## WHY
In modern JavaScript applications, you constantly receive massive JSON objects from backend APIs. If an API sends you a `user` object with 50 properties, and you only need their `firstName` and `email`, writing `const name = user.firstName; const email = user.email;` over and over again is incredibly tedious and clutters your code.

**Destructuring** is an elegant ES6 syntax that allows you to "unpack" or "extract" specific properties from Objects (or items from Arrays) directly into distinct variables in a single, clean line of code. It is one of the most heavily used features in modern JavaScript, and is the absolute foundation of how React Components receive properties (`props`).

## THEORY
### Unpacking Logic
Destructuring does NOT destroy the original object or array. It simply looks inside, copies the data you requested, and assigns it to brand new variables.

- **Object Destructuring**: Uses `{ }` brackets. It maps values based on the exact string **Key Name**. The order you extract them in does not matter at all.
- **Array Destructuring**: Uses `[ ]` brackets. Arrays don't have keys, so it maps values strictly based on **Index Order** (position).

## CODE

### Level 1 — Beginner (Object Destructuring)
```javascript
const user = {
  id: 101,
  username: "Alice",
  role: "Admin",
  status: "Active"
};

// THE OLD WAY:
// const username = user.username;
// const role = user.role;

// THE MODERN ES6 WAY:
// We open { } on the LEFT side of the equals sign.
// This tells JS: "Look inside the 'user' object. Find the key 'username' and make it a const. 
// Find the key 'role' and make it a const."
const { username, role } = user;

console.log(username); // "Alice"
console.log(role);     // "Admin"

// Note: If you ask for a key that doesn't exist, you just get undefined!
const { salary } = user;
console.log(salary); // undefined
```

### Level 2 — Building Blocks (Array Destructuring)
```javascript
const rgbColor = [255, 100, 50];

// THE OLD WAY:
// const red = rgbColor[0];
// const green = rgbColor[1];

// THE MODERN ES6 WAY:
// We use [ ] because it's an array. 
// Because arrays have no keys, it purely goes by ORDER!
// 255 goes into 'red', 100 goes into 'green'.
const [red, green, blue] = rgbColor;

console.log(red); // 255

// What if we only want the FIRST and THIRD items, but not the second?
// We just leave an empty comma!
const [firstValue, , thirdValue] = rgbColor;

console.log(thirdValue); // 50
```

### Level 3 — Intermediate (Renaming and Default Values)
```javascript
const apiResponse = {
  userName: "Bob123",
  // Notice 'bio' is completely missing!
};

// 1. RENAMING ALIASES
// 'userName' is a bad variable name for our app. Let's rename it to 'handle' instantly!
// Syntax: originalKey: newVariableName
const { userName: handle } = apiResponse;

console.log(handle); // "Bob123"
// console.log(userName); // ERROR! userName variable was never created.

// 2. DEFAULT FALLBACK VALUES
// Since 'bio' is missing from the API, it would normally be 'undefined'.
// We can provide a default value that ONLY applies if the actual value is undefined!
const { bio = "No biography provided." } = apiResponse;

console.log(bio); // "No biography provided."

// 3. DOING BOTH SIMULTANEOUSLY!
const { userName: handleName = "Guest" } = apiResponse;
```

### Level 4 — Advanced (Nested Destructuring & Function Params)
```javascript
const employee = {
  id: 5,
  personalInfo: {
    firstName: "Sarah",
    lastName: "Connor",
    contact: {
      email: "sarah@skynet.com",
      phone: "555-1234"
    }
  }
};

// 1. DEEP NESTED DESTRUCTURING
// We want to extract 'email' directly into a variable.
// The colon : means "dig deeper into this object". It does NOT create a 'personalInfo' variable!
const { 
  personalInfo: { 
    contact: { email } 
  } 
} = employee;

console.log(email); // "sarah@skynet.com"


// 2. FUNCTION PARAMETER DESTRUCTURING (Incredible power!)
// Instead of passing the whole 'employee' object and doing employee.id, 
// we can destructure the exact properties we need DIRECTLY in the parameter list!

function printBadge({ id, personalInfo: { firstName } }) {
  console.log(`Badge ID: ${id} | Name: ${firstName}`);
}

// We pass the massive object in, and the function unpacks exactly what it needs!
printBadge(employee); // "Badge ID: 5 | Name: Sarah"
```

## REAL_WORLD
If you look at any **React** code, destructuring is literally everywhere. 
In React, when a parent component passes data to a child component, the data is bundled into a massive object called `props`. Instead of accessing `props.title` and `props.isActive` inside the child component, React developers destructure the `props` object directly in the function definition!

```jsx
// React Component WITHOUT Destructuring (Verbose)
function UserCard(props) {
  return (
    <div>
      <h1>{props.name}</h1>
      <p>{props.email}</p>
    </div>
  );
}

// React Component WITH Destructuring (Industry Standard)
// We instantly unpack 'name' and 'email' from the incoming props object!
function UserCard({ name, email }) {
  return (
    <div>
      <h1>{name}</h1>
      <p>{email}</p>
    </div>
  );
}
```

## INTERVIEW
**Q: Can you destructure an Array into an Object?**
A: Technically yes, because Arrays are secretly Objects under the hood where the keys are the numerical indexes! If you do `const { 0: firstItem, 2: thirdItem } = ["Apple", "Banana", "Cherry"]`, the variable `firstItem` will be `"Apple"` and `thirdItem` will be `"Cherry"`. However, doing this is considered a bad practice as standard Array `[]` destructuring is much clearer.

**Q: In Object Destructuring, does the order of the variables matter?**
A: Absolutely not. `const { age, name } = user;` is identical to `const { name, age } = user;`. The V8 engine does not care about the order; it maps the data strictly by the literal string Key Name matching the property inside the object.

**Q: What happens if you try to destructure a property from `null` or `undefined`?**
A: It will trigger a fatal `TypeError` and crash your application (`Cannot destructure property 'name' of 'undefined' as it is undefined`). You must always ensure the object you are destructuring actually exists. If it might be null, you can use a fallback empty object: `const { name } = user || {};`.

## FEYNMAN CHECK

### Explain Destructuring Like I'm 10 Years Old
> Destructuring is SYNTAX SUGAR for pulling values out of arrays/objects into named variables in ONE line. `const {name, age} = user` is equivalent to `const name = user.name; const age = user.age`. `const [first, second] = arr` pulls the first two array elements. With defaults: `const {timeout = 5000} = config`. With renaming: `const {n: name} = obj`. With nested: `const {addr: {city}} = user`. With rest: `const [head, ...tail] = arr`. The non-obvious detail: destructuring on `undefined` or `null` THROWS — `const {x} = null` is TypeError. Always provide a default: `const {x} = config ?? {}`. Object destructuring in function PARAMS is the modern way to express named arguments without losing IDE help.

### 5 Deep Conceptual Questions

**Q1: When is destructuring worth it vs plain property access?**
> **A:** Use destructuring when you're pulling MORE THAN ONE value, OR you want defaults, OR you want renaming. For a single value, `config.url` is clearer than `const {url} = config`. For function parameters with 3+ options, destructuring is far better than a positional argument list — call sites are self-documenting (`fetch(url, { method: 'POST', headers })`).

**Q2: Mental model for destructuring?**
> **A:** Read destructuring as "describe the SHAPE on the left, JS will try to MATCH it on the right." `const {a, b: {c}} = obj` says: obj must have a property a (extract it), and a property b (which itself must have a c — extract that). It's pattern-matching-lite. Defaults apply when the value is undefined (not null, not 0, not ''). Renaming reverses syntax intuition: `const {a: x} = {a: 5}` means "extract a, call it x" — x = 5.

**Q3: Most dangerous misconception?**
> **A:** Destructuring null:
> ```js
> // ❌ Throws TypeError
> const { x } = null;          // TypeError: Cannot destructure property 'x' of 'null'
> const { x } = undefined;     // Same
>
> // ❌ Also: defaults only fire on undefined, not null
> const { count = 0 } = { count: null };
> console.log(count);           // null — default did NOT apply
>
> // ✅ Defend against null AND missing
> const { count = 0 } = obj ?? {};
> // Or normalise: const count = obj?.count ?? 0;
> ```

**Q4: How does parameter destructuring interact with arity?**
> **A:** `function f({a, b})` declares ONE parameter (an object). `f.length` is 1. The default arity-zero case: `function f({a, b} = {})` lets you call f() without args. Without the `= {}`, calling f() throws because you can't destructure undefined. The pattern `function f({a = 1, b = 2} = {})` is the canonical "named args with defaults" — every modern lib uses it.

**Q5: FAANG-grade definition?**
> **A:** "Destructuring assignment is JavaScript syntax that extracts values from arrays (positional) or objects (by key) into individual bindings — supporting defaults (for undefined), renaming via `prop: localName`, nested patterns, rest collection, and use in function parameters as the idiomatic named-arguments pattern — throwing TypeError when destructuring null or undefined without a fallback."

// options.js

// HTTP fetch wrapper — every option destructured, defaulted, validated
export async function request(url, {
  method = 'GET',
  headers = {},
  body,
  timeout = 5000,
  retries: { count: retryCount = 3, delay: retryDelay = 1000 } = {},
  signal,
} = {}) {
  const controller = new AbortController();
  const timer = setTimeout(() => controller.abort(), timeout);

  // Combine external signal with our timeout
  signal?.addEventListener('abort', () => controller.abort());

  let lastErr;
  for (let attempt = 1; attempt <= retryCount; attempt++) {
    try {
      const res = await fetch(url, {
        method, headers, body, signal: controller.signal,
      });
      clearTimeout(timer);
      return res;
    } catch (e) {
      lastErr = e;
      if (attempt < retryCount) await new Promise(r => setTimeout(r, retryDelay));
    }
  }
  clearTimeout(timer);
  throw lastErr;
}

// Demo: every form of destructuring in call sites
const cfg = {
  url: 'https://api.example.com',
  retries: { count: 5, delay: 2000 },
};

// 1. Destructure from object literal
const { url, retries: { count } = {} } = cfg;
console.log('URL:', url, 'retries:', count);

// 2. With rename + default
const { url: endpoint = '/fallback' } = cfg;

// 3. From function return

const [lat, lng] = getCoords();
console.log({ lat, lng });

// 4. Swap variables
let a = 1, b = 2;
[a, b] = [b, a];

#### Step 3 — Schema-Driven Options
```js
// schema.js
export function defineOptions(schema) {
  return (input = {}) => {
    const out = {};
    for (const [key, { default: def, required, validate }] of Object.entries(schema)) {
      const value = input[key] !== undefined ? input[key] : def;
      if (required && value === undefined) {
        throw new Error(`Missing required option: ${key}`);
      }
      if (validate && !validate(value)) {
        throw new Error(`Invalid option: ${key}`);
      }
      out[key] = value;
    }
    return out;
  };
}

const parseDbConfig = defineOptions({
  host: { default: 'localhost' },
  port: { default: 5432, validate: (n) => Number.isInteger(n) && n > 0 },
  user: { required: true },
  pool: { default: 10 },
});

console.log(parseDbConfig({ user: 'admin' }));
```

#### Step 4 — Error Handling: Destructure From Maybe-Null
```js
// safe.js — defensive destructuring
function getProfile(user) {
  // ❌ Throws if user is null/undefined:
  // const { name, email } = user;

  // ✅ Default to empty object to allow destructuring
  const { name = '<unknown>', email = '<none>' } = user ?? {};

  // ✅ For deeper nesting, combine optional chaining + ??
  const street = user?.address?.street ?? '<no street>';
  return { name, email, street };
}

console.log(getProfile(null));
console.log(getProfile({ name: 'Ana' }));
console.log(getProfile({ name: 'Ana', address: { street: 'Main St' } }));
```
