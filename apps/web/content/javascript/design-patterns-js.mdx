## WHY
As your JavaScript application grows from 1,000 lines of code to 100,000 lines of code, ad-hoc programming falls apart. How do you ensure only one instance of a Database Connection exists? How do you allow 50 different UI components to react when a single "User Logged In" event occurs without tightly coupling them together?
**Design Patterns** are standardized, time-tested architectural solutions to common software problems. Mastering them elevates you from someone who "writes code that works" to a Software Architect who designs scalable systems.

## THEORY
### Categories of Patterns
The "Gang of Four" (GoF) defined 23 classic design patterns in 1994, which are generally divided into three categories:
1. **Creational Patterns:** Deal with object creation mechanisms (e.g., Singleton, Factory).
2. **Structural Patterns:** Deal with object composition and relationships (e.g., Module, Proxy, Facade).
3. **Behavioral Patterns:** Deal with communication between objects (e.g., Observer, Mediator).

*Note: Because JavaScript is a dynamic, prototype-based language with First-Class Functions, many classic OOP patterns are natively built into the language or simplified!*

## CODE

### Level 1 — The Singleton Pattern (Creational)
A Singleton guarantees that exactly ONE instance of a class exists throughout the entire application lifecycle, and provides a global point of access to it.
Use cases: Database Connections, Logger Utilities, Global Application State.

```javascript
class DatabaseConnection {
  constructor() {
    // 1. Check if an instance already exists on the constructor itself!
    if (DatabaseConnection.instance) {
      // Return the cached instance instead of creating a new one!
      return DatabaseConnection.instance;
    }

    // 2. Otherwise, create the connection
    this.connectionString = "postgres://localhost:5432";
    this.isConnected = true;
    console.log("Expensive Database Connection Established!");

    // 3. Cache it for future calls!
    DatabaseConnection.instance = this;
  }

  query(sql) {
    console.log(`Executing: ${sql}`);
  }
}

// Testing the Singleton
const db1 = new DatabaseConnection(); // Prints: "Expensive Database Connection Established!"
const db2 = new DatabaseConnection(); // Does NOT print! Returns the cached instance.

console.log(db1 === db2); // TRUE! They are the exact same memory reference.
```
*Modern Alternative:* In ES6 Modules, if you `export const db = new DatabaseConnection()`, Node.js and Webpack automatically cache the exported module. Importing it into 50 different files will inherently act as a Singleton!

### Level 2 — The Factory Pattern (Creational)
Instead of using `new` directly everywhere in your code, you create a "Factory" function. The Factory encapsulates the complex logic of deciding *which* specific class to instantiate.

```javascript
// The Underlying Classes
class Car {
  constructor(doors) { this.doors = doors; this.type = "Car"; }
}
class Truck {
  constructor(doors) { this.doors = doors; this.type = "Truck"; }
}

// The Factory!
class VehicleFactory {
  // It takes a configuration object and decides what to build!
  static createVehicle(options) {
    if (options.vehicleType === "car") {
      return new Car(options.doors);
    } else if (options.vehicleType === "truck") {
      return new Truck(options.doors);
    }
    throw new Error("Unknown Vehicle Type");
  }
}

// The client doesn't need to know the specific classes exist!
const myCar = VehicleFactory.createVehicle({ vehicleType: "car", doors: 4 });
```

### Level 3 — The Observer Pattern (Behavioral)
This is the most important pattern in frontend development (it is the foundation of React Context, Redux, and DOM Event Listeners).
It defines a one-to-many relationship: One object (the Subject) holds state. When its state changes, it automatically notifies a list of dependents (the Observers).

```javascript
class Subject {
  constructor() {
    this.observers = []; // The list of subscribers
  }

  // Allow objects to subscribe
  subscribe(observerFunction) {
    this.observers.push(observerFunction);
  }

  // Allow objects to unsubscribe
  unsubscribe(observerFunction) {
    this.observers = this.observers.filter(obs => obs !== observerFunction);
  }

  // When an event happens, yell it to EVERYONE on the list!
  notify(data) {
    this.observers.forEach(observer => observer(data));
  }
}

// --- Usage ---
const userStore = new Subject();

// Observer 1 (e.g., A Header Component)
const header = (user) => console.log(`Header: Welcome ${user.name}!`);

// Observer 2 (e.g., An Analytics Tracker)
const analytics = (user) => console.log(`Analytics: User ${user.id} logged in.`);

// Both components subscribe to the global store!
userStore.subscribe(header);
userStore.subscribe(analytics);

// Somwhere else in the app, the user logs in:
// The store instantly notifies the header and the analytics system!
userStore.notify({ id: 1, name: "Alice" }); 
```

### Level 4 — The Module Pattern (Structural)
Before ES6 Classes and ES6 Modules existed, JavaScript had no concept of `private` variables. Everything attached to an object was public.
The Module Pattern uses **Closures** and **IIFEs** (Immediately Invoked Function Expressions) to emulate private variables!

```javascript
const ShoppingCartModule = (function () {
  // --- PRIVATE STATE ---
  // This array is trapped inside the closure. 
  // Nobody from the outside can ever access it directly!
  const cart = []; 

  function calculateTotal() {
    return cart.length * 10;
  }

  // --- PUBLIC API ---
  // We return an object containing the methods we WANT to expose.
  return {
    addItem: function (item) {
      cart.push(item);
      console.log(`${item} added!`);
    },
    checkout: function () {
      console.log(`Total is $${calculateTotal()}`);
    }
  };
})(); // Notice the () at the end? The function executes immediately!

ShoppingCartModule.addItem("Apple");
ShoppingCartModule.checkout();

// Attempting to hack the private state FAILS!
console.log(ShoppingCartModule.cart); // undefined
```

## REAL_WORLD
**The Facade Pattern**
If you look at the source code of `jQuery` or `Axios`, you are looking at the Facade Pattern.
Under the hood, making an AJAX request requires dealing with raw `XMLHttpRequest` objects, handling Internet Explorer 6 activeX fallbacks, managing headers, and parsing JSON manually. It is 50 lines of brutal code.
The library provides a **Facade**: a simple, beautiful, single method (`$.ajax()` or `axios.get()`) that completely hides the horrific complexity of the underlying subsystem. You should build Facades in your own codebase to hide complex third-party library integrations!

## INTERVIEW
**Q: You decide to use the Singleton Pattern to manage the global configuration (`AppConfig.getInstance()`) in a massive frontend application. Your Senior Architect rejects the PR, claiming Singletons are an "Anti-Pattern" in modern testing environments. Why are Singletons considered bad for Unit Testing?**
A: **Global State Persists Across Tests!**
Unit tests must run in complete isolation. If Test A runs, modifies the Singleton (e.g., `AppConfig.setTheme('dark')`), and finishes, the Singleton *remains in memory*.
When Test B runs, it expects a fresh, clean environment, but it gets the polluted Singleton from Test A. This causes "Flaky Tests"—tests that pass when run individually but fail randomly when run in a suite. 
*The Fix:* Instead of hardcoding `AppConfig.getInstance()` inside your components, you should pass the configuration into the component as a parameter via **Dependency Injection**. This allows the test runner to easily pass in a fake, temporary configuration object for each specific test!
