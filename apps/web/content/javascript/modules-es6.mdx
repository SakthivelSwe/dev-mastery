## WHY
In the early days of the web, JavaScript had no module system. Every variable and function you wrote was dumped into a single, massive global namespace (`window`). If File A defined `var name = "Alice"`, and File B defined `var name = "Bob"`, File B would silently overwrite File A's variable. This made building large applications a terrifying exercise in avoiding naming collisions.
**ES6 Modules (ESM)** introduced an official, standardized way to encapsulate code. Modules allow you to explicitly define what a file `exports` (makes public) and what it `imports` (consumes from other files), keeping the internal logic strictly private and scoped!

## THEORY
### The Module Scope
When you declare a file as an ES6 Module, it immediately gets its own private scope. Variables defined inside the module do NOT leak into the global `window` object. 
To share code between modules, you must use the `export` and `import` keywords.

## CODE

### Level 1 — Named Exports and Imports
You can export multiple specific variables, functions, or classes from a single file. These are called **Named Exports**.

```javascript
// --- math.js (The Provider) ---

// Exporting variables directly
export const PI = 3.14159;

// Exporting functions directly
export function add(a, b) {
  return a + b;
}

// A private variable! It is NOT exported.
// No other file can ever access this!
const internalSecret = "xyz"; 
```

When importing, you must use the exact same name, wrapped in curly braces.
```javascript
// --- app.js (The Consumer) ---

// Import exactly what you need
import { PI, add } from './math.js';

console.log(PI); // 3.14159
console.log(add(5, 10)); // 15
```

### Level 2 — Default Exports
If a file's primary purpose is to export *one main thing* (like a React Component), you should use a **Default Export**. A module can have multiple Named exports, but only ONE Default export.

```javascript
// --- User.js ---

export default class User {
  constructor(name) {
    this.name = name;
  }
}

// You can still have named exports in the same file!
export const USER_ROLES = ["ADMIN", "GUEST"];
```

When importing a Default Export, you do NOT use curly braces, and you can name the import whatever you want!
```javascript
// --- app.js ---

// 'User' is the default import. 'USER_ROLES' is the named import.
import User, { USER_ROLES } from './User.js';

const u = new User("Alice");
```

### Level 3 — Aliasing and Namespace Imports
If you import a function named `add` from `math.js`, and you already have an `add` function in your current file, you will get a naming collision.
You can use the `as` keyword to rename imports on the fly!

```javascript
import { add as mathAdd } from './math.js';

function add() {
  console.log("Local add function");
}

mathAdd(5, 5); // Uses the imported function
```

Alternatively, if a file exports 50 different utilities, you can import them all into a single Namespace object.
```javascript
import * as MathUtils from './math.js';

console.log(MathUtils.PI);
console.log(MathUtils.add(5, 10));
```

### Level 4 — Dynamic Imports
Standard `import` statements must be placed at the very top of the file, and they are statically analyzed. The browser will download all imported files immediately before executing the script.
What if you only want to download a heavy charting library IF the user clicks the "View Analytics" button?
You use **Dynamic Imports** (`import()`), which return a Promise!

```javascript
document.getElementById('chartBtn').addEventListener('click', async () => {
  // The browser makes a network request to fetch 'heavyChart.js' right now!
  try {
    const chartModule = await import('./heavyChart.js');
    
    // Access the exported functions from the loaded module
    chartModule.renderChart();
  } catch (error) {
    console.error("Failed to load module!", error);
  }
});
```

## REAL_WORLD
**ESM vs. CommonJS**
JavaScript actually has *two* wildly popular module systems, which has caused immense pain in the ecosystem.
1. **CommonJS (CJS):** Designed for Node.js servers in 2009. Uses `require()` and `module.exports`. It is strictly synchronous (it blocks execution while reading the file from the hard drive).
2. **ES Modules (ESM):** Designed for the Browser in 2015. Uses `import` and `export`. It is asynchronous (allowing the browser to download multiple modules in parallel over the network).

For years, Node.js only supported CommonJS, while browsers only supported ESM. 
Today, Node.js supports ESM natively (if you add `"type": "module"` to your `package.json`), and the entire ecosystem is painfully migrating away from `require()` to `import`.

## INTERVIEW
**Q: You write an `index.html` file and link your JavaScript file using `<script src="app.js"></script>`. Inside `app.js`, you write `import { User } from './user.js'`. When you open the HTML file in Chrome, the console throws a massive error: "Uncaught SyntaxError: Cannot use import statement outside a module". Why did this happen and how do you fix it?**
A: **The Script vs. Module distinction.**
By default, browsers treat all `<script>` tags as legacy "Scripts" (which share the global `window` scope and do not support `import/export` syntax).
To tell the browser engine that `app.js` is an ES6 Module, you MUST add the `type="module"` attribute to the script tag in your HTML!
```html
<!-- The Fix -->
<script type="module" src="app.js"></script>
```
*Bonus:* When a script is loaded as `type="module"`, it automatically acts like it has a `defer` attribute, meaning the browser won't block HTML parsing while it downloads the JavaScript!
