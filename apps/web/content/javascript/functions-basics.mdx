## WHY
If you write 50 lines of code to process a user's login, and you need to process a login on 3 different pages, copying and pasting those 50 lines of code is a maintenance nightmare. If you find a bug, you have to fix it in 3 different places.

**Functions** are reusable blocks of code. You wrap those 50 lines into a single Function, give it a name (`processLogin`), and then simply "call" the name whenever you need that logic executed. 
In JavaScript, Functions are incredibly powerful. Unlike Java or C++, where functions are rigid structures attached to classes, JavaScript treats Functions as "First-Class Citizens". You can assign a function to a variable, pass a function into another function as an argument, and even return a function from a function!

## THEORY
### The Evolution of the Function
JavaScript has three primary ways to write a function:
1. **Function Declaration**: The oldest, most classic syntax. It is "hoisted" to the top of the file, meaning you can call it before you define it.
2. **Function Expression**: Storing an anonymous function inside a `const` variable. It forces you to write cleaner, top-down code because it is NOT hoisted.
3. **Arrow Functions (`=>`)**: Introduced in ES6 (2015). A shorter, cleaner syntax that completely changes how the `this` keyword behaves. This is the absolute industry standard in React and modern Node.js.

## CODE

### Level 1 — Beginner (Function Declarations & Arguments)
```javascript
// 1. Define the Function (It does NOT execute yet!)
// 'name' is a Parameter (a placeholder variable).
function greetUser(name) {
  console.log(`Welcome to the system, ${name}!`);
}

// 2. Execute (Call / Invoke) the Function
// "Alice" is an Argument (the actual data passed in).
greetUser("Alice");
greetUser("Bob");

// 3. Returning Data
// Functions can calculate data and 'return' it back to the main program.
function addNumbers(a, b) {
  return a + b; // The function instantly stops and spits this value out
}

const result = addNumbers(5, 10);
console.log(result); // 15
```

### Level 2 — Building Blocks (Function Expressions)
```javascript
// In JS, a function is just a piece of Data, exactly like a String or a Number!
// Therefore, we can store an "Anonymous" (unnamed) function directly into a variable!

const multiplyNumbers = function(a, b) {
  return a * b;
};

console.log(multiplyNumbers(4, 5)); // 20

// WHY DO THIS? 
// Because 'multiplyNumbers' is a 'const', it is NOT hoisted!
// If you try to call multiplyNumbers() at the top of the file, JS will throw a ReferenceError.
// This forces developers to structure their code logically (declare first, use later), 
// preventing massive spaghetti-code bugs.
```

### Level 3 — Intermediate (Arrow Functions)
```javascript
// In ES6, the Arrow Function was introduced to remove the bulky 'function' keyword.

// 1. Standard Arrow Function
const divide = (a, b) => {
  return a / b;
};

// 2. The Implicit Return (The Magic of Arrows!)
// If your function is exactly ONE line of code, you can delete the { braces } 
// AND delete the 'return' keyword! It automatically returns the result!
const subtract = (a, b) => a - b;

console.log(subtract(10, 3)); // 7

// 3. Single Parameter Shorthand
// If you have exactly ONE parameter, you can even delete the ( parentheses )!
const square = num => num * num;

console.log(square(5)); // 25
```

### Level 4 — Advanced (Default Parameters & Rest Operator)
```javascript
// What happens if you forget to pass an argument? JS assigns 'undefined'.
function calculateTax(price, taxRate) {
  return price + (price * taxRate); // 100 * undefined = NaN!
}
console.log(calculateTax(100)); // NaN (Disaster!)

// 1. DEFAULT PARAMETERS
// We can assign a fallback value directly in the parameter list!
function safeCalculateTax(price, taxRate = 0.05) {
  return price + (price * taxRate);
}
console.log(safeCalculateTax(100)); // 105 (Uses the default 0.05)
console.log(safeCalculateTax(100, 0.20)); // 120 (Overrides the default)

// 2. THE REST OPERATOR (...)
// What if you want a function to accept an infinite number of arguments?
// The ... operator gathers all remaining arguments into a pure Array!
function calculateTotal(discount, ...prices) {
  // 'prices' is an array: [10, 20, 30]
  let total = 0;
  for (const p of prices) {
    total += p;
  }
  return total - discount;
}

console.log(calculateTotal(5, 10, 20, 30)); // 55
```

## REAL_WORLD
Arrow functions are the undisputed king of **React** development. 
In modern React (using Functional Components), every single component you build, every event listener you attach, and every piece of state you update uses Arrow Functions.

```jsx
// A modern React Component written entirely with Arrow Functions
const CheckoutButton = ({ price }) => {
  
  // An arrow function for the click event
  const handlePayment = () => {
    console.log(`Charging card for ${price}`);
  };

  return (
    <button onClick={handlePayment}>
      Pay Now
    </button>
  );
};
```
If you are uncomfortable with the `const myFunc = () => {}` syntax, you must practice it until it becomes muscle memory. You will rarely, if ever, see the traditional `function` keyword in a modern enterprise frontend codebase.

## INTERVIEW
**Q: Are Functions Objects in JavaScript?**
A: Yes! Functions are technically "Callable Objects". Because they are objects under the hood, you can actually attach properties directly to a function (e.g., `greetUser.counter = 1`). This is how functions can be assigned to variables and passed around memory just like any other object.

**Q: What is the main architectural difference between a Function Declaration and an Arrow Function?**
A: The most critical difference (besides the syntax and hoisting) is how they handle the `this` keyword. A standard `function` creates its own `this` context based on *how* it was called (e.g., who executed it). An Arrow Function does NOT create its own `this` context; it inherits `this` directly from the surrounding scope (Lexical Scoping). This makes Arrow functions perfectly suited for callbacks inside classes or objects.

**Q: If a function does not have a `return` keyword, what does it return?**
A: `undefined`. If the JavaScript engine reaches the final `}` bracket of a function without encountering a `return` statement, it automatically returns `undefined` to the caller.

## FEYNMAN CHECK

### Explain JS Functions Like I'm 10 Years Old
> A function is a REUSABLE BLOCK OF CODE that takes inputs (parameters) and returns an output. JS has FOUR ways to declare them: function declaration (`function f(){}` — hoisted, gets a name), function expression (`const f = function(){}` — not hoisted, name optional), arrow function (`const f = () => {}` — no own `this`, no `arguments`, no `new`, compact), and method shorthand inside objects (`{ f() {} }`). Functions in JS are FIRST-CLASS VALUES — you can pass them as arguments, return them from other functions, store them in arrays, attach them as object properties. The non-obvious detail: every function has a `length` property (number of declared params before the first default), a `name` property (often inferred from assignment), and access to `arguments` (except arrow functions).

### 5 Deep Conceptual Questions

**Q1: When do you use function declaration vs arrow vs expression?**
> **A:** Use ARROW for: callbacks, methods that need lexical `this` (React render functions, array methods), short pure expressions. Use FUNCTION DECLARATION for: top-level utilities you want hoisted, anything that benefits from a stack trace name, generators. Use METHOD SHORTHAND in object literals. Avoid bare function expressions (`const f = function(){}`) — arrows are shorter and the lack of `this` is usually what you want.

**Q2: Mental model for hoisting?**
> **A:** `function declaration` is hoisted with its full body — you can call it BEFORE its source line. `function expression` and arrow function assigned to `var` are partially hoisted (the var binding is hoisted to undefined). `let/const` arrow functions are in TDZ before the line. Modern style: prefer arrows + const for safety, even though it means defining helpers above their callers (or extracting them).

**Q3: Most dangerous misconception?**
> **A:** Default parameters are evaluated lazily — once per call:
> ```js
> // ❌ Trap: shared default mutable object
> function append(item, list = []) { list.push(item); return list; }
> append(1);   // [1]
> append(2);   // [2]  — defaults are CREATED PER CALL, not shared
>
> // BUT — outer scope mutation can bite:
> function counter(start = ++globalSeed) { return start; }   // globalSeed advances per call
>
> // ✅ Default expressions can also reference earlier params
> function range(start, end = start + 10) { return [start, end]; }
> ```

**Q4: How do arrow functions differ from regular functions at runtime?**
> **A:** Arrow functions do not bind their own `this` — they capture it from the enclosing lexical scope (closure-style). They have no `arguments` (use rest params `...args`), no `prototype` property (can't be used with `new`), and no `super`. They are slightly cheaper to construct (no prototype object). React encourages arrows in render methods specifically BECAUSE they preserve `this`; class component methods that need handling outside (event handlers) require .bind or arrow class fields.

**Q5: FAANG-grade definition?**
> **A:** "JavaScript functions are first-class callable objects with closures over their lexical environment — declared via function declarations (hoisted), function expressions (assigned), arrow functions (lexical-this, no arguments, no prototype, no new), or method shorthand — with parameter defaults evaluated per-call, rest parameters collecting variadic arguments into a real array, and IIFEs providing pre-modules scope isolation."

#### Step 2 — Core Combinators
```js
// combinators.js

// pipe(f, g, h)(x) = h(g(f(x)))  — left-to-right
export const pipe = (...fns) => (x) => fns.reduce((acc, fn) => fn(acc), x);

// compose(f, g, h)(x) = f(g(h(x)))  — right-to-left (mathematical)
export const compose = (...fns) => (x) => fns.reduceRight((acc, fn) => fn(acc), x);

// curry(fn) — fn(a, b, c) becomes fn(a)(b)(c) or fn(a, b)(c) etc.
export function curry(fn) {
  return function curried(...args) {
    if (args.length >= fn.length) return fn.apply(this, args);
    return (...more) => curried.apply(this, [...args, ...more]);
  };
}

// partial(fn, ...preset) — locks the first few arguments
export const partial = (fn, ...preset) => (...rest) => fn(...preset, ...rest);

// once(fn) — runs fn at most one time, caches the result
export function once(fn) {
  let called = false, value;
  return function (...args) {
    if (called) return value;
    called = true;
    value = fn.apply(this, args);
    return value;
  };
}
```

#### Step 3 — Use Them For Data Transformation
```js
// demo.js
import { pipe, compose, curry, partial, once } from './combinators.js';

const trim   = (s) => s.trim();
const lower  = (s) => s.toLowerCase();
const slugify = pipe(trim, lower, (s) => s.replace(/\s+/g, '-'));
console.log(slugify('  Hello World  '));   // 'hello-world'

const add = curry((a, b, c) => a + b + c);
console.log(add(1)(2)(3));   // 6
console.log(add(1, 2)(3));   // 6
console.log(add(1)(2, 3));   // 6

const greet = (greeting, name) => `${greeting}, ${name}!`;
const hello = partial(greet, 'Hello');
console.log(hello('Ana'));   // 'Hello, Ana!'

const initDb = once(() => { console.log('actually connecting...'); return { connected: true }; });
initDb();  // 'actually connecting...'
initDb();  // (silent — returns cached value)
```

// tap(fn) runs side effect, returns the value unchanged — useful in pipes
export const tap = (fn) => (x) => { fn(x); return x; };
