## WHY
In most programming languages, when a function finishes executing, all the variables inside that function are instantly destroyed by the computer to save memory. 
JavaScript has a magical exception to this rule called a **Closure**. A Closure allows a function to "remember" the variables that existed around it at the exact moment it was born, even if the surrounding function finished executing hours ago. 

Understanding Closures is the defining characteristic that separates Junior JavaScript developers from Mid-level developers. Closures are the secret mechanism that powers Data Privacy, Event Listeners, and the entire `useState` Hook system in React.

## THEORY
### Lexical Scoping
JavaScript uses "Lexical Scoping" (Static Scoping). This means the JavaScript engine determines the scope of a variable strictly based on *where it is physically typed in the source code*, not based on who called the function.
An inner function will always have access to the variables of its outer function.

### The Closure Definition
A **Closure** is created when an Inner Function is returned (or exported) out of its Outer Function. 
Even though the Outer function finishes executing and dies, the Inner Function takes a "backpack" with it containing all the variables it needs from the Outer Function's scope. The V8 Garbage Collector sees this backpack and is mathematically forbidden from deleting those variables from the RAM.

## CODE

### Level 1 — Beginner (The Basic Closure)
```javascript
// 1. The Outer Function
function createGreeting(greetingWord) {
  
  // 2. The Inner Function
  // It uses 'greetingWord' from the outer scope!
  function sayHello(name) {
    console.log(`${greetingWord}, ${name}!`);
  }
  
  // 3. We RETURN the Inner Function itself! (We don't execute it here)
  return sayHello;
}

// 'createGreeting' executes, sets greetingWord to "Howdy", and dies.
// It returns the 'sayHello' function into this variable!
const sayHowdy = createGreeting("Howdy");

// Wait, createGreeting is dead. How does sayHowdy know what the word is?!
// Because of CLOSURE! sayHowdy dragged "Howdy" out in its backpack!
sayHowdy("Alice"); // "Howdy, Alice!"
sayHowdy("Bob");   // "Howdy, Bob!"
```

### Level 2 — Building Blocks (Data Privacy / Encapsulation)
```javascript
// Before modern JS Classes, Closures were the ONLY way to create "Private" variables!

function createBankAccount(initialBalance) {
  // This variable is trapped inside the function scope.
  // It is mathematically impossible to access it from the outside world!
  let balance = initialBalance; 
  
  // We return an object containing functions (Methods).
  // These methods form a Closure over the 'balance' variable!
  return {
    deposit: function(amount) {
      balance += amount;
      console.log(`Deposited ${amount}. New balance: ${balance}`);
    },
    withdraw: function(amount) {
      if (amount > balance) return console.log("Insufficient funds!");
      balance -= amount;
      console.log(`Withdrew ${amount}. New balance: ${balance}`);
    },
    getBalance: function() {
      return balance;
    }
  };
}

const myAccount = createBankAccount(100);

myAccount.deposit(50); // 150
myAccount.withdraw(20); // 130

// Try to hack the bank!
console.log(myAccount.balance); // undefined! (It is completely hidden!)
myAccount.balance = 1000000; // This just creates a useless public variable.
console.log(myAccount.getBalance()); // 130 (The real, private balance is safe!)
```

### Level 3 — Intermediate (Function Factories)
```javascript
// Closures allow you to write generic "Factory" functions that generate
// highly specialized functions on the fly!

// A generic multiplier factory
function createMultiplier(multiplier) {
  return function(number) {
    return number * multiplier;
  };
}

// We generate a specialized function that ALWAYS multiplies by 2
const double = createMultiplier(2);

// We generate a specialized function that ALWAYS multiplies by 10
const decuple = createMultiplier(10);

console.log(double(5));   // 10
console.log(double(100)); // 200

console.log(decuple(5));  // 50
```

### Level 4 — Advanced (The Infamous Loop Interview Question)
```javascript
// THE PROBLEM (Written with 'var'):
function createButtonsBad() {
  for (var i = 1; i <= 3; i++) {
    // We simulate attaching an event listener to 3 buttons
    setTimeout(function() {
      console.log(`Button ${i} clicked!`);
    }, 1000);
  }
}

// When the timers finish 1 second later, what prints?
// Output: "Button 4 clicked!" "Button 4 clicked!" "Button 4 clicked!"
// WHY?! Because 'var' is Function Scoped! There is only ONE 'i' variable in RAM.
// The loop finishes instantly, setting 'i' to 4. 
// When the closures finally run 1 second later, they all look at the exact same 'i'.

// THE SOLUTION (Written with 'let'):
function createButtonsGood() {
  for (let i = 1; i <= 3; i++) {
    setTimeout(function() {
      console.log(`Button ${i} clicked!`);
    }, 1000);
  }
}

// Output: "Button 1" "Button 2" "Button 3"
// WHY?! Because 'let' is Block Scoped! The JS engine mathematically creates 
// a BRAND NEW 'i' variable in RAM for every single iteration of the loop!
// Each timeout closure gets its own personal, locked copy of 'i'!
```

## REAL_WORLD
Closures are the invisible foundation of **React Hooks** (`useState` and `useEffect`).

When you call `const [count, setCount] = useState(0)` in a React Component, how does React remember the value of `count` between renders? The Component function finishes executing and dies every time the screen paints!
React uses a massive, hidden Closure architecture under the hood. The `useState` function reaches outside of your component, stores the `count` data in a private array in the React Engine, and returns a `setCount` function that has a Closure connection back to that exact memory slot. 

Furthermore, if you attach an event listener in vanilla JS, you are using closures:
```javascript
function attachClickTracking(buttonId) {
  let clickCount = 0; // Private state
  
  document.getElementById(buttonId).addEventListener('click', () => {
    clickCount++; // The event listener has a closure over clickCount!
    console.log(`Clicked ${clickCount} times!`);
  });
}
```

## INTERVIEW
**Q: What is a Closure?**
A: A closure is the combination of a function bundled together with references to its surrounding lexical state. Simply put, it gives an inner function access to an outer function's scope, even after the outer function has finished executing and returned.

**Q: Do Closures cause Memory Leaks?**
A: They can, if used improperly. Because a closure forces the V8 engine to keep outer variables alive in RAM (preventing them from being Garbage Collected), holding onto a massive variable (like a 10MB array) inside a closure that never dies will permanently consume that RAM. If you attach a closure to the global `window` object via an event listener and forget to remove it, you create a permanent memory leak.

**Q: How does Lexical Scoping differ from Dynamic Scoping?**
A: JavaScript uses Lexical Scoping, meaning scope is determined by the physical location of the code when it is *written* (compiled). Inner functions always look "up and out" to find variables in the surrounding text blocks. Dynamic Scoping (used in bash/Perl) determines scope based on the call stack at *runtime*—meaning a function's variables change depending on who called it. (JS partially simulates Dynamic Scoping via the `this` keyword, which is why `this` is so confusing!)
