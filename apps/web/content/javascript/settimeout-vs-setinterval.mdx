## WHY
In JavaScript, you often need to delay the execution of code (e.g., "Hide this toast notification after 3 seconds") or run code repeatedly (e.g., "Poll the server for new messages every 5 seconds").
Because JavaScript is single-threaded and non-blocking, it does not have a `sleep(5000)` function like Python or Java. If you use a `while` loop to kill time, you will freeze the browser.
Instead, JavaScript relies on the Browser's Timer APIs: **`setTimeout`** and **`setInterval`**.

## THEORY
### The "Minimum" Delay Guarantee
The most critical concept to understand about Timers is that **the time you specify is a minimum delay, not an exact execution time.**
When you write `setTimeout(myFunc, 1000)`, you are telling the browser: *"Wait 1000ms, and then place `myFunc` into the Task Queue."*
If the main thread (Call Stack) is busy doing heavy calculations for another 500ms, `myFunc` will be forced to wait in the queue. The actual execution time will be 1500ms.

## CODE

### Level 1 — `setTimeout` (One-time Execution)
`setTimeout` schedules a function to run exactly once after a specified delay. It returns a unique ID that you can use to cancel the timer before it fires!

```javascript
// Schedule the function to run after 3000ms (3 seconds)
const timerId = setTimeout(() => {
  console.log("This toast message is now hidden!");
}, 3000);

// If the user manually clicks "X" to close the toast before 3 seconds pass,
// we MUST cancel the timer so it doesn't fire anyway!
document.getElementById('closeBtn').addEventListener('click', () => {
  clearTimeout(timerId);
  console.log("Timer cancelled!");
});
```

### Level 2 — `setInterval` (Repeated Execution)
`setInterval` runs a function repeatedly, pausing for the specified delay *between* each execution.

```javascript
// Update a clock on the screen every 1000ms (1 second)
const intervalId = setInterval(() => {
  const time = new Date().toLocaleTimeString();
  document.getElementById('clock').innerText = time;
}, 1000);

// Stop the clock when a button is clicked
document.getElementById('stopClockBtn').addEventListener('click', () => {
  clearInterval(intervalId);
});
```

### Level 3 — The `setInterval` Drift Problem
`setInterval` is notoriously dangerous for operations that take time to execute (like an API call).
Imagine you poll the server every 2000ms using `setInterval`.
If the server is slow and the network request takes 3000ms to finish, the next `setInterval` will fire *before* the first one finished! The browser queues them up, creating a massive backlog of overlapping requests that will crash the browser and DDoS your own server.

**The Fix: Recursive `setTimeout`**
Instead of `setInterval`, senior developers use a recursive `setTimeout`. This guarantees that the next timer will NOT start ticking until the previous execution has completely finished!

```javascript
// BAD: setInterval (Prone to overlapping requests)
// setInterval(fetchNewMessages, 2000);

// GOOD: Recursive setTimeout (Self-adjusting polling)
async function pollServer() {
  try {
    // Wait for the slow API call to FINISH completely!
    await fetchNewMessages(); 
  } finally {
    // Only AFTER it finishes, schedule the next call 2000ms from now!
    setTimeout(pollServer, 2000);
  }
}

// Kick it off!
pollServer();
```

### Level 4 — Debouncing (The Classic UI Pattern)
When a user types into a Search input, you want to query the API. If you query the API on every single keystroke, you will fire 10 requests a second, destroying performance.
You use `setTimeout` to implement **Debouncing**: *"Wait until the user stops typing for 500ms before making the API call."*

```javascript
function debounce(func, delay) {
  let timerId;
  
  // Return a newly wrapped function
  return function (...args) {
    // 1. If the user types again, clear the old timer!
    clearTimeout(timerId);
    
    // 2. Set a brand new timer!
    timerId = setTimeout(() => {
      func.apply(this, args);
    }, delay);
  };
}

const performSearch = (query) => console.log(`Searching API for: ${query}`);

// Create the debounced version
const debouncedSearch = debounce(performSearch, 500);

// Attach to the input field
document.getElementById('searchInput').addEventListener('input', (e) => {
  debouncedSearch(e.target.value);
});
```

## REAL_WORLD
**requestAnimationFrame (rAF)**
If you are building an HTML5 Game or a complex physics animation, you should NEVER use `setInterval` or `setTimeout`.
Timers are entirely disconnected from the monitor's refresh rate. If your monitor refreshes at 60Hz, using `setInterval(draw, 16)` will eventually drift out of sync, causing visual stuttering (screen tearing).
Modern browsers provide `requestAnimationFrame(callback)`. This API tells the browser: *"Right before you paint the very next frame to the screen, execute my callback!"* It guarantees perfectly smooth 60FPS animations and automatically pauses itself if the user switches browser tabs (saving battery life!).

## INTERVIEW
**Q: You have a memory leak in a React component. The component fetches data using `setInterval`, but when the user navigates away to a different page, the network tab shows the API requests are still firing every 5 seconds. Why did this happen and how do you fix it?**
A: **Orphaned Timers.**
`setInterval` is attached to the global `window` object. It exists outside the lifecycle of a React component. 
When a React component is unmounted (destroyed), any intervals or timeouts created inside it will continue running in the background forever unless explicitly cancelled. The callback inside the interval will still try to update React state, causing the classic "Can't perform a React state update on an unmounted component" error, or firing ghost network requests.
*The Fix:* In React, you must use the `useEffect` cleanup function to clear the interval!
```javascript
useEffect(() => {
  const timerId = setInterval(fetchData, 5000);
  
  // React runs this cleanup function when the component unmounts!
  return () => clearInterval(timerId); 
}, []);
```
