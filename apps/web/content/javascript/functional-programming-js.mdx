## WHY
Object-Oriented Programming (OOP) groups data (properties) and logic (methods) together into Classes. While powerful, OOP in massive JavaScript applications often leads to highly complex state mutations, where a method magically changes an object's internal data, creating untraceable bugs.
**Functional Programming (FP)** is an alternative paradigm. It separates data from logic entirely. It treats functions as mathematical equations: given an input, always return the exact same output, without ever modifying the original data. 
React, Redux, and modern JavaScript heavily rely on FP concepts (like immutability and pure functions) to guarantee predictable UIs.

## THEORY
### The Core Tenets of FP
1. **First-Class Functions:** Functions are treated just like variables. You can pass them as arguments to other functions, or return them from functions.
2. **Pure Functions:** A function must always return the exact same output for the same input. It cannot rely on global variables.
3. **No Side Effects:** A function cannot mutate external data, modify the DOM, or make API calls. It only computes and returns a new value.
4. **Immutability:** Once data is created, it cannot be changed. Instead of modifying an array, you create a brand new copy of the array with the changes applied.

## CODE

### Level 1 — Pure vs. Impure Functions
An impure function relies on or modifies data outside its scope. A pure function is an isolated mathematical island.

```javascript
// BAD: Impure Function!
// It relies on a global variable. If 'taxRate' changes, the output changes!
let taxRate = 0.2;
function calculateTaxImpure(amount) {
  return amount * taxRate;
}

// GOOD: Pure Function!
// It relies strictly on its inputs. 
// calculateTaxPure(100, 0.2) will equal 20 today, tomorrow, and forever.
function calculateTaxPure(amount, rate) {
  return amount * rate;
}
```

### Level 2 — Immutability (Don't Mutate!)
In FP, if you want to add an item to an array, you do NOT use `.push()`, because `.push()` modifies the original array in memory (mutation). You use the Spread Operator to create a brand new array!

```javascript
const cart = ['Apple', 'Banana'];

// BAD: Mutates the original data! 
// This breaks React state updates!
function addItemMutative(arr, item) {
  arr.push(item);
  return arr;
}

// GOOD: Immutable! Creates a brand new array.
function addItemImmutable(arr, item) {
  return [...arr, item];
}

const newCart = addItemImmutable(cart, 'Orange');
// 'cart' is still exactly ['Apple', 'Banana']!
```

### Level 3 — Higher-Order Functions (HOFs)
Because functions are values, you can pass them into other functions! A Higher-Order Function is a function that takes a function as an argument, or returns a function.
JavaScript's array methods (`.map`, `.filter`, `.reduce`) are classic HOFs.

```javascript
const numbers = [1, 2, 3, 4, 5];

// We pass an anonymous pure function into the .filter() HOF!
// .filter() loops over the array, applying our function to each item.
const evens = numbers.filter((n) => n % 2 === 0);

// .map() creates a brand new array (Immutability!) based on the math function we pass!
const doubledEvens = evens.map((n) => n * 2);

console.log(doubledEvens); // [4, 8]
```

### Level 4 — Currying and Composition
Currying is transforming a function that takes multiple arguments (`f(a, b, c)`) into a sequence of functions that take one argument each (`f(a)(b)(c)`).
This allows you to "pre-load" a function with data and reuse it!

```javascript
// Standard function
const add = (a, b) => a + b;

// Curried function
const curriedAdd = (a) => {
  return (b) => {
    return a + b;
  };
};

// We pre-load the 'a' variable with 5!
const addFive = curriedAdd(5); 

// Now we have a specialized function we can use anywhere!
console.log(addFive(10)); // 15
console.log(addFive(20)); // 25
```

## REAL_WORLD
**Redux Reducers**
If you look at the architecture of Redux (the state management tool for React), it is built entirely on pure Functional Programming.
A Redux **Reducer** is defined as a Pure Function: `(previousState, action) => newState`.
It explicitly forbids API calls (Side Effects), and it explicitly forbids `state.value = 5` (Mutation). It forces you to return a brand new object.
By strictly adhering to FP principles, Redux guarantees that if you give it the same initial state and replay the same sequence of actions, the UI will arrive at the exact same final state every single time, making "Time Travel Debugging" possible!

## INTERVIEW
**Q: You want to sort an array of objects by their `price` property. You write: `const sorted = items.sort((a,b) => a.price - b.price);`. Later in the app, you notice that the original `items` array was also sorted, completely breaking the UI that depended on the original order. What FP rule did you break, and how do you fix it?**
A: **The `sort()` method is Mutative!**
Unlike `.map()` and `.filter()`, which immutably return brand new arrays, JavaScript's native `.sort()`, `.reverse()`, and `.splice()` methods physically mutate the original array in memory. You violated the rule of Immutability!
*The Fix:* You must clone the array before sorting it.
In older JavaScript, you used the spread operator to clone it: `const sorted = [...items].sort(...)`.
In modern JavaScript (ES2023+), you can use the new immutable version of the method: `const sorted = items.toSorted((a,b) => a.price - b.price);`. This method automatically returns a cloned, sorted array while leaving the original untouched!
