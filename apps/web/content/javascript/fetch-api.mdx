## WHY
For over a decade, JavaScript developers communicated with servers using the `XMLHttpRequest` (XHR) object. It was incredibly verbose, callback-heavy, and notoriously difficult to read.
In 2015, modern browsers introduced the **Fetch API**. It is a modern, clean, Promise-based interface for making HTTP requests. It completely replaced XHR and became the foundational network layer for all modern JavaScript applications.

## THEORY
### Request and Response Streams
The Fetch API does not immediately download the entire payload into memory. It returns a Promise that resolves as soon as the server sends the HTTP **Headers**, before the actual body (data) arrives.
This allows you to check the status code (`404 Not Found`, `200 OK`) or read the headers *before* deciding whether you want to spend the time and memory downloading a massive 50MB JSON payload!

## CODE

### Level 1 — The Basic GET Request
By default, `fetch()` performs an HTTP `GET` request.

```javascript
// fetch() returns a Promise!
fetch('https://jsonplaceholder.typicode.com/users')
  .then((response) => {
    // 1. The Promise resolves when the HEADERS arrive.
    // We must call .json() to read the BODY stream and parse it into an object.
    // Note: .json() ALSO returns a Promise!
    return response.json(); 
  })
  .then((data) => {
    // 2. This runs when the full body is downloaded and parsed.
    console.log("Users:", data);
  });
```

Using `async/await` makes it much cleaner:
```javascript
async function getUsers() {
  const response = await fetch('https://jsonplaceholder.typicode.com/users');
  const data = await response.json();
  console.log("Users:", data);
}
```

### Level 2 — POSTing Data
To send data to the server (e.g., creating a new user), you pass an options object as the second argument to configure the `method`, `headers`, and the `body` (which must be a string!).

```javascript
async function createUser(newUser) {
  const response = await fetch('https://api.example.com/users', {
    method: 'POST', // Use POST to create data
    headers: {
      // You MUST tell the server you are sending JSON!
      'Content-Type': 'application/json',
      // Pass authentication tokens here!
      'Authorization': 'Bearer YOUR_TOKEN_HERE' 
    },
    // You MUST stringify the JavaScript object into a JSON string!
    body: JSON.stringify(newUser) 
  });

  const data = await response.json();
  return data;
}

createUser({ name: "Alice", email: "alice@test.com" });
```

### Level 3 — The Error Handling Trap
The most dangerous thing about `fetch` is how it handles HTTP Errors (like `404 Not Found` or `500 Internal Server Error`).
**`fetch()` DOES NOT REJECT on HTTP Errors!**
The `fetch` Promise only rejects (triggers a `.catch()` block) if there is a literal network failure (e.g., the user's WiFi disconnected, or DNS resolution failed). If the server successfully returns a `500 Server Crash` response, the Promise *resolves*. 
You MUST manually check `response.ok`!

```javascript
// THE ONLY CORRECT WAY TO HANDLE FETCH ERRORS
async function getSecureData() {
  try {
    const response = await fetch('/api/secret');

    // 1. Check if the HTTP status is between 200-299
    if (!response.ok) {
      // 2. If it's a 404 or 500, throw an error manually!
      throw new Error(`HTTP Error! Status: ${response.status}`);
    }

    const data = await response.json();
    return data;
  } catch (error) {
    // 3. This block now catches BOTH Network Failures AND HTTP Errors!
    console.error("Fetch failed:", error.message);
  }
}
```

### Level 4 — Aborting Requests (AbortController)
What happens if the user navigates away from the page while a massive 5MB `fetch()` is downloading in the background? You want to cancel the request to save bandwidth!
Promises cannot be cancelled natively. You must use the `AbortController` API to send a kill signal to the `fetch` process.

```javascript
// 1. Create a controller
const controller = new AbortController();
const signal = controller.signal; // Extract the signal wire

async function downloadHugeFile() {
  try {
    // 2. Pass the signal wire into the fetch options
    const response = await fetch('/api/huge-file', { signal });
    const data = await response.json();
    console.log("Download complete!");
  } catch (error) {
    if (error.name === 'AbortError') {
      console.log("Fetch was successfully cancelled by the user!");
    } else {
      console.error("A real error occurred:", error);
    }
  }
}

// Start the download
downloadHugeFile();

// 3. User clicks the "Cancel" button 2 seconds later!
document.getElementById('cancelBtn').addEventListener('click', () => {
  controller.abort(); // Instantly kills the fetch request!
});
```

## REAL_WORLD
**Axios vs. Fetch**
For years, the library `axios` was the industry standard. Why use a third-party library when `fetch` is built into the browser?
Because `axios` handles the annoyances of `fetch` automatically:
1. It automatically throws errors on `404` and `500` status codes.
2. It automatically stringifies JSON payloads.
3. It automatically parses JSON responses.
4. It provides easy "interceptors" for injecting auth tokens into every request.

Today, native `fetch` is preferred in lightweight projects (to reduce bundle size), and in server environments (like Next.js) where `fetch` is globally polyfilled and aggressively patched to support advanced caching strategies. For heavy Client-Side React apps, many still prefer `axios` combined with `React Query`.

## INTERVIEW
**Q: You make a `fetch()` call to a third-party API (`https://api.weather.com`). The browser blocks the request and throws a massive red error in the console mentioning "CORS policy" and "No 'Access-Control-Allow-Origin' header is present". What is CORS and how do you fix this?**
A: **Cross-Origin Resource Sharing (CORS) Security.**
Browsers implement a strict security rule called the *Same-Origin Policy*. By default, a website running on `mywebsite.com` is forbidden from reading data retrieved from `weather.com`. This prevents malicious websites from silently making requests to your bank account API while you are logged in.
When you `fetch` a different domain, the browser requires the *target server* (weather.com) to include a specific HTTP header (`Access-Control-Allow-Origin: *`) explicitly granting permission for your website to read its data.
*The Fix:* If you own the API, you must configure your backend server (e.g., Express or Spring Boot) to emit the correct CORS headers. If you do not own the API, you cannot bypass CORS in the browser. You must set up a "Proxy Server" (using Node.js or Next.js API routes on your own domain) to make the request on behalf of the client, because the CORS restriction only exists in web browsers, not on backend servers!

## FEYNMAN CHECK

### Explain the Fetch API Like I'm 10 Years Old
> fetch() is the modern browser API for making HTTP requests. It returns a PROMISE that resolves to a Response object. The non-obvious trap: fetch() ONLY rejects the promise on NETWORK ERRORS (DNS failure, no internet). A 404 or 500 response is considered a "successful network call" — the Promise resolves, but `response.ok` is false. You MUST check `response.ok` manually. Response bodies are STREAMS — you can only read them ONCE. `response.json()` and `response.text()` return promises that buffer and parse the stream. Timeouts require AbortController — fetch has no built-in timeout option. Credentials (cookies, auth headers) are NOT sent by default to cross-origin requests — set `credentials: 'include'` for cookies.

### 5 Deep Conceptual Questions

**Q1: What replaced XMLHttpRequest and why is fetch better?**
> **A:** XHR was callback-based, verbose, had progress events but terrible composability. fetch uses Promises — chainable, async/await compatible. fetch supports Streams (read huge responses chunk by chunk without buffering), AbortSignal for cancellation, and has a cleaner API. The ONE thing XHR has that fetch originally didn't: upload progress events (now available via `request.body.pipeThrough` for modern fetch streams). For simple HTTP calls, fetch is objectively superior.

**Q2: Mental model for the response lifecycle?**
> **A:** fetch(url) → network happens → a Response is returned (headers available NOW, but body is a STREAM not yet read). Call `response.json()` / `response.text()` / `response.arrayBuffer()` / `response.blob()` to consume the body — each returns a Promise that resolves when buffering/parsing is done. Calling two body-consuming methods throws "body already consumed." Check `response.bodyUsed` or clone the response (`response.clone()`) for multiple reads.

**Q3: Most dangerous misconception?**
> **A:** fetch rejects on HTTP error status:
> ```js
> // ❌ This never catches 404 or 500 — they resolve successfully
> try {
>   const data = await fetch('/api/user/999').then(r => r.json());
> } catch (e) {
>   console.error('error', e);   // only runs on network failure
> }
>
> // ✅ Check response.ok
> const response = await fetch('/api/user/999');
> if (!response.ok) throw new Error(`HTTP ${response.status}: ${response.statusText}`);
> const data = await response.json();
> ```

**Q4: How does AbortController work with fetch?**
> **A:** `const ctrl = new AbortController()` creates a controller. Pass `signal: ctrl.signal` to fetch. Calling `ctrl.abort()` triggers an AbortError on the fetch Promise (and any attached streams). You can reuse one signal across multiple fetches to cancel them all. It's the standard pattern for cancelling stale requests in React useEffect cleanup functions.

**Q5: FAANG-grade definition?**
> **A:** "The Fetch API is a Promise-based HTTP client standardised in the browser environment (and available in Node.js 18+ natively) — returning a Response wrapping a ReadableStream body consumed exactly once via .json()/.text()/.blob()/.arrayBuffer() — resolving on any HTTP response (including 4xx/5xx) and rejecting only on network errors — with timeout, cancellation, and retry implemented externally via AbortController, Promise.race, and wrapper utilities."

#### Step 2 — The Wrapper
```js
// apiFetch.js

export class HttpError extends Error {
  constructor(status, statusText, body) {
    super(`HTTP ${status}: ${statusText}`);
    this.name = 'HttpError';
    this.status = status;
    this.body = body;
  }
}

export async function apiFetch(url, {
  method = 'GET',
  headers = {},
  body,
  timeout = 10_000,
  retries = 2,
  retryDelay = 500,
  signal: externalSignal,
} = {}) {
  const controller = new AbortController();
  const timer = setTimeout(() => controller.abort(new Error('Request timeout')), timeout);

  // Chain external signal with internal timeout signal
  externalSignal?.addEventListener('abort', () => controller.abort(externalSignal.reason));

  const attempt = async (n) => {
    try {
      const response = await fetch(url, {
        method,
        headers: { 'Content-Type': 'application/json', ...headers },
        body: body ? JSON.stringify(body) : undefined,
        signal: controller.signal,
      });

      if (!response.ok) {
        let errorBody;
        try { errorBody = await response.json(); } catch { errorBody = null; }
        throw new HttpError(response.status, response.statusText, errorBody);
      }

      const contentType = response.headers.get('content-type') ?? '';
      return contentType.includes('application/json')
        ? response.json()
        : response.text();

    } catch (err) {
      // Don't retry HTTP errors or aborts — only retry network failures
      if (err instanceof HttpError) throw err;
      if (err.name === 'AbortError') throw err;

      if (n > 0) {
        await new Promise(r => setTimeout(r, retryDelay * (retries - n + 1)));
        return attempt(n - 1);
      }
      throw err;
    }
  };

  try {
    return await attempt(retries);
  } finally {
    clearTimeout(timer);
  }
}
```

#### Step 3 — Use It
```js
// demo.js
import { apiFetch, HttpError } from './apiFetch.js';

// Basic GET
const user = await apiFetch('/api/users/1');

const created = await apiFetch('/api/users', {
  method: 'POST',

// Error handling
try {
  await apiFetch('/api/users/9999');
} catch (e) {
  if (e instanceof HttpError && e.status === 404) {
    console.log('User not found');
  } else {
    throw e;
  }
}

// Timeout + cancel
const ctrl = new AbortController();
setTimeout(() => ctrl.abort(), 500);
try {
  await apiFetch('/api/slow-endpoint', { timeout: 2000, signal: ctrl.signal });
} catch (e) {
  console.log('Cancelled:', e.name);
}
```

#### Step 4 — Error Handling: Streaming Body

```js
// Stream large response
async function streamJSON(url) {
  const res = await fetch(url);
  if (!res.ok) throw new Error(`HTTP ${res.status}`);
  const reader = res.body.getReader();
  const decoder = new TextDecoder();
  let chunks = '';
  while (true) {
    const { done, value } = await reader.read();
    if (done) break;
    chunks += decoder.decode(value, { stream: true });
    process.stdout.write('.');   // progress indicator
  }
  return JSON.parse(chunks);
}
```

```js
function makeResponse(status, body) {
```
