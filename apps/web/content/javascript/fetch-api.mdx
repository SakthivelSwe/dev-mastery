## WHY
For over a decade, JavaScript developers communicated with servers using the `XMLHttpRequest` (XHR) object. It was incredibly verbose, callback-heavy, and notoriously difficult to read.
In 2015, modern browsers introduced the **Fetch API**. It is a modern, clean, Promise-based interface for making HTTP requests. It completely replaced XHR and became the foundational network layer for all modern JavaScript applications.

## THEORY
### Request and Response Streams
The Fetch API does not immediately download the entire payload into memory. It returns a Promise that resolves as soon as the server sends the HTTP **Headers**, before the actual body (data) arrives.
This allows you to check the status code (`404 Not Found`, `200 OK`) or read the headers *before* deciding whether you want to spend the time and memory downloading a massive 50MB JSON payload!

## CODE

### Level 1 — The Basic GET Request
By default, `fetch()` performs an HTTP `GET` request.

```javascript
// fetch() returns a Promise!
fetch('https://jsonplaceholder.typicode.com/users')
  .then((response) => {
    // 1. The Promise resolves when the HEADERS arrive.
    // We must call .json() to read the BODY stream and parse it into an object.
    // Note: .json() ALSO returns a Promise!
    return response.json(); 
  })
  .then((data) => {
    // 2. This runs when the full body is downloaded and parsed.
    console.log("Users:", data);
  });
```

Using `async/await` makes it much cleaner:
```javascript
async function getUsers() {
  const response = await fetch('https://jsonplaceholder.typicode.com/users');
  const data = await response.json();
  console.log("Users:", data);
}
```

### Level 2 — POSTing Data
To send data to the server (e.g., creating a new user), you pass an options object as the second argument to configure the `method`, `headers`, and the `body` (which must be a string!).

```javascript
async function createUser(newUser) {
  const response = await fetch('https://api.example.com/users', {
    method: 'POST', // Use POST to create data
    headers: {
      // You MUST tell the server you are sending JSON!
      'Content-Type': 'application/json',
      // Pass authentication tokens here!
      'Authorization': 'Bearer YOUR_TOKEN_HERE' 
    },
    // You MUST stringify the JavaScript object into a JSON string!
    body: JSON.stringify(newUser) 
  });

  const data = await response.json();
  return data;
}

createUser({ name: "Alice", email: "alice@test.com" });
```

### Level 3 — The Error Handling Trap
The most dangerous thing about `fetch` is how it handles HTTP Errors (like `404 Not Found` or `500 Internal Server Error`).
**`fetch()` DOES NOT REJECT on HTTP Errors!**
The `fetch` Promise only rejects (triggers a `.catch()` block) if there is a literal network failure (e.g., the user's WiFi disconnected, or DNS resolution failed). If the server successfully returns a `500 Server Crash` response, the Promise *resolves*. 
You MUST manually check `response.ok`!

```javascript
// THE ONLY CORRECT WAY TO HANDLE FETCH ERRORS
async function getSecureData() {
  try {
    const response = await fetch('/api/secret');

    // 1. Check if the HTTP status is between 200-299
    if (!response.ok) {
      // 2. If it's a 404 or 500, throw an error manually!
      throw new Error(`HTTP Error! Status: ${response.status}`);
    }

    const data = await response.json();
    return data;
  } catch (error) {
    // 3. This block now catches BOTH Network Failures AND HTTP Errors!
    console.error("Fetch failed:", error.message);
  }
}
```

### Level 4 — Aborting Requests (AbortController)
What happens if the user navigates away from the page while a massive 5MB `fetch()` is downloading in the background? You want to cancel the request to save bandwidth!
Promises cannot be cancelled natively. You must use the `AbortController` API to send a kill signal to the `fetch` process.

```javascript
// 1. Create a controller
const controller = new AbortController();
const signal = controller.signal; // Extract the signal wire

async function downloadHugeFile() {
  try {
    // 2. Pass the signal wire into the fetch options
    const response = await fetch('/api/huge-file', { signal });
    const data = await response.json();
    console.log("Download complete!");
  } catch (error) {
    if (error.name === 'AbortError') {
      console.log("Fetch was successfully cancelled by the user!");
    } else {
      console.error("A real error occurred:", error);
    }
  }
}

// Start the download
downloadHugeFile();

// 3. User clicks the "Cancel" button 2 seconds later!
document.getElementById('cancelBtn').addEventListener('click', () => {
  controller.abort(); // Instantly kills the fetch request!
});
```

## REAL_WORLD
**Axios vs. Fetch**
For years, the library `axios` was the industry standard. Why use a third-party library when `fetch` is built into the browser?
Because `axios` handles the annoyances of `fetch` automatically:
1. It automatically throws errors on `404` and `500` status codes.
2. It automatically stringifies JSON payloads.
3. It automatically parses JSON responses.
4. It provides easy "interceptors" for injecting auth tokens into every request.

Today, native `fetch` is preferred in lightweight projects (to reduce bundle size), and in server environments (like Next.js) where `fetch` is globally polyfilled and aggressively patched to support advanced caching strategies. For heavy Client-Side React apps, many still prefer `axios` combined with `React Query`.

## INTERVIEW
**Q: You make a `fetch()` call to a third-party API (`https://api.weather.com`). The browser blocks the request and throws a massive red error in the console mentioning "CORS policy" and "No 'Access-Control-Allow-Origin' header is present". What is CORS and how do you fix this?**
A: **Cross-Origin Resource Sharing (CORS) Security.**
Browsers implement a strict security rule called the *Same-Origin Policy*. By default, a website running on `mywebsite.com` is forbidden from reading data retrieved from `weather.com`. This prevents malicious websites from silently making requests to your bank account API while you are logged in.
When you `fetch` a different domain, the browser requires the *target server* (weather.com) to include a specific HTTP header (`Access-Control-Allow-Origin: *`) explicitly granting permission for your website to read its data.
*The Fix:* If you own the API, you must configure your backend server (e.g., Express or Spring Boot) to emit the correct CORS headers. If you do not own the API, you cannot bypass CORS in the browser. You must set up a "Proxy Server" (using Node.js or Next.js API routes on your own domain) to make the request on behalf of the client, because the CORS restriction only exists in web browsers, not on backend servers!
