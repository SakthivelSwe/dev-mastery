## WHY
If code simply executed straight down from line 1 to line 100, software would be entirely useless. Every user would see the exact same thing, regardless of their actions.

**Control Flow** allows your code to make intelligent decisions based on runtime data. It allows the JavaScript engine to evaluate a condition (e.g., "Did the user provide a valid password?") and choose to branch off into a completely different block of logic. Mastering `if`, `else`, and `switch` statements allows you to build complex state machines, validation logic, and dynamic user interfaces.

## THEORY
### Branching Logic
There are three primary ways to control execution flow in JavaScript:
1. **The `if / else` statement**: The most common approach. It evaluates a boolean condition and executes a specific block of code based on the result.
2. **The Ternary Operator (`? :`)**: A concise, inline version of an `if/else` statement that is strictly used to *return a value*.
3. **The `switch` statement**: Used when you need to evaluate a single variable against many different possible exact matches. It is cleaner than writing 10 `if/else` statements in a row.

## CODE

### Level 1 — Beginner (If, Else If, Else)
```javascript
const userRole = "ADMIN";

// The (condition) is evaluated. If it is Truthy, the { block } executes.
if (userRole === "ADMIN") {
  console.log("Welcome to the Control Panel.");
} else if (userRole === "EDITOR") {
  console.log("You can edit articles.");
} else {
  // The 'else' block is the final fallback if ALL previous conditions fail.
  console.log("Access Denied. You are a standard user.");
}

// NOTE: JavaScript will only execute EXACTLY ONE of these blocks. 
// Once a condition is met, it skips the rest entirely.
```

### Level 2 — Building Blocks (The Ternary Operator)
```javascript
const age = 20;

// THE OLD WAY: Verbose and requires a 'let' variable.
let status;
if (age >= 18) {
  status = "Adult";
} else {
  status = "Minor";
}

// THE NEW WAY: The Ternary Operator. 
// Syntax: condition ? trueResult : falseResult;
// Because it returns a value directly, we can use the strict 'const'!
const newStatus = age >= 18 ? "Adult" : "Minor";

console.log(newStatus); // "Adult"
```

### Level 3 — Intermediate (The Switch Statement)
```javascript
// When checking the SAME variable against MANY specific values, use Switch.
const httpStatusCode = 404;

switch (httpStatusCode) {
  case 200:
    console.log("Success! Data loaded.");
    break; // CRITICAL: You MUST break, or JS will execute the next case too!
    
  case 201:
    console.log("Created successfully.");
    break;
    
  case 400:
  case 404:
    // Notice we stacked these! This is "Fall-through". 
    // Both 400 and 404 will trigger this exact block of code!
    console.log("Client Error: Invalid request or Not Found.");
    break;
    
  case 500:
    console.log("Server Error.");
    break;
    
  default:
    // The fallback if no cases match (Similar to 'else')
    console.log("Unknown Status Code.");
}
```

### Level 4 — Advanced (Object Maps over Switch)
```javascript
// In modern enterprise JavaScript, large 'switch' statements are considered 
// "code smells" because they are hard to read and easily broken if you forget a 'break'.

// The modern replacement is the "Object Dictionary Map" pattern.

const eventType = "CLICK";

// 1. Define a pure JavaScript Object mapping keys to values (or functions)
const eventHandlers = {
  "CLICK": () => console.log("Handling click event..."),
  "HOVER": () => console.log("Handling hover event..."),
  "SCROLL": () => console.log("Handling scroll event...")
};

// 2. We use Bracket Notation [] to look up the exact key dynamically!
// If the key doesn't exist, it returns undefined, so we use Optional Chaining (?.) 
// to safely execute it!
eventHandlers[eventType]?.(); 

// This executes the CLICK function instantly in O(1) time, without needing 
// 15 lines of switch/case boilerplate!
```

## REAL_WORLD
The Ternary operator (`? :`) is the absolute backbone of **React** development. 
Because React's JSX syntax (writing HTML inside JS) does not legally allow traditional `if/else` statements inside the return block, developers rely entirely on Ternary operators and Logical AND (`&&`) to conditionally render UI.

```jsx
// Real React Component Example
function CheckoutButton({ isCartEmpty }) {
  return (
    <button className={isCartEmpty ? "disabled-btn" : "primary-btn"}>
      {isCartEmpty ? "Your Cart is Empty" : "Proceed to Checkout"}
    </button>
  );
}
```
If you do not master the Ternary operator, you will find it physically impossible to build modern React applications.

## INTERVIEW
**Q: What happens if you forget the `break` keyword inside a `switch` statement?**
A: You will trigger a bug called "Fall-through". Once the JavaScript engine finds a matching `case`, it begins executing code. If it doesn't hit a `break` keyword, it completely ignores all subsequent `case` evaluations and blindly executes the code inside every single following block until it hits the end of the switch statement or finally encounters a `break`.

**Q: Can you nest Ternary operators?**
A: Yes, technically you can write `condition1 ? result1 : condition2 ? result2 : result3`. However, this is universally considered terrible practice. Nested ternaries are incredibly difficult to read and maintain. If your logic requires multiple branches, you should revert to standard `if/else if` statements or extract the logic into a separate helper function.

**Q: In an `if` statement, does the condition have to be a boolean?**
A: No! JavaScript uses Implicit Type Coercion. You can place any variable inside an `if(variable)` block. The JS Engine will automatically coerce the variable into a Boolean using Truthy/Falsy rules. For example, `if("hello")` will execute because strings are Truthy, and `if(0)` will not execute because `0` is Falsy.

## FEYNMAN CHECK

### Explain Control Flow Like I'm 10 Years Old
> Control flow is HOW your program DECIDES what to do next. JS gives you: `if/else` (branch on truthy/falsy), `switch` (multi-way branch on equality), ternary `? :` (compact if/else expression), `try/catch/finally` (handle errors), and `throw` (signal errors). The non-obvious detail: switch uses `===` (strict equality) — `case 1` does NOT match string '1'. Switch cases FALL THROUGH unless you `break` or `return` — a frequent bug. Modern JS prefers OBJECT LOOKUPS and EARLY RETURNS over deep switch/if chains. `try/catch` only catches SYNCHRONOUS throws and PROMISES THAT YOU AWAIT — fire-and-forget async errors slip past.

### 5 Deep Conceptual Questions

**Q1: When should you use switch vs if-else vs object lookup?**
> **A:** Use OBJECT LOOKUP `actions[type]?.()` for "map type to handler" — cleanest, fastest (O(1) hash), tree-shakable, no fall-through bugs. Use SWITCH when you need fall-through intentionally or pattern-matching that an object can't express. Use IF-ELSE for actual ranges/conditions (`if (n > 100)`). Modern codebases use ternaries for expressions, early returns for guard clauses, switch only when truly needed.

**Q2: Mental model for try/catch and async?**
> **A:** A try/catch catches anything that THROWS SYNCHRONOUSLY in its body, OR is awaited and rejects. `try { setTimeout(() => { throw new Error() }) }` does NOT catch — the throw happens in a separate event-loop tick. `try { await fetch('/bad') }` catches the rejection. Modern advice: don't pepper try/catch everywhere; let unhandled errors bubble to global error handlers (window.onerror, process.on('unhandledRejection')), catch only when you can RECOVER.

**Q3: Most dangerous misconception?**
> **A:** Switch fall-through:
> ```js
> // ❌ Forgot break — fall-through silently runs next case
> switch (status) {
>   case 'pending':
>     showSpinner();
>     // forgot break!
>   case 'error':
>     showError();        // runs even when status is 'pending'!
>     break;
>   case 'success':
>     hideSpinner();
>     break;
> }
>
> // ✅ Always use return or break, OR refactor to object lookup
> const handlers = { pending: showSpinner, error: showError, success: hideSpinner };
> handlers[status]?.();
> ```

**Q4: How does try/catch interact with the JS engine's optimisation?**
> **A:** Historically (pre-2018), code inside try blocks was DE-OPTIMISED in V8 — try blocks couldn't be optimised by Crankshaft. Modern V8 (Turbofan, 2018+) handles try/catch fully. So "avoid try/catch in hot loops" is OUTDATED advice. However, throwing is still expensive — capturing a stack trace costs microseconds, much more than returning a Result type. For control flow, use return values; for exceptional errors, throw.

**Q5: FAANG-grade definition?**
> **A:** "JavaScript control flow comprises branching constructs (if/else, switch with strict-equality dispatch and explicit fall-through, ternary expressions), iteration (for, while, do-while, for-of, for-in), exception handling (try/catch/finally with throw, scoped to synchronous + awaited async), and labelled break/continue for nested loop control — collectively forming the imperative skeleton onto which modern JS layers functional patterns like Array.map/filter/reduce."

#### Step 2 — State Machine With Switch
```js
// order.js
export function nextState(currentState) {
  switch (currentState) {
    case 'pending':   return 'paid';
    case 'paid':      return 'shipped';
    case 'shipped':   return 'delivered';
    case 'delivered': return 'delivered';  // terminal
    case 'cancelled': return 'cancelled';  // terminal
    default:
      // Exhaustiveness check — runtime error if a new state was added
      throw new Error(`Unknown order state: ${currentState}`);
  }
}

// Equivalent object-lookup version — cleaner, tree-shakable
const TRANSITIONS = {
  pending:   'paid',
  paid:      'shipped',
  shipped:   'delivered',
  delivered: 'delivered',
  cancelled: 'cancelled',
};

export function nextStateLookup(s) {
  const next = TRANSITIONS[s];
  if (next === undefined) throw new Error(`Unknown state: ${s}`);
  return next;
}
```

#### Step 3 — Process Order With Retry
```js
// processor.js
export async function processOrder(orderId, attempt = 1) {
  const MAX_ATTEMPTS = 3;
  try {
    const order = await fetchOrder(orderId);
    if (!order)            return { status: 'error', reason: 'not_found' };
    if (order.cancelled)   return { status: 'noop',  reason: 'already_cancelled' };

    // Guard clauses — early return, no nested ifs
    const newState = nextState(order.state);
    await persistState(orderId, newState);
    return { status: 'ok', newState };
  } catch (err) {
    // Recoverable: retry up to MAX_ATTEMPTS
    if (err.name === 'NetworkError' && attempt < MAX_ATTEMPTS) {
      await new Promise(r => setTimeout(r, 2 ** attempt * 100));
      return processOrder(orderId, attempt + 1);
    }
    return { status: 'error', reason: err.message };
  } finally {
    console.log(`Processed order ${orderId} (attempt ${attempt})`);
  }
}

async function fetchOrder(id) { /* stub */ return { id, state: 'pending', cancelled: false }; }
async function persistState(id, state) { /* stub */ }
```

#### Step 4 — Error Handling: Async Try/Catch Gotcha
```js
// ❌ This does NOT catch the timer's throw
async function badAsync() {
  try {
    setTimeout(() => { throw new Error('boom'); }, 0);  // uncaught — escapes try
  } catch (e) {
    console.log('caught:', e.message);   // never runs
  }
}

// ✅ Wrap in a Promise so you can await it
function laterError() {
  return new Promise((_, reject) => setTimeout(() => reject(new Error('boom')), 0));
}
async function goodAsync() {
  try { await laterError(); }
  catch (e) { console.log('caught:', e.message); }   // ✅ runs
}
```
