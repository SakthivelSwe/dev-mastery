## WHY
In JavaScript, object keys are normally Strings. If you write `obj['id'] = 1`, and a third-party library also writes `obj['id'] = 2`, the library silently overwrites your data. This naming collision is a nightmare when extending objects you don't own.
Furthermore, how does a `for...of` loop actually know how to loop over an Array or a Set, but throws an error if you try to loop over a standard Object?
ES6 introduced **Symbols** to solve the naming collision problem, and **Iterables/Iterators** (which are powered by Symbols) to define custom looping behavior for your own data structures!

## THEORY
### The Unique Symbol
A `Symbol` is a primitive data type (just like Number, String, and Boolean). 
Its defining characteristic is that **every Symbol is mathematically unique**. Even if you create two symbols with the exact same description, they will never equal each other.

```javascript
const sym1 = Symbol("myId");
const sym2 = Symbol("myId");

console.log(sym1 === sym2); // FALSE! They are completely unique.
```

### The Iterator Protocol
An object is "Iterable" (meaning you can use `for...of` or the `...` spread operator on it) IF AND ONLY IF it has a method attached to the special built-in Symbol: `Symbol.iterator`.
This method must return an "Iterator" object, which is an object that has a `.next()` method that returns `{ value: any, done: boolean }`.

## CODE

### Level 1 — Symbols as Hidden Object Keys
You can use a Symbol as a key in an object. This acts as a "hidden" property that cannot accidentally be overwritten by string keys, and is completely ignored by standard `for...in` loops and `JSON.stringify`!

```javascript
// We create a unique Symbol
const SECRET_KEY = Symbol('auth_token');

const user = {
  name: "Alice",
  age: 25,
  // We use bracket notation to set the Symbol as a key!
  [SECRET_KEY]: "xyz123-super-secret"
};

// 1. You can access it if you have the Symbol reference!
console.log(user[SECRET_KEY]); // "xyz123-super-secret"

// 2. It is completely hidden from standard loops!
for (let key in user) {
  console.log(key); // Prints: "name", "age". It IGNORES the Symbol!
}

// 3. It is ignored by JSON!
console.log(JSON.stringify(user)); // '{"name":"Alice","age":25}'
```

### Level 2 — Global Symbols (Symbol.for)
If Symbols are always unique, how can File A and File B access the exact same Symbol?
You use the Global Symbol Registry! `Symbol.for("key")` checks the registry. If it exists, it returns it. If it doesn't, it creates it and registers it globally.

```javascript
// --- FileA.js ---
const mySym = Symbol.for("app.database");
globalState[mySym] = "Connected";

// --- FileB.js ---
// This grabs the EXACT SAME symbol created in FileA!
const dbSym = Symbol.for("app.database");
console.log(globalState[dbSym]); // "Connected"
```

### Level 3 — Creating Custom Iterables
Standard JavaScript Objects are NOT iterable. If you try `for (const x of {})`, it crashes.
Let's build an object that holds a "start" and "end" number, and manually teach JavaScript how to loop over it using `Symbol.iterator`!

```javascript
const range = {
  start: 1,
  end: 3,
  
  // To make it iterable, we attach a method to the built-in Symbol.iterator!
  [Symbol.iterator]: function() {
    let current = this.start;
    const last = this.end;

    // We must return an Iterator object (an object with a next() method)
    return {
      next() {
        if (current <= last) {
          // Return the current value, and increment it for next time!
          return { value: current++, done: false };
        } else {
          // We reached the end!
          return { value: undefined, done: true };
        }
      }
    };
  }
};

// MAGIC! We can now use for...of on our custom object!
for (const num of range) {
  console.log(num); // Prints 1, then 2, then 3!
}

// We can also use the Spread Operator on it!
const rangeArray = [...range]; 
console.log(rangeArray); // [1, 2, 3]
```

### Level 4 — Generators as Iterators (The Cheat Code)
Writing the `{ next() { return { value, done } } }` object manually is incredibly tedious.
Because **Generators** (`function*`) automatically return Iterator objects, you can use a Generator directly inside `Symbol.iterator` to write infinitely cleaner code!

```javascript
const elegantRange = {
  start: 1,
  end: 3,
  
  // We make the Symbol.iterator a Generator Function!
  *[Symbol.iterator]() {
    for (let i = this.start; i <= this.end; i++) {
      // We just 'yield' the values! The generator handles the 'next/done' boilerplate!
      yield i;
    }
  }
};

for (const num of elegantRange) {
  console.log(num); // Prints 1, 2, 3!
}
```

## REAL_WORLD
**Well-Known Symbols**
JavaScript provides several built-in Symbols (called Well-Known Symbols) that allow you to hook into the core engine operations.
- `Symbol.iterator`: Defines how an object is looped over.
- `Symbol.toPrimitive`: Defines how an object is converted into a primitive. For example, if you write `console.log(userObject + 5)`, JS normally prints `[object Object]5`. If you implement `[Symbol.toPrimitive]`, you can tell JS: *"When they try to do math with me, return my ID number instead!"*
- `Symbol.toStringTag`: Changes what `Object.prototype.toString.call(obj)` returns (instead of `[object Object]`, it could return `[object MyCustomClass]`).

## INTERVIEW
**Q: You created a highly secure banking object. You used a `Symbol` to store the user's secret password, knowing that `for...in` loops and `Object.keys()` cannot see it. You tell your manager that the password is now perfectly 100% private and inaccessible to other developers. Are you correct?**
A: **No! Symbols do NOT provide true privacy.**
While it is true that Symbols are hidden from standard loops and `JSON.stringify()`, JavaScript provides a backdoor specifically designed to find them.
Any developer can simply run `Object.getOwnPropertySymbols(bankObject)`. This returns an array of every single Symbol attached to the object. They can then take that Symbol, do `bankObject[foundSymbol]`, and extract the secret password immediately!
If you need true, cryptographic privacy in JavaScript, you must use **Closures** (variables trapped inside a function scope) or the modern **Private Class Fields** syntax (`#password = "secret"`).
