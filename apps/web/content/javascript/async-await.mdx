## WHY
Before `async/await`, asynchronous JavaScript was handled using Raw Promises (`.then()` and `.catch()`). 
While Promises fixed "Callback Hell", they introduced their own issues: code was still nested, error handling was tricky because `try/catch` didn't work inside `.then()`, and reading a sequence of asynchronous steps felt unnatural.
Introduced in ES2017, **Async/Await** is syntactic sugar over Promises. It allows you to write asynchronous code that *looks* and *reads* like synchronous code, making complex API calls and database queries infinitely easier to understand.

## THEORY
### The Sugar
1. **`async`:** Placing this keyword before a function guarantees that the function will always return a Promise. If you return a primitive (like `return 5;`), JavaScript automatically wraps it in a resolved Promise (`Promise.resolve(5)`).
2. **`await`:** This keyword can *only* be used inside an `async` function. It tells the JavaScript engine: *"Pause the execution of this specific function right here. Wait for this Promise to resolve. Once it resolves, unpack the value and continue to the next line."*

*Crucially, `await` does not block the entire main thread!* It only pauses the execution of the `async` function itself. The rest of your application (like UI rendering and event listeners) continues running perfectly.

## CODE

### Level 1 — The Transformation
Look at how much cleaner the code becomes when moving from `.then()` to `await`.

```javascript
// BAD: The old Promise way
function fetchUserData(userId) {
  return fetch(`/api/users/${userId}`)
    .then(response => response.json())
    .then(data => {
      console.log(data);
      return data;
    });
}

// GOOD: The modern Async/Await way
async function fetchUserData(userId) {
  // Execution pauses here until 'fetch' completes!
  const response = await fetch(`/api/users/${userId}`);
  
  // Execution pauses here until 'json()' completes!
  const data = await response.json();
  
  console.log(data);
  return data;
}
```

### Level 2 — Error Handling (`try/catch`)
With raw Promises, you have to use `.catch()`. With `async/await`, you can use the native, standard `try/catch` block, allowing you to handle both synchronous and asynchronous errors in the exact same place!

```javascript
async function processPayment(cartData) {
  try {
    // 1. Synchronous error possibility (JSON parsing)
    if (!cartData) throw new Error("Cart is empty!");

    // 2. Asynchronous error possibility (Network failure)
    const response = await fetch('/api/charge', {
      method: 'POST',
      body: JSON.stringify(cartData)
    });

    if (!response.ok) throw new Error("Payment declined!");

    return await response.json();
  } catch (error) {
    // This single catch block handles ALL errors from above!
    console.error("Transaction Failed:", error.message);
    // You can also re-throw the error to the caller!
    throw error; 
  }
}
```

### Level 3 — The Await Waterfall (Anti-Pattern)
The most common mistake beginners make is overusing `await`, creating a "Waterfall". 
If you have three API calls that do not depend on each other, awaiting them sequentially is a massive performance bottleneck.

```javascript
// BAD: The Waterfall (Takes 3 seconds total)
async function getDashboardData() {
  const users = await fetchUsers(); // Takes 1s
  const posts = await fetchPosts(); // Waits 1s, then takes 1s
  const stats = await fetchStats(); // Waits 2s, then takes 1s
}
```

To fix this, you initiate all the Promises simultaneously, and then `await` them all at once using `Promise.all()`!

```javascript
// GOOD: Parallel Execution (Takes 1 second total)
async function getDashboardData() {
  // 1. Fire off all requests instantly (No 'await' here!)
  const usersPromise = fetchUsers();
  const postsPromise = fetchPosts();
  const statsPromise = fetchStats();

  // 2. Wait for all of them to finish simultaneously!
  const [users, posts, stats] = await Promise.all([
    usersPromise,
    postsPromise,
    statsPromise
  ]);
}
```

### Level 4 — Top-Level Await
Historically, `await` could only be used inside a function explicitly labeled `async`. You couldn't use it at the root level of a file.
In modern JavaScript (ES2022+), **Top-Level Await** is supported inside ES Modules.

```javascript
// database.js
import { connectToDatabase } from 'my-db-driver';

// This is the root of the file! No async function wrapper needed!
// The module system will halt and wait for the database to connect 
// before allowing any other files to import this one!
export const db = await connectToDatabase('mongodb://localhost:27017');
```

## REAL_WORLD
**Loops and Await**
How do you run an asynchronous task for every item in an array?
If you use `.forEach()`, it will **fail completely**. `forEach` is not promise-aware; it will instantly fire off all the tasks and finish immediately, without waiting for the `awaits` inside it!

```javascript
// BAD: forEach does NOT wait!
async function processFiles(files) {
  files.forEach(async (file) => {
    await uploadFile(file); // forEach ignores this await!
  });
  console.log("Done!"); // This prints BEFORE the uploads finish!
}

// GOOD: Use a standard 'for...of' loop for SEQUENTIAL execution
async function processFilesSequentially(files) {
  for (const file of files) {
    await uploadFile(file); // Waits for file 1 to finish before starting file 2
  }
  console.log("Done!");
}

// GOOD: Use Promise.all() + map() for PARALLEL execution
async function processFilesParallel(files) {
  const uploadPromises = files.map(file => uploadFile(file));
  await Promise.all(uploadPromises);
  console.log("Done!");
}
```

## INTERVIEW
**Q: You write `const result = await fetch('/data');` inside a standard React `useEffect` hook. React throws a warning: "Effect callbacks are synchronous to prevent race conditions". Why does this happen and how do you fix it?**
A: **The `useEffect` signature.**
React's `useEffect` hook is designed to optionally return a cleanup function (e.g., `return () => clearTimeout(timer);`).
If you make the effect itself `async` (`useEffect(async () => {...})`), the function automatically returns a Promise instead of a cleanup function. React has no idea what to do with a Promise in that position, so it crashes/warns you.
**The Fix:** You must declare the `async` function *inside* the effect, and then immediately call it.
```javascript
useEffect(() => {
  // 1. Declare the async function inside the effect
  const fetchData = async () => {
    const res = await fetch('/data');
    // ...
  };
  
  // 2. Call it!
  fetchData();
}, []);
```

## FEYNMAN CHECK

### Explain async/await Like I'm 10 Years Old
> async/await is SYNTACTIC SUGAR for Promises. `async function f()` always returns a Promise. `await p` PAUSES the function until p settles, then resumes with p's value (or throws if rejected). Inside an async function, you write code that LOOKS synchronous but ISN'T — it can use try/catch for errors and for-loops for sequencing. The non-obvious detail: `await` ONLY pauses the CURRENT async function — the caller keeps running. `fn()` where fn is async returns the promise IMMEDIATELY; you have to await fn() or chain .then. A common bug: forgetting to await — `async function() { saveToDb(user); }` is fire-and-forget; errors vanish silently. ESLint's no-floating-promises catches this.

---

### 5 Deep Conceptual Questions

**Q1: What problem does async/await fundamentally solve over raw Promises?**
> **A:** Readability and control flow. With raw Promises, branching/looping/error-handling around async ops required nested .then chains: `p.then(a => { if (a) return q.then(b => ...) else return ...; }).catch(...)`. With async/await, the same code is: `try { const a = await p; if (a) { const b = await q; ... } } catch (e) { ... }` — flat, debuggable, supports loops naturally.

**Q2: Mental model for async/await?**
> **A:** "Every await is a SUSPEND POINT — the engine puts your function aside, runs other things, and resumes when the awaited value is ready." The function's local variables are preserved on the heap (a Continuation). The caller never blocks. await DOES NOT make the program single-threaded blocking — the event loop runs other tasks while you're paused.

**Q3: Most dangerous misconception?**
> **A:** Sequential awaits in a loop when parallel would do:
> ```js
> // ❌ Slow — each fetch awaits the previous to complete
> for (const url of urls) {
>   const data = await fetch(url).then(r => r.json());
>   results.push(data);
> }
> // 10 URLs × 200ms each = 2000ms total
>
> // ✅ Parallel with Promise.all
> const results = await Promise.all(urls.map(url => fetch(url).then(r => r.json())));
> // All 10 in flight at once = ~200ms total
> ```

**Q4: How does async/await compile under the hood?**
> **A:** The compiler converts each async function into a STATE MACHINE. Every `await` becomes a state transition: save current state, return the awaited promise's continuation, schedule resume callback. When the promise settles, the runtime jumps back into the function at the saved state. Modern V8 has dedicated bytecode for async — making it nearly as fast as raw .then chains, with extra overhead only for try/catch around await.

**Q5: FAANG-grade definition?**
> **A:** "async/await is JavaScript syntactic sugar over Promises — async functions always return a Promise, await expressions suspend the function at compile-time-generated state-machine resume points, throwing rejected-promise reasons as synchronous errors catchable by try/catch — enabling synchronous-looking code over asynchronous operations with full support for loops, branches, try/finally, and structured error handling."

#### Step 2 — Sequential vs Parallel
```js
// loader.js

// Simulate image load with variable delay
function loadImage(url) {
  return new Promise((resolve) => {
    setTimeout(() => resolve({ url, bytes: Math.floor(Math.random() * 50000) }), 50 + Math.random() * 150);
  });
}

// SEQUENTIAL — slow, simple
export async function loadAllSeq(urls, onProgress) {
  const results = [];
  for (const url of urls) {
    const img = await loadImage(url);     // ❌ each waits for previous
    results.push(img);
    onProgress?.(results.length, urls.length);
  }
  return results;
}

// PARALLEL — fast, but progress callback is trickier
export async function loadAllPar(urls, onProgress) {
  let done = 0;
  return Promise.all(urls.map(async (url) => {
    const img = await loadImage(url);
    done++;
    onProgress?.(done, urls.length);
    return img;
  }));
}

// CONCURRENT WITH LIMIT — best of both worlds
export async function loadAllLimit(urls, { concurrency = 4, onProgress } = {}) {
  let done = 0;
  const results = new Array(urls.length);
  let idx = 0;
  async function worker() {
    while (idx < urls.length) {
      const i = idx++;
      results[i] = await loadImage(urls[i]);
      done++;
      onProgress?.(done, urls.length);
    }
  }
  await Promise.all(Array.from({ length: concurrency }, worker));
  return results;
}
```

#### Step 4 — Error Handling: try/catch With await
```js
// Robust version with error per task
export async function loadAllSafe(urls) {
  const settled = await Promise.allSettled(urls.map(loadImage));
  const ok    = settled.filter(s => s.status === 'fulfilled').map(s => s.value);
  const errors = settled.filter(s => s.status === 'rejected').map(s => s.reason);
  return { ok, errors };
}

// Top-level await (ES2022, modules only)
const { ok, errors } = await loadAllSafe(['/good', '/bad', '/ugly']);
console.log(`Loaded: ${ok.length}, Failed: ${errors.length}`);
```

import { describe, it, expect, vi } from 'vitest';
import { loadAllSeq, loadAllPar, loadAllLimit } from './loader.js';

describe('async/await loaders', () => {
  it('sequential is slower than parallel for same workload', async () => {
    const urls = ['a', 'b', 'c', 'd', 'e'];
    const startSeq = Date.now();
    await loadAllSeq(urls);
    const seqTime = Date.now() - startSeq;

    expect(onProgress).toHaveBeenCalledTimes(3);
    expect(onProgress).toHaveBeenLastCalledWith(3, 3);

    let active = 0, peak = 0;
