## WHY
Before 2015, JavaScript was considered a chaotic scripting language by Enterprise developers who were used to the strict Object-Oriented Programming (OOP) paradigms of Java and C#. To create "Classes" in JavaScript, you had to use weird Constructor Functions and manually attach methods to the `prototype` property.

To modernize the language and make it easier for millions of developers to adopt, ES6 introduced the `class` keyword. This brought familiar concepts like constructors, inheritance (`extends`), and static methods directly into JavaScript. While modern Frontend developers (React) have largely abandoned Classes in favor of Functional Programming, Backend developers (Node.js/NestJS) rely on Classes heavily to structure massive server architectures.

## THEORY
### Syntactic Sugar
It is critical to understand that the `class` keyword in JavaScript is purely **"Syntactic Sugar"**. 
It is a fake facade. JavaScript did not change its core architectural engine. There are no rigid blueprints or compilation-time memory boundaries. The JS Engine literally reads your `class` keyword and instantly translates it back into Constructor Functions and Prototypal Inheritance links (`__proto__`) behind the scenes!

### The Four Pillars of OOP
1. **Encapsulation**: Grouping related data (properties) and behavior (methods) together inside the class.
2. **Inheritance**: Creating child classes that inherit behaviors from parent classes (using `extends`).
3. **Polymorphism**: A child class overriding a parent's method to provide its own specific behavior.
4. **Abstraction**: Hiding complex internal logic and exposing only simple public methods.

## CODE

### Level 1 — Beginner (Creating a Class)
```javascript
// 1. Define the Class Blueprint
class User {
  // The constructor runs exactly ONCE when the object is instantiated!
  constructor(name, email) {
    this.name = name;
    this.email = email;
    this.isOnline = false; // Default value
  }

  // 2. Define Methods
  // You do NOT use the 'function' keyword inside a class!
  login() {
    this.isOnline = true;
    console.log(`${this.name} has logged in.`);
  }

  logout() {
    this.isOnline = false;
    console.log(`${this.name} has logged out.`);
  }
}

// 3. Instantiate the Object
// The 'new' keyword tells JS to build the object and run the constructor!
const alice = new User("Alice", "alice@test.com");

alice.login(); // "Alice has logged in."
console.log(alice.isOnline); // true
```

### Level 2 — Building Blocks (Inheritance and Polymorphism)
```javascript
// We want an Admin user. Admins are Users, but they can do more things.
// The 'extends' keyword links the Admin prototype to the User prototype!

class Admin extends User {
  constructor(name, email, permissionsLevel) {
    // CRITICAL: You MUST call super() before using 'this'!
    // super() executes the Parent's (User's) constructor!
    super(name, email); 
    
    // Now we can set Admin-specific properties
    this.permissionsLevel = permissionsLevel;
  }

  // A brand new method only Admins have
  deleteUser(user) {
    console.log(`Admin ${this.name} deleted user ${user.name}`);
  }

  // POLYMORPHISM: We overwrite the parent's login method!
  login() {
    console.log(`[SECURE SYSTEM] ADMIN ${this.name.toUpperCase()} LOGGED IN.`);
    this.isOnline = true;
  }
}

const adminBob = new Admin("Bob", "bob@admin.com", 5);
adminBob.login(); // Triggers the polymorphic overridden method!
adminBob.deleteUser({ name: "Hacker" }); 
```

### Level 3 — Intermediate (Private Fields & Static Methods)
```javascript
class BankAccount {
  // 1. PRIVATE FIELDS (Introduced in ES2022)
  // The # symbol marks the property as mathematically private.
  // It cannot be accessed or modified from outside the class!
  #balance = 0;

  constructor(owner) {
    this.owner = owner;
  }

  deposit(amount) {
    this.#balance += amount;
  }

  getBalance() {
    return this.#balance;
  }

  // 2. STATIC METHODS
  // A static method is attached to the Class itself, NOT the individual objects!
  // It is used for utility functions.
  static calculateInterestRate() {
    return 0.05;
  }
}

const myAccount = new BankAccount("Alice");
myAccount.deposit(100);

// console.log(myAccount.#balance); // ERROR! Private field '#balance' must be declared in an enclosing class
console.log(myAccount.getBalance()); // 100

// You call Static methods directly on the Class name!
console.log(BankAccount.calculateInterestRate()); // 0.05
```

### Level 4 — Advanced (Getters and Setters)
```javascript
// Getters and Setters allow you to intercept someone trying to read or write a property!

class Temperature {
  constructor(celsius) {
    this._celsius = celsius; // The underscore is a developer convention meaning "Please treat as private"
  }

  // A Getter acts like a property, but runs as a function!
  get fahrenheit() {
    return (this._celsius * 1.8) + 32;
  }

  // A Setter intercepts the assignment and validates it!
  set fahrenheit(value) {
    if (value < -459.67) {
      console.error("Error: Temperature below Absolute Zero is impossible.");
      return;
    }
    this._celsius = (value - 32) / 1.8;
  }
}

const temp = new Temperature(25);

// We access 'fahrenheit' exactly like a property, but it runs the Getter math!
console.log(temp.fahrenheit); // 77

// We assign it exactly like a property, but it runs the Setter validation!
temp.fahrenheit = 100;
console.log(temp._celsius); // 37.777...

// It intercepts the bad data!
temp.fahrenheit = -500; // "Error: Temperature below Absolute Zero is impossible."
```

## REAL_WORLD
If you are building an enterprise backend using **NestJS** or an ORM like **TypeORM**, Classes are the absolute core of your architecture. You will create `class UserController`, `class UserService`, and `class UserEntity`. 

However, if you are building a frontend using **React**, Classes are essentially dead. In 2018, React introduced "Hooks", which allowed developers to write everything using plain, functional JavaScript. Writing `class MyComponent extends React.Component` is now considered legacy code, and functional components `const MyComponent = () => {}` are the industry standard.

## INTERVIEW
**Q: Why must you call `super()` inside a child class constructor?**
A: In JavaScript, when a child class `extends` a parent, the child does not automatically get a `this` context. The parent class constructor is responsible for allocating the memory and building the core object structure. If you try to execute `this.name = "Bob"` before calling `super()`, the JavaScript engine will crash with a `ReferenceError` because the `this` object physically does not exist yet.

**Q: Can you achieve multiple inheritance in JavaScript (e.g., `class Dog extends Animal, Pet`)?**
A: No. JavaScript, like Java, strictly forbids multiple inheritance. An object can only have one direct prototype link (`__proto__`). If you need an object to share behaviors from multiple different sources, you must use "Composition" (combining multiple independent objects together) rather than Inheritance.

**Q: Are JavaScript Private Fields (`#`) truly secure?**
A: Yes. Historically, JS developers used the underscore convention (`this._balance = 0`) to signal to other developers that a property was private, but it was just a gentleman's agreement; you could still hack into it. The modern `#balance` syntax is enforced at the engine level. It is a true, hard-private field. Even if you loop over the object using `Object.keys()`, the private field is completely invisible.

## FEYNMAN CHECK

### Explain JS Classes Like I'm 10 Years Old
> JS classes (ES2015) are SYNTACTIC SUGAR over the prototype system. `class Dog extends Animal {}` compiles to a constructor function + prototype linkage. Modern classes have: constructor, instance fields, static fields/methods, getters/setters, true private fields (`#name`, ES2022 — actually private, not just convention), and inheritance via `extends` + `super`. The non-obvious detail: classes are NOT hoisted (unlike function declarations) — accessing before the class statement throws ReferenceError (TDZ). `class X {}` always runs in strict mode regardless of file-level setting. And methods defined in the class body are NON-ENUMERABLE — they don't show up in `for...in` loops or `Object.keys`.

### 5 Deep Conceptual Questions

**Q1: When do you choose class vs factory function vs plain object?**
> **A:** Use CLASS when you have STATEFUL OBJECTS with shared methods, hierarchy via extends, or framework requirements (Web Components, Spring). Use FACTORY function when you want true privacy via closures, easier composition over inheritance, no `this` confusion. Use PLAIN OBJECTS when no methods needed. Modern React/Vue compose state via hooks/refs instead of classes; backend Node.js still uses classes heavily (ORMs, services).

**Q2: Mental model for super and extends?**
> **A:** `class Cat extends Animal` makes `Cat.prototype` inherit from `Animal.prototype` (instance methods), and `Cat` itself inherit from `Animal` (static methods). `super()` in the constructor calls Animal's constructor and BINDS `this`. You cannot use `this` before `super()`. `super.method()` in an instance method calls the parent's version. `super.staticMethod()` in a static method calls the parent's static.

**Q3: Most dangerous misconception?**
> **A:** Class methods are bound to instances:
> ```js
> // ❌ Method loses this when detached
> class Counter {
>   count = 0;
>   inc() { this.count++; }    // prototype method — this NOT bound
> }
> const c = new Counter();
> const ref = c.inc;
> ref();   // TypeError: Cannot read 'count' of undefined
>
> // ✅ Arrow class field is bound at construction
> class CounterBound {
>   count = 0;
>   inc = () => { this.count++; }   // arrow field — per-instance, bound
> }
> // Trade-off: arrow fields use more memory (one function per instance vs shared prototype)
> ```

**Q4: How do true private fields differ from underscore convention?**
> **A:** `_name` is just a CONVENTION — anyone can read `obj._name`. `#name` (ES2022) is a HARD private field — accessing `obj.#name` from outside the class throws SyntaxError at parse time. Private fields exist in a separate WeakMap-like slot per instance, invisible to JSON, reflection, console.log, Proxy traps. Subclasses cannot access parent's privates. Use # for security/encapsulation that MATTERS; _ remains acceptable for "internal but introspectable".

**Q5: FAANG-grade definition?**
> **A:** "JavaScript classes are syntactic sugar over prototype-based inheritance — desugaring into a constructor function plus prototype methods — featuring instance fields, static fields/methods, getters/setters, hard private fields (#) with per-instance hidden slots, extends-based prototype chaining with super for parent invocation, and strict-mode-by-default semantics — distinct from class-based languages by lacking interfaces, traits, abstract enforcement, or method overloading at the language level."

#### Step 2 — Base User Class
```js
// user.js
export class User {
  // True private fields (#)
  #passwordHash;
  #createdAt;

  // Public instance fields
  id;
  email;
  name;

  constructor({ id, email, name, passwordHash }) {
    if (!email.includes('@')) throw new TypeError('Invalid email');
    this.id    = id;
    this.email = email;
    this.name  = name;
    this.#passwordHash = passwordHash;
    this.#createdAt    = new Date();
  }

  // Computed accessor
  get displayName() { return this.name || this.email.split('@')[0]; }

  get createdAt() { return this.#createdAt; }   // controlled exposure

  verifyPassword(plain) {
    return hash(plain) === this.#passwordHash;
  }

  toJSON() {
    // Notice: #passwordHash and #createdAt are NOT serialised — security
    return { id: this.id, email: this.email, name: this.name };
  }

  static fromJSON(json) {
    return new User({ ...JSON.parse(json), passwordHash: hash('default') });
  }
}

function hash(s) { return 'sha256$' + s.split('').reverse().join(''); }
```

#### Step 3 — Subclass With extends + super
```js
// admin.js
import { User } from './user.js';

export class AdminUser extends User {
  #permissions;

  constructor(opts) {
    super(opts);   // must come before any 'this' access
    this.#permissions = new Set(opts.permissions ?? ['read']);
  }

  hasPermission(perm) { return this.#permissions.has(perm); }
  grant(perm)         { this.#permissions.add(perm); }
  revoke(perm)        { this.#permissions.delete(perm); }

  // Override + extend parent
  toJSON() {
    return { ...super.toJSON(), permissions: [...this.#permissions] };
  }

  // Static factory using parent's parsing
  static promote(user, perms = ['read', 'write']) {
    return new AdminUser({
      id: user.id, email: user.email, name: user.name,
      passwordHash: 'preserved',
      permissions: perms,
    });
  }
}
```

import { describe, it, expect } from 'vitest';
