## WHY
In heavily typed languages, attempting to add a String and a Number (`"5" + 2`) results in a fatal compiler crash. The language forces you to be explicit about what you want to do.

JavaScript, designed in 10 days for novice web masters, took a different approach: it assumes you made a mistake and tries to "fix" it for you. This invisible conversion of data from one type to another is called **Type Coercion**. While the intention was to prevent websites from crashing, the result is a system of confusing, illogical rules that generate catastrophic runtime bugs. Understanding exactly how V8 coerces data is essential to debugging strange logical errors in JavaScript.

## THEORY
### Implicit vs Explicit Coercion
- **Explicit Coercion**: When you, the programmer, manually convert a type (e.g., `Number("5")`). This is safe and predictable.
- **Implicit Coercion**: When the JavaScript engine secretly converts a type in the background to force an operation to succeed (e.g., `"5" - 2`). This is highly dangerous.

### The Double vs Triple Equals
Because of Implicit Coercion, JavaScript has two different Equality operators:
- `==` (Loose Equality): Checks if the values are equal *after* attempting to silently coerce them to the same type.
- `===` (Strict Equality): Checks if both the value AND the data type are mathematically identical. **(Industry Standard)**

## CODE

### Level 1 — Beginner (The Plus Operator Chaos)
```javascript
// The + operator in JS does two different things:
// 1. Mathematical Addition
// 2. String Concatenation

// If JS sees a String anywhere near a + sign, it prioritizes Concatenation!
console.log(5 + 2);       // 7 (Math)
console.log("5" + 2);     // "52" (JS coerced the 2 into a string: "5" + "2")
console.log("Hello" + 5); // "Hello5"

// Watch the order of operations!
console.log(1 + 2 + "3"); // "33" (1+2 is 3, then "3" + "3" is "33")
console.log("1" + 2 + 3); // "123" ("1"+"2" is "12", then "12"+"3" is "123")
```

### Level 2 — Building Blocks (The Subtraction Trick)
```javascript
// The - , * , and / operators ONLY perform Mathematics.
// Therefore, JS assumes you want to do Math, and coerces Strings into Numbers!

console.log("5" - 2);   // 3 (JS coerced "5" into the number 5)
console.log("10" * 2);  // 20
console.log("10" / "2"); // 5

// What if the string can't be converted to a number?
console.log("Apple" - 2); // NaN (Not a Number)

// TRICK: You can quickly explicitly coerce a string to a number by putting a + in front of it!
const input = "42";
const num = +input; 
console.log(typeof num); // "number"
```

### Level 3 — Intermediate (Truthy and Falsy)
```javascript
// In if() statements, JS coerces whatever you give it into a Boolean.
// Everything in JS coerces to TRUE ("Truthy"), EXCEPT for exactly 6 specific values!

// THE 6 FALSY VALUES:
// 1. false
// 2. 0
// 3. "" (Empty String)
// 4. null
// 5. undefined
// 6. NaN

if (0) {
  console.log("This will never run, because 0 is Falsy!");
}

if ("Hello") {
  console.log("This WILL run, because all non-empty strings are Truthy!");
}

if ([]) {
  console.log("This WILL run! Empty arrays are objects, and ALL objects are Truthy!");
}
```

### Level 4 — Advanced (The Double Equals Nightmare)
```javascript
// The Loose Equality operator (==) uses complex coercion rules that make zero sense.

console.log(5 == "5");     // true (Coerced string to number)
console.log(0 == false);   // true (Coerced false to 0)
console.log("" == false);  // true (Coerced both to 0)

// THE NIGHTMARE:
console.log(null == undefined); // true
console.log(null == 0);         // false
console.log([] == 0);           // true (Empty array coerces to "", which coerces to 0)
console.log([1,2] == "1,2");    // true (Arrays coerce to a string of their values)

// THE SOLUTION: NEVER USE DOUBLE EQUALS.
// Always use Strict Equality (===). It prevents coercion!
console.log(5 === "5");   // false
console.log(0 === false); // false
console.log([] === 0);    // false
```

## REAL_WORLD
Type coercion is the number one source of mathematical bugs in eCommerce platforms.
Imagine an HTML `<input type="number">` where the user enters the price of an item. Even though the HTML says `type="number"`, the DOM API (`input.value`) **always returns a String**.

If your backend sends down a `tax` variable (which is a Number `5`), and you attempt to calculate the total:
`const total = input.value + tax;`
If the user entered `50`, the total will be `"505"`, not `55`. You will accidentally charge the user five hundred and five dollars, resulting in a catastrophic financial bug!

To prevent this, you must **Always Explicitly Coerce** inputs before doing math:
`const total = Number(input.value) + tax;`

In massive enterprise systems, developers rely entirely on **TypeScript** to catch these exact coercion bugs at compile-time before the code ever reaches production.

## INTERVIEW
**Q: What is the output of `[] == ![]`?**
A: `true`. This is a classic trick question that highlights how broken double equals is. 
1. The right side `![]` turns the truthy array into a boolean `false`. So it becomes `[] == false`.
2. The `false` is coerced into `0`. It becomes `[] == 0`.
3. The empty array `[]` is coerced into an empty string `""`.
4. The empty string `""` is coerced into `0`.
5. `0 == 0` is `true`.
*Conclusion: Always use `===`!*

**Q: How do you quickly convert a value to a boolean in JS?**
A: Use the double-bang (`!!`) operator. The first `!` coerces the value to a boolean and inverses it. The second `!` inverses it back to its original Truthy/Falsy state. For example: `!!"Hello"` becomes `true`. `!!0` becomes `false`.

**Q: Are Objects Truthy or Falsy?**
A: All Objects, including Arrays and Functions, are inherently Truthy. Even an empty object `{}` or an empty array `[]` will evaluate to `true` in an `if()` statement. If you want to check if an array is empty, you cannot do `if(!arr)`, you must check its length: `if(arr.length === 0)`.

## FEYNMAN CHECK

### Explain Type Coercion Like I'm 10 Years Old
> Type coercion is JavaScript AUTOMATICALLY converting one type to another when an operator needs a specific type. `1 + '2'` → '12' (number coerced to string by +). `'5' - 1` → 4 (string coerced to number by -). `if (0)` → false (0 coerced to boolean). The rules are: `+` prefers string if EITHER operand is a string; `- * / %` always convert to number; `==` does AbstractEqualityComparison; Boolean coercion rules are: 0, '', null, undefined, NaN, false are FALSY, everything else is truthy. The non-obvious traps: `[] + []` → ''; `[] + {}` → '[object Object]'; `{} + []` → 0 (the `{}` is parsed as a block, not an object!); `true + true` → 2. These come up in live coding interviews.

### 5 Deep Conceptual Questions

**Q1: What algorithm does `==` actually run?**
> **A:** The Abstract Equality Comparison (ECMAScript §7.2.14): if types are equal, use strict equality. If null/undefined, they equal each other only. If one is number and other is string, convert string to number. If one is boolean, convert it to number first. If one is object and other is string/number/symbol, call ToPrimitive (valueOf then toString) on the object. This explains `'' == 0` (true), `[0] == 0` (true — [0].valueOf() → [0], [0].toString() → '0', '0' to number → 0), `null == undefined` (true by spec), `null == 0` (false — null only equals null/undefined).

**Q2: Mental model for ToPrimitive?**
> **A:** When an object needs to become a primitive (for +, ==, template literal, etc.), JavaScript calls `[Symbol.toPrimitive](hint)` first (if defined), then tries `.valueOf()` (default returns the object itself for plain objects — no coercion), then `.toString()`. For most plain objects, valueOf returns the object unchanged, so toString() wins: `{}.toString()` → '[object Object]'. Arrays: `[1,2].toString()` → '1,2'. Custom objects can override Symbol.toPrimitive for clean coercion behaviour.

**Q3: Most dangerous misconception?**
> **A:** All falsy values behave identically:
> ```js
> // ❌ null and undefined are NOT interchangeable in ==
> null == 0;         // false
> undefined == 0;   // false
> null == undefined; // true  (only each other)
>
> // ❌ NaN is not equal to itself
> NaN == NaN;   // false
> NaN === NaN;  // false
> Number.isNaN(NaN);  // true — only safe check
>
> // ❌ '0' is truthy but == to false
> Boolean('0');     // true (non-empty string!)
> '0' == false;     // true (coercion: false → 0, '0' → 0)
> if ('0') console.log('truthy');   // logs — confusing!
> ```

**Q4: How does implicit coercion interact with custom classes?**
> **A:** You can control coercion by defining `[Symbol.toPrimitive](hint)`, `valueOf()`, and `toString()`. `hint` is 'number', 'string', or 'default'. This powers library patterns: `moment().valueOf()` returns milliseconds so date comparison works (d1 < d2); BigDecimal libraries override toPrimitive so arithmetic with + is meaningful. Overuse leads to confusing APIs — prefer explicit `.toNumber()` methods.

**Q5: FAANG-grade definition?**
> **A:** "JavaScript implicit type coercion is the automatic application of the ToPrimitive, ToNumber, ToString, and ToBoolean abstract operations by operators and comparison algorithms — with `+` string-preferring, arithmetic operators number-converting, the Abstract Equality Comparison performing cross-type coercion including ToPrimitive for objects, and boolean coercion treating 0/NaN/''/null/undefined/false as falsy — producing a rich taxonomy of surprising results that motivated the prefer-`===` and prefer-explicit-coercion guidelines in modern style guides."

#### Step 3 — Custom toPrimitive
```js
// toPrimitive.js
export class Temperature {
  constructor(celsius) { this.celsius = celsius; }

  [Symbol.toPrimitive](hint) {
    if (hint === 'number') return this.celsius;
    if (hint === 'string') return `${this.celsius}°C`;
    return this.celsius;   // 'default' hint
  }
}

const temp = new Temperature(100);
console.log(`Boiling point: ${temp}`);   // "Boiling point: 100°C"
console.log(temp + 0);                    // 100 (number hint from arithmetic)
console.log(temp < 37);                   // false (body temp)
console.log(temp == 100);                 // true (default hint → number)
```

#### Step 4 — Explicit Coercion (Best Practice)
```js
// Prefer explicit over implicit
const userInput = '42px';

// ❌ Implicit
const implicit = +userInput;   // NaN (parseInt would work but + doesn't)

// ✅ Explicit — clear about what you want
const explicit = parseInt(userInput, 10);   // 42 — clear base 10
const asFloat  = parseFloat(userInput);      // 42 — stops at non-numeric
const asNum    = Number('42');               // 42
const asStr    = String(42);                 // '42'
const asBool   = Boolean(0);                 // false

// Safe string-to-number that returns null for invalid input
function toSafeInt(s) {
  const n = parseInt(s, 10);
  return Number.isFinite(n) ? n : null;
}
```
