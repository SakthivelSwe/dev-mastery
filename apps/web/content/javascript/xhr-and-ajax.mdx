## WHY
Before the early 2000s, the web was static. If you wanted to submit a form or fetch new data from the database, you clicked a button, and the browser physically refreshed the entire page, downloading all the HTML, CSS, and images again. The user experience was terrible.
In 2005, a technique called **AJAX** (Asynchronous JavaScript and XML) changed the world. It allowed developers to fetch data from a server *in the background*, without reloading the page, and update small sections of the DOM dynamically. The engine behind AJAX was the `XMLHttpRequest` (XHR) object. While modern apps use `fetch()`, understanding XHR is crucial for maintaining legacy codebases and understanding the history of the modern Web.

## THEORY
### The AJAX Paradigm
AJAX is not a technology, it is a paradigm. It combines:
1. **HTML/CSS:** For presentation.
2. **DOM:** For dynamic display and interaction.
3. **XML (or JSON):** For data interchange.
4. **XMLHttpRequest:** For asynchronous communication.

Despite the name having "XML" in it, modern AJAX relies almost entirely on JSON, as it is much lighter and easier to parse in JavaScript.

## CODE

### Level 1 — The Anatomy of an XHR Request
Creating an XHR request is highly verbose. You must create an object, open a connection, define event listeners, and then manually send it.

```javascript
// 1. Create the object
const xhr = new XMLHttpRequest();

// 2. Configure it: GET method, URL, and True for Asynchronous
xhr.open('GET', 'https://jsonplaceholder.typicode.com/users', true);

// 3. Setup the event listener for when the request finishes
xhr.onload = function() {
  // 4. Check if the HTTP status was successful
  if (xhr.status >= 200 && xhr.status < 300) {
    // 5. The response is a pure String. We must parse it into JSON!
    const data = JSON.parse(xhr.responseText);
    console.log("Users fetched:", data);
  } else {
    console.error("The server returned an error:", xhr.status);
  }
};

// Setup an event listener for physical network failures (e.g. WiFi down)
xhr.onerror = function() {
  console.error("A network error occurred.");
};

// 6. Finally, send the request over the wire!
xhr.send();
```

### Level 2 — Ready States
Under the hood, an XHR request goes through 5 distinct "Ready States" during its lifecycle. Before the modern `onload` event existed, developers had to manually listen to `onreadystatechange` and check the integer state!

```javascript
const xhr = new XMLHttpRequest();
xhr.open('GET', '/api/data');

xhr.onreadystatechange = function() {
  /*
    Ready States:
    0: UNSENT (Client has been created. open() not called yet.)
    1: OPENED (open() has been called.)
    2: HEADERS_RECEIVED (send() has been called, headers and status are available.)
    3: LOADING (Downloading the body; responseText holds partial data.)
    4: DONE (The operation is complete.)
  */
  
  if (xhr.readyState === 4) { // State 4 is DONE!
    if (xhr.status === 200) {
      console.log(JSON.parse(xhr.responseText));
    }
  }
};

xhr.send();
```

### Level 3 — Sending Data (POST)
To send data to a server using XHR, you must manually set the HTTP Headers, and you must manually stringify the JSON payload before passing it into `xhr.send()`.

```javascript
const xhr = new XMLHttpRequest();
xhr.open('POST', '/api/users');

// You MUST set the Content-Type header so the server knows how to parse the string!
xhr.setRequestHeader('Content-Type', 'application/json;charset=UTF-8');

xhr.onload = function() {
  if (xhr.status === 201) {
    console.log("User Created!");
  }
};

const payload = {
  name: "Alice",
  role: "Admin"
};

// The payload MUST be stringified!
xhr.send(JSON.stringify(payload));
```

### Level 4 — Progress Events (File Uploads)
Why would anyone still use XHR in the modern era of `fetch`?
**Upload Progress Indicators.**
For many years, the `fetch` API did not support tracking the percentage of a file upload. If you wanted to show a smooth 0% to 100% progress bar while uploading a 500MB video, you *had* to use XHR.

```javascript
const fileInput = document.getElementById('fileUpload');

fileInput.addEventListener('change', (e) => {
  const file = e.target.files[0];
  const formData = new FormData();
  formData.append('video', file);

  const xhr = new XMLHttpRequest();
  
  // Attach an event listener to the internal UPLOAD object!
  xhr.upload.onprogress = function(event) {
    if (event.lengthComputable) {
      // Calculate the percentage uploaded!
      const percentComplete = (event.loaded / event.total) * 100;
      console.log(`Upload is ${Math.round(percentComplete)}% complete`);
      document.getElementById('progressBar').style.width = `${percentComplete}%`;
    }
  };

  xhr.open('POST', '/api/upload');
  xhr.send(formData); // FormData handles headers automatically!
});
```

## REAL_WORLD
**jQuery and `$.ajax`**
In the late 2000s, dealing with raw XHR objects, handling Internet Explorer 6 quirks (which used `ActiveXObject` instead of XHR), and parsing JSON manually was a nightmare.
The library **jQuery** became the most popular JavaScript library in the world primarily because it provided the `$.ajax()` wrapper. It hid the horrors of XHR behind a clean, unified API.

```javascript
// The jQuery magic that revolutionized the web
$.ajax({
  url: '/api/users',
  method: 'GET',
  success: function(data) {
    // jQuery automatically parsed the JSON for you!
    console.log(data);
  },
  error: function(err) {
    console.error("Failed", err);
  }
});
```
Today, `fetch()` and `axios` have replaced both raw XHR and `$.ajax()`, but modern Single Page Applications (SPAs) owe their entire existence to the original AJAX revolution!

## INTERVIEW
**Q: You notice legacy code using XHR with a boolean flag set to `false`: `xhr.open('GET', '/api/data', false);`. What does this boolean do, and why does the browser console throw a massive deprecation warning about it?**
A: **Synchronous XHR (The Browser Freezer).**
The third argument in `xhr.open` is `async` (defaults to `true`). If you explicitly set it to `false`, you force the HTTP request to execute synchronously.
Because JavaScript is single-threaded, a Synchronous XHR request completely locks the main thread until the server responds! The user cannot scroll, animations freeze, and the entire browser tab goes completely dead. If the server takes 10 seconds to respond, the tab is frozen for 10 seconds.
This is so fundamentally destructive to the User Experience that modern browser vendors completely deprecated Synchronous XHR on the main thread, throwing massive warnings, and eventually planning to remove it entirely.
