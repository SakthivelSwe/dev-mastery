## WHY
Docker Hub is the world's largest public container registry with 15+ million images. It hosts official images for virtually every popular software: databases, runtimes, web servers, message queues. Understanding Docker Hub lets you leverage this ecosystem instead of building everything from scratch.

## THEORY
### What's on Docker Hub
- **Official Images:** Curated, security-scanned images maintained by Docker and the software vendors themselves (e.g., `nginx`, `postgres`, `redis`, `node`, `python`). These are the gold standard.
- **Verified Publisher Images:** Images from commercial vendors verified by Docker (e.g., `bitnami/postgresql`, `datadog/agent`).
- **Community Images:** Images from individual developers. Quality varies greatly. Always check the pull count, stars, and last update date.
- **Automated Builds:** Images automatically built from a connected GitHub repository.

## CODE

### Working with Docker Hub
```bash
# Search Docker Hub from the CLI
docker search nginx
# NAME                    DESCRIPTION                    STARS     OFFICIAL
# nginx                   Official build of Nginx...     20000     [OK]      ← Use this one!
# bitnami/nginx           Bitnami nginx Docker Image...  200

# Pull an official image
docker pull nginx:1.25-alpine

# Pull a specific digest (exact, immutable version — better than tags!)
docker pull nginx@sha256:abc123...

# Login to Docker Hub
docker login
# Enter your Docker Hub username and password

# Create a repository on Docker Hub: https://hub.docker.com/
# Repository name: myusername/my-app

# Tag and push your image
docker tag my-app:1.0.0 myusername/my-app:1.0.0
docker push myusername/my-app:1.0.0

# Logout (important on shared machines!)
docker logout
```

### Reading Docker Hub Image Tags
```bash
# Look at official postgres image tags to understand conventions:
# postgres:16          ← Latest patch of Postgres 16 (e.g., 16.2)
# postgres:16.2        ← Exact version
# postgres:16-alpine   ← Postgres 16 on Alpine Linux (smaller)
# postgres:16-bookworm ← Postgres 16 on Debian Bookworm (full)
# postgres:latest      ← WARNING: moves forward, never use in production!

# The pattern: {version}-{os-variant}
# Always prefer alpine variants for production (80% smaller)
# Use exact version tags for production
```

### Docker Hub API (Programmatic Access)
```bash
# List available tags for an image using the Docker Hub API
curl -s "https://hub.docker.com/v2/repositories/library/node/tags/?page_size=10" | \
  python3 -m json.tool | grep '"name"'

# Check image details
curl -s "https://hub.docker.com/v2/repositories/library/nginx/" | \
  python3 -m json.tool
```

## REAL_WORLD
**Evaluating Community Images for Safety**
When you must use a community image (not official), check these:
1. **Pull count:** 1M+ pulls suggests it's commonly trusted
2. **Stars:** Indicator of community quality
3. **Last updated:** Recent update = maintained; 2+ years old = risky
4. **Dockerfile source:** Reputable images link to their Dockerfile source code — audit it!
5. **Automated builds:** Image built automatically from source = code matches image
6. **Vulnerability scan:** Docker Hub shows vulnerability counts on verified images

Better alternative: Pull the official image and build your own image on top of it rather than trusting a community image.

## INTERVIEW
**Q: Why is it bad practice to use `FROM ubuntu:latest` or `FROM node:latest` in a production Dockerfile?**
A: The `latest` tag is a **mutable reference** — it points to whatever the most recent image is at the time of pull. This creates several problems:

1. **Non-reproducible builds:** Your CI server builds Monday with `node:latest = 20.11.0`. On Thursday, `latest` has been updated to `20.12.0`. Your Friday build uses a different Node.js version without you changing anything. Subtle differences between versions can cause mysterious bugs.

2. **Unexpected breaking changes:** If a major version is released (e.g., `latest` jumps from Node 20 to Node 22), your application might break silently.

3. **Compliance issues:** Security and compliance require knowing exactly what software version is running in production. "latest" doesn't tell you.

**The fix:** Always pin to a specific version:
```dockerfile
FROM node:20.11.1-alpine3.19  # Exact, immutable, reproducible
```

Even better, use the SHA256 digest which is truly immutable even if someone re-tags the version:
```dockerfile
FROM node:20-alpine@sha256:exact-immutable-hash
```
