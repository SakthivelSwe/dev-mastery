## WHY
Before Docker, the most common phrase in software development was: *"It works on my machine."*
A developer builds an application on their Mac laptop (macOS, Python 3.11, dependency version 2.4). They commit the code to GitHub. The CI/CD server tries to run it (Linux, Python 3.9, dependency version 2.1). It crashes.
The Operations team finally deploys it to Production (Linux, different library versions). It crashes differently.
**Docker** eliminates this fundamental problem by packaging your application and ALL of its dependencies (libraries, runtime, environment variables, config files) into a single, portable unit called a **Container**. The container runs identically on every machine that has Docker installed — developer laptops, CI servers, cloud VMs — guaranteed.

## THEORY
### The Core Concept: Shipping Containers
Docker borrowed an idea from the physical shipping industry. Before standardized shipping containers in the 1950s, unloading cargo was manual and inconsistent. Standardized containers meant any ship, any crane, any truck could handle them.
Docker containers are the software equivalent. Your application is packaged in a standardized container that any Docker host can run, regardless of the underlying OS or hardware.

### The Three Pillars
1. **Dockerfile:** The recipe. A text file with instructions for building your container image.
2. **Image:** The frozen, read-only blueprint built from the Dockerfile (like a class in OOP).
3. **Container:** A running instance of an Image (like an object instantiated from a class). You can run hundreds of containers from the same image.

## CODE

### Installing and Verifying Docker
```bash
# Verify Docker is installed correctly
docker --version
# Docker version 25.0.3

# Pull and run your first container instantly!
# This downloads the 'hello-world' image and runs it.
docker run hello-world

# Run an interactive Ubuntu container!
# -it: interactive terminal | ubuntu: image name
docker run -it ubuntu bash

# Inside the container, you are now root in an isolated Linux environment!
root@abc123:/# echo "I'm inside Docker!"
root@abc123:/# exit
```

### The "Works on My Machine" Fix
```bash
# Build your app image (we'll cover Dockerfile in detail later)
docker build -t my-node-app .

# Run it on YOUR machine
docker run my-node-app
# ✅ Works perfectly!

# Push it to Docker Hub (a free image registry)
docker push myusername/my-node-app:1.0

# Your teammate pulls and runs it on THEIR machine
docker pull myusername/my-node-app:1.0
docker run myusername/my-node-app:1.0
# ✅ Works IDENTICALLY — same Node.js version, same libraries, everything!
```

## REAL_WORLD
**Before and After Docker at a Startup**

*Before:*
- New developers spend 2 days setting up their environment.
- "Works on my machine" bugs take days to diagnose.
- Production deployments require a 20-page runbook.
- Different developers have slightly different library versions causing subtle bugs.

*After Docker:*
- New developers run `docker compose up` and have the entire stack running in 5 minutes.
- Environments are guaranteed identical — dev, staging, production all run the exact same containers.
- Deployments are atomic: swap Container V1 for Container V2.

## INTERVIEW
**Q: What is the difference between a Docker Image and a Docker Container?**
A: A **Docker Image** is a read-only, static blueprint or template. It is the result of running `docker build`. It contains the OS layers, runtime, libraries, and application code all frozen in place. Images are stored in registries (like Docker Hub) and can be shared.

A **Docker Container** is a live, running instance created from an Image. It has its own filesystem (a writable layer on top of the read-only Image), its own process space, and its own network interface. A single Image can spawn hundreds of simultaneous Containers. Think of it as: **Image = class, Container = object instance**.
