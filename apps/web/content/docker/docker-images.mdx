## WHY
A Docker Image is the fundamental unit of Docker. Everything starts from an image — you can't run a container without one. Understanding how images are structured, where they come from, and how to manage them is foundational to working effectively with Docker.
Images are like pre-packaged environments: pull an `nginx` image and you instantly have a configured web server. Pull a `postgres:15` image and you have a fully configured database. No manual installation required.

## THEORY
### Image Layers
Docker images use a **Union File System** and are built from layers. Each instruction in a Dockerfile creates a new layer on top of the previous. Layers are:
- **Read-only** after creation
- **Cached** — unchanged layers are reused between builds and pulls
- **Shared** — multiple images can share the same base layers (saving disk space)

When you run a container, Docker adds a **thin, writable layer** on top of the read-only image layers. All changes the container makes go into this writable layer, which is discarded when the container is deleted.

### Image Tags
Images use the format `name:tag` (e.g., `node:20-alpine`). The `latest` tag is applied when no tag is specified, but it's bad practice in production — always pin to a specific version tag.

## CODE

### Working with Images
```bash
# Pull from Docker Hub (the default public registry)
docker pull ubuntu:22.04
docker pull postgres:15-alpine
docker pull nginx:1.25

# List all locally cached images
docker images
# REPOSITORY   TAG         IMAGE ID       SIZE
# ubuntu       22.04       abc123...      77.8MB
# postgres     15-alpine   def456...      240MB
# nginx        1.25        ghi789...      187MB

# Inspect image layers
docker history nginx:1.25
# IMAGE         CREATED      CREATED BY                  SIZE
# abc123...    2 weeks ago  CMD ["nginx", "-g", ...      0B
# def456...    2 weeks ago  COPY file:...                3.27kB
# ghi789...    2 weeks ago  RUN /bin/sh -c ...           45.2MB

# Get detailed image info as JSON
docker inspect nginx:1.25

# Search Docker Hub for images
docker search redis

# Remove an image
docker rmi ubuntu:22.04

# Remove ALL unused (dangling) images
docker image prune

# Remove ALL unused images (not just dangling)
docker image prune -a
```

### Tagging Images
```bash
# Build an image and tag it
docker build -t my-app:1.0.0 .

# Add additional tags to an existing image
docker tag my-app:1.0.0 my-app:latest
docker tag my-app:1.0.0 myusername/my-app:1.0.0  # Tag for Docker Hub

# Push to Docker Hub
docker push myusername/my-app:1.0.0

# Pull from Docker Hub later
docker pull myusername/my-app:1.0.0
```

### Saving and Loading Images (Air-Gapped Environments)
```bash
# Export an image to a tar file (for transferring without a registry)
docker save -o my-app.tar my-app:1.0.0

# Load an image from a tar file
docker load -i my-app.tar
```

## REAL_WORLD
**Using Official Images vs Custom Images**
Docker Hub has thousands of official images maintained by the software vendors themselves. Always start from official images:

```dockerfile
# GOOD: Official, maintained images
FROM node:20-alpine        # Maintained by Node.js team
FROM postgres:15-alpine    # Maintained by PostgreSQL team
FROM nginx:1.25-alpine     # Maintained by Nginx team

# BAD: Random images from unknown users
FROM randomuser/mynode     # Who built this? What's in it?
```

**Alpine variants are almost always preferred:**
- `ubuntu` = 77 MB
- `ubuntu:22.04-slim` = 28 MB
- `node:20` = 1.1 GB (includes lots of build tools)
- `node:20-alpine` = 126 MB (just what you need!)
- `node:20-slim` = 246 MB

## INTERVIEW
**Q: Docker images are made of layers. Explain how this layer system benefits teams in terms of storage and build speed.**
A: The layer system provides two critical benefits:

1. **Shared Storage:** If 10 different images all start with `FROM node:20-alpine`, that base layer (126 MB) is stored on disk only **once**. Docker's Union File System presents each image as if it has its own complete filesystem, but the underlying storage is shared. This means 10 Node.js apps that each appear to be 150 MB might only occupy 160 MB total on disk.

2. **Build Cache:** Docker caches each layer after the first build. On subsequent builds, if an instruction and its context (files it copied) haven't changed, Docker reuses the cached layer and skips executing it. This means that changing a single source code file in a well-structured Dockerfile only rebuilds the final 1-2 layers, not the entire image. Dependency installations (`npm install`, `mvn dependency:resolve`) — the slowest operations — are cached and skipped on subsequent builds.

The key insight: **layer ordering in your Dockerfile determines cache effectiveness**. Put slow, rarely-changing operations first.
