## WHY
Docker's entire power is accessible through its CLI (Command Line Interface). Every Docker operation — pulling images, running containers, inspecting logs, managing networks — happens through the `docker` command.
Understanding the CLI is non-negotiable. Even if you later use Docker Compose or Kubernetes, these tools build on top of the same Docker primitives you master through the CLI.

## THEORY
### CLI Structure
Docker CLI commands follow a clear pattern: `docker <object> <action> [options] [args]`
- Objects: `container`, `image`, `volume`, `network`, `compose`
- Actions: `run`, `start`, `stop`, `rm`, `ls`, `inspect`, `logs`, `exec`

### Key Flags to Know
- `-d` (detached): Run container in background
- `-p` (publish): Map host port to container port
- `-e` (env): Set environment variable
- `-v` (volume): Mount a volume or bind mount
- `-it` (interactive + tty): Attach an interactive terminal
- `--name`: Give the container a human-readable name
- `--rm`: Automatically delete the container when it exits

## CODE

### Essential Container Commands
```bash
# --- RUNNING CONTAINERS ---

# Run nginx in background, name it 'web', map port 80
docker run -d --name web -p 8080:80 nginx

# Run an interactive Ubuntu session (--rm removes container on exit)
docker run -it --rm ubuntu bash

# Run with environment variables
docker run -d \
  -e POSTGRES_DB=myapp \
  -e POSTGRES_PASSWORD=secret \
  -p 5432:5432 \
  postgres:15

# --- MANAGING CONTAINERS ---

# List RUNNING containers
docker ps

# List ALL containers (including stopped)
docker ps -a

# Stop a container gracefully (sends SIGTERM, waits 10s, then SIGKILL)
docker stop web

# Start a stopped container
docker start web

# Force-kill immediately (SIGKILL)
docker kill web

# Remove a stopped container
docker rm web

# Remove a running container (force)
docker rm -f web

# --- LOGS ---

# View container logs
docker logs web

# Follow logs in real-time (like tail -f)
docker logs -f web

# Show last 100 lines
docker logs --tail=100 web

# --- EXEC (run commands inside a running container) ---

# Open a bash shell inside a running container
docker exec -it web bash

# Run a single command
docker exec web cat /etc/nginx/nginx.conf

# --- IMAGES ---

# Pull an image
docker pull redis:7

# List local images
docker images

# Remove an image
docker rmi redis:7

# Remove all unused images
docker image prune -a

# --- INSPECT ---

# Get full JSON details about a container (IP, mounts, env vars...)
docker inspect web

# Get just the IP address
docker inspect web --format '{{.NetworkSettings.IPAddress}}'

# Show port mappings
docker port web

# Show live resource usage (CPU, RAM, network)
docker stats
```

## REAL_WORLD
**Debugging a Crashed Container**
The most common debugging workflow:
```bash
# 1. Find the crashed container (Exit code non-zero)
docker ps -a
# CONTAINER ID   IMAGE     STATUS                      NAMES
# abc123         my-app    Exited (1) 2 minutes ago    api

# 2. Check what error caused the crash
docker logs abc123
# Error: Cannot connect to database at 'localhost:5432'
# Oops! The app is trying to connect to 'localhost' but the DB is in another container!

# 3. Inspect environment variables
docker inspect abc123 --format '{{json .Config.Env}}'
# ["DATABASE_URL=localhost:5432/mydb"]  ← Bug found!

# 4. Run with correct networking
docker run --network my-network -e DATABASE_URL=db:5432/mydb my-app
```

## INTERVIEW
**Q: What's the difference between `docker stop` and `docker kill`?**
A: Both stop a container, but the process differs:

`docker stop` sends **SIGTERM** (a polite "please shut down") to the container's main process (PID 1). This gives the application time to finish processing requests, close database connections, flush buffers, and shut down gracefully. After a configurable timeout (default 10 seconds), if the process hasn't stopped, Docker sends SIGKILL to forcefully terminate it.

`docker kill` immediately sends **SIGKILL** (a forceful termination) that the process cannot catch or ignore. The process is instantly terminated with no chance for cleanup.

Always use `docker stop` in production. Use `docker kill` only when a container is frozen and not responding to SIGTERM.
