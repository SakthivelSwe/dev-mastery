## WHY
While Docker Images are the blueprint, **Containers** are where the actual work happens. Understanding the full container lifecycle — creation, running, pausing, stopping, removal — and how to inspect and interact with running containers is the core of daily Docker usage.

## THEORY
### Container Lifecycle
1. **Created:** Container is created from an image but not started (`docker create`)
2. **Running:** Container is actively executing its main process (`docker start`/`docker run`)
3. **Paused:** Container processes are frozen in memory (SIGSTOP)
4. **Stopped:** Main process has exited. Container still exists, writable layer preserved
5. **Removed:** Container and its writable layer are permanently deleted

### Container ID vs Name
Every container gets a random hex ID (e.g., `abc123def456`) and optionally a human-readable name. You can use either to reference a container in commands. Always give containers meaningful names in production with `--name`.

## CODE

### Container Lifecycle Commands
```bash
# Create a container WITHOUT starting it
docker create --name mydb postgres:15

# Start a created container
docker start mydb

# Run = create + start in one step (most common)
docker run -d --name mydb postgres:15

# Pause all processes in a container (freezes everything in memory)
docker pause mydb

# Resume paused container
docker unpause mydb

# Gracefully stop (SIGTERM → SIGKILL after 10s)
docker stop mydb

# Force stop immediately (SIGKILL)
docker kill mydb

# Restart (stop + start)
docker restart mydb

# Remove stopped container
docker rm mydb

# Remove running container (force)
docker rm -f mydb

# Remove ALL stopped containers
docker container prune
```

### Inspecting Containers
```bash
# See all containers (running + stopped)
docker ps -a

# Detailed JSON information about a container
docker inspect mydb

# Extract specific fields with Go templates
docker inspect mydb --format '{{.NetworkSettings.IPAddress}}'
docker inspect mydb --format '{{.State.Status}}'
docker inspect mydb --format '{{.HostConfig.Memory}}'

# Live resource usage (CPU, RAM, network I/O, disk I/O)
docker stats                  # All containers
docker stats mydb             # Specific container
docker stats --no-stream mydb # One snapshot, then exit

# View running processes inside a container
docker top mydb
```

### Interacting with Running Containers
```bash
# Execute a command inside a running container
docker exec mydb psql -U postgres -c '\l'

# Open an interactive shell inside a running container
docker exec -it mydb bash    # If bash is available
docker exec -it mydb sh      # Alpine images use sh, not bash

# Copy files to/from containers
docker cp mydb:/var/log/postgresql/server.log ./local-copy.log
docker cp ./my-config.conf mydb:/etc/postgresql/postgresql.conf
```

## REAL_WORLD
**Debugging a Crashing Container (Container Exits Immediately)**
One of the most common frustrations: you run a container and it immediately exits.

```bash
# Container exits instantly after 'docker run'
docker run my-app

# Step 1: Check exit code and logs
docker ps -a
# STATUS: Exited (1) 2 seconds ago  ← Exit code 1 = error

docker logs $(docker ps -lq)  # Last container's logs
# Error: Port 3000 is already in use!

# Step 2: If there are no logs (app crashes before logging)
# Override CMD to start a shell instead, investigate manually
docker run -it --entrypoint sh my-app
# Now you're inside the container! Manually run your app:
> node server.js
# See the exact error output
```

## INTERVIEW
**Q: A container is running in production and shows high memory usage. You need to inspect it without stopping it. What commands would you use to diagnose the issue?**
A: Use these non-disruptive commands:

```bash
# 1. Check resource usage (CPU, Memory, Network, Disk I/O)
docker stats api --no-stream
# CONTAINER   CPU%    MEM USAGE / LIMIT    MEM%
# api         15.2%   485MiB / 512MiB      94.7%  ← Near the limit!

# 2. See all processes inside the container
docker top api
# PID     USER    COMMAND
# 12345   node    node server.js
# 12346   node    node worker.js  ← Unexpected second Node process?

# 3. Check recent logs for errors
docker logs --tail=200 --since=10m api

# 4. Open an interactive session to inspect runtime state
docker exec -it api sh
> cat /proc/meminfo        # Memory details
> ps aux                   # All processes
> ls /tmp                  # Check for temp files piling up
```

The key insight: `docker exec` lets you run commands inside a running container without stopping or restarting it, making it the primary tool for live debugging.
