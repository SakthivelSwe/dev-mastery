## WHY
You could manually run a container, install your dependencies inside it, copy your code, and run your app. But if that container dies, all your setup disappears.
A **Dockerfile** is an immutable, version-controlled recipe that automates the entire process of building your Docker Image. It's a plain text file containing a sequence of instructions. Every instruction creates a new layer in the image.
By committing the Dockerfile to Git alongside your application code, your entire environment definition is treated as code — **Infrastructure as Code**. Anyone can reproduce the exact same environment by running `docker build`.

## THEORY
### Dockerfile Instructions (The Most Important)
- `FROM`: Defines the base image to start from. Every Dockerfile starts with this.
- `WORKDIR`: Sets the working directory inside the container (like `cd` into a folder).
- `COPY` / `ADD`: Copies files from your host into the container.
- `RUN`: Executes a shell command during the **build** phase (e.g., `npm install`, `mvn package`).
- `ENV`: Sets environment variables available at both build-time and runtime.
- `EXPOSE`: Documents which port the container listens on (informational only — doesn't actually publish the port!).
- `CMD`: The default command to run when the container **starts** at runtime.
- `ENTRYPOINT`: Similar to CMD but harder to override; defines the container's main process.

### The Layer Cache (The Secret to Fast Builds)
Docker caches each layer. If you run `docker build` again and the instruction hasn't changed, Docker reuses the cached layer and skips rebuilding it. This makes subsequent builds incredibly fast. Ordering instructions strategically (slow-changing things first) dramatically speeds up development.

## CODE

### A Production-Grade Node.js Dockerfile
```dockerfile
# 1. FROM: Start from the official Node.js 20 image (LTS version)
# Use 'alpine' variant for a tiny 5MB OS instead of the full 100MB+ Ubuntu
FROM node:20-alpine

# 2. WORKDIR: All subsequent commands run inside /app in the container
WORKDIR /app

# 3. COPY package files FIRST! (Cache optimization - see REAL_WORLD below)
# Only these files are in this layer; if they don't change, npm install is skipped!
COPY package.json package-lock.json ./

# 4. RUN: Execute npm install INSIDE the container during build
RUN npm ci --only=production

# 5. COPY the rest of your source code AFTER npm install
COPY . .

# 6. ENV: Set environment variables
ENV NODE_ENV=production
ENV PORT=3000

# 7. EXPOSE: Document the port (doesn't publish it — use -p flag when running)
EXPOSE 3000

# 8. CMD: The command to run when the container starts
# Use array format (exec form) — NOT string format (shell form) for proper signal handling!
CMD ["node", "server.js"]
```

### Building and Running
```bash
# Build the image: -t gives it a name:tag
docker build -t my-node-app:1.0 .

# Run it: -p publishes port 3000 inside container to port 8080 on your machine
docker run -p 8080:3000 my-node-app:1.0

# Now visit http://localhost:8080!
```

### A Spring Boot Dockerfile
```dockerfile
# Stage 1: Build (Maven)
FROM maven:3.9-eclipse-temurin-21 AS build
WORKDIR /build
COPY pom.xml .
RUN mvn dependency:go-offline -q  # Cache dependencies
COPY src ./src
RUN mvn package -DskipTests -q

# Stage 2: Run (just the JRE, not the full JDK + Maven!)
FROM eclipse-temurin:21-jre-alpine
WORKDIR /app
COPY --from=build /build/target/*.jar app.jar
EXPOSE 8080
CMD ["java", "-jar", "app.jar"]
```

## REAL_WORLD
**The Layer Cache Trick — Order Matters Massively!**
Consider this **BAD** Dockerfile order:
```dockerfile
# BAD ORDER
COPY . .           # Copy ALL source code first
RUN npm install    # Install dependencies
```
Every time you change ANY file (even a comment in `README.md`), the `COPY . .` layer cache is invalidated. npm install reruns from scratch — 2 full minutes wasted every build.

The **CORRECT** order:
```dockerfile
# GOOD ORDER
COPY package.json package-lock.json ./   # Only copy dependency files
RUN npm install                          # This layer is CACHED as long as package.json doesn't change!
COPY . .                                 # Copy source code last — only this layer is rebuilt
```
Now changing your source code only rebuilds the last `COPY . .` layer in seconds. This single optimization can save teams hours of CI/CD time daily.

## INTERVIEW
**Q: What is the difference between `CMD` and `ENTRYPOINT` in a Dockerfile? Give an example of when you'd use each.**

A: Both define what runs when a container starts, but they differ in overridability.

**`CMD`** provides the default command that is easily overridden when running the container:
```bash
docker run my-image ls -la  # CMD is completely replaced by 'ls -la'
```

**`ENTRYPOINT`** defines the main executable and is NOT easily overridden. Arguments passed to `docker run` are APPENDED to ENTRYPOINT:
```bash
# Dockerfile: ENTRYPOINT ["ping"]
docker run my-ping google.com  # Runs: ping google.com
docker run my-ping facebook.com  # Runs: ping facebook.com
```

**Best pattern:** Use both together:
```dockerfile
ENTRYPOINT ["java", "-jar", "app.jar"]
CMD ["--spring.profiles.active=prod"]  # Default argument
```
Running `docker run my-app --spring.profiles.active=dev` overrides just the CMD argument, keeping the ENTRYPOINT. This makes the container behave like an executable with configurable defaults.
