## WHY
Software development is inherently complex. Without guiding principles, codebases naturally tend toward chaos, duplication, and over-engineering. Three famous acronyms—DRY, KISS, and YAGNI—serve as the foundational mantras for writing sane, maintainable code. They are not absolute rules, but rather heuristics to help you avoid the most common traps in software engineering.

## THEORY
### 1. DRY (Don't Repeat Yourself)
Every piece of knowledge must have a single, unambiguous, authoritative representation within a system.
**The Trap:** Copying and pasting code. If you find a bug in the original code, you have to remember to fix it in all 5 places you pasted it.

### 2. KISS (Keep It Simple, Stupid)
Systems work best if they are kept simple rather than made complicated.
**The Trap:** Trying to show off by writing a 1-line regex puzzle instead of 4 lines of highly readable, boring `if/else` statements.

### 3. YAGNI (You Aren't Gonna Need It)
Always implement things when you actually need them, never when you just foresee that you need them.
**The Trap:** Building a massive, scalable plugin architecture for a feature that the business might pivot away from next week.

## CODE

### Level 1 — DRY (Don't Repeat Yourself)
```javascript
// ❌ BAD: WET (Write Everything Twice)
function calculateAdminDiscount(price) {
    const tax = price * 0.2;
    const finalPrice = price + tax;
    return finalPrice * 0.8; // 20% discount
}

function calculateUserDiscount(price) {
    const tax = price * 0.2; // Duplicated logic!
    const finalPrice = price + tax;
    return finalPrice * 0.9; // 10% discount
}

// ✅ GOOD: DRY
function calculateTax(price) {
    return price * 0.2;
}

function calculateDiscount(price, discountRate) {
    const finalPrice = price + calculateTax(price);
    return finalPrice * (1 - discountRate);
}
```

### Level 2 — KISS (Keep It Simple, Stupid)
Developers often conflate "fewer lines of code" with "better code."

```javascript
// ❌ BAD: Overly clever, hard to read, hard to debug.
const isEligible = (u) => u.age > 18 ? u.hasLicense ? u.tickets === 0 ? true : false : false : false;

// ✅ GOOD: Boring, readable, easy to step through in a debugger.
const isEligible = (user) => {
    if (user.age <= 18) return false;
    if (!user.hasLicense) return false;
    if (user.tickets > 0) return false;
    
    return true;
};
```

### Level 3 — YAGNI (You Aren't Gonna Need It)
Imagine you are asked to build a function that fetches a User from the database.

```java
// ❌ BAD: YAGNI Violation. 
// "Maybe one day we will need to fetch from Redis, or an XML file, or a GraphQL API!"
public interface UserFetcher { ... }
public class DatabaseUserFetcher implements UserFetcher { ... }
public class CacheUserFetcher implements UserFetcher { ... }
public class UserFetcherFactory { ... }

// ✅ GOOD: Build exactly what is needed TODAY.
public class UserRepository {
    public User fetchUser(String id) {
        return db.query("SELECT * FROM users WHERE id = ?", id);
    }
}
```
If the requirements change next year, *then* you can refactor it into an Interface. Until then, the Factory and Interface are just dead weight slowing down the team.

### Level 4 — Advanced (The Rule of Three)
The biggest mistake Junior Developers make is **Premature Abstraction** (Applying DRY too early).
If you see two pieces of code that look similar, and you immediately try to extract them into a shared function, you might accidentally bind two unrelated concepts together. When one concept needs to change but the other doesn't, your shared function becomes a nightmare of `if/else` flags.

**The Rule of Three:**
1. First time: Write the code.
2. Second time: Copy and paste the code (Violate DRY).
3. Third time: Now that you have 3 distinct use cases, you actually understand the exact boundaries of the pattern. *Now* you refactor into a shared, DRY abstraction.

*Duplication is far cheaper than the wrong abstraction.*

## REAL_WORLD
**When DRY goes wrong (Microservices)**
In a Monolithic application, if you have a `User` entity, you share it everywhere (DRY).
When moving to Microservices, teams often try to create a shared `common-lib` library holding the `User` entity that all 50 microservices must import. This violates the core tenet of Microservices (Independent Deployability). If the Billing team adds a field to the `User` object, they have to force the Inventory team to update their `common-lib` version and re-deploy. 

In Microservices, **WET (Write Everything Twice) is often preferred over DRY.** The Billing service should have its own representation of a `User`, and the Inventory service should have its own representation.

## INTERVIEW
**Q: How do you balance DRY and KISS?**
A: DRY and KISS are often at odds. Extracting duplicated code into a highly configurable generic factory (DRY) often introduces massive complexity, making the code harder to read (violating KISS). As a general rule, readability and simplicity (KISS) should trump deduplication (DRY). If extracting 10 lines of duplicated code requires passing 5 boolean flags into a complex generic function, it is better to leave the duplication alone.
