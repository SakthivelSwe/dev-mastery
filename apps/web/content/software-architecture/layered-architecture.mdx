## WHY
If you write your database SQL queries inside the same function that renders your HTML buttons, your application will quickly become a monolithic, untestable nightmare. 
**Layered Architecture (N-Tier)** is the most common architectural pattern in the world. By dividing an application into horizontal layers—each with a specific responsibility—you ensure that changes to the UI don't accidentally break the database, and you make it possible to swap out underlying technologies in the future.

## THEORY
### The Layers
Most enterprise applications are divided into three or four standard layers:
1. **Presentation Layer (Web/API):** Handles HTTP requests, JSON parsing, and routing. (Controllers)
2. **Business/Domain Layer:** The heart of the application. Contains the rules and logic (e.g., "A user cannot withdraw more money than their balance"). (Services)
3. **Data Access / Persistence Layer:** Handles communication with the physical database. (Repositories / DAOs)
4. **Database Layer:** The actual PostgreSQL or MongoDB server.

### The Golden Rule of Dependencies
**Dependencies must always point strictly downward.**
The Presentation layer is allowed to call the Business layer.
The Business layer is allowed to call the Persistence layer.
The Persistence layer is **never** allowed to call the Business layer. Upward communication is strictly forbidden.

## CODE

### Level 1 — Beginner (The Presentation Layer)
The Presentation Layer (usually a REST Controller) should be incredibly thin. It does absolutely no business logic. Its only job is to receive HTTP traffic, pass it to the Business Layer, and return an HTTP response.

```java
// UserController.java (Presentation Layer)
@RestController
@RequestMapping("/users")
public class UserController {
    
    // Dependency injection points downwards to the Business Layer
    private final UserService userService;

    public UserController(UserService userService) {
        this.userService = userService;
    }

    @PostMapping
    public ResponseEntity<User> createUser(@RequestBody CreateUserDto dto) {
        // No logic here! Just passing data down.
        User createdUser = userService.registerNewUser(dto);
        return ResponseEntity.status(201).body(createdUser);
    }
}
```

### Level 2 — Building Blocks (The Business Layer)
The Business Layer (Service) contains the actual rules of the application. It knows nothing about HTTP, JSON, or SQL.

```java
// UserService.java (Business Layer)
@Service
public class UserService {
    
    // Dependency injection points downwards to the Persistence Layer
    private final UserRepository userRepository;

    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    public User registerNewUser(CreateUserDto dto) {
        // Core Business Logic!
        if (dto.getAge() < 18) {
            throw new IllegalArgumentException("User must be 18 or older.");
        }
        
        if (userRepository.existsByEmail(dto.getEmail())) {
            throw new IllegalStateException("Email already in use.");
        }

        User user = new User(dto.getName(), dto.getEmail());
        
        // Pass to the persistence layer to be saved
        return userRepository.save(user);
    }
}
```

### Level 3 — Intermediate (The Persistence Layer)
The Persistence Layer (Repository) handles the ugly details of talking to the database. It knows nothing about business rules.

```java
// UserRepository.java (Persistence Layer)
@Repository
public class UserRepository {
    
    private final JdbcTemplate db;

    public UserRepository(JdbcTemplate db) {
        this.db = db;
    }

    public boolean existsByEmail(String email) {
        String sql = "SELECT COUNT(*) FROM users WHERE email = ?";
        int count = db.queryForObject(sql, Integer.class, email);
        return count > 0;
    }

    public User save(User user) {
        String sql = "INSERT INTO users (name, email) VALUES (?, ?) RETURNING id";
        String id = db.queryForObject(sql, String.class, user.getName(), user.getEmail());
        user.setId(id);
        return user;
    }
}
```

### Level 4 — Advanced (The Sinkhole Anti-Pattern)
A common problem in Layered Architecture is the "Sinkhole Anti-Pattern" (also known as the Anemic Domain Model).
If 80% of the methods in your Business Layer look like this:
```java
public User getUserById(String id) {
    return userRepository.findById(id); // Literally just passing through
}
```
...then your Business Layer is acting as an empty sinkhole. The request flows through the Presentation layer, through the empty Business layer, straight to the Database. This usually happens when an application is entirely CRUD (Create, Read, Update, Delete) driven and has no actual business rules. If your application is pure CRUD, strict layered architecture might be overkill!

## REAL_WORLD
**Skipping Layers**
Is the Presentation layer allowed to skip the Business layer and talk directly to the Database layer? 
- **Strict Layered Architecture:** No. Layers must be closed. A request must pass through the layer immediately below it. This ensures the Business layer can always intercept requests to add caching or auditing later.
- **Relaxed Layered Architecture:** Yes. Some teams allow "open" layers, letting the Controller talk directly to the Repository for simple read-only queries (like fetching a dropdown list), avoiding the Sinkhole Anti-Pattern.

## INTERVIEW
**Q: What is the primary weakness of Layered Architecture compared to Clean/Hexagonal Architecture?**
A: In Layered Architecture, the Database sits at the very bottom, meaning the Business Layer depends on the Database Layer. Because dependencies flow downwards, the Business logic (the most important part of the app) becomes tightly coupled to the specific database technology (e.g., SQL tables). In Clean/Hexagonal Architecture, the Dependency Inversion Principle is used to flip this relationship, putting the Business Logic at the absolute center of the application, completely isolated from both the UI and the Database.
