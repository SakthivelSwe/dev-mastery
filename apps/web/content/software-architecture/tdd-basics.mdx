## WHY
If you write 500 lines of complex business logic and then spend 2 hours trying to write unit tests for it, you will likely discover that your code is incredibly difficult to test. You might have hardcoded database connections or deeply coupled dependencies. 

**Test-Driven Development (TDD)** flips this paradigm. By writing the test *before* the code, you force yourself to design an API that is inherently modular, loosely coupled, and highly testable. TDD is not just a testing strategy; it is a software design methodology.

## THEORY
### The Red-Green-Refactor Cycle
TDD is driven by a microscopic, rapid feedback loop that should take no longer than 2 to 5 minutes per cycle.

1. **RED:** Write a single, tiny test for a feature that doesn't exist yet. Run the test. It *must* fail (or fail to compile). This proves the test is actually testing something.
2. **GREEN:** Write the absolute minimum amount of messy, terrible code required to make the test pass. Do not write a generic algorithm. If the test expects `5`, just `return 5;`.
3. **REFACTOR:** Now that the test is green (your safety net is active), clean up the messy code. Extract methods, rename variables, and remove duplication.

## CODE

### Step 1: RED (Writing the Test First)
Imagine we need to build a `StringCalculator` that adds numbers from a string. 
Requirement 1: An empty string should return `0`.

We write the test before the class even exists!
```java
// StringCalculatorTest.java
import org.junit.jupiter.api.Test;
import static org.junit.jupiter.api.Assertions.assertEquals;

class StringCalculatorTest {
    
    @Test
    void emptyStringReturnsZero() {
        StringCalculator calc = new StringCalculator(); // Won't compile! Class doesn't exist.
        assertEquals(0, calc.add(""));
    }
}
```

### Step 2: GREEN (The Minimum Code)
We create the class and write the absolute bare minimum to make the test pass.
```java
// StringCalculator.java
public class StringCalculator {
    public int add(String numbers) {
        return 0; // Literally hardcoded just to pass the test!
    }
}
```
*We run the test. It passes (GREEN). We move on.*

### Step 3: RED (The Next Requirement)
Requirement 2: A single number returns the value.

```java
// StringCalculatorTest.java
@Test
void singleNumberReturnsValue() {
    StringCalculator calc = new StringCalculator();
    assertEquals(1, calc.add("1"));
}
```
*We run the test. It fails (RED) because our code hardcodes a `0`.*

### Step 4: GREEN (Updating the Logic)
```java
public class StringCalculator {
    public int add(String numbers) {
        if (numbers.isEmpty()) {
            return 0;
        }
        return Integer.parseInt(numbers); // Bare minimum to make both tests pass!
    }
}
```

### Step 5: RED (Complexity Increases)
Requirement 3: Two numbers separated by a comma return the sum.
```java
@Test
void twoNumbersCommaDelimitedReturnsSum() {
    StringCalculator calc = new StringCalculator();
    assertEquals(3, calc.add("1,2"));
}
```

### Step 6: GREEN & REFACTOR
```java
public class StringCalculator {
    public int add(String numbers) {
        if (numbers.isEmpty()) return 0;
        
        // This is getting messy, let's write the logic to pass...
        if (numbers.contains(",")) {
            String[] parts = numbers.split(",");
            return Integer.parseInt(parts[0]) + Integer.parseInt(parts[1]);
        }
        
        return Integer.parseInt(numbers);
    }
}
```
Now that all 3 tests pass, we enter the **REFACTOR** stage. We can clean up the algorithm without fear, because if we break it, the tests will instantly fail!

```java
// REFACTORED CODE
public class StringCalculator {
    public int add(String numbers) {
        if (numbers.isEmpty()) return 0;
        
        int sum = 0;
        for (String num : numbers.split(",")) {
            sum += Integer.parseInt(num);
        }
        return sum;
    }
}
```

## REAL_WORLD
**When NOT to use TDD**
TDD is incredible for complex business logic, algorithms, and core domain rules.
However, TDD is historically terrible for UI Development and Exploratory Coding (Prototyping). 
If you don't actually know what the final API should look like, or if you are tweaking CSS pixels to see what looks good on screen, writing a test first is an exercise in frustration. Many pragmatic developers use TDD for their backend business layer, but use manual testing for their frontend React components.

## INTERVIEW
**Q: What is "Mockist" vs "Classicist" TDD?**
A: 
- **Classicists (Chicago School):** Prefer testing real objects. If an `Order` object depends on a `Product` object, they instantiate a real `Product`. They only use Mock objects for things that cross system boundaries (like an HTTP call or Database connection).
- **Mockists (London School):** Believe a unit test should test exactly one class in absolute isolation. If `Order` depends on `Product`, they will inject a completely fake, Mocked version of `Product`. This ensures that if the test fails, the bug is guaranteed to be in `Order`, not `Product`.

#### Step 4 — Write Tests

```javascript
// Simple test runner (no dependencies needed)
function test(description, fn) {
  try {
    fn();
    console.log(`  ✅ PASS: ${description}`);
  } catch (e) {
    console.error(`  ❌ FAIL: ${description} — ${e.message}`);
  }
}

function assert(actual, expected, msg = '') {
  if (JSON.stringify(actual) !== JSON.stringify(expected)) {
    throw new Error(`Expected ${JSON.stringify(expected)} but got ${JSON.stringify(actual)}. ${msg}`);
  }
}

test('initialises correctly', () => {
  const m = createTest-DrivenDevelopment(TDD)();
  assert(typeof m.init, 'function');
  assert(typeof m.process, 'function');
});

test('processes valid input', () => {
  const m = createTest-DrivenDevelopment(TDD)();
  m.init('test');
  const out = m.process('test');
  assert(out !== null && out !== undefined, true, 'output should not be null');
});

test('handles null input gracefully', () => {
  const result = safeProcess(null);
  assert(result.success, false);
  assert(typeof result.error, 'string');
```
