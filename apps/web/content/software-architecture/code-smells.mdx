## WHY
A "Code Smell" is a surface indication that usually corresponds to a deeper problem in your system. It doesn't mean the code is broken (it might pass all unit tests!), but it indicates weakness in design that will slow down future development or breed bugs. Learning to identify these smells allows you to refactor proactively before the codebase rots completely.

## THEORY
### What is a Code Smell?
Coined by Kent Beck in Martin Fowler's legendary book *Refactoring*, a code smell is an intuition. It's the moment you look at a file and think, "Something feels wrong here."

Code smells are divided into categories:
1. **Bloaters:** Code, methods, or classes that have grown so large they are hard to work with.
2. **Object-Orientation Abusers:** Incomplete or incorrect application of OOP principles.
3. **Change Preventers:** Code that forces you to change multiple files just to make one simple modification.
4. **Dispensables:** Pointless, dead, or redundant code.
5. **Couplers:** Excessive coupling between classes.

## CODE

### Bloater 1: Long Method
The most common smell. A method should ideally fit on a single laptop screen (roughly 20-30 lines). If a method is 200 lines long, it is doing too much.
**The Fix:** Extract Method. Highlight blocks of logic and pull them into their own well-named private methods.

### Bloater 2: Primitive Obsession
Using basic primitives (`string`, `int`, `array`) to represent complex domain concepts.

```java
// ❌ BAD: Primitive Obsession
public void processOrder(String currency, double amount, String zipCode, String street) { ... }

// ✅ GOOD: Wrap primitives in Domain Objects
public void processOrder(Money price, Address address) { ... }
```
*Why?* Because a `zipCode` isn't just a String. It has validation rules (it must be 5 digits). If you use a String, you have to validate it everywhere. If you use an `Address` object, the object validates itself upon creation!

### Change Preventer: Shotgun Surgery
You want to add a new "PayPal" payment type. You find yourself opening `Order.java`, `Invoice.java`, `PaymentController.java`, and `DatabaseMapper.java`, adding a small `if (type == "paypal")` to every single file.
This is Shotgun Surgery: a single responsibility is spread across the entire codebase.
**The Fix:** Move Method / Move Field. Consolidate the logic into a single `PaymentProcessor` Strategy class.

### Object-Orientation Abuser: Switch Statements
In pure Object-Oriented code, long `switch` statements or massive `if/else if` chains based on an object's "type" are considered a smell.

```java
// ❌ BAD: A switch statement based on type
public double calculateShipping(Package pkg) {
    switch (pkg.type) {
        case "STANDARD": return 5.00;
        case "EXPRESS": return 15.00;
        case "OVERNIGHT": return 25.00;
    }
}
```
**The Fix:** Polymorphism. The `Package` interface should define a `calculateShipping()` method. The `StandardPackage` and `ExpressPackage` classes should implement their own specific logic.

### Coupler: Feature Envy
When a method in Class A spends more time talking to Class B than it does talking to its own class.

```java
// ❌ BAD: The User class is envious of the ShoppingCart's data!
class User {
    public double getCartTotal(ShoppingCart cart) {
        double total = 0;
        for (Item item : cart.getItems()) {
            total += item.getPrice() * cart.getTaxRate();
        }
        return total;
    }
}
```
**The Fix:** Move Method. The `getCartTotal()` logic clearly belongs inside the `ShoppingCart` class, not the `User` class!

## REAL_WORLD
**Data Clumps**
Do you often see the exact same three variables traveling together as parameters across 15 different methods?
`startDate` and `endDate`.
`x`, `y`, and `z`.

These are called Data Clumps. If data always travels together, it wants to be an Object.
Refactor `(Date startDate, Date endDate)` into a single `DateRange range` object. This shrinks method signatures instantly and gives you a place to put logic (like `range.contains(someDate)`).

## INTERVIEW
**Q: What is a "God Object" and how do you fix it?**
A: A God Object (or God Class) is a massive class that knows too much or does too much (e.g., a `SystemManager` or `ApplicationCore` class with 5,000 lines of code and 100 methods). It violates the Single Responsibility Principle and becomes a bottleneck because every developer is constantly modifying the same file, causing merge conflicts. You fix it by identifying distinct responsibilities within the class and extracting them into smaller, focused classes using the "Extract Class" refactoring technique.
