## WHY
Code is never perfect on the first try. As requirements change, the underlying structure of the code must adapt. **Refactoring** is the disciplined process of changing a software system in such a way that it does not alter the external behavior of the code, yet improves its internal structure. Without refactoring, a codebase slowly accumulates "Technical Debt," eventually making it impossible to add new features without breaking existing ones.

## THEORY
### The Golden Rule of Refactoring
**You cannot refactor code that doesn't have automated tests.**
If you change the internal structure of a system without unit tests, you are not refactoring; you are just moving bugs around and hoping for the best. Tests act as your safety net, instantly telling you if your structural changes accidentally altered the application's behavior.

### The Two Hats
Martin Fowler describes software development as wearing two distinct hats:
1. **The Adding Function Hat:** You write new code to add a feature. You might write messy code just to get it working. You are NOT allowed to change existing architecture.
2. **The Refactoring Hat:** You are NOT allowed to add new features. You only restructure existing code to make it cleaner.

*Never wear both hats at the same time.* Get the feature working first, commit it, then put on the Refactoring hat and clean it up.

## CODE

### Technique 1: Extract Method
The most common refactoring technique. You take a massive function and pull distinct blocks of logic into their own private methods.

```java
// BEFORE
public void printOwing() {
    printBanner();
    
    // Calculate outstanding
    double outstanding = 0;
    for (Order order : orders) {
        outstanding += order.getAmount();
    }
    
    // Print details
    System.out.println("Name: " + name);
    System.out.println("Amount: " + outstanding);
}

// AFTER
public void printOwing() {
    printBanner();
    double outstanding = calculateOutstanding();
    printDetails(outstanding);
}

private double calculateOutstanding() { ... }
private void printDetails(double outstanding) { ... }
```

### Technique 2: Replace Magic Number with Symbolic Constant
A magic number is a direct usage of a number in code with no context.

```java
// BEFORE
public double potentialEnergy(double mass, double height) {
    return mass * height * 9.81; // What is 9.81?
}

// AFTER
static final double GRAVITATIONAL_CONSTANT = 9.81;

public double potentialEnergy(double mass, double height) {
    return mass * height * GRAVITATIONAL_CONSTANT;
}
```

### Technique 3: Introduce Explaining Variable
Complex boolean conditions are hard to read. Extract them into highly readable boolean variables.

```java
// BEFORE
if ((platform.toUpperCase().indexOf("MAC") > -1) && 
    (browser.toUpperCase().indexOf("IE") > -1) && 
    wasInitialized() && resize > 0) {
    // do something
}

// AFTER
final boolean isMacOs = platform.toUpperCase().indexOf("MAC") > -1;
final boolean isIEBrowser = browser.toUpperCase().indexOf("IE") > -1;
final boolean wasResized = resize > 0;

if (isMacOs && isIEBrowser && wasInitialized() && wasResized) {
    // do something
}
```

### Technique 4: Replace Conditional with Polymorphism
If you have massive `switch` statements checking an object's type to perform different behavior, you should push that behavior down into subclasses.

```java
// BEFORE
public double getSpeed(Bird bird) {
    switch (bird.type) {
        case EUROPEAN: return getBaseSpeed();
        case AFRICAN: return getBaseSpeed() - getLoadFactor() * bird.numberOfCoconuts;
        case NORWEGIAN_BLUE: return bird.isNailed ? 0 : getBaseSpeed(bird.voltage);
    }
}

// AFTER
abstract class Bird {
    abstract double getSpeed();
}

class European extends Bird {
    double getSpeed() { return getBaseSpeed(); }
}

class African extends Bird {
    double getSpeed() { return getBaseSpeed() - getLoadFactor() * numberOfCoconuts; }
}
```

## REAL_WORLD
**Continuous Refactoring vs Planned Refactoring**
Many teams ask their Product Managers for a "Refactoring Sprint" (a full month dedicated to paying down technical debt). Product Managers almost always reject this because it delivers zero business value to the customer.

Instead, practice **Continuous Refactoring** (The Boy Scout Rule). When a developer picks up a ticket to add a button to the UI, they should spend 10 minutes refactoring the messy UI component *before* adding the button. Refactoring should be baked into the daily workflow of building features, not isolated as a separate task.

## INTERVIEW
**Q: What is the difference between Refactoring and Rewriting?**
A: 
- **Refactoring** is done in tiny, incremental, isolated steps. The application must be fully functional and compile successfully after *every single step*.
- **Rewriting** is starting from scratch. It is incredibly risky, often takes 3 times longer than estimated, and halts new feature development for months. You should almost always heavily favor aggressive refactoring over a full rewrite.
