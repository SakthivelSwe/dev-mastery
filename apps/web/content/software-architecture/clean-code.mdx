## WHY
Code is read 10 times more often than it is written. If you write clever, cryptic code to save 5 minutes of typing today, you will cost your team 5 hours of debugging next month. "Clean Code," a concept popularized by Robert C. Martin (Uncle Bob), is a set of guidelines focused on making code legible, maintainable, and predictable.

Your goal as a professional engineer is not just to make the computer understand your instructions—it is to make the next *human* understand your intent.

## THEORY
### The Core Tenets
1. **Meaningful Names:** Variables should reveal their intent.
2. **Small Functions:** Functions should do exactly one thing.
3. **Avoid Comments:** Code should document itself. Comments lie.
4. **Formatting:** Consistent indentation and vertical spacing.

## CODE

### Level 1 — Beginner (Meaningful Names)
Names must answer three questions: Why it exists, what it does, and how it is used. Avoid single-letter variables (except in short loops) and abbreviations.

```javascript
// ❌ BAD: What is 'd'? What is 'usr'?
let d = 0; // elapsed time in days
function getUsr(id) { ... }

// ✅ GOOD: Intention-Revealing Names
let elapsedTimeInDays = 0;
function fetchUserById(userId) { ... }

// ❌ BAD: Magic Numbers
if (user.age > 21 && user.status === 2) { ... }

// ✅ GOOD: Searchable Constants
const LEGAL_DRINKING_AGE = 21;
const STATUS_ACTIVE = 2;
if (user.age > LEGAL_DRINKING_AGE && user.status === STATUS_ACTIVE) { ... }
```

### Level 2 — Building Blocks (Functions Should Do One Thing)
A function should do one thing, do it well, and do it only. If your function name has the word "And" in it (e.g., `validateUserAndSaveToDatabase()`), it is doing more than one thing.

```javascript
// ❌ BAD: Doing three things (Validating, Saving, Emailing)
function createUser(user) {
    if (!user.email || !user.password) {
        throw new Error("Invalid input");
    }
    
    db.users.insert(user);
    
    emailService.sendWelcomeEmail(user.email);
}

// ✅ GOOD: Composed functions, each doing ONE thing
function createUser(user) {
    validateUserPayload(user);
    persistUser(user);
    notifyNewUser(user);
}

function validateUserPayload(user) { ... }
function persistUser(user) { ... }
function notifyNewUser(user) { ... }
```
*Notice how the `createUser` function now reads like a table of contents!*

### Level 3 — Intermediate (Don't Write Comments)
Comments are generally considered a failure to express yourself in code. Code is the single source of truth; comments often rot and become outdated, leading developers astray.

```javascript
// ❌ BAD: The comment explains complex, messy code
// Check if employee is eligible for full benefits
if ((employee.flags & HOURLY_FLAG) && employee.age > 65) { ... }

// ✅ GOOD: The function name makes the comment obsolete!
if (employee.isEligibleForFullBenefits()) { ... }
```
**The Exceptions:** You *should* write comments to explain the *Why* (e.g., "We are using a slow regex here because of a known bug in Safari version 14"), or to generate documentation for public APIs (JSDoc/JavaDoc).

### Level 4 — Advanced (Error Handling over Null Checks)
Sprinkling `if (result != null)` throughout your codebase creates a phenomenon called "Null Pointer paranoia" and obscures the actual business logic. 

```javascript
// ❌ BAD: Returning null forces the caller to check for null
function getConfiguration() {
    let file = readFile("config.json");
    if (!file) return null;
    return JSON.parse(file);
}

// ✅ GOOD: Throwing exceptions delegates error handling to the appropriate layer
function getConfiguration() {
    let file = readFile("config.json");
    if (!file) {
        throw new Error("Missing critical config.json file");
    }
    return JSON.parse(file);
}
```

## REAL_WORLD
**The Boy Scout Rule**
"Always leave the campground cleaner than you found it." 
In professional environments, code rots over time. If you are fixing a bug in a 1,000-line file, take 5 minutes to rename a confusing variable, extract a massive 100-line function into two smaller ones, or delete dead code. You don't need permission to make small, safe structural improvements while completing your primary task. This incremental refactoring is what keeps massive codebases alive for decades.

## INTERVIEW
**Q: What is the ideal number of arguments for a function?**
A: According to Clean Code:
- Zero arguments (Niladic) is ideal.
- One argument (Monadic) is very good.
- Two arguments (Dyadic) is acceptable, but requires caution (it's easy to accidentally swap `expected` and `actual` in testing).
- Three arguments (Triadic) should be avoided.
- Four or more (Polyadic) requires special justification. 

If a function requires 4+ arguments, those concepts are usually related and should be wrapped into a single Object or Class (e.g., instead of passing `x, y, radius, color`, pass a `CircleOptions` object).

#### Step 3 — Add Error Handling

```javascript
// Robust error handling for production use
function safeProcess(input) {
  try {
    if (input === null || input === undefined) {
      throw new TypeError(`[Clean Code Principles] Input cannot be null or undefined`);
    }
    const instance = createCleanCodePrinciples();
    return { success: true, data: instance.process(input) };
  } catch (err) {
    console.error(`[Clean Code Principles] Error:`, err.message);
    return { success: false, error: err.message };
  }
}

console.log(safeProcess('valid input'));   // { success: true, data: '...' }
console.log(safeProcess(null));            // { success: false, error: '...' }
```
