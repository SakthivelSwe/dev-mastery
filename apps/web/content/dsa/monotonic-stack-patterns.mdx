## WHY
You are given an array of stock prices over 10 days: `[73, 74, 75, 71, 69, 72, 76, 73]`.
For every single day, you must calculate exactly how many days you have to wait until the price is higher than today's price.
If you use a nested loop (for every day, loop forward to find a higher price), the time complexity is $O(N^2)$.
The **Monotonic Stack** is a hyper-specialized data structure that solves "Next Greater Element" problems in linear $O(N)$ time. By mathematically enforcing that elements inside the Stack are strictly increasing or strictly decreasing, it can instantly resolve queries across massive arrays without ever looping backwards!

## THEORY
### What is a Monotonic Stack?
A standard Stack pushes and pops items blindly.
A **Monotonic Stack** enforces a mathematical rule before pushing an item:
- **Monotonically Decreasing Stack:** The elements inside the stack MUST be getting smaller and smaller from bottom to top (e.g., `[100, 80, 60, 40]`).
- **The Rule:** If you try to push the number `90` onto that stack, the rule is violated! `90` is bigger than `40`. 
- **The Magic:** Before pushing `90`, the Stack violently pops out `40`, `60`, and `80`. *Why?* Because `90` is mathematically the "Next Greater Element" for those three numbers! We just answered three queries simultaneously! Then we safely push `90`. The stack is now `[100, 90]`.

## CODE

### Pattern 1 — Next Greater Element (Daily Temperatures)
*Problem:* Given an array of integers `temperatures` represents the daily temperatures, return an array `answer` such that `answer[i]` is the number of days you have to wait after the $i^{th}$ day to get a warmer temperature.

**1. The Stack Data:** Instead of storing the actual temperatures in the stack, we store the **Indices** of those temperatures. We need the indices to calculate "how many days passed"!

```java
import java.util.Stack;

public int[] dailyTemperatures(int[] temperatures) {
    int[] answer = new int[temperatures.length];
    
    // The Stack will strictly store the INDICES of the temperatures.
    // We enforce a Monotonically Decreasing pattern!
    Stack<Integer> stack = new Stack<>();

    for (int i = 0; i < temperatures.length; i++) {
        int currentTemp = temperatures[i];

        // RULE VIOLATION! The current temp is hotter than the temp at the top of the stack!
        while (!stack.isEmpty() && temperatures[stack.peek()] < currentTemp) {
            
            // Pop the index of the cold day!
            int coldDayIndex = stack.pop();
            
            // We just found the Next Greater Temperature for that cold day!
            // Calculate the distance in days!
            int daysWaited = i - coldDayIndex;
            
            // Save the answer!
            answer[coldDayIndex] = daysWaited;
        }

        // The stack is now perfectly monotonic. Safe to push today's index!
        stack.push(i);
    }

    // Any indices left in the stack never found a warmer day.
    // The Java array is automatically initialized to 0, which is exactly what we want!
    return answer;
}
```

### Pattern 2 — Previous Smaller Element (Largest Rectangle in Histogram)
*Problem:* You are given an array of bar heights `[2, 1, 5, 6, 2, 3]`. Find the area of the largest rectangle you can form inside the histogram.
*Why Monotonic?* To find the area of a rectangle, you need to know the width. The width of a bar is bounded by the very first bar to its Left that is shorter than it, and the very first bar to its Right that is shorter than it!

We use a Monotonically **Increasing** Stack! When a short bar arrives, it pops all the tall bars out of the stack, instantly calculating their maximum possible rectangle areas!

```java
public int largestRectangleArea(int[] heights) {
    int maxArea = 0;
    Stack<Integer> stack = new Stack<>();
    
    // We loop up to N (inclusive) to force a final "flush" of the stack at the end
    for (int i = 0; i <= heights.length; i++) {
        
        // If we reached the end, use a height of 0 to flush everything!
        int currentHeight = (i == heights.length) ? 0 : heights[i];

        // RULE VIOLATION! A shorter bar arrived! The tall bars must be popped!
        while (!stack.isEmpty() && heights[stack.peek()] > currentHeight) {
            // 1. Pop the tall bar. This is the height of our rectangle!
            int rectHeight = heights[stack.pop()];

            // 2. Calculate the Width!
            // The Right boundary is 'i' (the short bar that caused the violation).
            // The Left boundary is whatever is left sitting directly beneath it in the stack!
            int rectWidth;
            if (stack.isEmpty()) {
                rectWidth = i; // It extends all the way to the absolute beginning!
            } else {
                rectWidth = i - stack.peek() - 1; 
            }

            // 3. Calculate Area
            maxArea = Math.max(maxArea, rectHeight * rectWidth);
        }
        
        stack.push(i);
    }
    return maxArea;
}
```

## REAL_WORLD
**Stock Market Algorithms (High-Frequency Trading)**
A Monotonic Stack is practically a required skill for building low-latency trading systems.
Imagine processing a live stream of ticks (price changes) for a specific stock. You are writing an algorithm to detect "Price Support Breakdowns". You need to know, for every previous peak price over the last 10,000 trades, the exact millisecond the stock price violently plummeted below that peak.
A standard loop would take millions of operations, introducing 500+ milliseconds of latency to your trading bot. 
By feeding the live data stream directly into a Monotonically Increasing Stack, the exact instant the stock crashes, the Stack violently pops all the vulnerable peaks in $O(1)$ amortized time. The system executes the protective "Sell" orders with microsecond latency!

## INTERVIEW
**Q: Why does the Monotonic Stack operate in $O(N)$ time if there is a `while` loop nested inside a `for` loop? Doesn't a nested loop automatically make it $O(N^2)$?**
A: **No! Look at the Push/Pop lifecycle!**
It is true that the `while` loop might iterate 50 times in a single step (if 50 items are suddenly popped from the stack at once).
However, you must analyze the algorithm globally, not locally.
Every single element in the array is `pushed` onto the stack EXACTLY once.
Every single element in the array is `popped` from the stack AT MOST once.
If the array has $N=100$ items, there can be a maximum of 100 pushes and 100 pops across the *entire execution of the algorithm*. The `while` loop cannot magically conjure elements to pop. The total number of operations across all iterations of the inner loop is mathematically capped at $2N$. Therefore, the amortized time complexity is definitively $O(N)$.
