## WHY
Imagine you are trying to crack a 3-digit combination padlock. You start with `0-0-0`. If that fails, you try `0-0-1`. You systematically try every single combination until the lock opens.
Now imagine trying to solve a Sudoku puzzle. If you blindly try every possible number in every empty square, the number of permutations is astronomically large ($9^{81}$), and the universe will end before your code finishes.
**Backtracking** is an algorithmic technique for exhaustive search. It systematically builds candidates for a solution, but the moment it realizes a partial candidate *cannot possibly* be completed to a valid solution (e.g., you placed two '5's in the same Sudoku row), it instantly abandons that path, "backtracks" to the previous state, and tries a different path. This massive pruning of dead ends is what allows us to solve NP-Hard puzzles!

## THEORY
### The Backtracking Template
Every backtracking algorithm in existence follows the exact same recursive template:
1. **Choose:** Make a choice (add an element to our current path).
2. **Explore:** Call recursion to continue building the path with this new choice.
3. **Un-Choose (Backtrack):** Remove the choice we just made, returning the state back to exactly how we found it, so the loop can try the *next* choice in a clean environment.

## CODE

### Pattern 1 — Permutations (All Possible Orders)
*Problem:* Given an array `[1, 2, 3]`, return all possible permutations: `[1,2,3], [1,3,2], [2,1,3]...`

```java
import java.util.*;

public class Permutations {
    
    public List<List<Integer>> permute(int[] nums) {
        List<List<Integer>> results = new ArrayList<>();
        // The 'path' stores the current combination we are building!
        backtrack(results, new ArrayList<>(), nums);
        return results;
    }

    private void backtrack(List<List<Integer>> results, List<Integer> path, int[] nums) {
        // Base Case: Our path is full! We found a valid permutation.
        if (path.size() == nums.length) {
            // CRITICAL: We must make a deep copy of the path!
            // If we don't, the backtracking will empty it later!
            results.add(new ArrayList<>(path));
            return;
        }

        for (int i = 0; i < nums.length; i++) {
            // Permutations cannot reuse numbers. Skip if it's already in the path!
            // (Note: Using a boolean[] visited array is much faster than path.contains())
            if (path.contains(nums[i])) continue;

            // 1. CHOOSE
            path.add(nums[i]);

            // 2. EXPLORE
            backtrack(results, path, nums);

            // 3. UN-CHOOSE (BACKTRACK)
            // Remove the last element we just added, so the loop can try the next number!
            path.remove(path.size() - 1);
        }
    }
}
```

### Pattern 2 — Subsets (Combinations)
*Problem:* Given an array `[1, 2, 3]`, return all possible subsets (the Power Set): `[], [1], [2], [1,2], [1,2,3]...`
Unlike permutations, order doesn't matter (`[1,2]` is the same subset as `[2,1]`). To prevent duplicates, we pass an `index` to the recursion, forcing the loop to only look FORWARD!

```java
public class Subsets {
    
    public List<List<Integer>> subsets(int[] nums) {
        List<List<Integer>> results = new ArrayList<>();
        backtrack(results, new ArrayList<>(), nums, 0); // Start index is 0
        return results;
    }

    private void backtrack(List<List<Integer>> results, List<Integer> path, int[] nums, int startIdx) {
        // Every single path we generate is a valid subset! Add it immediately.
        results.add(new ArrayList<>(path));

        // Loop starting from 'startIdx' to prevent backwards duplicates!
        for (int i = startIdx; i < nums.length; i++) {
            
            path.add(nums[i]); // CHOOSE
            
            // EXPLORE: Pass i + 1 to force the next level to only look at FUTURE numbers!
            backtrack(results, path, nums, i + 1);
            
            path.remove(path.size() - 1); // UN-CHOOSE
        }
    }
}
```

### Level 3 — The Pruning Power (N-Queens)
*Problem:* Place `N` chess Queens on an `N x N` board so that no two queens attack each other.
If we use brute force, $8$ queens on a 64-square board is $64^8$ permutations (281 Trillion).
By using Backtracking, if we place Queen 1 at `(0,0)`, and Queen 2 at `(1,0)`, they are attacking each other! We instantly `return;` (prune the tree). We never bother trying to place Queens 3-8, saving Trillions of useless computations!

```java
public void solveNQueens(int n) {
    char[][] board = new char[n][n];
    for (int i = 0; i < n; i++) Arrays.fill(board[i], '.');
    
    backtrackQueens(board, 0, n);
}
```

## REAL_WORLD
**Regular Expressions (Regex Engine)**
When you run a Regex like `/a*b*c/` against a massive string in JavaScript, how does the engine work?
It uses Backtracking! The `*` quantifier means "match 0 or more". The engine is greedy; it matches as many `a`s as possible. If the string suddenly fails to match the `b`, the engine *backtracks*, un-matches one `a`, and tries to proceed again!
If a Regex is poorly written (Catastrophic Backtracking), the engine gets trapped in millions of recursive backtrack loops, causing the browser tab to freeze completely and trigger a Denial of Service (ReDoS) vulnerability!

## INTERVIEW
**Q: In your Permutations code, you added the result like this: `results.add(new ArrayList<>(path));`. The interviewer asks: "Why did you create a `new ArrayList`? Why couldn't you just do `results.add(path)`?"**
A: **Object Reference Mutation.**
In Java, `path` is an Object Reference pointing to a physical location in memory.
When you write `results.add(path)`, you are not copying the numbers; you are putting a *pointer* to that single `path` array inside the `results` list.
The recursion finishes and executes the "Un-choose" phase: `path.remove()`. Because the `results` list is holding a pointer to that exact same array, the `remove()` operation deletes the numbers from inside your final results!
By the time the algorithm finishes, your `results` list will contain 6 identical pointers, all pointing to a single, completely empty `[]` array.
You MUST create a deep copy `new ArrayList<>(path)` to take a "snapshot" of the numbers at that exact moment in time, freezing them safely inside the results list!
