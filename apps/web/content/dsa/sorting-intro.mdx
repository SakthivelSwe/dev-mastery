## WHY
Imagine looking for the word "Zebra" in a dictionary where the pages are completely randomized. You would have to flip through every single page one by one (`O(n)`). Because a real dictionary is sorted alphabetically, you instantly open to the back, cutting the search space in half repeatedly until you find it (`O(log n)`). 

Data structuring is fundamentally about organizing data so it can be retrieved efficiently. Sorting is the most critical organizational algorithm in computer science. Without sorting, databases couldn't use indexes, search engines couldn't rank results, and 3D graphics engines couldn't determine which objects render in front of others.

## THEORY
### O(N²) vs O(N log N)
There are two main categories of sorting algorithms you must know:
1. **The Naive Sorts (Bubble, Selection, Insertion)**: These are intuitive but brutally slow. They rely on nested loops, comparing every element to almost every other element. Time complexity is `O(n^2)`. If you sort 100,000 items, they perform 10 Billion operations and freeze the computer.
2. **The Divide & Conquer Sorts (Merge, Quick)**: These are complex but mathematically beautiful. They chop the array into tiny pieces, sort the pieces, and stitch them back together. Time complexity is `O(n log n)`. Sorting 100,000 items takes only 1.6 Million operations (instantly fast).

## CODE

### Level 1 — Beginner (Bubble Sort)
```java
public class BubbleSort {
    // The simplest and worst algorithm. 
    // It "bubbles" the largest element to the end by repeatedly swapping adjacent items.
    // Time: O(N^2) | Space: O(1)
    
    public static void sort(int[] arr) {
        int n = arr.length;
        
        // Outer loop: how many times we traverse
        for (int i = 0; i < n - 1; i++) {
            
            // Inner loop: compare adjacent elements
            // Notice: 'n - i - 1' because the last 'i' elements are already sorted!
            for (int j = 0; j < n - i - 1; j++) {
                
                // If the left element is bigger, swap them!
                if (arr[j] > arr[j + 1]) {
                    int temp = arr[j];
                    arr[j] = arr[j + 1];
                    arr[j + 1] = temp;
                }
            }
        }
    }
}
```

### Level 2 — Building Blocks (Selection Sort)
```java
public class SelectionSort {
    // Searches the unsorted portion of the array to select the absolute smallest element,
    // and places it at the very beginning. 
    // Time: O(N^2) | Space: O(1)
    
    public static void sort(int[] arr) {
        int n = arr.length;
        
        for (int i = 0; i < n - 1; i++) {
            // Assume the current index is the minimum
            int minIndex = i;
            
            // Scan the rest of the array to see if there's anything smaller
            for (int j = i + 1; j < n; j++) {
                if (arr[j] < arr[minIndex]) {
                    minIndex = j; // We found a new minimum!
                }
            }
            
            // Swap the smallest element we found into the 'i' position
            int temp = arr[minIndex];
            arr[minIndex] = arr[i];
            arr[i] = temp;
        }
    }
}
```

### Level 3 — Intermediate (Insertion Sort)
```java
public class InsertionSort {
    // Acts exactly like a person sorting playing cards in their hand.
    // Time: O(N^2) worst case. BUT! O(N) best case if array is almost sorted!
    
    public static void sort(int[] arr) {
        // Start from index 1 (assume index 0 is already a "sorted" sub-array of 1 card)
        for (int i = 1; i < arr.length; i++) {
            int currentCard = arr[i];
            int j = i - 1;
            
            // Look left. As long as the cards on the left are bigger, 
            // shift them one slot to the right to make room!
            while (j >= 0 && arr[j] > currentCard) {
                arr[j + 1] = arr[j];
                j--;
            }
            
            // Drop our current card into the empty slot we created
            arr[j + 1] = currentCard;
        }
    }
}
```

### Level 4 — Advanced (Merge Sort Concept)
```java
// Note: The full recursive implementation of Merge Sort is covered in the Recursion topic.
// Here is the foundational mental model:

// 1. DIVIDE: You have an array of 8 elements.
//    Split it into two arrays of 4. Split those into arrays of 2. Split into arrays of 1.
//    (An array of 1 element is mathematically sorted).

// 2. CONQUER: Take two sorted arrays of 1 element, and "merge" them into a sorted array of 2.
//    Take two sorted arrays of 2, merge them into a sorted array of 4.

// The Merge Step (Merging two already-sorted arrays into one):
public class MergeLogic {
    public static int[] mergeTwoSortedArrays(int[] arr1, int[] arr2) {
        int[] result = new int[arr1.length + arr2.length];
        int i = 0, j = 0, k = 0;
        
        // Compare the fronts of both arrays. Take the smaller one!
        while (i < arr1.length && j < arr2.length) {
            if (arr1[i] <= arr2[j]) {
                result[k++] = arr1[i++];
            } else {
                result[k++] = arr2[j++];
            }
        }
        
        // If one array ran out first, just copy the rest of the other array
        while (i < arr1.length) result[k++] = arr1[i++];
        while (j < arr2.length) result[k++] = arr2[j++];
        
        return result;
    }
}
```

## REAL_WORLD
In the real world, you will never write Bubble Sort. You will simply call `Arrays.sort(arr)` or `Collections.sort(list)`. 

What algorithm does Java actually use under the hood?
For primitive arrays (`int[]`, `double[]`), Java uses **Dual-Pivot Quicksort**. It is blindingly fast but "Unstable" (it might flip the order of identical elements).
For Object arrays (`String[]`, `User[]`), Java uses **TimSort** (a hybrid of Merge Sort and Insertion Sort). TimSort is "Stable", meaning if you have two users with the exact same age, they will remain in the exact same original order. TimSort also capitalizes on the fact that real-world data is often partially sorted, achieving `O(n)` performance on partially sorted arrays!

## INTERVIEW
**Q: Why would anyone use Insertion Sort if Quicksort is faster?**
A: Insertion Sort has virtually no overhead. For extremely tiny arrays (less than 10-20 elements), the mathematical overhead of recursion in Quicksort makes it slower than simple nested loops! Modern production algorithms (like TimSort and IntroSort) actually use recursion to chop the array down to tiny chunks, and then dynamically switch to Insertion Sort to finish the job locally.

**Q: What does it mean for a sorting algorithm to be "Stable"?**
A: Stability guarantees that two elements with identical values will appear in the final sorted output in the exact same relative order they appeared in the unsorted input. This is critical in applications like Excel; if you sort a table by "Last Name", and then sort by "First Name", a stable sort ensures that the Last Names remain sorted within the groups of identical First Names. (Bubble, Insertion, and Merge sort are stable. Selection and Quick sort are unstable).

**Q: Are there any sorting algorithms faster than `O(N log N)`?**
A: Yes, but only under extremely strict mathematical conditions. If you are sorting an array of 1 million integers, and you are absolutely guaranteed that every single integer is between 0 and 100 (e.g., student test scores), you can use **Counting Sort**. Counting Sort bypasses comparisons entirely, using the values as array indices to sort the data in strictly `O(N)` linear time.
