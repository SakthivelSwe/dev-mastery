## WHY
As we learned in Dijkstra's Algorithm, Dijkstra is blazing fast ($O(E \log V)$), but it completely breaks if the graph has **Negative Edge Weights**. 
Why would a graph have negative weights? 
Imagine a financial system tracking currency exchange rates. You trade USD to EUR (cost), EUR to JPY (cost), but if you trade JPY back to USD at the perfect time, you make a profit (a negative cost!).
The **Bellman-Ford Algorithm** is the definitive algorithm for finding the shortest path in graphs with negative weights. More importantly, it is the *only* algorithm that can detect if a **Negative Weight Cycle** exists (an infinite money glitch!).

## THEORY
### Edge Relaxation
Instead of using a clever Priority Queue to guess which node to visit next, Bellman-Ford takes the brute-force approach.
It uses an operation called **Relaxation**.
To relax an edge `A -> B` with weight `W`:
```text
If DistanceTo(A) + W < DistanceTo(B) 
Then DistanceTo(B) = DistanceTo(A) + W
```
Bellman-Ford simply relaxes *every single edge in the entire graph*. 
And it doesn't just do it once. It repeats the relaxation of ALL edges exactly $V - 1$ times (where $V$ is the number of vertices).

### Why V - 1 times?
The absolute longest possible simple path (no cycles) in a graph with $V$ vertices contains exactly $V - 1$ edges. By relaxing every edge $V - 1$ times, the algorithm guarantees that the optimal distances have enough time to "propagate" from the start node all the way across the longest possible chain to the furthest node.

## CODE

### Level 1 — The Bellman-Ford Algorithm
Because we are just blindly iterating over every edge, an Adjacency List is overkill. We can just use a flat list of `Edge` objects!

```java
import java.util.*;

class Edge {
    int source, dest, weight;
    public Edge(int source, int dest, int weight) {
        this.source = source;
        this.dest = dest;
        this.weight = weight;
    }
}

public class BellmanFord {

    public int[] findShortestPaths(int numNodes, List<Edge> allEdges, int startNode) {
        
        // 1. Initialize distances to Infinity
        int[] distances = new int[numNodes];
        Arrays.fill(distances, Integer.MAX_VALUE);
        distances[startNode] = 0; // Start node is 0

        // 2. Relax all edges EXACTLY (V - 1) times!
        for (int i = 1; i <= numNodes - 1; i++) {
            boolean anyChanges = false;

            for (Edge edge : allEdges) {
                int u = edge.source;
                int v = edge.dest;
                int weight = edge.weight;

                // If 'u' is reachable, and the path through 'u' is cheaper...
                if (distances[u] != Integer.MAX_VALUE && distances[u] + weight < distances[v]) {
                    distances[v] = distances[u] + weight; // RELAX!
                    anyChanges = true;
                }
            }

            // OPTIMIZATION: If we looped through all edges and NOTHING changed, 
            // the algorithm has converged! We can stop early!
            if (!anyChanges) break; 
        }

        return distances;
    }
}
```
**Time Complexity:** We loop $V$ times, and inside the loop, we iterate over $E$ edges. Time Complexity is exactly **$O(V \times E)$**. This is significantly slower than Dijkstra, which is why Bellman-Ford is only used when negative weights exist.

### Level 2 — Detecting Negative Cycles (The Infinite Money Glitch)
Imagine a cycle in the graph: `A -> B` (cost 5), `B -> C` (cost 5), `C -> A` (cost -20). 
The total cost of the cycle is `5 + 5 - 20 = -10`. 
Every time you travel around this loop, your total path cost decreases by 10! You could loop infinitely and reach a distance of Negative Infinity. If a graph contains a Negative Weight Cycle reachable from the start node, the concept of a "Shortest Path" mathematically ceases to exist.
**How do we detect it?**
We know the longest valid path is $V - 1$ edges. So we run the Relaxation loop one *extra* time (the $V$th time).
If any distance STILL updates on the $V$th loop, it mathematically proves that an infinite negative cycle exists!

```java
public boolean hasNegativeCycle(int numNodes, List<Edge> allEdges, int[] distances) {
    // Run one final relaxation pass!
    for (Edge edge : allEdges) {
        int u = edge.source;
        int v = edge.dest;
        int weight = edge.weight;

        if (distances[u] != Integer.MAX_VALUE && distances[u] + weight < distances[v]) {
            // A distance updated on the Vth pass! A cycle exists!
            System.out.println("CRITICAL ERROR: Negative Weight Cycle Detected!");
            return true; 
        }
    }
    return false;
}
```

## REAL_WORLD
**Currency Arbitrage Trading**
High-frequency trading firms use Bellman-Ford to find "Arbitrage" opportunities in the foreign exchange (Forex) market.
They map global currencies as Vertices (USD, EUR, JPY). They map the exchange rates as Edges.
By applying a logarithmic transformation to the exchange rates and negating them, they turn the math of "multiplying exchange rates to find profit" into "adding negative edge weights". 
They run Bellman-Ford on the live currency graph. If the `hasNegativeCycle` function returns `true`, it means there is a circular sequence of trades that results in guaranteed free money! The algorithms instantly execute millions of dollars in trades before the market corrects itself.

## INTERVIEW
**Q: Can Dijkstra's Algorithm detect Negative Weight Cycles?**
A: **No.**
Dijkstra's Algorithm uses a `visited` array (or `boolean[] locked`) to permanently lock a node once it is popped from the Priority Queue. It physically forbids the algorithm from ever revisiting or updating a locked node.
Because of this strict locking mechanism, Dijkstra will never traverse a cycle infinitely. It will just blindly lock the nodes, completely ignore the mathematical paradox occurring behind it, and return a fundamentally incorrect array of shortest distances. 
Only Bellman-Ford (or Floyd-Warshall) can accurately detect and report negative cycles!
