## WHY
In the real world, data is rarely a single point in time. It exists as ranges.
A doctor's appointment is from 1:00 PM to 2:30 PM `[1300, 1430]`. A user's active session is from timestamp A to timestamp B. A camera records motion from second 15 to second 45.
When you aggregate thousands of these ranges, you must answer complex questions: *"Do any of these appointments overlap?"* *"What is the total continuous time the camera recorded motion?"* *"How many meeting rooms do I need to accommodate all these meetings?"*
**Interval Problems** are a specific category of arrays where the elements are pairs of start and end times `[[start, end]]`. They require a specific combination of Sorting and Greedy logic to solve efficiently.

## THEORY
### The Golden Rule of Intervals
You cannot solve interval problems if the data is chaotic.
**You must almost always sort the intervals by their START time first!**
Once the intervals are sorted chronologically by their start time, an incredibly powerful mathematical property emerges: **Interval $B$ can ONLY overlap with Interval $A$ if Interval $B$'s start time is LESS THAN Interval $A$'s end time.**
Because they are sorted, you only ever need to compare the current interval with the previous interval, allowing you to traverse the array in a single $O(N)$ pass!

## CODE

### Pattern 1 — Merge Intervals
*Problem:* Given an array of intervals where `intervals[i] = [starti, endi]`, merge all overlapping intervals and return an array of the non-overlapping intervals that cover all the input.
*Example:* `[[1,3], [2,6], [8,10], [15,18]]` becomes `[[1,6], [8,10], [15,18]]`.

```java
import java.util.*;

public int[][] merge(int[][] intervals) {
    if (intervals.length <= 1) return intervals;

    // 1. Sort the intervals by their START time!
    Arrays.sort(intervals, (a, b) -> Integer.compare(a[0], b[0]));

    List<int[]> merged = new ArrayList<>();
    
    // We grab the very first interval to start our comparison block
    int[] currentInterval = intervals[0];
    merged.add(currentInterval); // Add the reference to the list!

    // 2. Iterate through the rest of the sorted intervals
    for (int[] nextInterval : intervals) {
        int currentEnd = currentInterval[1];
        int nextStart = nextInterval[0];
        int nextEnd = nextInterval[1];

        // Do they overlap? (Does the next one start before the current one ends?)
        if (nextStart <= currentEnd) {
            // MERGE! The start time remains the same.
            // We greedily expand the end time to the maximum of both!
            // Example: [1, 5] and [2, 3]. We don't shrink to 3! Max is 5!
            currentInterval[1] = Math.max(currentEnd, nextEnd);
        } else {
            // No overlap! The current block is finished. 
            // We add the new interval to the list and reset our pointer.
            currentInterval = nextInterval;
            merged.add(currentInterval);
        }
    }

    // Convert the ArrayList back into a standard 2D Array for the answer
    return merged.toArray(new int[merged.size()][]);
}
```

### Pattern 2 — Insert Interval
*Problem:* You have a perfectly merged and sorted array of intervals `[[1,3], [6,9]]`. You must insert a brand new interval `[2,5]` into the array and merge any overlapping intervals.
*Why is this different?* Because the array is already sorted, we don't need to waste $O(N \log N)$ time sorting! We can just walk through the array in 3 distinct phases: Left, Overlap, Right.

```java
public int[][] insert(int[][] intervals, int[] newInterval) {
    List<int[]> result = new ArrayList<>();
    int i = 0;
    int n = intervals.length;

    // Phase 1: Add all intervals that perfectly finish BEFORE the new interval starts.
    while (i < n && intervals[i][1] < newInterval[0]) {
        result.add(intervals[i]);
        i++;
    }

    // Phase 2: The Overlap Zone!
    // While the existing interval starts BEFORE our new interval ends...
    while (i < n && intervals[i][0] <= newInterval[1]) {
        // Merge them into the 'newInterval' variable!
        newInterval[0] = Math.min(newInterval[0], intervals[i][0]);
        newInterval[1] = Math.max(newInterval[1], intervals[i][1]);
        i++;
    }
    // We finally finished merging the giant blob. Add it!
    result.add(newInterval);

    // Phase 3: Add all remaining intervals that start AFTER our massive merged blob.
    while (i < n) {
        result.add(intervals[i]);
        i++;
    }

    return result.toArray(new int[result.size()][]);
}
```

### Pattern 3 — Meeting Rooms II (The Min-Heap Pattern)
*Problem:* Given an array of meeting time intervals, return the minimum number of conference rooms required.
*Example:* `[[0, 30], [5, 10], [15, 20]]`.
You need 2 rooms. (Room 1 hosts the 0-30 meeting. Room 2 hosts the 5-10 and 15-20 meetings).

*The Logic:* We sort by start time. But we can't just merge intervals. We need to know: *"When I start a new meeting, has any previous meeting finished so I can steal its room?"*
We use a **Priority Queue (Min-Heap)** to track the END TIMES of the meetings currently occupying rooms. The Min-Heap guarantees the meeting ending the earliest is always at the top!

```java
import java.util.*;

public int minMeetingRooms(int[][] intervals) {
    if (intervals == null || intervals.length == 0) return 0;

    // 1. Sort the meetings by START time
    Arrays.sort(intervals, (a, b) -> Integer.compare(a[0], b[0]));

    // 2. Create a Min-Heap to track the END times of active meetings
    PriorityQueue<Integer> activeRooms = new PriorityQueue<>();

    // Throw the first meeting's end time into the heap (Takes 1 room)
    activeRooms.add(intervals[0][1]);

    for (int i = 1; i < intervals.length; i++) {
        int nextStart = intervals[i][0];
        int nextEnd = intervals[i][1];

        // Look at the top of the heap. Has the earliest meeting finished?
        if (nextStart >= activeRooms.peek()) {
            // YES! The room is empty. We steal it by popping the old end time!
            activeRooms.poll();
        }

        // NO? We couldn't steal a room. We just add our end time to the heap.
        // This physically represents opening a brand new conference room!
        activeRooms.add(nextEnd);
    }

    // The size of the heap is exactly the total number of simultaneous rooms needed!
    return activeRooms.size();
}
```

## REAL_WORLD
**Database Compaction (LSM Trees)**
NoSQL databases like Cassandra and LevelDB use a data structure called Log-Structured Merge (LSM) Trees.
When you write data, it writes sequentially to disk files called SSTables. Because data is appended instantly, if a user updates their profile 5 times, there are 5 different intervals of timestamped data spread across the files.
Over time, the database runs a background thread called "Compaction". Compaction physically reads all the files and runs the **Merge Intervals** algorithm! It finds the overlapping timestamp bounds for the same database key, deletes the old overlapping historical intervals, merges them into a single definitive truth, and writes it back to disk, drastically speeding up future read times and saving massive amounts of storage!

## INTERVIEW
**Q: You implemented `Arrays.sort(intervals, (a, b) -> a[0] - b[0])` to sort the intervals by start time. The interviewer tells you this code has a critical bug that will fail edge cases. What is the bug?**
A: **Integer Underflow during Subtraction.**
The lambda `a[0] - b[0]` is a classic Java shortcut for comparators.
However, if `a[0]` is a massive positive integer (e.g., `2 Billion`), and `b[0]` is a massive negative integer (e.g., `-2 Billion`), the math becomes `2000000000 - (-2000000000)`.
This evaluates to `4 Billion`. Because the maximum limit of a 32-bit integer is `2.14 Billion`, the math silently overflows the binary limit and wraps around to a negative number (`-294967296`).
The Java `Arrays.sort()` function sees the negative number and falsely believes that `a[0]` is SMALLER than `b[0]`, sorting the massive array completely backwards!
*The Fix:* NEVER use subtraction for Comparators unless you mathematically guarantee the domain is strictly small numbers. ALWAYS use `Integer.compare(a[0], b[0])`, which uses safe conditional logic (`<`, `>`, `==`) internally and mathematically cannot overflow!
