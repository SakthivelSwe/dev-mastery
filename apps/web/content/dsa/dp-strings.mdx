## WHY
String manipulation is the bread and butter of software engineering.
If a user misspells "receive" as "recieve", how does Microsoft Word instantly know to suggest "receive"?
If you type a prompt into Google, how does it know your prompt is 99% similar to another previously searched prompt?
**Dynamic Programming on Strings** allows us to measure the mathematical "distance" between two words, calculate how many edits it takes to transform one word into another, or find hidden identical sub-patterns within massive blocks of text.

## THEORY
### The Edit Distance Concept
When comparing two strings, `String A` and `String B`, DP algorithms generally look at the last character of the current prefixes.
If `A[i] == B[j]`, great! No edit is required. The problem shrinks to `A[i-1]` and `B[j-1]`.
If `A[i] != B[j]`, we have 3 choices to fix it:
1. **Insert** a character into A.
2. **Delete** a character from A.
3. **Replace** a character in A.

By trying all 3 choices and taking the minimum cost, DP perfectly maps the cheapest transformation path!

## CODE

### Pattern 1 — Edit Distance (Levenshtein Distance)
*Problem:* Given two strings `word1` and `word2`, return the minimum number of operations required to convert `word1` into `word2`.
This is the legendary algorithm that powers Spell Checkers!

**1. State:** `dp[r][c]` = Minimum edits to convert the first `r` characters of `word1` into the first `c` characters of `word2`.
**2. Base Cases:** Converting a string to an *empty* string requires deleting all its characters.
**3. Recurrence Relation:** If characters match, grab the diagonal. If they don't, take the Minimum of (Left, Top, Diagonal) + 1!

```java
public int minDistance(String word1, String word2) {
    int m = word1.length();
    int n = word2.length();
    
    int[][] dp = new int[m + 1][n + 1];
    
    // Base Case 1: Convert word1 to an empty word2 (Requires i deletions)
    for (int i = 0; i <= m; i++) dp[i][0] = i;
    
    // Base Case 2: Convert empty word1 to word2 (Requires j insertions)
    for (int j = 0; j <= n; j++) dp[0][j] = j;
    
    // Fill the matrix
    for (int r = 1; r <= m; r++) {
        for (int c = 1; c <= n; c++) {
            
            if (word1.charAt(r - 1) == word2.charAt(c - 1)) {
                // Characters match! Cost is 0. Grab the diagonal.
                dp[r][c] = dp[r - 1][c - 1];
            } else {
                // Characters don't match. Find the cheapest edit!
                int insertCost = dp[r][c - 1];    // Left
                int deleteCost = dp[r - 1][c];    // Top
                int replaceCost = dp[r - 1][c - 1]; // Diagonal
                
                // Add 1 to the cheapest option!
                dp[r][c] = 1 + Math.min(insertCost, Math.min(deleteCost, replaceCost));
            }
        }
    }
    
    return dp[m][n];
}
```

### Pattern 2 — Palindromic Substrings
*Problem:* Given a string `s`, return the longest palindromic substring in `s`. (A palindrome reads the same forwards and backwards, like "racecar").
There are two ways to solve this. Expand-Around-Center is faster ($O(N^2)$ space $O(1)$), but the DP approach teaches a critical string pattern:

**1. State:** `dp[i][j]` = `true` if the substring from index `i` to `j` is a palindrome.
**2. Recurrence Relation:** The substring `s[i...j]` is a palindrome IF the outer characters match (`s[i] == s[j]`) AND the inner string `s[i+1...j-1]` is ALSO a palindrome!
**3. Traversal Trick:** Because calculating `(i, j)` requires knowing the inner string `(i+1, j-1)` FIRST, we cannot do a standard Top-to-Bottom loop! We must loop by **substring length**, starting at length 1 and growing!

```java
public String longestPalindromeDP(String s) {
    int n = s.length();
    boolean[][] dp = new boolean[n][n];
    
    int start = 0;
    int maxLength = 1;

    // Base Case: All single letters are palindromes
    for (int i = 0; i < n; i++) {
        dp[i][i] = true;
    }

    // Iterate by Length of the substring (len = 2, then 3, then 4...)
    for (int len = 2; len <= n; len++) {
        // Iterate over starting indices
        for (int i = 0; i <= n - len; i++) {
            int j = i + len - 1; // Calculate ending index

            if (s.charAt(i) == s.charAt(j)) {
                // If length is 2 ("aa"), it's a palindrome!
                // If length > 2, check if the INNER string is a palindrome!
                if (len == 2 || dp[i + 1][j - 1] == true) {
                    dp[i][j] = true;
                    
                    // Keep track of the longest one we found!
                    if (len > maxLength) {
                        maxLength = len;
                        start = i;
                    }
                }
            }
        }
    }
    return s.substring(start, start + maxLength);
}
```

## REAL_WORLD
**Fuzzy Search (ElasticSearch)**
When you search for a product on Amazon and misspell it, how does it still find the right product?
Databases like ElasticSearch use the **Levenshtein Edit Distance** algorithm under the hood for "Fuzzy Matching".
You can configure a query to allow a "Fuzziness of 2", meaning ElasticSearch will use DP to calculate the Edit Distance between your search query and every product title in the database. If the DP matrix returns `1` or `2`, the database accepts the match despite the typo and returns it to the user!

## INTERVIEW
**Q: You are given a massive string of 100,000 characters and asked to find the Longest Palindromic Substring. Your DP Matrix implementation fails with a `Memory Limit Exceeded` error on LeetCode. Why, and how do you fix it?**
A: **The Matrix overhead is fatal.**
A boolean matrix `boolean[100000][100000]` contains 10 Billion cells. In Java, this will consume tens of gigabytes of RAM, instantly crashing the environment.
*The Fix:* While DP is a great educational tool for palindromes, it is terrible for production palindromes. You must abandon DP and use the **Expand-Around-Center** technique. 
You loop through the string, and for every character, you treat it as the center of a palindrome. You use two pointers (`left` and `right`) and expand outwards as long as the characters match! This completely bypasses the need for a 2D Matrix, reducing Space Complexity to a pure $O(1)$ while maintaining the exact same $O(N^2)$ Time Complexity! (For absolute mathematical overkill, Manacher's Algorithm can do it in $O(N)$ time).
