## WHY
Arrays and Trees are hierarchical (Parent -> Child). But the real world is chaotic.
How do you represent a map of roads where any city can connect to any other city? How do you represent a Social Network where Alice is friends with Bob, Bob is friends with Charlie, and Charlie is friends with Alice?
You need a **Graph**. Graphs are the foundation of GPS navigation, social networks, the internet (routers connecting to routers), and Recommendation Engines (users connecting to products). Before you can run complex algorithms like Dijkstra's to find the shortest path, you must know how to physically code the Graph into memory!

## THEORY
### Vertices and Edges
A Graph consists of two things:
1. **Vertices (Nodes):** The entities (Cities, Users, Computers). Usually denoted as $V$.
2. **Edges:** The connections between the entities (Roads, Friendships). Usually denoted as $E$.

### Types of Graphs
- **Undirected Graph:** The connection goes both ways. (If Alice is friends with Bob, Bob is friends with Alice).
- **Directed Graph (Digraph):** One-way connections. (Alice follows Taylor Swift on Twitter, but Taylor Swift does not follow Alice).
- **Weighted Graph:** The edges have a "cost" or "distance" (The road from NY to LA is 2,800 miles long).

## CODE

### Level 1 — The Adjacency Matrix
An Adjacency Matrix uses a massive 2D Array ($V \times V$) to represent the graph. If there is a connection between Node 0 and Node 2, we set `matrix[0][2] = 1`.

```java
public class AdjacencyMatrix {
    int[][] matrix;
    int numVertices;

    public AdjacencyMatrix(int numVertices) {
        this.numVertices = numVertices;
        // Automatically initialized to 0 in Java
        matrix = new int[numVertices][numVertices]; 
    }

    public void addEdge(int source, int destination) {
        // Directed Graph: Just set the source to destination
        matrix[source][destination] = 1;

        // If it was UNDIRECTED, we would also set the reverse!
        // matrix[destination][source] = 1;
    }

    public boolean isConnected(int source, int destination) {
        // Checking a connection is blazingly fast: O(1) time!
        return matrix[source][destination] == 1;
    }
}
```
**Pros:** Checking if a connection exists takes $O(1)$ time. Easy to code.
**Cons:** If you have 100,000 users on a social network, you must create a `100000 x 100000` array. That requires 40 Gigabytes of RAM, even if each user only has 2 friends! It completely wastes memory on "empty" space (a Sparse Graph).

### Level 2 — The Adjacency List
This is the industry standard. Instead of a massive matrix, we use an Array of Lists! (Or a `HashMap<Integer, List<Integer>>`).
Node `0` simply holds a list of the exact nodes it connects to.

```java
import java.util.ArrayList;
import java.util.List;

public class AdjacencyList {
    // An Array where each index holds a List of connected nodes!
    List<List<Integer>> adjList;

    public AdjacencyList(int numVertices) {
        adjList = new ArrayList<>(numVertices);
        // Initialize an empty list for every vertex
        for (int i = 0; i < numVertices; i++) {
            adjList.add(new ArrayList<>());
        }
    }

    public void addEdge(int source, int destination) {
        // We just append the destination to the source's list!
        adjList.get(source).add(destination);
        
        // If UNDIRECTED: adjList.get(destination).add(source);
    }
}
```
**Pros:** Highly memory efficient. For 100,000 users with 2 friends each, it only uses a few Megabytes of RAM!
**Cons:** Checking if Node A connects to Node B requires looping through Node A's list, taking $O(\text{Degree})$ time instead of $O(1)$.

### Level 3 — Weighted Graphs
To add distances/weights to an Adjacency List, the List can no longer hold simple Integers. It must hold custom `Edge` objects containing the destination and the weight!

```java
class Edge {
    int destination;
    int weight;

    public Edge(int destination, int weight) {
        this.destination = destination;
        this.weight = weight;
    }
}

public class WeightedGraph {
    List<List<Edge>> adjList;

    public WeightedGraph(int numVertices) {
        adjList = new ArrayList<>();
        for (int i = 0; i < numVertices; i++) {
            adjList.add(new ArrayList<>());
        }
    }

    public void addEdge(int source, int dest, int weight) {
        // We add the Edge object containing the weight!
        adjList.get(source).add(new Edge(dest, weight));
    }
}
```

### Level 4 — String Nodes (HashMaps)
In LeetCode problems, Nodes are beautifully numbered `0` to `N-1`. But in the real world, nodes are UUIDs, Strings, or Objects (e.g., "New York", "London"). You cannot use an Array if the node identifier is a String! You must use a `HashMap`.

```java
import java.util.HashMap;
import java.util.List;
import java.util.ArrayList;

public class StringGraph {
    // Key: City Name (String) -> Value: List of connected Cities (Strings)
    HashMap<String, List<String>> adjList = new HashMap<>();

    public void addVertex(String city) {
        adjList.putIfAbsent(city, new ArrayList<>());
    }

    public void addEdge(String cityA, String cityB) {
        addVertex(cityA);
        addVertex(cityB);
        // Add the connection both ways! (Undirected)
        adjList.get(cityA).add(cityB);
        adjList.get(cityB).add(cityA);
    }
}
```

## REAL_WORLD
**Dense vs Sparse Graphs**
Choosing between a Matrix and a List depends entirely on Graph Density.
- **Sparse Graph:** A social network. A user has 500 friends out of 2 Billion total users. Use an **Adjacency List**. A Matrix would require exabytes of RAM and crash.
- **Dense Graph:** A highly interconnected system where almost every node connects to every other node. Use an **Adjacency Matrix**. It provides blazingly fast $O(1)$ lookups, and because the connections are dense, the memory isn't "wasted" on empty zeroes.

## INTERVIEW
**Q: You implement a social network graph using an Adjacency List. An influencer named 'Alice' decides to delete her account. You execute `adjList.remove("Alice")`. However, when you run analytics, the app crashes with NullPointerExceptions. Why didn't removing Alice's key from the HashMap properly delete her from the graph?**
A: **Dangling Edges in Undirected Graphs.**
In an Adjacency List representing an undirected graph, every connection is written twice. Alice's list contains "Bob", and Bob's list contains "Alice".
When you call `adjList.remove("Alice")`, you successfully deleted Alice's vertex and all her outgoing edges. However, "Alice" still physically exists as a String inside Bob's list, Charlie's list, and Dave's list!
When your analytics algorithm loops through Bob's friends and attempts to look up "Alice" in the primary HashMap, it returns `null` and crashes.
*The Fix:* Before you delete a vertex in an Adjacency List, you MUST iterate through all of that vertex's neighbors, go into their individual lists, and manually delete the incoming edge pointing back to the deleted vertex. This deletion operation takes $O(V + E)$ time, making node deletion incredibly slow in Adjacency Lists compared to Adjacency Matrices!

### 5 Deep Conceptual Questions
**Q1: List vs matrix?** > **A:** List O(V+E) memory, fast neighbors; matrix O(V²) but O(1) edge check.
**Q2: ONE model?** > **A:** "Sparse→list, dense→matrix."
**Q3: Misconception+code.** > **A:** Matrix on sparse wastes memory.
**Q4: Directed?** > **A:** Add one-way edge.
**Q5: Senior one-liner.** > **A:** "Adjacency list O(V+E) is default; matrix only when dense/edge-query heavy."
