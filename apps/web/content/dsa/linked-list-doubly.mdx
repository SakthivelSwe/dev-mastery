## WHY
In a Singly Linked List, you can only travel forward. If you traverse 10,000 nodes to the end of the list and suddenly realize you need to check the previous node, you are trapped! You cannot move backwards. The only way to find the previous node is to restart the entire search from the Head pointer and travel 9,999 steps forward again.

A **Doubly Linked List** solves this fundamental limitation. Every Node is given a second pointer: `prev`. By pointing backwards as well as forwards, you can traverse the list in both directions instantly. This slight increase in memory (storing one extra memory reference per node) opens up a massive world of `O(1)` performance optimizations.

## THEORY
### Bidirectional Traversal
A Doubly Linked List Node contains three fields:
1. `prev` pointer (Points to the previous node)
2. `val` (The actual data)
3. `next` pointer (Points to the next node)

Because we can move in both directions, we usually maintain two pointers at the List level: a `head` pointer and a `tail` pointer. If we want to append an element to the end of the list, we do not need to traverse from the head! We simply grab the `tail` pointer, attach the new node, and update the tail in `O(1)` time.

## CODE

### Level 1 — Beginner (Creating the Nodes)
```java
// The structure of a Doubly Linked List Node
class DoubleNode {
    int val;
    DoubleNode prev; // The magical backward pointer
    DoubleNode next;
    
    DoubleNode(int val) {
        this.val = val;
        this.prev = null;
        this.next = null;
    }
}

public class BasicDoublyList {
    public static void main(String[] args) {
        DoubleNode node1 = new DoubleNode(10);
        DoubleNode node2 = new DoubleNode(20);
        
        // We must wire BOTH pointers to establish the bidirectional link!
        node1.next = node2;
        node2.prev = node1;
    }
}
```

### Level 2 — Building Blocks (O(1) Tail Insertion)
```java
class DoublyLinkedList {
    DoubleNode head;
    DoubleNode tail; // Keep a reference to the very end!
    
    public void append(int val) {
        DoubleNode newNode = new DoubleNode(val);
        
        // Edge Case: The list is currently empty
        if (head == null) {
            head = newNode;
            tail = newNode;
            return;
        }
        
        // We do NOT need to traverse! We use the tail pointer!
        // Time Complexity: O(1) Instant.
        tail.next = newNode;
        newNode.prev = tail;
        
        // The new node is officially the new tail
        tail = newNode; 
    }
    
    public void traverseBackwards() {
        DoubleNode current = tail;
        while (current != null) {
            System.out.print(current.val + " -> ");
            current = current.prev; // Move BACKWARDS!
        }
        System.out.println("null");
    }
}
```

### Level 3 — Intermediate (O(1) Node Deletion)
```java
public class DeleteOptimized {
    // In a Singly Linked List, if I give you the EXACT Node you want to delete,
    // you STILL have to traverse the list from the Head to find the node BEFORE it!
    
    // In a Doubly Linked List, if you have a reference to the Node to delete,
    // you can delete it instantly in O(1) time without any traversal!
    
    public static void deleteNode(DoubleNode nodeToDelete) {
        // Edge Case: Can't delete null
        if (nodeToDelete == null) return;
        
        // We need the neighbors
        DoubleNode leftNeighbor = nodeToDelete.prev;
        DoubleNode rightNeighbor = nodeToDelete.next;
        
        // Rewire the left neighbor to point to the right neighbor (skipping our node)
        if (leftNeighbor != null) {
            leftNeighbor.next = rightNeighbor;
        }
        
        // Rewire the right neighbor to point to the left neighbor (skipping our node)
        if (rightNeighbor != null) {
            rightNeighbor.prev = leftNeighbor;
        }
        
        // Sever the ties of the deleted node completely
        nodeToDelete.prev = null;
        nodeToDelete.next = null;
    }
}
```

### Level 4 — Advanced (LRU Cache Implementation)
```java
import java.util.HashMap;

public class LRUCache {
    // The most famous FAANG interview question in the world.
    // Build a cache that evicts the "Least Recently Used" item when full, 
    // and both GET and PUT must be strictly O(1) time!
    
    // To achieve this, we combine a HashMap (for O(1) GET) 
    // with a Doubly Linked List (for O(1) re-ordering of nodes)!
    
    class CacheNode {
        int key, val;
        CacheNode prev, next;
        CacheNode(int k, int v) { key = k; val = v; }
    }
    
    private final int capacity;
    private final HashMap<Integer, CacheNode> map;
    private final CacheNode head; // Dummy Head
    private final CacheNode tail; // Dummy Tail
    
    public LRUCache(int capacity) {
        this.capacity = capacity;
        this.map = new HashMap<>();
        head = new CacheNode(0, 0);
        tail = new CacheNode(0, 0);
        head.next = tail;
        tail.prev = head;
    }
    
    // When a node is accessed, we remove it from its current position 
    // and move it right next to the Head (Most Recently Used).
    private void moveToFront(CacheNode node) {
        // Remove from current
        node.prev.next = node.next;
        node.next.prev = node.prev;
        
        // Insert right after Dummy Head
        node.next = head.next;
        node.prev = head;
        head.next.prev = node;
        head.next = node;
    }
    
    public int get(int key) {
        if (!map.containsKey(key)) return -1;
        CacheNode node = map.get(key);
        moveToFront(node); // Mark as recently used!
        return node.val;
    }
}
```

## REAL_WORLD
Have you ever clicked the "Back" and "Forward" buttons on your web browser?
The browser history is implemented using a **Doubly Linked List**. 
When you visit a new page, it is appended as the new `Tail`. If you click "Back", the browser simply follows the `prev` pointer of the current Node to load the previous page. Because you might click "Forward" again, it cannot delete the nodes; it must maintain the bidirectional links to allow seamless navigation in both directions.

Furthermore, Java's actual internal `java.util.LinkedList` is implemented as a Doubly Linked List. It maintains both a `first` and `last` pointer under the hood, allowing `addLast()` and `removeLast()` to execute in `O(1)` time.

## INTERVIEW
**Q: What is the main disadvantage of a Doubly Linked List compared to a Singly Linked List?**
A: Memory consumption and complexity. Every single node now requires an extra 8 bytes (on a 64-bit JVM) just to store the `prev` pointer. If you have 10 million nodes, you are wasting 80 Megabytes of RAM. Additionally, every time you insert or delete a node, you must carefully update 4 distinct pointers instead of 2. If you forget to update just one `prev` pointer, the list will quietly break in half.

**Q: In the LRU Cache, why do we need the HashMap? Why not just use the Doubly Linked List?**
A: A Doubly Linked List allows us to reorder nodes and delete the `Tail` (evicting the Least Recently Used item) in `O(1)` time. However, if a user calls `get("userId:5")`, finding that node in a Linked List requires traversing the entire list, which takes `O(N)` time! By storing the memory references of the Nodes inside a `HashMap`, we get `O(1)` instant lookup, and `O(1)` instant reordering. They cover each other's weaknesses perfectly.

**Q: Why does the LRU Cache use "Dummy" Head and Tail nodes?**
A: The Dummy nodes (`head` and `tail`) do not store real data. They act as permanent anchors. By placing them at the absolute boundaries of the list, we guarantee that the real data nodes always exist *between* two other nodes. This completely eliminates Null Pointer Exceptions and complex edge cases (like "What if the node I'm deleting is the only node in the list?"). The code to remove a node becomes incredibly simple and unified.

### Explain Doubly Linked Lists Like I'm 10 Years Old
> A doubly linked list is a train where each car couples to both neighbours. You can walk forward or back, and unhook any car in O(1) because it knows both sides. The cost: two coupling per car (extra memory). This is why LRU caches use it — pop any item instantly.
