## WHY
You already learned the Segment Tree, which can update an element and query a range sum in $O(\log N)$ time.
However, the Segment Tree code is extremely long (100+ lines), highly recursive, and requires an array of size $4N$.
What if you are in a high-pressure coding interview or competitive programming contest, and you only have 5 minutes to write the code?
The **Fenwick Tree** (also known as a Binary Indexed Tree, or BIT) does the exact same thing as a Segment Tree (Point Updates and Prefix Sum Queries in $O(\log N)$ time), but it takes exactly 15 lines of code to write, requires absolutely no recursion, and uses an array of exactly size $N+1$. It is a masterpiece of bitwise magic.

## THEORY
### The Core Concept
A Fenwick Tree relies on the binary representation of indices.
Every integer can be represented as a sum of powers of 2. For example, $13$ is $8 + 4 + 1$ ($2^3 + 2^2 + 2^0$).
A Fenwick Tree stores partial sums of the array. The value at `BIT[i]` stores the sum of the array elements starting from `i` going downwards, for a length equal to the *value of the least significant bit (LSB) of `i`*.
Because of this mathematical structure, getting the sum of the first 13 elements simply requires jumping through the `BIT` array at indices `13`, `12` (which is 13 - 1), and `8` (which is 12 - 4).

## CODE

### Level 1 — The Bitwise Magic (Isolating the LSB)
To navigate a Fenwick Tree, you must be able to isolate the Lowest Set Bit (LSB) of a number.
For example, if $x = 10$ (binary `1010`), the LSB is the `10` part, which has a decimal value of $2$.
In Two's Complement binary math, there is a magical formula to instantly extract the LSB of any integer:

```java
// The most important line of code in a Fenwick Tree
int lsb = i & (-i); 
```
*Why it works:* In Java, `-i` is calculated by inverting all the bits of `i` and adding `1`. If you perform a Bitwise AND (`&`) between `i` and `-i`, every single bit cancels out to `0`, *except* for the Lowest Set Bit!

### Level 2 — Building the BIT Array
A Fenwick Tree uses a 1-indexed array. We leave `BIT[0]` completely empty to make the bitwise math work perfectly.

```java
public class FenwickTree {
    int[] BIT;
    int n;

    // We initialize the array to 0. Size is N + 1.
    public FenwickTree(int size) {
        this.n = size;
        this.BIT = new int[n + 1]; 
    }
}
```

### Level 3 — The Update Operation (Adding a Value)
If we want to add a value `delta` to index `i` (1-indexed), we must update `BIT[i]`, and then update all the other `BIT` nodes that "cover" index `i`.
We do this by *adding* the LSB to the current index until we go out of bounds.

```java
// Add 'delta' to the element at 1-based index 'i'
public void add(int i, int delta) {
    // Keep climbing UP the tree
    while (i <= n) {
        BIT[i] += delta;
        // Move to the next node that is responsible for our current node
        i += i & (-i); // MAGIC!
    }
}
```
*Example:* If we update index 5 (binary `0101`). 
1. We update `BIT[5]`. LSB is 1. `5 + 1 = 6`.
2. We update `BIT[6]`. LSB is 2. `6 + 2 = 8`.
3. We update `BIT[8]`. LSB is 8. `8 + 8 = 16`.
In just 3 jumps, we updated the entire N=16 array! Time complexity: $O(\log N)$.

### Level 4 — The Query Operation (Prefix Sum)
If we want to find the sum of all elements from index `1` to `i`, we sum the value at `BIT[i]`, and then move *downwards* by subtracting the LSB.

```java
// Returns the sum from index 1 to 'i'
public int query(int i) {
    int sum = 0;
    // Keep climbing DOWN the tree
    while (i > 0) {
        sum += BIT[i];
        // Move to the previous block of the sum
        i -= i & (-i); // MAGIC!
    }
    return sum;
}

// To find the sum of a specific range [L, R]
public int queryRange(int L, int R) {
    // Sum up to R, minus the sum up to L-1
    return query(R) - query(L - 1);
}
```

## REAL_WORLD
**Fenwick Tree vs Segment Tree**
If Fenwick Trees are shorter, use less memory, and are generally faster due to lack of recursion, why do Segment Trees exist at all?
1. **Limitations:** A Fenwick Tree requires the operation to be *invertible*. You can easily calculate `RangeSum(L, R)` by doing `PrefixSum(R) - PrefixSum(L-1)` because subtraction is the inverse of addition.
2. **The Breaking Point:** You *cannot* easily use a Fenwick Tree to find the Minimum or Maximum in a range! There is no inverse operation for `Math.max()`. If `Max[0..10] = 50`, and you subtract `Max[0..5] = 20`, you have no mathematical way to know what `Max[6..10]` is! A Segment Tree can handle Min/Max queries effortlessly.

## INTERVIEW
**Q: The input array provided by the interviewer is 0-indexed (e.g., `arr[0], arr[1]`). Why does the Fenwick Tree fundamentally break if you try to run `add(0, 5)` to update the 0th index? How do you fix it?**
A: **The Infinite Loop of Zero.**
The bitwise logic of the Fenwick Tree relies entirely on the LSB: `i += i & (-i)`.
If you pass `0` into the `add` function, the bitwise math becomes `0 & (-0)`. The binary representation of `0` has no set bits! The LSB is `0`.
The while loop executes: `i += 0`, meaning `i` remains `0` forever, crashing the program in an infinite loop!
*The Fix:* The Fenwick Tree math mathematically requires a 1-based index to function. To fix this, simply add `1` to all queries and updates you receive from the user. If the user wants to update the 0th index of their array, you internally call `add(0 + 1, value)`, routing the data to `BIT[1]`.
