## WHY
1D Dynamic Programming operates on a single array or timeline. But what if your data physically exists in two dimensions? 
Imagine a Robot navigating a grid map, trying to find the path that collects the most coins. Or imagine comparing two entirely different strings to find their longest common overlapping characters.
**2D Dynamic Programming** introduces matrices. The logic is identical to 1D DP, but your State requires *two* variables (`row` and `col`), meaning your cache becomes a 2D Array! Mastering 2D DP unlocks the ability to solve the most difficult problems on LeetCode.

## THEORY
### The Grid Mindset
In 2D DP, `dp[r][c]` typically represents the answer for the subproblem ending exactly at Row `r` and Column `c`.
To calculate `dp[r][c]`, you almost always look at the cells immediately surrounding it:
- **Left:** `dp[r][c-1]`
- **Top:** `dp[r-1][c]`
- **Diagonal:** `dp[r-1][c-1]`

By filling the 2D matrix from top-left to bottom-right, you eventually calculate the final answer at the very bottom-right corner!

## CODE

### Pattern 1 — Grid Navigation (Unique Paths)
*Problem:* A robot is located in the top-left corner of an `M x N` grid. It can only move DOWN or RIGHT. How many unique paths are there to reach the bottom-right corner?

**1. State:** `dp[r][c]` = Total unique paths to reach cell `(r, c)`.
**2. Recurrence Relation:** Because the robot only moves Down or Right, to arrive at `(r, c)`, the robot MUST have moved down from the cell above `(r-1, c)`, OR moved right from the cell to the left `(r, c-1)`. 
Therefore: `dp[r][c] = dp[r-1][c] + dp[r][c-1]`.
**3. Base Cases:** The entire first row and first column only have 1 path (just move right infinitely, or move down infinitely).

```java
public int uniquePaths(int m, int n) {
    int[][] dp = new int[m][n];
    
    // Fill the first row with 1s
    for (int c = 0; c < n; c++) dp[0][c] = 1;
    
    // Fill the first col with 1s
    for (int r = 0; r < m; r++) dp[r][0] = 1;
    
    // Fill the rest of the matrix!
    for (int r = 1; r < m; r++) {
        for (int c = 1; c < n; c++) {
            // The paths from Above + The paths from the Left!
            dp[r][c] = dp[r - 1][c] + dp[r][c - 1];
        }
    }
    
    // The final answer is at the bottom-right corner!
    return dp[m - 1][n - 1];
}
```

### Pattern 2 — Two Sequences (Longest Common Subsequence)
*Problem:* Given two strings `text1 = "abcde"` and `text2 = "ace"`, return the length of their longest common subsequence. (Answer: 3, for "ace").
*Why is this 2D?* Because we must compare every prefix of String A against every prefix of String B!

**1. State:** `dp[r][c]` = Length of LCS for `text1` up to index `r-1` and `text2` up to index `c-1`.
**2. Recurrence Relation:** 
   - If the characters match (`text1[r] == text2[c]`), we add 1 to the diagonal! `dp[r][c] = 1 + dp[r-1][c-1]`.
   - If they DON'T match, we take the maximum of either skipping a letter in A, or skipping a letter in B! `dp[r][c] = Math.max(dp[r-1][c], dp[r][c-1])`.
**3. Base Cases:** We add a dummy row 0 and column 0 filled with `0` to represent comparing against an empty string.

```java
public int longestCommonSubsequence(String text1, String text2) {
    int rLen = text1.length();
    int cLen = text2.length();
    
    // +1 for the dummy 0th row/col
    int[][] dp = new int[rLen + 1][cLen + 1]; 
    
    for (int r = 1; r <= rLen; r++) {
        for (int c = 1; c <= cLen; c++) {
            
            if (text1.charAt(r - 1) == text2.charAt(c - 1)) {
                // MATCH! Grab the Diagonal and add 1!
                dp[r][c] = 1 + dp[r - 1][c - 1];
            } else {
                // NO MATCH. Take the max of Top or Left!
                dp[r][c] = Math.max(dp[r - 1][c], dp[r][c - 1]);
            }
            
        }
    }
    
    return dp[rLen][cLen];
}
```

### Level 3 — Matrix State Reduction ($O(N)$ Space)
In the `uniquePaths` problem, look at the inner loop: `dp[r][c] = dp[r-1][c] + dp[r][c-1]`.
When calculating Row 5, we look at data in Row 4 and data in Row 5. Do we ever look at Row 3? No! Do we ever look at Row 0? No!
Therefore, keeping a massive `m x n` matrix in memory is a complete waste of RAM. We can optimize the Space Complexity from $O(M \times N)$ down to $O(N)$ by only keeping TWO rows in memory: a `prevRow` and a `currentRow`!

```java
public int uniquePathsOptimized(int m, int n) {
    int[] prevRow = new int[n];
    
    // Initialize first row to 1s
    for (int c = 0; c < n; c++) prevRow[c] = 1;
    
    for (int r = 1; r < m; r++) {
        int[] currRow = new int[n];
        currRow[0] = 1; // First column is always 1
        
        for (int c = 1; c < n; c++) {
            // currRow[c-1] is the Left cell. prevRow[c] is the Top cell!
            currRow[c] = prevRow[c] + currRow[c - 1];
        }
        
        // The current row becomes the previous row for the next loop!
        prevRow = currRow; 
    }
    
    return prevRow[n - 1];
}
```
*Note: With an even more advanced trick, you can actually reduce it to a SINGLE 1D Array!*

## REAL_WORLD
**DNA Sequencing**
The "Longest Common Subsequence" (LCS) algorithm is not just a LeetCode problem; it is the foundation of modern bioinformatics.
When scientists extract DNA from a fossilized Neanderthal, the DNA sequence is severely degraded (missing bases). They must align it with modern Human DNA to see how closely related they are. 
Human DNA is 3.2 Billion base pairs long. Running a 2D LCS algorithm creates a grid so massive it requires terabytes of RAM. Bioinformatics uses massively distributed computing and highly optimized 1D array state-reductions (like the Needleman-Wunsch algorithm) to calculate exactly how many evolutionary "edits" exist between the two sequences!

## INTERVIEW
**Q: You successfully solved Unique Paths using a 2D DP Matrix. The interviewer asks: "Is it mathematically possible to solve this in $O(1)$ Space and $O(M)$ time using Combinatorics instead of DP?"**
A: **Yes, using Pascals Triangle / Combinatorics!**
The path from Top-Left to Bottom-Right on an `m x n` grid will ALWAYS require exactly `(m - 1)` down moves and `(n - 1)` right moves.
No matter what path the robot takes, the total number of physical movements it must make is `(m - 1) + (n - 1)`.
The problem literally boils down to a high-school statistics question: *"Out of `totalMoves`, how many ways can I choose exactly `downMoves`?"*
This is the **"N Choose K"** combination formula: `C(n, k) = n! / (k! * (n - k)!)`.
By mathematically calculating `C(TotalMoves, DownMoves)`, you bypass the entire matrix generation and arrive at the exact same answer using mathematical factorials in $O(1)$ space!
