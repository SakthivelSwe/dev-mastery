## WHY
Breadth-First Search (BFS) is perfect for finding the shortest path in an *unweighted* graph (e.g., finding the fewest number of flights to get from NY to Tokyo).
But what if the flights have different costs? A 2-flight route might cost $400, while a direct 1-flight route costs $1,500. BFS will always choose the direct flight because it has fewer "edges", completely ignoring the cost!
To find the cheapest, fastest, or shortest path in a **Weighted Graph**, you need the most famous graph algorithm in computer science: **Dijkstra's Algorithm**.

## THEORY
### The Greedy Priority Queue
Dijkstra's Algorithm is a Greedy algorithm. It maintains a table of the shortest known distance to every city.
Instead of a standard Queue (FIFO), Dijkstra uses a **Priority Queue (Min-Heap)**.
At every step, it asks: *"Of all the cities I have discovered but haven't fully explored yet, which one is mathematically the closest to my starting point?"* 
It pulls that closest city from the Min-Heap, locks in its distance permanently, and then explores all roads leading out of that city, updating the distances to neighboring cities if a cheaper path is found.

## CODE

### Level 1 — The Setup
We need an `Edge` class to hold the destination node and the weight. We also need a `NodeDistance` class for our Priority Queue, so the Priority Queue knows how to sort the nodes by their total distance from the start!

```java
import java.util.*;

class Edge {
    int target;
    int weight;
    public Edge(int target, int weight) {
        this.target = target;
        this.weight = weight;
    }
}

class NodeDistance {
    int node;
    int totalDistance; // Distance from START to this node!
    public NodeDistance(int node, int totalDistance) {
        this.node = node;
        this.totalDistance = totalDistance;
    }
}
```

### Level 2 — Dijkstra's Algorithm
Let's find the shortest distance from the `startNode` to all other nodes in the graph!

```java
public class Dijkstra {
    
    public int[] findShortestPaths(int numNodes, List<List<Edge>> adjList, int startNode) {
        // 1. Track the shortest known distance to every node. 
        // Fill it with Infinity initially!
        int[] distances = new int[numNodes];
        Arrays.fill(distances, Integer.MAX_VALUE);
        
        // The distance from the start to itself is 0.
        distances[startNode] = 0;

        // 2. The Min-Heap! Sort by the totalDistance ascending.
        PriorityQueue<NodeDistance> minHeap = new PriorityQueue<>(
            (a, b) -> Integer.compare(a.totalDistance, b.totalDistance)
        );
        minHeap.add(new NodeDistance(startNode, 0));

        // 3. Track which nodes have been permanently locked in
        boolean[] visited = new boolean[numNodes];

        while (!minHeap.isEmpty()) {
            // Greedy: Extract the absolutely closest node!
            NodeDistance current = minHeap.poll();
            int u = current.node;

            // If we already locked this node in, ignore it (Optimization)
            if (visited[u]) continue;
            
            // Lock it in permanently!
            visited[u] = true;

            // 4. Explore all roads leaving this node
            for (Edge edge : adjList.get(u)) {
                int v = edge.target;
                int weight = edge.weight;

                // Is the path through 'u' cheaper than the previous best path to 'v'?
                if (!visited[v] && distances[u] + weight < distances[v]) {
                    // Update the distance table!
                    distances[v] = distances[u] + weight;
                    // Push the new, better distance into the Min-Heap!
                    minHeap.add(new NodeDistance(v, distances[v]));
                }
            }
        }

        return distances; // Returns the shortest path to EVERY node!
    }
}
```

### Level 3 — Time Complexity Breakdown
Dijkstra's Time Complexity is $O(E \log V)$, where $E$ is Edges and $V$ is Vertices.
- We pull a node from the Priority Queue $V$ times. Pulling takes $\log V$. -> $O(V \log V)$
- We iterate over every edge exactly once, and push to the Priority Queue $E$ times. Pushing takes $\log V$. -> $O(E \log V)$
- Combined: $O((V + E) \log V)$. In dense graphs, $E$ dominates, so it simplifies to $O(E \log V)$.

### Level 4 — Network Routing (OSPF)
How does the Internet work? When your laptop requests a website, your ISP router receives the packet. The router must decide which of its 5 physical cables it should forward the packet down to reach the destination server fastest.
The Open Shortest Path First (OSPF) protocol literally runs Dijkstra's algorithm! Every router in the world mathematically calculates the shortest path through the massive web of global routers to guarantee your packet arrives instantly.

## REAL_WORLD
**Why Google Maps doesn't use pure Dijkstra**
Dijkstra is brilliant, but it searches radially in a massive expanding circle. If you ask for directions from NY to LA, Dijkstra will thoroughly explore every road in Boston, Washington DC, and Canada before reaching LA, because it blindly searches the closest things first!
Google Maps uses **A\* (A-Star) Search**. A\* is a modification of Dijkstra that adds a "Heuristic" (a guess). It calculates: `Cost = DistanceFromStart(Dijkstra) + EstimatedDistanceToTarget(Heuristic)`. 
By using the physical GPS latitude/longitude as the Heuristic, A\* heavily penalizes roads going East, and heavily favors roads going West, forcing the algorithm to aggressively laser-focus its search towards LA, skipping millions of irrelevant roads!

## INTERVIEW
**Q: You use Dijkstra's Algorithm on a graph that has negative edge weights (e.g., traveling down a road gives you $5 instead of costing you $5). The algorithm returns completely incorrect distances. Why does Dijkstra mathematically break when negative numbers are introduced?**
A: **The Permanent Lock Assumption.**
Dijkstra is a Greedy algorithm. The absolute core assumption of Dijkstra is: *"Once I pop a node from the Priority Queue and mark it `visited`, I have definitively found the absolute shortest path to that node. I never need to check it again."*
Why is this assumption safe? Because in the real world, adding distance only makes the path longer.
If you introduce negative edges, this assumption is destroyed. You might lock in Node A with a distance of `10`. Later, you travel down a bizarre 500-mile path that suddenly has an edge of `-5000`. You just found a cheaper path to Node A! But Dijkstra marked it `visited` and permanently locked it!
If your graph has negative edges, you cannot use Dijkstra. You MUST use the **Bellman-Ford Algorithm**, which sacrifices speed to safely handle negative weights.
