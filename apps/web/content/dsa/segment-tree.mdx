## WHY
Imagine you have an array of 100,000 numbers. You need to answer thousands of queries like: *"What is the sum of the numbers between index 15,000 and 85,000?"* or *"What is the minimum value between index 2,000 and 5,000?"*
If you simply loop through the array for every query, it takes $O(N)$ time per query. For $M$ queries, that's $O(N \times M)$, which will trigger a Time Limit Exceeded (TLE) error in competitive programming.
If you just precompute a Prefix Sum array, you can answer the queries in $O(1)$! But what if the array is constantly *changing* (e.g., `arr[4200] = 99`)? Updating a Prefix Sum array takes $O(N)$ time!
A **Segment Tree** solves this perfectly. It allows you to both **Update** an array element and **Query** a range in exactly $O(\log N)$ time!

## THEORY
### The Structure
A Segment Tree is a Binary Tree where:
- The **root node** represents the entire array (from index `0` to `N-1`).
- Each **internal node** represents the union of its two children. If the root represents `[0, 3]`, the left child represents `[0, 1]` and the right child represents `[2, 3]`.
- The **leaf nodes** represent the individual elements of the array.

If the Segment Tree is tracking "Sums", then each parent node simply stores the sum of its two children!

## CODE

### Level 1 — Building the Segment Tree
Unlike standard Binary Trees, we don't use pointers (`left`, `right`). Because the Segment Tree is a perfectly balanced tree, we can store it directly in a flat array!
If a parent is at index `i`, its left child is at `2*i + 1`, and its right child is at `2*i + 2`.
The size of the segment tree array needs to be at most `4 * N`.

```java
public class SegmentTree {
    int[] tree;
    int[] arr;
    int n;

    public SegmentTree(int[] input) {
        this.arr = input;
        this.n = input.length;
        this.tree = new int[4 * n];
        build(0, 0, n - 1);
    }

    // node = current index in tree array
    // start, end = the range in the original array this node represents
    private void build(int node, int start, int end) {
        if (start == end) {
            // Leaf node: store the array value
            tree[node] = arr[start];
            return;
        }

        int mid = start + (end - start) / 2;
        int leftChild = 2 * node + 1;
        int rightChild = 2 * node + 2;

        // Recursively build the left and right subtrees
        build(leftChild, start, mid);
        build(rightChild, mid + 1, end);

        // The current node's sum is the sum of its children!
        tree[node] = tree[leftChild] + tree[rightChild];
    }
}
```

### Level 2 — Range Query in $O(\log N)$
To find the sum of a specific range `[L, R]`, we traverse down the tree.
1. **Total Overlap:** If the current node's range is completely inside `[L, R]`, return the node's value immediately!
2. **No Overlap:** If the current node's range is completely outside `[L, R]`, return 0.
3. **Partial Overlap:** If the range partially overlaps, we split the query and search both the left and right children.

```java
// Public API
public int query(int L, int R) {
    return queryRec(0, 0, n - 1, L, R);
}

private int queryRec(int node, int start, int end, int L, int R) {
    // 1. No Overlap
    if (R < start || L > end) {
        return 0; // Returning 0 doesn't affect a Sum query
    }

    // 2. Total Overlap
    if (L <= start && R >= end) {
        return tree[node];
    }

    // 3. Partial Overlap: Split and search both children
    int mid = start + (end - start) / 2;
    int leftSum = queryRec(2 * node + 1, start, mid, L, R);
    int rightSum = queryRec(2 * node + 2, mid + 1, end, L, R);

    return leftSum + rightSum;
}
```

### Level 3 — Point Update in $O(\log N)$
If we change `arr[2] = 5`, we must update the leaf node for index `2`, and then recursively update all of its parents all the way up to the root!

```java
// Public API: Update index 'idx' to 'val'
public void update(int idx, int val) {
    updateRec(0, 0, n - 1, idx, val);
}

private void updateRec(int node, int start, int end, int idx, int val) {
    if (start == end) {
        // We found the exact leaf node!
        arr[idx] = val;      // Update the original array
        tree[node] = val;    // Update the segment tree
        return;
    }

    int mid = start + (end - start) / 2;
    int leftChild = 2 * node + 1;
    int rightChild = 2 * node + 2;

    if (idx <= mid) {
        // The index is in the left half
        updateRec(leftChild, start, mid, idx, val);
    } else {
        // The index is in the right half
        updateRec(rightChild, mid + 1, end, idx, val);
    }

    // After the child updates, recalculate the current node's sum!
    tree[node] = tree[leftChild] + tree[rightChild];
}
```

### Level 4 — Lazy Propagation
Imagine a query that says: *"Add 5 to every single element between index 2,000 and 9,000"*.
If we use our `Point Update` function above 7,000 times, it will take $O(7000 \times \log N)$, which is too slow!
**Lazy Propagation** solves this. If a node representing the range `[2000, 9000]` needs to be updated, we update the node's sum, but we *delay* (lazy) updating its millions of children. We store the "pending" update in a separate `lazy` array. We only push the pending updates down to the children when a future query physically tries to read them! This allows Range Updates to run in $O(\log N)$ time!

## REAL_WORLD
- **Geographic Information Systems (GIS):** Used for rapid 2D range queries (like finding all restaurants within a specific latitude/longitude bounding box).
- **Computational Geometry:** Used to find intersecting line segments or calculate the total area of overlapping rectangles.
- **Stock Market Analysis:** Rapidly finding the minimum or maximum price of a stock over a specific historical time window, while live prices are constantly streaming in.

## INTERVIEW
**Q: You implement a Segment Tree to track the Minimum value in a range instead of the Sum. In the `query` function, for the "No Overlap" case, you return `0`. Your code fails on all LeetCode tests. Why?**
A: **The Identity Element.**
When searching for the Sum of a range, the Identity Element (the value that does not affect the calculation) is `0`, because `A + 0 = A`. Therefore, returning `0` for an invalid path is correct.
However, when searching for the Minimum value, if you return `0` from an invalid path, the math becomes `Math.min(actualValue, 0)`. If your array contains positive numbers, `0` will always win! Your query will incorrectly return `0`.
*The Fix:* For a Minimum Segment Tree, the Identity Element is Infinity. In the "No Overlap" case, you must return `Integer.MAX_VALUE`! (For a Maximum Segment Tree, return `Integer.MIN_VALUE`).
