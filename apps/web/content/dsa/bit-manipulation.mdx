## WHY
At the absolute lowest level of physics, computers do not understand Strings, Objects, or even the number `42`. They only understand transistors turning ON (`1`) and OFF (`0`).
Most modern programming abstracts this away, allowing developers to write Python or Java without ever thinking about Binary. 
However, **Bit Manipulation** bypasses this abstraction. By using Bitwise Operators (AND, OR, XOR, Shifts), you command the CPU to execute math at the raw silicon level. Bit manipulation executes in just a single clock cycle, making it thousands of times faster than standard mathematical operations (`*`, `/`, `%`). It is the secret language of cryptography, compression, and high-performance gaming engines.

## THEORY
### The Core Bitwise Operators
- **AND (`&`):** Results in 1 *only* if BOTH bits are 1. Used as a "Mask" to isolate specific bits.
- **OR (`|`):** Results in 1 if AT LEAST ONE bit is 1. Used to turn specific bits ON.
- **XOR (`^`):** Results in 1 if the bits are DIFFERENT. (Exclusive OR). Used to toggle bits, and famously possesses the mathematical property: `A ^ A = 0` (Any number XOR'd by itself perfectly cancels out to zero).
- **NOT (`~`):** Inverts all bits. (0 becomes 1, 1 becomes 0).
- **Left Shift (`<<`):** Shifts all bits to the left, adding 0s to the right. Mathematically exactly equivalent to multiplying by 2.
- **Right Shift (`>>`):** Shifts all bits to the right. Mathematically exactly equivalent to dividing by 2.

## CODE

### Pattern 1 — The Single Number (The Magic of XOR)
*Problem:* Given an array of integers where every element appears exactly twice except for one. Find that single one. You must implement a solution with a linear runtime complexity and use only constant extra space $O(1)$.
*Example:* `[4, 1, 2, 1, 2]` -> Returns `4`.

If you use a HashSet, it takes $O(N)$ space.
If we use XOR, we remember the golden rules:
1. `A ^ A = 0` (Pairs cancel each other out completely!)
2. `A ^ 0 = A` (XORing with 0 changes nothing!)
3. Order doesn't matter (Commutative).

```java
public int singleNumber(int[] nums) {
    int result = 0;
    for (int num : nums) {
        // As we XOR every number, all pairs will cancel out to 0!
        // The only bits left standing will belong to the single number!
        result ^= num; 
    }
    return result;
}
```

### Pattern 2 — Counting Set Bits (Brian Kernighan’s Algorithm)
*Problem:* Given an integer `n`, return the number of `1` bits it has (also known as the Hamming Weight).
*The Slow Way:* Shift right by 1 and check `n & 1` 32 times.
*The Genius Way:* `n & (n - 1)`. When you subtract 1 from a binary number, all the bits flip starting from the right up to the very first `1`. When you AND them together, it magically deletes the lowest set bit in exactly 1 clock cycle!

```java
public int hammingWeight(int n) {
    int count = 0;
    while (n != 0) {
        count++;
        // Magic: Deletes the right-most '1' bit instantly!
        // If n = 10 (binary 1010). n - 1 = 9 (1001).
        // 1010 & 1001 = 1000! The lowest bit was deleted!
        n &= (n - 1); 
    }
    // Loop only runs as many times as there are '1's!
    return count; 
}
```

### Pattern 3 — Bit Masking (Subsets)
*Problem:* Generate all possible combinations (subsets) of an array `[A, B, C]`.
In the Backtracking module, we used recursion to choose or skip elements. We can do this effortlessly using pure binary counting!
If the array has 3 elements, there are $2^3 = 8$ subsets. If we simply count from 0 to 7 in binary, the 1s and 0s perfectly represent every possible combination!
- `000` = []
- `001` = [C]
- `101` = [A, C]
- `111` = [A, B, C]

```java
import java.util.*;

public List<List<Integer>> subsets(int[] nums) {
    List<List<Integer>> result = new ArrayList<>();
    int n = nums.length;
    // 1 << n is mathematically 2^N!
    int totalSubsets = 1 << n; 

    // Loop from 0 to 7 (for an array of size 3)
    for (int mask = 0; mask < totalSubsets; mask++) {
        List<Integer> subset = new ArrayList<>();
        
        // Loop through the 3 bit positions (0, 1, 2)
        for (int i = 0; i < n; i++) {
            // Is the i-th bit turned ON in our current mask?
            // e.g., mask = 5 (101). Left shift 1 by i to create a probe (001, 010, 100).
            // AND them together to see if it hits!
            if ((mask & (1 << i)) != 0) {
                subset.add(nums[i]);
            }
        }
        result.add(subset);
    }
    return result;
}
```

## REAL_WORLD
**Bit Flags in Operating Systems**
When you write a file to a Linux file system, the file has permissions: `Read`, `Write`, and `Execute`.
Linux does not store three boolean variables in the database. It stores a single integer (a Bitmask)!
- Read = 4 (Binary `100`)
- Write = 2 (Binary `010`)
- Execute = 1 (Binary `001`)

If you want to grant a user Read and Execute permissions, you simply use the Bitwise OR operator: `4 | 1 = 5` (`101`).
If the OS needs to verify if the user has Write permission, it uses the Bitwise AND operator: `(5 & 2)`. Because `101 & 010 == 000`, the OS instantly denies access in 1 clock cycle. This is why `chmod 777` grants all permissions! (`4 + 2 + 1 = 7`).

## INTERVIEW
**Q: You want to multiply a number by 2. A senior engineer tells you to use `x << 1` instead of `x * 2` because it's "faster". Are they correct?**
A: **Technically yes, but practically no (Modern Compilers).**
In the 1980s, the CPU multiplier circuits were incredibly slow. Executing a `MUL` assembly instruction took 10+ clock cycles, whereas a Shift Left `SHL` instruction took 1 clock cycle. In those days, replacing `* 2` with `<< 1` was a critical optimization.
However, modern compilers (like JIT in Java or GCC/Clang in C++) are astonishingly smart. When you type `x * 2` in your source code, the compiler recognizes the constant multiplier and *automatically* converts the compiled machine code into a Bitwise Left Shift! 
Therefore, you should write `x * 2` for human readability, knowing the compiler will perfectly optimize it to raw bitwise operations under the hood. Only use Bitwise shifts manually if you are explicitly dealing with bits, masks, or cryptography!
