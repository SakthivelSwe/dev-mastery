## WHY
You have a massive graph of friends on a social network. You want to find out if there is a connection between You and Kevin Bacon.
If you use Depth-First Search (DFS), you might follow a single chain of friends down a rabbit hole of 5,000 people across the world, completely missing the fact that your direct coworker knows Kevin Bacon!
**Breadth-First Search (BFS)** is an algorithm that searches outward in concentric circles (like ripples in a pond). It checks all your direct friends first (Level 1), then checks all of their friends (Level 2). Because it searches level by level, BFS mathematically guarantees that the first time it finds the target, it is the **Shortest Path** (in an unweighted graph)!

## THEORY
### The Queue Structure
DFS uses a Stack (LIFO) to plunge deep. BFS uses a **Queue (FIFO)** to maintain order.
1. Put the starting node in the Queue.
2. Dequeue a node, and check if it's the target.
3. If not, add all of its unseen neighbors to the *back* of the Queue.
4. Repeat until the Queue is empty.

Because new neighbors are added to the back, the Queue physically enforces that all nodes at Level 1 are processed completely before Level 2 is ever touched!

## CODE

### Level 1 — Basic BFS Traversal
Let's traverse a graph using an Adjacency List. The most critical part of BFS is the `visited` array/set. In Trees, you can only move downwards. In Graphs, connections go everywhere, including loops! If A points to B, and B points to A, without a `visited` tracker, your BFS will bounce between them in an infinite loop!

```java
import java.util.*;

public class GraphBFS {
    // Using a HashMap Adjacency List
    HashMap<Integer, List<Integer>> adjList = new HashMap<>();

    public void traverse(int startNode) {
        // 1. The Queue controls the Level-by-Level flow!
        Queue<Integer> queue = new LinkedList<>();
        // 2. The Visited Set prevents infinite loops!
        HashSet<Integer> visited = new HashSet<>();

        // Start the algorithm
        queue.add(startNode);
        visited.add(startNode);

        while (!queue.isEmpty()) {
            // Take the front node out of the queue
            int current = queue.poll();
            System.out.println("Visited Node: " + current);

            // Get all neighbors of the current node
            List<Integer> neighbors = adjList.getOrDefault(current, new ArrayList<>());

            for (int neighbor : neighbors) {
                // If we haven't seen this neighbor before...
                if (!visited.contains(neighbor)) {
                    visited.add(neighbor); // Mark it seen IMMEDIATELY!
                    queue.add(neighbor);   // Put it at the back of the line!
                }
            }
        }
    }
}
```

### Level 2 — Finding the Shortest Path (Tracking Levels)
To find the actual distance (number of edges) from the start node to the target, we must count the "Levels" (ripples).
To do this, we measure the size of the queue at the start of the while loop! That size tells us exactly how many nodes exist on the *current* level.

```java
public int getShortestDistance(int startNode, int targetNode) {
    Queue<Integer> queue = new LinkedList<>();
    HashSet<Integer> visited = new HashSet<>();

    queue.add(startNode);
    visited.add(startNode);
    
    int distance = 0; // Track the "ripples"

    while (!queue.isEmpty()) {
        // MAGIC: Take a snapshot of how many nodes are on the current level!
        int nodesInCurrentLevel = queue.size();

        // Process EXACTLY that many nodes!
        for (int i = 0; i < nodesInCurrentLevel; i++) {
            int current = queue.poll();

            // Did we find it?
            if (current == targetNode) return distance;

            for (int neighbor : adjList.getOrDefault(current, new ArrayList<>())) {
                if (!visited.contains(neighbor)) {
                    visited.add(neighbor);
                    queue.add(neighbor);
                }
            }
        }
        // Once the inner loop finishes, we have fully processed the current level!
        // Increment the distance for the next level!
        distance++; 
    }

    return -1; // Target not found
}
```

### Level 3 — Reconstructing the Path
Knowing the distance is 3 is great, but what if you need turn-by-turn directions? We need the actual path `A -> C -> D -> Target`.
We use a `parentMap`. Every time we add a neighbor to the queue, we record *who* added them!

```java
public List<Integer> reconstructPath(int start, int target) {
    Queue<Integer> queue = new LinkedList<>();
    HashSet<Integer> visited = new HashSet<>();
    // Key: Node -> Value: The parent node that discovered it!
    HashMap<Integer, Integer> parentMap = new HashMap<>();

    queue.add(start);
    visited.add(start);

    while (!queue.isEmpty()) {
        int current = queue.poll();

        if (current == target) {
            // TARGET FOUND! Build the path backwards using the Parent Map!
            List<Integer> path = new ArrayList<>();
            int step = target;
            
            while (step != start) {
                path.add(step);
                step = parentMap.get(step); // Trace backwards!
            }
            path.add(start);
            Collections.reverse(path); // Reverse it so it goes Start -> Target
            return path;
        }

        for (int neighbor : adjList.get(current)) {
            if (!visited.contains(neighbor)) {
                visited.add(neighbor);
                queue.add(neighbor);
                // Record history! "I reached 'neighbor' via 'current'"
                parentMap.put(neighbor, current); 
            }
        }
    }
    return new ArrayList<>(); // Empty list if no path
}
```

## REAL_WORLD
**Multi-Source BFS**
Standard BFS starts from a single node, like a single drop of water creating ripples.
What if a city has 5 Fire Stations, and you need to find the distance from EVERY house in the city to its *closest* Fire Station?
You use **Multi-Source BFS**! 
Instead of putting a single node in the Queue at the start, you push ALL 5 Fire Stations into the Queue simultaneously, and mark all 5 as visited.
When the BFS runs, the ripples expand outward from all 5 stations simultaneously. Whichever ripple touches a house first is mathematically guaranteed to be the closest Fire Station to that house!

## INTERVIEW
**Q: You write a BFS. Inside your loop, you pop a node, and THEN you mark it as `visited`. Your code causes the Queue to grow to a massive size and throws an OutOfMemoryError on LeetCode. A Senior Engineer tells you to mark nodes as `visited` the exact moment you *add* them to the queue, not when you *pop* them. Why does this order matter so much?**
A: **Queue Duplication.**
If you wait to mark a node as `visited` until you *pop* it from the Queue, the node is physically sitting inside the Queue completely unprotected.
Imagine Node A and Node B both connect to Node C. 
Node A processes its neighbors and pushes C into the Queue. 
Node B processes its neighbors. It checks if C is in the `visited` set. Because C hasn't been popped yet, it is not marked visited! Node B pushes a SECOND copy of Node C into the Queue!
In a highly interconnected dense graph, this causes the same node to be pushed into the Queue hundreds of times, exponentially exploding the Queue size and crashing the server.
*The Fix:* Always mark nodes as `visited` the exact millisecond you push them into the Queue. This guarantees a node can only ever enter the Queue exactly once.
