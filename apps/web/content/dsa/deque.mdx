## WHY
A Stack restricts you to one end (LIFO). A Queue restricts you to opposite ends (FIFO). 
But what if you need a data structure that acts like a Queue, but occasionally allows someone to "cut the line" and jump straight to the front? Or what if you are tracking browser history (a Stack), but you want to limit the history to exactly 50 pages; when the 51st page is pushed onto the top, you need to delete the oldest page from the absolute bottom?

A **Deque** (pronounced "Deck") is a **Double-Ended Queue**. It completely removes the restrictions of Stacks and Queues. It allows you to insert, remove, and peek at BOTH the absolute Front and the absolute Rear of the data structure, with all four operations executing in lightning-fast `O(1)` time. 

## THEORY
### The Ultimate Hybrid
A Deque is the ultimate hybrid data structure. 
- If you only use `addFirst()` and `removeFirst()`, it behaves exactly like a **Stack**.
- If you only use `addLast()` and `removeFirst()`, it behaves exactly like a **Queue**.

Because it is so versatile, Java officially deprecated the legacy `java.util.Stack` class and tells developers to use `Deque` for all Stack-related operations. 

### Implementation Under the Hood
In Java, the `Deque` interface is primarily implemented by two classes:
1. `LinkedList`: Uses a Doubly Linked List under the hood. Manipulating the Head and Tail is naturally `O(1)`, but memory overhead is high.
2. `ArrayDeque`: Uses a Circular Array under the hood. It is significantly faster and more memory-efficient than `LinkedList` because the CPU can pre-fetch the contiguous memory array. You should almost always use `ArrayDeque`.

## CODE

### Level 1 — Beginner (Deque Methods)
```java
import java.util.Deque;
import java.util.ArrayDeque;

public class BasicDeque {
    public static void main(String[] args) {
        Deque<String> deque = new ArrayDeque<>();
        
        // Add to the REAR (Like a normal Queue)
        deque.addLast("Alice");
        deque.addLast("Bob");
        
        // Add to the FRONT (Cutting the line!)
        deque.addFirst("Charlie");
        
        // Current state: [Front] Charlie -> Alice -> Bob [Rear]
        
        // Remove from the FRONT
        System.out.println("Served: " + deque.removeFirst()); // "Charlie"
        
        // Remove from the REAR (Like an Undo operation)
        System.out.println("Left the line: " + deque.removeLast()); // "Bob"
    }
}
```

### Level 2 — Building Blocks (Palindrome Checker)
```java
import java.util.Deque;
import java.util.ArrayDeque;

public class PalindromeDeque {
    // Problem: Check if a word reads the same forwards and backwards.
    // We already solved this with Two-Pointers, but a Deque is conceptually beautiful!
    
    public static boolean isPalindrome(String word) {
        Deque<Character> deque = new ArrayDeque<>();
        
        // Add all characters to the Deque
        for (char c : word.toCharArray()) {
            deque.addLast(c);
        }
        
        // While there is more than 1 character left...
        while (deque.size() > 1) {
            // Pop from the absolute front and absolute back simultaneously!
            char front = deque.removeFirst();
            char back = deque.removeLast();
            
            // If they don't match, it's not a palindrome!
            if (front != back) return false;
        }
        
        return true;
    }
}
```

### Level 3 — Intermediate (Maximum in Sliding Window)
```java
import java.util.Deque;
import java.util.ArrayDeque;

public class SlidingWindowMaximum {
    // This is a famously brutal FAANG interview question.
    // Problem: Given an array and a sliding window of size K, 
    // find the maximum element inside the window as it slides from left to right.
    // Example: arr=[1, 3, -1, -3, 5, 3, 6, 7], k=3
    // We MUST do this in strictly O(N) time!
    
    public static int[] maxSlidingWindow(int[] nums, int k) {
        if (nums == null || k <= 0) return new int[0];
        
        int[] result = new int[nums.length - k + 1];
        int resultIndex = 0;
        
        // The Deque will store INDICES, not the actual values!
        // We will maintain a "Monotonically Decreasing" Deque.
        Deque<Integer> deque = new ArrayDeque<>();
        
        for (int i = 0; i < nums.length; i++) {
            
            // Step 1: Remove indices that are out of the current window (too old)
            // The left side of our window is at index (i - k + 1). 
            if (!deque.isEmpty() && deque.peekFirst() < i - k + 1) {
                deque.pollFirst();
            }
            
            // Step 2: Remove elements from the REAR that are smaller than the current number.
            // Why? Because they are smaller, AND they are older! They can NEVER be the maximum!
            // We crush them instantly to keep the Deque clean.
            while (!deque.isEmpty() && nums[deque.peekLast()] < nums[i]) {
                deque.pollLast();
            }
            
            // Step 3: Add the current index to the REAR of the Deque
            deque.offerLast(i);
            
            // Step 4: The FRONT of the Deque is mathematically guaranteed 
            // to be the maximum value in the current window!
            if (i >= k - 1) {
                result[resultIndex++] = nums[deque.peekFirst()];
            }
        }
        return result;
    }
}
```

### Level 4 — Advanced (Custom ArrayDeque Implementation)
```java
public class CustomArrayDeque {
    // Understanding how ArrayDeque achieves O(1) adds to both ends using a Circular Array
    private int[] data;
    private int head;
    private int tail;
    private int size;

    public CustomArrayDeque(int capacity) {
        data = new int[capacity];
        head = 0;
        tail = 0;
        size = 0;
    }

    public void addFirst(int val) {
        if (size == data.length) throw new RuntimeException("Deque Full");
        
        // Move head BACKWARDS. If it goes below 0, wrap it to the absolute end of the array!
        head = (head - 1 + data.length) % data.length;
        data[head] = val;
        size++;
    }

    public void addLast(int val) {
        if (size == data.length) throw new RuntimeException("Deque Full");
        
        data[tail] = val;
        // Move tail FORWARDS. If it goes past the array limit, wrap it to 0!
        tail = (tail + 1) % data.length;
        size++;
    }
    
    // removeFirst and removeLast involve moving the pointers in the opposite directions.
}
```

## REAL_WORLD
While Deques are rarely explicitly requested by business logic, they are the architectural secret behind **Work-Stealing Thread Pools**.

Java's `ForkJoinPool` (which powers `CompletableFuture`, Parallel Streams, and Virtual Threads) uses an army of worker threads. Every single worker thread is assigned its own private Deque of tasks. 
Normally, a thread acts like a Stack (LIFO): it pushes a task to the `head` of its Deque, and later pops it from the `head` to execute it. This is perfectly synchronized because only the owner thread touches the `head`.

But what happens if Thread A finishes all its tasks and is sitting idle, while Thread B is overwhelmed with 1,000 tasks? 
Thread A becomes a "Thief"! It sneaks over to Thread B's Deque, and secretly steals a task from the `tail` (the opposite end!). Because the owner works at the `head`, and the thief steals from the `tail`, they almost never collide, completely eliminating the need for expensive thread locking. This `Deque` architecture is why modern Java concurrency is so unbelievably fast.

## INTERVIEW
**Q: If `ArrayDeque` uses an array under the hood, what happens if it gets full? Doesn't resizing it ruin the `O(1)` time complexity?**
A: Excellent question. When an `ArrayDeque` hits maximum capacity, it creates a new array that is exactly Double the size, and copies the elements over. This resize operation takes `O(N)` time. However, because it doubles the size every time, resizing happens extremely rarely. In algorithmic terms, we say the insertion time is `O(1) Amortized`. This means if you average the cost out over a million inserts, the rare `O(N)` resize penalty is mathematically diluted to `O(1)`.

**Q: Why do we use a "Monotonically Decreasing Deque" for the Sliding Window Maximum problem?**
A: The word "Monotonic" means strictly moving in one direction. By aggressively popping smaller numbers from the `tail` before we insert a new larger number, we guarantee that the values in the Deque are strictly sorted from Largest (at the `head`) down to Smallest (at the `tail`). This structure perfectly preserves the current maximum at the `head`, while keeping the smaller (but newer) numbers safely waiting behind it in case the massive `head` element falls out of the window!
