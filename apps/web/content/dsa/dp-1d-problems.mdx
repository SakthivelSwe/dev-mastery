## WHY
Once you understand the basic concept of DP (caching subproblems to avoid recursion explosions), the hardest part of DP is learning how to spot it in an interview.
Most 1-Dimensional DP problems disguise themselves as "Optimization" or "Combinatorics" problems.
*"Find the MAX profit"*, *"Find the MINIMUM number of jumps"*, or *"How many WAYS can you reach the end?"*
If a problem asks for an optimal value (Max/Min) or a total count of permutations, and the data is a 1D array or a single number line, it is almost always a 1D Dynamic Programming problem!

## THEORY
### The 3 Steps to Solve ANY DP Problem
You should never write code immediately for a DP problem. You must establish the mathematical formula on a whiteboard first!
1. **Define the State:** What does `dp[i]` represent in plain English?
2. **Find the Recurrence Relation:** How can I calculate `dp[i]` if I already know the answers to `dp[i-1]` and `dp[i-2]`?
3. **Establish Base Cases:** What is the answer for `dp[0]` and `dp[1]`?

## CODE

### Pattern 1 — Counting Ways (Climbing Stairs)
*Problem:* You are climbing a staircase. It takes `N` steps to reach the top. Each time you can either climb 1 step or 2 steps. How many distinct ways can you climb to the top?

**1. State:** `dp[i]` = The total number of distinct ways to reach step `i`.
**2. Recurrence Relation:** To reach step 10, you MUST have either jumped from step 9 (a 1-step jump), or jumped from step 8 (a 2-step jump). Therefore, the total ways to reach step 10 is simply the ways to reach 9 + the ways to reach 8! `dp[i] = dp[i-1] + dp[i-2]`.
**3. Base Cases:** `dp[1] = 1` (Only one way to reach step 1). `dp[2] = 2` (1+1 or 2).

*Wait a minute... this is just the Fibonacci sequence in disguise!*

```java
public int climbStairs(int n) {
    if (n <= 2) return n;
    
    int[] dp = new int[n + 1];
    dp[1] = 1;
    dp[2] = 2;
    
    for (int i = 3; i <= n; i++) {
        dp[i] = dp[i - 1] + dp[i - 2];
    }
    return dp[n];
}
```

### Pattern 2 — Take It or Leave It (House Robber)
*Problem:* You are a robber planning to rob houses along a street. Each house has a certain amount of money `nums[i]`. If you rob two adjacent houses, the alarm triggers! What is the MAXIMUM money you can rob?

**1. State:** `dp[i]` = The maximum money you can rob from the first `i` houses.
**2. Recurrence Relation:** When standing at house `i`, you have two choices:
   - **Take It:** Rob house `i`! But this means you CANNOT have robbed house `i-1`. So your total is `nums[i] + dp[i-2]`.
   - **Leave It:** Skip house `i`. This means your total is just whatever maximum money you had at the previous house `dp[i-1]`.
   - The formula is: `dp[i] = Math.max(Take It, Leave It)` -> `dp[i] = Math.max(nums[i] + dp[i-2], dp[i-1])`.
**3. Base Cases:** `dp[0] = nums[0]`. `dp[1] = Math.max(nums[0], nums[1])`.

```java
public int rob(int[] nums) {
    if (nums.length == 0) return 0;
    if (nums.length == 1) return nums[0];
    
    int[] dp = new int[nums.length];
    dp[0] = nums[0];
    dp[1] = Math.max(nums[0], nums[1]); // We can only rob the richest of the first two!
    
    for (int i = 2; i < nums.length; i++) {
        // Take it OR Leave it!
        dp[i] = Math.max(nums[i] + dp[i - 2], dp[i - 1]);
    }
    
    return dp[nums.length - 1]; // The final answer is at the end of the array
}
```

### Pattern 3 — Minimum Costs (Coin Change)
*Problem:* You have coins of different denominations (e.g., [1, 2, 5]) and a total amount (e.g., 11). What is the MINIMUM number of coins needed to make up that amount?

**1. State:** `dp[i]` = The absolute minimum coins required to make the amount `i`.
**2. Recurrence Relation:** To make amount `11`, if I use a `5` coin, I suddenly only need to make amount `6`. So the coins needed is `1 (for the 5 coin) + dp[6]`. We test EVERY coin and find the absolute minimum!
   `dp[i] = Math.min( dp[i], 1 + dp[i - coin_value] )`
**3. Base Cases:** `dp[0] = 0` (It takes 0 coins to make $0). Initialize all other amounts to Infinity.

```java
import java.util.Arrays;

public int coinChange(int[] coins, int amount) {
    int[] dp = new int[amount + 1];
    
    // Fill with a "Max" value (use amount + 1 as it represents Infinity)
    Arrays.fill(dp, amount + 1);
    dp[0] = 0; // Base case
    
    for (int i = 1; i <= amount; i++) { // Loop over every dollar amount!
        for (int coin : coins) {        // Try every coin for this amount!
            if (i - coin >= 0) {
                // Keep the minimum!
                dp[i] = Math.min(dp[i], 1 + dp[i - coin]);
            }
        }
    }
    
    // If dp[amount] is still Infinity, it's impossible to make that amount!
    return dp[amount] > amount ? -1 : dp[amount];
}
```

## REAL_WORLD
**Video Encoding (Keyframes)**
When streaming a video on YouTube, the video file does not contain 60 full-resolution images per second. That would take massive bandwidth.
Instead, it contains "Keyframes" (full images) and "Delta frames" (only the pixels that changed).
Video encoders use 1D Dynamic Programming algorithms (like Viterbi or shortest-path approximations) to analyze a sequence of frames and determine the *optimal* mathematically perfect moments to insert a Keyframe versus a Delta frame, balancing the exact trade-off between minimum file size and maximum visual quality across the 1D timeline of the video!

## INTERVIEW
**Q: You write the `coinChange` algorithm above. For the "Infinity" placeholder, you use `Integer.MAX_VALUE`. When you run the code, it crashes returning completely incorrect negative numbers! Why is `Integer.MAX_VALUE` dangerous in DP math?**
A: **Integer Overflow.**
Look closely at the recurrence relation: `1 + dp[i - coin]`.
If `dp[i - coin]` was unable to be formed by any coins, it remains at "Infinity" (`Integer.MAX_VALUE`).
When the algorithm runs `1 + Integer.MAX_VALUE`, the 32-bit integer overflows its binary limit and wraps around to `-2,147,483,648`.
The `Math.min()` function then sees this massive negative number, assumes it is a spectacularly good result, and locks it into the `dp` array!
*The Fix:* Always use a "safe" maximum. For Coin Change, because the absolute worst case scenario is using all `1` dollar coins, the maximum coins you could ever use is `amount`. Therefore, `amount + 1` acts as a perfectly safe, mathematically indestructible "Infinity" that will never trigger an overflow!
