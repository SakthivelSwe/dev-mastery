## WHY
If you open Google and type "app", the search engine instantly suggests "apple", "app store", and "application" out of billions of possible phrases in milliseconds.
If you store all these strings in a massive Array or a HashSet, finding all words that *start with* "app" requires checking every single word, taking $O(N)$ time (where $N$ is the number of words in the dictionary).
A **Trie** (pronounced "try", from the word re**trie**val) is a specialized tree data structure designed specifically for blazing-fast prefix matching. It can find all words starting with a given prefix in $O(L)$ time, where $L$ is simply the length of the prefix string, completely ignoring the billions of other words in the dictionary!

## THEORY
### The Structure
Unlike a Binary Tree which stores a full word in a single node, a Trie stores *characters* on the edges or nodes.
- The root node is empty.
- If you insert the word "CAT", the root points to 'C', which points to 'A', which points to 'T'.
- If you then insert the word "CAR", it reuses the 'C' and 'A' nodes from "CAT", and simply adds an 'R' branch splitting off from the 'A'!
- Every node contains a boolean flag `isEndOfWord` to indicate if the path from the root to this node forms a valid, complete word in the dictionary.

## CODE

### Level 1 — The TrieNode Class
A Trie node needs a way to point to its children. Because the English alphabet has 26 lowercase letters, we can use a simple array of size 26! (Index 0 = 'a', Index 1 = 'b').

```java
class TrieNode {
    // Array pointing to children nodes (null if the child doesn't exist)
    TrieNode[] children;
    // Flag to mark if a complete word ends at this specific node
    boolean isEndOfWord;

    public TrieNode() {
        children = new TrieNode[26]; // 26 letters in English alphabet
        isEndOfWord = false;
    }
}
```

### Level 2 — Inserting Words
To insert a word, we loop through each character. If the path doesn't exist, we create the node. We move our pointer down the tree until the word is finished, then we set the boolean flag.

```java
public class Trie {
    private TrieNode root;

    public Trie() {
        root = new TrieNode(); // The empty root node
    }

    // Time Complexity: O(L) where L is the length of the word
    public void insert(String word) {
        TrieNode current = root;

        for (int i = 0; i < word.length(); i++) {
            char ch = word.charAt(i);
            int index = ch - 'a'; // Convert 'a' to 0, 'b' to 1

            // If the branch doesn't exist, create it!
            if (current.children[index] == null) {
                current.children[index] = new TrieNode();
            }
            
            // Move down the tree to the child node
            current = current.children[index];
        }

        // We finished the word! Mark this specific node as a valid end.
        current.isEndOfWord = true;
    }
}
```

### Level 3 — Searching for Exact Words
To check if a word exists, we trace the path. If the path breaks, the word doesn't exist. If we reach the end, we MUST check `isEndOfWord` (because "APP" might be a prefix in the tree for "APPLE", but "APP" itself might not be a registered word!).

```java
// Time Complexity: O(L) where L is the length of the word
public boolean search(String word) {
    TrieNode current = root;

    for (int i = 0; i < word.length(); i++) {
        int index = word.charAt(i) - 'a';
        
        // The path broke! The word isn't here.
        if (current.children[index] == null) {
            return false; 
        }
        current = current.children[index];
    }

    // We finished traversing. Is it a registered word?
    return current.isEndOfWord; 
}
```

### Level 4 — The Autocomplete Power (Prefix Search)
This is why Tries exist! We want to know if *any* word starts with a specific prefix. It is the exact same code as `search()`, but we don't care about `isEndOfWord`! If the path exists at all, then words with that prefix exist!

```java
public boolean startsWith(String prefix) {
    TrieNode current = root;

    for (int i = 0; i < prefix.length(); i++) {
        int index = prefix.charAt(i) - 'a';
        
        if (current.children[index] == null) {
            return false;
        }
        current = current.children[index];
    }
    
    // We successfully traced the entire prefix! 
    // This means there is AT LEAST one word continuing down this path.
    return true; 
}
```

## REAL_WORLD
**Memory Optimization (HashMap vs Array)**
In the code above, we hardcoded `TrieNode[] children = new TrieNode[26]`. This is blazingly fast because array lookups take $O(1)$ time. 
However, it is a massive memory hog! Even if a node only has one child (e.g., 'Q' only points to 'U'), we still allocate 26 memory slots, leaving 25 of them completely empty (`null`). For a massive dictionary, this wastes gigabytes of RAM.
If your Trie needs to support the entire Unicode character set (Chinese characters, Emojis, Cyrillic) which has 140,000+ characters, creating a 140,000 element array for *every single node* will crash the server instantly.
In enterprise systems, `TrieNode[]` is often replaced with `HashMap<Character, TrieNode>`. A HashMap only uses memory for the children that actually exist. The tradeoff is that HashMaps are slightly slower than raw Arrays due to hash collisions, but they save 90% of your RAM.

## INTERVIEW
**Q: You are given a massive 2D Grid of letters (like a Boggle board) and an array of 10,000 dictionary words. You must find all words from the dictionary that can be formed by tracing paths in the grid. If you just run a DFS from every cell and check the HashSet of words, it takes way too long. How does a Trie optimize this to pass the LeetCode Hard time limit?**
A: **Early Termination during Backtracking.**
This is the classic "Word Search II" LeetCode problem.
If you use a HashSet, when your DFS traces the path `"X-Y-Z"`, it has no way of knowing if any words in the dictionary actually start with "XYZ". The DFS will blindly continue searching 10 levels deep ("XYZABCDEFG") before realizing it's a dead end, wasting massive amounts of compute time.
If you load the 10,000 words into a Trie first, you can pass the `TrieNode` pointer directly into your DFS.
When the DFS tries the letter 'Z', it checks `node.children['Z']`. If it's `null`, it proves that mathematically, NO word in the dictionary starts with "XYZ". The DFS instantly terminates that branch (`return;`), pruning millions of useless recursive calls!
