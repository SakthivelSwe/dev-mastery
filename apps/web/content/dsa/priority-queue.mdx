## WHY
A standard Queue operates on the **FIFO** (First-In-First-Out) principle: the first person in line gets the ticket first.
But in reality, systems rarely treat everyone equally. An airline boards First Class passengers before Economy, regardless of who arrived at the gate first. A server processing background jobs will process an "Urgent Payment" job before processing a "Weekly Email Newsletter" job.
A **Priority Queue** is an Abstract Data Type (ADT) that fundamentally breaks the FIFO rule. It guarantees that the element with the *highest priority* is always dequeued next, regardless of its arrival time!

## THEORY
### Abstract Data Type vs Implementation
A Priority Queue is a *concept* (an ADT). It simply states the rules:
1. `insert(item, priority)`
2. `getHighestPriority()`
3. `deleteHighestPriority()`

How you physically code these rules determines the performance:
- **Unsorted Array:** Insertion is $O(1)$, but finding the highest priority requires scanning the whole array $O(N)$.
- **Sorted Array:** Finding the highest is $O(1)$, but inserting requires shifting all elements $O(N)$.
- **Heap (The Standard):** A Binary Heap perfectly balances this! Both Insertion and Deletion take $O(\log N)$ time, making it the mathematically optimal structure for Priority Queues!

## CODE

### Level 1 — Priority Queue in Java
In Java, the Priority Queue concept is natively implemented via the `java.util.PriorityQueue` class, which uses a Min-Heap under the hood.

```java
import java.util.PriorityQueue;

public class BasicPQ {
    public static void main(String[] args) {
        // By default, it prioritizes the SMALLEST numbers first (Min-Heap)
        PriorityQueue<Integer> pq = new PriorityQueue<>();

        // Elements arrive in random order
        pq.offer(50);
        pq.offer(10);
        pq.offer(30);

        // They are processed in STRICT Priority order!
        System.out.println(pq.poll()); // 10
        System.out.println(pq.poll()); // 30
        System.out.println(pq.poll()); // 50
    }
}
```

### Level 2 — Custom Objects and Comparators
When prioritizing complex objects (like Tasks or Users), the Priority Queue doesn't magically know what makes one task more important than another. You must define a `Comparator`.

```java
import java.util.PriorityQueue;

class Task {
    String name;
    int priority; // 1 is highest priority, 10 is lowest

    public Task(String name, int priority) {
        this.name = name;
        this.priority = priority;
    }
}

public class TaskManager {
    public static void main(String[] args) {
        // We teach the PriorityQueue how to compare Tasks using a Lambda!
        // (t1, t2) -> t1.priority - t2.priority creates Ascending Order.
        PriorityQueue<Task> queue = new PriorityQueue<>(
            (t1, t2) -> Integer.compare(t1.priority, t2.priority)
        );

        queue.offer(new Task("Send Newsletter", 5));
        queue.offer(new Task("Process Payment", 1));
        queue.offer(new Task("Backup Database", 3));

        // The Payment processes first, because 1 is the smallest number!
        System.out.println(queue.poll().name); // "Process Payment"
        System.out.println(queue.poll().name); // "Backup Database"
    }
}
```

### Level 3 — Multiple Priority Factors
What if two Tasks have the exact same priority? In an operating system, if two processes have the same CPU priority, the system uses their arrival time as a tie-breaker (FIFO fallback).
We can implement this multi-level logic directly in our Comparator!

```java
class Process {
    String name;
    int priority; 
    long timestamp; // Arrival time

    public Process(String name, int priority, long timestamp) {
        this.name = name;
        this.priority = priority;
        this.timestamp = timestamp;
    }
}

// Multi-level Comparator!
PriorityQueue<Process> osQueue = new PriorityQueue<>((p1, p2) -> {
    // 1. First, compare by priority (Lowest number = highest priority)
    int priorityDiff = Integer.compare(p1.priority, p2.priority);
    
    if (priorityDiff != 0) {
        return priorityDiff; // Priorities are different, use this!
    }
    
    // 2. TIE BREAKER! If priorities are identical, compare by timestamp!
    // The older timestamp (smaller number) goes first!
    return Long.compare(p1.timestamp, p2.timestamp);
});
```

## REAL_WORLD
**Top K Problems**
Priority Queues are the absolute master of the "Top K" problem in interviews.
*Question:* Given an array of 1,000,000 integers, find the 5 largest numbers.
If you sort the array, it takes $O(N \log N)$ time and modifies the array.
Instead, use a **Min-Heap Priority Queue of size K (size 5)**.
You loop through the million numbers, pushing them into the PQ. The PQ automatically keeps the 5 largest numbers. Wait, a Min-Heap for the *largest* numbers? YES! 
Because it's a Min-Heap, the *smallest* of those 5 numbers sits at the top. As you iterate, if the new number from the array is larger than the top of the Min-Heap, you instantly `poll()` the top, and `offer()` the new number! 
At the end of the loop, the PQ holds the Top 5 largest numbers. Time Complexity: $O(N \log K)$, which is astronomically faster than sorting!

## INTERVIEW
**Q: You use a `PriorityQueue` in Java to sort objects. You print the queue by running a standard enhanced for-loop: `for (Task t : queue) { System.out.println(t.name); }`. The interviewer points out that the items printed to the console are completely out of order! They are not sorted by priority at all. Why?**
A: **Iterating a Heap does not guarantee sorted order!**
A `PriorityQueue` in Java is physically stored as an Array that maintains the Heap Property (Parent < Child). It is NOT a perfectly sorted array! The left child and right child have no mathematical relationship.
When you run a standard `for` loop over a `PriorityQueue`, Java simply iterates through the underlying array from index 0 to N. It prints the raw, unsorted tree structure.
*The Fix:* The ONLY way to guarantee the items are retrieved in perfect sorted order is to use the `poll()` method repeatedly until the queue is empty. `poll()` legally extracts the root node and rebalances the tree every single time, guaranteeing perfect mathematical priority order!

### 5 Deep Conceptual Questions
**Q1: Why heap not sorted array?** > **A:** Heap inserts O(log n) vs sorted insert O(n); only the extreme is needed.
**Q2: ONE model?** > **A:** "Partial order tree: parent ≤ children; root is min."
**Q3: Misconception+code.** > **A:** Wrong comparator: `new PriorityQueue<>(Collections.reverseOrder())` for max.
**Q4: Top-K?** > **A:** Min-heap of size k; pop smaller — O(n log k).
**Q5: Senior one-liner.** > **A:** "A PQ is a heap giving O(log n) insert and O(1) peek-extreme — Dijkstra, Top-K, scheduling."
