## WHY
Everyone knows how to use a `HashMap` to achieve $O(1)$ constant time lookups. It is the most commonly used data structure in programming interviews.
But what happens when you need to store incredibly complex data (like multi-dimensional arrays or entire trees) as Keys in a HashMap? What happens when a malicious user sends you 100,000 distinct strings that all intentionally produce the exact same HashCode, forcing your $O(1)$ server to degrade into an $O(N)$ LinkedList, crashing the system?
**Advanced Hashing** goes beyond `.put()` and `.get()`. It explores Custom Hashing, Rolling Hashes, and collision resolution strategies required for building secure, scalable backend systems.

## THEORY
### How HashMaps Actually Work
A HashMap is just a standard Array under the hood.
1. When you call `map.put("Alice", 50)`, the HashMap calls the `.hashCode()` function on the String "Alice". It returns a massive integer (e.g., `64024`).
2. The HashMap takes that number and applies modulo math to fit it into the underlying Array: `64024 % array.length`. If the array size is 10, the index is `4`.
3. It stores the value `50` at `array[4]`.

**Collisions:** If you try to insert "Bob", and its hash modulo also equals `4`, a Collision occurs! The HashMap creates a LinkedList at `array[4]` and stores both entries there. If you have too many collisions, finding "Bob" requires iterating through the entire LinkedList, destroying the $O(1)$ performance!

## CODE

### Pattern 1 — Custom HashCodes for Objects
*Problem:* You want to cache API responses. The API is called with a custom `UserRequest` object. You use the object as the Key in a HashMap. You construct an identical request and query the map, but it returns `null`!

*Why?* Because by default, Java computes the HashCode of an Object based on its physical memory address (RAM pointer), NOT its data! Two identical objects in different memory locations have completely different HashCodes!
You MUST override `equals()` and `hashCode()`.

```java
import java.util.Objects;

class UserRequest {
    int userId;
    String query;

    public UserRequest(int userId, String query) {
        this.userId = userId;
        this.query = query;
    }

    // 1. Tell Java exactly when two objects hold the same DATA
    @Override
    public boolean equals(Object o) {
        if (this == o) return true; // Same memory address? Instant true.
        if (o == null || getClass() != o.getClass()) return false;
        UserRequest that = (UserRequest) o;
        // The data must match perfectly!
        return userId == that.userId && Objects.equals(query, that.query);
    }

    // 2. Mathematically combine the data into a single Integer!
    @Override
    public int hashCode() {
        // Objects.hash() brilliantly multiplies the fields by prime numbers (usually 31) 
        // to mathematically guarantee a wide distribution and avoid collisions!
        return Objects.hash(userId, query);
    }
}
```

### Pattern 2 — The Rolling Hash (Rabin-Karp Algorithm)
*Problem:* Find if the substring "CAT" exists inside a massive 1-Billion-character DNA string.
If you use standard substring matching `s.substring(i, i+3).equals("CAT")`, comparing the 3 letters over and over again takes $O(N \times M)$ time.
A **Rolling Hash** turns a sliding window of characters into a single integer. When the window slides to the right (dropping the first letter, adding the next letter), the algorithm does not recalculate the hash from scratch! It applies a blazing-fast mathematical formula to update the hash in $O(1)$ time!

```java
// A simplified Rolling Hash concept (without complex modulo arithmetic for overflow)
public int searchSubString(String text, String pattern) {
    int patternLen = pattern.length();
    int textLen = text.length();
    if (patternLen > textLen) return -1;

    // Calculate the target hash of "CAT"
    int targetHash = 0;
    int currentHash = 0;
    for (int i = 0; i < patternLen; i++) {
        targetHash += pattern.charAt(i); 
        currentHash += text.charAt(i);
    }

    // Slide the window!
    for (int i = 0; i <= textLen - patternLen; i++) {
        // If the hashes match, VERIFY the actual strings (to prevent Hash Collisions!)
        if (currentHash == targetHash) {
            if (text.substring(i, i + patternLen).equals(pattern)) {
                return i; // Found it!
            }
        }

        // Slide the window: Drop the left character, add the new right character!
        if (i < textLen - patternLen) {
            currentHash -= text.charAt(i);                 // Drop Left
            currentHash += text.charAt(i + patternLen);    // Add Right
        }
    }
    return -1;
}
```
*Note: Real Rabin-Karp uses Polynomial Hashing (`val * 31^2 + val * 31...`) to prevent collisions where "CAT" and "TAC" produce the same sum!*

### Pattern 3 — Hashing Arrays/States (Memoization)
In complex DP problems, your State might be an entire Array! You cannot use a Java `int[]` array as a Key in a HashMap, because Java uses memory references for arrays. 
The easiest trick in competitive programming is to serialize the array into a `String`!

```java
import java.util.*;

public class StateHashing {
    // Cache map: String representation of Array -> Answer
    HashMap<String, Integer> cache = new HashMap<>();

    public int solvePuzzle(int[] state) {
        // Convert [1, 2, 3] to a string "1,2,3"
        String key = Arrays.toString(state); 

        if (cache.containsKey(key)) {
            return cache.get(key); // Return the cached answer!
        }

        int answer = performExpensiveComputation(state);
        cache.put(key, answer);
        return answer;
    }

    private int performExpensiveComputation(int[] state) { return 42; }
}
```

## REAL_WORLD
**Hash DoS Attacks & Java 8 Optimization**
In the early 2010s, a devastating cyberattack was discovered: The **Hash DoS Attack**.
Hackers realized that Java HashMaps used simple LinkedLists to handle collisions. The hackers pre-calculated thousands of bizarre Strings that all magically produced the exact same HashCode (e.g., `"AaAa"`, `"BBBB"`). 
When the hackers sent these thousands of Strings in an HTTP Request to a Tomcat web server, the server attempted to put them into a HashMap. Because every single String caused a collision, the server ended up building a gigantic LinkedList with 100,000 nodes inside a single array slot.
The HashMap degraded to $O(N)$ insertion time. Adding the 100,000th string required traversing 99,999 nodes. This spiked the server CPU to 100%, causing the entire internet infrastructure of companies to crash offline!
*The Fix:* In Java 8, the engineers modified the HashMap. If a specific array slot experiences more than 8 collisions (a LinkedList of size 8), Java instantly destroys the LinkedList and automatically replaces it with a **Red-Black Tree**! This limits the worst-case degradation to $O(\log N)$ instead of $O(N)$, completely neutralizing the Hash DoS attack!

## INTERVIEW
**Q: Why does the Java `hashCode()` implementation for Strings use the prime number `31` to multiply characters? (`hash = hash * 31 + char`)**
A: **Bitwise Optimization and Collision Reduction.**
Prime numbers are heavily used in hashing algorithms because they do not share common factors with other numbers, drastically reducing the mathematical probability of two different strings generating the exact same hash (Hash Collisions) when wrapped around an array boundary.
But why specifically `31` and not `37` or `41`? 
Because `31` is exactly `32 - 1`. And `32` is $2^5$.
Because it relates perfectly to a power of 2, the Java compiler can optimize the expensive mathematical multiplication `(hash * 31)` into an ultra-fast raw bitwise operation at the hardware level: `(hash << 5) - hash`. This provides incredible security and distribution without sacrificing execution speed!
