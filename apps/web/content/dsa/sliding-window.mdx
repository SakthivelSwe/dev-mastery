## WHY
Imagine you are given an array of 10,000 numbers representing daily stock prices, and you are asked to find the highest sum of any 5 consecutive days. 
The naive approach is to use nested loops: for every day, calculate the sum of the next 5 days. For day 0, sum `[0..4]`. For day 1, sum `[1..5]`. For day 2, sum `[2..6]`. 
Notice the massive inefficiency? The sum for day 1 `[1..5]` recalculates elements 1, 2, 3, and 4, which were *already calculated* for day 0! This redundant work forces the algorithm into `O(n * k)` time complexity.

The **Sliding Window Pattern** eliminates this redundant calculation completely. Instead of recalculating from scratch, you treat the 5 elements as a "window". To shift the window to the right, you simply subtract the element that falls out of the left side of the window, and add the new element that enters the right side. This instantly drops the time complexity to `O(n)`.

## THEORY
### Fixed vs Variable Windows
There are two main variations of this pattern:
1. **Fixed Window**: The size of the window is strictly defined by the problem (e.g., "Find the max sum of exactly `K` elements"). You build the initial window of size K, and then slide it exactly one step at a time, performing one subtraction and one addition.
2. **Variable Window**: The window expands or shrinks dynamically based on a condition (e.g., "Find the *longest* subarray whose sum is less than `Target`"). You expand the `right` pointer to ingest data until the condition breaks, then you shrink the `left` pointer to drop data until the condition is valid again.

## CODE

### Level 1 — Beginner (Naive vs Fixed Window)
```java
public class FixedWindow {
    // Problem: Find the maximum sum of a contiguous subarray of size K
    
    // ❌ The Naive Approach: O(n * k)
    // Avoid this! It recalculates the overlap repeatedly.
    public static int maxSubArraySumNaive(int[] arr, int k) {
        int maxSum = 0;
        for (int i = 0; i <= arr.length - k; i++) {
            int currentSum = 0;
            for (int j = i; j < i + k; j++) {
                currentSum += arr[j];
            }
            maxSum = Math.max(maxSum, currentSum);
        }
        return maxSum;
    }
}
```

### Level 2 — Building Blocks (The Fixed Window Solution)
```java
public class FixedWindowOptimized {
    
    // ✅ The Sliding Window Approach: O(n)
    public static int maxSubArraySum(int[] arr, int k) {
        if (arr.length < k) return -1; // Edge case
        
        int maxSum = 0;
        int windowSum = 0;
        
        // Step 1: Create the initial window of size K
        for (int i = 0; i < k; i++) {
            windowSum += arr[i];
        }
        maxSum = windowSum;
        
        // Step 2: Slide the window from index K to the end
        for (int i = k; i < arr.length; i++) {
            // Add the new right element, subtract the old left element
            windowSum = windowSum + arr[i] - arr[i - k];
            
            // Record the maximum we've seen so far
            maxSum = Math.max(maxSum, windowSum);
        }
        
        return maxSum;
    }
}
```

### Level 3 — Intermediate (Variable Window)
```java
public class VariableWindow {
    // Problem: Find the minimum length of a contiguous subarray 
    // whose sum is greater than or equal to the target.
    
    // O(n) Time | O(1) Space
    public static int minSubArrayLen(int target, int[] nums) {
        int minLength = Integer.MAX_VALUE;
        int windowSum = 0;
        int left = 0; // The left edge of our dynamic window
        
        // The right pointer aggressively expands the window
        for (int right = 0; right < nums.length; right++) {
            windowSum += nums[right]; 
            
            // While the condition is met, try to shrink the window to find a smaller valid length!
            while (windowSum >= target) {
                // Record the current valid length
                minLength = Math.max(minLength, right - left + 1);
                
                // Subtract the left element and move the left pointer forward
                windowSum -= nums[left];
                left++; 
            }
        }
        
        return minLength == Integer.MAX_VALUE ? 0 : minLength;
    }
}
```

### Level 4 — Advanced (String Window with HashMap)
```java
import java.util.HashMap;
import java.util.Map;

public class LongestSubstring {
    // Problem: Find the length of the longest substring without repeating characters.
    // Example: "abcabcbb" -> "abc" (Length 3)
    
    public static int lengthOfLongestSubstring(String s) {
        // Map to store character -> its most recent index
        Map<Character, Integer> charIndexMap = new HashMap<>();
        
        int maxLength = 0;
        int left = 0;
        
        for (int right = 0; right < s.length(); right++) {
            char currentChar = s.charAt(right);
            
            // If we've seen this character BEFORE, and it's INSIDE our current window...
            if (charIndexMap.containsKey(currentChar) && charIndexMap.get(currentChar) >= left) {
                // Instantly shrink the window by teleporting the left pointer 
                // just past the previous occurrence of this character!
                left = charIndexMap.get(currentChar) + 1;
            }
            
            // Update the most recent index for this character
            charIndexMap.put(currentChar, right);
            
            // Calculate max length
            maxLength = Math.max(maxLength, right - left + 1);
        }
        
        return maxLength;
    }
}
```

## REAL_WORLD
While you won't use sliding windows to manipulate raw arrays in enterprise web development, the algorithm itself is the foundation of **Rate Limiting** in API Gateways (like Spring Cloud Gateway or Kong).

If you want to restrict an API user to "100 requests per minute", you cannot simply reset their counter to 0 at the top of every minute (Fixed Window limit). If you do that, a user could send 100 requests at 1:59:59, and another 100 requests at 2:00:01. They bypassed your limit, executing 200 requests in 2 seconds!

To prevent this, production systems (using Redis) implement a **Sliding Window Log** algorithm. Every time a request comes in, it is timestamped. The system dynamically drops timestamps older than 60 seconds (moving the `left` pointer), and counts the remaining timestamps in the window. If the count exceeds 100, the request is rejected with `HTTP 429 Too Many Requests`. The algorithmic logic is identical to the Variable Window code pattern!

## INTERVIEW
**Q: How do you identify that a problem requires a Sliding Window?**
A: Look for three specific keywords in the prompt: 
1. **"Contiguous"**, "Subarray", or "Substring" (meaning the elements must be adjacent). If it asks for "Subsequences", sliding window will fail.
2. **"Maximum/Minimum"** or "Longest/Shortest".
3. **"Of size K"** (Fixed Window) or **"Sums to target"** (Variable Window).

**Q: In a Variable Window, is the time complexity `O(n^2)` because there is a `while` loop inside the `for` loop?**
A: No, it is strictly `O(n)`. This is a classic interview trap! Even though there is a nested `while` loop, the `left` pointer and `right` pointer only ever move *forward*. The `right` pointer traverses the array exactly once (N steps). The `left` pointer traverses the array exactly once (N steps). The total maximum steps across the entire algorithm is `2N`, which simplifies to `O(n)` linear time.

**Q: What is the difference between Two-Pointers and Sliding Window?**
A: Sliding Window is a specific sub-category of the Two-Pointer technique. In general Two-Pointer algorithms (like Array Reversal or Binary Search), the pointers often start at opposite ends and move towards each other. In a Sliding Window, both pointers start at the beginning (index 0) and move in the exact same direction, forming a "caterpillar" effect that stretches and shrinks as it moves right.

## FEYNMAN CHECK
### Explain Sliding Window Like I'm 10 Years Old
> A window is a clear ruler sliding over numbers, summing only what's under it. Instead of re-adding every time, you add the new right and drop the old left — one pass. This is why "longest substring no repeat" is O(n).

**Q1: Why O(n)?** > **A:** Each element enters and leaves the window once; 2n ops.
**Q2: ONE model?** > **A:** "Grow right, shrink left to keep validity."
**Q3: Misconception+code.** > **A:** Recompute sum = O(nk); instead `sum+=a[r]-a[l]`.
**Q4: Variable window?** > **A:** Expand until invalid, contract until valid.
