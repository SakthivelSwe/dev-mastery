## WHY
Dynamic Programming explores every possible path to guarantee the absolute best mathematical answer. But sometimes, exploring every path takes too much time and memory.
What if we don't need to look at the big picture? What if simply making the *best possible immediate decision right now* naturally leads to the best overall solution?
**Greedy Algorithms** do exactly this. At every step, they blindly pick the absolute best local option (the "greedy" choice) and never look back. They never backtrack. They never cache subproblems. When Greedy works, it produces blazing-fast $O(N \log N)$ or $O(N)$ solutions for seemingly impossible problems!

## THEORY
### When does Greedy Work?
Greedy algorithms only work if a problem exhibits the **Greedy Choice Property**.
This mathematically proves that a global optimum can be arrived at by selecting a local optimum.
- **Example where Greedy WORKS:** Giving change for $0.85 using standard US Coins (Quarters, Dimes, Nickels, Pennies). The greedy approach says: *"Always use the biggest coin possible."* 3 Quarters (75c) + 1 Dime (10c) = 4 coins. This perfectly matches the mathematical minimum!
- **Example where Greedy FAILS:** Giving change for $0.15 using a bizarre coin system of [1, 7, 10]. Greedy says: *"Use the biggest coin! 10!"* Then it uses five 1s. Total = 6 coins. But the *actual* best answer is two 7s + one 1 (3 coins!). Because Greedy never looks back, it fell into a trap! If Greedy fails, you MUST use Dynamic Programming.

## CODE

### Pattern 1 — Activity Selection (Non-Overlapping Intervals)
*Problem:* You are given an array of meetings `[[start, end]]`. You want to attend the maximum number of meetings possible. You cannot attend overlapping meetings.

*The Greedy Logic:* Should you pick the meeting that starts earliest? No, what if the earliest meeting lasts 10 hours? It blocks everything!
The correct greedy choice: **Always pick the meeting that ENDS the earliest!** The sooner a meeting gets out of the way, the more time is left in the day for other meetings!

```java
import java.util.Arrays;

public int maxEvents(int[][] events) {
    if (events.length == 0) return 0;

    // 1. Sort the events by their END time (Ascending)
    Arrays.sort(events, (a, b) -> Integer.compare(a[1], b[1]));

    int count = 1;
    // Track when the last meeting we attended finished
    int lastEndTime = events[0][1];

    // 2. Iterate through the rest of the meetings
    for (int i = 1; i < events.length; i++) {
        // If the next meeting STARTS after our last meeting ENDED...
        if (events[i][0] >= lastEndTime) {
            count++; // We can attend it!
            // Update our tracker to this new meeting's end time!
            lastEndTime = events[i][1]; 
        }
    }

    return count;
}
```

### Pattern 2 — Jump Game
*Problem:* You are given an array `nums`. Each element represents your maximum jump length at that position. Starting at index 0, return `true` if you can reach the last index.
*Array:* `[2, 3, 1, 1, 4]`.

*The Greedy Logic:* We don't need DP to check every possible jump combination. We just need to track the "Maximum Reachable Index". If we are standing at index `i`, and our jump power is `nums[i]`, we can reach index `i + nums[i]`. We just keep expanding our "Max Reach" globally!

```java
public boolean canJump(int[] nums) {
    int maxReach = 0; // The furthest index we are allowed to visit

    for (int i = 0; i < nums.length; i++) {
        // CRITICAL: If our current index 'i' is physically further 
        // than our maximum allowed reach, we are stuck! We fell in a gap!
        if (i > maxReach) {
            return false;
        }

        // Greedy Choice: Does jumping from this current spot 
        // push our global reach further than it was before?
        maxReach = Math.max(maxReach, i + nums[i]);

        // Optimization: If our reach already extends past the array, we won!
        if (maxReach >= nums.length - 1) {
            return true;
        }
    }
    
    return true;
}
```

### Pattern 3 — Two City Scheduling
*Problem:* You have `2N` people. You must fly exactly `N` to City A and `N` to City B. You are given an array `costs` where `costs[i] = [costA, costB]`. Minimize the total cost.

*The Greedy Logic:* Don't just sort by who is cheapest for City A. A person might cost $10 for City A, and $15 for City B. Sending them to City A saves $5.
But another person might cost $1000 for City A, and $10 for City B! Sending them to City A wastes $990!
The greedy choice is based on the **Opportunity Cost** (The difference: `costA - costB`). Sort by the difference to see who absolutely MUST go to City A to avoid a financial disaster!

```java
import java.util.Arrays;

public int twoCitySchedCost(int[][] costs) {
    // Sort by Opportunity Cost: (Cost of A - Cost of B)
    // A highly negative number means A is massively cheaper than B!
    Arrays.sort(costs, (p1, p2) -> Integer.compare(p1[0] - p1[1], p2[0] - p2[1]));

    int totalCost = 0;
    int n = costs.length / 2;

    // Send the first N people to City A (Because A was much cheaper for them!)
    for (int i = 0; i < n; i++) {
        totalCost += costs[i][0];
    }

    // Send the remaining N people to City B
    for (int i = n; i < costs.length; i++) {
        totalCost += costs[i][1];
    }

    return totalCost;
}
```

## REAL_WORLD
**Data Compression (Huffman Coding)**
When you ZIP a file, how does the computer compress the text?
It uses a Greedy algorithm called **Huffman Coding**. It counts the frequency of every letter in your document. It then creates a Binary Tree, greedily pairing the *least frequent* letters together at the bottom, and pushing the *most frequent* letters (like 'e' and 't') to the very top.
Because frequent letters are at the top, they are assigned very short binary codes (like `01`). Rare letters (like 'z') get long codes (like `10110`). This greedy prefix-tree assignment mathematically guarantees the absolute smallest possible file size!

## INTERVIEW
**Q: How do you prove that your Greedy Algorithm is actually correct in an interview?**
A: **You usually can't (formally), but you can use logical contradiction.**
In a real computer science exam, proving a greedy algorithm requires complex mathematical proofs (Exchange Arguments or Matroid Theory). 
In a coding interview, if you propose a Greedy solution, the interviewer will likely ask: *"Are you sure?"*
You must try to invent a counter-example that breaks your logic. 
If your greedy rule is "Pick the heaviest item", immediately draw a scenario where picking a medium item and a light item produces a better outcome. If you can break your own rule, Greedy FAILS, and you must pivot to Dynamic Programming. If you struggle for 2 minutes and cannot invent a scenario that breaks your rule, it's highly likely your Greedy logic is mathematically sound!
