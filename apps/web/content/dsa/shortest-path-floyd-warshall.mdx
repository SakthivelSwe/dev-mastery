## WHY
Dijkstra's Algorithm finds the shortest path from a **Single Source** (e.g., New York) to all other cities.
If you are building an airline ticketing engine, and you need to pre-compute a massive matrix showing the absolute cheapest flight cost between *every possible pair of cities in the world* (All-Pairs Shortest Path), you would have to run Dijkstra's Algorithm $V$ separate times (once for each city). If the graph contains negative flight prices, Dijkstra won't even work!
The **Floyd-Warshall Algorithm** elegantly solves the All-Pairs Shortest Path problem for graphs with negative weights in a brilliant $O(V^3)$ algorithm that requires fewer than 10 lines of code.

## THEORY
### Dynamic Programming on Graphs
Floyd-Warshall is a Dynamic Programming algorithm. It uses an Adjacency Matrix to track distances.
The core concept is to pick a "middleman" city (let's call it $K$).
For every pair of cities $(I, J)$, the algorithm asks: *"Is the direct flight from $I \rightarrow J$ cheaper than flying from $I \rightarrow K$ and then taking a connecting flight from $K \rightarrow J$?"*
It iterates through every possible city to act as the "middleman" $K$. By the time the algorithm finishes, it has implicitly checked every single possible multi-stop route between every pair of cities!

## CODE

### Level 1 — Initialization
Because Floyd-Warshall checks all pairs, we MUST use an Adjacency Matrix (`int[][] dist`).
If a direct edge exists between $A$ and $B$, `dist[A][B]` is the weight. If no direct edge exists, the distance is Infinity. The distance from a node to itself is always `0`.

```java
public class FloydWarshall {
    final static int INF = 999999; // Use a large number, but NOT Integer.MAX_VALUE to avoid overflow!

    // Assume 'graph' is an initialized V x V Adjacency Matrix
    public int[][] findAllPairsShortestPaths(int[][] graph, int V) {
        int[][] dist = new int[V][V];

        // 1. Copy the initial graph into our distance matrix
        for (int i = 0; i < V; i++) {
            for (int j = 0; j < V; j++) {
                dist[i][j] = graph[i][j];
            }
        }

        return runAlgorithm(dist, V);
    }
}
```

### Level 2 — The Algorithm (Three Nested Loops)
The code is astonishingly simple. Three nested loops.
- The outer loop `k` selects the "middleman" node.
- The middle loop `i` selects the starting node.
- The inner loop `j` selects the destination node.

```java
private int[][] runAlgorithm(int[][] dist, int V) {
    // 1. Pick the middleman node 'k'
    for (int k = 0; k < V; k++) {
        
        // 2. Pick all starting nodes 'i'
        for (int i = 0; i < V; i++) {
            
            // 3. Pick all destination nodes 'j'
            for (int j = 0; j < V; j++) {
                
                // If the path through the middleman 'k' is cheaper than the direct path...
                if (dist[i][k] + dist[k][j] < dist[i][j]) {
                    // Update the shortest path matrix!
                    dist[i][j] = dist[i][k] + dist[k][j];
                }
            }
        }
    }

    return dist;
}
```
**Time Complexity:** 3 nested loops counting to $V$. Time Complexity is exactly **$O(V^3)$**.
**Space Complexity:** $O(V^2)$ for the matrix. Because of the $O(V^3)$ time, this algorithm is only suitable for small, dense graphs (e.g., $V < 500$).

### Level 3 — Reconstructing the Path
Just knowing the cheapest cost between $I$ and $J$ is usually not enough; we need to print out the connecting cities! We use a `next[][]` matrix to track the route.

```java
// next[i][j] stores the FIRST step you must take to get from 'i' to 'j'
int[][] next = new int[V][V];

// Initialize the 'next' matrix
for (int i = 0; i < V; i++) {
    for (int j = 0; j < V; j++) {
        if (graph[i][j] != INF && i != j) {
            next[i][j] = j; // The first step is the direct edge
        } else {
            next[i][j] = -1; // No path yet
        }
    }
}

// Inside the Floyd-Warshall loops...
if (dist[i][k] + dist[k][j] < dist[i][j]) {
    dist[i][j] = dist[i][k] + dist[k][j];
    // CRITICAL: The first step from i to j is now the first step from i to k!
    next[i][j] = next[i][k]; 
}
```

To print the path, we just follow the `next` pointers:
```java
public void printPath(int u, int v, int[][] next) {
    if (next[u][v] == -1) {
        System.out.println("No path exists.");
        return;
    }
    List<Integer> path = new ArrayList<>();
    path.add(u);
    while (u != v) {
        u = next[u][v];
        path.add(u);
    }
    System.out.println("Path: " + path);
}
```

## REAL_WORLD
**Dense vs Sparse Graphs**
Is Floyd-Warshall always the best for All-Pairs Shortest Path? No!
If the graph has thousands of vertices but very few edges (A Sparse Graph), $O(V^3)$ will take far too long.
Instead, it is significantly faster to simply run **Dijkstra's Algorithm $V$ times** (once from every node). 
Running Dijkstra $V$ times yields a total time complexity of $O(V \times E \log V)$. 
If the graph only has $E = 100$ edges and $V = 1000$ vertices, $1000 \times 100 \times 10$ operations is massively faster than $1000^3$ (One Billion operations). 
Floyd-Warshall should only be used for dense graphs, or when negative edge weights are present (where Dijkstra fails).

## INTERVIEW
**Q: You initialize your matrix with `Integer.MAX_VALUE` to represent "Infinity". When the Floyd-Warshall algorithm runs, the application crashes with negative number errors! Why shouldn't you use `Integer.MAX_VALUE` in Floyd-Warshall?**
A: **Integer Overflow.**
Look at the core condition of Floyd-Warshall: `dist[i][k] + dist[k][j] < dist[i][j]`.
If there is no direct connection between `i` and `k`, `dist[i][k]` is Infinity. If you set it to `Integer.MAX_VALUE` (which is `2,147,483,647`), and you add literally anything to it, the binary math overflows the 32-bit limit and wraps around to `-2,147,483,648`!
Suddenly, your math condition evaluates `(-2 Billion) < dist[i][j]`, which is true! The algorithm incorrectly updates the matrix with negative garbage data.
*The Fix:* Always initialize Infinity as a safely large number that will not overflow when doubled (e.g., `999999`), or explicitly add an `if (dist[i][k] != Integer.MAX_VALUE)` check before performing the addition.
