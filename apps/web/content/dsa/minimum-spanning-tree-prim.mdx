## WHY
Kruskal's Algorithm solves the Minimum Spanning Tree (MST) problem by sorting all edges globally ($O(E \log E)$) and using a Union-Find data structure to pick edges randomly across the map, slowly merging them together.
If the graph is incredibly dense (millions of edges), sorting all of them is too slow. 
**Prim's Algorithm** takes a completely different approach. Instead of looking at the graph globally, it starts at a single random city and grows outward like a puddle of water. It maintains a Priority Queue (Min-Heap) to constantly ask: *"Of all the cities adjacent to my currently connected puddle, which city is the cheapest to annex?"*
For dense graphs, Prim's mathematically outperforms Kruskal's!

## THEORY
### The Growing Tree
1. Pick any random node to be the start of your MST. Mark it visited.
2. Throw all edges connected to that node into a Min-Heap.
3. Pop the cheapest edge from the Min-Heap. If the destination node is *unvisited*, permanently weld that edge into your MST! Mark the destination node as visited.
4. Throw all the new outgoing edges from that newly annexed node into the Min-Heap.
5. Repeat until all $V$ nodes are visited!

## CODE

### Level 1 — Prim's Algorithm Setup
We need an Adjacency List to easily find the neighbors of the cities we annex, and a `visited` array to ensure we don't accidentally create cycles by reconnecting to cities already in our MST!

```java
import java.util.*;

class Edge {
    int dest;
    int weight;
    public Edge(int dest, int weight) {
        this.dest = dest;
        this.weight = weight;
    }
}
```

### Level 2 — The Priority Queue Approach
Notice how similar this code looks to Dijkstra's Algorithm. Both algorithms greedily extract the minimum value from a Heap.
However, Dijkstra sorts the Heap by the *Total Cumulative Distance* from the start. Prim's sorts the Heap by the *Absolute Value of the Single Edge*. Prim's doesn't care how far you are from the start; it only cares about the cheapest bridge to cross right now!

```java
public class PrimMST {
    
    public int findMSTWeight(int numNodes, List<List<Edge>> adjList) {
        boolean[] visited = new boolean[numNodes];
        
        // Min-Heap sorting strictly by the raw weight of the Edge!
        PriorityQueue<Edge> minHeap = new PriorityQueue<>(
            (a, b) -> Integer.compare(a.weight, b.weight)
        );

        int totalMSTWeight = 0;
        int edgesAdded = 0;

        // 1. Start arbitrarily at Node 0. Mark it visited!
        visited[0] = true;

        // 2. Add all of Node 0's outgoing edges into the Heap
        for (Edge edge : adjList.get(0)) {
            minHeap.add(edge);
        }

        // 3. Grow the MST!
        while (!minHeap.isEmpty() && edgesAdded < numNodes - 1) {
            
            // Greedily extract the absolute cheapest edge currently available
            Edge cheapest = minHeap.poll();

            // CRITICAL: Does this edge point to a city we ALREADY annexed?
            // If so, using this edge would create a Cycle! Ignore it!
            if (visited[cheapest.dest]) continue;

            // 4. Annex the city!
            visited[cheapest.dest] = true;
            totalMSTWeight += cheapest.weight;
            edgesAdded++;

            // 5. Expand the frontier! Add all edges from this new city to the Heap!
            for (Edge nextEdge : adjList.get(cheapest.dest)) {
                // Optimization: Don't add edges pointing back into the MST
                if (!visited[nextEdge.dest]) {
                    minHeap.add(nextEdge);
                }
            }
        }

        // Check if the graph was disconnected!
        if (edgesAdded != numNodes - 1) return -1;

        return totalMSTWeight;
    }
}
```

### Level 3 — Time Complexity
Prim's Time Complexity using a standard Binary Min-Heap is **$O(E \log E)$**, which is mathematically equivalent to $O(E \log V)$ because $E \le V^2$.
- Every edge is added to the Heap at most once.
- Inserting into a Heap takes $\log E$.
- Therefore, $E$ insertions take $O(E \log E)$.
While this looks identical to Kruskal's $O(E \log E)$ sort time, Prim's is often faster in reality for dense graphs because we can stop the `while` loop early as soon as `edgesAdded == V - 1`, skipping the processing of millions of remaining edges in the Heap!

## REAL_WORLD
**Fibonacci Heaps (The Ultimate Optimization)**
In competitive programming and standard Java, we use a standard Binary Priority Queue, giving Prim's a time of $O(E \log V)$.
However, in massive telecom grid computations, engineers use an advanced data structure called a **Fibonacci Heap**. 
A Fibonacci Heap performs `insert()` in mathematically amortized $O(1)$ time, rather than $O(\log N)$ time! This single change reduces the total time complexity of Prim's Algorithm to **$O(E + V \log V)$**. For dense graphs where $E$ is near $V^2$, this optimization shaves hours off server computation time, cementing Prim's as the undeniable king of dense Minimum Spanning Trees!

## INTERVIEW
**Q: Dijkstra's Algorithm and Prim's Algorithm both use a `visited` array and a `PriorityQueue`. What is the fundamental difference in what they store in their Priority Queues?**
A: **The Sorting Metric.**
- **Dijkstra** is trying to find the shortest *path* from the origin. The Priority Queue stores the **Cumulative Distance** (`dist[u] + weight`). It asks: *"What is the cheapest total path from the Start to City X?"*
- **Prim's** is trying to build a global Minimum Spanning Tree. The Priority Queue stores the **Raw Edge Weight** (`weight`). It asks: *"Regardless of where the Start node was, what is the absolute cheapest physical bridge that connects our existing blob to a new unvisited city?"*
Because of this distinction, Dijkstra builds paths that shoot out aggressively towards the target, while Prim's builds a sprawling, tree-like web that organically overtakes the entire map.
