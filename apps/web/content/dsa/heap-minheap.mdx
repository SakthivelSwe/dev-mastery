## WHY
Imagine you are building the backend for a massive hospital emergency room. Patients arrive randomly. However, doctors don't treat patients in the order they arrived (FIFO like a standard Queue). They treat the patient with the highest priority (e.g., heart attack treated before a sprained ankle).
If you store the patients in an Array and sort it every time a new patient arrives, it takes $O(N \log N)$ time. For millions of patients, the hospital system will crash.
A **Heap** (specifically a Min-Heap or Max-Heap) is the ultimate data structure for Priority Queues. It guarantees that extracting the most important element takes $O(\log N)$ time, and inserting a new element takes $O(\log N)$ time!

## THEORY
### The Structure
A Min-Heap is a Complete Binary Tree with one sacred mathematical rule:
**The Heap Property:** The value of every parent node is always LESS THAN OR EQUAL TO the value of its children.
Because of this rule, the absolute smallest value in the entire tree is guaranteed to be sitting at the very top (the root).

### Arrays as Trees
Just like a Segment Tree, a Heap is a Complete Binary Tree, meaning it is perfectly balanced from left to right. Because there are no "gaps" in the tree, we don't need `Node` classes with `left` and `right` pointers! We can store the entire tree flatly inside a standard Array!
If a parent is at index `i`:
- Left Child = `2 * i + 1`
- Right Child = `2 * i + 2`
- Parent of `i` = `(i - 1) / 2`

## CODE

### Level 1 — Initialization
We build the Heap using a dynamic array (like `ArrayList` in Java) so it can grow infinitely.

```java
import java.util.ArrayList;

public class MinHeap {
    private ArrayList<Integer> heap;

    public MinHeap() {
        heap = new ArrayList<>();
    }

    // Helper methods for array math
    private int parent(int i) { return (i - 1) / 2; }
    private int leftChild(int i) { return 2 * i + 1; }
    private int rightChild(int i) { return 2 * i + 2; }

    // Helper to swap two elements in the array
    private void swap(int i, int j) {
        int temp = heap.get(i);
        heap.set(i, heap.get(j));
        heap.set(j, temp);
    }
}
```

### Level 2 — Insertion (Heapify Up)
When a new patient arrives, we push them to the very end of the array (the bottom-leftmost leaf of the tree).
Then, we compare them to their parent. If they are smaller than their parent, the Heap Property is violated! We swap them with their parent. We repeat this process, "bubbling up" the tree until they settle into the correct spot.

```java
// Time Complexity: O(log N)
public void insert(int value) {
    // 1. Add to the end of the array
    heap.add(value); 
    
    // 2. Bubble it up to restore the Heap Property!
    heapifyUp(heap.size() - 1);
}

private void heapifyUp(int index) {
    // While we aren't the root, AND we are smaller than our parent...
    while (index > 0 && heap.get(index) < heap.get(parent(index))) {
        // Swap with the parent!
        swap(index, parent(index));
        // Update our index to the parent's location, and check again!
        index = parent(index);
    }
}
```

### Level 3 — Extraction (Heapify Down)
We want the smallest element. It's guaranteed to be at index `0`. We take it.
But now index `0` is empty! The tree is broken.
We take the *very last* element in the array and move it to index `0` to plug the hole.
Now the root has a massive number in it, violating the Heap Property. We compare it to its two children, find the smallest child, and swap them. We repeat this, "bubbling down" until the tree is fixed!

```java
// Time Complexity: O(log N)
public int extractMin() {
    if (heap.isEmpty()) throw new RuntimeException("Heap is empty!");
    
    // 1. Grab the minimum value to return later
    int min = heap.get(0);

    // 2. Move the very last element to the root
    int lastElement = heap.remove(heap.size() - 1);
    
    if (!heap.isEmpty()) {
        heap.set(0, lastElement);
        // 3. Bubble it down to restore the Heap Property!
        heapifyDown(0);
    }

    return min;
}

private void heapifyDown(int index) {
    int smallest = index;
    int left = leftChild(index);
    int right = rightChild(index);

    // If the left child exists and is smaller than us...
    if (left < heap.size() && heap.get(left) < heap.get(smallest)) {
        smallest = left;
    }

    // If the right child exists and is even smaller...
    if (right < heap.size() && heap.get(right) < heap.get(smallest)) {
        smallest = right;
    }

    // If we are NOT the smallest, we must swap with the smallest child!
    if (smallest != index) {
        swap(index, smallest);
        // Recursively bubble down from the new position!
        heapifyDown(smallest);
    }
}
```

### Level 4 — Build Heap in $O(N)$
If you are given an unsorted array `[5, 2, 8, 1, 9]`, and you insert the elements one by one into an empty heap using `insert()`, it will take $O(N \log N)$ time.
However, there is a magical algorithm called **Floyd's Build-Heap Algorithm** that converts an unsorted array into a perfect Heap in strictly $O(N)$ time!
You dump the array directly into the tree. Then, starting from the *bottom* of the tree and working up to the root, you call `heapifyDown()` on every parent node. Because the bottom nodes travel very short distances, the complex math proves this runs in linear time! This is the foundation of the famous "HeapSort" algorithm.

## REAL_WORLD
**Dijkstra's Algorithm**
In network routing (e.g., Google Maps finding the shortest path from New York to LA), Dijkstra's Algorithm explores cities. It must always explore the city with the shortest known distance next.
If it searches a standard array for the shortest distance, the algorithm is $O(V^2)$ and takes forever. By using a Min-Heap to store the cities, finding the next city drops to $O(\log V)$, making Google Maps blazing fast!

## INTERVIEW
**Q: A Max-Heap is guaranteed to have the largest value at the root. Does a Max-Heap also guarantee that a Left-to-Right reading of the array is perfectly sorted in descending order? If not, how do you search for a specific number (like 42) in the Heap?**
A: **No, Heaps are NOT sorted arrays, and they cannot be searched efficiently!**
The Heap Property only governs the vertical Parent-Child relationship (Parent > Child). It enforces absolutely zero relationship between siblings. The left child can be larger than the right child, or the right child can be larger than the left. A valid Max-Heap array might look like `[100, 19, 36, 17, 3, 25, 1, 2, 7]`. It is not sorted!
Because it is not a Binary Search Tree (where Left is always smaller, Right is always bigger), you cannot use Binary Search. If you need to find if the number `42` exists in a Heap, you are forced to do a linear scan $O(N)$ through the entire array. Heaps are strictly designed for finding the *Extreme* values (Min or Max) rapidly, not for searching!
