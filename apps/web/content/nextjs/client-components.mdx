## WHY
In the Next.js App Router, all components are Server Components by default. They run on the server, output HTML, and send zero JavaScript to the browser. However, Server Components cannot use React state (`useState`), lifecycle hooks (`useEffect`), or handle DOM events (`onClick`).
When you need interactivity, browser APIs, or state management, you must explicitly opt-in to creating a **Client Component**.

## THEORY
### Server vs Client Boundaries
The `'use client'` directive marks the boundary between the server and the client.
- A component without the directive is a Server Component.
- Adding `'use client'` at the very top of a file makes it a Client Component.
- **The "Waterfall" Rule:** Any component imported *into* a Client Component automatically becomes a Client Component too! The boundary flows downward.

### The Misconception
"Client Components" are actually rendered on the server during SSR (generating initial HTML for fast load times), and *then* they are "hydrated" (made interactive) on the client using JavaScript. They run in both places!

## CODE

### Defining a Client Component
```tsx
// app/components/Counter.tsx
'use client'; // This directive must be at the very top!

import { useState } from 'react';

export default function Counter() {
  const [count, setCount] = useState(0);

  return (
    <div>
      <p>You clicked {count} times</p>
      {/* Event listeners require Client Components */}
      <button onClick={() => setCount(count + 1)}>Click me</button>
    </div>
  );
}
```

### The Boundary (Mixing Server and Client)
```tsx
// app/page.tsx (Server Component)
import Counter from './components/Counter'; // Client Component
import DatabaseInfo from './components/DatabaseInfo'; // Server Component

export default function HomePage() {
  return (
    <main>
      <h1>Welcome</h1>
      {/* You can render a Client Component inside a Server Component! */}
      <Counter />
      <DatabaseInfo />
    </main>
  );
}
```

### Passing Server Components as Children
You **cannot** import a Server Component directly into a Client Component (it would force the Server Component to become a Client Component). But you *can* pass a Server Component as a `children` prop!

```tsx
// app/components/ClientWrapper.tsx
'use client';
import { useState } from 'react';

// This is a Client Component
export default function CollapsiblePanel({ children }: { children: React.ReactNode }) {
  const [isOpen, setIsOpen] = useState(false);
  return (
    <div>
      <button onClick={() => setIsOpen(!isOpen)}>Toggle</button>
      {isOpen && <div>{children}</div>}
    </div>
  );
}
```

```tsx
// app/page.tsx (Server Component)
import CollapsiblePanel from './components/ClientWrapper';
import HeavyServerChart from './components/HeavyServerChart'; // Connects to DB!

export default function Page() {
  return (
    <CollapsiblePanel>
      {/* HeavyServerChart runs on the server, its HTML is passed down as 'children'! */}
      <HeavyServerChart />
    </CollapsiblePanel>
  );
}
```

## REAL_WORLD
**When to use `'use client'`:**
- Need `useState`, `useEffect`, `useReducer`, `useRef`.
- Need DOM event listeners (`onClick`, `onChange`).
- Using browser APIs (`window`, `document`, `localStorage`).
- Using custom React hooks that depend on state/effects.
- Using Context Providers (e.g., Redux, ThemeProvider).

**Pushing the Boundary Down**
A common mistake is putting `'use client'` at the top of a page layout. This makes your entire application a Client Component, destroying the performance benefits of Next.js 13.
Always push the `'use client'` directive as far down the component tree as possible. If only the "Like" button needs state, extract *just* the `<LikeButton>` into a Client Component, leaving the rest of the `<Article>` as a Server Component.

## INTERVIEW
**Q: You get the error: `"Event handlers cannot be passed to Client Component props. If you need interactivity, consider converting part of this to a Client Component."` Why did this happen?**

A: You tried to pass a function (like an `onClick` handler) from a Server Component down to a Client Component.

Server Components run on the server. Client Components run in the browser. You cannot serialize a JavaScript function and send it over the network from the server to the client.

**How to fix:**
Move the state and the event handler *into* the Client Component itself. The Server Component should only pass serializable data (strings, numbers, simple objects) as props to Client Components.
