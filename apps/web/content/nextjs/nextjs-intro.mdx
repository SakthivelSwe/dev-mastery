## WHY
React is a fantastic library for building user interfaces, but a plain React app has serious production challenges:
- **SEO:** React renders client-side. Search engine bots see an empty HTML page with just a `<div id="root">`. Your content is invisible to Google.
- **Performance:** Users download megabytes of JavaScript before seeing anything. First Contentful Paint is slow.
- **Routing:** You must install and configure React Router separately.
- **API:** You need a separate backend service for server-side logic.

**Next.js** solves all of these. It is a full-stack React framework built by Vercel that provides:
- Server-Side Rendering (SSR) and Static Site Generation (SSG) for SEO
- File-system-based routing (no configuration)
- Built-in API routes
- Image optimization
- Automatic code splitting

## THEORY
### The Framework Advantage
Next.js is "React with opinions." It makes the right architectural choices for you:
- **Rendering Strategy:** SSR, SSG, ISR, or Client-Side per page
- **Routing:** Folder structure = URL structure
- **Full-Stack:** React components + Node.js API routes in one project

### App Router vs Pages Router
Next.js 13+ introduced the **App Router** (`app/` directory) alongside the legacy **Pages Router** (`pages/` directory). The App Router is the future — it supports React Server Components, streaming, and Suspense natively.

## CODE

### Creating a Next.js Project
```bash
# Create with the official CLI
npx create-next-app@latest my-app

# Prompts:
# ✓ TypeScript? Yes
# ✓ ESLint? Yes
# ✓ Tailwind CSS? Yes
# ✓ src/ directory? No
# ✓ App Router? Yes ← Use the new App Router!
# ✓ Import alias @/*? Yes

cd my-app
npm run dev
# Server running at http://localhost:3000
```

### App Router File Structure
```
my-app/
├── app/
│   ├── layout.tsx        # Root layout (persists across pages)
│   ├── page.tsx          # Home page: renders at /
│   ├── about/
│   │   └── page.tsx      # About page: renders at /about
│   ├── blog/
│   │   ├── page.tsx      # Blog index: renders at /blog
│   │   └── [slug]/
│   │       └── page.tsx  # Dynamic route: /blog/my-post-title
│   └── api/
│       └── users/
│           └── route.ts  # API route: POST/GET /api/users
├── public/               # Static files (images, favicons)
├── next.config.js        # Next.js configuration
└── package.json
```

### Your First Pages
```tsx
// app/page.tsx — Home page (Server Component by default!)
export default function HomePage() {
  return (
    <main>
      <h1>Welcome to Next.js</h1>
      <p>This HTML was generated on the SERVER before being sent to the browser!</p>
    </main>
  );
}

// app/about/page.tsx — About page
export default function AboutPage() {
  return <h1>About Us</h1>;
}
// Accessible at: http://localhost:3000/about — no routing config needed!
```

### App Router Layout (Shared UI)
```tsx
// app/layout.tsx — Wraps EVERY page in the app
import { Inter } from 'next/font/google';
import './globals.css';

const inter = Inter({ subsets: ['latin'] });

export const metadata = {
  title: 'My App',
  description: 'Built with Next.js 14',
};

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="en">
      <body className={inter.className}>
        <nav>Navigation bar</nav>
        {children}          {/* Page content renders here */}
        <footer>Footer</footer>
      </body>
    </html>
  );
}
```

## REAL_WORLD
**Who Uses Next.js?**
Next.js powers some of the most trafficked websites on the internet:
- **Vercel** (the creators — obviously!)
- **Notion** (document collaboration)
- **Hulu** (streaming service)
- **TikTok** (the web version)
- **Twitch** (game streaming)
- **GitHub** (Copilot Chat interface)

The combination of React familiarity + SSR + full-stack capabilities makes Next.js the dominant choice for production React applications.

## INTERVIEW
**Q: Why would you choose Next.js over a plain Create React App (CRA) or Vite React project for a marketing website that also has an e-commerce store?**

A: For a marketing + e-commerce site, Next.js provides critical capabilities that CRA/Vite can't:

1. **SEO is mandatory for marketing:** Search engines need to index product descriptions, blog posts, and landing pages. CRA/Vite ship empty HTML shells — the JS renders content client-side. Google technically indexes JavaScript, but it's slower and less reliable. Next.js SSG/SSR sends fully-rendered HTML in the first response, perfect for crawlers.

2. **Performance for e-commerce:** Slow sites lose sales. Next.js SSG pre-generates product pages at build time — they load instantly from a CDN. No waiting for React to boot.

3. **API routes for backend logic:** Processing payments, managing sessions, and cart logic needs server-side code. Next.js API routes let you handle these in the same project. CRA requires a separate Express/Node backend.

4. **Image optimization:** Next.js `<Image>` component automatically converts images to WebP, serves correct sizes for each device, and lazy-loads — e-commerce is image-heavy, this is a significant performance win.

CRA/Vite are fine for internal dashboards or apps behind authentication where SEO doesn't matter. For public-facing sites, Next.js is the clear choice.
