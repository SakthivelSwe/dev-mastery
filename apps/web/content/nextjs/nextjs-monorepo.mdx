## WHY
As your company scales, you might have a marketing site (`example.com`), a web app (`app.example.com`), and an admin dashboard (`admin.example.com`). All three are Next.js applications, and they all need to share the same UI components (buttons, navbars), Tailwind configuration, and TypeScript types.
If you use separate repositories, sharing code requires publishing packages to npm, which is slow and tedious. A **Monorepo** (using tools like Turborepo or Nx) allows multiple Next.js apps to live in the same repository and seamlessly import local shared packages without publishing them.

## THEORY
### Workspaces and Tooling
- **Package Manager Workspaces (npm/yarn/pnpm):** Tells the package manager to link local folders together. If App A imports `@repo/ui`, it links to the local `packages/ui` folder instead of downloading from npm.
- **Turborepo:** A high-performance build system for JavaScript monorepos. It understands the dependency graph (App A depends on Package B) and caches build outputs. If you change App A, Turborepo knows it doesn't need to rebuild App C.

## CODE

### Standard Turborepo Structure
```text
my-monorepo/
├── apps/
│   ├── web/               ← Next.js App (marketing)
│   └── docs/              ← Next.js App (documentation)
├── packages/
│   ├── ui/                ← Shared React components
│   ├── config-tailwind/   ← Shared Tailwind config
│   ├── config-typescript/ ← Shared tsconfig.json
│   └── database/          ← Shared Prisma client
├── package.json           ← Defines workspaces
└── turbo.json             ← Defines build pipelines
```

### The Transpile Packages Feature
By default, Next.js only compiles the code *inside* its own directory. If you import a React component from a shared local package (`packages/ui`), Next.js will crash because the shared package isn't compiled.
Next.js 13+ introduced `transpilePackages` to solve this natively.

```javascript
// apps/web/next.config.js
/** @type {import('next').NextConfig} */
const nextConfig = {
  // Tell Next.js to compile these shared packages alongside the app!
  transpilePackages: ["@repo/ui", "@repo/database"],
};

module.exports = nextConfig;
```

### Creating a Shared UI Package
```json
// packages/ui/package.json
{
  "name": "@repo/ui",
  "version": "1.0.0",
  "main": "./index.tsx",
  "types": "./index.tsx",
  "dependencies": {
    "react": "^18.2.0"
  }
}
```

```tsx
// packages/ui/index.tsx
export * from './Button';
export * from './Card';

// packages/ui/Button.tsx
export function Button({ children }: { children: React.ReactNode }) {
  return <button className="bg-blue-500 text-white p-2 rounded">{children}</button>;
}
```

### Consuming the Shared Package
```json
// apps/web/package.json
{
  "name": "web",
  "dependencies": {
    "next": "^14.0.0",
    "react": "^18.2.0",
    "@repo/ui": "*"      // The * tells it to use the local workspace version
  }
}
```

```tsx
// apps/web/app/page.tsx
import { Button } from '@repo/ui'; // Imports directly from packages/ui !

export default function Home() {
  return <Button>Click Me</Button>;
}
```

## REAL_WORLD
**Sharing Tailwind Configs**
Sharing CSS configurations is notoriously tricky in monorepos. If `@repo/ui` uses Tailwind classes, the Next.js app needs to scan the UI package's files to generate the CSS.

```javascript
// apps/web/tailwind.config.js
// Import the shared base config
const sharedConfig = require('@repo/config-tailwind/tailwind.config.js');

module.exports = {
  // Spread the shared config
  ...sharedConfig,
  content: [
    // Scan the app's files
    './app/**/*.{js,ts,jsx,tsx,mdx}',
    // CRITICAL: Tell Tailwind to scan the shared UI package for classes!
    '../../packages/ui/**/*.{js,ts,jsx,tsx}',
  ],
};
```

## INTERVIEW
**Q: You have a Next.js app (`apps/web`) and an Express API (`apps/api`) in a Turborepo monorepo. Both depend on a shared Prisma database client (`packages/db`). When you deploy the Next.js app to Vercel, how does Vercel know how to build it since the repository root is not a Next.js app?**

A: Vercel has native support for monorepos, specifically Turborepo.

When you import a GitHub repository into Vercel, you specify the **Root Directory** as `apps/web`. Vercel understands that it is inside a monorepo.

During deployment, Vercel will:
1. Clone the *entire* monorepo (not just the `apps/web` folder).
2. Install all dependencies at the root workspace level, ensuring `packages/db` is linked correctly.
3. Automatically configure the build command to `npx turbo run build --filter=web`.
4. Deploy only the compiled output of the `apps/web` directory to the edge network.

This allows you to deploy individual apps from a massive monorepo independently, while Vercel handles the complex dependency linking behind the scenes.
