## WHY
Without automated tests, every new feature you add to your application has the potential to break existing functionality. Testing Next.js applications involves multiple layers: Unit testing isolated components, Integration testing complex Server/Client component interactions, and End-to-End (E2E) testing the fully built application in a real browser.

## THEORY
### The Testing Pyramid in Next.js
1. **Unit Tests (Jest / Vitest):** Testing pure functions and isolated UI components. Fast to run, but doesn't guarantee the app works together.
2. **Component/Integration Tests (React Testing Library):** Testing how components render and respond to user interactions (clicks, typing).
3. **End-to-End Tests (Cypress / Playwright):** Testing the full flow (Browser → Server Component → Database). Slower, but provides the highest confidence.

## CODE

### Unit Testing a Component (Jest + React Testing Library)
Client components are tested just like standard React components.

```tsx
// components/Button.tsx
'use client'
export default function Button({ label, onClick }: { label: string, onClick: () => void }) {
  return <button onClick={onClick} className="btn-blue">{label}</button>;
}
```

```tsx
// __tests__/Button.test.tsx
import { render, screen } from '@testing-library/react';
import userEvent from '@testing-library/user-event';
import Button from '@/components/Button';

describe('Button', () => {
  it('renders the label and handles clicks', async () => {
    const mockOnClick = jest.fn();
    render(<Button label="Submit" onClick={mockOnClick} />);
    
    // Assert rendering
    const button = screen.getByRole('button', { name: 'Submit' });
    expect(button).toBeInTheDocument();
    
    // Assert interaction
    await userEvent.click(button);
    expect(mockOnClick).toHaveBeenCalledTimes(1);
  });
});
```

### Mocking Next.js Router
When testing components that use `next/navigation`, you must mock the router.

```tsx
// __mocks__/next/navigation.js
module.exports = {
  useRouter: () => ({
    push: jest.fn(),
    replace: jest.fn(),
    prefetch: jest.fn(),
  }),
  usePathname: () => '/current-path',
  useSearchParams: () => new URLSearchParams('?query=test'),
};
```

### End-to-End Testing (Playwright)
Because Server Components involve database queries and server-side rendering, unit testing them with Jest is notoriously difficult. The modern best practice is to test Server Components using E2E tools like Playwright.

```typescript
// e2e/login.spec.ts
import { test, expect } from '@playwright/test';

test('user can log in and view dashboard', async ({ page }) => {
  // 1. Navigate to the login page
  await page.goto('/login');
  
  // 2. Fill out the form
  await page.fill('input[name="email"]', 'test@example.com');
  await page.fill('input[name="password"]', 'password123');
  
  // 3. Submit the form (Triggering a Server Action!)
  await page.click('button[type="submit"]');
  
  // 4. Assert that the server redirected us and loaded the DB data
  await expect(page).toHaveURL('/dashboard');
  await expect(page.locator('h1')).toContainText('Welcome back, Test User');
});
```

## REAL_WORLD
**The Shift to E2E Testing**
With the introduction of React Server Components, the traditional React testing strategy (using Jest to mount a component and mock the API responses) is becoming less practical. Server components often connect directly to databases or the file system. Mocking an entire Prisma database client inside Jest is painful and brittle.

Instead, the community is shifting heavily towards **Playwright**. You spin up a test database, start the Next.js server (`npm run build && npm start`), and run Playwright tests against it. This tests the *actual* Server Components, the *actual* Server Actions, and the *actual* database queries in one go, providing massive confidence that the app works.

## INTERVIEW
**Q: You want to write a Jest test for a Server Component that uses `cookies()` from `next/headers` to check if a user is logged in. How do you approach this?**

A: It is highly recommended **not** to unit test Server Components with Jest.

Server Components rely on the Next.js server environment (Webpack, request context, headers, cookies). Jest is a Node.js test runner that doesn't natively understand Next.js server contexts. While it is technically possible to aggressively mock `next/headers` and `next/cache`, the test becomes a test of your mocks, not a test of your code.

**The correct approach:**
Use an E2E testing framework like Playwright or Cypress. These tools hit the actual running application, meaning the `cookies()` function will behave exactly as it does in production, reading the actual HTTP headers sent by the browser during the test.
