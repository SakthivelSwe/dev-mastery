## WHY
Before Next.js 13 introduced the App Router (`app/`), the entire Next.js ecosystem relied on the Pages Router (`pages/`). While the App Router is the modern standard, millions of existing Next.js applications still use the Pages Router. If you get a job working on an existing Next.js codebase, you *must* understand the Pages Router. Furthermore, understanding the limitations of the Pages Router helps you appreciate *why* the App Router was created.

## THEORY
### File-System Routing in `pages/`
The fundamental concept is the same: the file system dictates the URL. However, the conventions and data fetching methods are entirely different from the App Router.
- `pages/index.tsx` → `/`
- `pages/about.tsx` → `/about`
- `pages/blog/[slug].tsx` → `/blog/my-post`

### Data Fetching
In the App Router, you use standard async/await React Server Components. In the Pages Router, components are *not* async. Instead, you must export specific data-fetching functions:
- `getServerSideProps`: Fetches data on every request (SSR).
- `getStaticProps`: Fetches data at build time (SSG).
- `getStaticPaths`: Generates dynamic paths at build time.

## CODE

### Basic Page and Routing
```tsx
// pages/about.tsx
// Renders at the /about URL
export default function AboutPage() {
  return <div>About Us</div>;
}
```

### Server-Side Rendering (SSR)
If a page exports `getServerSideProps`, Next.js will render the page on the server for *every* request.

```tsx
// pages/user/[id].tsx
import { GetServerSideProps } from 'next';

interface UserPageProps {
  user: { name: string; email: string };
}

export default function UserProfile({ user }: UserPageProps) {
  return <h1>{user.name}'s Profile</h1>;
}

// This function runs ONLY on the server, before the component renders
export const getServerSideProps: GetServerSideProps = async (context) => {
  const id = context.params?.id;
  
  // Safe to use server-side secrets here (like DB passwords)
  const res = await fetch(`https://api.example.com/users/${id}`);
  const user = await res.json();

  if (!user) {
    return { notFound: true }; // Triggers a 404 page
  }

  // The returned props are passed to the component above
  return {
    props: { user },
  };
};
```

### Static Site Generation (SSG)
If a page exports `getStaticProps`, Next.js will fetch the data *once* during `npm run build` and generate a static HTML file.

```tsx
// pages/blog.tsx
import { GetStaticProps } from 'next';

export default function BlogList({ posts }: { posts: any[] }) {
  return (
    <ul>
      {posts.map(post => <li key={post.id}>{post.title}</li>)}
    </ul>
  );
}

// Runs at BUILD TIME, not at request time!
export const getStaticProps: GetStaticProps = async () => {
  const posts = await fetch('https://api.example.com/posts').then(r => r.json());
  
  return {
    props: { posts },
  };
};
```

### `_app.tsx` and `_document.tsx`
The Pages Router requires two special files to control the global layout and HTML structure.
```tsx
// pages/_app.tsx
// This component wraps EVERY page. Used for global CSS, Providers, etc.
import type { AppProps } from 'next/app'
import '../styles/globals.css'

export default function MyApp({ Component, pageProps }: AppProps) {
  return <Component {...pageProps} />
}

// pages/_document.tsx
// Only runs on the server. Used to customize the <html> and <body> tags.
import { Html, Head, Main, NextScript } from 'next/document'

export default function Document() {
  return (
    <Html lang="en">
      <Head />
      <body>
        <Main />
        <NextScript />
      </body>
    </Html>
  )
}
```

## REAL_WORLD
**The App Router Migration**
Many companies run "hybrid" Next.js applications, utilizing both `pages/` and `app/` directories simultaneously. Next.js supports this natively! 
You can incrementally migrate a large app by moving one route at a time from `pages/blog.tsx` to `app/blog/page.tsx`.

## INTERVIEW
**Q: What is the main structural limitation of the Pages Router that led the Next.js team to create the App Router?**

A: **Nested Layouts and State Preservation.**

In the Pages Router, `_app.tsx` is the only global layout file. If you wanted a specific layout (like a sidebar) *only* for the `/dashboard/*` routes, you had to hack it using the "Per-Page Layouts" pattern (attaching a `getLayout` property to the page component). 

Furthermore, when navigating between pages in the Pages Router, the entire page component tree is unmounted and remounted. You couldn't easily have a persistent navigation sidebar that maintained its scroll position or React state while the main content area changed.

The App Router solves this natively with `layout.tsx` files that can be nested at any folder level, and these layouts purposefully persist their state and DOM across child navigations.
