## WHY
Traditional React components are "client components" — they render in the browser using JavaScript. This means:
- Users must download ALL your React code before seeing anything
- Database queries, API calls, and secrets cannot be done safely in the browser
- Sensitive data (API keys, business logic) is exposed to the client

**React Server Components (RSC)** are a revolutionary paradigm shift. They render on the server at request time. The server fetches data, renders the HTML, and sends only static HTML (no JavaScript!) to the browser. Zero JavaScript bundle size. Zero data loading states. Zero API endpoints needed for server data.

## THEORY
### Server Components vs Client Components
In Next.js 13+ App Router, ALL components are Server Components by default. You opt into Client Components with `'use client'`.

**Server Components:**
- Run only on the server (never in browser)
- Can access databases, file system, environment secrets directly
- Cannot use `useState`, `useEffect`, event handlers, or browser APIs
- Zero JavaScript added to the client bundle!

**Client Components:**
- Run in the browser (and also SSR'd for initial HTML)
- Can use all React hooks, event handlers, browser APIs
- Add JavaScript to the client bundle
- Cannot access server-side resources directly

## CODE

### Server Component (Default — No 'use client')
```tsx
// app/users/page.tsx — Server Component!
// This runs ONLY on the server. Users never see this code.

// You can query the database DIRECTLY — no API needed!
import { db } from '@/lib/database';

export default async function UsersPage() {
  // Async/await works directly in Server Components!
  // This fetch happens on the SERVER — no loading spinner needed!
  const users = await db.query('SELECT id, name, email FROM users LIMIT 20');

  return (
    <main>
      <h1>Users ({users.length})</h1>
      <ul>
        {users.map(user => (
          <li key={user.id}>{user.name} — {user.email}</li>
        ))}
      </ul>
    </main>
  );
  // The user receives fully rendered HTML with the list!
  // Zero loading state. Zero skeleton. Zero API request from the browser.
}
```

### Client Component (Opt-in for Interactivity)
```tsx
// app/components/LikeButton.tsx
'use client';  // This directive makes it a Client Component!

import { useState } from 'react';

export default function LikeButton({ initialCount }: { initialCount: number }) {
  const [likes, setLikes] = useState(initialCount);

  return (
    <button onClick={() => setLikes(l => l + 1)}>
      ❤️ {likes}
    </button>
  );
}
```

### Mixing Server and Client Components
```tsx
// app/post/[id]/page.tsx — Server Component
import { db } from '@/lib/database';
import LikeButton from '@/components/LikeButton';  // Client Component!
import CommentSection from '@/components/CommentSection';  // Client Component!

export default async function PostPage({ params }: { params: { id: string } }) {
  // Server data fetching — fast, secure, direct DB access
  const post = await db.posts.findUnique({ where: { id: params.id } });
  const likes = await db.likes.count({ where: { postId: params.id } });

  return (
    <article>
      <h1>{post.title}</h1>          {/* Server rendered */}
      <p>{post.content}</p>          {/* Server rendered */}
      
      {/* LikeButton is a Client Component — it adds interactivity! */}
      {/* You pass SERVER data (likes) as props to the Client Component */}
      <LikeButton initialCount={likes} />
      
      <CommentSection postId={post.id} />
    </article>
  );
}
```

### The Component Boundary Rule
```tsx
// ❌ WRONG: Can't import a client component in a server component if it uses server-only things
// 'use client';  ← If you add this, you can't use db directly!
// import { db } from '@/lib/database';  ← This would fail!

// ✅ RIGHT: Keep data fetching in Server Components, pass as props to Client Components
// Server Component fetches, Client Component renders interactively
```

## REAL_WORLD
**The Performance Impact**
Before Server Components: A blog post page sends ~200KB of React JavaScript to the browser. The browser must parse and execute it before showing any content.

After Server Components: The blog post page sends ~5KB of HTML (the rendered content). Zero JavaScript for the content itself. Only the LikeButton and CommentSection (Client Components) add a small amount of JavaScript.

Result:
- Lighthouse Performance score: 45 → 95
- Time to Interactive: 4.2s → 0.8s
- Bundle size: 200KB → 15KB

## INTERVIEW
**Q: A Server Component needs to display a "Like" button that users can click. But `useState` doesn't work in Server Components. How do you implement this?**

A: You use **composition** — keep the data fetching in the Server Component, and extract just the interactive part into a Client Component.

```tsx
// Server Component: fetches data, renders static content
async function PostPage({ params }) {
  const likes = await db.likes.count({ where: { postId: params.id } });
  
  return (
    <div>
      <PostContent />                    {/* Server Component — fast! */}
      <LikeButton initialCount={likes} /> {/* Client Component — interactive! */}
    </div>
  );
}
```

```tsx
// Client Component: only the interactive piece
'use client';
import { useState } from 'react';

function LikeButton({ initialCount }: { initialCount: number }) {
  const [count, setCount] = useState(initialCount);
  return <button onClick={() => setCount(c => c + 1)}>❤️ {count}</button>;
}
```

The key insight: you should have as many Server Components as possible (no JS bundle) and as few Client Components as possible (each adds to the bundle). Push the `'use client'` boundary as deep down the tree as possible — only the interactive leaf nodes need to be Client Components.
