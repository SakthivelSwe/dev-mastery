## WHY
If your application serves a global audience, you need to support multiple languages and regions. This involves translating text, formatting dates and currencies correctly, and ensuring Search Engines index the correct language versions of your pages.
**Internationalization (i18n)** in Next.js involves routing requests to locale-specific URLs (e.g., `/en/about` vs `/fr/about`) and loading the appropriate translation dictionaries on the server.

## THEORY
### The App Router i18n Strategy
Unlike the Pages Router which had built-in i18n routing, the App Router requires you to build i18n routing yourself using Middleware and dynamic route segments.
1. **Middleware:** Detects the user's preferred language (via the `Accept-Language` header) and redirects them to the correct locale-prefixed URL (`/fr/dashboard`).
2. **Dynamic Segment (`[locale]`):** The root of your `app/` directory becomes a dynamic segment `app/[locale]/...` so every page has access to the current locale.
3. **Dictionaries:** JSON files containing the translated strings. The server loads the correct JSON file based on the `[locale]` param.

### 2. Loading the Dictionary
```typescript
// app/dictionaries.ts
import 'server-only';

const dictionaries = {
  en: () => import('../dictionaries/en.json').then((module) => module.default),
  fr: () => import('../dictionaries/fr.json').then((module) => module.default),
};

export const getDictionary = async (locale: 'en' | 'fr') => {
  return dictionaries[locale]();
};
```

### 4. The Page Component
```tsx
// app/[locale]/page.tsx
import { getDictionary } from '../../dictionaries';

export default async function HomePage({ params: { locale } }: { params: { locale: 'en'|'fr' } }) {
  // Load the dictionary on the server! No massive JSON files sent to the client.
  const dict = await getDictionary(locale);
  
  return (
    <main>
      <h1>{dict.home.welcome}</h1>
    </main>
  );
}
```

### 5. The Middleware (Routing)
```typescript
// middleware.ts
import { NextResponse } from "next/server";
import { match } from '@formatjs/intl-localematcher';
import Negotiator from 'negotiator';

let locales = ['en', 'fr', 'es'];
let defaultLocale = 'en';

function getLocale(request: Request) {
  const headers = { 'accept-language': request.headers.get('accept-language') || '' };
  const languages = new Negotiator({ headers }).languages();
  return match(languages, locales, defaultLocale);
}

export function middleware(request: Request) {
  // Check if there is any supported locale in the pathname
  const pathname = new URL(request.url).pathname;
  const pathnameIsMissingLocale = locales.every(
    (locale) => !pathname.startsWith(`/${locale}/`) && pathname !== `/${locale}`
  );

  // Redirect if there is no locale
  if (pathnameIsMissingLocale) {
    const locale = getLocale(request);
    return NextResponse.redirect(new URL(`/${locale}${pathname}`, request.url));
  }
}

export const config = {
  matcher: ['/((?!api|_next/static|_next/image|favicon.ico).*)'],
};
```

## REAL_WORLD
**Server-Side Translation Advantage**
In traditional SPAs (React + React-i18next), the browser has to download the entire `fr.json` file before it can render the page. If your app has thousands of strings, this file can be megabytes in size, severely hurting load times.

With the Next.js App Router pattern shown above, the `getDictionary` function runs *exclusively on the server*. The server reads the JSON file, generates the HTML with the French text, and sends *only the HTML* to the browser. The browser never downloads the JSON dictionary. This results in incredibly fast, SEO-friendly multi-lingual sites.

## INTERVIEW
**Q: You need to translate a string inside a Client Component (`'use client'`). You cannot use `async/await` inside a Client Component to call `getDictionary()`. How do you translate text in Client Components?**

A: You should fetch the dictionary in the parent Server Component and pass the specific strings as props to the Client Component.

```tsx
// Server Component (app/[locale]/page.tsx)
import { getDictionary } from '@/dictionaries';
import { InteractiveButton } from './InteractiveButton';

export default async function Page({ params }) {
  const dict = await getDictionary(params.locale);
  
  return (
    <InteractiveButton 
      // Pass only the strings the client component needs!
      labels={{ 
        clickMe: dict.button.clickMe, 
        loading: dict.button.loading 
      }} 
    />
  );
}
```
If passing props becomes too tedious for deeply nested client components, you can pass the dictionary to a Client-side Context Provider at the top level, allowing any nested Client component to consume it via a custom hook (e.g., `useTranslations()`).
