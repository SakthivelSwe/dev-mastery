## WHY
A fast website directly impacts business metrics: Google uses page speed in rankings, Amazon found every 100ms of latency costs 1% in sales, and 53% of mobile users abandon sites that take more than 3 seconds to load.
Next.js has excellent built-in performance defaults, but knowing how to measure and optimize beyond the defaults is what separates good Next.js developers from great ones.

## THEORY
### Core Web Vitals (Google's Metrics)
- **LCP (Largest Contentful Paint):** How fast does the main content appear? Target: < 2.5s
- **FID/INP (Interaction to Next Paint):** How responsive is the page? Target: < 200ms
- **CLS (Cumulative Layout Shift):** Does content jump around? Target: < 0.1

## CODE

### Image Optimization
```tsx
import Image from 'next/image';

// BAD: Regular img tag
<img src="/hero.jpg" alt="Hero" width={1200} height={600} />
// No automatic optimization, no lazy loading, wrong sizes served to mobile

// GOOD: Next.js Image component
<Image
  src="/hero.jpg"
  alt="Hero"
  width={1200}
  height={600}
  priority          // LCP image — load eagerly (no lazy loading)
  quality={85}      // 85% quality (default 75%)
  placeholder="blur"  // Show blur placeholder while loading
  blurDataURL="data:image/jpeg;base64,..."  // Tiny inline base64 preview
/>
// Automatically: WebP format, right size for device, lazy loading, CDN
```

### Font Optimization
```tsx
// app/layout.tsx
import { Inter, Roboto_Mono } from 'next/font/google';

// Next.js downloads fonts at BUILD TIME and self-hosts them!
// No external requests → zero CLS from font loading
const inter = Inter({
  subsets: ['latin'],
  display: 'swap',    // FOUT instead of FOIT — always show text
  variable: '--font-inter',  // CSS variable for Tailwind
});

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="en" className={inter.variable}>
      <body>{children}</body>
    </html>
  );
}
```

### Code Splitting and Lazy Loading
```tsx
import dynamic from 'next/dynamic';

// Lazy load heavy components (they won't be in the initial bundle!)
const RichTextEditor = dynamic(
  () => import('@/components/RichTextEditor'),
  {
    loading: () => <div>Loading editor...</div>,
    ssr: false,  // Don't server-render (if component uses browser APIs)
  }
);

const ChartComponent = dynamic(
  () => import('@/components/Chart'),
  { ssr: false }  // Chart.js uses browser APIs — must be client-only!
);

export function BlogEditor() {
  return (
    <div>
      <RichTextEditor />  {/* Only loaded when this page is visited! */}
    </div>
  );
}
```

### Streaming and Suspense
```tsx
import { Suspense } from 'react';

// Stream slow data-fetching components progressively!
export default function DashboardPage() {
  return (
    <main>
      <h1>Dashboard</h1>
      
      {/* This renders instantly! */}
      <QuickStats />
      
      {/* This streams in when ready — shows skeleton while loading */}
      <Suspense fallback={<RecentOrdersSkeleton />}>
        <RecentOrders />   {/* Fetches slowly from DB */}
      </Suspense>
      
      <Suspense fallback={<AnalyticsSkeleton />}>
        <Analytics />     {/* Fetches from external API */}
      </Suspense>
    </main>
  );
}
```

## REAL_WORLD
**Measuring Performance**
```bash
# Lighthouse (built into Chrome DevTools)
# Run in Chrome: F12 → Lighthouse → Analyze page load

# Next.js built-in analytics
# next.config.js:
module.exports = {
  experimental: {
    webVitalsAttribution: ['CLS', 'LCP'],
  }
};

# In your layout, report Web Vitals to your analytics:
export function reportWebVitals(metric) {
  if (metric.label === 'web-vital') {
    analytics.track('Web Vital', {
      name: metric.name,
      value: metric.value,
      rating: metric.rating,  // 'good', 'needs-improvement', 'poor'
    });
  }
}
```

## INTERVIEW
**Q: Your Next.js product page has a Lighthouse Performance score of 45. The LCP is 4.2 seconds and the bundle is 850KB. What steps do you take to improve it?**

A: Systematic approach:

1. **Fix LCP (4.2s → < 2.5s):**
   - Identify the LCP element (usually the hero image or heading)
   - If it's an image: add `priority` prop to the `<Image>` component (prevents lazy loading for LCP)
   - If data fetching causes the delay: use Streaming with Suspense so the page shell renders immediately
   - Move the page from CSR to SSR or SSG so HTML arrives pre-rendered

2. **Reduce bundle size (850KB → target <200KB):**
   - Run `ANALYZE=true npm run build` to see what's large
   - Replace heavy libraries: moment.js → date-fns, lodash → native JS
   - Use `dynamic()` imports for heavy components that aren't needed on page load
   - Audit tree shaking: `import { specific } from 'lib'` not `import * from 'lib'`
   - Move server-only code to Server Components (zero bundle impact)

3. **Other optimizations:**
   - Use `next/font` for zero-CLS font loading
   - Set `Cache-Control` headers appropriately for static assets
   - Enable HTTP/2 and compression on the server

After these changes, typical result: LCP drops to 1.2s, bundle to 180KB, Lighthouse score: 92.
