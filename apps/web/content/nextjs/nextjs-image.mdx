## WHY
Images are often the largest assets on a web page, significantly impacting loading performance. Without optimization, a single high-resolution photo can be 3-5MB. On mobile with a slow connection, that single image could take 10+ seconds to load.
Next.js's `<Image>` component automatically handles: format conversion (WebP/AVIF), responsive resizing for each device, lazy loading, and preventing layout shift (CLS) — all with a single component swap.

## THEORY
### What Next.js Image Does Automatically
1. **Format conversion:** Serves WebP (30% smaller than JPEG) or AVIF (50% smaller) based on browser support
2. **Resizing:** Generates appropriately-sized images for each screen size
3. **Lazy loading:** Off-screen images load only when scrolled into view
4. **CLS prevention:** Reserves space for the image before it loads (no layout jump)
5. **Caching:** Optimized images cached on CDN or server file system

## CODE

### Basic Image Usage
```tsx
import Image from 'next/image';

// Local image (automatic sizing from import!)
import heroImage from '/public/hero.jpg';

export function HeroSection() {
  return (
    <Image
      src={heroImage}          // TypeScript-typed! Width/height inferred.
      alt="Hero image"         // Required for accessibility!
      priority                 // LCP image — don't lazy load!
      placeholder="blur"       // Show low-quality blur while loading
    />
  );
}

// Remote image (must specify width and height)
export function ProductCard({ product }: { product: Product }) {
  return (
    <Image
      src={product.imageUrl}
      alt={product.name}
      width={400}
      height={300}
      className="rounded-lg object-cover"
    />
  );
}
```

### Responsive Images
```tsx
import Image from 'next/image';

// Fill the parent container (great for hero sections)
<div className="relative h-96 w-full">
  <Image
    src="/hero.jpg"
    alt="Hero"
    fill               // Fills parent div!
    sizes="100vw"      // Hint: this image is full width
    style={{ objectFit: 'cover' }}
    priority
  />
</div>

// Responsive sizes (serve smaller image to smaller screens)
<Image
  src="/product.jpg"
  alt="Product"
  width={800}
  height={600}
  sizes="(max-width: 768px) 100vw,
         (max-width: 1200px) 50vw,
         33vw"
  // Browser chooses: full-width on mobile, half on tablet, third on desktop
/>
```

### Configuring Remote Domains
```javascript
// next.config.js — Required for external images!
/** @type {import('next').NextConfig} */
module.exports = {
  images: {
    remotePatterns: [
      {
        protocol: 'https',
        hostname: 'images.unsplash.com',  // Allow Unsplash images
      },
      {
        protocol: 'https',
        hostname: '**.amazonaws.com',      // Any S3 subdomain
        pathname: '/my-bucket/**',         // Specific bucket only
      },
    ],
    formats: ['image/avif', 'image/webp'],  // Prefer AVIF (50% smaller!)
    deviceSizes: [640, 750, 828, 1080, 1200, 1920, 2048, 3840],
    imageSizes: [16, 32, 48, 64, 96, 128, 256, 384],
  },
};
```

## REAL_WORLD
**Image Size Comparison**
```
Original JPEG:  3.2 MB
WebP:           2.1 MB  (34% smaller)
AVIF:           1.6 MB  (50% smaller)
WebP + resize for 375px mobile:  124 KB  (96% smaller!)
```

Next.js serves mobile users the ~124KB version, tablet users ~380KB, desktop users ~1.6MB AVIF. All automatically, with a single `<Image>` component.

## INTERVIEW
**Q: Why must you specify width and height (or use fill) for remote images in Next.js `<Image>`, but not for local images?**

A: For **local images** (imported from the file system), Next.js analyzes the image file at build time and knows the exact dimensions. It embeds them in the generated TypeScript types, so the component automatically gets the correct width/height without you specifying them. This also prevents CLS because the browser can calculate the space to reserve before the image loads.

For **remote images** (fetched from external URLs), Next.js cannot know the image dimensions at build time without downloading every image during the build — which is impractical. You must provide `width` and `height` explicitly so Next.js can:
1. Set the correct aspect ratio (reserving space to prevent CLS)
2. Generate appropriate srcset entries for responsive loading

**Alternative for remote images of unknown size:** Use `fill` + a parent container with defined dimensions:
```tsx
<div style={{ position: 'relative', width: '100%', height: '300px' }}>
  <Image src={url} alt="Product" fill style={{ objectFit: 'contain' }} />
</div>
```
This way you define the container size, and the image fills it without needing explicit width/height.
