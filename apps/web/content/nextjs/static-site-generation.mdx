## WHY
Performance is critical for web applications. The fastest possible way to serve a web page is to generate its HTML in advance and serve it directly from a CDN, bypassing the server and database entirely.
**Static Site Generation (SSG)** is the process of building these HTML pages at compile time (`npm run build`). In Next.js, SSG is the default rendering behavior, making it perfect for marketing sites, blogs, and documentation where content doesn't change on every single user request.

## THEORY
### How SSG Works in the App Router
By default, Next.js assumes all pages are static. During the build process, Next.js crawls your application, fetches any necessary data, executes the React components on the server, and outputs pure HTML and CSS files.

When a user visits the URL, the CDN instantly returns the pre-built HTML file.

### Dynamic Routes and SSG
If you have a dynamic route like `/blog/[slug]`, Next.js doesn't know which slugs exist at build time. To statically generate them, you must provide a list of all possible slugs using the `generateStaticParams` function.

## CODE

### Static Data Fetching (Default)
In the App Router, a simple `fetch()` without special configurations is automatically cached and the page is statically generated at build time.

```tsx
// app/about/page.tsx
export default async function AboutPage() {
  // This fetch happens ONCE during `npm run build`
  const res = await fetch('https://api.example.com/company-info');
  const data = await res.json();

  return (
    <main>
      <h1>About Us</h1>
      <p>Founded: {data.foundedYear}</p>
    </main>
  );
}
```

### Generating Static Params for Dynamic Routes
If you want `/blog/post-1` and `/blog/post-2` to be statically generated at build time, use `generateStaticParams`.

```tsx
// app/blog/[slug]/page.tsx
import { db } from '@/lib/db';

// 1. Tell Next.js which paths to generate at build time
export async function generateStaticParams() {
  const posts = await db.post.findMany({ select: { slug: true } });
  
  // Return an array of objects matching the dynamic segment ([slug])
  return posts.map((post) => ({
    slug: post.slug,
  }));
}

// 2. The page component uses the params generated above
export default async function BlogPost({ params }: { params: { slug: string } }) {
  // This runs at build time for each slug returned above!
  const post = await db.post.findUnique({ where: { slug: params.slug } });
  
  return (
    <article>
      <h1>{post.title}</h1>
      <div>{post.content}</div>
    </article>
  );
}
```

### Controlling Fallback Behavior
What happens if a user navigates to `/blog/new-post` after the build has finished?

```tsx
// app/blog/[slug]/page.tsx
export const dynamicParams = true; // (Default)

// If true: Next.js will dynamically render the new page on the first request, 
// and then cache it for subsequent visitors.

// If false: Next.js will return a 404 for any path not returned by generateStaticParams.
```

## REAL_WORLD
**Build Times vs SSG**
While SSG provides ultimate performance, it has a scaling problem. If you have an e-commerce store with 100,000 products, running `generateStaticParams` to build 100,000 HTML files at compile time could take hours. Every time you change a typo in the header, you have to wait hours for the build.

**The Solution:**
Only statically generate the top 100 most popular products at build time. Leave `dynamicParams = true`. The remaining 99,900 products will be generated on-demand when the first user requests them, and then cached globally.

## INTERVIEW
**Q: You want a page to be statically generated at build time, but it includes `import { cookies } from 'next/headers'`. What happens during `npm run build`?**

A: The page will **opt out of Static Site Generation** and become dynamically rendered (Server-Side Rendered on every request).

Next.js automatically detects "Dynamic Functions" like `cookies()`, `headers()`, or `searchParams`. Since it's impossible to know what a user's cookie or search parameter will be during the build step, Next.js cannot generate a static HTML file. The page is forced into dynamic rendering.

If you absolutely need the page to be static, you must remove the usage of dynamic functions from the Server Component.
