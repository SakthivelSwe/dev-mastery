## WHY
Static Site Generation (SSG) is great, but it falls apart when data must be personalized per user (like a dashboard), or when data changes constantly (like a live stock ticker).
**Server-Side Rendering (SSR)** solves this. With SSR, the HTML is generated on the server *dynamically* at the exact moment the user requests the page. The server fetches the latest data, injects it into the React components, renders the HTML, and sends it to the browser.

## THEORY
### SSR in the App Router
In the App Router, Next.js automatically switches a route to SSR (Dynamic Rendering) if it detects that the page relies on information that can only be known at request time.
This happens automatically if you use:
- `cookies()` or `headers()`
- The `searchParams` prop
- A `fetch()` request with `{ cache: 'no-store' }`
- A dynamic route without `generateStaticParams`

## CODE

### Opting into SSR via Dynamic Functions
Using a dynamic function automatically forces the route to render on every request.

```tsx
// app/dashboard/page.tsx
import { cookies } from 'next/headers';
import { db } from '@/lib/db';
import { redirect } from 'next/navigation';

export default async function Dashboard() {
  // Using cookies() opts the entire route into SSR!
  const cookieStore = cookies();
  const sessionToken = cookieStore.get('session')?.value;

  if (!sessionToken) {
    redirect('/login');
  }

  // Always fetches the freshest data for this specific user
  const userData = await db.user.findUnique({ where: { session: sessionToken } });

  return (
    <main>
      <h1>Welcome back, {userData.name}</h1>
      <p>Current balance: ${userData.balance}</p>
    </main>
  );
}
```

### Opting into SSR via Fetch Cache
You can force a specific fetch request to never be cached, which in turn forces the page to be dynamically rendered.

```tsx
// app/stock-ticker/page.tsx
export default async function StockTicker() {
  // 'no-store' means fetch on EVERY request
  const res = await fetch('https://api.stocks.com/live', { 
    cache: 'no-store' 
  });
  const data = await res.json();

  return (
    <div>
      <h1>AAPL: ${data.price}</h1>
      <small>Last updated: {new Date().toLocaleTimeString()}</small>
    </div>
  );
}
```

### Forcing Dynamic Rendering Manually
Sometimes you are using a database client (like Prisma) which doesn't use `fetch()`, so you can't pass `{ cache: 'no-store' }`. You can force the file to be dynamic using route segment configs.

```tsx
// app/feed/page.tsx
import { db } from '@/lib/db';

// Force Next.js to SSR this page on every request
export const dynamic = 'force-dynamic';

export default async function FeedPage() {
  // Now this DB query runs on every request
  const latestPosts = await db.post.findMany({ orderBy: { createdAt: 'desc' } });
  
  return (
    <ul>
      {latestPosts.map(post => <li key={post.id}>{post.title}</li>)}
    </ul>
  );
}
```

## REAL_WORLD
**The TTFB Tradeoff**
SSR gives you fresh data and SEO benefits (unlike client-side data fetching), but it has a cost: Time to First Byte (TTFB).

When a user requests an SSG page, the CDN responds in ~30ms.
When a user requests an SSR page, the request hits your server. The server must connect to the database (100ms), wait for the query to execute (200ms), render the React tree to HTML (50ms), and then send it back. The user is staring at a blank white screen for 350+ milliseconds before the page begins to load.

**Solution:** Use Next.js `loading.tsx` files! Next.js will instantly stream the layout and a loading skeleton to the user, while the server finishes the SSR database queries in the background.

## INTERVIEW
**Q: A page in your App Router uses `searchParams` to handle pagination (`/products?page=2`). During the build process, you notice Next.js marks this page as `λ (Server)`. Why? Can it be statically generated?**

A: It is marked as `λ (Server)` because it uses `searchParams`, which forces the page into Dynamic Rendering (SSR).

No, it cannot be completely statically generated. Search parameters (`?page=2`, `?sort=asc`) are not part of the file path. They can be any arbitrary value provided by the user at request time. Since Next.js cannot know all possible search parameter combinations at build time, it must render the page dynamically on the server for each request to read those parameters.
