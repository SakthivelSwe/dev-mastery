## WHY
If you set your website's main font size to `16px` (pixels), it might look perfect on your laptop. But if a visually impaired user relies on their browser's accessibility settings to force all text to be 24px so they can read it, your website will stubbornly refuse to scale! The text remains 16px, completely overriding the user's medical needs. 

Understanding the difference between **Absolute Units** and **Relative Units** is the foundation of building accessible, responsive, and maintainable web applications.

## THEORY
### Absolute vs Relative
- **Absolute Units (px, cm, in):** These are hardcoded physical measurements. 100px will mathematically be 100px, regardless of screen size, user preferences, or parent containers. (Great for borders, terrible for fonts).
- **Relative Units (em, rem, %, vh, vw):** These units do not have a fixed size. Their final size is calculated dynamically based on *something else* (like the parent container, the root document, or the physical screen size).

## CODE

### Level 1 — Beginner (Percentages `%`)
Percentages are relative to the **Parent Container**. If the parent is 1000px wide, and the child is `50%`, the child evaluates to 500px.

```css
.container {
  width: 1000px;
}

.child {
  /* Evaluates to 500px wide! */
  width: 50%;
}
```

### Level 2 — Building Blocks (Viewport Units: `vh`, `vw`)
Viewport units are relative to the **Browser Window (The Viewport)**.
- `1vw` = 1% of the viewport's width.
- `1vh` = 1% of the viewport's height.

If you want a Hero Banner image to perfectly fill the user's entire screen (whether they are on an iPhone or a 70-inch TV), you cannot use `height: 100%` (because `%` looks at the parent container, which might not have a defined height). You use `vh`!

```css
.hero-banner {
  /* 100 Viewport Height: Fills 100% of the physical screen height perfectly! */
  height: 100vh;
  
  /* 100 Viewport Width */
  width: 100vw; 
}
```

### Level 3 — Intermediate (The `rem` Unit)
`rem` stands for "Root Element". It is relative to the `font-size` defined on the root `<html>` element. 
By default, every browser in the world sets the root font-size to exactly `16px` (unless the user changed it for accessibility).

Therefore, by default: `1rem = 16px`.
If you want your button font to be 32px, you write `2rem` (2 * 16). 

```css
/* If a visually impaired user changes their browser default to 24px... */
/* <html> automatically becomes 24px. */

.button {
  /* If default is 16px, this button is 32px. */
  /* If default is 24px (accessibility), this button scales beautifully to 48px! */
  font-size: 2rem;
  
  /* We also use rem for padding, so the box scales with the text! */
  padding: 1rem 2rem;
}
```
**Golden Rule:** NEVER use `px` for `font-size`. ALWAYS use `rem`.

### Level 4 — Advanced (The `em` Unit)
`rem` is relative to the Root. `em` is relative to the **Direct Parent Element's font size**.
`em` is notoriously dangerous because it creates a compounding math nightmare if you nest elements.

```html
<div class="parent">
  Parent Text
  <div class="child">
    Child Text
    <div class="grandchild">Grandchild Text</div>
  </div>
</div>
```
```css
.parent { font-size: 20px; }
/* The child is 1.5 * 20 = 30px */
.child { font-size: 1.5em; }
/* The grandchild is 1.5 * the child (30px) = 45px! */
.grandchild { font-size: 1.5em; }
```
Because `em` compounds exponentially as you nest deeper into the HTML, developers almost universally abandoned it for `font-size` in favor of the much safer `rem`. 
However, `em` is still incredibly useful for Margins and Padding! If you build a button and set its padding to `1em`, the padding will dynamically grow and shrink perfectly in proportion to whatever font size the button currently has.

## REAL_WORLD
**The 62.5% Hack**
Doing math with `rem` is annoying. If you have a design mockup that says a title should be `24px`, you have to pull out a calculator and divide 24 by 16 (`1.5rem`). If the mockup says `18px`, you have to calculate `18/16 = 1.125rem`.

To solve this, developers invented a brilliant hack. They change the root font size to exactly `62.5%` of the default.
62.5% of 16px is exactly **10px**.
Now, the math is incredibly easy! `1rem = 10px`. 
If you want 24px, you type `2.4rem`. If you want 18px, you type `1.8rem`.

```css
html {
  /* Resets 1rem to perfectly equal 10px, while still respecting 
     user accessibility settings if they zoom in! */
  font-size: 62.5%; 
}

h1 {
  /* Evaluates to 32px */
  font-size: 3.2rem;
}
```

## INTERVIEW
**Q: I have a `<div>` inside a `<body>`. I set the div to `width: 100vw`. Suddenly, a horizontal scrollbar appeared at the bottom of the screen! Why didn't it fit perfectly?**
A: `vw` (Viewport Width) includes the width of the browser's vertical scrollbar! If the user's screen is 1000px wide, but the browser has a 15px vertical scrollbar on the right side, the available space for the website is only 985px. But `100vw` forces the div to be exactly 1000px wide, causing it to smash into the scrollbar and overflow horizontally. 
For widths, it is almost always safer to use `width: 100%`, because `%` calculates width *excluding* the scrollbar!

## FEYNMAN CHECK

### Explain CSS Units Like I'm 10 Years Old
> CSS values use ABSOLUTE units (`px`, `pt`, `cm` — fixed sizes) and RELATIVE units (`%`, `em`, `rem`, `vw`, `vh`, `ch`, `fr` — calculated from something else). `rem` is "relative to the ROOT font-size" (default 16px); `em` is "relative to THIS element's font-size" (which cascades and stacks confusingly); `%` is relative to the parent's matching dimension; `vw`/`vh` are 1% of the viewport. Modern apps use `rem` for typography (respects user zoom), `%` for fluid widths, `ch` for text-line widths, and `fr` for grid tracks. Using `px` for font-size is an accessibility violation because it ignores the user's browser-zoom font-size preference.

### 5 Deep Conceptual Questions

**Q1: What problem do CSS units fundamentally solve?**
> **A:** They let one stylesheet adapt to many contexts (different viewports, font sizes, user zoom levels, container widths) without recompilation. Absolute units (px) work for fixed-resolution prints; relative units enable responsive design. Without relative units, every breakpoint would require a hand-tuned px value.

**Q2: What is the ONE mental model that makes units click?**
> **A:** Every relative unit answers "relative to WHAT?" — and the answer is different per unit. `em` = relative to MY computed font-size. `rem` = relative to ROOT font-size. `%` = relative to parent's matching property (width % → parent width, padding % → parent WIDTH not height). `vw` = relative to viewport. `fr` = relative to other tracks in the same grid. Memorising these "anchor points" is 90% of the battle.

**Q3: What is the most dangerous misconception? Show it with code.**
> **A:** That `em` is safe to use everywhere — it COMPOUNDS through nesting and produces "shrinking text" bugs:
> ```css
> /* ❌ Each nested .item shrinks its children by 10% */
> .item { font-size: 0.9em; }
>
> /* HTML: <div class="item"><div class="item"><div class="item">Tiny</div></div></div> */
> /* Effective size: 1em → 0.9em → 0.81em → 0.729em (Tiny is ~73% of root) */
>
> /* ✅ rem is always anchored to the root — no compounding */
> .item { font-size: 0.9rem; }
> /* Every .item, no matter how nested, is 0.9 * root size = 14.4px (if root is 16px) */
> ```

**Q4: How does `vw`/`vh` interact with the mobile browser address bar at the runtime level?**
> **A:** On mobile, the visible viewport size CHANGES as you scroll (the address bar slides up/away). The traditional `vh` unit refers to the LARGEST viewport (no address bar), so `height: 100vh` on mobile creates a section TALLER than the visible screen on load. The 2022 spec added `svh` (small — smallest possible viewport), `lvh` (large — biggest possible), and `dvh` (dynamic — current). Use `dvh` for "fill the visible screen right now" — it updates as the address bar moves.

**Q5: Write a one-sentence definition of CSS units that a senior engineer at a FAANG company would find technically precise.**
> **A:** "CSS units are typed numeric values that resolve into used pixel values via a context-sensitive computation — absolute units map 1:1 to CSS pixels regardless of context, while relative units (`em`, `%`, `vw`, `fr`, `ch`) resolve against an anchor (element font-size, containing block, viewport, grid track total, or 0-character advance) — which is why typography stacks built on `em` introduce compounding bugs and viewport-based layouts must distinguish `vh` (legacy), `svh`, `lvh`, and `dvh` to handle mobile browser-chrome resizing correctly."

#### Step 3 — Live Computed-Size Readouts

// verify.js
function refresh() {
  for (const row of document.querySelectorAll('.row')) {
    const span   = row.querySelector('span:nth-of-type(2)') ?? row.children[1];
    const out    = row.querySelector('output');
    const size   = parseFloat(getComputedStyle(span).fontSize);
    out.textContent = size.toFixed(1) + 'px';
  }
}

window.addEventListener('resize', refresh);

#### Step 4 — Edge Cases: User Zoom + Min Font-Size Setting
```javascript
// verify.js (continued) — detect and warn if px-sized text won't respect zoom
function warnAboutAccessibility() {
  for (const el of document.querySelectorAll('[class*="px"]')) {
    const cs = getComputedStyle(el);
    if (cs.fontSize.endsWith('px') && el.tagName.match(/H[1-6]|P|SPAN|DIV/i)) {
      // Browsers DO scale px on Cmd-+ zoom, but NOT for user's
      // "minimum font size" or "default font size" browser settings.
      console.warn('px font-size — will not honour user minimum font-size:',
        el.textContent.slice(0, 30));
    }
  }
}
warnAboutAccessibility();

// Walk the user through testing:
console.log('Press Cmd/Ctrl-+ to zoom — every unit scales correctly.');
console.log('Now: Settings → Appearance → Font size → Largest.');
console.log('rem and % scale; px does NOT. That is the a11y violation.');
```
