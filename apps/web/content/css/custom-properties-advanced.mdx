## WHY
CSS Variables (officially called "Custom Properties") revolutionized CSS. By defining `--primary-color: blue`, you can reuse that color across hundreds of files and update it in one place.

But if you treat Custom Properties just like SASS/LESS static variables, you are missing 90% of their power. Unlike SASS variables (which are compiled away into static strings by Node.js before the browser ever sees them), CSS Custom Properties are **alive in the DOM**. They inherit through the HTML tree, they can be manipulated in real-time by JavaScript, and they can drive wildly complex responsive animations.

## THEORY
### The DOM Inheritance Model
SASS variables are scoped to the file. CSS Custom Properties are scoped to the **DOM Element**.
If you define `--bg-color` on the `<body>`, every child element inherits it. If you redefine `--bg-color` on a `<section class="dark-mode">`, that specific section and ALL of its children instantly recalculate their colors based on the new variable! This makes Custom Properties the ultimate tool for Theming and Component APIs.

## CODE

### Level 1 — Beginner (The Fallback Mechanism)
The `var()` function accepts a second argument: a fallback value. This is critical for building resilient components that won't break if a variable isn't defined.

```css
.btn {
  /* If --btn-bg is defined somewhere in the parent tree, use it.
     If it is undefined, fall back to 'black'. */
  background-color: var(--btn-bg, black);
  
  /* Fallbacks can be nested! */
  color: var(--btn-text, var(--global-text, white));
}
```

### Level 2 — Building Blocks (Component APIs)
Instead of writing heavy utility classes (like `.btn-red`, `.btn-blue`, `.btn-green`), you can expose a Custom Property "API" for your component. The component handles the complex logic, and the consumer only passes in the variable!

```css
/* 1. The Component Internal Logic */
.card {
  /* The card uses the variable to calculate multiple complementary colors via color-mix! */
  background-color: var(--theme-color, gray);
  border: 2px solid color-mix(in srgb, var(--theme-color) 50%, black);
  box-shadow: 0 4px 10px color-mix(in srgb, var(--theme-color) 30%, transparent);
}

/* 2. The Consumer (HTML/CSS) */
/* We don't need a new class. We just override the variable in a specific context! */
.header-section .card {
  --theme-color: #3498db; 
}
.warning-section .card {
  --theme-color: #e74c3c;
}
```
*Notice how we didn't redefine the `border` or `box-shadow`? Because they depend on `--theme-color`, they instantly recalculate themselves when the variable changes!*

### Level 3 — Intermediate (Theming / Dark Mode)
The absolute best way to implement Dark Mode is by redefining your global tokens at the root level using media queries. The rest of your CSS never needs to know if it's light or dark.

```css
/* 1. Define Light Mode (Default) on the :root (the <html> tag) */
:root {
  --bg-primary: #ffffff;
  --text-primary: #111111;
  --brand-color: #0066cc;
}

/* 2. Define Dark Mode using the OS preference media query */
@media (prefers-color-scheme: dark) {
  :root {
    /* Simply overwrite the variables! */
    --bg-primary: #111111;
    --text-primary: #eeeeee;
    --brand-color: #66b3ff;
  }
}

/* 3. Usage */
/* No .dark-mode classes required anywhere in your app! */
body {
  background-color: var(--bg-primary);
  color: var(--text-primary);
}
a {
  color: var(--brand-color);
}
```

### Level 4 — Advanced (JS Interactivity & Animation)
Because CSS variables live in the DOM, JavaScript can read and write them in real-time at 60 Frames Per Second.

```html
<!-- HTML -->
<div class="interactive-box" id="box">Hover over me!</div>
```

```css
/* CSS */
.interactive-box {
  /* We define variables for the mouse coordinates. Default to center (50%) */
  --mouse-x: 50%;
  --mouse-y: 50%;
  
  width: 300px;
  height: 300px;
  background: #222;
  
  /* Create a spotlight gradient that follows the mouse! */
  background-image: radial-gradient(
    circle at var(--mouse-x) var(--mouse-y), 
    rgba(255, 255, 255, 0.2) 0%, 
    transparent 50%
  );
}
```

```javascript
/* JavaScript */
const box = document.getElementById('box');

box.addEventListener('mousemove', (e) => {
  // Calculate mouse position relative to the box dimensions
  const rect = box.getBoundingClientRect();
  const x = ((e.clientX - rect.left) / rect.width) * 100;
  const y = ((e.clientY - rect.top) / rect.height) * 100;
  
  // INJECT the variables directly into the element's inline styles!
  // The CSS engine instantly redraws the gradient!
  box.style.setProperty('--mouse-x', `${x}%`);
  box.style.setProperty('--mouse-y', `${y}%`);
});
```

## REAL_WORLD
**React/Vue Dynamic Styling**
In modern JS frameworks like React, dynamically styling elements based on state (e.g., a progress bar filling up) used to require heavy inline styles or complex CSS-in-JS libraries.
Today, the most performant way to bridge React and CSS is to pass state directly into Custom Properties via the `style` object:
`<div className="progress-bar" style={{ "--progress": `${userProgress}%` }}></div>`
The React component stays incredibly clean, and the CSS file handles the actual width, transitions, and colors using `var(--progress)`.

## INTERVIEW
**Q: My team uses SASS. I defined a SASS variable `$main-color: red;` and tried to use it inside a CSS `calc()` function, but it didn't work. Why not? Should I use a CSS Custom Property instead?**
A: You should use a CSS Custom Property (`--main-color`). SASS variables are resolved during the build step. If you pass a SASS variable into a native CSS function like `calc()`, the browser just sees the literal string `$main-color`, which is invalid syntax. 
*(Note: SASS has a syntax interpolation workaround `calc(#{$main-color} + 10px)` to force compilation, but this defeats the fluid nature of modern CSS. Custom Properties are natively understood by the browser's math engine and are universally preferred for dynamic calculations).*
