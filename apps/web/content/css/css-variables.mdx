## WHY
Before CSS Variables, if you wanted to use a specific brand blue (`#2563EB`) in 50 different places across your website, you had to hardcode that hex value 50 times. If the marketing team decided to change the brand color to purple, you had to perform a massive "Find and Replace", hoping you didn't accidentally change a completely different element that happened to use the same blue. 

Preprocessors like Sass introduced variables (`$primary-color: #2563EB;`), but they are compiled away at build time. **CSS Custom Properties (Variables)** are native to the browser, dynamic, and can be updated instantly with JavaScript or Media Queries without recompiling your CSS!

## THEORY
### Scope and Inheritance
CSS variables follow standard CSS inheritance rules. 
- Variables declared on the `:root` pseudo-class (which represents the `<html>` element) are **Global** — they can be accessed anywhere.
- Variables declared on a specific class (e.g., `.card`) are **Local** — they can only be accessed by the `.card` element and its children.

### Syntax
- **Declaration:** Start with two dashes: `--primary-color: blue;`
- **Usage:** Wrap the name in the `var()` function: `color: var(--primary-color);`

## CODE

### Level 1 — Beginner (Declaring and Using)
```css
/* 1. Declare variables in the global root scope */
:root {
  --primary: #4F46E5;
  --secondary: #10B981;
  --text-dark: #1F2937;
  --spacing-md: 16px;
  --border-radius: 8px;
}

/* 2. Use them throughout your CSS */
.button {
  background-color: var(--primary);
  color: white;
  padding: var(--spacing-md);
  border-radius: var(--border-radius);
}

.card {
  border: 1px solid var(--primary);
  padding: var(--spacing-md);
}
```

### Level 2 — Building Blocks (Fallbacks)
What happens if a variable isn't defined? The CSS property becomes invalid. You can provide a fallback value directly in the `var()` function.

```css
.alert {
  /* If --alert-bg is not defined, it will fall back to #FEE2E2 */
  background-color: var(--alert-bg, #FEE2E2);
  
  /* You can even nest fallbacks! */
  color: var(--alert-text, var(--primary, #000000));
}
```

### Level 3 — Intermediate (Local Scope for Theming Components)
You don't always want variables on the `:root`. Using local scope allows you to create highly reusable, themable components.

```css
/* Define the default component structure */
.badge {
  /* Use local variables that expect to be provided by modifier classes */
  background-color: var(--badge-bg, #E5E7EB); /* Default gray fallback */
  color: var(--badge-text, #374151);
  padding: 4px 8px;
  border-radius: 12px;
}

/* Create modifier classes that ONLY redefine the variables! */
.badge.success {
  --badge-bg: #D1FAE5;
  --badge-text: #065F46;
}

.badge.danger {
  --badge-bg: #FEE2E2;
  --badge-text: #991B1B;
}

/* Now your HTML is super clean: 
   <span class="badge success">Approved</span> 
*/
```

### Level 4 — Advanced (Dynamic Manipulation)
Because CSS variables live in the browser's DOM (unlike Sass variables), you can change them dynamically based on user interaction or media queries!

**1. Dark Mode using Media Queries (Zero JS!):**
```css
:root {
  --bg-color: #FFFFFF;
  --text-color: #000000;
}

/* When the user's OS is in Dark Mode, automatically redefine the variables! */
@media (prefers-color-scheme: dark) {
  :root {
    --bg-color: #121212;
    --text-color: #FFFFFF;
  }
}

body {
  background-color: var(--bg-color); /* Automatically switches! */
  color: var(--text-color);
}
```

**2. Updating variables with JavaScript:**
```javascript
// Imagine a user sliding a "Theme Color" slider in settings:
const slider = document.querySelector('#color-slider');

slider.addEventListener('input', (event) => {
  const newHue = event.target.value;
  
  // Instantly update the root CSS variable!
  // Every element using var(--theme-hue) will update instantly!
  document.documentElement.style.setProperty('--theme-hue', newHue);
});
```

## REAL_WORLD
Modern CSS frameworks rely heavily on CSS Variables.
**Tailwind CSS** uses CSS variables under the hood to handle arbitrary values and opacity modifiers. For example, `bg-blue-500/50` (blue with 50% opacity) works by using a CSS variable for the RGB color channels and inserting the opacity variable into an `rgba()` function.

## INTERVIEW
**Q: What is the difference between a Sass variable (`$color`) and a CSS variable (`--color`)?**
A: Sass variables are static and compiled at build-time. The browser never sees them; they are replaced with raw values in the final CSS file. Therefore, they cannot be changed via JavaScript, nor can they be updated dynamically inside an `@media` query. CSS variables are dynamic, live in the browser's DOM, inherit down the DOM tree, and can be updated at runtime.

**Q: Does using thousands of CSS variables hurt browser performance?**
A: Changing a CSS variable on the `:root` element can trigger a recalculation of styles for the entire page, which can be expensive if you are animating it rapidly (e.g., updating a variable 60 times a second tied to mouse movement). For static theming or infrequent changes, the performance impact is entirely negligible.

## FEYNMAN CHECK

### Explain CSS Variables Like I'm 10 Years Old
> CSS variables (custom properties) are TOKENS that live in the cascade. Declare `--accent: blue` on `:root` and use it with `var(--accent)` anywhere. Unlike SCSS variables (resolved at build time), CSS variables are LIVE — change them at runtime via JavaScript or media queries and every using-element updates instantly. They INHERIT down the DOM tree by default, which is how you get scoped theming (set `--accent: red` on a section and only THAT section gets red). The non-obvious power: combined with `@property`, you can animate them with type safety; combined with `calc()`, you have a programmable design system.

### 5 Deep Conceptual Questions

**Q1: What problem do CSS variables solve that SCSS variables don't?**
> **A:** Runtime mutability and cascade inheritance. SCSS `$accent` is replaced at compile time with the literal value — once your bundle ships, you can't change it. CSS variables are read by the browser at computed-value time, so JavaScript can mutate them, media queries can override them, dark mode can swap them — all live, all reactive. They also inherit through the DOM, so scoped themes "just work."

**Q2: Mental model?**
> **A:** A CSS variable is an INHERITED CSS property whose name starts with `--`. It cascades and inherits exactly like `color` does. `var(--x, fallback)` reads it (with optional fallback for undefined cases). The browser resolves variables at COMPUTED-VALUE time, AFTER the cascade picks a winner — so you can override one variable in a media query and every `var(--x)` rule recomputes.

**Q3: Most dangerous misconception? Show with code.**
> **A:** Trying to use `var()` inside selectors or values where it's not allowed:
> ```css
> /* ❌ var() doesn't work inside @media query feature values */
> @media (min-width: var(--breakpoint)) { ... }
>
> /* ❌ var() can't compose selectors */
> .var(--class-name) { color: red; }
>
> /* ✅ var() works inside property VALUES only */
> .button { padding: var(--btn-padding); color: var(--accent); }
> ```

**Q4: How do CSS variables interact with animation at runtime?**
> **A:** By default, custom properties are `<*>` type — animating them switches between values instantly without interpolation. To animate smoothly, register them with `@property`: `@property --hue { syntax: "<number>"; initial-value: 0; inherits: false; }` — now `--hue: 0 → 360` interpolates frame-by-frame. This unlocks animatable gradients, hue rotation, fluid transitions — impossible before `@property`.

**Q5: FAANG-grade definition?**
> **A:** "CSS custom properties are inheritable, cascade-resolved properties whose names follow the `--*` syntax and whose values are evaluated at computed-value time via `var()` substitution — when registered through @property with a syntax descriptor they become typed and animatable, forming the foundation of runtime-themeable, JavaScript-bridged design token systems."

#### Step 3 — Variable System + Animated Gradient
```css
@property --hue {
  syntax: "<number>";
  initial-value: 220;
  inherits: true;
}

:root {
  --hue: 220;
  --primary: oklch(0.62 0.18 var(--hue));
  --primary-text: oklch(0.98 0 0);
  --bg: oklch(0.97 0.01 var(--hue));
  --fg: oklch(0.18 0.02 var(--hue));
  --radius: 6px;
}

.brand-emerald { --hue: 160; }   /* scoped override — emerald subsection */

body { background: var(--bg); color: var(--fg);
       font: 16px system-ui; margin: 0; padding: 2rem; transition: background 0.3s; }

.hero {
  padding: 3rem 2rem; border-radius: 12px; color: white;
  background:
    linear-gradient(135deg,
      oklch(0.6 0.2 var(--hue)),
      oklch(0.5 0.2 calc(var(--hue) + 60)));
  margin-bottom: 2rem;
  transition: --hue 0.5s ease;     /* @property makes this work */
}

button { padding: 0.6em 1.2em; border-radius: var(--radius); border: 0;
         background: var(--primary); color: var(--primary-text);
         font: inherit; cursor: pointer; margin-right: 0.5rem; }
button.ghost { background: transparent; color: var(--primary); border: 2px solid var(--primary); }

section { padding: 1.5rem; background: var(--bg); border-radius: var(--radius);
          margin-top: 1rem; }
```

#### Step 4 — Error Handling: Bridge JS, persist preference
```javascript
const slider = document.getElementById('hue');
const root = document.documentElement;
slider.addEventListener('input', e => {
  root.style.setProperty('--hue', e.target.value);
  localStorage.setItem('hue', e.target.value);
});
// Restore on load
const saved = localStorage.getItem('hue');
if (saved) { slider.value = saved; root.style.setProperty('--hue', saved); }

// Honour OS dark mode
const dark = matchMedia('(prefers-color-scheme: dark)');
function applyScheme() {
  root.style.setProperty('--bg', dark.matches
    ? `oklch(0.18 0.02 var(--hue))`
    : `oklch(0.97 0.01 var(--hue))`);
}
dark.addEventListener('change', applyScheme); applyScheme();
```
