## WHY
Dealing with time is notoriously the most frustrating part of software engineering. If you store timestamps as raw strings (`"2023-10-01"`) or raw UNIX integers (`1696118400`), it is incredibly difficult to answer questions like, "How much revenue did we make *last Tuesday*?" By using native SQL `DATE` and `TIMESTAMP` data types, you unlock powerful Date Functions that allow the database to handle leap years, timezones, and complex calendar math for you.

## THEORY
### Date vs Timestamp
It is crucial to understand the data types before manipulating them:
- **DATE:** Stores only the calendar date (e.g., `2023-10-01`). No time attached.
- **TIMESTAMP:** (Or `DATETIME` in MySQL) Stores the date AND the exact time (e.g., `2023-10-01 14:30:00`).
- **TIMESTAMP WITH TIME ZONE:** Stores the absolute exact moment in history, taking timezone offsets into account. (Highly recommended for all global applications!)

## CODE

### Level 1 — Beginner (Getting the Current Time)
When inserting a new row, you often want to record exactly when it happened.

```sql
-- Standard SQL / PostgreSQL
INSERT INTO logs (message, created_at) 
VALUES ('User logged in', CURRENT_TIMESTAMP);

-- Returns just the date (e.g., 2023-10-01)
SELECT CURRENT_DATE;

-- MySQL uses NOW() instead of CURRENT_TIMESTAMP
-- SELECT NOW();
```

### Level 2 — Building Blocks (Extracting Parts of a Date)
If you want to group your sales data by Month or Year, you use the `EXTRACT()` function (or `DATE_PART()` in Postgres).

```sql
-- Extract the Year (Returns 2023)
SELECT EXTRACT(YEAR FROM '2023-10-01'::DATE);

-- Extract the Month (Returns 10)
SELECT EXTRACT(MONTH FROM created_at) FROM users;

-- Grouping by Month! (How many users registered each month?)
SELECT 
    EXTRACT(MONTH FROM created_at) AS reg_month, 
    COUNT(*) 
FROM users 
GROUP BY reg_month;
```
*(Note: MySQL provides simpler helper functions like `YEAR(created_at)` and `MONTH(created_at)`).*

### Level 3 — Intermediate (Date Math / Intervals)
What if you want to find all trial accounts that expire in exactly 7 days? Or find users who haven't logged in for 3 months? You perform Date Math using `INTERVAL`s.

```sql
-- PostgreSQL / Standard SQL
-- Add 7 days to the current date
SELECT CURRENT_DATE + INTERVAL '7 days';

-- Subtract 3 months
SELECT CURRENT_DATE - INTERVAL '3 months';

-- Find expired trials
SELECT * FROM subscriptions 
WHERE expiration_date < CURRENT_DATE - INTERVAL '30 days';
```
*(Note: MySQL syntax is slightly different: `DATE_ADD(NOW(), INTERVAL 7 DAY)`).*

### Level 4 — Advanced (Timezones)
If your database server is physically located in London (UTC), but your user is in New York (EST), when they buy a product at 8:00 PM EST on Monday, the database might record it as 1:00 AM on Tuesday! If you generate a daily revenue report, the money will appear on the wrong day.

You must explicitly tell the database to convert the timestamp to the user's timezone before grouping the data.

```sql
-- PostgreSQL: Convert UTC timestamp to New York time
SELECT created_at AT TIME ZONE 'America/New_York'
FROM orders;

-- Grouping revenue by the day it occurred in New York!
SELECT 
    DATE(created_at AT TIME ZONE 'America/New_York') AS ny_date,
    SUM(amount)
FROM orders
GROUP BY ny_date;
```

## REAL_WORLD
**The `BETWEEN` Trap with Timestamps**
A very common junior developer mistake is querying timestamps using `BETWEEN` with simple dates.

```sql
-- ❌ DANGEROUS!
SELECT * FROM orders 
WHERE created_at BETWEEN '2023-10-01' AND '2023-10-31';
```
Why is this dangerous? Because the database secretly converts those simple dates into absolute timestamps at exactly midnight (`2023-10-31 00:00:00`). If a user places an order at noon on Halloween (`2023-10-31 12:00:00`), **it will be completely excluded from the report!**

To fix this, you should avoid `BETWEEN` for timestamps and use explicit `<` boundaries:
```sql
-- ✅ SAFE!
SELECT * FROM orders 
WHERE created_at >= '2023-10-01' 
  AND created_at < '2023-11-01'; -- Strictly less than the 1st of November
```

## INTERVIEW
**Q: My application has users in 50 different countries. Should I store timestamps in the database using their local timezone (e.g., storing `2023-10-01 14:00:00 EST` for the NY user, and `19:00:00 BST` for the London user)?**
A: **Absolutely not.** You should NEVER store local time in a database. You must always convert the time to strict **UTC** (Coordinated Universal Time) in your backend server before saving it. The database should only contain UTC timestamps. When you retrieve the data and send it to the frontend, the frontend's JavaScript (`Intl.DateTimeFormat`) will read the UTC timestamp and automatically convert it into the user's local timezone for display. Storing UTC ensures your database can accurately sort and compare events globally.
