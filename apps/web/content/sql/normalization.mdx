## WHY
When junior developers design a database, they often put everything into one massive table: `order_id, customer_name, customer_email, product_name, product_price`. 
This creates horrific problems. If the customer changes their email address, you have to update 50 different rows (one for each of their past orders). If you miss one, the database is now mathematically inconsistent. This is called an **Update Anomaly**.

**Normalization** is the mathematical process of breaking a database down into smaller, highly structured tables to eliminate data redundancy. It guarantees that every unique piece of information (like a customer's email) exists in exactly *one* place in the entire database.

## THEORY
### The Normal Forms
Database design is governed by strict rules called Normal Forms (1NF, 2NF, 3NF). 
While academics define up to 6 Normal Forms, 99% of professional software engineering only requires achieving **Third Normal Form (3NF)**.

- **1NF (Atomic Values):** A column can only hold one single value. No arrays or comma-separated strings!
- **2NF (No Partial Dependencies):** A column must depend on the *entire* Primary Key, not just part of it. (Usually solved by giving every table a single, unique `id`).
- **3NF (No Transitive Dependencies):** "Every non-key column must depend on the key, the whole key, and *nothing but the key*." (If Column B depends on Column A, and Column A depends on the ID, move Column B to a new table).

## CODE

### Level 1 — Beginner (First Normal Form - 1NF)
**The Rule:** Eliminate repeating groups. Every intersection of a row and column must contain exactly one value.

```text
-- THE TERRIBLE WAY (Not 1NF)
-- John has two phone numbers stored as a comma-separated string!
-- If we want to search for '555-9999', the query will be incredibly slow.
| id | name  | phones                  |
|----|-------|-------------------------|
| 1  | John  | 555-1234, 555-9999      |
| 2  | Mary  | 555-4321                |
```

```sql
-- THE 1NF WAY
-- We move the phones into a separate table! 
-- Now, every row/column intersection has exactly ONE value.
CREATE TABLE users (
    id INT PRIMARY KEY,
    name VARCHAR
);

CREATE TABLE user_phones (
    id INT PRIMARY KEY,
    user_id INT REFERENCES users(id),
    phone_number VARCHAR
);
```

### Level 2 — Building Blocks (Second Normal Form - 2NF)
**The Rule:** Must be in 1NF, and all non-key columns must depend on the *entire* Primary Key.
*(Note: 2NF is only an issue if your table uses a Composite Primary Key made of multiple columns. If you just use an auto-incrementing `id`, you are usually in 2NF by default!)*

```text
-- THE TERRIBLE WAY (Not 2NF)
-- Primary Key is (student_id, class_id)
-- The 'class_name' depends on 'class_id', but does NOT depend on 'student_id'.
-- If no students enroll in 'CS101', the class completely disappears from the database! (Deletion Anomaly)
| student_id | class_id | student_grade | class_name   |
|------------|----------|---------------|--------------|
| 1          | 101      | A             | Intro to CS  |
| 1          | 102      | B             | Physics      |
```

```sql
-- THE 2NF WAY
-- We break it into three tables!
CREATE TABLE students (id INT PRIMARY KEY, name VARCHAR);
CREATE TABLE classes (id INT PRIMARY KEY, class_name VARCHAR);

-- The Join Table (Grade depends on BOTH the student AND the class)
CREATE TABLE enrollments (
    student_id INT REFERENCES students(id),
    class_id INT REFERENCES classes(id),
    grade VARCHAR,
    PRIMARY KEY (student_id, class_id)
);
```

### Level 3 — Intermediate (Third Normal Form - 3NF)
**The Rule:** Must be in 2NF, and there must be no Transitive Dependencies. (A non-key column cannot depend on *another* non-key column).

```text
-- THE TERRIBLE WAY (Not 3NF)
-- The Primary Key is 'order_id'.
-- 'customer_name' depends on 'customer_id', NOT 'order_id'.
-- If customer #5 changes their name, we have to update 10,000 order rows!
| order_id | amount | customer_id | customer_name |
|----------|--------|-------------|---------------|
| 100      | 50.00  | 5           | John Doe      |
| 101      | 25.00  | 5           | John Doe      |
```

```sql
-- THE 3NF WAY
-- The customer data is completely isolated. 
-- Changing John's name requires exactly ONE row update in the entire database!
CREATE TABLE customers (
    id INT PRIMARY KEY,
    name VARCHAR
);

CREATE TABLE orders (
    id INT PRIMARY KEY,
    amount NUMERIC,
    customer_id INT REFERENCES customers(id)
);
```

### Level 4 — Advanced (Denormalization for Performance)
Once you perfectly normalize a database into 3NF, the data integrity is flawless. 
However, there is a massive downside: **Read Performance**.
If you want to view a single Order on the screen, you might have to write a SQL query with 7 `JOIN`s to fetch the customer name, the shipping address, the product names, and the category names.

**Denormalization** is the intentional, strategic breaking of 3NF rules to improve Read performance at the expense of Write performance.

```sql
-- A fully 3NF database requires us to constantly SUM() the order items 
-- to find the total order price. This is slow if we do it 10,000 times a second.

-- DENORMALIZATION:
-- We intentionally add a redundant 'total_price' column to the orders table.
ALTER TABLE orders ADD COLUMN total_price NUMERIC;

-- Now, reads are instant! (SELECT total_price FROM orders WHERE id = 1)
-- BUT, we take on the burden of ensuring this value stays mathematically accurate.
-- We must use Database Triggers or Application logic to constantly update this 
-- column whenever an order_item is added or removed!
```

## REAL_WORLD
**Data Warehousing (Star Schema)**
If Normalization is so important, why do Data Analysts hate it?
Because analysts run massive aggregation queries (`GROUP BY`, `SUM()`) across millions of rows to generate business reports. If the data is scattered across 15 highly-normalized tables, the database engine will choke trying to execute 15 massive `JOIN`s.

In Business Intelligence (BI) and Data Warehousing, engineers intentionally use highly **Denormalized** designs, like the **Star Schema**. They create one massive, wide "Fact Table" in the center (containing metrics like `sales_amount` and redundant string data like `customer_state`), surrounded by a few "Dimension Tables". This guarantees data analysts can generate reports without writing complex, slow joins.
*Rule of thumb: OLTP (Online Transaction Processing for the live app) should be strictly Normalized. OLAP (Online Analytical Processing for BI) should be Denormalized.*

## INTERVIEW
**Q: We have a table `employees(id, name, zip_code, city, state)`. Is this table in 3NF?**
A: **No, it contains a Transitive Dependency.**
The `city` and `state` columns do not depend directly on the employee's `id`. They depend on the `zip_code`. 
If a city changes its name, you would have to update every single employee record that shares that zip code. To achieve strict 3NF, you must remove `city` and `state` from the `employees` table, create a new `zip_codes` table `(zip_code, city, state)`, and set up a Foreign Key relationship.
*(Note: In the real world, developers often intentionally leave Zip/City/State denormalized in the users table because the performance penalty of joining a zip code table on every single query outweighs the rare risk of a city changing its name!)*
