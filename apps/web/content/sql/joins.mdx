## WHY
A database normalized to avoid data duplication splits data across multiple tables. User data is in `users`, order data is in `orders`, product data is in `products`. 

If you want "Show me all orders placed by Alice, with the product name she ordered", that data lives in three separate tables! **SQL JOINs** are the mechanism to combine data from multiple tables based on a related column (usually a Foreign Key), returning it as a single, unified result set.

## THEORY
### The Types of JOINs (Visualized)
Think of two circles overlapping (a Venn diagram), where the Left circle is Table A and the Right circle is Table B.
- **INNER JOIN:** Only the overlapping center (rows that exist in BOTH tables).
- **LEFT JOIN:** Everything in the Left circle (ALL rows from A, plus matching B data).
- **RIGHT JOIN:** Everything in the Right circle (ALL rows from B, plus matching A data).
- **FULL OUTER JOIN:** Both circles entirely (ALL rows from both A and B).

## CODE

### Level 1 — Beginner (INNER JOIN)
Returns only rows where there is a match in BOTH tables. If an order has no matching user, it is excluded.

```sql
-- Tables setup:
-- users (id, username, email)
-- orders (id, user_id, product_name, total_amount, order_date)

-- INNER JOIN: Get orders AND their associated user username
SELECT
    u.username,
    u.email,
    o.product_name,
    o.total_amount,
    o.order_date
FROM orders o
INNER JOIN users u ON o.user_id = u.id
ORDER BY o.order_date DESC;

-- If a user has no orders, they don't appear in the result.
-- If an order has an invalid user_id, it doesn't appear either.
```

### Level 2 — Building Blocks (LEFT JOIN)
Returns ALL rows from the LEFT table, plus matching data from the RIGHT table. If there's no match on the right, the right columns come back as NULL.

```sql
-- Get ALL users, showing their total orders (including users with zero orders)
SELECT
    u.username,
    COUNT(o.id) AS total_orders,    -- COUNT(column) ignores NULLs
    COALESCE(SUM(o.total_amount), 0) AS lifetime_spend
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.id, u.username
ORDER BY lifetime_spend DESC;

-- Users with 0 orders WILL appear (with total_orders = 0 and lifetime_spend = 0)
-- With INNER JOIN, they would be excluded!
```

### Level 3 — Intermediate (Multi-Table Joins & Aliases)
Real queries often join 3, 4, or more tables. You must alias every table to keep column references unambiguous.

```sql
-- Three-table join: orders + users + products
SELECT
    u.username                          AS customer_name,
    p.name                              AS product_name,
    p.category,
    oi.quantity,
    oi.unit_price,
    (oi.quantity * oi.unit_price)       AS line_total,
    o.order_date
FROM orders o
INNER JOIN users u      ON o.user_id    = u.id
INNER JOIN order_items oi ON oi.order_id = o.id
INNER JOIN products p   ON oi.product_id = p.id
WHERE o.order_date >= '2025-01-01'
    AND p.category = 'Electronics'
ORDER BY o.order_date DESC, line_total DESC;
```

### Level 4 — Advanced (SELF JOIN)
A table can be joined to itself! This is useful for hierarchical data (like an employee/manager relationship where both are in the same `employees` table).

```sql
-- employees table: id, name, manager_id (references employees.id)
SELECT
    emp.name    AS employee_name,
    mgr.name    AS manager_name
FROM employees emp
LEFT JOIN employees mgr ON emp.manager_id = mgr.id
ORDER BY mgr.name, emp.name;

-- Using a CTE to find multi-level hierarchy (recursive CTE)
WITH RECURSIVE employee_hierarchy AS (
    -- Base case: The CEO (no manager)
    SELECT id, name, manager_id, 0 AS level
    FROM employees
    WHERE manager_id IS NULL

    UNION ALL

    -- Recursive case: Employees under the previous level
    SELECT e.id, e.name, e.manager_id, eh.level + 1
    FROM employees e
    INNER JOIN employee_hierarchy eh ON e.manager_id = eh.id
)
SELECT name, level FROM employee_hierarchy ORDER BY level, name;
```

## REAL_WORLD
The most critical performance consideration with JOINs is ensuring you have **indexes on the Foreign Key columns** used in the `ON` condition.

If you JOIN `orders` to `users` on `orders.user_id = users.id`, PostgreSQL will use a fast Binary Search (B-Tree Index) on `users.id` (the Primary Key is always indexed). However, unless you explicitly create an index on `orders.user_id`, PostgreSQL will perform a slow **Full Sequential Scan** through the entire `orders` table for every user.

```sql
-- Always add an index to foreign key columns used in JOINs!
CREATE INDEX idx_orders_user_id ON orders (user_id);
```

This single line can take a query from 5 seconds to 5 milliseconds on a large table.

## INTERVIEW
**Q: What is the difference between a CROSS JOIN and a FULL OUTER JOIN?**
A: A `CROSS JOIN` (also called a Cartesian Product) joins EVERY row from Table A with EVERY row from Table B with NO condition. If A has 100 rows and B has 100 rows, you get 10,000 rows. It's almost never intentional. A `FULL OUTER JOIN` combines a LEFT JOIN and a RIGHT JOIN — it returns all rows from both tables, with NULLs where there is no match.

**Q: What does `ON` vs `USING` do in a JOIN?**
A: `JOIN users ON orders.user_id = users.id` is the full explicit form. `JOIN users USING (user_id)` is a shorthand that only works when both tables have a column with the exact same name. The result is the same, but `USING` removes the duplicate column from the result set (showing `user_id` once instead of twice).

**Q: When would you use a RIGHT JOIN?**
A: Almost never. Every `RIGHT JOIN` can be rewritten as a `LEFT JOIN` by simply swapping the table order. `FROM A RIGHT JOIN B` is identical to `FROM B LEFT JOIN A`. Most SQL developers exclusively use `LEFT JOIN` for consistency.
