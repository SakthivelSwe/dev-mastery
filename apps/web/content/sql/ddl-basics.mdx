## WHY
Before you can insert data into a database, you must build the structure to hold it. **Data Definition Language (DDL)** is the subset of SQL used to create, modify, and destroy the actual tables, databases, and schemas. If a database is a filing cabinet, DDL is the act of buying the cabinet and labeling the drawers, while DML (Data Manipulation Language) is the act of putting papers inside.

## THEORY
### The Core DDL Commands
1. **CREATE:** Build a brand new object (Database, Table, Index, View).
2. **ALTER:** Modify the structure of an existing object (e.g., add a new column to a table).
3. **DROP:** Completely destroy an object and all of the data inside it.
4. **TRUNCATE:** Empty a table completely, but keep the structure intact.

## CODE

### Level 1 — Beginner (CREATE TABLE)
When creating a table, you must specify the exact data type for every single column. This is what makes SQL "strongly typed" compared to NoSQL databases.

```sql
-- Create a simple Users table
CREATE TABLE users (
    id INT PRIMARY KEY,
    username VARCHAR(50) NOT NULL,
    email VARCHAR(100) UNIQUE NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### Level 2 — Building Blocks (ALTER TABLE)
Requirements always change. Six months after launching, the Product Manager asks you to add a "Phone Number" field to the user profiles. You do not drop the table and recreate it! You use `ALTER`.

```sql
-- Add a new column to an existing table
ALTER TABLE users 
ADD COLUMN phone_number VARCHAR(20);

-- Change the data type of an existing column (e.g., make it longer)
ALTER TABLE users 
ALTER COLUMN username TYPE VARCHAR(100);

-- Delete a column completely
ALTER TABLE users 
DROP COLUMN phone_number;
```

### Level 3 — Intermediate (DROP vs TRUNCATE)
Both of these commands destroy data, but they operate very differently under the hood.

```sql
-- TRUNCATE deletes all the rows, but the 'logs' table still exists!
-- You can immediately start inserting new rows into it.
TRUNCATE TABLE logs;

-- DROP completely destroys the 'logs' table.
-- If you try to insert into it, the database will throw an error.
DROP TABLE logs;
```
**Why use TRUNCATE instead of `DELETE FROM logs`?** 
`DELETE` scans the table row by row, deleting each one and writing the deletion to the transaction log (so it can be undone). `TRUNCATE` is a DDL command; it simply drops the physical data file on the hard drive and creates a new empty one. `TRUNCATE` can delete 1 billion rows in a fraction of a millisecond, whereas `DELETE` might take 10 minutes!

### Level 4 — Advanced (Constraints)
A database shouldn't just hold data; it should protect the integrity of the data. You enforce rules using DDL Constraints.

```sql
CREATE TABLE products (
    id SERIAL PRIMARY KEY,
    name VARCHAR(200) NOT NULL,
    price DECIMAL(10, 2) NOT NULL,
    
    -- CHECK Constraint: The database will physically reject 
    -- any INSERT or UPDATE where the price is negative!
    CONSTRAINT positive_price CHECK (price >= 0),
    
    -- FOREIGN KEY Constraint: This ensures that 'category_id'
    -- MUST perfectly match an ID that exists in the 'categories' table!
    category_id INT,
    CONSTRAINT fk_category 
      FOREIGN KEY (category_id) 
      REFERENCES categories(id)
      ON DELETE CASCADE
);
```

## REAL_WORLD
**Database Migrations (Flyway / Liquibase)**
In a professional environment, you never open a SQL console and manually type `ALTER TABLE` on the production database. If you make a typo, you could destroy the database.

Instead, developers use **Migration Tools** (like Flyway or Prisma). You write your DDL statements into versioned text files (e.g., `V1__Create_Users.sql`, `V2__Add_Phone_Column.sql`) and commit them to Git. When the application boots up, the migration tool automatically scans the database, determines which scripts haven't been run yet, and safely executes them in order.

## INTERVIEW
**Q: Can a DDL statement (like `ALTER TABLE`) be rolled back inside a Transaction?**
A: **It depends on the database engine!** 
In **PostgreSQL**, DDL statements are fully transactional. If you run `BEGIN; ALTER TABLE users DROP COLUMN email; ROLLBACK;`, the column will be restored instantly.
In **MySQL** and **Oracle**, DDL statements implicitly force an auto-commit. The moment you type `ALTER TABLE`, the transaction is instantly committed to disk and it **cannot** be rolled back. This is a crucial architectural difference to remember when writing migration scripts!
