## WHY
`GROUP BY` aggregations collapse multiple rows into one. If you want the average salary per department, `GROUP BY department` gives you one row per department. But what if you want to see each individual employee AND their department average salary on the same row? With `GROUP BY`, you'd have to run two separate queries and join them.

**Window Functions** solve this. They perform calculations across a "window" of rows related to the current row WITHOUT collapsing those rows. Every row remains in the output, but each row gains a new computed column derived from the aggregate calculation across the window.

## THEORY
### The `OVER()` Clause
The key difference between a regular aggregate and a window function is the `OVER()` clause. Appending `OVER()` to an aggregate function transforms it into a window function.

- `SUM(salary) OVER(PARTITION BY department)` — Calculates the total salary for each department AND displays it next to every individual employee row.
- `RANK() OVER(ORDER BY salary DESC)` — Ranks every employee by salary without collapsing the rows.

## CODE

### Level 1 — Beginner (Basic Aggregation over a Window)
```sql
-- WITHOUT Window Function: You get the average, but lose individual rows
SELECT department, AVG(salary) AS dept_avg
FROM employees
GROUP BY department;

-- WITH Window Function: Individual rows PRESERVED + dept avg!
SELECT
    name,
    department,
    salary,
    AVG(salary) OVER(PARTITION BY department) AS dept_avg_salary,
    salary - AVG(salary) OVER(PARTITION BY department) AS deviation_from_avg
FROM employees
ORDER BY department, salary DESC;

/* Output:
name     | department | salary | dept_avg_salary | deviation_from_avg
---------|------------|--------|-----------------|-------------------
Alice    | Engineering| 120000 | 100000          | +20000
Bob      | Engineering| 90000  | 100000          | -10000
Charlie  | Engineering| 90000  | 100000          | -10000
Dave     | Marketing  | 70000  | 65000           | +5000
Eve      | Marketing  | 60000  | 65000           | -5000
*/
```

### Level 2 — Building Blocks (Ranking Functions)
```sql
-- RANK() vs DENSE_RANK() vs ROW_NUMBER()
SELECT
    name,
    department,
    salary,
    
    -- ROW_NUMBER: Unique, sequential number for every row (no ties)
    ROW_NUMBER() OVER(PARTITION BY department ORDER BY salary DESC) AS row_num,
    
    -- RANK: Ties get the same rank, next rank SKIPS numbers (1, 1, 3)
    RANK() OVER(PARTITION BY department ORDER BY salary DESC) AS salary_rank,
    
    -- DENSE_RANK: Ties get the same rank, but next rank does NOT skip (1, 1, 2)
    DENSE_RANK() OVER(PARTITION BY department ORDER BY salary DESC) AS dense_rank
FROM employees;

-- Practical use: Get the TOP earner in each department
SELECT name, department, salary
FROM (
    SELECT
        name, department, salary,
        RANK() OVER(PARTITION BY department ORDER BY salary DESC) AS r
    FROM employees
) ranked
WHERE r = 1;
```

### Level 3 — Intermediate (LAG and LEAD — Accessing Adjacent Rows)
```sql
-- LAG: Access data from a PREVIOUS row in the result set
-- LEAD: Access data from a NEXT row in the result set

-- Month-over-month revenue comparison
SELECT
    month_year,
    total_revenue,
    LAG(total_revenue) OVER(ORDER BY month_year) AS prev_month_revenue,
    total_revenue - LAG(total_revenue, 1, 0) OVER(ORDER BY month_year) AS revenue_change,
    ROUND(
        (total_revenue - LAG(total_revenue) OVER(ORDER BY month_year))
        / LAG(total_revenue) OVER(ORDER BY month_year) * 100, 2
    ) AS growth_percentage
FROM monthly_revenue
ORDER BY month_year;
```

### Level 4 — Advanced (Frames — ROWS BETWEEN)
Window Functions can define a sliding "Frame" around the current row to create Moving Averages and Running Totals.

```sql
-- RUNNING TOTAL (cumulative sum)
SELECT
    order_date,
    daily_sales,
    SUM(daily_sales) OVER(
        ORDER BY order_date
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW  -- All rows up to current
    ) AS cumulative_sales
FROM daily_sales_summary;

-- 7-DAY MOVING AVERAGE (smooths out noise in time-series data)
SELECT
    order_date,
    daily_sales,
    ROUND(AVG(daily_sales) OVER(
        ORDER BY order_date
        ROWS BETWEEN 6 PRECEDING AND CURRENT ROW  -- Current row + previous 6 rows
    ), 2) AS seven_day_moving_avg
FROM daily_sales_summary;
```

## REAL_WORLD
Window functions are essential for building **Analytical Dashboards and Business Intelligence (BI) reports** without pulling data into application memory.

Rather than fetching all 1 million sales records into Node.js/Python and calculating rankings, averages, and trends in code, you push the computation down to the database engine. PostgreSQL can process millions of rows and compute window functions in a fraction of a second using optimized C code, returning only the pre-computed final table to your application.

## INTERVIEW
**Q: What is the execution order of SQL clauses? Where do Window Functions execute?**
A: SQL executes clauses in this order: `FROM` → `WHERE` → `GROUP BY` → `HAVING` → **Window Functions (inside SELECT)** → `ORDER BY` → `LIMIT`. Because Window Functions execute after `WHERE` and `GROUP BY` but before `ORDER BY`, you cannot filter by a window function result in a `WHERE` clause. You must wrap it in a subquery or CTE.

**Q: Can you combine a `GROUP BY` with Window Functions in the same query?**
A: Yes! First, the `GROUP BY` collapses the data. Then, the Window Function operates on the collapsed, aggregated result set. However, you can no longer reference individual pre-aggregated columns.

**Q: What is the performance difference between `ROW_NUMBER()`, `RANK()`, and `DENSE_RANK()`?**
A: They all have essentially the same performance since they all require sorting the window. The choice between them is purely semantic — based on how you want ties to be handled in the ranking output.
