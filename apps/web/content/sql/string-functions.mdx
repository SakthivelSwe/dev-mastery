## WHY
Data entered by human users is famously dirty. A user might type `"   jOhn.smith@Test.com  "`. If you search for `'john.smith@test.com'`, your query will fail. Instead of pulling millions of rows into your Node.js application and using JavaScript to clean the text, it is drastically faster to have the database engine clean, format, and manipulate the strings natively using SQL **String Functions**.

## THEORY
### Database Dialects
While basic SQL (`SELECT`, `WHERE`) is identical across all databases, String Functions are highly dialect-specific. A function that works perfectly in PostgreSQL might crash in SQL Server or MySQL. The examples below focus on the widely accepted ANSI standard or note Postgres/MySQL equivalents.

## CODE

### Level 1 — Beginner (Cleaning Data: TRIM, UPPER, LOWER)
The most common operation is standardizing messy user input before comparing it.

```sql
-- LOWER() converts the string to completely lowercase
-- UPPER() converts it to uppercase
SELECT LOWER(email), UPPER(last_name) FROM users;

-- TRIM() removes leading and trailing spaces!
-- Very useful for cleaning up copy-paste errors.
SELECT TRIM('   hello world   '); -- Returns 'hello world'

-- Combining them in a WHERE clause for safe searching:
SELECT * FROM users 
WHERE LOWER(TRIM(email)) = 'john@test.com';
```

### Level 2 — Building Blocks (Concatenation)
Often, the database stores the `first_name` and `last_name` in separate columns, but your API needs to return a single `full_name`.

```sql
-- Standard SQL (PostgreSQL, SQLite, Oracle) uses the || operator
SELECT first_name || ' ' || last_name AS full_name 
FROM users;

-- MySQL does NOT use ||. It requires the CONCAT() function!
SELECT CONCAT(first_name, ' ', last_name) AS full_name 
FROM users;
```

### Level 3 — Intermediate (Extracting Substrings)
Sometimes you need to pull a specific piece of text out of a larger string (e.g., getting the first 3 letters of a phone number for the area code).

```sql
-- SUBSTRING(string FROM start_position FOR length)
-- Note: SQL strings are 1-indexed! The first character is 1, not 0!
SELECT SUBSTRING('PostgreSQL' FROM 1 FOR 4); 
-- Returns 'Post'

-- LEFT() and RIGHT() are common shortcuts in many dialects
SELECT LEFT('123-456-7890', 3); -- Returns '123'
SELECT RIGHT('123-456-7890', 4); -- Returns '7890'
```

### Level 4 — Advanced (LENGTH and REPLACE)
You can search within strings and replace specific characters.

```sql
-- LENGTH() returns the number of characters
-- Find all users who have unusually short usernames
SELECT username FROM users 
WHERE LENGTH(username) < 4;

-- REPLACE(string, old_text, new_text)
-- Fix a massive typo across the entire database!
UPDATE employees
SET department = REPLACE(department, 'Salse', 'Sales');
```

## REAL_WORLD
**Parsing Emails with POSITION**
Imagine your CEO wants a report of all the unique Email Domains (e.g., `gmail.com`, `yahoo.com`) your users have. The `users` table only has a full `email` column. You have to write SQL to dynamically slice the string exactly where the `@` symbol is!

```sql
-- 1. POSITION (or STRPOS) finds the numeric index of the '@' symbol
-- 2. We use SUBSTRING to grab everything AFTER that index!

SELECT 
    SUBSTRING(
        email FROM POSITION('@' IN email) + 1
    ) AS email_domain,
    COUNT(*)
FROM users
GROUP BY email_domain;
```

## INTERVIEW
**Q: I have a massive table. I added `WHERE LOWER(email) = 'test@test.com'`, but my query is taking 5 seconds to run, even though I have an index on the `email` column. Why?**
A: Because you wrapped the database column in a function (`LOWER(email)`), you **destroyed the index**. The B-Tree index is built on the exact raw strings (e.g., `Test@Test.com`). When you apply a function to the column, the database cannot use the B-Tree; it is forced to run a Full Table Scan, applying the `LOWER()` function to every single row in memory before comparing it. 
To fix this, you must either create a specialized **Function-Based Index** (`CREATE INDEX idx_lower_email ON users (LOWER(email))`), or use a Case-Insensitive Collation on the column.
