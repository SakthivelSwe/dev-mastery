## WHY
Sometimes, the answer to your query depends on the answer to *another* query. For example: "Find all employees who make more than the average salary of the company." You don't know the average salary until you query it! Instead of running two separate queries from your Node.js backend, you can nest one query completely inside another using a **Subquery**.

## THEORY
### What is a Subquery?
A subquery (or Inner Query) is simply a `SELECT` statement nested inside another SQL statement. 
The database engine executes the Inner Query first, takes its result, and passes it directly to the Outer Query. 

Subqueries must always be enclosed in `() ` parentheses.

## CODE

### Level 1 — Beginner (Scalar Subqueries)
A scalar subquery returns exactly one row and one column (a single value). You can use it anywhere you would normally type a single number or string.

```sql
-- Step 1: What is the average salary? (Returns a single number, e.g., 60000)
-- SELECT AVG(salary) FROM employees;

-- Step 2: Use that number in a WHERE clause!
SELECT first_name, last_name, salary 
FROM employees
WHERE salary > (SELECT AVG(salary) FROM employees);
```
*The database runs the inner query, calculates 60000, and effectively transforms the outer query into `WHERE salary > 60000`.*

### Level 2 — Building Blocks (Multi-Row Subqueries)
What if the subquery returns an entire column of values? (e.g., "Find all departments that have offices in New York"). You cannot use the `=` operator, because a single string cannot equal an array of strings. You must use the `IN` operator.

```sql
-- Find all employees who work in a department located in New York
SELECT first_name, last_name 
FROM employees
WHERE department_id IN (
    SELECT id FROM departments WHERE location = 'New York'
);
```

### Level 3 — Intermediate (Correlated Subqueries)
The previous examples were **Uncorrelated Subqueries**. The inner query was completely independent; you could highlight it, run it by itself, and get an answer.

A **Correlated Subquery** depends on data from the Outer Query. It cannot run independently. Because it depends on the outer query, the database is forced to execute the subquery *over and over again, once for every single row* in the outer table!

```sql
-- Find employees who make more than the average salary OF THEIR SPECIFIC DEPARTMENT.
SELECT e.first_name, e.salary, e.department_id
FROM employees e
WHERE e.salary > (
    -- This inner query uses 'e.department_id' from the outer query!
    SELECT AVG(salary) 
    FROM employees 
    WHERE department_id = e.department_id
);
```
*Warning: Correlated subqueries can be incredibly slow on large tables because they execute repeatedly in a loop.*

### Level 4 — Advanced (Subqueries in the FROM Clause)
A `SELECT` statement returns a table of data. Therefore, you can put a subquery in the `FROM` clause and query it as if it were a real, physical table! This is known as a **Derived Table** or an **Inline View**.

*(Note: Every derived table MUST be given an alias!)*

```sql
-- We want to find the maximum order total for each user, 
-- and then find the average of those maximums!

SELECT AVG(max_order) AS average_of_maximums
FROM (
    -- This subquery creates a temporary 'table' in memory
    SELECT user_id, MAX(total_amount) AS max_order
    FROM orders
    GROUP BY user_id
) AS user_max_orders; -- You MUST alias the derived table!
```

## REAL_WORLD
**Subqueries vs JOINs**
Many subqueries (especially those using `IN`) can be rewritten as `JOIN` statements.
```sql
-- Using a Subquery
SELECT name FROM employees WHERE dept_id IN (SELECT id FROM departments WHERE name = 'IT');

-- Using a JOIN
SELECT e.name 
FROM employees e 
JOIN departments d ON e.dept_id = d.id 
WHERE d.name = 'IT';
```
Which is better? Historically, `JOIN`s were much faster because database optimizers struggled with subqueries. Today, modern databases (like Postgres 13+) have incredibly smart query optimizers that will automatically rewrite your subquery into a `JOIN` under the hood before executing it. You should prioritize whichever syntax is easier for you and your team to read and maintain!

## INTERVIEW
**Q: I have a query: `SELECT * FROM users WHERE id NOT IN (SELECT user_id FROM orders)`. It is running incredibly slow. How can I optimize it?**
A: The `NOT IN` operator with a subquery is notoriously slow and dangerous, especially if the subquery returns `NULL` values (which will cause the entire query to return 0 rows!). The industry standard best practice is to rewrite `NOT IN` queries using a `LEFT JOIN` and checking for `NULL`, or using the `NOT EXISTS` operator.
```sql
-- Drastically faster and safer
SELECT u.* 
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE o.id IS NULL;
```
