## WHY
Fetching an entire table is rarely useful. You almost always need a specific subset of data: users who registered today, products that are out of stock, or invoices that are unpaid. The `WHERE` clause is the gatekeeper of SQL. It evaluates every single row in the table against a boolean condition, only returning the rows that evaluate to `TRUE`.

## THEORY
### Boolean Logic
The `WHERE` clause relies entirely on boolean logic. For every row, the database asks: "Is this condition True, False, or Unknown?" 
- If True, the row is included in the result.
- If False, the row is discarded.
- If Unknown (which happens when comparing against `NULL`), the row is discarded.

## CODE

### Level 1 — Beginner (Comparison Operators)
The basic operators are `=`, `!=` (or `<>`), `>`, `<`, `>=`, and `<=`.

```sql
-- Exact match
SELECT * FROM employees WHERE department = 'Sales';

-- Not equal
SELECT * FROM employees WHERE department != 'HR';

-- Numeric comparison
SELECT * FROM employees WHERE salary >= 80000;
```

### Level 2 — Building Blocks (Logical Operators: AND, OR)
You can chain multiple conditions together. 
*Warning:* `AND` has higher precedence than `OR`. Always use parentheses to make your logic explicit!

```sql
-- Both conditions must be true
SELECT * FROM employees 
WHERE department = 'Sales' AND salary > 80000;

-- Without parentheses, this means:
-- (Sales AND Salary > 80000) OR (HR)
SELECT * FROM employees 
WHERE department = 'Sales' AND salary > 80000 OR department = 'HR';

-- ✅ GOOD: Explicit parentheses
-- The employee must make > 80k AND be in either Sales or HR.
SELECT * FROM employees 
WHERE (department = 'Sales' OR department = 'HR') AND salary > 80000;
```

### Level 3 — Intermediate (IN and BETWEEN)
Writing massive `OR` chains (`WHERE dept = 'HR' OR dept = 'IT' OR dept = 'Sales'`) is tedious. The `IN` operator simplifies this drastically.

```sql
-- Match any value in the list
SELECT * FROM employees 
WHERE department IN ('HR', 'IT', 'Sales');

-- Match ranges using BETWEEN (Inclusive!)
-- This includes exactly 50000 and exactly 80000
SELECT * FROM employees 
WHERE salary BETWEEN 50000 AND 80000;
```

### Level 4 — Advanced (Pattern Matching with LIKE)
What if you need to find an employee, but you only know their last name starts with "Sm"? 
You use `LIKE` with wildcard characters:
- `%` represents zero, one, or multiple characters.
- `_` represents exactly one single character.

```sql
-- Starts with 'Sm'
SELECT * FROM employees WHERE last_name LIKE 'Sm%';

-- Ends with 'son'
SELECT * FROM employees WHERE last_name LIKE '%son';

-- Contains 'admin' anywhere in the string
SELECT * FROM employees WHERE role LIKE '%admin%';

-- The second character must be 'a'
SELECT * FROM employees WHERE first_name LIKE '_a%';
```

## REAL_WORLD
**Index Invalidation**
The `LIKE` operator is incredibly dangerous for database performance. 
If you have a B-Tree index on `last_name`, and you query `WHERE last_name LIKE 'Sm%'`, the database can use the index perfectly (it jumps directly to the "S" section of the index, just like looking up "Smith" in a phone book).

However, if you put a wildcard at the *beginning* of the string (`WHERE last_name LIKE '%son'`), the database cannot use the index! It has no idea what letter the word starts with, so it is forced to read every single row in the entire table (a Full Table Scan). Never use leading wildcards (`%text`) on massive tables; use dedicated Full-Text Search engines (like Elasticsearch or Postgres `tsvector`) instead.

## INTERVIEW
**Q: Why does `SELECT * FROM users WHERE age != 30` not return users whose age is `NULL`?**
A: In SQL, `NULL` does not mean "empty"; it means "Unknown." 
If you ask the database, "Is an Unknown age not equal to 30?", the database evaluates that as "Unknown" (not True). Because the `WHERE` clause only returns rows that evaluate to exactly `TRUE`, the `NULL` rows are discarded. To include them, you must explicitly ask for them: `WHERE age != 30 OR age IS NULL`.
