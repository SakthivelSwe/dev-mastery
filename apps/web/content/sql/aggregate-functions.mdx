## WHY
If your CEO asks, "What was our total revenue last month?", running `SELECT price FROM orders` and trying to add up 5 million numbers manually in Excel is impossible. Databases are optimized for math. **Aggregate Functions** allow you to perform calculations across millions of rows instantly, summarizing massive datasets into single, actionable numbers. 

## THEORY
### The Collapse
Standard functions (like `UPPER(name)`) operate on a single row and return a single result per row. Aggregate functions operate on a *group* of rows and "collapse" them down into a single result. You cannot mix raw row data with aggregated data without explicitly telling the database how to group the raw data!

## CODE

### Level 1 — Beginner (The Core 5 Functions)
SQL provides five standard aggregate functions: `COUNT()`, `SUM()`, `AVG()`, `MAX()`, and `MIN()`.

```sql
-- How many total orders exist?
SELECT COUNT(*) FROM orders;

-- What is the total revenue?
SELECT SUM(price) FROM orders;

-- What is the average price of a product?
SELECT AVG(price) FROM products;

-- What are the cheapest and most expensive products?
SELECT MIN(price), MAX(price) FROM products;
```
*Note: If you run these without a `GROUP BY` clause, the database treats the entire table as one single massive group, returning exactly one row of output.*

### Level 2 — Building Blocks (GROUP BY)
What if the CEO asks, "What is the total revenue *per department*?"
You must use the `GROUP BY` clause. This tells the database to split the table into distinct buckets (one for each department), and then run the aggregate function independently inside each bucket.

```sql
SELECT 
    department, 
    SUM(salary) AS total_payroll
FROM employees
GROUP BY department;
```
**The Golden Rule of GROUP BY:**
If you have a column in your `SELECT` clause (like `department`), and it is NOT inside an aggregate function (like `SUM`), it **MUST** be listed in the `GROUP BY` clause! If you try to `SELECT department, employee_name, SUM(salary) GROUP BY department`, the query will crash because the database doesn't know which `employee_name` to show!

### Level 3 — Intermediate (HAVING)
What if you want to find only the departments where the total payroll is greater than $1,000,000?
You might try this:
```sql
-- ❌ ERROR! This will crash!
SELECT department, SUM(salary) 
FROM employees 
WHERE SUM(salary) > 1000000 
GROUP BY department;
```
Why does it crash? Because of the SQL Order of Execution! The `WHERE` clause runs *before* the `GROUP BY`. You cannot filter on an aggregate sum that hasn't been calculated yet!

To filter *after* aggregation, you must use the `HAVING` clause:
```sql
-- ✅ CORRECT
SELECT department, SUM(salary) 
FROM employees 
GROUP BY department
HAVING SUM(salary) > 1000000;
```

### Level 4 — Advanced (COUNT(DISTINCT))
A common reporting mistake involves duplicate data.
If a user logs in 5 times a day, the `logins` table will have 5 rows for that user. If you want to know how many *unique* users logged in today, `COUNT(*)` will give you the wrong answer (it will return 5, not 1).

You combine `COUNT` with `DISTINCT` to count unique occurrences:
```sql
-- Total number of login events today
SELECT COUNT(*) FROM logins WHERE date = CURRENT_DATE;

-- Number of UNIQUE users who logged in today
SELECT COUNT(DISTINCT user_id) FROM logins WHERE date = CURRENT_DATE;
```

## REAL_WORLD
**NULL Values in Aggregates**
Aggregate functions completely ignore `NULL` values!
Imagine a table with 3 employees. Their salaries are: `$100,000`, `$50,000`, and `NULL` (a contractor).
If you run `SELECT AVG(salary) FROM employees`, the database will calculate `(100,000 + 50,000) / 2 = $75,000`. It completely ignores the third employee. 

If you wanted the `NULL` to count as `$0`, you must use the `COALESCE` function to convert the `NULL` to a `0` before averaging it: 
`SELECT AVG(COALESCE(salary, 0)) FROM employees` -> `(100,000 + 50,000 + 0) / 3 = $50,000`.

## INTERVIEW
**Q: What is the difference between `WHERE` and `HAVING`?**
A: Both clauses are used to filter data, but they execute at completely different stages of the query. 
- **`WHERE`** runs *before* aggregation. It filters individual rows (e.g., `WHERE status = 'active'`). It cannot contain aggregate functions.
- **`HAVING`** runs *after* aggregation. It filters the grouped buckets (e.g., `HAVING SUM(amount) > 100`). It is almost always used in conjunction with a `GROUP BY` clause.
