## WHY
Sometimes, the data in your database doesn't perfectly match what you want to display on the frontend. A user might have a `role_id` of `1` or `2`, but the frontend needs to render the text `"Admin"` or `"User"`. Instead of pulling the raw integers into your Node.js backend and running a massive `switch` statement over 10,000 rows in JavaScript, you can use the SQL **CASE Expression** to execute `If/Else` logic natively inside the database engine before the data is even returned!

## THEORY
### Expressions, Not Statements
It is crucial to understand that `CASE` is an *expression*, not a statement. It evaluates to a single value. You cannot use it to execute different SQL queries (e.g., "If condition A, run this `SELECT`; else run this `UPDATE`"). You use it inline, exactly where a column name or a value would go.

## CODE

### Level 1 — Beginner (Simple CASE)
The Simple CASE expression compares a single column against a list of exact equality matches. It acts exactly like a `switch` statement.

```sql
-- Convert numeric status codes into readable text for the UI!
SELECT 
    order_id,
    status_code,
    CASE status_code
        WHEN 1 THEN 'Pending'
        WHEN 2 THEN 'Processing'
        WHEN 3 THEN 'Shipped'
        WHEN 4 THEN 'Delivered'
        ELSE 'Unknown Status'
    END AS status_label
FROM orders;
```
*(If you forget the `ELSE` clause, and a row doesn't match any of the `WHEN` conditions, the `CASE` expression will return `NULL` for that row).*

### Level 2 — Building Blocks (Searched CASE)
What if you need to use greater-than/less-than logic? The Simple CASE cannot do this. You must use the **Searched CASE** expression, which evaluates full boolean conditions.

```sql
-- Categorize products based on their price
SELECT 
    name,
    price,
    CASE
        WHEN price > 1000 THEN 'Luxury'
        WHEN price > 500  THEN 'Premium'
        WHEN price > 100  THEN 'Standard'
        ELSE 'Budget'
    END AS price_tier
FROM products;
```
*Note: A `CASE` expression evaluates top-to-bottom. The very first `WHEN` clause that evaluates to `TRUE` wins, and the database completely ignores the remaining clauses!*

### Level 3 — Intermediate (Conditional Aggregation)
This is one of the most powerful techniques in data engineering. 
What if the CEO wants a report showing the total number of orders, but split into three columns: "Pending", "Shipped", and "Canceled"? 

You could run 3 separate `SELECT COUNT(*)` queries with 3 different `WHERE` clauses. Or, you can use a `CASE` inside an aggregate function to build a Pivot Table!

```sql
-- 1 single query returns 3 distinct counts!
SELECT 
    -- If it's pending, return 1, otherwise return NULL.
    -- COUNT() ignores NULLs, so it only counts the pending ones!
    COUNT(CASE WHEN status = 'Pending' THEN 1 END) AS pending_orders,
    
    COUNT(CASE WHEN status = 'Shipped' THEN 1 END) AS shipped_orders,
    
    COUNT(CASE WHEN status = 'Canceled' THEN 1 END) AS canceled_orders
FROM orders;
```
*(In PostgreSQL 9.4+, you can use the much cleaner `FILTER` clause: `COUNT(*) FILTER (WHERE status = 'Pending')`)*

### Level 4 — Advanced (Updating with CASE)
You can use `CASE` inside an `UPDATE` statement to execute incredibly complex, dynamic updates across a massive table in a single atomic transaction.

```sql
-- Give all employees a raise!
-- Admins get 5%, Managers get 10%, everyone else gets 2%.
UPDATE employees
SET salary = salary * (
    CASE 
        WHEN role = 'Admin'   THEN 1.05
        WHEN role = 'Manager' THEN 1.10
        ELSE 1.02
    END
);
```

## REAL_WORLD
**Preventing Division by Zero**
As learned in the NULL Handling module, `NULLIF` is the standard way to prevent division by zero. However, `CASE` expressions give you complete programmatic control over mathematical errors.

```sql
-- Calculate a safe conversion rate!
SELECT 
    clicks, 
    purchases,
    CASE 
        WHEN clicks = 0 THEN 0.00 -- If clicks are 0, the rate is 0.
        ELSE (purchases::DECIMAL / clicks) * 100 
    END AS conversion_rate_percentage
FROM ad_campaigns;
```

## INTERVIEW
**Q: Can you put a `CASE` expression inside an `ORDER BY` clause?**
A: **Yes!** And it is an incredibly powerful trick. Imagine you have an e-commerce store, and the CEO demands that "Out of Stock" items are forced to the very bottom of the search results, but everything else is sorted by price.
You simply put a `CASE` expression in the `ORDER BY`!
```sql
SELECT name, price, stock_status 
FROM products
ORDER BY 
    CASE WHEN stock_status = 'Out of Stock' THEN 2 ELSE 1 END ASC,
    price ASC;
```
This forces all "In Stock" items to be grouped together at the top (Value 1) and sorted by price, while pushing the "Out of Stock" items to the bottom (Value 2).
