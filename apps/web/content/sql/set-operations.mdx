## WHY
A `JOIN` combines columns horizontally (adding the 'department_name' column to the 'employee' row). But what if you have a `current_employees` table and a `former_employees` table, and you want to combine them vertically into one massive list? 

**Set Operations** allow you to stack the results of two completely independent `SELECT` queries on top of each other, compare them to find overlaps, or subtract them to find differences.

## THEORY
### The Strict Rules of Set Operations
To use a Set Operator (like `UNION`), the two `SELECT` queries you are combining must strictly follow these rules:
1. They must have the exact same **number of columns**.
2. The columns must be in the exact same **order**.
3. The data types of the corresponding columns must **match** (or be highly compatible).

## CODE

### Level 1 — Beginner (UNION)
The `UNION` operator stacks the results of two queries and **automatically removes all duplicates**.

```sql
-- Get a master list of all email addresses we have on file.
-- If Alice is both an employee and a customer, her email will only appear ONCE.
SELECT email FROM employees
UNION
SELECT email FROM customers;
```

### Level 2 — Building Blocks (UNION ALL)
Removing duplicates is a very expensive mathematical operation for the database engine. It has to sort the entire massive combined list and filter out matching rows. 
If you know for a fact that there are no duplicates, or if you actually *want* the duplicates, you should use `UNION ALL`.

```sql
-- Stack the lists instantly. Do not waste CPU checking for duplicates!
-- If Alice is in both tables, her email will appear TWICE.
SELECT email FROM employees
UNION ALL
SELECT email FROM customers;
```
*Best Practice: Always use `UNION ALL` unless you specifically require duplicate removal.*

### Level 3 — Intermediate (INTERSECT)
`INTERSECT` returns *only* the rows that exist in **both** queries. It acts like an `AND` for entire rows.

```sql
-- Which of our employees are also paying customers?
SELECT email FROM employees
INTERSECT
SELECT email FROM customers;
```
*Note: MySQL does not natively support the `INTERSECT` keyword. In MySQL, you must mimic this behavior using an `INNER JOIN`.*

### Level 4 — Advanced (EXCEPT / MINUS)
`EXCEPT` (called `MINUS` in Oracle) acts as subtraction. It takes the results of the first query, and removes any rows that also appear in the second query.

```sql
-- Give me all the employees EXCEPT those who are also customers.
-- (Only returns employees who have never bought anything).
SELECT email FROM employees
EXCEPT
SELECT email FROM customers;
```

**Order Matters!**
Unlike `UNION` and `INTERSECT` (which are commutative), the order of queries in an `EXCEPT` statement completely changes the result.
```sql
-- Give me all the customers EXCEPT those who are also employees.
-- (Returns normal public customers).
SELECT email FROM customers
EXCEPT
SELECT email FROM employees;
```

## REAL_WORLD
**Faking Missing Columns**
What if the `employees` table has `id, name, department`, but the `customers` table only has `id, name`? If you try to `UNION` them, the database will throw an error because the column counts don't match (3 vs 2).

You can "fake" the missing column in the second query by hardcoding a `NULL` (or a static string)!

```sql
-- Stacking tables with different schemas!
SELECT id, name, department 
FROM employees

UNION ALL

SELECT id, name, NULL AS department 
FROM customers;
```

## INTERVIEW
**Q: I have two tables with 1 million rows each. I need to stack them together. Should I use `UNION` or `UNION ALL`?**
A: You should almost always use `UNION ALL`. `UNION` forces the database to sort 2 million rows in memory to identify and remove duplicates. This sorting process on massive datasets will likely spill from RAM onto the physical hard drive, taking minutes to execute. If you know the two tables are mutually exclusive (e.g., `archived_logs_2022` and `archived_logs_2023`), or if duplicates don't matter, `UNION ALL` will execute instantly because it simply appends the data streams without any processing.
