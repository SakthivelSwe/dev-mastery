## WHY
In SQL, `NULL` is the source of more bugs than almost any other concept. If you ask a database `SELECT 5 + NULL`, it doesn't throw an error, and it doesn't return `5`. It returns `NULL`. 

`NULL` does not mean "Zero" and it does not mean "Empty String". `NULL` means **"Unknown"** or **"Missing"**. If you do not explicitly handle `NULL` values in your queries, your mathematical calculations, string concatenations, and `WHERE` clauses will fail silently and return catastrophically incorrect data to your users.

## THEORY
### Three-Valued Logic
Most programming languages use Boolean Logic: an expression is either `True` or `False`. 
SQL uses **Three-Valued Logic**: an expression can be `True`, `False`, or `Unknown` (`NULL`).

If you check `WHERE age > 18`, and a row has a `NULL` age, the database evaluates the expression as `Unknown`. The `WHERE` clause strictly requires `True` to include a row, so the `Unknown` row is thrown away.

## CODE

### Level 1 — Beginner (IS NULL and IS NOT NULL)
You can NEVER use the standard equals sign (`=`) to check for `NULL`.
If you type `WHERE age = NULL`, you are asking "Is 25 equal to Unknown?". The answer is Unknown! The query will return 0 rows.

```sql
-- ❌ BAD: This will always return 0 rows!
SELECT * FROM employees WHERE bonus = NULL;

-- ✅ GOOD: You must use the IS operator!
SELECT * FROM employees WHERE bonus IS NULL;

-- Find employees who DO have a bonus
SELECT * FROM employees WHERE bonus IS NOT NULL;
```

### Level 2 — Building Blocks (COALESCE)
The most important `NULL` handling function in all of SQL. 
`COALESCE(val1, val2, val3)` accepts a list of arguments and returns the very first argument that is **NOT NULL**.

```sql
-- If 'bonus' is NULL, return a 0 instead!
SELECT first_name, salary, COALESCE(bonus, 0) AS safe_bonus 
FROM employees;

-- Safe Math!
-- If bonus is NULL, 'salary + bonus' becomes NULL.
-- By wrapping it in COALESCE, we guarantee a safe number!
SELECT salary + COALESCE(bonus, 0) AS total_compensation 
FROM employees;
```

### Level 3 — Intermediate (NULLIF)
`NULLIF(val1, val2)` does the exact opposite. It returns `NULL` if the two arguments are completely identical. If they are different, it returns the first argument.

Why is this useful? To prevent the most dreaded mathematical error: **Division by Zero!**
If you try to calculate `revenue / transactions`, and `transactions` is `0`, the query crashes.

```sql
-- If transactions is 0, NULLIF turns it into NULL.
-- revenue / NULL evaluates to NULL (safe!), instead of crashing the database!
SELECT 
    revenue, 
    transactions, 
    revenue / NULLIF(transactions, 0) AS average_order_value 
FROM sales;
```

### Level 4 — Advanced (NULLs in Sorting)
When you use `ORDER BY`, where do the `NULL` values go? Do they sort to the very top, or the very bottom?
It depends entirely on the database dialect! (Postgres puts them at the bottom for DESC, Oracle puts them at the top).

To ensure your API behaves consistently regardless of the database engine, you must explicitly declare where `NULL`s should go using `NULLS FIRST` or `NULLS LAST`.

```sql
-- Ensure users with no 'last_login' date are pushed to the very bottom
-- of the list, regardless of the database engine!
SELECT username, last_login 
FROM users
ORDER BY last_login DESC NULLS LAST;
```

## REAL_WORLD
**String Concatenation with NULLs**
If you want to combine a user's `first_name`, `middle_name`, and `last_name`, you might use standard concatenation:
```sql
SELECT first_name || ' ' || middle_name || ' ' || last_name AS full_name;
```
If the user doesn't have a middle name (it is `NULL`), the entire `full_name` will instantly become `NULL`! `String + NULL = NULL`.

You must wrap optional strings in `COALESCE`:
```sql
SELECT first_name || ' ' || COALESCE(middle_name, '') || ' ' || last_name;
-- (Note: MySQL's CONCAT() function actually ignores NULLs automatically, 
-- but Postgres and Oracle standard || operators do not!)
```

## INTERVIEW
**Q: I have a table of 100 users. I run `SELECT COUNT(age) FROM users` and it returns 90. Why didn't it return 100?**
A: Aggregate functions (like `SUM`, `AVG`, `COUNT(column)`) completely ignore `NULL` values. The result of `90` means there are exactly 10 users in the table whose `age` column is `NULL`. If you want to count every single physical row in the table regardless of what data is inside it, you must use `COUNT(*)`.
