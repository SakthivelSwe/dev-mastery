## WHY
The `SELECT` statement is the most frequently used command in SQL. It is the core mechanism for retrieving data from a database. While `SELECT * FROM users` is easy to learn, mastering how to alias columns, eliminate duplicates, and shape the returned result set is essential for building efficient APIs.

## THEORY
### Projection
In relational algebra, selecting specific columns from a table is known as **Projection**. When you run a `SELECT` statement, you are projecting a subset of the table's columns into a brand new, temporary result table.

### The Order of Execution
It is absolutely critical to understand that SQL is a declarative language. You write the query in a specific order (e.g., `SELECT ... FROM ... WHERE ...`), but the database engine executes it in a completely different order!
1. `FROM` (Locate the tables)
2. `WHERE` (Filter the rows)
3. `SELECT` (Extract the requested columns)

Because `SELECT` happens *after* `WHERE`, you cannot use a column alias defined in the `SELECT` clause inside your `WHERE` clause!

## CODE

### Level 1 — Beginner (Basic SELECT)
The asterisk (`*`) is a wildcard that returns every single column in the table.

```sql
-- Fetch all columns and all rows
SELECT * FROM products;

-- Fetch ONLY the specific columns you need (Best Practice)
SELECT name, price FROM products;
```
*Why is `SELECT *` bad in production?* If your table has 50 columns (including massive text descriptions) and your API only needs the product name and price, `SELECT *` forces the database to read massive amounts of unnecessary data from the hard drive and send it over the network, destroying performance. Always list your columns explicitly!

### Level 2 — Building Blocks (Aliases with AS)
If your database column is named `usr_fst_nm`, you don't want to send that ugly key back in your JSON API response. You can rename columns on the fly using the `AS` keyword.

```sql
-- Rename columns in the output
SELECT 
    usr_fst_nm AS first_name, 
    usr_lst_nm AS last_name 
FROM users;

-- You can also use aliases to name the result of mathematical operations!
SELECT 
    name, 
    price, 
    price * 0.20 AS tax_amount 
FROM products;
```

### Level 3 — Intermediate (DISTINCT)
If you have 10,000 users, and you want a list of all the different countries they live in, running `SELECT country FROM users` will return 10,000 rows, with "USA" repeating thousands of times.

The `DISTINCT` keyword removes duplicate rows from the result set.

```sql
-- Returns a list of unique countries
SELECT DISTINCT country FROM users;

-- You can use DISTINCT across multiple columns.
-- This returns unique COMBINATIONS of country and city.
SELECT DISTINCT country, city FROM users;
```

### Level 4 — Advanced (LIMIT and OFFSET)
If you run `SELECT * FROM transactions`, and the table has 50 million rows, your database will try to send 50 million rows to your Node.js backend. Your Node.js server will instantly run out of RAM and crash.

You must implement **Pagination**.
- `LIMIT`: Restricts the maximum number of rows returned.
- `OFFSET`: Skips a specific number of rows before beginning to return data.

```sql
-- Page 1: Fetch the first 10 products
SELECT name, price FROM products
ORDER BY id ASC
LIMIT 10 OFFSET 0;

-- Page 2: Skip the first 10, fetch the next 10
SELECT name, price FROM products
ORDER BY id ASC
LIMIT 10 OFFSET 10;
```
*(Always use `ORDER BY` when using `LIMIT`/`OFFSET`, otherwise the database will return rows in a completely random, unpredictable order!)*

## REAL_WORLD
**Keyset Pagination (Cursor Pagination)**
`OFFSET` is notoriously slow for massive tables. If you run `LIMIT 10 OFFSET 1000000`, the database engine actually has to read 1,000,010 rows from the hard drive, throw away the first million, and return the last 10. 

Modern APIs (like Twitter or Instagram) use **Cursor Pagination** instead.
Instead of passing an offset, the client passes the ID of the very last item they saw on the previous page:
```sql
-- Extremely fast, because it uses the B-Tree Index to instantly jump to ID 1000000!
SELECT name, price FROM products
WHERE id > 1000000
ORDER BY id ASC
LIMIT 10;
```

## INTERVIEW
**Q: You run `SELECT count(name) FROM users` and it returns 95. You run `SELECT count(*) FROM users` and it returns 100. Why are the numbers different?**
A: `COUNT(*)` counts the absolute number of physical rows in the table, regardless of what data is inside them. `COUNT(column_name)` counts the number of rows where that specific column is **NOT NULL**. The difference means that there are exactly 5 users in the database whose `name` column is currently set to `NULL`.
