## WHY
Not all files in your project should be tracked by Git. Node modules, compiled output, local environment configs, IDE settings, and operating system files should never be committed. Without a proper `.gitignore`, these clutter your repository and can expose sensitive data.

## THEORY
### What to Exclude
- **Dependencies:** `node_modules/`, `vendor/`, `target/` (compiled classes)
- **Build output:** `dist/`, `build/`, `.next/`, `out/`
- **Environment files:** `.env`, `.env.local`, `.env.production` — these contain secrets!
- **IDE files:** `.idea/`, `.vscode/settings.json`, `*.iml`
- **OS files:** `.DS_Store` (Mac), `Thumbs.db` (Windows)
- **Logs:** `*.log`, `npm-debug.log*`

### .gitignore Patterns
```bash
# Exact file
secret.key

# All files with extension
*.log
*.env

# Directory (trailing slash)
node_modules/
build/

# All files in a directory
logs/**

# Specific path
src/temp/
/root-level-only.txt   # Only root, not subdirectories

# Negate (un-ignore)
!important.log   # Track this log even though *.log is ignored

# Wildcards
*.{jpg,png,gif}  # All image files
docs/**/*.pdf    # PDFs anywhere in docs/
```

### Fixing Already-Committed Files
```bash
# You accidentally committed node_modules!
# Step 1: Add to .gitignore
echo "node_modules/" >> .gitignore

# Step 2: Remove from Git tracking WITHOUT deleting the files
git rm -r --cached node_modules/

# Step 3: Commit the removal
git add .gitignore
git commit -m "Remove node_modules from tracking"
git push

# Now node_modules/ is in Git history but won't be tracked going forward
# Other developers can delete it from their machines (git pull won't restore it)
```

### Global Gitignore (For Your Machine)
```bash
# Ignore IDE files everywhere without putting them in every project's .gitignore
git config --global core.excludesfile ~/.gitignore_global

cat ~/.gitignore_global
# .DS_Store
# .idea/
# *.iml
# .vscode/
# Thumbs.db
```

## REAL_WORLD
**Using gitignore.io**
The website [gitignore.io](https://gitignore.io) generates `.gitignore` files based on your tech stack:

```bash
# Generate .gitignore for Node, Java, IntelliJ, Mac
curl -sL https://gitignore.io/api/node,java,intellij,macos > .gitignore
```

## INTERVIEW
**Q: Your team member committed a `.env` file containing a database password to a public GitHub repository 3 months ago. They removed the file in the next commit, but the damage is done. What must you do?**

A: Removing the file in a later commit doesn't help — the `.env` is still visible in the commit history. Anyone can run `git log` and see it.

**Immediate action (most critical):**
1. **Rotate the database password NOW** — assume it's compromised. Update it in the database and all deployment environments. This is non-negotiable.
2. Check your database access logs for any unauthorized connections since the exposure.

**Clean up the repository:**
3. Use BFG Repo Cleaner to permanently remove the file from all commit history:
```bash
java -jar bfg.jar --delete-files .env
git reflog expire --expire=now --all && git gc --prune=now
git push --force --all
git push --force --tags
```
4. Contact GitHub Support to purge their cached copies.

**Prevention:**
5. Add `.env*` to `.gitignore` in ALL projects (use global gitignore!)
6. Set up `gitleaks` as a pre-commit hook to scan for secrets:
```bash
brew install gitleaks
gitleaks protect --source=. --staged  # Run before commit
```
7. Enable GitHub's secret scanning on all repositories (detects many credential patterns automatically).
