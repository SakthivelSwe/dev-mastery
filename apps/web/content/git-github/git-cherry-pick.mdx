## WHY
Sometimes you need just ONE specific commit from another branch — not the entire branch. Maybe a hotfix on the `develop` branch needs to go to `main` without merging everything else on `develop`. Or a useful utility function was committed to someone else's feature branch and you need it now.
**Git Cherry-Pick** copies one or more specific commits from one branch to another.

## THEORY
### What Cherry-Pick Does
Cherry-pick takes the diff (changes) introduced by a commit and applies them to your current branch as a NEW commit. The new commit has a different SHA-1 hash (different parent) but the same content changes.

### Cherry-Pick vs Merge
- **Merge:** Combines entire branch histories
- **Cherry-Pick:** Copies specific commit(s) only

## CODE

### Basic Cherry-Pick
```bash
# Scenario: develop branch has a security fix that also needs to go to main
git log develop --oneline
# d4e5f6c Add security headers middleware   ← Need this commit!
# c3d4e5f Add dark mode feature
# b2c3d4e Refactor user service

git checkout main
git cherry-pick d4e5f6c

# Result: The security fix is applied to main as a NEW commit
git log main --oneline
# a9b8c7d Add security headers middleware   ← Same content, NEW hash!
# e5f4a3b Previous main commit
```

### Cherry-Pick Multiple Commits
```bash
# Pick a range of commits
git cherry-pick a1b2c3d..d4e5f6c  # All commits between (exclusive start)
git cherry-pick a1b2c3d^..d4e5f6c  # All commits including start

# Pick non-contiguous commits
git cherry-pick abc1234 def5678 ghi9012  # Three specific commits

# Cherry-pick WITHOUT committing (apply changes only)
git cherry-pick -n abc1234   # -n = --no-commit
# Makes the changes but doesn't commit — you can edit them first!
git commit -m "Apply security fix with additional changes"
```

### Handling Cherry-Pick Conflicts
```bash
git cherry-pick d4e5f6c
# CONFLICT (content): Merge conflict in src/middleware.js

# Resolve the conflict, then:
git add src/middleware.js
git cherry-pick --continue   # Complete the cherry-pick

# Or abandon it:
git cherry-pick --abort
```

## REAL_WORLD
**Hotfix Workflow with Cherry-Pick**
```bash
# Scenario: Production bug discovered!
# You can't deploy all of develop to production — it has unreleased features.
# But the fix is already committed on develop.

git log develop --oneline
# g7h8i9 Critical payment bug fix         ← Need this!
# f6g7h8 Add experimental recommendation engine (not ready!)
# e5f6g7 Add beta features flag

# 1. Branch from main (production state)
git checkout main
git checkout -b hotfix/payment-fix

# 2. Cherry-pick only the fix
git cherry-pick g7h8i9

# 3. Deploy the hotfix branch to production
git tag -a v2.3.1 -m "Hotfix: payment calculation bug"
git push origin hotfix/payment-fix

# 4. Merge the hotfix to main
git checkout main
git merge hotfix/payment-fix

# The fix is now in production without any unreleased features!
```

## INTERVIEW
**Q: You cherry-picked a commit from branch A to branch B. Now a teammate merged branch A to main. Will there be a duplicate commit or conflict?**

A: You'll likely see a **duplicate commit** — two commits with the same logical change but different SHA-1 hashes appear in the history. This is generally harmless (the content is the same), but it can create confusion.

When branch A is later merged to main, Git sees the cherry-picked commit and the original commit as different commits (different hashes). The merge will apply the original commit's changes, but since the changes are already in main (via cherry-pick), there may be a conflict — or Git may silently include the changes twice.

**To avoid this:**
- If you know branch A will eventually be merged to main, use `git rebase` or a merge instead of cherry-picking
- Cherry-pick is best for permanent "copy to another branch" operations, not temporary ones
- If you're cherry-picking to avoid including unrelated commits, consider a separate branch for just that fix from the start

In practice, most teams accept the duplicate commit in history when cherry-picking hotfixes, as the alternatives (blocking the hotfix until branch A is ready) are worse.
