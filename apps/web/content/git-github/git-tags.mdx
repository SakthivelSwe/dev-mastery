## WHY
How do you mark the exact commit that went to production as "v2.3.1"? How do you go back to exactly the code that was deployed on March 15th?
**Git Tags** are permanent labels attached to specific commits. Unlike branches (which move forward with each commit), tags are immutable — they always point to the same commit forever. They are the standard way to mark release versions in Git.

## THEORY
### Lightweight vs Annotated Tags
- **Lightweight tag:** Just a pointer to a commit (no metadata). Like a branch that never moves.
- **Annotated tag:** A full Git object with tagger name, email, date, and message. Recommended for releases.

## CODE

### Creating and Managing Tags
```bash
# Create a lightweight tag
git tag v1.0.0

# Create an annotated tag (recommended for releases!)
git tag -a v2.1.0 -m "Release 2.1.0: Add payment gateway integration"

# Tag a specific past commit
git tag -a v1.5.0 -m "Retroactive tag for Jan release" abc1234

# List all tags
git tag

# Filter tags
git tag -l "v1.*"
# v1.0.0
# v1.1.0
# v1.2.0

# Show tag details
git show v2.1.0
# tag v2.1.0
# Tagger: Alice Smith <alice@example.com>
# Date:   Mon Jan 28 15:00:00 2024
#
# Release 2.1.0: Add payment gateway integration
#
# commit abc1234 (HEAD -> main, tag: v2.1.0)
# ...

# Push tags to remote (tags are NOT pushed with regular git push!)
git push origin v2.1.0        # Push specific tag
git push origin --tags        # Push ALL tags

# Delete a local tag
git tag -d v1.0.0-beta

# Delete a remote tag
git push origin --delete v1.0.0-beta

# Checkout to a tag (creates detached HEAD)
git checkout v2.0.0
```

### Semantic Versioning with Git Tags
```bash
# Semantic Versioning: MAJOR.MINOR.PATCH
# MAJOR: Breaking changes (incompatible API changes)
# MINOR: New features (backward-compatible)
# PATCH: Bug fixes (backward-compatible)

git tag -a v1.0.0 -m "Initial stable release"
git tag -a v1.1.0 -m "Add dark mode feature"
git tag -a v1.1.1 -m "Fix dark mode text color bug"
git tag -a v2.0.0 -m "New API: Breaking changes to authentication"

# GitHub shows tags as Releases with downloadable archives!
```

## REAL_WORLD
**CI/CD Triggered by Tags**
Many teams use tags to trigger production deployments:
```yaml
# .github/workflows/release.yml
on:
  push:
    tags:
      - 'v*'    # Trigger on any tag starting with 'v'

jobs:
  deploy:
    steps:
      - name: Deploy to Production
        run: deploy.sh ${{ github.ref_name }}  # v2.1.0
```

```bash
# To deploy to production:
git tag -a v2.1.0 -m "Release 2.1.0"
git push origin v2.1.0
# ← This triggers the GitHub Actions deployment automatically!
```

## INTERVIEW
**Q: You need to find which tag corresponds to the version that introduced a bug. You know the bug appeared sometime in Q3 2023. How do you find the relevant tags?**

```bash
# List tags with dates (annotated tags show dates!)
git log --tags --simplify-by-decoration --pretty="format:%d %ai" \
  --after="2023-07-01" --before="2023-10-01"

# Output:
# (tag: v3.2.0) 2023-09-15 14:30:00
# (tag: v3.1.0) 2023-08-20 10:15:00
# (tag: v3.0.0) 2023-07-05 09:00:00

# Check out each tag to test if the bug exists:
git checkout v3.1.0
./run-tests.sh  # Test passes? Bug introduced after v3.1.0

git checkout v3.2.0
./run-tests.sh  # Test fails? Bug introduced in v3.2.0!

# Find the exact commit using git bisect between these two tags:
git bisect start v3.2.0 v3.1.0
```

A: Combine `git log` with `--tags` and date filters to list releases in the time period. Then use `git bisect` between the last good tag and first bad tag to pinpoint the exact commit.
