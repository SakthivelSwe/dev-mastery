## WHY
Having branches isn't enough — your team needs a shared agreement on HOW to use them. Without a workflow, every developer branches and merges differently, creating chaos and frequent broken builds.
**Git Workflows** are standardized branching strategies that define rules for when to create branches, how to name them, and how/when to merge them. The right workflow reduces merge conflicts, enables parallel development, and keeps releases predictable.

## THEORY
### The Main Git Workflows

**1. GitHub Flow** — Simple, recommended for most teams:
- One long-lived branch: `main`
- Feature branches → PR → merge to main → deploy

**2. Git Flow** — Complex, for teams with scheduled releases:
- Long-lived: `main`, `develop`
- Short-lived: `feature/*`, `release/*`, `hotfix/*`

**3. Trunk-Based Development (TBD)** — Extreme CI/CD:
- Everyone commits to `main` (or very short-lived branches of 1-2 days)
- Feature flags for incomplete features

## CODE

### GitHub Flow (Recommended for Most Teams)
```bash
# The entire workflow in 7 steps:

# 1. Update main
git checkout main && git pull origin main

# 2. Create feature branch
git checkout -b feature/stripe-payment

# 3. Work and commit
git add . && git commit -m "Add Stripe SDK integration"
git add . && git commit -m "Add payment processing service"

# 4. Push branch
git push origin feature/stripe-payment

# 5. Open PR → Code Review → CI passes

# 6. Merge (via GitHub UI or CLI)
gh pr merge --squash --delete-branch

# 7. Deploy main (automatic via CI/CD)

# REPEAT for every feature!
```

### Git Flow (Complex Scheduled Releases)
```bash
# Initialize git flow (install: brew install git-flow)
git flow init

# Start a feature
git flow feature start user-authentication
# Creates branch: feature/user-authentication from develop

# Finish the feature (merges to develop, deletes feature branch)
git flow feature finish user-authentication

# Start a release
git flow release start 1.2.0
# Creates branch: release/1.2.0 from develop

# On release branch: only bug fixes, version bumps, docs
git commit -m "Bump version to 1.2.0"
git commit -m "Fix last-minute checkout bug"

# Finish release (merges to both main AND develop, creates tag)
git flow release finish 1.2.0

# Start a hotfix (for critical production bugs)
git flow hotfix start payment-security-patch
# Creates branch: hotfix/payment-security-patch from main!

git flow hotfix finish payment-security-patch
# Merges to both main AND develop, creates tag
```

### Trunk-Based Development (TBD)
```bash
# In TBD, you either commit directly to main, or use very short branches
# Feature flags are used to hide incomplete features

git checkout main
git pull origin main
git checkout -b feature/new-recommendation-engine  # Max 1-2 days!

# ... write code with feature flag ...
if (process.env.ENABLE_NEW_RECOMMENDATIONS === 'true') {
  return newRecommendationEngine.getSuggestions(user);
}
return legacyEngine.getSuggestions(user);

git commit -m "Add new recommendation engine (behind feature flag)"
git push origin feature/new-recommendation-engine

# Create PR → merge to main TODAY or tomorrow
# Feature is deployed but hidden by flag
# Enable flag gradually: 1% → 10% → 100% of users
```

## REAL_WORLD
**Choosing the Right Workflow**

| Team Size | Release Cadence | Recommended Workflow |
|---|---|---|
| 1-5 devs | Continuous (multiple times/day) | GitHub Flow or TBD |
| 5-20 devs | Weekly | GitHub Flow |
| 20+ devs | Scheduled (biweekly/monthly) | Git Flow |
| Any size | Continuous with feature flags | Trunk-Based Development |

**Most modern teams** (including Google, Facebook, Netflix) use Trunk-Based Development or GitHub Flow. Git Flow has fallen out of favor because long-lived branches cause "integration hell" — merging a 3-week-old branch to main can cause dozens of conflicts.

## INTERVIEW
**Q: Your team currently uses Git Flow with 2-week release cycles. The CEO wants to move to deploying 10 times per day. What workflow changes would you recommend and what technical practices must the team adopt?**
A: Moving from Git Flow to Continuous Deployment requires:

**Workflow Change:** Adopt **Trunk-Based Development**:
- Delete `develop` branch — everyone pushes to `main` (or ultra-short feature branches of 1-2 days)
- Delete `release` branches — releases happen automatically from `main`
- Hotfixes become just regular commits with immediate deployment

**Technical Requirements (the hard part):**
1. **Feature Flags:** Every incomplete feature must be behind a feature flag. You deploy code, but the feature is hidden until the flag is enabled.
2. **Comprehensive CI:** Tests must pass in < 10 minutes (fast feedback). Slow tests kill CD.
3. **Automated Testing:** 80%+ code coverage with automated integration/E2E tests so you trust the pipeline.
4. **Database Migrations:** Must be backward-compatible. Cannot break the current running version. Use expand-contract pattern.
5. **Observability:** Real-time monitoring and instant rollback capability (canary deployments, feature flag kill switches).
6. **Deployment Pipeline:** Automated blue-green or canary deployments — no manual deployment steps.

This is a 6-12 month transformation, not a weekend project.
