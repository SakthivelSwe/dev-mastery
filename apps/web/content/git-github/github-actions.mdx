## WHY
Every software project needs automated checks: run tests when code is pushed, build Docker images, deploy to staging, check code quality. Without automation, developers must remember to do all of this manually — and they forget.
**GitHub Actions** is GitHub's built-in CI/CD platform. It lets you define automated workflows that trigger on Git events (push, pull request, release, schedule). It's deeply integrated with GitHub and has a rich marketplace of pre-built actions.

## THEORY
### Core Concepts
- **Workflow:** A YAML file in `.github/workflows/`. Defines the automation.
- **Event:** What triggers the workflow (push, pull_request, schedule, manual)
- **Job:** A group of steps that run on the same virtual machine
- **Step:** A single task (run a command or use an Action)
- **Action:** A reusable unit of work from the GitHub Marketplace (e.g., checkout code, setup Node.js, login to Docker)
- **Runner:** The virtual machine that executes the workflow (GitHub provides Ubuntu, Windows, Mac)

## CODE

### CI Pipeline: Test and Build
```yaml
# .github/workflows/ci.yml
name: CI Pipeline

on:
  push:
    branches: [main, develop]       # Run on push to these branches
  pull_request:
    branches: [main]                # Run on PRs targeting main

jobs:
  test:
    name: Run Tests
    runs-on: ubuntu-latest          # Free GitHub-hosted runner!

    steps:
      # 1. Checkout your code
      - name: Checkout code
        uses: actions/checkout@v4

      # 2. Setup Node.js (from the marketplace!)
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'              # Cache npm dependencies automatically!

      # 3. Install dependencies
      - name: Install dependencies
        run: npm ci

      # 4. Run linting
      - name: Lint
        run: npm run lint

      # 5. Run tests with coverage
      - name: Test
        run: npm test -- --coverage

      # 6. Upload coverage to Codecov
      - name: Upload coverage
        uses: codecov/codecov-action@v3

  build:
    name: Build Docker Image
    needs: test            # Only run AFTER tests pass!
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - name: Login to GHCR
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}  # Auto-provided by GitHub!

      - name: Build and Push
        uses: docker/build-push-action@v5
        with:
          push: ${{ github.ref == 'refs/heads/main' }}  # Only push on main
          tags: ghcr.io/${{ github.repository }}:${{ github.sha }}
          cache-from: type=gha
          cache-to: type=gha,mode=max
```

### CD Pipeline: Deploy to Production
```yaml
# .github/workflows/deploy.yml
name: Deploy to Production

on:
  push:
    tags: ['v*']    # Trigger when a version tag is pushed

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: production    # Requires manual approval (configured in GitHub settings)!

    steps:
      - uses: actions/checkout@v4

      - name: Deploy to Server
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.PROD_HOST }}
          username: deploy
          key: ${{ secrets.SSH_PRIVATE_KEY }}
          script: |
            cd /opt/myapp
            docker compose pull
            docker compose up -d --no-deps api
            docker system prune -f
```

## REAL_WORLD
**Matrix Builds: Test Across Multiple Node Versions**
```yaml
jobs:
  test:
    strategy:
      matrix:
        node: [18, 20, 22]
        os: [ubuntu-latest, windows-latest]
    runs-on: ${{ matrix.os }}
    steps:
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: npm test
# Creates 6 parallel jobs: 3 node versions × 2 operating systems!
```

## INTERVIEW
**Q: A developer added `secrets.AWS_ACCESS_KEY_ID` to the workflow YAML file's `env` block (hardcoded the secret value instead of using `${{ secrets.AWS_ACCESS_KEY_ID }}`). The file was pushed to a public GitHub repository. What's the risk and immediate response?**

A: **This is a critical security incident.** The AWS key is now publicly visible to anyone on the internet. GitHub also scans for this pattern and will alert you, but the damage is done — bots continuously scrape GitHub for exposed credentials.

**Immediate response:**
1. **Revoke the key NOW** in the AWS IAM console — before doing anything else. Time matters; attackers may be already using it.
2. Check AWS CloudTrail for any unauthorized activity since the key was exposed
3. Remove the hardcoded key from the code and force-push (`git push --force`) to remove it from the latest commit
4. Note: Force-pushing does NOT remove the key from git history. Run `git filter-branch` or `BFG Repo Cleaner` to scrub history, then contact GitHub support if needed for a public repo
5. Add the correct approach: store in `Settings → Secrets`, reference as `${{ secrets.AWS_ACCESS_KEY_ID }}`
6. Set up a pre-commit hook or `gitleaks` scan to detect secrets before commits

**Prevention:** Never store credentials in code. Use GitHub Secrets, AWS Secrets Manager, or HashiCorp Vault.
