## WHY
`git status` and `git log` are the two commands you run most often. Status tells you "what's happening right now." Log tells you "what happened in the past." Together, they give you complete situational awareness of your repository.

## THEORY
### Git Status
`git status` compares three things:
1. What's in the last commit (HEAD)
2. What's in the staging area
3. What's in your working directory

The output tells you the difference between these three states.

### Git Log
`git log` displays the commit history. The default display is verbose — most developers customize it with flags.

## CODE

### git status
```bash
# Basic status
git status
# On branch feature/auth
# Changes to be committed:
#   new file:   src/auth/OAuthProvider.java     ← Staged (ready to commit)
#   modified:   src/auth/SecurityConfig.java    ← Staged
#
# Changes not staged for commit:
#   modified:   src/api/UserController.java     ← Modified but NOT staged
#
# Untracked files:
#   .env.local                                   ← New file, not tracked by Git

# Short format (one line per file)
git status -s
# A  src/auth/OAuthProvider.java    (A=Added, staged)
# M  src/auth/SecurityConfig.java   (M=Modified, staged)
#  M src/api/UserController.java    (second M=Modified, not staged)
# ?? .env.local                     (??=Untracked)
```

### git log
```bash
# Full default log (too verbose for daily use)
git log

# One-line per commit
git log --oneline
# a3f82b7 (HEAD -> main, origin/main) Add OAuth2 Google login
# d4e5f6c Add JWT token validation
# 1c4d89e Add User entity and repository

# With branch graph visualization
git log --oneline --graph --all
# * a3f82b7 (HEAD -> main) Add OAuth2 Google login
# * d4e5f6c Add JWT token validation
# | * b2c3d4 (feature/payment) Add Stripe integration
# |/
# * 1c4d89e Add User entity

# Last 5 commits
git log -5

# Commits by author
git log --author="Alice"

# Commits since/until date
git log --since="2024-01-01" --until="2024-01-31"

# Search commit messages
git log --grep="payment"

# Show commits that changed a specific file
git log -- src/api/user.js

# Show what actually changed in each commit
git log -p   # Full diff per commit (verbose!)
git log --stat  # Files changed + line counts per commit

# Formatted output
git log --pretty=format:"%h | %an | %ad | %s" --date=short
# a3f82b7 | Alice Smith | 2024-01-28 | Add OAuth2 Google login
# d4e5f6c | Bob Jones   | 2024-01-27 | Add JWT validation
```

## REAL_WORLD
**Useful git log Aliases**
```bash
# Add to ~/.gitconfig
[alias]
    lg = log --oneline --graph --all --decorate
    recent = log --oneline -10
    who = log --pretty=format:'%an: %s' --since=1.week.ago
```

Now `git lg` gives you a beautiful branch graph, and `git who` shows what your team worked on this week.

## INTERVIEW
**Q: How do you find which commit introduced a specific bug in a file? You know it was working 6 months ago but is broken now.**
A: Use **`git bisect`** — a binary search through commit history!

```bash
# Start bisect
git bisect start

# Mark current commit (HEAD) as BAD (has the bug)
git bisect bad

# Mark a known-good commit from 6 months ago
git bisect good abc1234  # Or: git bisect good 6.months.ago

# Git checks out the midpoint commit:
# "Bisecting: 47 revisions left to test (roughly 6 steps)"

# Test if the bug exists in this checkout:
./run-tests.sh

# Tell git the result:
git bisect good  # OR: git bisect bad

# Git jumps to the next midpoint...
# Repeat ~6 more times (binary search!)

# Git identifies the exact commit that introduced the bug:
# "a3f82b7 is the first bad commit"

git bisect reset  # Return to HEAD

# View what changed in that commit:
git show a3f82b7
```

Binary search through commit history is far more efficient than checking commits one-by-one. For 100 commits, you find the culprit in 7 steps!
