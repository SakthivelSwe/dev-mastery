## WHY
Without Pull Requests (PRs), developers push directly to `main` and hope for the best. Broken code goes straight to production. No one reviews code for security vulnerabilities, bugs, or design problems before they're deployed.
**Pull Requests** are the standard mechanism for collaborative code review. A developer finishes a feature on a branch, opens a PR, and asks teammates to review the code before it's merged. This catches bugs, improves code quality, shares knowledge, and maintains a clear audit trail of all changes.

## THEORY
### Pull Request Workflow
1. Developer creates a feature branch and commits changes
2. Developer pushes branch to GitHub and opens a PR
3. Reviewers are notified and examine the diff
4. Reviewers leave comments, request changes, or approve
5. Developer makes requested changes and pushes more commits
6. Once approved (and CI passes), the PR is merged to main

### PR vs MR
GitHub calls them **Pull Requests (PRs)**, GitLab calls them **Merge Requests (MRs)**. Same concept.

## CODE

### Creating a PR via GitHub CLI
```bash
# Install GitHub CLI
# brew install gh  (Mac)
# winget install GitHub.cli  (Windows)

# Authenticate
gh auth login

# Push your branch
git push origin feature/user-auth

# Create a PR
gh pr create \
  --title "Add OAuth2 Google Login" \
  --body "This PR implements Google OAuth2 login using Spring Security.

## What Changed
- Added oauth2-spring boot starter
- Configured Google provider in SecurityConfig
- Added UserController to expose /api/me endpoint

## Testing
- Manual: Verified login flow with test Google account
- Unit tests: Added SecurityConfig tests

Closes #142" \
  --reviewer alice,bob \
  --assignee me \
  --label "feature,authentication"

# List open PRs
gh pr list

# Check PR status (CI checks, reviews)
gh pr status

# Merge the PR (after approval)
gh pr merge 47 --squash --delete-branch

# View PR in browser
gh pr view 47 --web
```

```markdown

Brief explanation of what this PR does and WHY it was needed.

- [ ] Added SecurityFilterChain configuration
```

### Branch Protection Rules (Enforcing PR Process)
```
# GitHub Settings → Branches → Branch protection rules → Add rule

Branch name pattern: main

✅ Require a pull request before merging
  ✅ Require approvals: 1
  ✅ Dismiss stale pull request approvals when new commits are pushed
✅ Require status checks to pass before merging
  ✅ Require branches to be up to date
  Status checks: CI Pipeline, Build
✅ Restrict who can push to matching branches
✅ Do not allow bypassing the above settings
```

With these rules, **nobody** (not even repo owners) can push directly to main without a PR.

## REAL_WORLD
**Draft PRs for Early Feedback**
If you want early feedback before the feature is complete:
- Open the PR as a **Draft** on GitHub
- Reviewers know it's not ready for final review
- You get early architectural feedback
- CI runs on every commit so you see test results early
- When ready: convert Draft → Ready for Review

```bash
gh pr create --draft --title "WIP: Add payment gateway"
```

## INTERVIEW
**Q: A senior developer reviews your PR and requests 15 changes. You believe 13 of them are valid, but 2 are subjective style preferences that you disagree with. How do you handle this professionally?**
A: 

1. **Address the 13 valid changes** with actual code commits and mark each comment as "Resolved."

2. **For the 2 subjective disagreements**, engage professionally in the PR comment thread:
   - Acknowledge the reviewer's perspective: "I understand why you prefer X approach"
   - Clearly explain your reasoning: "I chose Y because [specific technical reason]"
   - Propose compromise if possible: "Would approach Z work for you?"
   - Ask for a team standard: "Should we add this to our style guide so everyone follows the same pattern?"
   - Escalate to a third-party for an objective opinion if needed

3. **Never merge without resolving all comments**, even if you disagree. Either change the code or get explicit acknowledgment that the reviewer is OK with your alternative approach (mark as "Resolved" with the reviewer's blessing).

The goal of code review is to improve the code AND maintain team relationships. Being right but adversarial is worse than being collaborative.
