## WHY
Imagine you're writing code. You make a change. It breaks everything. You try to undo it but can't remember what the code looked like before.
Or: You and a colleague both edit the same file simultaneously. One of you overwrites the other's work.
Or: You deploy a new feature and it breaks production. You need to instantly revert to last week's version.
**Git** solves all of these problems. It is the world's most widely used **distributed version control system**. Git tracks every change to every file, who made it, when, and why. It allows multiple developers to work simultaneously without overwriting each other's work, and lets you time-travel to any previous state of your code instantly.

## THEORY
### Git's Mental Model: Snapshots, Not Differences
Unlike older version control systems (SVN), Git doesn't store file differences. Git stores **snapshots** of the entire project at a point in time.
Each snapshot is called a **commit**. Each commit contains:
- A complete snapshot of all project files
- A unique identifier (SHA-1 hash: `a3f82b...`)
- The author's name and email
- A timestamp
- A commit message (description of what changed)
- A pointer to the parent commit (creating a chain of history)

### The Three Areas
1. **Working Directory:** Your actual files on disk. Where you make changes.
2. **Staging Area (Index):** A "draft" of your next commit. You add files here before committing.
3. **Repository (.git folder):** The database of all commits and history.

### The Basic Git Workflow
```bash
# 1. Create a new project
mkdir my-project && cd my-project
git init   # Creates the .git folder (the repository)

# 2. Create some files
echo "# My Project" > README.md
echo "console.log('Hello')" > app.js

# 3. Check status (what has changed?)
git status
# Untracked files: README.md, app.js

# 4. Stage files (add to the "draft" commit)
git add README.md     # Stage one file
git add .             # Stage ALL changes

# 5. Create a commit (save the snapshot!)
git commit -m "Initial commit: add README and app"

# 6. Make changes
echo "console.log('World')" >> app.js

# 7. See what changed
git diff              # See exact changes in working directory
git status            # See which files changed

# 8. Stage and commit again
git add app.js
git commit -m "Add World greeting"

# 9. See history
git log
# commit a3f82b... (HEAD -> main)
# Author: Alice Smith <alice@example.com>
# Date:   Thu Jan 28 15:00:00 2024
#     Add World greeting
#
# commit 1c4d89...
#     Initial commit: add README and app

git log --oneline    # Compact view
# a3f82b (HEAD -> main) Add World greeting
# 1c4d89 Initial commit: add README and app
```

## REAL_WORLD
**Git: The DNA of Modern Software Development**
Git is not just a tool — it's the foundation of:
- **GitHub, GitLab, Bitbucket:** Platforms built on top of Git repositories
- **CI/CD:** Every pipeline triggers on a `git push`
- **Code review:** Pull requests are comparisons between Git branches
- **DevOps:** Infrastructure-as-Code lives in Git repositories
- **Releases:** Git tags mark release versions

Companies run their entire engineering organizations around Git workflows. Understanding Git deeply is the single most universally valuable technical skill in software development.

## INTERVIEW
**Q: What is the difference between `git add` and `git commit`? Why does Git have this two-step process?**
A: 

**`git add`** moves changes from the Working Directory to the **Staging Area** (also called the Index). The Staging Area is a buffer — it holds the changes you want to include in your next commit.

**`git commit`** takes everything in the Staging Area and creates a permanent snapshot in the Git repository.

**Why the two-step process?**
It gives you precise control over what goes into each commit. Consider this scenario: You've been working on two unrelated bug fixes simultaneously — fixing a login bug AND fixing a display issue. You want to commit them separately (best practice!) so each commit represents one logical change.

Without the staging area, you'd be forced to commit both together or undo one fix temporarily. With staging, you can:
```bash
git add src/auth/login.js
git commit -m "Fix login validation bug"

git add src/components/Header.js
git commit -m "Fix header display on mobile"
```

Two clean, logical commits from one working session. This makes code review, bisecting bugs, and reverting specific changes much easier.
