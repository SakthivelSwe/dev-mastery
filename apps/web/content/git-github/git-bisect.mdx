## WHY
Someone introduced a bug in the codebase, but you don't know exactly when. The bug is currently in `main`, but you know it wasn't there a month ago (which is 500 commits ago).
Checking out and testing each of the 500 commits manually would take days. **Git Bisect** automates a binary search through your commit history to find the exact commit that introduced the bug in a fraction of the time.

## THEORY
### Binary Search
Instead of checking commit 1, then 2, then 3... Git Bisect checks the commit halfway between the "known good" and "known bad" points.
1. 500 commits to check. Git checks commit 250.
2. If 250 is GOOD, the bug must be in commits 251-500. Git checks 375.
3. If 375 is BAD, the bug must be in commits 251-375. Git checks 313.
This O(log n) approach finds the bug in 500 commits using at most 9 steps!

## CODE

### Manual Bisect
```bash
# 1. Start the bisect wizard
git bisect start

# 2. Tell Git the current commit is BAD (has the bug)
git bisect bad

# 3. Tell Git an older commit that is GOOD (doesn't have the bug)
# You can use a commit hash, tag, or branch name
git bisect good v1.2.0
# Output: Bisecting: 250 revisions left to test after this (roughly 8 steps)
# [git automatically checks out the middle commit]

# 4. Now, TEST the code! (Run your app, check for the bug)
# If the bug is present:
git bisect bad
# If the bug is gone:
git bisect good

# Git will check out the next middle commit. Repeat step 4 until Git says:
# "a1b2c3d4 is the first bad commit"

# 5. End the bisect session and return to where you started
git bisect reset
```

### Automated Bisect (The Magic Trick)
If you can write a script that returns `0` when the code is good and `1` (or any non-zero) when the code is bad, Git can run the entire bisect process *automatically* while you get coffee.

```bash
# Provide a test script (e.g., test.sh)
# test.sh:
#   npm install
#   npm test -- -t "Login Component"

# Start the bisect
git bisect start HEAD v1.2.0

# Tell Git to run the script for every step!
git bisect run ./test.sh

# Git will automatically checkout, run the script, read the exit code,
# and move to the next commit until it finds the exact culprit!
```

## REAL_WORLD
**Finding Performance Regressions**
Bisect isn't just for hard crashes. Imagine your page load time went from 1s to 5s over the last month.

Write a script `check-speed.sh`:
```bash
#!/bin/bash
npm run build
npm start &
PID=$!
sleep 5 # Wait for server to start
# Hit the API and get response time in milliseconds
TIME=$(curl -o /dev/null -s -w '%{time_total}\n' http://localhost:3000)
kill $PID

# If time > 2 seconds, exit with error code 1 (BAD)
if (( $(echo "$TIME > 2.0" |bc -l) )); then
  exit 1
else
  exit 0
fi
```
Run `git bisect run ./check-speed.sh` to find exactly which commit slowed down your app.

## INTERVIEW
**Q: You are in the middle of a `git bisect` session. Git checks out a commit, you try to run your tests, but the code won't even compile because of an unrelated typo in that specific commit. You can't tell if the bug exists because you can't run the app. What do you do?**

A: You use `git bisect skip`.

If you tell Git `bad` or `good`, you're feeding it false information because you couldn't actually test the feature.

By typing `git bisect skip`, you tell Git: "I cannot test this specific commit. Pick a different nearby commit to test instead." Git will move slightly forward or backward from the broken commit to find one that compiles, allowing the binary search to continue.
