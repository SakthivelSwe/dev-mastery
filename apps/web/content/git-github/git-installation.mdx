## WHY
Before using Git, you need it installed and properly configured. The initial setup — especially configuring your identity and SSH keys for GitHub — saves countless authentication headaches later.

## THEORY
### Global vs Local Configuration
Git config has three levels:
- **System:** Applies to all users on the machine (`/etc/gitconfig`)
- **Global:** Applies to all repositories for your user (`~/.gitconfig`)
- **Local:** Applies to a specific repository (`.git/config`)
Local overrides Global, which overrides System.

### Essential Configuration
```bash
# Identity (required for commits — use your real name and email!)
git config --global user.name "Alice Smith"
git config --global user.email "alice@example.com"

# Set default editor
git config --global core.editor "code --wait"     # VS Code
git config --global core.editor "vim"              # Vim
git config --global core.editor "nano"             # Nano (beginner-friendly)

# Set default branch name to 'main' (modern standard)
git config --global init.defaultBranch main

# Better diff visualization
git config --global core.autocrlf input  # Mac/Linux
git config --global core.autocrlf true   # Windows

# View all configuration
git config --list --global
```

### SSH Key Setup for GitHub (One-Time)
```bash
# Generate SSH key pair
ssh-keygen -t ed25519 -C "alice@example.com"
# Press Enter for default location (~/.ssh/id_ed25519)
# Enter a passphrase (recommended!)

# Start the SSH agent
eval "$(ssh-agent -s)"

# Add key to agent
ssh-add ~/.ssh/id_ed25519

# Copy public key (add this to GitHub Settings → SSH Keys)
cat ~/.ssh/id_ed25519.pub
# ssh-ed25519 AAAAC3NzaC1lZDI1NTE5... alice@example.com

# Test connection
ssh -T git@github.com
# Hi alice! You've successfully authenticated...
```

## REAL_WORLD
**SSH vs HTTPS for Remote Operations**
When cloning/pushing to GitHub, you can use SSH or HTTPS:

```bash
# HTTPS (simpler, uses username/password or token)
git clone https://github.com/user/repo.git

# SSH (more secure, uses key pair — no password needed!)
git clone git@github.com:user/repo.git
```

In 2021, GitHub deprecated password authentication for HTTPS. Now HTTPS requires a **Personal Access Token (PAT)** — a long random string that replaces your password.

Most developers prefer SSH for its one-time setup (add key once, never authenticate again) vs HTTPS which requires managing tokens.

## INTERVIEW
**Q: Your team uses Git with both Windows and Mac developers. Sometimes files show up as modified even when nothing has changed. What's likely causing this?**
A: **Line ending differences (CRLF vs LF)**.

Windows uses `\r\n` (CRLF) for line endings. Mac/Linux uses `\n` (LF). When a Mac developer commits a file with LF endings, a Windows developer's editor saves it with CRLF — Git sees every line as changed, even though the visible content is identical.

**Fix with `.gitattributes`** (the correct team-wide solution, not individual `core.autocrlf`):
```
# .gitattributes (committed to the repository)
* text=auto         # Git normalizes line endings automatically
*.sh text eol=lf    # Shell scripts: always LF (Linux servers need this)
*.bat text eol=crlf # Batch files: always CRLF (Windows)
*.png binary        # Binary files: no line ending conversion
```

Or configure each developer's Git:
- Windows: `git config --global core.autocrlf true` (CRLF on checkout, LF on commit)
- Mac/Linux: `git config --global core.autocrlf input` (LF on commit, untouched on checkout)
