## WHY
Before committing changes, you want to review exactly what you've changed. `git diff` shows you line-by-line differences between any two states of your code — working directory vs staged, staged vs committed, or any two commits.
Understanding diff output is fundamental to reviewing your own work and understanding others' changes.

## THEORY
### Reading Diff Output
Git diffs use a unified diff format:
- Lines starting with `-` (red): Lines removed
- Lines starting with `+` (green): Lines added
- Lines with no prefix: Context lines (unchanged, shown for context)

## CODE

### diff Variations
```bash
# Compare working directory to staging area (unstaged changes)
git diff

# Compare staging area to last commit (what will be committed)
git diff --staged
git diff --cached   # Same thing

# Compare working directory to last commit (all changes)
git diff HEAD

# Compare two commits
git diff a3f82b7 d4e5f6c

# Compare two branches
git diff main feature/payment

# Diff a specific file only
git diff src/api/user.js
git diff HEAD -- src/api/user.js

# Summary (just changed files, no line-by-line)
git diff --stat
# src/api/user.js  | 15 +++++++---
# src/model/User.java | 5 +++--
# 2 files changed, 15 insertions(+), 5 deletions(-)

# Word-level diff (more granular than line-level)
git diff --word-diff
```


## REAL_WORLD
**Reviewing Before a PR**
```bash
# Before creating a PR, compare your branch to main:
git diff main..feature/payment

# See which files changed (for a quick overview):
git diff main..feature/payment --stat

# Diff from the branching point (ignoring other changes to main):
git diff main...feature/payment  # Note: three dots!
```

## INTERVIEW
**Q: What's the difference between `git diff main..feature` and `git diff main...feature` (two dots vs three dots)?**

A: 
- **Two dots (`..`):** Shows all differences between the tip of `main` and the tip of `feature`. This includes any commits that were added to `main` AFTER you branched off.

- **Three dots (`...`):** Shows differences between the COMMON ANCESTOR of both branches and the tip of `feature`. This is the "changes you made on your feature branch" — it excludes any changes that were added to `main` after you branched.

Example:
```
main:     A → B → C → D (D added after feature was created)
feature:  A → B → E → F

git diff main..feature  → Shows: remove D's changes, add E+F's changes
git diff main...feature → Shows: only E+F's changes (what YOU wrote)
```

For reviewing a PR's changes, `...` is what you want — it shows what the feature branch actually contributed, without noise from main's changes.
