## WHY
You want to format your code, run the linter, or ensure the commit message matches a specific format *before* the commit is finalized. Or you want to run tests before a `git push`.
**Git Hooks** are scripts that Git executes automatically before or after events such as commit, push, and receive. They allow you to automate tasks and enforce standards locally, preventing bad code from ever leaving the developer's machine.

## THEORY
### Types of Hooks
- **Client-Side Hooks:** Run on the developer's local machine.
  - `pre-commit`: Runs before the commit message is typed. Great for linting/formatting.
  - `commit-msg`: Runs after the message is typed but before the commit is finalized. Great for validating message formats (e.g., Conventional Commits).
  - `pre-push`: Runs before pushing to a remote. Great for running fast tests.
- **Server-Side Hooks:** Run on the remote server (e.g., GitHub, GitLab).
  - `pre-receive`: Runs before processing pushed commits. Can reject the push entirely if it violates policies.

## CODE

### Creating a Native Git Hook
Git hooks are stored in the `.git/hooks/` directory. By default, they contain `.sample` files. Remove the `.sample` extension to make them active.

```bash
# Create a pre-commit hook
cat << 'EOF' > .git/hooks/pre-commit
#!/bin/sh
echo "Running pre-commit hook: Linting code..."
npm run lint

# If the linter fails (exit code > 0), the commit is aborted!
if [ $? -ne 0 ]; then
  echo "Linting failed. Commit aborted."
  exit 1
fi
echo "Linting passed. Committing..."
EOF

# Make the script executable (Required!)
chmod +x .git/hooks/pre-commit
```

### Husky: Modern Git Hooks for Node.js
Managing native `.git/hooks/` across a team is hard because the `.git` directory is not version controlled. **Husky** solves this by putting hooks in your repository files.

```bash
# Install Husky
npm install husky --save-dev

# Initialize Husky (creates .husky/ directory and updates package.json scripts)
npx husky init

# Add a pre-commit hook to run tests
echo "npm test" > .husky/pre-commit

# Add a commit-msg hook to enforce Conventional Commits (using commitlint)
npm install --save-dev @commitlint/{config-conventional,cli}
echo "module.exports = {extends: ['@commitlint/config-conventional']}" > commitlint.config.js
echo "npx --no -- commitlint --edit \$1" > .husky/commit-msg
```

### Bypassing Hooks
Sometimes you really need to commit something despite failing linting (e.g., saving work-in-progress locally).

```bash
# Bypass the pre-commit and commit-msg hooks
git commit --no-verify -m "WIP: saving broken state"

# Bypass pre-push hooks
git push --no-verify
```

## REAL_WORLD
**Pre-commit vs CI/CD**
Why use a `pre-commit` hook to lint code when your GitHub Actions CI pipeline lints it anyway?
- **Fast feedback loop:** The hook fails instantly on the developer's machine (1 second) vs waiting for the PR CI to run (3 minutes).
- **Cleaner history:** You don't end up with commits like `"Fix lint error"`, `"Fix another lint error"` polluting the PR history.
- **Resource saving:** Reduces load on the CI servers.

However, **hooks do not replace CI**. A developer can always bypass hooks with `--no-verify`. The CI pipeline is the ultimate, inescapable gatekeeper.

## INTERVIEW
**Q: Your team uses a `pre-commit` hook to run unit tests. As the codebase grew, the tests now take 3 minutes to run. Developers are complaining that every commit takes 3 minutes, so they're all using `--no-verify`. How do you solve this?**

A: `pre-commit` hooks must be fast (< 5 seconds), or developers will bypass them, defeating their purpose.

1. **Lint-staged:** Instead of running tests, run fast linters/formatters (Prettier, ESLint) *only on the staged files*.
   ```bash
   # package.json
   "lint-staged": {
     "*.js": "eslint --fix"
   }
   ```
2. **Move tests to pre-push:** A developer commits frequently but pushes less often. Moving tests to the `pre-push` hook is more tolerable.
3. **Move to CI entirely:** If tests are too slow for `pre-push`, remove the hook entirely and rely on the CI pipeline running on the Pull Request.
4. **Run specific tests:** Write a script that only runs tests related to the changed files, rather than the whole suite.
