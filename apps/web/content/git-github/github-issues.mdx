## WHY
Tracking bugs, feature requests, and tasks in an ad-hoc manner (emails, Slack messages, sticky notes) leads to lost work and lack of visibility. GitHub Issues provides a centralized, searchable, and structured way to track work directly alongside the code. They integrate seamlessly with Pull Requests, project boards, and automation.

## THEORY
### Core Concepts
- **Issue:** A single unit of work (bug report, feature request, task).
- **Labels:** Categorize issues (e.g., `bug`, `enhancement`, `help wanted`, `good first issue`).
- **Assignees:** Who is responsible for the issue (up to 10 people).
- **Milestones:** Group issues into a specific release or time-bound goal.
- **Projects:** Kanban boards to track issue progress across columns (To Do, In Progress, Done).

## CODE

### Creating and Managing Issues via CLI
```bash
# Create an issue
gh issue create --title "Login page crashes on mobile" --body "When clicking the submit button on an iPhone 13, the app crashes. See attached logs."

# List open issues
gh issue list

# List issues assigned to you
gh issue list --assignee "@me"

# List issues with specific labels
gh issue list --label "bug" --label "high-priority"

# View a specific issue
gh issue view 42

# Comment on an issue
gh issue comment 42 --body "I'm looking into this now."

# Close an issue
gh issue close 42

# Edit an issue
gh issue edit 42 --title "Login page crashes on iOS 16"
```

### Linking Pull Requests to Issues
You can automatically close an issue when a PR is merged by including specific keywords in the PR description:

```markdown
# PR Description
This PR fixes the mobile login crash.

Closes #42
```

Valid keywords: `close`, `closes`, `closed`, `fix`, `fixes`, `fixed`, `resolve`, `resolves`, `resolved`.

### Issue Templates
Standardize how users report bugs or request features using templates:

```yaml
# .github/ISSUE_TEMPLATE/bug_report.yml
name: Bug Report
description: File a bug report to help us improve
title: "[Bug]: "
labels: ["bug", "triage"]
body:
  - type: textarea
    id: description
    attributes:
      label: Description
      description: A clear and concise description of what the bug is.
    validations:
      required: true
  - type: textarea
    id: reproduce
    attributes:
      label: Steps To Reproduce
      description: Steps to reproduce the behavior.
  - type: input
    id: version
    attributes:
      label: Version
      description: What version of the software are you using?
```

## REAL_WORLD
**The "Good First Issue" Label**
In open-source projects, maintainers use the `good first issue` label to highlight bugs or features that are well-scoped, well-documented, and relatively easy to implement. This encourages new contributors to get involved without feeling overwhelmed. If you're looking to contribute to open source, searching for this label is the best place to start!

## INTERVIEW
**Q: You have an issue assigned to you, and you've just pushed a fix in a new branch. How do you communicate this clearly and automate the issue tracking?**

A: 
1. **Create a PR:** Open a Pull Request from your branch.
2. **Use Closing Keywords:** In the PR description, write `Fixes #ISSUE_NUMBER`.
3. **Link them (if not using keywords):** Use the "Linked issues" section in the GitHub UI sidebar of the PR.

This automatically adds a reference in the issue timeline that a PR was opened to address it. When the PR is merged, the issue will automatically close, keeping the project board and issue tracker clean without manual intervention.
