## WHY
You're working on a feature branch with half-finished changes when your manager says "Critical bug in production — drop everything!"
You need to switch to the main branch to fix the bug, but your half-finished changes would either go with you (polluting the hotfix) or be lost.
**Git Stash** saves your uncommitted changes to a temporary storage area (a stack), lets you switch branches freely, and restores your changes when you return. It's a context-switching lifesaver.

## THEORY
### What Gets Stashed?
By default, `git stash` saves:
- Modified tracked files
- Staged changes

By default, it does NOT save:
- Untracked new files (unless you use `-u`)
- Ignored files (unless you use `-a`)

## CODE

### Basic Stash Operations
```bash
# You have uncommitted changes you need to temporarily set aside
git status
# Modified: src/feature/PaymentService.java
# Modified: src/feature/PaymentController.java

# Stash your changes
git stash

# Now your working directory is clean
git status
# nothing to commit, working tree clean

# You can now switch branches, pull changes, fix the bug, etc.
git checkout main
git checkout -b hotfix/payment-crash
# ... fix the bug, commit, push ...

# Return to your feature branch
git checkout feature/payment

# Restore your stashed changes
git stash pop   # Apply AND remove from stash stack

# OR: Apply WITHOUT removing from stash (keep the stash)
git stash apply
```

### Working with Multiple Stashes
```bash
# Stash with a descriptive name (essential for multiple stashes!)
git stash push -m "WIP: payment gateway Stripe integration"

# List all stashes
git stash list
# stash@{0}: On feature/payment: WIP: payment gateway Stripe integration
# stash@{1}: On feature/auth: Halfway through OAuth implementation
# stash@{2}: WIP on main: Quick experiment

# Apply a specific stash
git stash apply stash@{1}

# Drop a specific stash
git stash drop stash@{2}

# Clear all stashes
git stash clear

# Show what's in a stash (without applying)
git stash show stash@{0}
git stash show -p stash@{0}  # Full diff
```

### Stash Untracked Files
```bash
# By default, new untracked files are NOT stashed!
touch new-feature.js   # New file

git stash
git status
# Untracked files: new-feature.js  ← Still here!

# Use -u to include untracked files
git stash -u

# Or -a to include all files (even .gitignore'd)
git stash -a
```

## REAL_WORLD
**Stash vs WIP Commit — Which to Use?**
Two common approaches for temporarily saving work:

**Option 1: Stash (temporary, not in history)**
```bash
git stash push -m "WIP: feature in progress"
# Fast, easy, no commit history
# Risk: stashes can be lost (git stash clear, disk corruption)
```

**Option 2: WIP commit (in history)**
```bash
git add .
git commit -m "WIP: save progress before hotfix"
# Work is safely in Git history!
# Use git commit --amend or git rebase -i to clean up later
```

For short interruptions (minutes), use stash. For longer interruptions or at end of day, commit a WIP (you can always squash it later with interactive rebase).

## INTERVIEW
**Q: You have unstaged changes and you run `git stash`. Later you run `git stash pop` and see a merge conflict. What happened and how do you resolve it?**
A: While your changes were stashed, someone else's code was merged into your branch (or you applied the stash to a different branch where the same lines were modified differently).

When `git stash pop` tries to reapply your stashed changes, it encounters a conflict because the same lines have been changed both in the stash AND in the current branch state.

**Resolution:**
```bash
# After the conflict message, check status
git status
# BOTH modified: src/api/user.js

# Open the conflicted file — you'll see conflict markers:
# <<<<<<< Updated upstream
# function getUser() { return db.findById(id); }  ← current branch
# =======
# function getUser() { return cache.get(id); }    ← your stashed version
# >>>>>>> Stashed changes

# Edit to keep the correct code (or combine both)
# Then mark as resolved:
git add src/api/user.js

# If stash pop created a conflict, the stash is STILL in the stash list!
# Once you've resolved and staged, just commit normally (no special stash command needed)
git commit -m "Merge stashed payment changes"

# Or if you used 'git stash apply' (not pop), drop the stash manually:
git stash drop stash@{0}
```
