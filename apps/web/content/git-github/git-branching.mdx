## WHY
Without branches, every developer pushes directly to the main codebase. A half-finished feature breaks everyone's work. A broken experiment corrupts the production release.
**Branches** are parallel universes for your code. A branch is an isolated copy of the codebase where you can experiment, build features, and fix bugs — completely independent of other developers' work.
When your feature is done and tested, you merge it back into the main branch. This is the fundamental workflow that makes professional software development possible.

## THEORY
### What is a Branch?
In Git, a branch is simply a **lightweight pointer to a commit**. That's it. Creating a branch doesn't copy any files — it just creates a new named pointer that advances with each commit you make on it.

### HEAD
`HEAD` is a special pointer that tells Git "where you are right now." When you switch branches, `HEAD` moves to point to that branch.

## CODE

### Branch Commands
```bash
# List all branches (* = current branch)
git branch
# * main
#   feature/user-auth
#   fix/login-bug

# List with last commit info
git branch -v

# Create a new branch
git branch feature/payment-gateway

# Switch to a branch
git checkout feature/payment-gateway

# Create AND switch in one command (modern syntax)
git checkout -b feature/payment-gateway
git switch -c feature/payment-gateway  # New preferred syntax

# Switch back to main
git checkout main
git switch main

# Delete a merged branch (safe — won't delete unmerged!)
git branch -d feature/payment-gateway

# Force delete a branch (even if unmerged)
git branch -D feature/payment-gateway

# Rename a branch
git branch -m old-name new-name
```

### The Feature Branch Workflow
```bash
# 1. Always start from the latest main
git checkout main
git pull origin main   # Make sure you have the latest code!

# 2. Create a feature branch
git checkout -b feature/user-profile-page

# 3. Work on your feature
echo "// Profile component" > src/components/Profile.jsx
git add .
git commit -m "Add Profile component skeleton"

# 4. Keep working
echo "// Profile API" > src/api/profile.js
git add .
git commit -m "Add Profile API service"

# 5. Push to remote for backup/collaboration
git push origin feature/user-profile-page

# 6. Create a Pull Request on GitHub for review

# 7. After approval, merge to main (done on GitHub or locally)
git checkout main
git merge feature/user-profile-page

# 8. Clean up
git branch -d feature/user-profile-page
git push origin --delete feature/user-profile-page  # Delete remote branch too
```

## INTERVIEW
**Q: What is a "detached HEAD" state in Git, and how do you exit it safely?**
A: A detached HEAD occurs when `HEAD` points directly to a commit hash instead of a branch name.

This happens when you checkout a specific commit:
```bash
git checkout a3f82b  # Now HEAD = a3f82b, not 'main'!
```
You're in "detached HEAD" — you can look at the code, even make commits, but they're "floating" in history with no branch name pointing to them. If you switch to another branch, these commits become unreachable (orphaned) and will eventually be garbage collected!

**Exiting safely:**
```bash
# Option 1: Go back to a branch without keeping your changes
git checkout main

# Option 2: Create a NEW branch to preserve any commits you made in detached HEAD
git checkout -b my-experiment-branch
# Now your commits are safe on 'my-experiment-branch'!

# Option 3: If you're just exploring history (no new commits), 
# just checkout any branch to reattach HEAD
git switch main
```
