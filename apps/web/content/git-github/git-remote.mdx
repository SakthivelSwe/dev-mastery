## WHY
A local Git repository exists only on your computer. If your laptop dies, all your code history is gone. And how do you share code with teammates?
**Remote repositories** are hosted versions of your Git repository on services like GitHub, GitLab, or Bitbucket. They serve as the central source of truth that all team members synchronize with.

## THEORY
### What is a Remote?
A remote is a named reference to a URL of another Git repository. By convention, the primary remote is named `origin`. When you clone a repository, Git automatically sets up `origin` pointing to where you cloned from.

### Key Operations
- **push:** Send your local commits to the remote
- **pull:** Fetch remote commits AND merge them into your current branch
- **fetch:** Download remote commits WITHOUT merging (safe — lets you see what changed first)

## CODE

### Working with Remotes
```bash
# List configured remotes
git remote -v
# origin  https://github.com/user/repo.git (fetch)
# origin  https://github.com/user/repo.git (push)

# Add a remote
git remote add origin https://github.com/user/repo.git

# Change the URL of a remote (e.g., switching from HTTPS to SSH)
git remote set-url origin git@github.com:user/repo.git

# Add a second remote (e.g., a backup, or deploying to different services)
git remote add upstream https://github.com/original-project/repo.git
git remote add heroku https://git.heroku.com/myapp.git

# Remove a remote
git remote remove upstream
```

### Push, Pull, Fetch
```bash
# Push local branch to remote
git push origin main           # Push 'main' to 'origin'
git push                       # Short form (if tracking is set)
git push -u origin main        # Push AND set upstream tracking

# Push a new branch to remote
git push origin feature/my-feature

# Delete a remote branch
git push origin --delete feature/old-branch

# Fetch (download changes without merging)
git fetch origin               # Download all remote changes
git fetch origin main          # Download only main
# Examine what changed without affecting your code:
git log origin/main --oneline  # See remote main's commits
git diff main origin/main      # Compare local vs remote

# Pull (fetch + merge)
git pull origin main           # Pull and merge main
git pull --rebase              # Pull and rebase instead of merge (cleaner history)

# Clone (download an entire repository)
git clone https://github.com/user/repo.git
git clone git@github.com:user/repo.git  # SSH (preferred)
git clone --depth=1 https://...         # Shallow clone (only latest snapshot)
```

### Keeping Your Fork Synced
```bash
# Working with a fork of an open-source project:

# Add the original repo as 'upstream'
git remote add upstream https://github.com/original-org/project.git

# Get latest changes from the original
git fetch upstream

# Merge upstream changes into your main
git checkout main
git merge upstream/main  # or: git rebase upstream/main

# Push your updated fork to GitHub
git push origin main
```

## REAL_WORLD
**git fetch vs git pull — Why to Prefer fetch**
Many developers use `git pull` instinctively, but `git fetch` is safer:

```bash
# git pull = git fetch + git merge (or rebase)
# If there are remote changes + your local changes, pull can create surprise merge commits!

# Better workflow:
git fetch origin                  # Safe — doesn't touch your working directory
git log HEAD..origin/main         # See what's new on remote before merging
git merge origin/main             # Now merge when you're ready
# OR
git rebase origin/main            # Rebase for cleaner history
```

## INTERVIEW
**Q: You pushed sensitive data to GitHub accidentally. The file containing the data has been committed and pushed. What are the exact steps to remove it from the repository history?**
A: This is a multi-step process:

**Step 1: Immediately invalidate the secret** (do this first, before fixing the code — assume it's compromised!)

**Step 2: Use BFG Repo Cleaner (faster than git filter-branch) to remove from history:**
```bash
# Download BFG
java -jar bfg.jar --delete-files config.env

# Or remove specific text (a password string):
java -jar bfg.jar --replace-text passwords.txt
# passwords.txt contains: supersecretpassword123 → REMOVED

# After BFG, clean up refs:
git reflog expire --expire=now --all
git gc --prune=now --aggressive

# Force-push ALL branches (rewrites history!)
git push --force --all
git push --force --tags
```

**Step 3: Contact GitHub Support** to remove cached views of the leaked data from GitHub's servers.

**Step 4: Notify your security team** if the data was sensitive (API keys, credentials, PII).

**Prevention:** Add `pre-commit` hooks with `gitleaks` or `truffleHog` to detect secrets before they're committed.
