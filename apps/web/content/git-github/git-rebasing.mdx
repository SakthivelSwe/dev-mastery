## WHY
When you've been working on a feature branch for a week, the main branch has moved forward with 20 new commits from other developers. Your feature branch is "behind."
You could `git merge main` into your feature branch, but this creates a messy merge commit and a non-linear history that's hard to read.
**Git Rebase** is an alternative that replays your commits on top of the latest main branch, resulting in a perfectly clean, linear history — as if you started your feature branch today.

## THEORY
### Merge vs Rebase
- **Merge:** Combines two branches by creating a new "merge commit." Preserves the true history (when branches diverged and merged). History is non-linear but accurate.
- **Rebase:** Takes your commits and replays them one-by-one on top of the target branch. Creates a linear history. Rewrites commit hashes (new parent commits = new SHA-1 hashes).

### The Golden Rule of Rebasing
**Never rebase a public/shared branch!**
Rebasing rewrites commit history. If you rebase `main` (which other developers have based their work on), their local histories become incompatible with yours. This causes massive merge conflicts for the entire team.

Only rebase **your own private feature branches** that nobody else has pushed to.

## CODE

### Basic Rebase
```bash
# Scenario: You're on feature/payment, main has moved forward with new commits

git checkout feature/payment
git log --oneline
# d4e5f6 Add stripe integration
# c3d4e5 Add payment model
# a1b2c3 (start) Initial payment work ← branched from here

git checkout main
git log --oneline
# j9k0l1 Add user notifications
# h7i8j9 Add dark mode
# a1b2c3 (original branch point)

# Rebase: replay our feature commits on top of latest main
git checkout feature/payment
git rebase main

# Git now:
# 1. Finds the common ancestor (a1b2c3)
# 2. Saves your commits temporarily
# 3. Moves HEAD to match main (j9k0l1)
# 4. Replays your commits one by one on top

git log --oneline
# n2o3p4 Add stripe integration (NEW hash — replay of d4e5f6)
# m1n2o3 Add payment model (NEW hash — replay of c3d4e5)
# j9k0l1 Add user notifications
# h7i8j9 Add dark mode
# a1b2c3 Initial setup
# Clean linear history!
```

### Interactive Rebase (The Power Feature)
```bash
# Clean up your commits before merging to main
git rebase -i HEAD~5   # Interactively rebase last 5 commits

# Opens your editor with:
# pick a1b2c3 WIP - working on auth
# pick d4e5f6 Fix typo
# pick g7h8i9 More auth work
# pick j0k1l2 Fix another typo
# pick m3n4o5 Auth complete - add tests

# Change 'pick' to:
# squash (s): Combine with previous commit
# reword (r): Change the commit message
# edit (e): Stop and amend the commit
# drop (d): Delete the commit entirely

# After editing:
# pick a1b2c3 WIP - working on auth
# squash d4e5f6 Fix typo
# squash g7h8i9 More auth work
# squash j0k1l2 Fix another typo
# reword m3n4o5 Auth complete - add tests

# Result: 5 messy WIP commits → 1 clean commit with a proper message
```

### Handling Rebase Conflicts
```bash
git rebase main
# CONFLICT (content): Merge conflict in src/api/user.js
# error: could not apply abc123... Add payment

# Fix the conflict in the file, then:
git add src/api/user.js
git rebase --continue      # Continue replaying next commit

# Or give up and go back to before the rebase
git rebase --abort
```

## REAL_WORLD
**The Team Convention: Rebase Before PR**
Many engineering teams require:
1. Feature branches must be rebased on latest `main` before creating a PR
2. All commits are squashed to 1-3 meaningful commits before merge

This keeps `git log` on `main` clean and readable:
```
abc123 Add OAuth2 social login feature
def456 Implement payment gateway integration
ghi789 Add user profile management
```

Instead of:
```
abc123 Merge pull request #347 from feature/auth
def456 WIP
ghi789 Fix tests again
jkl012 Fix tests
mno345 Merge branch 'main' into feature/auth
pqr678 Actually fix the thing
```

## INTERVIEW
**Q: You've been working on a feature branch for 3 days. You just found out a teammate merged a critical security fix to main that your feature also needs. How do you get that fix into your branch without creating a merge commit?**

A: Use `git rebase`:
```bash
# 1. Ensure you have the latest main
git fetch origin
git checkout main
git pull origin main

# 2. Rebase your feature branch onto the updated main
git checkout feature/my-feature
git rebase origin/main

# 3. If there are conflicts, resolve them and continue
# git add <resolved-files>
# git rebase --continue
```

Your feature branch is now replayed on top of the security fix. When your PR is reviewed, reviewers will see a clean, linear diff with no merge commits.

**Important caveat:** If you've already pushed this feature branch to remote, you'll need to force-push after rebasing because the commit hashes changed:
```bash
git push --force-with-lease origin feature/my-feature
# --force-with-lease is safer than --force: it only force-pushes
# if nobody else has pushed to the remote branch since your last fetch!
```
