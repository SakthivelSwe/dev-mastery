## WHY
Code review is more than finding bugs — it's the primary mechanism for knowledge sharing, maintaining code quality standards, onboarding new developers, and preventing security vulnerabilities from reaching production.
Done well, code review improves both the code AND the team. Done poorly, it creates friction, slows development, and damages morale.

## THEORY
### The Goals of Code Review
1. **Defect detection:** Catch bugs, logic errors, security issues before production
2. **Knowledge sharing:** Reviewers learn what changed; authors learn from feedback
3. **Standards enforcement:** Consistent patterns, architecture alignment
4. **Mentoring:** Senior developers guide junior developers through PR feedback

## CODE

### What to Look for When Reviewing
```
REVIEWER CHECKLIST:
□ Logic: Does the code actually do what it claims to do?
□ Edge cases: What happens with null, empty list, 0, -1?
□ Security: SQL injection, XSS, exposed secrets, authentication holes?
□ Performance: O(n²) where O(n) is possible? N+1 queries? Missing indexes?
□ Tests: Are critical paths tested? Edge cases tested?
□ Error handling: What happens when things fail? Are exceptions handled?
□ Naming: Are variables/functions named clearly and consistently?
□ Documentation: Complex logic needs comments. Public APIs need docs.
□ Dependencies: Is a new library necessary? Is it maintained and trusted?
□ Breaking changes: Does this change break existing API consumers?
```

### GitHub Code Review Features
```bash
# Review workflow on GitHub:

# 1. See the full diff
gh pr view 47 --web     # Opens in browser

# 2. Leave inline comments on specific lines
# (Click the '+' button next to a line in the GitHub diff view)

# 3. Suggest specific code changes (author can apply with one click!)
# In a review comment, use a suggestion block:
# ```suggestion
# const user = await db.users.findUnique({ where: { id } });
# ```

# 4. Submit review
gh pr review 47 --approve --body "LGTM! Great work on the caching."
gh pr review 47 --request-changes --body "Please add input validation for the email field."
gh pr review 47 --comment --body "Have you considered using a Set here for O(1) lookups?"
```

### Writing Constructive Review Comments
```
BAD: "This is wrong."
BAD: "Why did you do this???"
BAD: "I would never write it this way."

GOOD: "This might cause an issue with empty arrays. Consider adding: if (!arr.length) return []"
GOOD: "Question: why did we choose LinkedHashMap here instead of HashMap? Documenting the reasoning would help future readers."
GOOD: "Suggestion: extracting this into a separate method would make it easier to test in isolation. Not a blocker, just a thought!"
GOOD: "nit: Minor style preference — consider renaming `x` to `userId` for clarity."

GOOD PREFIXES (make intent clear):
"nit:" - Minor, non-blocking stylistic preference
"question:" - Genuine question, not necessarily a change request
"blocker:" - Must be addressed before merge
"suggestion:" - Take it or leave it
"thinking out loud:" - Not a change request, just sharing context
```

## REAL_WORLD
**Review Size Guidelines**
Research shows:
- Optimal PR size for effective review: **200-400 lines**
- At 500+ lines: Reviewers start rubber-stamping (approving without really reviewing)
- At 1000+ lines: The PR is too large to review effectively — should be split

**How to split large PRs:**
1. Refactor separately from feature addition
2. Database migration PR → Feature PR
3. Tests PR → Implementation PR (though usually combined)
4. Split by layer: API changes → UI changes

## INTERVIEW
**Q: You're reviewing a PR that implements a new feature. You notice the code works correctly, but uses a completely different pattern than the rest of the codebase (e.g., uses a different error handling approach, different state management). How do you handle this?**

A: The approach depends on whether the new pattern is genuinely better or just different.

**If the new pattern is better:**
```
"I notice this PR introduces a new error handling pattern using Result types. 
I think this is actually an improvement over our current approach! Before we 
merge, can we: 1) Document this as our new standard, 2) Consider creating a 
ticket to migrate existing code gradually? I'll approve this PR, but I want 
to make sure we're intentional about adopting this pattern team-wide."
```

**If it's just different (no clear improvement):**
```
"This works correctly, but I notice we're using a different error handling 
approach than the rest of the codebase. To maintain consistency, could you 
align with the pattern in UserService.java? If you think this pattern is 
better, let's discuss it in our next team meeting and decide as a team before 
adopting it."
```

The key is separating "this is wrong" from "this is inconsistent." Consistency is valuable. But don't reject genuinely better approaches just because they're new — bring them through the proper channel for team decision-making.
