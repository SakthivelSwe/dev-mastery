## WHY
You just accidentally deleted an unpushed branch. Or you ran `git reset --hard` to the wrong commit. Or a rebase went terribly wrong and you want to undo it.
You might panic, thinking the work is gone forever. But Git almost never deletes data immediately. **Git Reflog** is the ultimate safety net. It records every time the tip of a branch (HEAD) moves, allowing you to "time travel" back to states that are no longer visible in `git log`.

## THEORY
### Log vs Reflog
- `git log`: Shows the history of the *current branch*. If a commit isn't reachable from the current branch, it won't show up.
- `git reflog`: Shows the history of *your actions* on your local machine. It logs every time `HEAD` moved: checkouts, commits, resets, rebases, merges.

Git keeps unreachable (orphaned) commits in its database for 90 days by default before the garbage collector deletes them. The reflog is how you find them.

## CODE

### Using Reflog to Undo a Disaster
```bash
# Disaster scenario: You did a hard reset and lost 3 commits
git reset --hard HEAD~3

# git log shows they are GONE
git log --oneline

# 1. Open the reflog
git reflog
# Output:
# a1b2c3d HEAD@{0}: reset: moving to HEAD~3      <-- You are here
# e5f6g7h HEAD@{1}: commit: Add styling          <-- This is what you lost!
# 8i9j0k1 HEAD@{2}: commit: Add logic
# 2l3m4n5 HEAD@{3}: commit: Start feature

# 2. Reset back to the state before the disaster!
# You want to go back to HEAD@{1}
git reset --hard HEAD@{1}
# OR use the exact commit hash:
git reset --hard e5f6g7h

# Everything is restored!
```

### Recovering a Deleted Branch
```bash
# You deleted a branch that wasn't pushed!
git branch -D feature/important-work

# 1. Find the last commit of that branch in the reflog
git reflog
# 7x8y9z0 HEAD@{4}: checkout: moving from feature/important-work to main
# a1b2c3d HEAD@{5}: commit: Finish important work

# 2. Recreate the branch from that specific commit hash
git checkout -b feature/important-work a1b2c3d
```

### Reflogs for Specific Branches
```bash
# Show HEAD movements across the whole repo
git reflog

# Show movements only for the 'main' branch
git reflog show main

# Time-based reflog queries
git reflog show main@{yesterday}
git reflog show main@{1.week.ago}
```

## REAL_WORLD
**When Reflog Won't Help You**
Reflog is powerful, but it has limits:
1. **It only tracks local actions:** If you delete a branch on someone else's machine or on GitHub, your local reflog doesn't know about it. Reflog tracks *your* repository's `HEAD`.
2. **It only tracks committed work:** If you run `git reset --hard` and you had *unstaged or uncommitted* changes in your working directory, they are gone forever. Git only protects what you have committed.
3. **It expires:** Orphaned commits are eventually garbage collected (default 30 days for unreachable reflog entries, 90 days for reachable ones).

## INTERVIEW
**Q: You started a complex `git rebase` to squash 5 commits into 1. Halfway through resolving conflicts, you realize you've made a mess. You run `git rebase --abort`. However, after aborting, your branch seems to be missing commits. How do you use the reflog to get exactly back to the state right before you typed `git rebase`?**

A: 
1. Run `git reflog`.
2. Look through the output for the entry right before the rebase started. It will look something like this:
   ```
   a1b2c3d HEAD@{0}: rebase: aborting
   ...
   f9e8d7c HEAD@{5}: checkout: moving from main to my-feature  <-- Right here!
   ```
3. Copy the commit hash (`f9e8d7c`) or the reflog index (`HEAD@{5}`).
4. Run `git reset --hard HEAD@{5}` (or the hash).

This forcibly moves your branch pointer and working directory back to the exact state it was in before the rebase disaster began.
