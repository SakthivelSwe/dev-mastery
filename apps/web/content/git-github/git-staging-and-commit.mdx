## WHY
The two most fundamental Git operations are staging (`git add`) and committing (`git commit`). You do these dozens of times per day. Understanding them deeply — including how to write good commit messages, how to undo mistakes, and the nuances of partial staging — makes your Git history a powerful communication tool.

## THEORY
### What is a Commit?
A commit is a snapshot of your project. Every commit has:
- A unique SHA-1 hash (e.g., `a3f82b7`)
- Author name and email
- Timestamp
- Commit message
- Pointer to the parent commit
- Snapshot of ALL tracked files

Git stores commits efficiently using content-addressing — identical files share the same stored object.

## CODE

### Staging Commands
```bash
# Stage a specific file
git add src/api/user.js

# Stage all changes in a directory
git add src/

# Stage ALL changes (new files, modifications, deletions)
git add .

# Stage only modifications and deletions (NOT new untracked files)
git add -u

# Interactive staging (choose which hunks of a file to stage)
git add -p
# This is POWERFUL! You can stage PART of a file:
# You changed 3 functions — stage only 1 for a focused commit!
# For each hunk: (y)es, (n)o, (s)plit, (e)dit, (q)uit

# See what's staged vs unstaged
git status
git diff               # Unstaged changes
git diff --staged      # Staged changes (what will be committed)
```

### Commit Commands
```bash
# Basic commit (opens editor for message)
git commit

# Commit with inline message
git commit -m "Add user authentication middleware"

# Stage ALL tracked files AND commit in one step
git commit -am "Fix login redirect bug"
# Note: -a doesn't stage NEW untracked files! Only tracked+modified.

# Amend the last commit (BEFORE pushing — don't amend pushed commits!)
git add forgotten-file.js
git commit --amend --no-edit  # Add file to last commit, keep message

# Amend commit message
git commit --amend -m "Better commit message for the last commit"

# Create an empty commit (useful for CI triggers, changelog markers)
git commit --allow-empty -m "Deploy: trigger CI pipeline"
```

### Writing Good Commit Messages
```bash
# BAD commit messages:
git commit -m "fix"
git commit -m "changes"
git commit -m "WIP"
git commit -m "asdfgh"

# GOOD commit messages (follow Conventional Commits):
git commit -m "feat: add OAuth2 Google login"
git commit -m "fix: resolve race condition in payment processing"
git commit -m "docs: update API authentication section"
git commit -m "refactor: extract auth logic into separate service"
git commit -m "test: add unit tests for UserService"
git commit -m "chore: update dependencies to latest versions"

# Multi-line commit message (subject + body):
git commit -m "feat: add email verification flow

Users must now verify their email address within 24 hours of
registration. Unverified accounts are restricted to read-only
access. Verification emails are sent via SendGrid.

Closes #234"
```

### Undoing Staged/Committed Changes
```bash
# Unstage a file (keep changes in working directory)
git restore --staged src/api/user.js

# Discard working directory changes (CAREFUL — irreversible!)
git restore src/api/user.js

# Undo the last commit but keep changes staged
git reset --soft HEAD~1

# Undo the last commit and unstage (keep changes in working dir)
git reset HEAD~1

# DANGEROUS: Undo commit AND discard all changes (no undo!)
git reset --hard HEAD~1
```

## REAL_WORLD
**Conventional Commits**
Many teams use the Conventional Commits specification:
`<type>(<scope>): <description>`
- `feat`: New feature → triggers minor version bump in semantic versioning
- `fix`: Bug fix → triggers patch version bump
- `BREAKING CHANGE`: Incompatible change → triggers major version bump

Tools like `commitlint` enforce this in CI, and tools like `semantic-release` use the commit history to automatically version and publish your packages.

## INTERVIEW
**Q: You just ran `git commit --amend` and realized it changed the commit hash. Why did the hash change? And why is this dangerous if you've already pushed the commit?**
A: Every Git commit hash is computed from ALL of the commit's contents: the timestamp, the author, the commit message, the tree (file snapshot), AND the parent commit hash. When you amend, you create a NEW commit (with a new timestamp and possibly new content) — so it gets a different hash.

**The danger:** If you had pushed the original commit and a teammate had pulled it, their Git history now has commit `a3f82b`. You've replaced it locally with `c9d4e2`. When you force-push, your remote now shows `c9d4e2`. Your teammate's branch still points to `a3f82b` (which is now "orphaned" — no longer on any remote branch). When they pull, Git sees two diverging histories and they'll get a confusing conflict.

**Rule:** Only amend commits that have **NOT been pushed to a shared remote**. Once a commit is pushed and others might have it, treat it as immutable. Instead of amending, create a new commit that fixes the problem.
