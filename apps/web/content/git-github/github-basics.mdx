## WHY
Git is a distributed version control tool. GitHub is a cloud platform built on top of Git that adds collaboration features: repository hosting, code review (Pull Requests), issue tracking, project management, CI/CD (GitHub Actions), and a social layer for open-source contribution.
Understanding GitHub is essential — it's where virtually all modern software development collaboration happens, and your GitHub profile is your portfolio to the world.

## THEORY
### Git vs GitHub
- **Git:** The version control tool (free, open-source, runs on your machine)
- **GitHub:** A cloud service that hosts Git repositories + collaboration tools
- **Alternatives:** GitLab (self-hostable), Bitbucket (Atlassian)

GitHub is not Git — it's a company (owned by Microsoft) that built a product on top of Git.

## CODE

### Essential GitHub Concepts
```bash
# Repository (Repo): Your project + its Git history
# Public repos: Visible to everyone on the internet
# Private repos: Only you and collaborators can see

# Forking: Create your own copy of someone else's public repo
# Used for: Contributing to open-source projects
# Your fork: github.com/yourusername/original-project

# Stars: Bookmark/upvote a repo (like a 'like' button)
# Watching: Get notifications for a repo's activity
```

### GitHub CLI (Essential Tool)
```bash
# Install GitHub CLI
# Mac: brew install gh
# Windows: winget install GitHub.cli

# Authenticate
gh auth login

# Create a new repository on GitHub (from your local project)
gh repo create my-project --public --source=. --push

# Clone a repo
gh repo clone facebook/react

# Open current repo in browser
gh repo view --web

# Create an issue
gh issue create --title "Login button not working on mobile" \
  --body "Steps to reproduce: 1. Go to /login 2. On mobile viewport..."

# List open issues
gh issue list

# Create a PR
gh pr create --title "Fix mobile login button" --body "..."

# Review a PR
gh pr review 45 --approve
gh pr review 45 --request-changes --body "Please add tests"

# Merge a PR
gh pr merge 45 --squash --delete-branch
```

### GitHub Repository Structure
```
github.com/user/repo/
├── Code tab          # Browse files, navigate code
├── Issues tab        # Bug reports, feature requests
├── Pull Requests tab # Code review workflow
├── Actions tab       # CI/CD pipelines
├── Projects tab      # Kanban boards, project management
├── Wiki tab          # Documentation
├── Settings tab      # Collaborators, branch protection, secrets

# Special files GitHub auto-renders:
├── README.md         # Project description (shows on repo homepage)
├── CONTRIBUTING.md   # How to contribute guide
├── CODE_OF_CONDUCT.md # Community guidelines
├── LICENSE           # Open-source license
├── .github/
│   ├── ISSUE_TEMPLATE/ # Template for new issues
│   ├── PULL_REQUEST_TEMPLATE.md
│   └── workflows/    # GitHub Actions CI/CD
```

## REAL_WORLD
**GitHub for Open Source Contribution**
```bash
# Contributing to an open-source project:

# 1. Fork the repository (on GitHub.com — click Fork button)

# 2. Clone YOUR fork
git clone git@github.com:yourusername/react.git

# 3. Add the ORIGINAL as upstream
git remote add upstream git@github.com:facebook/react.git

# 4. Create a branch for your fix
git checkout -b fix/typo-in-readme

# 5. Make your change and commit
git commit -m "Fix typo in CONTRIBUTING.md"

# 6. Push to YOUR fork
git push origin fix/typo-in-readme

# 7. Create a PR from YOUR fork to the ORIGINAL repo (on GitHub.com)
gh pr create --repo facebook/react --title "Fix typo" --body "..."

# 8. The maintainers review and (hopefully) merge!
```

## INTERVIEW
**Q: What is the difference between Forking a repository and Cloning it?**

A: Both copy a repository, but they operate differently:

**Cloning** (`git clone`) downloads the repository to your local machine. It's a local copy. You can push changes only if you have write access to the original repository.

**Forking** is a GitHub-level operation (not a Git concept). It creates a copy of the repository under YOUR GitHub account. You have full write access to your fork. The original repository shows that yours is a "fork." You can submit Pull Requests from your fork to the original.

**When to use each:**
- **Clone only:** You have write access to the repo (it's your team's project) — just clone and work directly
- **Fork first:** Contributing to someone else's public repo (open source) where you don't have write access — fork, clone YOUR fork, make changes, then PR to the original
