## WHY
Branches let you work in isolation. But eventually, your feature needs to join the main codebase. **Git Merge** combines the work from one branch into another.
Understanding merge types and how to resolve conflicts is essential — merge conflicts are one of the most common and dreaded moments in collaborative development, but they're straightforward once you understand them.

## THEORY
### Types of Merges
- **Fast-Forward Merge:** If the target branch hasn't diverged (no new commits on main since you branched), Git simply moves the branch pointer forward. No merge commit created. Linear history.
- **Three-Way Merge:** If both branches have new commits, Git finds the common ancestor and creates a new "merge commit" combining both branches. History is non-linear but accurate.
- **Squash Merge:** All commits from the feature branch are squashed into ONE commit and applied to main. Clean history, but you lose individual commit history.

## CODE

### Basic Merging
```bash
# Merge feature/auth into main
git checkout main
git merge feature/auth

# If no conflicts: auto-merged!
# Git opens editor for merge commit message (for 3-way merge)

# Fast-forward (no extra commit created):
git merge --ff-only feature/auth

# Always create a merge commit (even if fast-forward is possible):
git merge --no-ff feature/auth

# Squash: combine all feature commits into one
git merge --squash feature/auth
git commit -m "Add user authentication feature"
```

### Resolving Merge Conflicts
```bash
git merge feature/auth
# CONFLICT (content): Merge conflict in src/api/user.js
# Automatic merge failed; fix conflicts and then commit the result.

# Open the conflicted file — Git adds markers:
cat src/api/user.js

# <<<<<<< HEAD (your current branch - main)
# function getUser() { return db.findUser(); }
# =======
# function getUser(id) { return cache.get(id) || db.findUser(id); }
# >>>>>>> feature/auth (incoming branch)

# Choose what you want to keep:
# Option A: Keep main's version (delete everything else)
# Option B: Keep feature's version (delete everything else)
# Option C: Combine both (the most common choice!)
# function getUser(id) { return cache.get(id) || db.findUser(id); }

# After resolving ALL conflicts, stage and commit:
git add src/api/user.js
git commit -m "Merge feature/auth into main"
# Or use a merge tool for a visual editor:
git mergetool  # Opens VS Code or another configured diff tool
```

### Checking Merge Status
```bash
# See all unresolved conflicts
git status

# See what conflicts look like diff-style
git diff

# See merged branches (safe to delete)
git branch --merged

# See unmerged branches (NOT safe to delete — still have unique commits)
git branch --no-merged
```

### Aborting a Merge
```bash
# Made a mess? Start over!
git merge --abort
# Returns everything to pre-merge state
```

## REAL_WORLD
**Merge Strategy: Squash vs Regular Merge**

| Strategy | Use When |
|---|---|
| Fast-forward | Clean feature branches, linear history preferred |
| Regular merge | Want to preserve when features were integrated |
| Squash merge | Messy WIP commits, want clean main branch history |

GitHub Pull Request merging options match these: "Create a merge commit", "Squash and merge", "Rebase and merge".

Most teams pick **one strategy** and enforce it through GitHub branch protection rules to keep history consistent.

## INTERVIEW
**Q: During a merge, you have a conflict in a binary file (like an image). Git shows the file as conflicted but you can't see diff markers. How do you resolve it?**
A: Binary files can't be merged by Git — it can't do line-by-line diffing on binary content. Git marks the file as conflicted but shows the entire file content from both sides.

You must manually choose which version to keep:

```bash
# Keep YOUR version (main branch's version):
git checkout HEAD -- assets/logo.png

# Keep THEIR version (the branch you're merging):
git checkout MERGE_HEAD -- assets/logo.png

# Then stage it:
git add assets/logo.png
git commit -m "Merge: keep new logo design from feature branch"
```

If you need to actually combine binary files (e.g., merging two Photoshop files), that's a design/content management workflow problem — not something Git can solve. In practice, binary files that change frequently should be managed in a separate system (Git LFS for large files, or a design tool like Figma that handles versioning natively).
