## WHY
Collaboration requires syncing code between your local repository and the remote. Push, pull, and fetch are the three operations that move commits between local and remote repositories. Understanding the subtle difference between pull and fetch prevents many common git headaches.

## THEORY
### The Three Remote Operations
- **`git fetch`:** Downloads commits from remote → local mirror (`origin/main`). Does NOT touch your working branch.
- **`git pull`:** `git fetch` + `git merge` (or rebase). Downloads AND integrates.
- **`git push`:** Sends your local commits to the remote repository.

## CODE

### git push
```bash
# Push current branch to its upstream remote branch
git push

# Explicitly specify remote and branch
git push origin feature/auth

# Push and set upstream tracking (first push of a new branch)
git push -u origin feature/auth
# After -u, just 'git push' works without specifying origin/branch!

# Force push (DANGEROUS on shared branches!)
git push --force
git push --force-with-lease  # Safer: only force-push if no one else pushed!

# Push a tag
git push origin v1.2.0

# Push all tags
git push origin --tags

# Delete a remote branch
git push origin --delete feature/old-branch
```

### git fetch
```bash
# Fetch all remotes (downloads, doesn't merge)
git fetch

# Fetch a specific remote
git fetch origin

# Fetch a specific branch
git fetch origin main

# After fetch, see what changed:
git log HEAD..origin/main --oneline  # Remote commits not yet in local
git diff main origin/main             # Exact file differences
```

### git pull
```bash
# Pull (fetch + merge)
git pull

# Pull with rebase instead of merge (cleaner history!)
git pull --rebase

# Set rebase as default pull behavior (recommended!)
git config --global pull.rebase true

# Pull a specific branch
git pull origin main
```

### Tracking Branches
```bash
# See upstream tracking for each local branch
git branch -vv
# * feature/auth  a3f82b7 [origin/feature/auth] Add OAuth2
#   main          1c4d89e [origin/main] Initial commit

# How many commits ahead/behind the remote:
git status
# Your branch is ahead of 'origin/main' by 3 commits.
# (use "git push" to publish your local commits)

# Or:
# Your branch is behind 'origin/main' by 5 commits, and can be fast-forwarded.
# (use "git pull" to update your local branch)
```

## REAL_WORLD
**The Safe Daily Workflow**
```bash
# Morning: start by fetching (safe, non-destructive)
git fetch origin

# See what teammates pushed overnight
git log HEAD..origin/main --oneline

# Review changes before merging
git diff main origin/main

# When ready, merge or rebase
git rebase origin/main   # Preferred: cleaner history
# OR:
git merge origin/main
```

## INTERVIEW
**Q: Your `git push` fails with "rejected — non-fast-forward." What does this mean and how do you fix it?**
A: This means someone else pushed commits to the remote branch AFTER your last fetch. Your local history and the remote history have diverged — you can't simply add your commits on top because they'd skip the other person's commits.

```
Remote: A → B → C → D  (someone pushed C and D while you were working)
Local:  A → B → E      (you committed E locally)
```

**Fix:**
```bash
# 1. Fetch the remote changes (non-destructive)
git fetch origin

# 2. Rebase your changes on top of the remote (recommended)
git rebase origin/main
# Now: A → B → C → D → E (your E replayed on top of D)

# 3. Push successfully now!
git push origin main
```

**Alternative (creates merge commit):**
```bash
git pull origin main   # Fetch + merge
git push               # Now works, but has a merge commit
```

**NEVER use `--force` to resolve this** (unless it's your own branch no one else is using). Force-pushing would overwrite C and D, erasing your teammate's work!
