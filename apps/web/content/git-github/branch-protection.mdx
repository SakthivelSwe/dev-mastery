## WHY
In a team environment, anyone can accidentally run `git push origin main --force` and delete the entire repository's history. Or they might push broken code directly to the `main` branch, breaking production.
**Branch Protection rules** enforce workflows and prevent destructive actions. They ensure that all code entering important branches (like `main` or `release`) meets specific quality and review standards.

## THEORY
### Common Protection Rules
- **Require Pull Request reviews:** Code cannot be pushed directly; it must be reviewed by N other developers via a PR.
- **Require status checks to pass:** The CI pipeline (tests, linters, security scans) must succeed before merging.
- **Require signed commits:** Commits must be cryptographically signed (GPG/SSH) to verify identity.
- **Prevent force pushes:** Stops anyone from rewriting history on the protected branch.
- **Prevent deletions:** Stops anyone from deleting the branch.

### Configuring via GitHub API/CLI (or UI)
While usually configured via the GitHub Settings UI (Settings -> Branches -> Add branch protection rule), it can also be done programmatically.

```bash
# Example logic conceptually (GitHub doesn't have a direct CLI command for all rules yet, usually done via API)
curl -X PUT \
  -H "Accept: application/vnd.github.v3+json" \
  -H "Authorization: token $GITHUB_TOKEN" \
  https://api.github.com/repos/OWNER/REPO/branches/main/protection \
  -d '{
    "required_status_checks": {
      "strict": true,
      "contexts": ["ci/github-actions"]
    },
    "enforce_admins": true,
    "required_pull_request_reviews": {
      "dismiss_stale_reviews": true,
      "require_code_owner_reviews": true,
      "required_approving_review_count": 1
    },
    "restrictions": null
  }'
```

### The Codeowners File
A `CODEOWNERS` file defines who is responsible for specific files or directories in a repository. When combined with branch protection, you can mandate that changes to certain files *must* be approved by their specific owners.

```gitignore
# .github/CODEOWNERS

# Default owners for everything
*       @frontend-team @backend-team

# Only the security team can approve changes to auth/
/src/auth/      @security-team

# Only DB admins can approve migration changes
/prisma/migrations/  @database-admins

# Specific individuals
/docs/          @tech-writer-alice
```

## REAL_WORLD
**The "Break Glass" Scenario**
Sometimes, a critical production issue occurs, and the CI pipeline is broken or the required reviewers are asleep. How do you merge the hotfix?
- **Option 1 (Dangerous):** Temporarily disable branch protection. (Bad audit trail).
- **Option 2 (Better):** Use the `enforce_admins: false` setting. This allows repository administrators to bypass the rules (force merge) when necessary, while still keeping the rules active for regular contributors. GitHub clearly marks bypassed merges in the audit log.

## INTERVIEW
**Q: You have branch protection enabled requiring CI status checks to pass before merging. Your PR is approved, but the "Deploy Preview" CI job is failing because the preview server is down. The unit tests passed. How do you merge the PR?**

A: If the "Deploy Preview" check is marked as a *Required Status Check* in the branch protection settings, the PR physically cannot be merged (the button is disabled) until it passes, unless you are an admin and bypass rules are allowed.

**Solutions:**
1. **Fix the preview server:** The best approach. Make the check pass.
2. **Admin Override:** An administrator can force-merge if `enforce_admins` is disabled, but this bypasses *all* checks.
3. **Change the Protection Rules:** A repo admin can temporarily remove "Deploy Preview" from the list of *required* checks. It will still run, but its failure won't block the merge. This is common for flaky non-critical checks. You should only make critical checks (like unit tests or security scans) required.
