## WHY
React Components are supposed to be "Pure Functions." They take inputs (Props) and return an output (JSX). 
However, real-world applications need to do "Impure" things: fetching data from an external API over the network, manually manipulating the DOM to initialize a third-party charting library, or setting up a WebSocket connection.

These actions are called **Side Effects**. If you put an HTTP request directly inside the main body of your component, React will execute that HTTP request *every single time* the component re-renders (which could be 60 times a second if the user is typing). This will instantly crash the server.

You need a way to tell React: *"Wait until the component has finished rendering the HTML to the screen, and THEN execute this Side Effect just once."* We do this using the **`useEffect`** hook.

## THEORY
### The Two Arguments of useEffect
1. **The Callback Function**: The code you want to run (e.g., the API fetch).
2. **The Dependency Array**: An array of variables. React watches these variables. If (and only if) one of these variables changes between renders, React will run the Callback Function again.

### The Component Lifecycle
Using the Dependency Array, `useEffect` allows you to hook into three distinct moments of a Component's life:
- **Mounting**: When the component first appears on the screen.
- **Updating**: When specific State or Props change.
- **Unmounting**: When the component is destroyed and removed from the screen.

## CODE

### Level 1 — Beginner (Run Once on Mount)
If you provide an **Empty Array `[]`** as the second argument, React will run the effect exactly *once* when the component first loads, and never again. This is where you fetch initial data!

```javascript
import { useState, useEffect } from 'react';

export default function UserProfile() {
  const [userData, setUserData] = useState(null);

  useEffect(() => {
    // This code runs AFTER the initial HTML is painted to the screen!
    console.log("Fetching data from API...");
    
    fetch('https://api.example.com/user/1')
      .then(res => res.json())
      .then(data => {
        // When the data arrives, we update State.
        // This triggers a re-render, and the screen updates!
        setUserData(data); 
      });
      
  }, []); // <-- The Empty Array means "Only run on Mount!"

  if (!userData) return <p>Loading...</p>;
  return <h1>{userData.name}</h1>;
}
```

### Level 2 — Building Blocks (Run on State Change)
What if the User ID is dynamic, passed in as a Prop? If the Prop changes from `1` to `2`, you need to fetch the new user! You must put the dynamic variable inside the Dependency Array.

```javascript
export default function UserProfile({ userId }) {
  const [userData, setUserData] = useState(null);

  useEffect(() => {
    // This runs on Mount, AND it runs again anytime 'userId' changes!
    console.log(`Fetching data for User ${userId}...`);
    
    fetch(`https://api.example.com/user/${userId}`)
      .then(res => res.json())
      .then(data => setUserData(data));
      
  }, [userId]); // <-- React is watching 'userId'!

  if (!userData) return <p>Loading...</p>;
  return <h1>{userData.name}</h1>;
}
```

### Level 3 — Intermediate (The Cleanup Function)
If you subscribe to a WebSocket, or start a `setInterval`, that process will keep running forever in the background. If the user navigates away to a different page, the Component is destroyed, but the timer keeps ticking in memory, causing a massive Memory Leak!

To fix this, your `useEffect` callback can `return` a **Cleanup Function**. React will execute this cleanup function right before the component is destroyed.

```javascript
import { useState, useEffect } from 'react';

export default function Timer() {
  const [seconds, setSeconds] = useState(0);

  useEffect(() => {
    // 1. We start an infinite timer
    const intervalId = setInterval(() => {
      setSeconds(prev => prev + 1);
    }, 1000);

    // 2. We RETURN a function to clean it up!
    // When the component unmounts, React runs this automatically!
    return () => {
      console.log("Component destroyed! Clearing timer...");
      clearInterval(intervalId);
    };
    
  }, []); 

  return <h1>Seconds: {seconds}</h1>;
}
```

### Level 4 — Advanced (Async/Await inside useEffect)
You cannot make the `useEffect` callback itself an `async` function (e.g., `useEffect(async () => {})`). Why? Because `useEffect` expects you to return either *nothing* or a *Cleanup Function*. An `async` function automatically returns a *Promise*, which completely breaks the cleanup mechanism!

To use `async/await`, you must define the function *inside* the effect, and then call it immediately.

```javascript
export default function DataFetcher() {
  
  useEffect(() => {
    // 1. Define the async function
    const fetchData = async () => {
      try {
        const response = await fetch('/api/data');
        const json = await response.json();
        console.log(json);
      } catch (error) {
        console.error("Fetch failed", error);
      }
    };

    // 2. Execute it!
    fetchData();
  }, []);

  return <div>Data loaded</div>;
}
```

## REAL_WORLD
`useEffect` is the most abused hook in React. Junior developers use it for everything, creating horrifying "Effect Chains" where updating State A triggers Effect B, which updates State C, which triggers Effect D. This makes the application impossible to debug and incredibly slow.

**The Golden Rule of React:** *You don’t need Effects to transform data for rendering.*
If you have an array of `users` in State, and you want to calculate the `activeUsers`, DO NOT use a `useEffect` to filter them and save them into a second State variable! Just calculate it directly during render: `const activeUsers = users.filter(u => u.active);`. 

Furthermore, in modern Enterprise React (Next.js), fetching data inside `useEffect` is considered an anti-pattern. You should use **React Server Components** or libraries like **React Query (TanStack Query)**, which handle caching, deduping, and race conditions automatically.

## INTERVIEW
**Q: What happens if you completely omit the Dependency Array? (`useEffect(() => {...})`)**
A: The effect will run after the initial mount, AND it will run after *every single re-render* of the component. This is almost never what you want and is a common source of infinite loops (if the effect updates state, which causes a re-render, which runs the effect, which updates state...).

**Q: In React 18 Strict Mode, why does my `useEffect` with an empty array `[]` run twice when the page loads?**
A: In development mode only, React 18 intentionally mounts your component, instantly unmounts it, and then mounts it again. It does this to aggressively test your Cleanup Functions! If your app breaks because the effect ran twice, it proves you forgot to write a cleanup function (like `clearInterval`), and you have a hidden memory leak!

**Q: Can you put `useEffect` inside an `if` statement?**
A: No! Hooks must be called unconditionally at the top level of the component. If you only want the effect to run conditionally, put the `if` statement *inside* the `useEffect` callback function!

## FEYNMAN CHECK

Explain to a 10-year-old:
> `useEffect` runs AFTER the browser paints — not during render. It is for synchronising React with the outside world: APIs, timers, DOM, WebSockets. The cleanup function runs before the NEXT effect and on unmount — like cleaning your desk before starting the next task. The dependency array tells React when to re-run the effect: `[]` means once (mount only), `[id]` means every time `id` changes, no array means every render. React StrictMode runs effects TWICE in development to catch cleanup omissions.
