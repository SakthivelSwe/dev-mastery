## WHY
In React, data flows strictly from Top to Bottom using Props. If the `<App>` component holds the `currentUser` state, and a `<ProfileAvatar>` component 6 levels deep needs that data, you are forced to pass the `currentUser` prop down through 5 intermediate components that don't care about the user at all.

This anti-pattern is called **Prop Drilling**, and it destroys the readability of a codebase.

React provides the **Context API** to solve this. Context creates a "Wormhole" or "Teleporter" for data. You can place the `currentUser` data into a Context Provider at the top of the tree, and *any* component, no matter how deep, can reach into the wormhole and extract the data instantly using the **`useContext`** hook, bypassing the middle layers entirely!

## THEORY
### The Three Steps of Context
1. **Create the Context**: You create the Wormhole (usually in a separate file).
2. **Provide the Context**: You wrap your application in a `<Provider>` and stuff the data into the Wormhole.
3. **Consume the Context**: A deep Child component uses `useContext()` to pull the data out of the Wormhole.

## CODE

### Level 1 — Beginner (Creating and Providing)
```javascript
// 1. ThemeContext.js (Create the Wormhole)
import { createContext } from 'react';

// We export the Context so other files can import it!
// The "light" is just a default fallback value if no Provider is found.
export const ThemeContext = createContext("light");
```

```javascript
// 2. App.jsx (Provide the Data)
import { useState } from 'react';
import { ThemeContext } from './ThemeContext';
import DeepLayout from './DeepLayout';

export default function App() {
  const [theme, setTheme] = useState("dark");

  return (
    // We wrap our app in the Provider and pass the 'value' prop!
    // ANY component inside this Provider can access "dark"!
    <ThemeContext.Provider value={theme}>
      
      {/* We do NOT pass the theme as a prop! */}
      <DeepLayout />
      
      <button onClick={() => setTheme(theme === 'dark' ? 'light' : 'dark')}>
        Toggle Theme
      </button>
      
    </ThemeContext.Provider>
  );
}
```

### Level 2 — Building Blocks (Consuming the Data)
```javascript
// 3. DeepAvatar.jsx (Consume the Data 5 levels deep!)
import { useContext } from 'react';
import { ThemeContext } from './ThemeContext';

export default function DeepAvatar() {
  
  // We use the Hook to reach into the Wormhole!
  const currentTheme = useContext(ThemeContext);

  return (
    <div className={`avatar avatar-${currentTheme}`}>
      <img src="/face.jpg" alt="User" />
    </div>
  );
}
```
*Magic: When the `<App>` toggles the theme from 'dark' to 'light', React will automatically re-render `<DeepAvatar>`, instantly applying the new theme!*

### Level 3 — Intermediate (Providing Data and Setters)
Passing read-only data is great, but usually, the deep Child component wants to *change* the data! You can pass an Object containing both the State AND the Setter function into the Context!

```javascript
// AuthContext.jsx
import { createContext, useState } from 'react';

// 1. Create the Context
export const AuthContext = createContext(null);

// 2. Create a Custom Provider Component!
// This is an industry standard pattern. We encapsulate the State logic 
// entirely inside this custom Provider component!
export function AuthProvider({ children }) {
  const [user, setUser] = useState(null);

  const login = (name) => setUser({ name });
  const logout = () => setUser(null);

  return (
    // We pass an Object containing the user, login, and logout functions!
    <AuthContext.Provider value={{ user, login, logout }}>
      {children}
    </AuthContext.Provider>
  );
}
```

```javascript
// index.jsx
// Wrap the entire app in the Custom Provider
root.render(
  <AuthProvider>
    <App />
  </AuthProvider>
);
```

```javascript
// Navbar.jsx
import { useContext } from 'react';
import { AuthContext } from './AuthContext';

export default function Navbar() {
  // Destructure the object we placed in the Provider!
  const { user, login, logout } = useContext(AuthContext);

  if (user) {
    return <button onClick={logout}>Logout {user.name}</button>;
  }
  return <button onClick={() => login("Alice")}>Login</button>;
}
```

### Level 4 — Advanced (The Performance Trap)
The Context API has a massive, deadly flaw. 

**When the `value` of a Provider changes, EVERY single component that consumes that Context will instantly re-render.**

If you have a massive Global State object holding the `theme`, `currentUser`, and `shoppingCart`, and you update the `shoppingCart`, the `<DeepAvatar>` component (which only cares about the `theme`) will be forced to re-render because the overall Context Object changed! This causes catastrophic performance issues in large apps.

## REAL_WORLD
Because of the Performance Trap, **Context is NOT a State Management tool.** 

Context is an incredibly powerful Dependency Injection tool for data that changes *very rarely* (like the User's Theme, their Authenticated JWT Token, or their Language preference). 

If you have highly dynamic data that changes 60 times a second (like dragging an item across a Kanban board, or live chat messages), you MUST NOT use the Context API. You should use a dedicated State Manager like **Redux**, **Zustand**, or **Jotai**. These libraries are scientifically engineered to surgically update *only* the specific components that care about the exact piece of data that changed.

## INTERVIEW
**Q: How do you access Context in a Class Component?**
A: Since Hooks (`useContext`) cannot be used inside Classes, you have to use the `<ThemeContext.Consumer>` wrapper component, which requires a messy Render Props pattern. This is one of the main reasons the industry abandoned Class Components.

**Q: Can a component consume two different Contexts?**
A: Absolutely. A component can call `useContext(ThemeContext)` and `useContext(AuthContext)` perfectly fine, assuming the component is wrapped by both Providers higher up in the tree.

**Q: What happens if you call `useContext` but forget to wrap the app in a `<Provider>`?**
A: The application will not crash! Instead, `useContext` will silently return the default value you provided when you ran `createContext("default_value")`. This often leads to confusing bugs where the data appears completely frozen.

## FEYNMAN CHECK

Explain to a 10-year-old:
> Context broadcasts a value to all descendants without prop-drilling through intermediates. Every component that consumes the context re-renders when the context value changes. This is the critical performance trap: putting a frequently-updating value (e.g., mouse position) in a context at the app root causes EVERY consuming component to re-render on every mouse move. Solution: split contexts into slow-changing (theme, user) and fast-changing (UI state), or use `useSyncExternalStore` for granular subscriptions.
