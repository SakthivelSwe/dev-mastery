## WHY
Every time a Component's State or Prop changes, React throws away the old UI and re-executes the entire Component function from the very first line of code.

If you have a heavy mathematical calculation inside that function (e.g., sorting an array of 50,000 items), React will re-sort that array *every single time* the user types a letter in an unrelated search box, freezing the browser.

Similarly, if you declare a function `const handleClick = () => {}` inside your component, React creates a *brand new memory address* for that function on every render. If you pass that function as a prop to a Child component, the Child sees a new memory address, assumes the function changed, and aggressively re-renders itself!

React provides **`useMemo`** and **`useCallback`** to cache (memoize) calculations and functions between renders, preventing catastrophic performance leaks.

## THEORY
### useMemo (Caching Values)
`useMemo` caches the result of a calculation. It takes a function and a Dependency Array. It only re-calculates the value if one of the dependencies actually changes. Otherwise, it simply returns the cached result from the previous render.

### useCallback (Caching Functions)
`useCallback` caches the *memory address* of a function. It ensures that the function reference remains identical across re-renders, preventing Child components from re-rendering unnecessarily.

## CODE

### Level 1 — Beginner (The Problem with Heavy Math)
```javascript
import { useState } from 'react';

export default function SlowComponent() {
  const [count, setCount] = useState(0);
  const [text, setText] = useState("");

  // This is a brutal 500ms algorithm.
  // BUG: Because React re-executes the whole file, this math runs EVERY time!
  // If the user types in the input, the text updates, triggering a re-render.
  // The user typing will be delayed by 500ms per keystroke!
  const heavyResult = slowMathAlgorithm(count);

  return (
    <div>
      <h1>Result: {heavyResult}</h1>
      <button onClick={() => setCount(count + 1)}>Increment Math</button>
      
      {/* Typing here will lag the entire app! */}
      <input value={text} onChange={e => setText(e.target.value)} />
    </div>
  );
}
```

### Level 2 — Building Blocks (Fixing it with useMemo)
We wrap the heavy calculation in `useMemo`.

```javascript
import { useState, useMemo } from 'react';

export default function FastComponent() {
  const [count, setCount] = useState(0);
  const [text, setText] = useState("");

  // 1. We wrap the algorithm in useMemo!
  // 2. We provide [count] as the dependency array!
  const heavyResult = useMemo(() => {
    console.log("Running heavy math...");
    return slowMathAlgorithm(count);
  }, [count]); 

  // MAGIC: If the user types in the text box, 'text' changes.
  // React re-renders. It hits useMemo. It checks 'count'. 'count' hasn't changed!
  // React SKIPS the algorithm completely and instantly returns the cached result!
  // The typing is blazing fast!

  return (
    <div>
      <h1>Result: {heavyResult}</h1>
      <button onClick={() => setCount(count + 1)}>Increment Math</button>
      <input value={text} onChange={e => setText(e.target.value)} />
    </div>
  );
}
```

### Level 3 — Intermediate (The Problem with Functions)
Imagine a highly optimized `<Graph />` component that is wrapped in `React.memo` (meaning it only re-renders if its Props change).

```javascript
import { useState, useCallback } from 'react';
import HighlyOptimizedGraph from './Graph';

export default function Dashboard() {
  const [theme, setTheme] = useState("dark");

  // Every time 'theme' changes, Dashboard re-renders.
  // A brand new handleDataClick function is created in memory!
  const handleDataClick = () => {
    console.log("Graph clicked!");
  };

  return (
    <div>
      <button onClick={() => setTheme(theme === 'dark' ? 'light' : 'dark')}>
        Toggle Theme
      </button>

      {/* 
        BUG: The Graph sees a brand new 'onDataClick' memory address.
        It assumes the function is completely different, and it re-renders 
        the entire massive Graph just because the Theme changed!
      */}
      <HighlyOptimizedGraph onDataClick={handleDataClick} />
    </div>
  );
}
```

### Level 4 — Advanced (Fixing it with useCallback)
We wrap the function in `useCallback` to freeze its memory address!

```javascript
export default function Dashboard() {
  const [theme, setTheme] = useState("dark");

  // MAGIC: useCallback caches the memory address of the function!
  // When 'theme' changes, React returns the EXACT SAME memory address 
  // from the previous render!
  const handleDataClick = useCallback(() => {
    console.log("Graph clicked!");
  }, []); // <-- Empty array means this function NEVER changes.

  return (
    <div>
      <button onClick={() => setTheme('light')}>Toggle</button>
      
      {/* 
        The Graph sees the exact same memory address for onDataClick!
        The Graph skips rendering completely, saving massive CPU cycles!
      */}
      <HighlyOptimizedGraph onDataClick={handleDataClick} />
    </div>
  );
}
```

## REAL_WORLD
Junior developers often learn about `useMemo` and start wrapping **EVERYTHING** in it: every variable, every array, every function.

**This is a massive anti-pattern.** `useMemo` has an internal performance cost. React has to allocate memory for the cache, and it has to perform a mathematical loop to check the Dependency Array on every render. If your calculation takes 1 millisecond, wrapping it in `useMemo` might actually make your app *slower* because the caching logic takes 2 milliseconds!

You should only use `useMemo` for:
1. Genuinely heavy mathematical loops (thousands of items).
2. Creating an Object/Array that you intend to pass as a Prop to a heavily optimized Child component, to prevent breaking the Child's `React.memo`.

## INTERVIEW
**Q: What is the exact difference between `useMemo` and `useCallback`?**
A: They are mathematically identical. `useCallback(fn, deps)` is just syntactic sugar for `useMemo(() => fn, deps)`. `useMemo` is used to cache the *result* of calling a function. `useCallback` is used to cache the *function itself*.

**Q: Can I use `useMemo` instead of `useEffect` to fetch data?**
A: **Absolutely not.** `useMemo` runs *synchronously during the render phase*. If you put a network request in it, you will literally pause the browser tab and freeze the UI until the network request finishes. `useEffect` runs asynchronously *after* the render phase, allowing the UI to paint first.

**Q: What is `React.memo`?**
A: It is a Higher-Order Component. If you wrap a component `export default React.memo(MyComponent)`, you are telling React: *"Only re-render this component if its Props have mathematically changed."* It works hand-in-hand with `useCallback` (which ensures the Props don't mathematically change).

## FEYNMAN CHECK

Explain to a 10-year-old:
> `useMemo` caches the RESULT of a computation. `useCallback` caches the FUNCTION REFERENCE. Both recompute when their dependency array changes. The golden rule: don't add them speculatively — profile first. Wrapping every function in `useCallback` is NOT free: React still calls the hook on every render, compares deps, and only skips the allocation if deps didn't change. The real use case: `useCallback` prevents a child wrapped in `React.memo` from re-rendering when the parent re-renders.
