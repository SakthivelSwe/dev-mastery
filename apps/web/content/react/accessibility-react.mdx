## WHY
In modern web development, creating beautiful User Interfaces is not enough. You must ensure that your application is usable by *everyone*, including people using Screen Readers, people relying entirely on keyboard navigation (no mouse), and people with visual impairments.
Ignoring Accessibility (a11y) is not just bad UX—it is literally illegal in many jurisdictions, leading to massive lawsuits against companies whose websites do not meet WCAG (Web Content Accessibility Guidelines) compliance. 
React developers often build custom UI components (like a custom `<Select>` dropdown) that look great but are completely invisible or unusable to screen readers.

## THEORY
### Semantic HTML vs. `<div>` Soup
The easiest way to make a React app accessible is to use native HTML elements exactly as they were designed.
A `<button>` inherently knows how to receive keyboard focus (via the Tab key) and inherently responds to the Enter and Spacebar keys.
If you build a custom button using `<div onClick={handleClick}>Click Me</div>`, you have destroyed all of that built-in accessibility. A screen reader sees a `<div>` and ignores it. A keyboard user cannot Tab to it. It is a dead element to them.

## CODE

### Level 1 — The Keyboard Trap
Always ensure interactive elements are reachable via the `Tab` key.
If you MUST make a `<div>` clickable, you are responsible for manually reinventing everything a `<button>` does for free.

```jsx
// TERRIBLE: Inaccessible. 
// 1. Cannot be reached by Tab key.
// 2. Cannot be activated via Enter key.
// 3. Screen readers announce it as plain text.
<div className="btn" onClick={submitForm}>Submit</div>

// GOOD: Use the native element!
<button className="btn" onClick={submitForm}>Submit</button>

// ACCEPTABLE (But why?): Manually recreating accessibility
<div 
  className="btn" 
  onClick={submitForm}
  role="button" // Tell screen readers it acts like a button
  tabIndex={0} // Make it reachable via the Tab key!
  onKeyDown={(e) => {
    // Buttons must activate on Enter AND Space!
    if (e.key === 'Enter' || e.key === ' ') {
      e.preventDefault();
      submitForm();
    }
  }}
>
  Submit
</div>
```

### Level 2 — ARIA Attributes
ARIA (Accessible Rich Internet Applications) attributes are HTML properties that provide extra context to screen readers when visual cues are not enough.

```jsx
// A visual-only icon button. A sighted user sees a Trash Can and knows it means "Delete".
// A blind user hears absolutely nothing.
// BAD
<button onClick={deleteUser}>
  <TrashIcon />
</button>

// GOOD: Use aria-label to provide an invisible text description for screen readers!
<button onClick={deleteUser} aria-label="Delete User">
  <TrashIcon />
</button>
```

When building complex components like Modals or Accordions, you must use `aria-expanded` and `aria-hidden` to communicate state changes.

```jsx
function Accordion({ title, children }) {
  const [isOpen, setIsOpen] = useState(false);

  return (
    <div>
      {/* aria-expanded tells the screen reader the current state! */}
      <button 
        onClick={() => setIsOpen(!isOpen)}
        aria-expanded={isOpen}
        aria-controls="accordion-content"
      >
        {title}
      </button>
      
      {/* aria-hidden explicitly hides the content from the screen reader when closed */}
      <div 
        id="accordion-content" 
        hidden={!isOpen}
        aria-hidden={!isOpen}
      >
        {children}
      </div>
    </div>
  );
}
```

### Level 3 — Focus Management (Modals)
When a Modal opens, a sighted user's attention is drawn to the center of the screen. 
A screen reader user doesn't know the Modal opened. Their keyboard focus is still buried in the background content behind the modal!
You must manually move the `focus()` into the modal when it opens, and you must "trap" the focus inside the modal so they don't accidentally tab back into the background.

```jsx
import { useEffect, useRef } from 'react';

function Modal({ isOpen, onClose, children }) {
  const modalRef = useRef(null);

  useEffect(() => {
    if (isOpen && modalRef.current) {
      // 1. Move focus into the modal immediately when it opens!
      modalRef.current.focus();
    }
  }, [isOpen]);

  if (!isOpen) return null;

  return (
    <div className="fixed inset-0 bg-black/50">
      <div 
        role="dialog" 
        aria-modal="true" // Tells screen reader background is inactive
        ref={modalRef} 
        tabIndex={-1} // Programmatically focusable, but skipped by manual Tabbing
        className="bg-white p-6 outline-none"
      >
        <button onClick={onClose} aria-label="Close modal">X</button>
        {children}
      </div>
    </div>
  );
}
```

### Level 4 — Live Regions (`aria-live`)
If an AJAX request finishes and a "Success! File Saved!" banner pops up at the top of the screen, sighted users see it. Screen reader users have no idea the text appeared unless their cursor is actively on it.
You use `aria-live` to force the screen reader to interrupt whatever it is doing and announce the dynamic update immediately!

```jsx
function NotificationArea({ message }) {
  return (
    // 'polite' waits for the screen reader to finish its current sentence.
    // 'assertive' interrupts the screen reader instantly (use for critical errors).
    <div aria-live="polite" className="sr-only">
      {/* The 'sr-only' CSS class hides the text visually, but keeps it in the DOM for screen readers! */}
      {message}
    </div>
  );
}
```

## REAL_WORLD
**eslint-plugin-jsx-a11y**
Nobody remembers all 150 ARIA rules. The industry standard is to automate compliance by installing `eslint-plugin-jsx-a11y`.
If you write `<img src="dog.jpg" />`, the linter will instantly throw a compiler error: *“img elements must have an alt prop”*.
If you write `<div onClick={handleClick}>`, the linter will scream: *“Visible, non-interactive elements with click handlers must have at least one keyboard listener”*.
Enabling this ESLint plugin is the single most effective way to guarantee your React app remains accessible as the team grows.

## INTERVIEW
**Q: You have a standard HTML `<form>` with an `<input>` and a submit `<button>`. You added `aria-label="Email Address"` to the input so screen readers can understand it. A senior engineer rejects your PR and tells you to use a `<label>` element instead. Why is `<label>` vastly superior to `aria-label` here?**
A: **The Hitbox Extension (Usability).**
`aria-label` ONLY helps screen reader users. It is completely invisible to users with motor-control disabilities or anyone using a mouse.
When you use a proper HTML `<label htmlFor="emailInput">Email Address</label>`, the browser natively links the text to the input. 
Crucially, if a user clicks on the word "Email Address" anywhere on the screen, the browser will automatically focus the input field! This massively increases the clickable "hitbox" of the input, making it significantly easier to use for users with shaky hands (and mobile users with "fat fingers"). ARIA attributes should only be used as a last resort when native HTML elements cannot accomplish the goal!
