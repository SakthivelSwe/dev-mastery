## WHY
In React, when Component A renders Component B, the resulting HTML for B is strictly nested inside the HTML for A. 
This strict HTML nesting causes massive CSS headaches for components that need to "break out" of their containers—specifically **Modals, Tooltips, and Dropdowns**.
If a Modal is deeply nested inside a `div` that has `overflow: hidden` or `z-index: 1`, the Modal will be clipped or hidden behind other elements, completely ruining the UI.
**React Portals** provide a magical escape hatch. They allow a React component to exist logically inside the React Tree, but render its HTML into a completely different part of the physical DOM (like directly inside the `<body>` tag)!

## THEORY
### The Logical Tree vs. The Physical Tree
With Portals, the React Component Tree and the actual HTML DOM Tree diverge.
Even if a Portal renders its HTML at the very bottom of the `<body>`, it is still "logically" a child of the component that rendered it.
This is critical because it means React features like **Context API** and **Event Bubbling** still work exactly as expected! An `onClick` event fired inside the Portal will bubble up to its logical React parent, even though they are miles apart in the physical HTML.

## CODE

### Level 1 — Creating a Portal
To use a Portal, you use `createPortal` from `react-dom`. It takes two arguments:
1. The JSX you want to render.
2. The physical DOM node where you want to inject it.

First, your `index.html` usually needs a target `div` outside the main React root:
```html
<body>
  <div id="root"></div> <!-- Your entire React app lives here -->
  <div id="modal-root"></div> <!-- Portals will be teleported here! -->
</body>
```

Now, create the Portal component:
```jsx
import { createPortal } from 'react-dom';

function Modal({ children, isOpen }) {
  if (!isOpen) return null;

  // We are NOT returning standard JSX. We are returning a Portal!
  return createPortal(
    <div className="fixed inset-0 bg-black/50 flex justify-center items-center">
      <div className="bg-white p-6 rounded-lg z-50">
        {children}
      </div>
    </div>,
    document.getElementById('modal-root') // The teleport destination!
  );
}
```

### Level 2 — Using the Portal
You use the `Modal` exactly like a normal component. You don't have to worry about where it physically renders.

```jsx
export function Dashboard() {
  const [showModal, setShowModal] = useState(false);

  return (
    // This div is a tiny box with overflow: hidden!
    // Without a portal, the modal would be trapped and invisible.
    <div className="w-32 h-32 overflow-hidden border border-red-500">
      <button onClick={() => setShowModal(true)}>Delete Account</button>
      
      {/* 
        Logically, the Modal is right here. 
        Physically, React teleports the HTML to <div id="modal-root">!
        It completely escapes the overflow: hidden trap!
      */}
      <Modal isOpen={showModal}>
        <h2>Are you sure?</h2>
        <button onClick={() => setShowModal(false)}>Cancel</button>
      </Modal>
    </div>
  );
}
```

### Level 3 — Event Bubbling Magic
Because the Portal is logically still inside the `Dashboard` component, events bubble upwards through the React Tree, NOT the HTML tree.

```jsx
function App() {
  const handleWrapperClick = () => {
    console.log("Wrapper was clicked!");
  };

  return (
    // We put the onClick on the parent...
    <div onClick={handleWrapperClick}>
      <Modal isOpen={true}>
        {/* Even though this button is physically injected into the <body>, 
            clicking it will STILL trigger handleWrapperClick! */}
        <button>Click Me</button>
      </Modal>
    </div>
  );
}
```

### Level 4 — Dynamic Portals
Hardcoding `document.getElementById('modal-root')` works, but it isn't flexible (what if the element doesn't exist?). Robust component libraries dynamically create the DOM node on the fly!

```jsx
import { useEffect, useState } from 'react';
import { createPortal } from 'react-dom';

function Tooltip({ children }) {
  const [portalNode, setPortalNode] = useState(null);

  useEffect(() => {
    // 1. Create a brand new div when the component mounts
    const div = document.createElement('div');
    // 2. Append it directly to the body
    document.body.appendChild(div);
    // 3. Save it to state so we can render into it
    setPortalNode(div);

    // 4. Cleanup! Remove the div when the Tooltip unmounts!
    return () => {
      document.body.removeChild(div);
    };
  }, []);

  if (!portalNode) return null;

  return createPortal(
    <div className="absolute top-0 right-0 bg-gray-800 text-white p-2">
      {children}
    </div>,
    portalNode
  );
}
```

## REAL_WORLD
**Headless UI Libraries**
If you inspect the DOM of an enterprise application using Radix UI or Material UI, and you open a Dropdown Menu, you will notice the dropdown's HTML is injected at the very bottom of the `<body>` tag.
They all use Portals under the hood to guarantee that your dropdown is never clipped by a random `overflow: hidden` on a parent layout container. 

## INTERVIEW
**Q: You implemented a Portal for a Modal. Inside the Modal, you use a Context Consumer (e.g., `useTheme()`). Will the Portal successfully receive the context values from the Context Provider wrapping the main `<App />`, even though the Modal's HTML is rendered entirely outside the `<div id="root">`?**
A: **Yes.**
Context is tied strictly to the React Component Tree (the logical tree), not the physical DOM tree. Because the Portal component was rendered *logically* inside a component that was wrapped by the Context Provider, it has full access to the Context, regardless of where `createPortal` physically injects the resulting DOM nodes.

## FEYNMAN CHECK

Explain to a 10-year-old:
> Portals render a React component's output OUTSIDE its parent DOM node while keeping it INSIDE the React component tree. This means the portal component still receives context from its React ancestors and event bubbling propagates through React's tree, not the DOM tree. The primary use case: modals, tooltips, and dropdowns that must escape `overflow: hidden` or `z-index` stacking contexts of their parent.
