## WHY
React is a "Library", not a "Framework". Out of the box, React only provides UI rendering. It does not provide Routing, Server-Side Rendering (SSR), SEO optimizations, API endpoints, or Image Optimization.
If you build a large application using pure React (like Create React App or Vite), you will eventually hit a wall where your site takes 5 seconds to load on mobile devices, and Google's SEO web crawlers cannot read your content because it is hidden behind a massive JavaScript bundle.
**Next.js** is the official, industry-standard Framework built *on top* of React. It solves all of these problems by moving rendering to the Server, providing file-based routing, and acting as the full-stack architecture for modern React.

## THEORY
### The Rendering Spectrums
Next.js supports every rendering paradigm simultaneously, allowing you to choose the exact strategy per page:
1. **CSR (Client-Side Rendering):** The standard React way. Downloads a blank HTML file and massive JS. Slow initial load, fast subsequent navigations.
2. **SSR (Server-Side Rendering):** The server generates the HTML dynamically *on every single request*. Great for highly dynamic, personalized data, but requires a fast server.
3. **SSG (Static Site Generation):** The HTML is generated exactly once during the `npm run build` process. Served instantly via a CDN. Perfect for Blogs and Marketing pages.
4. **ISR (Incremental Static Regeneration):** The holy grail. The page is statically generated, but the server rebuilds it in the background every X seconds without requiring a full redeploy!

## CODE

### Level 1 — The App Router (File-Based Routing)
In standard React, you use `react-router-dom` and manually map URL paths to components.
In Next.js, the file system *is* the router. If you create a file at `app/dashboard/settings/page.jsx`, Next.js automatically creates the `/dashboard/settings` URL!

```jsx
// app/blog/[slug]/page.jsx
// The [slug] bracket notation means this is a Dynamic Route! 
// It matches /blog/hello-world or /blog/react-guide

export default function BlogPost({ params }) {
  // 'params.slug' will be 'hello-world'
  return <h1>Reading post: {params.slug}</h1>;
}
```

### Level 2 — Data Fetching (Server Components)
Next.js natively implements React Server Components. All components inside the `app/` directory run on the server by default.

```jsx
// app/products/page.jsx
// This runs on the Node.js server! 
export default async function ProductsPage() {
  
  // Next.js patches the native `fetch` API to automatically cache data!
  const res = await fetch('https://api.example.com/products', {
    // ISR Magic: Next.js caches this result for 3600 seconds (1 hour).
    // If a user visits the page after an hour, Next.js instantly serves 
    // the stale page, then rebuilds a fresh HTML page in the background!
    next: { revalidate: 3600 } 
  });
  
  const products = await res.json();

  return (
    <ul>
      {products.map(p => <li key={p.id}>{p.name}</li>)}
    </ul>
  );
}
```

### Level 3 — API Routes (Full Stack React)
Next.js allows you to build your backend API directly inside the same repository! You don't need a separate Express.js server.

```javascript
// app/api/users/route.js
import { NextResponse } from 'next/server';
import { db } from '@/lib/db';

export async function GET(request) {
  // Run backend code directly! Connect to databases!
  const users = await db.query("SELECT * FROM users");
  
  return NextResponse.json(users);
}

export async function POST(request) {
  const body = await request.json();
  // ... save to database
  return NextResponse.json({ success: true }, { status: 201 });
}
```

Next.js includes powerful built-in components that dramatically improve Google Lighthouse scores and Core Web Vitals.


## REAL_WORLD
**Metadata and SEO**
The primary reason businesses abandon standard React (CSR) for Next.js is SEO (Search Engine Optimization).
If you inspect the HTML source code of a standard React app, it looks like this: `<div id="root"></div>`. When the Google Bot scrapes the page, it sees absolutely no content. It has to wait to execute the heavy JavaScript to see the text, which drastically penalizes your ranking.
In Next.js, because the server generates the final HTML, the Google Bot instantly sees all the content. Furthermore, Next.js allows you to dynamically generate `<title>` and `<meta>` tags based on the database data!

```jsx
// app/products/[id]/page.jsx
export async function generateMetadata({ params }) {
  const product = await fetchProduct(params.id);
  
  // This dynamically injects the SEO tags into the HTML <head>!
  return {
    title: `${product.name} | MyStore`,
    description: product.summary,
    openGraph: { images: [product.thumbnail] } // Renders Twitter/Discord preview cards!
  };
}
```

## INTERVIEW
**Q: You use `fetch('api/data')` inside a Server Component in the `app/` directory. When you run `npm run build`, the build process suddenly crashes, saying it cannot reach the API. Why does data fetching break the build process in Next.js?**
A: **Static Site Generation (SSG) runs during the Build Step!**
By default, Next.js attempts to statically generate as many pages as possible *during the build process* to make them instantly fast at runtime.
When Next.js encounters your Server Component during the build, it actually *executes* the component! It hits the `fetch` statement and attempts to make a network request to your API. If your API server is not running, or the database is inaccessible during the CI/CD build pipeline, the fetch fails, and the Next.js build crashes.
*The Fix:* If the data is highly dynamic and you *want* it to fetch at runtime (not build time), you must opt the page out of Static Generation by adding `export const dynamic = 'force-dynamic';` at the top of the file, or using `no-store` in the fetch options!

## FEYNMAN CHECK

Explain to a 10-year-old:
> Next.js is React + a server. It can render pages on the server (SSR — fresh on every request), at build time (SSG — pre-baked HTML), incrementally (ISR — revalidate after N seconds), or not at all (CSR — pure client React). The App Router (Next.js 13+) uses React Server Components by default — components are server components unless you add `'use client'`. This means your data fetching logic stays on the server, never ships to the browser.
