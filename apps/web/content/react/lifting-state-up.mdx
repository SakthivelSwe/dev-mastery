## WHY
In React, data flows strictly downwards (Unidirectional Data Flow). A parent can pass data to a child via `props`, but a child cannot directly modify the parent's data, nor can sibling components talk to each other directly.

What happens when you have two sibling components (like a `SearchInput` and a `ProductList`) that need to share the exact same state? The `SearchInput` needs to update the search term, and the `ProductList` needs to read it to filter the items. 
Because siblings cannot communicate, you must **Lift the State Up** to their closest common ancestor.

## THEORY
### The Shared Ancestor
To share state between two components, you find the closest parent component that wraps them both. 
You move the `useState` definition into that parent. 
Then, you pass the `state` value down to the component that needs to *read* it, and you pass the `setState` function down to the component that needs to *update* it.

## CODE

### Level 1 — The Problem (Isolated State)
Here, the `AccordionPanel` components are completely isolated. Clicking one panel opens it, but it doesn't close the others! We want an Accordion where only ONE panel can be open at a time.

```jsx
// BAD: Each panel manages its own isolated state!
function AccordionPanel({ title, children }) {
  const [isActive, setIsActive] = useState(false);

  return (
    <div className="border p-4">
      <h3 onClick={() => setIsActive(!isActive)} className="cursor-pointer">
        {title}
      </h3>
      {isActive && <p>{children}</p>}
    </div>
  );
}

export function Accordion() {
  return (
    <div>
      <AccordionPanel title="Panel 1">Content 1</AccordionPanel>
      <AccordionPanel title="Panel 2">Content 2</AccordionPanel>
    </div>
  );
}
```

### Level 2 — Lifting the State
To fix the Accordion, the Parent (`Accordion`) needs to know which panel is currently active. We must strip the state out of the children and move it to the parent!

```jsx
// The Child is now "Controlled" by the Parent! It has no state of its own.
function AccordionPanel({ title, isActive, onShow, children }) {
  return (
    <div className="border p-4">
      <h3 onClick={onShow} className="cursor-pointer font-bold">
        {title}
      </h3>
      {isActive && <p className="mt-2">{children}</p>}
    </div>
  );
}

export function Accordion() {
  // THE STATE LIVES HERE NOW!
  // It tracks the index of the currently active panel.
  const [activeIndex, setActiveIndex] = useState(0);

  return (
    <div>
      <AccordionPanel 
        title="Panel 1" 
        isActive={activeIndex === 0} 
        onShow={() => setActiveIndex(0)}
      >
        Content 1
      </AccordionPanel>
      
      <AccordionPanel 
        title="Panel 2" 
        isActive={activeIndex === 1} 
        onShow={() => setActiveIndex(1)}
      >
        Content 2
      </AccordionPanel>
    </div>
  );
}
```

### Level 3 — Prop Drilling (The Consequence)
Lifting state is the correct React pattern, but it introduces a new problem: **Prop Drilling**.
If the Parent is 5 levels deep from the Child that actually needs the state, you must manually pass the `props` down through 4 intermediate components that don't even care about the data!

```jsx
// The state lives at the very top.
function App() {
  const [theme, setTheme] = useState("light");
  // We have to pass 'theme' and 'setTheme' down...
  return <Layout theme={theme} setTheme={setTheme} />;
}

// Layout doesn't care about theme, but it has to accept the props just to pass them down!
function Layout({ theme, setTheme }) {
  return <Header theme={theme} setTheme={setTheme} />;
}

function Header({ theme, setTheme }) {
  return <ThemeToggle theme={theme} setTheme={setTheme} />;
}

function ThemeToggle({ theme, setTheme }) {
  return (
    <button onClick={() => setTheme(theme === "light" ? "dark" : "light")}>
      Toggle Theme
    </button>
  );
}
```
Prop drilling creates brittle, hard-to-refactor code.

## REAL_WORLD
**When NOT to Lift State Up**
Junior developers often lift *everything* to the root `App` component, creating a massive, unmaintainable god-object that forces the entire application to re-render whenever a single input changes.
*Rule of Thumb:* Only lift state to the **closest** common ancestor. If a modal's `isOpen` state only affects the `Dashboard` component, put it in `Dashboard`. Do not put it in `App`!

How do we solve Prop Drilling if we shouldn't lift state endlessly?
We use two advanced patterns:
1. **Component Composition** (Passing JSX as props)
2. **Context API / Global State** (Zustand, Redux)

## INTERVIEW
**Q: You have a `Form` component that maintains a large state object for user inputs. Inside the form, you have a tiny `Tooltip` component that has its own `isHovered` state to show/hide a message. Should you lift the `isHovered` state up into the `Form` component's state?**
A: **No.** 
State should be kept as local as possible. The `isHovered` state is ephemeral UI state that only matters to the `Tooltip` component itself. The `Form` component does not need to know or care if the user is currently hovering over the tooltip. If you lift `isHovered` into the `Form`, every time the user moves their mouse over the tooltip, the *entire* massive form will re-render, destroying performance. Keep state local unless another component explicitly depends on it!

## FEYNMAN CHECK

Explain to a 10-year-old:
> When two sibling components need to share state, lift it to their common ancestor. The parent owns the state and passes it down as props plus a callback setter. The siblings never communicate directly — they only talk through the parent. If you find yourself syncing state between siblings (updating one when the other changes), stop — you have duplicate state, and it will desynchronise. One source of truth, always.
