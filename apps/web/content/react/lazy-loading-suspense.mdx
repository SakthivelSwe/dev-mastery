## WHY
When you build a React application with Webpack or Vite, all your JavaScript is bundled into one massive file (e.g., `main.bundle.js`). 
If your app has a heavy 5-Megabyte `DataAnalyticsDashboard` component, the user must download that entire 5MB file before they can see the login screen—even if they never actually visit the dashboard! This results in terrible Initial Page Load times.

**Lazy Loading (Code Splitting)** allows you to break your app into smaller chunks. React will only download the JavaScript for the Dashboard when the user physically navigates to that page.

## THEORY
### Dynamic Imports vs. Static Imports
Standard static imports (`import { Dashboard } from './Dashboard'`) force the bundler to include the code in the main bundle immediately.
Dynamic imports (`import('./Dashboard')`) return a Promise. The bundler sees this and splits `Dashboard.jsx` into a completely separate file (e.g., `chunk-482.js`).
React's `lazy()` function wraps this dynamic import, allowing you to render the lazily-loaded component just like a normal React component.

## CODE

### Level 1 — `React.lazy`
To lazy-load a component, you replace the standard import with `React.lazy`.
*(Note: The component you are lazy-loading MUST be the `default export` of its file!).*

```jsx
import React, { lazy, useState } from 'react';

// BAD (Static Import): Downloaded immediately!
// import HeavyChart from './HeavyChart';

// GOOD (Lazy Import): Returns a Promise!
const HeavyChart = lazy(() => import('./HeavyChart'));

export function App() {
  const [showChart, setShowChart] = useState(false);

  return (
    <div>
      <button onClick={() => setShowChart(true)}>View Analytics</button>
      
      {/* 
        When showChart becomes true, React initiates the network request 
        to download the chunk-xxx.js file containing HeavyChart!
      */}
      {showChart && <HeavyChart />}
    </div>
  );
}
```

### Level 2 — React `<Suspense>`
If you run the code in Level 1, your app will crash with an error!
When the user clicks the button, React starts downloading the Javascript file. But rendering in React is synchronous. React doesn't know what to put on the screen while it waits for the network download to finish!

You must wrap lazy-loaded components in a `<Suspense>` boundary. The boundary catches the "Wait" signal and displays a fallback UI (like a spinner).

```jsx
import React, { lazy, Suspense } from 'react';

const HeavyChart = lazy(() => import('./HeavyChart'));

export function App() {
  return (
    <div>
      <h1>Analytics</h1>
      
      {/* If any component inside Suspense is "suspended" (downloading), 
          the fallback UI is shown instead! */}
      <Suspense fallback={<div className="spinner">Downloading component...</div>}>
        <HeavyChart />
      </Suspense>
    </div>
  );
}
```

### Level 3 — Route-Based Code Splitting
The most common and impactful place to use Lazy Loading is at the Routing layer.
If you have a `/login` page and an `/admin` page, a user logging in shouldn't download the admin code until they successfully authenticate.

```jsx
import { BrowserRouter, Routes, Route } from 'react-router-dom';
import { lazy, Suspense } from 'react';

// Split the app by Routes!
const Home = lazy(() => import('./pages/Home'));
const AdminDashboard = lazy(() => import('./pages/AdminDashboard'));
const Settings = lazy(() => import('./pages/Settings'));

export function App() {
  return (
    <BrowserRouter>
      {/* A single Suspense boundary at the top of the routes! */}
      <Suspense fallback={<div>Loading Page...</div>}>
        <Routes>
          <Route path="/" element={<Home />} />
          <Route path="/admin" element={<AdminDashboard />} />
          <Route path="/settings" element={<Settings />} />
        </Routes>
      </Suspense>
    </BrowserRouter>
  );
}
```

### Level 4 — Suspense for Data Fetching
Originally, `<Suspense>` was only for lazy-loading JavaScript files.
However, React's vision for Suspense is much grander. It is designed to suspend rendering while waiting for *anything asynchronous*—most importantly, **Data Fetching**.
Modern libraries like React Query or Next.js use this heavily. 

```jsx
// Imagine this component is using React Query with 'suspense: true'
function UserProfile() {
  // It doesn't need to check "isLoading" anymore!
  // If the data isn't ready, the hook throws a Promise, 
  // triggering the Suspense boundary above it automatically!
  const { data } = useSuspenseQuery({ queryKey: ['user'], queryFn: fetchUser });
  
  return <div>{data.name}</div>;
}

export function App() {
  return (
    // This single boundary handles BOTH downloading the JS file 
    // AND waiting for the API to return the user data!
    <Suspense fallback={<SkeletonLoader />}>
      <UserProfile />
    </Suspense>
  );
}
```

## REAL_WORLD
**The "Named Export" Workaround**
`React.lazy` strictly requires the module to have a `default export`.
If your component uses named exports (`export const MyComponent = () => {}`), `React.lazy` will crash.
To fix this, you must write a slightly ugly Promise resolver:

```jsx
// Workaround for Named Exports
const MyComponent = lazy(() => 
  import('./MyComponents').then(module => ({ default: module.MyComponent }))
);
```
*(Because of this annoyance, many teams enforce a rule: standard components use named exports, but Page-level components use default exports specifically to make routing and lazy-loading easier).*

## INTERVIEW
**Q: You wrap a massive component in `<Suspense>` and `React.lazy`. You deploy the app. A user loads the app, turns their phone on Airplane mode (losing internet), and clicks the button to view the massive component. What happens, and how do you handle it?**
A: **The Promise Rejects and crashes the App.**
When the user clicks the button, `React.lazy` attempts to download the JavaScript chunk over the network. Because the network is disconnected, the HTTP request fails. The Promise returned by the dynamic `import()` rejects.
React `<Suspense>` does *not* catch errors; it only catches pending Promises! 
The rejected Promise will bubble up as an unhandled error and crash the entire React application, resulting in a white screen of death.
**The Fix:** You must wrap the `<Suspense>` boundary inside an **Error Boundary** component. The Error Boundary will catch the network failure and display a fallback UI like "Network error. Please check your connection and click to retry."
