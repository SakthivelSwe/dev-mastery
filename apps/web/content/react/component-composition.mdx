## WHY
Prop Drilling occurs when you pass data through 5 layers of components just so the 5th component can use it. It makes your code incredibly brittle.
Most developers immediately reach for the Context API or Redux to fix this. But 90% of the time, that is overkill. The official React documentation recommends solving Prop Drilling using a much simpler, more powerful pattern: **Component Composition** (passing JSX as props).

## THEORY
### Inversion of Control
Instead of Component A passing raw data to Component B, and Component B passing it to Component C to render it... 
Component A just renders Component C directly, and passes the *entire rendered JSX block* into Component B!
This "Inversion of Control" removes the need for intermediate components to know anything about the data.

## CODE

### Level 1 — The Prop Drilling Nightmare
Here is a classic layout that suffers from Prop Drilling. `Layout` and `Sidebar` don't care about the user data, but they have to accept it just to pass it down to `UserProfile`.

```jsx
// BAD: Deep Prop Drilling
function App() {
  const user = { name: "John", role: "Admin" };
  return <Layout user={user} />;
}

function Layout({ user }) {
  return (
    <div className="flex">
      <Sidebar user={user} />
      <main>Main Content</main>
    </div>
  );
}

function Sidebar({ user }) {
  return (
    <aside>
      <nav>Links...</nav>
      {/* Sidebar shouldn't have to manage UserProfile! */}
      <UserProfile user={user} />
    </aside>
  );
}

function UserProfile({ user }) {
  return <div className="p-4">Logged in as {user.name}</div>;
}
```

### Level 2 — The Solution: The `children` Prop
We can fix this by using React's special `children` prop. `Layout` and `Sidebar` become "Containers". They don't know what is inside them, they just render whatever JSX is passed between their opening and closing tags!

```jsx
// GOOD: Composition using 'children'
function App() {
  const user = { name: "John", role: "Admin" };
  
  // App completely controls what goes inside the Sidebar!
  // No data needs to be passed down as props!
  return (
    <Layout>
      <Sidebar>
        <UserProfile user={user} />
      </Sidebar>
    </Layout>
  );
}

// Layout just wraps whatever is passed to it!
function Layout({ children }) {
  return (
    <div className="flex">
      {children}
      <main>Main Content</main>
    </div>
  );
}

// Sidebar is now a generic container! It knows nothing about 'user'!
function Sidebar({ children }) {
  return (
    <aside>
      <nav>Links...</nav>
      {/* Render the UserProfile (or anything else) here! */}
      {children}
    </aside>
  );
}
```
*Look at `Sidebar`! We completely deleted the `user` prop! The Prop Drilling is gone!*

### Level 3 — Slots (Multiple JSX Props)
What if a component has multiple distinct areas? For example, a `Modal` component needs a Header, a Body, and a Footer. You can't just use `children` because you need to place the JSX in 3 different spots.
You simply pass the JSX into named props! (This is called "Slots" in Vue/Svelte).

```jsx
// Modal expects 3 different chunks of JSX!
function Modal({ header, footer, children }) {
  return (
    <div className="modal-wrapper">
      <div className="modal-header">{header}</div>
      <div className="modal-body">{children}</div>
      <div className="modal-footer">{footer}</div>
    </div>
  );
}

function App() {
  // We pass entire React elements (JSX) into the props!
  return (
    <Modal 
      header={<h2 className="text-xl">Delete File?</h2>}
      footer={
        <div className="flex gap-2">
           <button>Cancel</button>
           <button className="bg-red-500">Delete</button>
        </div>
      }
    >
      <p>Are you sure you want to delete this file? This cannot be undone.</p>
    </Modal>
  );
}
```

## REAL_WORLD
**Headless UI Components**
Component Composition is the foundational concept behind modern "Headless" UI libraries (like Radix UI or Headless UI).
Instead of giving you a monolithic `<Dropdown items={myItems} />` component that is impossible to customize, they use composition to give you complete control over the rendering:

```jsx
// Radix UI Composition Pattern
<DropdownMenu.Root>
  <DropdownMenu.Trigger asChild>
    <button>Options</button>
  </DropdownMenu.Trigger>

  <DropdownMenu.Portal>
    <DropdownMenu.Content>
      <DropdownMenu.Item>Profile</DropdownMenu.Item>
      <DropdownMenu.Item>Settings</DropdownMenu.Item>
    </DropdownMenu.Content>
  </DropdownMenu.Portal>
</DropdownMenu.Root>
```
Because the components are composed, you can insert your own CSS classes, icons, or tooltips exactly where you want them, without waiting for the library author to add a specific `prop` to support your use case!

## INTERVIEW
**Q: Is passing a component via props (e.g., `<Layout sidebar={<Sidebar />} />`) bad for performance? Will React re-render `<Sidebar />` every time `<Layout />` re-renders because it's a new prop?**
A: **No, it is actually GREAT for performance (Render Bailing).**
This is a massive misconception. When the parent `App` renders `<Layout sidebar={<Sidebar />} />`, the `Sidebar` element is created by `App`.
If `Layout` experiences a state change and re-renders, it simply returns the *exact same React Element reference* for `sidebar` that it received from `App`. Because the reference hasn't changed, React completely skips re-rendering `Sidebar`!
Using the `children` prop or passing JSX as props is one of the most effective ways to optimize React applications and prevent unnecessary re-renders in heavy layout components!
