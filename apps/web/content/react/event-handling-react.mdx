## WHY
A User Interface is useless if it doesn't respond to the user. When a user clicks a button, types in an input, or hovers over a card, the browser fires an "Event." 

In vanilla JavaScript, you would attach an event listener by writing `document.getElementById('btn').addEventListener('click', myFunction)`. In React, because we write our UI declaratively in JSX, we can attach Event Listeners directly to the JSX elements themselves, making the code vastly easier to read and maintain.

## THEORY
### React Synthetic Events
When you attach an `onClick` listener in React, you aren't actually using the raw Browser DOM event. React wraps the native browser event in a **SyntheticEvent** wrapper. 

Why? Because Chrome, Firefox, and Safari all have slightly different implementations of events under the hood. React's SyntheticEvent normalizes these differences, guaranteeing that your event handling code behaves exactly the same across every browser in existence.

### Naming Conventions
In HTML, event names are lowercase: `<button onclick="doSomething()">`.
In React JSX, events are strictly **camelCase**: `<button onClick={doSomething}>`.

## CODE

### Level 1 — Beginner (The Basics)
You pass a *reference* to a function directly into the event prop.

```javascript
export default function ClickButton() {
  
  // 1. Define the event handler function
  function handleClick() {
    alert("Button was clicked!");
  }

  return (
    // 2. Pass the function REFERENCE to the onClick prop
    // ❌ DO NOT write onClick={handleClick()} with parentheses! 
    // If you add parentheses, the function will execute immediately 
    // the moment the page loads, instead of waiting for the click!
    <button onClick={handleClick}>
      Click Me
    </button>
  );
}
```

### Level 2 — Building Blocks (Passing Arguments)
What if you have three buttons, and you want them all to use the same function, but pass a different ID? If you write `onClick={deleteItem(id)}`, it executes immediately. 

To pass arguments without executing immediately, you must wrap the call in an **Anonymous Arrow Function**.

```javascript
export default function ProductList() {
  
  function deleteItem(productId) {
    console.log("Deleting product ID:", productId);
  }

  return (
    <div>
      {/* 
        We pass an arrow function! 
        When the button is clicked, it executes the arrow function, 
        which THEN calls deleteItem(1) 
      */}
      <button onClick={() => deleteItem(1)}>Delete Product 1</button>
      <button onClick={() => deleteItem(2)}>Delete Product 2</button>
    </div>
  );
}
```

### Level 3 — Intermediate (The Event Object)
When the event fires, React automatically passes an Event Object (`e`) to your handler function. This object contains crucial information about the event (like the X/Y coordinates of the mouse, or what letter the user just typed).

```javascript
export default function MouseTracker() {
  
  // The 'e' object contains all the event data!
  function handleHover(e) {
    console.log("Mouse X:", e.clientX);
    console.log("Mouse Y:", e.clientY);
  }

  return (
    <div 
      onMouseMove={handleHover} 
      className="w-64 h-64 bg-blue-200"
    >
      Hover over me!
    </div>
  );
}
```

### Level 4 — Advanced (Preventing Default Behavior & Propagation)
Browsers have deeply ingrained default behaviors. For example, clicking an `<a href="">` tag instantly refreshes the page. Submitting a `<form>` instantly refreshes the page. 
In a React Single Page Application, page refreshes destroy your entire application state. You MUST stop them using `e.preventDefault()`.

Additionally, if you click a button inside a card, the click event "bubbles up" and triggers the card's click event too. You stop this using `e.stopPropagation()`.

```javascript
export default function LoginForm() {
  
  function handleSubmit(e) {
    // 1. STOP THE BROWSER FROM REFRESHING THE PAGE!
    e.preventDefault();
    
    console.log("Form submitted via AJAX!");
  }

  function handleCardClick() {
    console.log("Card was clicked!");
  }

  function handleButtonClick(e) {
    // 2. STOP THE CLICK FROM BUBBLING UP TO THE CARD!
    e.stopPropagation();
    
    console.log("Only the button was clicked!");
  }

  return (
    <div onClick={handleCardClick} className="card">
      <form onSubmit={handleSubmit}>
        <input type="text" placeholder="Username" />
        <button type="submit" onClick={handleButtonClick}>
          Login
        </button>
      </form>
    </div>
  );
}
```

## REAL_WORLD
When building complex UI elements (like dragging and dropping, or handling a custom right-click context menu), developers don't usually write raw Event Handlers from scratch.

Enterprise teams rely on robust libraries like `react-use` or `framer-motion` which provide pre-built Hooks for complex gestures (e.g., `useScroll`, `useHover`, `useKeyPress`), saving hundreds of hours of manual DOM math.

## INTERVIEW
**Q: Can you define the event handler arrow function directly inline in the JSX?**
A: Yes, you can write `<button onClick={() => console.log('Click')}>`. However, if the logic is more than one line, you should extract it into a named function above the `return` statement. Extracting it makes the code significantly easier to read and test.

**Q: In React, does attaching `onClick` to 1,000 list items create 1,000 separate event listeners in the DOM?**
A: No! This is a massive performance feature of React. React uses **Event Delegation**. It attaches exactly *one* global event listener to the root of the document. When a click happens anywhere on the screen, React intercepts it, figures out which specific JSX component the click originated from, and fires the corresponding synthetic event. This saves massive amounts of memory.

**Q: What is the difference between `onChange` in React and `onchange` in native HTML?**
A: In native HTML, an input's `onchange` event only fires *after* the user types something and then clicks outside the input box (loses focus). In React, `onChange` is magically re-wired to fire on every single keystroke instantly, behaving like HTML's `oninput` event.

## FEYNMAN CHECK

Explain to a 10-year-old:
> React uses SyntheticEvents — wrappers around native browser events that normalise cross-browser differences. React attaches ONE event listener to the root element (event delegation), not one per DOM node. This is why removing a React component automatically removes its event handlers. Calling `e.stopPropagation()` stops the synthetic event from bubbling up through React's component tree, but it does NOT stop the native DOM event — events attached via `addEventListener` on the same element may still fire.
