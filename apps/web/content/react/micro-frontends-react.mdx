## WHY
As a tech company scales to hundreds of engineers, the single monolithic React repository (the "Frontend Monolith") becomes a nightmare. 
Fifty developers pushing code to the same repository daily causes merge conflicts, 45-minute CI/CD build times, and the terrifying risk that a tiny bug in the "Shopping Cart" component will crash the entire "Video Player" page.
**Micro-Frontends** solve this. Just like Microservices split the backend, Micro-Frontends split the frontend into completely independent applications. Team A builds and deploys the Shopping Cart using React 18, Team B builds the Product Page using Vue, and the user's browser stitches them together seamlessly into a single web page.

## THEORY
### Module Federation
There are many ways to implement Micro-Frontends (iframes, NGINX routing), but the modern industry standard is **Webpack Module Federation** (or Vite Module Federation).
Module Federation allows a JavaScript application to dynamically load code from *another* completely separate application over the internet at runtime!
1. **The Host (Shell):** The main application frame (e.g., Header, Footer, Routing).
2. **The Remote (Micro-Frontend):** An independent app (e.g., Checkout flow) that exposes specific components for the Host to consume.

## CODE

### Level 1 — Configuring the Remote (Exposing a Component)
Imagine Team A maintains the `Checkout` application. It is its own GitHub repo, its own Webpack build, and runs on `https://checkout.example.com`.
They configure their `webpack.config.js` to expose their `<Cart />` component to the world!

```javascript
// webpack.config.js (Team A - The Remote)
const ModuleFederationPlugin = require("webpack/lib/container/ModuleFederationPlugin");

module.exports = {
  plugins: [
    new ModuleFederationPlugin({
      name: "checkoutApp", // The globally unique name of this micro-frontend
      filename: "remoteEntry.js", // The manifest file generated by Webpack
      exposes: {
        // Expose the internal React component to the public!
        "./CartWidget": "./src/components/Cart", 
      },
      shared: { react: { singleton: true }, "react-dom": { singleton: true } },
    }),
  ],
};
```

### Level 2 — Configuring the Host (Consuming the Component)
Team B maintains the main `Storefront` application running on `https://store.example.com`.
They configure their Webpack to point to Team A's remote server.

```javascript
// webpack.config.js (Team B - The Host)
const ModuleFederationPlugin = require("webpack/lib/container/ModuleFederationPlugin");

module.exports = {
  plugins: [
    new ModuleFederationPlugin({
      name: "hostApp",
      remotes: {
        // "checkout" is the alias. 
        // It points to Team A's deployed JavaScript manifest!
        checkout: "checkoutApp@https://checkout.example.com/remoteEntry.js",
      },
      shared: { react: { singleton: true }, "react-dom": { singleton: true } },
    }),
  ],
};
```

### Level 3 — Dynamic Rendering in React
Inside the Host React application, you dynamically import the component over the network as if it were a local file!
Because it downloads over the internet at runtime, you MUST wrap it in `React.lazy` and `<Suspense>`.

```jsx
// App.jsx (Team B - The Host)
import React, { Suspense } from 'react';

// MAGIC: We are importing a component from a completely different server!
// Webpack intercepts "checkout/CartWidget" and downloads the JS from Team A.
const RemoteCartWidget = React.lazy(() => import('checkout/CartWidget'));

export function App() {
  return (
    <div>
      <header>My Global Store</header>
      
      <main>
        <h1>Products</h1>
        {/* Render Team A's component seamlessly! */}
        <Suspense fallback={<div>Loading Cart...</div>}>
          <RemoteCartWidget />
        </Suspense>
      </main>
    </div>
  );
}
```

### Level 4 — Sharing State (The Global Event Bus)
The Golden Rule of Micro-Frontends is **Isolation**. They should *not* share a Redux store. If Team A changes the Redux schema, Team B's app will crash.
How do they communicate? (e.g., clicking "Add to Cart" in the Host updates the count in the Remote).
They communicate using native browser CustomEvents, acting as a decentralized Pub/Sub system!

```jsx
// In the Host App (Clicking a button)
const handleAddToCart = (product) => {
  // Dispatch a native browser event containing the payload!
  const event = new CustomEvent('ADD_TO_CART', { detail: { id: product.id } });
  window.dispatchEvent(event);
};

// In the Remote Cart App (Listening for the event)
useEffect(() => {
  const listener = (event) => {
    console.log("Adding product ID to cart:", event.detail.id);
    updateCartLogic(event.detail.id);
  };

  window.addEventListener('ADD_TO_CART', listener);
  return () => window.removeEventListener('ADD_TO_CART', listener);
}, []);
```

## REAL_WORLD
**Independent Deployments**
The true superpower of Module Federation is CI/CD.
If Team A fixes a bug in the `<CartWidget />`, they run their pipeline and deploy `checkoutApp` to production.
**Team B does NOTHING.**
The Host app (`store.example.com`) does not need to rebuild. It does not need to deploy. The next time a user hits refresh on the Host app, their browser fetches `remoteEntry.js` from Team A's server, instantly downloading the bug fix. The architecture is completely decoupled!

## INTERVIEW
**Q: Both the Host app and the Remote app use React 18. If the browser downloads the Host JS bundle (which includes React), and then downloads the Remote JS bundle (which also includes React), won't the browser download React twice, destroying performance and causing version conflicts?**
A: **No, because of Shared Dependencies.**
In the `webpack.config.js` for both apps, we added `shared: { react: { singleton: true } }`.
This is a critical feature of Module Federation. When the Host app boots up, it loads its copy of React into a shared global scope. When the Remote app downloads, Webpack checks the global scope. It sees that React is already loaded. Instead of downloading React again from Team A's server, the Remote app seamlessly hooks into the Host's existing React instance! This guarantees optimal performance and prevents the dreaded "multiple instances of React" error.
