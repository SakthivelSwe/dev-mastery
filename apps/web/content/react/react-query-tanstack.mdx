## WHY
Historically, developers fetched data in React using `useEffect` and stored the result in `useState`. They also needed a `loading` state, an `error` state, and maybe a Redux store if the data needed to be shared across pages.
This manual approach is deeply flawed. It doesn't handle caching, it doesn't automatically retry failed requests, it doesn't background-refresh stale data, and it leads to race conditions.

**TanStack Query (React Query)** completely eliminates this nightmare. It treats fetching not as "State Management", but as **"Server State Synchronization"**. It is arguably the most important library in the modern React ecosystem.

## THEORY
### The Cache is King
React Query sits between your components and your API.
When Component A requests data (e.g., `fetchUser(1)`), React Query fetches it, stores it in a massive internal cache, and gives it to Component A.
If Component B suddenly mounts and requests the exact same data (`fetchUser(1)`), React Query intercepts the request. It instantly gives Component B the cached data (zero loading time!), while silently doing a background re-fetch to ensure the data is still accurate.

## CODE

### Level 1 — Queries (`useQuery`)
You wrap your app in a `<QueryClientProvider>`. Then, you use `useQuery` to fetch data.
A Query needs two things: a unique `queryKey` (how the cache identifies the data), and a `queryFn` (the actual Promise that fetches the data).

```jsx
import { useQuery } from '@tanstack/react-query';
import axios from 'axios';

function UserProfile({ userId }) {
  // The magic hook!
  const { data, isLoading, isError, error } = useQuery({
    // The Cache Key! An array. If the userId changes, React Query 
    // knows it's a completely different cache entry!
    queryKey: ['user', userId],
    // The Fetch Function
    queryFn: () => axios.get(`/api/users/${userId}`).then(res => res.data)
  });

  if (isLoading) return <div>Loading user skeleton...</div>;
  if (isError) return <div>Failed to load: {error.message}</div>;

  return <h1>Welcome, {data.name}!</h1>;
}
```

### Level 2 — Mutations (`useMutation`)
While `useQuery` is for reading data (GET), `useMutation` is for creating, updating, or deleting data (POST, PUT, DELETE).
Mutations don't run automatically on mount; they give you a trigger function (e.g., `mutate()`).

```jsx
import { useMutation, useQueryClient } from '@tanstack/react-query';
import axios from 'axios';

function AddUserForm() {
  const queryClient = useQueryClient();

  const { mutate, isPending } = useMutation({
    // The function that executes the POST request
    mutationFn: (newUser) => axios.post('/api/users', newUser),
    
    // What happens when the POST succeeds?
    onSuccess: () => {
      // THE MAGIC: Tell the Cache that the 'users' list is now Stale!
      // React Query will instantly background-refresh any components 
      // currently looking at the 'users' list!
      queryClient.invalidateQueries({ queryKey: ['users'] });
    }
  });

  const handleSubmit = (e) => {
    e.preventDefault();
    mutate({ name: "Alice", role: "Admin" });
  };

  return (
    <form onSubmit={handleSubmit}>
      <button disabled={isPending}>
        {isPending ? 'Saving...' : 'Create User'}
      </button>
    </form>
  );
}
```

### Level 3 — StaleTime vs CacheTime
React Query is aggressive by default. The default `staleTime` is **0 milliseconds**.
This means the instant a query finishes, it is considered "stale". If the user switches tabs and comes back, React Query will instantly trigger a background re-fetch.

```jsx
const { data } = useQuery({
  queryKey: ['settings'],
  queryFn: fetchSettings,
  // Tell React Query: "This data is fresh for 5 minutes. 
  // Do NOT re-fetch it in the background until 5 minutes pass!"
  staleTime: 1000 * 60 * 5, 
});
```
*Note:* `gcTime` (formerly `cacheTime`) is different. It determines how long the data stays in memory *after the component unmounts* before being garbage collected.

### Level 4 — Optimistic Updates
When a user clicks a "Like" button, they expect the heart icon to turn red instantly. They don't want to wait 500ms for the API to respond.
**Optimistic Updates** manually update the cache *before* the API responds. If the API fails, it rolls the cache back to the original state.

```jsx
const { mutate } = useMutation({
  mutationFn: likePost,
  // Run this IMMEDIATELY when mutate() is called!
  onMutate: async (postId) => {
    // 1. Cancel any outgoing refetches so they don't overwrite our optimistic update
    await queryClient.cancelQueries({ queryKey: ['posts'] });

    // 2. Snapshot the previous value (in case we need to roll back)
    const previousPosts = queryClient.getQueryData(['posts']);

    // 3. Optimistically update the cache! The UI updates instantly!
    queryClient.setQueryData(['posts'], (old) => {
      return old.map(p => p.id === postId ? { ...p, likes: p.likes + 1 } : p);
    });

    // 4. Return a context object with the snapshotted value
    return { previousPosts };
  },
  // If the mutation fails, use the context returned from onMutate to roll back
  onError: (err, newTodo, context) => {
    queryClient.setQueryData(['posts'], context.previousPosts);
  },
  // Always refetch after error or success to ensure we are perfectly synced
  onSettled: () => {
    queryClient.invalidateQueries({ queryKey: ['posts'] });
  },
});
```

## REAL_WORLD
**The "State Management" Illusion**
Before React Query, developers dumped all their API data into Redux.
Once you adopt React Query, you realize that 90% of your Redux store was just cached API responses. 
When you move API fetching to React Query, your Redux store suddenly becomes almost empty. You only need global state for true "Client State" (like `isSidebarOpen` or `theme: dark`). Because Client State is so small, you can usually drop Redux entirely and just use Zustand or Context for the remaining 10%!

## INTERVIEW
**Q: You use `useQuery` to fetch a list of Users. In a different component on the same page, you need to display just the *count* of those users. If you call `useQuery({ queryKey: ['users'] })` again in the second component, does it trigger two network requests?**
A: **No.**
This is the core power of React Query's deduplication. If multiple components call `useQuery` with the exact same `queryKey` at the exact same time, React Query merges them into a single network request. When the API responds, it hands the data to both components simultaneously. 
You never need to fetch data in a parent and pass it down via props again. Any component that needs the data simply calls `useQuery` with the correct key, and the cache handles the rest!

## FEYNMAN CHECK

### Explain TanStack Query Like I'm 10 Years Old
> TanStack Query manages server state. `useQuery({ queryKey: ['user', id], queryFn: () => fetchUser(id) })` returns `{ data, isLoading, error }`. React Query caches by `queryKey`. 10 components requesting same key = 1 network call. Cache has `staleTime` — after expiry, background refetch silently updates. `useMutation` handles writes. It solves: loading states, deduplication, caching, background updates, optimistic updates, refetch on window focus — all declaratively, eliminating 90% of useEffect data fetching.

### 5 Deep Conceptual Questions

**Q1: How does queryKey determine caching?**
> **A:** queryKey is the cache key — a serialisable array. `['user', 1]` and `['user', 2]` are separate entries. Changing any key element causes a new fetch. Must contain ALL variables queryFn depends on (same as useEffect deps). Hierarchical: `queryClient.invalidateQueries({ queryKey: ['user'] })` invalidates ALL user queries.

**Q2: staleTime vs gcTime?**
> **A:** STALETIME: how long data is FRESH — no background refetch during this window. After: STALE — refetch runs in background on next access, user sees old data first. GCTIME (formerly cacheTime): how long UNUSED cache entries stay in memory. Default gcTime=5min. Pattern: `staleTime: 60_000` (1min fresh), `gcTime: 300_000` (5min in memory). Navigate away and back within 1min → no refetch.

**Q3: Most dangerous misconception?**
> **A:** Mutation automatically updates cache:
> ```tsx
> // ❌ Mutation fires but stale data stays — no cache update
> const { mutate } = useMutation({ mutationFn: updateUser });
> // List still shows old name until page refresh
>
> // ✅ Invalidate after success
> const qc = useQueryClient();
> const { mutate } = useMutation({
>   mutationFn: updateUser,
>   onSuccess: () => qc.invalidateQueries({ queryKey: ['users'] }),
> });
>
> // ✅ Optimistic update with rollback
> const { mutate } = useMutation({
>   mutationFn: updateUser,
>   onMutate: async (newUser) => {
>     await qc.cancelQueries({ queryKey: ['users'] });
>     const snapshot = qc.getQueryData<User[]>(['users']);
>     qc.setQueryData<User[]>(['users'], prev => prev!.map(u => u.id === newUser.id ? newUser : u));
>     return { snapshot };
>   },
>   onError: (_, __, ctx) => qc.setQueryData(['users'], ctx?.snapshot),
>   onSettled: () => qc.invalidateQueries({ queryKey: ['users'] }),
> });
> ```

**Q4: How do you implement infinite scroll?**
> **A:** `useInfiniteQuery` with `initialPageParam`, `getNextPageParam` (extracts next cursor from each page), `fetchNextPage()` triggered by IntersectionObserver at the bottom. React Query manages the pages array; navigating away and back restores all pages without refetch.

**Q5: FAANG-grade definition.**
> **A:** "TanStack Query: server-state library with declarative data fetching — queryKey as cache ID, staleTime controls refetch trigger, gcTime controls garbage collection, deduplication (single request per key), automatic refetch on window focus/reconnect/invalidation — useMutation with onMutate optimistic update + rollback pattern — eliminating useEffect boilerplate for loading/error/caching/sync — and extending to infinite scroll via useInfiniteQuery and suspense via useSuspenseQuery."

#### Step 2 — Setup + API
```tsx
// src/main.tsx
import { QueryClient, QueryClientProvider } from '@tanstack/react-query';
import { ReactQueryDevtools } from '@tanstack/react-query-devtools';
import { createRoot } from 'react-dom/client';
import App from './App';
const qc = new QueryClient({ defaultOptions: { queries: { staleTime: 30_000 } } });
createRoot(document.getElementById('root')!).render(
  <QueryClientProvider client={qc}><App /><ReactQueryDevtools /></QueryClientProvider>
);

// src/api.ts
export interface User { id: number; name: string; email: string; }
const BASE = 'https://jsonplaceholder.typicode.com';
export const getUsers = (): Promise<User[]> => fetch(BASE+'/users').then(r => r.json());
export const createUser = (d: Omit<User,'id'>): Promise<User> =>
  fetch(BASE+'/users', { method:'POST', body: JSON.stringify(d), headers: {'Content-Type':'application/json'} }).then(r => r.json());
export const deleteUser = (id: number) =>
  fetch(BASE+'/users/'+id, { method:'DELETE' }).then(() => {});
```

#### Step 3 — App
```tsx
// src/App.tsx
import { useState } from 'react';
import { useQuery, useMutation, useQueryClient } from '@tanstack/react-query';
import { getUsers, createUser, deleteUser, User } from './api';

const KEYS = { users: () => ['users'] as const };

export default function App() {
  const qc = useQueryClient();
  const [name, setName] = useState('');
  const [email, setEmail] = useState('');

  const { data: users, isLoading, isFetching } = useQuery({ queryKey: KEYS.users(), queryFn: getUsers });

  const addMutation = useMutation({
    mutationFn: createUser,
    onMutate: async (newUser) => {
      await qc.cancelQueries({ queryKey: KEYS.users() });
      const snap = qc.getQueryData<User[]>(KEYS.users());
      qc.setQueryData<User[]>(KEYS.users(), p => [...(p??[]), { ...newUser, id: -Date.now() }]);
      return { snap };
    },
    onError: (_, __, ctx) => qc.setQueryData(KEYS.users(), ctx?.snap),
    onSettled: () => qc.invalidateQueries({ queryKey: KEYS.users() }),
  });

  const delMutation = useMutation({
    mutationFn: deleteUser,
    onMutate: async (id) => {
      await qc.cancelQueries({ queryKey: KEYS.users() });
      const snap = qc.getQueryData<User[]>(KEYS.users());
      qc.setQueryData<User[]>(KEYS.users(), p => p?.filter(u => u.id !== id));
      return { snap };
    },
    onError: (_, __, ctx) => qc.setQueryData(KEYS.users(), ctx?.snap),
    onSettled: () => qc.invalidateQueries({ queryKey: KEYS.users() }),
  });

  if (isLoading) return <div className="p-4">Loading...</div>;

  return (
    <main className="max-w-xl mx-auto p-8">
      <h1 className="text-2xl font-bold mb-2">Users {isFetching && <span className="text-xs text-gray-400">(syncing...)</span>}</h1>
      <div className="flex gap-2 mb-4">
        <input value={name} onChange={e => setName(e.target.value)} placeholder="Name" className="flex-1 border px-2 py-1 rounded" />
        <input value={email} onChange={e => setEmail(e.target.value)} placeholder="Email" className="flex-1 border px-2 py-1 rounded" />
        <button onClick={() => { addMutation.mutate({ name, email }); setName(''); setEmail(''); }}
          disabled={addMutation.isPending} className="px-3 py-1 bg-blue-500 text-white rounded disabled:opacity-50">
          {addMutation.isPending ? '...' : 'Add'}
        </button>
      </div>
      <ul className="border rounded">
        {users?.map(u => (
          <li key={u.id} className={`flex justify-between p-3 border-b ${u.id < 0 ? 'opacity-50' : ''}`}>
            <div><p className="font-medium">{u.name}</p><p className="text-sm text-gray-500">{u.email}</p></div>
            <button onClick={() => delMutation.mutate(u.id)} className="text-red-500">Delete</button>
          </li>
        ))}
      </ul>
    </main>
  );
}
```
