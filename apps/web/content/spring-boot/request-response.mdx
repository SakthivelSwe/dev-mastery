## WHY
In Spring Boot, the HTTP request-response cycle is the fundamental unit of work for every REST API. A request arrives at your controller, your method processes it, and you must return a proper HTTP response with the correct status code, headers, and body.

Understanding how Spring Boot maps the incoming HTTP request to your method parameters, and how it serializes your return value into a JSON response body, is the cornerstone of building a production-quality REST API.

## THEORY
### The Journey of a Request
1. An HTTP request hits the server.
2. **DispatcherServlet** (Spring's front controller) receives it and consults the **HandlerMapping** to find which Controller method should handle it.
3. The request is routed to the matched Controller method.
4. The **Jackson** library serializes the return value into a JSON string.
5. The JSON string is written into the HTTP Response body and sent back to the client.

## CODE

### Level 1 — Beginner (Extracting Data from the Request)
Spring provides dedicated annotations to extract different parts of an incoming HTTP request into your method parameters.

```java
@RestController
@RequestMapping("/api/users")
public class UserController {

    // 1. @PathVariable: Extracts from the URL path segment
    // GET /api/users/42
    @GetMapping("/{id}")
    public String getUserById(@PathVariable Long id) {
        return "User ID: " + id;
    }

    // 2. @RequestParam: Extracts from the URL query string (?key=value)
    // GET /api/users?name=Alice&page=1
    @GetMapping
    public String searchUsers(
            @RequestParam String name,
            @RequestParam(defaultValue = "0") int page) {
        return "Searching for: " + name + " page: " + page;
    }

    // 3. @RequestHeader: Extracts from the HTTP Header
    @GetMapping("/whoami")
    public String whoAmI(@RequestHeader("Authorization") String token) {
        return "Token: " + token;
    }

    // 4. @RequestBody: Deserializes the JSON body into a Java Object
    @PostMapping
    public String createUser(@RequestBody UserRequest request) {
        return "Created user: " + request.name();
    }
}

record UserRequest(String name, String email) {}
```

### Level 2 — Building Blocks (ResponseEntity)
`@ResponseBody` (included in `@RestController`) is great for simple success responses. But what if you need to set a custom HTTP status code (like `201 Created` after a `POST`), or add custom headers (like a `Location` header pointing to the new resource)?

Use `ResponseEntity<T>`.

```java
@PostMapping
public ResponseEntity<UserResponse> createUser(@RequestBody UserRequest request) {
    UserResponse saved = userService.create(request);

    // Build the Location header pointing to the newly created resource
    URI location = URI.create("/api/users/" + saved.id());

    return ResponseEntity
            .created(location) // Sets status to 201, and adds Location header!
            .body(saved);
}

@GetMapping("/{id}")
public ResponseEntity<UserResponse> getUser(@PathVariable Long id) {
    return userService.findById(id)
            .map(ResponseEntity::ok)                      // 200 OK if found
            .orElse(ResponseEntity.notFound().build());   // 404 if not found
}
```

### Level 3 — Intermediate (Validation)
Before processing a request body, you should validate it. Spring Boot integrates with the Java Bean Validation API (`jakarta.validation`). Add `@Valid` to the `@RequestBody` annotation to trigger validation.

```java
import jakarta.validation.constraints.*;

public record UserRequest(
        @NotBlank(message = "Name is required")
        String name,

        @Email(message = "Must be a valid email")
        @NotBlank
        String email,

        @Min(value = 18, message = "Must be at least 18 years old")
        int age
) {}

@PostMapping
public ResponseEntity<UserResponse> createUser(@Valid @RequestBody UserRequest request) {
    // If validation fails, Spring automatically returns a 400 Bad Request
    // before your method body even executes!
    UserResponse saved = userService.create(request);
    return ResponseEntity.ok(saved);
}
```

### Level 4 — Advanced (Content Negotiation via @RequestMapping produces/consumes)
You can restrict a Controller method to only accept or produce specific content types using `consumes` and `produces`.

```java
@PostMapping(
    path = "/upload",
    // This method ONLY fires if the incoming request has 'Content-Type: application/json'
    consumes = MediaType.APPLICATION_JSON_VALUE,
    // This method ONLY fires if the client includes 'Accept: application/json' header
    produces = MediaType.APPLICATION_JSON_VALUE
)
public ResponseEntity<ApiResponse> handleJsonUpload(@RequestBody DataPayload payload) {
    return ResponseEntity.ok(new ApiResponse("Processed JSON!"));
}
```

## REAL_WORLD
In Enterprise REST APIs, every response body should be wrapped in a consistent envelope structure. Instead of returning a raw `User` JSON object, you return a `ApiResponse<User>` wrapper that includes metadata like `status`, `message`, and `data`. This makes client-side error handling drastically easier.

```java
public record ApiResponse<T>(
        String status,  // "SUCCESS" or "ERROR"
        String message,
        T data
) {}
```

## INTERVIEW
**Q: What is the difference between `@Controller` and `@RestController`?**
A: `@RestController` is a convenience annotation that combines `@Controller` and `@ResponseBody`. `@ResponseBody` tells Spring to serialize the return value of every method directly into the HTTP response body. Without it, Spring assumes you are returning a View Name (like `"home.html"`) for a traditional MVC web application.

**Q: What happens if `@RequestParam` is missing from the URL?**
A: By default, `@RequestParam` is mandatory. If the parameter is missing, Spring returns a `400 Bad Request`. You can make it optional by adding `required = false` or providing a `defaultValue`.
