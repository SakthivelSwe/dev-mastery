## WHY
When you build an application on your laptop, it usually connects to a local, empty database (`localhost:5432`) and sends fake emails. When you deploy that exact same code to Production, it must connect to a secure AWS RDS database and send real emails to paying customers.

You cannot rewrite your code or manually change your `application.yml` file every time you deploy. You need a way to tell the Spring ApplicationContext: *"If I am running in Development, instantiate THESE Beans and use THESE properties. But if I am running in Production, instantiate THOSE Beans and use THOSE properties."* This is solved by **Spring Profiles**.

## THEORY
### What is a Profile?
A Profile is simply a named logical grouping of Beans and Configuration properties. You can name a profile anything you want (e.g., `dev`, `test`, `staging`, `prod`, `aws`, `on-prem`). 

When Spring boots up, you specify which profile(s) are currently "Active". Spring then filters out any Beans or properties that do not belong to the active profile.

### Two Modes of Profiling
1. **Property Profiles**: Creating separate `application-{profile}.yml` files to hold environment-specific variables.
2. **Bean Profiles (`@Profile`)**: Annotating specific `@Component` or `@Bean` classes so they are only created if a specific profile is active.

## CODE

### Level 1 — Beginner (Property Profiles)
Instead of having one massive `application.yml`, you create multiple files.

**1. `application.yml` (The Default/Base file):**
```yaml
# This file is ALWAYS loaded first, regardless of the active profile.
# Put common configuration here.
spring:
  application:
    name: "payment-service"
```

**2. `application-dev.yml` (The 'dev' profile):**
```yaml
# Only loaded if the 'dev' profile is active!
spring:
  datasource:
    url: "jdbc:postgresql://localhost:5432/local_db"
payment:
  gateway: "https://sandbox.stripe.com"
```

**3. `application-prod.yml` (The 'prod' profile):**
```yaml
# Only loaded if the 'prod' profile is active!
spring:
  datasource:
    url: "jdbc:postgresql://rds.aws.internal:5432/prod_db"
payment:
  gateway: "https://api.stripe.com"
```

### Level 2 — Building Blocks (Activating a Profile)
How do you tell Spring which profile to use? You set the `spring.profiles.active` property!

**Option A: Command Line Argument (Production Standard)**
```bash
java -jar my-app.jar --spring.profiles.active=prod
```

**Option B: Environment Variable (Docker/K8s Standard)**
```bash
export SPRING_PROFILES_ACTIVE=staging
java -jar my-app.jar
```

**Option C: Hardcoding in `application.yml` (Local Development)**
```yaml
spring:
  profiles:
    active: dev
```

### Level 3 — Intermediate (The @Profile Annotation)
Sometimes, changing properties isn't enough. You need to completely swap out entire Java implementations.

```java
public interface EmailSender {
    void send(String to, String msg);
}

// ----------------------------------------------------

import org.springframework.context.annotation.Profile;
import org.springframework.stereotype.Service;

// This Bean is ONLY created if the 'dev' OR 'test' profile is active!
@Service
@Profile({"dev", "test"})
public class MockEmailSender implements EmailSender {
    public void send(String to, String msg) {
        System.out.println("FAKE EMAIL sent to " + to + ": " + msg);
    }
}

// ----------------------------------------------------

// This Bean is ONLY created if the 'prod' profile is active!
@Service
@Profile("prod")
public class AwsSesEmailSender implements EmailSender {
    public void send(String to, String msg) {
        // Connect to AWS Simple Email Service and bill the company!
        amazonSesClient.sendEmail(to, msg); 
    }
}
```

### Level 4 — Advanced (Profile Expressions & Not Profiles)
The `@Profile` annotation supports logical expressions (AND, OR, NOT).

```java
import org.springframework.context.annotation.Profile;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class MetricsConfig {

    // Activate this Bean ONLY if 'prod' is active AND 'aws' is active.
    @Bean
    @Profile("prod & aws")
    public MetricsPublisher cloudWatchPublisher() { ... }

    // Activate this Bean in EVERY environment EXCEPT 'test'.
    // Very useful to disable heavy integrations during JUnit testing!
    @Bean
    @Profile("!test")
    public ExpensiveCache preWarmer() { ... }
}
```

## REAL_WORLD
A highly advanced, modern Spring Boot 2.4+ configuration pattern is to use **Multi-Document YAML files** instead of creating 5 different files.

You can put all your environments into one single `application.yml` file, separated by three dashes `---`.

```yaml
# src/main/resources/application.yml

# --- Base Config (Always Active) ---
spring:
  application:
    name: "user-service"

---
# --- Dev Profile Section ---
spring:
  config:
    activate:
      on-profile: dev
server:
  port: 8080

---
# --- Prod Profile Section ---
spring:
  config:
    activate:
      on-profile: prod
server:
  port: 80
```

## INTERVIEW
**Q: What happens if you don't explicitly set an active profile?**
A: Spring will automatically activate a fallback profile named `"default"`. If you have a file named `application-default.yml` (or Beans annotated with `@Profile("default")`), they will be loaded.

**Q: Can you have multiple profiles active at the same time?**
A: Yes! It is extremely common. You might run: `SPRING_PROFILES_ACTIVE=prod,aws,db-migration`. Spring will merge the properties from `application-prod.yml`, `application-aws.yml`, and `application-db-migration.yml`.

**Q: In what order does Spring resolve properties if multiple profiles are active?**
A: Spring uses a strict "Last One Wins" hierarchy. If `SPRING_PROFILES_ACTIVE=dev,prod`, the properties in the `prod` profile will overwrite the exact same properties defined in the `dev` profile. Furthermore, Command Line arguments (`--server.port=9090`) will always overwrite *everything* defined in any YAML file.
