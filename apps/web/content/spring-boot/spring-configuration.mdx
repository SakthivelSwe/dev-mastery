## WHY
An application is only useful if it can adapt to different environments without requiring a code recompilation. Hardcoding database passwords, API keys, or server port numbers directly into your Java classes is a massive security risk and an architectural failure.

Spring solves this by separating **Configuration Data** (the variables that change) from **Configuration Code** (the blueprints that build the Beans). Understanding how to inject properties from `application.yml` files into your Java classes using `@Value` and `@ConfigurationProperties` is a fundamental requirement for building 12-Factor Apps.

## THEORY
### Application Properties
Spring Boot automatically looks for a file named `application.properties` (or `application.yml`) in your `src/main/resources` folder. When the application boots, Spring reads this entire file into memory as an `Environment` object.

### Injecting Properties
Once the data is loaded, Spring provides two primary ways to inject that data into your Beans:
1. **`@Value`**: Quick, dirty, and easy. Used for injecting a single variable into a single field.
2. **`@ConfigurationProperties`**: Robust, type-safe, and object-oriented. Used for mapping entire blocks of YAML into structured Java Objects.

## CODE

### Level 1 — Beginner (The application.yml file)
```yaml
# src/main/resources/application.yml
server:
  port: 8080

payment:
  gateway:
    url: "https://api.stripe.com/v1"
    timeout-ms: 5000
    enabled: true
```

### Level 2 — Building Blocks (The @Value Annotation)
```java
import org.springframework.beans.factory.annotation.Value;
import org.springframework.stereotype.Service;

@Service
public class PaymentClient {

    // We use the Spring Expression Language (SpEL) syntax: ${property.path}
    // Spring extracts the value from the YAML and injects it into this field!
    
    @Value("${payment.gateway.url}")
    private String apiUrl;

    @Value("${payment.gateway.timeout-ms}")
    private int timeout;

    // You can also provide a default value just in case the property is missing!
    // If 'payment.gateway.enabled' isn't in the YAML, it defaults to 'false'.
    @Value("${payment.gateway.enabled:false}")
    private boolean isEnabled;

    public void process() {
        if (isEnabled) {
            System.out.println("Connecting to: " + apiUrl + " with timeout " + timeout);
        }
    }
}
```

### Level 3 — Intermediate (@ConfigurationProperties)
`@Value` becomes a nightmare if you have 20 different fields. Instead, we map the YAML tree directly to a Java Object.

```java
import org.springframework.boot.context.properties.ConfigurationProperties;
import org.springframework.stereotype.Component;

// 1. Tell Spring: "Map everything under the 'payment.gateway' prefix to this class!"
@Component
@ConfigurationProperties(prefix = "payment.gateway")
public class PaymentConfig {

    // 2. The variable names MUST exactly match the YAML keys!
    private String url;
    private int timeoutMs; // Maps to 'timeout-ms' in YAML automatically!
    private boolean enabled;

    // 3. Spring uses Standard Getters and Setters to inject the data!
    // (If you use Lombok, just slap @Data on the class).
    
    public String getUrl() { return url; }
    public void setUrl(String url) { this.url = url; }
    
    public int getTimeoutMs() { return timeoutMs; }
    public void setTimeoutMs(int timeoutMs) { this.timeoutMs = timeoutMs; }
    
    public boolean isEnabled() { return enabled; }
    public void setEnabled(boolean enabled) { this.enabled = enabled; }
}

// Now, in your services, you just inject the entire PaymentConfig object!
@Service
public class CheckoutService {
    private final PaymentConfig config;

    public CheckoutService(PaymentConfig config) {
        this.config = config;
    }
}
```

### Level 4 — Advanced (Record-based ConfigurationProperties)
In modern Java (Java 16+) and Spring Boot 3, you can use **Records** for perfectly immutable, constructor-bound configuration properties!

```java
import org.springframework.boot.context.properties.ConfigurationProperties;
import org.springframework.boot.context.properties.bind.DefaultValue;

// No @Component needed! No getters or setters!
// Just pure, immutable, thread-safe configuration.

@ConfigurationProperties(prefix = "payment.gateway")
public record PaymentProperties(
    String url,
    int timeoutMs,
    @DefaultValue("false") boolean enabled
) {}

// To use this, you must explicitly enable it on your main Application class:
// @ConfigurationPropertiesScan
// @SpringBootApplication
// public class Application { ... }
```

## REAL_WORLD
In an Enterprise environment, your `application.yml` file is often empty or missing entirely. 

Why? Because Spring Boot follows the **Environment Property Abstraction**. Spring doesn't just read `application.yml`; it reads System Environment Variables, JVM Command Line Arguments (`-D`), and OS-level properties.

When deploying to Kubernetes, infrastructure engineers will set `PAYMENT_GATEWAY_URL` as a Kubernetes Environment Variable. Spring Boot's "Relaxed Binding" engine will automatically detect that `PAYMENT_GATEWAY_URL` maps to `payment.gateway.url`, and flawlessly inject it into your `@ConfigurationProperties` without you changing a single line of Java code.

## INTERVIEW
**Q: Can you use `@Value` on a static field?**
A: No. `@Value` is processed during the Bean Instantiation lifecycle. Static variables belong to the Class itself, not the instantiated Object. Spring will silently ignore the `@Value` annotation on a static field, leaving it as `null`.

**Q: What is the difference between `@PropertySource` and `application.yml`?**
A: Spring Boot auto-loads `application.properties/yml` entirely by magic. If you have a custom properties file (like `legacy-database.properties`) sitting somewhere obscure, you must use `@PropertySource("classpath:legacy-database.properties")` on a `@Configuration` class to manually force Spring to read it. Note: `@PropertySource` does NOT support YAML files out-of-the-box, only `.properties`.

**Q: How do you validate Configuration Properties?**
A: You can apply standard Jakarta Bean Validation annotations directly to your config classes! Just put `@Validated` at the top of the class, and use `@NotBlank` or `@Min(100)` on your fields. If the `application.yml` contains invalid data, Spring will violently crash at startup, preventing a broken application from starting.
