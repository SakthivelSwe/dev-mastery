## WHY
The primary purpose of most modern Spring Boot applications is to act as a backend REST API for frontend applications (React, Angular, mobile apps). 

To do this, the application must be able to intercept incoming HTTP requests (like `GET /users/123`), parse the URL, extract the data, run business logic, and return a JSON response (like `{"name": "Alice"}`). 

The **Spring Web MVC** module handles this using the `@RestController` annotation. It completely hides the complex, low-level Java Servlet API, allowing you to map URLs directly to simple Java methods using highly expressive annotations.

## THEORY
### @Controller vs @RestController
Historically, Spring was used to build monolithic web applications that returned HTML pages. Developers used `@Controller`, and the methods returned the name of an HTML template file (like `"index.html"`).

Today, modern APIs do not return HTML; they return raw data, usually formatted as JSON. The `@RestController` annotation is a specialized version of `@Controller`. It automatically converts whatever Java object you return from your method directly into a JSON string and writes it straight into the HTTP Response body. 

*(Technically, `@RestController` is a meta-annotation that simply combines `@Controller` and `@ResponseBody` together).*

## CODE

### Level 1 — Beginner (The Basic GET Request)
```java
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RestController;
import java.util.List;

// 1. Mark this class as a Web API endpoint!
// Spring will register it as a Bean and hook it into the internal HTTP router.

@RestController
public class UserController {

    // 2. Map this specific method to an HTTP GET request at the "/api/users" URL!
    @GetMapping("/api/users")
    public List<String> getAllUsers() {
        
        // 3. Just return standard Java Objects!
        // Spring uses the Jackson library internally to automatically convert this 
        // Java List into a JSON array: ["Alice", "Bob", "Charlie"]
        return List.of("Alice", "Bob", "Charlie");
    }
}
```

### Level 2 — Building Blocks (Class-Level Routing)
Instead of typing `/api/users` on every single method, we can set a base URL for the entire class using `@RequestMapping`.

```java
import org.springframework.web.bind.annotation.*;
import java.util.List;

@RestController
// Base URL for ALL methods in this class!
@RequestMapping("/api/v1/products") 
public class ProductController {

    // Matches: GET /api/v1/products
    @GetMapping
    public List<String> getProducts() {
        return List.of("Laptop", "Keyboard");
    }

    // Matches: POST /api/v1/products
    @PostMapping
    public String createProduct() {
        return "Product created successfully!";
    }

    // Matches: DELETE /api/v1/products
    @DeleteMapping
    public String deleteAll() {
        return "All products deleted!";
    }
}
```

### Level 3 — Intermediate (Injecting Services)
A Controller should **never** contain business logic or database queries. Its only responsibility is to receive the HTTP request, hand the data off to a `@Service`, and return the result.

```java
import org.springframework.web.bind.annotation.*;

@RestController
@RequestMapping("/api/orders")
public class OrderController {

    // 1. Inject the Business Logic Service!
    private final OrderService orderService;

    public OrderController(OrderService orderService) {
        this.orderService = orderService;
    }

    @PostMapping
    public Order createOrder() {
        // 2. The Controller does NO heavy lifting. 
        // It delegates entirely to the Service!
        return orderService.processNewOrder();
    }
}
```

### Level 4 — Advanced (Content Negotiation)
What if the client requesting data doesn't want JSON? What if they want XML?

Spring Boot has built-in **Content Negotiation**. It looks at the HTTP `Accept` header sent by the client. If the client sends `Accept: application/xml`, Spring will automatically try to convert the returned Java Object into XML instead of JSON!

```java
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RequestMapping;
import org.springframework.web.bind.annotation.RestController;

@RestController
@RequestMapping("/api/reports")
public class ReportController {

    // You can explicitly restrict what Content-Types this endpoint supports!
    // If the client asks for XML, Spring will reject them with a 406 Not Acceptable error.
    
    @GetMapping(produces = "application/json")
    public ReportData getReport() {
        return new ReportData("Monthly Sales", 50000);
    }
}
```

## REAL_WORLD
In Enterprise environments, REST Controllers are heavily integrated with OpenAPI (Swagger) documentation. 

Instead of writing manual API documentation in a Wiki, developers use the `springdoc-openapi` library. They place annotations directly on the Controller methods, and Spring automatically generates a beautiful, interactive web page where frontend developers can test the APIs in real-time.

```java
import io.swagger.v3.oas.annotations.Operation;
import io.swagger.v3.oas.annotations.tags.Tag;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RestController;

@RestController
@Tag(name = "User Management", description = "Endpoints for managing users")
public class UserController {

    @GetMapping("/users")
    @Operation(summary = "Get all users", description = "Returns a paginated list of all active users.")
    public List<User> getUsers() {
        return List.of();
    }
}
```

## INTERVIEW
**Q: What is the underlying architecture that routes requests to a `@RestController`?**
A: Spring Web MVC uses the **Front Controller** design pattern. Deep inside Tomcat, there is a single massive Servlet called the `DispatcherServlet`. Every single HTTP request hits the `DispatcherServlet` first. The Dispatcher then looks at the URL, checks its internal map of `@RestController` methods (the HandlerMappings), and delegates the request to the correct Java method.

**Q: Can you return an HTML string from a `@RestController`?**
A: Yes, if your method returns `String`, Spring will return that literal string in the response body. If the string happens to be `"<h1>Hello</h1>"`, the browser will render it as HTML. However, if you want to return an HTML *template file* (like a Thymeleaf view), you cannot use `@RestController`. You must use `@Controller` so Spring knows to resolve the string as a filename, not raw data.

**Q: Are REST Controllers thread-safe?**
A: No! Just like `@Service` and `@Repository`, Controllers are Singletons by default. If 10,000 HTTP requests arrive simultaneously, 10,000 threads will execute the methods inside the exact same Controller instance. You must never use mutable instance variables inside a Controller.
