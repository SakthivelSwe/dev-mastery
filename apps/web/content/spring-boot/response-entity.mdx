## WHY
When building a REST API, you don't just return data (like a JSON object). You also need to return metadata about the response. This metadata primarily consists of HTTP Status Codes (e.g., 200 OK, 404 Not Found, 201 Created) and HTTP Headers.
If a Spring Boot controller simply returns a raw object `return new User();`, Spring automatically assumes it is a successful operation and defaults to sending a `200 OK` status code. 
However, what if the user requested an ID that doesn't exist? You cannot just return `null` with a `200 OK` status; you must return a `404 Not Found`!
**`ResponseEntity`** is a powerful wrapper class in Spring that represents the entire HTTP response. It gives you absolute, fine-grained control over the status code, the headers, and the body of the response sent back to the client.

## THEORY
### The Structure of ResponseEntity
`ResponseEntity` uses the Builder Pattern to construct the final HTTP response.
It acts as a generic envelope: `ResponseEntity<T>`, where `T` is the type of the body you are sending back (e.g., `ResponseEntity<User>`, `ResponseEntity<String>`, or `ResponseEntity<Void>` if there is no body).

### Common HTTP Status Codes
- **200 OK:** The standard success code. (Default if not specified).
- **201 Created:** A new resource was successfully created in the database (usually in response to a `POST` request).
- **204 No Content:** The operation succeeded, but there is no data to send back (commonly used for `DELETE` requests).
- **400 Bad Request:** The client sent invalid data (e.g., missing a required field).
- **404 Not Found:** The requested resource does not exist.

## CODE

### 1. Basic Usage (Returning 200 OK)
While returning raw objects is allowed, wrapping them in `ResponseEntity.ok()` is the best practice for consistency.

```java
@RestController
@RequestMapping("/api/products")
public class ProductController {

    // BAD: Implicit 200 OK. Lacks fine-grained control.
    @GetMapping("/bad/{id}")
    public Product getProductBad(@PathVariable Long id) {
        return new Product(id, "Laptop");
    }

    // GOOD: Explicitly defines the Response Entity!
    @GetMapping("/good/{id}")
    public ResponseEntity<Product> getProductGood(@PathVariable Long id) {
        Product product = new Product(id, "Laptop");
        
        // .ok() is a static builder method that sets the status to 200 OK!
        return ResponseEntity.ok(product);
    }
}
```

### 2. Returning Custom Status Codes (e.g., 201 Created)
When handling a `POST` request, you should always return `201 Created` instead of `200 OK`.

```java
@PostMapping

    // Imagine we save the product to the database here...
```

### 3. Handling Errors (404 Not Found)
If you try to find a user in the database and they don't exist, you must change the status code dynamically!

```java
@GetMapping("/{id}")
public ResponseEntity<User> getUserById(@PathVariable Long id) {
    Optional<User> optionalUser = userRepository.findById(id);

    if (optionalUser.isPresent()) {
        // User exists! Return 200 OK with the User JSON.
        return ResponseEntity.ok(optionalUser.get());
    } else {
        // User does not exist! Return 404 Not Found with NO body.
        return ResponseEntity.notFound().build();
    }
}
```

### 4. Adding Custom HTTP Headers
Sometimes the client needs extra metadata in the headers, completely separate from the JSON body.

```java
@GetMapping("/download/{fileId}")
public ResponseEntity<String> downloadFile(@PathVariable String fileId) {
    String fileData = "This is the secure file content.";

    return ResponseEntity.ok()
            // .header() allows you to inject custom HTTP headers!
            .header("X-Custom-Header", "Spring-Boot-Is-Awesome")
            .header("Content-Disposition", "attachment; filename=secret.txt")
            .body(fileData);
}
```

## REAL_WORLD
**Location Headers for Created Resources**
In perfectly strict REST API design, when you create a new resource (`POST`), you should return a `201 Created` status code, AND you should include a `Location` HTTP header. 
The `Location` header tells the client the exact URL where they can find the newly created resource!

```java
@PostMapping
public ResponseEntity<Void> createOrder(@RequestBody Order order) {
    Order savedOrder = orderService.save(order);
    
    // Dynamically build the URL to the new resource (e.g., /api/orders/42)
    URI location = ServletUriComponentsBuilder
                    .fromCurrentRequest()
                    .path("/{id}")
                    .buildAndExpand(savedOrder.getId())
                    .toUri();

    // .created() automatically sets the 201 status AND the Location header!
    return ResponseEntity.created(location).build();
}
```

## INTERVIEW
**Q: You are writing a Controller method. You want to handle the case where a resource is not found. You can write an `if` statement and return `ResponseEntity.notFound().build();`. However, an interviewer says: "Writing `if/else` checks for 404s in every single Controller method causes massive code duplication. How can we completely eliminate `ResponseEntity` from our Controllers but still return custom HTTP Status Codes when errors occur?"**
A: **Global Exception Handling (`@ControllerAdvice`).**
While `ResponseEntity` is incredibly powerful, cluttering business logic with `if/else` status code checks is an anti-pattern in massive Spring applications.
Instead of returning a `ResponseEntity`, your Controller should just blindly trust the Service layer and return the raw Object. 
If the Service layer cannot find the user, it should simply `throw new ResourceNotFoundException("User not found");`.
You then create a separate class annotated with `@ControllerAdvice` and `@ExceptionHandler(ResourceNotFoundException.class)`. This Global Exception Handler will automatically catch the thrown exception anywhere in the entire application, construct a beautiful error JSON payload, wrap it in a `ResponseEntity.status(HttpStatus.NOT_FOUND)`, and send it to the client! This keeps Controllers incredibly clean and delegates status code management to a centralized error handler!
