## WHY
In the early 2000s, building Enterprise Java applications was a nightmare. Developers had to use Java EE (Enterprise Edition), which required massive, complex, and heavy "Application Servers" to run even the simplest code. The configuration was endless, and the code was tightly coupled to the server infrastructure.

In 2003, Rod Johnson released the **Spring Framework**. It revolutionized Java development by introducing a lightweight, POJO-based (Plain Old Java Object) architecture. Instead of forcing developers to inherit massive framework classes, Spring simply took over the responsibility of creating, managing, and destroying objects for you. This concept, known as **Inversion of Control (IoC)**, fundamentally changed how modern backend applications are architected.

## THEORY
### The Spring ApplicationContext
At the heart of the Spring Framework is the **IoC Container**, technically known as the `ApplicationContext`. 

Think of the `ApplicationContext` as a massive, invisible factory running in the background of your application. When your application starts, the factory wakes up, looks at your configuration, and physically creates all the Java Objects (called **Beans**) your application needs to function. It wires them all together automatically, and then hands them to you when you ask for them.

### Why is this better?
- **Decoupling**: Classes no longer need to use the `new` keyword to create their dependencies. They just ask Spring for them.
- **Testing**: Because dependencies are injected, it is trivially easy to swap a real Database object for a fake "Mock" object during unit testing.
- **Lifecycle Management**: Spring handles all the complex setup and teardown of network connections, thread pools, and transactions automatically.

## CODE

### Level 1 — Beginner (The Problem: Tightly Coupled Code)
```java
// BEFORE SPRING: The Tightly Coupled Nightmare

class EmailService {
    public void sendEmail(String msg) {
        System.out.println("Sending: " + msg);
    }
}

class NotificationSystem {
    // 1. The Class is tightly coupled to the specific implementation!
    private EmailService emailService;

    public NotificationSystem() {
        // 2. We are manually managing the lifecycle using 'new'.
        // If EmailService suddenly requires a database connection in its constructor,
        // we have to rewrite THIS file and 50 other files that use 'new EmailService()'!
        this.emailService = new EmailService();
    }

    public void alertUser(String msg) {
        emailService.sendEmail(msg);
    }
}
```

### Level 2 — Building Blocks (Inversion of Control)
```java
// WITH SPRING: Inversion of Control (IoC)

class EmailService {
    public void sendEmail(String msg) {
        System.out.println("Sending: " + msg);
    }
}

class NotificationSystem {
    private EmailService emailService;

    // 1. We completely removed the 'new' keyword!
    // 2. We force the outside world (Spring) to pass the dependency into us.
    // This is called "Constructor Injection".
    public NotificationSystem(EmailService emailService) {
        this.emailService = emailService;
    }

    public void alertUser(String msg) {
        emailService.sendEmail(msg);
    }
}
```

### Level 3 — Intermediate (The Spring Container Configuration)
```java
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

// We must tell the Spring Factory how to build our objects!
// A class annotated with @Configuration is a blueprint for the Spring IoC Container.

@Configuration
public class AppConfig {

    // The @Bean annotation tells Spring: "Execute this method, take the 
    // returned object, and store it permanently inside the ApplicationContext."
    
    @Bean
    public EmailService emailService() {
        return new EmailService(); // Spring creates this once and holds it!
    }

    @Bean
    public NotificationSystem notificationSystem() {
        // Spring says: "NotificationSystem needs an EmailService! Let me grab 
        // the one I just created above and inject it automatically!"
        return new NotificationSystem(emailService());
    }
}
```

### Level 4 — Advanced (Bootstrapping the Context)
```java
import org.springframework.context.ApplicationContext;
import org.springframework.context.annotation.AnnotationConfigApplicationContext;

public class MainApplication {
    public static void main(String[] args) {
        
        // 1. Boot up the Spring IoC Container!
        // We pass in our AppConfig blueprint so it knows what to build.
        ApplicationContext context = new AnnotationConfigApplicationContext(AppConfig.class);

        // 2. The container is now running. All Beans have been constructed and wired together!
        
        // 3. We simply ASK the container for the fully-built NotificationSystem!
        NotificationSystem system = context.getBean(NotificationSystem.class);
        
        // 4. It works perfectly, and we never had to write 'new NotificationSystem()' ourselves!
        system.alertUser("Server is going down in 5 minutes!");
    }
}
```

## REAL_WORLD
In modern Spring Boot applications, you will almost *never* see the `new AnnotationConfigApplicationContext()` code written in Level 4. Spring Boot hides the entire startup sequence inside a single method: `SpringApplication.run()`. 

Furthermore, you will rarely write manual `@Bean` methods for every single class. Spring uses a feature called **Component Scanning** (which we cover in a later topic) to automatically search your entire project, find your classes, and build them into Beans dynamically without any manual configuration at all.

However, understanding that the `ApplicationContext` is a massive `Map<String, Object>` sitting in memory holding all your singletons is the prerequisite to debugging complex application crashes.

## INTERVIEW
**Q: What is the difference between IoC (Inversion of Control) and DI (Dependency Injection)?**
A: **IoC** is a broad architectural principle. It simply means surrendering control of an aspect of your program (like object creation) to an external framework. **Dependency Injection (DI)** is the specific *implementation* or *pattern* used to achieve IoC. Spring achieves IoC *by* injecting dependencies into constructors or fields.

**Q: Does Spring use the Factory Pattern?**
A: Yes. The `ApplicationContext` (and its parent interface, the `BeanFactory`) are literal implementations of the Gang of Four **Factory Design Pattern**. Instead of the developer acting as the factory floor worker piecing objects together, the Spring framework handles the instantiation logic.

**Q: Can you run Spring without Spring Boot?**
A: Absolutely. Spring Core (the IoC container) is a standalone library. Spring Boot is simply an opinionated wrapper built *on top* of Spring Core that automatically configures Web Servers (like Tomcat) and database connections so you don't have to write hundreds of lines of XML or Java configuration manually.
