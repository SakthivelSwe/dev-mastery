## WHY
Never trust client input.
If you build a registration API, a malicious (or confused) user might send a `POST` request with an empty string for their password, a negative number for their age, or "not-an-email" for their email address.
If you don't validate this data, it will crash your database or corrupt your application's logic.
You *could* write massive blocks of `if/else` statements in your Controller: `if (user.getAge() < 0) { return error; }`. But this is tedious, ugly, and hard to maintain.
**Spring Boot Validation (Jakarta Bean Validation)** solves this elegantly. It allows you to place simple Annotations (like `@NotNull` or `@Email`) directly onto your Java Classes. Spring intercepts the incoming HTTP JSON payload, automatically runs all the validation rules, and instantly rejects the request with a `400 Bad Request` if any rules are broken, before the code ever reaches your database!

## THEORY
### Jakarta Bean Validation
Spring Boot doesn't actually write the validation logic itself; it uses the standard **Jakarta Bean Validation API** (formerly Java EE), usually implemented by the **Hibernate Validator** engine under the hood.

### The Two-Step Process
To make validation work, you must do two things:
1. **Define the Rules:** Place validation annotations (e.g., `@NotBlank`, `@Min`) on the fields of your DTO (Data Transfer Object).
2. **Trigger the Rules:** Place the `@Valid` annotation on the `@RequestBody` parameter in your Controller. If you forget `@Valid`, Spring will completely ignore the rules!

## CODE

To use validation, you must ensure the dependency is in your `pom.xml` (it is not included by default in `spring-boot-starter-web` anymore!).
```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-validation</artifactId>
</dependency>
```

### 1. Defining the Rules (The DTO)
We define a plain Java class that represents the expected JSON payload, and we decorate it with rules!

```java
import jakarta.validation.constraints.*;

public class UserRegistrationDto {

    // Cannot be null, and cannot be an empty string ("") or just spaces ("   ")
    @NotBlank(message = "Username is absolutely required!")
    @Size(min = 3, max = 20, message = "Username must be between 3 and 20 characters.")
    private String username;

    // Automatically checks if the string contains an '@' and a valid domain format!
    @NotBlank(message = "Email is required")
    @Email(message = "Please provide a valid email address.")
    private String email;

    @NotNull(message = "Age cannot be null")
    @Min(value = 18, message = "You must be at least 18 years old to register.")
    @Max(value = 120, message = "Age must be realistic.")
    private Integer age;

    // Getters and Setters omitted...
}
```

### 2. Triggering the Rules (The Controller)
We use the `@Valid` annotation to tell Spring: *"Hey, before you execute this method, check the JSON against the rules!"*

```java
@RestController
@RequestMapping("/api/users")
public class UserController {

    @PostMapping("/register")
    public ResponseEntity<String> registerUser(
        // CRITICAL: The @Valid annotation triggers the Hibernate Validator!
        @Valid @RequestBody UserRegistrationDto dto
    ) {
        
        // If the code reaches this line, it is MATHEMATICALLY GUARANTEED 
        // that the username is not blank, the email is valid, and the age >= 18!
        // We do not need a single if/else statement!
        
        System.out.println("Registering user: " + dto.getUsername());
        return ResponseEntity.ok("Registration successful!");
    }
}
```

## REAL_WORLD
**Custom Validation Annotations**
Sometimes the built-in annotations (`@Email`, `@Min`, `@Size`) aren't enough. 
Imagine you are building a banking app, and you need to validate that an employee ID strictly starts with the letters "EMP-" followed by exactly 4 digits.
You can create your very own custom validation annotations!
1. You create a custom Java interface `@interface ValidEmployeeId`.
2. You create a Validator class `EmployeeIdValidator implements ConstraintValidator<ValidEmployeeId, String>` containing an `isValid` method where you write custom Regex logic.
3. You can now use your custom `@ValidEmployeeId` annotation on any field across your entire enterprise application, ensuring perfectly reusable, DRY (Don't Repeat Yourself) business logic!

## INTERVIEW
**Q: You annotate your DTO with `@NotBlank`, and your Controller parameter with `@Valid`. The frontend sends an invalid JSON payload with an age of 12. Spring automatically blocks the request and throws a `MethodArgumentNotValidException`. However, the user's browser receives a massive, ugly, 500-line Java stack trace. How do you intercept this exception and send a clean JSON object containing exactly which fields failed validation?**
A: **By using an `@ExceptionHandler` inside an `@ControllerAdvice` class.**
When validation fails, Spring throws an exception before the Controller method even executes. To intercept this, you must set up Global Exception Handling.
You create a class annotated with `@ControllerAdvice` and define a method annotated with `@ExceptionHandler(MethodArgumentNotValidException.class)`. 
Inside this method, you can catch the exception, extract the `BindingResult` (which contains a list of every single field that failed and its associated error message), map those errors into a clean Java `Map<String, String>`, and return that Map wrapped in a `ResponseEntity.badRequest()`. This translates the ugly Java stack trace into a beautiful, machine-readable JSON error object for the frontend!
