## WHY
In a standard Java application, you create objects using the `new` keyword. Once you create them, it is your responsibility to figure out where to store them and how to pass them around memory. In a massive enterprise backend, manually managing 5,000 different objects and their memory references is mathematically impossible.

To solve this, Spring introduces the concept of the **Bean**. A "Bean" is not a special type of Java class; it is literally just a Plain Old Java Object (POJO) that you have surrendered to the Spring IoC Container. Once an object is designated as a Bean, Spring will automatically instantiate it, store it securely in the `ApplicationContext`, and track its entire lifecycle from creation to destruction.

## THEORY
### How to define a Bean
There are two primary ways to tell Spring to turn an object into a Bean:
1. **The Method-Level Approach (`@Bean`)**: You write a method inside a `@Configuration` class that returns an object. Spring calls the method, takes the object, and stores it. You use this approach when you are instantiating classes from third-party libraries (where you can't edit their source code).
2. **The Class-Level Approach (`@Component`)**: You place an annotation directly on top of your own class definition. Spring scans your project, finds the class, and automatically instantiates it for you. This is the industry standard for 95% of your code.

### Stereotype Annotations
Spring provides specialized versions of `@Component` called "Stereotypes". They all do the exact same thing (register a Bean), but they signal to other developers *what* the Bean actually does:
- `@Component`: A generic, plain Bean.
- `@Service`: A Bean that holds complex business logic.
- `@Repository`: A Bean that interacts directly with a database.
- `@Controller`: A Bean that intercepts HTTP REST API requests.

## CODE

### Level 1 — Beginner (The Method-Level @Bean)
```java
// Imagine we imported this class from an external AWS SDK Library.
// We CANNOT put @Component on it because we don't own the source code!
// class AmazonS3Client { ... }

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class CloudConfig {

    // Because we can't edit the Amazon class, we manually instantiate it here.
    // The @Bean annotation tells Spring to intercept the return value!
    
    @Bean
    public AmazonS3Client s3Client() {
        AmazonS3Client client = new AmazonS3Client();
        client.setRegion("us-east-1");
        // Spring takes this fully configured object and registers it as a Bean!
        return client; 
    }
}
```

### Level 2 — Building Blocks (The Class-Level @Component)
```java
import org.springframework.stereotype.Component;

// For our OWN code, we don't need to write manual @Bean methods.
// We just slap @Component on top of the class!

@Component
public class PaymentValidator {
    
    public boolean isValid(String creditCard) {
        return creditCard.length() == 16;
    }
}

// When Spring boots up, it sees @Component. It immediately runs:
// PaymentValidator bean = new PaymentValidator();
// And stores it in the ApplicationContext!
```

### Level 3 — Intermediate (Stereotype Annotations)
```java
import org.springframework.stereotype.Service;
import org.springframework.stereotype.Repository;

// Under the hood, @Service and @Repository are literally just @Component.
// However, using them creates self-documenting code. 

@Repository
public class UserRepository {
    public void saveToDatabase() {
        System.out.println("Executing SQL INSERT...");
    }
}

@Service
public class UserService {
    private final UserRepository repository;

    // Spring injects the @Repository bean into the @Service bean!
    public UserService(UserRepository repository) {
        this.repository = repository;
    }

    public void registerUser() {
        System.out.println("Running Business Logic...");
        repository.saveToDatabase();
    }
}
```

### Level 4 — Advanced (Naming and Overriding Beans)
```java
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class DatabaseConfig {

    // By default, the name of the Bean in the IoC container is the name of the method 
    // (or the class name with a lowercase first letter).
    
    @Bean(name = "primaryDatabaseConnection")
    public DatabaseConnection primaryDb() {
        return new DatabaseConnection("jdbc:mysql://primary-db");
    }

    @Bean(name = "readOnlyDatabaseConnection")
    public DatabaseConnection secondaryDb() {
        return new DatabaseConnection("jdbc:mysql://replica-db");
    }
    
    // WARNING: If two @Beans have the exact same name, Spring will crash at startup 
    // to prevent memory conflicts, unless 'spring.main.allow-bean-definition-overriding=true' 
    // is explicitly set in your properties file!
}
```

## REAL_WORLD
When you look at a massive enterprise Spring Boot application, you will almost never see the `new` keyword used to instantiate business services, controllers, or database clients. Every single architectural layer is defined using Stereotype Annotations (`@Controller` → `@Service` → `@Repository`).

The only time you will ever use the `new` keyword in a Spring application is when you are instantiating basic Data Transfer Objects (DTOs) or Domain Entities (like `new User("Alice")`). Data models are **not** Beans, because they represent transient data that flows through the system and dies, not permanent infrastructural services.

## INTERVIEW
**Q: Can you put `@Bean` directly on a class definition?**
A: No. `@Bean` is strictly a *Method-Level* annotation. It must be placed on a method inside a `@Configuration` class. If you want to put an annotation on a class definition, you must use `@Component` (or a stereotype).

**Q: If I create an interface `PaymentProcessor` and two implementation classes `StripeProcessor` and `PaypalProcessor`, and mark both with `@Component`, what happens?**
A: The application will crash at startup with a `NoUniqueBeanDefinitionException`. When a service asks Spring for a `PaymentProcessor`, Spring looks in the container, finds two Beans that match the interface, panics, and crashes because it doesn't know which one you want. You must resolve this using the `@Primary` annotation on one of the implementations, or use the `@Qualifier("stripeProcessor")` annotation at the injection point to specify the exact Bean name.

**Q: Are Spring Beans thread-safe?**
A: **NO.** By default, Spring Beans are created as "Singletons" (meaning only exactly one instance exists in memory). If multiple HTTP requests hit your server simultaneously, multiple threads will access the exact same Service Bean at the exact same time. If your Service Bean contains mutable class-level instance variables (like `private int counter;`), those variables will suffer from race conditions and data corruption. Business logic Beans must always remain stateless!
