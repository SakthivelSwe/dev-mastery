## WHY
When your React frontend calls your Spring Boot API, it almost always expects data in **JSON** format. 
But what if a legacy enterprise application calls your API and demands the data in **XML** format? What if a data analyst runs a script that requires the data in **CSV** (Comma Separated Values) format?
Do you create three separate endpoints (`/api/users/json`, `/api/users/xml`, `/api/users/csv`)? No! That violates REST principles. The URL should represent the resource (`/api/users`), not the format!
**Content Negotiation** is the mechanism where the Client and the Server peacefully negotiate what format the data should be sent in, allowing a single Spring Boot endpoint to dynamically return JSON, XML, or any other format based entirely on what the client specifically asks for!

## THEORY
### The 'Accept' HTTP Header
Content Negotiation relies entirely on standard HTTP Headers.
When a Client makes a `GET` request, it sends an **`Accept`** header. This header explicitly lists the MIME types (Media Types) the client understands.
- `Accept: application/json` (Client wants JSON)
- `Accept: application/xml` (Client wants XML)

### HttpMessageConverters
When your Spring Controller returns a Java Object (e.g., `return new User();`), Spring looks at the client's `Accept` header. 
Spring then searches its internal list of **HttpMessageConverters**. 
- If the client wants JSON, Spring finds the `MappingJackson2HttpMessageConverter` and translates the Java Object to JSON. 
- If the client wants XML, Spring finds the XML converter and translates the Java Object to XML!

## CODE

By default, Spring Boot only configures JSON (via the Jackson library). If a client sends `Accept: application/xml`, Spring will crash and return a `406 Not Acceptable` error because it doesn't know how to speak XML!

### 1. Enabling XML Support
To allow your API to dynamically serve XML alongside JSON, you simply need to add the XML extension for the Jackson library to your `pom.xml`.

```xml
<dependency>
    <groupId>com.fasterxml.jackson.dataformat</groupId>
    <artifactId>jackson-dataformat-xml</artifactId>
</dependency>
```
That's it! By merely having this dependency on the classpath, Spring Boot auto-configures the XML `HttpMessageConverter`.

### 2. The Controller (Zero Code Changes!)
Your Java code does not change at all. The Controller remains completely ignorant of formats.

```java
@RestController
@RequestMapping("/api/products")
public class ProductController {

    @GetMapping("/{id}")
    public Product getProduct(@PathVariable Long id) {
        // We just return a plain Java Object!
        return new Product(id, "Laptop", 999.99);
    }
}
```

### 3. Testing the Negotiation (cURL)
Now we can use a terminal to fire HTTP requests at the exact same URL, changing only the header!

**Client requests JSON:**
```bash
curl -H "Accept: application/json" http://localhost:8080/api/products/1

# Response:
# { "id": 1, "name": "Laptop", "price": 999.99 }
```

**Client requests XML:**
```bash

# <Product>
#     <id>1</id>
#     <name>Laptop</name>
#     <price>999.99</price>
# </Product>
```

## REAL_WORLD
**Restricting Accepted Formats (produces & consumes)**
While Content Negotiation is magical, sometimes you explicitly *want* an endpoint to strictly refuse certain formats.
For example, your modern microservice might strictly drop support for legacy XML to save processing power.
You can use the `produces` and `consumes` attributes inside your `@RequestMapping` annotations to hardcode the negotiation rules!

```java
@RestController
@RequestMapping("/api/modern-users")
public class UserController {

    // This endpoint explicitly REFUSES to speak anything other than JSON!
    // If a client asks for XML, Spring immediately throws a 406 Not Acceptable.
    @GetMapping(value = "/{id}", produces = "application/json")
    public User getUser(@PathVariable Long id) {
        return new User();
    }

    // This endpoint explicitly REFUSES to accept anything other than JSON!
    // If a client sends a POST request with an XML body, Spring throws a 415 Unsupported Media Type.
    @PostMapping(consumes = "application/json")
    public void createUser(@RequestBody User user) {
        // save user...
    }
}
```

## INTERVIEW
**Q: A client sends a request to your API with the header: `Accept: application/json;q=0.8, application/xml;q=0.9, text/html;q=0.2`. Your Spring Boot application supports both JSON and XML. Which format will Spring Boot return to the client, and what do the `q` values mean?**
A: **Spring Boot will return XML.**
The `q` values stand for **Quality Factor (or Weight)**. 
Content Negotiation isn't always a binary "I only want JSON" request. Browsers and complex clients often provide a fallback list of acceptable formats, ranking them by preference! 
The `q` value ranges from 0.0 to 1.0 (with 1.0 being the highest priority default if not specified).
In the provided header:
- `application/xml` has a weight of 0.9.
- `application/json` has a weight of 0.8.
- `text/html` has a weight of 0.2.
Because `application/xml` has the highest priority weight (`q=0.9`), Spring's Content Negotiation engine will inspect its available converters, realize it is capable of generating XML, and format the Java object into an XML string to fulfill the client's highest-weighted preference!
