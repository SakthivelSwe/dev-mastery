## WHY
`HashMap` is lightning fast, but it scrambles your data. If you insert elements A, B, and C, iterating over the map might return C, A, B. 
`TreeMap` keeps data perfectly sorted (A, B, C), but it slows down insertions to `O(log n)`.

What if you want the blistering `O(1)` speed of a `HashMap`, but you also want to remember the exact order in which elements were inserted? Or, what if you are building an LRU (Least Recently Used) cache and need to know exactly which element was accessed the longest time ago so you can delete it? 

Enter `LinkedHashMap`. It bridges the gap by wrapping a standard `HashMap` with an independent doubly-linked list running through every single entry, providing `O(1)` performance with predictable iteration order.

## THEORY
### The Hybrid Architecture
A `LinkedHashMap` extends `HashMap`. It maintains the exact same bucket array and hash functions. 
However, it overrides the internal `Node` class to include `before` and `after` pointers. 
Every time you insert a new key-value pair, it does two things:
1. It hashes the key and places the Node in the array bucket (just like `HashMap`).
2. It links the new Node to the tail of a global doubly-linked list.
When you iterate over a `LinkedHashMap`, it completely ignores the hash buckets and simply walks the doubly-linked list from head to tail. This guarantees that iteration is ALWAYS in **Insertion Order**.

### Access Order and LRU Caches
By default, `LinkedHashMap` maintains Insertion Order. But you can pass a special flag to its constructor: `accessOrder = true`.
When Access Order is enabled, every time you call `.get(key)`, the `LinkedHashMap` physically removes that node from its current position in the linked list and moves it to the tail. Thus, the head of the list is always the *least recently used* item, and the tail is the *most recently used* item. 
It also provides a protected method: `removeEldestEntry()`. By overriding this method to return `true` when the map size exceeds a limit (e.g., 100), the map will automatically delete the head node, giving you a production-grade LRU cache in about 5 lines of code.

## CODE

### Level 1 — Beginner
```java
import java.util.HashMap;
import java.util.LinkedHashMap;
import java.util.Map;

public class InsertionOrder {
    public static void main(String[] args) {
        Map<String, Integer> hash = new HashMap<>();
        hash.put("Zebra", 1); hash.put("Apple", 2); hash.put("Monkey", 3);
        // Prints scrambled! (e.g. Apple, Zebra, Monkey)
        System.out.println("HashMap: " + hash.keySet()); 
        
        Map<String, Integer> linked = new LinkedHashMap<>();
        linked.put("Zebra", 1); linked.put("Apple", 2); linked.put("Monkey", 3);
        // GUARANTEED to print Zebra, Apple, Monkey
        System.out.println("Linked: " + linked.keySet());
    }
}
```

### Level 2 — Building Blocks
```java
import java.util.LinkedHashMap;
import java.util.Map;

public class AccessOrder {
    public static void main(String[] args) {
        // The 3rd parameter 'true' turns on Access Order
        Map<Integer, String> cache = new LinkedHashMap<>(16, 0.75f, true);
        
        cache.put(1, "A");
        cache.put(2, "B");
        cache.put(3, "C");
        
        // Reading key 1 moves it to the END of the linked list
        cache.get(1); 
        
        // Prints: 2, 3, 1
        System.out.println(cache.keySet());
    }
}
```

### Level 3 — Intermediate
```java
import java.util.LinkedHashMap;
import java.util.Map;

// Building a production-grade LRU Cache in 5 lines
public class LRUCache<K, V> extends LinkedHashMap<K, V> {
    private final int MAX_ENTRIES;

    public LRUCache(int maxEntries) {
        // Enable access order
        super(maxEntries + 1, 1.0f, true);
        this.MAX_ENTRIES = maxEntries;
    }

    // The JVM automatically calls this after every put().
    // If it returns true, it deletes the oldest item!
    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        return size() > MAX_ENTRIES;
    }
}
```

### Level 4 — Advanced
```java
public class CacheUsage {
    public static void main(String[] args) {
        LRUCache<Integer, String> imageCache = new LRUCache<>(3);
        
        imageCache.put(1, "image1.png");
        imageCache.put(2, "image2.png");
        imageCache.put(3, "image3.png"); // Map is full (Size 3)
        
        imageCache.get(1); // 1 is now the MOST recently used
        
        imageCache.put(4, "image4.png"); // Triggers removeEldestEntry!
        
        // Key 2 was the LEAST recently used, so it was evicted!
        // Prints: {3=image3.png, 1=image1.png, 4=image4.png}
        System.out.println(imageCache);
    }
}
```

## REAL_WORLD
In Spring Boot applications, developers often need to fetch a list of roles from the database and send them to the frontend UI as JSON. 
If the database query sorts the roles by `priority` using `ORDER BY priority DESC`, the JDBC driver returns the rows in perfect order. 
However, if the developer blindly maps those rows into a standard `HashMap<String, RoleDTO>`, the Map destroys the sorting! When the JSON is serialized and sent to the frontend, the UI renders the roles out of order, leading to a horrible user experience.

To preserve the `ORDER BY` clause from a database when constructing a Map, you **must** use a `LinkedHashMap`. It guarantees that the JSON serialization will process the keys in the exact order they were inserted.

## INTERVIEW
**Q: How does `LinkedHashMap` guarantee insertion order?**
A: It maintains a global doubly-linked list connecting all entries. When a new entry is put into the map, it is appended to the tail of this list. When you iterate over the map, it simply follows the linked list pointers instead of scanning the array buckets.

**Q: Does `LinkedHashMap` consume more memory than `HashMap`?**
A: Yes. Every `Node` in a `LinkedHashMap` has two extra pointers (`before` and `after`) to maintain the doubly-linked list. On a 64-bit JVM, this adds 16 bytes of overhead to every single entry in the map.

**Q: How do you configure a `LinkedHashMap` to act as an LRU Cache?**
A: You must invoke the constructor with the `accessOrder` boolean set to `true`. Then, you override the `removeEldestEntry(Map.Entry eldest)` method to return `true` when the map's size exceeds your desired maximum capacity. The map will automatically evict the least recently accessed item.
