# Function Components and Composition

## Components as Pure Functions of Props

In modern React, components are represented exclusively as pure JavaScript functions. A functional component receives an object of properties (`props`) and returns a React Node (typically JSX) representing the UI.

```typescript
interface ProfileProps {
  username: string;
  avatarUrl: string;
}

export const Profile: React.FC<ProfileProps> = ({ username, avatarUrl }) => {
  return (
    <div className="profile-card">
      <img src={avatarUrl} alt={`${username}'s avatar`} />
      <h3>{username}</h3>
    </div>
  );
};
```

### Element vs. Component Naming Convention
React relies on the casing of element tags to differentiate between native HTML elements and custom React components during compilation:
* **Lowercase tags** (e.g., `<div />`, `<span />`) compile to strings representing host elements: `_jsx('div', {})`.
* **Capitalized tags** (e.g., `<Profile />`, `<Button />`) compile directly to the function references: `_jsx(Profile, {})`.

## The Rule of Purity

React is built on the paradigm of functional programming. Components must be **pure functions** during their render phase.

### Rules of Pure Components:
1. **Same Props, Same Output**: Given the same inputs, a component must always return the identical JSX structure.
2. **No Side Effects during Render**: A component must not modify any variables or objects that existed before rendering. It must not perform network requests, mutate global variables, or write to the DOM during execution.

```typescript
// BAD: Impure Component
let renderCount = 0; // Global state mutation

export const ImpureComponent: React.FC = () => {
  renderCount++; // SIDE EFFECT DURING RENDER!
  return <div>Render count: {renderCount}</div>;
};

// GOOD: Pure Component
interface PureProps {
  count: number;
}

export const PureComponent: React.FC<PureProps> = ({ count }) => {
  return <div>Render count: {count}</div>;
};
```

## Component Composition over Inheritance

React does not support class-based inheritance patterns. Instead, it leverages **Composition**—building complex, feature-rich components by nesting and combining simpler, focused components.

### The `children` Prop and `React.ReactNode`
The primary mechanism of composition is the `children` prop, which allows a component to pass arbitrary elements or nodes directly into its output template.

```typescript
import React from 'react';

interface CardProps {
  title: string;
  children: React.ReactNode; // Strongly typed children supporting elements, strings, etc.
}

export const Card: React.FC<CardProps> = ({ title, children }) => {
  return (
    <div className="card">
      <div className="card-header">
        <h2>{title}</h2>
      </div>
      {/* Composition point */}
      <div className="card-body">{children}</div>
    </div>
  );
};

// Usage:
export const App: React.FC = () => {
  return (
    <Card title="User Metrics">
      <p>Performance is currently optimal.</p>
      <button onClick={() => alert('Details')}>View Logs</button>
    </Card>
  );
};
```

## When to Split Components

Monolithic components degrade codebase maintainability and trigger unnecessary re-renders. Split components based on three architectural metrics:

1. **Single Responsibility Principle (SRP)**: A component should do one thing. If a component displays a user profile, handles edit forms, and renders billing charts, it should be split into three sub-components.
2. **Reusability**: If a UI pattern or logic block is used in multiple places (e.g., buttons, input fields, modal dialogs), abstract it.
3. **State Isolation (Performance)**: If a single part of a large component updates frequently (e.g., a search input or real-time ticker), isolate that interactive element into its own component. This ensures only the small component re-renders, preventing expensive tree-wide reconciliations.

## Colocating State: Lifting and Lowering

State should be kept as close to where it is used as possible. This is known as **State Colocation**.

* **Lowering State**: If state is only used by a single subtree, move it down into that specific component.
* **Lifting State Up**: If two or more sibling components need access to the same state, move the state to their nearest common ancestor. Pass the state down as props, and pass state-mutator callbacks back up.

```typescript
import React, { useState } from 'react';

// Sibling A: Input
interface InputProps {
  value: string;
  onChange: (val: string) => void;
}

const SearchInput: React.FC<InputProps> = ({ value, onChange }) => (
  <input type="text" value={value} onChange={(e) => onChange(e.target.value)} />
);

// Sibling B: Previewer
const Previewer: React.FC<{ content: string }> = ({ content }) => (
  <p>Search Term: {content}</p>
);

// Common Ancestor: Manages the lifted state
export const ParentContainer: React.FC = () => {
  const [searchTerm, setSearchTerm] = useState<string>('');

  return (
    <div className="container">
      <SearchInput value={searchTerm} onChange={setSearchTerm} />
      <Previewer content={searchTerm} />
    </div>
  );
};
```
