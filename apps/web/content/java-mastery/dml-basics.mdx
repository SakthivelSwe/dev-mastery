# DML — INSERT, UPDATE, DELETE, SELECT

### 1. Architectural & Theoretical Foundations

#### MVCC and DML Internals
Every DML operation in PostgreSQL interacts with Multi-Version Concurrency Control (MVCC) metadata fields stored inside each tuple header:
*   `xmin`: The Transaction ID (TxID) of the inserting transaction.
*   `xmax`: The TxID of the deleting/updating transaction. For active rows, this is 0.

*   **INSERT:** Allocates a new physical slot in a table page, writes the tuple data, and writes the current TxID to `xmin`.
*   **DELETE:** Does not physically erase the row. It sets the `xmax` of the existing tuple to the current TxID. The row remains on disk until a `VACUUM` process reclaims it after all transactions that could see it have completed.
*   **UPDATE:** Conceptually executes a `DELETE` followed by an `INSERT`. It sets `xmax` of the old tuple to the current TxID and inserts a new tuple with `xmin` set to the current TxID.

#### TRUNCATE vs. DELETE
*   `DELETE`: A DML operation that scans the table and flags each row as deleted (by setting `xmax`). It generates substantial WAL overhead and triggers row-level triggers.
*   `TRUNCATE`: A DDL operation that bypasses row-by-row deletes. It creates a new, empty data file (relation fork) for the table and unlinks the old one. It releases disk space immediately, does not fire individual delete triggers, and requires an exclusive access lock (`AccessExclusiveLock`) on the table, blocking all concurrent operations.

#### PostgreSQL RETURNING Clause
The `RETURNING` clause is a PostgreSQL extension that returns the values of columns modified by `INSERT`, `UPDATE`, or `DELETE` statements. This avoids the need for a separate `SELECT` query, saving a network round-trip and ensuring atomic read-after-write operations.

### 2. DDL & DML Code Implementation

```sql
-- Setup Schema
CREATE TABLE inventory_items (
    item_id BIGINT GENERATED ALWAYS AS IDENTITY,
    sku VARCHAR(100) NOT NULL,
    quantity INTEGER NOT NULL,
    price NUMERIC(10, 2) NOT NULL,
    last_updated TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT pk_inventory_items PRIMARY KEY (item_id),
    CONSTRAINT uq_sku UNIQUE (sku)
);

CREATE TABLE supplier_discounts (
    sku VARCHAR(100) NOT NULL,
    discount_pct NUMERIC(5, 2) NOT NULL,
    CONSTRAINT pk_supplier_discounts PRIMARY KEY (sku)
);

-- Seed Data
INSERT INTO inventory_items (sku, quantity, price) 
VALUES 
('SKU-A', 100, 50.00),
('SKU-B', 15, 120.00),
('SKU-C', 5, 15.00);

INSERT INTO supplier_discounts (sku, discount_pct) 
VALUES 
('SKU-B', 0.15),
('SKU-C', 0.50);

-- DML 1: INSERT with RETURNING
INSERT INTO inventory_items (sku, quantity, price)
VALUES ('SKU-D', 200, 8.50)
RETURNING item_id, sku, (quantity * price) AS total_value;

-- DML 2: UPDATE with Subquery / JOIN equivalent
UPDATE inventory_items i
SET price = i.price * (1 - d.discount_pct),
    last_updated = CURRENT_TIMESTAMP
FROM supplier_discounts d
WHERE i.sku = d.sku
RETURNING i.item_id, i.sku, i.price AS discounted_price;

-- DML 3: DELETE using USING clause
DELETE FROM inventory_items i
USING supplier_discounts d
WHERE i.sku = d.sku AND d.discount_pct >= 0.50
RETURNING i.sku, i.quantity;
```
