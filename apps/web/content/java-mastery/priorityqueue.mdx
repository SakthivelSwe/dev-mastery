## WHY
Imagine you are building the backend for a hospital Emergency Room. Patients arrive in a specific order (a Queue). If you use a standard `LinkedList` (FIFO - First In, First Out), the patient who arrived first gets treated first. But what if the 10th patient who arrives is having a heart attack, while the 1st patient just has a mild cough? 
A FIFO Queue will process the cough first, with fatal consequences.

You need a data structure that completely ignores insertion order and instead processes elements based on their **Priority** or severity. 
`PriorityQueue` solves this exact problem. It guarantees that every time you pull an element from the queue, you are instantly given the absolute highest priority item available, making it the backbone of scheduling algorithms, Dijkstra's pathfinding, and task processing engines.

## THEORY
### The Min-Heap / Max-Heap Architecture
Unlike `LinkedList` which uses node pointers, a `PriorityQueue` in Java is backed by a **Binary Min-Heap** (by default). 
A Binary Heap is a complete binary tree that obeys one strict rule: *Every parent node must be strictly smaller than or equal to its children*.
- **The Root**: Because of the heap rule, the root node of the tree is mathematically guaranteed to be the smallest (or highest priority) element in the entire collection.
- **Array Storage**: Even though it's a "Tree" conceptually, a Heap is actually flattened and stored inside a standard `Object[]` array! The children of the element at index `i` are located exactly at `2*i + 1` and `2*i + 2`. This makes it incredibly memory efficient.

### Time Complexity
- **`peek()`**: `O(1)`. Finding the highest priority item is instantaneous because it is always at index `0` of the array.
- **`add()` / `poll()`**: `O(log n)`. When you add a new item, it's placed at the end of the array. The queue then performs an operation called "Sift Up" (or bubble up), swapping the new item with its parent until the heap rule is satisfied. When you remove the root, it takes the last item, puts it at the root, and "Sifts Down".

## CODE

### Level 1 — Beginner
```java
import java.util.PriorityQueue;
import java.util.Queue;

public class BasicPQ {
    public static void main(String[] args) {
        // By default, it's a Min-Heap (smallest numbers come out first)
        Queue<Integer> numbers = new PriorityQueue<>();
        
        numbers.add(50);
        numbers.add(10);
        numbers.add(30);
        
        // Iterating a PriorityQueue DOES NOT print in priority order!
        // It prints the raw flattened array (10, 50, 30).
        System.out.println("Raw structure: " + numbers);
        
        // You MUST use poll() to extract items in priority order
        System.out.println(numbers.poll()); // 10
        System.out.println(numbers.poll()); // 30
        System.out.println(numbers.poll()); // 50
    }
}
```

### Level 2 — Building Blocks
```java
import java.util.PriorityQueue;
import java.util.Comparator;

public class MaxHeap {
    public static void main(String[] args) {
        // Providing a reverse comparator turns it into a Max-Heap
        // (Largest numbers come out first)
        PriorityQueue<Integer> maxHeap = new PriorityQueue<>(Comparator.reverseOrder());
        
        maxHeap.add(5);
        maxHeap.add(100);
        maxHeap.add(20);
        
        System.out.println(maxHeap.poll()); // 100
    }
}
```

### Level 3 — Intermediate
```java
import java.util.PriorityQueue;
import java.util.Comparator;

class Patient {
    String name;
    int severity; // 10 is critical, 1 is mild
    Patient(String name, int severity) { this.name = name; this.severity = severity; }
}

public class HospitalQueue {
    public static void main(String[] args) {
        // Custom Comparator: Sort by severity DESCENDING
        PriorityQueue<Patient> er = new PriorityQueue<>(
            (p1, p2) -> Integer.compare(p2.severity, p1.severity)
        );
        
        er.add(new Patient("Alice", 2)); // Cough
        er.add(new Patient("Bob", 10));  // Heart Attack
        er.add(new Patient("Charlie", 5)); // Broken Arm
        
        // Bob is extracted first!
        System.out.println(er.poll().name); // Bob
    }
}
```

### Level 4 — Advanced
```java
import java.util.PriorityQueue;

public class TopKElements {
    // Advanced Algorithm: Find the top K largest elements in a massive stream of data
    public static void findTopK(int[] massiveData, int k) {
        // Keep a Min-Heap of size K
        PriorityQueue<Integer> minHeap = new PriorityQueue<>();
        
        for (int num : massiveData) {
            minHeap.add(num);
            // If the heap grows larger than K, remove the SMALLEST element
            if (minHeap.size() > k) {
                minHeap.poll(); 
            }
        }
        
        // The heap now contains EXACTLY the K largest elements!
        System.out.println(minHeap);
    }
}
```

## REAL_WORLD
In Spring Boot microservices, you often rely on external message brokers (like RabbitMQ or Apache Kafka) for queueing tasks. However, if you are building an in-memory task scheduler (like Quartz or Java's `ScheduledThreadPoolExecutor`), `PriorityQueue` is the engine that drives it.

`ScheduledThreadPoolExecutor` uses a special variant called `DelayQueue` (which is internally powered by a `PriorityQueue`). When you schedule a task to run in 5 minutes, it wraps the task in a node where the "priority" is the exact Execution Timestamp. The task with the *closest* timestamp always bubbles up to the root of the Min-Heap. The background thread simply calls `.peek()` to check the root. If the timestamp is in the future, the thread goes to sleep. When it wakes up, it pops the root task in `O(log n)` time and executes it. This is how JVM timers work under the hood!

## INTERVIEW
**Q: How is a `PriorityQueue` stored in memory?**
A: Despite acting conceptually like a Binary Tree (a Heap), `PriorityQueue` is stored flattened in a standard contiguous `Object[]` array. The JVM navigates the "tree" using mathematical indices: the children of index `i` are at `2*i + 1` and `2*i + 2`. This eliminates the need for expensive node objects and pointers.

**Q: If you iterate over a `PriorityQueue` using a standard `for-each` loop, will the elements print in sorted order?**
A: No! This is a classic trap. Iterating over a `PriorityQueue` simply iterates over the backing array sequentially. A Binary Heap only guarantees that the parent is smaller than the children; it does not guarantee that the array itself is perfectly sorted. The ONLY way to retrieve elements in sorted order is to repeatedly call `.poll()`.

**Q: What is the time complexity of `remove(Object o)` in a PriorityQueue?**
A: It is `O(n)` linear time! Because a Heap only guarantees the position of the root node, searching for a specific arbitrary object requires scanning the entire underlying array sequentially. Once found, removing it and re-sifting the heap takes an additional `O(log n)`.
