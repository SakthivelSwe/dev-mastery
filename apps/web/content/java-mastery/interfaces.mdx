## WHY
In Java, a class can only extend one parent class (Single Inheritance) to avoid the infamous Diamond Problem. But what happens when an object needs to exhibit multiple distinct behaviors? A `Smartphone` IS-A `Computer`, but it CAN-DO `Camera` things and CAN-DO `Phone` things. 

Interfaces solve this by providing a pure behavioral contract. They define exactly *what* an object can do, without dictating *how* it does it. This allows completely unrelated classes to interact seamlessly. For example, both a `PDFDocument` and a `UserAvatar` can implement the `Printable` interface, allowing a single `Printer` class to process both of them without knowing anything about their internal structure.

## THEORY
### The Interface Contract
An interface is a 100% abstract blueprint. By default, every method declared in an interface is implicitly `public abstract`. Every field declared in an interface is implicitly `public static final` (a constant). 
Because interfaces have no state (no instance variables) and (traditionally) no method bodies, a class can safely implement an infinite number of interfaces without any memory collision issues.

### Java 8 Evolution
Historically, once an interface was published and implemented by third parties, adding a new method to it would break the entire world (because every implementing class would fail to compile).
To solve this, Java 8 introduced **Default Methods**. You can now provide a concrete implementation inside the interface using the `default` keyword. Existing classes inherit this default behavior automatically, preserving backward compatibility.

## CODE

### Level 1 — Beginner
```java
public interface Drivable {
    // implicitly public abstract
    void startEngine(); 
    void accelerate();
}

public class Car implements Drivable {
    @Override
    public void startEngine() { System.out.println("Vroom"); }
    
    @Override
    public void accelerate() { System.out.println("Speeding up"); }
}
```

### Level 2 — Building Blocks
```java
// A class extending a class AND implementing multiple interfaces
public class Smartphone extends Computer implements Camera, Phone {
    @Override
    public void takePhoto() { System.out.println("Snap"); }
    
    @Override
    public void makeCall(String number) { System.out.println("Calling " + number); }
}
```

### Level 3 — Intermediate
```java
public interface PaymentGateway {
    void charge(double amount);
    
    // Java 8 Default method. 
    // Implementing classes don't need to override this unless they want custom behavior.
    default void refund(double amount) {
        System.out.println("Generic refund logic for " + amount);
    }
}
```

### Level 4 — Advanced
```java
public interface Cache {
    void put(String key, Object value);
    
    // Java 9 Private Methods
    // Used to share internal logic between default methods without exposing it
    default void putAll(Map<String, Object> map) {
        log("Batch insert starting");
        map.forEach(this::put);
    }
    
    private void log(String message) {
        System.out.println("CACHE SYSTEM: " + message);
    }
}
```

## REAL_WORLD
Interfaces are the absolute foundation of the Spring Framework (Dependency Injection) and modern testing practices.

If your Spring Controller directly instantiated a `StripePaymentService`, you could never test your controller without actually pinging Stripe's servers. By depending on a `PaymentGateway` interface instead, Spring can inject the real `StripePaymentService` in production, but during Unit Testing, you can inject a `MockPaymentService` (using Mockito) that instantly returns a fake success response. This pattern (Programming to an Interface) is critical for enterprise software.

## INTERVIEW
**Q: Can an Interface extend another Interface?**
A: Yes! An interface can extend multiple other interfaces using the `extends` keyword (e.g., `interface SmartDevice extends Camera, Phone`). However, a class uses the `implements` keyword to inherit from an interface.

**Q: What is a Marker Interface?**
A: A Marker Interface is an interface that contains absolutely no methods or fields (e.g., `java.io.Serializable` or `java.lang.Cloneable`). It is used simply to "mark" a class so the JVM or a framework knows it possesses a certain property and can be processed in a special way at runtime using `instanceof` checks.
