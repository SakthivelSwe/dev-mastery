## WHY
Java enforces single class inheritance to avoid the Diamond Problem — a class can only `extend` one parent. But what if an object genuinely needs to exhibit multiple, unrelated capabilities? A `Smartphone` IS-A `Computer`, but it also CAN take photos (Camera behavior) AND make calls (Phone behavior) AND play music (MediaPlayer behavior).

Interfaces solve this by allowing a class to **implement multiple behavioral contracts** simultaneously. Unlike classes, interfaces define **WHAT** an object can do without any inheritance of state — they're pure contracts.

Why interfaces matter in practice:
1. **Multiple behaviors**: `class Duck extends Animal implements Flyable, Swimmable`
2. **Testability**: Code against interfaces, inject mocks in tests — never import concrete classes in your business layer
3. **Plugin systems**: Third-party code can implement your interface and integrate seamlessly
4. **Functional programming**: Lambda expressions are interface implementations — `Runnable`, `Comparator`, `Predicate` are all interfaces

In modern Java, interfaces are the **#1 tool for writing decoupled, testable, maintainable code**.

## THEORY

### Interface Members — What's Allowed

| Member Type | Java Version | Notes |
|------------|-------------|-------|
| `public abstract` method | All | Default implicit modifier |
| `public static final` constant | All | Default — no instance fields |
| `default` method | Java 8+ | Concrete implementation in interface |
| `static` method | Java 8+ | Utility methods on the interface |
| `private` method | Java 9+ | Internal helper between default methods |
| `private static` method | Java 9+ | Internal static helper |

### Multiple Interface Implementation
A class can implement unlimited interfaces — no Diamond Problem since interfaces (traditionally) have no state:

```java
class SmartTV extends ElectronicDevice
             implements Streamable, Connectable, VoiceControlled, ScreenSharing {
    // Must implement ALL abstract methods from all 4 interfaces
}
```

### Marker Interfaces
An interface with zero methods. Used purely to "mark" a class so the JVM or frameworks treat it specially:
- `java.io.Serializable` — JVM allows this object to be serialized to bytes
- `java.lang.Cloneable` — JVM allows `clone()` to be called
- `java.util.RandomAccess` — Collections framework optimizes for random access

Modern alternative: Annotations (`@interface`) are now preferred over marker interfaces.

### Interface Inheritance
Interfaces can extend multiple other interfaces:
```java
interface ReadableRepository<T> {
    T findById(Long id);
    List<T> findAll();
}

interface WritableRepository<T> {
    T save(T entity);
    void delete(Long id);
}

// Combines both — no body needed!
interface Repository<T> extends ReadableRepository<T>, WritableRepository<T> {}
```

### Sealed Interfaces (Java 17+)
Restrict which classes may implement an interface:
```java
public sealed interface Shape
    permits Circle, Rectangle, Triangle {}

// Only Circle, Rectangle, Triangle can implement Shape
// Enables exhaustive pattern matching in switch expressions
```

### Interface vs Abstract Class — Quick Reference

| | Interface | Abstract Class |
|-|-----------|----------------|
| Instance fields | ❌ | ✅ |
| Constructors | ❌ | ✅ |
| Multiple inheritance | ✅ | ❌ |
| `default` methods | ✅ | ✅ (concrete methods) |
| IS-A relationship | CAN-DO | IS-A |
| When to use | Unrelated classes sharing behavior | Related classes sharing state |

## CODE

### Level 1 — Beginner: Basic Interface Implementation
```java
// Interface — defines WHAT, not HOW
public interface Printable {
    void print();       // implicitly public abstract
    void preview();     // implicitly public abstract

    // Constant — implicitly public static final
    int MAX_PAGES = 1000;
}

public interface Exportable {
    String export(String format);  // Returns exported content
}

// A Document can BOTH print and export
public class PDFDocument implements Printable, Exportable {
    private String title;
    private String content;

    public PDFDocument(String title, String content) {
        this.title = title;
        this.content = content;
    }

    @Override
    public void print() {
        System.out.println("[PDF PRINTER] Printing: " + title);
        System.out.println(content);
    }

    @Override
    public void preview() {
        System.out.println("[PREVIEW] " + title + " — " + content.substring(0, Math.min(50, content.length())) + "...");
    }

    @Override
    public String export(String format) {
        return switch (format.toUpperCase()) {
            case "HTML" -> "<html><body><h1>" + title + "</h1><p>" + content + "</p></body></html>";
            case "MARKDOWN" -> "# " + title + "\n\n" + content;
            default -> throw new UnsupportedOperationException("Unsupported format: " + format);
        };
    }
}

public class InterfaceBasics {
    public static void main(String[] args) {
        PDFDocument doc = new PDFDocument("Java Guide", "Java is a powerful language.");

        // Use as Printable reference
        Printable printer = doc;
        printer.print();
        printer.preview();

        // Use as Exportable reference
        Exportable exporter = doc;
        System.out.println(exporter.export("HTML"));
        System.out.println(exporter.export("MARKDOWN"));

        // Interface constant
        System.out.println("Max pages: " + Printable.MAX_PAGES); // 1000
    }
}
```

### Level 2 — Building Blocks: Repository Pattern with Interfaces
```java
import java.util.*;

// Generic repository interface — the core contract
public interface Repository<T, ID> {
    T save(T entity);
    Optional<T> findById(ID id);
    List<T> findAll();
    void delete(ID id);
    long count();
}

// User-specific interface extending generic contract
public interface UserRepository extends Repository<User, Long> {
    Optional<User> findByEmail(String email);
    List<User> findByRole(String role);
    boolean existsByEmail(String email);
}

// Simple entity
public class User {
    private Long id;
    private String name;
    private String email;
    private String role;

    // Constructor, getters, setters...
    public User(String name, String email, String role) {
        this.name = name;
        this.email = email;
        this.role = role;
    }

    public Long getId() { return id; }
    public void setId(Long id) { this.id = id; }
    public String getName() { return name; }
    public String getEmail() { return email; }
    public String getRole() { return role; }

    @Override
    public String toString() {
        return String.format("User{id=%d, name=%s, role=%s}", id, name, role);
    }
}

// In-memory implementation — perfect for tests
public class InMemoryUserRepository implements UserRepository {
    private final Map<Long, User> store = new HashMap<>();
    private long idSeq = 1;

    @Override
    public User save(User user) {
        if (user.getId() == null) user.setId(idSeq++);
        store.put(user.getId(), user);
        return user;
    }

    @Override
    public Optional<User> findById(Long id) {
        return Optional.ofNullable(store.get(id));
    }

    @Override
    public List<User> findAll() { return new ArrayList<>(store.values()); }

    @Override
    public void delete(Long id) { store.remove(id); }

    @Override
    public long count() { return store.size(); }

    @Override
    public Optional<User> findByEmail(String email) {
        return store.values().stream()
                    .filter(u -> u.getEmail().equals(email))
                    .findFirst();
    }

    @Override
    public List<User> findByRole(String role) {
        return store.values().stream()
                    .filter(u -> u.getRole().equals(role))
                    .toList();
    }

    @Override
    public boolean existsByEmail(String email) {
        return store.values().stream().anyMatch(u -> u.getEmail().equals(email));
    }
}

class RepositoryDemo {
    public static void main(String[] args) {
        UserRepository repo = new InMemoryUserRepository();

        repo.save(new User("Alice", "alice@example.com", "ADMIN"));
        repo.save(new User("Bob", "bob@example.com", "USER"));
        repo.save(new User("Carol", "carol@example.com", "USER"));

        System.out.println(repo.count());                          // 3
        System.out.println(repo.findByRole("USER"));               // [Bob, Carol]
        System.out.println(repo.existsByEmail("alice@example.com")); // true
    }
}
```

### Level 3 — Intermediate: Default Methods and Interface Evolution
```java
import java.util.Collection;

// Before Java 8, adding this method would break ALL existing List implementations
public interface Sortable<T extends Comparable<T>> {
    Collection<T> getItems();

    // Default method — new behavior added without breaking existing code
    default List<T> sorted() {
        List<T> copy = new ArrayList<>(getItems());
        Collections.sort(copy);
        return copy;
    }

    default T min() {
        return getItems().stream()
                         .min(Comparator.naturalOrder())
                         .orElseThrow(() -> new NoSuchElementException("Empty collection"));
    }

    default T max() {
        return getItems().stream()
                         .max(Comparator.naturalOrder())
                         .orElseThrow(() -> new NoSuchElementException("Empty collection"));
    }
}

// Static factory method on interface (Java 8+)
public interface Validator<T> {
    boolean isValid(T value);
    String getErrorMessage();

    static <T> Validator<T> of(java.util.function.Predicate<T> predicate, String errorMsg) {
        return new Validator<T>() {
            @Override public boolean isValid(T value) { return predicate.test(value); }
            @Override public String getErrorMessage() { return errorMsg; }
        };
    }

    default Validator<T> and(Validator<T> other) {
        return new Validator<T>() {
            @Override public boolean isValid(T value) {
                return Validator.this.isValid(value) && other.isValid(value);
            }
            @Override public String getErrorMessage() {
                return Validator.this.getErrorMessage() + " AND " + other.getErrorMessage();
            }
        };
    }
}
```

### Level 4 — Advanced: Sealed Interfaces and Pattern Matching
```java
// Java 17+ Sealed Interfaces — closed hierarchy for exhaustive matching
public sealed interface PaymentResult
    permits PaymentResult.Success, PaymentResult.Failure, PaymentResult.Pending {

    record Success(String transactionId, double amount) implements PaymentResult {}
    record Failure(String code, String reason) implements PaymentResult {}
    record Pending(String referenceId, String message) implements PaymentResult {}
}

public class PaymentProcessor {

    public String processResult(PaymentResult result) {
        // Java 21 switch expression with pattern matching — EXHAUSTIVE!
        // Compiler guarantees all sealed subtypes are handled
        return switch (result) {
            case PaymentResult.Success s ->
                String.format("✅ Payment successful! TxnId=%s, Amount=$%.2f",
                              s.transactionId(), s.amount());

            case PaymentResult.Failure f ->
                String.format("❌ Payment failed! Code=%s, Reason=%s",
                              f.code(), f.reason());

            case PaymentResult.Pending p ->
                String.format("⏳ Payment pending. Ref=%s, Message=%s",
                              p.referenceId(), p.message());
        };
        // No default needed — sealed hierarchy is exhaustive!
    }

    public static void main(String[] args) {
        PaymentProcessor processor = new PaymentProcessor();

        System.out.println(processor.processResult(
            new PaymentResult.Success("TXN-001", 99.99)));

        System.out.println(processor.processResult(
            new PaymentResult.Failure("INSUFFICIENT_FUNDS", "Card declined")));

        System.out.println(processor.processResult(
            new PaymentResult.Pending("REF-123", "3DS verification required")));
    }
}
```

### Level 5 — Expert: Functional Composition with Interface Chains
```java
import java.util.function.*;
import java.util.*;
import java.util.stream.*;

// Building a type-safe pipeline using interface composition
public interface Pipeline<T, R> {
    R execute(T input);

    default <V> Pipeline<T, V> then(Pipeline<R, V> next) {
        return input -> next.execute(this.execute(input));
    }
    
    static <T> Pipeline<T, T> identity() {
        return input -> input;
    }
}

public class PipelineDemo {
    public static void main(String[] args) {
        // Each step is a lambda implementing the Pipeline interface
        Pipeline<String, String>         trim       = String::trim;
        Pipeline<String, String>         lowercase  = String::toLowerCase;
        Pipeline<String, String[]>       split      = s -> s.split("\\s+");
        Pipeline<String[], List<String>> toList     = Arrays::asList;
        Pipeline<List<String>, String>   join       = l -> String.join("-", l);

        // Compose a pipeline: trim → lowercase → split → toList → join
        Pipeline<String, String> slugify = trim
            .then(lowercase)
            .then(split)
            .then(toList)
            .then(join);

        System.out.println(slugify.execute("  Java OOP Concepts  ")); // "java-oop-concepts"
        System.out.println(slugify.execute("Spring Boot Security"));   // "spring-boot-security"
    }
}
```

## REAL_WORLD

### Spring Data JPA — Interface-Based Repositories
Spring Data JPA is the prime example of interface power in production:

```java
// You write ONLY the interface — Spring generates the implementation at runtime!
public interface UserRepository extends JpaRepository<User, Long> {

    // Spring derives the SQL query from the method name!
    Optional<User> findByEmail(String email);
    List<User> findByRoleAndActiveTrue(String role);

    // Custom query via annotation
    @Query("SELECT u FROM User u WHERE u.createdAt > :date ORDER BY u.createdAt DESC")
    List<User> findRecentUsers(@Param("date") LocalDateTime date);

    // Projection interface — only fetch needed columns
    interface UserSummary {
        Long getId();
        String getUsername();
        String getEmail();
    }

    List<UserSummary> findAllProjectedBy();
}
```

Spring creates a Proxy object at runtime that implements your interface with real Hibernate/JDBC calls. Your service code never knows about databases — it only talks to the interface.

### Mockito — Testing via Interfaces
The reason we code against interfaces becomes clear in testing:

```java
// Without interface: can't test UserService without real database
@Service
public class UserService {
    private final UserRepository userRepository; // Interface!

    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    public User createUser(String name, String email) {
        if (userRepository.existsByEmail(email)) {
            throw new DuplicateEmailException("Email already in use");
        }
        return userRepository.save(new User(name, email, "USER"));
    }
}

// Test — inject a Mockito mock instead of real database
@ExtendWith(MockitoExtension.class)
class UserServiceTest {
    @Mock
    UserRepository userRepository;    // Mockito creates a fake implementation

    @InjectMocks
    UserService userService;

    @Test
    void shouldThrowWhenEmailExists() {
        when(userRepository.existsByEmail("alice@test.com")).thenReturn(true);

        assertThrows(DuplicateEmailException.class,
            () -> userService.createUser("Alice", "alice@test.com"));
    }
}
```

## INTERVIEW

**Q: Can an interface extend another interface?**
A: Yes! An interface can `extend` multiple other interfaces. Example: `interface Repository<T> extends ReadableRepository<T>, WritableRepository<T>`. Unlike a class, interfaces don't inherit state — only method contracts — so there's no Diamond Problem for method signatures (though conflicts in default method bodies must still be resolved).

**Q: What is a Marker Interface?**
A: An interface with no methods. Used to "tag" a class for special treatment by the JVM or framework. `java.io.Serializable` marks that the JVM should serialize the object. Modern Java prefers annotations for this purpose since annotations can carry metadata.

**Q: Why should you code against interfaces rather than concrete classes?**
A: Loose coupling. Code that depends on `UserRepository` (interface) can work with ANY implementation — JPA, MongoDB, in-memory, mock. If you depend on `JpaUserRepository` directly, switching databases requires changing all dependent code, and testing requires a real database connection.

**Q: What is the difference between a default method in an interface and a method in an abstract class?**
A: Both provide concrete implementations. Key differences: (1) interface default methods cannot access instance fields (interfaces have none); (2) a class can implement multiple interfaces getting all their default methods, but can only extend one abstract class; (3) abstract class methods can have `protected` or package-private access, while interface default methods must be `public`.

**Q: What are sealed interfaces and why are they useful?**
A: Sealed interfaces (Java 17+) restrict which classes can implement them using the `permits` clause. They enable exhaustive pattern matching in switch expressions — the compiler knows all possible types, so no `default` case is needed. This is perfect for result types like `Success | Failure | Pending` where you want compile-time guarantee that all cases are handled.

## FEYNMAN CHECK

Explain to a 10-year-old:
> An interface is like a **job contract**. The contract says "any employee hired for this job must be able to `cook()` and `clean()`." But the contract doesn't say HOW — Chef Alice might cook French cuisine and clean with bleach; Chef Bob might cook sushi and clean with vinegar. The restaurant manager just knows: "If I hired you with this contract, I can call `cook()` and `clean()` on you."
