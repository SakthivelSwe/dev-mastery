# JavaScript Event Loop — Complete Internal Model

The JavaScript runtime environment is designed around a single-threaded execution model. To achieve highly concurrent, non-blocking I/O operations without the overhead of native OS-level multi-threading (such as context-switching overhead and race conditions), JavaScript environments rely on a cooperative, event-driven architecture. 

This architecture is governed by the **V8 Engine** (which handles synchronous execution) and an external event loop coordinator: **Libuv** in Node.js, or the browser’s **Blink/Web APIs** platform orchestration layer.

---

## 1. The V8 Call Stack & Execution Mechanics

At the core of JavaScript execution is the V8 Call Stack. The call stack is a standard LIFO (Last-In, First-Out) data structure that tracks the execution state of the program.

```
+------------------------------------------+
|  Stack Frame: inner()                    |  <-- Active Execution Context
+------------------------------------------+
|  Stack Frame: outer()                    |
+------------------------------------------+
|  Stack Frame: anonymous (Global Context) |
+------------------------------------------+
```

### Stack Frames and Execution Contexts
Every time a JavaScript function is invoked, V8 creates an **Execution Context** (represented internally in V8 by `v8::internal::StackFrame` defined in `src/execution/frames.h`). This context contains:
1. **Local Variables and Parameters**: Values scoped to the function.
2. **Scope Chain**: References to outer lexical environments.
3. **The `this` Binding**: Resolves dynamically or lexically.
4. **Return Address**: The instruction pointer to resume execution in the calling frame once the current frame pops.

### Stack Overflow (Recursion Limits)
Because physical memory is finite, the call stack has a hard limit. In V8, this limit is defined by the security and stability boundaries of the underlying platform (typically 1MB to 10MB depending on the OS and environment). 

If a function recursively calls itself without a base case:
```javascript
function recurse() {
  recurse();
}
recurse(); // RangeError: Maximum call stack size exceeded
```
V8 triggers a **Stack Overflow**. Internally, V8 tracks the stack pointer (`sp` register). During the prologue of every function call, V8 runs a stack-guard check:

$$\text{Stack Pointer} < \text{Stack Limit Address}$$

If the stack pointer exceeds the guard boundary, V8 throws a `RangeError`. This prevents the process from crashing due to a hard segmentation fault.

## 2. Web APIs and Libuv: Out-of-Thread Execution

Synchronous execution blocks the thread. To handle asynchronous tasks (network requests, timers, file system I/O), V8 delegates work to external environments. V8 **does not** have built-in APIs for `setTimeout`, `fetch`, or `fs.readFile`.

```
+-------------------------------------------------------------+
|                     BROWSER ENVIRONMENT                     |
|                                                             |
|  +------------------+             +----------------------+  |
|  |    V8 ENGINE     |             |       WEB APIs       |  |
|  |                  |             |                      |  |
|  |  +------------+  |  Delegates  |  - setTimeout        |  |
|  |  | Call Stack |  |------------>|  - fetch (Network)   |  |
|  |  +------------+  |             |  - DOM Events        |  |
|  +--------^---------+             +----------|-----------+  |
|           |                                  |              |
|           | Pushes Callback                  | Completes    |
|           |                                  v              |
|  +--------|---------+             +----------------------+  |
|  |    EVENT LOOP    |<------------|   CALLBACK QUEUES    |  |
|  +------------------+             +----------------------+  |
+-------------------------------------------------------------+
```

### In the Browser (Web APIs)
When you call `fetch()` or `setTimeout()`, the call stack executes the binding code and immediately returns. The actual work is handed off to browser-managed threads (e.g., Chromium's network service thread pool, storage thread pool, or timer utility threads). 
* Once the network card receives the HTTP response, or the hardware timer interrupts, the browser's thread pushes the associated callback into the appropriate queue.

### In Node.js (Libuv)
Node.js uses **Libuv** (written in C) to abstract OS-level asynchronous interfaces. 
* For socket I/O, Libuv uses non-blocking, event-driven OS mechanisms: `epoll` on Linux, `kqueue` on macOS, and `IOCP` on Windows.
* For file I/O, DNS lookups, and user CPU-bound tasks, Libuv maintains a default thread pool of 4 threads (configurable via `UV_THREADPOOL_SIZE`). When a file read completes on a Libuv thread, the result is queued to be processed by the Node.js event loop thread.

## 3. The Callback (Macrotask) vs. Microtask Queues

The event loop monitors two primary types of queues:

### The Macrotask Queue (Callback Queue / Task Queue)
Holds discrete, self-contained operations. Each macrotask runs to completion before any other macrotask is allowed to start.
* **Examples**: `setTimeout`, `setInterval`, `setImmediate` (Node.js), I/O operations, UI rendering events, user interactions (clicks, keypresses).

### The Microtask Queue
Holds short-lived operations that must execute immediately after the current synchronous execution context finishes and before the paint/render pipeline or next macrotask runs.
* **Examples**: `Promise.prototype.then()`, `queueMicrotask()`, `MutationObserver` callbacks, `process.nextTick` (Node.js-specific, sits in its own high-priority microtask-like phase).

## 4. The Precise Event Loop Algorithm

The event loop executes a highly structured loop. Below is the precise operational algorithm of a browser-compliant event loop (conforming to the HTML5 Living Standard):

```
                       +-------------------+
                       |   Start Loop      |
                       +---------+---------+
                                 |
                                 v
                       +-------------------+
                       | Execute Sync Code | <---+
                       | (Drain Call Stack)|     |
                       +---------+---------+     |
                                 |               |
                                 v               |
            +--------------------+--------------------+
            |                                         |
            v                                         v
  [Microtask Queue Empty?]                  [Macrotask Queue Empty?]
       /           \                             /           \
     Yes            No                         Yes            No
     /               \                         /               \
    /      +----------v-----------+           /       +--------v---------+
   /       | Pop & Run Microtask  |          /        | Pop & Run ONE    |
  /        | (Can enqueue more)   |         /         | Macrotask        |
 /         +----------+-----------+        /          +--------+---------+
|                     |                   |                    |
|                     +-------------------+                    |
v                                                              v
+--------------------------------------------------------------+
| Render Steps (Style, Layout, Paint) [If frame budget allows] |
+-------------------------------+------------------------------+
                                |
                                v
                       +-------------------+
                       |    Repeat Loop    |
                       +-------------------+
```

### Algorithmic Steps:

1. **Execute Synchronous Code**: Run the initial script. This step drains the V8 Call Stack down to the global execution context.
2. **Drain Microtask Queue**:
   * Loop through the Microtask Queue.
   * If a microtask schedules *another* microtask, append it to the end of the current queue.
   * **Crucial Behavior**: The loop will keep processing microtasks until the queue is completely empty. If you recursively enqueue microtasks (e.g., via an infinite loop of `Promise.resolve().then(fn)`), you will **starve** the event loop, freezing the application and UI rendering.
3. **Check Rendering Opportunity**:
   * Determine if a repaint is needed (typically every $16.6\text{ms}$ for $60\text{Hz}$ displays, or $8.3\text{ms}$ for $120\text{Hz}$).
   * If yes, run the rendering pipeline:
     1. Fire `requestAnimationFrame` (rAF) callbacks.
     2. Perform Style Calculation, Layout (Reflow), and Paint (Repaint).
4. **Process One Macrotask**:
   * Retrieve the oldest task from the Macrotask Queue.
   * Push its callback onto the V8 Call Stack.
   * Execute it to completion (call stack becomes empty).
5. **Repeat**: Go back to Step 2.

### Visualizing the Promise vs. setTimeout Guarantee

Because microtasks are processed immediately after the call stack clears and before the next macrotask, `Promise.then()` will **always** execute before `setTimeout(fn, 0)`.

```javascript
console.log("1: Sync");

setTimeout(() => {
  console.log("4: Macrotask (setTimeout)");
}, 0);

Promise.resolve().then(() => {
  console.log("3: Microtask (Promise)");
});

console.log("2: Sync End");

// Output Order:
// 1: Sync
// 2: Sync End
// 3: Microtask (Promise)
// 4: Macrotask (setTimeout)
```

## 5. The Node.js Event Loop (Libuv Architecture)

The Node.js event loop differs from the browser's HTML5 event loop. It is managed by Libuv and consists of **6 distinct phases**. Each phase has a FIFO queue of callbacks to execute.

```
   ┌───────────────────────────┐
┌─>│          TIMERS           │ <--- setTimeout(), setInterval()
│  └─────────────┬─────────────┘
│                v
│  ┌───────────────────────────┐
│  │     PENDING CALLBACKS     │ <--- System errors (e.g., TCP connection refused)
│  └─────────────┬─────────────┘
│                v
│  ┌───────────────────────────┐
│  │       IDLE, PREPARE       │ <--- Node.js internal utilities
│  └─────────────┬─────────────┘
│                v
│  ┌───────────────────────────┐
│  │           POLL            │ <--- Incoming I/O, connections, data
│  └─────────────┬─────────────┘
│                v
│  ┌───────────────────────────┐
│  │           CHECK           │ <--- setImmediate()
│  └─────────────┬─────────────┘
│                v
│  ┌───────────────────────────┐
│  │      CLOSE CALLBACKS      │ <--- socket.on('close', ...)
└───────────────┬──────────────┘
                |
          [Exit Loop?]
```

### Detailed Phase Mechanics:

1. **Timers**: Executes callbacks scheduled by `setTimeout()` and `setInterval()`. Libuv checks if the current time matches or exceeds the timer's threshold.
2. **Pending Callbacks**: Executes I/O callbacks deferred from the previous loop iteration (e.g., a TCP socket error like `ECONNREFUSED`).
3. **Idle, Prepare**: Used only internally by Node.js for engine optimization and book-keeping.
4. **Poll**: Retrieves new I/O events. 
   * If the poll queue is not empty, Node.js processes them synchronously.
   * If the poll queue is empty, Node.js checks for scheduled `setImmediate` or timer callbacks. If none exist, the event loop will block here and wait for incoming I/O events to execute them immediately.
5. **Check**: Executes callbacks scheduled by `setImmediate()`. This phase allows developers to execute code immediately after the Poll phase completes.
6. **Close Callbacks**: Executes cleanup callbacks (e.g., `socket.on('close', ...)`).

### Intermediate Microtask Draining in Node.js
In modern Node.js (v11+), the behavior aligns with browsers: **microtasks and `process.nextTick` are executed between every individual callback execution**, not just when switching phases.

* `process.nextTick()` has its own queue (`nextTickQueue`) which has a higher priority than the standard Promise microtask queue (`microtaskQueue`).
* When the call stack finishes executing an operation, Node.js will first drain the `nextTickQueue` completely, and then drain the `microtaskQueue`.
