Title: MongoDB — Document Model vs Relational | Level: 2

### Architectural Divergence: WiredTiger vs. PostgreSQL Storage Engine

To understand why MongoDB scales differently than relational databases like PostgreSQL, we must analyze their underlying storage engines and serialization formats.

```
+---------------------------------------------------------------------------------+
|                                 STORAGE ENGINE                                  |
+---------------------------------------------------------------------------------+
|  POSTGRESQL (Heap-Based Storage)          |  MONGODB (WiredTiger Storage Engine)|
|                                           |                                     |
|  +-------------------------------------+  |  +-------------------------------+  |
|  | Page (8KB)                          |  |  | Block (Allocated dynamically) |  |
|  | +---------------------------------+ |  |  | +---------------------------+ |  |
|  | | Pointer Array (Line Pointers)   | |  |  | | B-Tree Node (Prefix Comp.)| |  |
|  | +---------------------------------+ |  |  | | +-----------------------+ | |  |
|  | | Tuple 1 (Fixed Columns)         | |  |  | | | Key: ObjectId         | | |  |
|  | | Tuple 2 (Fixed Columns)         | |  |  | | | Value: BSON Payload   | | |  |
|  | | Tuple 3 (TOAST Pointer) --------|-|--|--|-|--> [Out-of-line Blob]   | | |  |
|  | +---------------------------------+ |  |  | | +-----------------------+ | |  |
|  +-------------------------------------+  |  | +---------------------------+ |  |
|                                           |  +-------------------------------+  |
+-------------------------------------------+-------------------------------------+
```

#### PostgreSQL (Heap-Based Storage)
PostgreSQL stores data in fixed-size pages (typically 8KB). A table is a collection of heap pages where rows (tuples) are written. If a row exceeds the page limit, PostgreSQL uses **TOAST** (The Oversized-Attribute Storage Technique) to compress and split the attribute into a separate physical table. 
*   **Joins:** Relational integrity relies on primary/foreign key constraints. Resolving relations requires index-scans on multiple tables and merging them in memory (Nested Loop, Hash Join, or Merge Join).
*   **Concurrency:** Uses Multi-Version Concurrency Control (MVCC) by appending new tuple versions directly to the heap, relying on a background process (`VACUUM`) to reclaim space.

#### MongoDB (WiredTiger Storage Engine)
WiredTiger uses a B-Tree-based architecture for both collections and indexes. Data is stored as contiguous blocks on disk, leveraging prefix compression for keys and block compression (Snappy or Zlib) for values.
*   **Locality:** Related data is nested inside a single document. Reading a document is a single sequential I/O operation, bypassing the CPU and memory overhead of multi-table joins.
*   **Concurrency:** WiredTiger uses ticket-based concurrency control (optimistic execution) at the document level. It avoids global or collection-level locks for most operations, allowing high-throughput concurrent writes.

### BSON: Binary JSON Specification and Internal Types

MongoDB stores data as **BSON** (Binary JSON). While JSON is text-based and supports only basic types (string, number, boolean, null, array, object), BSON is a binary representation that encodes length prefixing and explicit type markers, allowing fast traversal without parsing the entire payload.

#### Key BSON Types

##### 1. `ObjectId` (12 Bytes)
A unique identifier designed for distributed systems to avoid central coordination:
*   **4-byte timestamp** (Unix epoch seconds).
*   **5-byte random value** (unique per process/machine).
*   **3-byte counter** (starting with a random value).

##### 2. `Date` (8 Bytes)
64-bit integer representing milliseconds since the Unix epoch. Unlike JSON strings, this enables direct temporal queries and indexing without string parsing.

##### 3. `NumberDecimal` (16 Bytes)
IEEE 754 decimal128 format supporting 34 decimal digits of precision. This is critical for financial calculations to prevent floating-point rounding errors (inherent in BSON's 8-byte `Double`).

##### 4. `Binary` (Variable)
Stores arbitrary binary data (e.g., UUIDs, encrypted payloads, small files) with a subtype byte indicating how the driver should deserialize it.

#### BSON Payload Layout Example
```
\x16\x00\x00\x00                  // Total document size (22 bytes)
\x02                              // Type: String
hello\x00                         // Field name: "hello"
\x06\x00\x00\x00world\x00         // Value: "world" (length + data)
\x00                              // Document terminator
```

### Flexible Schema: Benefits, Hard Realities, and Mitigation

The phrase **"schemaless" is a misnomer**. MongoDB has a **dynamic schema**, meaning the database engine does not enforce structural rules unless configured to do so. The schema is defined by how the application writes and reads data.

#### Benefits
*   **Polymorphic Documents:** A single collection can store documents with different structures (e.g., a payment collection containing credit card transactions, PayPal transactions, and crypto transfers, each with unique fields).
*   **Zero-Downtime Migrations:** Adding a new field does not require executing an expensive `ALTER TABLE` statement that locks the database. The application simply starts writing the new field.

#### Challenges
*   **Application-Level Pollution:** If the application changes a field name from `phoneNumber` to `phone`, older documents remain unchanged. The application must handle both formats, leading to legacy fallback code.
*   **Memory Overhead:** Every document carries its own field names as BSON keys. Storing `"first_name": "John"` in millions of documents duplicates the string `"first_name"` millions of times.

#### Mitigation: JSON Schema Validation
MongoDB allows enforcing schema constraints at the collection level using JSON Schema.

```javascript
// mongosh: Enforcing schema validation on a 'customers' collection
db.createCollection("customers", {
   validator: {
      $jsonSchema: {
         bsonType: "object",
         required: [ "name", "email", "status" ],
         properties: {
            name: {
               bsonType: "string",
               description: "Must be a string and is required"
            },
            email: {
               bsonType: "string",
               pattern: "^.+@.+$",
               description: "Must be a valid email address and is required"
            },
            status: {
               enum: [ "ACTIVE", "INACTIVE", "PENDING" ],
               description: "Can only be one of the allowed enum values"
            },
            balance: {
               bsonType: "decimal",
               description: "Must be a NumberDecimal for financial precision"
            }
         }
      }
   },
   validationAction: "error" // Rejects invalid documents; use "warn" for soft migrations
});
```

### The 16MB Document Limit: Architectural Rationale

MongoDB enforces a strict **16MB limit** on individual BSON documents.

#### Why does this limit exist?
1.  **Memory Allocation Strategy:** WiredTiger allocates memory blocks based on document boundaries. Allowing multi-gigabyte documents would cause massive memory fragmentation and degrade garbage collection performance in Java/Node.js drivers.
2.  **Network Overhead:** MongoDB transfers entire documents over the wire. If a document is 100MB, retrieving it to read a single field wastes network bandwidth and blocks the socket.
3.  **Index Limitations:** Nested arrays inside a massive document can lead to an index entry explosion (multikey indexing), causing write amplification.

*If your data model requires documents larger than 16MB, you must use **GridFS** (which splits files into chunks of 255KB) or redesign your schema to use referencing.*

### MongoDB vs. PostgreSQL: When to Use What

| Dimension | MongoDB 7.x | PostgreSQL 16 |
| :--- | :--- | :--- |
| **Data Model** | Document-oriented (BSON). | Relational (Tables, Rows) + JSONB support. |
| **Write Throughput** | Extremely high. Lock-free document writes, optimized for append-heavy workloads. | Moderate-to-high. Bound by WAL bottlenecks and MVCC write amplification. |
| **Complex Joins** | Supported via `$lookup` and `$graphLookup`, but carries high memory and execution cost. | Native, highly optimized join algorithms with robust optimizer statistics. |
| **Schema Evolution** | Dynamic. Instantaneous. Application-driven. | Strict. Requires `ALTER TABLE` (though modern PG handles many non-blocking migrations). |
| **ACID Transactions** | Multi-document ACID supported since 4.0 (distributed since 4.2). Overhead is high. | Native, zero-overhead ACID. Built for transactional integrity. |
| **Horizontal Scaling** | Native Sharding out-of-the-box (Config servers, Routers, Shards). | Third-party extensions (Citus) or read-replicas. Manual sharding required. |

#### When MongoDB Wins
*   **Unstructured/Semi-structured Data:** Content management systems, product catalogs with dynamic attributes, IoT telemetry with variable sensors.
*   **Rapid Prototyping and Microservices:** When the domain model is evolving rapidly, and deploying database migrations slows down the CI/CD pipeline.
*   **Horizontal Scale Requirements:** When write volume or data size exceeds the limits of a single large physical machine, requiring automatic sharding.

#### When PostgreSQL Wins
*   **Complex Relational Analytics:** Systems requiring deep analytical queries across dozens of tables with complex joins and window functions.
*   **Financial Ledgers & Double-Entry Bookkeeping:** Where absolute transactional integrity, foreign key constraints, and zero-tolerance for orphaned records are non-negotiable.
*   **Hybrid Relational + JSON Workloads:** When 90% of the schema is rigid and relational, but you need to store arbitrary parameters in a `JSONB` column.

### Spring Data MongoDB Implementation

Let's define a robust Spring Data MongoDB configuration and document model representing a polymorphic customer with custom BSON type handling.

#### Maven Dependencies (`pom.xml`)
```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-mongodb</artifactId>
</dependency>
```

#### Document Model (`Customer.java`)
```java
package com.devmastery.mongodb.model;

import lombok.Data;
import lombok.NoArgsConstructor;
import lombok.AllArgsConstructor;
import org.bson.types.Binary;
import org.bson.types.ObjectId;
import org.springframework.data.annotation.Id;
import org.springframework.data.annotation.TypeAlias;
import org.springframework.data.mongodb.core.mapping.Document;
import org.springframework.data.mongodb.core.mapping.Field;
import org.springframework.data.mongodb.core.mapping.FieldType;

import java.math.BigDecimal;
import java.util.Date;
import java.util.List;

@Data
@NoArgsConstructor
@AllArgsConstructor
@Document(collection = "customers")
@TypeAlias("Customer") // Avoids writing full package name to class type field
public class Customer {

    @Id
    private ObjectId id;

    @Field(name = "full_name")
    private String fullName;

    @Field(name = "email")
    private String email;

    @Field(name = "balance", targetType = FieldType.DECIMAL128)
    private BigDecimal balance; // Maps directly to BSON NumberDecimal

    @Field(name = "created_at")
    private Date createdAt;

    @Field(name = "avatar_data")
    private Binary avatarData; // Maps to BSON Binary

    @Field(name = "tags")
    private List<String> tags;
}
```

#### Spring Configuration for Custom Conversions (`MongoConfig.java`)
```java
package com.devmastery.mongodb.config;

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.data.mongodb.config.AbstractMongoClientConfiguration;
import org.springframework.data.mongodb.core.convert.MongoCustomConversions;
import com.mongodb.client.MongoClient;
import com.mongodb.client.MongoClients;

import java.util.ArrayList;
import java.util.List;

@Configuration
public class MongoConfig extends AbstractMongoClientConfiguration {

    @Override
    protected String getDatabaseName() {
        return "devmastery_db";
    }

    @Override
    @Bean
    public MongoClient mongoClient() {
        // Connects to local replica set or standalone
        return MongoClients.create("mongodb://localhost:27017/devmastery_db");
    }

    @Bean
    @Override
    public MongoCustomConversions customConversions() {
        List<Object> converters = new ArrayList<>();
        // Add custom converters here if you need specialized mapping (e.g., JSR-310 dates to legacy dates)
        return new MongoCustomConversions(converters);
    }
}
```
