## WHY
Imagine you are building a `BankAccount` class with a `balance` field. If you make `balance` public, any class in the entire system can write `account.balance = -1000000;`. There is no validation, no security check, no logging. The internal state of the object is completely compromised.

**Without encapsulation, you cannot trust your own system.** Any bug anywhere in a million-line codebase could silently corrupt critical data.

Encapsulation is the defensive shield of OOP. It dictates that:
1. An object should **hide its internal state** (`private` fields)
2. Expose only a **strictly controlled public API** (methods) to interact with that state

By forcing external code to call `withdraw(amount)` instead of directly modifying balance, you can inject validation rules, transaction logging, and synchronization logic into ONE place — without trusting the caller to remember the rules.

Real impact:
- **Security**: Prevents unauthorized direct mutation
- **Maintainability**: Business rules live in one method, not scattered everywhere
- **Testability**: You test the method's behavior, not the field's raw value
- **Flexibility**: You can change the internal representation without breaking callers

## THEORY

### Access Modifiers — The Four Visibility Levels
Java enforces encapsulation at **compile time** using four access levels:

| Modifier | Same Class | Same Package | Subclass (diff pkg) | Everywhere |
|----------|-----------|--------------|---------------------|-----------|
| `private` | ✅ | ❌ | ❌ | ❌ |
| (default/package-private) | ✅ | ✅ | ❌ | ❌ |
| `protected` | ✅ | ✅ | ✅ | ❌ |
| `public` | ✅ | ✅ | ✅ | ✅ |

**Rule of thumb**: Always start with `private`. Widen access only when explicitly needed.

### Getters/Setters — The Common Misconception
A common beginner mistake is generating getters AND setters for every single field. This is the **Anemic Domain Model anti-pattern** — it satisfies the syntax of encapsulation but not the spirit.

```java
// WRONG — Anemic Domain Model
public class BankAccount {
    private double balance;
    public double getBalance() { return balance; }
    public void setBalance(double balance) { this.balance = balance; } // Exposes raw mutation!
}

// RIGHT — Rich Domain Model
public class BankAccount {
    private double balance;
    public double getBalance() { return balance; }
    public void deposit(double amount) { /* validation + logging here */ }
    public void withdraw(double amount) { /* validation + balance check here */ }
    // No setBalance() — nobody should set balance directly!
}
```

### Immutability — The Ultimate Encapsulation
The strongest form of encapsulation is **immutability** — making an object's state completely unchangeable after construction. Benefits:
- **Thread-safe by design** — multiple threads can share it without locking
- **Cacheable** — safe to cache and reuse (e.g., String Pool)
- **Simple to reason about** — state never unexpectedly changes

To make a class immutable:
1. `final` class (prevent subclassing that could break the contract)
2. All fields `private final`
3. No setters
4. Deep-copy mutable arguments in the constructor
5. Deep-copy mutable return values in getters

### The `final` Keyword on Fields
```java
private final String name;        // Cannot be reassigned after constructor
private final List<String> items; // The reference is final, but the LIST CONTENTS can still change!
```
`final` makes the reference immutable, NOT the object it points to. A `final List` can still have elements added to it.

### Java Records (Java 16+) — Instant Immutability
Java 16 introduced `record` — a concise way to create immutable data carriers:
```java
// Old way — 15 lines of boilerplate
public final class Point {
    private final int x, y;
    public Point(int x, int y) { this.x = x; this.y = y; }
    public int x() { return x; }
    public int y() { return y; }
    @Override public boolean equals(Object o) { ... }
    @Override public int hashCode() { ... }
    @Override public String toString() { ... }
}

// New way — 1 line
public record Point(int x, int y) {}
```
Records are automatically: `final`, immutable, have canonical constructor, `equals`, `hashCode`, `toString` generated.

## CODE

### Level 1 — Beginner: BankAccount with Encapsulation
```java
public class BankAccount {
    // 1. Hide the state
    private double balance;
    private String owner;
    private int transactionCount;

    public BankAccount(String owner, double initialBalance) {
        if (initialBalance < 0) throw new IllegalArgumentException("Initial balance cannot be negative");
        this.owner = owner;
        this.balance = initialBalance;
        this.transactionCount = 0;
    }
    
    // 2. Read-only access (getter without setter — balance changes only via methods)
    public double getBalance() { return balance; }
    public String getOwner()   { return owner; }
    public int getTransactionCount() { return transactionCount; }

    // 3. Controlled write access with validation
    public void deposit(double amount) {
        if (amount <= 0) throw new IllegalArgumentException("Deposit amount must be positive");
        this.balance += amount;
        this.transactionCount++;
        System.out.printf("[DEPOSIT] %s: +%.2f. New balance: %.2f%n", owner, amount, balance);
    }

    public void withdraw(double amount) {
        if (amount <= 0) throw new IllegalArgumentException("Withdrawal must be positive");
        if (amount > balance) throw new IllegalStateException("Insufficient funds");
        this.balance -= amount;
        this.transactionCount++;
        System.out.printf("[WITHDRAW] %s: -%.2f. New balance: %.2f%n", owner, amount, balance);
    }

    // transfer() — coordinated operation across two accounts
    public void transfer(BankAccount target, double amount) {
        this.withdraw(amount);
        target.deposit(amount);
    }

    @Override
    public String toString() {
        return String.format("BankAccount[owner=%s, balance=%.2f, txns=%d]",
                             owner, balance, transactionCount);
    }

    public static void main(String[] args) {
        BankAccount alice = new BankAccount("Alice", 1000.0);
        BankAccount bob   = new BankAccount("Bob", 500.0);

        alice.deposit(250.0);
        alice.transfer(bob, 200.0);

        System.out.println(alice); // balance=1050.00, txns=2
        System.out.println(bob);   // balance=700.00, txns=1
    }
}
```

### Level 2 — Building Blocks: Access Modifiers in Practice
```java
package com.devmastery.hr;

public class Employee {
    // private — only this class can access
    private String ssn;
    private double salary;

    // package-private (no modifier) — same package can access
    String department;

    // protected — same package + subclasses
    protected String name;

    // public — accessible from anywhere
    public String employeeId;

    public Employee(String id, String name, String department, double salary, String ssn) {
        this.employeeId = id;
        this.name = name;
        this.department = department;
        this.salary = salary;
        this.ssn = ssn;
    }

    // Method encapsulates salary access with audit
    public double getSalary() {
        auditLog("Salary accessed for: " + employeeId);
        return salary;
    }

    // Controlled mutation
    public void raiseSalary(double percentage) {
        if (percentage < 0 || percentage > 50) {
            throw new IllegalArgumentException("Raise must be between 0% and 50%");
        }
        this.salary *= (1 + percentage / 100);
        auditLog(String.format("Salary raised by %.1f%% for %s", percentage, name));
    }

    private void auditLog(String message) {
        System.out.println("[AUDIT] " + message); // private helper method
    }
}

// In a different package — can only access public and inherited protected members
package com.devmastery.payroll;
import com.devmastery.hr.Employee;

class Payroll extends Employee {
    Payroll(String id, String name) {
        super(id, name, null, 0, null);
    }
    
    void printInfo() {
        System.out.println(name);        // ✅ protected — accessible via inheritance
        System.out.println(employeeId);  // ✅ public
        // System.out.println(ssn);      // ❌ private — compilation error
        // System.out.println(department);// ❌ package-private — different package
    }
}
```

### Level 3 — Intermediate: Immutable Value Objects
```java
import java.util.ArrayList;
import java.util.Collections;
import java.util.List;

// Fully immutable class — thread-safe, cacheable
public final class Money {
    private final double amount;
    private final String currency;

    // Public constructor — sets everything via constructor, no setters
    public Money(double amount, String currency) {
        if (amount < 0) throw new IllegalArgumentException("Amount cannot be negative");
        if (currency == null || currency.isBlank()) throw new IllegalArgumentException("Currency required");
        this.amount = amount;
        this.currency = currency.toUpperCase();
    }

    public double getAmount()   { return amount; }
    public String getCurrency() { return currency; }

    // Returning a NEW object instead of mutating — immutable style!
    public Money add(Money other) {
        if (!this.currency.equals(other.currency)) {
            throw new IllegalArgumentException("Cannot add different currencies");
        }
        return new Money(this.amount + other.amount, this.currency);
    }

    public Money multiply(double factor) {
        return new Money(this.amount * factor, this.currency);
    }

    @Override
    public boolean equals(Object o) {
        if (!(o instanceof Money m)) return false;
        return Double.compare(amount, m.amount) == 0 && currency.equals(m.currency);
    }

    @Override
    public int hashCode() { return java.util.Objects.hash(amount, currency); }

    @Override
    public String toString() { return String.format("%s %.2f", currency, amount); }

    public static void main(String[] args) {
        Money price   = new Money(99.99, "USD");
        Money tax     = new Money(8.00, "USD");
        Money total   = price.add(tax);  // Returns a NEW Money object
        Money doubled = total.multiply(2);

        System.out.println(total);   // USD 107.99
        System.out.println(doubled); // USD 215.98
        // price and tax are UNCHANGED — immutable!
    }
}
```

### Level 4 — Advanced: Reflection Bypass and Java Records
```java
import java.lang.reflect.Field;

public class EncapsulationDeepDive {

    // --- Reflection can bypass private at runtime ---
    static void reflectionBreach() throws Exception {
        BankAccount account = new BankAccount("Alice", 100.0);

        // Get the private field by name
        Field balanceField = BankAccount.class.getDeclaredField("balance");
        balanceField.setAccessible(true); // Bypasses the access modifier check!

        // Read the private field
        double stolen = (double) balanceField.get(account);
        System.out.println("Read private balance: " + stolen); // 100.0

        // Overwrite the private field
        balanceField.set(account, 1_000_000.0);
        System.out.println(account.getBalance()); // 1000000.0 — encapsulation bypassed!
    }

    // --- Java Modules (Java 9+) protect against reflection ---
    // module-info.java:
    // module com.devmastery {
    //     opens com.devmastery.bank; // Only opened packages are accessible via reflection
    // }
    // Without opens, setAccessible() throws InaccessibleObjectException

    // --- Java 16+ Records — ultimate encapsulation shorthand ---
    record ProductSnapshot(String sku, double price, int quantity) {
        // Compact canonical constructor — add validation
        ProductSnapshot {
            if (price < 0) throw new IllegalArgumentException("Price cannot be negative");
            if (quantity < 0) throw new IllegalArgumentException("Quantity cannot be negative");
        }
    }

    public static void main(String[] args) throws Exception {
        // Record: immutable, equals/hashCode/toString auto-generated
        ProductSnapshot p1 = new ProductSnapshot("SKU-001", 29.99, 100);
        ProductSnapshot p2 = new ProductSnapshot("SKU-001", 29.99, 100);

        System.out.println(p1.sku());      // "SKU-001"
        System.out.println(p1.equals(p2)); // true — value-based equality
        System.out.println(p1);            // ProductSnapshot[sku=SKU-001, price=29.99, quantity=100]

        // Records are immutable — no setters exist
        // p1.sku = "HACK"; // ❌ Compilation error

        reflectionBreach();
    }
}
```

### Level 5 — Expert: Thread-Safe Encapsulation with `volatile` and `synchronized`
```java
import java.util.concurrent.atomic.AtomicInteger;
import java.util.concurrent.locks.ReadWriteLock;
import java.util.concurrent.locks.ReentrantReadWriteLock;

// Thread-safe BankAccount — encapsulation extended to concurrency
public class ThreadSafeBankAccount {
    private volatile double balance; // volatile ensures all threads see the latest write
    private final String owner;
    private final AtomicInteger transactionCount = new AtomicInteger(0);

    // ReadWriteLock: multiple threads can READ simultaneously, only one can WRITE
    private final ReadWriteLock lock = new ReentrantReadWriteLock();

    public ThreadSafeBankAccount(String owner, double initialBalance) {
        this.owner = owner;
        this.balance = initialBalance;
    }

    public double getBalance() {
        lock.readLock().lock();
        try {
            return balance; // Multiple threads can read concurrently
        } finally {
            lock.readLock().unlock();
        }
    }

    public void deposit(double amount) {
        lock.writeLock().lock(); // Only ONE thread at a time can write
        try {
            if (amount <= 0) throw new IllegalArgumentException("Invalid amount");
            balance += amount;
            transactionCount.incrementAndGet();
        } finally {
            lock.writeLock().unlock();
        }
    }

    public void withdraw(double amount) {
        lock.writeLock().lock();
        try {
            if (amount > balance) throw new IllegalStateException("Insufficient funds");
            balance -= amount;
            transactionCount.incrementAndGet();
        } finally {
            lock.writeLock().unlock();
        }
    }

    // Atomic compound operation — check-then-act must be in one lock
    public boolean transferIfSufficient(ThreadSafeBankAccount target, double amount) {
        lock.writeLock().lock();
        try {
            if (balance < amount) return false;
            balance -= amount;
            target.deposit(amount);
            return true;
        } finally {
            lock.writeLock().unlock();
        }
    }
}
```

## REAL_WORLD

### Spring Boot JPA: The Anemic Entity Problem
In many production Spring codebases, `@Entity` classes are abused:

```java
// ANTI-PATTERN: Anemic Entity with Lombok @Data
@Entity
@Data  // Generates public setters for EVERY field
public class Order {
    @Id @GeneratedValue private Long id;
    private String status;          // setStatus("ANYTHING") allowed!
    private LocalDateTime createdAt; // setCreatedAt(yesterday) allowed!
    private BigDecimal total;        // setTotal(-9999) allowed!
}

// Any code can do: order.setStatus("INVALID_STATUS_CODE") — no validation!
// Or: order.setTotal(BigDecimal.NEGATIVE_INFINITY) — total nonsense!
```

**The Production Pattern**: Rich domain model with encapsulated state transitions:

```java
@Entity
@Getter  // Only getters via Lombok — no setters!
public class Order {
    @Id @GeneratedValue private Long id;

    @Column(nullable = false)
    private String status = "PENDING";

    @Column(nullable = false, updatable = false)
    private LocalDateTime createdAt = LocalDateTime.now();

    @Column(nullable = false)
    private BigDecimal total;

    // Domain methods with business logic — the ONLY way to change state
    public void confirm() {
        if (!"PENDING".equals(status)) throw new IllegalStateException("Can only confirm PENDING orders");
        this.status = "CONFIRMED";
    }

    public void ship() {
        if (!"CONFIRMED".equals(status)) throw new IllegalStateException("Can only ship CONFIRMED orders");
        this.status = "SHIPPED";
    }

    public void cancel() {
        if ("SHIPPED".equals(status) || "DELIVERED".equals(status)) {
            throw new IllegalStateException("Cannot cancel shipped or delivered orders");
        }
        this.status = "CANCELLED";
    }
}
```
Now `status` can ONLY change through valid business transitions — the entity enforces its own invariants.

## INTERVIEW

**Q: What is the difference between `protected` and package-private?**
A: Package-private (no modifier) allows access only within the same package. `protected` allows the same, plus any subclass in a different package can access the member via inheritance. However, `protected` does NOT allow any random class in a different package to access it — only subclasses.

**Q: Can you achieve 100% true encapsulation in Java?**
A: At compile time, yes — `private` is strictly enforced. At runtime, no — Java Reflection can bypass `private` via `field.setAccessible(true)`. However, Java 9+ Modules can make this extremely difficult by not `open`-ing the package, causing `setAccessible()` to throw `InaccessibleObjectException`.

**Q: What is the Anemic Domain Model anti-pattern?**
A: When a class has `private` fields but exposes `public` setters for every one of them, the encapsulation is only superficial. External code can still freely modify any field via the setter, bypassing all validation. The solution is to expose only semantically meaningful domain methods (`confirm()`, `ship()`, `cancel()`) that enforce business rules before modifying state.

**Q: What is the difference between `final` on a field vs `final` on a class?**
A: `final` on a field means the field's reference cannot be reassigned after construction (but the referenced object's contents can still change). `final` on a class means the class cannot be subclassed (e.g., `String` is final). `final` on a method means it cannot be overridden.

**Q: What are Java Records and how do they help encapsulation?**
A: Records (Java 16+) are compact immutable data carriers. `record Point(int x, int y) {}` automatically generates: private final fields, a canonical constructor, public accessor methods `x()` and `y()`, value-based `equals`/`hashCode`, and `toString`. Records cannot have setters — they are the strongest encapsulation available short of manual immutable classes.

## FEYNMAN CHECK

Explain to a 10-year-old:
> Imagine a piggy bank (the object). The money inside is hidden — you can't reach in and grab it (that's `private`). The ONLY way to interact with it is through the slot on top (that's the controlled `public` method). The slot lets coins in (deposit) but the rules of the piggy bank decide what's valid. You can't put a negative amount in, you can't read the amount without shaking it (calling `getBalance()`). The piggy bank **controls its own state**.
