## WHY
Imagine you are building a `BankAccount` class with a `balance` field. If you make `balance` public, any other class in the entire system can simply write `account.balance = -1000000;`. There is no validation, no security check, and no logging. The internal state of the object is completely compromised.

Encapsulation is the defensive shield of Object-Oriented Programming. It dictates that an object should bundle its state (fields) and hide it from the outside world, exposing only a strictly controlled API (methods) to interact with that state. This is primarily achieved in Java through Access Modifiers (`private`, `protected`, `public`, and package-private). By forcing external code to use a `withdraw(amount)` method instead of modifying the balance directly, you retain absolute control to inject validation rules, logging, or synchronization logic without ever trusting the caller.

## THEORY
### Access Modifiers
Java enforces encapsulation at compile time using four access levels:
1. **`private`**: Visible ONLY within the exact same class. This is the foundation of encapsulation. All fields should default to private.
2. **Package-Private (Default / No Keyword)**: Visible to any class in the exact same package folder.
3. **`protected`**: Visible to classes in the same package, AND any child class that extends this class (even if the child is in a different package).
4. **`public`**: Visible to the entire universe. Use this exclusively for the methods that form the object's API contract.

### Getters and Setters (Mutators)
A common pattern in Java is to make fields `private` and provide `public getField()` and `public setField(val)` methods. 
While this satisfies the syntax of encapsulation, blindly generating getters and setters for every field is an **anti-pattern** known as "Anemic Domain Models". True encapsulation doesn't just hide the field; it hides the *business logic* associated with the field. Instead of `setBalance()`, you should have `deposit()` and `withdraw()`.

## CODE

### Level 1 — Beginner
```java
public class BankAccount {
    // 1. Hide the state (Data Hiding)
    private double balance;
    
    public BankAccount(double initialBalance) {
        this.balance = initialBalance;
    }
    
    // 2. Expose safe read access
    public double getBalance() {
        return balance;
    }
    
    // 3. Expose controlled write access with validation
    public void deposit(double amount) {
        if (amount <= 0) throw new IllegalArgumentException("Invalid amount");
        this.balance += amount;
    }
}
```

### Level 2 — Building Blocks
```java
// Access modifier demonstration
package com.devmastery.core;

public class Employee {
    public String publicName;     // Anywhere
    protected double salary;      // Same package + Subclasses
    int departmentId;             // Same package only
    private String ssn;           // This class only
}
```

### Level 3 — Intermediate
```java
public class ImmutablePerson {
    // True encapsulation often leads to Immutability.
    // By omitting setters entirely and marking fields final, 
    // the state is permanently protected after construction.
    private final String name;
    
    public ImmutablePerson(String name) {
        this.name = name;
    }
    
    public String getName() { return name; }
}
```

### Level 4 — Advanced
```java
import java.lang.reflect.Field;

public class ReflectionBreach {
    public static void main(String[] args) throws Exception {
        BankAccount account = new BankAccount(100);
        
        // Encapsulation is enforced by the compiler, but can be broken at runtime!
        // Java Reflection can bypass 'private' modifiers.
        Field balanceField = BankAccount.class.getDeclaredField("balance");
        balanceField.setAccessible(true); // Disable JVM access checks
        
        // Maliciously overwrite the private field
        balanceField.set(account, 1000000.0);
        System.out.println(account.getBalance()); // Prints 1000000.0
    }
}
```

## REAL_WORLD
In the Spring Framework ecosystem, you will frequently write JPA Entities (e.g., `@Entity class User`). 
Historically, developers used the Lombok `@Data` annotation, which automatically generates `public` getters and setters for every single field. 

```java
// ANTI-PATTERN: Completely breaks encapsulation
@Entity
@Data // Generates setId(), setCreatedAt() etc.
public class Order {
    private Long id;
    private LocalDateTime createdAt;
    private String status;
}
```
This is a massive violation of encapsulation. Why should any code be allowed to call `order.setId(99)` or `order.setCreatedAt(yesterday)`? Those fields should be strictly controlled by the Database and the Hibernate lifecycle. The modern production pattern is to use Lombok's `@Getter` (to read data) but manually write specific domain methods (like `order.markAsShipped()`) to mutate the status, completely omitting the `@Setter` annotation to protect the entity's internal consistency.

## INTERVIEW
**Q: What is the difference between `protected` and package-private (default)?**
A: Package-private (no modifier) makes the member visible only to classes within the exact same package. `protected` does exactly the same thing, but with one addition: it also allows child classes located in *different* packages to access the member via inheritance.

**Q: Can you achieve 100% true encapsulation in Java?**
A: At compile time, yes. At runtime, no. A developer with sufficient privileges can use the Java Reflection API (`Field.setAccessible(true)`) to bypass all access modifiers and read or mutate private fields. (Note: Java 9+ Modules System introduced strict boundaries that make reflection-based breaches much harder by default).
