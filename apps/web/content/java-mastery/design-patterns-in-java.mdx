## WHY

When you build a small application, you can organize code however you like. But when building enterprise systems with millions of lines, ad-hoc structures collapse into spaghetti code. Bugs multiply because developers don't share a common vocabulary for design decisions.

**Design Patterns** are battle-tested solutions to recurring software design problems. The "Gang of Four" (GoF) book (1994) catalogued 23 patterns that still appear in every modern Java codebase. They matter because:

1. **Communication**: "We should use an Observer here" instantly conveys architecture to any engineer — no explanation needed.
2. **Tested solutions**: These patterns survived decades of production use. Inventing your own is almost always inferior.
3. **Framework implementations**: Spring Boot is literally built on patterns — IoC Container (Dependency Injection), Template Method (JdbcTemplate), Proxy (AOP), Observer (ApplicationEvents), Decorator (BeanPostProcessors).
4. **Interview currency**: FAANG and banking interviews constantly test pattern recognition — "Refactor this God class" means Strategy. "How would you notify multiple services?" means Observer.
5. **OCP compliance**: Patterns like Strategy, Observer, and Factory let you add features without modifying existing code — the Open-Closed Principle in practice.

Understanding patterns also helps you recognize **anti-patterns** — the equally important catalog of what NOT to do.

## THEORY

### The Three Categories

| Category | Purpose | Examples |
|----------|---------|---------|
| **Creational** | Object creation mechanisms — hide `new` keyword | Singleton, Factory, Abstract Factory, Builder, Prototype |
| **Structural** | Object composition — how to assemble objects | Adapter, Decorator, Facade, Proxy, Composite, Flyweight |
| **Behavioral** | Object communication — who calls whom | Observer, Strategy, Command, Iterator, Template Method, Chain of Responsibility |

### Pattern Decision Guide

```
PROBLEM                                  PATTERN
────────────────────────────────────     ───────────────────
Need exactly one instance?           →   Singleton
Need to create objects without knowing
  exact class at compile time?       →   Factory / Abstract Factory
Complex object with many optional fields? → Builder
Family of interchangeable algorithms?  → Strategy
Need to notify many objects of events? → Observer
Need to wrap functionality at runtime? → Decorator
Need to adapt incompatible interfaces? → Adapter
Need to simplify a complex subsystem?  → Facade
Need to define skeleton of algorithm
  but defer steps to subclasses?     →   Template Method
Need to undo/redo operations?        →   Command
```

### Key Patterns Deep Dive

**Singleton**: One instance, globally accessible. Thread-safety requires `volatile` + double-checked locking, or better: `enum` Singleton (JVM guarantees single instance) or Spring `@Bean` (let the container manage it).

**Factory Method**: Abstracts object creation. Caller asks "give me a Transport" — factory decides whether to return `Truck` or `Ship` based on runtime config. Enables OCP: adding `Drone` transport means adding a class, not modifying the factory logic.

**Builder**: Replaces telescoping constructors (overloaded constructors with 6+ parameters). Produces immutable objects via fluent chaining. Lombok's `@Builder` generates this at compile time.

**Strategy**: Encapsulates an algorithm family behind an interface. Context holds a reference to the interface, not a concrete class. Algorithm swapped at runtime without changing the context. Replaces entire `if-else` blocks.

**Observer**: Subject maintains a list of observers. When state changes, notifies all observers. Spring's `ApplicationEventPublisher` / `@EventListener` is Observer. Stock tickers, webhooks, UI event systems.

**Decorator**: Wraps an object in another object implementing the same interface. Adds behavior at runtime without inheritance. Java IO: `new BufferedReader(new InputStreamReader(new FileInputStream(...)))` — each wrapper adds buffering/charset-decoding.

**Proxy**: Controls access to another object. Used for: lazy initialization, caching, security checks, logging. Spring AOP creates dynamic proxies for `@Transactional`, `@Cacheable`, `@Secured`. JDK dynamic proxies use `Proxy.newProxyInstance()`.

## CODE

### Level 1 — Beginner
```java
// SINGLETON: Thread-safe implementations

// BAD: Not thread-safe! Two threads can both see null and create two instances
class UnsafeSingleton {
    private static UnsafeSingleton instance;
    private UnsafeSingleton() {}
    public static UnsafeSingleton getInstance() {
        if (instance == null) instance = new UnsafeSingleton();  // RACE CONDITION!
        return instance;
    }
}

// GOOD 1: Double-checked locking with volatile
class SafeSingleton {
    // volatile: prevents CPU reordering of memory writes; ensures fully initialized
    // before any thread can see the reference
    private static volatile SafeSingleton instance;
    private SafeSingleton() {}

    public static SafeSingleton getInstance() {
        if (instance == null) {              // First check (no locking — fast path)
            synchronized (SafeSingleton.class) {
                if (instance == null) {      // Second check (with locking — safe path)
                    instance = new SafeSingleton();
                }
            }
        }
        return instance;
    }
}

// BEST: Enum Singleton — JVM guarantees single instance, serialization-safe
enum DatabaseConfig {
    INSTANCE;

    private final String url = "jdbc:postgresql://localhost:5432/devmastery";
    public String getUrl() { return url; }
}

// Usage:
// DatabaseConfig.INSTANCE.getUrl()  ← thread-safe, no double-checked locking needed

// ENTERPRISE BEST: Let Spring manage it
// @Configuration
// public class Config {
//     @Bean  // @Bean is Singleton by default — Spring manages the single instance
//     public DatabaseConfig databaseConfig() { return new DatabaseConfig(); }
// }
```

### Level 2 — Building Blocks
```java
// FACTORY PATTERN: Create objects without knowing their concrete type

// 1. Common interface
interface Notification {
    void send(String recipient, String message);
}

// 2. Concrete implementations
class EmailNotification implements Notification {
    public void send(String to, String msg) {
        System.out.printf("EMAIL to %s: %s%n", to, msg);
        // In reality: JavaMailSender.send(...)
    }
}

class SMSNotification implements Notification {
    public void send(String to, String msg) {
        System.out.printf("SMS to %s: %s%n", to, msg);
        // In reality: TwilioClient.send(...)
    }
}

class PushNotification implements Notification {
    public void send(String to, String msg) {
        System.out.printf("PUSH to %s: %s%n", to, msg);
        // In reality: FirebaseMessaging.send(...)
    }
}

// 3. Factory — encapsulates creation logic
class NotificationFactory {
    // Returns correct type based on runtime config — caller never knows which class
    public static Notification create(String channel) {
        return switch (channel.toUpperCase()) {
            case "EMAIL" -> new EmailNotification();
            case "SMS"   -> new SMSNotification();
            case "PUSH"  -> new PushNotification();
            default      -> throw new IllegalArgumentException("Unknown channel: " + channel);
        };
    }
}

public class FactoryDemo {
    public static void main(String[] args) {
        // Channel determined by user preference stored in DB — runtime decision
        String preferredChannel = "EMAIL";  // from user.notificationPreference

        Notification notifier = NotificationFactory.create(preferredChannel);
        notifier.send("user@example.com", "Your order has shipped!");
        // Adding "SLACK" channel? Add class + one case. Zero other changes.
    }
}
```

### Level 3 — Intermediate
```java
// BUILDER PATTERN: Complex object construction with validation

public class HttpRequest {
    // All fields are final — Builder produces immutable objects
    private final String method;     // Required
    private final String url;        // Required
    private final java.util.Map<String, String> headers;  // Optional
    private final String body;       // Optional
    private final int timeoutMs;     // Optional, default 5000
    private final int retries;       // Optional, default 3

    // Private constructor — only Builder can create instances
    private HttpRequest(Builder builder) {
        this.method = builder.method;
        this.url = builder.url;
        this.headers = java.util.Collections.unmodifiableMap(builder.headers);
        this.body = builder.body;
        this.timeoutMs = builder.timeoutMs;
        this.retries = builder.retries;
    }

    public String getMethod() { return method; }
    public String getUrl()    { return url; }

    // Fluent Builder (static nested class)
    public static class Builder {
        // Required fields — set via constructor
        private final String method;
        private final String url;

        // Optional with defaults
        private java.util.Map<String, String> headers = new java.util.HashMap<>();
        private String body = null;
        private int timeoutMs = 5_000;
        private int retries = 3;

        public Builder(String method, String url) {
            if (method == null || url == null) throw new IllegalArgumentException("method/url required");
            this.method = method.toUpperCase();
            this.url = url;
        }

        public Builder header(String key, String value) {
            this.headers.put(key, value);
            return this;  // Return this — enables chaining
        }
        public Builder body(String body)       { this.body = body;           return this; }
        public Builder timeout(int ms)         { this.timeoutMs = ms;        return this; }
        public Builder retries(int retries)    { this.retries = retries;     return this; }

        public HttpRequest build() {
            // Validation before object creation
            if (method.equals("GET") && body != null) {
                throw new IllegalStateException("GET requests cannot have a body");
            }
            return new HttpRequest(this);
        }
    }

    @Override
    public String toString() {
        return method + " " + url + " (timeout=" + timeoutMs + "ms, retries=" + retries + ")";
    }

    public static void main(String[] args) {
        // BEFORE Builder — telescoping constructor nightmare:
        // new HttpRequest("GET", "/api/users", null, null, 3000, 5, false, null, true);

        // AFTER Builder — readable and self-documenting:
        HttpRequest request = new HttpRequest.Builder("GET", "https://api.example.com/users")
            .header("Authorization", "Bearer token123")
            .header("Accept", "application/json")
            .timeout(3_000)
            .retries(5)
            .build();

        System.out.println(request);
        // GET https://api.example.com/users (timeout=3000ms, retries=5)
    }
}
```

### Level 4 — Advanced
```java
// OBSERVER + STRATEGY: Event-driven architecture
import java.util.*;
import java.util.function.Consumer;

// OBSERVER PATTERN: Event system
interface OrderEventListener {
    void onOrderPlaced(Order order);
}

record Order(long id, String customerId, double total) {}

class EmailConfirmationListener implements OrderEventListener {
    public void onOrderPlaced(Order order) {
        System.out.printf("EMAIL: Confirming order #%d to customer %s%n",
            order.id(), order.customerId());
    }
}

class InventoryUpdateListener implements OrderEventListener {
    public void onOrderPlaced(Order order) {
        System.out.printf("INVENTORY: Reserving stock for order #%d%n", order.id());
    }
}

class FraudDetectionListener implements OrderEventListener {
    public void onOrderPlaced(Order order) {
        if (order.total() > 10_000) {
            System.out.printf("FRAUD ALERT: High-value order #%d ($%.2f) flagged%n",
                order.id(), order.total());
        }
    }
}

class OrderService {
    private final List<OrderEventListener> listeners = new ArrayList<>();

    public void addListener(OrderEventListener listener) {
        listeners.add(listener);
    }

    public void placeOrder(Order order) {
        System.out.println("Placing order #" + order.id());
        // Notify ALL listeners — adding new behavior = add listener, NOT modify this class
        listeners.forEach(l -> l.onOrderPlaced(order));
    }
}

// STRATEGY PATTERN: Pluggable discount algorithms
interface DiscountStrategy {
    double apply(double originalPrice);
    String name();
}

public class PatternsDemo {
    public static void main(String[] args) {
        // Observer
        OrderService service = new OrderService();
        service.addListener(new EmailConfirmationListener());
        service.addListener(new InventoryUpdateListener());
        service.addListener(new FraudDetectionListener());

        service.placeOrder(new Order(1001, "user-42", 250.00));
        service.placeOrder(new Order(1002, "user-07", 15_000.00));

        // Strategy implementations
        DiscountStrategy noDiscount = new DiscountStrategy() {
            public double apply(double p) { return p; }
            public String name() { return "No discount"; }
        };
        DiscountStrategy vipDiscount = new DiscountStrategy() {
            public double apply(double p) { return p * 0.80; }  // 20% off
            public String name() { return "VIP 20% off"; }
        };
        DiscountStrategy flashSale = new DiscountStrategy() {
            public double apply(double p) { return Math.min(p, p - 50); }  // $50 off
            public String name() { return "Flash Sale -$50"; }
        };

        double price = 200.0;
        for (DiscountStrategy strategy : new DiscountStrategy[]{noDiscount, vipDiscount, flashSale}) {
            System.out.printf("%s: $%.2f → $%.2f%n",
                strategy.name(), price, strategy.apply(price));
        }
    }
}
```

### Level 5 — Expert
```java
// DECORATOR + PROXY PATTERNS: Dynamic behavior injection
import java.lang.reflect.*;
import java.util.function.*;

// DECORATOR: Add behavior to objects at runtime without inheritance
interface DataProcessor {
    String process(String data);
}

class BaseProcessor implements DataProcessor {
    public String process(String data) { return data.trim(); }
}

// Decorators wrap DataProcessor — add behavior before/after
class EncryptionDecorator implements DataProcessor {
    private final DataProcessor wrapped;
    public EncryptionDecorator(DataProcessor dp) { this.wrapped = dp; }

    public String process(String data) {
        String processed = wrapped.process(data);  // Call wrapped first
        return "[ENC:" + new StringBuilder(processed).reverse() + "]";  // Simulated encryption
    }
}

class CompressionDecorator implements DataProcessor {
    private final DataProcessor wrapped;
    public CompressionDecorator(DataProcessor dp) { this.wrapped = dp; }

    public String process(String data) {
        String processed = wrapped.process(data);
        return "[ZIPPED:" + processed.length() + "bytes]";  // Simulated compression
    }
}

class LoggingDecorator implements DataProcessor {
    private final DataProcessor wrapped;
    public LoggingDecorator(DataProcessor dp) { this.wrapped = dp; }

    public String process(String data) {
        System.out.println("Processing: " + data.substring(0, Math.min(20, data.length())));
        String result = wrapped.process(data);
        System.out.println("Completed: " + result);
        return result;
    }
}

// JDK DYNAMIC PROXY: Spring AOP under the hood
interface UserRepository {
    String findById(long id);
}

public class DecoratorProxyDemo {

    // Create proxy that adds timing to ANY interface implementation
    @SuppressWarnings("unchecked")
    static <T> T withTiming(T target, Class<T> iface) {
        return (T) Proxy.newProxyInstance(
            iface.getClassLoader(),
            new Class[]{iface},
            (proxy, method, args) -> {
                long start = System.nanoTime();
                try {
                    return method.invoke(target, args);
                } finally {
                    System.out.printf("[TIMING] %s: %d μs%n",
                        method.getName(), (System.nanoTime() - start) / 1000);
                }
            }
        );
    }

    public static void main(String[] args) {
        // Decorator pattern — stack decorators like Java IO streams
        DataProcessor pipeline =
            new LoggingDecorator(
                new EncryptionDecorator(
                    new CompressionDecorator(
                        new BaseProcessor()
                    )
                )
            );

        pipeline.process("  Hello World  ");
        // Processes: BaseProcessor → Compress → Encrypt → Log

        // Dynamic Proxy — adds timing without modifying UserRepository
        UserRepository repo = id -> "User#" + id;
        UserRepository timedRepo = withTiming(repo, UserRepository.class);

        timedRepo.findById(42);
        // [TIMING] findById: 45 μs
        // This is how Spring @Cacheable, @Transactional, @Secured work!
    }
}
```

### Spring Boot Patterns in Action
```java
// 1. FACTORY: Spring uses FactoryBean for complex bean creation
@Configuration
public class AppConfig {
    @Bean  // Spring manages this as Singleton — enterprise Singleton pattern
    public ObjectMapper objectMapper() {
        return new ObjectMapper()
            .configure(DeserializationFeature.FAIL_ON_UNKNOWN_PROPERTIES, false);
    }
}

// 2. BUILDER: Spring's ResponseEntity and UriComponentsBuilder
@RestController
public class UserController {

    @GetMapping("/users/{id}")
    public ResponseEntity<User> getUser(@PathVariable Long id) {
        // Builder pattern — ResponseEntity uses builder for clean HTTP response construction
        return ResponseEntity.ok()
            .header("X-Cache", "MISS")
            .contentType(MediaType.APPLICATION_JSON)
            .body(new User(id, "Alice"));
    }
}

// 3. OBSERVER: Spring ApplicationEvents — decoupled event publishing
@Service
public class OrderService {

    @Autowired
    private ApplicationEventPublisher eventPublisher;

    public void placeOrder(Order order) {
        // save to DB
        // Then publish event — OrderService doesn't know who handles it
        eventPublisher.publishEvent(new OrderPlacedEvent(this, order));
    }
}

// Any number of listeners can react — zero coupling to OrderService
@Component
public class WelcomeEmailListener {
    @EventListener
    public void onOrderPlaced(OrderPlacedEvent event) {
        emailService.sendConfirmation(event.getOrder().getCustomerEmail());
    }
}

// 4. TEMPLATE METHOD: JdbcTemplate / RestTemplate
// Spring JdbcTemplate eliminates boilerplate while letting you customize SQL
@Repository
public class UserRepo {
    @Autowired JdbcTemplate jdbc;

    public User findById(long id) {
        // Template method: JdbcTemplate handles connection, PreparedStatement, ResultSet
        // You only provide: SQL + RowMapper (the customizable step)
        return jdbc.queryForObject(
            "SELECT * FROM users WHERE id = ?",
            (rs, rowNum) -> new User(rs.getLong("id"), rs.getString("name")),
            id
        );
    }
}
```

## INTERVIEW

### Q1: What is the difference between Factory and Abstract Factory patterns?
**Difficulty:** Medium | **Companies:** Google, Amazon

❌ **Weak:** "Factory creates objects, Abstract Factory creates families of objects."

✅ **Strong:** "Factory Method creates one product via a single method — the subclass decides the concrete type. Abstract Factory creates a **family of related products** across multiple methods. For example, a `UIThemeFactory` interface might have `createButton()`, `createDialog()`, and `createInput()` — a `DarkThemeFactory` implements all three to create dark-styled components that work together, while `LightThemeFactory` creates light-styled components. The key is that Abstract Factory ensures related products are used together (they come from the same factory)."

---

### Q2: How do you make a Singleton thread-safe in Java?
**Difficulty:** Hard | **Companies:** All

❌ **Weak:** "Use synchronized."

✅ **Strong:** "There are 4 approaches: (1) **Enum Singleton** — simplest, JVM-guaranteed single instance, serialization-safe, reflection-resistant. `enum Config { INSTANCE; }` (2) **Double-checked locking** with `volatile` — lazy initialization, fast after first creation. Requires `volatile` to prevent CPU reordering. (3) **Initialization-on-demand holder** (Bill Pugh idiom) — JVM class loading guarantees thread safety without `synchronized`: `private static class Holder { static final X INSTANCE = new X(); }`. (4) **Spring @Bean** — the IoC container manages singleton lifecycle. In Spring apps, never roll your own Singleton — let the container do it."

---

### Q3: When would you use Decorator over inheritance?
**Difficulty:** Medium | **Companies:** Netflix, Uber

❌ **Weak:** "When you want to add behavior without changing the class."

✅ **Strong:** "Use Decorator when: (1) You need to add behavior at **runtime** based on configuration (inheritance is static — decided at compile time). (2) You need to **combine multiple behaviors** independently — with inheritance, N behaviors requires 2^N subclasses; with Decorator, you simply stack wrappers. (3) You're working with classes from external libraries you can't modify. Classic example: Java IO — `new BufferedReader(new InputStreamReader(socket.getInputStream()))` stacks buffering + charset conversion without inheritance. Spring AOP creates Decorator-like proxies for `@Transactional` and `@Cacheable` at runtime."

---

### Q4: How does the Observer pattern differ from Publisher-Subscriber?
**Difficulty:** Hard | **Companies:** Palantir, Stripe

❌ **Weak:** "They're basically the same thing."

✅ **Strong:** "Observer is a **tightly coupled** pattern — the Subject holds direct references to Observer instances and calls them directly. Observers know who publishes. Publisher-Subscriber adds a **message broker** (event bus, topic) between publishers and subscribers — publishers fire events to a topic, subscribers listen to topics. Neither side knows about the other. Spring ApplicationEvents is pure Observer (ApplicationEventPublisher holds listener references). Kafka/RabbitMQ is Publisher-Subscriber (producers write to topics, consumers subscribe independently). Use Observer for in-process eventing; Pub-Sub for distributed, decoupled, or cross-service communication."

---

### Q5: Explain how Spring AOP implements the Proxy pattern internally.
**Difficulty:** Hard | **Companies:** Pivotal, Goldman Sachs

❌ **Weak:** "Spring creates proxy objects that intercept method calls."

✅ **Strong:** "When you annotate a Spring bean method with `@Transactional`, Spring creates a **proxy** wrapping your bean. There are two proxy mechanisms: (1) **JDK Dynamic Proxy** — used when the bean implements at least one interface. `Proxy.newProxyInstance()` creates a runtime class implementing the same interfaces; `InvocationHandler.invoke()` intercepts calls, adds transaction management, then delegates to the real bean. (2) **CGLIB Proxy** — used for concrete classes without interfaces. CGLIB generates a subclass at runtime that overrides all methods to add cross-cutting behavior. The bean you `@Autowire` is actually the proxy, not the real class — which is why `@Transactional` methods called from within the same class (i.e., `this.method()`) bypass the proxy and lose transaction management."

## FEYNMAN CHECK

Design patterns are like IKEA furniture assembly instructions — proven ways to put standard pieces together. Instead of inventing a new way to assemble a chair every time, you follow the tested blueprint. **Singleton** = there's only one master key to the building. **Factory** = you tell the factory "I need a vehicle" and it decides car or truck based on your needs. **Observer** = you subscribe to email notifications; when the store restocks, you're automatically notified. **Strategy** = you have a GPS that can route via fastest, shortest, or scenic — you swap the algorithm without changing the GPS itself. **Decorator** = you add whipped cream, sprinkles, and syrup to coffee — each topping is a decorator that wraps and adds to the previous layer.
