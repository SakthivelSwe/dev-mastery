## WHY
When you build a small application, you can organize your code however you like. But when you build an enterprise system with millions of lines of code, ad-hoc structures collapse into spaghetti code.
**Design Patterns** are battle-tested solutions to common software design problems. Introduced by the "Gang of Four" (GoF), these patterns provide a universal vocabulary for developers. When a developer says, "I used a Singleton factory," every other Java engineer in the world instantly understands the architecture without reading a single line of code.

## THEORY
### The Three Categories
1. **Creational Patterns:** Deal with object creation mechanisms, hiding the creation logic rather than instantiating objects directly using `new` (e.g., Singleton, Factory, Builder).
2. **Structural Patterns:** Deal with object composition, ensuring that if one part of a system changes, the entire structure doesn't collapse (e.g., Adapter, Decorator, Facade).
3. **Behavioral Patterns:** Deal with communication and assignment of responsibilities between objects (e.g., Observer, Strategy, Command).

## CODE

### Level 1 — Beginner (The Singleton Pattern)
*Problem: You need exactly ONE instance of a class across the entire application (e.g., a Database Connection Manager or a Configuration Logger).*

```java
public class DatabaseConfig {
    // 1. The single instance is static and private!
    private static DatabaseConfig instance;
    private String databaseUrl;

    // 2. The constructor is PRIVATE! No one can call `new DatabaseConfig()`!
    private DatabaseConfig() {
        this.databaseUrl = "jdbc:postgresql://localhost:5432/mydb";
    }

    // 3. The global access method
    public static DatabaseConfig getInstance() {
        if (instance == null) {
            // Lazy initialization: Only create it when first requested
            instance = new DatabaseConfig();
        }
        return instance;
    }
    
    public String getDatabaseUrl() { return databaseUrl; }
}

// Usage:
// DatabaseConfig config = DatabaseConfig.getInstance();
```
*(Warning: The simple Singleton above is NOT thread-safe. In enterprise Java, we usually rely on Spring Boot's `@Bean` annotations to manage Singletons securely).*

### Level 2 — Building Blocks (The Factory Pattern)
*Problem: You have multiple subclasses (e.g., `Car`, `Truck`, `Motorcycle`). You want to create them without hardcoding the exact class names all over your codebase.*

```java
// 1. The Common Interface
public interface Transport {
    void deliver();
}

public class Truck implements Transport {
    public void deliver() { System.out.println("Delivering by land in a box."); }
}

public class Ship implements Transport {
    public void deliver() { System.out.println("Delivering by sea in a container."); }
}

// 2. The Factory Class
public class TransportFactory {
    // The Factory Method encapsulates the `new` keyword!
    public static Transport createTransport(String type) {
        if (type == null) return null;
        
        switch (type.toLowerCase()) {
            case "truck":
                return new Truck();
            case "ship":
                return new Ship();
            default:
                throw new IllegalArgumentException("Unknown transport type");
        }
    }
}

// Usage:
// Transport myTransport = TransportFactory.createTransport("ship");
// myTransport.deliver();
```

### Level 3 — Intermediate (The Builder Pattern)
*Problem: You have a class with 15 fields. Writing a constructor `new User("John", null, null, 25, null, true)` is unreadable and error-prone.*

The Builder pattern separates the construction of a complex object from its representation.

```java
public class User {
    private final String firstName; // Required
    private final String lastName;  // Required
    private final int age;          // Optional
    private final String phone;     // Optional

    // Private constructor! Only the Builder can call this.
    private User(UserBuilder builder) {
        this.firstName = builder.firstName;
        this.lastName = builder.lastName;
        this.age = builder.age;
        this.phone = builder.phone;
    }

    // Static nested Builder class
    public static class UserBuilder {
        private final String firstName;
        private final String lastName;
        private int age = 0;
        private String phone = "";

        public UserBuilder(String firstName, String lastName) {
            this.firstName = firstName;
            this.lastName = lastName;
        }

        // Methods return 'this' to allow method chaining!
        public UserBuilder age(int age) {
            this.age = age;
            return this;
        }

        public UserBuilder phone(String phone) {
            this.phone = phone;
            return this;
        }

        public User build() {
            return new User(this);
        }
    }
}

// Usage:
// User user = new User.UserBuilder("John", "Doe")
//                 .age(30)
//                 .phone("555-1234")
//                 .build();
```

### Level 4 — Advanced (The Strategy Pattern)
*Problem: You have an E-commerce app. A user can pay with Credit Card, PayPal, or Crypto. Writing a massive `if-else` block inside the `CheckoutService` violates the Open-Closed Principle.*

The Strategy Pattern defines a family of algorithms, encapsulates each one, and makes them interchangeable at runtime!

```java
// 1. The Strategy Interface
public interface PaymentStrategy {
    void pay(int amount);
}

// 2. Concrete Strategies
public class CreditCardStrategy implements PaymentStrategy {
    private String cardNumber;
    public CreditCardStrategy(String cardNumber) { this.cardNumber = cardNumber; }
    
    public void pay(int amount) {
        System.out.println(amount + " paid with Credit Card.");
    }
}

public class PayPalStrategy implements PaymentStrategy {
    private String email;
    public PayPalStrategy(String email) { this.email = email; }
    
    public void pay(int amount) {
        System.out.println(amount + " paid using PayPal.");
    }
}

// 3. The Context Class (The one that uses the strategy)
public class ShoppingCart {
    // The cart doesn't care HOW the payment is processed.
    // It just holds a reference to the Interface!
    private PaymentStrategy paymentStrategy;

    // We inject the strategy dynamically!
    public void setPaymentStrategy(PaymentStrategy paymentStrategy) {
        this.paymentStrategy = paymentStrategy;
    }

    public void checkout(int amount) {
        paymentStrategy.pay(amount);
    }
}

// Usage:
// ShoppingCart cart = new ShoppingCart();
// cart.setPaymentStrategy(new PayPalStrategy("john@example.com"));
// cart.checkout(100);
```

## REAL_WORLD
**The Java Standard Library**
You have likely used Design Patterns without even realizing it:
- **Iterator Pattern:** Every time you use a `for-each` loop on an `ArrayList` or `HashSet`, you are using Java's internal `Iterator` pattern to traverse elements without exposing the underlying data structure.
- **Decorator Pattern:** Java IO streams (`new BufferedReader(new FileReader("file.txt"))`). The `BufferedReader` dynamically adds caching behavior to the `FileReader` at runtime.
- **Observer Pattern:** ActionListeners in Java Swing or Spring ApplicationEvents. A component shouts "I was clicked!", and any registered observers magically react.

## INTERVIEW
**Q: I implemented a Singleton in Java exactly like your Level 1 example. The interviewer said it will break in a multithreaded environment. Why?**
A: **The Race Condition.** If Thread A and Thread B both call `getInstance()` at the exact same millisecond, they both see `if (instance == null)` as `true`. They both execute `new DatabaseConfig()`, creating TWO distinct instances and destroying the Singleton!
To fix this, you must use **Double-Checked Locking** with the `volatile` keyword:
```java
private static volatile DatabaseConfig instance;
public static DatabaseConfig getInstance() {
    if (instance == null) {
        synchronized (DatabaseConfig.class) {
            if (instance == null) instance = new DatabaseConfig();
        }
    }
    return instance;
}
```
*(Alternatively, just use an `enum` for the Singleton, which the JVM guarantees is strictly thread-safe!)*
