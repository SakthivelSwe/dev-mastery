## WHY

In C/C++, the entire program is compiled into a single binary. Every function, class, and library is statically linked before execution. If you want to add a plugin, you recompile everything.

Java takes a radically different approach: **dynamic class loading on demand**. The JVM loads `.class` files lazily — only when first referenced. This enables five powerful capabilities that define the Java ecosystem:

1. **Multi-application isolation**: Tomcat runs 50 web apps in one JVM. Each app has its own ClassLoader — they can't see each other's classes even if they both have `com.example.Config`. Without ClassLoaders, apps would conflict.
2. **Plugin architectures**: IntelliJ IDEA plugins, OSGi bundles, and Maven plugins each use custom ClassLoaders to load and unload code at runtime without restarting.
3. **Zero-downtime hot-swap**: Deploy a new version of a service by creating a new ClassLoader, loading the new code, atomically swapping the reference. Old ClassLoader becomes GC-eligible.
4. **Framework magic**: Spring, Hibernate, and AspectJ use ClassLoaders to manipulate bytecode at load-time (instrumentation), enabling `@Transactional`, `@Cacheable`, and lazy-loading proxies.
5. **Security boundary**: The Bootstrap ClassLoader loads `java.lang.String`. Without the delegation model, a malicious jar could provide a fake `String` class that steals data. The delegation hierarchy prevents this.

Without understanding ClassLoaders, you cannot diagnose `ClassNotFoundException` vs `NoClassDefFoundError`, understand why `instanceof` returns false for the "same" class, or architect a plugin system.

## THEORY

### ClassLoader Hierarchy

```
Bootstrap ClassLoader (C++ native code)
    │  Loads: java.lang.*, java.util.*, core JDK modules
    │  Source: $JAVA_HOME/lib/rt.jar (Java 8) or $JAVA_HOME/jmods/ (Java 9+)
    │  Java ref: String.class.getClassLoader() == null  ← no Java object!
    │
    ▼
Platform ClassLoader (Java 9+, formerly Extension ClassLoader)
    │  Loads: javax.*, java.sql.*, security modules
    │  Source: $JAVA_HOME/lib/ext/, module path
    │
    ▼
Application ClassLoader (System ClassLoader)
    │  Loads: YOUR code, third-party jars
    │  Source: CLASSPATH, -cp argument, Maven/Gradle classpath
    │  Java ref: ClassLoader.getSystemClassLoader()
    │
    ▼
Custom ClassLoaders (optional)
    │  Examples: URLClassLoader, Tomcat WebAppClassLoader
    │  Created by: frameworks, application servers, plugin systems
```

### The Parent-Delegation Model (Algorithm)

When `ClassLoader CL` is asked to load class `Foo`:

```
1. Check CL's cache: if already loaded, return cached Class<?> object
2. Delegate to parent: parent.loadClass("Foo")
   a. Parent delegates to ITS parent (recursion up the chain)
   b. Bootstrap ClassLoader tries first: can it find Foo in rt.jar?
   c. If yes → return it. ALL ClassLoaders see this class.
   d. If no → fall back to Platform → Application → Custom
3. If ALL parents fail: CL tries to load it itself via findClass()
4. If CL fails: throw ClassNotFoundException
```

**Why delegation?** Security. `java.lang.String` is loaded by Bootstrap. Even if your malicious `evil.jar` has a class named `java.lang.String`, the Bootstrap ClassLoader finds the real String first — your fake class is never loaded.

**The Golden Rule:**
- Child ClassLoader **CAN** see parent's classes (upward visibility)
- Parent ClassLoader **CANNOT** see child's classes (no downward visibility)

### Class Loading Phases

| Phase | What Happens |
|-------|-------------|
| **Loading** | Read `.class` file bytes; create `java.lang.Class` object in Metaspace |
| **Verification** | Bytecode Verifier checks: no invalid memory access, valid bytecode format |
| **Preparation** | `static` fields allocated in Metaspace; set to defaults (0, null, false) |
| **Resolution** | Replace symbolic references in constant pool with direct memory pointers |
| **Initialization** | Run `<clinit>` (static initializers + static field assignments) |

**Initialization is lazy**: `<clinit>` runs only when the class is first **actively used** (first `new`, first `static` call, first static field access). Reading a class reference doesn't trigger it.

### ClassNotFoundException vs NoClassDefFoundError

| Error | When | Cause |
|-------|------|-------|
| `ClassNotFoundException` | `Class.forName("...")`, `ClassLoader.loadClass(...)` | Class not found on classpath at runtime |
| `NoClassDefFoundError` | Normal class reference | Class WAS available at compile time, NOT found at runtime |

`NoClassDefFoundError` is trickier — it usually means the jar is on the compile classpath but not the runtime classpath (common with build tools misconfiguration).

### Thread Context ClassLoader — Breaking the Hierarchy

**The SPI Problem**: `java.sql.DriverManager` (Bootstrap-loaded) needs to find `org.postgresql.Driver` (App-loaded). But Bootstrap can't see App classes!

**Solution**: Thread Context ClassLoader (TCCL) — each thread carries a reference to a ClassLoader, set by the creator:

```java
// Any code can do:
ClassLoader tccl = Thread.currentThread().getContextClassLoader();
// Use tccl to load classes that the calling code's ClassLoader can't see
```

Spring, JDBC, JNDI all use TCCL to "reach down" to application ClassLoaders from framework code.

### Metaspace and Class Unloading

Classes live in **Metaspace** (off-heap in Java 8+). A class can be **unloaded** (GC-eligible) when:
1. All instances of the class are GC'd
2. The `ClassLoader` that loaded it has no remaining references
3. The `java.lang.Class` object itself has no references

This is how hot-swap and plugin unloading work: discard the ClassLoader reference → GC removes the ClassLoader and ALL its associated classes from Metaspace.

## CODE

### Level 1 — Beginner
```java
// Inspecting the ClassLoader hierarchy
public class ClassLoaderHierarchy {
    public static void main(String[] args) {
        // YOUR class — loaded by Application ClassLoader
        ClassLoader appLoader = ClassLoaderHierarchy.class.getClassLoader();
        System.out.println("App ClassLoader: " + appLoader);
        // sun.misc.Launcher$AppClassLoader@...  or  jdk.internal.loader.ClassLoaders$AppClassLoader

        // Platform ClassLoader (Java 9+) — parent of App ClassLoader
        ClassLoader platformLoader = appLoader.getParent();
        System.out.println("Platform ClassLoader: " + platformLoader);
        // jdk.internal.loader.ClassLoaders$PlatformClassLoader

        // Bootstrap ClassLoader — parent of Platform, but returns null (it's native C++)
        ClassLoader bootstrapLoader = platformLoader.getParent();
        System.out.println("Bootstrap ClassLoader: " + bootstrapLoader);  // null!

        // Core classes loaded by Bootstrap — always return null
        System.out.println("String ClassLoader: " + String.class.getClassLoader());     // null
        System.out.println("ArrayList ClassLoader: " + java.util.ArrayList.class.getClassLoader()); // null

        // Third-party library — loaded by App ClassLoader
        // System.out.println(com.fasterxml.jackson.ObjectMapper.class.getClassLoader());

        // Walk the entire hierarchy
        ClassLoader cl = ClassLoaderHierarchy.class.getClassLoader();
        while (cl != null) {
            System.out.println("Level: " + cl.getClass().getName());
            cl = cl.getParent();
        }
        System.out.println("Level: Bootstrap (null)");
    }
}
```

### Level 2 — Building Blocks
```java
import java.net.*;

// URLClassLoader: loading classes from external JAR at runtime
public class DynamicClassLoading {

    @SuppressWarnings("deprecation")
    public static void main(String[] args) throws Exception {
        // Load a class from an external JAR at runtime — no compile-time dependency!
        URL jarUrl = new URL("file:///opt/plugins/analytics-plugin.jar");
        URLClassLoader pluginLoader = new URLClassLoader(
            new URL[]{jarUrl},
            DynamicClassLoading.class.getClassLoader()  // Parent = App ClassLoader
        );

        // Dynamically load the class — no compile-time import needed!
        Class<?> pluginClass = pluginLoader.loadClass("com.analytics.ReportGenerator");

        // Invoke via reflection or via a known interface
        Object plugin = pluginClass.getDeclaredConstructor().newInstance();
        pluginClass.getMethod("generate", String.class).invoke(plugin, "Q4-2024");

        // CRITICAL: Close the ClassLoader when done
        // This allows GC to unload the classes from Metaspace
        pluginLoader.close();

        // Demonstrate: same class loaded by different ClassLoaders = different Class objects!
        URL jar2 = new URL("file:///opt/plugins/analytics-plugin-v2.jar");
        URLClassLoader loaderV2 = new URLClassLoader(new URL[]{jar2}, Thread.currentThread().getContextClassLoader());

        Class<?> v1 = pluginLoader.loadClass("com.analytics.ReportGenerator");
        Class<?> v2 = loaderV2.loadClass("com.analytics.ReportGenerator");

        System.out.println("Same class? " + (v1 == v2));  // false!
        // Even though same name: loaded by different ClassLoaders → different Class objects
        // instanceof would also return false!
    }
}
```

### Level 3 — Intermediate
```java
// Class.forName vs ClassLoader.loadClass: the difference
public class ClassLoadingModes {

    public static void main(String[] args) throws Exception {
        String className = "com.example.HeavyInitializer";

        // DIFFERENCE 1: Class.forName() — loads AND initializes (runs <clinit>)
        // static initializers run immediately
        Class<?> forNameClass = Class.forName(className);
        // <clinit> ran: all static initializers and static field assignments executed

        // Class.forName with explicit ClassLoader — still initializes
        Class<?> forNameWithLoader = Class.forName(className, true, Thread.currentThread().getContextClassLoader());

        // DIFFERENCE 2: ClassLoader.loadClass() — loads but does NOT initialize
        // <clinit> deferred until class is first actively used
        Class<?> loadedClass = ClassLoaderHierarchy.class.getClassLoader().loadClass(className);
        // <clinit> NOT yet run! Static fields are at default values.

        // When is initialization triggered? First:
        //   - new Instance()
        //   - static field/method access
        //   - Class.forName() with initialize=true
        Object instance = loadedClass.getDeclaredConstructor().newInstance();
        // NOW <clinit> runs for loadedClass

        // THREAD SAFETY: <clinit> is thread-safe by JVM spec
        // If 100 threads all try to use the class first simultaneously,
        // only ONE executes <clinit> — others wait. This is the basis of the
        // Initialization-on-Demand Holder Singleton pattern.

        System.out.println("Loaded class: " + loadedClass.getName());
    }
}
```

### Level 4 — Advanced
```java
// Custom ClassLoader: loading encrypted bytecode
import java.io.*;

public class SecurePluginClassLoader extends ClassLoader {

    private final String pluginDir;
    private final byte xorKey;

    public SecurePluginClassLoader(String pluginDir, byte xorKey, ClassLoader parent) {
        super(parent);  // Register parent for delegation model
        this.pluginDir = pluginDir;
        this.xorKey = xorKey;
    }

    // ALWAYS override findClass, NEVER override loadClass
    // loadClass implements the delegation model — overriding it breaks security
    // findClass is called only after parents have failed
    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        try {
            byte[] encrypted = readBytes(name);
            byte[] decrypted = xorDecrypt(encrypted);
            // defineClass registers the bytecode with the JVM
            return defineClass(name, decrypted, 0, decrypted.length);
        } catch (IOException e) {
            throw new ClassNotFoundException("Cannot find encrypted class: " + name, e);
        }
    }

    private byte[] readBytes(String className) throws IOException {
        String path = pluginDir + "/" + className.replace('.', '/') + ".enc";
        try (InputStream is = new FileInputStream(path);
             ByteArrayOutputStream baos = new ByteArrayOutputStream()) {
            is.transferTo(baos);
            return baos.toByteArray();
        }
    }

    private byte[] xorDecrypt(byte[] data) {
        byte[] result = new byte[data.length];
        for (int i = 0; i < data.length; i++) result[i] = (byte)(data[i] ^ xorKey);
        return result;
    }
}

// Usage:
class PluginUsage {
    interface Plugin { String execute(String input); }

    void loadPlugin() throws Exception {
        // Plugin interface loaded by App ClassLoader (parent)
        // Plugin implementation loaded by SecurePluginClassLoader (child)
        SecurePluginClassLoader cl = new SecurePluginClassLoader("/plugins", (byte)0x45,
            PluginUsage.class.getClassLoader());

        Class<?> pluginClass = cl.loadClass("com.example.AnalyticsPlugin");
        // Cast via interface — parent ClassLoader loaded the interface, so it's compatible
        Plugin plugin = (Plugin) pluginClass.getDeclaredConstructor().newInstance();
        System.out.println(plugin.execute("data"));
    }
}
```

### Level 5 — Expert
```java
// Hot-swap architecture + ClassLoader leak detection
import java.lang.ref.*;
import java.net.*;

public class HotSwapArchitecture {

    // Interface loaded by Application ClassLoader (stable)
    interface ReportService { String generateReport(String type); }

    // Hot-swappable service manager
    static class ServiceManager {
        private volatile ReportService activeService;
        private WeakReference<ClassLoader> oldLoaderRef;

        public ServiceManager() throws Exception {
            activeService = loadVersion("/services/v1/");
        }

        // Zero-downtime hot-swap
        public void upgrade(String newVersionPath) throws Exception {
            ReportService newService = loadVersion(newVersionPath);

            // Atomic reference swap — in-flight requests finish with old service
            ReportService old = activeService;
            activeService = newService;

            // OLD ClassLoader becomes weakly reachable
            // GC will unload it + all its classes from Metaspace eventually
            System.out.println("Upgraded service. Old loader GC-eligible.");

            // Optional: verify GC (for testing)
            old = null;
            System.gc();  // Not guaranteed, but signals intent
        }

        private ReportService loadVersion(String path) throws Exception {
            URL jarUrl = new URL("file://" + path + "report-service-impl.jar");
            // Child ClassLoader — parent is App ClassLoader
            URLClassLoader childLoader = new URLClassLoader(
                new URL[]{jarUrl},
                ServiceManager.class.getClassLoader()
            );
            oldLoaderRef = new WeakReference<>(childLoader);

            Class<?> implClass = childLoader.loadClass("com.example.ReportServiceImpl");
            return (ReportService) implClass.getDeclaredConstructor().newInstance();
        }

        public boolean isOldLoaderGarbageCollected() {
            return oldLoaderRef.get() == null;  // null = GC collected it
        }

        public ReportService get() { return activeService; }
    }

    // CLASSLOADER LEAK DETECTION: Common source of Metaspace OOM
    // A ClassLoader leak = keeping a strong reference to the ClassLoader (or any of its classes)
    // after you intended to unload it
    static void demonstrateLeakRisk() throws Exception {
        URLClassLoader leak = new URLClassLoader(new URL[]{new URL("file:///opt/plugin.jar")});
        Class<?> loadedClass = leak.loadClass("com.plugin.Config");

        // LEAK: Storing a static reference to a class loaded by the custom loader
        // ConfigHolder.config = loadedClass.newInstance();  ← ClassLoader can never be GC'd!

        // SAFE: Use WeakReference for optional caching
        WeakReference<Class<?>> weakRef = new WeakReference<>(loadedClass);
        leak.close();  // Signal intent to unload (JDK 7+ URLClassLoader.close())
        loadedClass = null;
        // Now only WeakReference holds it — GC can collect
    }

    public static void main(String[] args) throws Exception {
        // Tomcat-style multi-app isolation
        String[] apps = {"/webapps/app1/", "/webapps/app2/"};
        for (String appPath : apps) {
            // Each app gets its own ClassLoader — completely isolated namespaces
            URLClassLoader appLoader = new URLClassLoader(
                new URL[]{new URL("file://" + appPath + "WEB-INF/classes/")},
                HotSwapArchitecture.class.getClassLoader()
            );
            // app1's com.example.Config != app2's com.example.Config
            // Even if same bytes — different ClassLoader = different Class objects
            System.out.println("App " + appPath + ": " +
                appLoader.loadClass("com.example.Config").getClassLoader());
        }
    }
}
```

## INTERVIEW

### Q1: What is the parent delegation model and why does it exist?
**Difficulty:** Medium | **Companies:** Oracle, Google

❌ **Weak:** "ClassLoaders check parents first before loading a class."

✅ **Strong:** "When a ClassLoader is asked to load a class, it first delegates to its parent, which delegates to its parent, recursively up to Bootstrap. Only if Bootstrap can't find it does it fall back down. This guarantees that core classes like `java.lang.String` are always loaded by Bootstrap — you cannot override them with a custom malicious class. It also ensures consistency: `java.lang.Object` is the same object regardless of which ClassLoader requested it. The hierarchy is Bootstrap → Platform → Application → Custom. A class loaded by Bootstrap is visible to all ClassLoaders below it; a class loaded by Application ClassLoader is invisible to Bootstrap."

---

### Q2: What is the difference between ClassNotFoundException and NoClassDefFoundError?
**Difficulty:** Medium | **Companies:** All

❌ **Weak:** "Both mean the class wasn't found."

✅ **Strong:** "`ClassNotFoundException` is a checked exception thrown when `Class.forName()`, `ClassLoader.loadClass()`, or similar explicit loading APIs can't find the class. It means the class was never on the runtime classpath. `NoClassDefFoundError` is an Error (not Exception) thrown when a class reference in bytecode (like `new Foo()`) can't be resolved — meaning the class WAS available at compile time but is missing at runtime. Common cause: the class is in the compile classpath but missing from the runtime classpath (e.g., a jar that's a `compileOnly` dependency). `NoClassDefFoundError` can also be caused by a failed static initializer — the class was found but its `<clinit>` threw an exception, leaving it in an invalid state."

---

### Q3: How does Tomcat achieve class isolation between deployed web applications?
**Difficulty:** Hard | **Companies:** Red Hat, VMware

❌ **Weak:** "Each app has a separate ClassLoader."

✅ **Strong:** "Tomcat creates a separate `WebAppClassLoader` for each deployed web application. Each loader has a dedicated classpath pointing to `/WEB-INF/classes/` and `/WEB-INF/lib/*.jar` of that specific app. Because different apps have different ClassLoaders, even if both apps have `com.example.Config.class` with identical bytecode, they are two different `Class` objects — completely isolated namespaces. App1's static fields don't affect App2's statics. Critically, `WebAppClassLoader` uses **inverted delegation** for web app classes: it tries to load from the app's WEB-INF first before delegating to parent — this allows the app to override framework classes with its own version (e.g., different log4j version than Tomcat's)."

---

### Q4: What is a ClassLoader leak and how do you detect it?
**Difficulty:** Hard | **Companies:** Netflix, Spotify

❌ **Weak:** "When a ClassLoader isn't garbage collected."

✅ **Strong:** "A ClassLoader leak occurs when a custom ClassLoader can never be GC'd because something in the 'parent' class graph holds a strong reference to a class loaded by the custom loader. Since a Class holds a reference to its ClassLoader, any lingering reference to a loaded class keeps the entire ClassLoader alive in Metaspace — along with ALL classes it loaded. Classic causes: (1) `ThreadLocal` holding a class instance from the plugin loader. (2) Static fields in Bootstrap/App classes storing objects of classes from the plugin loader. (3) Shutdown hooks registered with the Runtime. Detection: heap dump + Eclipse Memory Analyzer Tool (MAT) → look for ClassLoader objects in 'Retained Heap' view. Monitor `Metaspace Used` via JMX — steady growth across redeployments = classic leak symptom."

---

### Q5: Why is `Class.forName()` different from `ClassLoader.loadClass()`?
**Difficulty:** Medium | **Companies:** Oracle, JetBrains

❌ **Weak:** "forName loads and initializes, loadClass just loads."

✅ **Strong:** "`Class.forName(name)` is equivalent to `Class.forName(name, true, currentClassLoader)` — it loads the class AND runs `<clinit>` (static initializer blocks + static field assignments). `ClassLoader.loadClass(name)` loads the binary class data and creates the Class object but defers `<clinit>` until the class is first actively used. This matters for classes with expensive or side-effectful static initializers. Also, `Class.forName()` uses the calling class's ClassLoader by default (via stack walk), while `ClassLoader.loadClass()` uses the specific ClassLoader you call it on — important for plugin architectures where you need to load from a specific loader, not the current thread's loader."

## FEYNMAN CHECK

ClassLoaders are like separate universes with a shared rulebook. Java has three built-in universes: Bootstrap (knows core Java), Platform (knows extensions), Application (knows your code). When you ask "what is String?", you always ask Bootstrap first. Bootstrap knows it — everyone shares the same String. When Tomcat runs your app, it creates a new mini-universe just for your app. Your app's `Logger` lives in that universe. Another app's `Logger` lives in a completely separate universe. Even if they look identical (same bytes), they're different objects in different universes — they can't accidentally mix. When your app is undeployed, its universe collapses — all its classes vanish from Metaspace.

## BUILD

**Challenge 1:** Write a `ClassLoader` that loads classes from Base64-encoded strings (inline bytecode). Load a class named `Hello` that prints "Hello from inline class!"

**Challenge 2:** Build a simple plugin system: define a `Plugin` interface in your main code, write two implementations in separate JARs, load them dynamically with `URLClassLoader`. Verify they're isolated (same class name in both plugins = different Class objects).

**Challenge 3:** Create a hot-swap service: implement `ServiceManager.upgrade(jarPath)` that loads a new implementation, atomically swaps the reference, and verifies (using `WeakReference`) that the old ClassLoader is eventually garbage collected.
