# CSS — Cascade, Specificity, Inheritance | Level: 1

The CSS engine is built on a highly structured pipeline that resolves conflicts, applies default rules, and calculates the exact visual values of elements. To write predictable, high-performance stylesheets, you must master the mechanics of the Cascade, Specificity calculations, Inheritance rules, and the Value Resolution pipeline.

---

## 1. The Cascade: Order of Precedence

The **Cascade** is the algorithm that resolves conflicts when multiple CSS rules match the same element. It processes declarations through a series of sorting steps in order of importance:

1.  **Origin and Importance**: Declarations are sorted into buckets based on where they are defined and whether they use `!important`.
2.  **Scope**: Rules scoped via inline styles, `@scope`, or Shadow DOM boundaries.
3.  **Specificity**: Calculated based on the selector components.
4.  **Order of Appearance**: The last declared rule wins if all other criteria are equal.

### The Cascade Origin Hierarchy (Highest to Lowest)

When resolving conflicts, the engine evaluates origins in this strict order:

```
[Highest Priority]
 1. Transition declarations (active transitions)
 2. User Agent !important (browser defaults, overrides)
 3. User !important (user-defined accessibility styles)
 4. Author !important (your application's overrides)
 5. Animation declarations (active keyframe animations)
 6. Author Normal (your application's standard CSS)
 7. User Normal (user-defined custom styles)
 8. User Agent Normal (default browser stylesheets)
[Lowest Priority]
```

## 2. Specificity Calculation

Specificity is a four-digit vector `(A, B, C, D)` that determines which selector takes precedence within the same Cascade origin:

*   **A (Inline Styles)**: Styles declared directly in the HTML `style` attribute. Specificity: `(1, 0, 0, 0)`.
*   **B (ID Selectors)**: e.g., `#header`. Specificity: `(0, 1, 0, 0)`.
*   **C (Class, Attribute, and Pseudo-classes)**: e.g., `.card`, `[type="text"]`, `:hover`. Specificity: `(0, 0, 1, 0)`.
*   **D (Type and Pseudo-elements)**: e.g., `div`, `::before`. Specificity: `(0, 0, 0, 1)`.

*Note: The universal selector (`*`), combinators (`+`, `>`, `~`, space), and the `:where()` pseudo-class have zero specificity `(0, 0, 0, 0)`.*

### Specificity Matchup

| Selector | A | B | C | D | Calculated Specificity |
| :--- | :--- | :--- | :--- | :--- | :--- |
| `div` | 0 | 0 | 0 | 1 | `(0, 0, 0, 1)` |
| `div.card` | 0 | 0 | 1 | 1 | `(0, 0, 1, 1)` |
| `#sidebar .nav-item a:hover` | 0 | 1 | 2 | 1 | `(0, 1, 2, 1)` |
| `style="..."` (Inline) | 1 | 0 | 0 | 0 | `(1, 0, 0, 0)` |

```css
/* Specificity: (0, 0, 1, 1) */
div.card {
  color: oklch(0.6 0.25 140);
}

/* Specificity: (0, 1, 0, 0) - Wins over the rule above due to ID column */
#hero-card {
  color: oklch(0.5 0.2 280);
}
```

## 3. The `!important` Escape Hatch

The `!important` flag does not increase specificity. Instead, it **moves the declaration to a higher-priority Cascade origin** (Author `!important`). 

### Why Avoid `!important`?
It bypasses the standard specificity tree, making selectors hard to override. If you need to override an `!important` rule, you must write an even more specific selector also flagged with `!important`. This leads to "specificity wars" that clutter your codebase.

### Modern Alternative: Cascade Layers (`@layer`)
Cascade Layers allow you to group styles into explicit priority blocks, rendering specificity wars obsolete.

```css
@layer reset, framework, utilities;

@layer utilities {
  /* This wins because the utilities layer is declared last, 
     regardless of selector specificity in earlier layers */
  .text-primary {
    color: oklch(0.62 0.17 220);
  }
}

@layer framework {
  #main-layout .profile-card div.header {
    color: oklch(0.3 0.1 200); /* High specificity, but loses to utilities layer */
  }
}
```

## 4. Inheritance Mechanics

Some CSS properties automatically inherit their computed values from their parent elements, while others do not.

*   **Inherited Properties (Text & Font)**: `color`, `font-family`, `font-size`, `line-height`, `letter-spacing`, `visibility`, `list-style`.
*   **Non-Inherited Properties (Layout & Geometry)**: `margin`, `padding`, `border`, `background`, `width`, `height`, `position`, `transform`, `display`.

### Explicit Control Keywords
You can override default inheritance behavior using four global keywords:
*   `inherit`: Forces an element to adopt the computed value of its parent for that property.
*   `initial`: Resets the property to its initial value as defined in the CSS specification (e.g., `inline` for `display`, `static` for `position`).
*   `unset`: Acts as `inherit` if the property is naturally inherited, and `initial` if it is not.
*   `revert`: Rolls back the cascade to the value defined by the user-agent or user stylesheet.

---

## 5. The Value Resolution Pipeline

The browser processes CSS properties through a multi-step pipeline before rendering them to the screen:

$$\text{Declared Value} \longrightarrow \text{Specified Value} \longrightarrow \text{Computed Value} \longrightarrow \text{Used Value} \longrightarrow \text{Resolved/Actual Value}$$

1.  **Declared Value**: All declarations matching the element (e.g., `width: auto`, `width: 50vw`, `width: 20rem`).
2.  **Specified Value**: The single value chosen by the Cascade. If no value matches, it uses the inherited or initial value.
3.  **Computed Value**: Absolute values resolved during path computation, before rendering layout (e.g., relative font sizes like `2rem` are converted to absolute pixels like `32px`). Relative lengths like percentages (`%` or `vw`) remain relative because layout has not run yet.
4.  **Used Value**: The absolute value calculated after laying out the document. Relative percentages and viewport units are resolved to absolute pixel dimensions (e.g., `50vw` becomes `960px`).
5.  **Resolved (Actual) Value**: The final value applied to the element, rounded to fit the physical pixel grid of the display device (e.g., `960.333px` rounded to `960px`).

### Accessing Resolved Values with JS
The JavaScript method `window.getComputedStyle(element)` returns the **Resolved Value** (which is usually the Used Value for layout properties like `width`, `height`, and `margin`, but the Computed Value for others).

## 6. Practical Implementation & HTML Demo

Here is a working demonstration of cascade layers, specificity overrides, and logical properties utilizing the modern `oklch()` color space.

### HTML
```html
<div class="theme-wrapper">
  <article id="featured-card" class="card">
    <h1 class="card-title">Understanding Cascade & Specificity</h1>
    <p class="card-body">
      This card demonstrates how CSS layers and specificity resolve conflicts.
    </p>
    <button class="btn btn-primary">Action Button</button>
  </article>
</div>
```

### CSS
```css
/* 1. Define Cascade Layers */
@layer base, components, overrides;

/* 2. Base Layer Styles */
@layer base {
  :root {
    --bg-surface: oklch(0.98 0.01 200);
    --text-main: oklch(0.2 0.02 240);
    --accent: oklch(0.6 0.18 250);
  }

  .theme-wrapper {
    background-color: var(--bg-surface);
    color: var(--text-main);
    padding-block: 2rem; /* Logical property */
    padding-inline: 1.5rem;
  }
}

/* 3. Component Layer Styles */
@layer components {
  .card {
    background-color: oklch(1 0 0);
    border: 1px solid oklch(0.9 0.01 240);
    border-radius: 12px;
    padding: 1.5rem;
    box-shadow: 0 4px 12px oklch(0 0 0 / 0.05);
  }

  .card-title {
    font-size: 1.5rem;
    margin-block-end: 1rem;
    color: var(--accent);
  }

  .btn {
    font: inherit;
    padding-block: 0.5rem;
    padding-inline: 1.25rem;
    border: none;
    border-radius: 6px;
    cursor: pointer;
    background-color: oklch(0.9 0.02 240);
    color: var(--text-main);
  }

  /* Specificity: (0, 0, 2, 0) */
  .card .btn-primary {
    background-color: var(--accent);
    color: oklch(1 0 0);
  }
}

/* 4. Overrides Layer Styles */
@layer overrides {
  /* This rule wins because it is in the overrides layer,
     even though .card .btn-primary in components has higher specificity */
  .btn-primary {
    background-color: oklch(0.55 0.22 140); /* Green accent override */
  }
}
```

## 7. Browser DevTools & Inspection Mechanics

To inspect how the browser resolves the cascade:
1.  Open Chrome or Firefox DevTools, right-click the button element, and select **Inspect**.
2.  Look at the **Styles** pane in the right column.
3.  You will see `.card .btn-primary` crossed out with a line through its properties. Underneath or above it, you'll see `.btn-primary` within the `@layer overrides` block active.
4.  Switch to the **Computed** tab.
5.  Search for `background-color`. Expand the property to see the resolution history: it shows the initial values, user-agent defaults, and every matched rule, indicating which one was selected.

## 8. Browser Compatibility & Specifications

*   **Cascade Layers (`@layer`)**: Supported in all major evergreen browsers since March 2022. Global support: **~98%** (CanIUse reference: `css-cascade-layers`).
*   **oklch() Color Space**: Supported in Chrome 111+, Safari 16.2+, Firefox 113+. Global support: **~95%** (CanIUse reference: `css-color-4`).
*   **Logical Properties (`padding-block`, `padding-inline`)**: Supported since late 2020. Global support: **~98%** (CanIUse reference: `css-logical-props`).

---

## 9. Performance Impact: Layout vs. Paint vs. Composite

*   **Property Selection**: Modifying properties like `color` or `background-color` bypasses the **Layout** phase and triggers only **Paint** and **Composite** operations.
*   **Logical Properties**: Properties like `padding-block` do not introduce extra performance overhead compared to physical properties (`padding-top`). They are resolved to physical properties during the Computed-to-Used value resolution step before layout calculations begin.
