# Concurrency Basics — Threads, Shared State, and Race Conditions

In Java, concurrency is built upon the execution model of the underlying Operating System. To write high-throughput, low-latency concurrent systems, you must understand how the Java Virtual Machine (JVM) maps its abstractions down to physical hardware, how memory is synchronized, and how the CPU reorders operations to optimize execution.

---

## 1. Deep Internal Mechanics of Java Threads

### 1:1 Thread Mapping on HotSpot
On modern HotSpot JVMs running on Linux, macOS, or Windows, a Java thread (`java.lang.Thread`) has a **1:1 mapping** to an operating system kernel thread (typically a `pthread` on POSIX systems). 

```
+---------------------------+
|    java.lang.Thread       |  (Java API)
+---------------------------+
              |
              v (1:1 Mapping via JVM)
+---------------------------+
|       OS Thread           |  (e.g., pthread on Linux)
+---------------------------+
              |
              v
+---------------------------+
|     Kernel Thread         |  (Managed by OS Scheduler)
+---------------------------+
```

When you call `Thread.start()`, the JVM invokes native code (via JNI) to allocate OS resources and spawn a kernel thread. This model introduces significant overhead:
*   **Kernel Thread Creation Cost:** Spawning a thread requires context switching into kernel space, allocating OS-level thread descriptors, and initializing system-level registers.
*   **Memory Footprint:** Each platform thread is allocated its own **JVM stack** (configured via the `-Xss` flag, defaulting to 1024 KB on 64-bit systems). This stack space is allocated off-heap (native memory) and holds local variables, frame data, and method call states.
*   **Context Switching:** The OS scheduler manages thread execution using preemptive time-slicing. When context switching occurs, the CPU must save the current thread's state (registers, program counter) and load the target thread's state, flushing CPU instruction pipelines and degrading L1/L2/L3 cache locality.

### Shared Heap vs. Private Stack
While each thread possesses its own private stack, all threads share the **JVM Heap**. Objects reside on the heap; local variables, primitives, and object references reside on the stack. 

```
               JVM HEAP (Shared Memory)
+-----------------------------------------------------+
|  Object A { field1, field2 }   Object B { ... }     |
+-----------------------------------------------------+
        ^                                 ^
        | (Reference)                     | (Reference)
+-----------------------+         +-----------------------+
| Thread-1 Stack        |         | Thread-2 Stack        |
|  - Local variables    |         |  - Local variables    |
|  - Frame pointers     |         |  - Frame pointers     |
+-----------------------+         +-----------------------+
```

Any thread with a reference to an object on the heap can read and write to its fields. This shared-memory model is the root cause of race conditions and memory visibility issues.

## 2. Race Conditions and Bytecode Deconstruction

A **race condition** occurs when the correctness of a program depends on the relative timing or interleaving of threads executing non-atomic operations on shared state.

Consider the classic non-thread-safe counter increment:

```java
private int counter = 0;

public void increment() {
    counter++;
}
```

At the Java syntax level, `counter++` looks like a single operation. However, compiling this code reveals that it is translated into multiple JVM bytecode instructions.

### Bytecode Deconstruction
Using `javap -c`, we can inspect the generated bytecode for `increment()`:

```bytecode
public void increment();
  Code:
   0: aload_0             // Load 'this' reference onto the operand stack
   1: dup                 // Duplicate the 'this' reference
   2: getfield      #2    // Retrieve the value of field 'counter' (reads from Heap)
   5: iconst_1            // Push integer 1 onto the operand stack
   6: iadd                // Add the top two stack values (counter + 1)
   7: putfield      #2    // Write the result back to field 'counter' (writes to Heap)
   10: return
```

The operation consists of three distinct steps:
1.  **Read:** `getfield` fetches the value from the heap.
2.  **Modify:** `iadd` performs the addition in the thread's local operand stack.
3.  **Write:** `putfield` writes the modified value back to the heap.

### Interleaving Scenario
If Thread A and Thread B concurrently invoke `increment()` when `counter = 5`, the execution can interleave as follows:

| Time | Thread A | Thread B | Heap State (`counter`) | Thread A Stack | Thread B Stack |
| :--- | :--- | :--- | :--- | :--- | :--- |
| T1 | `getfield` (reads 5) | | 5 | `[5]` | `[]` |
| T2 | | `getfield` (reads 5) | 5 | `[5]` | `[5]` |
| T3 | `iadd` (computes 6) | | 5 | `[6]` | `[5]` |
| T4 | | `iadd` (computes 6) | 5 | `[6]` | `[6]` |
| T5 | `putfield` (writes 6)| | 6 | `[]` | `[6]` |
| T6 | | `putfield` (writes 6)| 6 | `[]` | `[]` |

Despite two increments occurring, the final value of `counter` is `6` instead of `7`.

## 3. The Java Memory Model (JMM) & Visibility

The **Java Memory Model (JMM)** defines the legal behaviors of the JVM when threads read and write to shared memory. 

To maximize execution speed, modern CPUs do not write directly to main memory (DRAM). Instead, they write to local store buffers and utilize multi-level caches (L1, L2, L3). 

```
+-------------------------------------------------------+
|                     Main Memory                       |
+-------------------------------------------------------+
                           ^
                           | (Bus / Cache Coherency Protocol)
                           v
            +-----------------------------+
            |          L3 Cache           |
            +-----------------------------+
             /                           \
            v                             v
+-----------------------+     +-----------------------+
|       L1/L2 Cache     |     |       L1/L2 Cache     |
+-----------------------+     +-----------------------+
|  CPU Core 1 Registers |     |  CPU Core 2 Registers |
+-----------------------+     +-----------------------+
|    Thread 1 Working   |     |    Thread 2 Working   |
|         Memory        |     |         Memory        |
+-----------------------+     +-----------------------+
```

### The Visibility Problem
When Thread 1 writes a value to a shared variable, the updated value may reside only in its local CPU registers or write buffers. It has not yet been flushed to L3 cache or main memory. 
Concurrently, Thread 2 might read the same variable from its own local L1/L2 cache, resulting in a **stale read**.

## 4. Deep Dive: `synchronized` and the Object Monitor

The `synchronized` keyword provides mutual exclusion and establishes memory visibility guarantees.

### The Object Header and the Mark Word
Every Java object in the HotSpot JVM has an object header containing a **Mark Word** (64 bits on a 64-bit JVM) and a **Klass Word** (pointing to the metadata of the class).

The Mark Word changes state depending on the locking level of the object:

| Lock State | Biased Lock Bit | Lock Tag Bits | Mark Word Contents |
| :--- | :--- | :--- | :--- |
| **Unlocked** | `0` | `01` | Hashcode (31 bits), Age (4 bits) |
| **Biased** (Deprecated) | `1` | `01` | Thread ID (54 bits), Epoch (2 bits), Age (4 bits) |
| **Lightweight Locked** | `0` | `00` | Pointer to lock record on owner's thread stack |
| **Heavyweight Locked** | `0` | `10` | Pointer to native `ObjectMonitor` structure |
| **Marked for GC** | `0` | `11` | Used internally by the Garbage Collector |

### Lock Inflation (Escalation)
1.  **Biased Locking:** Historically, if a lock was only ever acquired by a single thread, the JVM "biased" the lock to that thread. (Note: Biased locking is disabled or deprecated in modern JDKs due to execution complexity).
2.  **Lightweight Locking (Thin Lock):** If multiple threads attempt to acquire the lock but there is no active contention, the JVM uses CAS (Compare-And-Swap) to write a pointer to the thread's execution stack into the Mark Word. This avoids OS-level mutexes.
3.  **Heavyweight Locking (Fat Lock):** If a thread attempts to acquire a lightweight lock and fails (due to active contention), the lock **inflates** to a heavyweight lock. The JVM allocates an OS-level `ObjectMonitor` structure. The Mark Word is updated to point to this monitor, and contending threads are parked using kernel-level synchronization primitives (e.g., `pthread_mutex` or `futex` on Linux).

### Memory Flush Semantics
When a thread enters a `synchronized` block:
1.  It acquires the monitor associated with the object.
2.  The JMM invalidates the thread's local cache, forcing it to read all shared variables directly from main memory.

When a thread exits a `synchronized` block:
1.  All modifications made within the block are flushed to main memory.
2.  The monitor is released, allowing parked threads to compete for acquisition.

## 5. Deep Dive: `volatile` Mechanics

The `volatile` keyword guarantees **visibility** and **instruction ordering**, but does *not* guarantee atomicity for compound operations (like `i++`).

```java
private volatile boolean flag = true;
```

### Preventing Reordering via Memory Barriers
CPUs and JIT compilers frequently reorder instructions to keep execution pipelines full. To prevent this, the compiler inserts hardware-specific **Memory Barriers** (also called memory fences) into the instruction stream around volatile reads and writes.

At the JVM level, the JMM specifies the following barrier placements:

*   **Volatile Write:**
    *   Inserts a `StoreStore` barrier *before* the write. This ensures all preceding writes are flushed before the volatile write occurs.
    *   Inserts a `StoreLoad` barrier *after* the write. This prevents the volatile write from being reordered with subsequent reads/writes.
*   **Volatile Read:**
    *   Inserts a `LoadLoad` barrier *after* the read. This ensures subsequent reads cannot read stale data.
    *   Inserts a `LoadStore` barrier *after* the read. This prevents subsequent writes from occurring before the read completes.

On an x86 architecture (which has a strong memory model), many of these barriers compile down to no-ops because the hardware naturally guarantees sequential consistency for most operations. However, the JIT compiler will emit instructions like `lock addl` to act as a CPU-level fence when a `StoreLoad` barrier is required. On weaker memory architectures like ARM, explicit fence instructions (e.g., `dmb`) are emitted.

## 6. Happens-Before Relationships

The **Happens-Before** relationship is the formal semantic model defined by the JMM to guarantee memory visibility. If action $A$ happens-before action $B$ ($A \rightarrow B$), then the results of action $A$ are guaranteed to be visible to action $B$, and the JVM is prohibited from reordering them.

Key JMM Happens-Before rules include:
1.  **Program Order Rule:** Each action in a single thread happens-before any subsequent action in that same thread (following the source code order).
2.  **Monitor Lock Rule:** An unlock on a monitor happens-before every subsequent lock on that *same* monitor.
3.  **Volatile Field Rule:** A write to a volatile field happens-before every subsequent read of that *same* volatile field.
4.  **Thread Start Rule:** A call to `Thread.start()` on a thread happens-before any actions in the started thread.
5.  **Thread Join Rule:** Any action in a thread happens-before any other thread successfully returns from a `join()` on that terminated thread.
6.  **Transitivity:** If $A \rightarrow B$ and $B \rightarrow C$, then $A \rightarrow C$.

## 7. Thread-Safe Implementation: Lock-Free Custom Counter

The following code implements a thread-safe, high-throughput counter. Instead of using heavy synchronized blocks, it leverages lock-free synchronization via `VarHandle` (introduced in Java 9), which exposes atomic CPU operations (CAS) directly.

```java
import java.lang.invoke.MethodHandles;
import java.lang.invoke.VarHandle;

public final class LockFreeCounter {

    // Volatile state is mandatory for VarHandle memory visibility guarantees
    private volatile long count = 0L;

    // VarHandle to perform atomic CAS operations on the 'count' field
    private static final VarHandle COUNT_HANDLE;

    static {
        try {
            COUNT_HANDLE = MethodHandles.lookup()
                .findVarHandle(LockFreeCounter.class, "count", long.class);
        } catch (ReflectiveOperationException e) {
            throw new ExceptionInInitializerError(e);
        }
    }

    /**
     * Increments the counter atomically using a lock-free CAS loop.
     */
    public void increment() {
        long current;
        long next;
        do {
            // Volatile read of the current state
            current = (long) COUNT_HANDLE.getVolatile(this);
            next = current + 1;
            // CAS operation: Attempts to set the value to 'next' ONLY if the
            // current value in memory matches 'current'.
        } while (!COUNT_HANDLE.compareAndSet(this, current, next));
    }

    /**
     * Retrieves the current count.
     * Guaranteed to return the most up-to-date value due to volatile read semantics.
     */
    public long get() {
        return (long) COUNT_HANDLE.getVolatile(this);
    }
}
```

### Why This Code is Thread-Safe
1.  **Volatile Memory Semantics:** The `count` field is marked `volatile`. When `COUNT_HANDLE.getVolatile(this)` is invoked, it forces a fresh read from main memory, preventing stale values.
2.  **Atomic CAS (Compare-And-Swap):** The method `compareAndSet` compiles down to a single atomic CPU instruction (e.g., `lock cmpxchg` on x86). If another thread modified the `count` field between the read and the write steps, the CAS fails, and the loop retries.
3.  **No Deadlocks or Thread Parking:** Because it does not block, this code is immune to thread deadlock scenarios and avoids the high context-switching costs associated with OS-level mutexes.
