# REST — Constraints, Richardson Maturity Model

### 1. Architectural Deep Dive & Core Mechanics

Representational State Transfer (REST), formulated by Roy Fielding in his 2000 doctoral dissertation, is an architectural style for distributed hypermedia systems. It is not a protocol, but a set of six architectural constraints designed to optimize network efficiency, scalability, and system longevity:

1. **Client-Server Architecture**: Separation of concerns. Clients manage user interface and state; servers manage data storage, business logic, and security. This separation allows independent evolution of both tiers.
2. **Statelessness**: Every request from a client must contain all the information necessary to understand and process the request. The server must not store any session context about the client. This improves scalability (as requests can be routed to any server instance) and reliability.
3. **Cacheability**: Responses must implicitly or explicitly define themselves as cacheable or non-cacheable. Efficient caching eliminates some client-server interactions, improving performance and scalability.
4. **Layered System**: A client cannot ordinarily tell whether it is connected directly to the end server or to an intermediary (e.g., load balancer, reverse proxy, API gateway). This enables security policies, caching layers, and horizontal scalability without client-side modifications.
5. **Code on Demand (Optional)**: Servers can temporarily extend or customize client functionality by transferring executable code (e.g., compiled components, JavaScript scripts).
6. **Uniform Interface**: This is the defining constraint of REST. It simplifies the system architecture by enforcing a decoupled, standard interface based on four sub-constraints:
   - **Identification of Resources**: Individual resources are identified in requests using URIs. The resources themselves are conceptually separate from the representations returned to the client.
   - **Manipulation of Resources through Representations**: When a client holds a representation of a resource (including metadata), it has enough information to modify or delete the resource on the server, provided it has permission.
   - **Self-Descriptive Messages**: Each message includes enough information to describe how to process it. For example, the `Content-Type` header specifies the media type of the payload.
   - **Hypermedia As The Engine Of Application State (HATEOAS)**: Clients discover available actions dynamically through hypermedia links embedded within resource representations.

#### The Richardson Maturity Model (RMM)

Leonard Richardson categorized REST APIs into four levels based on their adherence to RESTful principles:

```
[ Level 3: Hypermedia Controls (HATEOAS) ]
                    │
[ Level 2: HTTP Verbs & Status Codes    ]
                    │
[ Level 1: Resources (Individual URIs)  ]
                    │
[ Level 0: The Swamp of POX (Single URI)]
```

*   **Level 0 (The Swamp of POX)**: Uses HTTP as a transport protocol for Remote Procedure Calls (RPC). There is a single URI (e.g., `/api/service`) and typically one method (usually `POST`). The payload contains the method name and parameters (e.g., XML-RPC or SOAP).
*   **Level 1 (Resources)**: Introduces the concept of individual resources. Instead of sending all requests to a single endpoint, requests are sent to individual resource URIs (e.g., `/books`, `/books/42`). However, it still typically uses a single HTTP method (usually `POST`) and ignores HTTP status codes.
*   **Level 2 (HTTP Verbs & Status Codes)**: Leverages HTTP semantics. It uses correct HTTP verbs (`GET` for retrieval, `POST` for creation, `PUT`/`PATCH` for updates, `DELETE` for removal) and returns standard HTTP status codes (e.g., `200 OK`, `201 Created`, `400 Bad Request`, `404 Not Found`).
*   **Level 3 (Hypermedia Controls)**: Introduces HATEOAS. The resource representations contain links that guide the client on what actions are possible next. This decouples the client from the server’s URI structure, allowing the server to change its URI scheme without breaking clients.

### 2. Protocol Specifications & Interface Contracts

Below is an OpenAPI 3.1.0 specification representing a Level 3 RESTful resource with HATEOAS links.

```yaml
openapi: 3.1.0
info:
  title: Order Management API (HATEOAS)
  version: 1.0.0
paths:
  /orders/{orderId}:
    get:
      summary: Retrieve an order by ID
      parameters:
        - name: orderId
          in: path
          required: true
          schema:
            type: string
            format: uuid
      responses:
        '200':
          description: Standard Level 3 HATEOAS representation of an order.
          content:
            application/hal+json:
              schema:
                $ref: '#/components/schemas/OrderRepresentation'
components:
  schemas:
    OrderRepresentation:
      type: object
      properties:
        id:
          type: string
          format: uuid
        amount:
          type: number
          format: double
        status:
          type: string
          enum: [PENDING, PAID, SHIPPED, CANCELLED]
        _links:
          type: object
          properties:
            self:
              $ref: '#/components/schemas/Link'
            payment:
              $ref: '#/components/schemas/Link'
            cancel:
              $ref: '#/components/schemas/Link'
    Link:
      type: object
      properties:
        href:
          type: string
          format: uri
        method:
          type: string
          default: GET
```

### 3. Production-Grade Spring Boot Implementation

This implementation uses Spring HATEOAS to build a Level 3 Richardson Maturity Model controller.

```java
package com.example.api.rest;

import org.springframework.hateoas.CollectionModel;
import org.springframework.hateoas.EntityModel;
import org.springframework.hateoas.Link;
import org.springframework.hateoas.RepresentationModel;
import org.springframework.hateoas.server.mvc.WebMvcLinkBuilder;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.*;

import java.util.UUID;

import static org.springframework.hateoas.server.mvc.WebMvcLinkBuilder.*;

enum OrderStatus { PENDING, PAID, SHIPPED, CANCELLED }

class OrderModel extends RepresentationModel<OrderModel> {
    private final UUID id;
    private final double amount;
    private final OrderStatus status;

    public OrderModel(UUID id, double amount, OrderStatus status) {
        this.id = id;
        this.amount = amount;
        this.status = status;
    }

    public UUID getId() { return id; }
    public double getAmount() { return amount; }
    public OrderStatus getStatus() { return status; }
}

@RestController
@RequestMapping(value = "/orders", produces = "application/hal+json")
public class OrderController {

    @GetMapping("/{id}")
    public ResponseEntity<OrderModel> getOrder(@PathVariable UUID id) {
        // Mock DB fetch
        OrderModel order = new OrderModel(id, 250.00, OrderStatus.PENDING);
        
        // Build self link
        order.add(linkTo(methodOn(OrderController.class).getOrder(id)).withSelfRel());
        
        // Build conditional state transition links (HATEOAS)
        if (order.getStatus() == OrderStatus.PENDING) {
            order.add(linkTo(methodOn(OrderController.class).payOrder(id)).withRel("payment"));
            order.add(linkTo(methodOn(OrderController.class).cancelOrder(id)).withRel("cancel"));
        }
        
        return ResponseEntity.ok(order);
    }

    @PostMapping("/{id}/payments")
    public ResponseEntity<Void> payOrder(@PathVariable UUID id) {
        // Process payment...
        return ResponseEntity.noContent().build();
    }

    @DeleteMapping("/{id}")
    public ResponseEntity<Void> cancelOrder(@PathVariable UUID id) {
        // Process cancellation...
        return ResponseEntity.noContent().build();
    }
}
```

### 4. Security Architecture & OWASP Alignment

*   **OWASP API1:2023 - Broken Object Level Authorization (BOLA)**: In REST APIs, resource identifiers (like UUIDs) are exposed directly in URIs. Attackers can exploit this by guessing other IDs.
    *   *Mitigation*: Implement fine-grained authorization checks at the service/data layer. Never rely on the client hiding links. The server must verify if the authenticated user owns or has access to `/orders/{id}` before processing the request.
*   **OWASP API3:2023 - Broken Object Property Level Authorization (BOPLA)**: Returning a complete representation of a resource (especially when using generic serializers) can leak sensitive properties.
    *   *Mitigation*: Use dedicated DTOs (like `OrderModel` above) instead of raw JPA entities. This prevents exposing internal database fields or user records.

### 6. Anti-Patterns & Production Checklist

#### Anti-Patterns
*   **RPC-over-HTTP (Level 0/1 disguised as REST)**: Exposing endpoints like `/orders/create` or `/orders/deleteOrder?id=5` using `POST` or `GET`.
*   **Ignoring HTTP Cache Directives**: Not utilizing `ETag`, `Last-Modified`, or `Cache-Control` headers, which forces the backend to execute database queries for static or unchanged resources.

#### Production Checklist
- [ ] Every API endpoint is mapped to a specific, noun-based resource path.
- [ ] Correct HTTP verbs are utilized (`GET` for reading, `POST` for creating, `PUT`/`PATCH` for updates, `DELETE` for removal).
- [ ] No raw database entities are serialized; DTOs/Representations are strictly used.
- [ ] HATEOAS links are dynamically generated using context-aware URI builders (respecting `X-Forwarded-Proto` and `X-Forwarded-Host` headers).
- [ ] All sensitive endpoints verify authorization at the object level, regardless of link visibility.
