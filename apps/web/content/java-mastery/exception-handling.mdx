## WHY
In C and Go, errors are handled by returning error codes (e.g., `-1` for file not found). You have to manually check `if (result == -1)` after every single function call. If you forget to check it once, the program continues executing with corrupted state until it spectacularly crashes millions of instructions later.

Java solved this using Exceptions. When a catastrophic event occurs (like dividing by zero or a database timeout), Java stops normal execution immediately and "throws" an Exception object up the Call Stack. This forces the developer to handle the error explicitly (`try/catch`), or the JVM violently terminates the thread. It ensures that errors cannot be silently ignored.

## THEORY
### The Exception Hierarchy
All exceptions in Java inherit from `java.lang.Throwable`.
1. **Error**: Severe JVM-level issues that you should NEVER try to catch (e.g., `OutOfMemoryError`, `StackOverflowError`). If these happen, the JVM is already dead.
2. **Exception**: Issues your program *can* recover from.
   - **Checked Exceptions** (extends `Exception`): The compiler strictly forces you to handle them (e.g., `IOException`, `SQLException`). You must use a `try/catch` block, or declare `throws` on your method signature.
   - **Unchecked Exceptions** (extends `RuntimeException`): Programming errors (e.g., `NullPointerException`, `IllegalArgumentException`). The compiler does *not* force you to handle these.

### The Stack Trace
When an exception is thrown, the JVM takes a snapshot of the current Thread Stack (every method that is currently executing) and embeds it into the Exception object. This is called the Stack Trace. It is an extremely expensive operation! Creating an exception takes microseconds (which is an eternity for a CPU) because it has to traverse and record memory frames.

## CODE

### Level 1 — Beginner
```java
public class BasicTryCatch {
    public static void main(String[] args) {
        try {
            int result = 10 / 0; // Throws ArithmeticException
        } catch (ArithmeticException e) {
            // JVM jumps here instantly, skipping remaining try block code
            System.out.println("Cannot divide by zero!");
        } finally {
            // The finally block ALWAYS executes, even if an exception occurs
            // Perfect for closing resources.
            System.out.println("Cleanup executed.");
        }
    }
}
```

### Level 2 — Building Blocks
```java
import java.io.IOException;

public class CheckedExceptions {
    // Because IOException is Checked, we MUST declare 'throws' if we don't catch it
    public void readFile() throws IOException {
        throw new IOException("File not found");
    }
    
    public void process() {
        try {
            readFile();
        } catch (IOException e) {
            System.out.println("Handled the checked exception.");
        }
    }
}
```

### Level 3 — Intermediate
```java
// Custom Unchecked Exception
public class UserNotFoundException extends RuntimeException {
    public UserNotFoundException(String message) {
        super(message);
    }
}

public class ThrowingCustom {
    public void findUser(String id) {
        if (id == null) {
            throw new UserNotFoundException("User ID cannot be null");
        }
    }
}
```

### Level 4 — Advanced
```java
// Java 7+ Try-With-Resources
import java.io.BufferedReader;
import java.io.FileReader;
import java.io.IOException;

public class AutoClose {
    public void readData() {
        // Any object implementing AutoCloseable can go in the () block.
        // The JVM will automatically call .close() on it, even if exceptions occur!
        // No finally block needed!
        try (BufferedReader br = new BufferedReader(new FileReader("data.txt"))) {
            System.out.println(br.readLine());
        } catch (IOException e) {
            e.printStackTrace();
        }
    }
}
```

## REAL_WORLD
In modern Spring Boot applications, the standard practice is to use **Unchecked Exceptions (`RuntimeException`) exclusively**.

Historically, J2EE forced developers to use Checked Exceptions. This resulted in disastrous "Catch and Ignore" patterns where developers got tired of the compiler yelling at them, so they wrote `try { ... } catch (Exception e) {}` and swallowed the error silently. 

Spring Data completely wraps all Checked `SQLExceptions` into Unchecked `DataAccessException`s.
When writing a Spring Boot REST API, the production pattern is to throw Unchecked exceptions deep in your Service layer, and use a global `@RestControllerAdvice` to catch them at the top level and format them into beautiful JSON HTTP responses.

```java
// 1. Service throws unchecked exception
public void checkout() {
    throw new OutOfStockException("Item 123 is gone");
}

// 2. Global Handler intercepts it automatically
@RestControllerAdvice
public class GlobalExceptionHandler {
    
    @ExceptionHandler(OutOfStockException.class)
    @ResponseStatus(HttpStatus.CONFLICT) // Returns HTTP 409
    public ErrorResponse handleOutOfStock(OutOfStockException ex) {
        return new ErrorResponse("OUT_OF_STOCK", ex.getMessage());
    }
}
```
Furthermore, you should NEVER use Exceptions for normal business control flow (e.g., throwing an exception just to exit a loop). Because generating the Stack Trace is incredibly slow, using Exceptions for normal logic will destroy the performance of your application.

## INTERVIEW
**Q: What is the difference between a Checked and Unchecked Exception?**
A: Checked exceptions extend `Exception` and must be handled explicitly at compile time using `try-catch` or the `throws` keyword. Unchecked exceptions extend `RuntimeException` and represent programming errors; the compiler does not force you to handle them.

**Q: Does a `finally` block always execute?**
A: Almost always. It will execute even if a `return` statement is inside the `try` block or an Exception is thrown. The only scenarios where `finally` does NOT execute is if you call `System.exit(0)`, if the JVM crashes entirely, or if the thread executing the block is violently killed by the OS.

**Q: Why are Exceptions slow to instantiate?**
A: When you instantiate an Exception using `new Exception()`, the JVM calls the native `fillInStackTrace()` method. This method has to traverse the entire Thread Stack, reading the memory frames to figure out exactly which methods were called to get to this exact line of code. This memory traversal is highly expensive for the CPU.
