## WHY

In Spring Boot, a **Bean** is any object that the IoC (Inversion of Control) container creates, configures, and manages. You don't write `new UserService()` — Spring creates it, wires its dependencies, and hands it to you. This is the core philosophy of Spring.

But "Spring Bean" encompasses more than just creation. The bean lifecycle, stereotype annotations, bean scopes, and lifecycle callbacks are the machinery that powers every Spring Boot application. Getting these wrong causes:

1. **`NullPointerException` in production**: Using a bean before its `@PostConstruct` runs, or using `this` in a constructor when fields aren't injected yet.
2. **Stale beans**: Injecting a `prototype`-scoped bean into a `singleton` — the singleton is created once with that prototype instance, never getting a fresh copy again.
3. **Missing exception translation**: Using `@Component` instead of `@Repository` means JPA/Hibernate exceptions propagate untranslated — your service layer gets `javax.persistence.OptimisticLockException` instead of Spring's `DataAccessException`.
4. **Slow startup**: Doing heavy initialization in constructors or `@PostConstruct`. The entire application blocks until it completes.
5. **Memory leaks**: Not understanding bean scope — storing per-request state in a singleton bean.

Understanding the full Spring Bean lifecycle — from classpath scanning through BeanFactoryPostProcessor, BeanPostProcessor, and AOP proxy creation — is the foundation for Spring debugging.

## THEORY

### Stereotype Annotations: What They Really Do

All four annotations ultimately register beans with the application context. The differences are semantic and AOP-based:

| Annotation | Meta-Annotation | Special Behavior |
|------------|-----------------|------------------|
| `@Component` | None (root) | Generic Spring component — no special post-processing |
| `@Service` | `@Component` | Marks business logic; used as pointcut for `@Transactional` advice |
| `@Repository` | `@Component` | **Exception translation**: JPA/JDBC exceptions → `DataAccessException` |
| `@Controller` | `@Component` | Registered with `RequestMappingHandlerMapping` for HTTP routing |
| `@RestController` | `@Controller` + `@ResponseBody` | Controller + automatic JSON response body serialization |

### Spring Bean Lifecycle (Complete)

```
1. CLASSPATH SCANNING
   ClassPathScanningCandidateComponentProvider uses ASM (reads .class bytes
   without loading into JVM) to find @Component stereotypes
   → Produces ScannedGenericBeanDefinition objects

2. BEAN DEFINITION REGISTRATION
   BeanDefinitions stored in DefaultListableBeanFactory registry
   Metadata: class name, scope, constructor args, lazy-init flag

3. BEAN FACTORY POST-PROCESSING
   BeanFactoryPostProcessor runs BEFORE any beans are instantiated
   Example: ConfigurationClassPostProcessor processes @Configuration,
            PropertySourcesPlaceholderConfigurer resolves ${...} placeholders

4. BEAN INSTANTIATION
   Topological sort of dependency graph → create beans in dependency order
   Constructor.newInstance(args) called with injected dependencies

5. DEPENDENCY INJECTION
   Constructor injection (done in step 4)
   Field/setter injection via AutowiredAnnotationBeanPostProcessor

6. AWARE INTERFACE CALLBACKS
   If bean implements BeanNameAware → setBeanName(name) called
   If bean implements ApplicationContextAware → setApplicationContext(ctx) called

7. BEAN POST-PROCESSING (BEFORE INIT)
   BeanPostProcessor.postProcessBeforeInitialization() called
   e.g., ApplicationContextAwareProcessor sets ApplicationContext on Aware beans

8. INITIALIZATION
   @PostConstruct methods run (JSR-250)
   InitializingBean.afterPropertiesSet() called
   @Bean(initMethod="init") called

9. BEAN POST-PROCESSING (AFTER INIT)
   BeanPostProcessor.postProcessAfterInitialization() called
   *** AOP PROXY CREATION HAPPENS HERE ***
   - @Transactional beans get CGLIB/JDK proxy
   - @Cacheable beans get proxy
   - @Async beans get proxy

10. BEAN READY
    Stored in singleton cache. Ready to serve requests.

11. CONTEXT CLOSE (SHUTDOWN)
    @PreDestroy methods run
    DisposableBean.destroy() called
    @Bean(destroyMethod="cleanup") called
```

### `@Repository` Exception Translation

`@Repository` adds an AOP interceptor (`PersistenceExceptionTranslationInterceptor`) around the DAO bean. When a JPA/Hibernate/JDBC exception occurs, the interceptor catches it and translates it to Spring's `DataAccessException` hierarchy:

```
Original exception:
  javax.persistence.OptimisticLockException
    → org.springframework.dao.OptimisticLockingFailureException

  org.hibernate.exception.ConstraintViolationException
    → org.springframework.dao.DataIntegrityViolationException

  java.sql.SQLTimeoutException
    → org.springframework.dao.QueryTimeoutException
```

**Why this matters**: Without translation, your `@Service` layer must `catch (HibernateException e)` — coupling business logic to your persistence technology. With translation, it only catches `DataAccessException` — technology-agnostic.

### Bean Scopes

| Scope | One per... | Default | Thread-safe |
|-------|-----------|---------|-------------|
| `singleton` | ApplicationContext | ✅ YES | Your responsibility |
| `prototype` | Injection/`getBean()` | No | N/A (new each time) |
| `request` | HTTP request | No (web only) | Yes |
| `session` | HTTP session | No (web only) | Yes (synchronized) |
| `application` | ServletContext | No (web only) | Your responsibility |

**The prototype-in-singleton problem**: If a `singleton` injects a `prototype` bean, the singleton holds a permanent reference to the SAME prototype instance — you only get one prototype, not a fresh one each time. Use `ObjectProvider<PrototypeBean>` to get a new instance each call.

### Bean Naming

Spring generates bean names from class names:
- `UserService.class` → `"userService"` (lowercase first letter)
- `@Service("customName")` → `"customName"` (explicit)
- `@Bean` method name → method name as bean name

## CODE

### Level 1 — Beginner
```java
import org.springframework.stereotype.*;
import org.springframework.boot.autoconfigure.*;
import org.springframework.boot.*;

// THE FOUR STEREOTYPES: What they are and when to use each
@SpringBootApplication
public class SpringBeansApp {
    public static void main(String[] args) {
        SpringApplication.run(SpringBeansApp.class, args);
    }
}

// @Component: Generic utility — doesn't fit service/repository/controller
@Component
class UuidGenerator {
    public String generate() {
        return java.util.UUID.randomUUID().toString();
    }
}

// @Repository: Data access — MUST use for JPA/JDBC exception translation
@Repository
interface UserRepository extends org.springframework.data.jpa.repository.JpaRepository<User, Long> {
    // Spring Data auto-generates implementation
}

// @Service: Business logic — transaction boundaries, orchestration
@Service
class UserService {
    private final UserRepository userRepository;  // constructor injection

    UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    public User findUser(Long id) {
        return userRepository.findById(id)
            .orElseThrow(() -> new RuntimeException("User not found: " + id));
    }
}

// @RestController: HTTP endpoint — @Controller + @ResponseBody
@org.springframework.web.bind.annotation.RestController
@org.springframework.web.bind.annotation.RequestMapping("/api/users")
class UserController {
    private final UserService userService;

    UserController(UserService userService) {
        this.userService = userService;
    }

    @org.springframework.web.bind.annotation.GetMapping("/{id}")
    public User getUser(@org.springframework.web.bind.annotation.PathVariable Long id) {
        return userService.findUser(id);
    }
}

@jakarta.persistence.Entity
class User {
    @jakarta.persistence.Id
    @jakarta.persistence.GeneratedValue
    private Long id;
    private String name;
    // getters/setters
}
```

### Level 3 — Intermediate
```java
import org.springframework.context.annotation.*;
import org.springframework.stereotype.*;
import org.springframework.beans.factory.config.*;
import org.springframework.beans.factory.*;

// SCOPES: singleton, prototype, and the prototype-in-singleton trap

// Singleton (default): ONE instance for entire application lifetime
@Service  // scope = singleton by default
class OrderService {
    // Stateless — safe as singleton
    public String processOrder(String orderId) {
        return "Processed: " + orderId;
    }
}

// Prototype: NEW instance every time it's injected or getBean() is called
@Component
@Scope(ConfigurableBeanFactory.SCOPE_PROTOTYPE)
class ShoppingCart {
    private final java.util.List<String> items = new java.util.ArrayList<>();

    // Stateful — MUST be prototype, not singleton!
    // If singleton: all users would share the same cart!
    public void addItem(String item) { items.add(item); }
    public java.util.List<String> getItems() { return items; }
}

// THE PROTOTYPE-IN-SINGLETON TRAP
@Service
class BadCheckoutService {
    // PROBLEM: singleton holds ONE prototype instance — never gets a fresh cart!
    @org.springframework.beans.factory.annotation.Autowired
    private ShoppingCart sharedCart;  // ← WRONG: this is always the same cart!

    public void addToCart(String item) {
        sharedCart.addItem(item);  // All users add to the SAME cart!
    }
}

// FIX: Use ObjectProvider to get a fresh prototype per call
@Service
class GoodCheckoutService {
    private final ObjectProvider<ShoppingCart> cartProvider;

    GoodCheckoutService(ObjectProvider<ShoppingCart> cartProvider) {
        this.cartProvider = cartProvider;
    }

    public void addToCart(String item) {
        ShoppingCart freshCart = cartProvider.getObject();  // NEW instance each call
        freshCart.addItem(item);
    }
}
```

### Level 4 — Advanced
```java
import org.springframework.beans.factory.config.*;
import org.springframework.context.annotation.*;
import org.springframework.stereotype.*;

// BEANFACTORYPOSTPROCESSOR: Modify bean definitions BEFORE instantiation
@Component
public class CustomBeanPostProcessors {

    // BeanFactoryPostProcessor: modifies BeanDefinition BEFORE instantiation
    @Configuration
    static class MetadataModifier implements BeanFactoryPostProcessor {
        @Override
        public void postProcessBeanFactory(ConfigurableListableBeanFactory factory) {
            // Example: make all @Service beans lazy-init unless explicitly eager
            String[] serviceNames = factory.getBeanNamesForAnnotation(Service.class);
            for (String name : serviceNames) {
                BeanDefinition bd = factory.getBeanDefinition(name);
                if (!name.contains("Critical")) {
                    bd.setLazyInit(true);  // Defer instantiation until first use
                    System.out.println("Set lazy: " + name);
                }
            }
        }
    }

    // BeanPostProcessor: intercepts EVERY bean after creation, before use
    @Component
    static class PerformanceTracker implements BeanPostProcessor {
        private final java.util.Set<String> trackedBeans = new java.util.HashSet<>();

        @Override
        public Object postProcessBeforeInitialization(Object bean, String beanName) {
            // Called before @PostConstruct
            System.out.println("Pre-init: " + beanName);
            return bean;  // MUST return bean (or a proxy)
        }

        @Override
        public Object postProcessAfterInitialization(Object bean, String beanName) {
            // Called after @PostConstruct — AOP proxies created here!
            // Spring's @Transactional proxy is created in this phase
            trackedBeans.add(beanName);
            return bean;  // Can return a proxy wrapping the original bean
        }
    }

    // ApplicationReadyEvent: heavy tasks after FULL startup complete
    @Component
    static class StartupTaskRunner {
        @org.springframework.context.event.EventListener(
            org.springframework.boot.context.event.ApplicationReadyEvent.class)
        public void onReady() {
            System.out.println("App fully started — running heavy initialization...");
            // Safe to do expensive work here — all beans ready, app serving requests
        }
    }
}
```

## REAL_WORLD

### The `@Repository` Exception Translation in Practice
```java
// Without @Repository: JPA exception leaks through all layers
@Component  // WRONG: should be @Repository
class BadUserDao {
    @PersistenceContext EntityManager em;

    public User findById(Long id) {
        User user = em.find(User.class, id);
        if (user == null) throw new javax.persistence.EntityNotFoundException("id: " + id);
        // ← Hibernate-specific exception propagates to @Service, violating layer separation!
        return user;
    }
}

// With @Repository: translated to Spring's DataAccessException
@Repository  // CORRECT: exception translation AOP applied
class GoodUserDao {
    @PersistenceContext EntityManager em;

    public User findById(Long id) {
        User user = em.find(User.class, id);
        if (user == null) throw new javax.persistence.EntityNotFoundException("id: " + id);
        // Spring translates → org.springframework.dao.EmptyResultDataAccessException
        return user;
    }
}

@Service
class UserServiceLayer {
    private final GoodUserDao dao;
    UserServiceLayer(GoodUserDao dao) { this.dao = dao; }

    public UserDTO getUser(Long id) {
        try {
            return mapToDto(dao.findById(id));
        } catch (org.springframework.dao.DataAccessException e) {
            // Technology-agnostic catch — works whether you switch to MongoDB or Postgres
            throw new UserNotFoundException("User " + id + " not found");
        }
    }
}
```

## INTERVIEW

### Q1: What is the difference between @Component, @Service, @Repository, and @Controller?
**Difficulty:** Easy | **Companies:** All Spring shops

❌ **Weak:** "They're all the same, just semantic labels."

✅ **Strong:** "All four are `@Component` specializations that register beans with the IoC container. The functional differences are: `@Repository` adds an AOP interceptor (`PersistenceExceptionTranslationInterceptor`) that translates JPA/Hibernate/JDBC exceptions into Spring's `DataAccessException` hierarchy — critical for technology-agnostic error handling. `@Controller` gets picked up by `RequestMappingHandlerMapping` to register HTTP endpoints. `@Service` is a semantic marker without special post-processing, but it's the natural pointcut for `@Transactional` advice in service layers. `@Component` is the generic fallback for utilities that don't fit the other categories."

---

### Q2: Describe the complete Spring Bean lifecycle.
**Difficulty:** Hard | **Companies:** Large Spring shops, banking

❌ **Weak:** "Spring creates beans, injects dependencies, and calls @PostConstruct."

✅ **Strong:** "The lifecycle has 10 phases: (1) Classpath scanning finds `@Component` candidates using ASM bytecode reading. (2) BeanDefinition registration stores metadata. (3) BeanFactoryPostProcessor modifies definitions before instantiation. (4) Bean instantiation via constructor. (5) Dependency injection. (6) Aware interface callbacks (BeanNameAware, ApplicationContextAware). (7) BeanPostProcessor.postProcessBeforeInitialization(). (8) @PostConstruct and InitializingBean.afterPropertiesSet(). (9) BeanPostProcessor.postProcessAfterInitialization() — **this is where AOP proxies for @Transactional and @Cacheable are created**. (10) At shutdown: @PreDestroy, DisposableBean.destroy()."

---

### Q3: What is the prototype-in-singleton problem and how do you fix it?
**Difficulty:** Hard | **Companies:** Netflix, fintech companies

❌ **Weak:** "You should use prototype scope for stateful beans."

✅ **Strong:** "When a `singleton` bean has a field `@Autowired` with a `prototype`-scoped bean, Spring injects the prototype exactly ONCE when creating the singleton — the singleton holds that one prototype instance forever. Subsequent calls to the singleton reuse the same prototype — defeating its purpose. Fix: Inject `ObjectProvider<PrototypeBean>` instead of the bean directly. Call `provider.getObject()` each time a fresh instance is needed. For web scope, `@Scope(proxyMode = ScopedProxyMode.TARGET_CLASS)` creates a scoped proxy that fetches the current request/session-scoped bean from the appropriate scope on each method call."

---

### Q4: What happens in BeanPostProcessor.postProcessAfterInitialization()?
**Difficulty:** Hard | **Companies:** Spring-heavy enterprise shops

❌ **Weak:** "It runs custom code after initialization."

✅ **Strong:** "`postProcessAfterInitialization()` is the last phase before a bean enters the singleton cache. Critically, **this is where Spring creates AOP proxies** for beans annotated with `@Transactional`, `@Cacheable`, `@Async`, `@Retryable`, etc. Spring's `AnnotationAwareAspectJAutoProxyCreator` (itself a BeanPostProcessor) inspects each bean, and if it matches any pointcut (method annotations, class annotations), wraps it in a CGLIB or JDK dynamic proxy. The bean you get from `@Autowired` after this phase is the proxy, not the original object. This is why calling `@Transactional` methods via `this.method()` bypasses the transaction — `this` is the raw bean, not the proxy."

---

### Q5: Why does calling a @Transactional method from within the same class not work?
**Difficulty:** Hard | **Companies:** All companies using Spring transactions

❌ **Weak:** "Self-invocation bypasses the proxy."

✅ **Strong:** "Spring's `@Transactional` is implemented via AOP proxy. When Spring creates your service bean, it wraps it in a proxy (CGLIB or JDK dynamic proxy). The proxy intercepts method calls and adds transaction management. When external code calls `service.save()`, it calls the PROXY, which starts the transaction, then delegates to the real bean. When you call `this.helper()` from within `save()`, `this` refers to the REAL BEAN, not the proxy — the proxy is bypassed entirely, no transaction is started. Fix: (1) Inject the service into itself: `@Autowired MyService self; self.helper()`. (2) Fetch the proxy from ApplicationContext. (3) Refactor `helper()` into a separate `@Transactional` bean. (4) Use `AopContext.currentProxy()` (requires `exposeProxy=true`)."

## FEYNMAN CHECK

A Spring Bean is like a hotel guest registered at the front desk. When you check in (Spring scans @Component), the hotel records your preferences (BeanDefinition). Before you enter your room (bean creation), hotel staff prepare it (BeanFactoryPostProcessor). When you enter, your bags are carried (dependency injection). A bellhop greets you (BeanPostProcessor). You unpack and settle in (@PostConstruct). The hotel manages your stay (singleton lifecycle). When you check out (context close), you're escorted out (@PreDestroy). The AOP proxy is like a personal security guard who intercepts every call to you and checks if a transaction should be started — but only if people call through the front desk, not if you call yourself on your personal phone.
