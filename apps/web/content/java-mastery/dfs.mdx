# Depth-First Search (DFS)

Depth-First Search (DFS) is a graph traversal algorithm that explores as deep as possible along each branch before backtracking.

## 1. Core Theory & Mathematical Foundations

### The Stack Mechanism
DFS relies on a **Last-In, First-Out (LIFO)** policy, implemented either implicitly via the system call stack (recursive) or explicitly via a heap-allocated stack structure (iterative). The stack ensures that the most recently discovered vertex is processed next, forcing the path exploration to go deep rather than broad.

### DFS Tree Edge Classification
When running DFS on a graph, we construct a DFS spanning forest. Edges are classified into four types based on the discovery time interval $[d[u], f[u]]$ where $d[u]$ is the discovery time and $f[u]$ is the finish time of vertex $u$:

1. **Tree Edges**: Edges in the DFS forest. $v$ was discovered for the first time from $u$.
2. **Back Edges**: Edges $(u, v)$ pointing from a descendant $u$ to an ancestor $v$ in the DFS tree. These indicate the presence of a cycle.
3. **Forward Edges**: Non-tree edges $(u, v)$ pointing from an ancestor $u$ to a descendant $v$.
4. **Cross Edges**: All other edges. They connect vertices in different subtrees or vertices with no ancestor/descendant relationship.

```text
Interval Inclusion Properties:
- If [d[u], f[u]] is entirely nested within [d[v], f[v]], then u is a descendant of v.
- If [d[u], f[u]] and [d[v], f[v]] are entirely disjoint, they represent cross edges.
```

### Connected Components
For disconnected graphs, running DFS from a single source will not visit all vertices. We must iterate through all vertices $v \in V$, and if $v$ is unvisited, initiate a new DFS root. The number of times we initiate a new root equals the number of connected components (in undirected graphs) or strongly connected components (using Kosaraju's or Tarjan's algorithm for directed graphs).

### Recursive Risks: Stack Overflow
Recursive DFS utilizes the call stack. The default stack size in JVM is typically 1MB (configured via `-Xss`). For a degenerate graph (a straight linked list of $V$ nodes), recursive DFS reaches a depth of $V$. If $V > 10,000$, this routinely triggers a `StackOverflowError`. Production-grade engines use an explicit stack on the heap to avoid this limit.

## 3. Implementation (Java 21)

Here is a robust iterative DFS implementation using an explicit stack to prevent stack overflow, alongside discovery and finish time tracking.

```java
import java.util.*;

public class DepthFirstSearch {
    
    public record DFSResult(int[] discovery, int[] finish, List<Integer> topoOrder) {}

    public static DFSResult runDFS(int numVertices, Map<Integer, List<Integer>> adj) {
        int[] discovery = new int[numVertices];
        int[] finish = new int[numVertices];
        boolean[] visited = new boolean[numVertices];
        List<Integer> topoOrder = new ArrayList<>();
        int[] globalTime = new int[]{0}; // Use array to pass by reference

        for (int i = 0; i < numVertices; i++) {
            if (!visited[i]) {
                dfsIterative(i, adj, visited, discovery, finish, globalTime, topoOrder);
            }
        }
        Collections.reverse(topoOrder);
        return new DFSResult(discovery, finish, topoOrder);
    }

    private static void dfsIterative(int start, Map<Integer, List<Integer>> adj, 
                                     boolean[] visited, int[] discovery, 
                                     int[] finish, int[] globalTime, List<Integer> topoOrder) {
        
        // Stack stores: [node, neighborIndex]
        Deque<int[]> stack = new ArrayDeque<>();
        stack.push(new int[]{start, 0});
        visited[start] = true;
        globalTime[0]++;
        discovery[start] = globalTime[0];

        while (!stack.isEmpty()) {
            int[] frame = stack.peek();
            int curr = frame[0];
            int neighborIdx = frame[1];

            List<Integer> neighbors = adj.getOrDefault(curr, Collections.emptyList());

            if (neighborIdx < neighbors.size()) {
                frame[1]++; // Advance neighbor pointer for when we return
                int neighbor = neighbors.get(neighborIdx);
                if (!visited[neighbor]) {
                    visited[neighbor] = true;
                    globalTime[0]++;
                    discovery[neighbor] = globalTime[0];
                    stack.push(new int[]{neighbor, 0});
                }
            } else {
                stack.pop();
                globalTime[0]++;
                finish[curr] = globalTime[0];
                topoOrder.add(curr);
            }
        }
    }
}
```

## 4. Complexity Analysis

### Time Complexity: $\mathcal{O}(V + E)$
* Each vertex is pushed onto the stack exactly once because of the `visited` array.
* The algorithm iterates through every outgoing edge of each vertex exactly once when scanning neighbors.
* Thus, total time is linear with respect to vertices and edges: $\mathcal{O}(V + E)$.

### Space Complexity: $\mathcal{O}(V)$
* **Explicit Stack**: In the worst-case scenario (a linear graph), the stack contains $V$ frames.
* **Visited and Time Arrays**: Require $3 \times V$ slots, which simplifies to $\mathcal{O}(V)$.

## 5. Advanced Interview Practice

### Why is post-order DFS used for topological sorting instead of pre-order DFS?
Pre-order DFS processes a node before its descendants. If we used pre-order, a node $u$ with an edge $u \to v$ would be processed before $v$, which seems correct. However, if there are alternative paths or cross-edges, pre-order does not guarantee that all dependencies are fully resolved. Post-order processing guarantees that a node $u$ is only finished *after* all of its reachable descendants are finished. Reversing this post-order sequence ensures all dependencies are placed before their targets.

### How do you detect cycles in a directed graph vs. an undirected graph using DFS?
* **Directed Graph**: Maintain an active recursion stack set (or mark nodes as "currently visiting" using a tri-color scheme: White = unvisited, Gray = visiting, Black = finished). If you encounter a Gray node during traversal, it is a back-edge, indicating a cycle.
* **Undirected Graph**: A simple visited set is not enough because a node can see its parent. To detect a cycle, you must ignore the direct parent node. If you encounter any visited node that is *not* the direct parent of the current node, a cycle exists.
