## WHY
In OOP, sometimes a class represents a concept so high-level that it **shouldn't actually be instantiated**. You can have a `Dog` or `Cat`, but what does it mean to instantiate a generic `Animal`? An "Animal" in isolation makes no sense — it has no specific sound, no specific movement.

Furthermore, you often need to **define a strict contract**: every plugin written by a third-party developer for your browser must have a `load()` and `unload()` method. Every payment processor in your fintech app must implement `charge()` and `refund()`. You need to guarantee the API without dictating the implementation.

Abstraction solves both:
1. **Abstract Classes** — templates that cannot be instantiated; force child classes to implement specific behavior while sharing common state
2. **Interfaces** — pure contracts that define exactly what an object CAN DO, without caring about HOW it does it

The result: you can write code that works with `PaymentGateway` without knowing anything about Stripe, PayPal, or Apple Pay internally.

## THEORY

### Abstract Classes
A class prefixed with the `abstract` keyword. Key rules:

| Property | Abstract Class |
|---------|----------------|
| Can be instantiated? | ❌ No — `new Animal()` is a compile error |
| Can have state (fields)? | ✅ Yes |
| Can have constructors? | ✅ Yes (called via child's `super()`) |
| Can have concrete methods? | ✅ Yes |
| Can have abstract methods? | ✅ Yes — child MUST implement them |
| Multiple abstract parents? | ❌ No — single inheritance only |

**Use abstract class when**: sharing both state (fields) and behavior (methods) across IS-A related classes.

### Interfaces
A pure behavioral contract. Defined with `interface`:

| Property | Interface (Java 8+) |
|---------|---------------------|
| Can be instantiated? | ❌ No |
| Instance fields? | ❌ No — only `public static final` constants |
| Constructors? | ❌ No |
| Methods? | `abstract`, `default`, `static`, `private` (Java 9+) |
| Implement multiple? | ✅ Yes — a class can implement many interfaces |
| Extend other interfaces? | ✅ Yes — an interface can extend multiple interfaces |

**Use interface when**: defining CAN-DO capabilities across unrelated classes (e.g., both `Document` and `User` can `Printable`).

### Abstract Class vs Interface — The Decision Table

| Scenario | Use |
|---------|-----|
| Share code (fields + methods) between related IS-A classes | Abstract Class |
| Define a pure behavioral contract (CAN-DO) | Interface |
| Achieve multiple inheritance of behavior | Interface |
| Version compatibility — add methods without breaking existing code | Interface (default methods) |
| Need a constructor for setup | Abstract Class |
| Template Method Pattern | Abstract Class |

### Java 8+ Interface Evolution
Before Java 8, adding any new method to a published interface **broke every single implementing class worldwide**. Java 8 added:
- **`default` methods** — provide a default implementation, existing classes inherit it automatically
- **`static` methods** — utility methods on the interface itself

Java 9 added:
- **`private` methods** — shared helpers between `default` methods (not visible to implementing classes)

### Template Method Pattern with Abstract Classes
Abstract classes are perfect for the Template Method Pattern: define the algorithm skeleton in a final method, with abstract methods as "fill in the blank" hooks:
```java
abstract class DataProcessor {
    // Template method — sealed algorithm
    public final void process() {
        readData();     // abstract — child fills in
        transformData(); // abstract — child fills in
        writeResult();  // concrete — shared logic
    }

    protected abstract void readData();
    protected abstract void transformData();

    private void writeResult() {
        System.out.println("Writing to output...");
    }
}
```

## CODE

### Level 1 — Beginner: Abstract Class with Animal Hierarchy
```java
// Abstract class — cannot be instantiated directly
abstract class Animal {
    protected String name;
    protected int age;

    // Constructor — called via super() from child classes
    public Animal(String name, int age) {
        this.name = name;
        this.age = age;
    }

    // Abstract methods — child MUST implement these
    public abstract void makeSound();
    public abstract String getMovement();

    // Concrete method — shared by all animals
    public void sleep() {
        System.out.println(name + " is sleeping... Zzz");
    }

    public void eat(String food) {
        System.out.println(name + " is eating " + food);
    }

    @Override
    public String toString() {
        return getClass().getSimpleName() + "{name=" + name + ", age=" + age +
               ", moves=" + getMovement() + "}";
    }
}

class Dog extends Animal {
    private String breed;

    public Dog(String name, int age, String breed) {
        super(name, age);  // Must call abstract class constructor
        this.breed = breed;
    }

    @Override
    public void makeSound() { System.out.println(name + " barks: Woof! Woof!"); }

    @Override
    public String getMovement() { return "runs"; }

    public void fetch() { System.out.println(name + " fetches the ball!"); }
}

class Bird extends Animal {
    private double wingspan;

    public Bird(String name, int age, double wingspan) {
        super(name, age);
        this.wingspan = wingspan;
    }

    @Override
    public void makeSound() { System.out.println(name + " chirps: Tweet tweet!"); }

    @Override
    public String getMovement() { return "flies (wingspan: " + wingspan + "m)"; }
}

public class AnimalDemo {
    public static void main(String[] args) {
        // Animal a = new Animal("Test", 1); // ❌ Compile error — cannot instantiate abstract class!

        Animal[] animals = {
            new Dog("Rex", 3, "German Shepherd"),
            new Bird("Tweety", 1, 0.3)
        };

        for (Animal a : animals) {
            a.makeSound();    // Polymorphic
            a.sleep();        // Inherited concrete method
            System.out.println(a);
        }
    }
}
```

### Level 2 — Building Blocks: Interface with Multiple Implementation
```java
// Interface — pure contract
interface Flyable {
    int MAX_ALTITUDE = 10000; // implicitly public static final

    void fly();
    void land();

    default int getMaxAltitude() { return MAX_ALTITUDE; }
}

interface Swimmable {
    void swim();
    void dive(int depthMeters);
}

// Duck can BOTH fly and swim — multiple interface implementation
class Duck extends Animal implements Flyable, Swimmable {
    public Duck(String name) { super(name, 2); }

    @Override public void makeSound() { System.out.println(name + ": Quack!"); }
    @Override public String getMovement() { return "walks, flies, swims"; }

    // Must implement ALL interface methods
    @Override public void fly()  { System.out.println(name + " flaps wings and takes off!"); }
    @Override public void land() { System.out.println(name + " glides to a smooth landing"); }
    @Override public void swim() { System.out.println(name + " paddles in the water"); }
    @Override public void dive(int depth) { System.out.println(name + " dives to " + depth + "m"); }
}

class Fish extends Animal implements Swimmable {
    public Fish(String name) { super(name, 1); }

    @Override public void makeSound() { System.out.println(name + " blows bubbles"); }
    @Override public String getMovement() { return "swims"; }
    @Override public void swim()            { System.out.println(name + " glides through water"); }
    @Override public void dive(int depth)   { System.out.println(name + " dives effortlessly to " + depth + "m"); }
}

public class InterfaceDemo {
    public static void main(String[] args) {
        Duck duck = new Duck("Donald");

        // Polymorphism via interface references
        Flyable flier = duck;
        Swimmable swimmer = duck;

        flier.fly();
        swimmer.swim();
        duck.makeSound(); // Via Animal reference

        // A list of only Swimmable things
        java.util.List<Swimmable> aquatics = java.util.List.of(duck, new Fish("Nemo"));
        aquatics.forEach(s -> s.swim());
    }
}
```

### Level 3 — Intermediate: Default Methods and Conflict Resolution
```java
interface Logger {
    void log(String message);

    // Java 8 default method — backward-compatible addition
    default void logInfo(String message)  { log("[INFO] " + message); }
    default void logError(String message) { log("[ERROR] " + message); }
    default void logWarn(String message)  { log("[WARN] " + message); }
}

interface Auditable {
    void audit(String event);

    // Another default method
    default void logInfo(String message) {
        audit("[AUDIT-INFO] " + message);
    }
}

// Conflict! Both Logger and Auditable define logInfo()
// Java forces you to explicitly resolve it
class ApplicationService implements Logger, Auditable {
    @Override
    public void log(String message) {
        System.out.println("LOG: " + message);
    }

    @Override
    public void audit(String event) {
        System.out.println("AUDIT: " + event);
    }

    // MUST override to resolve the conflict
    @Override
    public void logInfo(String message) {
        Logger.super.logInfo(message);   // Explicitly choose which default
        Auditable.super.logInfo(message); // Can call both if needed
    }
}

// Java 9+ private interface methods
interface DataProcessor {
    default void processCSV(String data)  { process(data, "CSV"); }
    default void processJSON(String data) { process(data, "JSON"); }

    // private — shared internal logic, not visible to implementing classes
    private void process(String data, String format) {
        validate(data);
        System.out.println("Processing " + format + ": " + data);
    }

    private void validate(String data) {
        if (data == null || data.isBlank()) throw new IllegalArgumentException("Data required");
    }
}
```

### Level 4 — Advanced: Template Method Pattern and Abstract Factory
```java
// Template Method Pattern — abstract class defines algorithm skeleton
abstract class ReportGenerator {

    // Template method — FINAL prevents overriding the algorithm structure
    public final String generate(String title) {
        String header  = createHeader(title);   // abstract — child implements
        String content = createContent();       // abstract — child implements
        String footer  = createFooter();        // concrete — shared

        return applyFormatting(header + "\n" + content + "\n" + footer);
    }

    protected abstract String createHeader(String title);
    protected abstract String createContent();

    protected String createFooter() {
        return "Generated at: " + java.time.LocalDateTime.now();
    }

    // Hook method — optional override (default does nothing)
    protected String applyFormatting(String content) {
        return content; // Default: no formatting
    }
}

class PDFReportGenerator extends ReportGenerator {
    private final java.util.List<String> rows;

    PDFReportGenerator(java.util.List<String> rows) { this.rows = rows; }

    @Override
    protected String createHeader(String title) {
        return "=== PDF: " + title.toUpperCase() + " ===";
    }

    @Override
    protected String createContent() {
        return String.join("\n", rows);
    }

    @Override
    protected String applyFormatting(String content) {
        return content.replace("===", "***"); // PDF-specific formatting
    }
}

class HTMLReportGenerator extends ReportGenerator {
    private final java.util.List<String> rows;

    HTMLReportGenerator(java.util.List<String> rows) { this.rows = rows; }

    @Override
    protected String createHeader(String title) {
        return "<h1>" + title + "</h1>";
    }

    @Override
    protected String createContent() {
        StringBuilder sb = new StringBuilder("<ul>");
        rows.forEach(r -> sb.append("<li>").append(r).append("</li>"));
        return sb.append("</ul>").toString();
    }
}

public class TemplateMethodDemo {
    public static void main(String[] args) {
        java.util.List<String> data = java.util.List.of("Alice: $1000", "Bob: $800");

        ReportGenerator pdf  = new PDFReportGenerator(data);
        ReportGenerator html = new HTMLReportGenerator(data);

        System.out.println(pdf.generate("Sales Report"));
        System.out.println("---");
        System.out.println(html.generate("Sales Report"));
    }
}
```

### Level 5 — Expert: Functional Interfaces and the Abstraction of Behavior
```java
import java.util.function.*;
import java.util.List;

// Functional Interface — interface with exactly ONE abstract method
// Can be implemented by a lambda expression
@FunctionalInterface
interface Validator<T> {
    boolean validate(T value);

    // Default combinator methods
    default Validator<T> and(Validator<T> other) {
        return value -> this.validate(value) && other.validate(value);
    }

    default Validator<T> or(Validator<T> other) {
        return value -> this.validate(value) || other.validate(value);
    }

    default Validator<T> negate() {
        return value -> !this.validate(value);
    }
}

// Usage: compose validators without writing new classes
public class ValidatorDemo {
    public static void main(String[] args) {
        Validator<String> notNull     = s -> s != null;
        Validator<String> notEmpty    = s -> !s.isBlank();
        Validator<String> minLength   = s -> s.length() >= 8;
        Validator<String> hasUpper    = s -> s.chars().anyMatch(Character::isUpperCase);
        Validator<String> hasDigit    = s -> s.chars().anyMatch(Character::isDigit);

        // Compose validators: all conditions must pass
        Validator<String> passwordPolicy = notNull
            .and(notEmpty)
            .and(minLength)
            .and(hasUpper)
            .and(hasDigit);

        List<String> passwords = List.of("weak", "StrongPass1", "nodigits", "N0Short");

        passwords.forEach(p ->
            System.out.printf("'%s' -> %s%n", p, passwordPolicy.validate(p) ? "VALID" : "INVALID")
        );
        // 'weak'       -> INVALID (too short, no digit, no upper)
        // 'StrongPass1'-> VALID
        // 'nodigits'   -> INVALID (no digit, no upper)
        // 'N0Short'    -> INVALID (too short)
    }
}
```

## REAL_WORLD

### Hexagonal Architecture (Ports and Adapters) — Built on Abstraction
The most important architectural pattern in modern enterprise Java uses interfaces as **Ports** that completely isolate the business core from infrastructure:

```java
// PORT — pure interface (abstraction) — the business core depends only on this
public interface UserRepository {
    Optional<User> findById(Long id);
    User save(User user);
    List<User> findByEmail(String email);
}

// ADAPTER 1 — JPA implementation (lives in infrastructure layer)
@Repository
public class JpaUserRepository implements UserRepository {
    @PersistenceContext private EntityManager em;

    @Override
    public Optional<User> findById(Long id) {
        return Optional.ofNullable(em.find(User.class, id));
    }
    // ...
}

// ADAPTER 2 — In-memory implementation for testing
public class InMemoryUserRepository implements UserRepository {
    private final Map<Long, User> store = new HashMap<>();
    private long idCounter = 1;

    @Override
    public Optional<User> findById(Long id) { return Optional.ofNullable(store.get(id)); }

    @Override
    public User save(User user) {
        user.setId(idCounter++);
        store.put(user.getId(), user);
        return user;
    }
    // ...
}

// BUSINESS CORE — uses only the Port interface, knows nothing about databases
@Service
public class UserService {
    private final UserRepository userRepository; // Port reference

    // Spring injects JpaUserRepository in production
    // Tests inject InMemoryUserRepository for speed
    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }
}
```
This is abstraction at the architectural level — the business logic is completely isolated from database technology.

## INTERVIEW

**Q: What is the difference between an Abstract Class and an Interface?**
A: An abstract class can have instance fields, constructors, methods with any access modifier, and concrete implementations. A class can only extend ONE abstract class. An interface cannot have instance fields or constructors (only `public static final` constants), and a class can implement MULTIPLE interfaces. Use abstract class for IS-A shared state/behavior; use interface for CAN-DO contracts across unrelated classes.

**Q: Why did Java introduce `default` methods in interfaces (Java 8)?**
A: For backward compatibility. When the JDK team added `stream()`, `forEach()` to `Collection` in Java 8, every existing class implementing `Collection` would have broken if they were abstract methods. Making them `default` methods lets the interface provide a default body, so existing code compiles unmodified.

**Q: Can you instantiate an abstract class?**
A: No. You cannot use `new AbstractClass()`. However, when you instantiate a concrete child class, the abstract class's constructor IS called via the `super()` chain to initialize its fields. Also, anonymous inner classes technically instantiate an abstract class inline: `Animal a = new Animal("test") { void makeSound() {} }`.

**Q: What is the Template Method Pattern?**
A: Define an algorithm's skeleton in a `final` method in an abstract class, with `abstract` method "hooks" that subclasses fill in. The algorithm's structure is fixed (final), but specific steps are customizable. Example: `DataProcessor.process()` always calls `readData()` → `transform()` → `write()`, but each subclass implements `readData()` and `transform()` differently (CSV vs JSON).

**Q: What is a Functional Interface?**
A: An interface with exactly one abstract method (SAM — Single Abstract Method). It can be implemented by a lambda expression. The `@FunctionalInterface` annotation asks the compiler to verify this. Examples: `Runnable`, `Callable`, `Comparator`, `Predicate<T>`, `Function<T,R>`.

## FEYNMAN CHECK

Explain to a 10-year-old:
> An abstract class is like a **recipe template** that says "this recipe MUST have a 'main ingredient' step and a 'cooking method' step, but I'll leave those blank for you to fill in." All my cookie recipes, cake recipes, and bread recipes follow the same template but fill in the blanks differently. An interface is like a **job description** — "Any delivery driver must be able to `pickUp()` and `deliver()`." I don't care if you use a bike, car, or drone — just fulfill the job description.
