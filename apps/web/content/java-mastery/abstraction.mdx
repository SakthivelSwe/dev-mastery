## WHY
In OOP, sometimes a class represents a concept so high-level that it shouldn't actually exist on its own. For example, you can instantiate a `Dog` or a `Cat`, but what does it mean to instantiate a generic `Animal`? It doesn't make sense.

Furthermore, you often want to define a strict contract. If you are building a plug-in system for a web browser, you need to guarantee that every plugin written by a third-party developer has a `load()` method and an `unload()` method. 

Abstraction solves both of these problems. It allows you to define templates (Abstract Classes) that force child classes to implement certain behaviors, and Contracts (Interfaces) that define exactly what a class must do, without caring at all about how it does it. 

## THEORY
### Abstract Classes
An abstract class is a class prefixed with the `abstract` keyword. 
- **Cannot be instantiated**: You cannot do `new Animal()`.
- **Can have state**: It can have fields, constructors, and fully implemented methods.
- **Can have abstract methods**: Methods without a body. Any non-abstract child class *must* override and implement these methods.
Use an abstract class when classes share a fundamental "IS-A" identity and need to share common state (fields).

### Interfaces
An interface defines a pure behavioral contract.
- **No state**: Prior to Java 8, interfaces could only contain `public static final` constants and `public abstract` methods. They cannot have instance fields or constructors.
- **Multiple Inheritance**: A class can implement multiple interfaces (e.g., `class Car implements Drivable, Sellable`).
Use an interface when defining a "CAN-DO" capability across completely unrelated classes (e.g., a `Document` and a `User` can both implement `Printable`).

### Java 8+ Interface Evolution
Historically, adding a new method to an Interface broke every single class that implemented it worldwide. To fix this, Java 8 introduced `default` methods in interfaces. This allows an interface to provide a default implementation, meaning existing classes don't have to implement the new method. Java 9 added `private` methods to interfaces to help organize these default methods.

## CODE

### Level 1 — Beginner
```java
// ABSTRACT CLASS
abstract class Animal {
    String name; // Can have state
    
    Animal(String name) { this.name = name; }
    
    // Abstract method (no body)
    abstract void makeSound(); 
}

class Dog extends Animal {
    Dog() { super("Dog"); }
    
    @Override
    void makeSound() { System.out.println("Woof"); }
}
```

### Level 2 — Building Blocks
```java
// INTERFACE
interface Flyable {
    // Implicitly public abstract
    void fly(); 
}

// A class can extend one class, but implement many interfaces
class Bird extends Animal implements Flyable {
    Bird() { super("Bird"); }
    
    @Override void makeSound() { System.out.println("Tweet"); }
    @Override public void fly() { System.out.println("Flapping wings"); }
}
```

### Level 3 — Intermediate
```java
interface Logger {
    void log(String msg);
    
    // Java 8 Default Method
    // Doesn't break implementing classes if added later
    default void logError(String msg) {
        log("ERROR: " + msg);
    }
}
```

### Level 4 — Advanced
```java
// Resolving Default Method Conflicts
interface A { default void show() { System.out.println("A"); } }
interface B { default void show() { System.out.println("B"); } }

// If a class implements both, the compiler forces you to resolve the clash
class C implements A, B {
    @Override
    public void show() {
        // Explicitly choose which default implementation to run
        A.super.show(); 
    }
}
```

## REAL_WORLD
Interfaces are the foundational building block of the **Spring Framework** and **Hexagonal Architecture**.

In a real enterprise application, you never expose concrete implementations between layers. A `PaymentController` talks to a `PaymentService` interface, NOT the `StripePaymentServiceImpl` concrete class.
```java
// The Interface Contract
public interface PaymentService {
    void process(double amount);
}

// The Concrete Implementation
@Service
public class StripePaymentServiceImpl implements PaymentService { ... }
```
Why? Because when writing Unit Tests for the Controller, you can effortlessly create a mock `PaymentService` (using Mockito) without actually charging credit cards. If the Controller depended on the concrete Stripe class, testing would be impossible without making real API calls. Interfaces decouple dependencies.

## INTERVIEW
**Q: What is the difference between an Abstract Class and an Interface?**
A: An Abstract class can have instance fields, constructors, and any access modifiers (`protected`, `private`), and a class can only extend one abstract class (IS-A relationship). An Interface cannot have instance fields or constructors (all fields are implicitly `public static final`), and a class can implement multiple interfaces (CAN-DO behavior contract).

**Q: Why did Java introduce `default` methods in interfaces?**
A: To enable backward compatibility. When the JDK maintainers wanted to add the `stream()` method to the `Collection` interface in Java 8, it would have broken millions of existing classes worldwide that implemented `Collection`. By making `stream()` a `default` method, existing classes automatically inherited the behavior without failing to compile.

**Q: Can you instantiate an abstract class?**
A: No, you cannot instantiate an abstract class using the `new` keyword. However, when you instantiate a concrete child class, the abstract class's constructor *is* executed via the `super()` chain to initialize its fields.
