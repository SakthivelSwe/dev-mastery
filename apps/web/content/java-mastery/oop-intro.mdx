## WHY
Before Object-Oriented Programming (OOP), developers wrote procedural code. Data (like `int[] userAges`) and functions (`void printUser(int age)`) were completely separate. As systems grew to millions of lines, managing which function was allowed to modify which global variable became a nightmare, leading to uncontrollable side effects.

OOP solved this by bundling Data (State) and Functions (Behavior) together into a single logical unit: the **Object**. A Class is the blueprint; an Object is the physical manifestation in memory. This paradigm allows us to model real-world concepts (like a `BankAccount` with a `balance` and a `withdraw()` method) securely, ensuring that the state cannot be corrupted by external, unrelated functions.

## THEORY
### The Object Header and Heap Layout
When you write `User u = new User();`, three things happen:
1. **Memory Allocation**: The JVM allocates contiguous space on the **Heap**.
2. **Object Header**: The JVM prepends a 12-16 byte header. This contains the **Mark Word** (used for Garbage Collection age, locking/synchronization state, and identity hashcode) and the **Klass Pointer** (pointing to the class definition in Metaspace).
3. **Initialization**: The JVM zeroes out the memory, executes the constructor, and then places the reference (pointer) on the Stack.

### The `this` Reference
Inside a class method, how does the method know *which* object's data to manipulate? The JVM implicitly passes a hidden parameter to every non-static method: the `this` reference. `this` is simply a pointer to the current object on the Heap.

## CODE

### Level 1 — Beginner
```java
// The Blueprint (Class)
public class Car {
    String model; // State (Field)
    
    // Constructor
    public Car(String model) {
        this.model = model; 
    }
    
    // Behavior (Method)
    public void start() {
        System.out.println(model + " starting...");
    }
}
```

### Level 2 — Building Blocks
```java
public class Main {
    public static void main(String[] args) {
        // Instantiation: 'new' allocates Heap memory.
        // myCar is just a pointer on the Stack.
        Car myCar = new Car("Tesla");
        myCar.start();
    }
}
```

### Level 3 — Intermediate
```java
public class Singleton {
    // Static fields belong to the Class (Metaspace), not the Object (Heap)
    private static Singleton instance;
    
    // Private constructor prevents 'new'
    private Singleton() {}
    
    public static Singleton getInstance() {
        if (instance == null) instance = new Singleton();
        return instance;
    }
}
```

### Level 4 — Advanced
```java
import org.openjdk.jol.info.ClassLayout;

public class ObjectSize {
    public static void main(String[] args) {
        // Using Java Object Layout (JOL) tool to inspect the JVM memory layout
        Object obj = new Object();
        // Prints the 12-byte header + 4-byte padding (due to 8-byte alignment)
        System.out.println(ClassLayout.parseInstance(obj).toPrintable());
    }
}
```

## REAL_WORLD
In Spring Boot, almost every class you write is a `@Component` or `@Service`. By default, Spring treats these classes as **Singletons**. 
Spring creates exactly ONE Object from your Class blueprint when the application starts, and stores it in the Spring Application Context. Every time a web request comes in, multiple threads use that exact same single Object. This is why you must **never** store user-specific state in a Spring Service field—it will cause massive cross-thread data corruption!

## INTERVIEW
**Q: What is the difference between a Class and an Object?**
A: A Class is a compile-time logical blueprint stored in the JVM's Metaspace. An Object is a runtime, physical instance of that blueprint stored on the Heap, complete with an Object Header and memory allocated for its specific field values.

**Q: Does every object have an Object Header? What is inside it?**
A: Yes. Every object in Java has a header (typically 12-16 bytes). It contains the Mark Word (stores GC generation age, locking state for synchronized blocks, and the identity hashcode) and a Klass Pointer (pointing to the class definition in Metaspace so the JVM knows the object's type).
