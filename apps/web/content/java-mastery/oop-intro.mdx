## WHY
Before Object-Oriented Programming (OOP), developers wrote procedural code — long C programs where data lived in global variables and functions operated on them from anywhere. A program managing bank accounts might have a global `int balance` variable and fifty different functions across fifty different files all modifying it with zero access control. When a bug appeared — "How did balance become negative?" — you had to grep through all fifty files to find the culprit.

OOP solved this with a revolutionary idea: **bundle the data and the functions that operate on it into a single, self-governing unit called an Object**. The data (state) became `private` fields. The functions became `public` methods that enforce rules before touching the data. Now `balance` can only change via `deposit()` and `withdraw()`, and both methods enforce their own rules. External code cannot bypass them.

Four pillars emerged from this idea:
1. **Encapsulation** — Hide state, expose behavior
2. **Inheritance** — Child classes inherit parent behavior
3. **Polymorphism** — One method, many behaviors
4. **Abstraction** — Expose what, hide how

This topic covers the foundation: **Classes and Objects**.

## THEORY

### What is a Class?
A **Class** is a compile-time blueprint. It defines:
- **Fields** — the state an object will hold (e.g., `String name`, `int age`)
- **Methods** — the behavior the object can perform (e.g., `void greet()`)
- **Constructors** — special methods that set up the object's initial state

A class itself uses NO memory at runtime. It exists only in the JVM's Metaspace as bytecode.

### What is an Object?
An **Object** is a runtime instance of a class, allocated on the **JVM Heap**. Each object has:
- Its own copy of every non-static field
- A hidden 12-16 byte **Object Header** (Mark Word + Klass Pointer)
- NO copy of methods — all objects of the same class share the same bytecode

```
Stack                        Heap
------                  ----------------
| car1 |  --------->   | Object Header |
| car2 |  ----.        | model: "Tesla" |
------        |        | color: "Red"   |
              |        ----------------
              `------> | Object Header |
                       | model: "BMW"  |
                       | color: "Blue" |
                       ----------------
```

### The Object Header (JVM Internals)
When you write `new Car()`, the JVM allocates these bytes on the Heap **before** your fields:

| Part | Size | Purpose |
|------|------|---------|
| Mark Word | 8 bytes | GC age, lock state (for `synchronized`), identity hashcode |
| Klass Pointer | 4 bytes (compressed OOPs) | Pointer to class metadata in Metaspace |
| Padding | 0-4 bytes | Aligns object size to 8-byte boundary |

Total minimum size of ANY Java object: **16 bytes** (before your own fields).

### Static vs Instance Members

| Feature | `static` | instance |
|---------|----------|---------|
| Belongs to | The Class (Metaspace) | The Object (Heap) |
| Memory | One copy shared by all | One copy per object |
| Access | `ClassName.field` | `objectRef.field` |
| Use case | Constants, utility methods | Per-object data |

### The `this` Reference
Every non-static method receives a hidden first parameter: `this` — a pointer to the current object on the Heap. The compiler rewrites:
```java
car.start()  →  Car.start(car)   // 'car' becomes 'this' inside the method
```
This is how the method knows which object's `model` to print.

### Constructor Chaining
If you don't define a constructor, the compiler generates a no-arg constructor. If you define any constructor, the compiler removes the generated one. Constructors can call each other using `this(...)`:
```java
public Car() {
    this("Unknown", "White"); // calls the 2-arg constructor
}
public Car(String model, String color) {
    this.model = model;
    this.color = color;
}
```

## CODE

### Level 1 — Beginner: Your First Class and Object
```java
// The Blueprint (Class definition)
public class Car {
    // Fields — per-object state
    String model;
    String color;
    int speed;

    // Constructor — initializes the object
    public Car(String model, String color) {
        this.model = model;  // 'this.model' = the object's field
        this.color = color;  // 'color' alone = the constructor parameter
    }
    
    // Method — behavior
    public void start() {
        System.out.println(model + " is starting...");
    }

    public void accelerate(int amount) {
        speed += amount;
        System.out.println(model + " speed: " + speed + " km/h");
    }
}

public class Main {
    public static void main(String[] args) {
        // 'new' allocates memory on the Heap
        // 'myCar' is a reference (pointer) on the Stack
        Car myCar = new Car("Tesla", "Red");
        myCar.start();            // Output: Tesla is starting...
        myCar.accelerate(60);     // Output: Tesla speed: 60 km/h

        // Create a second object — completely independent state
        Car other = new Car("BMW", "Blue");
        other.accelerate(100);    // Output: BMW speed: 100 km/h

        System.out.println(myCar.speed);  // Still 60 — not 100!
    }
}
```

### Level 2 — Building Blocks: Multiple Constructors and `static`
```java
public class BankAccount {
    // static field — ONE shared copy for all BankAccount objects
    private static int totalAccounts = 0;

    // instance fields — each object has its own copy
    private final String accountNumber;
    private double balance;
    private String owner;

    // No-arg constructor chains to full constructor
    public BankAccount() {
        this("ANON-" + System.currentTimeMillis(), 0.0);
    }

    // Full constructor
    public BankAccount(String owner, double initialBalance) {
        this.owner = owner;
        this.balance = initialBalance;
        this.accountNumber = "ACC-" + ++totalAccounts;
    }

    public void deposit(double amount) {
        if (amount <= 0) throw new IllegalArgumentException("Deposit must be positive");
        this.balance += amount;
        System.out.printf("[%s] Deposited %.2f. Balance: %.2f%n",
                          accountNumber, amount, balance);
    }

    public void withdraw(double amount) {
        if (amount > balance) throw new IllegalStateException("Insufficient funds");
        this.balance -= amount;
    }

    public double getBalance() { return balance; }

    // static method — can only access static fields
    public static int getTotalAccounts() {
        return totalAccounts;
    }

    @Override
    public String toString() {
        return String.format("BankAccount{id=%s, owner=%s, balance=%.2f}",
                             accountNumber, owner, balance);
    }

    public static void main(String[] args) {
        BankAccount alice = new BankAccount("Alice", 1000);
        BankAccount bob   = new BankAccount("Bob",   500);

        alice.deposit(250);
        bob.deposit(100);

        System.out.println(BankAccount.getTotalAccounts()); // 2
        System.out.println(alice);  // BankAccount{id=ACC-1, owner=Alice, balance=1250.00}
    }
}
```

### Level 3 — Intermediate: Object References, Null, and `equals()`
```java
public class ReferenceDemo {
    public static void main(String[] args) {
        // ---- Reference Equality vs Value Equality ----
        BankAccount acc1 = new BankAccount("Alice", 500);
        BankAccount acc2 = new BankAccount("Alice", 500);

        // == compares memory addresses (ARE THEY THE SAME OBJECT?)
        System.out.println(acc1 == acc2);          // false — different Heap addresses

        // .equals() compares logical CONTENT (by default, same as ==)
        // You must override equals() to define logical equality
        System.out.println(acc1.equals(acc2));     // false unless you override equals

        // ---- Aliasing Trap ----
        BankAccount alias = acc1;  // alias and acc1 point to the SAME object
        alias.deposit(1000);
        System.out.println(acc1.getBalance()); // 1500 — BOTH changed!

        // ---- The Null Pointer ----
        BankAccount phantom = null;  // A reference that points to NOTHING
        try {
            phantom.deposit(100);    // NullPointerException!
        } catch (NullPointerException e) {
            System.out.println("Cannot call methods on a null reference");
        }
    }

    // Java passes references BY VALUE — this does NOT swap the original vars
    static void swapAttempt(BankAccount a, BankAccount b) {
        BankAccount temp = a;
        a = b;      // Changes local copy only
        b = temp;   // Changes local copy only
    }
}
```

### Level 4 — Advanced: Singleton Pattern and Builder Pattern
```java
import java.util.Objects;

// Thread-Safe Singleton — Bill Pugh / Initialization-on-demand Holder
public class DatabaseConnection {
    private final String url;

    private DatabaseConnection() {
        this.url = "jdbc:postgresql://localhost:5432/devmastery";
    }

    // Inner class is loaded only when getInstance() is first called
    // JVM class loading is inherently thread-safe
    private static class Holder {
        static final DatabaseConnection INSTANCE = new DatabaseConnection();
    }

    public static DatabaseConnection getInstance() {
        return Holder.INSTANCE;
    }

    public void query(String sql) {
        System.out.println("Executing on [" + url + "]: " + sql);
    }
}

// Builder pattern for objects with many optional parameters
public class HttpRequest {
    private final String method;
    private final String url;
    private final String body;
    private final int timeout;

    private HttpRequest(Builder b) {
        this.method  = b.method;
        this.url     = b.url;
        this.body    = b.body;
        this.timeout = b.timeout;
    }

    public static class Builder {
        private String method = "GET";
        private String url;
        private String body;
        private int timeout = 30;

        public Builder url(String url)       { this.url = url; return this; }
        public Builder method(String method) { this.method = method; return this; }
        public Builder body(String body)     { this.body = body; return this; }
        public Builder timeout(int s)        { this.timeout = s; return this; }
        public HttpRequest build()           { return new HttpRequest(this); }
    }

    // Always override both equals() AND hashCode() together!
    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof HttpRequest)) return false;
        HttpRequest r = (HttpRequest) o;
        return timeout == r.timeout &&
               Objects.equals(method, r.method) &&
               Objects.equals(url, r.url);
    }

    @Override
    public int hashCode() {
        return Objects.hash(method, url, timeout);
    }

    public static void main(String[] args) {
        HttpRequest req = new HttpRequest.Builder()
            .url("https://api.devmastery.com/users")
            .method("POST")
            .body("{\"name\":\"Alice\"}")
            .timeout(60)
            .build();
    }
}
```

### Level 5 — Expert: JVM Object Layout with JOL
```java
import org.openjdk.jol.info.ClassLayout;
import org.openjdk.jol.info.GraphLayout;
import org.openjdk.jol.vm.VM;

public class JvmObjectInternals {

    static class Point {
        int x;
        int y;
    }

    static class Node {
        int value;
        Node next;  // Reference = 4 bytes (compressed OOPs)
    }

    public static void main(String[] args) {
        System.out.println(VM.current().details());

        // --- Object Header and Field Layout ---
        // OFFSET SIZE TYPE DESCRIPTION
        //      0   12      (object header: mark word + klass pointer)
        //     12    4  int Point.x
        //     16    4  int Point.y
        //     20    4      (alignment padding)
        // Instance size: 24 bytes
        System.out.println(ClassLayout.parseClass(Point.class).toPrintable());

        // --- Identity HashCode changes the Mark Word ---
        Point p = new Point();
        System.out.println("Before hashCode:");
        System.out.println(ClassLayout.parseInstance(p).toPrintable());

        int hash = System.identityHashCode(p);
        System.out.println("Hash: " + hash);

        System.out.println("After hashCode (25 bits now stored in Mark Word):");
        System.out.println(ClassLayout.parseInstance(p).toPrintable());

        // --- Total Object Graph Deep Memory ---
        Node head = new Node();
        head.next = new Node();
        head.next.next = new Node();

        // Total 3 Node objects * 24 bytes = 72 bytes
        System.out.println(GraphLayout.parseInstance(head).toFootprint());
    }
}
```

## REAL_WORLD

### Spring Boot: Every Bean IS a Singleton Object
In Spring Boot, classes annotated with `@Component`, `@Service`, `@Repository`, or `@Controller` are managed as **Singleton Beans** — Spring creates exactly **one object instance** from your class blueprint at startup and stores it in the ApplicationContext.

```java
@Service
public class UserService {
    // DANGER: Mutable instance field in a Singleton!
    private List<String> recentSearches = new ArrayList<>(); // WRONG!

    // ALL web requests share this ONE object — User A's data leaks to User B!
    public void search(String term) {
        recentSearches.add(term); // Thread-unsafe AND logically broken
    }
}
```

**The Correct Pattern**: Beans should be **stateless**. All data flows in via method parameters.

```java
@Service
public class UserService {
    private final UserRepository userRepo; // Injected dependency — safe!

    public UserService(UserRepository userRepo) {
        this.userRepo = userRepo; // Constructor injection — Spring best practice
    }

    public List<User> search(String term) {
        // Local variable — thread-safe, request-scoped
        return userRepo.findByNameContaining(term);
    }
}
```

### JPA Entities and `equals()`/`hashCode()`
When `@Entity` classes are used with `HashSet` or as `HashMap` keys, broken `equals`/`hashCode` causes entities to "disappear" from collections after Hibernate assigns their database ID:

```java
@Entity
public class User {
    @Id @GeneratedValue
    private Long id;
    private String username;

    // Correct: base equals/hashCode on business key, not generated id
    @Override
    public boolean equals(Object o) {
        if (!(o instanceof User)) return false;
        User u = (User) o;
        return username != null && username.equals(u.username);
    }

    @Override
    public int hashCode() { return Objects.hash(username); }
}
```

## INTERVIEW

**Q: What is the difference between a Class and an Object?**
A: A Class is a compile-time logical blueprint stored as bytecode in the JVM's Metaspace — no runtime memory consumed. An Object is a runtime physical instance allocated on the JVM Heap with its own copy of all instance fields plus a 12-16 byte Object Header. Multiple objects can exist from a single class.

**Q: What is inside the Object Header?**
A: Two parts: (1) **Mark Word** (8 bytes) — stores identity hashcode (25 bits), GC generation age (4 bits), and lock state for `synchronized`; (2) **Klass Pointer** (4 bytes with compressed OOPs) — points to class metadata in Metaspace for type checking and method dispatch.

**Q: Why does `==` fail for objects but work for primitives?**
A: For primitives, `==` compares actual bit values. For object references, `==` compares Heap addresses. Two separate `new String("hello")` objects hold identical content at different addresses, so `==` returns `false`. Use `.equals()` to compare content.

**Q: What happens when no references point to an object?**
A: It becomes eligible for Garbage Collection. The GC traces from GC roots and reclaims Heap memory for unreachable objects. `System.gc()` is a hint only — the JVM decides when to actually collect.

**Q: What is the difference between `static` and instance methods?**
A: Instance methods receive the implicit `this` pointer and access both instance and static members. Static methods belong to the class, receive no `this`, and can only access static members directly. Static methods are resolved at compile-time (no vtable lookup).

**Q: Why is the Singleton pattern used and what are its dangers?**
A: Singletons ensure one global instance — useful for stateless services, loggers, connection pools. Dangers: shared mutable state causes thread-safety bugs; testability is harder because you can't inject a mock; hidden coupling between classes.
