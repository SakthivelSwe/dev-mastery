# Objects, Interfaces, and Type Aliases | Level: 2

In TypeScript, we define object shapes using **Interfaces** or **Type Aliases**. While highly similar, they have distinct behaviors, compile-time mechanics, and performance characteristics.

---

### Interfaces vs. Type Aliases

*   **Interface**: Describes the shape of an object. It can be extended and supports **declaration merging**.
*   **Type Alias**: Can define any type, including primitives, unions, intersections, tuples, and mapped types. It cannot participate in declaration merging.

#### Declaration Merging (Interfaces only)
If you declare two interfaces with the same name in the same scope, TypeScript merges them into a single definition. This is highly useful for extending third-party library types.

```typescript
interface User {
  name: string;
}
interface User {
  role: "admin" | "user";
}

// Merged User interface has both properties
const admin: User = { name: "Bob", role: "admin" };
```

#### Type Aliases are Closed
Type aliases cannot be redeclared or merged.

```typescript
type Vehicle = { brand: string };
type Vehicle = { model: string }; 
// error TS2300: Duplicate identifier 'Vehicle'.
```

---

### Extending Shapes

#### Extending Interfaces (via `extends`)
```typescript
interface Animal {
  legs: number;
}
interface Dog extends Animal {
  bark(): void;
}
```

#### Extending Types (via Intersection `&`)
```typescript
type Point2D = { x: number; y: number };
type Point3D = Point2D & { z: number };
```

*Performance Tip*: The TypeScript compiler optimizes interface inheritance (`extends`) better than type intersections (`&`) because it caches interface shapes.

### Modifiers: Optional and Readonly

#### Optional Properties (`?`)
Optional properties can be omitted during initialization. Their type implicitly becomes `Type | undefined`.

```typescript
interface Profile {
  username: string;
  avatarUrl?: string; // string | undefined
}
```
