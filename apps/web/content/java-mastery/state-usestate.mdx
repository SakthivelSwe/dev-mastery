# useState — State Batching, Functional Updates, Initialization

## State vs. Variables: Triggering the Reconciliation Loop

Standard local variables do not persist across renders. When a component re-renders, local variables are re-initialized to their default values. Furthermore, mutating a local variable does not notify React that it needs to update the UI.

`useState` registers a state cell on the component’s internal Fiber node. This cell persists across renders, and its setter function triggers the reconciliation loop to update the DOM.

```typescript
import { useState } from 'react';

export const Counter: React.FC = () => {
  let localCount = 0; // Will reset on every render, doesn't trigger UI updates
  const [stateCount, setStateCount] = useState<number>(0);

  return (
    <div>
      {/* Mutating this does nothing visually */}
      <button onClick={() => { localCount++; }}>Local: {localCount}</button>
      
      {/* This triggers reconciliation and updates the screen */}
      <button onClick={() => setStateCount(stateCount + 1)}>State: {stateCount}</button>
    </div>
  );
};
```

## Automatic State Batching (React 18+)

In React 18 and 19, multiple state updates inside event handlers, promises, timeouts, or native events are automatically **batched** into a single re-render. This prevents the browser from performing intermediate layout calculations.

```typescript
import { useState } from 'react';

export const BatchingDemo: React.FC = () => {
  const [count, setCount] = useState(0);
  const [flag, setFlag] = useState(false);

  const handleAsyncClick = () => {
    // Even though these are asynchronous, React 18+ batches them
    fetch('/api/data').then(() => {
      setCount(c => c + 1);
      setFlag(f => !f);
      // Result: Only ONE re-render occurs here!
    });
  };

  return <button onClick={handleAsyncClick}>Increment & Toggle</button>;
};
```

## The Functional Update Form: Solving Stale Closures

Because state updates are batched and asynchronous, calling a state setter multiple times in a single render block using the standard value form will result in unexpected behavior due to **stale closures**.

```typescript
import { useState } from 'react';

export const CounterFix: React.FC = () => {
  const [count, setCount] = useState(0);

  const handleStaleIncrement = () => {
    // Both calls capture the value of `count` from the current render scope (e.g., 0)
    setCount(count + 1); // setCount(0 + 1)
    setCount(count + 1); // setCount(0 + 1)
    // Result after click: count is 1, not 2!
  };

  const handleFunctionalIncrement = () => {
    // The functional form passes the absolute, up-to-date state from the fiber queue
    setCount(prev => prev + 1);
    setCount(prev => prev + 1);
    // Result after click: count is 2!
  };

  return (
    <div>
      <button onClick={handleStaleIncrement}>Stale (Incr 1)</button>
      <button onClick={handleFunctionalIncrement}>Functional (Incr 2)</button>
    </div>
  );
};
```

## Lazy State Initialization

If your default state requires an expensive calculation (e.g., reading from `localStorage`, parsing a large JSON string, or performing heavy array operations), passing the value directly to `useState` runs that calculation on **every single render**, even though React only uses the result on the initial mount.

To prevent this performance leak, pass a **callback function** (a "lazy initializer"). React will execute this function exactly once, on mount.

```typescript
import { useState } from 'react';

interface Settings {
  theme: 'light' | 'dark';
}

export const SettingsPanel: React.FC = () => {
  // GOOD: Lazy initialization. This function runs ONCE on mount.
  const [settings, setSettings] = useState<Settings>(() => {
    const saved = localStorage.getItem('user_settings');
    if (saved) {
      try {
        return JSON.parse(saved) as Settings;
      } catch {
        // Fallback
      }
    }
    return { theme: 'dark' };
  });

  return <div>Current Theme: {settings.theme}</div>;
};
```

## State as a Snapshot

State in React behaves like a snapshot in time. When React renders a component, it calls your function and captures the state variables for *that specific render*. They do not change within that execution frame, even if you call the setter
