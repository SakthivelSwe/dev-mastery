## WHY

If you couldn't group lines of code into named blocks, your program would just be one infinite, unreadable scroll of instructions. Methods (known as functions in other languages) exist to encapsulate logic into reusable, testable, and named units. 

But calling a method isn't magic. It physically alters the memory state of the application. Every time a method is called, the JVM has to pause the current execution, allocate a new block of memory (a Stack Frame) to hold local variables, pass data across the boundary, execute the logic, and then destroy that memory block before returning to the caller. 

Understanding exactly how parameters cross this boundary is the source of endless confusion for junior Java developers. "Is Java pass-by-reference or pass-by-value?" Not knowing the answer leads to bugs where you accidentally mutate an object inside a helper method and corrupt the data across your entire application. Furthermore, understanding how the compiler resolves Method Overloading (methods with the same name but different parameters) is crucial when dealing with complex object hierarchies and autoboxing.

## THEORY

### Stack Frames
The JVM manages execution using the **Thread Stack**. Every thread has its own stack. 
When a method is invoked, the JVM creates a **Stack Frame** and pushes it onto the top of the stack. This frame contains:
1. **Local Variable Array**: Holds all local variables and method parameters. 
2. **Operand Stack**: The actual workspace where bytecode instructions push and pop values to perform calculations.
3. **Reference to Constant Pool**: Allows the method to access class-level constants and static variables.

When the method finishes (hits a `return` statement or the closing brace), its Stack Frame is immediately popped off the stack and destroyed. This is why local variables "disappear" after a method ends. If you call methods recursively too many times without stopping, you fill up the stack memory, causing a fatal `StackOverflowError`.

### Pass-By-Value
Java is strictly, 100% **Pass-By-Value**. It never passes references to variables themselves (like C++ `&` references).
When you pass an argument to a method, the JVM copies the *binary value* of that variable into the new Stack Frame's local variable array.
- **For Primitives** (`int`, `double`): The actual numeric value (e.g., `42`) is copied. Changing the parameter inside the method has zero effect on the caller's variable.
- **For Reference Types** (Objects, Arrays): The value copied is the *memory address pointer* (e.g., `0xAB12`). The new parameter now points to the exact same Object on the Heap. If you mutate the object's fields inside the method, the caller sees the changes because both pointers look at the same Heap location. However, if you reassign the parameter to a `new` object, you only change the local copy of the pointer; the caller's pointer is unaffected.

### Method Overloading and Varargs
**Overloading** is defining multiple methods in the same class with the same name, but different parameter signatures. The compiler decides which method to call *at compile time* (Static Dispatch) using strict resolution rules:
1. Exact match by type.
2. Widening of primitives (e.g., passing an `int` to a `long` parameter).
3. Autoboxing (e.g., passing an `int` to an `Integer` parameter).
4. Varargs.

**Varargs** (`Type... args`) allow a method to accept zero or more arguments of a type. At compile time, the Java compiler literally replaces this syntax with an array creation. `printAll("A", "B")` becomes `printAll(new String[]{"A", "B"})`.

## CODE

### Level 1 — Beginner
```java
public class BasicMethods {
    public static void main(String[] args) {
        // Calling a method creates a new Stack Frame
        int result = add(10, 20);
        System.out.println(result);
    }
    
    // Method signature: name + parameter types
    public static int add(int a, int b) {
        return a + b; // Frame destroyed upon return
    }
}
```

### Level 2 — Building Blocks
```java
public class PassByValueProof {
    public static void main(String[] args) {
        int primitive = 10;
        int[] reference = {10, 20};
        
        attemptToChange(primitive, reference);
        
        // Primitive was COPIED. The original is untouched.
        System.out.println("Primitive: " + primitive); // Prints 10
        
        // Reference (pointer) was COPIED. Modifying the array mutates the Heap.
        System.out.println("Array[0]: " + reference[0]); // Prints 99
    }
    
    public static void attemptToChange(int p, int[] r) {
        p = 99; // Changes local copy only
        r[0] = 99; // Follows pointer to heap and mutates data
        
        // If we reassign 'r', it only changes the local pointer copy
        r = new int[]{100, 200}; // Caller will NOT see this!
    }
}
```

### Level 3 — Intermediate
```java
public class OverloadingResolution {
    // 1. Exact Match
    public void process(int i) { System.out.println("Exact int"); }
    
    // 2. Widening (int fits into long)
    public void process(long l) { System.out.println("Widened to long"); }
    
    // 3. Autoboxing (int boxed to Integer object)
    public void process(Integer i) { System.out.println("Autoboxed"); }
    
    // 4. Varargs (Array)
    public void process(int... i) { System.out.println("Varargs"); }
    
    public static void main(String[] args) {
        OverloadingResolution obj = new OverloadingResolution();
        
        // Compiler resolves this at compile-time by searching in the exact order above
        obj.process(5); 
    }
}
```

### Level 4 — Advanced
```java
public class MethodSignatures {
    // Return types are NOT part of the method signature for overloading!
    // The following two methods would cause a Compiler Error if uncommented:
    
    /*
    public int calculate(String data) { return 1; }
    public String calculate(String data) { return "1"; } // ERROR: Already defined!
    */
    
    // Varargs must ALWAYS be the absolute last parameter
    public void printLogs(String prefix, String... messages) {
        // 'messages' is strictly compiled into a String[] array
        for (String msg : messages) {
            System.out.println(prefix + ": " + msg);
        }
    }
}
```

## REAL_WORLD

A dangerous real-world anti-pattern involves mixing Autoboxing and Method Overloading in heavily utilized APIs, particularly around Collections.

A classic bug happens when working with `List<Integer>`:
```java
List<Integer> ids = new ArrayList<>();
ids.add(10);
ids.add(20);
ids.add(30);

// DEVELOPER INTENT: "I want to remove the value 20 from the list"
ids.remove(20); // BUG! This crashes!
```
Why does this crash with an `IndexOutOfBoundsException`? 
Because the `List` interface has TWO overloaded remove methods:
1. `remove(int index)` — Removes the element at the specified index.
2. `remove(Object o)` — Removes the first occurrence of the specified object.

Because `20` is an `int` primitive, the compiler strictly resolves to the **exact match** `remove(int index)`. It tries to remove the item at index 20, which doesn't exist.
To fix this, you must force the compiler to use the `Object` overload by manually autoboxing:
```java
// Correct way to remove the object value '20'
ids.remove(Integer.valueOf(20));
```

Additionally, **Varargs** are heavily used in modern frameworks like Spring Boot (e.g., `RestTemplate.getForObject(String url, Class<T> responseType, Object... uriVariables)`). While convenient, every time you call a varargs method, the JVM allocates a brand new array on the Heap. If you call a varargs method inside a tight loop executing millions of times, you are creating millions of garbage arrays, which will trigger GC pauses. High-performance libraries will often explicitly overload methods up to 3 or 4 parameters (e.g., `log(String a)`, `log(String a, String b)`) to avoid the varargs array allocation penalty for common use cases.

## INTERVIEW

**Q: Is Java pass-by-reference or pass-by-value?**
A: Java is strictly **pass-by-value**. There is no pass-by-reference in Java. When you pass an Object to a method, you are passing a copy of the *reference pointer* by value. Because the copied pointer points to the same object on the Heap, you can mutate the object's fields. However, if you assign a completely new object to the parameter variable, it only affects the local copy of the pointer, and the original caller's reference remains completely unchanged.

**Q: What causes a `StackOverflowError`?**
A: A `StackOverflowError` occurs when a thread exhausts its allocated stack memory. Every time a method is called, a new Stack Frame is allocated to hold local variables and execution state. This most commonly happens due to an infinite recursive method call with no base case, causing frames to be pushed onto the stack until memory runs out.

**Q: Can you overload a method by only changing its return type?**
A: No. Method overloading is resolved by the compiler using the Method Signature. In Java, a Method Signature consists exclusively of the method's name and its parameter types/order. The return type is explicitly excluded from the signature. Therefore, two methods with the same name and parameters but different return types will cause a compilation error.

**Q: In what order does the Java compiler resolve overloaded methods?**
A: The compiler follows a strict priority chain:
1. Exact type match.
2. Primitive widening (e.g., `int` to `long`).
3. Autoboxing (e.g., `int` to `Integer`).
4. Varargs (`int...`).
If the compiler finds two equally valid matches at the same level (which can happen with complex inheritance hierarchies), it will throw an ambiguous method call compilation error.
