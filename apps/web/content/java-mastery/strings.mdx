## WHY

In any software application—from web servers parsing HTTP headers to databases querying JSON—text is the most ubiquitous form of data. Because text is so prevalent, Strings typically consume 25% to 40% of the total memory in a running Java application.

If Strings in Java were standard, mutable objects that you could change freely, the JVM would have a catastrophic memory problem. If an app receives the HTTP header `"GET"` 10,000 times a second, the JVM would have to create and store 10,000 distinct `"GET"` objects on the heap every single second, completely choking the Garbage Collector.

Furthermore, if Strings were mutable, security would be compromised. Imagine verifying a user's password, and then an attacking thread mutates that exact `String` object in memory right before the JVM grants access. 

Java solved these problems with two radical design choices: **Strings are completely Immutable** (they cannot be changed once created), and **Literal Strings are Pooled** (cached and reused). Understanding how the String Pool works, when to use `.intern()`, and the severe performance penalty of using the `+` operator inside loops is an absolute necessity for any professional Java developer.

## THEORY

### Immutability
A `String` in Java is backed by a final array of characters (in Java 9+, it's a `byte[]` to save space using Compact Strings). Once a String object is created on the heap, its state cannot be modified. 
Why is this critical?
1. **Security**: Strings are used for database URLs, network connections, and file paths. If a String were mutable, a hacker could pass a valid file path to a security check, and then mutate the String to `/etc/shadow` right before the OS opens the file (a Time-of-Check to Time-of-Use vulnerability).
2. **Thread Safety**: Because they cannot change, Strings are inherently thread-safe. Thousands of threads can share the exact same String object without locking.
3. **HashCode Caching**: HashMaps use Strings as keys extensively. Because a String never changes, its HashCode never changes. Java calculates the HashCode once and caches it in a private field, making HashMap lookups lightning fast.

### The String Pool (Metaspace/Heap)
The **String Pool** (or String Intern Pool) is a special caching area. Historically (before Java 7), it was in the PermGen space. Today, it resides directly in the main Java Heap. 
When you declare a string literal like `String s = "devmastery"`, the JVM does a lookup in the String Pool:
- If `"devmastery"` already exists, the JVM simply returns a pointer to the existing object.
- If it does not exist, the JVM creates the object in the pool and returns the pointer.
This means if you declare `"devmastery"` 10,000 times in your code, only ONE object is created in memory.

### String Concatenation and StringBuilder
Because Strings are immutable, modifying a String creates a **brand new String object**. 
```java
String a = "Hello";
a = a + " World"; 
```
Behind the scenes, the JVM abandons the `"Hello"` object and creates a completely new `"Hello World"` object. If you do this in a loop 10,000 times, you generate 10,000 abandoned objects that the Garbage Collector must desperately clean up, grinding your application to a halt.
To solve this, Java provides **`StringBuilder`** (and its older, synchronized brother `StringBuffer`). These act as mutable arrays of characters that you can append to without creating new objects until you finally call `.toString()`.

## CODE

### Level 1 — Beginner
```java
public class StringBasics {
    public static void main(String[] args) {
        // Creates ONE object in the String Pool
        String literal1 = "Java";
        String literal2 = "Java";
        
        // Since they point to the exact same pooled object, == is true
        System.out.println(literal1 == literal2); // Prints true
        
        // Bypasses the pool and forces a BRAND NEW object on the main heap
        String objectStr = new String("Java");
        
        // Different memory addresses!
        System.out.println(literal1 == objectStr); // Prints FALSE
        
        // Always compare String VALUES with .equals()
        System.out.println(literal1.equals(objectStr)); // Prints true
    }
}
```

### Level 2 — Building Blocks
```java
public class ConcatenationTrap {
    public static void main(String[] args) {
        // ANTI-PATTERN: String concatenation in a loop
        // This creates 10,000 garbage objects in memory!
        String slow = "";
        for (int i = 0; i < 10000; i++) {
            slow += i; 
        }
        
        // GOOD PATTERN: Use StringBuilder for loops
        // Only 1 object is manipulated, resizing its internal array when needed.
        StringBuilder fast = new StringBuilder();
        for (int i = 0; i < 10000; i++) {
            fast.append(i);
        }
        String result = fast.toString(); // Finally generates the single String
    }
}
```

### Level 3 — Intermediate
```java
public class StringInterning {
    public static void main(String[] args) {
        // Dynamic string created at runtime (e.g. from a database or file)
        // This lives on the main heap, not the String Pool
        String dbResult = new String("admin");
        
        // .intern() asks the JVM: "Is 'admin' in the String Pool?"
        // If yes, it returns the pooled reference. 
        // If no, it moves this string into the pool and returns it.
        String pooledStr = dbResult.intern();
        
        String literal = "admin";
        
        System.out.println(dbResult == literal);   // false
        System.out.println(pooledStr == literal);  // true
    }
}
```

### Level 4 — Advanced
```java
public class ModernConcatenation {
    public static void main(String[] args) {
        String a = "Hello";
        String b = "World";
        
        // Prior to Java 9, the compiler literally rewrote this as:
        // new StringBuilder().append(a).append(" ").append(b).toString();
        // Since Java 9, it uses invokedynamic to highly optimize this at runtime
        // without instantiating extra StringBuilders.
        String result = a + " " + b; 
        
        // However, String.format is notoriously slow. 
        // It uses Regex parsing internally. Never use it in a hot performance loop.
        String formatted = String.format("%s %s", a, b);
        
        // Java 15+ introduced Text Blocks for readable multi-line strings
        String json = """
            {
                "greeting": "%s",
                "target": "%s"
            }
            """.formatted(a, b);
    }
}
```

## REAL_WORLD

In large-scale production applications, excessive memory consumption by Strings is the #1 cause of OutOfMemory (OOM) crashes. 

A common scenario: A Spring Boot application reads 1,000,000 rows from a CSV file. Each row has a "status" column with the value `"ACTIVE"`. If the code uses `new String("ACTIVE")` or generates the string dynamically via substring parsing, the Heap fills up with 1,000,000 duplicate `"ACTIVE"` objects, instantly exhausting GBs of RAM.

To solve this, high-performance data processing pipelines heavily utilize the **`.intern()`** method. By calling `.intern()` on the incoming status strings, the JVM deduplicates them. The 1,000,000 strings are suddenly reduced to just 1 object in the String Pool, saving massive amounts of memory. (Note: In Java 8+, the String Pool is in the main Heap and is garbage collected, so you don't have to worry about filling up the old PermGen space like in Java 7).

Additionally, always prefer `StringBuilder` over `StringBuffer` in modern codebases. `StringBuffer` was introduced in Java 1.0 and every single method is synchronized (thread-safe). This introduces significant locking overhead. 99% of the time, string building happens entirely within the local scope of a single method (on a single thread). `StringBuilder` (introduced in Java 1.5) drops the synchronization, making it incredibly fast.

## INTERVIEW

**Q: Why is String immutable in Java?**
A: Strings are immutable for three main reasons:
1. **Security**: Strings are heavily used for sensitive parameters (network connections, database URLs, file paths). Immutability prevents a malicious thread from modifying a string after it has been validated but before it is used.
2. **String Pool**: Immutability makes the String Pool possible. Because the string value can never change, the JVM can safely share a single instance among thousands of references to save memory.
3. **HashCode Caching**: HashMaps rely on keys having stable hash codes. Because a String never changes, it caches its `hash` value upon the first calculation, making future HashMap lookups extremely fast.

**Q: What is the difference between `String`, `StringBuilder`, and `StringBuffer`?**
A: `String` is completely immutable. `StringBuilder` is mutable and not thread-safe (fastest, used for single-threaded local string construction). `StringBuffer` is mutable and thread-safe (synchronized, slower, rarely used in modern Java).

**Q: If I do `String a = "Hello" + " World";`, how many objects are created?**
A: Only ONE object. Because `"Hello"` and `"World"` are compile-time constants (literals), the Java compiler performs **Constant Folding**. It resolves the concatenation at compile time and literally embeds the single string `"Hello World"` directly into the bytecode's constant pool.

**Q: When should you use the `+` operator vs `StringBuilder`?**
A: You should use the `+` operator for simple, single-statement concatenations (e.g., `String msg = "Hello " + name;`), because the modern Java compiler optimizes this seamlessly using `invokedynamic`. However, you must **NEVER** use the `+` operator inside a loop (e.g., `for` or `while`). In a loop, the compiler is forced to instantiate a brand new `StringBuilder` and a brand new `String` on every single iteration, flooding the Heap with garbage. Always use a manual `StringBuilder` outside the loop in that scenario.
