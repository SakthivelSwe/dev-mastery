# Dependency Injection — Constructor vs Field vs Setter | Level: 2

---

## 1. Architectural Deep Dive & Internal Mechanics

Dependency Injection (DI) is the concrete implementation of Inversion of Control. Spring supports three primary injection styles: **Constructor Injection**, **Field Injection**, and **Setter Injection**.

```
[ Constructor Injection ]
   │
   ├── Instantiation & Injection occur simultaneously
   └── Guarantees final fields, prevents circular dependencies at startup

[ Field Injection ]
   │
   ├── Class instantiated with empty constructor
   └── ReflectionUtils.setField() modifies private fields (Requires running Context)

[ Setter Injection ]
   │
   ├── Class instantiated with empty constructor
   └── Mutators invoked post-instantiation (Allows optional/mutable dependencies)
```

### Constructor Injection
Constructor injection is the industry standard. The container instantiates the bean by invoking its constructor, passing all dependencies as arguments. This guarantees:
* **Immutability**: Fields can be declared `final`, preventing post-initialization modification.
* **Fail-Fast Startup**: Missing dependencies are detected at compilation or application startup rather than runtime.
* **No Reflection Required**: Dependencies can be injected programmatically in unit tests without starting an application context.

At runtime, Spring uses reflection via `Constructor.newInstance()` to instantiate the bean, passing resolved arguments from the `BeanFactory`.

### Field Injection
Field injection uses the `@Autowired` annotation directly on private fields. Under the hood, Spring uses `ReflectionUtils.makeAccessible(Field field)` to bypass Java access controls and invokes `field.set(bean, dependency)`. 

* **The Problem**: Field injection decouples the class from its dependency declaration. The class cannot be instantiated cleanly in standard unit tests without using mock frameworks or Spring's internal testing utilities (like `ReflectionTestUtils`). It also hides class coupling issues, making it easy to violate the Single Responsibility Principle.

### Setter Injection
Setter injection instantiates the bean using its default constructor, then calls setter methods annotated with `@Autowired`. This style is suitable for optional dependencies that can be reconfigured or changed at runtime. However, it exposes the bean to mutability and can lead to partially initialized objects if not carefully managed.

### Circular Dependencies
A circular dependency occurs when `BeanA` requires `BeanB`, and `BeanB` requires `BeanA`.

* **Constructor Injection**: Spring detects circular dependencies during the instantiation phase. Because both beans require the other to be constructed first, the container throws a `BeanCurrentlyInCreationException` immediately at startup.
* **Field/Setter Injection**: Spring can occasionally resolve circular dependencies by instantiating the beans using empty constructors, registering them in its "singleton objects currently in creation" cache, and injecting the field dependencies later. However, this relies on internal bean lifecycle caching and can fail or lead to null references if the circular path is executed during initialization.
* **The `@Lazy` Escape Hatch**: The `@Lazy` annotation resolves circular dependencies by injecting a dynamic proxy instead of the actual bean. The proxy implements the target interface and defers loading the real bean until a method is first called on the proxy.

## 2. Blueprint & Production-Ready Code

The following code demonstrates constructor injection, setter injection, and how to resolve circular dependencies using `@Lazy`.

```java
package com.devmastery.di;

import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import org.springframework.context.annotation.Lazy;
import org.springframework.stereotype.Component;
import org.springframework.stereotype.Service;

@SpringBootApplication
public class DependencyInjectionApplication {
    public static void main(String[] args) {
        SpringApplication.run(DependencyInjectionApplication.class, args);
    }
}

// ==========================================
// 1. RECOMMENDED: Constructor Injection (Immutable)
// ==========================================
@Service
class OrderService {
    private final PaymentProcessor paymentProcessor;

    // Single constructor: @Autowired is implicit in Spring 4.3+
    public OrderService(PaymentProcessor paymentProcessor) {
        if (paymentProcessor == null) {
            throw new IllegalArgumentException("PaymentProcessor must not be null");
        }
        this.paymentProcessor = paymentProcessor;
    }

    public void processOrder(double amount) {
        paymentProcessor.pay(amount);
    }
}

@Component
class PaymentProcessor {
    private static final Logger log = LoggerFactory.getLogger(PaymentProcessor.class);

    public void pay(double amount) {
        log.info("Processing payment of ${}", amount);
    }
}

// ==========================================
// 2. Setter Injection (Optional/Mutable)
// ==========================================
@Service
class NotificationService {
    private static final Logger log = LoggerFactory.getLogger(NotificationService.class);
    private SmsGateway smsGateway;

    // Setter injection allows this dependency to be optional or swapped
    @org.springframework.beans.factory.annotation.Autowired(required = false)
    public void setSmsGateway(SmsGateway smsGateway) {
        this.smsGateway = smsGateway;
    }

    public void notifyUser(String message) {
        if (smsGateway != null) {
            smsGateway.send(message);
        } else {
            log.warn("SmsGateway not available. Notification logged: {}", message);
        }
    }
}

@Component
class SmsGateway {
    private static final Logger log = LoggerFactory.getLogger(SmsGateway.class);
    public void send(String msg) {
        log.info("SMS Sent: {}", msg);
    }
}

// ==========================================
// 3. Circular Dependency Mitigation with @Lazy
// ==========================================
@Component
class CircularServiceA {
    private static final Logger log = LoggerFactory.getLogger(CircularServiceA.class);
    private final CircularServiceB serviceB;

    // Use @Lazy to inject a proxy of ServiceB, breaking the constructor loop
    public CircularServiceA(@Lazy CircularServiceB serviceB) {
        this.serviceB = serviceB;
    }

    public void doWork() {
        log.info("ServiceA working...");
        serviceB.doSubWork();
    }
}

@Component
class CircularServiceB {
    private static final Logger log = LoggerFactory.getLogger(CircularServiceB.class);
    private final CircularServiceA serviceA;

    public CircularServiceB(CircularServiceA serviceA) {
        this.serviceA = serviceA;
    }

    public void doSubWork() {
        log.info("ServiceB working...");
    }

    public void triggerCascade() {
        serviceA.doWork();
    }
}
```

## 3. Configuration & Properties

```yaml
spring:
  application:
    name: spring-di-deepdive
  main:
    # Set to false to ensure Spring fails fast on circular dependencies
    allow-circular-references: false
```

### Property Explanations
* `spring.main.allow-circular-references`: Setting this to `false` (the default since Spring Boot 2.6) ensures that Spring throws a `BeanCurrentlyInCreationException` during startup if a circular reference is detected, unless it is explicitly mitigated using `@Lazy`.

## 4. Advanced Patterns & Pitfalls (SonarQube Compliant)

### Common Pitfall: Field Injection in Production Code
Using `@Autowired` on private fields makes unit testing difficult and hides dependency bloat.

```java
// INSECURE / UNTESTABLE ANTI-PATTERN
@Service
public class BadService {
    @Autowired
    private DependencyA depA;
    @Autowired
    private DependencyB depB;
    // No constructor or setter. Testing requires ReflectionTestUtils.
}
```

*Correction*: Use constructor injection. It allows you to instantiate the class in a plain JUnit test using `new BadService(mockA, mockB)`.

### SonarQube Compliance: S3749 (Members of Spring components should be injected)
Ensure all fields in a Spring component are either injected via constructor, initialized with default values, or marked as transient if they do not represent state.

### Performance Consideration: Heavy Lazy Initialization Proxying
Using `@Lazy` introduces a small runtime overhead because every method call must pass through a dynamic proxy. Avoid using `@Lazy` as a broad solution for poor design. Instead, resolve circular dependencies by refactoring the shared logic into a third, independent bean.

## 5. Verification & Testing

This test verifies that constructor injection works without starting a Spring Context, and that circular dependency resolution using `@Lazy` functions as expected.

```java
package com.devmastery.di;

import org.junit.jupiter.api.Test;
import org.mockito.Mockito;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.boot.test.context.SpringBootTest;

import static org.junit.jupiter.api.Assertions.assertNotNull;
import static org.mockito.Mockito.verify;

@SpringBootTest(classes = DependencyInjectionApplication.class)
class DependencyInjectionTest {

    @Autowired
    private CircularServiceB circularServiceB;

    @Test
    void testConstructorInjectionWithoutSpring() {
        // Verify we can test OrderService without loading any Spring Context
        PaymentProcessor mockProcessor = Mockito.mock(PaymentProcessor.class);
        OrderService orderService = new OrderService(mockProcessor);

        orderService.processOrder(100.0);

        verify(mockProcessor).pay(100.0);
    }

    @Test
    void testCircularDependencyResolution() {
        // Verify that the context starts successfully and the proxy resolves calls
        assertNotNull(circularServiceB);
        circularServiceB.triggerCascade();
    }
}
```
