## WHY
If you write a method to process payments, you don't want to write three separate methods: `processCreditCard(CreditCard c)`, `processPayPal(PayPal p)`, and `processCrypto(Crypto c)`. Every time a new payment method is added to the company, you would have to modify the core processing engine. 

Polymorphism (meaning "many forms") allows you to write one single method: `processPayment(PaymentMethod p)`. At runtime, you can pass any child object (`CreditCard`, `PayPal`) into that method. The program dynamically figures out which specific implementation to execute. This decouples the core logic from the specific implementations, achieving the Holy Grail of software engineering: the Open-Closed Principle (code should be open for extension, but closed for modification).

## THEORY
### Upcasting and Downcasting
When you do `Animal myDog = new Dog();`, you are **Upcasting**. You create a `Dog` object on the heap, but the reference variable on the stack is of type `Animal`. 
- **Compile-time**: The compiler only checks the reference type (`Animal`). You can only call methods defined in `Animal`. If `Dog` has a `bark()` method, `myDog.bark()` will cause a compilation error.
- **Runtime**: The JVM looks at the actual Heap object (`Dog`). If you call `myDog.eat()`, and `Dog` overrode that method, the JVM executes the `Dog`'s version using Dynamic Dispatch (via the vtable).

To call `bark()`, you must explicitly **Downcast**: `((Dog) myDog).bark()`. If you try to downcast an object to the wrong type, the JVM throws a `ClassCastException` at runtime.

### Static vs Dynamic Binding
- **Static Binding (Compile-time Polymorphism)**: Method Overloading. The compiler knows exactly which method to call based on the parameter types.
- **Dynamic Binding (Runtime Polymorphism)**: Method Overriding. The compiler doesn't know the exact object type, so the JVM resolves the method call at runtime by inspecting the object's vtable.

## CODE

### Level 1 — Beginner
```java
class Payment {
    void pay() { System.out.println("Processing generic payment"); }
}
class PayPal extends Payment {
    @Override
    void pay() { System.out.println("Processing PayPal"); }
}

public class Main {
    public static void main(String[] args) {
        // Upcasting: Parent reference, Child object
        Payment p = new PayPal();
        
        // Dynamic Binding executes the PayPal overridden method
        p.pay(); 
    }
}
```

### Level 2 — Building Blocks
```java
// Polymorphic arrays allow grouping different children
public class Processor {
    public static void main(String[] args) {
        Payment[] payments = { new PayPal(), new CreditCard(), new Payment() };
        
        // Process all payments without knowing their specific types!
        for (Payment p : payments) {
            p.pay(); 
        }
    }
}
```

### Level 3 — Intermediate
```java
public class Casting {
    public static void handle(Payment p) {
        // Safe downcasting using instanceOf to prevent ClassCastException
        if (p instanceof PayPal) {
            PayPal pp = (PayPal) p;
            pp.loginToPayPal(); // Access child-specific method
        }
    }
}
```

### Level 4 — Advanced
```java
public class ModernPatternMatching {
    public static void handle(Payment p) {
        // Java 16+ Pattern Matching for instanceof
        // Automatically casts 'p' to 'PayPal' and assigns to 'pp'
        if (p instanceof PayPal pp) {
            pp.loginToPayPal();
        }
    }
}
```

## REAL_WORLD
Polymorphism is the absolute backbone of Dependency Injection in Spring Boot. 

When you write a Spring Controller, you inject an Interface (e.g., `UserRepository`). 
```java
@RestController
public class UserController {
    @Autowired
    private UserRepository repository; // Interface
}
```
At compile time, the Controller knows nothing about databases. At runtime, Spring's IoC container instantiates a concrete class (like a Hibernate-backed Postgres repository) and injects it. Because of polymorphism, the Controller can call `repository.save()` seamlessly. If the company migrates to MongoDB tomorrow, you just swap the concrete bean. The Controller code never changes.

## INTERVIEW
**Q: What is the difference between Compile-time and Runtime Polymorphism?**
A: Compile-time polymorphism is achieved via Method Overloading. The compiler determines which method to execute based on the method signature. Runtime polymorphism is achieved via Method Overriding and Upcasting. The compiler doesn't know the exact method to call; the JVM uses the object's vtable at runtime to execute the overridden method.

**Q: If a parent and child class both have a `public String name` field, and you do `Parent p = new Child(); System.out.println(p.name);`, which field is accessed?**
A: The Parent's field. **Variables are not polymorphic in Java.** Only methods are dynamically dispatched. Variables are resolved strictly at compile time based on the reference type (Static Binding).
