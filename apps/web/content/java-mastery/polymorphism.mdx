## WHY
If you write a payment processing engine, you don't want to write three separate methods:
```java
void processCreditCard(CreditCard c) { ... }
void processPayPal(PayPal p) { ... }
void processBitcoin(Bitcoin b) { ... }
```
Every time the company adds a new payment method (Apple Pay, Google Pay, BNPL), you have to **modify the core engine**. This violates the Open-Closed Principle: code should be **open for extension, closed for modification**.

Polymorphism (Greek: "many forms") solves this. Write one method:
```java
void processPayment(PaymentMethod p) { p.pay(); }
```
Pass ANY payment type. The JVM dynamically selects the correct `pay()` implementation at runtime. The core engine never changes — you only add new `PaymentMethod` implementations.

This is why polymorphism is called the **backbone of maintainable software design**.

## THEORY

### Two Types of Polymorphism

| Type | Also Called | Mechanism | Resolved At |
|------|-------------|-----------|------------|
| Method Overloading | Compile-time polymorphism | Same name, different parameter types | Compile time |
| Method Overriding | Runtime polymorphism | Child redefines parent's method | Runtime (vtable) |

### Upcasting — The Foundation of Runtime Polymorphism
When you assign a child object to a parent-type reference, you **upcast**:
```java
Animal a = new Dog(); // Dog is upcast to Animal
```
- **Compile-time**: Compiler only knows the reference type (`Animal`) — you can only call `Animal`'s methods.
- **Runtime**: JVM looks at the actual Heap object (`Dog`) — the vtable dispatches to `Dog`'s overridden methods.

```
Stack                Heap
------          ----------------
|  a  | -----> | Object Header |  ← actual type: Dog
|     |        | name: "Rex"   |
------          | [vtable ptr]--> Dog's eat() method
```

### Dynamic Dispatch (vtable Lookup)
When you call `a.eat()`:
1. JVM follows the reference to the Heap object
2. Reads the Klass Pointer from the Object Header
3. Jumps to Dog's vtable in Metaspace
4. Dog's vtable has a pointer to Dog's `eat()` bytecode
5. Dog's `eat()` executes

The `Animal` reference variable is irrelevant — it's the **actual Heap object type** that determines behavior.

### Downcasting and `ClassCastException`
To access Dog-specific methods (`bark()`), you must **downcast**:
```java
Animal a = new Dog("Rex");
Dog d = (Dog) a;         // Safe — 'a' actually IS a Dog
d.bark();

Cat c = (Cat) a;         // RUNTIME ClassCastException! 'a' is a Dog, not a Cat
```

Always use `instanceof` before downcasting:
```java
if (a instanceof Dog d) {  // Java 16+ pattern matching
    d.bark();
}
```

### Method Overloading (Compile-time Polymorphism)
```java
class Printer {
    void print(int n)    { System.out.println("int: " + n); }
    void print(double d) { System.out.println("double: " + d); }
    void print(String s) { System.out.println("String: " + s); }
}
// Compiler selects which print() to call based on argument type at compile time
```

### Variables are NOT Polymorphic
Only **methods** are dynamically dispatched. Fields (variables) are resolved at compile time based on the reference type:
```java
class Parent { String name = "Parent"; }
class Child extends Parent { String name = "Child"; }

Parent p = new Child();
System.out.println(p.name); // "Parent" — field access is static!
// To get "Child", you need an actual Child reference
```

## CODE

### Level 1 — Beginner: Runtime Polymorphism in Action
```java
// Parent class
class Shape {
    String color;

    Shape(String color) { this.color = color; }

    // This method will be overridden
    public double area() {
        return 0.0;
    }

    public String describe() {
        return color + " " + getClass().getSimpleName() + " with area=" + String.format("%.2f", area());
    }
}

class Circle extends Shape {
    double radius;

    Circle(String color, double radius) {
        super(color);
        this.radius = radius;
    }

    @Override
    public double area() {
        return Math.PI * radius * radius; // Overrides parent
    }
}

class Rectangle extends Shape {
    double width, height;

    Rectangle(String color, double width, double height) {
        super(color);
        this.width = width;
        this.height = height;
    }

    @Override
    public double area() {
        return width * height; // Different implementation, same method name
    }
}

public class Main {
    public static void main(String[] args) {
        // Polymorphic array — parent reference, child objects
        Shape[] shapes = {
            new Circle("Red", 5.0),
            new Rectangle("Blue", 4.0, 6.0),
            new Circle("Green", 3.0)
        };

        // One loop processes ALL shape types without if/else branching
        double totalArea = 0;
        for (Shape s : shapes) {
            System.out.println(s.describe());  // Dynamic dispatch!
            totalArea += s.area();             // Calls correct override
        }
        System.out.printf("Total area: %.2f%n", totalArea);
    }
}
```

### Level 2 — Building Blocks: Polymorphism with Interfaces
```java
interface Notification {
    void send(String recipient, String message);

    default String formatMessage(String message) {
        return "[DevMastery] " + message;
    }
}

class EmailNotification implements Notification {
    @Override
    public void send(String recipient, String message) {
        System.out.println("EMAIL to " + recipient + ": " + formatMessage(message));
    }
}

class SMSNotification implements Notification {
    @Override
    public void send(String recipient, String message) {
        // SMS has character limit
        String truncated = message.length() > 160 ? message.substring(0, 157) + "..." : message;
        System.out.println("SMS to " + recipient + ": " + truncated);
    }
}

class PushNotification implements Notification {
    @Override
    public void send(String recipient, String message) {
        System.out.println("PUSH to device[" + recipient + "]: " + message);
    }
}

public class NotificationService {
    // Accepts ANY Notification implementation — pure polymorphism
    public void notifyUser(String user, String msg, Notification... channels) {
        for (Notification channel : channels) {
            channel.send(user, msg);  // Each channel handles it differently
        }
    }

    public static void main(String[] args) {
        NotificationService service = new NotificationService();

        // Mix and match notification channels
        service.notifyUser("alice@example.com",
                           "Your order has shipped!",
                           new EmailNotification(),
                           new SMSNotification(),
                           new PushNotification());
    }
}
```

### Level 3 — Intermediate: Upcasting, Downcasting, and Pattern Matching
```java
abstract class Animal {
    String name;
    Animal(String name) { this.name = name; }
    abstract void makeSound();
}

class Dog extends Animal {
    Dog(String name) { super(name); }
    @Override public void makeSound() { System.out.println(name + ": Woof!"); }
    public void fetch() { System.out.println(name + " fetches the ball!"); }
}

class Cat extends Animal {
    Cat(String name) { super(name); }
    @Override public void makeSound() { System.out.println(name + ": Meow!"); }
    public void purr() { System.out.println(name + " purrs..."); }
}

class Parrot extends Animal {
    String phrase;
    Parrot(String name, String phrase) { super(name); this.phrase = phrase; }
    @Override public void makeSound() { System.out.println(name + ": " + phrase); }
}

public class CastingDemo {

    // Old-style downcasting with instanceof
    static void handleOldStyle(Animal animal) {
        animal.makeSound();  // Always works — polymorphic

        if (animal instanceof Dog) {
            Dog dog = (Dog) animal;  // Manual cast
            dog.fetch();
        } else if (animal instanceof Cat) {
            Cat cat = (Cat) animal;
            cat.purr();
        }
    }

    // Modern Java 16+ Pattern Matching — cleaner and safer
    static void handleModern(Animal animal) {
        animal.makeSound();

        // Pattern matching: checks type AND casts in one step
        if (animal instanceof Dog d) {
            d.fetch();                 // 'd' is already a Dog here
        } else if (animal instanceof Cat c) {
            c.purr();
        } else if (animal instanceof Parrot p && p.phrase.contains("hello")) {
            System.out.println(p.name + " is friendly!");
        }
    }

    public static void main(String[] args) {
        Animal[] zoo = {
            new Dog("Rex"),
            new Cat("Whiskers"),
            new Parrot("Polly", "hello there")
        };

        for (Animal a : zoo) {
            handleModern(a);
        }
    }
}
```

### Level 4 — Advanced: Method Overloading Resolution and Ambiguity
```java
public class OverloadingRules {
    // Method overloading resolution: compiler picks the most specific type
    void process(Object obj)  { System.out.println("Object"); }
    void process(String str)  { System.out.println("String"); }
    void process(Number num)  { System.out.println("Number"); }
    void process(Integer i)   { System.out.println("Integer"); }

    public static void main(String[] args) {
        OverloadingRules o = new OverloadingRules();

        o.process("hello");     // "String" — most specific match
        o.process(42);          // "Integer" — 42 is auto-boxed to Integer
        o.process(3.14);        // "Number" — Double extends Number, no Double overload
        o.process(new Object());// "Object" — most general match

        // Tricky: null can match multiple overloads
        // o.process(null);     // Ambiguity! Compiler error if String and Number both match
        o.process((String) null); // Explicit cast to resolve ambiguity

        // Widening vs Boxing — widening takes priority
        byte b = 10;
        o.process(b);  // "Number" (widened to int, then Number) NOT "Integer"
        // Widening: byte→short→int→long→float→double
        // Boxing: byte→Byte (only if no widening match exists)
    }
}
```

### Level 5 — Expert: Polymorphism via Strategy Pattern in Production
```java
import java.math.BigDecimal;
import java.util.Map;

// Strategy Pattern — pure polymorphism for interchangeable algorithms
public interface PricingStrategy {
    BigDecimal calculatePrice(BigDecimal basePrice, int quantity);
    String getName();
}

class RegularPricing implements PricingStrategy {
    @Override
    public BigDecimal calculatePrice(BigDecimal basePrice, int quantity) {
        return basePrice.multiply(BigDecimal.valueOf(quantity));
    }
    @Override public String getName() { return "REGULAR"; }
}

class BulkDiscountPricing implements PricingStrategy {
    private final int threshold;
    private final double discountRate;

    BulkDiscountPricing(int threshold, double discountRate) {
        this.threshold = threshold;
        this.discountRate = discountRate;
    }

    @Override
    public BigDecimal calculatePrice(BigDecimal basePrice, int quantity) {
        BigDecimal total = basePrice.multiply(BigDecimal.valueOf(quantity));
        if (quantity >= threshold) {
            BigDecimal discount = total.multiply(BigDecimal.valueOf(discountRate));
            return total.subtract(discount);
        }
        return total;
    }
    @Override public String getName() { return "BULK_DISCOUNT"; }
}

class FlashSalePricing implements PricingStrategy {
    private final double flashRate;

    FlashSalePricing(double flashRate) { this.flashRate = flashRate; }

    @Override
    public BigDecimal calculatePrice(BigDecimal basePrice, int quantity) {
        return basePrice.multiply(BigDecimal.valueOf(flashRate))
                        .multiply(BigDecimal.valueOf(quantity));
    }
    @Override public String getName() { return "FLASH_SALE"; }
}

// The engine NEVER changes — only strategies change
class OrderCalculator {
    private final Map<String, PricingStrategy> strategies = Map.of(
        "REGULAR",    new RegularPricing(),
        "BULK",       new BulkDiscountPricing(10, 0.15),
        "FLASH_SALE", new FlashSalePricing(0.50)
    );

    public BigDecimal calculate(String strategyKey, BigDecimal price, int qty) {
        PricingStrategy strategy = strategies.getOrDefault(strategyKey, strategies.get("REGULAR"));
        BigDecimal result = strategy.calculatePrice(price, qty);
        System.out.printf("[%s] %d x $%.2f = $%.2f%n",
                          strategy.getName(), qty, price, result);
        return result;
    }

    public static void main(String[] args) {
        OrderCalculator calc = new OrderCalculator();
        BigDecimal price = new BigDecimal("10.00");

        calc.calculate("REGULAR",    price, 5);   // $50.00
        calc.calculate("BULK",       price, 15);  // $127.50 (15% off)
        calc.calculate("FLASH_SALE", price, 5);   // $25.00 (50% off)
    }
}
```

## INTERVIEW

**Q: What is the difference between Compile-time and Runtime Polymorphism?**
A: Compile-time polymorphism (overloading) — the compiler selects which method to call based on the method signature at compile time. Runtime polymorphism (overriding + upcasting) — the compiler cannot determine which method to call; the JVM uses the actual Heap object's vtable at runtime (dynamic dispatch) to execute the overridden method.

**Q: If `Parent p = new Child(); System.out.println(p.name);`, which `name` prints?**
A: The Parent's `name`. **Fields are not polymorphic** in Java. Field access is resolved at compile time based on the reference type (static binding), not the actual object type. Only methods use dynamic dispatch.

**Q: What is upcasting and why is it safe?**
A: Upcasting is assigning a child object to a parent-type reference (`Animal a = new Dog()`). It's always safe because a `Dog` IS-AN `Animal` — it has all of Animal's fields and methods. No casting operator needed; the compiler allows this implicitly.

**Q: What is downcasting and when can it fail?**
A: Downcasting is casting a parent reference to a child type (`Dog d = (Dog) a`). It can fail at runtime with `ClassCastException` if the actual Heap object is NOT an instance of the target type. Always use `instanceof` (or Java 16+ pattern matching) before downcasting.

**Q: Explain the Strategy Design Pattern and how it uses polymorphism.**
A: Define a family of algorithms behind a common interface. Each algorithm is a concrete class implementing that interface. The context class stores a reference to the interface (not a specific implementation), so algorithms can be swapped at runtime without changing the context. This is pure polymorphism applied to algorithm selection.

**Q: Why can't `static` methods be overridden?**
A: Static methods belong to the class, not to any object instance. They don't participate in the vtable. When you "override" a static method in a child class, you're actually **hiding** it — the JVM still determines which version to call based on the compile-time reference type, not the runtime object type. This makes static method calls non-polymorphic.

## FEYNMAN CHECK

Explain to a 10-year-old:
> Imagine a TV remote has a VOLUME UP button. Every TV brand (Samsung, LG, Sony) has its own way of increasing volume internally. But every remote works the same way — you press VOLUME UP. The remote doesn't care what brand the TV is; it just sends the signal, and each TV handles it in its own way. That's polymorphism — one message, many responses.
