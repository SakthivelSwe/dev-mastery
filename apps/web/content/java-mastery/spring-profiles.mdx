# Spring Profiles | Level: 2

---

## 1. Architectural Deep Dive & Internal Mechanics

Spring Profiles provide a way to segregate parts of your application configuration and make them available only in specific environments.

```
[ Application Bootstrap ]
       │
       ▼
[ Environment ] (Loads properties from OS Env, System Props, application.yml)
       │
       ├── Reads active profiles (spring.profiles.active)
       │
       ▼
[ ConfigurationCondition Evaluation ]
       │
       ├── Evaluates @Profile annotations
       └── Skips bean registration if condition is not met
```

### The Environment Abstraction
At the core of Spring's profile management is the `Environment` interface, which extends `PropertyResolver`. The `Environment` represents the profile configuration of the running application. It maintains two main sets of profiles:
1. **Active Profiles**: The profiles currently enabled for the application run.
2. **Default Profiles**: The profiles enabled by default if no active profiles are specified (typically `default`).

### Profile Evaluation Mechanics
When Spring processes configurations, any class or `@Bean` method annotated with `@Profile` is evaluated by a `ProfileCondition` class, which implements `ConfigurationCondition`. 

During the `ConfigurationClassPostProcessor` execution phase, the container checks if the profiles specified in `@Profile` match the active profiles in the `Environment`. If the condition fails, the `BeanDefinition` is discarded, and the bean is never instantiated.

### Profile Activation Priority
Profiles can be activated using multiple strategies, evaluated in the following order of precedence (highest to lowest):
1. Programmatic activation via `SpringApplication.setAdditionalProfiles(...)` or `WebApplicationInitializer`.
2. JVM System Properties: `-Dspring.profiles.active=prod`.
3. OS Environment Variables: `SPRING_PROFILES_ACTIVE=prod`.
4. Properties file configuration: `spring.profiles.active` defined inside `application.yml`.

### Profile Groups (Spring Boot 2.4+)
Profile groups allow you to group multiple profiles under a single logical name. For example, activating a `production` profile group can automatically activate underlying profiles like `prod-database`, `prod-security`, and `prod-metrics`.

## 2. Blueprint & Production-Ready Code

The following example demonstrates profile-specific bean creation, custom profile conditions, and profile groups.

```java
package com.devmastery.profiles;

import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Condition;
import org.springframework.context.annotation.ConditionContext;
import org.springframework.context.annotation.Conditional;
import org.springframework.context.annotation.Configuration;
import org.springframework.context.annotation.Profile;
import org.springframework.core.type.AnnotatedTypeMetadata;
import org.springframework.stereotype.Service;

@SpringBootApplication
public class ProfilesApplication {
    public static void main(String[] args) {
        SpringApplication.run(ProfilesApplication.class, args);
    }
}

// ==========================================
// 1. Define Service Interface
// ==========================================
interface DatasourceConnector {
    String connect();
}

// ==========================================
// 2. Profile-Specific Implementations
// ==========================================
@Service
@Profile("dev")
class DevDatasourceConnector implements DatasourceConnector {
    @Override
    public String connect() {
        return "Connected to Local H2 Database (DEV Environment)";
    }
}

@Service
@Profile("prod")
class ProdDatasourceConnector implements DatasourceConnector {
    @Override
    public String connect() {
        return "Connected to Production PostgreSQL Cluster (PROD Environment)";
    }
}

// ==========================================
// 3. Custom Profile Condition (Fallback)
// ==========================================
class OnLocalCondition implements Condition {
    @Override
    public boolean matches(ConditionContext context, AnnotatedTypeMetadata metadata) {
        String[] activeProfiles = context.getEnvironment().getActiveProfiles();
        return activeProfiles.length == 0 || "local".equalsIgnoreCase(activeProfiles[0]);
    }
}

@Configuration
class LocalConfiguration {

    @Bean
    @Conditional(OnLocalCondition.class)
    public DatasourceConnector localConnector() {
        return () -> "Connected to Local Dockerized Database (LOCAL fallback)";
    }
}
```

## 3. Configuration & Properties

Here is the `application.yml` file demonstrating profile groups and profile fallback structures.

```yaml
spring:
  application:
    name: spring-profiles-deepdive
  profiles:
    active: dev
    # Define Profile Groups (Spring Boot 2.4+)
    group:
      development:
        - dev
        - debug-logging
        - mock-email
      production:
        - prod
        - secure-ssl
        - live-email

---
# Development Profile Configuration
spring:
  config:
    activate:
      on-profile: dev
logging:
  level:
    root: DEBUG
    org.springframework: INFO

---
# Production Profile Configuration
spring:
  config:
    activate:
      on-profile: prod
logging:
  level:
    root: ERROR
    org.springframework: WARN
```
