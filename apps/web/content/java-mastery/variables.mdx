# var, let, const — Scope, Hoisting, TDZ | Level: 1

Variable declaration in JavaScript has undergone a significant evolution. Understanding the exact mechanical differences between `var`, `let`, and `const` requires analyzing how the JavaScript engine structures memory and scopes during parsing and execution.

---

## 1. Scope: Function Scope vs. Block Scope

Scope determines where variables are accessible within your code.

### Function Scope (`var`)
Variables declared with `var` are scoped to the nearest containing **function**. They completely ignore block boundaries such as `if` statements, `switch` blocks, and `for` loops.

```javascript
function scopeDemo() {
  if (true) {
    var functionScoped = "I am accessible anywhere in this function";
  }
  console.log(functionScoped); // Prints: "I am accessible anywhere in this function"
}
scopeDemo();
```

### Block Scope (`let` and `const`)
Variables declared with `let` and `const` are scoped to the nearest containing **block** (defined by curly braces `{}`).

```javascript
function blockScopeDemo() {
  if (true) {
    let blockScoped = "I am trapped in this block";
    const alsoBlockScoped = "Me too";
  }
  // ReferenceError: blockScoped is not defined
  console.log(blockScoped); 
}
```

## 2. Hoisting and the Temporal Dead Zone (TDZ)

To understand hoisting, we must look at how the JavaScript engine processes code. The engine runs code in two distinct phases: **Parsing (Creation Phase)** and **Execution Phase**.

### The `var` Hoisting Mechanism
During the parsing phase, when the engine encounters a `var` declaration, it allocates memory for that variable in the enclosing function scope (or global scope) and immediately initializes its value to `undefined`.

```javascript
console.log(hoistedVar); // Prints: undefined
var hoistedVar = "value";
```

Under the hood, the engine interprets the above code as:
```javascript
var hoistedVar; // Declared and initialized to undefined
console.log(hoistedVar);
hoistedVar = "value"; // Assignment happens in place
```

### The `let` and `const` Hoisting Mechanism
Variables declared with `let` and `const` are also hoisted. During the parsing phase, the engine registers their identifiers in memory. However, unlike `var`, **they are not initialized**. They remain uninitialized.

```javascript
// Temporal Dead Zone starts here for 'hoistedLet'
console.log(hoistedLet); // ReferenceError: Cannot access 'hoistedLet' before initialization
let hoistedLet = "value"; // TDZ ends here
```

### The Temporal Dead Zone (TDZ)
The TDZ is the physical block of code from the start of the block scope until the line where the variable is officially declared and initialized. Accessing a variable within its TDZ triggers a `ReferenceError`.

```javascript
{ // Block scope starts
  // TDZ for x is active
  const func = () => console.log(x); // This is fine to define, but not to execute yet
  
  // x is in TDZ here
  
  let x = 10; // TDZ for x ends
  func(); // Safe to execute now: prints 10
}
```

## 3. The Immutability of `const`

A common misconception is that `const` creates a completely immutable value. In reality, `const` creates an **immutable binding**.

```javascript
const immutableBinding = 42;
immutableBinding = 100; // TypeError: Assignment to constant variable.
```

When you assign an object or an array to a `const` variable, the variable holds a **reference** (a memory pointer) to that object on the heap. While you cannot change the pointer itself, you can freely mutate the properties of the object it points to.

```javascript
const user = { name: "Alice" };

// This is perfectly valid (mutating the heap value):
user.name = "Bob"; 
user.age = 30;

// This is invalid (attempting to rebind the pointer):
user = { name: "Charlie" }; // TypeError: Assignment to constant variable.
```

If you require true immutability of an object's values, you must use `Object.freeze()`:

```javascript
const frozenUser = Object.freeze({ name: "Alice" });
frozenUser.name = "Bob"; // Silently fails in sloppy mode, throws TypeError in strict mode
console.log(frozenUser.name); // "Alice"
```

## 4. Why `var` Should Be Avoided in Modern JavaScript

### 1. Block Leakage and Loop Variables
When using `var` in a loop, the variable leaks out into the surrounding scope. This causes notorious asynchronous behavior bugs:

```javascript
// Buggy pattern using var
for (var i = 0; i < 3; i++) {
  setTimeout(() => {
    console.log(i); // Prints 3, 3, 3 (because 'i' is shared and mutated)
  }, 100);
}

// Correct pattern using let
for (let j = 0; j < 3; j++) {
  setTimeout(() => {
    console.log(j); // Prints 0, 1, 2 (because 'j' is scoped to each loop iteration block)
  }, 100);
}
```

### 2. Accidental Global Pollution
If you assign a value to a variable without declaring it (only possible with `var`/sloppy mode), it becomes a global property:

```javascript
function leak() {
  accidentalGlobal = "Oops"; // Becomes window.accidentalGlobal / global.accidentalGlobal
}
leak();
```

## 5. Memory Model: V8 Internal representation

In modern JavaScript engines like V8, variables are tracked using **Environment Records**.

*   **VariableEnvironment:** Holds bindings created by `var` declarations. These are allocated on the function's activation record (stack frame) or context object.
*   **LexicalEnvironment:** Holds bindings created by `let` and `const` declarations.

When a block `{}` is entered, a new block-scoped LexicalEnvironment is created. If a loop runs, a new LexicalEnvironment is generated for *each iteration*, capturing the current value of loop-declared `let` variables. This is why the `setTimeout` example works correctly with `let` but fails with `var`.

```
Execution Context (Function)
 ├── VariableEnvironment: { i: 3 }  <-- Shared var
 └── LexicalEnvironment (Block Iteration 1): { j: 0 }
 └── LexicalEnvironment (Block Iteration 2): { j: 1 }
 └── LexicalEnvironment (Block Iteration 3): { j: 2 }
