## WHY
Before Java 5, if you wanted to configure a web framework (like Struts or early Spring) or a database mapping framework (like Hibernate), you had to write massive, hundreds-of-lines-long XML files. The XML files were completely disconnected from the Java code. If you renamed a Java class, the XML file broke, but the compiler couldn't catch the error, leading to catastrophic runtime crashes.

Java 5 solved this by introducing **Annotations**. Annotations are metadata that you attach directly to your classes, methods, or fields. They allow you to embed configuration logic right next to the code it configures. Instead of writing `<bean id="userController" class="com.app.UserController">` in an XML file, you simply write `@RestController` at the top of your class, and the framework automatically reads the metadata at runtime to configure the application.

## THEORY
### What is an Annotation?
An annotation (e.g., `@Override`, `@Deprecated`) is not executable code. By itself, it does absolutely nothing. It is purely an informational label. 
Annotations only possess power when a **Processor** (a piece of code using the Reflection API) scans the classes, finds the labels, and executes logic based on their presence.

### The Meta-Annotations
When you create a custom annotation, you must define its properties using "Meta-Annotations" (annotations applied to other annotations!):
1. **`@Retention`**: When does the annotation exist?
   - `SOURCE`: Thrown away by the compiler (e.g., `@Override`).
   - `CLASS`: Saved in the `.class` file, but ignored by the JVM at runtime (default).
   - `RUNTIME`: Loaded into JVM memory, allowing frameworks to read it via Reflection.
2. **`@Target`**: Where can it be applied? (Methods, Fields, Classes, Parameters).

## CODE

### Level 1 — Beginner (Built-in Annotations)
```java
class Parent {
    public void executeTask() {}
}

class Child extends Parent {
    // 1. @Override tells the compiler to double-check our spelling.
    // If we misspell it as 'executeTsk', the compiler throws an error!
    @Override 
    public void executeTask() {}
    
    // 2. @SuppressWarnings tells the compiler to ignore specific warnings.
    @SuppressWarnings("unused")
    private int deadVariable = 5;
}
```

### Level 2 — Building Blocks (Creating an Annotation)
```java
import java.lang.annotation.Retention;
import java.lang.annotation.RetentionPolicy;
import java.lang.annotation.Target;
import java.lang.annotation.ElementType;

// 1. We must retain it at RUNTIME so our reflection code can see it!
@Retention(RetentionPolicy.RUNTIME)
// 2. We only want this applied to Methods
@Target(ElementType.METHOD)
public @interface Retry {
    // Annotation fields act like methods. They can have default values.
    int maxAttempts() default 3;
    long delayMs() default 1000;
}
```

### Level 3 — Intermediate (Applying the Annotation)
```java
public class NetworkClient {
    
    // Applying our custom annotation and overriding the defaults!
    @Retry(maxAttempts = 5, delayMs = 2000)
    public void connectToServer() {
        System.out.println("Attempting connection...");
        throw new RuntimeException("Connection Timeout!");
    }
}
```

### Level 4 — Advanced (Processing the Annotation)
```java
import java.lang.reflect.Method;

public class AnnotationProcessor {
    public static void main(String[] args) throws Exception {
        NetworkClient client = new NetworkClient();
        
        // 1. Use Reflection to get the method
        Method method = NetworkClient.class.getMethod("connectToServer");
        
        // 2. Check if our @Retry annotation is present
        if (method.isAnnotationPresent(Retry.class)) {
            // 3. Extract the metadata configuration
            Retry retryConfig = method.getAnnotation(Retry.class);
            int attempts = retryConfig.maxAttempts();
            
            // 4. Execute the framework logic!
            for (int i = 0; i < attempts; i++) {
                try {
                    method.invoke(client);
                    break; // Success!
                } catch (Exception e) {
                    System.out.println("Failed. Retries left: " + (attempts - i - 1));
                }
            }
        }
    }
}
```

## REAL_WORLD
Modern Spring Boot is almost 100% annotation-driven. 
If you define an entity with `@Entity` and `@Table(name = "users")`, Spring Data JPA's internal annotation processors read that metadata at runtime and automatically generate the SQL `CREATE TABLE` and `INSERT` statements to match your Java class structure.

However, an even more powerful use case is **Compile-Time Annotation Processing**. 
Project **Lombok** uses annotations like `@Getter` and `@Setter`. But Lombok does NOT use Reflection at runtime! Instead, it hooks directly into the Java Compiler (`javac`). During compilation, when it sees `@Getter`, it intercepts the AST (Abstract Syntax Tree) and injects the actual `public String getName()` bytecode directly into the `.class` file. This is why Lombok has zero runtime performance penalty—it generates the code at compile time!

## INTERVIEW
**Q: Can an annotation contain business logic?**
A: No. An annotation (`@interface`) is purely a metadata container. It can hold values (Strings, ints, arrays), but it cannot contain any executable method bodies. The actual business logic is always executed by a separate component (a Processor) that uses Reflection to find the annotation and act upon its values.

**Q: What happens if you forget to add `@Retention(RetentionPolicy.RUNTIME)` to your custom annotation?**
A: By default, annotations have a retention policy of `CLASS`. This means the annotation is compiled into the `.class` file, but the JVM discards it when loading the class into memory. If you try to use Reflection (`method.isAnnotationPresent(...)`) at runtime, it will return `false`, and your framework logic will silently fail.

**Q: How do you pass an array of values into an annotation?**
A: If the annotation defines a field like `String[] roles();`, you can pass multiple values using curly braces: `@Secured(roles = {"ADMIN", "USER"})`. If there is only one value, the compiler allows you to omit the braces: `@Secured(roles = "ADMIN")`.
