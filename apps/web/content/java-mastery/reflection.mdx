## WHY
In Java, if a class field is marked as `private`, it is strictly impossible for an external class to read or modify it. That is the absolute bedrock of Encapsulation.
But wait. How does a library like Google GSON or Jackson convert a JSON string into a Java Object if your fields are `private` and you didn't write any setter methods? How does Spring Boot scan your entire project, find methods marked with `@GetMapping`, and execute them dynamically when an HTTP request arrives, without you ever explicitly calling the method?

They use the **Reflection API**. Reflection is a magical, dangerous "backdoor" that allows Java code to inspect itself, read private fields, execute private methods, and dynamically instantiate classes at Runtime without knowing anything about them at Compile Time. It is the engine that powers every major Java framework in existence.

## THEORY
### The `Class` Object
In Java, everything is an object, including classes themselves! 
When the JVM loads your `User` class, it creates a special metaphysical object of type `java.lang.Class<User>`. This object contains the complete blueprint of your class: a list of every constructor, every field, every method, and every annotation. By grabbing this `Class` object, you can programmatically analyze and manipulate the blueprint.

### Performance and Security Costs
Reflection is incredibly powerful, but it comes with severe penalties:
1. **Performance**: Calling a method via Reflection is significantly slower than a direct method call because the JVM cannot perform JIT compiler optimizations (like method inlining) on reflective calls.
2. **Security**: Reflection completely bypasses the compiler's safety checks. You can accidentally crash the JVM or violate critical security sandbox rules if you randomly execute unknown methods.

## CODE

### Level 1 — Beginner (Inspecting a Class)
```java
import java.lang.reflect.Method;
import java.lang.reflect.Field;

class SecretAgent {
    private String codeName = "007";
    public void executeMission() { System.out.println("Mission Go!"); }
}

public class BasicReflection {
    public static void main(String[] args) throws Exception {
        // 1. Get the Blueprint (Class object)
        Class<?> clazz = SecretAgent.class;
        
        System.out.println("Class Name: " + clazz.getName()); // SecretAgent
        
        // 2. Inspect its Methods dynamically
        for (Method m : clazz.getDeclaredMethods()) {
            System.out.println("Found method: " + m.getName());
        }
    }
}
```

### Level 2 — Building Blocks (Breaking Encapsulation)
```java
import java.lang.reflect.Field;

public class BreakEncapsulation {
    public static void main(String[] args) throws Exception {
        SecretAgent agent = new SecretAgent();
        Class<?> clazz = agent.getClass();
        
        // 1. Find the exact private field named "codeName"
        Field secretField = clazz.getDeclaredField("codeName");
        
        // 2. OVERRIDE ENCAPSULATION! (The magic backdoor)
        // This disables Java's access checks, allowing us to read/write private data!
        secretField.setAccessible(true);
        
        // 3. Read the private data
        String stolenName = (String) secretField.get(agent);
        System.out.println("Stolen: " + stolenName); // 007
        
        // 4. Modify the private data!
        secretField.set(agent, "006"); 
    }
}
```

### Level 3 — Intermediate (Dynamic Execution)
```java
import java.lang.reflect.Method;

public class DynamicExecution {
    public static void main(String[] args) throws Exception {
        // Imagine this string came from a database or a config file
        String classNameToLoad = "SecretAgent";
        
        // 1. Dynamically load the class at runtime
        Class<?> clazz = Class.forName(classNameToLoad);
        
        // 2. Instantiate an object of this class dynamically!
        Object instance = clazz.getDeclaredConstructor().newInstance();
        
        // 3. Find the method and execute it dynamically!
        Method missionMethod = clazz.getDeclaredMethod("executeMission");
        
        // Invoke requires the instance to run on, and any parameters
        missionMethod.invoke(instance); 
    }
}
```

### Level 4 — Advanced (Dynamic Proxies)
```java
import java.lang.reflect.Proxy;
import java.lang.reflect.InvocationHandler;
import java.lang.reflect.Method;

// Advanced Architecture: Intercepting method calls!
public class ProxyDemo {
    interface Database { void save(); }
    
    public static void main(String[] args) {
        InvocationHandler handler = new InvocationHandler() {
            @Override
            public Object invoke(Object proxy, Method method, Object[] args) {
                System.out.println("TRANSACTION STARTED...");
                System.out.println("Executing: " + method.getName());
                System.out.println("TRANSACTION COMMITTED!");
                return null;
            }
        };
        
        // The JVM dynamically generates a FAKE class that implements Database at runtime!
        Database db = (Database) Proxy.newProxyInstance(
            Database.class.getClassLoader(),
            new Class<?>[]{ Database.class },
            handler
        );
        
        // When we call save(), it is intercepted by our InvocationHandler!
        db.save();
    }
}
```

## REAL_WORLD
Reflection and Dynamic Proxies are the beating heart of the **Spring Framework**.

When you write a Spring `@Transactional` or `@Cacheable` annotation on a method, how does Spring actually wrap your code in a database transaction? 
Spring uses `java.lang.reflect.Proxy` (or CGLIB) to generate a fake subclass of your Service at runtime. When the REST Controller calls your Service, it is actually calling the Dynamic Proxy. The Proxy starts the database transaction, uses Reflection (`method.invoke()`) to call your real business logic, and then commits the transaction when your method finishes.

Similarly, when you declare `@Autowired private UserService userService;`, Spring Boot uses Reflection to find the private field, calls `setAccessible(true)`, and forcibly injects the dependency into the field without needing a setter method. Without the Reflection API, Dependency Injection and modern Java frameworks simply would not exist.

## INTERVIEW
**Q: What does `setAccessible(true)` do?**
A: It overrides the Java compiler's access control checks (public/private/protected) for that specific `Field`, `Method`, or `Constructor` object during runtime. It allows external classes to read and write to private fields, or execute private methods, completely bypassing encapsulation.

**Q: Can you prevent someone from using Reflection to read your private fields?**
A: Historically, you could use the Java SecurityManager to block reflection, but this was deprecated. In modern Java (Java 9+), the Module System (JPMS) fundamentally locks down reflection. If you put your code in a strict Module and do not explicitly `opens` your package to other modules, `setAccessible(true)` will throw an `InaccessibleObjectException`, completely blocking frameworks like Spring or Hibernate from hacking into your private data.

**Q: Why is Reflection considered slow?**
A: When you write `obj.method()`, the JVM's JIT compiler optimizes it, calculates the exact memory offset, and can often inline the method entirely. When you use `method.invoke()`, the JVM has to perform security access checks, unbox/box arguments, and dynamically resolve the method pointer at runtime, completely defeating the JIT optimizer.
