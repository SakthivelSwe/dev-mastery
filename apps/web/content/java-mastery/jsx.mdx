## JSX is Not HTML: Strict Typings and Key Differences

While JSX resembles HTML syntax, it evaluates to JavaScript objects and maps directly to the DOM properties of browser elements rather than HTML attributes. This requires adhering to specific syntactic and structural rules.

### 1. CamelCase Attribute Naming
Because JSX compiles to JavaScript objects, property names must be valid camelCase identifiers.
* `class` becomes `className`
* `for` becomes `htmlFor`
* `tabindex` becomes `tabIndex`

### 2. Inline Styles as Objects
The `style` attribute in JSX accepts a strongly typed JavaScript object, not a string. CSS property names must be camelCased (e.g., `backgroundColor` instead of `background-color`).

```typescript
import React from 'react';

const StyleExample: React.FC = () => {
  const containerStyle: React.CSSProperties = {
    backgroundColor: '#1a1a1a',
    padding: '24px',
    borderRadius: '8px',
    border: '1px solid #333'
  };

  return <div style={containerStyle}>Styled Container</div>;
};
```

## JSX Expressions: Embedding Dynamic Logic

Any valid JavaScript expression can be embedded in JSX by wrapping it in curly braces `{}`. An expression must evaluate to a value that React can render.

### Renderable vs. Non-Renderable Values
* **Renderable**: Strings, Numbers, React Elements, Arrays of these types.
* **Ignored (Not Rendered)**: Booleans (`true`/`false`), `null`, `undefined`.
* **Invalid (Throws Error)**: Plain, non-element JavaScript objects.

```typescript
const ExpressionDemo: React.FC = () => {
  const user = { name: 'Sarah', age: 32 }; // Plain object
  const items = ['React', 'TypeScript', 'Vite'];

  return (
    <div>
      {/* Strings and numbers render directly */}
      <p>User Name: {user.name}</p>
      
      {/* Arrays are flattened and rendered */}
      <ul>{items.map(item => <li key={item}>{item}</li>)}</ul>

      {/* CRITICAL: Plain objects cannot be rendered directly */}
      {/* <div>{user}</div> -> This will throw a runtime error! */}
    </div>
  );
};
```

## Conditional Rendering: Avoid the "Zero" Trap

Conditional rendering in React is typically handled using ternary operators or logical AND (`&&`) expressions. However, the logical `&&` operator has a dangerous edge case when dealing with numbers.

### The `&&` Operator Mechanics
In JavaScript, `expression1 && expression2` returns `expression1` if it is falsy. If `expression1` is the number `0`, React will evaluate the expression to `0` and render it to the DOM, as `0` is a renderable number.

```typescript
interface DashboardProps {
  notifications: string[];
}

export const Dashboard: React.FC<DashboardProps> = ({ notifications }) => {
  return (
    <div className="dashboard">
      {/* PITFALL: If notifications is empty ([]), notifications.length is 0. */}
      {/* This renders "0" to the UI instead of rendering nothing! */}
      {notifications.length && <p>You have notifications!</p>}

      {/* SAFE APPROACH 1: Explicit boolean cast */}
      {!!notifications.length && <p>You have notifications!</p>}

      {/* SAFE APPROACH 2: Comparison check */}
      {notifications.length > 0 && <p>You have notifications!</p>}

      {/* SAFE APPROACH 3: Ternary operator */}
      {notifications.length > 0 ? (
        <p>You have notifications!</p>
      ) : (
        <p>No notifications.</p>
      )}
    </div>
  );
};
```

## Rendering Lists and Key Stability

When mapping arrays to JSX, every sibling element must have a unique, stable, and predictable `key` prop.

### Why Array Indexes are Dangerous Keys
If a list can be re-sorted, filtered, or have items inserted/deleted at arbitrary indices, using the array index as a key will cause React to map state to the wrong DOM nodes.

```typescript
interface Task {
  id: string; // Stable UUID from database
  text: string;
}

export const TaskList: React.FC<{ tasks: Task[] }> = ({ tasks }) => {
  return (
    <ul>
      {tasks.map((task, index) => (
        // BAD: key={index}
        // If tasks are reordered, inputs inside this component will retain
        // state based on the index, causing UI-state mismatch.
        
        // GOOD: Stable, unique ID
        <li key={task.id}>
          <input type="text" defaultValue={task.text} />
        </li>
      ))}
    </ul>
  );
};
```

## React Fragments: Grouping Elements Without DOM Pollution

A React component must return a single root element. If you want to return multiple elements, wrapping them in a `<div>` adds unnecessary depth to the browser's DOM tree, which can break CSS Flexbox/Grid layouts.

React provides **Fragments** to group children without adding extra nodes to the DOM.

### Short Syntax vs. Explicit Fragment Syntax
The short syntax `<></>` cannot accept attributes or keys. If you need to pass a `key` to a fragment (e.g., when rendering a list of description terms and details), you must use the explicit `<React.Fragment>` syntax.

```typescript
import React from 'react';

interface Term {
  id: string;
  word: string;
  definition: string;
}

export const Glossary: React.FC<{ terms: Term[] }> = ({ terms }) => {
  return (
    <dl>
      {terms.map((term) => (
        // When mapping to fragments, you must use explicit import
        // and provide a unique key. Short syntax <></> will compile error here.
        <React.Fragment key={term.id}>
          <dt>{term.word}</dt>
          <dd>{term.definition}</dd>
        </React.Fragment>
      ))}
    </dl>
  );
};
```

## The React 17+ JSX Transform

In older React codebases, you had to import React at the top of every file containing JSX:

```typescript
import React from 'react'; // Required before React 17
export const App = () => <h1>Hello</h1>;
```

With the modern compiler-integrated JSX transform, the compiler automatically injects the necessary runtime functions, freeing developers from adding boilerplate imports and reducing final bundle sizes.

```typescript
// Modern React 17+ / React 18 / React 19: No React import needed!
export const App = () => <h1>Hello</h1>;
```
