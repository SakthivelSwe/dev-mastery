# TypeScript Introduction — Why Types, tsc, tsconfig | Level: 2

TypeScript is a strongly typed, static superset of JavaScript that compiles down to plain JavaScript. At its core, TypeScript adds a **static type system** to JavaScript's dynamic runtime environment. 

To master TypeScript, you must understand its foundational philosophy: **structural subtyping**, **type erasure**, and how the compiler (`tsc`) operates separately from execution.

---

### Structural vs. Nominal Typing

Unlike languages like Java, C++, or C# which use **nominal typing** (where compatibility is determined by explicit class declarations and inheritance hierarchies), TypeScript uses **structural typing** (often called structural subtyping or "duck typing"). 

In a structural type system, type compatibility is determined solely by the *shape* and *structure* of the members (properties and methods) of a type. If two objects have the same shape, they are considered to be of the same type, regardless of their nominal declarations.

```typescript
// TypeScript
class User {
  id!: string;
  username!: string;
}

class Customer {
  id!: string;
  username!: string;
}

// Structurally identical; allowed in TypeScript!
const user: User = new Customer(); 
```

---

### Type Erasure: Compile-Time vs. Runtime

TypeScript types exist **only** during compilation. The compiler performs static analysis to ensure your program is type-safe, and then it strips away all type annotations, interfaces, type aliases, and generic parameters. This process is called **type erasure**. 

At runtime, only standard JavaScript is executed. This means there is **zero runtime overhead** introduced by TypeScript's type system.

#### Side-by-Side: TypeScript vs. Compiled JavaScript

Here is an example of type erasure in action. Notice how the type annotations and interface definitions completely disappear in the compiled output.

```typescript
// TypeScript (input.ts)
interface Config {
  port: number;
  host: string;
}

function startServer(config: Config): string {
  return `Server running at http://${config.host}:${config.port}`;
}

const myConfig: Config = { port: 8080, host: "localhost" };
startServer(myConfig);
```

```javascript
// Compiled JavaScript (output.js - Target: ES2022)
function startServer(config) {
    return `Server running at http://${config.host}:${config.port}`;
}
const myConfig = { port: 8080, host: "localhost" };
startServer(myConfig);
```

### The `tsc` Compiler: Validation vs. Compilation

The TypeScript compiler (`tsc`) has two primary, independent responsibilities:
1. **Type Checking**: Analyzing your code for type safety and emitting errors if rules are broken.
2. **Transpilation**: Converting modern TypeScript/JavaScript syntax into a target version of JavaScript (e.g., ES5, ES6, ES2022).

By default, `tsc` will still emit compiled `.js` files even if there are type errors. This is because your code might still be valid JavaScript at runtime. You can override this behavior in your build pipeline using the `--noEmitOnError` compiler flag.

---

### Configuring `tsconfig.json`

The `tsconfig.json` file specifies the root files and compiler options required to compile your project. Below is a production-grade, strict configuration:

```json
{
  "compilerOptions": {
    "target": "ES2022",                          /* Specify ECMAScript target version */
    "module": "NodeNext",                        /* Specify module code generation */
    "moduleResolution": "NodeNext",              /* Specify how modules get resolved */
    "strict": true,                              /* Enable all strict type-checking options */
    "noEmitOnError": true,                       /* Do not emit outputs if any errors were reported */
    "esModuleInterop": true,                     /* Emit additional JavaScript to ease support for importing CommonJS modules */
    "forceConsistentCasingInFileNames": true,    /* Ensure that casing is correct in imports */
    "skipLibCheck": true                         /* Skip type checking of declaration files (.d.ts) */
  }
}
```

#### What `strict: true` Enables Under the Hood
Enabling `"strict": true` is non-negotiable for modern TypeScript applications. It acts as an umbrella flag that auto-enables several critical strictness options:

*   `noImplicitAny`: Raises an error on expressions and declarations with an implied `any` type.
*   `strictNullChecks`: When enabled, `null` and `undefined` have their own distinct types, preventing "billion-dollar mistake" runtime crashes.
*   `strictFunctionTypes`: Ensures function parameters are checked contravariantly.
*   `strictBindCallApply`: Ensures the arguments to `bind`, `call`, and `apply` match the original function signature.
*   `strictPropertyInitialization`: Ensures class properties declared but not initialized are set in the constructor.
*   `useUnknownInCatchVariables`: Changes the type of the catch clause variable from `any` to `unknown`.
*   `alwaysStrict`: Parses your files in JavaScript's strict mode and emits `"use strict";` at the top of every output file.

### TypeScript Errors vs. Runtime Errors

TypeScript catches errors at **compile time** (static analysis) that would otherwise manifest as crashes or logical bugs at **runtime**.

```typescript
// TypeScript
const user = {
  name: "Alice",
  getProfile() {
    return this.name;
  }
};

// Compile-time error! TS knows 'age' does not exist on this object.
console.log(user.age); 
```

If compiled with `tsc`, TypeScript halts the build and outputs:
```text
error TS2339: Property 'age' does not exist on type '{ name: string; getProfile(): string; }'.
```

In plain JavaScript, this code would run without errors and print `undefined`, which could lead to silent bugs down the line (e.g., trying to read properties of `undefined`).

---

### When TypeScript Helps Most

1.  **Refactoring**: Safely changing property names or function signatures across thousands of files. The compiler instantly flags every broken reference.
2.  **API Contracts**: Establishing rigorous contracts between frontend and backend systems, ensuring data payloads match expected types.
3.  **Team Codebases**: Serving as living, self-updating documentation. Developers can understand an API simply by hovering over its types in their IDE.
