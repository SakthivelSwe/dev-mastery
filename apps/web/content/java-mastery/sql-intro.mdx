# SQL and Relational Model — Tables, Keys, ACID

### 1. Architectural & Theoretical Foundations

#### The Relational Model & Codd's Rules
Edgar F. Codd’s relational model (1970) decoupled physical data storage from logical representation. The core premise is that all data is represented as mathematical **relations** (tables) containing **tuples** (rows) and **attributes** (columns). 

Codd defined 12 rules (actually 13, numbered 0 to 12) to govern relational database management systems (RDBMS). Key rules include:
*   **Rule 1: The Information Rule:** All information is represented explicitly at the logical level as values in tables.
*   **Rule 2: Guaranteed Access Rule:** Every datum (atomic value) is logically addressable by combining the table name, primary key value, and column name.
*   **Rule 4: Dynamic On-Line Catalog:** The system catalog (database metadata) must be stored in relational tables, queryable via the same language used for user data.
*   **Rule 9: Logical Data Independence:** Application programs are insulated from changes in the logical schema (e.g., splitting or combining tables) without altering queries.

#### Key Mechanics: Primary and Foreign Keys
*   **Primary Key (PK):** A minimal set of attributes that uniquely identifies a tuple within a relation. Physically, RDBMSs enforce uniqueness by automatically backing primary keys with a unique index (typically a B-Tree). Primary keys must be immutable and enforce a non-null constraint (`NOT NULL`).
*   **Foreign Key (FK):** A referential constraint between two tables. It identifies a column or set of columns in one (referencing) table that refers to a column or set of columns in another (referenced) table. 
*   **Referential Integrity:** Enforced via `ON DELETE` and `ON UPDATE` actions:
    *   `CASCADE`: Automatically delete or update matching rows in the child table.
    *   `RESTRICT` / `NO ACTION`: Prevent deletion or update of parent rows if child references exist. (In PostgreSQL, `NO ACTION` allows deferring the check to the end of the transaction, whereas `RESTRICT` checks immediately).
    *   `SET NULL` / `SET DEFAULT`: Set child columns to `NULL` or default values.

#### ACID Properties: Deep Dive
*   **Atomicity:** The "all or nothing" guarantee. If any part of a transaction fails, the entire transaction is aborted and rolled back. Enforced via the **Write-Ahead Log (WAL)** in PostgreSQL or the **Undo/Redo Logs** (InnoDB) in MySQL.
*   **Consistency:** Transactions transition the database from one valid state to another, maintaining all schema constraints (PK, FK, CHECK, UNIQUE). If a constraint is violated, the transaction is aborted.
*   **Isolation:** Controls how concurrent transactions see changes made by others. Enforced using lock managers and **Multi-Version Concurrency Control (MVCC)**.
*   **Durability:** Once committed, transaction effects survive system crashes. Enforced by writing transaction logs (WAL) to non-volatile storage (via `fsync()`) before confirming the commit to the client.

#### SQL vs. NoSQL Decision Matrix

| Dimension | Relational (SQL) | NoSQL (Document/Key-Value/Graph) |
| :--- | :--- | :--- |
| **Data Schema** | Strict, static, normalized schema | Dynamic, schema-less, denormalized |
| **Consistency** | Strong Consistency (ACID) | Eventual Consistency (BASE: Basically Available, Soft state, Eventual consistency) |
| **Scaling** | Vertical (scale up); Horizontal scaling requires sharding/clustering overhead | Horizontal (scale out) natively via partitioning |
| **Joins** | Highly optimized relational joins in memory/disk | Application-level joins or denormalization |
| **Transactions** | Multi-row ACID transactions | Single-document/row ACID; multi-document transactions are limited or costly |

#### Engine-Level Comparisons

*   **PostgreSQL 16:**
    *   *Storage Engine:* Append-only storage model with MVCC. Updates create a new version of the row (tuple), marking the old one as dead. Requires background **VACUUM** processes to reclaim space.
    *   *Concurrency:* Highly advanced lock manager. Readers do not block writers, and writers do not block readers.
    *   *Extensibility:* Strong support for custom data types (JSONB, GIS, Arrays).
*   **MySQL 8.0 (InnoDB):**
    *   *Storage Engine:* Index-Organized Tables (clustered index on PK). Updates occur in-place; old versions are written to an **Undo Log** space.
    *   *Concurrency:* Row-level locking with next-key locking to prevent phantom reads at the Repeatable Read isolation level.
*   **SQLite 3:**
    *   *Storage Engine:* Single-file database.
    *   *Concurrency:* Database-level locking (one writer at a time). Lacks native MVCC (though WAL mode provides some reader/writer concurrency). Ideal for embedded, serverless, or local storage.

### 2. DDL & DML Code Implementation

```sql
-- Setup Schema representing a standard parent-child relationship
CREATE TABLE customers (
    customer_id BIGINT GENERATED ALWAYS AS IDENTITY,
    email VARCHAR(255) NOT NULL,
    company_name VARCHAR(150),
    created_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT pk_customers PRIMARY KEY (customer_id),
    CONSTRAINT uq_customer_email UNIQUE (email)
);

CREATE TABLE orders (
    order_id BIGINT GENERATED ALWAYS AS IDENTITY,
    customer_id BIGINT NOT NULL,
    order_date TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
    total_amount NUMERIC(12, 2) NOT NULL,
    status VARCHAR(50) DEFAULT 'PENDING',
    CONSTRAINT pk_orders PRIMARY KEY (order_id),
    CONSTRAINT fk_orders_customer FOREIGN KEY (customer_id) 
        REFERENCES customers (customer_id) 
        ON DELETE RESTRICT 
        ON UPDATE CASCADE,
    CONSTRAINT chk_positive_amount CHECK (total_amount >= 0)
);
```

### 3. Real-World Object-Relational Mapping (ORM)

```java
package com.devmastery.sql.domain;

import jakarta.persistence.*;
import lombok.Getter;
import lombok.Setter;
import java.math.BigDecimal;
import java.time.OffsetDateTime;
import java.util.ArrayList;
import java.util.List;

@Entity
@Table(name = "customers", uniqueConstraints = {
    @UniqueConstraint(name = "uq_customer_email", columnNames = "email")
})
@Getter
@Setter
public class Customer {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    @Column(name = "customer_id")
    private Long id;

    @Column(name = "email", nullable = false, length = 255)
    private String email;

    @Column(name = "company_name", length = 150)
    private String companyName;

    @Column(name = "created_at", nullable = false, updatable = false)
    private OffsetDateTime createdAt = OffsetDateTime.now();

    @OneToMany(mappedBy = "customer", cascade = CascadeType.PERSIST, orphanRemoval = false)
    private List<Order> orders = new ArrayList<>();
}
```

```java
package com.devmastery.sql.domain;

import jakarta.persistence.*;
import lombok.Getter;
import lombok.Setter;
import java.math.BigDecimal;
import java.time.OffsetDateTime;

@Entity
@Table(name = "orders")
@Getter
@Setter
public class Order {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    @Column(name = "order_id")
    private Long id;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(
        name = "customer_id", 
        nullable = false, 
        foreignKey = @ForeignKey(name = "fk_orders_customer")
    )
    private Customer customer;

    @Column(name = "order_date", nullable = false)
    private OffsetDateTime orderDate = OffsetDateTime.now();

    @Column(name = "total_amount", nullable = false, precision = 12, scale = 2)
    private BigDecimal totalAmount;

    @Column(name = "status", nullable = false, length = 50)
    private String status = "PENDING";
}
```
