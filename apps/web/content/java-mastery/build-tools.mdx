## WHY
If you only have two Java files, compiling them via the command line (`javac Main.java`) is easy. 
But enterprise applications have thousands of files, require dozens of third-party libraries (like Spring Boot, Jackson, and PostgreSQL drivers), and need to be compiled, tested, packaged into a `.jar`, and deployed automatically. Doing this manually is impossible.

**Build Tools** (like Maven and Gradle) automate the entire lifecycle of a Java project. They connect to global repositories (like Maven Central) to automatically download dependencies, compile your code, run your unit tests, and package the final executable application.

## THEORY
### Dependency Management
Before Build Tools, if you wanted to use a JSON parsing library, you had to manually download the `.jar` file from a website, place it in a `lib/` folder, and configure your IDE classpath. If that library depended on 5 *other* libraries, you had to manually find and download those too!

Build Tools introduced **Dependency Resolution**. You write one line of configuration ("I need Jackson v2.15"), and the tool automatically connects to the internet, downloads Jackson, and recursively downloads every single library that Jackson relies on (Transitive Dependencies).

## CODE

### Level 1 — Beginner (Maven Basics)
Maven is the oldest and most widely used build tool in Java. It relies on a strict, declarative XML file called `pom.xml` (Project Object Model).

```xml
<!-- pom.xml -->
<project xmlns="http://maven.apache.org/POM/4.0.0">
    <modelVersion>4.0.0</modelVersion>

    <!-- The identity of YOUR project -->
    <groupId>com.mycompany</groupId>
    <artifactId>my-app</artifactId>
    <version>1.0.0</version>

    <!-- The dependencies your project needs! -->
    <dependencies>
        <!-- I want to use Spring Boot Web -->
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-web</artifactId>
            <version>3.1.0</version>
        </dependency>
        
        <!-- I want to use JUnit, but ONLY during the test phase! -->
        <dependency>
            <groupId>org.junit.jupiter</groupId>
            <artifactId>junit-jupiter</artifactId>
            <version>5.9.2</version>
            <scope>test</scope> <!-- Excludes this from the final production .jar! -->
        </dependency>
    </dependencies>
</project>
```
*To build the project, you run:* `mvn clean install`

### Level 2 — Building Blocks (Maven Lifecycles)
Maven enforces a strict, rigid lifecycle. When you run `mvn install`, Maven executes a predefined sequence of phases:
1. **validate:** Checks if the project is correct.
2. **compile:** Compiles the source code.
3. **test:** Runs the unit tests using a framework like JUnit. *(If tests fail, the build instantly stops!)*
4. **package:** Takes the compiled code and packages it in its distributable format (a `.jar` or `.war` file).
5. **verify:** Runs integration tests.
6. **install:** Installs the package into your local repository, making it available as a dependency for other local projects.

### Level 3 — Intermediate (Gradle Basics)
Maven's XML is verbose and rigid. **Gradle** is the modern alternative (and the official build tool for Android). Instead of XML, it uses a programmatic DSL (Domain Specific Language) based on Groovy or Kotlin.

```kotlin
// build.gradle.kts (Kotlin DSL)

plugins {
    java // Tell Gradle this is a Java project
}

group = "com.mycompany"
version = "1.0.0"

// Where should Gradle download the libraries from?
repositories {
    mavenCentral() 
}

// The dependencies! Much cleaner than XML.
dependencies {
    // Standard library for production code
    implementation("org.springframework.boot:spring-boot-starter-web:3.1.0")
    
    // Test-only library
    testImplementation("org.junit.jupiter:junit-jupiter:5.9.2")
}

// Configure the test task to use JUnit Platform
tasks.test {
    useJUnitPlatform()
}
```
*To build the project, you run:* `./gradlew build`

### Level 4 — Advanced (Gradle Custom Tasks)
Because Gradle uses a real programming language (Kotlin/Groovy), you aren't locked into a rigid lifecycle like Maven. You can write custom scripts and logic directly inside your build file!

```kotlin
// build.gradle.kts

// Register a brand new custom task
tasks.register("generateVersionFile") {
    // Describe what this task does
    description = "Generates a txt file containing the current build version"
    
    // The action to perform when the task runs
    doLast {
        val versionFile = file("${buildDir}/version.txt")
        versionFile.writeText("App Version: $version\nBuild Time: ${java.time.Instant.now()}")
        println("Successfully generated version file at ${versionFile.path}")
    }
}

// Force the standard 'build' task to DEPEND ON our custom task!
// Our custom file will be generated automatically every time we build the app!
tasks.named("build") {
    dependsOn("generateVersionFile")
}
```

## REAL_WORLD
**Continuous Integration (CI/CD)**
When you push code to GitHub, how does the company guarantee it works before deploying it to AWS? 
They use CI/CD platforms (like GitHub Actions or Jenkins). These platforms spin up an empty Linux server in the cloud, clone your repository, and execute a single command: `./gradlew build` (or `mvn clean install`).
If a developer accidentally broke a feature, the `test` phase of the build tool will fail. The build tool returns an exit code of `1` (Error). The CI server sees the error, marks the Pull Request with a massive red X, and completely blocks the deployment to production, saving the company from an outage.

## INTERVIEW
**Q: I tried to run `gradle build` on a new server, but it threw an error saying "gradle command not found". Do I have to manually install Gradle on every server before I can build the project?**
A: **No, you should use the Wrapper!**
A core principle of modern build tools is deterministic builds. If Developer A has Gradle 7 installed, and Developer B has Gradle 8, the build might behave differently. 
To fix this, Gradle provides the **Gradle Wrapper** (`gradlew` for Linux/Mac, `gradlew.bat` for Windows). This is a tiny script checked into your Git repository. When you run `./gradlew build`, the script automatically checks if the exact correct version of Gradle is installed. If not, it downloads it instantly, and *then* runs the build. You never need to pre-install Gradle on a machine; the wrapper guarantees the exact right version is used automatically.

**Q: Maven vs Gradle — when do you pick which, and why?**
A: **Maven** wins when the team values *predictability and convention* — a rigid lifecycle, declarative XML, and a massive plugin ecosystem make it almost impossible to write a "weird" build. **Gradle** wins when the project is large or non-standard — incremental compilation, build caching, parallel task execution, and a real Kotlin/Groovy DSL give it 2–10× faster builds on monorepos, plus the freedom to script custom tasks. Android, Spring's own multi-module repo, and most modern microservice stacks ship Gradle. Legacy enterprises and most public OSS libraries still ship Maven.

**Q: What is a transitive dependency, and what is the "dependency hell" problem?**
A: A transitive dependency is a library pulled in *indirectly* — e.g., your project depends on Spring Boot, which depends on Jackson, which depends on `jackson-core`. Dependency hell occurs when two of your direct dependencies pull in *different versions* of the same transitive library. Maven uses "nearest wins" (closest in the tree), Gradle uses "highest wins" by default. Both let you force a specific version via `<dependencyManagement>` or `dependencyResolution.strategy { force(...) }`.

**Q: What is a SNAPSHOT vs a RELEASE version?**
A: A `SNAPSHOT` (e.g., `1.0.0-SNAPSHOT`) is a mutable, in-development version. Maven/Gradle re-check the remote repository for a fresher copy on every build. A release version (`1.0.0`) is immutable — once published it must never change. Using `SNAPSHOT` in production is an antipattern because the same version coordinate can resolve to two different artifacts at different times, destroying build reproducibility.

**Q: How does the Maven build lifecycle differ from a Gradle task graph?**
A: Maven has a fixed, linear sequence of *phases* (validate → compile → test → package → verify → install → deploy). You bind plugin *goals* to phases. Gradle has no fixed lifecycle; it builds a *directed acyclic graph (DAG) of tasks* on the fly based on `dependsOn` declarations and input/output annotations. Gradle then walks the DAG, runs independent tasks in parallel, and skips tasks whose inputs are unchanged ("up-to-date checks"). This is why Gradle can be dramatically faster than Maven on incremental builds.

## FEYNMAN CHECK

Explain a build tool to a non-programmer in 60 seconds:

> Imagine you are baking a five-tier wedding cake. The recipe has 200 steps and uses 40 ingredients sourced from 12 different shops, and some ingredients require *other* ingredients to be made first. Doing this by memory every time would be insane — you'd forget steps, buy the wrong flour, burn the icing. A **build tool** is the master chef's clipboard: it knows the exact recipe, the exact suppliers, the exact order of steps, and exactly which steps can be done in parallel. You say "bake the cake" and it handles everything. If a single ingredient supplier is missing, it stops with a clear error before you waste hours on the wrong cake.

## BUILD

**Mini-project: a multi-module Gradle build with custom tasks and a CI pipeline.**

Goal: build a small "URL Shortener" project as a **Gradle multi-module monorepo** with `:core`, `:web`, and `:cli` submodules, and produce a GitHub Actions workflow that builds, tests, and publishes Docker images.

### Requirements
1. Create a root `settings.gradle.kts` declaring three sub-modules.
2. Place common configuration (`java`, `repositories`, JUnit 5) inside `buildSrc/src/main/kotlin/build-conventions.gradle.kts` as a convention plugin. Each subproject applies the plugin with `plugins { id("build-conventions") }`.
3. `:core` exposes a `ShortenerService` — pure Java, no Spring. `:web` depends on `:core` and adds `spring-boot-starter-web`. `:cli` depends on `:core` and adds `picocli`.
4. Write a custom Gradle task `generateBuildInfo` that writes a JSON file `build/buildInfo.json` containing the git SHA, the build timestamp, and the Gradle version. Wire it as a `dependsOn` of `processResources` so it is automatically included in the final jar.
5. Configure `application` plugin on `:web` and `:cli` so `./gradlew :web:bootRun` and `./gradlew :cli:run -- --url=https://example.com` both work.
6. Add the Spring Boot dependency-management plugin and pull a `org.springframework.boot:spring-boot-dependencies` BOM. Confirm with `./gradlew :web:dependencyInsight --dependency jackson-core` that the BOM-managed version wins.
7. Write a `.github/workflows/ci.yml` that:
   * Caches `~/.gradle/caches` keyed by hash of `*.gradle.kts`.
   * Runs `./gradlew build --scan` on every push.
   * Publishes a build scan link as a comment on the PR.
   * On `main` only, builds and pushes Docker images via the `bootBuildImage` task.

### Stretch goals
* Add a third-party plugin: `com.diffplug.spotless` for code formatting.
* Wire `jacoco` coverage and fail the build below 80 % line coverage in `:core`.
* Use Gradle composite builds to include a separate `:test-utils` repo without publishing it to Maven Local.

This exercise covers every senior-level build concept: convention plugins, custom tasks, BOMs, dependency insight, multi-module DAGs, build caching, build scans, and CI integration.
