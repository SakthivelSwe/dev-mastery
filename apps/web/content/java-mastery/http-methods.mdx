# HTTP Methods — Idempotency, Safe Methods

### 1. Architectural Deep Dive & Core Mechanics

HTTP methods (verbs) are defined by RFC 9110 to establish semantic properties for resource interactions. Understanding these properties is critical for designing predictable distributed systems.

#### Safe Methods
A method is **safe** if its semantics are defined as entirely read-only. Safe methods must not cause any observable side effects on the server (e.g., database writes, state changes).
*   **Safe Methods**: `GET`, `HEAD`, `OPTIONS`, `TRACE`.
*   *System Implications*: Safe methods can be pre-fetched, aggressively cached, and safely retried by network intermediaries (proxies, CDNs) without consulting the origin server.

#### Idempotent Methods
A method is **idempotent** if the side-effects of making multiple identical requests are the same as making a single request.
*   **Idempotent Methods**: `GET`, `HEAD`, `OPTIONS`, `TRACE`, `PUT`, `DELETE`.
*   *Non-Idempotent Methods**: `POST`, `PATCH`.
*   *System Implications*: If a network timeout or connection reset occurs during an idempotent request, the client (or a proxy) can safely retry the request automatically.
