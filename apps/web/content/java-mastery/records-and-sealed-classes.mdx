## WHY
For decades, the biggest criticism of Java was its extreme verbosity. If you wanted to create a simple Data Transfer Object (DTO) to hold an X and Y coordinate, you had to write a `class`, define two private variables, write a constructor, write two getters, write an `equals()` method, a `hashCode()` method, and a `toString()` method. That's 50 lines of boilerplate for 2 pieces of data! Developers relied heavily on the third-party **Lombok** library (`@Data`) just to maintain sanity.

Java 14 and Java 15 solved this at the language level by introducing **Records** (immutable data carriers) and **Sealed Classes** (strict inheritance hierarchies). Together, they form the foundation of "Data-Oriented Programming" in modern Java, eliminating boilerplate and making switch-statement pattern matching mathematically exhaustive.

## THEORY
### Records (Data Carriers)
A Record is a special type of class designed exclusively to act as a transparent carrier for immutable data. 
When you declare a Record, the Java Compiler automatically generates the constructor, the getter methods (named exactly after the fields, e.g., `x()` instead of `getX()`), `equals()`, `hashCode()`, and `toString()`. 
Records are strictly immutable. Their fields are `private final`. You cannot extend a Record, and a Record cannot extend another class (though it can implement interfaces).

### Sealed Classes (Controlled Inheritance)
Historically, if you created an abstract class or interface, any developer anywhere could extend it. This made it impossible to know all possible subclasses. 
A Sealed Class uses the `sealed` and `permits` keywords to strictly define exactly which classes are allowed to extend it. If a class is not in the `permits` list, the compiler violently rejects the inheritance. This allows the compiler to know the exact mathematical bounds of the hierarchy.

## CODE

### Level 1 — Beginner (Records)
```java
// 1 Line of Code replaces 50 lines of boilerplate!
// The compiler auto-generates the constructor, getters, equals, hashCode, and toString.
public record Point(int x, int y) {}

public class RecordDemo {
    public static void main(String[] args) {
        Point p = new Point(10, 20);
        
        // Getters do NOT have the "get" prefix!
        System.out.println("X: " + p.x()); 
        
        // toString() is perfectly formatted automatically
        System.out.println(p); // Prints: Point[x=10, y=20]
        
        // equals() correctly compares the internal data, not the memory address
        Point p2 = new Point(10, 20);
        System.out.println(p.equals(p2)); // true
    }
}
```

### Level 2 — Building Blocks (Record Constructors)
```java
public record UserDTO(String username, int age) {
    
    // Compact Constructor!
    // We don't need to write 'this.age = age'. The compiler handles the assignment.
    // We just use this space to add validation logic before the assignment happens!
    public UserDTO {
        if (age < 0) {
            throw new IllegalArgumentException("Age cannot be negative");
        }
        
        // We can reassign the parameters before they are saved to the fields
        username = username.trim().toLowerCase();
    }
}
```

### Level 3 — Intermediate (Sealed Classes)
```java
// We strictly define that ONLY Circle, Square, and Rectangle can implement Shape.
public sealed interface Shape permits Circle, Square, Rectangle {}

// Subclasses of a sealed class MUST declare exactly one of three modifiers:
// 1. final: Cannot be extended further.
public final class Circle implements Shape {}

// 2. non-sealed: Re-opens the hierarchy, allowing anyone to extend it.
public non-sealed class Square implements Shape {}

// 3. sealed: Continues the strict hierarchy.
public sealed class Rectangle implements Shape permits RoundedRectangle {}
public final class RoundedRectangle extends Rectangle {}
```

### Level 4 — Advanced (Pattern Matching with Switch)
```java
// The true power of Sealed Classes and Records combined!
public sealed interface Response permits Success, Failure {}
public record Success(String jsonPayload) implements Response {}
public record Failure(int httpCode, String errorMsg) implements Response {}

public class ApiHandler {
    public static void process(Response response) {
        
        // Java 21 Pattern Matching for Switch!
        // Notice there is NO 'default' branch! 
        // Because the Response interface is SEALED, the compiler mathematically guarantees 
        // that Success and Failure are the ONLY two possibilities in the universe.
        String result = switch (response) {
            
            // We match the type AND instantly extract the record fields!
            case Success(String payload) -> "Process Data: " + payload;
            
            case Failure(int code, String msg) -> "Error " + code + ": " + msg;
        };
        
        System.out.println(result);
    }
}
```

## REAL_WORLD
Records have completely decimated the usage of Lombok `@Data` in modern Spring Boot applications.
When you build a `@RestController`, you typically receive JSON payloads from the frontend. Spring Boot's Jackson library perfectly understands Java Records.

```java
// A perfectly immutable, clean DTO for a REST API
public record CreateUserRequest(
    @NotBlank String email,
    @Size(min = 8) String password
) {}

@RestController
public class UserController {
    @PostMapping("/users")
    public ResponseEntity<String> register(@Valid @RequestBody CreateUserRequest request) {
        // We safely access request.email() knowing it is immutable
        return ResponseEntity.ok("User created: " + request.email());
    }
}
```
Using Records for DTOs (Data Transfer Objects), API Requests, and API Responses is now the absolute industry standard. It guarantees immutability, which prevents bugs where a junior developer accidentally modifies a Request object deep inside a Service layer, completely breaking the data flow.

## INTERVIEW
**Q: Can a Record extend another class?**
A: No. Under the hood, every Record automatically extends `java.lang.Record`. Because Java does not support multiple inheritance for classes, a Record cannot extend any other class. However, a Record *can* implement as many interfaces as it wants.

**Q: Are Records completely immutable?**
A: A Record itself is shallowly immutable (its fields are `final` and the references cannot be reassigned). However, if one of the fields is a mutable object (like a `java.util.List` or an array), the *contents* of that list can still be modified by whoever has the reference! To make a Record deeply immutable, you must use `List.copyOf()` in the compact constructor.

**Q: What is the purpose of a Sealed Class?**
A: Sealed classes restrict the inheritance hierarchy. They allow the author of an API to strictly define all possible subtypes. The primary benefit is realized in modern `switch` expressions: the Java compiler can perform exhaustive checking, ensuring you have handled every possible subclass without needing a useless `default` branch.
