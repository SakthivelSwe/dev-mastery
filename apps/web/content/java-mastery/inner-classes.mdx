## WHY
Sometimes, a class is only useful within the context of exactly one other class. For example, a `Node` class that holds data for a `LinkedList` has absolutely no reason to exist or be accessed outside of the `LinkedList` logic. 

If you make `Node` a top-level, public class, you pollute your package namespace, break encapsulation, and allow other developers to accidentally instantiate `Node` objects for completely unrelated purposes. 

Inner classes allow you to nest a class definition directly inside another class. This provides extreme encapsulation (you can make a class `private`!), and crucially, gives the inner class special access rights to the outer class's private members.

## THEORY
### Types of Inner Classes
Java has four distinct types of nested classes:
1. **Static Nested Class**: Behaves exactly like a normal top-level class, just grouped inside the outer class namespace. It does NOT have access to the outer class's instance variables. 
2. **Member Inner Class (Non-Static)**: Intimately tied to a specific *instance* of the outer class. It secretly holds a hidden reference pointer to the outer object, allowing it to directly read/write the outer object's `private` fields.
3. **Local Inner Class**: A class defined entirely inside a single method body. Rarely used today.
4. **Anonymous Inner Class**: A class defined and instantiated on the fly in a single expression, without a name. Heavily used for simple event listeners before Java 8 Lambdas replaced them.

### Memory Leaks
Member Inner classes are a notorious source of memory leaks in Java. Because the inner object holds a hidden, implicit pointer to the outer object (e.g., `OuterClass.this`), if you pass the inner object to a long-lived cache or thread, the Garbage Collector cannot destroy the massive Outer object, because the inner object is keeping it alive.

## CODE

### Level 1 — Beginner
```java
public class LinkedList {
    private Node head;
    
    // STATIC NESTED CLASS
    // Just a structural grouping. Doesn't have access to LinkedList's instance variables.
    private static class Node {
        int data;
        Node next;
        Node(int data) { this.data = data; }
    }
}
```

### Level 2 — Building Blocks
```java
public class Car {
    private String engineStatus = "OFF";
    
    // MEMBER INNER CLASS (Non-Static)
    // Bound to a specific instance of Car
    public class Engine {
        public void start() {
            // Direct access to outer class private field!
            engineStatus = "ON"; 
            System.out.println("Engine started");
        }
    }
    
    public void turnKey() {
        Engine e = new Engine();
        e.start();
    }
}
```

### Level 3 — Intermediate
```java
public class AnonymousExample {
    public static void main(String[] args) {
        // ANONYMOUS INNER CLASS
        // Creates a one-off class implementing Runnable on the fly
        Thread t = new Thread(new Runnable() {
            @Override
            public void run() {
                System.out.println("Running in a new thread");
            }
        });
        t.start();
        
        // Note: In modern Java, this is replaced by a Lambda: () -> System.out.println(...)
    }
}
```

### Level 4 — Advanced
```java
public class Shadowing {
    private int x = 10;
    
    public class Inner {
        private int x = 20; // Shadows the outer 'x'
        
        public void printValues(int x) {
            System.out.println(x); // Local parameter
            System.out.println(this.x); // Inner class field (20)
            
            // Advanced syntax to access the shadowed outer field!
            System.out.println(Shadowing.this.x); // (10)
        }
    }
}
```

## REAL_WORLD
Static Nested Classes are universally used to implement the **Builder Pattern** in enterprise Java. 

When you have a class with 15 fields, a constructor with 15 parameters is unreadable and prone to errors. You create a `public static class Builder` inside the target class. 
```java
public class User {
    private final String name;
    private final String email;
    
    private User(Builder b) {
        this.name = b.name;
        this.email = b.email;
    }
    
    // The Builder is a Static Nested Class
    public static class Builder {
        private String name;
        private String email;
        
        public Builder name(String name) { this.name = name; return this; }
        public Builder email(String email) { this.email = email; return this; }
        public User build() { return new User(this); }
    }
}
// Usage: User u = new User.Builder().name("Alice").email("a@b.com").build();
```
This pattern is so ubiquitous that Project Lombok's `@Builder` annotation automatically generates this exact Static Nested Class bytecode for you at compile time.

## INTERVIEW
**Q: What is the difference between a Static Nested Class and a Member Inner Class?**
A: A Static Nested Class is independent; it does not require an instance of the outer class to be instantiated, and it cannot access the non-static fields of the outer class. A Member Inner class requires an instance of the outer class to exist first. It holds a hidden reference (`Outer.this`) to that instance, allowing it to directly read and mutate the outer class's private fields.

**Q: Why do Anonymous Inner Classes cause memory leaks in Android development?**
A: In Android, developers historically used Anonymous Inner classes for click listeners or background threads inside an `Activity`. Because it's an inner class, it holds a hidden reference to the massive `Activity` object. If the background thread continues running after the user rotates the screen (which destroys the `Activity`), the Garbage Collector cannot clean up the destroyed `Activity` because the running inner class still holds a pointer to it, causing an OutOfMemory crash.
