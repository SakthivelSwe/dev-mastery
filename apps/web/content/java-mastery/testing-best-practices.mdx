## WHY
"It works on my machine."
When you change a line of code in the Payment module, how do you know you didn't accidentally break the User Registration module? Without automated tests, you are forced to manually click through the entire application every time you make a change, a process that doesn't scale.

Writing comprehensive tests ensures your code does exactly what it is supposed to do. It documents the expected behavior, protects against regressions, and gives developers the confidence to ruthlessly refactor ugly code, knowing the tests will catch any mistakes.

## THEORY
### The Testing Pyramid
1. **Unit Tests (Bottom):** Test a single method or class in absolute isolation. They must run in milliseconds. They form 70% of your test suite.
2. **Integration Tests (Middle):** Test how multiple components work together (e.g., testing if the Repository class actually reads from a real Database). Slower, but critical.
3. **End-to-End Tests (Top):** Simulate a real user clicking through the UI and hitting the full backend. Very slow, brittle, and expensive. Use sparingly.

## CODE

### Level 1 — Beginner (JUnit 5 Basics)
JUnit is the industry standard framework for writing tests in Java.

```java
import org.junit.jupiter.api.*;
import static org.junit.jupiter.api.Assertions.*;

class CalculatorTest {
    private Calculator calc;

    // Runs before EVERY SINGLE test method! Perfect for fresh initialization.
    @BeforeEach
    void setUp() {
        calc = new Calculator();
    }

    // The @Test annotation tells JUnit to run this method
    @Test
    @DisplayName("Should correctly add two positive numbers")
    void testAddition() {
        // Arrange
        int a = 5;
        int b = 3;
        
        // Act
        int result = calc.add(a, b);
        
        // Assert (Did it do what we expected?)
        assertEquals(8, result, "5 + 3 should equal 8");
    }

    @Test
    void testDivisionByZero() {
        // Assert that a specific Exception is thrown!
        assertThrows(ArithmeticException.class, () -> {
            calc.divide(10, 0);
        });
    }
}
```

### Level 2 — Building Blocks (Mocking with Mockito)
Unit tests must run in isolation. If your `UserService` needs to send an email via the `EmailService`, you DO NOT want your test to actually send a real email! 
**Mockito** is a library that creates fake (Mock) versions of dependencies. You can explicitly program the Mock to return specific data, allowing you to test the Service logic in perfect isolation.

```java
import org.junit.jupiter.api.Test;
import org.junit.jupiter.api.extension.ExtendWith;
import org.mockito.InjectMocks;
import org.mockito.Mock;
import org.mockito.junit.jupiter.MockitoExtension;
import static org.mockito.Mockito.*;
import static org.junit.jupiter.api.Assertions.*;

// Tells JUnit to initialize Mockito annotations!
@ExtendWith(MockitoExtension.class)
class UserServiceTest {

    // 1. Create a Fake Version of the database repository
    @Mock
    private UserRepository userRepository;

    // 2. Inject the Fake repository into the Real service we want to test!
    @InjectMocks
    private UserService userService;

    @Test
    void testGetUserFullName() {
        // Arrange: Program the Mock! 
        // "When anyone calls findById(1), return this specific User object."
        User mockUser = new User("John", "Doe");
        when(userRepository.findById(1L)).thenReturn(mockUser);

        // Act: Call the real method
        String fullName = userService.getUserFullName(1L);

        // Assert: Check the result
        assertEquals("John Doe", fullName);
        
        // Verify: Ensure the Service actually called the Database exactly 1 time!
        verify(userRepository, times(1)).findById(1L);
    }
}
```

### Level 3 — Intermediate (Integration Testing with Testcontainers)
Mocking the database is great for Unit Tests, but eventually, you need to know if your SQL queries actually work against a real PostgreSQL database. 
Historically, developers used H2 (an in-memory database) for tests, but H2 has different syntax than PostgreSQL, leading to tests that pass locally but fail in production!

**Testcontainers** solves this by spinning up a real Docker container for your database during the test phase!

```java
import org.junit.jupiter.api.Test;
import org.testcontainers.containers.PostgreSQLContainer;
import org.testcontainers.junit.jupiter.Container;
import org.testcontainers.junit.jupiter.Testcontainers;

// Tells JUnit to manage the Docker containers!
@Testcontainers
class UserRepositoryIntegrationTest {

    // Spin up a real, ephemeral PostgreSQL database strictly for this test!
    @Container
    public static PostgreSQLContainer<?> postgres = new PostgreSQLContainer<>("postgres:15-alpine")
            .withDatabaseName("testdb")
            .withUsername("user")
            .withPassword("password");

    @Test
    void testDatabaseConnectionAndQuery() {
        // The container provides the dynamic JDBC URL!
        String jdbcUrl = postgres.getJdbcUrl();
        
        // Inject this URL into your UserRepository and run your SQL queries...
        // ...
        
        assertTrue(postgres.isRunning());
    }
}
```

### Level 4 — Advanced (Parameterized Tests)
Writing 5 separate test methods to check boundary conditions (`testAddZero`, `testAddNegative`, `testAddLargeValues`) is exhausting. JUnit 5 allows you to run the exact same test method multiple times with different data inputs!

```java
import org.junit.jupiter.params.ParameterizedTest;
import org.junit.jupiter.params.provider.CsvSource;
import static org.junit.jupiter.api.Assertions.assertEquals;

class MathUtilsTest {

    // Provide the inputs and the expected output in a CSV format!
    @ParameterizedTest(name = "{0} + {1} should equal {2}")
    @CsvSource({
        "1, 1, 2",
        "-5, 10, 5",
        "0, 0, 0",
        "100, -100, 0"
    })
    void testAdditionCombinations(int a, int b, int expectedResult) {
        Calculator calc = new Calculator();
        assertEquals(expectedResult, calc.add(a, b));
    }
}
```

## REAL_WORLD
**Test-Driven Development (TDD)**
TDD is a software engineering practice where you literally write the Tests *before* you write the production code. 
1. **Red:** Write a failing test for a feature that doesn't exist yet.
2. **Green:** Write the absolute minimum amount of messy code required to make the test pass.
3. **Refactor:** Clean up the messy code, knowing the test will immediately alert you if you break the logic.
While dogmatic TDD isn't used by every team, writing tests alongside your code (rather than as an afterthought 3 weeks later) is mandatory in all professional Agile environments.

## INTERVIEW
**Q: I wrote a Unit Test that generates a random number, saves it to the database, and then reads it back. Sometimes the test passes, and sometimes it fails if another test runs at the same time. What principle did I violate?**
A: **You violated Test Isolation.**
A Unit Test must be completely deterministic and independent. If a test relies on a shared resource (like an external database) or relies on the execution order of other tests, it is not a Unit Test; it is a brittle, non-deterministic Integration Test (often called a "Flaky Test"). 
Unit tests should mock external dependencies (using Mockito) so they run perfectly in memory, 100% of the time, regardless of what other tests are running in parallel.

**Q: What is the difference between a Mock, a Stub, a Fake, and a Spy?**
A: All four are **test doubles**.
* **Stub** — returns canned values; no verification (`when(repo.findById(1)).thenReturn(user)`).
* **Mock** — also verifies *how* it was called (`verify(repo, times(1)).save(any())`).
* **Spy** — wraps a real object; calls go to the real method unless overridden. Useful for partial mocking of legacy code.
* **Fake** — a working but lightweight implementation (e.g., an in-memory `Map`-based repository). The Spring Boot team calls this "the best kind of test double" because it behaves like real code without the network cost.

**Q: What is the Test Pyramid?**
A: A model proposed by Mike Cohn: at the base sit many fast, isolated **unit tests** (~70 %), in the middle sit **integration / component tests** (~20 %), at the top sit slow, brittle **end-to-end UI tests** (~10 %). Inverting the pyramid (lots of E2E, few units) is the classic anti-pattern that creates 2-hour CI builds and flaky pipelines.

**Q: What does code coverage tell you, and what does it NOT tell you?**
A: Coverage tells you which lines of production code were *executed* by tests. It does **not** tell you whether the tests *assert* anything meaningful, whether edge cases are covered, or whether the tests would catch real regressions. A class with 100 % line coverage and zero `assertEquals` is worthless. Use coverage as a floor (e.g., "block merges below 80 %"), never as a ceiling.

**Q: What is Mutation Testing and why is it better than line coverage?**
A: Mutation testing tools (e.g., PIT, Stryker) automatically alter your production code — flipping `<` to `<=`, changing `+` to `-`, replacing a return value with `null` — and re-run your test suite. If a test still passes after the mutation, the test is too weak. The score is the percentage of mutants your tests *killed*. This actually measures test quality, not just code reach.

**Q: How do you make tests deterministic when they depend on time?**
A: Inject a `Clock` (Java 8+) into the class under test, default it to `Clock.systemUTC()` in production, and inject `Clock.fixed(Instant.parse("2025-01-01T00:00:00Z"), ZoneOffset.UTC)` in tests. **Never** call `Instant.now()` or `new Date()` directly in business code — that single discipline eliminates an entire class of flaky tests.

## FEYNMAN CHECK

Explain testing to a non-engineer in 60 seconds:

> Imagine you build a bridge. Before letting cars on it, you don't just look at it and hope — you drive a test truck across, drop weights on it, simulate an earthquake. Every test answers one question: *under this exact load, does the bridge still hold?* Software tests are exactly the same. A **unit test** is "does this single bolt hold under this load?" An **integration test** is "do these three connected beams hold together?" An **end-to-end test** is "can a real truck drive the full span?" If you skip the bolt tests, you find out the bolts are bad only after the bridge collapses — in production, under a customer.

**Self-test questions:**
1. What are the three "A"s of a good unit test? (**Arrange, Act, Assert**)
2. What makes a test "deterministic" and why does it matter for CI?
3. What is the difference between `@BeforeEach` and `@BeforeAll` in JUnit 5?
4. What is a "test double" and what are the four common kinds?
5. Name three things `@SpringBootTest` does under the hood.
6. How do you test private methods? (Trick question — you don't; test the public API that uses them.)
7. What is mutation testing, and what does a high mutation score prove that high line coverage does not?

If you can answer all seven you have a senior-level grip on Java testing.

## BUILD

**Mini-project: a tested banking transfer service.**

Goal: build a `TransferService` that moves money between two accounts, persists the transactions, and publishes a domain event. Cover it with the full **test pyramid** — unit, integration, and end-to-end.

### Requirements
1. **Unit tests (JUnit 5 + Mockito).** Mock the `AccountRepository`, the `EventPublisher`, and the `Clock`. Cover at least:
   * Happy path (debit + credit + event).
   * Insufficient funds rejection.
   * Idempotent retry — same `transferId` must not double-debit.
   * Concurrent transfer races (use `CountDownLatch` to force a deadlock window).
2. **Parameterized tests** for at least 10 boundary amounts: 0, -1, 0.01, `Long.MAX_VALUE`, etc.
3. **Integration tests** with `@SpringBootTest(webEnvironment = NONE)` and an in-memory H2 schema. Verify that the real JPA repository, Flyway migrations, and Spring transaction boundaries all cooperate.
4. **End-to-end tests** with `@SpringBootTest(webEnvironment = RANDOM_PORT)` + **Testcontainers** running real PostgreSQL. Hit the REST endpoint with `WebTestClient`, then `SELECT` directly from the DB to confirm rows landed.
5. Wire **Jacoco** coverage and **PIT mutation testing**. Aim for ≥ 90 % line coverage and ≥ 80 % mutation score in `TransferService`.
6. Configure **flaky-test detection** — re-run failing tests once in CI; if a test passes on retry, fail the build but tag the test as "flaky candidate" for human review.

### Stretch goals
* Add **Pact** consumer-driven contract tests against a fake downstream notification service.
* Add **Awaitility** to test asynchronous event publication without `Thread.sleep`.
* Add **ArchUnit** rules so no controller can call a repository directly, and no service can have a `public` field.

This single project exercises every senior Java testing skill: mocking, parameterization, transactions, containers, contracts, coverage, mutation, async, and architecture verification.
