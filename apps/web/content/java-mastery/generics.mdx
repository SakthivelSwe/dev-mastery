## WHY
Before Java 5, collections like `ArrayList` only held `Object` references. If you wanted a list of Strings, you did `list.add("Hello")`. But because the list only knew about `Object`, another developer could do `list.add(new Integer(5))`. The compiler allowed this.
When you looped through the list later, you had to manually cast every item back to a String: `String s = (String) list.get(0)`. When the loop hit the `Integer`, the program violently crashed with a `ClassCastException` at runtime.

Generics were introduced to provide **Compile-Time Type Safety**. By defining the list as `ArrayList<String>`, you tell the compiler exactly what type of objects are allowed. If someone tries to add an `Integer`, the code simply refuses to compile. By shifting errors from Runtime (crashing the app) to Compile-Time (blocking the build), Generics fundamentally revolutionized Java reliability.

## THEORY
### Type Erasure
The most critical thing to understand about Java Generics is **Type Erasure**. 
When you compile `List<String> list = new ArrayList<>();`, the Java compiler strictly enforces the `<String>` rules. But during the actual compilation into bytecode, the compiler *erases* the `<String>` information completely. 
In the final `.class` bytecode, it just becomes `List list = new ArrayList();` holding generic `Object` references. 
Why did Java do this? **Backward Compatibility**. They had to ensure that Java 5 code using generics could still interoperate perfectly with Java 1.4 code that knew nothing about generics. Because of Type Erasure, you cannot check generic types at runtime (e.g., `if (list instanceof List<String>)` is illegal and won't compile).

### Bounded Type Parameters
Sometimes you want a generic method to accept any type, but with restrictions. 
- `<T extends Number>`: Accepts `Number` or any of its children (`Integer`, `Double`).
- `<T super Integer>`: Accepts `Integer` or any of its parents (`Number`, `Object`).
This is heavily used in generic sorting and filtering algorithms.

## CODE

### Level 1 — Beginner
```java
import java.util.ArrayList;
import java.util.List;

public class BasicGenerics {
    public static void main(String[] args) {
        // The Diamond Operator <> (Java 7+) infers the type from the left side
        List<String> names = new ArrayList<>();
        
        names.add("Alice");
        // names.add(100); // COMPILER ERROR! Type safety in action.
        
        // No manual casting required!
        String first = names.get(0); 
    }
}
```

### Level 2 — Building Blocks
```java
// Writing your own Generic Class
public class Box<T> {
    private T item;
    
    public void setItem(T item) { this.item = item; }
    public T getItem() { return item; }
}

public class Usage {
    public static void main(String[] args) {
        Box<Integer> intBox = new Box<>();
        intBox.setItem(42);
        Integer val = intBox.getItem();
    }
}
```

### Level 3 — Intermediate
```java
// Generic Methods and Bounded Types
public class GenericMethods {
    
    // This method accepts ANY type of Number (Integer, Double, etc.)
    public static <T extends Number> double sum(T num1, T num2) {
        // Because of the bound, we know .doubleValue() exists!
        return num1.doubleValue() + num2.doubleValue();
    }
    
    public static void main(String[] args) {
        System.out.println(sum(10, 20.5)); 
    }
}
```

### Level 4 — Advanced
```java
// PECS Rule: Producer Extends, Consumer Super
import java.util.List;

public class Wildcards {
    // PRODUCER: We are READing data from this list. 
    // It produces data. Use <? extends T>.
    public static void printNumbers(List<? extends Number> list) {
        for (Number n : list) { System.out.println(n); }
        // list.add(10); // ERROR! You cannot add to an extends wildcard!
    }
    
    // CONSUMER: We are WRITING data to this list. 
    // It consumes data. Use <? super T>.
    public static void addIntegers(List<? super Integer> list) {
        list.add(10);
        list.add(20);
        // Number n = list.get(0); // ERROR! You cannot safely read from a super wildcard!
    }
}
```

## REAL_WORLD
Generics are heavily used in modern Repository layers (like Spring Data JPA) to eliminate boilerplate CRUD (Create, Read, Update, Delete) code.

Instead of writing a `save()` method for `User`, and another `save()` for `Order`, you write one abstract Generic interface:
```java
public interface CrudRepository<T, ID> {
    T save(T entity);
    Optional<T> findById(ID id);
    void delete(T entity);
}

// Spring automatically generates the implementation at runtime!
public interface UserRepository extends CrudRepository<User, Long> {
}
```
In this pattern, `<T, ID>` are generic type parameters. When `UserRepository` extends it, it locks `T` to `User` and `ID` to `Long`. Spring analyzes these generic boundaries at startup using reflection (technically reading the erased Generic Signature metadata) and builds a flawless, type-safe database proxy specifically for the `User` table.

## INTERVIEW
**Q: What is Type Erasure?**
A: Type Erasure is the process by which the Java compiler removes all Generic type information during compilation to bytecode. `List<String>` becomes a raw `List` holding `Object` references. This was done to maintain strict backward compatibility with older Java versions. As a result, you cannot check generic types at runtime (`instanceof List<String>` is illegal).

**Q: Explain the PECS rule.**
A: PECS stands for **Producer Extends, Consumer Super**. It is a mnemonic for dealing with Generic Wildcards (`?`). If you have a method that solely reads data from a generic collection, that collection is a Producer, and you should use `<? extends T>` to allow children types to be read. If your method solely inserts data into a collection, the collection is a Consumer, and you should use `<? super T>` to ensure you can safely write to it.

**Q: Can you create a generic array, like `T[] array = new T[10];`?**
A: No, this will cause a compilation error. Because of Type Erasure, `T` is erased to `Object` (or its bound). Arrays, however, must know their exact type at runtime to enforce bounds and type checking (e.g., throwing `ArrayStoreException`). Because the runtime doesn't know what `T` is, it cannot safely construct the array. You must use `Object[]` and cast it, or use `ArrayList<T>`.
