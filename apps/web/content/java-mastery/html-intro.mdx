# HTML — Document Structure, DOCTYPE, Semantic Purpose

### 1. Deep-Dive Technical Explanation

#### The DOCTYPE Declaration and Quirks Mode
The modern `<!DOCTYPE html>` declaration is not an HTML tag; it is an instruction to the browser's parser that the document is written in HTML5. Historically, the Document Type Definition (DTD) referenced complex SGML schemas. In HTML5, it exists solely to trigger **Standards Mode** in the browser. 

If the DOCTYPE is omitted or malformed, browsers fall back to **Quirks Mode**, rendering the document using non-standard layout rules mimicking late-1990s browsers (such as IE5). This alters the CSS box model (where width includes padding and borders) and disables many modern layout features.

#### HTML Parsing Algorithm and Tokenization
The HTML parser is unique compared to typical compiler parsers because it cannot fail. It operates on a state-machine model consisting of two main phases:
1. **Tokenization**: Converting the raw character stream into tokens (StartTag, EndTag, Character, Comment, DOCTYPE).
2. **Tree Construction**: Consuming these tokens to build the Document Object Model (DOM) tree.

The parser handles malformed HTML using explicit error-recovery rules specified in the HTML Living Standard. For example, if you open a `p` tag and then open a `div` tag, the parser automatically closes the `p` tag because a block-level `div` cannot reside inside a phrasing-content paragraph.

```
[Character Stream] -> [Tokenizer] -> [Tokens] -> [Tree Construction] -> [DOM Tree]
                                                        ^
                                                        | (Interrupted by synchronous scripts)
```

#### DOM Tree vs. Accessibility Object Model (AOM)
As the browser constructs the DOM, it concurrently builds the **Accessibility Tree** (and the emerging Accessibility Object Model, or AOM). The Accessibility Tree is a subset of the DOM containing only elements that are meaningful to assistive technologies (like screen readers). 

```
DOM Tree:
└── body
    ├── header
    │   └── h1 ("Title")
    └── main
        └── p ("Text")

Accessibility Tree:
└── banner (Header landmark)
    └── heading level 1 ("Title")
└── main (Main landmark)
    └── text ("Text")
```

Non-semantic elements like `div` and `span` do not generate nodes in the Accessibility Tree unless they are explicitly assigned ARIA roles. Semantic elements like `<header>`, `<main>`, and `<article>` map directly to accessible landmark roles.

### 2. Complete, Valid HTML5 Example

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>Semantic Document Architecture</title>
  <link rel="stylesheet" href="style.css">
</head>
<body>

  <!-- Skip link for keyboard navigation -->
  <a href="#main-content" class="skip-link">Skip to main content</a>

  <header class="site-header">
    <div class="logo">DevMastery</div>
    <nav aria-label="Main Site Navigation">
      <ul class="nav-links">
        <li><a href="#home">Home</a></li>
        <li><a href="#docs">Documentation</a></li>
        <li><a href="#contact">Contact</a></li>
      </ul>
    </nav>
  </header>

  <main id="main-content" class="main-layout" tabindex="-1">
    <article class="hero-section">
      <h1>The Evolution of Web Semantics</h1>
      <p>Understanding the underlying parser rules and accessibility mappings empowers developers to construct high-performance, inclusive web applications.</p>
    </article>
  </main>

  <footer class="site-footer">
    <p>&copy; 2026 DevMastery Content Engine. All rights reserved.</p>
  </footer>

</body>
</html>
```

### 5. Accessibility Analysis

- **Document Language**: The `<html lang="en">` attribute is present. This is critical because it tells screen readers which speech synthesis engine to load (e.g., English pronunciation instead of French).
- **Landmark Regions**:
  - `<header>` maps to `role="banner"`.
  - `<nav>` maps to `role="navigation"`. The `aria-label="Main Site Navigation"` distinguishes it from other navigation bars on the page.
  - `<main>` maps to `role="main"`.
  - `<footer>` maps to `role="contentinfo"`.
- **Keyboard Navigation**: The `skip-link` allows keyboard users to bypass the navigation links and jump straight to the main content. The `<main>` element has `tabindex="-1"` so that when the skip link is clicked, programmatic focus moves to the main content area without adding the main tag to the natural tab order.

### 6. WCAG 2.1 AA Compliance Note & Validation

- **WCAG 2.1 AA Success Criterion 3.1.1 (Language of Page)**: Satisfied via the `lang="en"` attribute on the `<html>` tag.
- **WCAG 2.1 AA Success Criterion 2.4.1 (Bypass Blocks)**: Satisfied via the implementation of the functional `.skip-link` and semantic landmark elements.
- **Validation**: All tags are correctly nested, attributes are properly quoted, and the document contains exactly one `<main>` element, ensuring 100% compliance with the W3C HTML5 specification.
