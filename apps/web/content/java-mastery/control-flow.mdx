## WHY

Imagine writing a program without control flow. Code would execute top-to-bottom, exactly once, with no ability to skip lines or repeat actions. If you wanted to process 1,000 users, you would literally have to copy and paste the processing logic 1,000 times. If you wanted to show an error message *only* if the user's password was wrong, you couldn't; the error message would print for everyone.

Control flow constructs (`if`, `switch`, `for`, `while`) are the steering wheel and engine of programming. They allow the execution path of the code to branch dynamically based on runtime data, and to loop over operations iteratively.

However, traditional control flow has always been a major source of bugs. The old C-style `switch` statement in Java, for example, forces you to manually write `break;` at the end of every case. If you forget it, the execution silently "falls through" to the next case, causing catastrophic bugs in production. Java recognized this pain and introduced modern **Switch Expressions** (Java 14+) that eliminate fall-through entirely and allow control flow to *return* a value directly. Understanding how the JVM compiles these different constructs into bytecode is essential for writing both safe and extremely fast code.

## THEORY

### `if` vs `switch` at the Bytecode Level
When you write an `if-else if` chain, the Java compiler generates conditional branch instructions (like `ifeq`, `if_icmplt`). The JVM evaluates these conditions sequentially. If you have 50 `if-else if` branches, evaluating the 50th branch takes 50 comparisons. 

When you write a `switch` statement on an `int` or `String`, the compiler generates entirely different bytecode. It creates a `tableswitch` or a `lookupswitch`. 
- **`tableswitch`**: Used when case values are dense (e.g., 1, 2, 3, 4). The JVM essentially creates an array of memory addresses. It uses the `switch` variable as an index to jump *instantly* in O(1) time to the correct block of code.
- **`lookupswitch`**: Used when case values are sparse (e.g., 10, 1000, 50000). The JVM creates a sorted table of keys and uses Binary Search to find the jump target in O(log n) time.

Because of this, a `switch` statement is almost always faster than a long `if-else if` chain.

### Traditional `switch` vs Enhanced Switch Expressions
The traditional `switch` is a **statement**. It performs actions but doesn't resolve to a value. It also suffers from "fall-through": if a case matches, it executes that case and *every case below it* unless stopped by a `break;`. Fall-through exists because C programmers used it for Duff's Device (loop unrolling), but in modern Java, it's mostly a liability.

Java 14 introduced the **Switch Expression**.
1. It uses `->` syntax instead of `:`.
2. It completely eliminates fall-through. No `break;` needed.
3. It resolves to a single value, making it perfect for assignments.
4. If a block of code is needed, you use the `yield` keyword to return the value from that block.

### Labeled `break` and `continue`
In deeply nested loops, a normal `break` only exits the immediate, innermost loop. Java does not have `goto` (it is a reserved but unused keyword). Instead, Java provides labeled breaks. By tagging a loop with a label (e.g., `OUTER_LOOP:`), you can forcefully terminate the entire outer execution context from deep within a nested loop.

## CODE

### Level 1 — Beginner
```java
public class BasicIf {
    public static void main(String[] args) {
        int age = 15;
        
        // Basic sequential branching
        if (age >= 18) {
            System.out.println("Adult");
        } else if (age >= 13) {
            System.out.println("Teen");
        } else {
            System.out.println("Child");
        }
    }
}
```

### Level 2 — Building Blocks
```java
public class OldSwitch {
    public static void main(String[] args) {
        String role = "ADMIN";
        
        // TRADITIONAL SWITCH (Statement)
        switch (role) {
            case "ADMIN":
                System.out.println("Full Access");
                break; // If you forget this, it falls through to USER!
            case "USER":
                System.out.println("Read Access");
                break;
            default:
                System.out.println("No Access");
        }
    }
}
```

### Level 3 — Intermediate
```java
public class EnhancedSwitch {
    public static void main(String[] args) {
        String role = "ADMIN";
        
        // JAVA 14+ SWITCH EXPRESSION
        // 1. Evaluates to a value (assigned to 'access')
        // 2. Uses arrow syntax '->'
        // 3. NO FALL-THROUGH possible
        int accessLevel = switch (role) {
            case "ADMIN" -> 100;
            case "EDITOR", "MODERATOR" -> 50; // Multiple matching allowed
            case "USER" -> {
                // If you need complex logic inside a branch, use a block
                System.out.println("Calculating user access...");
                // Use 'yield' to return the value from the block
                yield 10; 
            }
            default -> 0; // Exhaustiveness is enforced by compiler!
        };
        
        System.out.println("Access Level: " + accessLevel);
    }
}
```

### Level 4 — Advanced
```java
public class LabeledBreak {
    public static void main(String[] args) {
        int[][] matrix = {
            {1, 2, 3},
            {4, 5, 6},
            {7, 8, 9}
        };
        
        int target = 5;
        boolean found = false;
        
        // Label the outer loop
        SEARCH_MATRIX:
        for (int row = 0; row < matrix.length; row++) {
            for (int col = 0; col < matrix[row].length; col++) {
                if (matrix[row][col] == target) {
                    found = true;
                    // Breaks the OUTER loop entirely, skipping remaining rows
                    break SEARCH_MATRIX; 
                }
            }
        }
    }
}
```

## REAL_WORLD

In production Spring Boot applications, developers often misuse massive `if-else if` chains to route incoming webhook events or apply strategy patterns. 

```java
// ANTI-PATTERN: The "God Method" of if-else
public void handleWebhook(String eventType, String payload) {
    if (eventType.equals("USER_CREATED")) {
        userHandler.handle(payload);
    } else if (eventType.equals("USER_DELETED")) {
        deleteHandler.handle(payload);
    } else if (eventType.equals("PAYMENT_FAILED")) {
        paymentHandler.handle(payload);
    } // ... 50 more branches
}
```
This is an anti-pattern because adding a new event type requires modifying the core routing file, violating the Open-Closed Principle (OCP).

However, if you *must* use direct routing logic, modern Java dictates you should use a Switch Expression, as the compiler enforces **Exhaustiveness**. If `eventType` was an `enum` rather than a `String`, the compiler would physically prevent the code from building if you forgot to handle a specific event type.

The absolute best production pattern to replace complex `if-else` control flow is using a `Map` of strategies or the Strategy Pattern combined with Spring's `@Component` mapping.
```java
// GOOD PATTERN: Eliminating control flow entirely via Maps
@Service
public class WebhookRouter {
    // Spring automatically injects all beans implementing WebhookHandler
    private final Map<String, WebhookHandler> handlers;
    
    public void route(String eventType, String payload) {
        WebhookHandler handler = handlers.get(eventType);
        if (handler != null) {
            handler.handle(payload);
        }
    }
}
```

## INTERVIEW

**Q: Does a `switch` statement perform faster than an `if-else if` chain? Why?**
A: Yes, generally. For an `if-else if` chain, the JVM executes sequential branch comparisons (O(n) complexity). For a `switch` statement, the compiler generates a `tableswitch` (O(1) direct memory jump for dense values) or a `lookupswitch` (O(log n) binary search for sparse values). This makes `switch` significantly faster for a large number of conditions.

**Q: What is the difference between `break` and `yield` inside a switch block?**
A: `break` is used in traditional switch *statements* to prevent fall-through to the next case block and exit the switch. `yield` is used in Java 14+ switch *expressions* (when a `{}` block is utilized) to return a specific value back to the caller of the expression.

**Q: Can you use a `switch` statement on objects in Java?**
A: Historically, no. You could only switch on primitives (`int`, `char`, etc.), `Enum` types, and (since Java 7) `String`. However, starting with Java 17 and Java 21's Pattern Matching for Switch, you can now switch on any Object type, and even extract variables simultaneously (e.g., `case User u -> process(u)`).

**Q: Explain how labeled `break` differs from the `goto` statement found in C++.**
A: `goto` allows jumping to any arbitrary line of code anywhere in a function, which leads to unpredictable, "spaghetti" control flow. Java's labeled `break` is strictly constrained: you can only jump outward to the end of a specific enclosing loop block. You cannot jump into a loop, nor can you jump backward in execution.
