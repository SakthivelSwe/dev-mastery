## WHY

Imagine writing a program without control flow. Code would execute top-to-bottom, exactly once, with no ability to skip lines or repeat actions. To process 1,000 users, you'd copy-paste the logic 1,000 times. To show an error only on wrong passwords, you couldn't — the error would print for everyone.

Control flow constructs are the **steering wheel and engine** of programming. But they go far deeper than syntax — they directly affect:

1. **Performance**: A 50-branch `if-else` chain is O(n). A `switch` with `tableswitch` bytecode is O(1). Wrong choice = measurable slowdown at scale.
2. **Safety**: C-style `switch` fall-through silently executes wrong cases in production. Java 14+ switch expressions eliminate this by design.
3. **Readability**: Deeply nested `if-else` creates the "arrow anti-pattern" — code that pyramids rightward, impossible to read or test.
4. **Compiler guarantees**: `switch` expressions over `enum` and sealed classes enforce exhaustiveness — the compiler refuses to build if you miss a case.
5. **Modern Java patterns**: Pattern Matching for Switch (Java 21) lets you type-check and destructure in one expression, eliminating entire `instanceof` + cast chains.

Senior developers don't just know the syntax — they know which construct the JVM compiles to `tableswitch` vs `lookupswitch`, when to replace `if-else if` chains with `Map<String, Strategy>`, and how Java 21 pattern matching eliminates verbosity.

## THEORY

### `if-else` vs `switch` at the Bytecode Level

When you write `if-else if`, the compiler generates **sequential branch bytecodes**:
```
iload_1          // push variable onto operand stack
bipush 18        // push constant 18
if_icmpge L1     // if var >= 18, jump to L1
...
L1: ...
```
With 50 branches, the 50th evaluation requires 50 comparisons. Performance degrades linearly: O(n).

When you write a `switch` on integers/strings, the compiler generates fundamentally different bytecode:

| Bytecode | When Used | Complexity | Mechanism |
|----------|-----------|------------|-----------|
| `tableswitch` | Dense cases (1,2,3,4,5) | **O(1)** | Array of jump addresses; uses value as index |
| `lookupswitch` | Sparse cases (1, 1000, 5000) | **O(log n)** | Sorted key-value pairs; binary search |

The `tableswitch` is essentially a jump table — the JVM takes the switch variable, uses it as an array index, and jumps directly to the correct code block in a single operation. This is why `switch` dramatically outperforms `if-else if` for large numbers of cases.

### Traditional `switch` Statement Problems

The old C-style switch is a **statement** (performs actions, produces no value) with two critical flaws:

**Flaw 1: Fall-through** — execution continues to the next case unless stopped by `break`:
```java
switch (status) {
    case "NEW":
        initOrder();      // Executes
        // ← Missing break!
    case "PROCESSING":
        processOrder();   // Also executes — SILENT BUG
        break;
}
```

**Flaw 2: No exhaustiveness check** — if you handle 5 of 6 enum values, the compiler doesn't warn you. The uncovered case silently falls to `default` (or does nothing if no default exists).

### Java 14+ Switch Expressions (The Modern Standard)

The switch **expression** (preview Java 12-13, stable Java 14+) solves both problems:

| Feature | Old switch Statement | New switch Expression |
|---------|---------------------|----------------------|
| Syntax | `case X:` with `:` | `case X ->` with `->` |
| Fall-through | Yes (by default) | **Impossible** |
| Returns value | No | **Yes** — assigns to variable |
| Exhaustiveness | Not checked | **Compiler enforced** |
| Multiple labels | No | Yes — `case A, B, C ->` |
| Block syntax | N/A | Use `yield` to return from block |

### Pattern Matching for Switch (Java 21)

Java 21 added type patterns, guarded patterns, and `null` handling:

```java
// Before Java 21 — verbose instanceof + cast
String describe(Object obj) {
    if (obj instanceof String s) {
        return "String of length " + s.length();
    } else if (obj instanceof Integer i && i > 0) {
        return "Positive int: " + i;
    } else if (obj == null) {
        return "null";
    }
    return "other";
}

// Java 21 — clean pattern matching switch
String describe(Object obj) {
    return switch (obj) {
        case String s  -> "String of length " + s.length();
        case Integer i when i > 0 -> "Positive int: " + i;
        case null      -> "null";
        default        -> "other: " + obj.getClass().getSimpleName();
    };
}
```

### Loop Types and Use Cases

| Loop | When to Use | Key Feature |
|------|-------------|-------------|
| `for` | Known iteration count | Compact; `i` scoped to loop |
| `while` | Unknown count, condition-first | Tests before each iteration |
| `do-while` | Must run at least once | Tests after each iteration |
| Enhanced `for-each` | Iterating collections/arrays | Clean; no index needed |
| `Iterator` | Need `remove()` during iteration | Avoids `ConcurrentModificationException` |

### Labeled Break and Continue

Java has no `goto` (reserved keyword but never used). Instead, labels let you break/continue outer loops:

```
OUTER:                    ← label on outer loop
for (...) {
    INNER:                ← label on inner loop
    for (...) {
        break OUTER;      ← exits outer loop entirely
        continue OUTER;   ← skips to next outer iteration
    }
}
```

Labels are legitimate — not spaghetti code — for matrix traversal, graph search, and multi-level state machine exit conditions.

### The Arrow Anti-Pattern (and How to Fix It)

```java
// ARROW ANTI-PATTERN — impossible to read or test
if (user != null) {
    if (user.isActive()) {
        if (user.hasPermission("ADMIN")) {
            if (request != null) {
                // actual logic buried 4 levels deep
            }
        }
    }
}

// GUARD CLAUSE pattern — early return flattens nesting
if (user == null) throw new IllegalArgumentException("user required");
if (!user.isActive()) throw new IllegalStateException("user inactive");
if (!user.hasPermission("ADMIN")) throw new SecurityException("denied");
if (request == null) throw new IllegalArgumentException("request required");
// actual logic at top level — easy to read and test
```

## CODE

### Level 1 — Beginner
```java
// Basic control flow: if-else, for, while, do-while
public class BasicControlFlow {
    public static void main(String[] args) {
        int age = 15;

        // if-else if-else: sequential branching
        if (age >= 18) {
            System.out.println("Adult");
        } else if (age >= 13) {
            System.out.println("Teen");      // prints this
        } else {
            System.out.println("Child");
        }

        // Classic for loop
        System.out.print("Counting: ");
        for (int i = 1; i <= 5; i++) {
            System.out.print(i + " ");       // 1 2 3 4 5
        }
        System.out.println();

        // while loop — unknown iteration count
        int n = 1;
        while (n < 100) {
            n *= 2;
        }
        System.out.println("First power of 2 >= 100: " + n);  // 128

        // do-while — runs at least once
        int attempts = 0;
        do {
            attempts++;
            System.out.println("Attempt " + attempts);
        } while (attempts < 3);  // runs 3 times
    }
}
```

### Level 2 — Building Blocks
```java
// Traditional switch vs modern switch expression
public class SwitchComparison {
    public static void main(String[] args) {
        String role = "ADMIN";

        // OLD switch statement — fall-through danger
        switch (role) {
            case "ADMIN":
                System.out.println("Full access");
                break;  // ← REQUIRED — without this, falls through to USER!
            case "EDITOR":
            case "MODERATOR":
                System.out.println("Write access");
                break;
            case "USER":
                System.out.println("Read access");
                break;
            default:
                System.out.println("No access");
        }

        // NEW switch expression (Java 14+) — no fall-through possible
        int accessLevel = switch (role) {
            case "ADMIN"               -> 100;  // Arrow → no fall-through
            case "EDITOR", "MODERATOR" -> 50;   // Multiple labels in one case
            case "USER"                -> 10;
            default                    -> 0;
        };
        System.out.println("Access level: " + accessLevel);  // 100

        // Switch expression with block + yield
        String description = switch (role) {
            case "ADMIN" -> {
                System.out.println("Computing admin description...");
                yield "Administrator with full system access";  // yield returns value
            }
            default -> "Standard user";
        };
        System.out.println(description);
    }
}
```

### Level 3 — Intermediate
```java
// Loop control: break, continue, labeled break
public class LoopControl {

    // Find first prime in a 2D grid
    static int[] findFirstPrime(int[][] grid) {
        GRID_SEARCH:                          // label on outer loop
        for (int row = 0; row < grid.length; row++) {
            for (int col = 0; col < grid[row].length; col++) {
                if (isPrime(grid[row][col])) {
                    System.out.println("Found prime " + grid[row][col]
                        + " at [" + row + "][" + col + "]");
                    break GRID_SEARCH;        // exits BOTH loops immediately
                }
            }
        }
        return new int[]{0, 0};
    }

    static boolean isPrime(int n) {
        if (n < 2) return false;
        for (int i = 2; i * i <= n; i++) {
            if (n % i == 0) return false;
        }
        return true;
    }

    // Labeled continue: skip rest of OUTER iteration
    static void printOddRows(int[][] matrix) {
        OUTER:
        for (int i = 0; i < matrix.length; i++) {
            for (int j = 0; j < matrix[i].length; j++) {
                if (matrix[i][j] % 2 == 0) {
                    continue OUTER;  // Row has even number — skip entire row
                }
            }
            System.out.println("All-odd row: " + java.util.Arrays.toString(matrix[i]));
        }
    }

    public static void main(String[] args) {
        int[][] grid = {{4, 6, 8}, {9, 10, 13}, {15, 17, 19}};
        findFirstPrime(grid);

        int[][] matrix = {{1, 3, 5}, {1, 2, 3}, {7, 9, 11}};
        printOddRows(matrix);  // Prints row 0 and row 2 only
    }
}
```

### Level 4 — Advanced
```java
// Pattern Matching for Switch (Java 21) + Sealed Classes
public class PatternMatchingSwitch {

    // Sealed class hierarchy — compiler knows all subtypes
    sealed interface Shape permits Circle, Rectangle, Triangle {}
    record Circle(double radius)                    implements Shape {}
    record Rectangle(double width, double height)   implements Shape {}
    record Triangle(double base, double height)     implements Shape {}

    // EXHAUSTIVE switch — compiler enforces coverage of all subtypes
    static double area(Shape shape) {
        return switch (shape) {
            case Circle c          -> Math.PI * c.radius() * c.radius();
            case Rectangle r       -> r.width() * r.height();
            case Triangle t        -> 0.5 * t.base() * t.height();
            // No default needed! sealed interface + all cases = exhaustive
        };
    }

    // Guarded patterns with 'when' clause (Java 21)
    static String classify(Object obj) {
        return switch (obj) {
            case Integer i when i < 0    -> "Negative int: " + i;
            case Integer i when i == 0   -> "Zero";
            case Integer i               -> "Positive int: " + i;
            case String s when s.isEmpty() -> "Empty string";
            case String s                -> "String: " + s;
            case null                    -> "null value";
            default                      -> "Unknown: " + obj.getClass().getSimpleName();
        };
    }

    public static void main(String[] args) {
        Shape[] shapes = {
            new Circle(5),
            new Rectangle(4, 6),
            new Triangle(3, 8)
        };

        for (Shape s : shapes) {
            System.out.printf("Area of %s = %.2f%n",
                s.getClass().getSimpleName(), area(s));
        }
        // Area of Circle = 78.54
        // Area of Rectangle = 24.00
        // Area of Triangle = 12.00

        System.out.println(classify(-5));       // Negative int: -5
        System.out.println(classify(0));        // Zero
        System.out.println(classify("hello"));  // String: hello
        System.out.println(classify(""));       // Empty string
        System.out.println(classify(null));     // null value
    }
}
```

### Level 5 — Expert
```java
// ADVANCED: Eliminating control flow with Maps and Strategy pattern
import java.util.*;
import java.util.function.*;

public class EliminateControlFlow {

    // STRATEGY PATTERN: Replace if-else chains with Map<String, Function>
    // Before: 10-branch if-else for tax calculation
    // After: lookup table — O(1), easy to extend without modifying existing code

    interface TaxStrategy {
        double calculate(double income);
    }

    static final Map<String, TaxStrategy> TAX_RULES = Map.of(
        "EMPLOYEE",   income -> income * 0.30,
        "CONTRACTOR", income -> income * 0.35,
        "FREELANCER", income -> income > 100_000 ? income * 0.28 : income * 0.20,
        "STUDENT",    income -> income > 20_000   ? income * 0.10 : 0.0
    );

    static double calculateTax(String employmentType, double income) {
        TaxStrategy strategy = TAX_RULES.get(employmentType);
        if (strategy == null) {
            throw new IllegalArgumentException("Unknown type: " + employmentType);
        }
        return strategy.calculate(income);
    }

    // LOOP OPTIMIZATION: Prefer iteration patterns that the JIT can vectorize
    static long sumArray(int[] arr) {
        long sum = 0;
        // Simple indexed loop — JIT can auto-vectorize with SIMD instructions
        for (int i = 0; i < arr.length; i++) {
            sum += arr[i];
        }
        return sum;
    }

    // ITERATOR WITH REMOVAL — avoids ConcurrentModificationException
    static void removeExpiredSessions(List<String> sessions) {
        Iterator<String> it = sessions.iterator();
        while (it.hasNext()) {
            String session = it.next();
            if (session.startsWith("EXPIRED_")) {
                it.remove();  // Safe removal via Iterator — not sessions.remove()
            }
        }
    }

    // TRAMPOLINE PATTERN — tail-recursive algorithms without StackOverflow
    @FunctionalInterface
    interface Trampoline<T> {
        T get();
        default boolean isDone() { return true; }

        static <T> T run(Trampoline<T> t) {
            while (!t.isDone()) {
                t = (Trampoline<T>) t.get();
            }
            return t.get();
        }
    }

    public static void main(String[] args) {
        // Strategy pattern tax calculation
        System.out.println(calculateTax("EMPLOYEE", 80_000));     // 24000.0
        System.out.println(calculateTax("FREELANCER", 120_000));  // 33600.0

        // Array sum
        int[] data = new int[1_000_000];
        Arrays.fill(data, 1);
        System.out.println(sumArray(data));  // 1000000

        // Safe removal
        List<String> sessions = new ArrayList<>(List.of(
            "ACTIVE_abc", "EXPIRED_xyz", "ACTIVE_def", "EXPIRED_123"
        ));
        removeExpiredSessions(sessions);
        System.out.println(sessions);  // [ACTIVE_abc, ACTIVE_def]
    }
}
```

## REAL_WORLD

### Spring Boot Webhook Router — Eliminating if-else with Map
```java
// ANTI-PATTERN: "God method" — every new event type requires modifying this file
@Service
public class WebhookServiceBad {
    public void handle(String eventType, String payload) {
        if (eventType.equals("USER_CREATED"))        userHandler.handle(payload);
        else if (eventType.equals("USER_DELETED"))   deleteHandler.handle(payload);
        else if (eventType.equals("PAYMENT_FAILED")) paymentHandler.handle(payload);
        // Adding PAYMENT_REFUNDED requires editing this file — violates OCP
    }
}

// GOOD PATTERN: Spring auto-discovers handlers by event type
public interface WebhookHandler {
    String eventType();
    void handle(String payload);
}

@Component
public class UserCreatedHandler implements WebhookHandler {
    public String eventType() { return "USER_CREATED"; }
    public void handle(String payload) { /* ... */ }
}

@Service
public class WebhookRouter {
    private final Map<String, WebhookHandler> handlers;

    // Spring injects all WebhookHandler beans; we index them by eventType
    public WebhookRouter(List<WebhookHandler> handlers) {
        this.handlers = handlers.stream()
            .collect(java.util.stream.Collectors.toMap(
                WebhookHandler::eventType,
                h -> h
            ));
    }

    public void route(String eventType, String payload) {
        WebhookHandler handler = handlers.get(eventType);
        if (handler == null) throw new IllegalArgumentException("No handler for: " + eventType);
        handler.handle(payload);
    }
    // Adding a new event type = add a new @Component. ZERO modification to router.
}
```

### Enum Switch with Exhaustiveness
```java
// Using enum + switch expression for state machine with compiler-checked exhaustiveness
public enum OrderStatus { PENDING, CONFIRMED, SHIPPED, DELIVERED, CANCELLED }

@Service
public class OrderStateMachine {

    public OrderStatus transition(OrderStatus current, String action) {
        return switch (current) {
            case PENDING   -> action.equals("confirm") ? CONFIRMED : CANCELLED;
            case CONFIRMED -> action.equals("ship")    ? SHIPPED   : CONFIRMED;
            case SHIPPED   -> DELIVERED;   // Can only deliver after shipping
            case DELIVERED, CANCELLED -> {
                throw new IllegalStateException("Terminal state: " + current);
                // yield not needed after throw
            }
        };
        // If OrderStatus gains a new value (e.g., REFUNDED),
        // compiler FAILS with "switch expression is not exhaustive"
        // — you MUST handle the new case. Bug found at compile time, not runtime!
    }
}
```

## INTERVIEW

### Q1: What bytecode does the JVM use for switch statements and why is it faster?
**Difficulty:** Hard | **Companies:** Oracle, Netflix

❌ **Weak:** "Switch is optimized so it's faster."

✅ **Strong:** "The JVM compiles switch statements into one of two bytecodes. `tableswitch` is used for dense integer cases (e.g., 1-10) and creates an array of jump addresses — the switch value indexes directly into the array for O(1) dispatch. `lookupswitch` is used for sparse cases and stores sorted key-value pairs searched with binary search for O(log n) dispatch. Both are faster than an `if-else if` chain which sequentially evaluates each condition at O(n)."

---

### Q2: What is the difference between a switch statement and a switch expression in Java?
**Difficulty:** Medium | **Companies:** Amazon, Goldman Sachs

❌ **Weak:** "Switch expression uses arrows and returns values."

✅ **Strong:** "A switch **statement** (pre-Java 14 style) executes side effects but produces no value. It uses `case X:` colon syntax and requires explicit `break` to prevent fall-through. A switch **expression** (Java 14+) evaluates to a value that can be assigned to a variable or returned. It uses `case X ->` arrow syntax with no fall-through possible. When a block is needed, `yield` returns the value from the block. Switch expressions over enums and sealed classes are exhaustiveness-checked by the compiler."

---

### Q3: How do you safely remove items from a List while iterating over it?
**Difficulty:** Medium | **Companies:** Spotify, Airbnb

❌ **Weak:** "Use a for-each loop and call remove."

✅ **Strong:** "You cannot call `list.remove()` inside a for-each loop — it throws `ConcurrentModificationException` because the for-each loop uses an `Iterator` internally, and direct list modification while an iterator is active fails the modification check. The correct approaches are: (1) Use the `Iterator` directly and call `iterator.remove()`. (2) Use `list.removeIf(predicate)` for filter-based removal. (3) Use `list.stream().filter(...).collect(...)` to produce a new list. Never use indexed `for` loop with `remove(int index)` while iterating forward — indices shift."

---

### Q4: What is fall-through in switch statements and when is it intentional?
**Difficulty:** Easy | **Companies:** All

❌ **Weak:** "It's when you forget a break."

✅ **Strong:** "Fall-through means when a switch case matches, execution continues into the next case block even though its case label doesn't match, unless stopped by `break`. It's usually a bug when unintentional. It's intentionally used when multiple cases share the same logic — e.g., `case 'a': case 'e': case 'i':` all fall through to vowel-handling code. However, Java 14+ switch expressions make fall-through physically impossible with arrow syntax, which is why they're preferred in modern code."

---

### Q5: Explain pattern matching for switch in Java 21 and its benefits.
**Difficulty:** Hard | **Companies:** Google, Palantir

❌ **Weak:** "You can use instanceof in switch cases."

✅ **Strong:** "Pattern matching for switch (Java 21, finalized) allows switching on the type of an `Object` reference, automatically binding the matched value to a typed variable. Instead of `if (obj instanceof String s) { use s; } else if (obj instanceof Integer i) { use i; }`, you write `switch (obj) { case String s -> use(s); case Integer i -> use(i); }`. Guarded patterns add `when` conditions for fine-grained matching. Combined with sealed classes, the compiler enforces exhaustiveness — if you add a new subtype and forget to handle it in a switch expression, the code won't compile. This eliminates entire categories of runtime ClassCastExceptions."

## FEYNMAN CHECK

Control flow is a set of decision signs on a road. `if-else` is a series of "turn left if X" signs — you read each sign in order until one applies. `switch` is a roundabout with numbered exits — you go directly to exit N without checking exits 1, 2, 3, ... N-1 first. The switch roundabout is faster when there are many exits. Pattern matching for switch is a smart security scanner — it checks your bag's contents, immediately identifies what type of item is inside, and routes you to the correct security lane — all in one step.
