## WHY
For 25 years, Java used a "1-to-1 Thread Model". Every time you wrote `new Thread().start()`, the JVM requested a massive, heavy, physical OS Thread from the Linux kernel. Because OS threads require 1MB of memory for the stack and require context-switching at the CPU kernel level, a single server maxes out at around 4,000 active threads before violently crashing. 

This created the "Thread-Per-Request Bottleneck". If 10,000 users call your Spring Boot API simultaneously, and your API takes 1 second to query a database, you need 10,000 threads. Since Tomcat limits you to 200 threads, 9,800 users get blocked, even though your CPU is sitting at 2% utilization! 
To solve this, developers were forced to use complex Reactive programming (RxJava, Spring WebFlux) which destroyed stack traces and made code unreadable. 

Java 21 introduced **Virtual Threads**. They completely shatter this bottleneck, allowing you to spawn literally 10 Million threads on a standard laptop, restoring simple, readable, blocking code while achieving infinite scalability.

## THEORY
### M-to-N Thread Model
Virtual Threads are managed entirely by the JVM in user-space, completely bypassing the OS kernel. 
The JVM maintains a tiny pool of physical OS Carrier Threads (equal to your CPU cores, e.g., 8 threads). 
When you spawn 1,000,000 Virtual Threads, the JVM maps them onto those 8 physical Carrier Threads.

### Continuation Passing (The Magic)
How do 1,000,000 Virtual Threads share 8 Physical Threads without freezing?
If Virtual Thread #1 executes an HTTP request or a Database Query, it performs a **Blocking I/O** call. 
The moment it blocks, the JVM physically unmounts the Virtual Thread's memory stack from the Carrier Thread and saves it to the Heap. The Carrier Thread is instantly freed up! The JVM immediately mounts Virtual Thread #2 onto the Carrier Thread and executes it. 
When the Database finally responds 1 second later, the JVM pulls Virtual Thread #1's memory stack from the Heap, mounts it back onto any available Carrier Thread, and resumes execution exactly where it left off.

## CODE

### Level 1 — Beginner (Spawning a Virtual Thread)
```java
public class BasicVirtualThread {
    public static void main(String[] args) throws InterruptedException {
        
        // OLD WAY: Spawns a heavy physical OS thread
        Thread osThread = new Thread(() -> System.out.println("OS Thread"));
        osThread.start();
        
        // NEW WAY (Java 21+): Spawns a lightweight Virtual Thread
        Thread vThread = Thread.ofVirtual().start(() -> {
            System.out.println("Hello from a Virtual Thread!");
        });
        
        // Main thread must wait, because virtual threads are daemon threads by default!
        vThread.join(); 
    }
}
```

### Level 2 — Building Blocks (The 10 Million Thread Test)
```java
import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;

public class MillionThreads {
    public static void main(String[] args) {
        // If you tried to use a CachedThreadPool here, your PC would crash instantly.
        // With Virtual Threads, this loop takes a few seconds and uses almost zero RAM.
        
        try (ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor()) {
            
            for (int i = 0; i < 1_000_000; i++) {
                executor.submit(() -> {
                    try {
                        // The thread sleeps for 1 second. 
                        // Instead of blocking the CPU, the JVM unmounts it!
                        Thread.sleep(1000); 
                    } catch (InterruptedException e) {}
                });
            }
            
            // The try-with-resources block automatically waits for all 1 Million threads to finish!
            System.out.println("1 Million Virtual Threads executed successfully!");
        }
    }
}
```

### Level 3 — Intermediate (Structured Concurrency)
```java
import java.util.concurrent.StructuredTaskScope;
import java.util.concurrent.StructuredTaskScope.Subtask;

public class StructuredConcurrency {
    public static void main(String[] args) throws Exception {
        // Java 21 Preview: Structured Concurrency
        // A new paradigm to manage thousands of virtual threads safely.
        
        try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
            
            // Fork spawns a virtual thread for each task
            Subtask<String> weatherTask = scope.fork(() -> fetchWeatherAPI());
            Subtask<String> newsTask = scope.fork(() -> fetchNewsAPI());
            
            // Wait for both to finish. If ANY task throws an exception, 
            // the scope instantly cancels the other tasks to prevent memory leaks!
            scope.join(); 
            scope.throwIfFailed();
            
            System.out.println(weatherTask.get() + " | " + newsTask.get());
        }
    }
    
    static String fetchWeatherAPI() throws InterruptedException { Thread.sleep(1000); return "Sunny"; }
    static String fetchNewsAPI() throws InterruptedException { Thread.sleep(1000); return "Markets Up"; }
}
```

### Level 4 — Advanced (ThreadLocal Pitfalls)
```java
public class ThreadLocalDanger {
    // ThreadLocals store data specific to the current thread.
    // In the old days with 200 Tomcat threads, this was fine.
    private static final ThreadLocal<byte[]> heavyData = new ThreadLocal<>();
    
    public static void main(String[] args) {
        
        Thread.ofVirtual().start(() -> {
            // DANGER! If you spawn 1 Million virtual threads, 
            // and every single one allocates a 1MB byte array into its ThreadLocal,
            // you will instantly consume 1 Terabyte of RAM and crash the server!
            heavyData.set(new byte[1024 * 1024]); 
        });
        
        // RULE: Never use ThreadLocals with Virtual Threads.
        // Java 21 introduced "Scoped Values" as the safe replacement.
    }
}
```

## REAL_WORLD
Virtual Threads are the biggest revolution in the Java ecosystem since Generics in Java 5.

If you are building a modern Spring Boot 3.2+ application on Java 21, you can enable Virtual Threads by simply adding one line to your `application.properties`:
`spring.threads.virtual.enabled=true`

That's it. Your Tomcat server instantly switches from a rigid pool of 200 physical threads to an infinite pool of Virtual Threads.
If your API makes a slow 2-second call to an external payment gateway, the old Tomcat server would max out at 100 requests per second. With Virtual Threads enabled, the exact same code can now handle 100,000 requests per second. You do not need to change a single line of your `@RestController` code, and you do not need to rewrite your application using complex Reactive frameworks like Spring WebFlux. Virtual Threads give you the insane scalability of Node.js and Go, while keeping the simple, readable, synchronous Java syntax you already know.

## INTERVIEW
**Q: Are Virtual Threads faster than OS Threads?**
A: No! This is a massive misconception. Virtual Threads do not execute code any faster than OS Threads. If you are doing pure CPU-bound work (like hashing passwords or calculating Pi), Virtual Threads offer absolutely zero performance benefit. Their sole purpose is to allow infinite scale during **Blocking I/O** (network calls, database queries, file reads) by allowing the JVM to multiplex millions of waiting tasks onto a handful of physical CPU cores.

**Q: What does "Pinning" mean in the context of Virtual Threads?**
A: Pinning is a critical performance bug. Normally, when a Virtual Thread hits a blocking operation, it unmounts from the Carrier Thread. However, if the Virtual Thread is executing inside a `synchronized` block or a native JNI C++ method when it blocks, the JVM is currently unable to unmount it! The Virtual Thread becomes "Pinned" to the physical Carrier Thread. If all your Carrier Threads get pinned, your entire application completely freezes. (To fix this, replace `synchronized` with `ReentrantLock`).

**Q: Can I use Thread Pools with Virtual Threads?**
A: You should never pool Virtual Threads. The whole point of pooling OS threads is because they are too expensive to create and destroy. Virtual Threads take nanoseconds to create and consume less than 1KB of memory. You should spawn a brand new Virtual Thread for every single task, and let the Garbage Collector destroy it when the task finishes.
