## WHY
When building large systems, you often find overlapping behaviors. A `Dog`, `Cat`, and `Bird` all need an `eat()` method, a `sleep()` method, and a `name` field. Copy-pasting this logic into 50 different animal classes is a maintenance disaster — if the feeding logic changes (e.g., log all meals to a database), you have to find and update 50 separate files with the same fix.

Inheritance solves this by establishing an **IS-A relationship**. By creating a parent `Animal` class with the shared logic, child classes automatically inherit all non-private fields and methods — with zero duplication. Furthermore, inheritance enables **polymorphism**: you can write one method that accepts an `Animal` and it automatically works with `Dog`, `Cat`, and `Bird` objects.

**When to use Inheritance vs Composition:**
- Use Inheritance for true IS-A relationships: `Dog IS-A Animal`, `Admin IS-A User`
- Use Composition for HAS-A relationships: `Car HAS-A Engine`, `UserService HAS-A UserRepository`
- Modern rule: **Prefer Composition over Inheritance** (GoF Design Patterns, Item 18 of Effective Java)

## THEORY

### vtable (Virtual Method Table)
At the JVM level, how does method overriding actually work? The JVM uses a **Virtual Method Table (vtable)**. Every class has a vtable in Metaspace — an array of method pointers. When a child class overrides a method, the child's vtable entry for that method is updated to point to the child's bytecode. At runtime, the JVM looks up the object's actual type vtable to dispatch the call.

```
Animal vtable:            Dog vtable (inherits + overrides):
+------------+           +------------------+
| eat()  --> [Animal] |  | eat()  --> [Dog] |   ← overridden!
| sleep()-> [Animal] |  | sleep()--> [Animal] | ← inherited
+------------+           | bark() --> [Dog] |   ← new method
                         +------------------+
```

### Constructor Chaining with `super()`
When you instantiate a child object (`new Dog()`), the JVM must construct the **parent part first**. Every constructor's first statement is implicitly `super()`. This chain propagates all the way to `java.lang.Object`. If you don't explicitly call `super(...)`, the compiler inserts a no-arg `super()` call — which fails if the parent has no no-arg constructor.

```
new Dog()
  → Dog() constructor starts
    → super() → Animal() constructor starts
      → super() → Object() constructor starts → Object header created
      → Animal fields initialized
    → Dog fields initialized
  → Dog object fully created
```

### Method Overriding Rules
| Rule | Explanation |
|------|-------------|
| `@Override` annotation | Compiler verifies you're actually overriding (not accidentally overloading) |
| Return type | Must be the same OR a subtype (covariant return type) |
| Access modifier | Cannot be more restrictive than parent (can widen access) |
| Exception | Cannot throw broader checked exceptions than parent |
| `private` methods | Cannot be overridden — they're invisible to child classes |
| `static` methods | Cannot be overridden — they are "hidden", not overridden |
| `final` methods | Cannot be overridden at all |

### Method Hiding vs Overriding
This is a critical distinction:
- **Overriding** (instance methods): The actual object type determines which version runs at runtime (vtable dispatch).
- **Hiding** (static methods): The reference variable type determines which version runs at compile time.

```java
Parent p = new Child();
p.instanceMethod(); // Calls Child version — OVERRIDING (runtime dispatch)
p.staticMethod();   // Calls Parent version — HIDING (compile-time binding)
```

### The Diamond Problem — Why Java Forbids Multiple Class Inheritance
If Java allowed `class C extends A, B`, and both A and B had a `doSomething()` method, the JVM couldn't know which version C's vtable should point to. Java solves this by:
1. Classes can only extend **one** parent class
2. Interfaces (not classes) can be implemented multiple times safely

### Inheritance Access Modifiers
| Modifier | Same Class | Same Package | Subclass | Other |
|----------|-----------|--------------|----------|-------|
| `private` | ✅ | ❌ | ❌ | ❌ |
| package | ✅ | ✅ | ❌ | ❌ |
| `protected` | ✅ | ✅ | ✅ | ❌ |
| `public` | ✅ | ✅ | ✅ | ✅ |

A child class **cannot** access the parent's `private` fields directly — it must use the parent's `protected` or `public` getters.

## CODE

### Level 1 — Beginner: Basic Inheritance and Method Overriding
```java
// Parent class
class Animal {
    String name;

    Animal(String name) {
        this.name = name;
    }

    public void eat() {
        System.out.println(name + " is eating...");
    }

    public void sleep() {
        System.out.println(name + " is sleeping...");
    }

    @Override
    public String toString() {
        return "Animal{name=" + name + "}";
    }
}

// Dog IS-A Animal
class Dog extends Animal {
    String breed;

    Dog(String name, String breed) {
        super(name);  // Must call parent constructor first!
        this.breed = breed;
    }

    // Override parent method
    @Override
    public void eat() {
        System.out.println(name + " (" + breed + ") gulps food enthusiastically!");
    }

    // New method — only exists in Dog
    public void bark() {
        System.out.println(name + " says: Woof!");
    }
}

// Cat IS-A Animal
class Cat extends Animal {
    Cat(String name) { super(name); }

    @Override
    public void eat() {
        System.out.println(name + " nibbles delicately...");
    }

    public void purr() {
        System.out.println(name + " purrs...");
    }
}

public class Main {
    public static void main(String[] args) {
        Dog dog = new Dog("Rex", "German Shepherd");
        Cat cat = new Cat("Whiskers");

        dog.eat();   // Dog's override: "Rex (German Shepherd) gulps food enthusiastically!"
        dog.sleep(); // Inherited: "Rex is sleeping..."
        dog.bark();  // Dog-specific: "Rex says: Woof!"

        cat.eat();   // Cat's override: "Whiskers nibbles delicately..."
        cat.purr();  // Cat-specific: "Whiskers purrs..."
    }
}
```

### Level 2 — Building Blocks: `super()`, Constructor Chaining
```java
class Vehicle {
    protected String brand;
    protected int year;
    protected double price;

    public Vehicle(String brand, int year, double price) {
        this.brand = brand;
        this.year = year;
        this.price = price;
        System.out.println("Vehicle created: " + brand);
    }

    public double calculateInsurance() {
        return price * 0.05; // 5% of price
    }

    public void display() {
        System.out.printf("Brand: %s | Year: %d | Price: $%.2f%n", brand, year, price);
    }
}

class Car extends Vehicle {
    private int numDoors;
    private String fuelType;

    public Car(String brand, int year, double price, int doors, String fuelType) {
        super(brand, year, price);  // Parent constructor runs first
        this.numDoors = doors;
        this.fuelType = fuelType;
        System.out.println("Car created with " + doors + " doors");
    }

    // Override — Car insurance is higher
    @Override
    public double calculateInsurance() {
        double base = super.calculateInsurance(); // Call parent's version
        return "Electric".equals(fuelType) ? base * 0.8 : base * 1.2; // 20% EV discount
    }

    @Override
    public void display() {
        super.display(); // Show parent info
        System.out.printf("Doors: %d | Fuel: %s | Insurance: $%.2f%n",
                          numDoors, fuelType, calculateInsurance());
    }
}

class ElectricCar extends Car {
    private int batteryRangeKm;

    public ElectricCar(String brand, int year, double price, int batteryRange) {
        super(brand, year, price, 4, "Electric"); // Chain through Car → Vehicle
        this.batteryRangeKm = batteryRange;
    }

    @Override
    public void display() {
        super.display();
        System.out.println("Battery Range: " + batteryRangeKm + " km");
    }
}

public class VehicleDemo {
    public static void main(String[] args) {
        ElectricCar tesla = new ElectricCar("Tesla Model 3", 2024, 45000, 560);
        // Prints: "Vehicle created: Tesla Model 3", "Car created with 4 doors"
        tesla.display();
    }
}
```

### Level 3 — Intermediate: `final`, Abstract, and Overriding Rules
```java
// Abstract parent — forces children to implement critical methods
abstract class Shape {
    private String color;

    public Shape(String color) {
        this.color = color;
    }

    // Abstract — MUST be overridden by non-abstract children
    public abstract double area();
    public abstract double perimeter();

    // Template method — defines algorithm, child fills in the blanks
    public final void printDetails() {  // final — cannot be overridden
        System.out.printf("[%s %s] Area=%.2f, Perimeter=%.2f%n",
                          color, getClass().getSimpleName(), area(), perimeter());
    }

    public String getColor() { return color; }
}

class Circle extends Shape {
    private final double radius;

    public Circle(String color, double radius) {
        super(color);
        this.radius = radius;
    }

    @Override
    public double area() { return Math.PI * radius * radius; }

    @Override
    public double perimeter() { return 2 * Math.PI * radius; }
}

class Rectangle extends Shape {
    private final double width, height;

    public Rectangle(String color, double width, double height) {
        super(color);
        this.width = width;
        this.height = height;
    }

    @Override
    public double area() { return width * height; }

    @Override
    public double perimeter() { return 2 * (width + height); }
}

public class ShapeDemo {
    public static void main(String[] args) {
        Shape[] shapes = {
            new Circle("Red", 5.0),
            new Rectangle("Blue", 4.0, 6.0)
        };

        double totalArea = 0;
        for (Shape s : shapes) {
            s.printDetails();          // Template Method called
            totalArea += s.area();     // Polymorphic dispatch via vtable
        }
        System.out.printf("Total area: %.2f%n", totalArea);
    }
}
```

### Level 4 — Advanced: Method Hiding, Covariant Return, and `instanceof`
```java
class Base {
    // Static method — can be "hidden" not "overridden"
    public static String type() { return "Base"; }

    // Instance method — can be overridden
    public Object create() { return new Object(); }
}

class Child extends Base {
    // This HIDES the parent static method — NOT polymorphic
    public static String type() { return "Child"; }

    // Covariant return type — Child can return a more specific type
    @Override
    public String create() { return "child object"; } // String IS-A Object
}

public class AdvancedInheritance {
    public static void main(String[] args) {
        // Method Hiding demonstration
        Base b = new Child();
        System.out.println(b.type());     // "Base" — static dispatch on reference type!
        System.out.println(Child.type()); // "Child"

        // Covariant return
        Base ref = new Child();
        Object result = ref.create();     // Returns "child object" (String)

        // instanceof checks
        System.out.println(result instanceof String);  // true
        System.out.println(result instanceof Object);  // true (all objects are Objects)

        // Java 16+ Pattern Matching
        if (b instanceof Child c) {
            System.out.println("Confirmed Child: " + c.create());
        }

        // Safe downcasting
        Base base = new Child();
        if (base instanceof Child child) {
            child.create(); // No ClassCastException risk
        }
    }
}
```

### Level 5 — Expert: Fragile Base Class and the Liskov Substitution Principle
```java
// Fragile Base Class Problem: Parent changes break children
class Stack<T> extends java.util.ArrayList<T> {
    // TERRIBLE design — extends ArrayList just for reuse
    // Problem: inherited remove(), clear(), addAll() bypass our push counter!

    private int pushCount = 0;

    public T push(T item) {
        add(item);
        pushCount++;
        return item;
    }

    public T pop() {
        if (isEmpty()) throw new java.util.EmptyStackException();
        return remove(size() - 1);
    }

    public int getPushCount() { return pushCount; }

    // BUG: stack.addAll(list) bypasses pushCount!
    // BUG: stack.remove(0) directly removes without our logic!
}

// Liskov Substitution Principle (LSP): Child must be substitutable for parent
// LSP VIOLATION — Square extends Rectangle
class Rectangle2 {
    protected int width, height;

    public void setWidth(int w)  { this.width = w; }
    public void setHeight(int h) { this.height = h; }
    public int area()            { return width * height; }
}

class Square extends Rectangle2 {
    @Override
    public void setWidth(int w)  { width = height = w; }  // Forces equal sides
    @Override
    public void setHeight(int h) { width = height = h; }  // Forces equal sides
}

class LspTest {
    static void testRectangle(Rectangle2 r) {
        r.setWidth(5);
        r.setHeight(4);
        // Any caller expects: area == 20 for a Rectangle
        assert r.area() == 20 : "Expected 20, got " + r.area();
    }

    public static void main(String[] args) {
        testRectangle(new Rectangle2()); // Passes
        testRectangle(new Square());     // FAILS — area = 16, not 20!
        // Square violates LSP: it cannot safely substitute for Rectangle
    }
}
```

## REAL_WORLD

### Spring Boot: Avoiding Deep Inheritance Trees
In enterprise Java, deep inheritance trees are considered an **Anti-Pattern** (the "Fragile Base Class" problem). Modern Spring Boot uses **Composition over Inheritance**:

```java
// ANTI-PATTERN: Deep hierarchy
class BaseService { ... }
class DatabaseService extends BaseService { ... }
class UserDatabaseService extends DatabaseService { ... }
class AdminUserDatabaseService extends UserDatabaseService { ... }
// Changing BaseService breaks ALL four classes

// MODERN PATTERN: Composition
@Service
public class UserService {
    private final UserRepository userRepo;        // Inject, don't extend
    private final AuditLogger auditLogger;         // Inject, don't extend
    private final PasswordEncoder passwordEncoder; // Inject, don't extend

    public UserService(UserRepository r, AuditLogger l, PasswordEncoder e) {
        this.userRepo = r;
        this.auditLogger = l;
        this.passwordEncoder = e;
    }
}
```

### JPA Entity Inheritance Strategies
When you DO need inheritance in JPA entities, Hibernate provides four strategies:

```java
// SINGLE_TABLE: All subclasses in one table (fast, but nullable columns)
@Entity
@Inheritance(strategy = InheritanceType.SINGLE_TABLE)
@DiscriminatorColumn(name = "user_type")
public abstract class User {
    @Id @GeneratedValue private Long id;
    private String email;
}

@Entity
@DiscriminatorValue("ADMIN")
public class AdminUser extends User {
    private String adminLevel;
}

@Entity
@DiscriminatorValue("CUSTOMER")
public class CustomerUser extends User {
    private String loyaltyCard;
}
```

## INTERVIEW

**Q: Why doesn't Java support multiple class inheritance?**
A: To avoid the Diamond Problem. If class C extended both A and B, and both had a `doSomething()` method with different implementations, the JVM's vtable for C would have conflicting entries — it wouldn't know which method pointer to store. Java solves this by allowing multiple interface implementation (which historically had no bodies) and single class extension.

**Q: What is the purpose of the `super` keyword?**
A: `super` accesses the direct parent class. `super()` invokes the parent's constructor (must be the first statement in a child constructor). `super.methodName()` calls the parent's version of an overridden method, allowing a child to extend rather than completely replace the parent's behavior.

**Q: What is the difference between method overriding and method hiding?**
A: Method overriding applies to **instance methods** — the actual runtime object type determines which version is called (dynamic dispatch via vtable). Method hiding applies to **static methods** — the compile-time reference type determines which version is called (static binding). Hiding is NOT polymorphic.

**Q: What is the Fragile Base Class problem?**
A: When a parent class changes its internal implementation, it can unexpectedly break all subclasses that relied on that behavior. Example: If you subclass `ArrayList` to count element additions, and `addAll()` internally calls `add()`, your override of `add()` accidentally double-counts when `addAll()` is used. This is why Effective Java Item 18 recommends preferring composition over inheritance.

**Q: What is the Liskov Substitution Principle?**
A: Objects of a subclass should be usable anywhere the parent class is expected, without breaking the program's correctness. A `Square extends Rectangle` is a textbook LSP violation — setting width independently of height (a contract of Rectangle) is impossible for Square, breaking any code that assumes `setWidth(5); setHeight(4)` produces `area() == 20`.

**Q: Can a constructor be inherited?**
A: No. Constructors are not inherited. A child class must define its own constructors. The parent's constructor is invoked via `super()` but is not itself inherited — you cannot call `new Dog()` and expect Dog to automatically have all of Animal's constructor signatures available.
