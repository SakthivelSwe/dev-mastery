# React — Virtual DOM, Reconciliation, JSX

## The Core Problem: The Real DOM is Slow and Imperative

In traditional web development, dynamic UI updates are achieved through direct manipulation of the browser's Document Object Model (DOM) using imperative APIs:

```typescript
// Imperative DOM manipulation: Fragile, error-prone, and slow
const container = document.getElementById('user-list');
if (container) {
  const item = document.createElement('div');
  item.className = 'user-item';
  item.textContent = 'Alice';
  container.appendChild(item);
}
```

This approach suffers from two fundamental architectural flaws:
1. **Inefficient Layout Engines**: Browsers must recalculate styles, compute layouts (reflow), and repaint the screen whenever the DOM changes. Uncoordinated DOM writes trigger multiple synchronous reflows, causing severe performance bottlenecks.
2. **State-UI Desynchronization**: As application state scales, manually tracking which DOM node corresponds to which state variable becomes mathematically intractable, leading to bugs where the UI diverges from the source of truth.

React solves this by treating the UI as a pure, declarative function of state: $UI = f(State)$. Developers describe *what* the UI should look like at any point in time, and React handles the *how*—optimally updating the browser DOM to match.

## The Virtual DOM: A Lightweight Blueprint

The **Virtual DOM (VDOM)** is a tree of plain JavaScript objects that mirrors the real DOM. It is lightweight because it does not contain the browser-specific layout, styling, and event-handling machinery of native DOM nodes.

A virtual node is represented as a simple object:

```typescript
interface VirtualNode {
  type: string | Function;
  props: {
    className?: string;
    children?: VirtualNode[] | string | null;
    [key: string]: any;
  };
  key: string | number | null;
}
```

Because allocating and garbage-collecting plain JavaScript objects is highly optimized in modern V8/JS engines, React can destroy and recreate this virtual tree on every state change without incurring layout or paint costs.

## Reconciliation: The $O(n)$ Diffing Heuristic

When a component's state changes, React generates a new Virtual DOM tree. Comparing two arbitrary trees has a theoretical time complexity of $O(n^3)$ (using algorithms like tree-to-tree edit distance). For 1,000 elements, this would require 1 billion operations—unusable for interactive 60fps UIs.

React implements a highly optimized **Reconciliation Algorithm** based on two heuristics, reducing the complexity to $O(n)$:

### Heuristic 1: Different Element Types Produce Different Trees
If two elements have different types (e.g., changing from `<div>` to `<span>`, or from `<ComponentA>` to `<ComponentB>`), React will tear down the old tree completely, destroy its state, and build the new tree from scratch.

```
Old Tree:               New Tree:
   <div>                   <span>
     <Counter />             <Counter />  <-- Destroyed & Re-mounted (State Reset)
   </div>                  </span>
```

### Heuristic 2: Same Element Type, Update Attributes
If two elements have the same type, React compares their properties (props and attributes), updates only the changed values, and recurses on their children.

```html
<!-- Old -->
<div className="active" style={{ color: 'red' }} />

<!-- New -->
<div className="inactive" style={{ color: 'red' }} />
<!-- React only runs: element.className = 'inactive'; -->
```

## Fiber Architecture: Work Units and Time Slicing

Before React 16, the reconciler (Stack Reconciler) traversed the tree recursively. This execution could not be interrupted. If a large render took 100ms, the main thread blocked, causing dropped frames and unresponsive inputs (jank).

**React Fiber** is a complete rewrite of the core reconciliation engine. It represents a virtual call stack where each element in the tree is a **Fiber Node**:

```typescript
interface FiberNode {
  type: any;
  key: null | string;
  stateNode: any; // Reference to host DOM node or component instance
  
  // Singly-linked list tree structure
  child: FiberNode | null;
  sibling: FiberNode | null;
  return: FiberNode | null; // Parent fiber
  
  // Work-in-progress properties
  pendingProps: any;
  memoizedProps: any;
  memoizedState: any;
  
  // Concurrency & Updates
  lanes: number; // Priority bitmask
  alternate: FiberNode | null; // Double-buffering pool link
}
```

### Key Capabilities of Fiber:
* **Time Slicing**: React breaks rendering work into small units (Fiber nodes). During a render, React periodically yields control back to the browser's main thread (using scheduler mechanisms like `requestIdleCallback` or microtask queues) to process high-priority user inputs (clicks, typing).
* **Double Buffering**: React maintains two trees: the `current` tree (visible on screen) and the `workInProgress` tree (constructed asynchronously in memory). Once the `workInProgress` tree is fully computed, React swaps the pointers instantly, preventing partial, broken UI states from rendering.
* **Phases of Fiber**:
  1. **Render Phase (Asynchronous/Interruptible)**: React walks the fiber tree, computes changes, and tags nodes with flags (e.g., `Placement`, `Update`, `Deletion`). This phase can be paused, discarded, or restarted.
  2. **Commit Phase (Synchronous/Uninterruptible)**: React applies all computed changes to the actual DOM in a single, blocking pass to ensure visual consistency.

## JSX: Syntactic Sugar Under the Hood

JSX is not valid JavaScript. It is a declarative syntax extension that compiles down to standard JavaScript function calls.

### Before React 17 (Classic Runtime):
JSX compiled to `React.createElement()` calls:

```typescript
// JSX Source
const element = (
  <div className="container">
    <h1 id="title">Hello</h1>
  </div>
);

// Transpiled Output
const element = React.createElement(
  'div',
  { className: 'container' },
  React.createElement('h1', { id: 'title' }, 'Hello')
);
```

### React 17+ (New JSX Runtime):
Compilers like Babel or SWC automatically import special entry points from the React package. You no longer need to import `React` explicitly just to use JSX.

```typescript
// Transpiled Output (React 17+)
import { jsx as _jsx } from 'react/jsx-runtime';

const element = _jsx('div', {
  className: 'container',
  children: _jsx('h1', { id: 'title', children: 'Hello' }),
});
```

## Why the `key` Prop Matters

When rendering lists of elements, React needs to match the children in the old tree with the children in the new tree. Without keys, React compares children index-by-index.

If you insert an element at the beginning of a list without keys, React will re-render every subsequent element because their positions (indexes) have shifted, destroying local component state and causing massive performance degradation.

```typescript
interface User {
  id: string;
  name: string;
}

interface UserListProps {
  users: User[];
}

export const UserList: React.FC<UserListProps> = ({ users }) => {
  return (
    <ul>
      {users.map((user) => (
        // The key must be a stable, unique identifier (e.g., database ID)
        <li key={user.id} className="user-item">
          {user.name}
        </li>
      ))}
    </ul>
  );
};
```

### Reconciler Behavior with Keys:
1. **With Key**: React maps elements by key. If an item moves from index 0 to index 2, React identifies that the key is identical, retains the existing DOM node and component state, and simply moves the physical DOM node in the layout.
2. **Without Key (or using Array Index)**: React sees that the element at index 0 has changed its properties, mutates the DOM at index 0, mutates the DOM at index 1, and so on. This forces unnecessary updates and breaks stateful inputs.
