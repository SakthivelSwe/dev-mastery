## WHY
Sorting a `List<Integer>` is easy. The JVM inherently knows that `5` is smaller than `10`.
But what happens if you have a `List<User>` and you call `Collections.sort(users)`? How does the JVM know which User is "smaller"? Should it sort by age? By last name? By account balance? 
It doesn't know, so the compiler will violently reject the code.

To sort custom objects, you must teach the JVM how to compare them. Java provides two distinct interfaces for this: `Comparable` (defining the default, "natural" sorting order baked directly into the class) and `Comparator` (external sorting rules that can be swapped dynamically on the fly). Understanding the mathematical contracts of these interfaces is crucial to prevent silent sorting corruption.

## THEORY
### The Mathematical Contract
Both interfaces rely on a single method that returns an integer (`int`).
When comparing Object A and Object B:
- **Returns negative `< 0`**: Object A is smaller (comes before B).
- **Returns `0`**: Object A and B are equal.
- **Returns positive `> 0`**: Object A is larger (comes after B).

### `Comparable` (Natural Ordering)
`java.lang.Comparable<T>` is implemented directly *on* the class itself (e.g., `class User implements Comparable<User>`). It forces you to override the `compareTo(T other)` method. This defines the one true, default sorting behavior for that class.

### `Comparator` (Custom Ordering)
`java.util.Comparator<T>` is an external interface used to create separate sorting rules. If a `User` class naturally sorts by ID, but the UI needs to display them sorted by Age, you can pass a custom `Comparator` to the sort method, overriding the natural order without modifying the `User` class itself. Since Java 8, `Comparator` is a Functional Interface, meaning you can pass lambdas directly into it!

## CODE

### Level 1 — Beginner
```java
// 1. Implement Comparable to define the Natural Order
class Player implements Comparable<Player> {
    String name;
    int score;
    
    Player(String n, int s) { name = n; score = s; }

    @Override
    public int compareTo(Player other) {
        // We want to sort by score ASCENDING
        // Integer wrapper has a built-in compare method!
        return Integer.compare(this.score, other.score);
    }
}

public class BasicComparable {
    public static void main(String[] args) {
        List<Player> list = new ArrayList<>(List.of(
            new Player("Alice", 50), new Player("Bob", 10)
        ));
        
        // This works automatically because Player implements Comparable!
        Collections.sort(list); 
    }
}
```

### Level 2 — Building Blocks
```java
import java.util.Comparator;
import java.util.List;
import java.util.ArrayList;

public class BasicComparator {
    public static void main(String[] args) {
        List<Player> list = new ArrayList<>(List.of(
            new Player("Alice", 50), new Player("Bob", 10)
        ));
        
        // 2. What if we want to sort by Name instead of Score?
        // We pass an external Comparator lambda!
        list.sort((p1, p2) -> p1.name.compareTo(p2.name));
        
        System.out.println(list.get(0).name); // Alice
    }
}
```

### Level 3 — Intermediate
```java
import java.util.Comparator;

public class Java8Comparators {
    public static void main(String[] args) {
        // Java 8 introduced beautiful static helper methods on the Comparator interface
        // This eliminates the need to write the mathematical (p1, p2) lambda manually!
        
        Comparator<Player> byName = Comparator.comparing(p -> p.name);
        
        // Reverse order (Descending)
        Comparator<Player> byScoreDesc = Comparator.comparingInt(p -> p.score).reversed();
    }
}
```

### Level 4 — Advanced
```java
import java.util.Comparator;
import java.util.List;

public class ChainedComparators {
    public static void main(String[] args) {
        List<Player> roster = // ...
        
        // What if two players have the exact same score?
        // We can CHAIN comparators together!
```

## REAL_WORLD
A legendary bug occurs when developers try to manually calculate the `int` return value using basic subtraction.

```java
// DANGEROUS ANTI-PATTERN!
@Override
public int compareTo(Player other) {
    // This looks correct: if my score is 10 and theirs is 5, it returns 5 (Positive).
    return this.score - other.score; 
}
```
Why is this a bug? **Integer Overflow**. 
If `this.score` is `2,000,000,000` and `other.score` is `-1,000,000,000`, subtracting them results in `3,000,000,000`. This exceeds the maximum 32-bit Integer limit (`2.14B`). The number overflows to a *negative* value. The sorting algorithm receives a negative number, thinks `2 Billion` is smaller than `-1 Billion`, and corrupts the entire sorted list!

To prevent this, you should NEVER use subtraction. Always use the safe helper methods: `Integer.compare(a, b)` or `Double.compare(a, b)`.

## INTERVIEW
**Q: What is the difference between `Comparable` and `Comparator`?**
A: `Comparable` is implemented on the domain object itself (`implements Comparable`) and defines its single, natural default sorting order via the `compareTo()` method. `Comparator` is an external interface used to define alternate sorting rules on the fly without modifying the domain class. 

**Q: Can you pass a class that doesn't implement `Comparable` into a `TreeSet`?**
A: If you use the empty constructor (`new TreeSet<>()`), it will compile, but it will instantly crash with a `ClassCastException` at runtime when it tries to insert the first element. To fix this, you must either make the class implement `Comparable`, OR you must pass a custom `Comparator` into the TreeSet constructor.

**Q: Why does the `compare` method return an `int` instead of a `boolean`?**
A: A boolean can only represent two states (True/False). Sorting requires three states: Less Than, Equal To, and Greater Than. Returning a negative, zero, or positive integer mathematically satisfies these three states in a single CPU instruction.
