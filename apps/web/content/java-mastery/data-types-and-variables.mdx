## WHY

In programming, memory is finite. If you want to store a number, you need to tell the computer exactly how much RAM to reserve for it. A boolean (true/false) might only need a single bit, while a high-precision decimal might need 64 bits.

Before high-level languages, programmers had to manually manage memory allocation, tracking exact byte offsets and sizes. This was error-prone and often led to catastrophic bugs where data would "overflow" into adjacent memory blocks.

Java solves this by being a **statically typed** language. When you declare a variable, you MUST explicitly state its type (e.g., `int x`). The Java compiler (`javac`) uses this type information to allocate the exact amount of memory needed and to enforce strict rules—preventing you from accidentally multiplying a string by a boolean. This strictness catches thousands of errors at compile-time, long before the code reaches a production environment. 

Understanding how Java stores these types differently (primitives on the Stack vs. Objects on the Heap) is the first step toward writing memory-efficient, bug-free applications.

## THEORY

Java divides its data types into two fundamental categories: **Primitives** and **Reference Types (Objects)**. Understanding the distinction is arguably the most important concept in Java memory management.

### 1. Primitive Types (The Basic 8)
Primitives are the simplest data types. They hold pure values, not objects. When you declare a primitive, the JVM allocates memory for it directly on the **Thread Stack** (the fast, per-thread memory area).

*   `byte`: 8-bit signed integer (-128 to 127)
*   `short`: 16-bit signed integer (-32,768 to 32,767)
*   `int`: 32-bit signed integer (default for numbers, ~ -2B to 2B)
*   `long`: 64-bit signed integer (use `L` suffix, e.g., `1000L`)
*   `float`: 32-bit floating point (use `f` suffix, e.g., `3.14f`)
*   `double`: 64-bit floating point (default for decimals, e.g., `3.14`)
*   `char`: 16-bit Unicode character (use single quotes, e.g., `'A'`)
*   `boolean`: `true` or `false` (size is JVM dependent, often 1 byte or 4 bytes for arrays)

### 2. Reference Types (Objects)
Everything that is not a primitive is a reference type (e.g., `String`, `Scanner`, custom classes, and *all arrays*). 
When you write `String name = new String("Alice");`, two things happen:
1.  The JVM allocates memory for the actual string object `"Alice"` on the **Heap** (the large, shared memory area).
2.  The JVM allocates memory for the variable `name` on the **Stack**. This variable does not hold the text "Alice"; it holds the memory address (reference/pointer) pointing to the object on the Heap.

### The Integer Cache & Autoboxing
Java provides wrapper classes for primitives (e.g., `Integer` for `int`). Converting between them automatically is called **autoboxing** (primitive to object) and **unboxing** (object to primitive).

Here is the famous Java trap: To save memory, the JVM caches `Integer` objects for values between **-128 and 127**. 
If you write `Integer a = 100; Integer b = 100;`, both references point to the *exact same cached object* in memory. Therefore, `a == b` is `true`.
However, if you write `Integer x = 200; Integer y = 200;`, they fall outside the cache. Two new distinct objects are created. Therefore, `x == y` is `false` (because `==` checks if they are the exact same memory address, not if they hold the same value). You must use `x.equals(y)`!

### `var` Keyword (Type Inference)
Since Java 10, you can use `var` for local variables. 
`var count = 10;` is exactly the same as `int count = 10;`. The compiler infers the type at compile-time. Java remains statically typed; `count` is locked as an `int` forever.

## CODE

### Level 1 — Beginner
```java
public class VariablesIntro {
    public static void main(String[] args) {
        // Declaring and initializing primitive variables
        int age = 25;           // 32-bit integer
        double price = 19.99;   // 64-bit decimal
        boolean isActive = true; // true/false flag
        char grade = 'A';       // Single character in single quotes
        
        // Declaring a Reference type (String)
        String name = "DevMastery"; // Text in double quotes
        
        System.out.println(name + " costs " + price);
    }
}
// Expected Output: DevMastery costs 19.99
```

### Level 2 — Building Blocks
```java
public class TypeConversion {
    public static void main(String[] args) {
        // Implicit casting (Widening): smaller type fits into larger type safely
        int smallNumber = 100;
        long largeNumber = smallNumber; // int (32-bit) naturally fits in long (64-bit)
        
        // Explicit casting (Narrowing): forcing larger type into smaller type (risky!)
        double preciseValue = 9.99;
        int truncatedValue = (int) preciseValue; // Drops the .99 completely!
        
        System.out.println("Original: " + preciseValue);
        System.out.println("Truncated: " + truncatedValue);
    }
}
// Expected Output:
// Original: 9.99
// Truncated: 9
```

### Level 3 — Intermediate
```java
public class AutoboxingTrap {
    public static void main(String[] args) {
        // Autoboxing: primitive 'int' is automatically wrapped into an 'Integer' object
        Integer val1 = 100;
        Integer val2 = 100;
        
        // Output is true because values between -128 and 127 are cached by the JVM!
        System.out.println("100 == 100? " + (val1 == val2)); 
        
        Integer val3 = 1000;
        Integer val4 = 1000;
        
        // Output is FALSE! 1000 is outside the cache, so two distinct objects are created on the Heap.
        // The '==' operator checks for memory reference equality, not value equality.
        System.out.println("1000 == 1000? " + (val3 == val4)); 
        
        // Correct way to compare objects:
        System.out.println("1000.equals(1000)? " + val3.equals(val4));
    }
}
// Expected Output:
// 100 == 100? true
// 1000 == 1000? false
// 1000.equals(1000)? true
```

### Level 4 — Advanced
```java
import java.math.BigDecimal;

public class FinancialMath {
    public static void main(String[] args) {
        // The danger of floating point arithmetic
        double a = 0.1;
        double b = 0.2;
        System.out.println("0.1 + 0.2 (double) = " + (a + b)); 
        // Output: 0.30000000000000004 (Base-2 floats cannot accurately represent Base-10 decimals!)

        // ALWAYS use BigDecimal for currency or financial calculations
        BigDecimal decA = new BigDecimal("0.1"); // Always use the String constructor!
        BigDecimal decB = new BigDecimal("0.2");
        BigDecimal result = decA.add(decB);
        
        System.out.println("0.1 + 0.2 (BigDecimal) = " + result); 
    }
}
// Expected Output:
// 0.1 + 0.2 (double) = 0.30000000000000004
// 0.1 + 0.2 (BigDecimal) = 0.3
```

### Level 5 — Expert
```java
public class ValueTypesPreview {
    public static void main(String[] args) {
        // In highly optimized systems, object overhead is a problem.
        // An Integer object takes up to 16 bytes of metadata overhead just to store a 4-byte int!
        
        // Project Valhalla (upcoming in future Java) introduces "Value Classes"
        // which behave like objects but allocate directly on the stack/inline in arrays like primitives.
        
        // Until then, if you have an array of 1,000,000 Integers:
        Integer[] objArray = new Integer[1000000]; 
        // This creates 1 array object AND 1,000,000 distinct Integer objects on the Heap. Terribly inefficient.
        
        // Instead, use primitive arrays for high performance:
        int[] primArray = new int[1000000]; 
        // This allocates exactly 4,000,000 bytes (plus a small array header) in one contiguous block of memory.
        
        System.out.println("Primitive array initialized efficiently.");
    }
}
// Expected Output: Primitive array initialized efficiently.
```

## REAL_WORLD

You will encounter autoboxing performance traps frequently in production environments, particularly when working with Java Collections or ORM frameworks like Hibernate/Spring Data JPA.

For example, `List<int>` is illegal in Java; generics only work with objects, so you must use `List<Integer>`. 

Consider this real-world anti-pattern in a Spring Boot service calculating total order revenue:

```java
public Integer calculateTotalRevenue(List<Order> orders) {
    Integer sum = 0; // The trap is here
    for (Order order : orders) {
        sum += order.getAmount(); 
    }
    return sum;
}
```

Because `sum` is an `Integer` object, and objects are immutable, the `+=` operator is actually doing this under the hood:
`sum = Integer.valueOf(sum.intValue() + order.getAmount());`

If there are 1,000,000 orders, this loop creates **one million temporary `Integer` objects on the heap**, triggering massive Garbage Collection pauses. The fix is simple: use a primitive `int sum = 0;` for the accumulator and only box it when returning.

Furthermore, when designing API responses using Spring `@RestController`, always use Wrapper types (like `Boolean`, `Integer`) for optional fields, and primitives (`boolean`, `int`) for required fields. 
If an incoming JSON omits an `int` field, Jackson defaults it to `0`. If it omits an `Integer` field, it defaults to `null`. Using `null` correctly allows you to distinguish between "the user set this to 0" and "the user didn't send this field at all."

## INTERVIEW

### Q1: What is the difference between primitive types and wrapper classes?
**Difficulty:** Easy | **Companies:** All

❌ **Weak answer:** "Primitives are basic numbers and wrappers are objects for those numbers."

✅ **Strong answer:** 
"Primitives (like `int`, `boolean`) hold pure values and are allocated on the thread stack, meaning they are incredibly fast and memory-efficient. Wrapper classes (like `Integer`, `Boolean`) are full Java objects allocated on the heap. They contain metadata overhead and are passed by reference value. You must use Wrapper classes when dealing with Java Collections (like `ArrayList`) or when a field needs to represent a `null` state (e.g., in a database entity)."

### Q2: Why shouldn't you use `double` for currency?
**Difficulty:** Medium | **Companies:** JP Morgan, Stripe, FinTechs

❌ **Weak answer:** "Because double isn't big enough."

✅ **Strong answer:** 
"Because `double` and `float` use base-2 (binary) floating-point representation (IEEE 754 standard). Many base-10 fractions (like 0.1) cannot be represented precisely in base-2, leading to rounding errors (e.g., `0.1 + 0.2` evaluates to `0.30000000000000004`). In financial applications where penny accuracy is legally required, you must use `java.math.BigDecimal` (specifically instantiated via its `String` constructor) or store the currency as an `int` or `long` representing the smallest unit (e.g., storing cents instead of dollars)."

### Q3: Explain the Integer cache in Java.
**Difficulty:** Hard | **Companies:** Amazon, Google

❌ **Weak answer:** "Java caches integers to make it faster."

✅ **Strong answer:** 
"To optimize memory and performance for frequently used numbers, the JVM caches `Integer` objects in the range of -128 to +127 (configurable via `-XX:AutoBoxCacheMax`). When autoboxing an `int` within this range, or calling `Integer.valueOf()`, the JVM returns a reference to the pre-existing cached object rather than instantiating a new one. This is why `Integer a = 100; Integer b = 100;` results in `a == b` evaluating to true, but doing the same for `1000` evaluates to false. It's a critical reason why you must always use `.equals()` to compare wrapper objects."

### Q4: What happens if you cast a `long` to an `int`?
**Difficulty:** Medium | **Companies:** Oracle, Microsoft

❌ **Weak answer:** "The compiler throws an error."

✅ **Strong answer:** 
"It requires an explicit cast (e.g., `(int) myLong`). Because an `int` is 32 bits and a `long` is 64 bits, this is a narrowing conversion. The JVM handles this by simply discarding the upper 32 bits of the `long`. If the `long` value was within the valid range of an `int`, the value remains correct. However, if the `long` was larger than `Integer.MAX_VALUE`, discarding the upper bits alters the sign bit and the magnitude, resulting in a completely different (and often negative) number, known as arithmetic overflow."

### Q5: What is Autoboxing and when does it hurt performance?
**Difficulty:** Medium | **Companies:** Atlassian, Bloomberg

❌ **Weak answer:** "It's when Java converts int to Integer automatically."

✅ **Strong answer:** 
"Autoboxing is the compiler automatically inserting `Integer.valueOf()` to convert a primitive to its corresponding wrapper object, and unboxing is inserting `.intValue()`. It hurts performance severely inside loops. If you accumulate a sum inside a loop using an `Integer` object instead of an `int` primitive, the immutability of wrapper objects means a brand new object is allocated on the heap for every single iteration. This causes high memory churn and forces the Garbage Collector to run frequently, crippling throughput."
