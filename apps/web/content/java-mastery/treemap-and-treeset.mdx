## WHY
`HashMap` and `HashSet` are extremely fast (`O(1)`), but they have one massive limitation: they completely destroy the ordering of your data. If you insert `Z`, `A`, `Q`, a `HashMap` will store them randomly based on their mathematical hash. If you iterate over the map, they will print out in unpredictable chaos.

What if you need to build a High Score leaderboard where players are always kept perfectly sorted by their score? Sorting a `List` of 10 million players every time a new score comes in is incredibly slow (`O(n log n)`). 

You need a data structure that maintains absolute sorted order *while* inserting data, and still provides extremely fast lookups. The `TreeMap` and `TreeSet` solve this by abandoning the array-and-hash approach entirely and using a self-balancing binary search tree.

## THEORY
### Red-Black Trees
`TreeMap` (and `TreeSet`, which is just a wrapper around a `TreeMap`) is powered by a **Red-Black Tree**.
This is a complex, self-balancing binary search tree. 
- Every node has a color (Red or Black) used to mathematically ensure the tree never becomes unbalanced.
- When you insert a new key, the tree compares it to the root. If it's smaller, it goes left. If larger, it goes right, traversing down until it finds an empty spot.
- If an insertion causes one side of the tree to become too deep, the tree performs a "Rotation" to rebalance itself.
Because the tree is strictly balanced, the maximum depth is always logarithmic. Therefore, `get()`, `put()`, and `remove()` operations are guaranteed to execute in **`O(log n)`** time.

### Comparable vs Comparator
Because a `TreeMap` must sort elements as they are inserted, it *must* know how to compare them.
- If you use objects as keys (e.g., `String`, `Integer`), they automatically implement the `Comparable` interface, which provides natural alphabetical/numerical sorting.
- If you use custom objects, or want to sort in reverse order, you must provide a custom `Comparator` to the `TreeMap` constructor. If you don't, it crashes with a `ClassCastException` at runtime!

## CODE

### Level 1 — Beginner
```java
import java.util.Set;
import java.util.TreeSet;

public class BasicTreeSet {
    public static void main(String[] args) {
        Set<String> names = new TreeSet<>();
        
        // Inserted in random order
        names.add("Zebra");
        names.add("Apple");
        names.add("Monkey");
        
        // Iterating prints them in PERFECT alphabetical order!
        // Apple, Monkey, Zebra
        for (String name : names) {
            System.out.println(name);
        }
    }
}
```

### Level 2 — Building Blocks
```java
import java.util.Map;
import java.util.TreeMap;

public class BasicTreeMap {
    public static void main(String[] args) {
        // Natural ordering (1, 2, 3...)
        Map<Integer, String> rankings = new TreeMap<>();
        rankings.put(3, "Bronze");
        rankings.put(1, "Gold");
        rankings.put(2, "Silver");
        
        // Prints: {1=Gold, 2=Silver, 3=Bronze}
        System.out.println(rankings);
    }
}
```

### Level 3 — Intermediate
```java
import java.util.Comparator;
import java.util.TreeMap;

public class CustomSorting {
    public static void main(String[] args) {
        // Supplying a custom Comparator to sort in REVERSE order
        TreeMap<String, Double> prices = new TreeMap<>(Comparator.reverseOrder());
        
        prices.put("Apple", 1.99);
        prices.put("Banana", 0.99);
        prices.put("Zucchini", 2.99);
        
        // Prints: {Zucchini=2.99, Banana=0.99, Apple=1.99}
        System.out.println(prices);
    }
}
```

### Level 4 — Advanced
```java
import java.util.NavigableMap;
import java.util.TreeMap;

public class NavigableFeatures {
    public static void main(String[] args) {
        TreeMap<Integer, String> logEntries = new TreeMap<>();
        logEntries.put(1000, "App Started");
        logEntries.put(1050, "User Logged In");
        logEntries.put(1200, "Error Occurred");
        logEntries.put(1300, "App Crashed");
        
        // TreeMap implements NavigableMap, giving us extremely powerful range queries!
        
        // Get the closest entry BEFORE timestamp 1100
        System.out.println(logEntries.floorEntry(1100)); // 1050=User Logged In
        
        // Get a sub-map of all events between 1000 and 1200 (inclusive/exclusive)
        NavigableMap<Integer, String> range = logEntries.subMap(1000, true, 1200, false);
        System.out.println(range); 
    }
}
```

## REAL_WORLD
In standard web applications, you should default to `HashMap`. `TreeMap` is slower (`O(log n)` vs `O(1)`), consumes more memory (due to tree node pointers), and is completely unnecessary if you just need key-value lookups.

However, `TreeMap` is the weapon of choice for **Time-Series Data** and **Scheduling Systems**.
If you are building an algorithmic trading engine, you receive thousands of price ticks per second out of order. You store them in a `TreeMap<Long, PriceData>` where the key is the timestamp. When the trading algorithm needs to calculate the moving average for the last 5 minutes, it doesn't loop over the entire database. It simply calls `treeMap.subMap(now - 5mins, now)`, which instantly isolates the exact slice of time in `O(log n)` time. This makes range queries exponentially faster than scanning a database.

## INTERVIEW
**Q: What is the time complexity of `get()` in a TreeMap vs HashMap?**
A: `HashMap` is `O(1)` average time because it uses a hash-based array index lookup. `TreeMap` is `O(log n)` time because it must traverse down a Red-Black binary search tree to find the key. 

**Q: Can you put `null` keys into a `TreeMap`?**
A: No. A `TreeMap` must compare keys to sort them. Calling `compareTo()` on a `null` key immediately throws a `NullPointerException`. (Note: `HashMap` DOES allow exactly one `null` key, mapping it to bucket 0).

**Q: Does a `TreeSet` use `equals()` or `compareTo()` to determine if an element is a duplicate?**
A: This is a massive trap! `TreeSet` and `TreeMap` completely ignore the `.equals()` method. They rely 100% on `.compareTo()` (or the provided `Comparator`). If `.compareTo()` returns `0`, the tree considers the objects to be identical duplicates, even if `.equals()` would return `false`. Always ensure your `compareTo` logic is consistent with `equals`.
