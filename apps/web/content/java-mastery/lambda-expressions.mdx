## WHY
Before Java 8, if you wanted to pass a simple piece of logic into a method—for example, passing a custom sorting rule into `Collections.sort()`—you had to create an entirely new class that implemented an interface (usually an Anonymous Inner Class). 
This resulted in absolute syntax bloat. You had to write 6 lines of structural boilerplate just to express 1 line of actual business logic. 

Functional programming languages had solved this decades ago by treating functions as "first-class citizens", allowing developers to pass functions around as easily as variables. Java 8 introduced **Lambda Expressions** to bring this immense power to Java. It allows you to write clean, concise, inline functions that dramatically reduce boilerplate and make APIs highly declarative.

## THEORY
### What is a Lambda?
A Lambda expression is essentially an anonymous function. It does not belong to any class, has no name, and doesn't require access modifiers.
The fundamental syntax is: `(Parameters) -> { Body }`
- **Parameters**: The input variables. If the compiler can infer the type, you don't need to write it (e.g., `(a, b)` instead of `(int a, int b)`).
- **Arrow `->`**: The lambda operator that separates the parameters from the body.
- **Body**: The logic. If the logic is a single statement, you can omit the `{}` brackets and the `return` keyword!

### `invokedynamic`
You might assume that the Java Compiler takes your Lambda expression and secretly converts it into an Anonymous Inner Class at compile time. **It does not.** 
To avoid generating thousands of hidden `.class` files and polluting the JVM Metaspace, Java 8 leverages the `invokedynamic` bytecode instruction. At runtime, the JVM dynamically generates the lambda function directly into memory. This makes lambdas significantly faster and more memory-efficient than the old anonymous inner classes they replaced.

## CODE

### Level 1 — Beginner
```java
import java.util.List;
import java.util.ArrayList;

public class BasicLambda {
    public static void main(String[] args) {
        List<String> names = new ArrayList<>(List.of("Alice", "Bob", "Charlie"));
        
        // Old Way: Enhanced For-Loop
        for (String name : names) {
            System.out.println(name);
        }
        
        // New Way: The forEach method accepts a Lambda!
        // No type declaration needed for 'name', no curly braces needed!
        names.forEach(name -> System.out.println(name));
    }
}
```

### Level 2 — Building Blocks
```java
import java.util.List;
import java.util.ArrayList;

public class LambdaSyntax {
    public static void main(String[] args) {
        List<Integer> numbers = new ArrayList<>(List.of(1, 2, 3, 4, 5));
        
        // The removeIf method accepts a Predicate lambda (returns a boolean)
        // This instantly removes all even numbers from the list!
        numbers.removeIf(n -> n % 2 == 0);
        
        // Multi-line lambda requires {} and the 'return' keyword
        numbers.replaceAll(n -> {
            int squared = n * n;
            return squared;
        });
        
        System.out.println(numbers); // [1, 9, 25]
    }
}
```

### Level 3 — Intermediate
```java
import java.util.Collections;
import java.util.List;
import java.util.ArrayList;

public class SortingLambdas {
    public static void main(String[] args) {
        List<String> words = new ArrayList<>(List.of("Apple", "Banana", "Kiwi"));
        
        // Sorting by String length using a Lambda (a, b)
        // We omit the types (String a, String b) because they are inferred from the List
        Collections.sort(words, (a, b) -> Integer.compare(a.length(), b.length()));
        
        System.out.println(words); // [Kiwi, Apple, Banana]
    }
}
```

### Level 4 — Advanced
```java
import java.util.List;

public class MethodReferences {
    public static void main(String[] args) {
        List<String> names = List.of("Alice", "Bob");
        
        // Sometimes a lambda literally just calls an existing method.
        // names.forEach(name -> System.out.println(name));
        
        // You can replace it with a Method Reference (::) for maximum conciseness!
        names.forEach(System.out::println);
        
        // Using method references to map Strings to their lengths
        names.stream()
             .map(String::length) // equivalent to: s -> s.length()
             .forEach(System.out::println);
    }
}
```

## REAL_WORLD
Lambdas are the absolute lifeblood of modern Java frameworks.

In Spring Security, the configuration file used to be an incomprehensible mess of XML or deeply nested builder chains. In Spring Boot 3, the entire security configuration relies entirely on custom Lambdas to provide incredibly clean, localized DSLs (Domain Specific Languages).

```java
@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    // Spring Security configuration using Lambdas!
    http
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/public/**").permitAll()
            .anyRequest().authenticated()
        )
        .formLogin(login -> login
            .loginPage("/custom-login")
            .defaultSuccessUrl("/dashboard")
        )
        .csrf(csrf -> csrf.disable());

    return http.build();
}
```
Notice how the `auth`, `login`, and `csrf` configurations are completely isolated inside their own lambda blocks? This prevents configuration scope bleed and makes the API vastly more readable.

## INTERVIEW
**Q: Can a Lambda expression modify variables defined outside of it?**
A: Yes, but only if the variable is "Effectively Final". A lambda can read any local variable defined in the surrounding method. However, if the lambda attempts to modify that variable, or if the variable is modified anywhere else in the method, the compiler will instantly throw an error. This restriction exists to prevent catastrophic concurrency bugs if the lambda is executed on a different thread.

**Q: What is a Method Reference?**
A: A Method Reference (`::`) is a shorthand syntax for a Lambda expression that does nothing but call a single existing method. For example, `String::toUpperCase` is the exact equivalent of the lambda `s -> s.toUpperCase()`. It is purely syntax sugar to improve readability.

**Q: How does the Java compiler know the types of the parameters in a Lambda?**
A: **Type Inference**. The compiler looks at the method signature that the lambda is being passed into. If you pass a lambda into `List<String>.forEach()`, the compiler knows that the `forEach` method expects a Consumer of Strings. Therefore, it automatically infers that the parameter in your lambda must be a String.
