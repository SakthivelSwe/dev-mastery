## WHY
If you need to retrieve a user's profile by their email address, searching a `List<User>` forces you to check every single user one by one (`O(n)`). If you have 10 million users, this takes millions of operations per search.

A Hash Table solves this by providing `O(1)` constant-time lookups. It mathematically maps the email address directly to a memory location. In Java, `HashMap` is the absolute king of collections. It is arguably the most frequently used data structure in the entire language. Understanding exactly how it calculates hashes, resolves collisions, and dynamically resizes its internal array is a mandatory requirement for passing any senior engineering interview.

## THEORY
### The Bucket Array and Hashing
Under the hood, a `HashMap` is an array of `Node<K,V>` objects, often called "buckets". By default, the array size is 16.
When you call `put(key, value)`:
1. The JVM calls `key.hashCode()`.
2. It mathematically compresses this hash to fit within the array bounds using a bitwise AND operation: `index = (n - 1) & hash`.
3. It places the `Node` at that exact array index.
When you call `get(key)`, it runs the exact same math, instantly calculating the index and returning the value in `O(1)` time.

### Collision Resolution (Java 8 Treeify)
What happens if two different keys calculate to the exact same array index? This is a **Collision**.
Historically, `HashMap` handled this via a **Linked List**. The new `Node` was simply attached to the existing `Node` at that index. However, if a malicious user intentionally fed the map 10,000 keys that all collided, the list would become 10,000 nodes long, destroying performance (`O(n)` lookup).
**Java 8 Optimization**: Once a bucket's linked list reaches 8 nodes (the `TREEIFY_THRESHOLD`), the `HashMap` dynamically transforms the linked list into a **Red-Black Tree**. This guarantees worst-case lookup performance degrades to `O(log n)` instead of `O(n)`, neutralizing Denial of Service (DoS) collision attacks.

### Load Factor and Rehashing
A `HashMap` has a default `Load Factor` of 0.75. This means when the bucket array is 75% full (12 out of 16 slots), the `HashMap` will completely resize itself. It creates a brand new array double the size (32), and recalculates the hash index for every single existing element, moving them to the new array. This is a very expensive operation called **Rehashing**.

## CODE

### Level 1 — Beginner
```java
import java.util.HashMap;
import java.util.Map;

public class BasicHashMap {
    public static void main(String[] args) {
        Map<String, Integer> scores = new HashMap<>();
        
        // O(1) Insert
        scores.put("Alice", 95);
        scores.put("Bob", 82);
        
        // O(1) Retrieval
        System.out.println(scores.get("Alice")); // 95
        
        // put() overwrites if the key exists
        scores.put("Alice", 100); 
    }
}
```

### Level 2 — Building Blocks

```java
public class MapIteration {
    public static void main(String[] args) {
        Map<String, String> config = new HashMap<>();
        config.put("host", "localhost");
        config.put("port", "8080");
        
        // BEST PRACTICE: Iterate over the EntrySet (gets both key and value simultaneously)
        for (Map.Entry<String, String> entry : config.entrySet()) {
            System.out.println(entry.getKey() + "=" + entry.getValue());
        }
        
        // Java 8+ lambda approach
        config.forEach((key, value) -> {
            System.out.println(key + "=" + value);
        });
    }
}
```

### Level 3 — Intermediate

```java
public class CapacityTuning {
    public static void main(String[] args) {
        // If we know we are adding exactly 10,000 items,
        // we should pre-size the HashMap to avoid expensive Rehashing.
        // Math: 10,000 / 0.75 (load factor) = 13,333. 
        // We round up to the next power of 2: 16,384.
        
        Map<Integer, String> optimizedMap = new HashMap<>(16384);
        
        for (int i = 0; i < 10000; i++) {
            optimizedMap.put(i, "Data");
        }
    }
}
```

### Level 4 — Advanced
```java
import java.util.Objects;

// Defining a custom object as a HashMap Key
public class CustomKey {
    private final String id;
    private final String tenant;

    public CustomKey(String id, String tenant) {
        this.id = id; this.tenant = tenant;
    }

    // MANDATORY: If you use a custom object as a Key, 
    // you MUST override equals() and hashCode() perfectly.
    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (o == null || getClass() != o.getClass()) return false;
        CustomKey that = (CustomKey) o;
        return Objects.equals(id, that.id) && Objects.equals(tenant, that.tenant);
    }

    @Override
    public int hashCode() {
        return Objects.hash(id, tenant);
    }
}
```

## REAL_WORLD
A legendary bug in early Java web applications was the "Mutable Key Trap". 

Developers would create a complex object, put it into a `HashMap` as a Key, and then *modify* the object's fields later. 
```java
User u = new User("alice@corp.com");
map.put(u, "Admin Profile");

// 100 lines later...
u.setEmail("alice@newcorp.com"); // BUG!

// Attempt to fetch the profile
map.get(u); // RETURNS NULL!
```
When `setEmail` was called, the object's `hashCode()` output changed. But the object was already placed in bucket #5 based on its old hash code. When you call `get(u)`, it calculates the new hash code, looks in bucket #12, finds nothing, and returns `null`. The object is permanently lost in the map like a ghost.

**Rule for Production:** Keys in a `HashMap` must ALWAYS be Immutable (like `String` or `Integer` or a Java `Record`). Never use a mutable JPA Entity as a Map key.

## INTERVIEW
**Q: How does `HashMap` handle collisions?**
A: When two keys hash to the same bucket index, `HashMap` stores them in a Linked List at that bucket. When retrieving, it traverses the list and uses the `.equals()` method to find the exact key. In Java 8+, if this list grows beyond 8 elements (`TREEIFY_THRESHOLD`), the map converts the list into a Red-Black Tree, improving search performance from `O(n)` to `O(log n)`.

**Q: What happens if you override `equals()` but do not override `hashCode()`?**
A: You break the fundamental contract of `HashMap`. If two objects are `.equals()`, they MUST return the exact same `hashCode()`. If you don't override `hashCode()`, two logically equal objects will return different memory-based hashes. They will be placed in completely different buckets in the map, making it impossible to retrieve the value using a seemingly identical key.

**Q: Why is the default `HashMap` array size (16) always a power of 2?**
A: Because of extreme CPU optimization. The standard formula to fit a hash into an array size `n` is modulo math: `hash % n`. However, modulo operations are slow at the CPU level. If (and only if) `n` is a power of 2, you can use bitwise AND masking: `hash & (n - 1)`. This produces the exact same result as modulo but executes drastically faster in the CPU logic gates.
