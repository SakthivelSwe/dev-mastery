## WHY
If you build a beautiful, high-performance web application, but a user who is blind (using a screen reader) or a user with motor impairments (using keyboard navigation) cannot use it, your product is broken. In many countries, it is also illegal and opens your company to massive lawsuits.

**Accessibility (a11y)** is not a feature you add at the end of a project. It is a fundamental requirement. By baking accessibility directly into the foundational components of a Design System, you guarantee that every developer who uses those components automatically builds accessible applications by default, without even thinking about it.

## THEORY
### The WCAG Standards
The Web Content Accessibility Guidelines (WCAG) define the international standards for accessibility.
They are organized around 4 principles (POUR):
1. **Perceivable:** Users must be able to perceive the information (e.g., sufficient color contrast, alt text for images).
2. **Operable:** Users must be able to operate the interface (e.g., full keyboard navigation, no keyboard traps).
3. **Understandable:** The information and operation must be understandable (e.g., clear error messages, predictable navigation).
4. **Robust:** Content must be robust enough to be interpreted by a wide variety of assistive technologies (e.g., semantically correct HTML).

## CODE

### Level 1 — Semantic HTML (The Foundation)
The easiest way to make a component accessible is to use the correct HTML element. 
If you build a button using a `<div>`, you break everything. A screen reader doesn't know it's a button, and a user cannot trigger it by pressing the `Enter` or `Space` key.

```jsx
// THE TERRIBLE WAY
// Fails Screen Readers. Fails Keyboard Navigation. Fails Focus Outlines.
<div className="btn bg-blue-500 text-white p-2 rounded" onClick={submit}>
  Submit
</div>

// THE DESIGN SYSTEM WAY
// By simply using the native <button> tag, the browser gives you 90% of accessibility for free!
<button className="btn bg-blue-500 text-white p-2 rounded" onClick={submit}>
  Submit
</button>
```

### Level 2 — Focus Management
Keyboard users (and power users) rely on the `Tab` key to navigate the interface. They must always know exactly where their cursor is. 
Designers often hate the default blue "focus ring" browsers provide, so junior developers write `outline: none;` in their CSS. **This completely destroys keyboard accessibility.**

If you remove the default outline, you MUST replace it with a custom, highly visible focus state in your Design System!

```css
/* THE TERRIBLE WAY */
.btn:focus {
  outline: none; /* The user is now completely lost on the page! */
}

/* THE DESIGN SYSTEM WAY */
.btn {
  /* Set up a custom transition for focus states */
  transition: box-shadow 0.2s ease;
}

.btn:focus-visible {
  /* focus-visible is incredible! It ONLY shows the ring if the user is navigating 
     with a keyboard! If they click with a mouse, the ring remains hidden, 
     keeping the designers happy! */
  outline: none;
  /* Use a brand-colored box-shadow or an offset outline to make it pop! */
  box-shadow: 0 0 0 3px var(--color-primary-200);
}
```

### Level 3 — ARIA Attributes (WAI-ARIA)
When native HTML isn't enough (e.g., building complex tabs, modals, or dropdown menus), you must use **ARIA** (Accessible Rich Internet Applications) attributes to explain the component's state to screen readers.

```jsx
// Accessible Dropdown/Disclosure Component
export const Accordion = ({ title, children, isOpen, onToggle }) => {
  return (
    <div>
      <button 
        onClick={onToggle}
        /* 1. Tell the screen reader this button controls another element */
        aria-controls="accordion-content"
        /* 2. Tell the screen reader if the controlled element is currently open or closed! */
        aria-expanded={isOpen}
      >
        {title}
        {/* Hide the visual chevron icon from the screen reader to prevent noise */}
        <ChevronIcon aria-hidden="true" />
      </button>

      <div 
        id="accordion-content" 
        /* 3. Completely hide the content from the screen reader when closed! */
        hidden={!isOpen}
      >
        {children}
      </div>
    </div>
  );
};
```

### Level 4 — Advanced Color and Status Reliance
Colorblindness affects roughly 8% of men. If your UI relies *solely* on color to convey information, it is broken.

Imagine a login form. If the user types a wrong password, the input border turns Red. A user with red-green colorblindness might not notice the change at all!

**Rule: Never use color as the ONLY visual means of conveying information.**

```jsx
// THE TERRIBLE WAY (Relies only on color)
<input className={hasError ? "border-red-500" : "border-gray-300"} />

// THE DESIGN SYSTEM WAY
export const Input = ({ hasError, errorMessage }) => {
  return (
    <div>
      <input 
        // 1. We still use the red border for sighted users
        className={`border-2 ${hasError ? "border-red-500" : "border-gray-300"}`}
        
        // 2. We programmatically tell the screen reader this input is invalid!
        aria-invalid={hasError}
        
        // 3. We link the input directly to the error message ID!
        aria-describedby={hasError ? "error-msg-id" : undefined}
      />
      
      {/* 4. We provide a VISUAL cue (the text and an icon) that does not rely on color! */}
      {hasError && (
        <span id="error-msg-id" className="text-red-500 flex items-center">
          <AlertTriangleIcon aria-hidden="true" className="mr-1" />
          {errorMessage}
        </span>
      )}
    </div>
  );
};
```

## REAL_WORLD
**Screen Reader Only Text (`.sr-only`)**
Sometimes a design is minimal. A "Close" button on a modal might just be an "X" icon. 
Visually, this is fine. But a screen reader will just read "Button". Close what?
Design Systems heavily utilize a CSS utility class called `.sr-only` (Screen Reader Only). It visually hides text from the screen without using `display: none;` (which would also hide it from the screen reader).

```css
/* The classic .sr-only utility class */
.sr-only {
  position: absolute;
  width: 1px;
  height: 1px;
  padding: 0;
  margin: -1px;
  overflow: hidden;
  clip: rect(0, 0, 0, 0);
  white-space: nowrap;
  border-width: 0;
}
```
```jsx
<button className="modal-close-btn" onClick={closeModal}>
  <CloseIcon aria-hidden="true" />
  {/* Visually hidden, but perfectly read by the screen reader! */}
  <span className="sr-only">Close Modal</span>
</button>
```

## INTERVIEW
**Q: I built a Modal (Dialog) component. It works perfectly with a mouse. However, accessibility testers reported a "Keyboard Trap" issue. When the modal is open, they can press 'Tab' and the focus accidentally moves to the links on the main page *behind* the modal! How do you fix this?**
A: **Focus Trapping.**
When a highly disruptive element like a Modal or a Drawer opens, it must "Trap" the keyboard focus. 
1. When the modal opens, you use JavaScript to automatically move the browser's `document.activeElement` (focus) to the very first focusable element *inside* the modal.
2. You write a global `keydown` event listener for the `Tab` key. 
3. If the user presses `Tab` while on the *last* focusable element inside the modal, you intercept the event (`e.preventDefault()`) and manually force the focus back to the *first* element inside the modal, creating a loop!
4. Additionally, you must add an event listener for the `Escape` key to instantly close the modal, and when the modal closes, you MUST return the focus back to the exact button the user clicked to open it originally! 
*(Note: Because this logic is incredibly complex, modern Design Systems use Headless UI libraries like Radix or HeadlessUI which handle Focus Trapping automatically!).*

## FEYNMAN CHECK

### Explain Accessibility in Design Like I'm 10
> Accessible design means everyone can use the product — keyboard users, screen-reader users, low-vision users, motor-impaired users. The non-obvious part: every interactive element needs a role, name, and state for assistive tech, and semantic HTML provides those for free. This is why "just use a div" silently excludes 15% of users.

### 5 Deep Questions
**Q1: Why is accessibility a design concern, not just a dev concern?** > A: Inaccessible mockups produce inaccessible products. Designers must specify focus states, error states, and contrast in Figma — not leave them for developers to invent.
**Q2: ONE model.** > A: "Every interactive element has role + name + state. Miss any and AT users are locked out."
**Q3: Misconception+code.** > A: Decorative aria roles on non-interactive elements. ```tsx
// ❌ <div role="button" onClick={...}> — no keyboard, no Enter handler
// ✅ <button onClick={...}>
```
**Q4: Interaction with design tokens?** > A: Colour tokens must encode contrast pairs verified at definition time.
**Q5: Senior one-liner.** > A: "Accessibility in design is encoding WCAG into design tokens, component states, and Figma mockups — not a post-development audit."
