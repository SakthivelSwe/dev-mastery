## WHY
If you build the most perfectly engineered, accessible, performant `<Button>` component in the world, but you don't tell anyone how to use it, developers will just build their own `<Button>` from scratch.

A Design System is not a component library; it is a **product**. 
Its users are the developers and designers within your company. Like any successful product, if the documentation is poor, adoption will fail. Comprehensive, interactive documentation bridges the gap between Figma and VSCode, ensuring the system is actually utilized.

## THEORY
### The Anatomy of Great Documentation
Every component in your system requires a dedicated documentation page containing three distinct sections:
1. **Design Guidelines (For Designers):** When to use it, when NOT to use it, and visual best practices.
2. **Code Implementation (For Developers):** Interactive playgrounds, code snippets, and API (Props) tables.
3. **Accessibility (For QA/Compliance):** Keyboard interactions, ARIA roles, and screen reader expectations.

## CODE

### Level 1 — The MDX Revolution
Historically, developers wrote documentation in raw HTML or plain Markdown. 
Markdown is great for text, but terrible for demonstrating React components. You had to take screenshots of the components and paste the images into the Markdown.

**MDX** (Markdown + JSX) changed everything. MDX allows you to literally import and execute live React components directly inside a Markdown text file!

```mdx
<!-- ButtonDocs.mdx -->

# Button Component

The Button is the primary call to action in our application.

import { Button } from '@mycompany/ui';

## Live Demo
<div style={{ padding: '20px', border: '1px solid gray' }}>
  {/* THIS IS A LIVE, CLICKABLE REACT COMPONENT INSIDE THE MARKDOWN! */}
  <Button variant="primary" onClick={() => alert("Clicked!")}>
    Interactive Demo
  </Button>
</div>

## Usage Rules
- **DO** use Primary buttons for the main action on a page.
- **DON'T** put two Primary buttons next to each other.
```

### Level 2 — Auto-Generating Prop Tables
Writing and maintaining a table of Component Props (e.g., `variant`, `size`, `disabled`) in Markdown is tedious. If a developer adds a new prop to the TypeScript interface, they will inevitably forget to update the documentation table.

Tools like **Storybook Docs** or **Nextra** use AST (Abstract Syntax Tree) parsing (via `react-docgen`) to read your actual TypeScript source code and automatically generate the documentation tables!

```tsx
// Button.tsx
export interface ButtonProps {
  /** 
   * The visual style of the button. 
   * Use 'danger' for destructive actions like Delete.
   */
  variant?: 'primary' | 'secondary' | 'danger';
  
  /** Disables click events and reduces opacity */
  disabled?: boolean;
}
export const Button = (props: ButtonProps) => { ... }
```
*The Magic:* The documentation site automatically reads those `/** comments */` and builds a beautiful, auto-updating HTML table showing the Prop Name, Type, Default Value, and Description. Zero manual maintenance required!

### Level 3 — Interactive Sandboxes (Live Code Editing)
A static live demo is good. A demo where the user can edit the code in real-time is incredible.
Tools like **Sandpack** (by CodeSandbox) or **React Live** allow you to embed a literal mini-IDE into your documentation.

```jsx
// Example of a React Live code block
// The developer reading the documentation can type inside this code block, 
// and the UI updates instantly above it!

import { LiveProvider, LiveEditor, LiveError, LivePreview } from 'react-live';
import { Button } from '@mycompany/ui';

<LiveProvider code={`<Button variant="danger">Delete Account</Button>`} scope={{ Button }}>
  <div className="preview-box">
    <LivePreview /> {/* The rendered button */}
  </div>
  <LiveEditor /> {/* The editable code box */}
  <LiveError />
</LiveProvider>
```

### Level 4 — Integrating Design Assets (Figma)
Developers live in code, Designers live in Figma. 
If the Figma file is updated, but the Developer documentation isn't, the system fractures.
Modern documentation sites (like Storybook) allow you to embed the live Figma artboards directly into the developer documentation.

```jsx
// Storybook Example: Embedding Figma
import { withDesign } from 'storybook-addon-designs';

export default {
  title: 'Components/Button',
  component: Button,
  decorators: [withDesign],
};

export const Primary = () => <Button>Click</Button>;

Primary.parameters = {
  design: {
    type: 'figma',
    // The developer can see the exact Figma file right next to the code!
    url: 'https://www.figma.com/file/12345/Design-System?node-id=1:2',
  },
};
```

## REAL_WORLD
**Do/Don't Visuals**
The most effective way to document design guidelines is not a wall of text. It is a visual "Do / Don't" board.
Instead of writing "Do not use long text in buttons", you provide two visual examples:
1. **[✅ DO]** A button that says "Save Profile" (with a green checkmark).
2. **[❌ DON'T]** A button that says "Click here to securely save your profile to the database" (with a red X).

A high-quality Design System site provides MDX utility components specifically for rendering these side-by-side comparison blocks, making the guidelines instantly scannable and undeniable.

## INTERVIEW
**Q: How do you measure if the Design System is actually being successfully adopted by the product teams, or if they are just ignoring it and writing custom CSS?**
A: **Static Analysis and Telemetry.**
Documentation alone doesn't guarantee adoption. You must measure it.
1. **Custom ESLint Rules:** You write an ESLint plugin that throws a warning if a developer tries to use `<button className="...">` instead of importing `<Button>` from the design system. 
2. **AST Scanning:** You write a script that runs weekly across all company repositories. It parses the Abstract Syntax Tree of the codebase and counts exactly how many times `import { Button } from '@mycompany/ui'` is used vs native HTML elements. 
3. **The Dashboard:** You publish these metrics to an internal dashboard. If the "Adoption Score" of a specific app is only 40%, you know the documentation failed or the Design System is missing critical features that the specific product team requires!

## FEYNMAN CHECK

### Explain Documenting Design Systems Like I'm 10
> Documentation IS the design system. A component nobody can find or understand might as well not exist. Docs include: usage guidelines, props API, dos/don'ts, accessibility notes, code examples. The non-obvious part: documentation must be co-located with components (MDX in Storybook, not a separate Confluence page) so it stays in sync with code. This is why Notion-based design system docs go stale within 3 months.

---

### 5 Deep Questions
**Q1: Why co-locate docs with components?** > A: Drift is the #1 documentation failure; co-location forces updates in the same PR.
**Q2: ONE model.** > A: "Storybook MDX = single source. Props auto-generated from TypeScript. Examples are live stories."
**Q3: Misconception+code.** > A: Manual prop tables go stale; auto-generate from TS via storybook-addon-docs.
**Q4: With Figma?** > A: Sync via component spec links; tokens defined in code, mirrored to Figma.
**Q5: Senior one-liner.** > A: "Design system docs are co-located, code-generated, example-driven MDX — anything else drifts."
