## WHY
If a user submits a form, the "Submit" button needs to disable itself, change its color to gray, and display a spinning loader.
If you manage this by manually toggling 5 different CSS classes (`.bg-blue-500`, `.bg-gray-400`, `.cursor-not-allowed`, `.opacity-50`) inside your React component, your code will quickly become an unreadable, bug-prone mess.

A **Design System** abstracts these combinations of CSS classes into highly controlled **Variants and States**, ensuring that UI components react predictably to user interaction without cluttering the application's business logic.

## THEORY
### The Three Dimensions of UI State
When designing a robust component, you must account for three separate dimensions that affect its visual appearance:
1. **Variants (The Style):** e.g., `primary`, `secondary`, `danger`, `ghost`, `outline`.
2. **Sizes (The Scale):** e.g., `sm`, `md`, `lg`.
3. **Interactive States (The Condition):** e.g., `hover`, `active`, `focus`, `disabled`, `loading`.

A truly robust component allows developers to mix and match these dimensions safely (`<Button variant="danger" size="sm" loading />`).

## CODE

### Level 1 — The Naive Approach (Template Literal Hell)
Junior developers often try to handle variants using raw string interpolation. This scales terribly and is almost impossible to maintain.

```jsx
// THE TERRIBLE WAY
export const Button = ({ variant, size, disabled, children }) => {
  return (
    <button
      className={`
        base-button-styles 
        ${variant === 'primary' ? 'bg-blue-500 text-white' : ''}
        ${variant === 'danger' ? 'bg-red-500 text-white' : ''}
        ${size === 'sm' ? 'text-sm p-2' : ''}
        ${size === 'lg' ? 'text-lg p-4' : ''}
        ${disabled ? 'opacity-50 cursor-not-allowed' : 'hover:opacity-80'}
      `}
      disabled={disabled}
    >
      {children}
    </button>
  );
};
```

### Level 2 — CVA (Class Variance Authority)
The modern industry standard for building Design System components (especially with Tailwind CSS) is **CVA (Class Variance Authority)**.
CVA allows you to map props directly to CSS classes using a beautiful, declarative configuration object.

```tsx
import { cva, type VariantProps } from "class-variance-authority";

// 1. Define the rules mathematically!
const buttonVariants = cva(
  // The BASE styles applied to EVERY button
  "inline-flex items-center justify-center rounded-md font-medium transition-colors focus-visible:outline-none focus-visible:ring-2 disabled:pointer-events-none disabled:opacity-50",
  {
    variants: {
      variant: {
        primary: "bg-blue-600 text-white hover:bg-blue-700",
        danger: "bg-red-600 text-white hover:bg-red-700",
        ghost: "hover:bg-slate-100 text-slate-900",
      },
      size: {
        sm: "h-9 px-3 text-xs",
        md: "h-10 px-4 py-2",
        lg: "h-11 px-8 text-lg",
      },
    },
    // The DEFAULT fallback props if the developer doesn't provide them!
    defaultVariants: {
      variant: "primary",
      size: "md",
    },
  }
);

// 2. Build the Component
export const Button = ({ variant, size, className, ...props }) => {
  return (
    <button 
      // CVA automatically calculates the exact string of classes!
      className={buttonVariants({ variant, size, className })} 
      {...props} 
    />
  );
};
```

### Level 3 — Compound Variants
Sometimes, mixing two dimensions requires a specific, unique style that breaks the standard rules.
For example, a `ghost` button is usually transparent. But if a `ghost` button is *also* a `danger` button, maybe it needs a faint red background when hovered.

CVA handles this perfectly with **Compound Variants**.

```tsx
const buttonVariants = cva("base-styles", {
  variants: {
    variant: { primary: "bg-blue", ghost: "bg-transparent" },
    intent: { info: "text-blue", danger: "text-red" }
  },
  // If BOTH of these props are true, apply these extra classes!
  compoundVariants: [
    {
      variant: "ghost",
      intent: "danger",
      className: "hover:bg-red-100", // The special exception rule!
    }
  ]
});
```

### Level 4 — Controlled vs. Uncontrolled Components
When dealing with interactive states (like an Accordion being Open or Closed), who "owns" the state? The Component itself, or the Application?

1. **Uncontrolled:** The Component manages its own internal React `useState`. The application just renders it. Great for simple things.
2. **Controlled:** The Application passes the exact state via props (`isOpen={true}`) and provides an `onChange` handler. The Component has ZERO internal state. Great for complex forms.

A robust Design System component should ideally support **both**!

```jsx
import { useState } from 'react';

// A component that supports both Controlled and Uncontrolled usage!
export const Toggle = ({ 
  isToggled: controlledToggled, // The App provides the state
  defaultToggled = false,       // The App wants the Component to manage it
  onChange 
}) => {
  // Internal state fallback
  const [internalToggled, setInternalToggled] = useState(defaultToggled);

  // Magic: If the App passed a prop, use it. Otherwise, use the internal state!
  const isControlled = controlledToggled !== undefined;
  const currentToggled = isControlled ? controlledToggled : internalToggled;

  const handleClick = () => {
    const newValue = !currentToggled;
    if (!isControlled) {
      setInternalToggled(newValue); // Update internal state
    }
    if (onChange) {
      onChange(newValue); // Always notify the parent
    }
  };

  return <button onClick={handleClick}>{currentToggled ? 'ON' : 'OFF'}</button>;
};
```

## REAL_WORLD
**Data Attributes for State Targeting**
How do you style a component based on complex internal states without passing boolean flags down to 5 different CSS files?
Modern Design Systems (like Radix UI) inject semantic `data-*` attributes directly into the DOM elements based on their state.

```html
<!-- The DOM output of an Accordion -->
<button data-state="open" data-disabled="false">
  Settings
</button>
```
Now, you don't need messy JavaScript string interpolation to handle the CSS! You just target the data attributes natively in CSS!
```css
/* CSS natively reacts to the HTML state! */
button[data-state="open"] {
  border-bottom: 2px solid blue;
}
button[data-state="closed"] {
  border-bottom: none;
}
```

## INTERVIEW
**Q: A developer complains that they cannot override the CSS of your Design System `Button`. They wrote `<Button className="bg-green-500" />`, but the button is still Blue. Why did this happen, and how do you fix it?**
A: **The CSS Specificity / Class Clash problem (Tailwind Merge).**
If the Design System uses Tailwind, the internal button might have the class `bg-blue-500`. If the developer passes `bg-green-500` via the `className` prop, the final HTML becomes `<button class="bg-blue-500 bg-green-500">`. 
Because CSS reads the stylesheet order (not the HTML string order), the Blue color might "win" the cascade, ignoring the developer's override.
**The Solution:** You MUST use a utility like `tailwind-merge` inside your components. 
```tsx
import { twMerge } from "tailwind-merge";

export const Button = ({ className }) => {
  // twMerge intelligently understands Tailwind. It sees two background colors, 
  // deletes the default blue one, and keeps the custom green one!
  const finalClasses = twMerge("bg-blue-500 text-white", className);
  
  return <button className={finalClasses} />;
};
```
This guarantees that consumer applications can cleanly override your components when absolutely necessary!

## FEYNMAN CHECK

### Explain UI State Management Like I'm 10
> Every component has internal states: idle, loading, error, success, disabled, focused, hovered, selected. UI state management means making all those states explicit and predictable. The non-obvious part: implicit state ("button.disabled = something") leads to undefined states (loading + disabled + error all true). A discriminated union forces one state at a time. This is why XState is popular for complex components.
