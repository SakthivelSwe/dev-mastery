## WHY
If a developer looks at a Figma file and sees an `<h1>` is 32px, an `<h2>` is 28px, and a paragraph is 15px, they will hardcode `font-size: 32px` into the CSS. 
Another developer on a different page might use `font-size: 30px` because it "looked better." 
Soon, your application will have 40 different, arbitrary font sizes. The visual hierarchy is destroyed, and updating the design across the app becomes impossible.

A **Typography Scale** enforces a strict mathematical progression of font sizes. It removes guesswork, ensures a harmonious rhythm throughout the application, and guarantees perfect responsiveness across mobile and desktop.

## THEORY
### The Modular Scale
In music, notes sound harmonious when their frequencies relate to each other by specific mathematical ratios. Typography works exactly the same way.
Instead of picking random font sizes, you choose a **Base Size** (usually 16px for body text) and a **Scale Ratio** (e.g., $1.25$ - Major Third).
To get the next size up, you multiply by 1.25:
- $16px \times 1.25 = 20px$
- $20px \times 1.25 = 25px$
- $25px \times 1.25 = 31.25px$

Every font size in your entire application MUST be picked from this mathematical scale.

## CODE

### Level 1 — The REM Unit (Accessibility)
Never use `px` for font sizes. If a visually impaired user changes their browser's default font size from 16px to 24px, any text hardcoded in `px` will *not* scale, breaking accessibility.
You must use `rem` (Root EM). 
`1rem` is exactly equal to the user's browser default (usually 16px). If you want 20px, you write `1.25rem`.

```css
/* THE TERRIBLE WAY (Fails Accessibility) */
h1 { font-size: 32px; }
p  { font-size: 16px; }

/* THE DESIGN SYSTEM WAY */
:root {
  /* Using a 1.25 ratio (Major Third) */
  --text-sm:   0.8rem;   /* ~12.8px */
  --text-base: 1rem;     /* ~16px (Browser Default) */
  --text-lg:   1.25rem;  /* ~20px */
  --text-xl:   1.563rem; /* ~25px */
  --text-2xl:  1.953rem; /* ~31px */
  --text-3xl:  2.441rem; /* ~39px */
  --text-4xl:  3.052rem; /* ~48px */
}

h1 { font-size: var(--text-4xl); }
p  { font-size: var(--text-base); }
```

### Level 2 — Line Height (Leading)
Font size is only half the battle. If lines of text are packed too tightly together, the eye loses its place when wrapping to the next line. This is called **Line Height** (or Leading).
*The Rule:* The wider the block of text, or the smaller the font size, the larger the Line Height must be to remain readable.

```css
:root {
  /* Tighter line heights for massive headings (they usually span 1-2 lines) */
  --leading-none:  1;
  --leading-tight: 1.1;
  --leading-snug:  1.25;
  
  /* Relaxed line heights for dense paragraph reading */
  --leading-normal: 1.5;
  --leading-loose:  2;
}

h1 {
  font-size: var(--text-4xl);
  line-height: var(--leading-tight); /* Prevents massive gaps between heading lines! */
}

p {
  font-size: var(--text-base);
  line-height: var(--leading-normal); /* Breathing room for the eyes! */
  max-width: 65ch; /* Pro-Tip: Never let a paragraph exceed 65 characters in width! */
}
```

### Level 3 — Font Weights and Tracking
You rarely need 5 different font families. You create visual hierarchy by contrasting **Font Weight** and **Letter Spacing** (Tracking).

```css
:root {
  --font-weight-normal: 400;
  --font-weight-medium: 500;
  --font-weight-bold:   700;
  
  --tracking-tighter: -0.05em;
  --tracking-normal:  0em;
  --tracking-wider:   0.05em;
}

/* Creating an "Eyebrow" heading (ALL CAPS subheader) */
.eyebrow {
  font-size: var(--text-sm);
  font-weight: var(--font-weight-bold);
  text-transform: uppercase;
  
  /* ALL CAPS text MUST have increased letter spacing, 
     otherwise the letters bleed into each other! */
  letter-spacing: var(--tracking-wider); 
  
  color: var(--color-primary-600);
}
```

### Level 4 — Fluid Typography (Advanced Responsive Design)
If an `<h1>` is 48px (`3rem`) on a 27-inch 4K monitor, it looks beautiful. If you load that exact same `<h1>` on an iPhone 12, the word gets chopped in half because the screen is too narrow.
Historically, developers used Media Queries to hardcode "jumps" in font sizes.
Modern Design Systems use **Fluid Typography** via the CSS `clamp()` function. The font size dynamically and smoothly scales between a minimum and maximum value based on the exact width of the viewport!

```css
h1 {
  /* clamp(MIN_SIZE, DYNAMIC_PREFERRED_SIZE, MAX_SIZE) */
  /* 
    If the screen is tiny, it locks at 2rem.
    If the screen is massive, it locks at 3.5rem.
    Anywhere in between, it mathematically calculates exactly 5% of the Viewport Width (vw) + 1rem!
  */
  font-size: clamp(2rem, 5vw + 1rem, 3.5rem);
}
```

## REAL_WORLD
**System Fonts vs. Web Fonts**
Designers love custom web fonts (like *Inter* or *Roboto*). However, loading a 300KB web font file blocks the browser from painting the text, causing the dreaded "Flash of Invisible Text" (FOIT).
For massive, high-performance dashboards (like the GitHub or Stripe dashboards), engineers often completely bypass web fonts and rely entirely on the **System Font Stack**.
`font-family: -apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, Helvetica, Arial, sans-serif;`
This incredibly clever string tells the browser: "If you are on a Mac, use Apple's San Francisco font. If you are on Windows, use Microsoft's Segoe UI. If you are on Android, use Roboto."
The result? Zero kilobytes downloaded, 0ms load time, and the application perfectly blends in with the user's native operating system!

## INTERVIEW
**Q: The designer hands you a Figma file. The headings use a Serif font (like Times New Roman), and the body text uses a Sans-Serif font (like Arial). Why did they make this choice, and how do you implement it cleanly?**
A: **Font Pairing for Visual Hierarchy.**
Using a highly stylized Serif font for large, short headings draws the eye and establishes brand identity. Using a clean Sans-Serif font for dense body paragraphs maximizes readability at small sizes.
To implement this in a Design System, you absolutely *do not* define font families on specific HTML tags (e.g., `h1 { font-family: 'Merriweather'; }`). 
You must define semantic Design Tokens for the *role* of the font.
```css
:root {
  --font-family-display: 'Merriweather', serif; /* For Headings */
  --font-family-body: 'Inter', sans-serif;      /* For Paragraphs */
  --font-family-mono: 'Fira Code', monospace;   /* For Code Blocks */
}
```
You then apply these tokens to your utility classes or component abstractions, ensuring the pairing rules are universally respected across the entire UI.
