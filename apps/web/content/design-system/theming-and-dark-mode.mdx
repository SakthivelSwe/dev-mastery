## WHY
Ten years ago, "Dark Mode" was a cool hack for developer tools. Today, it is an operating-system-level requirement. If a user has their iPhone set to Dark Mode, and they open your application at 2:00 AM in a dark room and are blinded by a massive, glaring white `#FFFFFF` background, they will close your app and never return.

Implementing theming (Dark Mode, High Contrast Mode, or Multi-Brand White-Labeling) requires a robust architectural strategy. If you try to hack it together with `!important` tags or complex JavaScript state, your application will become unmaintainable.

## THEORY
### The Three Approaches to Theming
1. **The Hardcoded Approach (Terrible):** Writing separate CSS classes for everything (e.g., `.bg-white .dark:bg-black`). This bloats your CSS and requires manual work on every single component.
2. **The JavaScript Approach (Slow):** Using a React Context (e.g., `Styled-Components` ThemeProvider) to pass JSON objects down the component tree. This is slow, causes massive React re-renders, and fails horribly during Server-Side Rendering (causing the "White Flash of Death").
3. **The CSS Variable Approach (The Standard):** Abstracting all colors into CSS Custom Properties (`--color-bg`). The React components never change. You simply swap the values of the CSS variables globally.

## CODE

### Level 1 — The CSS Variable Foundation
The absolute best way to handle themes is entirely outside of JavaScript. 

```css
/* 1. Define the Semantic Aliases */
:root {
  /* LIGHT MODE (Default) */
  --bg-primary: #ffffff;
  --text-primary: #111827;
  --border-color: #e5e7eb;
}

/* 2. The Dark Mode Override */
/* By attaching this to a [data-theme] attribute, we can toggle it via JS! */
[data-theme="dark"] {
  --bg-primary: #111827;
  --text-primary: #f9fafb;
  --border-color: #374151;
}

/* 3. The Components ONLY use the aliases! */
.card {
  background-color: var(--bg-primary);
  color: var(--text-primary);
  border: 1px solid var(--border-color);
}
```

### Level 2 — Toggling the Theme (React)
To change the theme, you simply need to change the `data-theme` attribute on the `<html>` or `<body>` tag.

```jsx
import { useEffect, useState } from 'react';

export const ThemeToggle = () => {
  const [theme, setTheme] = useState('light');

  // When the user clicks the button...
  const toggleTheme = () => {
    const newTheme = theme === 'light' ? 'dark' : 'light';
    setTheme(newTheme);
    
    // 1. Update the DOM attribute globally!
    document.documentElement.setAttribute('data-theme', newTheme);
    
    // 2. Save their preference in LocalStorage!
    localStorage.setItem('user-theme', newTheme);
  };

  return <button onClick={toggleTheme}>Switch to {theme === 'light' ? 'Dark' : 'Light'}</button>;
};
```

### Level 3 — The "White Flash of Death" (SSR Problem)
If you use Server-Side Rendering (Next.js/Remix), you have a massive problem.
1. The user requests the page. The Server renders the HTML in Light Mode (because servers don't know the user's local preference).
2. The browser downloads the HTML and renders a bright white page.
3. Milliseconds later, React hydrates, reads `localStorage.getItem('user-theme')`, realizes the user wants Dark Mode, and instantly flips the colors.
This causes a violent "Flash" of bright white light before the page turns dark.

**The Solution: An Injectable Blocking Script.**
You must execute a tiny, vanilla JavaScript snippet in the `<head>` of the HTML document. This script runs *before* the browser renders the `<body>`, instantly applying the correct `data-theme`!

```html
<!-- Inside the <head> of your index.html or Next.js _document.js -->
<script>
  (function() {
    try {
      // Check local storage OR the OS preference!
      var localTheme = localStorage.getItem('user-theme');
      var osTheme = window.matchMedia('(prefers-color-scheme: dark)').matches ? 'dark' : 'light';
      
      var finalTheme = localTheme || osTheme;
      document.documentElement.setAttribute('data-theme', finalTheme);
    } catch (e) {}
  })();
</script>
```
*(Note: Modern React frameworks often abstract this via libraries like `next-themes`).*

### Level 4 — Multi-Brand Theming (White Labeling)
What if you need 5 different themes (e.g., a "Halloween" theme, a "Cyberpunk" theme, or a "Client A" theme)?
Because we are using CSS variables, multi-theming is trivial.

```css
/* Theme: Cyberpunk */
[data-theme="cyberpunk"] {
  --bg-primary: #fce7f3; /* Neon Pink */
  --text-primary: #1e3a8a; /* Deep Blue */
  
  /* We can even theme spacing and typography! */
  --font-family: 'Courier New', monospace;
  --border-radius: 0px; /* Sharp, aggressive edges */
}
```
Switching a multi-million dollar web application to a totally different visual identity is now as simple as `document.documentElement.setAttribute('data-theme', 'cyberpunk')`.

## REAL_WORLD
**System Preference vs. User Override**
A professional theming system has three states, not two: `Light`, `Dark`, and `System`.
By default, the application should respect the operating system's setting using the CSS media query `@media (prefers-color-scheme: dark)`.
If the user's OS changes from Light to Dark at sunset, your web app should automatically switch with it!
However, you must provide a toggle button so the user can *override* the system setting if they prefer. If they click the override, you save that preference to `localStorage`, and from that point on, you ignore the OS setting and respect their explicit choice.

## INTERVIEW
**Q: In Dark Mode, our shadows look terrible. In Light Mode, we use a soft gray shadow `box-shadow: 0 4px 6px rgba(0,0,0, 0.1)`. In Dark Mode, the background is dark gray (`#1f2937`), so the 10% black shadow is completely invisible. How do you create depth in Dark Mode?**
A: **Elevation via Lightness, not Shadows.**
Shadows fundamentally do not work on dark backgrounds because you cannot cast a dark shadow on a dark surface.
In Material Design and modern Dark Mode systems, you simulate "depth" (elevation) by manipulating the *Lightness* of the surface, not by adding shadows.
- The base background (Level 0) is very dark (e.g., `#121212`).
- A Card resting on the background (Level 1) is slightly lighter (e.g., `#1e1e1e`).
- A Dropdown Menu floating above the card (Level 2) is even lighter (e.g., `#2c2c2c`).
By progressively lightening the background color of floating elements, you perfectly simulate the effect of a light source hitting elements that are closer to the user!

## FEYNMAN CHECK

### Explain Theming & Dark Mode Like I'm 10
> Dark mode swaps the colour palette without rewriting components. The trick: components reference semantic tokens (`--text-primary`), and dark mode remaps those tokens to dark-palette values. The non-obvious part: CSS custom properties cascade for free without React re-renders — switching themes is one DOM attribute change. This is why React-context theming causes 200ms jank that CSS theming doesn't.

### 5 Deep Questions
**Q1: CSS vars vs React context?** > A: CSS = zero re-renders; context = O(n consumers) re-renders.
**Q2: ONE model.** > A: "Tokens reference palette by semantic name; `[data-theme]` selector remaps."
**Q3: Misconception+code.** > A: Hardcoded `#fff` in components breaks dark mode entirely.
**Q4: SSR/flash?** > A: Sync script in <head> reads localStorage before paint.
**Q5: Senior one-liner.** > A: "Theming is a CSS cascade controlled by data-theme — the React-state approach is the wrong tool."
