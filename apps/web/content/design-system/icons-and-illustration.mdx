## WHY
If developers manually download `.png` or `.svg` icons from random websites, your application will suffer from three massive problems:
1. **Visual Inconsistency:** One icon has rounded corners and a 2px stroke, another is sharp with a 1px stroke. The UI looks fragmented.
2. **Performance:** Loading 50 separate `.svg` files causes 50 separate HTTP network requests, slowing down page loads.
3. **Inflexibility:** A `.png` icon cannot change its color when the user switches to Dark Mode.

An **Iconography System** standardizes how icons are designed, packaged, distributed, and rendered in code, ensuring perfect consistency and maximum performance.

## THEORY
### The Rules of Icon Design
In a strict Design System, icons must adhere to an inflexible mathematical grid (usually a 24x24 pixel grid).
- **Stroke Weight:** Every icon must use the exact same stroke width (e.g., 2px).
- **Corner Radius:** Every icon must use the same corner rounding (e.g., sharp 0px, or rounded 2px).
- **Optical Volume:** A tall, thin icon takes up less visual space than a solid square icon. Icons must be optically balanced within their 24x24 bounding box so they feel equally "heavy."

## CODE

### Level 1 — The Problem with Image Tags
The worst way to use an icon is with the HTML `<img>` tag.

```html
<!-- THE TERRIBLE WAY -->
<!-- 1. Triggers a slow HTTP network request! -->
<!-- 2. You CANNOT change the color of this image via CSS! -->
<img src="/icons/user.svg" alt="User Profile" class="icon" />
```

### Level 2 — Inline SVGs and the 'currentColor' Magic
To manipulate an icon dynamically, you must use **Inline SVGs**. The SVG code is physically injected directly into the HTML DOM.
The secret to Design System icons is the CSS `currentColor` keyword.

```jsx
// React Component Icon (The Standard Way)
export const UserIcon = ({ className }) => {
  return (
    <svg 
      xmlns="http://www.w3.org/2000/svg" 
      viewBox="0 0 24 24" 
      width="24" 
      height="24"
      fill="none" 
      /* MAGIC: By setting stroke to currentColor, the SVG will automatically 
         inherit whatever text color is currently applied to its parent container! */
      stroke="currentColor" 
      strokeWidth="2" 
      strokeLinecap="round" 
      strokeLinejoin="round"
      className={className}
    >
      <path d="M20 21v-2a4 4 0 0 0-4-4H8a4 4 0 0 0-4 4v2" />
      <circle cx="12" cy="7" r="4" />
    </svg>
  );
};

// USAGE: The icon will automatically be RED, because the parent text is red!
<div className="text-red-500 hover:text-blue-500">
  <UserIcon /> 
  <span>Delete Profile</span>
</div>
```

### Level 3 — Icon Sprites (The Performance Optimization)
If your page has 100 icons, injecting 100 massive inline SVGs into the DOM will bloat the HTML file size and slow down React's render cycle. 
For extreme performance, Design Systems use **SVG Sprites**.

An SVG Sprite is a single, hidden SVG file containing ALL the icons in your system, wrapped in `<symbol>` tags.

```html
<!-- The hidden sprite file (loaded once, cached by the browser) -->
<svg style="display: none;">
  <symbol id="icon-user" viewBox="0 0 24 24">
    <path d="..." />
  </symbol>
  <symbol id="icon-settings" viewBox="0 0 24 24">
    <path d="..." />
  </symbol>
</svg>

<!-- THE USAGE -->
<!-- The <use> tag points to the ID in the sprite! -->
<!-- This is blazingly fast, requires zero HTTP requests, keeps the DOM tiny, 
     AND supports 'currentColor'! -->
<svg class="icon">
  <use href="#icon-user"></use>
</svg>
```

### Level 4 — Automated Icon Pipelines (SVGR)
How do designers actually get icons into the codebase?
You absolutely DO NOT want developers manually copying and pasting SVG code from Figma into React components. Designers often export SVGs with useless meta-tags, hardcoded fill colors, and bloated path data.

Modern Design Systems use an automated pipeline with tools like **SVGR** and **SVGO**.

1. The Designer exports raw SVGs into a specific GitHub folder.
2. The CI/CD pipeline detects the new SVGs and runs **SVGO** (SVG Optimizer). It strips out Figma metadata, minimizes the paths, and aggressively shrinks the file size.
3. The pipeline runs **SVGR**. It automatically converts the optimized `.svg` files into perfect, ready-to-use React Components!
4. It dynamically replaces hardcoded colors like `stroke="#000"` with `stroke="currentColor"`.
5. It outputs the final `.tsx` files directly into the component library!

## REAL_WORLD
**Accessibility (Aria-Hidden)**
A massive issue with icons is Screen Readers (software used by visually impaired users).
If you have a button with a Trash Can icon and the word "Delete", a screen reader might read: *"Graphic, path data, Delete"*. This is incredibly annoying.
Icons in a Design System must be rigorously categorized into two types:
1. **Decorative Icons:** The icon is just visual flair next to text. You MUST hide it from screen readers using `aria-hidden="true"`.
2. **Standalone Icons:** A button that ONLY contains an icon (no text). You MUST provide an accessible label, either via `aria-label="Delete"` on the button, or a visually hidden `<span class="sr-only">Delete</span>` inside the button!

## INTERVIEW
**Q: Our marketing site uses massive, complex SVG illustrations (like a character sitting at a desk with plants). They are 150KB each. Should we convert these into React components and bundle them into our JavaScript payload like we do with our small UI icons?**
A: **No, absolutely not.**
UI Icons (like a magnifying glass or a chevron) are tiny (often <1KB), highly reusable, and need to change color dynamically (hover states, dark mode). They belong in the JavaScript bundle as React components or SVG Sprites.
Massive SVG Illustrations are fundamentally different. They are essentially static images. If you bundle 10 complex illustrations into your React app, your JavaScript bundle size will explode by 1.5MB, severely delaying the Time-to-Interactive (TTI) of your application. 
Illustrations should be treated as static assets. They should be hosted on a CDN, heavily cached, and loaded via standard `<img>` tags (or `background-image` in CSS) to ensure they are fetched asynchronously and do not block the parsing of the application's JavaScript.
