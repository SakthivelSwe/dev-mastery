## WHY
If you let 5 different frontend developers pick the color "blue" for their respective buttons, you will end up with `#0000FF`, `#1E90FF`, `#4169E1`, `#4682B4`, and `#87CEEB`. Your application will look like a chaotic, unprofessional mess.

A **Design System** establishes strict constraints. **Color Theory** isn't just about picking colors that look pretty; it's about establishing a semantic, mathematically scalable palette that ensures consistency, accessibility, and brand identity across thousands of pages.

## THEORY
### The Semantic Roles of Color
In a Design System, colors are never named by what they *look* like (e.g., `dark-blue`). They are named by what they *do* (e.g., `primary-600`).
1. **Primary:** The main brand color. Used for primary actions (Save, Submit, active states).
2. **Secondary/Accent:** Used sparingly to highlight secondary actions or specific brand elements.
3. **Neutrals:** Grays, blacks, and whites. These make up 90% of your UI (text, backgrounds, borders).
4. **Semantic (Feedback):** 
   - **Success (Green):** Completed actions.
   - **Warning (Yellow/Orange):** Proceed with caution.
   - **Danger/Error (Red):** Destructive actions (Delete).
   - **Info (Blue):** General neutral information.

## CODE

### Level 1 — The Number Scale (The 50-900 System)
You cannot just have one `primary` color. If you have a primary button (`primary`), what color is it when you hover over it? What color is its subtle background when disabled?
Modern design systems (like Tailwind or Material Design) use a **Lightness Scale**, usually from 50 (lightest) to 900 (darkest).

```css
/* A typical Primary Palette (e.g., Blue) */
:root {
  /* 50-100: Backgrounds for alerts or subtle active states */
  --color-primary-50:  #eff6ff;
  --color-primary-100: #dbeafe;
  
  /* 200-400: Borders, hover states for light buttons */
  --color-primary-200: #bfdbfe;
  --color-primary-300: #93c5fd;
  --color-primary-400: #60a5fa;
  
  /* 500: The base brand color! Used for standard buttons. */
  --color-primary-500: #3b82f6;
  
  /* 600-700: Hover states for base buttons */
  --color-primary-600: #2563eb;
  --color-primary-700: #1d4ed8;
  
  /* 800-900: Text colors on light backgrounds */
  --color-primary-800: #1e40af;
  --color-primary-900: #1e3a8a;
}
```

### Level 2 — HSL vs. Hex Codes
If you need to programmatically generate a color scale, or adjust it slightly for Dark Mode, Hex codes (`#3b82f6`) are terrible because they are completely opaque to human understanding.
Modern CSS uses **HSL (Hue, Saturation, Lightness)** or **OKLCH**.

```css
/* Using HSL makes generating a scale mathematically trivial! */
:root {
  /* Hue (Color): 217 (Blue) */
  /* Saturation: 91% (Vibrant) */
  
  /* We just decrease the Lightness (L) to get darker shades! */
  --color-primary-100: hsl(217, 91%, 90%); /* Light background */
  --color-primary-500: hsl(217, 91%, 60%); /* Base button */
  --color-primary-900: hsl(217, 91%, 20%); /* Dark text */
}

/* Creating an accessible hover state is as easy as subtracting 10% lightness! */
.button-primary:hover {
  background-color: hsl(217, 91%, 50%); 
}
```

### Level 3 — Semantic Mapping (The Alias Layer)
Never use a core palette color directly in your component CSS.
If you write `background-color: var(--color-primary-500);` on your button, what happens when the marketing team decides the brand color should be Purple instead of Blue?
You must create an **Alias Layer**.

```css
/* 1. Core Palette (The raw ingredients) */
:root {
  --blue-500: #3b82f6;
  --blue-600: #2563eb;
  --red-500:  #ef4444;
}

/* 2. Semantic Alias Layer (The meaning) */
/* This maps the brand meaning to the raw ingredient! */
:root {
  --color-brand-base: var(--blue-500);
  --color-brand-hover: var(--blue-600);
  
  --color-danger-base: var(--red-500);
}

/* 3. Component Layer (The usage) */
.btn-primary {
  background-color: var(--color-brand-base); /* PERFECT! */
}
.btn-primary:hover {
  background-color: var(--color-brand-hover);
}
```
Now, if the brand changes to purple, you only change the Alias Layer (`--color-brand-base: var(--purple-500);`), and your entire application updates instantly!

### Level 4 — Tinted Neutrals (The "Premium" Feel)
Why do websites like Stripe or Vercel look so premium, while an internal admin dashboard looks boring?
Admin dashboards use pure gray (`#808080`). Premium designs use **Tinted Neutrals**.
They inject a tiny amount of the brand's primary Hue into the gray palette.

```css
/* Pure Gray (Boring, sterile) */
--gray-900: hsl(0, 0%, 10%); /* 0% Saturation */

/* Slate / Blue-Gray (Premium, cohesive) */
/* We add the Brand Hue (217) and 15% Saturation to the grays! */
--slate-50:  hsl(217, 15%, 96%);
--slate-500: hsl(217, 15%, 50%);
--slate-900: hsl(217, 15%, 15%);

/* Now, use Slate for all your text and borders instead of pure Gray! */
body {
  color: var(--slate-900);
  background-color: var(--slate-50);
}
```

## REAL_WORLD
**Accessibility: WCAG Contrast Ratios**
A Design System is useless if visually impaired users cannot read it. 
The Web Content Accessibility Guidelines (WCAG) require that Text and its Background have a specific mathematical contrast ratio.
- **Normal Text (Level AA):** Must have a contrast ratio of at least `4.5:1`.
- **Large Text / UI Components:** Must have a contrast ratio of at least `3:1`.

If your Primary Button is `#3b82f6` (Lightness 60%), and you put White text (`#ffffff`) on it, the contrast ratio is only `3.2:1`. **This fails accessibility testing.** You must either make the button darker (`--primary-700`), or change the text color to Black.
Modern Design System color palettes are strictly mathematically tested. Usually, colors from `50` to `400` require black text, and `500` to `900` require white text.

## INTERVIEW
**Q: The marketing team wants a "Dark Mode". Should we just write CSS that inverts all the colors on the screen?**
A: **No, simple inversion creates terrible user interfaces.**
Dark mode is not an inverted Light mode. For example, in Light Mode, a shadow is dark gray. If you invert it, it becomes a glowing white aura. In Light Mode, you use a white background with dark gray cards. In Dark Mode, you do *not* use a pure black background with bright white cards; you use a dark slate background (`900`) with slightly lighter slate cards (`800`) to create depth.
To implement Dark Mode correctly in a Design System, you must use CSS Variables at the **Semantic Alias Layer**. 
```css
/* Light Mode (Default) */
:root {
  --bg-surface: var(--slate-50);
  --text-primary: var(--slate-900);
}

/* Dark Mode Override */
@media (prefers-color-scheme: dark) {
  :root {
    --bg-surface: var(--slate-900); /* Cards become dark */
    --text-primary: var(--slate-50); /* Text becomes light */
  }
}
```
The component CSS (`background: var(--bg-surface)`) never changes!

## FEYNMAN CHECK

### Explain Color Theory Like I'm 10
> Colours have three properties: hue (red vs blue), saturation (vivid vs grey), and lightness (light vs dark). A design system picks one base hue, then generates a 10-step scale by varying lightness. The non-obvious part: equal lightness steps look uneven to the eye — perceptually-uniform colour spaces (OKLCH, LCH) fix this. This is why `hsl()` palettes look "wrong" in the mid range.

---

### 5 Deep Questions
**Q1: HSL vs OKLCH?** > A: HSL is fast but perceptually non-uniform; OKLCH steps look evenly spaced to humans.
**Q2: ONE model.** > A: "Hue defines brand. Saturation defines vibrance. Lightness drives the scale. Pick a perceptual space."
**Q3: Misconception+code.** > A: ```css
/* ❌ HSL — mid-range looks muddy */
background: hsl(220 80% 50%);
/* ✅ OKLCH */
background: oklch(60% 0.2 240);
```
**Q4: With accessibility?** > A: Lightness directly determines contrast — design lightness pairs for WCAG 4.5:1.
**Q5: Senior one-liner.** > A: "Color theory operationalised for design systems = hue families × perceptually-uniform lightness scale × verified contrast pairs."
