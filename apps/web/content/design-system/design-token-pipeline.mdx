## WHY

In a traditional workflow, designers create a UI in Figma, hand it off to developers, and the developers manually copy HEX codes, font sizes, and border radii into CSS. When the designer inevitably tweaks the brand primary color a month later, developers have to hunt through the codebase with "Find and Replace", inevitably missing a few hardcoded `#3b82f6` values.

This manual handoff is a massive source of friction, bugs, and UI inconsistency. 

A **Design Token Pipeline** solves this by establishing a "Single Source of Truth". Designers define the tokens in Figma. An automated pipeline extracts those tokens and compiles them into CSS Variables, iOS Swift variables, and Android XML resources. When the designer updates a color in Figma, the code updates automatically via CI/CD.

---

## THEORY

### What are Design Tokens?
Design tokens are the smallest, indivisible pieces of your design system (Colors, Typography, Spacing, Shadows, Border Radii). They are stored in a platform-agnostic format, universally as **JSON**.

Instead of writing `color: #ff0000`, you write `color: {brand.primary.500}`. 

### The Toolchain: Style Dictionary
To translate JSON tokens into platform-specific code, the industry standard tool is **Amazon Style Dictionary**. It takes a raw JSON file containing token definitions and runs it through a build pipeline.
- For Web: It outputs `variables.css` or SCSS variables.
- For iOS: It outputs `Color.swift` or `.plist` files.
- For Android: It outputs `colors.xml` and `dimens.xml`.

## IMPLEMENTATION

### 1. The Token JSON Structure (The W3C Standard)
Here is an example of raw tokens exported from Figma (often using a plugin like *Tokens Studio*).

```json
// tokens/colors.json
{
  "color": {
    "primary": {
      "500": { "value": "#3b82f6", "type": "color" },
      "600": { "value": "#2563eb", "type": "color" }
    },
    "surface": {
      "default": { "value": "#ffffff", "type": "color" },
      "muted": { "value": "#f3f4f6", "type": "color" }
    }
  },
  "spacing": {
    "sm": { "value": "8px", "type": "spacing" },
    "md": { "value": "16px", "type": "spacing" }
  }
}
```

### 2. Style Dictionary Configuration
We configure Style Dictionary to read the JSON files and generate the platform artifacts.

```javascript
// build-tokens.js
const StyleDictionary = require('style-dictionary');

const sd = StyleDictionary.extend({
  source: ['tokens/**/*.json'],
  platforms: {
    // 1. Web Output (CSS Variables)
    css: {
      transformGroup: 'css',
      buildPath: 'packages/tokens/dist/css/',
      files: [{
        destination: 'variables.css',
        format: 'css/variables'
      }]
    },
    
    // 2. iOS Output (Swift classes)
    ios: {
      transformGroup: 'ios-swift',
      buildPath: 'packages/tokens/dist/ios/',
      files: [{
        destination: 'StyleDictionary.swift',
        format: 'ios-swift/class.swift'
      }]
    },
    
    // 3. Android Output (XML Resources)
    android: {
      transformGroup: 'android',
      buildPath: 'packages/tokens/dist/android/',
      files: [{
        destination: 'colors.xml',
        format: 'android/colors'
      }]
    }
  }
});

sd.buildAllPlatforms();
```

### 3. The Output
When you run `node build-tokens.js`, it automatically generates:

**Web (variables.css):**
```css
:root {
  --color-primary-500: #3b82f6;
  --color-primary-600: #2563eb;
  --color-surface-default: #ffffff;
  --spacing-md: 16px;
}
```

**Android (colors.xml):**
```xml
<?xml version="1.0" encoding="UTF-8"?>
<resources>
  <color name="color_primary_500">#ff3b82f6</color>
  <color name="color_primary_600">#ff2563eb</color>
</resources>
```

## SYSTEM_DESIGN

### The Fully Automated CI/CD Handoff
To truly act like a massive enterprise tech company, you must remove humans from the handoff process entirely.

1. **Design Phase:** The Designer updates the Primary Color from Blue to Indigo in Figma.
2. **Push to GitHub:** The Designer clicks "Push to GitHub" inside the Figma *Tokens Studio* plugin.
3. **Pull Request:** This creates a Pull Request in the company's `design-system` GitHub repository. The PR contains the updated `tokens.json` file.
4. **CI Build:** GitHub Actions detects the PR, runs `npm run build:tokens`, and automatically generates the new CSS, Swift, and XML files.
5. **Review & Merge:** The engineering team reviews the PR. Upon merging, a new NPM package version (e.g., `@my-company/design-tokens v2.1.0`) is published automatically.
6. **Consumption:** The web application's Dependabot automatically creates a PR to bump the token version. The application is redeployed, and the UI is instantly updated to Indigo.

This pipeline ensures that design and code are inextricably linked and mathematically synchronized.
