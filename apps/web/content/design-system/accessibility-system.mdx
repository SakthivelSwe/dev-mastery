## WHY

Accessibility (A11y) is not a feature you add at the end of a project. It is a fundamental architecture requirement. In many countries, WCAG 2.1 AA compliance is a strict legal requirement for enterprise and government software.

If your Design System is fundamentally inaccessible (e.g., buttons lack focus rings, modals don't trap focus, color contrasts fail), every single application built using your system will be legally non-compliant and unusable by millions of people who rely on screen readers or keyboard navigation.

Building an accessible Design System means baking WCAG compliance into the lowest-level primitives: color tokens, focus states, and component ARIA roles.

---

## THEORY

### The Three Pillars of Component Accessibility
To make a component fully accessible, it must pass three tests:
1. **Keyboard Navigation:** Every interactive element MUST be reachable via the `Tab` key. Custom components (like dropdowns or tabs) must handle `Arrow`, `Enter`, `Space`, and `Esc` keys according to W3C Authoring Practices.
2. **Screen Reader Context (ARIA):** Visual cues mean nothing to a screen reader. A red border on an input must be accompanied by `aria-invalid="true"`. A loading spinner must be announced via `aria-live="polite"`.
3. **Visual Contrast:** Text and interactive UI elements must maintain a contrast ratio of at least 4.5:1 against their backgrounds (3:1 for large text). 

### ARIA is a Promise (Do not break it)
The first rule of ARIA is: **No ARIA is better than bad ARIA.** 
If you add `role="menu"` to a `<ul>`, you are making a promise to the screen reader that the user can navigate the list items using their UP/DOWN arrow keys. If you haven't actually written the JavaScript to handle those keyboard events, the screen reader user will be completely trapped. If you don't have the JS, just leave it as a standard `<ul>`.

## IMPLEMENTATION

### 1. High-Contrast Focus Indicators
Browsers provide default focus rings, but designers often hate them. The worst thing you can do is `outline: none;`. The correct approach is to provide a beautiful, custom focus ring that only appears for keyboard users via `:focus-visible`.

```css
/* accessibility.css */

/* Global reset for mouse users */
:focus:not(:focus-visible) {
  outline: none;
}

/* Strict, highly visible focus ring for keyboard users */
:focus-visible {
  outline: 3px solid var(--color-primary-500);
  outline-offset: 2px;
  box-shadow: 0 0 0 4px var(--color-primary-100);
  border-radius: 2px;
}
```

### 2. The VisuallyHidden Utility
Sometimes you need to provide text strictly for screen readers (e.g., an icon-only close button). You cannot use `display: none` or `visibility: hidden` because screen readers ignore those! You must use the `VisuallyHidden` CSS pattern.

```tsx
import styled from 'styled-components';

export const VisuallyHidden = styled.span`
  position: absolute;
  width: 1px;
  height: 1px;
  padding: 0;
  margin: -1px;
  overflow: hidden;
  clip: rect(0, 0, 0, 0);
  white-space: nowrap;
  border: 0;
`;

// Usage:
// <button>
//   <Icon name="x" />
//   <VisuallyHidden>Close Modal</VisuallyHidden>
// </button>
```

### 3. Accessible Accordion (Disclosure Pattern)
Let's build a fully accessible Accordion according to the W3C Disclosure pattern. Notice the strict use of `aria-expanded` and `aria-controls`.

```tsx
import React, { useState, useId } from 'react';

export const Accordion = ({ title, children }) => {
  const [isOpen, setIsOpen] = useState(false);
  const contentId = useId();
  const buttonId = useId();

  return (
    <div className="accordion-wrapper">
      <h3>
        <button
          id={buttonId}
          aria-expanded={isOpen}
          aria-controls={contentId}
          onClick={() => setIsOpen(!isOpen)}
          className="accordion-trigger"
        >
          {title}
          <span className="icon" aria-hidden="true">
            {isOpen ? '▼' : '▶'}
          </span>
        </button>
      </h3>
      
      <div
        id={contentId}
        role="region"
        aria-labelledby={buttonId}
        hidden={!isOpen}
        className="accordion-content"
      >
        {children}
      </div>
    </div>
  );
};
```

## SYSTEM_DESIGN

### Automated Accessibility Auditing Pipeline
You cannot rely purely on manual QA for accessibility. A mature Design System enforces accessibility at the CI/CD pipeline level.

1. **Linting:** Enforce `eslint-plugin-jsx-a11y` in your repository. It will break the build if a developer adds an `onClick` to a `<div>` without giving it a `role="button"` and `tabIndex={0}`.
2. **DOM Auditing:** Use `axe-core` in your component integration tests (e.g., Jest + React Testing Library).
3. **E2E Testing:** Integrate Cypress Axe to scan the fully rendered application routes for contrast and ARIA violations before any code is merged to `main`.

```javascript
// Example Jest Test using jest-axe
import { render } from '@testing-library/react';
import { axe, toHaveNoViolations } from 'jest-axe';
import { Button } from './Button';

expect.extend(toHaveNoViolations);

it('Button should have no accessibility violations', async () => {
  const { container } = render(<Button>Click Me</Button>);
  const results = await axe(container);
  
  expect(results).toHaveNoViolations();
});
```
