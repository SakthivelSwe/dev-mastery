## WHY
In traditional web development, to test if a complex "Checkout Button" works, you have to: start your entire backend, run your frontend, navigate to the cart page, add an item, go to checkout, and *finally* see your button. What if you want to see what it looks like when it's in a "loading" state? Or a "disabled" state? You have to manually mock backend data or click around the UI endlessly.

**Storybook** is an industry-standard open-source tool that allows you to develop, test, and document UI components in complete **isolation** from the main application. 

## THEORY
### Component-Driven Development (CDD)
Storybook enforces Component-Driven Development. You build UIs from the bottom up:
1. Build individual components (Button, Input, Avatar) in isolation.
2. Combine them into composite components (LoginForm, Header).
3. Assemble those into pages (LoginPage, Dashboard).

A "Story" captures a single specific state of a UI component. A Button component might have 4 stories: `Primary`, `Secondary`, `Loading`, and `Disabled`.

## CODE

### Level 1 — Beginner (Writing a Story)
```javascript
// Button.stories.js (React Example)
import { Button } from './Button';

// 1. Default export defines the component and metadata
export default {
  title: 'Components/Button',
  component: Button,
  // Add auto-generated documentation for the props!
  tags: ['autodocs'],
};

// 2. Named exports define the individual "Stories" (states)
export const Primary = {
  args: {
    primary: true,
    label: 'Submit Order',
  },
};

export const Secondary = {
  args: {
    primary: false,
    label: 'Cancel',
  },
};

export const Large = {
  args: {
    size: 'large',
    label: 'Click Me!',
  },
};

export const Disabled = {
  args: {
    disabled: true,
    label: 'Not Allowed',
  },
};
```

### Level 2 — Building Blocks (Interactive Controls)
Storybook automatically generates a UI panel where you can interactively tweak the props (Arguments/Args) of your component in real-time.

```javascript
// You can explicitly define what controls appear in the Storybook UI
export default {
  title: 'Components/Alert',
  component: Alert,
  argTypes: {
    // Creates a dropdown menu in Storybook for the 'severity' prop
    severity: {
      control: 'select',
      options: ['info', 'success', 'warning', 'error']
    },
    // Creates a color picker in Storybook
    backgroundColor: { control: 'color' },
    // Creates a toggle switch
    isClosable: { control: 'boolean' }
  }
};

export const WarningAlert = {
  args: {
    severity: 'warning',
    message: 'Your password expires in 3 days.',
    isClosable: true
  }
};
```

### Level 3 — Intermediate (Mocking Data and Contexts)
Components often depend on external context (Routing, Redux, API calls). Storybook uses **Decorators** to wrap stories with the necessary providers.

```javascript
// .storybook/preview.js (Global config applied to ALL stories)
import { ThemeProvider } from 'styled-components';
import { theme } from '../src/theme';
import { BrowserRouter } from 'react-router-dom';

export const decorators = [
  (Story) => (
    // Wrap every component in the Theme and Router providers!
    <BrowserRouter>
      <ThemeProvider theme={theme}>
        <div style={{ margin: '3em' }}>
          <Story />
        </div>
      </ThemeProvider>
    </BrowserRouter>
  ),
];
```
To mock API calls inside isolated components, Storybook natively integrates with **MSW (Mock Service Worker)**, intercepting network requests at the browser level and returning mock JSON.

### Level 4 — Advanced (Visual Regression Testing)
When you have 500 components, how do you know a CSS change in `Button.css` didn't accidentally break the alignment of `UserProfileCard`?

**Chromatic** (created by the Storybook team) automates this:
1. Every time you push a PR, Chromatic boots up your Storybook in the cloud.
2. It takes a screenshot of EVERY single story.
3. It compares the new screenshots to the baseline (master branch) pixel-by-pixel.
4. If a button shifted 2 pixels to the left, it flags the PR for visual review.

This completely eliminates "CSS Regressions" in large design systems!

## REAL_WORLD
Storybook is used by nearly every major tech company (Airbnb, Github, Slack, Microsoft) to document their Design Systems. 

A standard enterprise workflow:
1. Designers create a new Component in Figma.
2. Frontend Engineers build it in isolation in Storybook.
3. It goes through Visual Review (Chromatic).
4. Once approved, the Storybook is published as a static website (the "Component Library Documentation") for other engineers in the company to reference.
5. Finally, the component is integrated into the actual application.

## INTERVIEW
**Q: Doesn't maintaining Storybook slow down development?**
A: In the very short term (a 1-week hackathon), yes. For any project lasting longer than a month, it massively accelerates development. You don't have to spin up backend services, you don't have to navigate through the app to find the UI state, and it forces you to write decoupled, pure components. Tightly coupled components (e.g., a Button that fetches its own data from a Redux store) are incredibly painful to put in Storybook — which is a great architectural smell indicating you should refactor!

## FEYNMAN CHECK

### Explain Storybook Like I'm 10 Years Old
> Storybook is a showroom for components — every component lives in its own window, in every possible state, without needing a full app around it. You can click through the "loading state" story, the "empty state" story, and the "error state" story without setting up a database. The non-obvious part: Storybook forces you to think about what states your component has, because you must write them explicitly as stories. A component with 47 undocumented states gets exposed immediately. This is why Storybook is as much a design tool as a documentation tool.

---

### 5 Deep Questions
**Q1: What is the "story" mental model?**
> **A:** A story is a named, reproducible snapshot of a component in a specific state with specific props. It is a living documentation entry AND an isolated test case. The discipline: every meaningful prop combination that represents a real product use case should have a story. "Loading", "Empty", "Error", "Populated", "Disabled" are stories, not developer afterthoughts.

**Q2: ONE model.**
> **A:** "Story = named prop set + optional play function. Component catalogue = all stories. Tests = play functions + snapshot tests from stories."

**Q3: Misconception with code.**
> **A:** Only writing the happy-path story:
> ```tsx
> // ❌ Only happy path — loading/empty/error never documented or tested
> export const Default: Story = { args: { data: MOCK_DATA } };
> // ✅ All meaningful states
> export const Loading: Story = { args: { isLoading: true } };
> export const Empty: Story   = { args: { data: [] } };
> export const WithError: Story = { args: { error: 'Failed to load' } };
> ```

**Q4: How do play functions enable interaction testing?**
> **A:** Play functions run after the story renders, using Testing Library to simulate user interactions and assert outcomes. `userEvent.click(button)` inside a play function verifies the component responds correctly — the test runs in a real browser, not jsdom, catching visual and interaction bugs that unit tests miss.

**Q5: Senior one-liner.**
> **A:** "Storybook is a component workbench that forces exhaustive state documentation through stories, enabling visual regression, interaction testing, and design system governance from a single source of truth."

## BUILD

### 🏗️ Mini Project: Storybook with All Component States + Interaction Test

**What you will build:** A Button component with 6 stories covering all states, plus a play function interaction test.
**Time estimate:** 35 minutes

#### Step 1 — Setup
```bash
npx create-next-app@latest storybook-demo --typescript --tailwind --no-src-dir --app
cd storybook-demo && npx storybook@latest init
touch stories/Button.stories.tsx
```

#### Step 2 — Stories
```tsx
// stories/Button.stories.tsx
import type { Meta, StoryObj } from '@storybook/react';
import { Button } from '../components/Button';
import { userEvent, within, expect } from '@storybook/test';

const meta: Meta<typeof Button> = {
  title: 'Design System/Button',
  component: Button,
  parameters: { layout: 'centered' },
  argTypes: { variant: { control: 'select', options: ['primary','secondary','destructive','plain'] } },
};
export default meta;
type Story = StoryObj<typeof Button>;

export const Primary: Story   = { args: { variant: 'primary',     children: 'Save changes' } };
export const Secondary: Story = { args: { variant: 'secondary',   children: 'Cancel' } };
export const Destructive: Story={ args: { variant:'destructive',  children: 'Delete account' } };
export const Loading: Story   = { args: { loading: true,          children: 'Saving…' } };
export const Disabled: Story  = { args: { disabled: true,         children: 'Unavailable' } };

export const ClickTest: Story = {
  args: { children: 'Click me', variant: 'primary' },
  play: async ({ canvasElement }) => {
    const canvas = within(canvasElement);
    const btn = canvas.getByRole('button', { name: 'Click me' });
    await userEvent.click(btn);
    await expect(btn).toBeVisible();
  },
};
```

#### Step 5 — Run
```bash
npm run storybook        # visual catalogue
npm run test-storybook   # runs all play functions
```

**Expected Output:**
```
✓ Button/Primary
✓ Button/Loading
✓ Button/ClickTest — play function passed
```
