## WHY
If you build a `UserProfileCard` component, and later the team needs a `ProductCard`, a developer might just copy-paste the HTML, change the variable names, and tweak the CSS. Six months later, you have 15 different "Card" components in your codebase. When the design team decides to round the corners of all cards from `4px` to `8px`, you have to find and update 15 different files.

**Atomic Design** (coined by Brad Frost) is a methodology for breaking UIs down into hierarchical, reusable building blocks. By understanding the anatomy of a component, you guarantee reusability, eliminate code duplication, and enforce visual consistency.

## THEORY
### The Atomic Design Hierarchy
1. **Atoms:** The absolute lowest level. They cannot be broken down further. (e.g., A single `<Button>`, an `<Input>`, an `<Icon>`, a `<Typography>` label).
2. **Molecules:** Groups of Atoms bonded together to do ONE specific thing. (e.g., A "Search Bar" Molecule = An `<Input>` Atom + a `<Button>` Atom).
3. **Organisms:** Complex UI sections composed of Molecules and Atoms. (e.g., A "Header Navigation" Organism = A Logo Atom + a Search Bar Molecule + a User Profile Molecule).
4. **Templates:** Page-level layouts containing Organisms, but with no actual data (just placeholders).
5. **Pages:** Specific instances of Templates populated with real data from the database.

## CODE

### Level 1 — Building an Atom (The Button)
An Atom must be incredibly flexible. It shouldn't know *what* it's doing, only *how it looks*.
It should accept props for variants (Primary/Secondary), sizes (Small/Large), and state (Disabled/Loading).

```jsx
// React Atom Example
import styles from './Button.module.css';

export const Button = ({ 
  children, 
  variant = 'primary', 
  size = 'md', 
  isLoading = false,
  ...props // Pass through native HTML attributes!
}) => {
  
  // Dynamically build the CSS classes based on the props!
  const classNames = `
    ${styles.base} 
    ${styles[variant]} 
    ${styles[size]} 
    ${isLoading ? styles.loading : ''}
  `;

  return (
    <button className={classNames} disabled={isLoading} {...props}>
      {isLoading && <Spinner className={styles.icon} />}
      {children}
    </button>
  );
};
```

### Level 2 — Building a Molecule (The Input Field)
A bare `<input>` tag is rarely used alone. It usually needs a Label above it, and an Error Message below it. This grouping forms a standard **Molecule**.

```jsx
// A complex Molecule composed of 3 Atoms!
export const FormField = ({ label, error, children }) => {
  return (
    <div className="flex flex-col gap-2">
      {/* Atom 1: The Label */}
      <Label className="text-sm font-medium">{label}</Label>
      
      {/* Atom 2: The actual Input (passed as children for flexibility) */}
      {children} 
      
      {/* Atom 3: The Error Message (Only renders if an error exists) */}
      {error && <span className="text-red-500 text-xs">{error}</span>}
    </div>
  );
};

// USAGE:
<FormField label="Email Address" error="Invalid email format">
  <Input type="email" placeholder="john@example.com" />
</FormField>
```

### Level 3 — Component Anatomy (Slots and Composition)
The biggest mistake in component design is creating "God Components." 
Imagine a `Card` component. You pass it `title`, `description`, `imageUrl`, and `buttonText`. 
But what if one page needs the Image at the *bottom* instead of the top? You have to add an `imagePosition` prop. What if another page needs *two* buttons? You add `buttonTwoText`. The component becomes a bloated nightmare.

**The Solution: Composition (Slots/Compound Components).**
Instead of one massive component, you export the anatomical *pieces* of the Card, and let the developer assemble them!

```jsx
// We export the anatomical pieces!
export const Card = ({ children }) => <div className="card-base">{children}</div>;
Card.Header = ({ children }) => <div className="card-header">{children}</div>;
Card.Body = ({ children }) => <div className="card-body">{children}</div>;
Card.Footer = ({ children }) => <div className="card-footer">{children}</div>;

// USAGE: The developer has ultimate flexibility over the layout!
<Card>
  <Card.Header>
    <h3>Product Name</h3>
  </Card.Header>
  <Card.Body>
    <img src="/shoes.jpg" alt="Shoes" />
    <p>These shoes are great.</p>
  </Card.Body>
  <Card.Footer>
    {/* I can put TWO buttons here easily, without changing the Card code! */}
    <Button>Buy</Button>
    <Button variant="ghost">Save for later</Button>
  </Card.Footer>
</Card>
```

### Level 4 — Headless Components
*Advanced Pattern:* What if the Design System needs to support React, Vue, and plain HTML? Or what if three different brands use the system, and they all look completely different, but need the exact same keyboard accessibility logic for a Dropdown Menu?

**Headless Components** (like Radix UI or Headless UI) provide the *logic, state, and accessibility (ARIA)* of a complex component, but render **absolutely zero CSS**. 
They are invisible brains. You wrap them around your own DOM elements to give them superpowers.

```jsx
// Using a Headless Dialog (Modal)
import { Dialog } from '@headlessui/react';

// The Headless Component handles the complex logic:
// 1. Trapping keyboard focus inside the modal.
// 2. Closing when hitting the 'Escape' key.
// 3. Managing the 'isOpen' state.
// BUT you write 100% of the CSS!

<Dialog open={isOpen} onClose={() => setIsOpen(false)}>
  {/* I can style this exact overlay however my specific brand needs! */}
  <div className="fixed inset-0 bg-black/50" aria-hidden="true" />
  
  <Dialog.Panel className="bg-white rounded-xl p-8">
    <Dialog.Title>Deactivate account</Dialog.Title>
    <p>Are you sure you want to deactivate your account?</p>
  </Dialog.Panel>
</Dialog>
```

## REAL_WORLD
**Prop Drilling vs. Context in Design Systems**
If you build a complex Organism like a `<Table>`, and you need to pass a `compactMode={true}` boolean down to every single `<td>` Atom so they reduce their padding, passing that prop through 4 layers of components (Prop Drilling) is exhausting.
Design Systems heavily utilize **React Context** internally. The parent `<Table>` component provides a Context containing `{ isCompact: true }`. The deepest `<TableCell>` Atoms simply read from that Context directly, instantly adapting their padding without requiring the developer to pass the prop manually!

## INTERVIEW
**Q: We need an `Avatar` component that displays a user's profile picture. If the picture fails to load, it should display their Initials. How would you design the API for this Atom?**
A: **A highly abstracted, robust interface.**
A naive implementation would just take an `imageUrl`. But to fulfill the requirements, the Component API should accept `src` (the image URL), `name` (to calculate the initials), and `size`. 
Internally, the component uses the native `onError` event of the `<img>` tag. If `onError` fires, the component flips an internal `imgFailed` state to `true`. When true, it unmounts the `<img>` tag and renders a `<div>` containing the first letters of the `name` prop, perfectly encapsulating the fallback logic inside the Atom so no other developer ever has to write it again!

## FEYNMAN CHECK

### Explain Component Anatomy Like I'm 10
> Every component is built from parts: container, content, decoration, interactive zones. A button = box + label + icon slot + focus ring. A card = container + header + body + footer slots. Naming the parts lets designers and devs talk the same language. The non-obvious part: components fail when "parts" leak across slots — putting padding on the body instead of the container produces inconsistent spacing.

---

### 5 Deep Questions
**Q1: Why name the parts?** > A: Common vocabulary for design/dev handoff; matches BEM/CSS naming.
**Q2: ONE model.** > A: "Container + slots + decoration. Every styling rule belongs to exactly one."
**Q3: Misconception+code.** > A: ```tsx
// ❌ Padding on body breaks alignment
// ✅ Padding on container, body fills available space
```
**Q4: With component API?** > A: Slots become compound sub-components (Card.Header, Card.Body).
**Q5: Senior one-liner.** > A: "Component anatomy is the named-parts contract that aligns Figma layers with React composition."
