## WHY
If you are building a `<DatePicker>` component, how do you test it?
Historically, a developer would spin up the entire `localhost:3000` web application, log in with test credentials, navigate to the User Settings page, find the specific form where the DatePicker lives, and click it. 
If they want to see what the DatePicker looks like in an "Error" state, they have to manually trigger a fake API error in the application. This is incredibly slow and fragile.

**Storybook** is an isolated development environment (a sandbox) for UI components. It allows you to build, view, and test the `<DatePicker>` completely separated from the complex business logic, routing, and APIs of the main application.

## THEORY
### Component-Driven Development (CDD)
CDD is a methodology where you build UIs from the bottom up.
1. You build the `<Button>` in isolation.
2. You build the `<Input>` in isolation.
3. You combine them to build the `<SearchForm>` in isolation.
4. Finally, you drop the complete `<SearchForm>` into the actual Next.js application page.

By building in isolation, you are forced to decouple the UI from the business logic. If your `<Button>` requires a Redux store or a React Router context to render, Storybook will crash, instantly exposing that you built a poorly architected, tightly-coupled component!

## CODE

### Level 1 — Writing Your First Story
A "Story" is simply a specific state of a component. 
If you have a Button, you should write a Story for the Primary state, the Secondary state, and the Disabled state.

Storybook uses the **Component Story Format (CSF)**, which is just standard ES Modules.

```jsx
// Button.stories.jsx
import { Button } from './Button';

// 1. Default Export: Metadata about the component
export default {
  title: 'Components/Button', // Where it lives in the Storybook sidebar
  component: Button,
};

// 2. Named Exports: The actual Stories!
// Story 1: The Primary State
export const Primary = () => <Button variant="primary">Click Me</Button>;

// Story 2: The Secondary State
export const Secondary = () => <Button variant="secondary">Cancel</Button>;

// Story 3: The Disabled State
export const Disabled = () => <Button disabled>Processing...</Button>;
```

### Level 2 — Args and Controls (Interactive Sandboxes)
Writing 50 different Named Exports for every possible combination of props is exhausting.
Storybook **Controls** allow you to write ONE base template. Storybook automatically reads your React component's Props (via PropTypes or TypeScript interfaces) and generates a beautiful UI panel with dropdowns, toggles, and color pickers. Designers can interactively change the props and watch the component update in real-time!

```jsx
// Button.stories.jsx
import { Button } from './Button';

export default {
  title: 'Components/Button',
  component: Button,
  // We can explicitly define the Controls UI!
  argTypes: {
    variant: {
      control: 'select',
      options: ['primary', 'secondary', 'danger'],
    },
    onClick: { action: 'clicked' } // Logs clicks in the 'Actions' panel!
  },
};

// Create a Template that accepts 'args' (props)
const Template = (args) => <Button {...args} />;

// Bind the Template to a Story and provide default arguments
export const Interactive = Template.bind({});
Interactive.args = {
  variant: 'primary',
  size: 'md',
  children: 'Interactive Button',
  disabled: false,
};
```

### Level 3 — Decorators (Context and Layout)
Some components need a specific environment to render correctly. 
For example, a Modal component might look terrible if it renders flush against the top-left corner of the iframe. A Theme Toggle component might absolutely require a `ThemeProvider` context wrapper to function.

**Decorators** allow you to wrap Stories with arbitrary React code.

```jsx
// Wrapping all stories in a centered div and a Theme Provider!
export default {
  title: 'Components/Modal',
  component: Modal,
  decorators: [
    (Story) => (
      <ThemeProvider theme="light">
        <div style={{ padding: '3rem', display: 'flex', justifyContent: 'center' }}>
          {/* This renders the actual component story inside the wrapper! */}
          <Story />
        </div>
      </ThemeProvider>
    ),
  ],
};
```

### Level 4 — Visual Regression Testing (Chromatic)
Unit testing a Button (`expect(button).toBeInTheDocument()`) is easy. But how do you test that the padding is exactly `16px`, or that the background color isn't accidentally overlapping the text? You cannot easily unit test CSS.

**Visual Regression Testing** is the superpower of Storybook.
Tools like **Chromatic** (built by the Storybook team) integrate into your CI/CD pipeline.
1. A developer submits a Pull Request changing some CSS.
2. Chromatic automatically boots up Storybook in the cloud.
3. It takes pixel-perfect screenshots of all 500 of your component stories using headless browsers (Chrome, Safari, Firefox).
4. It mathematically compares the new screenshots against the "Baseline" screenshots from the `main` branch.
5. If it detects a 1-pixel difference in the Button's padding, the CI/CD pipeline fails! The PR is blocked until a Designer explicitly reviews the visual diff and clicks "Approve" or "Reject".

## REAL_WORLD
**Storybook as Documentation (Living Guidelines)**
Historically, companies maintained separate "Design Guidelines" websites. They required manual updating and were always out of sync with the actual codebase.
Modern teams use Storybook with the **Docs Addon** (MDX).
You write standard Markdown documentation, but you *embed* live React components directly into the document! Because the documentation is literally rendering the actual source code, the documentation can never be out of sync. When you run `npm run build-storybook`, it outputs a static HTML website that acts as the ultimate, interactive Single Source of Truth for both Designers and Developers.

## INTERVIEW
**Q: I am trying to build a Storybook Story for my `<UserProfile>` component. However, the component contains an Apollo GraphQL `useQuery` hook to fetch the user's data. When I load the Story, it crashes with a Network Error because Storybook isn't connected to our backend database. How do I fix this?**
A: **Mocking Network Requests (MSW).**
You should never connect Storybook to a real backend API or database. Storybook must be deterministic.
To solve this, we use **Mock Service Worker (MSW)**. MSW is an addon that intercepts outgoing HTTP/GraphQL requests at the network level (using Service Workers) and returns fake JSON data!
```jsx
// UserProfile.stories.jsx
import { UserProfile } from './UserProfile';
import { rest } from 'msw';

export default {
  title: 'Pages/UserProfile',
  component: UserProfile,
};

export const SuccessState = () => <UserProfile />;

// We attach the mock data directly to the Story!
SuccessState.parameters = {
  msw: {
    handlers: [
      rest.get('/api/user', (req, res, ctx) => {
        // MSW intercepts the fetch() and returns this fake JSON instantly!
        return res(ctx.json({ name: 'John Doe', role: 'Admin' }));
      }),
    ],
  },
};
```
By providing the mock data, the component renders perfectly in Storybook, allowing you to test the exact UI layout without needing a real backend!

## FEYNMAN CHECK

### Explain Storybook Basics Like I'm 10
> Storybook is a development sandbox where each component lives in isolation, in every state. You write "stories" — one per prop combination — and Storybook renders them in a browsable UI. The non-obvious part: stories double as visual regression tests (Chromatic) and as interaction tests (play functions). One artefact, three uses. This is why every serious design system uses it.
