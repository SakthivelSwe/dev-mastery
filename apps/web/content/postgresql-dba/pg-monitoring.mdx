## WHY
"The database is slow." This is the most terrifying phrase a DBA can hear. 
If the CPU hits 100%, is it because there is a massive sudden spike in web traffic? Or did a junior developer deploy a single bad SQL query that is missing an index and doing a full table scan 500 times a second?
Without robust monitoring and telemetry, you are flying blind. PostgreSQL provides an incredibly powerful, native statistics collector that tracks every single byte, block, and query executed in the engine.

## THEORY
### The Statistics Collector
PostgreSQL maintains a rich set of internal system views (prefixed with `pg_stat_`). These views are continuously updated in memory.
- `pg_stat_activity`: Shows exactly what every open connection is doing *right now*.
- `pg_stat_database`: Shows aggregate statistics for the entire database (cache hit ratios, deadlocks, commits).
- `pg_stat_user_tables`: Shows how often specific tables are queried, updated, or sequentially scanned.
- `pg_stat_statements`: (An extension) Tracks the historical performance of *every unique query* ever executed.

## CODE

### Level 1 — Real-Time Activity (`pg_stat_activity`)
When the database suddenly spikes, your first reflex is to see what queries are currently running.

```sql
-- Find all queries that have been running for more than 5 minutes!
SELECT 
    pid, 
    usename, 
    state, 
    now() - query_start AS duration, 
    query 
FROM pg_stat_activity 
WHERE 
    state != 'idle' 
    AND now() - query_start > interval '5 minutes'
ORDER BY duration DESC;
```
*If a query is stuck in the `active` state for 2 hours, it is likely a forgotten transaction holding aggressive locks. You use `pg_terminate_backend(pid)` to kill it.*

### Level 2 — Cache Hit Ratio
PostgreSQL relies heavily on `shared_buffers` (RAM) to serve data quickly. If Postgres has to read from the physical Hard Drive (Disk), performance plummets. 
The Cache Hit Ratio tells you exactly what percentage of data is being served from RAM vs Disk. 

```sql
-- Calculate the global Cache Hit Ratio
SELECT 
    sum(blks_hit) * 100 / sum(blks_hit + blks_read) AS cache_hit_ratio
FROM pg_stat_database;
```
*Rule of Thumb:* In a healthy OLTP (web application) database, the Cache Hit Ratio should be **> 99%**. If it drops below 95%, your database is thrashing the disk. You either need to add more RAM to the server (increase `shared_buffers`), or find the bad queries that are scanning too much data.

### Level 3 — Identifying Missing Indexes
How do you know which tables need indexes? You look at `pg_stat_user_tables` to find tables with massive amounts of Sequential Scans (Full Table Scans).

```sql
-- Find tables that are sequentially scanned the most
SELECT 
    relname AS table_name, 
    seq_scan AS full_table_scans, 
    seq_tup_read AS rows_read_via_seq_scan, 
    idx_scan AS index_scans,
    -- If they do a seq scan, how many rows are they actually reading?
    seq_tup_read / seq_scan AS avg_rows_per_scan
FROM pg_stat_user_tables 
WHERE seq_scan > 0 
ORDER BY seq_tup_read DESC 
LIMIT 10;
```
*If `users` has 1,000,000 full table scans, and it reads 5,000,000 rows per scan, a developer is executing a bad query against the users table that desperately needs an index.*

### Level 4 — The Holy Grail: `pg_stat_statements`
`pg_stat_statements` is not enabled by default, but it is **mandatory** for production. It aggregates every query executed, strips out the variables, and records exactly how much CPU time and IO it consumes.

You must enable it in `postgresql.conf`: `shared_preload_libraries = 'pg_stat_statements'` and run `CREATE EXTENSION pg_stat_statements;`.

```sql
-- Find the Top 5 Queries consuming the most total CPU time!
SELECT 
    query, 
    calls, 
    total_exec_time / 1000 / 60 AS total_minutes, 
    (total_exec_time / calls) AS avg_ms_per_query, 
    rows / calls AS avg_rows_returned
FROM pg_stat_statements 
ORDER BY total_exec_time DESC 
LIMIT 5;
```
This is the most powerful query in a DBA's arsenal. It tells you exactly which 5 queries you need to optimize (by adding indexes or rewriting the SQL) to drastically reduce the CPU load on your entire server.

## REAL_WORLD
**Integrating with Prometheus and Grafana**
You don't want to run these SQL queries manually every day. 
In the real world, DBAs deploy the **Postgres Exporter** (an open-source Go binary). The exporter connects to Postgres, runs all these `pg_stat_` queries every 15 seconds, and translates them into Prometheus metrics. 
Grafana then ingests these metrics and displays beautiful, real-time dashboards showing Cache Hit Ratios, CPU load, Active Connections, and Transaction Commit Rates. When the Cache Hit Ratio drops below 95%, Prometheus automatically pages the DBA via PagerDuty.

## INTERVIEW
**Q: The CPU on the database server is suddenly at 100%. You run the `pg_stat_activity` query, but there are only 5 active queries, and they all look like simple, perfectly indexed `SELECT * FROM users WHERE id = $1` queries. If the queries are simple, why is the CPU at 100%?**
A: **Connection Churn (The Storm).**
PostgreSQL is process-based (it forks a heavy OS process for every connection), not thread-based.
If a poorly configured Node.js application is opening a brand new database connection, running a 1ms query, and immediately closing the connection, it might be doing this 1,000 times a second.
The actual SQL query takes 0% CPU. But the OS overhead of forking 1,000 heavy PostgreSQL processes every second instantly maxes out the CPU.
You diagnose this by looking at the connection rate in your metrics. The solution is immediately deploying a **Connection Pooler** like PgBouncer, which holds 100 persistent connections to Postgres and multiplexes the 1,000 incoming application requests over them, instantly dropping the CPU back to 5%.
