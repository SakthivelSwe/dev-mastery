## WHY
If your HTML file includes a 5MB hero image and a massive JavaScript bundle, the browser will freeze, the user will stare at a blank white screen for 10 seconds, and they will abandon your website. Amazon found that every 100ms of latency costs them 1% in sales.

Browsers execute code sequentially. As a Senior Developer, you must orchestrate exactly *when* the browser downloads and executes specific assets. By optimizing the `<head>` of your HTML document, you can drastically reduce the "Time to First Byte" (TTFB) and "Largest Contentful Paint" (LCP).

## THEORY
### The Critical Rendering Path
When a browser downloads your HTML, it reads it from top to bottom. 
If it encounters a `<script src="bundle.js">` tag, it entirely **PAUSES** the HTML parser. It refuses to render the rest of the page until the JavaScript file is fully downloaded across the network, parsed, and executed. This is called **Render-Blocking**.
Your goal is to eliminate Render-Blocking resources so the user sees the visual UI instantly, while the heavy lifting happens invisibly in the background.

## CODE

### Level 1 — Beginner (Script Loading)
Historically, developers put `<script>` tags at the very bottom of the `<body>` to prevent them from blocking the HTML rendering above them.
Today, we keep them in the `<head>` but use modern boolean attributes: `defer` and `async`.

```html
<head>
  <!-- ❌ TERRIBLE: Blocks the entire page from rendering until it finishes! -->
  <script src="analytics.js"></script>

  <!-- ✅ ASYNC: Downloads in the background, but executes IMMEDIATELY once downloaded 
          (Useful for isolated scripts that don't depend on the DOM, like Google Analytics) -->
  <script async src="analytics.js"></script>

  <!-- ✅ DEFER: Downloads in the background, but guarantees it will NOT execute 
          until the entire HTML document is fully parsed! (Best for your main app logic) -->
  <script defer src="app.js"></script>
</head>
```

### Level 2 — Building Blocks (Preloading Assets)
Sometimes, your CSS file contains a `@font-face` declaration that loads a custom Google Font. The browser doesn't know it needs to download the font until it finishes downloading and parsing the CSS! This causes a "Flash of Unstyled Text" (FOUT) where the text suddenly changes fonts 2 seconds after the page loads.

You can explicitly tell the browser to start downloading the font instantly the moment the HTML arrives, before it even looks at the CSS!

```html
<head>
  <!-- Preload the font immediately! -->
  <link rel="preload" href="/fonts/Inter-Bold.woff2" as="font" type="font/woff2" crossorigin>
  
  <!-- Preload a massive hero image so it's ready the millisecond the body renders -->
  <link rel="preload" href="/images/hero-banner.jpg" as="image">
</head>
```

### Level 3 — Intermediate (Preconnecting)
If your website makes a `fetch()` request to an external API (`api.stripe.com`), creating a secure TLS/SSL connection to that external server takes hundreds of milliseconds (DNS lookup, TCP handshake, TLS negotiation). 
You can tell the browser to establish that secure connection in the background while the user is still reading the page, so when they finally click "Buy", the connection is already hot and ready!

```html
<head>
  <!-- Establish a secure connection to the Stripe API immediately -->
  <link rel="preconnect" href="https://api.stripe.com">
  
  <!-- DNS-Prefetch is a lighter version that only does the DNS resolution, 
       saving ~50ms if you don't need a full connection yet -->
  <link rel="dns-prefetch" href="https://fonts.googleapis.com">
</head>
```

### Level 4 — Advanced (Lazy Loading)
If you have a blog post with 50 high-resolution images, you absolutely should not force the user's phone to download all 50 images the second they open the page. It will destroy their data plan and crash the browser.

You should only download images when the user scrolls down and the image is about to enter the screen! Historically, this required massive Javascript libraries (like IntersectionObserver). 
Today, it is a native HTML attribute!

```html
<!-- The browser will NOT download this image until the user scrolls near it! -->
<img src="heavy-diagram.png" alt="Architecture" loading="lazy">

<!-- You can also lazy-load entire external iframes (like YouTube videos) -->
<iframe src="https://youtube.com/embed/123" loading="lazy"></iframe>
```

## REAL_WORLD
**Image Optimization (The `<picture>` Element)**
If a user is on a massive 4K desktop monitor, they need the 3MB high-resolution hero image. If a user is on an iPhone 8 on a slow 3G cellular network, sending them the 3MB image is a crime. They need a heavily compressed, cropped 100KB version.

You can use the HTML5 `<picture>` tag to dynamically serve completely different image files based on the user's screen size or supported formats!

```html
<picture>
  <!-- 1. If the browser supports the ultra-fast modern WebP format, use this: -->
  <source srcset="hero.webp" type="image/webp">
  
  <!-- 2. If they are on a mobile phone (max width 600px), use the tiny mobile version! -->
  <source media="(max-width: 600px)" srcset="hero-mobile.jpg">
  
  <!-- 3. The fallback for everything else (and browsers that don't support <picture>) -->
  <img src="hero-desktop.jpg" alt="Hero Banner">
</picture>
```

## INTERVIEW
**Q: I added `rel="preload"` to all 50 of my JavaScript, CSS, and Image files to make the site load as fast as possible. But the site actually got slower! Why?**
A: `preload` is an emergency override switch. It tells the browser, "Drop whatever you are doing, this asset is a critical priority, fetch it immediately." If you preload *everything*, you completely destroy the browser's intelligent prioritization engine. It will try to download all 50 files simultaneously over a limited bandwidth pipe, causing severe network congestion. You should ONLY preload the absolute most critical assets required to render the very first screen the user sees (the Largest Contentful Paint).

## FEYNMAN CHECK

### Explain HTML Performance Like I'm 10 Years Old
> Every HTML byte the browser receives goes through: parse → build DOM → fetch sub-resources → run scripts → compute styles → layout → paint. `<script>` BLOCKS this pipeline; `<link rel="stylesheet">` blocks rendering but not parsing; `<img>` is downloaded in parallel by the preload scanner. The single biggest HTML-level perf decision is the ORDER of tags in `<head>` — putting a `<script src>` before your CSS stalls EVERYTHING. Modern hints (`async`, `defer`, `preload`, `fetchpriority`, `loading`) exist precisely to let you opt OUT of these defaults.

### 5 Deep Conceptual Questions

**Q1: What problem does HTML performance optimisation fundamentally solve?**
> **A:** It minimises Time to First Byte → First Contentful Paint → Largest Contentful Paint — the three Core Web Vitals that determine SEO ranking and user perception. A 100ms delay in LCP measurably drops conversion rates. HTML-level fixes are the cheapest perf wins because they don't require touching the framework or backend.

**Q2: What is the ONE mental model that makes HTML perf click?**
> **A:** The browser has TWO parsers: the main HTML parser (blocked by sync scripts) and the speculative preload scanner (looks AHEAD in the byte stream for `<img>`, `<script src>`, `<link href>` and starts fetching). Your job is to put the LCP image and critical CSS where the preload scanner finds them first, and to keep sync scripts out of the critical path.

**Q3: What is the most dangerous misconception? Show it with code.**
> **A:** That moving `<script>` to the bottom of `<body>` is still best practice in 2026:
> ```html
> <!-- ❌ Pre-2015 advice. The preload scanner is delayed — script discovered late. -->
> <!DOCTYPE html><html><head><title>Site</title></head>
> <body>
>   <main>...</main>
>   <script src="app.js"></script>           <!-- discovered last → fetched last -->
> </body></html>
>
> <!-- ✅ Modern: put scripts in <head> with defer — discovered early, executed after parse -->
> <!DOCTYPE html><html><head>
>   <title>Site</title>
>   <link rel="stylesheet" href="app.css">
>   <script src="app.js" defer></script>     <!-- parallel download, deferred execution -->
> </head>
> <body><main>...</main></body></html>
> ```

**Q4: How do `async` and `defer` interact with the HTML parser at runtime?**
> **A:** `async`: downloads in parallel with parsing, EXECUTES AS SOON AS DOWNLOADED (interrupts parsing, no guaranteed order). `defer`: downloads in parallel with parsing, EXECUTES AFTER parse completes (preserves source order, fires before `DOMContentLoaded`). For 99% of scripts, `defer` is correct because order matters; `async` is for independent scripts like analytics. Module scripts (`<script type="module">`) are `defer` by default.

**Q5: Write a one-sentence FAANG-precise definition.**
> **A:** "HTML performance is the practice of ordering and annotating `<head>` and `<body>` tags so the browser's preload scanner discovers critical resources (LCP image, critical CSS, deferred JS) in the right order with the right priorities — turning HTML markup itself into a network-scheduler that materially shifts First Contentful Paint, Largest Contentful Paint, and Cumulative Layout Shift before a single byte of JavaScript runs."

#### Step 2 — The "Before" (typical pre-optimisation)
```html
<!DOCTYPE html>
<html lang="en">
<head>
  <title>Landing page</title>
  <!-- ❌ Render-blocking, no preconnect to CDN -->
  <link rel="stylesheet" href="https://cdn.example.com/app.css">
  <!-- ❌ Render-blocking, in head with no defer -->
  <script src="https://cdn.example.com/big-app.js"></script>
  <style>
    @font-face {
      font-family: 'Brand'; src: url('/fonts/font.woff2') format('woff2');
      /* ❌ No font-display: invisible text until font loads (FOIT) */
    }
  </style>
</head>
<body>
  <!-- ❌ No width/height — causes CLS when loaded -->
  <img src="/img/hero.jpg" alt="Hero">
  <!-- ❌ Below-fold images eager-loaded -->
  <img src="/img/g1.jpg" alt="">
  <img src="/img/g2.jpg" alt="">
  <img src="/img/g3.jpg" alt="">
</body>
</html>
```

#### Step 3 — The "After" (HTML-only optimisations applied)
```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Landing page</title>

  <!-- ✅ Preconnect to CDN: saves 100-300ms on first fetch from that origin -->
  <link rel="preconnect" href="https://cdn.example.com" crossorigin>
  <link rel="dns-prefetch" href="https://cdn.example.com">

  <!-- ✅ Preload the LCP image (hero) so it starts BEFORE CSS parses -->
  <link rel="preload" as="image" href="/img/hero.jpg" fetchpriority="high">

  <!-- ✅ Preload the brand font — eliminates FOUT/FOIT -->
  <link rel="preload" as="font" type="font/woff2"
        href="/fonts/font.woff2" crossorigin>

  <link rel="stylesheet" href="https://cdn.example.com/app.css">

  <!-- ✅ Defer script: downloads in parallel, executes after parse -->
  <script src="https://cdn.example.com/big-app.js" defer></script>

  <style>
    @font-face {
      font-family: 'Brand';
      src: url('/fonts/font.woff2') format('woff2');
      font-display: swap;   /* ✅ Show fallback immediately, swap when ready */
    }
  </style>
</head>
<body>
  <!-- ✅ Hero: width/height prevents CLS, fetchpriority=high for LCP -->
  <img src="/img/hero.jpg" alt="Sunlit forest path"
       width="1200" height="600"
       fetchpriority="high" decoding="sync">

  <!-- ✅ Below-fold: lazy + async -->
  <img src="/img/g1.jpg" alt="" width="400" height="300"
       loading="lazy" decoding="async">
  <img src="/img/g2.jpg" alt="" width="400" height="300"
       loading="lazy" decoding="async">
  <img src="/img/g3.jpg" alt="" width="400" height="300"
       loading="lazy" decoding="async">
</body>
</html>
```

#### Step 4 — Edge Cases: Module Scripts, Third-Party Tags, Inline CSS
```html
<!-- ESM is defer-by-default. Don't add defer redundantly. -->
<script type="module" src="/app.mjs"></script>

<!-- Analytics: async is fine (independent, order doesn't matter) -->
<script async src="https://www.googletagmanager.com/gtag/js?id=GA"></script>

<!-- Inline critical CSS for above-the-fold, lazy-load the rest -->
<style>/* ~5KB of above-the-fold styles inlined */</style>
<link rel="stylesheet" href="/rest.css" media="print"
      onload="this.media='all'">                <!-- non-blocking pattern -->
```
