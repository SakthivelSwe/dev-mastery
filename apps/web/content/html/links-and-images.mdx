## WHY

Links are what make the web a *web* — without hyperlinks, the internet is just disconnected pages. Images are the most bandwidth-intensive part of most web pages — the difference between a 2-second and a 10-second page load is usually image optimization. Together, links and images are the two most-used non-text HTML elements, yet they're also the most frequently misused.

A broken link costs SEO ranking and user trust. An image with a missing `alt` attribute fails accessibility audits and may result in legal liability. An unoptimized image causes a poor Core Web Vitals score (specifically Largest Contentful Paint). A link opening in a new tab without `rel="noopener"` creates a security vulnerability. Every professional web developer must understand these elements deeply.

## THEORY

### The `<a>` Element — Complete Reference

| Attribute | Values | Purpose |
|-----------|--------|---------|
| `href` | URL, `#id`, `mailto:`, `tel:`, `javascript:void(0)` | Link destination |
| `target` | `_self` (default), `_blank`, `_parent`, `_top`, frame name | Where to open |
| `rel` | `noopener`, `noreferrer`, `nofollow`, `author`, `prev`, `next`, `canonical` | Relationship to destination |
| `download` | filename or boolean | Forces download instead of navigation |
| `hreflang` | BCP 47 language tag (`en`, `fr`, `zh`) | Language of destination page |
| `type` | MIME type | Hint about media type at destination |
| `ping` | URL list | Analytics ping URLs on click |
| `referrerpolicy` | `no-referrer`, `origin`, etc. | Controls Referer header |

### URL Types
```html
<!-- Absolute URL: full domain + path -->
<a href="https://developer.mozilla.org/en-US/docs/Web/HTML">MDN</a>

<!-- Root-relative: from domain root -->
<a href="/about">About Page</a>

<!-- Document-relative: from current page location -->
<a href="../images/photo.jpg">Photo</a>

<!-- Fragment: scroll to element with that id -->
<a href="#contact-section">Jump to Contact</a>

<!-- mailto: open email client -->
<a href="mailto:hello@example.com?subject=Hello&body=Hi there">Email Us</a>

<!-- tel: trigger phone call (on mobile) -->
<a href="tel:+15555551234">+1 (555) 555-1234</a>

<!-- Download: force download instead of navigation -->
<a href="/report.pdf" download="Annual-Report-2026.pdf">Download Report</a>
```

### The `<img>` Element — Complete Reference

| Attribute | Purpose |
|-----------|---------|
| `src` | Image URL (required) |
| `alt` | Alternative text (required — empty string for decorative images) |
| `width` / `height` | Intrinsic dimensions in pixels (prevents layout shift!) |
| `loading` | `lazy` (defer until near viewport) / `eager` (load immediately) |
| `decoding` | `async` (decode off main thread) / `sync` / `auto` |
| `fetchpriority` | `high` (LCP image) / `low` / `auto` |
| `srcset` | Multiple sources for different resolutions |
| `sizes` | Which source size to use at different viewport widths |
| `crossorigin` | `anonymous` / `use-credentials` (for CORS images) |
| `referrerpolicy` | Controls Referer header for image requests |
| `ismap` / `usemap` | Server-side / client-side image maps |

### Responsive Images with `srcset` and `sizes`
```html
<!-- Resolution switching: same image, different sizes -->
<img
  src="hero-800.jpg"
  srcset="hero-400.jpg 400w,
          hero-800.jpg 800w,
          hero-1200.jpg 1200w,
          hero-1600.jpg 1600w"
  sizes="(max-width: 600px) 100vw,
         (max-width: 900px) 80vw,
         60vw"
  alt="DevMastery hero image"
  loading="eager"
  fetchpriority="high"
  width="1200"
  height="600"
>
```

The browser picks the optimal source: if viewport is 375px wide at 2x DPR, it needs a 750px image, so it picks the 800w source.

### `<picture>` — Art Direction & Format Negotiation
```html
<picture>
  <!-- AVIF: best compression, modern browsers only -->
  <source type="image/avif" srcset="hero.avif">
  <!-- WebP: good compression, broad support -->
  <source type="image/webp" srcset="hero.webp">
  <!-- Art direction: different crop on mobile -->
  <source media="(max-width: 600px)" srcset="hero-portrait.jpg">
  <!-- Fallback: always required -->
  <img src="hero.jpg" alt="DevMastery learning platform" width="1200" height="600">
</picture>
```

### Level 1 — Beginner: Links and Images Basics
```html
<!DOCTYPE html>
<html lang="en">
<head><meta charset="UTF-8"><title>Links & Images</title></head>
<body>

  <h1>Links</h1>

  <!-- EXTERNAL LINK: always add rel="noopener noreferrer" for security -->
  <a href="https://developer.mozilla.org" target="_blank" rel="noopener noreferrer">
    MDN Web Docs (opens in new tab)
  </a>

  <!-- INTERNAL LINK: root-relative path -->
  <a href="/about">About Us</a>

  <!-- ANCHOR LINK: jump to section on same page -->
  <a href="#section2">Jump to Section 2</a>

  <!-- EMAIL LINK -->
  <a href="mailto:support@devmastery.com">Email Support</a>

  <!-- PHONE LINK: critical for mobile users -->
  <a href="tel:+15555551234">Call: (555) 555-1234</a>

  <a href="/files/cheatsheet.pdf" download="HTML-Cheatsheet.pdf">
    Download HTML Cheatsheet (PDF)
  </a>

    src="https://via.placeholder.com/400x200"

    width="400"
    height="200"
  >

  </a>

</html>
```

## REAL_WORLD

Image optimization is the single highest-ROI performance improvement for most websites. Google's Core Web Vitals measures Largest Contentful Paint (LCP) — the time until the largest visible element (usually an image) renders. An LCP under 2.5 seconds is "good"; above 4 seconds is "poor" and causes ranking penalties. The two most impactful fixes: (1) Add `fetchpriority="high"` to the above-the-fold hero image so the browser prioritizes it; (2) Use `<picture>` with AVIF/WebP formats — AVIF is 40% smaller than WebP and 80% smaller than JPEG for the same quality.

```jsx
// Next.js <Image> component — wraps HTML <picture> + srcset + lazy loading
import Image from 'next/image';

function HeroBanner() {
  return (
    <Image
      src="/hero.jpg"          // Automatically generates WebP/AVIF variants
      alt="DevMastery hero"
      width={1200}
      height={600}
      priority                  // = fetchpriority="high" + loading="eager"
      quality={85}              // Compression quality (1-100)
      sizes="(max-width: 768px) 100vw, 80vw"
    />
  );
}
```

For links, never use `<a href="javascript:void(0)" onclick="...">` for interactive elements — this is a significant accessibility failure because the `href` makes it look like a navigation link, but it doesn't navigate anywhere, confusing screen readers. Use `<button>` for actions, `<a href="/url">` for navigation. If you truly need an anchor-style button, use `role="button"` and handle both `click` and `keydown` (Enter/Space) events.

## INTERVIEW

### Q1: What is `alt` text and when should it be empty vs descriptive?
**Difficulty:** Easy | **Companies:** All

❌ **Weak answer:** "alt shows text when image doesn't load."

✅ **Strong answer:**
"The `alt` attribute provides an **alternative text description** of an image for screen readers, search engines, and when the image fails to load. When to make it descriptive: always, unless the image is purely decorative (adds no information). For **functional images** (logos, icons): describe the function, not appearance (`alt='DevMastery home page'`). For **content images** (diagrams, photos): describe what the image shows and its significance. For **decorative images** (background ornaments, spacer images): use `alt=''` (empty string) — this tells screen readers to skip it entirely (not `alt='decorative'` which forces them to read 'decorative'). Never omit `alt` entirely — some screen readers read the `src` filename instead."

### Q2: What is the difference between `srcset` and `<picture>`?
**Difficulty:** Medium | **Companies:** Google, Performance-focused companies

❌ **Weak answer:** "They both do responsive images."

✅ **Strong answer:**
"`srcset` on `<img>` handles **resolution switching** — you provide the same image at multiple sizes, and the browser picks the most appropriate based on viewport width and device pixel ratio. The browser has freedom to choose based on its own logic (network conditions, cache, user preferences). `<picture>` handles **art direction** — you have explicit control over which source is used via `media` queries, AND it handles **format negotiation** via `type` (AVIF → WebP → JPEG). Use `srcset` when you want the browser to optimize automatically (usually preferred). Use `<picture>` when you need to serve fundamentally different crops for different viewports (portrait on mobile, landscape on desktop), or when you need to serve modern formats to capable browsers with JPEG fallback."

### Q3: What does `rel="noopener noreferrer"` do and why is it needed?
**Difficulty:** Medium | **Companies:** Security-focused companies

❌ **Weak answer:** "It's a security best practice for links."

✅ **Strong answer:**
"When a link uses `target='_blank'`, the new tab gets a reference to the opening page via `window.opener`. A malicious page could execute `window.opener.location = 'https://phishing.com'` — silently redirecting the original tab (tab nabbing attack). `rel='noopener'` nullifies `window.opener`, preventing this attack. `rel='noreferrer'` additionally prevents the `Referer` HTTP header from being sent (user privacy) AND implicitly applies `noopener`. Modern browsers (Chrome 88+) automatically apply `noopener` to `target='_blank'` links, but explicitly adding it maintains compatibility with older browsers and makes intent clear. `nofollow` is a separate signal telling search engine crawlers not to follow or attribute PageRank to this link (used for untrusted user-submitted links or paid links)."

### Q4: What are Core Web Vitals and how do images affect them?
**Difficulty:** Hard | **Companies:** Google, Agencies, E-commerce

❌ **Weak answer:** "Core Web Vitals are Google's performance metrics."

✅ **Strong answer:**
"Core Web Vitals are Google's page experience signals that directly affect search rankings. The three metrics: **LCP (Largest Contentful Paint)** — time until the largest visible content renders (target: <2.5s). The LCP element is usually a hero image. Fix: `fetchpriority='high'` on the LCP image, use `<link rel='preload'>` in `<head>`, avoid lazy loading above-fold images. **CLS (Cumulative Layout Shift)** — unexpected visual shifts (target: <0.1). Fix: always set explicit `width` and `height` attributes on images so the browser reserves space before loading. **INP (Interaction to Next Paint)** — response time to user interactions (target: <200ms). Images affect CLS most directly — an `<img>` without dimensions causes the page to reflow when the image loads, shifting all content below it and creating a high CLS score."

### Q5: What are `loading="lazy"` and `decoding="async"` and how do they differ?
**Difficulty:** Medium | **Companies:** Performance-focused companies

❌ **Weak answer:** "They both make images load faster."

✅ **Strong answer:**
"`loading='lazy'` is a **network loading strategy** — the browser defers fetching the image resource until it's within a certain distance of the viewport (typically ~1-2 viewports ahead). This reduces initial page weight and speeds up the first paint. Never use it on above-the-fold images — it delays your LCP image. `decoding='async'` is a **rendering strategy** — it tells the browser to decode (decompress JPEG/PNG/WebP into pixel data) off the main thread, so it doesn't block rendering. Even if the image is already loaded in the cache, decoding takes CPU time. `async` allows the browser to paint everything it has and decode the image concurrently. Use `loading='eager'` + `fetchpriority='high'` + `decoding='async'` for LCP images; use `loading='lazy'` + `decoding='async'` for all below-the-fold images."

## FEYNMAN CHECK

### Explain Links and Images Like I'm 10 Years Old
> Links (`<a href>`) and images (`<img src>`) are the two ways HTML reaches out to other URLs. A link tells the browser "navigate here when clicked"; an image tells the browser "download this and render it inline." Both are subject to the same security rules (CORS, mixed content) but behave totally differently: a link is lazy (only fetched on click), an image is eager (fetched immediately during HTML parsing unless you say `loading="lazy"`). Getting `alt=""` wrong on images and `target="_blank"` wrong on links are the two most common accessibility AND security mistakes on the web.

### 5 Deep Conceptual Questions

**Q1: What problem do `<a>` and `<img>` fundamentally solve?**
> **A:** They are the original "rich-media composition" primitives — the reason the web outcompeted plain-text protocols. `<a>` invented hypertext navigation; `<img>` invented inline media. Every modern resource hint (`<link rel="preload">`, `fetchpriority`) descends from optimisation pressure on these two tags.

**Q2: What is the ONE mental model that makes them click?**
> **A:** Both are URL references with browser-managed fetch lifecycles, but `<a>` is user-initiated (click) and `<img>` is parser-initiated (downloaded during HTML parse). Once you internalise "the browser starts the `<img>` fetch BEFORE running your CSS or JS," lazy-loading, priority hints, and LCP optimisation all make sense.

**Q3: What is the most dangerous misconception? Show it with code.**
> **A:** That `target="_blank"` is safe by itself:
> ```html
> <!-- ❌ Tab-napping vulnerability — the new tab can navigate THIS tab via window.opener -->
> <a href="https://evil.com" target="_blank">External link</a>
>
> <!-- ✅ Safe: rel="noopener" severs the window.opener reference -->
> <a href="https://evil.com" target="_blank" rel="noopener noreferrer">External link</a>
> ```
> Modern browsers default `target="_blank"` to noopener since 2021, but legacy stacks (older WebViews, embedded browsers) still leak.

**Q4: How does an `<img>` interact with the browser's preload scanner at the runtime level?**
> **A:** While the main HTML parser runs synchronously (and is blocked by `<script>`), the browser ALSO runs a "preload scanner" in parallel that finds `<img src>`, `<link rel="stylesheet">`, and other URLs in the raw HTML bytes and starts fetching them immediately. This is why `<img loading="lazy">` works ONLY for images below the fold — for above-the-fold images, the preload scanner has already started the fetch before the parser reaches the tag. For LCP images, use `fetchpriority="high"` instead.

**Q5: Write a one-sentence FAANG-precise definition.**
> **A:** "`<a>` and `<img>` are URL-bearing elements with distinct fetch lifecycles: `<a>` defers the network request until user interaction while `<img>` triggers an eager parallel fetch via the browser's preload scanner — which is why `fetchpriority`, `loading`, `decoding`, and `rel` attributes are the most impactful single-attribute additions for Core Web Vitals."
