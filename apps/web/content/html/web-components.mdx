## WHY
In the early 2010s, if you built a beautiful, complex DatePicker UI, and you wanted to share it with another team, it was nearly impossible. The CSS from your DatePicker would accidentally overwrite the CSS on their website, and your JavaScript variables would clash with theirs. 

React and Angular solved this by introducing **Components** (encapsulated pieces of UI). However, a React component cannot be used in an Angular app.
**Native Web Components** solve this permanently at the browser level. They allow you to create custom HTML tags (like `<my-datepicker>`) that encapsulate their own CSS and JavaScript, and work flawlessly in *any* framework, or with no framework at all!

## THEORY
### The Three Pillars
Web Components are not a single technology; they are a suite of three distinct Web APIs:
1. **Custom Elements API:** Allows you to register a brand new HTML tag (e.g., `<user-card>`) and tie it to a JavaScript Class.
2. **Shadow DOM:** Allows you to attach an isolated, secret DOM tree inside your element. The CSS inside the Shadow DOM cannot leak out, and global CSS cannot leak in!
3. **HTML Templates (`<template>` and `<slot>`):** Provides the reusable, inert HTML blueprint (covered in the previous module).

## CODE

### Level 1 — Beginner (Custom Elements)
To create a custom element, you extend the native `HTMLElement` JavaScript class, and then register it with the browser using `customElements.define`.

```javascript
// 1. Define the logic for your custom element
class GreetingElement extends HTMLElement {
  connectedCallback() {
    // This lifecycle method fires the moment the element is injected into the DOM!
    this.innerHTML = `<h1>Hello, World!</h1>`;
  }
}

// 2. Register the tag name with the browser
// MANDATORY RULE: Custom element names MUST contain a hyphen (-) 
// to prevent clashing with future official HTML tags!
customElements.define('my-greeting', GreetingElement);
```

```html
<!-- 3. Use it anywhere in your HTML! -->
<my-greeting></my-greeting>
```

### Level 2 — Building Blocks (The Shadow DOM)
In Level 1, if the website has global CSS that says `h1 { color: red; }`, our GreetingElement will turn red. It is not encapsulated! 
To truly isolate the component, we must attach a **Shadow Root**.

```javascript
class UserCard extends HTMLElement {
  constructor() {
    super(); // Always call super() first in the constructor!
    
    // Attach the secret, isolated Shadow DOM! ('open' means JS can inspect it)
    this.attachShadow({ mode: 'open' }); 
    
    // Notice we append HTML to the SHADOW ROOT, not the standard DOM!
    this.shadowRoot.innerHTML = `
      <style>
        /* This CSS is trapped! It will NEVER affect the rest of the website! */
        h2 { color: blue; } 
      </style>
      <div class="card">
        <h2>Secret User</h2>
      </div>
    `;
  }
}
customElements.define('user-card', UserCard);
```

### Level 3 — Intermediate (Reactive Attributes)
Just like React props, you want your component to react when an HTML attribute changes (e.g., `<user-card name="Alice">`).

```javascript
class UserCard extends HTMLElement {
  // 1. Tell the browser exactly which attributes to watch for changes
  static get observedAttributes() {
    return ['name'];
  }

  // 2. This lifecycle method fires whenever an observed attribute changes!
  attributeChangedCallback(attributeName, oldValue, newValue) {
    if (attributeName === 'name') {
      // Update the UI dynamically!
      this.shadowRoot.querySelector('h2').textContent = newValue;
    }
  }
}
```

### Level 4 — Advanced (Combining all Three Pillars)
In production, you never write massive HTML strings in JavaScript. You define a `<template>` in your HTML file, and your Web Component clones it into its Shadow DOM!

```html
<!-- The Blueprint -->
<template id="card-template">
  <style> h2 { color: purple; } </style>
  <div class="wrapper">
    <slot name="title">Default Title</slot>
  </div>
</template>

<!-- The Usage -->
<purple-card>
  <span slot="title">I am injected via a slot!</span>
</purple-card>
```

```javascript
class PurpleCard extends HTMLElement {
  constructor() {
    super();
    this.attachShadow({ mode: 'open' });
    
    // Grab the blueprint, clone it, and shove it into the Shadow DOM!
    const template = document.getElementById('card-template');
    this.shadowRoot.appendChild(template.content.cloneNode(true));
  }
}
customElements.define('purple-card', PurpleCard);
```

## REAL_WORLD
**Micro-Frontends and Design Systems**
Massive enterprises (like Apple, GitHub, and Salesforce) use Web Components to build their internal **Design Systems**.
Why? If Apple builds a beautifully animated `<apple-button>` using React, the Vue team and the Angular team at Apple cannot use it. They would have to rewrite the button from scratch!

By writing the `<apple-button>` as a native Web Component, the company builds it exactly once. It is shipped as a tiny JavaScript file, and it works flawlessly inside React, Vue, Angular, Svelte, or plain HTML, guaranteeing visual consistency across a trillion-dollar company.

## INTERVIEW
**Q: Why must a custom element name always contain a hyphen (e.g., `<user-card>`)?**
A: To completely eliminate the risk of naming collisions. The W3C (the organization that dictates HTML standards) made a strict promise: *They will never, ever create an official HTML tag that contains a hyphen.* By forcing developers to use hyphens for custom components, it guarantees that your `<custom-tabs>` component will never accidentally conflict with an official `<tabs>` element that Google Chrome might release 10 years from now.
