## WHY
If you build a registration form, and a user types "apples" into the Age field, and submits it to your Node.js backend, your database query will crash. Worse, if they type malicious JavaScript into the Comments field, they could hack your entire application.

Data must be validated. By using **HTML5 Native Form Validation**, you can catch 90% of user errors directly in the browser *before* the data is ever sent over the network, providing instant, beautiful error messages to the user and saving your server from processing garbage data.

## THEORY
### Client-Side vs Server-Side Validation
**Client-Side Validation (HTML/JS):** This happens in the browser. It exists purely for **User Experience (UX)**. It prevents the user from waiting 3 seconds for a server response just to find out they forgot the "@" symbol in their email.
**Server-Side Validation (Node.js/Spring):** This happens on the backend. It exists for **Security**. 

*Critical Rule:* You must ALWAYS implement Server-Side Validation! A malicious hacker can open Chrome DevTools, delete your HTML validation attributes, and submit the form anyway. Client-Side Validation is a courtesy; Server-Side Validation is a fortress.

## CODE

### Level 1 ŌĆö Beginner (Required & Input Types)
The easiest way to validate data is to use the correct `type` attribute on your `<input>` tags, and add the `required` boolean attribute.

```html
<form>
  <!-- The browser will physically refuse to submit the form if this is empty! -->
  <label for="username">Username:</label>
  <input type="text" id="username" required>

  <!-- The browser will automatically verify that the text contains an '@' symbol! -->
  <label for="email">Email Address:</label>
  <input type="email" id="email" required>
  
  <button type="submit">Register</button>
</form>
```

### Level 2 ŌĆö Building Blocks (Length and Range Boundaries)
If you are asking for an Age, it shouldn't be negative, and it shouldn't be 1,000. If you are asking for a Password, it shouldn't be 2 characters long.

```html
<form>
  <!-- Using minlength and maxlength for text boundaries -->
  <label for="password">Password (8-20 characters):</label>
  <input type="password" id="password" minlength="8" maxlength="20" required>

  <!-- Using min and max for numerical boundaries -->
  <label for="age">Age (Must be 18+):</label>
  <input type="number" id="age" min="18" max="120" required>
  
  <!-- Using step to control increments (e.g., money values) -->
  <label for="price">Price (Multiples of $0.50):</label>
  <input type="number" id="price" min="0" step="0.50">
</form>
```

### Level 3 ŌĆö Intermediate (Regular Expressions with `pattern`)
What if you need the user to enter a Zip Code (exactly 5 digits) or a complex Part Number (3 letters followed by a dash and 4 numbers)? HTML5 provides the `pattern` attribute, which accepts a raw Regular Expression!

```html
<form>
  <!-- The pattern forces exactly 5 digits (0-9) -->
  <label for="zipcode">US Zip Code:</label>
  <input 
    type="text" 
    id="zipcode" 
    pattern="[0-9]{5}" 
    title="Five digit zip code (e.g., 90210)" 
    required
  >
</form>
```
*(Pro Tip: Always include the `title` attribute when using `pattern`. When the regex fails, the browser will display the text inside the `title` attribute as the error popup, explaining to the user exactly what format they need to use!)*

### Level 4 ŌĆö Advanced (Custom JS Validation with ValidityState)
Sometimes, native HTML attributes aren't enough. What if you need to verify that the "Confirm Password" field exactly matches the "Password" field? HTML cannot do this alone. You must use the **Constraint Validation API** in JavaScript!

```html
<form id="signupForm">
  <input type="password" id="pwd" required>
  <input type="password" id="pwdConfirm" required>
  <button type="submit">Submit</button>
</form>

<script>
  const pwd = document.getElementById('pwd');
  const confirmPwd = document.getElementById('pwdConfirm');

  // Listen for the user typing in the confirm field
  confirmPwd.addEventListener('input', () => {
    if (pwd.value !== confirmPwd.value) {
      // Set a custom error message! 
      // This tells the browser the input is mathematically INVALID!
      confirmPwd.setCustomValidity("Passwords do not match!");
    } else {
      // Clear the error (Empty string means it is VALID)
      confirmPwd.setCustomValidity("");
    }
  });
</script>
```

## REAL_WORLD
**Styling Valid/Invalid States with CSS**
The browser's default popup error messages are notoriously ugly and impossible to style. Modern web apps use CSS pseudo-classes (`:valid` and `:invalid`) to provide immediate, beautiful visual feedback to the user while they are typing.

```css
/* When the input is valid, give it a subtle green border */
input:valid {
  border: 2px solid #28a745;
}

/* When the input is invalid, give it a red border and a pink background */
input:invalid {
  border: 2px solid #dc3545;
  background-color: #f8d7da;
}
```
*(Warning: By default, empty `required` fields are instantly marked as `:invalid` the second the page loads! To fix this UX nightmare, modern CSS introduced the `:user-invalid` pseudo-class, which only turns red AFTER the user has clicked on the field and tried to type).*

## INTERVIEW
**Q: I have a `<form>` with a "Cancel" button. When the user clicks Cancel, I want to redirect them to the home page using JavaScript. But when they click it, the browser screams at them to fill out the `required` fields! How do I stop this?**
A: By default, any `<button>` inside a `<form>` is implicitly treated as `type="submit"`. If you click it, it triggers the entire HTML5 validation sequence! 
To fix this, you have two options:
1. Explicitly mark the cancel button as `type="button"`. This tells the browser, "This is just a dumb, clickable box; do not submit or validate the form."
2. Add the `formnovalidate` attribute directly to the submit button. This tells the browser to proceed with the form submission, but skip all validation checks!

## FEYNMAN CHECK

### Explain HTML5 Form Validation Like I'm 10 Years Old
> Before you call the server, the browser plays bouncer at the door: it checks every input against built-in rules (`required`, `type="email"`, `pattern`, `min/max`, `minlength/maxlength`) AND any custom rule you set via `setCustomValidity('...')`. If anything fails, the browser blocks the submission and pops up the FIRST invalid field's error ŌĆö without ONE line of JS. This is why "use the platform" exists: native validation gives you keyboard nav, screen-reader announcements, mobile-friendly UI, and built-in localised error text for free.

### 5 Deep Conceptual Questions

**Q1: What problem does native form validation fundamentally solve?**
> **A:** It eliminates the round-trip cost of submitting an obviously-invalid form, while ALSO giving the user instant feedback in a UX consistent with their OS and locale. A custom JS validator can do the same but you re-invent: focus management, ARIA live-region announcements, mobile native error styling, localised error strings, and the screen-reader form mode.

**Q2: What is the ONE mental model that makes form validation click?**
> **A:** Every form control has a live `validity` object whose 8 boolean flags (`valueMissing`, `typeMismatch`, `patternMismatch`, `tooLong`, `tooShort`, `rangeUnderflow`, `rangeOverflow`, `stepMismatch`, plus catch-all `customError`) summarise WHY the field is invalid. The browser uses these flags to block submission and show the tooltip. Custom validation is just `element.setCustomValidity(message)` ŌĆö empty string means valid.

**Q3: What is the most dangerous misconception? Show it with code.**
> **A:** That client-side validation is enough. It's purely a UX layer, NEVER security:
> ```javascript
> // ŌØī DANGEROUS ŌĆö attacker bypasses the form with curl
> // Server: app.post('/transfer', (req, res) => bank.transfer(req.body.amount));
>
> // Ō£ģ Server MUST re-validate every constraint the HTML claimed
> app.post('/transfer', (req, res) => {
>   const amt = Number(req.body.amount);
>   if (!Number.isFinite(amt) || amt <= 0 || amt > 10000) return res.sendStatus(400);
>   bank.transfer(amt);
> });
> ```

**Q4: How does `:invalid` CSS interact with the validation engine?**
> **A:** The browser keeps `validity.valid` in sync on every `input` event. `:invalid` matches whenever `validity.valid === false`. The pitfall: `:invalid` fires IMMEDIATELY on page load ŌĆö a required-but-empty input is already `:invalid` before the user touched it, producing ugly red borders on a fresh form. Fix: apply error styles via `:invalid:not(:placeholder-shown)` or gate them behind a JS-toggled `data-touched` attribute.

**Q5: Write a one-sentence FAANG-precise definition.**
> **A:** "HTML5 form validation is a declarative constraint API exposed via element attributes (`required`, `pattern`, `min`, `type`...) and a programmatic counterpart (`element.validity`, `setCustomValidity`, `reportValidity`) that the browser uses to block submission and surface localised error UI ŌĆö providing zero-JS UX that must always be re-checked on the server because it is purely a client-side hint, not a security boundary."

#### Step 3 ŌĆö Custom + Async Validation (`validate.js`)
```javascript
const form = document.getElementById('signup');
const u    = form.username;
const p    = form.password;
const p2   = form.passwordConfirm;

// (1) Mark fields as "touched" on first blur ŌĆö :invalid styling only shows then
for (const field of form.elements) {
  if (!field.name) continue;
  field.addEventListener('blur', () => field.dataset.touched = 'true', { once: true });
  field.addEventListener('input', () => showError(field));
}

// (2) Password-confirm custom validity ŌĆö runs on EITHER field changing
function syncConfirm() {
  const msg = p.value && p2.value && p.value !== p2.value
    ? 'Passwords do not match'
    : '';
  p2.setCustomValidity(msg);
  showError(p2);
}
p.addEventListener('input', syncConfirm);
p2.addEventListener('input', syncConfirm);

// (3) Async username availability ŌĆö debounced, AbortController-cancelled
let abort;
u.addEventListener('input', async () => {
  if (abort) abort.abort();
  abort = new AbortController();
  if (!u.checkValidity()) return;           // skip if native rules already fail
  await new Promise(r => setTimeout(r, 300));  // debounce
  try {
    const r = await fetch('/api/check?u=' + encodeURIComponent(u.value),
                          { signal: abort.signal });
    const { available } = await r.json();
    u.setCustomValidity(available ? '' : 'Username already taken');
  } catch (e) {
    if (e.name !== 'AbortError') u.setCustomValidity('');
  }
  showError(u);
});

// (4) Render validation message into the live region
function showError(field) {
  const slot = document.getElementById(field.id + '-err');
  slot.textContent = field.validity.valid ? '' : field.validationMessage;
  field.setAttribute('aria-invalid', String(!field.validity.valid));
}

// (5) On submit, re-check ALL and let browser show its UI
form.addEventListener('submit', (e) => {
  if (!form.checkValidity()) {
    e.preventDefault();
    form.reportValidity();                  // browser focuses first invalid field
  }
});
```
