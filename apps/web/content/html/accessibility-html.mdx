## WHY
Imagine trying to navigate a website blindfolded, using only your keyboard and a robot voice reading the screen to you. This is how millions of visually impaired users experience the web via Screen Readers. If you build a dropdown menu using a generic `<div>` with a JavaScript click handler, the Screen Reader doesn't know it's a menu. It just sees an empty box, and the user is completely blocked from using your site.

**Web Accessibility (often abbreviated as a11y)** is not just a moral obligation; in many countries (including the US and EU), it is a strict legal requirement for businesses.

## THEORY
### The Accessibility Tree
When a browser parses HTML, it builds the DOM (Document Object Model) for JavaScript. Simultaneously, it builds a secondary tree called the **Accessibility Tree**. This tree contains only the semantic meaning and state of elements (e.g., "This is a button, its name is 'Submit', and it is currently 'Disabled'"). Screen Readers read the Accessibility Tree, not the DOM. 
If your HTML lacks semantics, the Accessibility Tree is broken.

## CODE

### Level 1 — Beginner (Semantic Foundations)
The easiest way to make a website accessible is to simply use native HTML elements exactly as they were intended.

```html
<!-- ❌ TERRIBLE: A Screen Reader thinks this is just a random line of text. -->
<!-- A keyboard user cannot 'Tab' to this element! -->
<div class="my-fancy-button" onclick="submitForm()">Submit</div>

<!-- ✅ PERFECT: The browser automatically tells the Screen Reader it's a button, -->
<!-- AND it automatically allows keyboard users to focus on it and press 'Enter'! -->
<button onclick="submitForm()">Submit</button>
```

### Level 2 — Building Blocks (Forms and Labels)
If a blind user focuses on an input field, how do they know what they are supposed to type? You must associate a label with the input.

```html
<!-- ❌ BAD: The Screen Reader just says "Text Input". -->
Email: <input type="email" id="user_email">

<!-- ✅ GOOD: The Screen Reader says "Email, Text Input". -->
<!-- The 'for' attribute strictly links the Label to the Input's 'id'. -->
<label for="user_email">Email:</label>
<input type="email" id="user_email">
```
*(Bonus: Clicking the label text automatically focuses the input field, which is great for mobile users with big thumbs!)*

### Level 3 — Intermediate (ARIA Attributes)
Sometimes, native HTML isn't enough. If you are building a complex custom UI component (like a highly stylized, animated toggle switch built out of `div`s), there is no `<toggleswitch>` HTML tag. 

You must manually inject meaning into the Accessibility Tree using **WAI-ARIA** (Web Accessibility Initiative – Accessible Rich Internet Applications) attributes.

```html
<!-- We are forced to use a DIV for a custom switch -->
<div 
  class="custom-switch" 
  
  -- 1. ROLE: Tell the Screen Reader what this actually is
  role="switch" 
  
  -- 2. ARIA STATE: Tell the Screen Reader its current status
  aria-checked="true" 
  
  -- 3. TABINDEX: Allow keyboard users to 'Tab' to this element!
  tabindex="0"
>
  <div class="knob"></div>
</div>
```
*(Note: You must use JavaScript to manually toggle `aria-checked="false"` when the user clicks the switch!)*

### Level 4 — Advanced (aria-live and aria-hidden)
**Dynamic Updates (`aria-live`)**
If a user submits a form, and a red error message ("Invalid password") appears dynamically via React/JavaScript, a sighted user sees it instantly. A blind user has no idea it appeared because their Screen Reader is focused somewhere else.
You can use a "Live Region" to force the Screen Reader to interrupt the user and read the new text!
```html
<!-- When JS injects text into this div, the Screen Reader will announce it instantly -->
<div aria-live="assertive" class="error-message"></div>
```

**Hiding Clutter (`aria-hidden`)**
If you have a decorative icon next to a button, the Screen Reader might try to read the raw SVG code, frustrating the user.
```html
<button>
  <!-- Tell the Screen Reader to completely ignore this icon! -->
  <svg aria-hidden="true" class="icon-save"></svg>
  Save Document
</button>
```

## REAL_WORLD
**Color Contrast and Keyboard Traps**
Accessibility isn't just for Screen Readers!
1. **Low Vision (Color Contrast):** Gray text on a light-gray background is impossible to read for users with poor vision or color blindness. The WCAG (Web Content Accessibility Guidelines) mandates a strict contrast ratio (usually 4.5:1) for text.
2. **Motor Disabilities (Keyboard Navigation):** Many users cannot use a mouse. They navigate by mashing the `Tab` key. You must ensure that every single interactive element on your page is reachable via `Tab`, and that a visible "Focus Ring" (the blue outline) clearly shows where they currently are. Never write `outline: none;` in CSS without providing an alternative focus style!

## INTERVIEW
**Q: What is the first rule of ARIA?**
A: **No ARIA is better than bad ARIA.** (And native HTML is better than both!) 
ARIA attributes do not add functionality; they only add promises. If you add `role="button"` to a `div`, it promises the Screen Reader that it is a button, but it does NOT automatically add keyboard support or the ability to trigger it with the 'Enter' key. If you fail to write the complex JavaScript to fulfill that promise, you have actively made the site worse for disabled users. Whenever mathematically possible, use a native `<button>` instead of a custom `div` with ARIA.

## FEYNMAN CHECK

### Explain HTML Accessibility Like I'm 10 Years Old
> Accessibility is making your web page work when somebody isn't using a mouse, isn't looking at the screen, or has a brain that processes things differently. The browser builds a second tree alongside the DOM called the **accessibility tree** — that's what a screen reader actually reads. Native HTML tags (`<button>`, `<nav>`, `<main>`, `<input>`) automatically have the right "role" in that tree. The moment you reach for `<div onclick>`, the browser puts role=generic in the accessibility tree — invisible to a blind user, unreachable by the Tab key.

### 5 Deep Conceptual Questions

**Q1: What problem does HTML accessibility fundamentally solve?**
> **A:** It makes the web usable for the 15% of humans with a permanent disability AND for the temporarily disabled (broken arm, baby in hand, bright sun on phone screen). It also unlocks SEO, voice control, automated testing, and reader-mode features — accessibility is the foundation of every alternative interface to your site.

**Q2: What is the ONE mental model that makes accessibility click?**
> **A:** "Native first, ARIA second, never both." If a native element does what you need, use it (`<button>`, `<input type="checkbox">`). Only reach for ARIA roles when you must build a custom widget. NEVER add a redundant ARIA role to a native element (`<button role="button">` literally breaks some screen readers).

**Q3: What is the most dangerous misconception? Show it with code.**
> **A:** That `aria-label` fixes everything:
> ```html
> <!-- ❌ aria-label only patches the screen-reader name. Not keyboard-focusable. No click semantics. -->
> <div aria-label="Close" onclick="close()">X</div>
>
> <!-- ✅ Real button: focusable, Enter/Space works, screen-reader-announced, form-friendly -->
> <button type="button" aria-label="Close" onclick="close()">×</button>
> ```

**Q4: How does focus management interact with the accessibility tree?**
> **A:** The accessibility tree tracks WHICH node currently has focus (the OS announces it). Only certain elements are focusable by default (links with `href`, form controls, `tabindex`). Adding `tabindex="0"` makes any element focusable in the normal Tab order; `tabindex="-1"` makes it programmatically focusable (via `element.focus()`) but skipped by Tab. `tabindex="1+"` is almost always a bug — it overrides natural DOM order and confuses screen-reader users.

**Q5: Write a one-sentence FAANG-precise definition.**
> **A:** "HTML accessibility is the discipline of choosing semantic elements, providing text alternatives for non-text content, managing keyboard focus order, and using ARIA only as a polyfill for custom widgets — so that the browser's accessibility tree, populated automatically by native elements, conveys interactive affordances and content structure to assistive technologies without requiring JavaScript instrumentation."

## BUILD

### 🏗️ Mini Project: An Accessible Custom Modal Dialog From Scratch

**What you will build:** A fully WCAG 2.2-compliant modal dialog with: focus trap, ESC to close, focus restoration, screen-reader announcements, and `<dialog>` element fallback to ARIA for older browsers.
**Why this project:** Modals are the most commonly broken widget on the web. Building one correctly forces you to use every accessibility primitive.
**Time estimate:** 45 minutes

#### Step 1 — Setup
```bash
mkdir a11y-modal && cd a11y-modal
ni index.html, modal.js -ItemType File
```

#### Step 2 — Use the Native `<dialog>` Element
```html
<!DOCTYPE html>
<html lang="en"><head><meta charset="UTF-8"><title>Accessible modal</title></head>
<body>
  <main>
    <h1>Account settings</h1>
    <button id="open-btn" type="button">Delete account</button>
  </main>

  <dialog id="confirm" aria-labelledby="d-title" aria-describedby="d-desc">
    <h2 id="d-title">Are you sure?</h2>
    <p id="d-desc">This will permanently delete your account and all data.</p>
    <form method="dialog">
      <button value="cancel" type="submit">Cancel</button>
      <button value="confirm" type="submit" autofocus>Delete</button>
    </form>
  </dialog>

  <script type="module" src="./modal.js"></script>
</body></html>
```

#### Step 3 — Wire It Up
```javascript
const openBtn = document.getElementById('open-btn');
const dlg     = document.getElementById('confirm');

openBtn.addEventListener('click', () => {
  dlg.showModal();                     // native: focus trap + inert background
});

dlg.addEventListener('close', () => {
  // dlg.returnValue is 'confirm' or 'cancel' — set by the button's value=
  if (dlg.returnValue === 'confirm') deleteAccount();
  openBtn.focus();                     // restore focus to the trigger
});

function deleteAccount() {
  console.log('Account deletion requested');
}
```

#### Step 4 — Fallback for Older Browsers (No `<dialog>` Support)
```javascript
// Polyfill: trap focus inside a div if <dialog> is unsupported
function manualModal(modal, trigger) {
  const focusable = modal.querySelectorAll(
    'a, button, input, select, textarea, [tabindex]:not([tabindex="-1"])'
  );
  const first = focusable[0], last = focusable[focusable.length - 1];

  modal.setAttribute('role', 'dialog');
  modal.setAttribute('aria-modal', 'true');
  modal.removeAttribute('hidden');
  first.focus();

  function onKey(e) {
    if (e.key === 'Escape') return close();
    if (e.key !== 'Tab') return;
    // Loop focus inside the modal
    if (e.shiftKey && document.activeElement === first) {
      e.preventDefault(); last.focus();
    } else if (!e.shiftKey && document.activeElement === last) {
      e.preventDefault(); first.focus();
    }
  }
  function close() {
    modal.setAttribute('hidden', '');
    document.removeEventListener('keydown', onKey);
    trigger.focus();                   // restore focus
  }

  document.addEventListener('keydown', onKey);
  modal.querySelector('[data-close]')?.addEventListener('click', close);
}
```
