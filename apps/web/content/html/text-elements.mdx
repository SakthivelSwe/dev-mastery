## WHY

Text is the primary medium of the web — every article, product description, documentation page, and error message is text. Yet most developers use only 3-4 text elements (`<p>`, `<h1>`-`<h3>`, `<span>`) and miss the rich vocabulary HTML provides for marking up text with semantic meaning. Using the wrong element — or no element at all — wastes an opportunity to communicate meaning to browsers, search engines, and assistive technologies.

Using `<strong>` instead of CSS `font-weight: bold` tells search engines "this text is important" and tells screen readers to emphasize it in voice. Using `<time datetime="2026-06-23">` instead of plain text "June 23, 2026" lets browsers, calendars, and scrapers parse the date programmatically. Using `<abbr title="CSS">` makes abbreviations accessible to users who don't know the acronym. These aren't cosmetic choices — they are the difference between a web page and a semantically rich document.

### The `<time>` Element
The `<time>` element pairs a human-readable date with a machine-readable `datetime` attribute:

```html
<time datetime="2026-06-23">June 23, 2026</time>
<time datetime="2026-06-23T14:30:00">2:30 PM, June 23</time>
<time datetime="P3D">3 days</time>           <!-- Duration: 3 days -->
<time datetime="PT2H30M">2.5 hours</time>    <!-- Duration: 2h 30m -->
```

### Level 3 — Intermediate: Time, Data Attributes & Semantic Diffs
```html
<!DOCTYPE html>
<html lang="en">
<head><meta charset="UTF-8"><title>Time & Semantic Diffs</title></head>
<body>

  <!-- TIME ELEMENT: human-readable + machine-readable datetime -->
  <article>
    <h2>Release Notes</h2>

    <!-- Full date + time in ISO 8601 format -->
    <p>Released: <time datetime="2026-06-23T09:00:00+00:00">
      June 23, 2026 at 9:00 AM UTC</time></p>

    <!-- Date only -->
    <p>Scheduled maintenance: <time datetime="2026-07-01">July 1, 2026</time></p>

    <!-- Duration (P = period, T = time separator) -->
    <p>Estimated downtime: <time datetime="PT2H30M">2 hours 30 minutes</time></p>

    <!-- Relative time (combine with JavaScript for "2 days ago") -->
    <p>Last updated: <time datetime="2026-06-21" id="last-updated">2 days ago</time></p>
  </article>

  <!-- DIFF TRACKING: <ins> and <del> for document changes -->
  <article>
    <h2>Changelog — v2.1.0</h2>

    <p>API endpoint changed from:
      <del datetime="2026-06-01" cite="/changelog/v2.0.0">
        <code>/api/v1/users</code>
      </del>
      to
      <ins datetime="2026-06-23" cite="/changelog/v2.1.0">
        <code>/api/v2/users</code>
      </ins>
    </p>

    <p>Price updated:
      <del>$99/month</del>
      <ins>$79/month</ins>
    </p>
  </article>

  <!-- DETAILS/SUMMARY: Native accordion — no JavaScript! -->
  <section>
    <h2>FAQ</h2>

    <details>
      <summary>What is HTML?</summary>
      <p>HyperText Markup Language (HTML) is the standard markup language
         for creating web pages. It describes the structure of web content.</p>
    </details>

    <details>
      <summary>What is the DOM?</summary>
      <p>The Document Object Model (DOM) is a programming interface for
         HTML documents. It represents the page as a tree of nodes.</p>
    </details>

    <details open>
      <!-- 'open' attribute makes it expanded by default -->
      <summary>What is semantic HTML?</summary>
      <p>Semantic HTML uses elements that describe their meaning to both the
         browser and the developer (e.g., &lt;article&gt; instead of &lt;div&gt;).</p>
    </details>
  </section>

  <script>
    // Update relative times based on datetime attribute
    document.querySelectorAll('time[datetime]').forEach(el => {
      const date = new Date(el.getAttribute('datetime'));
      if (!isNaN(date)) {
        const diff = Math.floor((Date.now() - date) / 86400000);
        if (diff === 0) el.textContent = 'today';
        else if (diff === 1) el.textContent = 'yesterday';
        else if (diff < 7) el.textContent = `${diff} days ago`;
        // Otherwise keep the written date
      }
    });
  </script>

</body>
</html>
```

## REAL_WORLD

In content-heavy applications like documentation sites (MDN, Dev.to), CMS platforms (WordPress, Contentful), and e-learning platforms, the quality of text markup directly affects search engine rankings through TF-IDF scoring. Search engines weight `<strong>` text higher than plain text. The `<time>` element enables Google's rich results for articles and events — pages that use `datetime` attributes appear with event dates in Google search results (increasing click-through rate by up to 20%).

## INTERVIEW

### Q1: What is the difference between `<strong>` and CSS `font-weight: bold`?
**Difficulty:** Easy | **Companies:** All

❌ **Weak answer:** "strong makes text bold, CSS bold also makes it bold, they look the same."

✅ **Strong answer:**
"`<strong>` is a **semantic HTML element** that communicates **strong importance** — it affects the accessibility tree (screen readers may emphasize the word differently), search engine indexing (Google weights it higher in keyword analysis), and browser default styles (bold as a side effect). `font-weight: bold` in CSS is **purely presentational** — it changes the visual appearance only, with zero semantic impact. A screen reader on text styled `font-weight: bold` reads it identically to surrounding text. A screen reader on `<strong>` text announces it with emphasis. Best practice: use `<strong>` when the text is genuinely important/critical; use CSS bold for purely aesthetic purposes like product names or UI labels."

### Q2: When should you use `<time>` vs just writing a date as plain text?
**Difficulty:** Medium | **Companies:** News sites, Google, Content platforms

❌ **Weak answer:** "It's just a tag for dates, doesn't really matter."

✅ **Strong answer:**
"The `<time>` element pairs human-readable text with a **machine-readable `datetime` attribute** in ISO 8601 format. This enables: (1) **Google rich results** — article publish dates, event dates, and durations appear in search results when marked with `<time>` + proper Schema.org structured data. (2) **Browser/OS integration** — some browsers can add dates to calendar apps from `<time>` elements. (3) **JavaScript processing** — `timeEl.getAttribute('datetime')` gives a reliable, parseable string regardless of how the date is displayed (locale, abbreviation, relative). (4) **Internationalization** — you can display `June 23, 2026` in English but `23. Juni 2026` in German while the `datetime` remains universal. Plain text dates (`June 23, 2026`) are meaningless to machines; `<time datetime='2026-06-23'>` is unambiguous."

### Q3: What is the purpose of `<abbr>` and `<dfn>`?
**Difficulty:** Easy | **Companies:** Documentation platforms

❌ **Weak answer:** "abbr shows abbreviations, dfn defines terms."

✅ **Strong answer:**
"`<abbr title='full expansion'>abbrev</abbr>` marks up an abbreviation or acronym and provides its full form via the `title` attribute, displayed as a tooltip on hover and read by screen readers. It makes content accessible to users unfamiliar with the abbreviation. `<dfn>` marks the **defining instance** of a term — the first place a term is formally defined on the page. By convention, `<dfn>` should appear near the definition, and all subsequent uses can link back to it via `id`/`href`. They can be combined: `<dfn><abbr title='Cascading Style Sheets'>CSS</abbr></dfn>` for terms that are both being defined and are abbreviations. Search engines use `<dfn>` signals to identify glossary content and may surface it in Knowledge Panels."

### Q4: What is the `<kbd>` element and why does it matter for documentation?
**Difficulty:** Easy | **Companies:** Developer tools companies, Documentation

❌ **Weak answer:** "It shows keyboard shortcuts in a special style."

✅ **Strong answer:**
"`<kbd>` (Keyboard Input) semantically marks text that represents user keyboard input — keys to press, commands to type, keyboard shortcuts. Screen readers announce it with a different tone or phrasing. For example: `Press <kbd>Ctrl</kbd>+<kbd>S</kbd> to save` tells screen reader users and voice control software that these are keyboard keys, not just styled text. The browser's default styling (monospace font, sometimes a border) visually distinguishes it. Related elements: `<samp>` for **sample output** from a program, `<code>` for **code snippets**, `<var>` for **variables in math/code**. Using the wrong element (e.g., `<code>Ctrl+S</code>` for a keyboard shortcut) is technically incorrect though visually identical."

### Q5: How does the `<pre>` element work and what are its use cases?
**Difficulty:** Easy | **Companies:** All

❌ **Weak answer:** "pre shows text in monospace and keeps the formatting."

✅ **Strong answer:**
"`<pre>` (Preformatted Text) displays content with **all whitespace preserved** — spaces, tabs, and newlines are rendered exactly as written in the HTML (unlike normal HTML where whitespace collapses). It uses a monospace font by default. Primary use cases: (1) **Code samples** — always use `<pre><code>` together for code blocks: `<pre>` preserves indentation, `<code>` marks it semantically as code. (2) **ASCII art**. (3) **Terminal/command output** where column alignment is significant. Important gotcha: any HTML entities inside `<pre>` still need escaping — `<`, `>`, `&` must be `&lt;`, `&gt;`, `&amp;`. The CSS property `white-space: pre` on any element replicates `<pre>`'s whitespace behavior without the semantic meaning or monospace font."

## FEYNMAN CHECK

### Explain HTML Text Elements Like I'm 10 Years Old
> HTML has TWO types of "make it bold" and TWO types of "make it italic" — but they mean different things. `<strong>` means "this is IMPORTANT" (a screen reader changes its tone); `<b>` means "make it visually bold with no meaning." Same for `<em>` (emphasis, changes pronunciation) vs `<i>` (italic style — for ship names, foreign words). Picking the wrong one is invisible to sighted users but completely changes how the page sounds to a blind user. This is why "use `<strong>` not `<b>`" is one of the most cited beginner mistakes in WCAG audits.

### 5 Deep Conceptual Questions

**Q1: What problem does semantic text markup fundamentally solve?**
> **A:** It separates VISUAL presentation (CSS owns) from SEMANTIC importance (the document owns). Swap your design system and `<strong>` still gets announced with stress by screen readers; `<b>` won't — it's purely a style hook surviving for legacy reasons.

**Q2: What is the ONE mental model that makes text-element choice click?**
> **A:** Ask: "If I removed all CSS, does this word still need to look or sound different?" If YES → use a semantic tag (`<strong>`, `<em>`, `<mark>`, `<code>`, `<abbr>`). If NO → it's just styling, use `<span>`. The user-agent stylesheet exists precisely so semantic tags carry visual defaults too.

**Q3: What is the most dangerous misconception? Show it with code.**
> **A:** Using `<br>` for spacing instead of paragraph structure:
> ```html
> <!-- ❌ Screen reader announces "Line break, line break, line break" — exhausting -->
> Hello there.<br><br>
> Welcome to the site.<br><br>
>
> <!-- ✅ Three distinct paragraphs, properly announced and properly margined -->
> <p>Hello there.</p>
> <p>Welcome to the site.</p>
> ```

**Q4: How does `<code>` + `<pre>` interact with whitespace and the rendering engine?**
> **A:** `<pre>` preserves whitespace and newlines exactly (uses monospace); `<code>` is just inline monospace, no whitespace preservation. Combining `<pre><code>...</code></pre>` is what Markdown converts to — `<pre>` handles layout, `<code>` carries the "this is source code" semantic so screen readers announce it as code and syntax highlighters know what to target.

**Q5: Write a one-sentence FAANG-precise definition.**
> **A:** "Semantic text elements are inline tags whose default ARIA semantics influence how assistive tech announces emphasis, abbreviation expansions, code spans, deletions, and inserted content — which is why swapping `<b>` for `<strong>` in a 10-year-old codebase changes the audible experience for screen-reader users with zero visual change."

### 🏗️ Mini Project: Technical Docs Page Using Every Semantic Text Tag Correctly

**What you will build:** A "Keyboard shortcuts reference" page using every inline text semantic: `<code>`, `<kbd>`, `<samp>`, `<var>`, `<mark>`, `<abbr>`, `<del>`, `<ins>`, `<sup>`, `<sub>`, `<q>`, `<cite>`, `<time>`.
**Why this project:** Most devs only know `<strong>` and `<em>`. The full inline vocabulary separates a junior from a senior on accessibility-sensitive products.
**Time estimate:** 25 minutes

---

#### Step 1 — Setup
```bash
mkdir text-elements-demo && cd text-elements-demo
ni shortcuts.html -ItemType File
```

#### Step 2 — Core Document
```html
<!DOCTYPE html>
<html lang="en">
<head><meta charset="UTF-8"><title>Keyboard Shortcuts — VS Code</title></head>
<body>
  <main>
    <h1>Keyboard Shortcuts <small>(<abbr title="Visual Studio Code">VSC</abbr> 1.92)</small></h1>
    <p>Last updated <time datetime="2026-06-25">June 25, 2026</time>.</p>

    <h2>Editing</h2>
    <p>Duplicate the current line with
       <kbd>Shift</kbd>+<kbd>Alt</kbd>+<kbd>Down</kbd>.</p>

    <p>Open the command palette with
       <kbd><kbd>Ctrl</kbd>+<kbd>Shift</kbd>+<kbd>P</kbd></kbd>.</p>
  </main>
</body></html>
```

#### Step 3 — Inline Semantics for Code, Output, Variables
```html
<h2>Run a script</h2>
<p>Run <code>npm test</code>. You should see output like:</p>

<pre><samp>PASS  ./auth.test.js
  ✓ rejects invalid token (12 ms)
  ✓ accepts valid token  ( 8 ms)
</samp></pre>

<p>Replace <var>file</var> with the actual filename:
   <code>node <var>file</var>.js</code>.</p>
```

#### Step 4 — Highlights, Quotes, Edits, Math
```html
<h2>Recent changes</h2>
<p>The shortcut <del datetime="2026-04-01">Ctrl+B</del>
   <ins datetime="2026-04-01"><kbd>Ctrl</kbd>+<kbd>K</kbd> <kbd>Ctrl</kbd>+<kbd>B</kbd></ins>
   now toggles the sidebar.</p>

<p>As <cite>The VS Code Docs</cite> note: <q>Keybindings can be
   user-customised per workspace.</q></p>

<p>Search results for <mark>keybinding</mark> are highlighted in the margin.</p>

<p>Chemistry: H<sub>2</sub>O. Math: E = mc<sup>2</sup>.</p>
```

#### Step 5 — Tests
```bash
npx -y html-validate shortcuts.html

# Manual: open with NVDA / VoiceOver and confirm:
#   - <kbd> announced as "Ctrl" (not styled text)
#   - <mark> announced with "marked" or "highlight" prefix
#   - <abbr title=...> announced as full expansion on first read
#   - <del>/<ins> announced as "deleted" / "inserted"
#   - <q> wrapped in audible quote marks
```

**Expected Output:**
```
html-validate: 0 errors, 0 warnings
```
