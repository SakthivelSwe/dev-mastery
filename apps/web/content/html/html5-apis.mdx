## WHY
Before 2014, if you wanted to play a video on a website, save data locally, or get the user's GPS location, pure HTML/JavaScript could not do it. You had to force the user to download and install a heavy, insecure 3rd-party plugin called Adobe Flash. 

**HTML5** revolutionized the web by introducing native, standardized APIs built directly into the browser. It transformed the web from a collection of static text documents into a platform capable of running full-blown applications (like Netflix, Google Maps, and Spotify) without any plugins.

## THEORY
### What is a Web API?
An API (Application Programming Interface) is a set of rules that allows code to talk to other code. A "Web API" specifically refers to the tools the Browser exposes to your JavaScript code. You write JavaScript, the JS talks to the Web API, and the Web API talks to the underlying Operating System (Windows/macOS) to access the hard drive, microphone, or camera.

## CODE

### Level 1 — Beginner (Audio and Video)
HTML5 introduced native media tags. No Flash required! 
You can control the video purely through HTML attributes, or use JavaScript to build custom Play/Pause buttons.

```html
<!-- The 'controls' attribute tells the browser to automatically render 
     the play button, volume slider, and timeline! -->
<video width="640" controls>
  <!-- You provide multiple source files; the browser picks the first one it supports -->
  <source src="/media/tutorial.mp4" type="video/mp4">
  <source src="/media/tutorial.webm" type="video/webm">
  Your browser does not support the video tag.
</video>

<!-- Audio works the exact same way -->
<audio controls>
  <source src="/media/podcast.mp3" type="audio/mpeg">
</audio>
```

### Level 2 — Building Blocks (Web Storage API)
Historically, if you wanted to remember that a user preferred "Dark Mode", you had to save it in a Cookie. Cookies are terrible for storing app state because every single cookie is automatically sent to the server on every single HTTP request, wasting bandwidth.

HTML5 introduced **localStorage** and **sessionStorage**. They allow you to store up to 5MB of string data directly on the user's hard drive, and the data is *never* sent to the server automatically.

```javascript
// 1. localStorage: Persists forever (until the user manually clears their cache)
localStorage.setItem('theme', 'dark');

const userTheme = localStorage.getItem('theme');
console.log(userTheme); // 'dark'

// 2. sessionStorage: Automatically deleted the second the user closes the browser tab!
sessionStorage.setItem('shopping_cart_id', '12345');
```
*(Note: Because Web Storage only accepts Strings, if you want to save a complex Object or Array, you MUST use `JSON.stringify()` before saving, and `JSON.parse()` when reading!)*

### Level 3 — Intermediate (Geolocation API)
Want to show the user restaurants near them? You can ask the browser for their exact GPS coordinates! (The browser will pop up a strict security prompt asking the user for permission first).

```javascript
// Check if the browser supports Geolocation
if (navigator.geolocation) {
  
  // Ask for the location!
  navigator.geolocation.getCurrentPosition(
    (position) => {
      // Success Callback!
      console.log("Latitude:", position.coords.latitude);
      console.log("Longitude:", position.coords.longitude);
    },
    (error) => {
      // Error Callback! (User clicked "Deny" or GPS is turned off)
      console.error("Location access denied.");
    }
  );
  
} else {
  console.log("Geolocation is not supported by this browser.");
}
```

### Level 4 — Advanced (Canvas API)
If you want to build a 2D web game (like Mario) or draw complex charts and graphs, manipulating standard HTML `<div>` tags is far too slow. 
The HTML5 `<canvas>` tag provides a blank rectangular pixel drawing board. You use JavaScript to rapidly draw and erase shapes, lines, and images at 60 frames per second.

```html
<!-- The HTML is just an empty box -->
<canvas id="gameBoard" width="400" height="400"></canvas>
```

```javascript
// The JavaScript does the heavy lifting
const canvas = document.getElementById('gameBoard');
// Get the 2D rendering context
const ctx = canvas.getContext('2d');

// Draw a red rectangle
ctx.fillStyle = 'red';
ctx.fillRect(50, 50, 100, 100); // x: 50, y: 50, width: 100, height: 100

// Draw a blue circle
ctx.beginPath();
ctx.arc(200, 200, 40, 0, 2 * Math.PI); // x, y, radius, startAngle, endAngle
ctx.fillStyle = 'blue';
ctx.fill();
```

## REAL_WORLD
**Security and HTTPS**
Because HTML5 APIs are so incredibly powerful (accessing your GPS, accessing your webcam/microphone via the `getUserMedia` API, or reading your hard drive), browser vendors enforce strict security protocols. 
**Almost all modern Web APIs will completely fail to load if your website is not served over HTTPS!** 
If you are developing locally on `http://localhost`, the browser grants a special exception, but the moment you deploy to a real `.com` domain, you absolutely must have an SSL certificate configured, or the browser will block access to the Geolocation and Media APIs entirely to protect the user from malicious interception.

## INTERVIEW
**Q: Can I store a user's JWT (Authentication Token) or their Credit Card number in `localStorage`?**
A: **Absolutely NOT.** `localStorage` is vulnerable to **XSS (Cross-Site Scripting)** attacks. If a hacker manages to inject a single line of malicious JavaScript into your website (e.g., via a compromised NPM package or an unescaped comment forum), that script has full, unrestricted access to `localStorage`. The script can read the JWT and send it to the hacker's server, completely stealing the user's account. Authentication tokens should ideally be stored in `HttpOnly` Cookies, which are mathematically impossible for JavaScript (and therefore XSS attackers) to read.

## FEYNMAN CHECK

### Explain HTML5 APIs Like I'm 10 Years Old
> HTML5 quietly bundled JavaScript with about 40 new built-in APIs that browsers expose to your page — Geolocation, Canvas, Web Storage, History, Drag-and-Drop, File API, Web Workers, IndexedDB, Notifications, Web Audio, WebSockets, Service Workers, Intersection Observer, ResizeObserver, and more. They're all gated by a *permission model*: dangerous ones (camera, GPS, notifications) require explicit user consent; safe ones (localStorage, History) don't. Together they're why a browser tab can now replace what used to require a desktop app — Figma, VS Code, Google Maps all run on these APIs.

### 5 Deep Conceptual Questions

**Q1: What problem do HTML5 APIs fundamentally solve?**
> **A:** They give web pages access to platform capabilities (storage, location, sensors, files, threads, GPU, networking) that previously required a native app. Without them you couldn't build offline-capable web apps, real-time chat, in-browser image editing, or web-based games — every "Can I move this to the web?" decision since 2010 depended on which HTML5 API closed the gap.

**Q2: What is the ONE mental model that makes HTML5 APIs click?**
> **A:** Every API is in one of three tiers: (1) always-on (Web Storage, History, Canvas — no permission needed); (2) one-time-prompt (Geolocation, Notifications, Camera — user clicks "Allow"); (3) install-required (Service Worker, Push, Background Sync — only after the page meets PWA criteria over HTTPS). Knowing the tier tells you what UX to design around the API.

**Q3: What is the most dangerous misconception? Show it with code.**
> **A:** Treating `localStorage` as secure or large-capacity:
> ```javascript
> // ❌ DANGEROUS: synchronous (blocks main thread), readable by ANY script on the page, ~5MB cap
> localStorage.setItem('jwt', 'eyJhbGc...');         // XSS-stealable
> localStorage.setItem('photos', JSON.stringify(arr)); // hits quota after a few large images
>
> // ✅ Use the right API for the job
> // - Auth tokens: HttpOnly Secure cookie set by server (JS can't read)
> // - Large/structured data: IndexedDB (async, gigabytes, structured queries)
> const db = await indexedDB.open('photos', 1);      // async, ~unlimited
> ```

**Q4: How does the Intersection Observer API interact with the browser's rendering pipeline?**
> **A:** Instead of running expensive `getBoundingClientRect()` on every scroll event (which forces synchronous layout), Intersection Observer registers a callback that fires AFTER the browser has computed layout — batched, off the critical path. It's why infinite-scroll lists, lazy-loaded images, and viewport analytics scale to thousands of observed elements without jank. The browser groups intersection changes into one callback per animation frame.

**Q5: Write a one-sentence FAANG-precise definition.**
> **A:** "HTML5 APIs are the suite of browser-exposed platform capabilities — storage, sensors, threading, networking, lifecycle, observers — gated by a three-tier permission model and accessed via JavaScript globals (`navigator`, `window`, `document`, registered service workers) — which together close the gap between web pages and native apps and underpin every offline-capable, real-time, or hardware-aware product running in a browser tab."

#### Step 2 — Storage Layer (IndexedDB for notes, localStorage for settings)
```javascript
// app.js — IndexedDB wrapper for notes
const dbOpen = indexedDB.open('notes-db', 1);
dbOpen.onupgradeneeded = (e) => {
  e.target.result.createObjectStore('notes', { keyPath: 'id', autoIncrement: true });
};
const db = await new Promise(r => dbOpen.onsuccess = e => r(e.target.result));

async function addNote(text) {
  const tx = db.transaction('notes', 'readwrite');
  await tx.objectStore('notes').add({ text, created: Date.now() });
}

async function getAllNotes() {
  const tx = db.transaction('notes', 'readonly');
  return new Promise(r => {
    const req = tx.objectStore('notes').getAll();
    req.onsuccess = () => r(req.result);
  });
}

// localStorage for small settings only
const settings = {
  get theme()  { return localStorage.getItem('theme') ?? 'light'; },
  set theme(v) { localStorage.setItem('theme', v); }
};
```

#### Step 3 — History API for Deep Linking
```javascript
// Each opened note pushes a state — back button works without a router
function openNote(id) {
  history.pushState({ noteId: id }, '', '#note-' + id);
  render(id);
}

// Restore state when the user navigates back/forward
window.addEventListener('popstate', (e) => {
  if (e.state?.noteId) render(e.state.noteId);
  else renderList();
});
```

#### Step 4 — Service Worker for Offline + Notifications API for Reminders
```javascript
// app.js — register the service worker
if ('serviceWorker' in navigator) {
  await navigator.serviceWorker.register('/sw.js');
}

// Permission-gated notification (Tier 2)
async function remindMe(noteId, atTime) {
  const perm = await Notification.requestPermission();
  if (perm !== 'granted') return;            // user denied — fall back gracefully
  const delay = atTime - Date.now();
  setTimeout(() => {
    new Notification('Reminder', { body: 'Open note #' + noteId, tag: 'note-' + noteId });
  }, delay);
}
```
```javascript
// sw.js — cache-first for offline support
const CACHE = 'notes-v1';
self.addEventListener('install', (e) => {
  e.waitUntil(caches.open(CACHE).then(c => c.addAll(['/', '/app.js', '/manifest.webmanifest'])));
});
self.addEventListener('fetch', (e) => {
  e.respondWith(caches.match(e.request).then(r => r || fetch(e.request)));
});
```

#### Step 5 — Tests
```javascript
// Intersection Observer for lazy-loading note thumbnails
const io = new IntersectionObserver((entries) => {
  for (const entry of entries) {
    if (entry.isIntersecting) {
      const img = entry.target;
      img.src = img.dataset.src;             // swap data-src → src
      io.unobserve(img);
    }
  }
});
document.querySelectorAll('img[data-src]').forEach(img => io.observe(img));
```
