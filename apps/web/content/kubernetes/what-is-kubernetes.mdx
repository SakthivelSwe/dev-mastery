## WHY
Docker Compose runs multiple containers on **one machine**. But what happens in production when one machine can't handle all your traffic? You need to run containers across **many machines** (a cluster).
Running 100 containers across 20 servers manually is impossible. Who decides which server runs which container? What happens when a server crashes? How do you update your app without downtime?
**Kubernetes** (K8s) is the industry-standard platform that automates all of this. It manages a cluster of machines as a single unit, automatically scheduling containers, healing failures, scaling horizontally, and rolling out updates — with zero downtime.

## THEORY
### The Core Problem Kubernetes Solves
- **Scheduling:** "I have 100 containers to run and 20 servers. Put each container on the best available server."
- **Self-healing:** "Server 7 crashed. Automatically reschedule those 20 containers on other servers."
- **Scaling:** "Traffic spiked! Add 50 more containers instantly. Traffic dropped? Remove them."
- **Rolling updates:** "Deploy v2.0 without a single dropped request."
- **Service discovery:** "Container A needs to call Container B. Auto-resolve the IP."

### Why "K8s"?
Kubernetes comes from the Greek word for "helmsman" or "pilot." The abbreviated "K8s" replaces the 8 letters between K and s.

```bash
# Install kubectl (the Kubernetes CLI)
# Mac: brew install kubectl
# Windows: choco install kubernetes-cli

# Deploy a simple nginx application

# http://192.168.64.2:32100
```

## REAL_WORLD
**Who Uses Kubernetes?**
- **Google:** Kubernetes was built by Google, based on their internal system Borg. Google runs billions of containers per week.
- **Airbnb, Spotify, Uber, Netflix:** All run critical services on Kubernetes.
- **Cloud Providers:** AWS (EKS), Google Cloud (GKE), Azure (AKS) all offer managed Kubernetes. You just create a cluster — they manage the control plane.

**The managed Kubernetes advantage:** With EKS/GKE/AKS, the cloud provider operates the control plane (API servers, etcd, etc.). You only manage your worker nodes (or use Fargate/Autopilot for fully managed workers). This dramatically reduces operational overhead.

## INTERVIEW
**Q: A startup is running their application with Docker Compose on a single large server. When should they migrate to Kubernetes?**
A: Migration to Kubernetes is justified when you face one or more of these challenges:

1. **Need for horizontal scaling beyond one machine:** Traffic requires more compute than one server can provide.
2. **High availability requirements:** You need automatic failover if a server crashes (zero single-point-of-failure).
3. **Multiple services with complex dependencies:** 10+ microservices that need sophisticated networking and service discovery.
4. **Advanced deployment strategies:** Blue-green, canary deployments, or circuit breakers at the infrastructure level.
5. **Team has DevOps capacity:** Kubernetes requires significant operational expertise and dedicated maintenance.

**Warning signs that Kubernetes is premature:**
- Small team (< 5 engineers) with limited DevOps resources
- Simple application (1-3 services)
- Budget constraints (managed K8s clusters cost $100+/month minimum for production)

**Intermediate alternative:** Docker Swarm provides multi-machine orchestration with much less complexity, and can be a stepping stone to Kubernetes.
