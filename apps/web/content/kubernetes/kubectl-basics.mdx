## WHY
`kubectl` is your primary interface for interacting with Kubernetes. Every operation — deploying applications, debugging pods, checking logs, scaling services — goes through kubectl. Mastering it is the foundation of all Kubernetes work.

## THEORY
### The kubectl → API Server Pattern
Every kubectl command sends an HTTP request to the **Kubernetes API Server**. kubectl reads your `~/.kube/config` file to know which cluster to talk to and with which credentials.

### Context and Config
If you have multiple clusters (dev, staging, prod), kubectl uses **contexts** to switch between them. Getting this wrong means running `kubectl delete pod` on the wrong cluster!

## CODE

### Essential kubectl Commands
```bash
# --- GETTING RESOURCES ---
kubectl get pods                    # List pods in current namespace
kubectl get pods -n kube-system     # Different namespace
kubectl get pods --all-namespaces   # All namespaces
kubectl get pods -o wide            # Extra info (Node, IP)
kubectl get pods -o yaml            # Full YAML output
kubectl get pods -l app=api         # Filter by label
kubectl get all                     # Everything in namespace

# --- APPLYING/CREATING RESOURCES ---
kubectl apply -f deployment.yml     # Create or update
kubectl apply -f ./k8s/             # Apply all YAMLs in a directory
kubectl apply -k ./kustomization/   # Kustomize directory
kubectl delete -f deployment.yml    # Delete from file

# --- DESCRIBING RESOURCES ---
kubectl describe pod my-pod         # Detailed info + events
kubectl describe node worker-01     # Node capacity and conditions

# --- LOGS ---
kubectl logs my-pod                 # Logs from default container
kubectl logs my-pod -c sidecar      # Specific container
kubectl logs -f my-pod              # Follow logs
kubectl logs my-pod --previous      # Logs from crashed previous instance!

# --- EXEC ---
kubectl exec -it my-pod -- bash     # Interactive shell
kubectl exec my-pod -- env          # Run non-interactive command

# --- PORT FORWARDING ---
kubectl port-forward pod/my-pod 8080:3000      # Pod to localhost
kubectl port-forward svc/my-service 8080:80   # Service to localhost

# --- CONTEXT MANAGEMENT ---
kubectl config get-contexts         # List all contexts
kubectl config current-context      # Which cluster am I pointing to?
kubectl config use-context prod-cluster  # Switch to production
kubectl config use-context dev-cluster   # Switch to dev (SAFER!)
```

```bash
# Tired of typing -n myapp-namespace every time?
kubectl config set-context --current --namespace=myapp-namespace

# Now all commands default to myapp-namespace without -n flag!
kubectl get pods  # Same as: kubectl get pods -n myapp-namespace
```

## INTERVIEW
**Q: You accidentally ran `kubectl delete deployment api` in the production namespace instead of the dev namespace. You have no backups. How do you recover?**
A: First, stop panicking. Here's the recovery path:

```bash
# 1. Check if any pods are still running (delete is not always instant)
kubectl get pods -n production -l app=api

# 2. Check if the ReplicaSet still exists (RS survives after Deployment deletion briefly)
kubectl get replicaset -n production

# 3. Immediately re-apply the deployment from your Git repository!
# This is why all K8s manifests must be in version control!
git checkout main -- k8s/production/deployment.yaml
kubectl apply -f k8s/production/deployment.yaml -n production
```

The hard lesson: **never run `kubectl` directly against production**. All changes should go through GitOps (ArgoCD, Flux) or at minimum require PR review. `kubectl apply` to production should be done only by CI/CD, never manually. Also, `kubectl config use-context` should be the first thing you check before any destructive command.
