## WHY
Deploying a production-grade application to Kubernetes requires many YAML files: Deployment, Service, ConfigMap, Secret, Ingress, HPA, RBAC roles, ServiceAccount... easily 10-20 YAML files per application.
Managing all these files manually — customizing them for different environments (dev, staging, prod), sharing them with other teams, versioning them, and rolling back to previous configurations — is an operational nightmare.
**Helm** is the package manager for Kubernetes. It bundles all YAML files for an application into a **Chart**, allows parameterization via templates, and treats deployments like versioned releases that can be installed, upgraded, and rolled back with single commands.

## THEORY
### Helm Concepts
- **Chart:** A collection of YAML templates + a `values.yml` file. Like an npm package for Kubernetes.
- **Release:** A specific instance of a Chart installed in a cluster (you can install the same chart multiple times with different configurations, creating multiple releases).
- **Repository:** A collection of Charts (like Docker Hub, but for Helm charts). Public repos: ArtifactHub, Bitnami, etc.
- **Values:** Configuration parameters that customize a Chart's templates.

## CODE

### Installing Applications with Helm
```bash
# Add a Helm chart repository
helm repo add bitnami https://charts.bitnami.com/bitnami
helm repo update

# Install PostgreSQL from the Bitnami chart
helm install my-postgres bitnami/postgresql \
  --set auth.postgresPassword=mysecretpassword \
  --set primary.persistence.size=20Gi

# Install cert-manager for TLS certificates
helm repo add jetstack https://charts.jetstack.io
helm install cert-manager jetstack/cert-manager \
  --namespace cert-manager \
  --create-namespace \
  --version v1.13.0 \
  --set installCRDs=true

# List all helm releases
helm list
# NAME          NAMESPACE   REVISION   STATUS    CHART
# my-postgres   default     1          deployed  postgresql-13.2.0

# Upgrade a release
helm upgrade my-postgres bitnami/postgresql --set auth.postgresPassword=newpassword

# Rollback to previous revision
helm rollback my-postgres 1

# Uninstall
helm uninstall my-postgres
```

### Creating Your Own Chart
```bash
# Scaffold a new chart
helm create my-api-chart

# Chart structure:
# my-api-chart/
# ├── Chart.yaml        # Chart metadata (name, version, description)
# ├── values.yaml       # Default configuration values
# ├── templates/        # YAML templates with Go templating
# │   ├── deployment.yaml
# │   ├── service.yaml
# │   ├── ingress.yaml
# │   └── _helpers.tpl  # Reusable template fragments
```

```yaml
# values.yaml (default values — teams override these)
image:
  repository: my-api
  tag: "1.0.0"
  pullPolicy: IfNotPresent

replicaCount: 1

service:
  type: ClusterIP
  port: 80
  targetPort: 3000

resources:
  limits:
    memory: 512Mi
    cpu: 500m
  requests:
    memory: 256Mi
    cpu: 250m

ingress:
  enabled: false
  host: ""
```

```yaml
# templates/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ .Release.Name }}-deployment   # Release name injected!
spec:
  replicas: {{ .Values.replicaCount }}   # From values.yaml!
  template:
    spec:
      containers:
        - name: api
          image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
          resources: {{- toYaml .Values.resources | nindent 12 }}
```

### Installing with Environment-Specific Values
```bash
# values.prod.yaml
replicaCount: 5
image:
  tag: "2.1.0"
resources:
  limits:
    memory: 2Gi
    cpu: "2"
ingress:
  enabled: true
  host: api.mycompany.com

# Install for production
helm upgrade --install my-api ./my-api-chart \
  -f values.yaml \
  -f values.prod.yaml \
  --namespace production \
  --create-namespace
```

## REAL_WORLD
**ArtifactHub: The Helm Chart Registry**
ArtifactHub (artifacthub.io) is the public registry for Helm charts. You can find production-ready charts for almost any software:
- **Monitoring:** Prometheus, Grafana, AlertManager
- **Ingress:** Nginx Ingress Controller, Traefik
- **Certificates:** cert-manager (Let's Encrypt)
- **Databases:** PostgreSQL, MySQL, Redis, MongoDB
- **CI/CD:** ArgoCD, Tekton, Jenkins

Instead of writing 500 lines of Kubernetes YAML to deploy Prometheus, you run: `helm install prometheus prometheus-community/kube-prometheus-stack`. Done.

## INTERVIEW
**Q: You want to deploy the same application to 3 environments (dev, staging, production) with different configurations (replica count, resource limits, image tags). How do you manage this with Helm without duplicating YAML files?**
A: Use **Helm's values override system** with environment-specific values files:

```
charts/
└── my-app/
    ├── Chart.yaml
    ├── templates/         # Single set of templates (no duplication!)
    ├── values.yaml        # Base/default values (committed to git)
    ├── values.dev.yaml    # Dev overrides: replicaCount=1, image.tag=latest
    ├── values.staging.yaml  # Staging: replicaCount=2, image.tag=rc-1.2.0
    └── values.prod.yaml   # Prod: replicaCount=10, image.tag=1.2.0
```

Deploy with:
```bash
helm upgrade --install myapp ./charts/my-app -f values.yaml -f values.${ENV}.yaml
```

Helm merges the values files (right-to-left priority), so environment-specific values override the base defaults. This gives you:
- **Single source of truth** for templates
- **Environment-specific configuration** without duplication
- **Version control** over both templates and configurations
- **Easy comparison** between environments (just diff the values files)
