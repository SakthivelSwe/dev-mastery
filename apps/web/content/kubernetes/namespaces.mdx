## WHY
A Kubernetes cluster can be shared by multiple teams, environments, or projects. Without organization, every team's resources (Pods, Services, ConfigMaps) would all exist in the same flat space, causing name collisions and making it impossible to apply team-specific access controls or resource quotas.
**Namespaces** provide virtual partitioning within a cluster. Each namespace is an isolated scope for Kubernetes resources — separate from other namespaces.

## THEORY
### Default Namespaces
- **default:** Where resources go when you don't specify a namespace
- **kube-system:** Kubernetes internal components (DNS, API server, etc.)
- **kube-public:** Readable by all users; rarely used
- **kube-node-lease:** Node heartbeat leases for availability detection

### What's Namespace-Scoped vs Cluster-Scoped?
- **Namespace-scoped:** Pods, Services, Deployments, ConfigMaps, Secrets, Ingress
- **Cluster-scoped:** Nodes, PersistentVolumes, ClusterRoles, Namespaces themselves

## REAL_WORLD
**Namespace-per-Environment Pattern**
```
cluster/
├── namespace: production    # Team: prod engineers, strict RBAC
├── namespace: staging       # Team: QA + engineers
├── namespace: development   # Team: all developers
└── namespace: monitoring    # Prometheus, Grafana, AlertManager
```

Benefits:
- Team A cannot accidentally delete Team B's resources
- Resource quotas prevent one team from consuming all cluster resources
- Different deployment configurations per namespace
- Easy to delete an environment: `kubectl delete namespace staging` removes EVERYTHING in it!

## INTERVIEW
**Q: You have 10 microservices in the `production` namespace. A Service in service-A needs to call service-B by hostname. What hostname does service-A use? What is the full DNS name?**
A: Within the same namespace, service-A can use just the **service name**: `http://service-b:8080`.

The full DNS name follows this pattern:
`<service-name>.<namespace>.svc.cluster.local:<port>`

So: `http://service-b.production.svc.cluster.local:8080`

When you omit parts, Kubernetes DNS resolves them:
- `service-b` → resolves within the same namespace first
- `service-b.production` → resolves in the `production` namespace
- `service-b.production.svc.cluster.local` → fully qualified, always resolves correctly

If service-A (in `production` namespace) needs to call service-C in a different namespace (e.g., `monitoring` namespace), you must use the namespace-qualified name: `http://grafana.monitoring.svc.cluster.local:3000`.
