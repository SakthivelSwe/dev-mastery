## WHY
Applications need sensitive data: database passwords, API keys, TLS certificates, OAuth secrets. These must never be hardcoded in your Docker image or ConfigMaps.
**Kubernetes Secrets** provide a mechanism to store sensitive data separately from application code and inject it into Pods at runtime. Access is controlled by RBAC, and Secrets can be encrypted at rest.

## THEORY
### Important: Secrets are Base64, NOT Encrypted
The default Kubernetes Secret stores data as Base64-encoded strings — NOT encrypted! `echo -n "password" | base64` is just encoding, not security.
For real security, you must either:
1. Enable **Encryption at Rest** in the API server (`EncryptionConfiguration`)
2. Use an external secrets manager (AWS Secrets Manager, HashiCorp Vault)
3. Use **Sealed Secrets** (encrypted secrets stored in Git)

## CODE

### Creating Secrets
```bash
# From literal values
kubectl create secret generic db-secret \
  --from-literal=username=postgres \
  --from-literal=password=S3cur3P@ss!

# From a file
kubectl create secret generic tls-certs \
  --from-file=tls.crt=./cert.crt \
  --from-file=tls.key=./cert.key

# TLS secret (for Ingress HTTPS)
kubectl create secret tls my-tls-secret \
  --cert=tls.crt \
  --key=tls.key
```

```yaml
# secret.yml (values are base64-encoded)
apiVersion: v1
kind: Secret
metadata:
  name: db-secret
type: Opaque
data:
  username: cG9zdGdyZXM=    # echo -n "postgres" | base64
  password: UzNjdXIzUEBzcyE=  # echo -n "S3cur3P@ss!" | base64

# Or use stringData (auto-encoded — easier!)
stringData:
  username: postgres          # Plain text — Kubernetes encodes automatically
  password: "S3cur3P@ss!"
```

### Using Secrets in Pods
```yaml
spec:
  containers:
    - name: api
      image: my-api:1.0.0

      # Method 1: As environment variables
      env:
        - name: DB_USERNAME
          valueFrom:
            secretKeyRef:
              name: db-secret
              key: username
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:
              name: db-secret
              key: password

      # Method 2: Mount as files (more secure!)
      volumeMounts:
        - name: secret-volume
          mountPath: /app/secrets  # Files appear here!
          readOnly: true

  volumes:
    - name: secret-volume
      secret:
        secretName: db-secret
        # Files created: /app/secrets/username, /app/secrets/password
```

### External Secrets with AWS Secrets Manager (Production Best Practice)
```yaml
# external-secrets.yml (using External Secrets Operator)
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: db-credentials
spec:
  refreshInterval: 1h           # Re-sync from AWS every hour
  secretStoreRef:
    name: aws-secretsmanager    # Points to your AWS config
    kind: ClusterSecretStore
  target:
    name: db-secret             # Creates a K8s Secret with this name
  data:
    - secretKey: password       # Key in K8s Secret
      remoteRef:
        key: production/myapp/db  # Key in AWS Secrets Manager
        property: password         # JSON property within the secret
```

## REAL_WORLD
**GitOps-Safe Secret Management with Sealed Secrets**
You want your K8s configs in Git, but secrets can't be committed plainly. Sealed Secrets (by Bitnami) solve this: you encrypt secrets using a cluster-specific public key. The encrypted "SealedSecret" is safe to commit to Git. The controller in the cluster decrypts it using the private key.

```bash
# Install kubeseal CLI
brew install kubeseal

# Create a regular secret YAML, then seal it
kubectl create secret generic db-secret \
  --from-literal=password=S3cur3P@ss! \
  --dry-run=client -o yaml | \
  kubeseal --format yaml > db-sealedsecret.yaml

# Now commit db-sealedsecret.yaml to Git safely!
# The cluster controller decrypts it and creates the real Secret
```

## INTERVIEW
**Q: A security audit found that developers can read all Kubernetes Secrets in the `production` namespace using `kubectl get secret -n production`. They can see all production passwords. How do you fix this?**

A: Two layers of fixes:

1. **RBAC Restriction:** Implement the principle of least privilege:
```yaml
# Remove the ability to read secrets from developer role
kind: Role
metadata:
  namespace: production
  name: developer
rules:
  - apiGroups: [""]
    resources: ["pods", "deployments", "services", "configmaps"]
    verbs: ["get", "list", "watch"]
  # Intentionally NO secrets access!
```

2. **Use an External Secrets Manager:**
Move secrets out of Kubernetes-native Secrets into AWS Secrets Manager or HashiCorp Vault. Then:
- Developers have no access to Kubernetes at all in production (GitOps only)
- Actual credentials never exist in etcd — only references to external secrets
- Access to the actual secrets is controlled by AWS IAM/Vault policies with full audit logging

Additionally, enable **Kubernetes audit logging** to track all access to Secrets (who read what and when), and **Encryption at Rest** for the Kubernetes API server to encrypt secret data in etcd.
