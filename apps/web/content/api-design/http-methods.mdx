## WHY
HTTP (Hypertext Transfer Protocol) is the foundation of data communication on the World Wide Web. When a client (like a web browser or mobile app) talks to a server, it doesn't just send a URL; it sends a "Method" (also known as a Verb). 

The HTTP Method tells the server **what action the client wants to perform** on the resource identified by the URL. Using the correct method is critical for building predictable, secure, and caching-friendly APIs (like REST).

## THEORY
### The Core HTTP Methods (CRUD)
The most common HTTP methods map directly to standard database operations (Create, Read, Update, Delete):

1. **GET (Read):** Retrieve data. Should never modify state on the server.
2. **POST (Create):** Submit new data to the server to create a resource.
3. **PUT (Update/Replace):** Replace an entire existing resource with new data.
4. **PATCH (Update/Partial):** Partially update an existing resource.
5. **DELETE (Delete):** Remove a resource.

### Idempotency and Safety
Two critical concepts define how HTTP methods behave:
- **Safe Methods:** Methods that do not modify resources. (e.g., `GET`, `OPTIONS`, `HEAD`). You can call them safely without changing the database.
- **Idempotent Methods:** Methods that can be called multiple times with the SAME exact outcome as calling them once. (e.g., `GET`, `PUT`, `DELETE`). 

*Note: `POST` is neither safe nor idempotent. Calling `POST /users` 10 times will create 10 users!*

## CODE

### Level 1 — Beginner (GET and POST)
```javascript
// GET Request (Retrieving data)
// Notice: GET requests DO NOT have a body. Data is passed in the URL (query params).
fetch('https://api.devmastery.com/users?role=admin', {
  method: 'GET'
})
.then(response => response.json())
.then(data => console.log(data));

// POST Request (Creating data)
// Notice: POST requests include a 'body' containing the data to be created.
fetch('https://api.devmastery.com/users', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    name: 'Alice',
    role: 'admin'
  })
});
```

### Level 2 — Building Blocks (PUT vs PATCH)
Both update data, but they behave very differently.

```json
// Original Resource on the server:
// GET /users/1 
{ "id": 1, "name": "Alice", "age": 30, "role": "user" }

// PUT REQUEST: Completely replaces the resource!
// PUT /users/1
{ "name": "Alice Smith" }
// Resulting Resource:
{ "id": 1, "name": "Alice Smith", "age": null, "role": null } // You wiped out age & role!

// PATCH REQUEST: Partially updates the resource!
// PATCH /users/1
{ "name": "Alice Smith" }
// Resulting Resource:
{ "id": 1, "name": "Alice Smith", "age": 30, "role": "user" } // Age & role are preserved!
```

### Level 3 — Intermediate (DELETE and HEAD)
```javascript
// DELETE Request
// Deletes the resource. Note: Calling this twice on the same resource should 
// return a 200 OK the first time, and ideally a 404 Not Found the second time.
// (It's still considered idempotent because the end state is the same: the resource doesn't exist).
fetch('https://api.devmastery.com/users/1', {
  method: 'DELETE'
});

// HEAD Request
// Exactly like a GET request, but the server returns ONLY the headers, NOT the body!
// Useful for checking if a file exists, or how large it is, without downloading it.
fetch('https://api.devmastery.com/large-video.mp4', {
  method: 'HEAD'
}).then(response => {
  // Check Content-Length header to see file size before downloading!
  console.log(response.headers.get('Content-Length')); 
});
```

### Level 4 — Advanced (OPTIONS and CORS)
The `OPTIONS` method is rarely called manually by developers, but your browser calls it constantly behind the scenes. This is called a **CORS Preflight Request**.

```text
When a frontend on 'https://my-app.com' tries to POST to 'https://api.example.com',
the browser pauses the POST request and automatically sends an OPTIONS request first:

Browser: "OPTIONS /users" (Hey server, what methods are allowed from my-app.com?)

Server:  "200 OK"
         Access-Control-Allow-Origin: https://my-app.com
         Access-Control-Allow-Methods: GET, POST, OPTIONS

Browser: (Sees POST is allowed). Automatically sends the actual POST request.
```
If your frontend ever gets a "CORS Error", it's usually because your backend API is failing to respond correctly to the `OPTIONS` preflight request!

## REAL_WORLD
Web Browsers (HTML `<form>` tags) historically only support `GET` and `POST`. 
If you build a standard HTML form without JavaScript, you cannot send a `PUT` or `DELETE` request!

```html
<!-- This is impossible in pure HTML: -->
<form action="/users/1" method="DELETE"> ... </form>
```

**The Workaround:** Web frameworks (like Ruby on Rails, Laravel, or Spring) use "Method Spoofing". You send a `POST` request with a hidden field `_method=DELETE`. The backend framework sees this and pretends the request was an actual `DELETE` request.

## INTERVIEW
**Q: Is it safe to use a GET request to delete a user? (e.g., `GET /users/1/delete`)**
A: Absolutely not! This is a catastrophic security and architectural flaw. `GET` methods are considered "Safe". Web crawlers (like Googlebot) follow all `GET` links they find. If you use `GET` to delete, Googlebot will happily navigate through your site and delete all your users. Additionally, browsers aggressively pre-fetch and cache `GET` requests, which could trigger unintended deletions.

**Q: Can a GET request have a body?**
A: Technically, the HTTP specification does not explicitly forbid `GET` requests from having a body. However, it states that the payload has no defined semantics. In practice, almost all web servers (Nginx, Tomcat) and fetch clients (Axios, browser fetch) will either reject a `GET` request with a body or completely ignore the body. Never use a body in a `GET` request.
