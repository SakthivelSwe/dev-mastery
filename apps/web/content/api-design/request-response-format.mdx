## WHY
When a frontend application talks to a backend API, they are simply trading text over the internet. If they don't agree on exactly *how* that text is formatted, the system breaks down. Over the years, the industry evolved from clunky XML formats to JSON (JavaScript Object Notation), which is now the undisputed standard for modern REST APIs.

Understanding how to structure these payloads cleanly ensures your API is predictable, easy to consume, and robust against future changes.

## THEORY
### Content Negotiation
The client and server negotiate the format using HTTP Headers:
- **`Accept: application/json`** -> The client tells the server: "Please send the response as JSON."
- **`Content-Type: application/json`** -> The sender tells the receiver: "The body of this message is formatted as JSON."

### The Standard Envelope Pattern
Instead of returning raw arrays or raw strings, professional APIs wrap their responses in a standard JSON "envelope". This allows you to add metadata (like pagination, links, or success flags) alongside the actual data.

## CODE

### Level 1 — Beginner (Raw JSON vs Envelopes)
**❌ BAD: Returning a raw array**
If you return a raw array, you cannot easily add metadata (like `totalCount`) later without breaking the clients that are iterating directly over the array.
```json
[
  { "id": 1, "name": "Alice" },
  { "id": 2, "name": "Bob" }
]
```

**✅ GOOD: The Envelope Pattern**
```json
{
  "data": [
    { "id": 1, "name": "Alice" },
    { "id": 2, "name": "Bob" }
  ],
  "meta": {
    "totalCount": 150
  }
}
```

### Level 2 — Building Blocks (Standardizing Errors)
Error responses must be structurally consistent. If your API returns `{ "error": "Not found" }` on a 404, but `{ "message": "Invalid password", "code": 123 }` on a 400, frontend developers will hate you.

```json
// A standard Error Format (RFC 7807 Problem Details for HTTP APIs)
{
  "status": 400,
  "title": "Validation Error",
  "detail": "The provided payload violates business rules.",
  // Provide an array if there are multiple specific issues!
  "errors": [
    { "field": "email", "message": "Email format is invalid." },
    { "field": "age", "message": "Age must be over 18." }
  ]
}
```

### Level 3 — Intermediate (HATEOAS / Hypermedia)
HATEOAS (Hypermedia As The Engine Of Application State) is the final level of REST maturity. The API response includes hypermedia `links` that tell the client what actions are currently available based on the resource's state.

```json
{
  "data": {
    "id": "ord_123",
    "status": "pending_payment",
    "total": 50.00
  },
  // The server tells the client exactly what they can do next!
  "links": {
    "self": "/api/orders/ord_123",
    "cancel": "/api/orders/ord_123/cancel",
    "pay": "/api/orders/ord_123/pay"
  }
}
```
If the order status changes to `shipped`, the `cancel` and `pay` links will disappear from the response. The frontend doesn't need to hardcode business logic; it just renders buttons if the links exist!

### Level 4 — Advanced (Handling Dates and Timestamps)
**Never use formatted strings for dates in APIs.** Do not use `"02-05-2023"` (Is that Feb 5th or May 2nd?). Do not use `"2023/10/23 4:00 PM"` (What time zone is that?).

Always use **ISO 8601 UTC timestamps**. The `Z` at the end stands for Zulu time (UTC).
```json
{
  "createdAt": "2023-10-23T16:00:00.000Z"
}
```
The backend stores everything in UTC. The frontend receives UTC, and uses JavaScript's `Intl.DateTimeFormat` (or libraries like `date-fns`) to automatically convert it to the user's local timezone on their screen.

## REAL_WORLD
**CamelCase vs snake_case**
- JavaScript/TypeScript uses `camelCase`.
- Python/Ruby/Databases generally use `snake_case`.
What should the JSON API use?
JSON was invented for JavaScript, so the overwhelming industry standard for JSON APIs is `camelCase` (e.g., `firstName`, not `first_name`). Most modern backend frameworks (like Spring Boot, Express, or Django) will automatically convert the backend's internal naming convention to `camelCase` right before the JSON leaves the server.

## INTERVIEW
**Q: Why shouldn't we wrap *everything* in an envelope if HTTP already has a body?**
A: Some developers argue that the HTTP Status Code `200` already means success, so adding `{"success": true}` inside the JSON is redundant. They prefer returning just the raw object `{"id": 1, "name": "Alice"}`. However, the `data` envelope pattern (specifically popularized by the JSON:API specification) is generally preferred because it provides a dedicated namespace (`meta`, `links`, `included`) that prevents metadata fields from accidentally colliding with the actual properties of your database entity.
