## WHY
A well-designed URL is intuitive, predictable, and self-documenting. If developers can guess the URL of an endpoint without reading your documentation, you have succeeded. Conversely, chaotic URL design (`/getProducts`, `/products/delete/1`, `/api/v2/new_user`) leads to massive confusion, integration bugs, and endless support tickets.

In REST, URLs represent **Resources** (Nouns), not Actions (Verbs).

## THEORY
### Core URL Design Rules
1. **Use Plural Nouns:** Always use plural nouns for resources, even when referring to a single item. (`/users` not `/user`).
2. **Keep it Lowercase:** Use lowercase letters to avoid cross-platform case-sensitivity issues.
3. **Use Kebab-Case:** If a resource name has two words, separate them with a hyphen (`/blog-posts` not `/blogPosts` or `/blog_posts`).
4. **Never Use Verbs:** The HTTP Method (GET, POST, DELETE) provides the verb. The URL should only contain nouns.
5. **Nest for Relationships:** Use the URL path to represent parent-child relationships, but don't nest too deep.

## CODE

### Level 1 — Beginner (The Golden Rule of Nouns)
Let's design endpoints for a `Product` resource.

```text
❌ BAD DESIGN (RPC Style - using verbs):
GET  /getAllProducts
GET  /getProduct?id=123
POST /createProduct
POST /updateProduct?id=123
POST /deleteProduct?id=123

✅ GOOD DESIGN (RESTful - using nouns):
GET    /products          (Get a list of all products)
GET    /products/123      (Get the specific product with ID 123)
POST   /products          (Create a new product)
PUT    /products/123      (Update the product with ID 123)
DELETE /products/123      (Delete the product with ID 123)

Notice how the URL `/products/123` is exactly the same for GET, PUT, and DELETE. 
The HTTP Method is what changes the behavior!
```

### Level 2 — Building Blocks (Sub-Resources and Relationships)
How do you represent relationships? For example, an Order belongs to a User.

```text
To get all orders for user 123:
✅ GET /users/123/orders

To get a specific order (ID 456) for user 123:
✅ GET /users/123/orders/456

To create a new order for user 123:
✅ POST /users/123/orders

RULE OF THUMB: Don't nest more than one level deep!
❌ BAD: GET /users/123/orders/456/items/789
This is too deeply nested and brittle. What if an item can exist without an order?

✅ BETTER: Flatten the URL once you have the unique ID.
GET /users/123/orders
GET /orders/456           (Access the order directly by its ID)
GET /orders/456/items     (Get the items for that order)
```

### Level 3 — Intermediate (Filtering, Sorting, and Pagination)
Do not put filters or sorting instructions in the URL path. Use **Query Parameters** (the part of the URL after the `?`).

```text
Filtering:
✅ GET /users?role=admin
✅ GET /products?category=electronics&status=active

Sorting:
✅ GET /users?sort=created_at&order=desc

Pagination:
✅ GET /users?page=2&limit=50
✅ GET /users?offset=100&limit=50

Search:
✅ GET /users?search=john

Combined:
✅ GET /products?category=shoes&sort=price&order=asc&page=1
```

### Level 4 — Advanced (Handling Non-CRUD Actions)
What happens when you need to do something that doesn't map perfectly to CRUD? For example, "Publishing" an article, "Banning" a user, or "Translating" a document.

```text
APPROACH 1: Treat the action as a sub-resource (Best Practice)
POST /articles/123/publish
POST /users/456/ban
(The verb becomes a noun representing the "event" of publishing or banning).

APPROACH 2: Patch the state
PATCH /articles/123
Body: { "status": "published" }

APPROACH 3: Action Query Parameter (Acceptable, but less RESTful)
POST /articles/123?action=publish

What about complex multi-resource operations? (e.g., Search across everything)
It's perfectly fine to break REST rules when pragmatism demands it.
✅ GET /search?q=apple
(Search isn't technically a "resource", but trying to force it to be one is silly).
```

## REAL_WORLD
**API Versioning in the URL**
Almost all modern APIs include a version number directly in the URL path. Why? Because once an API is public, you can NEVER break the contract. If you need to introduce a breaking change (like changing a field name from `userId` to `user_identifier`), you release a new version.

```text
https://api.stripe.com/v1/customers
https://api.github.com/v3/users
```
There is debate about putting versions in the URL vs putting them in an HTTP Header (`Accept: application/vnd.github.v3+json`). The industry consensus has heavily favored URLs because they are easier to test in a browser, easier to copy/paste in Slack, and easier to route in API Gateways.

## INTERVIEW
**Q: When should I use plural nouns vs singular nouns?**
A: **Always use plural.** It keeps the API consistent. `/users` represents the collection. `/users/1` represents the item with ID 1 *within* the users collection. Using singular (`/user`) breaks this mental model. The only exception is a true singleton resource, like `/me` or `/profile`, where there can only ever be exactly one per authenticated user.

**Q: Is `GET /users/search?q=bob` good URL design?**
A: It's a common pattern, but technically flawed in strict REST. `/users/search` implies there is a user with the ID "search". A better approach is simply `GET /users?q=bob`. If the search is highly complex or spans multiple resource types, a dedicated top-level `GET /search?type=user&q=bob` is acceptable.
