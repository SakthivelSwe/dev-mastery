## WHY
If you build a mobile app, publish it to the App Store, and wait a year, 10% of your users will *never* have updated their app. If your backend API suddenly renames the `userId` field to `customer_id`, your old mobile app will instantly crash for those 10% of users. 

**APIs are contracts.** Once an API is public, you cannot make breaking changes to it without destroying the clients that rely on it. Versioning allows you to evolve your backend architecture and introduce new features while maintaining backwards compatibility for older clients.

## THEORY
### What is a Breaking Change?
- Renaming a field or an endpoint.
- Deleting a field or an endpoint.
- Changing a field's data type (e.g., from an integer to a string).
- Adding a *required* parameter to an existing request.

### What is a Non-Breaking Change?
- Adding a new endpoint.
- Adding a new *optional* parameter to a request.
- Adding a new field to a response (well-written clients ignore unknown fields).

## CODE

### Strategy 1: URL Path Versioning (The Standard)
This is the most common, intuitive, and widely adopted approach (used by Stripe, Twitter, Twilio). The version is explicitly baked into the URL path.

**Pros:** Extremely easy to route in your load balancer. Easy for developers to see what version they are using just by looking at the URL.
**Cons:** Breaks the pure REST principle that a URL should represent a *resource*, not a *version* of a resource.

```text
// Request
GET /api/v1/users/123 HTTP/1.1
Host: api.devmastery.com

// When you release a breaking change:
GET /api/v2/users/123 HTTP/1.1
```

### Strategy 2: Query Parameter Versioning
The version is passed as a query parameter. Used heavily by AWS.

**Pros:** Keeps the base URL clean. Easy to default to `v1` if the parameter is missing.
**Cons:** Query parameters are usually meant for filtering/sorting, not routing logic.

```text
// Request
GET /api/users/123?version=2 HTTP/1.1
Host: api.devmastery.com
```

### Strategy 3: Header Versioning (Custom Header)
The URL remains completely clean and pure. The version is passed in a custom HTTP header.

**Pros:** Pure REST. The URL strictly defines the resource.
**Cons:** Terrible developer experience. You cannot simply share a clickable link to an endpoint with a colleague, because the browser doesn't send custom headers when you paste a link into the address bar.

```text
// Request
GET /api/users/123 HTTP/1.1
Host: api.devmastery.com
X-API-Version: 2
```

### Strategy 4: Accept Header Versioning (Content Negotiation)
Used by GitHub. This is the most strictly "RESTful" approach. The client negotiates the exact structure of the payload using the standard HTTP `Accept` header.

**Pros:** Extremely powerful and semantically correct.
**Cons:** Very complex to implement on the backend and complex for junior frontend developers to consume.

```text
// Request
GET /api/users/123 HTTP/1.1
Host: api.devmastery.com
Accept: application/vnd.github.v3+json
```

## REAL_WORLD
**Stripe's Date-Based Versioning**
Stripe is famous for having the best API engineering in the world. Instead of using `v1` or `v2`, they use the exact date the API was released:
`Stripe-Version: 2023-10-31`.

Internally, Stripe doesn't maintain 50 different copies of their codebase. They maintain only the newest version of the API. When an old client sends a request requesting the `2019-05-01` version, the request goes through a massive chain of **Migration Middleware**. The modern response payload is literally passed backward through time, with 4 years worth of middleware scripts automatically renaming fields and stripping data until the payload looks exactly like it did in 2019, before it is finally sent back to the client!

## INTERVIEW
**Q: My company is migrating our database from MySQL to MongoDB. Does this require bumping our API from `v1` to `v2`?**
A: **No.** The client application does not care what database you use. If the JSON payload structure and the URL endpoints remain identical, there is no breaking change to the contract. Versioning is strictly for changes to the *interface*, not the *implementation*.
