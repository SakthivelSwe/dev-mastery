## WHY
When a client sends a request to a server, the server must communicate the result of that request. Did it succeed? Did the client send bad data? Is the server broken?

**HTTP Status Codes** are standardized 3-digit numbers that universally communicate the state of the response. If your API returns `{ "error": "Not Found" }` but sets the HTTP status code to `200 OK`, you are breaking the internet protocol. CDNs won't cache it correctly, load balancers will misread it, and client HTTP libraries won't throw exceptions when they should.

## THEORY
### The 5 Classes of Status Codes
Status codes are grouped into five blocks based on their first digit:

- **1xx (Informational):** Request received, continuing process (Rarely seen by typical devs).
- **2xx (Success):** The action was successfully received, understood, and accepted.
- **3xx (Redirection):** Further action needs to be taken by the client to complete the request.
- **4xx (Client Error):** The client messed up (bad syntax, unauthorized, not found).
- **5xx (Server Error):** The server messed up (crashed, timeout, unavailable).

## CODE

### Level 1 — Beginner (The Big Three)
If you only know three status codes, know these:

```text
200 OK
The request succeeded. For a GET request, the requested data is in the response body.

400 Bad Request
The server cannot process the request due to client error (e.g., malformed JSON, 
missing required fields).

500 Internal Server Error
A generic error message indicating the server encountered an unexpected condition 
(e.g., a NullPointerException in your Java code!).
```

### Level 2 — Building Blocks (2xx Success & 3xx Redirects)
```text
201 Created
Used primarily after a successful POST request. Indicates a new resource was created.
Best practice: Include a 'Location' header pointing to the new resource's URL.

204 No Content
The server successfully processed the request, but is not returning any content. 
Perfect for successful DELETE or PUT requests where the client already has the data.

301 Moved Permanently
This and all future requests should be directed to the given URI. 
Browsers aggressively cache 301s!

302 Found (Temporary Redirect)
The resource resides temporarily under a different URI. 
Browsers will follow it, but won't cache it forever.
```

### Level 3 — Intermediate (4xx Client Errors - Authentication)
```text
401 Unauthorized
Actually means "Unauthenticated". The client must log in / provide a valid API token 
to access this resource.

403 Forbidden
Actually means "Unauthorized". The client IS logged in (we know who they are), 
but they do not have permission to perform this action (e.g., a standard user 
trying to access the admin panel).

404 Not Found
The requested resource could not be found. (Also frequently used to hide the 
existence of a resource from unauthorized users instead of returning 403).

422 Unprocessable Entity
The request was well-formed JSON (not a 400), but contained semantic errors 
(e.g., 'email' was required but blank, or 'age' was a negative number).
```

### Level 4 — Advanced (429 and 5xx Errors)
```text
429 Too Many Requests
The user has sent too many requests in a given amount of time (Rate Limiting).
Best practice: Always include a 'Retry-After' header indicating how many seconds 
to wait before trying again.

502 Bad Gateway
The server, while acting as a gateway or proxy (like Nginx), received an 
invalid response from the upstream server (like your Node.js app). 
Usually means your backend app crashed or isn't running!

503 Service Unavailable
The server is currently unable to handle the request due to temporary 
overloading or maintenance. 

504 Gateway Timeout
The proxy (Nginx/API Gateway) did not receive a timely response from the upstream server. 
Usually means your database query is taking too long!
```

## REAL_WORLD
**The "GraphQL Trap"**
In standard REST, HTTP status codes are paramount. However, standard GraphQL APIs return `200 OK` for *everything* — even if the query fails completely! Instead of HTTP codes, GraphQL includes an `errors` array in the JSON body. This makes monitoring GraphQL APIs with traditional network tools (like Datadog or standard API gateways) very difficult, as all traffic looks like successful `200 OK` traffic at the HTTP layer.

## INTERVIEW
**Q: My API returns a `200 OK` with `{ error: "User not found" }`. Why is this a bad design?**
A: HTTP Status Codes operate at Layer 7 of the OSI model. Content Delivery Networks (CDNs), Load Balancers, and Proxies look at the *status code* to determine what to do. If you return a `200 OK` for a missing user, a CDN might permanently cache that error response, serving it to all future users. It also breaks standard HTTP client libraries (like Axios) which rely on 4xx/5xx codes to automatically throw JavaScript exceptions.

**Q: I'm building an API for a bank. If a user tries to access another user's bank account via `GET /accounts/999`, should I return 403 Forbidden or 404 Not Found?**
A: For highly secure applications, it is often best practice to return **404 Not Found** instead of 403. Returning 403 confirms to the attacker that account #999 *exists*, giving them valuable information to map out your database structure (an Enumeration Attack). Returning 404 hides the existence of the resource entirely from unauthorized parties.
