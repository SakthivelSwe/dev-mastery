## WHY
If you build an API that allows users to fetch their private banking details or transfer money, you must ensure that the person making the request is actually who they claim to be. **Authentication (AuthN)** answers a single question: *"Who are you?"* 

Understanding the evolution of API Authentication—from stateful Session Cookies to stateless JWTs and OAuth2—is the most critical security foundation for any backend developer.

## THEORY
### Authentication vs Authorization
These two terms are constantly confused, but they are entirely different concepts.
- **Authentication (AuthN):** Proving *who* you are (e.g., logging in with a username and password).
- **Authorization (AuthZ):** Proving *what you are allowed to do* (e.g., an admin can delete users, but a standard user cannot). 
*You must authenticate before you can authorize.*

### The Stateless HTTP Problem
HTTP is a stateless protocol. When a user logs in successfully, the server forgets who they are the very next millisecond. Every subsequent request must carry a piece of proof (a Token) that says, "I am Alice, I just logged in 5 minutes ago."

## CODE

### Level 1 — Beginner (Basic Auth)
The oldest and simplest form of API authentication. The client sends their raw username and password with *every single HTTP request*.

```text
// The client Base64 encodes "alice:mypassword123"
GET /api/bank/balance HTTP/1.1
Host: api.devmastery.com
Authorization: Basic YWxpY2U6bXlwYXNzd29yZDEyMw==
```
**Pros:** Incredibly easy to implement.
**Cons:** Very dangerous. You are transmitting raw passwords on every request. If the connection is not strictly HTTPS, anyone on the Wi-Fi network can steal the password. Never use this for modern web apps.

### Level 2 — Building Blocks (Session Cookies)
The standard for traditional web applications (like PHP or Ruby on Rails).

1. Alice sends her username and password to `/login`.
2. The server verifies the password, generates a random `Session ID` (e.g., `abc-123`), and saves it in a Database table.
3. The server sends the Session ID back to Alice in a `Set-Cookie` header.
4. Alice's browser automatically sends that Cookie with every future request.

**Pros:** Very secure. You can easily revoke Alice's access by deleting her Session ID from the database.
**Cons:** **Stateful.** Every time a request comes in, the server must query the database to verify the Session ID is still valid. If you have 10 million users, this database lookup becomes a massive performance bottleneck.

### Level 3 — Intermediate (JSON Web Tokens - JWT)
The standard for modern REST APIs and Microservices. 

A JWT (pronounced "jot") is a **Stateless Token**. 
1. Alice logs in. 
2. The server creates a JSON object: `{"userId": 5, "role": "admin"}`. 
3. The server mathematically signs this JSON using a highly secure, secret server key, creating a JWT.
4. Alice sends the JWT in the `Authorization: Bearer <token>` header on future requests.

When the server receives the JWT, it does **NOT** query the database. It simply runs a math equation to verify the cryptographic signature. If the signature is valid, the server trusts the data inside the token!

**Pros:** Blazingly fast. Zero database lookups required. Perfect for microservices.
**Cons:** **Impossible to revoke.** Because the server doesn't check the database, if a hacker steals Alice's JWT, the server has no way of knowing it was stolen. The hacker has full access until the token naturally expires. (This is why JWTs must have very short lifespans, typically 15 minutes).

### Level 4 — Advanced (OAuth 2.0 & OpenID Connect)
What if you want to let users log in to your app using a "Sign in with Google" button? You don't want to ask for their Google password!

**OAuth 2.0** is an industry-standard delegation protocol. 
1. Your app redirects Alice to Google.com.
2. Alice logs into Google directly.
3. Google asks Alice: "DevMastery wants to read your email address. Allow?"
4. Alice clicks Allow.
5. Google redirects Alice back to your app, handing your backend an **Access Token**.
6. Your backend uses this token to fetch Alice's data from Google's servers.

*Note: OAuth 2.0 is technically for Authorization (granting access). OpenID Connect (OIDC) is a layer built on top of OAuth 2.0 specifically designed for Authentication.*

## REAL_WORLD
**Refresh Tokens**
If a JWT expires every 15 minutes, does the user have to type their password 4 times an hour? No!
When Alice logs in, the server actually sends back TWO tokens:
1. **Access Token (JWT):** Expires in 15 minutes. Used for API requests.
2. **Refresh Token (Random String):** Expires in 30 days. Saved securely in an HttpOnly cookie.

When the Access Token expires, the frontend silently sends the Refresh Token to a special `/refresh` endpoint. The server checks the database to ensure the Refresh Token hasn't been revoked, and if it's valid, issues a brand new 15-minute Access Token! This balances the extreme performance of JWTs with the security and revokability of Sessions.

## INTERVIEW
**Q: A hacker steals a JWT from a user's browser via an XSS (Cross-Site Scripting) attack. They change the payload from `{"userId": 5, "role": "user"}` to `{"userId": 5, "role": "admin"}` and send it to your server. Does your server get hacked?**
A: **No.** The JWT payload is just Base64 encoded; anyone can decode and read it. However, the third part of a JWT is the Cryptographic Signature. When the hacker alters the payload, the existing signature becomes mathematically invalid. When your server receives the forged token, it will run the hash algorithm using its Secret Key, realize the signature doesn't match the payload, and instantly reject the request with a `401 Unauthorized`. You cannot forge or alter a JWT without knowing the server's private secret key.
