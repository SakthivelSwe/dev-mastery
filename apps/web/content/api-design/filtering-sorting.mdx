## WHY
When your database has 10,000 products, you cannot send the entire list to the frontend and let the browser sort it. The payload would be massively slow, and mobile devices would crash due to memory exhaustion. 

A professional REST API must provide mechanisms for the client to specifically filter, sort, and search through large collections of data directly on the server before the data is transmitted over the network.

## THEORY
### The Query String
In REST API design, the URL Path (e.g., `/api/products`) identifies the *resource collection*. The URL Query String (everything after the `?`) is used to *manipulate* that collection through filtering, sorting, and pagination.

## CODE

### Level 1 — Beginner (Exact Match Filtering)
The simplest form of filtering is checking for an exact match. The query parameter key should directly map to the resource's property name.

```text
// 1. Filter by a single property
GET /api/users?role=admin
GET /api/products?category=electronics

// 2. Filter by multiple properties (Implicit AND)
// "Find products that are electronics AND in stock"
GET /api/products?category=electronics&inStock=true

// 3. Filter by multiple values (Implicit OR within the same property)
// "Find products in the electronics OR clothing category"
GET /api/products?category=electronics,clothing
// (Alternatively, many APIs use repeated keys: ?category=electronics&category=clothing)
```

### Level 2 — Building Blocks (Advanced Operators)
Exact matching isn't always enough. How do you filter for products with a price *greater than* $50? 

**Approach A: Suffixes (The Stripe Approach)**
You append an operator suffix to the query parameter key.
```text
GET /api/products?price.gt=50      (Greater than 50)
GET /api/products?price.lte=100    (Less than or equal to 100)
GET /api/products?name.like=iphone (Fuzzy string search)
```

**Approach B: Bracket Notation (The JSON:API / MongoDB Approach)**
You pass the operator as a nested key within brackets.
```text
GET /api/products?price[gt]=50
GET /api/products?price[lte]=100
```
*Note: Both approaches are completely valid. The most important thing is picking one and being 100% consistent across your entire API.*

### Level 3 — Intermediate (Sorting)
Sorting requires two pieces of information: the field to sort by, and the direction (Ascending or Descending).

**Single Field Sorting:**
```text
// Often defaults to Ascending
GET /api/products?sort=price

// Descending is usually denoted by a minus sign (-) or a 'desc' suffix
GET /api/products?sort=-price
GET /api/products?sort=price:desc
```

**Multi-Field Sorting:**
What if you want to sort by Category first, and then sort by Price within each category?
```text
GET /api/products?sort=category,-price
```

### Level 4 — Advanced (Sparse Fieldsets)
Even if you filter the list down to 10 products, each product might have 50 properties (including a massive `description` string). If the frontend only wants to render a tiny dropdown list showing the product's `id` and `name`, fetching all 50 properties wastes massive amounts of bandwidth.

**Sparse Fieldsets** allow the client to specify exactly which properties the API should include in the response payload.

```text
// "Send me 10 products, but ONLY include the id and name fields."
GET /api/products?fields=id,name
```
This requires complex dynamic SQL on the backend, but provides GraphQL-like efficiency in a standard REST API!

## REAL_WORLD
**Global Search vs Field Filtering**
If an API consumer wants to find a user named "Alice", they might use `GET /api/users?name=Alice`. 
But what if the user doesn't know if "Alice" is the user's first name, last name, or email address? They just have a search box on the UI.

In this scenario, it is a best practice to expose a generic `q` (query) parameter for global text search across multiple fields:
```text
// The backend will dynamically search firstName, lastName, and email for "Alice"
GET /api/users?q=Alice
```

## INTERVIEW
**Q: How do you prevent SQL Injection when implementing dynamic filtering and sorting?**
A: You must never concatenate the user's query parameters directly into your SQL strings. 
1. For filtering values (e.g., `?name=Alice`), always use Parameterized Queries (Prepared Statements) so the database treats "Alice" strictly as a string, not executable code.
2. For sorting fields (e.g., `?sort=price`), you cannot use Prepared Statements for column names. Instead, you must use a **Whitelist approach**. The backend must maintain a strict array of allowed sortable columns (e.g., `['price', 'created_at', 'name']`). If the user passes `?sort=DROP TABLE users`, it is rejected by the whitelist before it ever touches the SQL string.
