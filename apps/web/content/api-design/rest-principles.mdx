## WHY
Before REST, web services used protocols like SOAP and XML-RPC, which were heavy, complex, and treated the web merely as a transport layer for remote procedure calls (RPC). 

**REST (Representational State Transfer)**, introduced by Roy Fielding in 2000, changed everything. It treats the web as a massive, distributed state machine where data entities are "Resources" (like Nouns) and HTTP methods are the actions (like Verbs). REST is the architectural style that powers 90% of the public APIs on the internet today.

## THEORY
### The 6 Guiding Principles of REST
For an API to be considered truly "RESTful", it must adhere to these six constraints:

1. **Client-Server Architecture:** The client (frontend) and server (backend) are completely separated. They can evolve independently as long as the interface (API) doesn't break.
2. **Statelessness:** Every request from the client must contain ALL the information needed by the server to process it. The server stores NO session state. (This makes scaling horizontally trivial).
3. **Cacheability:** Responses must implicitly or explicitly define themselves as cacheable or not to prevent clients from reusing stale data.
4. **Layered System:** A client cannot usually tell whether it is connected directly to the end server or an intermediary (Load Balancer, API Gateway). 
5. **Code on Demand (Optional):** Servers can temporarily extend the functionality of a client by transferring executable code (e.g., sending JavaScript to a browser).
6. **Uniform Interface:** The most important constraint! It decouples the architecture. (See below).

## CODE

### Level 1 — Beginner (Resources and URIs)
In REST, everything is a **Resource**. Resources are identified by URIs (Uniform Resource Identifiers).
URIs should be nouns (plural), never verbs.

```text
✅ GOOD (RESTful):
GET  /users              (Get all users)
GET  /users/123          (Get user 123)
POST /users              (Create a new user)
GET  /users/123/orders   (Get orders for user 123)

❌ BAD (RPC Style):
GET  /getAllUsers
POST /createUser
POST /updateUser?id=123
POST /deleteOrder?id=456
```

### Level 2 — Building Blocks (HTTP Methods as Verbs)
REST uses standard HTTP methods to perform CRUD operations on resources:

```text
| HTTP Method | CRUD Operation | Idempotent? | Description                               |
|-------------|----------------|-------------|-------------------------------------------|
| POST        | Create         | No          | Creates a new resource.                   |
| GET         | Read           | Yes         | Retrieves a resource. Side-effect free.   |
| PUT         | Update         | Yes         | Replaces the ENTIRE resource.             |
| PATCH       | Update         | No          | Partially updates a resource.             |
| DELETE      | Delete         | Yes         | Deletes a resource.                       |

What is "Idempotent"?
If you make the exact same request 100 times, the state of the server is the 
same as if you made the request 1 time.
- GET is idempotent.
- DELETE is idempotent (deleting it 100 times still results in it being deleted).
- PUT is idempotent (overwriting it with the same data 100 times has no side effects).
- POST is NOT idempotent (calling POST /users 100 times creates 100 users!).
```

### Level 3 — Intermediate (HATEOAS - The Uniform Interface)
**HATEOAS** stands for *Hypermedia As The Engine Of Application State*. 
It is the most advanced (and most often ignored) principle of REST.

Instead of the client hard-coding URLs for future actions, the server's response includes the URLs for all possible next actions the client can take. 

```json
// GET /accounts/12345
{
  "account_id": "12345",
  "balance": 100.00,
  "status": "active",
  
  // HATEOAS Links (The engine of application state!)
  // The client doesn't need to know HOW to construct a deposit URL.
  // It just looks for the 'deposit' link and follows it.
  "links": {
    "self":      { "href": "/accounts/12345" },
    "deposit":   { "href": "/accounts/12345/deposit" },
    "withdraw":  { "href": "/accounts/12345/withdraw" },
    "transfer":  { "href": "/accounts/12345/transfer" },
    "close":     { "href": "/accounts/12345/close" }
  }
}
```
If the account balance goes below 0, the server simply omits the "withdraw" link in the response. The client dynamically disables the withdraw button. The business logic lives entirely on the server!

### Level 4 — Advanced (Content Negotiation)
REST is not just JSON. A resource is an abstract concept; its "Representation" is what gets sent over the wire.

A client can request different representations using the `Accept` header (Content Negotiation):

```text
Request 1 (from a Web Browser):
GET /users/123
Accept: text/html
--> Server returns a rendered HTML profile page.

Request 2 (from a Mobile App):
GET /users/123
Accept: application/json
--> Server returns raw JSON data.

Request 3 (from a Legacy Enterprise System):
GET /users/123
Accept: application/xml
--> Server returns XML data.
```
This is the true power of REST — the same URI serves multiple clients based on their capabilities!

## REAL_WORLD
Most APIs today are "Pragmatic REST" (or REST-ish), not strict REST.
Stripe's API is famously considered one of the best designed APIs in the world. It perfectly uses nouns for URLs (`/v1/customers`, `/v1/charges`), standard HTTP verbs, and standard HTTP status codes. However, it completely ignores HATEOAS, because for developer-focused APIs, explicit documentation and SDKs are easier to consume than dynamic hypermedia links.

## INTERVIEW
**Q: What is the difference between PUT and PATCH?**
A: `PUT` is a complete replacement. If the user has `{name, email, age}` and you send `PUT /users/1` with just `{name}`, the server should delete `email` and `age` (or set them to null). `PUT` is idempotent. `PATCH` is a partial update. Sending `{name}` via `PATCH /users/1` will ONLY update the name and leave `email` and `age` intact. 

**Q: How do you design an API endpoint for an action that doesn't map to CRUD? (e.g., "Ban a User")**
A: You have two valid RESTful options:
1. Treat the state as a resource property: `PATCH /users/123` with body `{"status": "banned"}`.
2. Treat the *action* as a sub-resource: `POST /users/123/bans`. (This is often better, as you can include metadata like "reason" and "duration" in the ban resource).

// Robust error handling for production use
function safeProcess(input) {
