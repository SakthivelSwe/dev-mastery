## WHY
Functions are the core building blocks of JavaScript. In vanilla JS, you can call a function with the wrong number of arguments, or pass a string into a function that expects a number. The function will execute, fail silently, and return `NaN` or `undefined`.
TypeScript functions solve this by defining a strict "Contract". You explicitly define what types go IN (Parameters), and what type comes OUT (Return Type). If you violate the contract, the code won't even compile.

## THEORY
### The Function Signature
A Function Signature in TypeScript consists of two parts:
1. **Parameter Types:** Annotating the arguments inside the parenthesis `(name: string)`.
2. **Return Type:** Annotating the result after the parenthesis `(): number`.

If a function does not return anything (it just executes a side effect, like `console.log`), its return type is `void`.

## CODE

### Level 1 — Basic Function Typing
You must annotate the parameters. If you don't, TypeScript implicitly types them as `any`, which defeats the purpose of TypeScript!

```typescript
// Parameter 'price' is a number. 
// The ': string' after the parenthesis dictates what the function returns.
function formatPrice(price: number): string {
    return `$${price.toFixed(2)}`;
}

// ERROR: Expected 1 arguments, but got 0.
formatPrice(); 

// ERROR: Argument of type 'string' is not assignable to parameter of type 'number'.
formatPrice("50"); 

let result = formatPrice(50); // Valid. 'result' is strongly typed as a string!
```

### Level 2 — Optional and Default Parameters
In JavaScript, all parameters are technically optional. In TypeScript, parameters are strictly required unless you explicitly mark them as optional using the `?` operator.

```typescript
// 'greeting' is optional. If not provided, it will be undefined.
function sayHello(name: string, greeting?: string): string {
    if (greeting) {
        return `${greeting}, ${name}!`;
    }
    return `Hello, ${name}!`;
}
sayHello("Alice"); // Valid
sayHello("Alice", "Good morning"); // Valid

// Default Parameters (Type Inference happens automatically!)
// Because we assign "Hello", TypeScript infers that 'greeting' is a string.
function sayHelloDefault(name: string, greeting = "Hello"): string {
    return `${greeting}, ${name}!`;
}
```

### Level 3 — The `void` and `never` Types
What if a function doesn't return anything?

```typescript
// 'void' means the function returns nothing (technically it returns undefined in JS).
function logMessage(message: string): void {
    console.log(message);
    // return "Done"; // ERROR: Type 'string' is not assignable to type 'void'.
}
```

What if a function *never finishes executing*? (e.g., it throws an error and crashes the app, or it has an infinite while loop).
```typescript
// 'never' means this function physically cannot reach the end of its execution block.
function crashApp(errorMessage: string): never {
    throw new Error(errorMessage);
}
```

### Level 4 — Typing Function Variables (Callbacks)
When you pass a function *as an argument* to another function (a callback), you must define the signature of the function you are expecting.
You use the Arrow Function syntax to define a function type: `(arg: Type) => ReturnType`.

```typescript
// The second argument 'onComplete' MUST be a function that takes a string and returns nothing.
function fetchData(url: string, onComplete: (data: string) => void) {
    console.log("Fetching " + url);
    // Simulate network request
    onComplete("Here is your data!");
}

fetchData("/api/users", (data) => {
    // TypeScript knows 'data' is a string automatically!
    console.log(data.toUpperCase()); 
});
```

## REAL_WORLD
**Function Overloads**
In JavaScript, it is common to write a single function that does completely different things depending on what arguments you pass it. 
For example, a `getDate` function:
- If you pass a Timestamp (number), it returns a Date object.
- If you pass a Date object, it returns a Timestamp (number).

How do you type a single function with multiple different signatures? You use **Function Overloads**.

```typescript
// Overload Signature 1: Pass a number, get a Date
function transformDate(timestamp: number): Date;
// Overload Signature 2: Pass a Date, get a number
function transformDate(date: Date): number;

// The Actual Implementation (Uses 'any' or Union types internally)
function transformDate(value: any): any {
    if (typeof value === "number") {
        return new Date(value);
    } else {
        return value.getTime();
    }
}

// TypeScript enforces the Overloads perfectly!
let d = transformDate(1700000000); // d is typed as Date
let n = transformDate(new Date()); // n is typed as number
// transformDate("string"); // ERROR: No overload matches this call.
```

## INTERVIEW
**Q: What is the difference between typing a function's return type as `void` vs `undefined`?**
A: **Intent and Callback Compatibility.**
If you type a function as returning `undefined`, you are strictly enforcing that the function *must* physically execute `return undefined;` (or have no return statement at all).
If you type a return as `void`, it means "I don't care what this function returns, the caller should ignore it."
This is crucial for Callbacks. 
If an API expects a callback `(data: string) => void`, you are allowed to pass an arrow function that returns a value (e.g., `(data) => data.toUpperCase()`). TypeScript allows this because `void` tells the API to simply ignore the returned value. If the API expected `(data: string) => undefined`, passing that same arrow function would trigger a compiler error because it returns a string, not undefined!
