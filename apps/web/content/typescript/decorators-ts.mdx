## WHY
Imagine you have an API controller with 50 methods. You want to log the execution time of every single method. 
Without decorators, you have to manually wrap the code inside all 50 methods with `console.time()` and `console.timeEnd()`. This heavily clutters your business logic.
**Decorators** allow you to "wrap" or modify a class, method, or property dynamically at runtime, using a clean `@DecoratorName` syntax above the code. They are heavily used in enterprise frameworks like Angular and NestJS to inject metadata and handle cross-cutting concerns (like Logging, Authorization, and Validation) without polluting the core logic.

## THEORY
### Stage 3 vs. Legacy Decorators
TypeScript has had "Experimental Decorators" since 2015. Frameworks like Angular rely on this older standard.
However, in TypeScript 5.0+, the official **ECMAScript Stage 3 Decorators** were finally implemented natively. 
A Decorator is literally just a function. It receives information about the thing it is attached to (the target), and it can replace that thing with a new version of itself!

## CODE

### Level 1 — The Method Decorator
Let's write a standard TS 5.0 Method Decorator to log when a method is called.
A method decorator function takes two arguments: the original method, and a `context` object containing metadata.

```typescript
// 1. Define the Decorator Function
function LogExecution(originalMethod: any, context: ClassMethodDecoratorContext) {
    // We return a NEW function that wraps the original one!
    return function (this: any, ...args: any[]) {
        console.log(`[LOG]: Calling method '${context.name as string}' with args:`, args);
        
        // Execute the original method
        const result = originalMethod.apply(this, args);
        
        console.log(`[LOG]: Finished method '${context.name as string}'`);
        return result;
    };
}

class Calculator {
    // 2. Attach the Decorator!
    @LogExecution
    add(a: number, b: number) {
        return a + b;
    }
}

const calc = new Calculator();
calc.add(5, 10); 
// Output:
// [LOG]: Calling method 'add' with args: [5, 10]
// [LOG]: Finished method 'add'
```

### Level 2 — Decorator Factories (Passing Arguments)
What if you want to pass an argument to the decorator? For example, `@Auth("ADMIN_ROLE")`.
You must use a **Decorator Factory**. It is a function that *returns* the actual decorator function!

```typescript
// 1. The Factory Function takes the arguments
function Auth(requiredRole: string) {
    // 2. It returns the actual Decorator Function
    return function (originalMethod: any, context: ClassMethodDecoratorContext) {
        return function (this: any, ...args: any[]) {
            const currentUserRole = "USER"; // Simulate fetching from a session
            
            if (currentUserRole !== requiredRole) {
                throw new Error(`Unauthorized! Requires ${requiredRole}`);
            }
            return originalMethod.apply(this, args);
        };
    };
}

class AdminPanel {
    @Auth("ADMIN")
    deleteDatabase() {
        console.log("Database deleted.");
    }
}

const panel = new AdminPanel();
// panel.deleteDatabase(); // Throws Error: Unauthorized! Requires ADMIN
```

### Level 3 — Class Decorators
A Class Decorator is attached to the entire Class. It can actually intercept the `constructor` and wrap the entire instantiation process!

```typescript
// A class decorator receives the Class constructor
function Singleton<T extends { new (...args: any[]): {} }>(constructor: T, context: ClassDecoratorContext) {
    let instance: any = null;

    // Return a proxy class that intercepts the 'new' keyword!
    return class extends constructor {
        constructor(...args: any[]) {
            if (instance) {
                return instance; // Return the cached instance!
            }
            super(...args);
            instance = this; // Cache the first creation!
        }
    };
}

@Singleton
class DatabaseConnection {
    id = Math.random();
}

const db1 = new DatabaseConnection();
const db2 = new DatabaseConnection();
console.log(db1.id === db2.id); // true! They are the exact same instance!
```

### Level 4 — Property and Accessor Decorators
You can also decorate individual class properties or Getters/Setters.
A classic example is `@AutoBind` (which fixes the `this` context loss in React or vanilla JS).

```typescript
function AutoBind(originalMethod: any, context: ClassMethodDecoratorContext) {
    // We add an initialization phase to the class!
    context.addInitializer(function () {
        // 'this' refers to the class instance being created
        // We permanently bind the method to the instance!
        this[context.name] = this[context.name].bind(this);
    });
}

class Button {
    name = "SubmitBtn";

    @AutoBind
    handleClick() {
        // Without @AutoBind, 'this' would be undefined when passed as an event listener!
        console.log(`Clicked ${this.name}`);
    }
}
```

## REAL_WORLD
**NestJS / Angular Dependency Injection**
Frameworks like NestJS rely heavily on decorators to build their entire architecture.
When you write `@Controller('/users')` or `@Injectable()` in NestJS, the decorator is not wrapping a method. Instead, it is using a library called `reflect-metadata` to attach invisible metadata to the Class.
When the NestJS framework boots up, it scans all the classes, reads the metadata, and automatically wires up the HTTP routes and injects the required database connections based entirely on the decorators!

## INTERVIEW
**Q: I wrote a bunch of decorators in an older project using Angular. I upgraded to TypeScript 5.0, and now my decorators are throwing massive compiler errors regarding the `target` and `propertyKey` arguments. What changed?**
A: **The shift from Experimental to Standard ECMAScript Decorators.**
Before TypeScript 5.0, decorators were an "Experimental" feature enabled by `"experimentalDecorators": true` in `tsconfig.json`. The legacy method decorator signature was `(target, propertyKey, descriptor)`.
In TS 5.0, the official ECMAScript Stage 3 standard was implemented. The signature completely changed to `(originalMethod, context)`.
The old and new decorators are mathematically incompatible!
*The Fix:* If you are maintaining a legacy project (like Angular), you MUST ensure `"experimentalDecorators": true` remains in your `tsconfig.json` to force TypeScript to use the old parsing logic. If you want to use the new standard decorators, you must rewrite them to accept the new `context` object format.

## FEYNMAN CHECK

### Explain Decorators Like I'm 10 Years Old
> A decorator is a function that ANNOTATES a class, method, field, or accessor — wrapping or modifying it without changing the original code. `@Logger` above a method runs Logger(method) at class-definition time. There are TWO versions: (1) The LEGACY decorators (`experimentalDecorators: true` + `emitDecoratorMetadata`) used by Angular, NestJS, TypeORM. (2) The STAGE-3 decorators (TC39 standard, TS 5.0+) — different API, different semantics, future-proof. They cannot easily coexist. Decorators are powerful — they enable Dependency Injection, ORM mappings, validation rules, RPC stubs — but at the cost of "magic" and complex bug surfaces. The TS 5.0 stage-3 decorators fixed many issues but are not yet supported by Angular as of TS 5.x.

### 5 Deep Conceptual Questions

**Q1: Why do frameworks like Angular, NestJS, TypeORM love decorators?**
> **A:** Decorators provide a declarative way to attach METADATA to classes/methods. Angular uses `@Component`, `@Injectable` to register classes with the DI container. TypeORM uses `@Entity`, `@Column` to map classes to database tables. NestJS uses `@Controller`, `@Get` to define HTTP routes. The runtime metadata (via `reflect-metadata`) lets the framework introspect what was decorated. Without decorators, all this metadata would need to be in separate config files or builder calls.

**Q2: Mental model for decorator execution order?**
> **A:** Decorators are FUNCTIONS that run at class DEFINITION time (once per class), not per instance. Order: field initializers → constructor → method decorators (in source order, bottom to top for stacked) → class decorator (last). With multiple stacked decorators `@A @B method() {}`, B runs first, then A wraps the result.

**Q3: Most dangerous misconception? Show with code.**
> **A:** Mixing legacy and stage-3 decorators:
> ```ts
> // ❌ With experimentalDecorators: true, a stage-3 decorator silently fails
> function legacy(target: any, key: string, descriptor: PropertyDescriptor) { /* legacy */ }
> function stage3(value: Function, ctx: ClassMethodDecoratorContext) { /* stage 3 */ }
>
> class A {
>   @legacy
>   foo() {}
> }
> // If both decorators present, behaviour depends on the experimentalDecorators flag
>
> // ✅ Pick ONE — stage-3 for new projects, legacy ONLY when stuck with frameworks needing it
> ```

**Q4: How do stage-3 decorators differ from legacy?**
> **A:** Stage-3 decorators have a NEW SIGNATURE: `(value, context) => newValue`. The `context` carries metadata (`kind`, `name`, `addInitializer`). They support field decorators that return an initial-value function. They have NO emitDecoratorMetadata equivalent — frameworks must use `context.metadata` instead. They run later in class definition (after fields), with stricter ordering rules. Migration is non-trivial because the API surface is completely different.

**Q5: FAANG-grade definition?**
> **A:** "TypeScript decorators are functions applied to classes, methods, fields, and accessors at class-definition time — existing in two incompatible flavours: the legacy (`experimentalDecorators`) form used by Angular/Nest/TypeORM with reflect-metadata for runtime introspection, and the stage-3 standardised form (TS 5.0+) with a context-object API and built-in metadata support but limited framework adoption."

#### Step 2 — Mini DI Container
```ts
// src/di.ts
const REGISTRY = new Map<string, new (...args: any[]) => any>();

export function Injectable(name: string) {
  return function <T extends new (...args: any[]) => any>(
    constructor: T,
    _ctx: ClassDecoratorContext,
  ): T {
    REGISTRY.set(name, constructor);
    return constructor;
  };
}

export function resolve<T>(name: string): T {
  const ctor = REGISTRY.get(name);
  if (!ctor) throw new Error(`Service '${name}' not registered`);
  return new ctor();
}

// Method decorator: logs every call
export function LogCall<This, Args extends unknown[], Out>(
  originalMethod: (this: This, ...args: Args) => Out,
  context: ClassMethodDecoratorContext<This, (this: This, ...args: Args) => Out>,
) {
  const methodName = String(context.name);
  return function (this: This, ...args: Args): Out {
    console.log(`→ ${methodName}(${JSON.stringify(args)})`);
    const result = originalMethod.call(this, ...args);
    console.log(`← ${methodName} returned`, result);
    return result;
  };
}
```
