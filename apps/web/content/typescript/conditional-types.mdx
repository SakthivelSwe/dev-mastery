## WHY
Sometimes you want to create a generic type that makes decisions based on the type of data passed into it. 
For example, if you pass an Array to an API, you want the API to return the underlying element type (e.g., `string[]` becomes `string`). If you pass a bare primitive, you just want to return the primitive itself.
You need the ability to write `if/else` logic directly inside the Type System. 
**Conditional Types** provide this exact capability, turning TypeScript into a Turing-complete functional programming language operating entirely at compile-time.

## THEORY
### The `extends` Keyword as "If"
In standard code, you write `if (condition) { true } else { false }`.
In TypeScript's type system, you write `TypeA extends TypeB ? TrueType : FalseType`.

Think of `extends` as meaning "is assignable to". If `TypeA` fits inside the shape of `TypeB`, the condition is true.

## CODE

### Level 1 — Basic Conditional Logic
Let's create a type that checks if a type is a string.

```typescript
// If T is a string, return the literal type "yes", otherwise "no"
type IsString<T> = T extends string ? "yes" : "no";

type A = IsString<string>;  // Type is "yes"
type B = IsString<number>;  // Type is "no"
type C = IsString<any>;     // Type is "yes" | "no" (Wait, what? See Level 3!)
```

### Level 2 — The `infer` Keyword (Type Extraction)
The most powerful use of Conditional Types is extracting a nested type using the `infer` keyword. `infer` acts like a variable declaration inside the type condition!

Let's write a type that extracts the inner type from an Array. If it's not an Array, return the original type.

```typescript
// If T is an Array of SOME_TYPE (infer U), return U. Else, return T.
type UnpackArray<T> = T extends Array<infer U> ? U : T;

type StrArray = string[];
type Unpacked1 = UnpackArray<StrArray>; // Type is string!

type NumList = Array<number>;
type Unpacked2 = UnpackArray<NumList>;  // Type is number!

type JustBoolean = boolean;
type Unpacked3 = UnpackArray<JustBoolean>; // Type is boolean!
```
The `infer` keyword reached *inside* the array structure and pulled the inner type out for us to use!

### Level 3 — Distributive Conditional Types
What happens if you pass a **Union** type (`string | number`) into a Conditional Type?
TypeScript does something called **Distribution**. It splits the union, runs the condition on *each piece individually*, and then unions the results back together!

```typescript
type ToArray<T> = T extends any ? T[] : never;

// You might expect: (string | number)[]
// What actually happens: string[] | number[]
type StrOrNumArray = ToArray<string | number>;

// Step 1: ToArray<string> | ToArray<number>
// Step 2: string[] | number[]
```

*How do you prevent this?* Wrap the `T` in brackets!
```typescript
// By wrapping T in brackets, we disable distribution.
type ToArrayNoDist<T> = [T] extends [any] ? T[] : never;

// Now it stays together!
type SafeArray = ToArrayNoDist<string | number>; // (string | number)[]
```

### Level 4 — Advanced Filtering (Extract and Exclude)
Because of Distributive Conditional Types, you can easily filter Unions!
TypeScript provides global utility types built exactly this way: `Exclude` and `Extract`.

```typescript
// The source code for Exclude inside the TypeScript compiler:
// type Exclude<T, U> = T extends U ? never : T;

type Status = "success" | "warning" | "error" | "fatal";

// Let's exclude the bad statuses!
type GoodStatus = Exclude<Status, "error" | "fatal">;
// Result: "success" | "warning"

// How it worked under the hood:
// "success" extends ("error"|"fatal") ? never : "success"  -> "success"
// "error" extends ("error"|"fatal") ? never : "error"      -> never
// Result: "success" | never -> "success"
```

## REAL_WORLD
**API Return Types (`Awaited<T>`)**
When you work with Promises, you often have types like `Promise<Promise<string>>`. How do you recursively unwrap the Promises to get the actual underlying string value?
TypeScript provides the `Awaited<T>` utility type, which uses recursive Conditional Types and `infer`!

```typescript
// A simplified version of Awaited
type MyAwaited<T> = T extends Promise<infer U> ? MyAwaited<U> : T;

type Nested = Promise<Promise<number>>;
type Result = MyAwaited<Nested>; // Type is number!
```

## INTERVIEW
**Q: You want to write a type `ReturnType<T>` that takes a Function type and extracts whatever that function returns. How do you write this using Conditional Types and `infer`?**
A: **Use `infer R` in the return position of the function signature.**
```typescript
type ReturnType<T> = T extends (...args: any[]) => infer R ? R : any;
```
Explanation: The condition asks, "Is `T` a function that takes any arguments and returns *something* (`infer R`)?" If true, it returns `R` (the extracted return type). If `T` is not a function at all, it just falls back to `any`.
This is exactly how the built-in global `ReturnType` utility works in TypeScript!

## FEYNMAN CHECK

### Explain Conditional Types Like I'm 10 Years Old
> Conditional types are IF-ELSE for types. `T extends U ? X : Y` means: at compile time, check if T is assignable to U; if yes, produce X; if no, produce Y. Combined with `infer` (to extract sub-types), conditional types let you write `Returntype<F>`, `Parameters<F>`, `Awaited<T>`. The crucial subtlety: when T is a UNION and appears NAKED on the left of extends, the conditional DISTRIBUTES — `('a' | 'b') extends string ? true : false` becomes `true | true` = `true`, but `('a' | 1) extends string ? true : false` becomes `true | false`. Wrap in brackets to disable: `[('a'|1)] extends [string] ? true : false` evaluates once = `false`.

### 5 Deep Conceptual Questions

**Q1: What problem do conditional types solve?**
> **A:** Type-level conditional logic. Without them, you cannot express "this function returns T if input is X, else Y." Conditional types are the foundation of every utility type and most production TS libraries — Redux Toolkit's createSlice, tRPC's procedure builders, zod's z.infer, Drizzle ORM's typed queries — all depend on conditional + infer to derive output types from input shapes.

**Q2: Mental model for distribution?**
> **A:** A NAKED type parameter on the LEFT of extends triggers DISTRIBUTION over unions. `T extends string ? T[] : T` applied to T = 'a' | 1 produces `('a' extends string ? 'a'[] : 'a') | (1 extends string ? 1[] : 1)` = `'a'[] | 1`. To EVALUATE ONCE for the whole union, bracket both sides: `[T] extends [string]` — now the whole T is checked as a single unit.

**Q3: Most dangerous misconception? Show with code.**
> **A:** Forgetting distribution turns simple conditionals into surprising unions:
> ```ts
> // ❌ Trying to check "is T exactly string?"
> type IsString<T> = T extends string ? true : false;
> IsString<'a' | number>;   // = true | false (distributes) — WRONG
>
> // ✅ Disable distribution with brackets
> type IsStringStrict<T> = [T] extends [string] ? true : false;
> IsStringStrict<'a' | number>;   // = false (entire union is not string)
> ```

**Q4: How does `infer` interact with conditional types?**
> **A:** `infer` is only allowed inside the EXTENDS clause of a conditional. It pattern-matches part of the type and binds a name. `T extends Promise<infer U> ? U : T` says "if T is a Promise<something>, give that something the name U and use it." You can have multiple infers, even in nested positions: `T extends (...args: infer A) => infer R ? { args: A; returns: R } : never`.

**Q5: FAANG-grade definition?**
> **A:** "Conditional types are TypeScript's type-level ternary expressions `T extends U ? X : Y` — distributing over union members when T is a naked type parameter, supporting type extraction via the `infer` keyword inside the extends clause — collectively enabling the implementation of every utility type and the type-derivation patterns underpinning production ORMs, routers, and validators."

#### Step 2 — Build Utility Types From Scratch
```ts
// src/utils.ts
// 1. ReturnType
export type MyReturnType<F> = F extends (...args: any[]) => infer R ? R : never;

// 2. Parameters
export type MyParameters<F> = F extends (...args: infer A) => any ? A : never;

// 3. Awaited (TS 4.5+)
export type MyAwaited<T> =
  T extends Promise<infer U>
    ? MyAwaited<U>                       // recurse for Promise<Promise<X>>
    : T;

// 4. First and Last in a tuple
export type First<T extends unknown[]> = T extends [infer F, ...unknown[]] ? F : never;
export type Last<T extends unknown[]>  = T extends [...unknown[], infer L] ? L : never;

// 5. NonNullable from scratch
export type MyNonNullable<T> = T extends null | undefined ? never : T;
```

#### Step 3 — Use Them With Real Functions
```ts
// src/main.ts
import type {
  MyReturnType, MyParameters, MyAwaited,
  First, Last, MyNonNullable,
} from './utils.js';

function greet(name: string, age: number): { hi: string } { return { hi: name }; }

type R = MyReturnType<typeof greet>;        // { hi: string }
type P = MyParameters<typeof greet>;        // [string, number]
type A = MyAwaited<Promise<Promise<number>>>;   // number
type F = First<[1, 2, 3]>;                  // 1
type L = Last<['a', 'b', 'c']>;            // 'c'
type N = MyNonNullable<string | null>;      // string

console.log('Types compile cleanly!');
```

#### Step 4 — Error Handling: Distribution Demo
```ts
// Distributive (default)
type ToArray<T> = T extends any ? T[] : never;
type Distributed = ToArray<string | number>;   // string[] | number[]

// Non-distributive
type ToArrayStrict<T> = [T] extends [any] ? T[] : never;
type Single = ToArrayStrict<string | number>;   // (string | number)[]

// Common bug fix: NonNullable wrongly distributing
type WrongNonNull<T> = T extends null ? never : T;   // distributes — for T=null|undefined this is undefined
type RightNonNull<T> = T extends null | undefined ? never : T;
```
