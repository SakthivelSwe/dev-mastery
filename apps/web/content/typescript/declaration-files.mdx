## WHY
If you write 100% of your code in TypeScript, everything is perfectly typed. 
But what happens when you `npm install lodash` or use an obscure JavaScript library written 10 years ago? The TypeScript compiler panics because it has no idea what `_.debounce()` is, what arguments it takes, or what it returns. It sees pure, untyped JavaScript.

To bridge the gap between pure JavaScript libraries and the strict TypeScript compiler, we use **Declaration Files (`.d.ts`)**. These files act as a "Translation Dictionary", providing the missing type signatures for JavaScript code so the compiler can understand it.

## THEORY
### Ambient Declarations
A `.ts` file contains **Implementation** (actual executable code).
A `.d.ts` file contains **Declarations** (only types, interfaces, and signatures). It contains zero executable code. It physically disappears when compiled to JavaScript.

You use the `declare` keyword to tell TypeScript: *"Trust me, this variable or module exists at runtime. Here is its type."* This is called an **Ambient Declaration**.

## CODE

### Level 1 — Declaring Global Variables
Imagine you include a third-party script in your `index.html` (like Google Analytics). At runtime, a global variable `ga` exists on the `window` object. But if you type `ga('send', 'pageview');` in your TypeScript file, the compiler throws an error: `Cannot find name 'ga'`.

You must declare it in a `.d.ts` file (e.g., `global.d.ts`):

```typescript
// global.d.ts
// I am telling the compiler: "A global function named 'ga' exists!"
declare function ga(action: string, payload: string): void;

// Or for a global configuration object:
declare const AppConfig: {
    apiUrl: string;
    environment: "dev" | "prod";
};
```
Once this file is included in your project (usually picked up automatically by `tsconfig.json`), you can use `ga` and `AppConfig` anywhere in your app without importing them, and with full type safety!

### Level 2 — Declaring Modules (Untyped npm packages)
You run `npm install obscure-js-lib`. It has no types.
In your code, you write `import { doMagic } from 'obscure-js-lib';`.
TypeScript screams: `Could not find a declaration file for module 'obscure-js-lib'.`

**The Fix:** Create a module declaration.
```typescript
// types.d.ts
declare module 'obscure-js-lib' {
    // You map out the exact API surface of the library here!
    export function doMagic(input: string): number;
    export const version: string;
}
```
Now, TypeScript maps your `import` statement to this declaration block, and the error disappears!

### Level 3 — Extending Existing Modules (Module Augmentation)
What if an npm library *does* have types, but you want to add a custom property to it? 
For example, in Express.js, the `Request` object has types. But you wrote a custom authentication middleware that adds `req.user`. TypeScript doesn't know `req.user` exists, so it throws an error!

You use **Module Augmentation** to inject your new types into the existing library's definitions!

```typescript
// express-auth.d.ts
import { UserData } from './models';

// Open up the existing 'express' module...
declare module 'express-serve-static-core' {
    // ...and inject a new property into the existing Request interface!
    interface Request {
        user?: UserData;
    }
}
```
Because TypeScript interfaces are "Open" (they merge if declared multiple times), the compiler merges your custom `Request` interface with the official Express `Request` interface!

### Level 4 — Typing Non-Code Assets (CSS, SVG)
In modern bundlers (like Vite or Webpack), you can import CSS or SVG files directly into your JavaScript: `import logo from './logo.svg';`.
TypeScript will throw an error because an SVG is not a JS/TS module!

You use a wildcard module declaration to teach TypeScript how to handle non-code imports.

```typescript
// assets.d.ts

// Any import that ends in .svg is treated as a string!
declare module '*.svg' {
    const content: string;
    export default content;
}

// Any import that ends in .module.css is treated as an object of class names!
declare module '*.module.css' {
    const classes: { [key: string]: string };
    export default classes;
}
```

## REAL_WORLD
**DefinitelyTyped (`@types`)**
You rarely have to write declaration files for popular libraries. 
The open-source community maintains a massive GitHub repository called **DefinitelyTyped**. It contains community-written `.d.ts` files for over 8,000 JavaScript libraries!
When you install `lodash` (which is written in pure JS), you instantly fix the TypeScript errors by running:
`npm install --save-dev @types/lodash`
This downloads the community-authored declaration files directly into your `node_modules/@types` folder, which the TypeScript compiler reads automatically.

## INTERVIEW
**Q: You created a `global.d.ts` file and wrote `declare const API_URL: string;`. It worked perfectly. Then, you added `import { User } from './types'` to the top of the file. Suddenly, `API_URL` broke and is no longer available globally! Why did the `import` statement destroy the global declaration?**
A: **The file transitioned from a "Script" to a "Module".**
In TypeScript, a file without any top-level `import` or `export` statements is considered a "Script" (or "Global"). Anything declared in it pollutes the global namespace.
The moment you add an `import` or `export` statement to the top level, the file instantly becomes a "Module". Modules have isolated scopes! Therefore, `declare const API_URL` is now scoped *locally* to that module and is no longer global.
*The Fix:* If you must import types into a global declaration file, you must wrap your declarations in the `declare global` block:
```typescript
import { User } from './types';

declare global {
    const API_URL: string;
}
```

## FEYNMAN CHECK

### Explain Declaration Files Like I'm 10 Years Old
> A declaration file (`.d.ts`) is a TYPES-ONLY blueprint — no implementation, just the SHAPES. When you write TS source, the compiler can EMIT matching .d.ts files alongside .js — this is how libraries ship type information. When you use a JS-only library, you can WRITE a .d.ts file describing its shape so TS knows what's there. `@types/foo` packages on npm are community-maintained .d.ts for popular JS libraries (lodash, express, react-dom). Ambient declarations (`declare module 'foo' { ... }`) let you type a library inline. Triple-slash directives (`/// <reference path="..." />`) are legacy but still appear. The non-obvious power: declaration merging lets you AUGMENT existing types (extend Express's Request, add custom Window properties).

---

### 5 Deep Conceptual Questions

**Q1: When do you need to write a .d.ts file?**
> **A:** Three cases: (1) Your TS library publishing — `declaration: true` emits them automatically. (2) Using a JS-only library with no @types/* package — write your own .d.ts. (3) Augmenting global types — adding to `window.myFoo`, extending Express `Request.user`, adding custom CSS in JSX. For modern projects most libraries ship types; you usually only write .d.ts for project-specific augmentation.

**Q2: Mental model for declaration files?**
> **A:** Think of .d.ts as the "header file" of TypeScript (C analog). It describes WHAT exists without HOW. Pure types are erased at runtime; declaration files preserve them across compilation boundaries. They're how TS bridges to JS-only code, how IDE intellisense works for installed packages, and how libraries express their public API without forcing consumers to compile from source.

**Q3: Most dangerous misconception? Show with code.**
> **A:** Putting `import` in an ambient declaration accidentally makes it a module:
> ```ts
> // ❌ The import makes this file a MODULE — globals don't apply globally
> import { type Foo } from './foo';
> declare global {
>   interface Window { myApp: { foo: Foo } }   // works only when imported, not globally
> }
>
> // ✅ Wrap in declare global { } and avoid top-level non-type imports
> // For augmentation, use 'export {}' to make it explicitly a module
> declare global {
>   interface Window { myApp: { foo: string } }
> }
> export {};   // makes the file a module
> ```

**Q4: How does declaration merging work for libraries like Express?**
> **A:** Express ships `@types/express` with `interface Request { ... }`. In your app, you write `declare module 'express-serve-static-core' { interface Request { user?: User } }` — TS MERGES this with the original interface, adding your custom `user` property. Now every `req.user` is typed across your codebase. This is the canonical pattern for typing middleware-added properties.

**Q5: FAANG-grade definition?**
> **A:** "Declaration files (.d.ts) are TypeScript's type-only header files describing the public shape of modules without implementation — emitted automatically by tsc with `declaration: true`, hand-written for type augmentation (declare global, declare module), distributed via @types/* packages for JavaScript libraries, and combined through declaration merging to extend third-party types in application code."

#### Step 4 — Use Both Augmentations
```ts
// src/server.ts
import express, { type Request, type Response, type NextFunction } from 'express';
// @ts-ignore — local alias
import * as legacy from 'legacy-lib';

const app = express();

// Middleware: assign requestId + (fake) user
app.use((req: Request, _res: Response, next: NextFunction) => {
  req.requestId = Math.random().toString(36).slice(2);
  req.user = { id: 1, email: 'demo@example.com', role: 'admin' };
  next();
});

app.get('/profile', (req, res) => {
  // req.user is now typed — autocomplete works
  if (!req.user) return res.status(401).end();
  res.json({
    user: req.user,
    requestId: req.requestId,
    checksum: legacy.computeChecksum(req.user.email),
    libVersion: legacy.VERSION,
  });
});

app.listen(3000, () => console.log('Listening on 3000'));
```
