## WHY
In raw JavaScript, variables have no fixed type. A variable `count` can hold a number `10`, and on the next line, it can be reassigned to the string `"ten"`. This flexibility is convenient for writing tiny scripts, but in massive applications, it causes horrific runtime crashes (e.g., trying to execute `.toUpperCase()` on a number). 

TypeScript's Basic Types allow you to explicitly define the "shape" of your data. By telling the compiler exactly what type a variable *must* be, TypeScript can physically prevent you from writing code that will crash in the browser, shifting errors from "Runtime" to "Compile-Time".

## THEORY
### Type Annotations vs. Type Inference
- **Type Annotation:** You explicitly tell TypeScript what the type is using a colon (`: type`).
- **Type Inference:** You just assign a value, and TypeScript is smart enough to implicitly *guess* the type. You don't need to annotate everything!

TypeScript supports the JavaScript primitives (`string`, `number`, `boolean`, `null`, `undefined`, `symbol`, `bigint`), plus a few special types (`any`, `unknown`).

## CODE

### Level 1 — Primitives and Inference
The most common types are `string`, `number`, and `boolean`.

```typescript
// Explicit Type Annotation (The colon syntax)
let firstName: string = "John";
let age: number = 28;
let isActive: boolean = true;

// Type Inference (Best Practice)
// TypeScript sees "Alice" is a string, so it locks 'lastName' as a string automatically.
let lastName = "Alice"; 

// ERROR: Type 'number' is not assignable to type 'string'.
lastName = 42; 
```

### Level 2 — `null` and `undefined`
In JavaScript, `null` means "intentionally empty", and `undefined` means "not yet assigned".
By default (with `strictNullChecks` enabled), TypeScript treats them as completely distinct types that cannot be assigned to variables of other types.

```typescript
let missingValue: null = null;
let notAssigned: undefined = undefined;

let city: string = "New York";

// ERROR: Type 'null' is not assignable to type 'string'.
// This prevents the dreaded "Cannot read properties of null" runtime error!
city = null; 
```

### Level 3 — The Danger of `any`
`any` is an escape hatch. When you type a variable as `any`, you are telling the TypeScript compiler: *"Turn off all type checking for this variable. I know what I'm doing."*

```typescript
let wildcard: any = "Hello";
wildcard = 42;          // Allowed
wildcard = true;        // Allowed

// TypeScript will NOT catch this error! 
// This will crash at runtime because 42 does not have a .toUpperCase() method!
wildcard.toUpperCase(); 
```
**Rule of Thumb:** Never use `any`. Using `any` defeats the entire purpose of using TypeScript. If you don't know the type, use `unknown`.

### Level 4 — `unknown` (The Safe `any`)
`unknown` is similar to `any`, meaning it can accept any value. However, it is fundamentally safer: TypeScript forces you to *prove* what the type is before you are allowed to use it!

```typescript
let safeWildcard: unknown = "Hello World";
safeWildcard = 42;

// ERROR: 'safeWildcard' is of type 'unknown'. 
// TypeScript blocks this because it might not be a string!
// safeWildcard.toUpperCase(); 

// To use it, you must "Narrow" the type by proving what it is:
if (typeof safeWildcard === "string") {
    // Inside this block, TypeScript KNOWS it is a string!
    console.log(safeWildcard.toUpperCase());
}
```

## REAL_WORLD
**API Responses and `unknown`**
When you make a network request (e.g., `fetch('/api/data')`), the resulting JSON payload is entirely unpredictable. 
Developers often lazily cast the response as `any`. This is a massive vulnerability. 
Instead, you should cast raw API responses as `unknown`. This forces you (and your team) to write validation logic (like a Zod schema) to verify the data shape *before* the application attempts to process it.

## INTERVIEW
**Q: What is the difference between the `String` type (capital S) and the `string` type (lowercase s) in TypeScript? Which should you use?**
A: **Always use `string`.**
In JavaScript, `string` (lowercase) is the primitive data type. `String` (uppercase) is the global Wrapper Object used to create strings (e.g., `new String("Hello")`). 
In TypeScript, if you use the uppercase `String`, you are typing the variable as the *Wrapper Object*, not the primitive. This causes bizarre compatibility issues because a primitive `string` is not identical to a wrapper `String` object. You should always use lowercase primitives (`string`, `number`, `boolean`) in your type annotations.

## FEYNMAN CHECK

### Explain TS Basic Types Like I'm 10 Years Old
> TypeScript's basic types are the LEGO BRICKS you build everything else from: `string`, `number` (no separate int/float — all numbers are 64-bit floats like JS), `boolean`, `null`, `undefined`, `bigint`, `symbol`, plus the special types `any` (escape hatch — disables checking), `unknown` (safe `any` — must narrow before use), `void` (function returns nothing), and `never` (function never returns — throws or loops forever). Literal types like `'GET' | 'POST'` are exact-string types — a variable typed `'GET'` can ONLY hold the string "GET". The non-obvious detail: when you write `const x = "hi"`, x is typed `"hi"` (literal). When you write `let x = "hi"`, x is typed `string` (widened) — because let can be reassigned.

### 5 Deep Conceptual Questions

**Q1: Why `unknown` instead of `any`?**
> **A:** `any` opts OUT of type checking — assigning `any` to anything works, calling `.foo()` on `any` is allowed. `unknown` keeps the type system honest — you can ASSIGN anything to unknown (it's a top type) but you can DO NOTHING with it until you NARROW it via `typeof` or `instanceof` or a custom type guard. Use `unknown` for JSON-parsed data, event payloads, and any input from outside your system; use `any` only as a temporary escape hatch during migration.

**Q2: Mental model for literal types?**
> **A:** A literal type is a type with EXACTLY ONE value. `type Method = 'GET' | 'POST'` is a union of two literal string types. Combined with `const` (which produces literal types) and `as const` (which makes objects deeply literal), you get exhaustive switch checking, autocomplete, and no string-typo bugs.

**Q3: Most dangerous misconception? Show with code.**
> **A:** Believing `null` and `undefined` are interchangeable:
> ```ts
> // ❌ strict mode catches the difference
> interface User { name: string; phone: string | null }
> function call(u: User) {
>   if (u.phone) console.log(u.phone);    // OK
>   else console.log(u.phone.length);     // ❌ TS error: phone is null
> }
>
> // ✅ With strictNullChecks, null and undefined must be handled explicitly
> // Always prefer T | undefined for optionality; reserve T | null for "explicitly absent"
> ```

**Q4: How does `never` interact with type unions at compile time?**
> **A:** `never` is the EMPTY type — no value inhabits it. It's used as a return type for functions that throw or loop forever, and as the "default" in exhaustive switch checking: `const _: never = x` will compile only if x is `never` (i.e., all cases were handled). In unions, `T | never` simplifies to `T` — never is absorbed. Mapped types use `never` to FILTER keys: `{ [K in keyof T as T[K] extends Function ? K : never]: T[K] }` filters to only function-valued keys.

**Q5: FAANG-grade definition?**
> **A:** "TypeScript's primitive types map JavaScript's runtime types into the static type system, augmented by literal types (single-value subtypes of primitives), `unknown` (the safe top type requiring narrowing before use), `never` (the empty bottom type used for exhaustiveness checks), and special types `any` and `void` — all governed by strictNullChecks for null-safety and by literal widening rules in const vs let declarations."

#### Step 2 — Type-Safe HTTP Client
```ts
// src/http.ts
type Method = 'GET' | 'POST' | 'PUT' | 'DELETE';

export type FetchResult =
  | { kind: 'ok'; data: unknown }
  | { kind: 'http_error'; status: number; message: string }
  | { kind: 'network_error'; cause: Error };

export async function request(
  method: Method,
  url: string,
  body?: unknown,            // unknown forces caller to be intentional
): Promise<FetchResult> {
  try {
    const res = await fetch(url, {
      method,
      headers: body ? { 'Content-Type': 'application/json' } : {},
      body: body ? JSON.stringify(body) : undefined,
    });
    if (!res.ok) {
      return { kind: 'http_error', status: res.status, message: res.statusText };
    }
    const data: unknown = await res.json();
    return { kind: 'ok', data };
  } catch (e) {
    return { kind: 'network_error', cause: e as Error };
  }
}
```

#### Step 3 — Caller With Exhaustive Switch
```ts
// src/main.ts
import { request, type FetchResult } from './http.js';

function describe(result: FetchResult): string {
  switch (result.kind) {
    case 'ok':            return `OK — body has ${JSON.stringify(result.data).length} bytes`;
    case 'http_error':    return `HTTP ${result.status}: ${result.message}`;
    case 'network_error': return `Network failure: ${result.cause.message}`;
    default: {
      const _exhaustive: never = result;   // compile error if a kind was added but not handled
      return _exhaustive;
    }
  }
}

const r = await request('GET', 'https://api.github.com/users/torvalds');
console.log(describe(r));
```

#### Step 4 — Error Handling: Narrow `unknown` Data Safely
```ts
function isUserShape(x: unknown): x is { login: string; id: number } {
  return typeof x === 'object' && x !== null
      && 'login' in x && typeof (x as { login: unknown }).login === 'string'
      && 'id' in x    && typeof (x as { id:    unknown }).id    === 'number';
}

if (r.kind === 'ok' && isUserShape(r.data)) {
  console.log('GitHub login:', r.data.login, 'id:', r.data.id);
}
```
