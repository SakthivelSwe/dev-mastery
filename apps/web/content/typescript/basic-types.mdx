## WHY
In raw JavaScript, variables have no fixed type. A variable `count` can hold a number `10`, and on the next line, it can be reassigned to the string `"ten"`. This flexibility is convenient for writing tiny scripts, but in massive applications, it causes horrific runtime crashes (e.g., trying to execute `.toUpperCase()` on a number). 

TypeScript's Basic Types allow you to explicitly define the "shape" of your data. By telling the compiler exactly what type a variable *must* be, TypeScript can physically prevent you from writing code that will crash in the browser, shifting errors from "Runtime" to "Compile-Time".

## THEORY
### Type Annotations vs. Type Inference
- **Type Annotation:** You explicitly tell TypeScript what the type is using a colon (`: type`).
- **Type Inference:** You just assign a value, and TypeScript is smart enough to implicitly *guess* the type. You don't need to annotate everything!

TypeScript supports the JavaScript primitives (`string`, `number`, `boolean`, `null`, `undefined`, `symbol`, `bigint`), plus a few special types (`any`, `unknown`).

## CODE

### Level 1 — Primitives and Inference
The most common types are `string`, `number`, and `boolean`.

```typescript
// Explicit Type Annotation (The colon syntax)
let firstName: string = "John";
let age: number = 28;
let isActive: boolean = true;

// Type Inference (Best Practice)
// TypeScript sees "Alice" is a string, so it locks 'lastName' as a string automatically.
let lastName = "Alice"; 

// ERROR: Type 'number' is not assignable to type 'string'.
lastName = 42; 
```

### Level 2 — `null` and `undefined`
In JavaScript, `null` means "intentionally empty", and `undefined` means "not yet assigned".
By default (with `strictNullChecks` enabled), TypeScript treats them as completely distinct types that cannot be assigned to variables of other types.

```typescript
let missingValue: null = null;
let notAssigned: undefined = undefined;

let city: string = "New York";

// ERROR: Type 'null' is not assignable to type 'string'.
// This prevents the dreaded "Cannot read properties of null" runtime error!
city = null; 
```

### Level 3 — The Danger of `any`
`any` is an escape hatch. When you type a variable as `any`, you are telling the TypeScript compiler: *"Turn off all type checking for this variable. I know what I'm doing."*

```typescript
let wildcard: any = "Hello";
wildcard = 42;          // Allowed
wildcard = true;        // Allowed

// TypeScript will NOT catch this error! 
// This will crash at runtime because 42 does not have a .toUpperCase() method!
wildcard.toUpperCase(); 
```
**Rule of Thumb:** Never use `any`. Using `any` defeats the entire purpose of using TypeScript. If you don't know the type, use `unknown`.

### Level 4 — `unknown` (The Safe `any`)
`unknown` is similar to `any`, meaning it can accept any value. However, it is fundamentally safer: TypeScript forces you to *prove* what the type is before you are allowed to use it!

```typescript
let safeWildcard: unknown = "Hello World";
safeWildcard = 42;

// ERROR: 'safeWildcard' is of type 'unknown'. 
// TypeScript blocks this because it might not be a string!
// safeWildcard.toUpperCase(); 

// To use it, you must "Narrow" the type by proving what it is:
if (typeof safeWildcard === "string") {
    // Inside this block, TypeScript KNOWS it is a string!
    console.log(safeWildcard.toUpperCase());
}
```

## REAL_WORLD
**API Responses and `unknown`**
When you make a network request (e.g., `fetch('/api/data')`), the resulting JSON payload is entirely unpredictable. 
Developers often lazily cast the response as `any`. This is a massive vulnerability. 
Instead, you should cast raw API responses as `unknown`. This forces you (and your team) to write validation logic (like a Zod schema) to verify the data shape *before* the application attempts to process it.

## INTERVIEW
**Q: What is the difference between the `String` type (capital S) and the `string` type (lowercase s) in TypeScript? Which should you use?**
A: **Always use `string`.**
In JavaScript, `string` (lowercase) is the primitive data type. `String` (uppercase) is the global Wrapper Object used to create strings (e.g., `new String("Hello")`). 
In TypeScript, if you use the uppercase `String`, you are typing the variable as the *Wrapper Object*, not the primitive. This causes bizarre compatibility issues because a primitive `string` is not identical to a wrapper `String` object. You should always use lowercase primitives (`string`, `number`, `boolean`) in your type annotations.
