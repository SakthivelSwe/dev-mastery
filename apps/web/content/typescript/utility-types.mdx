## WHY
TypeScript ships with a powerful library of built-in "Utility Types" — generic types that transform other types. They solve the most common type manipulation problems without requiring you to write complex Mapped Types from scratch.

For example: You have a massive `User` interface with 20 fields. Your "Create User" form only needs 5 of those fields. Instead of defining a completely separate `CreateUserForm` interface and manually duplicating 5 properties (risking them falling out of sync), you can derive the form type directly from `User` using a Utility Type!

## THEORY
### Why Utility Types Matter
Every non-trivial TypeScript project uses Utility Types extensively. They enforce the **DRY (Don't Repeat Yourself) Principle** at the type level. When the source `User` interface is updated (e.g., a field is renamed), all derived Utility Types update automatically, eliminating an entire class of synchronization bugs.

## CODE

### Level 1 — Beginner (Partial & Required)
```typescript
interface User {
    id: number;
    username: string;
    email: string;
    bio?: string;   // Optional field
}

// Partial<T>: Makes ALL fields optional
// Perfect for "Update User" operations where only some fields change.
type UpdateUserDto = Partial<User>;

function updateUser(id: number, changes: UpdateUserDto): void {
    // changes.username might be undefined — and TypeScript knows it!
    console.log(changes.username?.toUpperCase());
}
updateUser(1, { username: "alice_new" }); // ✅ Only username, rest are optional


// Required<T>: Makes ALL optional fields required
// The opposite of Partial!
type FullUser = Required<User>;
const fullUser: FullUser = { id: 1, username: "alice", email: "alice@test.com", bio: "..." };
// bio is REQUIRED now!
```

### Level 2 — Building Blocks (Pick & Omit)
```typescript
interface User {
    id: number;
    username: string;
    email: string;
    passwordHash: string; // Sensitive field!
    createdAt: Date;
}

// Pick<T, Keys>: Creates a NEW type containing ONLY the specified keys
// Perfect for response DTOs that exclude sensitive data!
type PublicUserProfile = Pick<User, "id" | "username">;

const profile: PublicUserProfile = { id: 1, username: "alice" };
// profile.email; // ❌ ERROR! email was not picked.
// profile.passwordHash; // ❌ ERROR! Sensitive field not available!

// Omit<T, Keys>: Creates a NEW type EXCLUDING the specified keys
// Easier than Pick when you want most fields but need to exclude a few.
type CreateUserDto = Omit<User, "id" | "createdAt">;

const newUser: CreateUserDto = {
    username: "bob",
    email: "bob@test.com",
    passwordHash: "$2a$12$..."
};
```

### Level 3 — Intermediate (Record, Readonly, ReturnType)
```typescript
// Record<Keys, ValueType>: Creates a type that is a Map/Dictionary
type UserRoles = Record<string, "admin" | "user" | "viewer">;
const roles: UserRoles = {
    "alice": "admin",
    "bob": "user",
    "charlie": "viewer"
};

// Record with a union of literal keys — creates a strict enum-like dictionary
type StatusLabels = Record<"active" | "inactive" | "pending", string>;
const labels: StatusLabels = {
    active: "Active User",
    inactive: "Inactive Account",
    pending: "Awaiting Verification"
};

// Readonly<T>: Makes all properties immutable (cannot be reassigned)
const frozenUser: Readonly<User> = { id: 1, username: "alice", email: "...", passwordHash: "...", createdAt: new Date() };
// frozenUser.username = "bob"; // ❌ COMPILE ERROR! Cannot assign to read-only property.

// ReturnType<typeof fn>: Extracts the return type of a function
function getCurrentUser() {
    return { id: 1, username: "alice", role: "admin" };
}
type CurrentUser = ReturnType<typeof getCurrentUser>;
// CurrentUser is automatically { id: number, username: string, role: string }
```

### Level 4 — Advanced (Parameters, Awaited, NonNullable)
```typescript
// Parameters<typeof fn>: Extracts the parameters as a Tuple type
function createOrder(userId: number, productId: number, quantity: number): void {}
type OrderParams = Parameters<typeof createOrder>;
// OrderParams = [number, number, number]

// Useful for wrapper functions:
function wrappedCreateOrder(...args: OrderParams): void {
    console.log("Creating order...");
    createOrder(...args);
}

// Awaited<T>: Unwraps the resolved type of a Promise
type FetchedUser = Awaited<Promise<User>>;
// FetchedUser = User (not Promise<User>!)

async function fetchUser(): Promise<User> { /* ... */ return {} as User; }
type FetchResult = Awaited<ReturnType<typeof fetchUser>>;
// FetchResult = User ✅

// NonNullable<T>: Removes null and undefined from a type
type MaybeString = string | null | undefined;
type DefinitelyString = NonNullable<MaybeString>;
// DefinitelyString = string
```

## REAL_WORLD
Utility Types are the cornerstone of **Type-Safe DTOs (Data Transfer Objects)** in Enterprise APIs.

A common Enterprise pattern is the **DTO Transformation Chain**:
```typescript
interface User { id: number; username: string; email: string; passwordHash: string; }

type CreateUserDto = Omit<User, "id"> & { password: string };
// (Username + Email + Password, but no ID or hash yet)

type UpdateUserDto = Partial<Omit<User, "id" | "passwordHash">>;
// (Only username or email, optional, no ID or sensitive fields)

type UserResponseDto = Omit<User, "passwordHash">;
// (Safe to send to client — no password hash!)
```

## INTERVIEW
**Q: Can you use `Partial<T>` inside an API endpoint to allow partial updates (PATCH)?**
A: Yes! This is exactly the right use case. A `PATCH /users/{id}` endpoint should accept a `Partial<UpdateUserDto>`. The controller method takes the partial update, merges it with the existing database record (using `Object.assign` or spread), and saves the result.

**Q: What is the difference between `Pick` and `Omit`? When do you use each?**
A: Functionally they are opposites. Use `Pick` when you want a small subset of a large type (explicitly listing the few fields you want). Use `Omit` when you want most of a type but need to exclude a few fields (explicitly listing the few fields to exclude). If excluding 2 fields from a 20-field type, `Omit` is much cleaner than `Pick`ing all 18 remaining fields.
