## WHY
JavaScript added `class` syntax in ES6, but it lacked the powerful Object-Oriented (OOP) features found in languages like Java or C#. By default, all properties in a JavaScript class are completely public, meaning any other developer can accidentally mutate your class's internal state.

TypeScript supercharges JavaScript classes by adding **Access Modifiers** (public, private, protected), **Abstract Classes**, and strict **Interface Implementation**, bringing robust, enterprise-grade OOP to the frontend.

## THEORY
### Access Modifiers
TypeScript introduces three keywords to control who can see and modify class properties:
1. `public` (Default): Accessible from anywhere.
2. `private`: Accessible ONLY from within the class itself.
3. `protected`: Accessible from within the class AND any subclasses that extend it.

## CODE

### Level 1 — Beginner (Basic Classes & Access Modifiers)
```typescript
class BankAccount {
    // 1. Declare properties
    public owner: string;
    // Private properties prevent external mutation!
    private balance: number;

    // 2. Constructor initializes the properties
    constructor(owner: string, initialBalance: number) {
        this.owner = owner;
        this.balance = initialBalance;
    }

    public deposit(amount: number) {
        this.balance += amount;
    }

    public getBalance() {
        return this.balance;
    }
}

const account = new BankAccount("Alice", 100);
console.log(account.owner); // ✅ Valid (Public)

// ❌ Error: Property 'balance' is private and only accessible within class 'BankAccount'.
// account.balance = 50000; 

account.deposit(50); // ✅ Controlled mutation via a public method
```

### Level 2 — Building Blocks (Parameter Properties & Getters/Setters)
TypeScript offers a massive shortcut for defining and assigning class properties.

```typescript
class User {
    // Parameter Properties: 
    // Adding an access modifier (public/private/readonly) inside the constructor 
    // automatically declares and assigns the property!
    constructor(
        public readonly id: string, 
        private _firstName: string, 
        private _lastName: string
    ) {}

    // Getters and Setters allow you to intercept property access
    get fullName(): string {
        return `${this._firstName} ${this._lastName}`;
    }

    set firstName(newName: string) {
        if (newName.length < 2) throw new Error("Name too short");
        this._firstName = newName;
    }
}

const user = new User("123", "John", "Doe");
// Looks like a normal property access, but it invokes the Getter!
console.log(user.fullName); 
// Invokes the Setter!
user.firstName = "Jane"; 
```

### Level 3 — Intermediate (Implementing Interfaces)
In enterprise applications, you want to decouple the implementation from the contract. A Class can implement an Interface to guarantee it conforms to a specific shape.

```typescript
interface Logger {
    log(message: string): void;
    error(message: string): void;
}

// ❌ Error: Class 'ConsoleLogger' incorrectly implements interface 'Logger'.
// Property 'error' is missing in type 'ConsoleLogger'.
class ConsoleLogger implements Logger {
    log(message: string) {
        console.log(`[LOG]: ${message}`);
    }
    // Must implement error() to satisfy the compiler!
    error(message: string) {
        console.error(`[ERROR]: ${message}`);
    }
}
```

### Level 4 — Advanced (Abstract Classes)
Sometimes you want to create a "Base Class" that shares some functionality, but you want to force developers to implement specific methods in the subclasses. You can't use an Interface for this because Interfaces cannot hold implementation logic.

Enter **Abstract Classes**. They cannot be instantiated directly.

```typescript
// 1. Mark the class as abstract
abstract class PaymentProcessor {
    constructor(protected gatewayUrl: string) {}

    // A fully implemented shared method
    public logTransaction(amount: number) {
        console.log(`Processing $${amount} via ${this.gatewayUrl}`);
    }

    // 2. An abstract method forces the subclass to implement it!
    abstract processPayment(amount: number): boolean;
}

// ❌ Error: Cannot create an instance of an abstract class.
// const p = new PaymentProcessor("http...");

// ✅ Subclasses inherit shared logic and provide specific implementations
class StripeProcessor extends PaymentProcessor {
    processPayment(amount: number): boolean {
        // Stripe-specific logic here
        return true;
    }
}
```

## REAL_WORLD
**ECMAScript Private Fields (`#`) vs TypeScript `private`**
Modern JavaScript recently introduced native private fields using the `#` symbol (e.g., `#balance: number;`). 
- TypeScript's `private` keyword is purely a compile-time check. If you compile your TS to JS, the property becomes fully public at runtime. A malicious user inspecting your JS could easily modify `account.balance`.
- JavaScript's `#` syntax provides true **runtime privacy**. The property is impossible to access from the outside, even in the compiled JavaScript. 

Today, most developers still use TypeScript's `private` keyword because it looks cleaner and integrates better with parameter properties, but for strict security libraries, `#` is preferred.

## INTERVIEW
**Q: What is the difference between an Interface and an Abstract Class?**
A: 
- **Interfaces** contain absolutely zero implementation logic. They are completely erased during compilation and do not exist in the final JavaScript file. They simply describe a "Shape" or a contract. A class can implement *multiple* interfaces.
- **Abstract Classes** can contain full implementations (real functions and properties) alongside abstract signatures. They survive compilation and become real JavaScript classes. A class can only extend *one* abstract class (JavaScript does not support multiple inheritance).
