## WHY
If you open a 100,000-word text document and press `Ctrl+F` to search for the word "hello", how does the computer find it so instantly?
A naive string search uses a nested loop: check index 0, check index 1, check index 2. If the text is length $N$ and the word is length $M$, this takes $O(N \times M)$ time. 
For massive genomics data (searching for a DNA sequence inside a 3-billion-letter genome), $O(N \times M)$ is completely unviable. 

**Hard String Algorithms** like KMP and Rabin-Karp eliminate the inner loop, guaranteeing highly efficient string matching in $O(N + M)$ linear time.

## THEORY
### The Naive Problem
If you are searching for `"ABABC"` inside the text `"ABABABABC"`, you match `A`, `B`, `A`, `B`, and then fail on the 5th letter (you found `A`, but you needed `C`).
In a naive search, you throw all that work away, shift your starting position by 1, and start the comparison completely over from scratch!

**Knuth-Morris-Pratt (KMP)** realizes that because we already matched `"ABAB"`, we *know* the structure of the string we just scanned. Instead of shifting by 1 and starting over, KMP uses a precomputed table to intelligently shift the pattern forward multiple spaces, avoiding redundant comparisons!

## CODE

### Level 1 — Beginner (The Naive Search)
This is what you want to avoid, but you must understand it to appreciate the optimization.

```javascript
function naiveSearch(text, pattern) {
    const n = text.length;
    const m = pattern.length;
    
    // We only need to loop up to the point where the pattern can still fit!
    for (let i = 0; i <= n - m; i++) {
        let j = 0;
        
        // Check if the pattern matches starting at text[i]
        while (j < m && text[i + j] === pattern[j]) {
            j++;
        }
        
        // If we matched all 'm' characters, we found it!
        if (j === m) {
            return i; // Return the starting index
        }
    }
    
    return -1;
}
```

### Level 2 — Building Blocks (The KMP LPS Array)
The heart of KMP is the **LPS Array** (Longest Proper Prefix which is also Suffix). 
It pre-analyzes the pattern to answer one question: "If I fail at index `j`, where should I fallback to in the pattern to avoid starting over?"

```javascript
function buildLPS(pattern) {
    const m = pattern.length;
    const lps = new Array(m).fill(0);
    
    let length = 0; // Length of the previous longest prefix suffix
    let i = 1;
    
    while (i < m) {
        if (pattern[i] === pattern[length]) {
            length++;
            lps[i] = length;
            i++;
        } else {
            if (length !== 0) {
                // Fall back! Don't increment 'i' yet, we need to check the previous prefix!
                length = lps[length - 1];
            } else {
                // Absolute failure. No prefix matches.
                lps[i] = 0;
                i++;
            }
        }
    }
    
    return lps;
}
```

### Level 3 — Intermediate (Knuth-Morris-Pratt Algorithm)
Now we combine the LPS array with the search!

```javascript
function kmpSearch(text, pattern) {
    if (pattern.length === 0) return 0;
    
    const lps = buildLPS(pattern);
    
    let i = 0; // index for text
    let j = 0; // index for pattern
    
    while (i < text.length) {
        if (pattern[j] === text[i]) {
            i++;
            j++;
        }
        
        if (j === pattern.length) {
            // Match found! 
            return i - j; // Return starting index
            
            // If we wanted to find ALL occurrences, we would reset j:
            // j = lps[j - 1];
        } 
        else if (i < text.length && pattern[j] !== text[i]) {
            // Mismatch after 'j' matches
            if (j !== 0) {
                // Use the LPS array to avoid backing up 'i' !
                j = lps[j - 1];
            } else {
                i++;
            }
        }
    }
    
    return -1;
}
```

### Level 4 — Advanced (Rabin-Karp Rolling Hash)
While KMP is mathematically elegant, **Rabin-Karp** uses a different approach entirely: **Hashing**.
Instead of comparing strings character by character, it calculates an integer hash of the pattern (e.g., `hash("abc") = 123`). It then slides a window across the text, calculating the hash of every 3-letter block. 
Comparing integers (`123 === 123`) is instant $O(1)$!

*The Magic:* Recalculating a hash from scratch takes $O(M)$. Rabin-Karp uses a **Rolling Hash**. When the window slides from `"abc"` to `"bcd"`, it mathematically subtracts the `'a'` and adds the `'d'` in $O(1)$ time!

```javascript
function rabinKarpSearch(text, pattern) {
    const n = text.length;
    const m = pattern.length;
    if (m === 0) return 0;
    
    const PRIME = 101; // A prime number for our hashing algorithm
    const BASE = 256;  // Total number of characters in the alphabet
    
    let patternHash = 0;
    let windowHash = 0;
    let h = 1;
    
    // The value of h would be Math.pow(BASE, m-1) % PRIME
    for (let i = 0; i < m - 1; i++) {
        h = (h * BASE) % PRIME;
    }
    
    // Calculate the initial hash values for pattern and the first window
    for (let i = 0; i < m; i++) {
        patternHash = (BASE * patternHash + pattern.charCodeAt(i)) % PRIME;
        windowHash = (BASE * windowHash + text.charCodeAt(i)) % PRIME;
    }
    
    // Slide the window over text
    for (let i = 0; i <= n - m; i++) {
        // If the hashes match, we MUST verify character by character!
        // (Because two different strings can rarely produce the same hash: a Hash Collision)
        if (patternHash === windowHash) {
            let match = true;
            for (let j = 0; j < m; j++) {
                if (text[i + j] !== pattern[j]) {
                    match = false;
                    break;
                }
            }
            if (match) return i;
        }
        
        // Calculate the hash for the NEXT window (Rolling Hash!)
        if (i < n - m) {
            windowHash = (BASE * (windowHash - text.charCodeAt(i) * h) + text.charCodeAt(i + m)) % PRIME;
            
            // We might get negative value of windowHash, convert it to positive
            if (windowHash < 0) {
                windowHash = (windowHash + PRIME);
            }
        }
    }
    
    return -1;
}
```

## REAL_WORLD
**Plagiarism Detection Software**
Services like Turnitin detect if a student copied a Wikipedia article. Doing an exact string match (KMP) doesn't work because the student might have changed a few words. 
Plagiarism detectors use a variation of the Rabin-Karp Rolling Hash algorithm. They generate hashes for "n-grams" (chunks of 5 words) across the student's essay and check for mass collisions against a global database of billions of previously hashed documents. The Rolling Hash architecture makes this massive data processing mathematically feasible.

## INTERVIEW
**Q: During an interview, the problem was "Find the first occurrence of a string." I just wrote `text.indexOf(pattern)` and it passed all Leetcode tests. Why would I ever write KMP from scratch?**
A: **You shouldn't, unless specifically asked.** The built-in `.indexOf()` in modern browsers (V8 engine) is implemented in highly optimized C++, often using Boyer-Moore or a hardware-accelerated SIMD algorithm. It is vastly faster than any JavaScript KMP you can write.
