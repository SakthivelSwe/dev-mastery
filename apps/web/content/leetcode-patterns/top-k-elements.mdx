## WHY
If an interviewer asks you to find the "Largest", "Smallest", or "Most Frequent" $K$ elements in an array, your first instinct is to sort the entire array and slice the top $K$ elements. 
While this works, sorting the entire array takes $O(N \log N)$ time. If you have an array of 1,000,000 logs and only want the top 3, sorting the other 999,997 logs is a massive waste of CPU cycles.

The **Top K Elements Pattern** uses a **Heap** (Priority Queue) to elegantly maintain a running list of the top $K$ elements, drastically reducing the time complexity to $O(N \log K)$.

## THEORY
### Min-Heaps and Max-Heaps
A Heap is a specialized Tree structure that always keeps the "most extreme" element at the very top (the root).
- Finding the minimum element in a Min-Heap takes $O(1)$ time!
- Inserting or removing an element takes $O(\log K)$ time (where $K$ is the size of the heap).

*The Golden Rule:*
- If you need the Top K **Largest** elements, use a **Min-Heap** of size $K$. (The smallest of the largest elements sits at the top, ready to be kicked out if a bigger number comes along).
- If you need the Top K **Smallest** elements, use a **Max-Heap**.

## CODE

*(Note: JavaScript does not have a built-in Heap/Priority Queue class. In a real interview, you must quickly mention this and assume the existence of a `MinPriorityQueue` class provided by the platform, or write a quick, simple custom implementation).*

### Level 1 — Beginner (Kth Largest Element)
*Problem: Given an integer array `nums` and an integer `k`, return the $k^{th}$ largest element in the array.*

```javascript
// Assuming a standard MinPriorityQueue exists
function findKthLargest(nums, k) {
    const minHeap = new MinPriorityQueue();
    
    for (let num of nums) {
        // Push the number into the heap
        minHeap.enqueue(num);
        
        // If the heap grows larger than K, pop the SMALLEST element out!
        if (minHeap.size() > k) {
            minHeap.dequeue(); 
        }
    }
    
    // Because the heap size is exactly K, and it's a Min-Heap, 
    // the element sitting at the top is exactly the Kth largest!
    return minHeap.front().element;
}
```

### Level 2 — Building Blocks (Top K Frequent Elements)
*Problem: Given an integer array `nums` and an integer `k`, return the `k` most frequent elements.*
*Trick:* We combine the **Hashmap Frequency** pattern with the **Top K** pattern!

```javascript
function topKFrequent(nums, k) {
    // 1. Build the frequency map
    const map = new Map();
    for (let num of nums) {
        map.set(num, (map.get(num) || 0) + 1);
    }
    
    // 2. Initialize a MinHeap that sorts based on FREQUENCY (the map value)
    // In JS Leetcode, MinPriorityQueue accepts an 'options' object to define priority
    const minHeap = new MinPriorityQueue({ priority: (item) => item.frequency });
    
    // 3. Iterate through the unique numbers
    for (let [num, freq] of map.entries()) {
        minHeap.enqueue({ number: num, frequency: freq });
        
        // Keep the heap strictly at size K
        if (minHeap.size() > k) {
            minHeap.dequeue(); 
        }
    }
    
    // 4. Extract the remaining K numbers
    const result = [];
    while (!minHeap.isEmpty()) {
        result.push(minHeap.dequeue().element.number);
    }
    
    return result;
}
```

### Level 3 — Intermediate (K Closest Points to Origin)
*Problem: Given an array of `points` where `points[i] = [x, y]`, return the `k` closest points to the origin `(0, 0)`.*

To find the SMALLEST distances, we must use a **Max-Heap**. 

```javascript
function kClosest(points, k) {
    // Initialize a MaxHeap that sorts by Euclidean distance formula: (x^2 + y^2)
    const maxHeap = new MaxPriorityQueue({ 
        priority: (point) => point[0] ** 2 + point[1] ** 2 
    });
    
    for (let point of points) {
        maxHeap.enqueue(point);
        
        // If the heap exceeds K, the furthest point (max) sits at the top.
        // Kick it out!
        if (maxHeap.size() > k) {
            maxHeap.dequeue();
        }
    }
    
    const result = [];
    while (!maxHeap.isEmpty()) {
        result.push(maxHeap.dequeue().element);
    }
    
    return result;
}
```

### Level 4 — Advanced (Connect Ropes / Minimum Cost)
*Problem: You have $N$ ropes of different lengths. You need to connect all the ropes into one. The cost to connect two ropes is equal to the sum of their lengths. Find the minimum total cost to connect all ropes.*

*Trick:* To minimize the overall cost, you must always aggressively combine the two *shortest* ropes available at any given time. This requires constantly finding the minimums dynamically as the pool of ropes changes!

```javascript
function minCostToConnectRopes(ropes) {
    // Put ALL ropes into a Min-Heap
    const minHeap = new MinPriorityQueue();
    for (let rope of ropes) {
        minHeap.enqueue(rope);
    }
    
    let totalCost = 0;
    
    // While there is more than 1 rope left...
    while (minHeap.size() > 1) {
        // Grab the two absolute shortest ropes!
        const rope1 = minHeap.dequeue().element;
        const rope2 = minHeap.dequeue().element;
        
        // Combine them
        const currentCost = rope1 + rope2;
        totalCost += currentCost;
        
        // Throw the newly combined (longer) rope back into the heap!
        minHeap.enqueue(currentCost);
    }
    
    return totalCost;
}
```

## REAL_WORLD
**Search Engine Autocomplete**
When you type "How to..." into Google, it instantly shows the top 5 most frequent search queries. Google processes billions of searches. Sorting an array of 5 billion strings takes hours. Maintaining a distributed Heap/Priority Queue structure allows their systems to constantly track and update the "Top K" queries in real-time, providing sub-millisecond autocomplete suggestions to the user.

## INTERVIEW
**Q: Why don't we just sort the array and slice the top $K$ elements? Isn't `Array.sort((a,b) => b-a).slice(0,k)` faster to write?**
A: It is faster to write, and for a small array of 100 items, the performance difference is negligible. But in a systems design or algorithm interview, you are tested on scalability. 
If the array has $N=1,000,000$ items, and $K=10$:
- Sorting the array takes $O(N \log N) \approx 1,000,000 \times 20 = 20,000,000$ operations.
- The Heap Pattern takes $O(N \log K) \approx 1,000,000 \times 3.3 = 3,300,000$ operations.
The Heap approach is nearly an order of magnitude faster and prevents memory spikes from duplicating massive arrays.
