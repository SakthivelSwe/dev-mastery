## WHY
If you are repeatedly asked to calculate the sum of elements in a subarray (e.g., "Find the sum of items from index 2 to 7, then 4 to 9"), computing the sum from scratch every single time requires an $O(N)$ loop for *every* query. If there are $M$ queries, the time complexity becomes $O(N \times M)$, which will trigger a Time Limit Exceeded (TLE) error in coding interviews.

The **Prefix Sum Pattern** allows you to answer these sum queries in $O(1)$ constant time by doing a single $O(N)$ pre-computation pass over the array. 

## THEORY
### The Core Concept
A Prefix Sum array is an array of the exact same size where every element at index `i` is the sum of all elements from the beginning of the original array up to index `i`.

**Original Array:** `[2, 4, 1, 5, 3]`
**Prefix Sum Array:** `[2, 6, 7, 12, 15]`

Notice how `prefix[i] = prefix[i - 1] + original[i]`.

### The Magic Formula
If you want the sum of elements from index `L` (Left) to index `R` (Right), you don't need a loop!
**Sum(L to R) = `PrefixSum[R] - PrefixSum[L - 1]`**
*(If L is 0, the sum is simply `PrefixSum[R]`)*.

Example: Sum from index 1 to 3 in `[2, 4, 1, 5, 3]`.
`PrefixSum[3] - PrefixSum[0]` -> `12 - 2 = 10`. (Which is exactly `4 + 1 + 5`!).

## CODE

### Level 1 — Beginner (Range Sum Queries)
**Problem:** Given an integer array `nums`, handle multiple queries to calculate the sum of the elements between indices `left` and `right` inclusive.

```java
class NumArray {
    private int[] prefix;

    // Pre-compute the prefix sum array in O(N) time
    public NumArray(int[] nums) {
        prefix = new int[nums.length];
        prefix[0] = nums[0];
        
        for (int i = 1; i < nums.length; i++) {
            prefix[i] = prefix[i - 1] + nums[i];
        }
    }
    
    // Answer any range query in O(1) constant time!
    public int sumRange(int left, int right) {
        if (left == 0) return prefix[right];
        return prefix[right] - prefix[left - 1];
    }
}
```

### Level 2 — Building Blocks (Find the Pivot Index)
**Problem:** Given an array of integers, return the pivot index. The pivot index is the index where the sum of all the numbers strictly to the left of the index is equal to the sum of all the numbers strictly to the index's right.

```java
// Time Complexity: O(N) | Space Complexity: O(1)
// Notice we don't even need to build the full prefix array, 
// we just need the total sum and a running left sum!
public int pivotIndex(int[] nums) {
    int totalSum = 0;
    for (int num : nums) totalSum += num;
    
    int leftSum = 0;
    
    for (int i = 0; i < nums.length; i++) {
        // If totalSum - leftSum - currentElement == leftSum, we found it!
        int rightSum = totalSum - leftSum - nums[i];
        if (leftSum == rightSum) {
            return i;
        }
        
        // Add the current element to the left sum for the next iteration
        leftSum += nums[i];
    }
    
    return -1; // No pivot found
}
```

### Level 3 — Intermediate (Subarray Sum Equals K)
This is a legendary LeetCode Medium (LeetCode #560). We want to find the *total number* of continuous subarrays whose sum equals `k`. The numbers can be negative, so a Sliding Window will fail (Sliding Windows only work on positive numbers because adding an element must always increase the sum).

We solve this using **Prefix Sum + HashMap**.

```java
// Time Complexity: O(N) | Space Complexity: O(N)
public int subarraySum(int[] nums, int k) {
    int count = 0;
    int currentPrefixSum = 0;
    
    // Map stores: <PrefixSum, Number of times we've seen this sum>
    Map<Integer, Integer> prefixCounts = new HashMap<>();
    
    // Base case: We've seen the sum '0' exactly one time (before adding any elements)
    prefixCounts.put(0, 1);
    
    for (int num : nums) {
        currentPrefixSum += num;
        
        // If (currentPrefixSum - k) exists in the map, it means there is a valid 
        // subarray ending at this index that sums to k!
        int target = currentPrefixSum - k;
        if (prefixCounts.containsKey(target)) {
            count += prefixCounts.get(target);
        }
        
        // Add the current prefix sum to the map for future iterations
        prefixCounts.put(currentPrefixSum, prefixCounts.getOrDefault(currentPrefixSum, 0) + 1);
    }
    
    return count;
}
```

### Level 4 — Advanced (2D Prefix Sums)
What if you are given a 2D Matrix (a grid) and asked to find the sum of a specific rectangular region?

You can build a 2D Prefix Sum matrix where `Prefix[row][col]` stores the sum of all elements in the rectangle from the top-left corner `(0,0)` to `(row, col)`.
To calculate a specific region from `(row1, col1)` to `(row2, col2)`, you use the Inclusion-Exclusion principle:

`Sum = Prefix[r2][c2] - Prefix[r1-1][c2] - Prefix[r2][c1-1] + Prefix[r1-1][c1-1]`

*(You subtract the rectangle above your target, subtract the rectangle to the left of your target, and add back the top-left overlapping corner that you accidentally subtracted twice!)*

## REAL_WORLD
**Analytics Dashboards**
Imagine building an analytics dashboard that shows total revenue. The user can select any arbitrary date range (e.g., "March 15 to April 10"). If the database has 10 years of daily revenue data, computing the sum on the fly for every user request is expensive. By storing a pre-computed "Running Total" (a Prefix Sum) in the database, the backend simply queries two rows: the Running Total on April 10, minus the Running Total on March 14, delivering the exact revenue in 2 milliseconds regardless of the size of the date range!

## INTERVIEW
**Q: Why doesn't the Sliding Window pattern work if the array contains negative numbers?**
A: A Sliding Window relies on a strict monotonic property: If you expand the window, the sum *always* increases. If you shrink the window, the sum *always* decreases. This allows you to intelligently decide when to move the left or right pointers. If the array has negative numbers, expanding the window might actually *decrease* the sum. The logic breaks entirely, forcing you to use the Prefix Sum + HashMap pattern instead.
