## WHY
If you are iterating through a Linked List or an Array and need to find the exact middle element, how do you do it? 
The brute-force way is to iterate through the entire structure to count the total length, divide that length by two, and then iterate through the structure *again* until you reach that middle index. This takes two full passes ($2N$).

The **Fast & Slow Pointers** pattern (also known as Floyd's Cycle-Finding Algorithm or the Hare & Tortoise algorithm) solves this in a single pass ($O(N)$) using two pointers moving at different speeds.

## THEORY
### The Core Concept
You initialize two pointers at the start of the data structure. 
- **The Slow Pointer (Tortoise):** Moves forward 1 step at a time.
- **The Fast Pointer (Hare):** Moves forward 2 steps at a time.

Because the Fast pointer is moving exactly twice as fast, when the Fast pointer reaches the absolute end of the list, the Slow pointer will be sitting exactly in the middle! 

### When to use it?
1. Finding the middle of a Linked List.
2. Detecting if a Linked List has a cycle (a loop).
3. Finding the start of a cycle.
4. Detecting cycles in arrays where values point to indices.

## CODE

### Level 1 — Beginner (Middle of the Linked List)
**Problem:** Given the head of a singly linked list, return the middle node of the linked list. If there are two middle nodes (even length), return the second middle node.

```java
class ListNode {
    int val;
    ListNode next;
}

// Time: O(N) | Space: O(1)
public ListNode middleNode(ListNode head) {
    ListNode slow = head;
    ListNode fast = head;
    
    // Fast pointer checks 2 steps ahead. 
    // We must ensure fast AND fast.next are not null to avoid NullPointerExceptions!
    while (fast != null && fast.next != null) {
        slow = slow.next;         // Moves 1 step
        fast = fast.next.next;    // Moves 2 steps
    }
    
    // When fast hits the end, slow is exactly in the middle.
    return slow;
}
```

### Level 2 — Building Blocks (Linked List Cycle Detection)
**Problem:** Given head, the head of a linked list, determine if the linked list has a cycle in it. (A cycle occurs if a node's `next` pointer points back to a previous node).

Imagine two runners on a track. If the track is a straight line, the fast runner finishes the race and leaves the slow runner behind. If the track is a circular loop, the fast runner will eventually "lap" the slow runner from behind.

```java
// Time: O(N) | Space: O(1)
public boolean hasCycle(ListNode head) {
    ListNode slow = head;
    ListNode fast = head;
    
    while (fast != null && fast.next != null) {
        slow = slow.next;
        fast = fast.next.next;
        
        // If they ever stand on the exact same node at the same time, 
        // there is a loop!
        if (slow == fast) {
            return true;
        }
    }
    
    // Fast reached a null pointer, so it's a straight line (no loop).
    return false;
}
```

### Level 3 — Intermediate (Start of the Cycle)
**Problem:** Given the head of a linked list, return the node where the cycle begins. If there is no cycle, return null.

This requires a tiny bit of mathematical magic. Floyd's algorithm proves that if the two pointers meet inside a cycle, the distance from the head of the list to the start of the cycle is EXACTLY EQUAL to the distance from the meeting point to the start of the cycle.

```java
public ListNode detectCycle(ListNode head) {
    ListNode slow = head;
    ListNode fast = head;
    
    // Step 1: Find the meeting point (Confirm there is a cycle)
    while (fast != null && fast.next != null) {
        slow = slow.next;
        fast = fast.next.next;
        
        if (slow == fast) {
            // Cycle detected! Now find the exact start node.
            
            // Step 2: Reset ONE pointer back to the very beginning.
            ListNode pointer1 = head;
            ListNode pointer2 = slow; // Currently at the meeting point
            
            // Step 3: Move BOTH pointers at the same speed (1 step at a time).
            // They will collide exactly at the start of the cycle!
            while (pointer1 != pointer2) {
                pointer1 = pointer1.next;
                pointer2 = pointer2.next;
            }
            return pointer1;
        }
    }
    return null;
}
```

### Level 4 — Advanced (Happy Number)
**Problem:** Write an algorithm to determine if a number `n` is happy. A happy number is a number defined by the following process: Replace the number by the sum of the squares of its digits. Repeat the process until the number equals 1, or it loops endlessly in a cycle which does not include 1.

This doesn't look like a Linked List! But it is mathematically identical. Every number "points" to the next calculated number.
`19 -> 82 -> 68 -> 100 -> 1`.
If there is a cycle, the Fast/Slow pointers will catch it!

```java
public boolean isHappy(int n) {
    int slow = n;
    int fast = n;
    
    do {
        // Slow takes 1 step (calculates next number once)
        slow = sumOfSquaredDigits(slow);
        
        // Fast takes 2 steps (calculates next number twice)
        fast = sumOfSquaredDigits(sumOfSquaredDigits(fast));
        
    } while (slow != fast); // Loop until they meet
    
    // If they met at the number 1, it's a happy number!
    // If they met at any other number, they are trapped in a cycle!
    return slow == 1;
}

private int sumOfSquaredDigits(int number) {
    int totalSum = 0;
    while (number > 0) {
        int digit = number % 10;
        totalSum += (digit * digit);
        number /= 10;
    }
    return totalSum;
}
```

## REAL_WORLD
While cycle detection in linked lists feels like a purely academic interview question, the underlying mathematical concept (Cycle Finding) is heavily used in cryptography, specifically in the **Pollard's rho algorithm** for integer factorization. The algorithm uses the exact same pseudo-random sequence generation and a fast/slow pointer comparison to find prime factors of massive numbers!

## INTERVIEW
**Q: Can we move the fast pointer 3 steps instead of 2 steps to make it find the cycle faster?**
A: Mathematically, yes, moving the fast pointer 3 steps (or 4, or 5) will eventually detect a cycle. However, 2 steps is optimal. If the fast pointer moves 3 steps, it might "jump over" the slow pointer in a specific cycle configuration, requiring an entire extra lap around the cycle before they finally collide. A relative speed of 1 (Fast moves 2, Slow moves 1. 2 - 1 = 1) guarantees they will collide perfectly without skipping over each other.
