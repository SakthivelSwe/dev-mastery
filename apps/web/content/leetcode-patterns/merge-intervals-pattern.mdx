## WHY
Imagine you are building a calendar app (like Google Calendar). A user tries to schedule a meeting from 2:00 PM to 4:00 PM. You need to write an algorithm that checks their existing schedule to see if this new meeting overlaps with any others, or if it can be seamlessly merged.

The **Merge Intervals Pattern** is a foundational technique for dealing with overlapping time frames, scheduling problems, and continuous ranges. It is notoriously tricky because intervals can overlap completely, partially, or not at all.

## THEORY
### The Sorting Prerequisite
The secret to solving almost *any* interval problem is **Sorting**. 
If you have random intervals like `[10, 15], [1, 3], [8, 12]`, detecting overlaps requires an agonizing $O(N^2)$ nested loop to compare every interval against every other interval.

If you **sort the intervals by their starting times** first (`[1, 3], [8, 12], [10, 15]`), you guarantee that any overlapping intervals will be sitting directly next to each other in the array! This allows you to solve the problem in a single $O(N)$ pass.

## CODE

### Level 1 — Beginner (Checking for Overlap)
How do we mathematically know if two intervals overlap?

```javascript
// Interval A: [aStart, aEnd]
// Interval B: [bStart, bEnd]
// Assuming Interval A starts before Interval B (aStart <= bStart)

function doTheyOverlap(aStart, aEnd, bStart, bEnd) {
    // If the first interval ENDS after (or exactly when) the second interval STARTS,
    // they are mathematically colliding!
    return aEnd >= bStart; 
}
```

### Level 2 — Building Blocks (Merge Intervals)
*Problem: Given an array of intervals, merge all overlapping intervals, and return an array of the non-overlapping intervals.*
*(Example: `[[1,3], [2,6], [8,10]]` becomes `[[1,6], [8,10]]`)*

```javascript
function merge(intervals) {
    if (intervals.length <= 1) return intervals;
    
    // 1. Sort the intervals strictly by their STARTING time
    intervals.sort((a, b) => a[0] - b[0]);
    
    const merged = [intervals[0]];
    
    for (let i = 1; i < intervals.length; i++) {
        const currentInterval = intervals[i];
        const lastMergedInterval = merged[merged.length - 1];
        
        // 2. Do they overlap? (Does the last one end AFTER the current one starts?)
        if (lastMergedInterval[1] >= currentInterval[0]) {
            // YES! Merge them! 
            // The new END time is the maximum of BOTH end times
            lastMergedInterval[1] = Math.max(lastMergedInterval[1], currentInterval[1]);
        } else {
            // NO! No overlap. Safely push the current interval as a distinct block.
            merged.push(currentInterval);
        }
    }
    
    return merged;
}
```

### Level 3 — Intermediate (Insert Interval)
*Problem: Given a sorted array of non-overlapping intervals, insert a new interval into the array, merging it if necessary.*
*(Example: Insert `[4,8]` into `[[1,2], [3,5], [6,7], [8,10], [12,16]]`)*

While you *could* just push the new interval and run the Level 2 sort algorithm, that takes $O(N \log N)$ time. Because the array is already sorted, we can do this in $O(N)$ time!

```javascript
function insert(intervals, newInterval) {
    const result = [];
    let i = 0;
    
    // 1. Skip all intervals that end BEFORE the new interval starts. (No overlap)
    while (i < intervals.length && intervals[i][1] < newInterval[0]) {
        result.push(intervals[i]);
        i++;
    }
    
    // 2. Merge all overlapping intervals into the 'newInterval'
    while (i < intervals.length && intervals[i][0] <= newInterval[1]) {
        newInterval[0] = Math.min(newInterval[0], intervals[i][0]); // Min Start
        newInterval[1] = Math.max(newInterval[1], intervals[i][1]); // Max End
        i++;
    }
    // Push the massive, merged frankenstein interval
    result.push(newInterval);
    
    // 3. Push all remaining intervals that start AFTER the new interval ends.
    while (i < intervals.length) {
        result.push(intervals[i]);
        i++;
    }
    
    return result;
}
```

### Level 4 — Advanced (Meeting Rooms II)
*Problem: Given an array of meeting time intervals, find the minimum number of conference rooms required to host all the meetings.*
*(Example: `[[0, 30], [5, 10], [15, 20]]` requires 2 rooms, because [5,10] overlaps with [0,30]).*

*Trick:* We separate the start times and end times completely. We sort them independently, and use two pointers to simulate the flow of time!

```javascript
function minMeetingRooms(intervals) {
    if (!intervals || intervals.length === 0) return 0;
    
    const starts = [];
    const ends = [];
    
    // Separate the times
    for (let i = 0; i < intervals.length; i++) {
        starts.push(intervals[i][0]);
        ends.push(intervals[i][1]);
    }
    
    // Sort them independently!
    starts.sort((a, b) => a - b);
    ends.sort((a, b) => a - b);
    
    let rooms = 0;
    let endPointer = 0;
    
    // Simulate time moving forward by looking at start times
    for (let startPointer = 0; startPointer < starts.length; startPointer++) {
        
        // Is the current meeting starting BEFORE the earliest meeting ends?
        if (starts[startPointer] < ends[endPointer]) {
            // Overlap! We must allocate a brand new room.
            rooms++;
        } else {
            // A previous meeting ended! We can reuse that room.
            // Move the end pointer to the next ending time.
            endPointer++;
        }
    }
    
    return rooms;
}
```

## REAL_WORLD
**Garbage Collection and Memory Fragmentation**
When an Operating System or a language like C++ allocates RAM, it keeps track of free memory blocks as "intervals" (e.g., Memory bytes 1000 to 2000 are free). 
When a program deletes variables, new free intervals are created. Over time, memory becomes "fragmented" (hundreds of tiny little free intervals). The Memory Allocator periodically runs a strict **Merge Intervals algorithm** to combine adjacent free blocks `[1000, 2000]` and `[2000, 3500]` into a single massive block `[1000, 3500]` so large arrays can be safely allocated!

## INTERVIEW
**Q: What is the time complexity of the standard Merge Intervals algorithm?**
A: **$O(N \log N)$**. The actual merging loop runs in $O(N)$ linear time because we only iterate through the array exactly once. However, because we MUST sort the array by starting times before we can do the linear scan, the sorting step dominates the time complexity. The space complexity is usually $O(N)$ to store the newly merged output array (or $O(\log N)$ for the in-place sorting algorithm overhead).
