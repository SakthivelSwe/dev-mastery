## WHY
Most developers know how to use RxJS `subscribe()` to fetch data from `HttpClient`. But what happens when the user clicks a "Search" button 5 times in 1 second? If you just subscribe 5 times, you send 5 concurrent HTTP requests to the backend, causing massive server load and race conditions (Request #4 might resolve *after* Request #5, showing the wrong data!).
**Advanced RxJS Operators** are the secret weapon of Senior Angular Developers. They allow you to choreograph complex, multi-step asynchronous workflows (like typeaheads, polling, and retry logic) declaratively, without writing a single nested `if/else` statement.

## THEORY
### Higher-Order Mapping Operators
The most confusing part of RxJS is flattening. If you have an Observable of a String (from a search input), and you want to use that String to make an HTTP request (which returns an Observable of an Array), you end up with an `Observable<Observable<Array>>`. 
You must "flatten" the inner Observable. The four critical flattening operators dictate *how* concurrent requests are handled:
1. **`switchMap`:** Cancels the previous request if a new one arrives.
2. **`concatMap`:** Queues requests in order. Waits for the first to finish before starting the second.
3. **`mergeMap`:** Runs all requests concurrently (simultaneously) without caring about order.
4. **`exhaustMap`:** Ignores all new requests while the current one is still running.

## CODE

### Level 1 — The Perfect Typeahead Search (`switchMap`)
A user is typing into a search bar: "A", then "Ap", then "App", then "Apple". We want to fetch results from the API, but we don't want to make an HTTP request for every single keystroke.

```typescript
import { Component, OnInit } from '@angular/core';
import { FormControl } from '@angular/forms';
import { HttpClient } from '@angular/common/http';
import { debounceTime, distinctUntilChanged, switchMap, catchError } from 'rxjs/operators';
import { of, Observable } from 'rxjs';

export class SearchComponent implements OnInit {
  searchInput = new FormControl('');
  results$!: Observable<any[]>;

  constructor(private http: HttpClient) {}

  ngOnInit() {
    // valueChanges is an Observable of the user's keystrokes!
    this.results$ = this.searchInput.valueChanges.pipe(
      
      // 1. Wait 300ms after they STOP typing before continuing.
      debounceTime(300),
      
      // 2. If they type 'a', backspace, and type 'a' again within 300ms, ignore it.
      distinctUntilChanged(),
      
      // 3. MAGIC! Execute the HTTP request. 
      // If they type 'Apple', the request fires. If they immediately type 's' ('Apples'),
      // switchMap will CANCEL the pending 'Apple' HTTP request, and start the 'Apples' request!
      switchMap(searchTerm => this.http.get<any[]>(`/api/search?q=${searchTerm}`).pipe(
        // 4. CRITICAL: Catch errors on the INNER observable so the outer stream doesn't die!
        catchError(() => of([]))
      ))
    );
  }
}
```

### Level 2 — Safe Database Saves (`concatMap`)
If a user is editing a document and it auto-saves every few seconds, you cannot use `switchMap`. If you cancel a "Save" request halfway through, the database might become corrupted!
You must use `concatMap` to queue the saves strictly in order.

```typescript
import { Subject } from 'rxjs';
import { concatMap } from 'rxjs/operators';

export class DocumentEditor {
  private saveQueue$ = new Subject<DocumentData>();

  constructor(private http: HttpClient) {
    
    this.saveQueue$.pipe(
      // If 5 saves are requested simultaneously, concatMap processes them one by one.
      // Save 2 will not start the HTTP request until Save 1 completely finishes.
      concatMap(data => this.http.post('/api/save', data))
    ).subscribe();
    
  }

  triggerAutoSave(data: DocumentData) {
    this.saveQueue$.next(data); // Push into the queue!
  }
}
```

### Level 3 — Preventing Double-Submissions (`exhaustMap`)
A user is impatient and clicks the "Login" button 5 times rapidly. We only want to process the first click and completely ignore the rest until the login finishes.

```typescript
import { Subject } from 'rxjs';
import { exhaustMap } from 'rxjs/operators';

export class LoginComponent {
  private loginClicks$ = new Subject<Credentials>();

  constructor(private authService: AuthService) {
    
    this.loginClicks$.pipe(
      // exhaustMap executes the first request.
      // While the HTTP request is pending, it aggressively IGNORES all subsequent clicks!
      exhaustMap(creds => this.authService.login(creds))
    ).subscribe(result => console.log('Login successful!'));
    
  }

  onLoginButtonClicked(creds: Credentials) {
    this.loginClicks$.next(creds);
  }
}
```

### Level 4 — Retry Logic with Exponential Backoff
If the user's mobile connection drops for 2 seconds, the HTTP request shouldn't instantly fail and show an error screen. We should silently retry 3 times, waiting longer between each attempt.

```typescript
import { retry, timer, throwError } from 'rxjs';

export class WeatherService {
  
  getWeather() {
    return this.http.get('/api/weather').pipe(
      // Modern RxJS 7+ retry operator!
      retry({
        count: 3, // Try a maximum of 3 times
        delay: (error, retryCount) => {
          // If it's a 404 Not Found, don't retry! It will never work.
          if (error.status === 404) return throwError(() => error);
          
          console.log(`Attempt ${retryCount} failed. Retrying...`);
          // Exponential backoff: Wait 1s, then 2s, then 3s!
          return timer(retryCount * 1000); 
        }
      })
    );
  }
}
```

## REAL_WORLD
**Memory Leaks and `takeUntil`**
The number one cause of memory leaks in Angular is subscribing to an infinite Observable (like `Router.events` or a WebSocket) inside a component, and forgetting to unsubscribe when the component is destroyed.
If the user navigates back and forth to the component 50 times, 50 invisible subscriptions are created in memory, crashing the browser.
*The Classic Fix:* Use `takeUntil` and `ngOnDestroy`.
```typescript
private destroy$ = new Subject<void>();

ngOnInit() {
  this.infiniteStream$.pipe(
    takeUntil(this.destroy$) // Kills the stream when destroy$ emits!
  ).subscribe();
}

ngOnDestroy() {
  this.destroy$.next(); // Emit!
  this.destroy$.complete();
}
```
*The Modern Fix (Angular 16+):* Use `takeUntilDestroyed()`. It automatically ties the stream to the Component's lifecycle, requiring zero `ngOnDestroy` boilerplate!
```typescript
import { takeUntilDestroyed } from '@angular/core/rxjs-interop';

constructor() {
  this.infiniteStream$.pipe(takeUntilDestroyed()).subscribe();
}
```

## INTERVIEW
**Q: You have two Observables: `user$` (which fetches user details from the API) and `posts$` (which fetches the user's posts from a different API). You need to combine them so the template can render `<h1>{{ user.name }} - {{ posts.length }} Posts</h1>`. Both HTTP requests should execute simultaneously to save time. Which RxJS combination operator should you use: `combineLatest`, `zip`, or `forkJoin`?**
A: **`forkJoin`!**
- `combineLatest`: Emits whenever *any* of the streams emit. Excellent for UI filters (e.g., search text + category dropdown), but unnecessary for single HTTP requests.
- `zip`: Waits for both streams to emit index 1, then emits. Waits for both to emit index 2, then emits. Usually used for strict pairing of infinite streams.
- `forkJoin`: This is the RxJS equivalent of `Promise.all()`. It fires both HTTP requests simultaneously. It waits for BOTH observables to fully complete, and then emits exactly once with the final array of results `[userData, postsData]`. Because HTTP requests are finite observables that complete immediately after returning data, `forkJoin` is mathematically the perfect, most performant operator for combining multiple API calls!

## FEYNMAN CHECK

### Explain Advanced RxJS Like I'm 10 Years Old
> Once you know the basic operators, the advanced patterns are about: (1) SHARING — shareReplay(1) multicasts an Observable to many subscribers while caching the last value (use for HTTP calls shared across components). (2) ERROR HANDLING — catchError returns a fallback, retryWhen implements custom retry logic with exponential backoff. (3) CUSTOM OPERATORS — pipe() lets you compose operators into a named, reusable function. (4) SCHEDULERS — control the timing/execution context of operators (observeOn(asyncScheduler) defers to microtask queue). (5) MARBLE TESTING — TestScheduler lets you test time-based operators without actually waiting. The non-obvious depth: RxJS operators are CURRIED HIGHER-ORDER FUNCTIONS that return operator functions — you can compose them programmatically, creating custom operators with the same interface as built-ins.

---

### 5 Deep Conceptual Questions

**Q1: When and why does shareReplay matter?**
> **A:** Without shareReplay, each subscriber to an HTTP Observable triggers a NEW network request. shareReplay(1) multicasts the single request to all subscribers and replays the last value to late subscribers. This is the standard pattern for cached HTTP calls shared across multiple components: const users$ = this.http.get('/api/users').pipe(shareReplay(1)). With { refCount: false }, the subscription persists even when all consumers unsubscribe (permanent cache). With { refCount: true }, it re-subscribes when the first new subscriber arrives after all previous ones leave (re-fetches).

**Q2: Mental model for custom operators?**
> **A:** A custom operator is a function that takes an Observable and returns an Observable — exactly what pipe() chains together. Pattern: export function retryWithBackoff(maxRetries: number, delay: number) { return (source: Observable<T>) => source.pipe(retryWhen(errors => errors.pipe(scan((count, err) => ...), delay(...)))) }. Use it like any built-in: http.get().pipe(retryWithBackoff(3, 500)). This is how libraries like ng-query, angular-query, and ngrx/effects compose reusable stream logic.

**Q3: Most dangerous misconception about advanced RxJS?**
> **A:** shareReplay always prevents duplicate HTTP calls:
> ```typescript
> // WRONG: shareReplay(1) with refCount:true can re-fetch
> const users$ = this.http.get('/api/users').pipe(shareReplay(1));
> // If all components unsubscribe (route change) then re-subscribe,
> // the Observable re-runs — making a new HTTP request!
>
> // CORRECT for permanent cache:
> const users$ = this.http.get('/api/users').pipe(shareReplay({ bufferSize: 1, refCount: false }));
>
> // CORRECT for cache that refreshes when all leave:
> const users$ = this.http.get('/api/users').pipe(shareReplay({ bufferSize: 1, refCount: true }));
> ```

**Q4: How do RxJS schedulers affect Angular change detection?**
> **A:** By default, Observables emit synchronously or in macrotask/microtask context. Zone.js patches these contexts to notify Angular's change detection. With observeOn(asyncScheduler) or subscribeOn(asapScheduler), you can control WHICH async context values emit in. In performance-critical scenarios (large data processing), running work on queueScheduler (synchronous batch) or observeOn(animationFrameScheduler) for smooth rendering can prevent Zone.js from triggering CD on every intermediate value — batch on a scheduler that fires once per frame.

**Q5: FAANG-grade definition of advanced RxJS patterns.**
> **A:** "Advanced RxJS encompasses multicasting (shareReplay with refCount for HTTP caching, multicast with Subjects), custom operators (factory functions returning (src: Observable) => Observable), error handling (catchError for recovery, retryWhen with exponential backoff, throwError for propagation), marble testing (TestScheduler for deterministic time-based operator testing), scheduler control (asyncScheduler, animationFrameScheduler, queueScheduler for execution context), and operator composition (pipe with custom operator factories) — all building on the lazy, composable, cancellable Observable primitive."

#### Step 2 — retryWithBackoff Operator
```typescript
// operators.ts
import { Observable, throwError, timer } from 'rxjs';
import { retryWhen, scan, mergeMap, debounceTime, distinctUntilChanged, switchMap, takeWhile } from 'rxjs/operators';

export function retryWithBackoff<T>(maxRetries: number, baseDelayMs: number) {
  return (source: Observable<T>): Observable<T> =>
    source.pipe(
      retryWhen(errors =>
        errors.pipe(
          scan((retryCount, error) => {
            if (retryCount >= maxRetries) throw error;  // exhaust retries
            return retryCount + 1;
          }, 0),
          mergeMap(retryCount => timer(baseDelayMs * Math.pow(2, retryCount - 1))),
        )
      ),
    );
}
```

#### Step 3 — debounceDistinct Operator
```typescript
export function debounceDistinct<T>(dueTime: number) {
  return (source: Observable<T>): Observable<T> =>
    source.pipe(
      debounceTime(dueTime),
      distinctUntilChanged(),
    );
}
```

#### Step 4 — pollUntil Operator
```typescript
import { interval } from 'rxjs';

export function pollUntil<T>(
  pollFn: () => Observable<T>,
  conditionFn: (val: T) => boolean,
  intervalMs: number,
) {
  return interval(intervalMs).pipe(
    switchMap(() => pollFn()),
    takeWhile(val => !conditionFn(val), true),  // true = emit the final value
  );
}

// Usage: poll job status until complete
const jobStatus$ = pollUntil(
  () => http.get('/api/job/123'),
  (job: any) => job.status === 'complete',
  2000,
);
```
