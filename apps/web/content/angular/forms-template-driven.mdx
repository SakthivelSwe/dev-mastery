## WHY
Handling user input in vanilla HTML/JS is brutal. You have to write `document.getElementById('email').value`, manually attach `onKeyUp` listeners to 15 different inputs, manually write regex to validate the email format, manually check if the password is 8 characters long, and manually toggle error messages.

Angular provides two distinct architectures for handling Forms: **Template-Driven Forms** and **Reactive Forms**. 

Template-Driven Forms are the "quick and easy" way to build forms. As the name implies, 90% of the logic lives inside the HTML Template. Angular uses heavily abstracted directives (like `ngModel`) to magically track input changes, validate data, and bind it to your TypeScript class. They are perfect for simple login screens and basic user profiles.

## THEORY
### Two-Way Data Binding (`[(ngModel)]`)
The heart of Template-Driven Forms is the "Banana-in-a-Box" syntax: `[( )]`.
- The square brackets `[ ]` push data from the TypeScript Class *down* to the HTML.
- The parentheses `( )` pull DOM Events *up* from the HTML back to the Class.

When combined into `[(ngModel)]`, Angular creates a synchronized, real-time connection. If the user types in the input box, the TypeScript variable instantly updates. If the TypeScript variable changes programmatically, the input box text instantly updates.

## CODE

### Level 1 — Beginner (Setting up ngModel)
To use `ngModel`, you **MUST** import `FormsModule`. If you forget this, the Angular Compiler will crash violently.

```typescript
// For Standalone Components
import { Component } from '@angular/core';
import { FormsModule } from '@angular/forms'; // MANDATORY

@Component({
  standalone: true,
  imports: [FormsModule],
  template: `
    <!-- The banana-in-a-box syntax! -->
    <input type="text" [(ngModel)]="username">
    
    <!-- This heading will update in real-time as you type! -->
    <h1>Welcome, {{ username }}</h1>
  `
})
export class LoginComponent {
  // This variable is bound directly to the input box!
  username: string = 'Guest'; 
}
```

### Level 2 — Building Blocks (The form Element)
When you place `<form>` in your HTML, Angular secretly intercepts it and attaches an invisible directive called `NgForm`. You must give your inputs a `name` attribute so `NgForm` can track them.

```html
<!-- 1. We grab a reference to the hidden NgForm object using #loginForm="ngForm" -->
<!-- 2. We bind the standard (ngSubmit) event to our TypeScript method -->
<form #loginForm="ngForm" (ngSubmit)="onSubmit(loginForm.value)">
  
  <div>
    <label>Email:</label>
    <!-- The 'name' attribute is STRICTLY REQUIRED for Template-Driven Forms! -->
    <input type="email" name="userEmail" [(ngModel)]="email">
  </div>

  <div>
    <label>Password:</label>
    <input type="password" name="userPass" [(ngModel)]="password">
  </div>

  <button type="submit">Login</button>
  
</form>
```

```typescript
export class LoginComponent {
  email = '';
  password = '';

  onSubmit(formData: any) {
    // When submitted, ngForm bundles all the named inputs into a beautiful JSON object!
    console.log('Sending to API:', formData); 
    // Output: { userEmail: "test@test.com", userPass: "12345" }
  }
}
```

### Level 3 — Intermediate (Validation)
You don't need to write regex or custom JS to validate Template-Driven forms. You just use standard HTML5 validation attributes (`required`, `minlength`, `email`). Angular reads these attributes and automatically updates the form's state!

```html
<form #registerForm="ngForm" (ngSubmit)="submit()">
  
  <!-- 1. We attach standard HTML validators (required, minlength) -->
  <!-- 2. We expose this specific input's Angular State object using #nameInput="ngModel" -->
  <input 
     type="text" 
     name="username" 
     [(ngModel)]="user" 
     required 
     minlength="4"
     #nameInput="ngModel">

  <!-- 3. If the input is invalid AND the user has touched/typed in it, show the error! -->
  <div *ngIf="nameInput.invalid && (nameInput.dirty || nameInput.touched)" style="color: red;">
    
    <span *ngIf="nameInput.errors?.['required']">Name is strictly required!</span>
    <span *ngIf="nameInput.errors?.['minlength']">Name must be at least 4 characters.</span>
    
  </div>

  <!-- 4. Disable the submit button until the ENTIRE form is perfectly valid! -->
  <button type="submit" [disabled]="registerForm.invalid">Register</button>

</form>
```

### Level 4 — Advanced (CSS Validation Classes)
Angular doesn't just manage the state in memory; it physically injects CSS classes into the DOM elements in real-time as the user types.

If an input is valid, Angular adds the `.ng-valid` class. If it's invalid, `.ng-invalid`.
If a user hasn't clicked it, `.ng-untouched`. If they have, `.ng-touched`.

You can write incredibly powerful global CSS based on these classes without writing a single line of TypeScript:

```css
/* styles.css */

/* If the input is invalid, AND the user has interacted with it, make the border red! */
input.ng-invalid.ng-touched {
  border: 2px solid red;
  background-color: #ffeeee;
}

/* If the input is perfectly valid, make it green! */
input.ng-valid.ng-touched {
  border: 2px solid green;
}
```

## REAL_WORLD
Template-Driven forms are excellent for simple static pages, but they are generally frowned upon in massive Enterprise applications for three reasons:
1. **Testing is a Nightmare**: Because the validation logic (`required`, `minlength`) is physically written inside the HTML template, you cannot unit test the logic without starting up a fake browser (TestBed) and simulating fake mouse clicks.
2. **Dynamic Forms**: If your server sends you a JSON object that dictates what the form should look like (e.g., "Render 5 text boxes and 2 dropdowns dynamically"), Template-Driven forms cannot handle this effectively.
3. **Reactivity**: Listening to value changes requires dirty DOM bindings instead of clean RxJS observable streams.

Because of these limitations, Enterprise Angular developers almost exclusively use **Reactive Forms** (which we cover in the next topic) for anything more complex than a basic login screen.

## INTERVIEW
**Q: What is the difference between `ngModel` and `[(ngModel)]`?**
A: `ngModel` (no brackets) just registers the input with the `NgForm` tracker. `[ngModel]` is one-way binding (Class -> HTML). `[(ngModel)]` is two-way binding (Class <-> HTML). 

**Q: What happens if you forget to add the `name` attribute to an input using `[(ngModel)]` inside a `<form>`?**
A: Angular will violently crash in the browser console. The `NgForm` directive relies entirely on the `name` attribute to generate the keys for its internal JSON state object.

**Q: What are `touched` and `dirty` states?**
A: An input is `touched` the moment the user clicks inside it and then clicks away (blur event), regardless of whether they typed anything. An input is `dirty` the moment the user physically changes the value (types a key). These are crucial because you never want to show a screaming red "THIS FIELD IS REQUIRED" error message before the user has even had a chance to click on the input box.

## FEYNMAN CHECK

### Explain Angular Template-Driven Forms Like I'm 10 Years Old
> Template-driven forms build the form IN THE HTML using NgModel and validation attributes — Angular reads the template and creates a FormGroup/FormControl model automatically behind the scenes. You don't explicitly create the model in TypeScript; the template IS the source of truth. They are simpler for small forms (login, search, contact) but harder to test and extend for complex forms. ngModel binds a template variable to a component property and tracks validation state — adding CSS classes like ng-valid, ng-invalid, ng-touched automatically. The non-obvious depth: template-driven forms use DIRECTIVES to build the reactive model under the hood — NgModel is actually creating FormControl instances. The difference from reactive forms is WHERE the model is defined (template vs TypeScript), not how it works internally.

### 5 Deep Conceptual Questions

**Q1: How does NgModel create the form model?**
> **A:** When Angular processes a template with NgModel on an input, the NgModel directive instantiates a FormControl internally and registers it with the nearest NgForm ancestor. NgForm collects all NgModel controls into a FormGroup. You can access this model via a template reference variable: #form="ngForm". The key insight: there IS a reactive forms model under the hood — template-driven forms just build it declaratively from the template instead of imperatively in TypeScript.

**Q2: Mental model for ngModel with and without two-way binding?**
> **A:** [(ngModel)]="prop" is the full two-way binding — reads the value from prop AND writes back when the user types. [ngModel]="prop" is one-way binding (component to template only). (ngModelChange)="handler($event)" is the event-only version. You might use [ngModel] with (ngModelChange) separately when you need to process/validate the change before applying it — e.g., debouncing or sanitising input.

**Q3: Most dangerous misconception about template-driven forms?**
> **A:** Template-driven forms are always easier than reactive forms:
> ```typescript
> // For dynamic forms or complex cross-field validation, template-driven gets messy
>
> // TEMPLATE-DRIVEN (hard to do):
> // How do you add/remove fields dynamically?
> // How do you access the FormControl to set errors programmatically?
> // How do you unit-test validation without a rendered DOM?
>
> // REACTIVE (clean for these):
> this.form.setErrors({ serverError: 'Email already exists' });
> this.phones.push(this.createPhone());
> // Testable: validators are plain functions, no DOM needed
>
> // RULE: Simple forms (login, search) -> template-driven
> //       Complex/dynamic/heavily-validated -> reactive
> ```

**Q4: How do you add custom validation to template-driven forms?**
> **A:** Create a directive that implements Validator interface and provides itself as NG_VALIDATORS: @Directive({ selector: '[appMinAge]', providers: [{ provide: NG_VALIDATORS, useExisting: MinAgeDirective, multi: true }] }). Implement validate(control: AbstractControl). In the template: <input appMinAge="18" ngModel name="age">. The validator is called by Angular's forms system the same way built-in validators are. This is more complex than reactive form validators (plain functions) — one more reason complex validation belongs in reactive forms.

**Q5: FAANG-grade definition of template-driven forms.**
> **A:** "Angular template-driven forms use NgModel, NgForm, and NgModelGroup directives to implicitly construct a reactive forms model (FormControl/FormGroup) from the template HTML — with NgModel implementing ControlValueAccessor to bridge DOM values and the Angular forms model — validation through HTML5 attributes (required, minlength, pattern) or custom validator directives implementing Validator — and form state reflected via CSS class names (ng-valid, ng-invalid, ng-touched, ng-dirty) — appropriate for simple forms where template definition is sufficient and programmatic control is not required."

#### Step 2 — Login Form
```typescript
// login.component.ts
import { Component } from '@angular/core';
import { FormsModule } from '@angular/forms';
import { NgIf } from '@angular/common';

@Component({
  standalone: true,
  selector: 'app-login',
  imports: [FormsModule, NgIf],
  template: `
    <form #loginForm="ngForm" (ngSubmit)="onSubmit(loginForm)">
      <div>
        <label>Email</label>
        <input type="email" name="email" [(ngModel)]="credentials.email"
               required email #emailField="ngModel">
        <span *ngIf="emailField.touched && emailField.errors?.['required']">Required</span>
        <span *ngIf="emailField.touched && emailField.errors?.['email']">Invalid email</span>
      </div>
      <div>
        <label>Password</label>
        <input type="password" name="password" [(ngModel)]="credentials.password"
               required minlength="8" #passField="ngModel">
        <span *ngIf="passField.touched && passField.errors?.['required']">Required</span>
        <span *ngIf="passField.touched && passField.errors?.['minlength']">Min 8 characters</span>
      </div>
      <button type="submit" [disabled]="loginForm.invalid">Login</button>
    </form>
    <p *ngIf="submitted">Submitted: {{ credentials | json }}</p>
  `,
})
export class LoginComponent {
  credentials = { email: '', password: '' };
  submitted = false;

  onSubmit(form: any) {
    if (form.valid) {
      this.submitted = true;
      console.log('Login:', this.credentials);
    }
  }
}
```

#### Step 3 — Contact Form With NgModelGroup
```typescript
// contact.component.ts
import { Component } from '@angular/core';
import { FormsModule, NgForm } from '@angular/forms';
import { NgIf } from '@angular/common';

@Component({
  standalone: true,
  selector: 'app-contact',
  imports: [FormsModule, NgIf],
  template: `
    <form #contactForm="ngForm" (ngSubmit)="onSubmit(contactForm)">
      <div ngModelGroup="sender">
        <input name="name" [(ngModel)]="contact.sender.name" required placeholder="Name" #nameField="ngModel">
        <span *ngIf="nameField.invalid && nameField.touched">Name required</span>
        <input name="email" [(ngModel)]="contact.sender.email" required email placeholder="Email">
      </div>
      <textarea name="message" [(ngModel)]="contact.message" required minlength="10" placeholder="Message"></textarea>
      <button type="submit" [disabled]="contactForm.invalid">Send</button>
    </form>
    <pre *ngIf="result">{{ result | json }}</pre>
  `,
})
export class ContactComponent {
  contact = { sender: { name: '', email: '' }, message: '' };
  result: any = null;

  onSubmit(form: NgForm) {
    if (form.valid) {
      this.result = form.value;  // form.value reflects ngModelGroup structure
      form.resetForm();
    }
  }
}
```

#### Step 4 — Custom Validator Directive
```typescript
// min-age.directive.ts
import { Directive, Input } from '@angular/core';
import { AbstractControl, NG_VALIDATORS, ValidationErrors, Validator } from '@angular/forms';

@Directive({
  standalone: true,
  selector: '[appMinAge]',
  providers: [{ provide: NG_VALIDATORS, useExisting: MinAgeDirective, multi: true }],
})
export class MinAgeDirective implements Validator {
  @Input('appMinAge') minAge = 18;

  validate(control: AbstractControl): ValidationErrors | null {
    const age = parseInt(control.value, 10);
    return isNaN(age) || age < this.minAge ? { minAge: { required: this.minAge, actual: age } } : null;
  }
}

// Usage: <input type="number" name="age" [(ngModel)]="age" appMinAge="18">
```
