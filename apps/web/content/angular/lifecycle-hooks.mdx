## WHY
When a Component is rendered on the screen, it doesn't just magically appear. It goes through a highly structured, sequential Lifecycle. Angular creates the class, initializes the inputs, renders the HTML, monitors for changes, and eventually destroys the component when the user navigates away.

If you want to fetch data from an API, you cannot do it in the standard TypeScript `constructor()`, because at that exact millisecond, the Component's `@Input()` properties haven't been bound yet! 

Angular provides **Lifecycle Hooks**: special methods you can implement in your class that Angular will automatically call at specific moments during the component's life, allowing you to execute logic safely.

## THEORY
### The Core Lifecycle Sequence
While there are 8 lifecycle hooks, 95% of Enterprise development relies on these three:
1. **`ngOnChanges()`**: Fires immediately when an `@Input()` property receives new data from a parent component.
2. **`ngOnInit()`**: Fires exactly once after the component is initialized and all `@Input()` properties are locked in. This is where you fetch API data.
3. **`ngOnDestroy()`**: Fires exactly once right before Angular wipes the component from memory. This is critical for preventing Memory Leaks by unsubscribing from Observables.

## CODE

### Level 1 — Beginner (ngOnInit)
To use a lifecycle hook, you should formally implement the Interface (e.g., `OnInit`). This isn't strictly required by the JavaScript engine, but it forces the TypeScript compiler to alert you if you misspell the method name!

```typescript
import { Component, OnInit, inject } from '@angular/core';
import { HttpClient } from '@angular/common/http';

@Component({
  selector: 'app-dashboard',
  template: `<h1>Dashboard data loaded!</h1>`
})
export class DashboardComponent implements OnInit {
  private http = inject(HttpClient);

  // The Constructor is for pure dependency injection ONLY.
  // Never put heavy logic or HTTP calls here!
  constructor() {
    console.log('1. Constructor fired!');
  }

  // ngOnInit is the Angular standard for initialization logic.
  ngOnInit() {
    console.log('2. Component initialized! Fetching data...');
    this.http.get('/api/dashboard').subscribe();
  }
}
```

### Level 2 — Building Blocks (ngOnDestroy & Memory Leaks)
If you subscribe to an infinite RxJS stream (like a WebSocket or Router events), and the user navigates to a new page, the Component is destroyed. BUT, the RxJS Subscription remains active in memory! If the user navigates back and forth 10 times, you will have 10 duplicate subscriptions running simultaneously, crashing the browser.

You MUST use `ngOnDestroy` to clean up.

```typescript
import { Component, OnInit, OnDestroy } from '@angular/core';
import { Subscription, interval } from 'rxjs';

export class TimerComponent implements OnInit, OnDestroy {
  
  // 1. We must hold a reference to the Subscription!
  private timerSub!: Subscription;

  ngOnInit() {
    // interval(1000) emits a number every 1 second, forever!
    this.timerSub = interval(1000).subscribe(tick => {
      console.log('Tick:', tick);
    });
  }

  // 2. This fires milliseconds before the Component is deleted!
  ngOnDestroy() {
    console.log('Component dying! Killing the timer to save memory.');
    // 3. We manually severe the connection!
    this.timerSub.unsubscribe();
  }
}
```

### Level 3 — Intermediate (ngOnChanges)
If your Component relies on data passed from a Parent via `@Input()`, you might need to react every time the Parent changes that data. 

```typescript
import { Component, Input, OnChanges, SimpleChanges } from '@angular/core';

@Component({
  selector: 'app-user-card',
  template: `<h3>{{ userId }}</h3>`
})
export class UserCardComponent implements OnChanges {
  
  @Input() userId!: number;

  // This fires BEFORE ngOnInit, and fires AGAIN every single time 
  // the Parent passes a new 'userId'!
  ngOnChanges(changes: SimpleChanges) {
    
    // The SimpleChanges object tells us exactly what changed!
    if (changes['userId']) {
      const oldVal = changes['userId'].previousValue;
      const newVal = changes['userId'].currentValue;
      
      console.log(`User ID changed from ${oldVal} to ${newVal}`);
      
      // We can trigger a new API call here based on the new ID!
      this.fetchNewUserDetails(newVal);
    }
  }

  fetchNewUserDetails(id: number) {}
}
```

### Level 4 — Advanced (ngAfterViewInit)
Sometimes you need to physically manipulate the DOM (e.g., initializing a 3rd-party charting library like Chart.js on a `<canvas>` element). 

If you try to find the `<canvas>` element in `ngOnInit`, it will crash! `ngOnInit` fires *before* the HTML template has finished rendering. You must use `ngAfterViewInit`.

```typescript
import { Component, AfterViewInit, ViewChild, ElementRef } from '@angular/core';

export class ChartComponent implements AfterViewInit {
  
  // Grab a reference to the <canvas #myChart> tag in the HTML!
  @ViewChild('myChart') canvasRef!: ElementRef;

  // This fires exactly once, AFTER the HTML is 100% rendered on the screen!
  ngAfterViewInit() {
    // We are now guaranteed that the DOM element exists!
    const htmlElement = this.canvasRef.nativeElement;
    
    // Initialize standard JS library...
    // new Chart(htmlElement, { ... });
  }
}
```

## REAL_WORLD
Handling RxJS unsubscriptions in `ngOnDestroy` is the most tedious, error-prone part of Angular development.

In Angular 16+, a brilliant new utility was introduced: `takeUntilDestroyed()`. It hooks directly into the DI context and automatically destroys the subscription when the component dies, completely eliminating the need to write `ngOnDestroy` boilerplate!

```typescript
import { Component } from '@angular/core';
import { takeUntilDestroyed } from '@angular/core/rxjs-interop';
import { interval } from 'rxjs';

export class ModernTimerComponent {
  
  constructor() {
    // MAGIC: We pipe the stream through takeUntilDestroyed().
    // We never need to save a Subscription reference, and we never write ngOnDestroy!
    interval(1000).pipe(
      takeUntilDestroyed()
    ).subscribe(tick => console.log(tick));
  }
}
```

## INTERVIEW
**Q: What is the difference between `constructor()` and `ngOnInit()`?**
A: `constructor()` is a standard TypeScript feature called when the Class is instantiated in memory. At this point, Angular has NOT processed the `@Input()` bindings. `ngOnInit()` is an Angular-specific lifecycle hook that runs *after* Angular has fully wired up the component and injected the initial `@Input()` values. Always use `constructor` solely for Dependency Injection, and `ngOnInit` for actual initialization logic.

**Q: Do you need to unsubscribe from `HttpClient` Observables in `ngOnDestroy`?**
A: No! `HttpClient` observables are "finite." The moment the server responds with the data, the Observable emits the value and immediately calls `.complete()` on itself. A completed observable is automatically garbage collected. You only need to unsubscribe from "infinite" observables (like `Router.events`, `setInterval`, or `Subject` streams).

**Q: What is `ngDoCheck()`?**
A: It is a hook that fires on every single Change Detection cycle (potentially hundreds of times a second). It is designed to allow you to write custom dirty-checking logic if Angular's default change detection misses something. It is almost never used in standard application development and will severely degrade performance if used incorrectly.
