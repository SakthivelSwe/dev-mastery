## WHY
If you have a `UserListComponent` and a `UserProfileComponent`, and both need to fetch a list of users from your API, where do you write the `fetch()` logic? 

If you write it inside the `UserListComponent`, the `UserProfileComponent` can't access it. If you copy/paste the code into both components, you violate the DRY (Don't Repeat Yourself) principle. 

Angular solves this architectural problem using **Services** and **Dependency Injection (DI)**. A Service is a centralized TypeScript class that holds business logic, state, or HTTP calls. Components are completely stripped of business logic; they simply ask Angular's Dependency Injection system to hand them a copy of the Service, and they use its methods.

## THEORY
### Dependency Injection (DI)
Dependency Injection is an incredibly complex architectural pattern (Inversion of Control) that Angular completely hides from you. 
When your `UserComponent` says: *"I need the UserService to function!"*, it doesn't run `new UserService()`. Instead, it asks the Angular Framework for it. 
Angular looks in its internal "Registry" (the IoC Container), finds the `UserService`, creates exactly ONE instance of it (a Singleton), and injects that exact same instance into every component that asks for it.

Because it's a Singleton, if Component A updates a variable inside the `UserService`, Component B will instantly see the updated variable! This is how Angular manages global state.

## CODE

### Level 1 — Beginner (Creating a Service)
Always use the Angular CLI to generate services.
```bash
ng generate service user
# Or: ng g s user
```

This creates `user.service.ts`. Notice the `@Injectable` decorator. It tells Angular that this class is allowed to be injected into other classes.

```typescript
import { Injectable } from '@angular/core';

// The 'providedIn: root' tells Angular to create exactly ONE instance of this 
// service and make it available to the entire application (Singleton).
@Injectable({
  providedIn: 'root'
})
export class UserService {
  // Centralized state
  private users: string[] = ['Alice', 'Bob', 'Charlie'];

  // Centralized business logic
  getUsers(): string[] {
    return this.users;
  }

  addUser(name: string): void {
    this.users.push(name);
  }
}
```

### Level 2 — Building Blocks (Injecting the Service)
To use the service, a Component must declare it in its `constructor`. This is called **Constructor Injection**.

```typescript
import { Component, OnInit } from '@angular/core';
import { UserService } from './user.service';

@Component({
  selector: 'app-user-list',
  template: `
    <ul>
      <li *ngFor="let u of userList">{{ u }}</li>
    </ul>
    <button (click)="createNewUser()">Add User</button>
  `
})
export class UserListComponent implements OnInit {
  userList: string[] = [];

  // 1. Dependency Injection! 
  // By placing `private` in the constructor, TypeScript automatically creates 
  // a class property, and Angular automatically injects the Singleton into it.
  constructor(private userService: UserService) {}

  ngOnInit() {
    // 2. We use the service to get our data!
    this.userList = this.userService.getUsers();
  }

  createNewUser() {
    // 3. We use the service to modify global state!
    this.userService.addUser('Dave');
  }
}
```

### Level 3 — Intermediate (Injecting Services into Services)
Services can also use Dependency Injection to talk to *other* Services!

```typescript
import { Injectable } from '@angular/core';
import { LoggerService } from './logger.service';

@Injectable({
  providedIn: 'root'
})
export class PaymentService {

  // We inject the LoggerService into the PaymentService!
  constructor(private logger: LoggerService) {}

  processPayment(amount: number) {
    // Execute logic...
    this.logger.log(`Payment processed for $${amount}`);
  }
}
```
*(Note: In older versions of Angular, if a Service didn't have `@Injectable()` on it, it couldn't receive injected dependencies. Today, `@Injectable` is mandatory for all services).*

### Level 4 — Advanced (Hierarchical Injectors)
By default, `providedIn: 'root'` makes the Service a global Singleton. 

But what if you are building an email client, and you want each individual Email Draft window to have its *own independent copy* of the `DraftService`? You do not want them sharing state!

You can bypass the Root Injector and register the service directly onto the Component's metadata. This creates a **Component-Level Provider**.

```typescript
import { Component } from '@angular/core';
import { DraftService } from './draft.service';

@Component({
  selector: 'app-email-draft',
  template: `<textarea [(ngModel)]="draftService.content"></textarea>`,
  // MAGIC: We provide the service directly here!
  // Every time a new <app-email-draft> is created, Angular destroys the old DI rules 
  // and instantiates a brand new, isolated DraftService purely for this component!
  providers: [DraftService] 
})
export class EmailDraftComponent {
  constructor(public draftService: DraftService) {}
}
```

## REAL_WORLD
In Angular 14, the framework introduced an entirely new way to inject dependencies called the `inject()` function. It is rapidly replacing Constructor Injection in modern enterprise codebases because it is vastly cleaner, especially when dealing with inheritance.

**The Old Way (Constructor Boilerplate):**
```typescript
export class ProfileComponent {
  constructor(
    private userService: UserService,
    private authService: AuthService,
    private router: Router
  ) {}
}
```

**The New Way (Angular 14+ `inject()` function):**
```typescript
import { Component, inject } from '@angular/core';

export class ProfileComponent {
  // Flawlessly clean, acts just like a standard variable declaration!
  private userService = inject(UserService);
  private authService = inject(AuthService);
  private router = inject(Router);
}
```
You will see `inject()` everywhere in modern Angular signals and functional guards.

## INTERVIEW
**Q: What is the difference between an `@Injectable()` service and a standard TypeScript class?**
A: Technically, nothing. They are both just classes. The `@Injectable` decorator simply generates metadata during AOT compilation that allows the Angular Dependency Injection engine to find it, instantiate it, and track its memory lifecycle. If you try to inject a class without `@Injectable`, the compiler will crash.

**Q: Can you inject a Component into a Service?**
A: No. Components are tied to the DOM and have a specific lifecycle (`ngOnInit`, `ngOnDestroy`). Services are pure logical singletons. If a Service tries to hold a reference to a Component, it will create catastrophic memory leaks because the Service (which lives forever) will prevent the Garbage Collector from destroying the Component when the user navigates to a new page.

**Q: What is an InjectionToken?**
A: Usually, you inject classes (`UserService`). But what if you want to inject an Interface, a primitive string (like an API URL), or a configuration object? Interfaces disappear after TypeScript compilation, so Angular can't find them at runtime! You create an `InjectionToken<string>('API_URL')` to give Angular a physical object it can register in the DI container and inject into constructors.
