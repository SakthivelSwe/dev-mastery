## WHY
JavaScript natively handles asynchronous events using Callbacks and Promises. A Promise is fine for a single HTTP request (it executes once, returns data, and dies). But what if you are listening to a WebSocket that streams live stock prices every second? What if you are listening to the user typing in a search box, and you want to ignore keystrokes until they stop typing for 500 milliseconds? 

Promises mathematically cannot handle streams of continuous events or cancellation.

**RxJS (Reactive Extensions for JavaScript)** is a massive, third-party library that Angular adopted as a core architectural pillar. It introduces the **Observable**. If you don't understand Observables and the various RxJS "Operators" (like `map`, `filter`, and `switchMap`), you cannot write Enterprise Angular code.

## THEORY
### Observables vs Promises
- **Promise**: Executes immediately upon creation (Eager). Emits exactly one value. Cannot be canceled.
- **Observable**: Does absolutely nothing until you `.subscribe()` to it (Cold/Lazy). Can emit zero, one, or an infinite number of values over time. Can be canceled instantly by calling `.unsubscribe()`.

### Operators (The Pipe)
RxJS provides dozens of utility functions called "Operators." You use the `.pipe()` method to string these operators together, manipulating the stream of data *before* it reaches the `.subscribe()` block.

## CODE

### Level 1 — Beginner (Creating and Subscribing)
Usually, Angular creates Observables for you (via `HttpClient` or `ReactiveForms`). But you can create them manually.

```typescript
import { Observable } from 'rxjs';

// 1. Create a stream that emits 3 numbers, then completes.
const myStream$ = new Observable<number>(observer => {
  observer.next(1);
  observer.next(2);
  
  setTimeout(() => {
    observer.next(3);
    observer.complete(); // The stream is finished!
  }, 1000);
});

// 2. We subscribe to listen!
myStream$.subscribe({
  next: (val) => console.log('Received:', val),
  error: (err) => console.error('Error:', err),
  complete: () => console.log('Stream finished!')
});

// Output: 
// Received: 1
// Received: 2
// (1 second later...)
// Received: 3
// Stream finished!
```

### Level 2 — Building Blocks (Pipe, Map, and Filter)
Before the data reaches the subscriber, we can modify it.

```typescript
import { of } from 'rxjs';
import { map, filter } from 'rxjs/operators';

// 'of' is a helper that instantly creates an Observable from an array
const stream$ = of(1, 2, 3, 4, 5);

stream$.pipe(
  // Filter out odd numbers!
  filter(num => num % 2 === 0),
  
  // Multiply the remaining numbers by 10!
  map(num => num * 10)
).subscribe(result => {
  console.log(result); 
});

// Output: 
// 20
// 40
```

### Level 3 — Intermediate (The Subject)
An `Observable` is read-only. You can subscribe to it, but you can't force it to emit new data from the outside. 
A **Subject** is a special type of Observable that is BOTH read and write! You can call `.next()` on it to forcefully broadcast data to all subscribers. 

Subjects are the backbone of State Management in Angular.

```typescript
import { Injectable } from '@angular/core';
import { Subject, BehaviorSubject } from 'rxjs';

@Injectable({ providedIn: 'root' })
export class NotificationService {
  
  // 1. A BehaviorSubject requires an initial value. 
  // It always remembers the 'current' value for new subscribers!
  private messageSource = new BehaviorSubject<string>("Welcome!");

  // 2. We expose it as a READ-ONLY Observable so components can't mess with it directly.
  public currentMessage$ = this.messageSource.asObservable();

  // 3. Components call this method to update the global state!
  updateMessage(newMessage: string) {
    this.messageSource.next(newMessage); // Broadcast to everyone!
  }
}
```

### Level 4 — Advanced (Higher-Order Mapping: switchMap)
This is the most asked Angular interview question in existence.

Imagine a user typing into a search box. Every keystroke triggers an HTTP request.
If they type "A", Request 1 fires.
If they type "P", Request 2 fires.
If they type "P", Request 3 fires.
If Request 3 finishes *before* Request 1 (due to network lag), the UI will show the results for "A", destroying the app.

`switchMap` solves this by **canceling the previous HTTP request** if a new one arrives before it finishes!

```typescript
import { Component, OnInit } from '@angular/core';
import { FormControl } from '@angular/forms';
import { HttpClient } from '@angular/common/http';
import { debounceTime, distinctUntilChanged, switchMap } from 'rxjs/operators';

export class SearchComponent implements OnInit {
  searchControl = new FormControl('');

  constructor(private http: HttpClient) {}

  ngOnInit() {
    this.searchControl.valueChanges.pipe(
      
      // 1. Wait 300ms after the user stops typing before doing anything!
      debounceTime(300),
      
      // 2. If they hit Backspace and then the exact same letter again, ignore it!
      distinctUntilChanged(),
      
      // 3. SWITCHMAP: The magic operator. 
      // It takes the search string, creates a brand new HTTP Observable, 
      // and subscribes to it. IF the user types another letter while this HTTP 
      // request is pending, switchMap instantly cancels the old HTTP request!
      switchMap(searchTerm => {
        return this.http.get(`/api/search?q=${searchTerm}`);
      })
      
    ).subscribe(results => {
      // We are mathematically guaranteed that these results match 
      // the exact final string the user typed!
      console.log('Search Results:', results);
    });
  }
}
```

## REAL_WORLD
While RxJS is incredibly powerful, it is also notoriously difficult to learn. Developers frequently create "Subscription Hell" (nesting `.subscribe()` calls inside other `.subscribe()` calls), resulting in memory leaks and impossible-to-read code.

In **Angular 16**, the framework began a slow transition away from RxJS for UI State Management, introducing **Signals**. 
Signals handle synchronous state updates flawlessly without `.subscribe()` or `| async` pipes. However, RxJS will *always* remain the core engine for asynchronous network events (like `HttpClient` and complex race conditions). 

Enterprise developers now use **Signals for State, and RxJS for Events**.

## INTERVIEW
**Q: What is the difference between `Subject` and `BehaviorSubject`?**
A: A `Subject` does not have an initial value and does not remember past values. If Component A subscribes to a `Subject` *after* it emitted "Hello", Component A will hear nothing until the next emission. A `BehaviorSubject` requires an initial value (e.g., "Hello"). If Component A subscribes late, the `BehaviorSubject` instantly hands it the most recent value ("Hello").

**Q: What is the difference between `mergeMap`, `concatMap`, `exhaustMap`, and `switchMap`?**
A: 
- `switchMap`: Cancels the previous inner observable if a new one arrives (Great for Search).
- `concatMap`: Queues them up. If Request 1 is running, Request 2 waits in line until Request 1 finishes (Great for sending ordered chat messages).
- `mergeMap`: Runs them all simultaneously in parallel. Order is not guaranteed (Great for fetching 10 independent user profiles at once).
- `exhaustMap`: Ignores new events if one is already running. (Great for a "Login" button so the user can't spam it 50 times).

**Q: How do you handle Errors in an RxJS stream?**
A: You use the `catchError` operator in the `.pipe()`. It intercepts the error, allows you to log it or show a toast notification, and then you must return an observable (usually `of([])` or `throwError()`) to keep the stream alive or gracefully shut it down.

## FEYNMAN CHECK

### Explain RxJS in Angular Like I'm 10 Years Old
> RxJS is Angular's built-in reactive library — it models async events as STREAMS that you can transform with operators. Imagine a water pipe: the source (Observable) produces values over time; operators (map, filter, switchMap, debounceTime) transform them; subscribe consumes them. Angular uses RxJS for: HTTP responses (HttpClient returns Observables), routing events (Router.events), form value changes (form.valueChanges), custom event buses, and reactive state patterns. The non-obvious depth: operators are LAZY — nothing happens until subscribe(). switchMap is the most important operator for Angular — it cancels the previous inner Observable when a new source value arrives (perfect for search-as-you-type where you want to discard stale API results).

### 5 Deep Conceptual Questions

**Q1: What is the difference between Subject, BehaviorSubject, and ReplaySubject?**
> **A:** Subject is a plain multicast Observable — new subscribers only get FUTURE values. BehaviorSubject holds the CURRENT value and immediately emits it to new subscribers — essential for state management where late subscribers need the current state. ReplaySubject buffers N past values and replays them to new subscribers — useful for event logs or action streams where late subscribers need history. All three are both Observable (you can subscribe) AND Observer (you can call next/error/complete) — they bridge imperative code to reactive streams.

**Q2: Mental model for the "Higher-Order Observable" operators?**
> **A:** switchMap, mergeMap, concatMap, exhaustMap take each value from a source and MAP it to a new inner Observable. The difference is how they handle CONCURRENT inner Observables: switchMap CANCELS the previous inner when a new source value arrives (good for: search, autocomplete — discard stale results). mergeMap runs all inner Observables CONCURRENTLY (good for: parallel file downloads). concatMap queues inner Observables, running ONE AT A TIME in order (good for: sequential saves). exhaustMap IGNORES new source values while inner is active (good for: login button — ignore double-clicks).

**Q3: Most dangerous misconception about RxJS in Angular?**
> **A:** You need to unsubscribe from ALL observables:
> ```typescript
> // Observables that complete automatically (NO unsubscribe needed):
> this.http.get('/api/users')       // completes after response
> of(1, 2, 3)                       // completes immediately
> timer(1000)                       // completes after one emission
> this.route.paramMap               // completes when component destroyed (router managed)
>
> // Observables that NEVER complete (MUST unsubscribe or use takeUntilDestroyed):
> interval(1000)                    // infinite timer
> fromEvent(document, 'click')      // DOM events
> this.store.select(mySelector)     // NgRx selectors
> new BehaviorSubject(0)            // manual subjects
> webSocket('ws://localhost')       // websocket streams
> ```

**Q4: How do you compose multiple HTTP calls with RxJS?**
> **A:** forkJoin([obs1, obs2, obs3]) runs all in PARALLEL, emits when ALL complete — like Promise.all. combineLatest([obs1, obs2]) emits whenever ANY source emits, with latest values from each — good for combining filter + pagination + search. switchMap is used for SEQUENTIAL calls where the first response determines the second: first get the user, then get their orders: getUser(id).pipe(switchMap(user => getOrders(user.teamId))). withLatestFrom snapshots the current value of a secondary stream without subscribing to it.

**Q5: FAANG-grade definition of RxJS in Angular.**
> **A:** "RxJS (Reactive Extensions for JavaScript) is Angular's core reactive library — modelling asynchronous values as lazy Observable streams with first-class composability via operators (map, filter, switchMap, mergeMap, concatMap, exhaustMap, debounceTime, distinctUntilChanged, catchError, retry, shareReplay) — with Subjects (plain, Behavior, Replay, Async) bridging imperative and reactive code — and Angular integrating RxJS throughout: HttpClient, Router events, FormControl.valueChanges, EventEmitter internals, and async pipe — with takeUntilDestroyed/DestroyRef for lifecycle-aware subscription management."

#### Step 2 — Search Component
```typescript
// search.component.ts
import { Component, inject, signal } from '@angular/core';
import { FormControl, ReactiveFormsModule } from '@angular/forms';
import { HttpClient } from '@angular/common/http';
import { NgFor, NgIf, AsyncPipe } from '@angular/common';
import { debounceTime, distinctUntilChanged, switchMap, catchError, startWith, tap } from 'rxjs/operators';
import { of, Observable } from 'rxjs';

interface User { id: number; name: string; email: string; }

@Component({
  standalone: true,
  selector: 'app-search',
  imports: [ReactiveFormsModule, NgFor, NgIf],
  template: `
    <input [formControl]="searchControl" placeholder="Search users...">
    <p *ngIf="isLoading()">Searching...</p>
    <p *ngIf="error()">Error: {{ error() }}</p>
    <ul>
      <li *ngFor="let user of results()">{{ user.name }} ({{ user.email }})</li>
    </ul>
    <p *ngIf="!isLoading() && results().length === 0 && searchControl.value">
      No results for "{{ searchControl.value }}"
    </p>
  `,
})
export class SearchComponent {
  private http = inject(HttpClient);
  searchControl = new FormControl('');
  results = signal<User[]>([]);
  isLoading = signal(false);
  error = signal<string | null>(null);

  constructor() {
    this.searchControl.valueChanges.pipe(
      debounceTime(300),            // wait 300ms after last keystroke
      distinctUntilChanged(),       // skip if same value
      tap(() => { this.isLoading.set(true); this.error.set(null); }),
      switchMap(query => {          // cancel previous request on new keystroke
        if (!query?.trim()) return of([]);
        return this.http.get<User[]>('https://jsonplaceholder.typicode.com/users').pipe(
          // Client-side filter for demo — real app would pass query to server
          catchError(err => { this.error.set(err.message); return of([]); }),
        );
      }),
      tap(() => this.isLoading.set(false)),
    ).subscribe(results => {
      const q = this.searchControl.value?.toLowerCase() ?? '';
      this.results.set(results.filter(u => u.name.toLowerCase().includes(q)));
    });
  }
}
```

#### Step 3 — Combining Multiple Streams
```typescript
// Advanced: combine search + filter + pagination
import { combineLatest, BehaviorSubject } from 'rxjs';

export class AdvancedSearchComponent {
  private search$ = new BehaviorSubject('');
  private filter$ = new BehaviorSubject<'all' | 'active'>('all');
  private page$ = new BehaviorSubject(0);

  results$ = combineLatest([this.search$, this.filter$, this.page$]).pipe(
    debounceTime(100),
    switchMap(([query, filter, page]) =>
      this.http.get('/api/users', { params: { query, filter, page: page.toString() } })
    ),
  );

  setSearch(q: string) { this.search$.next(q); this.page$.next(0); }
  setFilter(f: 'all' | 'active') { this.filter$.next(f); this.page$.next(0); }
  nextPage() { this.page$.next(this.page$.value + 1); }
}
```

#### Step 4 — Error Handling: catchError and retry
```typescript
import { catchError, retry, throwError } from 'rxjs';

const resilientSearch$ = this.http.get('/api/search').pipe(
  retry({ count: 2, delay: 1000 }),
  catchError(err => {
    if (err.status === 0) return throwError(() => new Error('Network offline'));
    if (err.status === 429) return throwError(() => new Error('Rate limited'));
    return throwError(() => err);  // re-throw unknown errors
  }),
);
```
