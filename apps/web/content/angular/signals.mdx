## WHY
Since 2016, Angular's reactivity model has relied on a library called `Zone.js`. `Zone.js` works by "monkey-patching" the entire browser—hijacking every single click event, `setTimeout`, and HTTP request. When an event finishes, `Zone.js` screams, *"Something might have changed!"* and Angular is forced to check the entire component tree from top to bottom just to see if a single text variable updated. This is incredibly inefficient.
Introduced in Angular 16, **Signals** are the future of Angular. They provide "Fine-Grained Reactivity". Instead of checking the whole app, Angular knows *exactly* which variable changed, and surgical updates *only* the specific HTML `<p>` tag that binds to it. It makes Angular blindingly fast, and eliminates the steep learning curve of RxJS for basic state management!

## THEORY
### What is a Signal?
A Signal is a simple wrapper around a value. It can be a primitive (string, number) or an object.
It has three core properties:
1. It holds a value.
2. It can notify consumers when that value changes.
3. It tracks its dependencies automatically.

### The Three Pillars of Signals
1. **`signal()`:** A writable piece of state. You can set it, and update it.
2. **`computed()`:** A derived piece of state. It automatically recalculates when its dependent signals change. (Like a "getter" that caches its result).
3. **`effect()`:** A side-effect. A function that automatically runs when the signals it reads change (like logging to the console or saving to localStorage).

## CODE

### Level 1 — Creating and Using Signals
Let's build a simple counter. Compare this to standard property assignment.

```typescript
import { Component, signal } from '@angular/core';

@Component({
  selector: 'app-counter',
  template: `
    <!-- We MUST call the signal like a function () to read its value! -->
    <h1>Count: {{ count() }}</h1>
    <button (click)="increment()">Increment</button>
    <button (click)="reset()">Reset</button>
  `
})
export class CounterComponent {
  // 1. Create a Signal with an initial value of 0
  count = signal(0); 

  increment() {
    // 2. update() allows us to modify the value based on the previous value
    this.count.update(currentValue => currentValue + 1);
  }

  reset() {
    // 3. set() completely overwrites the value
    this.count.set(0);
  }
}
```

### Level 2 — Computed Signals (Derived State)
Imagine a shopping cart. If you have a signal holding `items`, you shouldn't manually calculate the `totalPrice` every time an item is added. You use `computed()`.

```typescript
import { Component, signal, computed } from '@angular/core';

export class CartComponent {
  // Writable Signals
  price = signal(100);
  quantity = signal(2);

  // Derived Signal (Computed)
  // Angular automatically knows this depends on 'price' and 'quantity'!
  // If neither of those change, Angular CACHES the result of total().
  total = computed(() => this.price() * this.quantity());

  applyDiscount() {
    // When we change the price...
    this.price.set(80);
    // ...the 'total' computed signal is INSTANTLY marked as stale, 
    // and will recalculate to 160 the next time the template reads it!
  }
}
```

### Level 3 — Effects (Reactivity outside the Template)
Sometimes you need to react to a signal change *outside* of the HTML template (e.g., syncing data to the backend when a user types, or debugging).

```typescript
import { Component, signal, effect } from '@angular/core';

export class SyncComponent {
  username = signal('');

  constructor() {
    // An effect runs at least once immediately.
    // It automatically tracks any Signals it reads inside its function body!
    effect(() => {
      // Because we read this.username(), Angular registers it as a dependency.
      // Every time username changes, this entire block runs again automatically!
      console.log(`The username is now: ${this.username()}`);
      
      // We can also trigger side effects!
      localStorage.setItem('user', this.username());
    });
  }

  updateName(newName: string) {
    this.username.set(newName); // Triggers the effect!
  }
}
```

### Level 4 — Working with Objects and Arrays
Signals check for equality using the standard JavaScript `===` operator.
If your Signal holds an array, and you push an item into the array, the memory reference doesn't change! The Signal won't know it updated.
You must use `.update()` to return a brand new array reference!

```typescript
interface Todo { id: number; text: string; }

export class TodoComponent {
  todos = signal<Todo[]>([]);

  addTodo(text: string) {
    // BAD: Mutating the array does NOT trigger reactivity!
    // this.todos().push({ id: 1, text });

    // GOOD: Return a brand new array!
    this.todos.update(currentTodos => [
      ...currentTodos, 
      { id: Date.now(), text }
    ]);
  }

  removeTodo(id: number) {
    // Array.filter creates a new array reference!
    this.todos.update(currentTodos => currentTodos.filter(t => t.id !== id));
  }
}
```

## REAL_WORLD
**Signals vs. RxJS (Observables)**
For years, Angular forced developers to use RxJS `BehaviorSubjects` for state management, which resulted in horrific template code filled with `| async` pipes, and memory leaks if developers forgot to unsubscribe.
Are Signals replacing RxJS?
**No. They complement each other.**
- **Signals** are designed for Synchronous State Management (What is the value of `counter` right now? Update the UI!).
- **RxJS** is designed for Asynchronous Event Streams (Listen to the WebSocket, debounce the user's typing for 500ms, then switchMap into an HTTP request).
Angular provides native interoperability: `toObservable(mySignal)` converts a Signal into a Stream, and `toSignal(myObservable)` converts a Stream into a synchronous Signal! You use RxJS to fetch the data, and Signals to display it.

## INTERVIEW
**Q: You write a `computed()` signal: `const fullName = computed(() => this.firstName() + ' ' + this.lastName());`. In your template, you call `{{ fullName() }}` 50 times inside a large list. The Code Reviewer is worried that this will execute the string concatenation math 50 times per render cycle, destroying performance. Are they correct?**
A: **No, they are incorrect! Computed Signals are heavily cached and lazily evaluated.**
This is the magic of Signals over standard getter functions (`get fullName() { ... }`).
When Angular evaluates the `computed` signal the first time, it performs the string concatenation and *caches the result*. The next 49 times the template asks for `fullName()`, Angular instantly returns the cached string without executing the math function again.
The cached value is only ever discarded (marked as "stale") if `firstName` or `lastName` specifically changes. Furthermore, the math isn't even recalculated immediately when the dependencies change; it is evaluated *lazily*, meaning it only recalculates when the template actually asks for the value again! This guarantees mathematically optimal performance.
