## WHY
Since 2016, Angular's reactivity model has relied on a library called `Zone.js`. `Zone.js` works by "monkey-patching" the entire browser—hijacking every single click event, `setTimeout`, and HTTP request. When an event finishes, `Zone.js` screams, *"Something might have changed!"* and Angular is forced to check the entire component tree from top to bottom just to see if a single text variable updated. This is incredibly inefficient.
Introduced in Angular 16, **Signals** are the future of Angular. They provide "Fine-Grained Reactivity". Instead of checking the whole app, Angular knows *exactly* which variable changed, and surgical updates *only* the specific HTML `<p>` tag that binds to it. It makes Angular blindingly fast, and eliminates the steep learning curve of RxJS for basic state management!

## THEORY
### What is a Signal?
A Signal is a simple wrapper around a value. It can be a primitive (string, number) or an object.
It has three core properties:
1. It holds a value.
2. It can notify consumers when that value changes.
3. It tracks its dependencies automatically.

### The Three Pillars of Signals
1. **`signal()`:** A writable piece of state. You can set it, and update it.
2. **`computed()`:** A derived piece of state. It automatically recalculates when its dependent signals change. (Like a "getter" that caches its result).
3. **`effect()`:** A side-effect. A function that automatically runs when the signals it reads change (like logging to the console or saving to localStorage).

## CODE

### Level 1 — Creating and Using Signals
Let's build a simple counter. Compare this to standard property assignment.

```typescript
import { Component, signal } from '@angular/core';

@Component({
  selector: 'app-counter',
  template: `
    <!-- We MUST call the signal like a function () to read its value! -->
    <h1>Count: {{ count() }}</h1>
    <button (click)="increment()">Increment</button>
    <button (click)="reset()">Reset</button>
  `
})
export class CounterComponent {
  // 1. Create a Signal with an initial value of 0
  count = signal(0); 

  increment() {
    // 2. update() allows us to modify the value based on the previous value
    this.count.update(currentValue => currentValue + 1);
  }

  reset() {
    // 3. set() completely overwrites the value
    this.count.set(0);
  }
}
```

### Level 2 — Computed Signals (Derived State)
Imagine a shopping cart. If you have a signal holding `items`, you shouldn't manually calculate the `totalPrice` every time an item is added. You use `computed()`.

```typescript
import { Component, signal, computed } from '@angular/core';

export class CartComponent {
  // Writable Signals
  price = signal(100);
  quantity = signal(2);

  // Derived Signal (Computed)
  // Angular automatically knows this depends on 'price' and 'quantity'!
  // If neither of those change, Angular CACHES the result of total().
  total = computed(() => this.price() * this.quantity());

  applyDiscount() {
    // When we change the price...
    this.price.set(80);
    // ...the 'total' computed signal is INSTANTLY marked as stale, 
    // and will recalculate to 160 the next time the template reads it!
  }
}
```

### Level 3 — Effects (Reactivity outside the Template)
Sometimes you need to react to a signal change *outside* of the HTML template (e.g., syncing data to the backend when a user types, or debugging).

```typescript
import { Component, signal, effect } from '@angular/core';

export class SyncComponent {
  username = signal('');

  constructor() {
    // An effect runs at least once immediately.
    // It automatically tracks any Signals it reads inside its function body!
    effect(() => {
      // Because we read this.username(), Angular registers it as a dependency.
      // Every time username changes, this entire block runs again automatically!
      console.log(`The username is now: ${this.username()}`);
      
      // We can also trigger side effects!
      localStorage.setItem('user', this.username());
    });
  }

  updateName(newName: string) {
    this.username.set(newName); // Triggers the effect!
  }
}
```

### Level 4 — Working with Objects and Arrays
Signals check for equality using the standard JavaScript `===` operator.
If your Signal holds an array, and you push an item into the array, the memory reference doesn't change! The Signal won't know it updated.
You must use `.update()` to return a brand new array reference!

```typescript
interface Todo { id: number; text: string; }

export class TodoComponent {
  todos = signal<Todo[]>([]);

  addTodo(text: string) {
    // BAD: Mutating the array does NOT trigger reactivity!
    // this.todos().push({ id: 1, text });

    // GOOD: Return a brand new array!
    this.todos.update(currentTodos => [
      ...currentTodos, 
      { id: Date.now(), text }
    ]);
  }

  removeTodo(id: number) {
    // Array.filter creates a new array reference!
    this.todos.update(currentTodos => currentTodos.filter(t => t.id !== id));
  }
}
```

## REAL_WORLD
**Signals vs. RxJS (Observables)**
For years, Angular forced developers to use RxJS `BehaviorSubjects` for state management, which resulted in horrific template code filled with `| async` pipes, and memory leaks if developers forgot to unsubscribe.
Are Signals replacing RxJS?
**No. They complement each other.**
- **Signals** are designed for Synchronous State Management (What is the value of `counter` right now? Update the UI!).
- **RxJS** is designed for Asynchronous Event Streams (Listen to the WebSocket, debounce the user's typing for 500ms, then switchMap into an HTTP request).
Angular provides native interoperability: `toObservable(mySignal)` converts a Signal into a Stream, and `toSignal(myObservable)` converts a Stream into a synchronous Signal! You use RxJS to fetch the data, and Signals to display it.

## INTERVIEW
**Q: You write a `computed()` signal: `const fullName = computed(() => this.firstName() + ' ' + this.lastName());`. In your template, you call `{{ fullName() }}` 50 times inside a large list. The Code Reviewer is worried that this will execute the string concatenation math 50 times per render cycle, destroying performance. Are they correct?**
A: **No, they are incorrect! Computed Signals are heavily cached and lazily evaluated.**
This is the magic of Signals over standard getter functions (`get fullName() { ... }`).
When Angular evaluates the `computed` signal the first time, it performs the string concatenation and *caches the result*. The next 49 times the template asks for `fullName()`, Angular instantly returns the cached string without executing the math function again.
The cached value is only ever discarded (marked as "stale") if `firstName` or `lastName` specifically changes. Furthermore, the math isn't even recalculated immediately when the dependencies change; it is evaluated *lazily*, meaning it only recalculates when the template actually asks for the value again! This guarantees mathematically optimal performance.

## FEYNMAN CHECK

### Explain Angular Signals Like I'm 10 Years Old
> Signals are Angular's reactive primitive — a container that holds a value AND tracks who reads it. When you read a signal (signal()), Angular registers that component or computed as a DEPENDENT. When the signal's value changes via .set() or .update(), Angular AUTOMATICALLY re-renders all dependents — no Zone.js needed, no markForCheck(), no manual subscription. Think of a signal as a spreadsheet cell: when you change cell A1, all cells that reference it (B1, C1) automatically recalculate. The non-obvious depth: computed() creates a DERIVED signal — its value is recalculated lazily only when a dependency changes and the computed is READ. effect() runs a SIDE EFFECT whenever its signal dependencies change — use for logging, syncing to localStorage, triggering animations.

### 5 Deep Conceptual Questions

**Q1: What are the three signal primitives?**
> **A:** (1) signal(initialValue) — a writable container. Read with signal(); write with signal.set(val), signal.update(fn), signal.mutate(fn) (for objects). (2) computed(() => expr) — a read-only derived signal; lazily evaluated when read after dependencies change; memoised. (3) effect(() => { /* runs when deps change */ }) — a side effect tied to signal reads; runs after render; auto-cleans up when the owning context is destroyed. All three are SYNCHRONOUS — no Promises or Observables involved.

**Q2: Mental model for computed() memoisation?**
> **A:** computed() is LAZY + MEMOISED. If userCount and productCount haven't changed since the last read of totalCount = computed(() => userCount() + productCount()), Angular returns the CACHED value without recalculating. This makes computed() safe to use in templates — even if the template reads a computed 100 times per render cycle, the function runs at most ONCE per dependency change. This contrasts sharply with template method calls (doCalculation()) which run on EVERY change detection cycle.

**Q3: Most dangerous misconception about signals?**
> **A:** You can use effect() for component state updates:
> ```typescript
> // WRONG: setting another signal inside effect() causes infinite loops
> const a = signal(0);
> const b = signal(0);
>
> effect(() => {
>   b.set(a() * 2);  // reading a() makes effect depend on a
>   // Setting b in effect triggers the effect again if b is also read here
> });
>
> // CORRECT: use computed() for derived state
> const b = computed(() => a() * 2);
>
> // Use effect() only for SIDE EFFECTS: localStorage, logging, DOM libraries
> effect(() => {
>   localStorage.setItem('lastValue', JSON.stringify(a()));  // OK
> });
> ```

**Q4: How do signals interact with OnPush change detection?**
> **A:** With signals, Angular marks a component as dirty specifically when a signal it reads changes — this is MORE granular than OnPush (which marks dirty on any @Input reference change). A component reading 10 signals but only 1 changing gets a targeted dirty mark for just that component. Angular 17+ uses a "reactive graph" — the dependency graph built from signal reads — to determine the minimal set of components to re-render. This is the foundation for zoneless Angular: no Zone.js patching needed when all async state flows through signals.

**Q5: FAANG-grade definition of Angular signals.**
> **A:** "Angular signals (Angular 16+) are reactive primitives implementing the observer pattern at the framework level — signal() for writable state containers, computed() for memoised derived values (lazily recalculated when dependencies change and the value is read), and effect() for side effects tied to signal-dependency changes — enabling fine-grained, synchronous reactivity without Zone.js, compatible with OnPush change detection, and forming the foundation of zoneless Angular applications via a reactive dependency graph that marks only signal-reading components as dirty on state change."

#### Step 2 — Cart Store Service
```typescript
// cart-store.service.ts
import { Injectable, signal, computed, effect } from '@angular/core';

export interface CartItem { id: string; name: string; price: number; qty: number; }

@Injectable({ providedIn: 'root' })
export class CartStoreService {
  private _items = signal<CartItem[]>(this.loadFromStorage());

  // Public read-only view
  items = this._items.asReadonly();

  // Derived signals — computed lazily, memoised
  itemCount = computed(() => this._items().reduce((s, i) => s + i.qty, 0));
  subtotal  = computed(() => this._items().reduce((s, i) => s + i.price * i.qty, 0));
  tax       = computed(() => Math.round(this.subtotal() * 0.10 * 100) / 100);
  total     = computed(() => this.subtotal() + this.tax());
  isEmpty   = computed(() => this._items().length === 0);

  constructor() {
    // Side effect: persist to localStorage whenever items change
    effect(() => {
      localStorage.setItem('cart', JSON.stringify(this._items()));
    });
  }

  addItem(item: Omit<CartItem, 'qty'>) {
    this._items.update(items => {
      const existing = items.find(i => i.id === item.id);
      if (existing) return items.map(i => i.id === item.id ? { ...i, qty: i.qty + 1 } : i);
      return [...items, { ...item, qty: 1 }];
    });
  }

  updateQty(id: string, qty: number) {
    if (qty <= 0) { this.removeItem(id); return; }
    this._items.update(items => items.map(i => i.id === id ? { ...i, qty } : i));
  }

  removeItem(id: string) {
    this._items.update(items => items.filter(i => i.id !== id));
  }

  clear() { this._items.set([]); }

  private loadFromStorage(): CartItem[] {
    try {
      const saved = localStorage.getItem('cart');
      return saved ? JSON.parse(saved) : [];
    } catch { return []; }
  }
}
```

#### Step 3 — Cart Component
```typescript
// cart.component.ts
import { Component, inject } from '@angular/core';
import { NgFor, NgIf, CurrencyPipe } from '@angular/common';
import { CartStoreService } from '../cart-store.service';

const PRODUCTS = [
  { id: 'p1', name: 'Angular T-Shirt', price: 29.99 },
  { id: 'p2', name: 'RxJS Hoodie', price: 49.99 },
  { id: 'p3', name: 'Signal Mug', price: 14.99 },
];

@Component({
  standalone: true,
  selector: 'app-cart',
  imports: [NgFor, NgIf, CurrencyPipe],
  template: `
    <section>
      <h2>Products</h2>
      <div *ngFor="let p of products">
        {{ p.name }} — {{ p.price | currency }}
        <button (click)="cart.addItem(p)">Add to Cart</button>
      </div>
    </section>

    <section>
      <h2>Cart ({{ cart.itemCount() }} items)</h2>
      <p *ngIf="cart.isEmpty()">Cart is empty</p>
      <div *ngFor="let item of cart.items()">
        {{ item.name }} x
        <input type="number" [value]="item.qty"
               (change)="cart.updateQty(item.id, +$any($event.target).value)"
               min="0" style="width:50px">
        = {{ item.price * item.qty | currency }}
        <button (click)="cart.removeItem(item.id)">×</button>
      </div>
      <hr>
      <p>Subtotal: {{ cart.subtotal() | currency }}</p>
      <p>Tax (10%): {{ cart.tax() | currency }}</p>
      <strong>Total: {{ cart.total() | currency }}</strong>
      <button (click)="cart.clear()">Clear Cart</button>
    </section>
  `,
})
export class CartComponent {
  cart = inject(CartStoreService);
  products = PRODUCTS;
}
```
