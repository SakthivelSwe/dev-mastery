## WHY
For the first 7 years of Angular's existence, every single Component, Directive, and Pipe you wrote had to be registered inside an `NgModule`.
`NgModules` were massive, confusing arrays of `declarations`, `imports`, and `providers`. They created a steep learning curve, made Lazy Loading incredibly convoluted, and caused frustrating errors like *"Can't bind to 'ngIf' since it isn't a known property"* (because you forgot to import `CommonModule`).
In Angular 14, the Angular Team introduced **Standalone Components**. It completely eliminates the need for `NgModules`. Components are now completely independent entities that directly import exactly what they need, bringing Angular's architecture in line with modern frameworks like React and Vue!

## THEORY
### The Paradigm Shift
- **The Old Way (Module-based):** A Component cannot exist on its own. It belongs to an `NgModule`. If Component A wants to use Component B, the Module owning Component A must import the Module owning Component B.
- **The New Way (Standalone):** The Component *is* the module. If Component A wants to use Component B, Component A simply imports Component B directly in its `@Component` decorator!

*Note: As of Angular 19, Standalone Components are the default. `NgModules` are officially considered legacy architecture.*

## CODE

### Level 1 — The Standalone Flag
To make a component standalone, you set `standalone: true` in the decorator.
(If you are using Angular 19+, this flag is true by default and can be omitted!).

```typescript
// button.component.ts
import { Component } from '@angular/core';

@Component({
  selector: 'app-button',
  standalone: true, // MAGIC: No NgModule required!
  template: `<button>Click Me</button>`
})
export class ButtonComponent {}
```

### Level 2 — Importing Dependencies
If our `CardComponent` wants to use the `ButtonComponent` and the `NgIf` directive, it imports them directly into its own decorator!

```typescript
// card.component.ts
import { Component } from '@angular/core';
import { CommonModule } from '@angular/common'; // Contains *ngIf
import { ButtonComponent } from './button.component'; // Import the Standalone Component

@Component({
  selector: 'app-card',
  standalone: true,
  // Declare exactly what THIS component needs to function!
  imports: [CommonModule, ButtonComponent], 
  template: `
    <div class="card" *ngIf="isVisible">
      <h2>Title</h2>
      <!-- We can use the button because we imported it! -->
      <app-button></app-button> 
    </div>
  `
})
export class CardComponent {
  isVisible = true;
}
```

### Level 3 — Bootstrapping the Application
Without an `AppModule`, how do you start the application?
You use `bootstrapApplication` in your `main.ts` file, passing in your root Standalone Component and a global configuration object!

```typescript
// main.ts
import { bootstrapApplication } from '@angular/platform-browser';
import { provideRouter } from '@angular/router';
import { provideHttpClient } from '@angular/common/http';
import { AppComponent } from './app/app.component'; // Standalone!
import { routes } from './app/app.routes';

// Start the app directly with the Component!
bootstrapApplication(AppComponent, {
  providers: [
    // Global Services and Configs go here!
    provideRouter(routes),
    provideHttpClient()
  ]
}).catch(err => console.error(err));
```

### Level 4 — Lazy Loading Made Trivial
In the `NgModule` era, Lazy Loading a single component was complex; you had to lazy-load an entire Module.
With Standalone Components, Lazy Loading a specific component is a one-liner!

```typescript
// app.routes.ts
import { Routes } from '@angular/router';

export const routes: Routes = [
  {
    path: 'dashboard',
    // MAGIC: The router dynamically imports the Standalone Component over the network 
    // ONLY when the user navigates to the /dashboard URL!
    loadComponent: () => import('./dashboard.component').then(m => m.DashboardComponent)
  }
];
```

## REAL_WORLD
**Migrating Legacy Codebases**
If your company has a massive Angular 13 codebase with 500 NgModules, do you have to rewrite the entire application to upgrade?
No! Standalone Components and NgModules are 100% interoperable.
- A Standalone Component can import an old `NgModule` (e.g., `imports: [MatButtonModule]`).
- An old `NgModule` can import a modern Standalone Component!
The Angular CLI even provides an automated schematic to migrate your entire codebase:
Run `ng generate @angular/core:standalone` in your terminal, and Angular will mathematically analyze your codebase, rip out all the NgModules, and convert every component to Standalone automatically!

## INTERVIEW
**Q: You convert a Component to `standalone: true`. Inside its template, you use the `| date` pipe and the `*ngIf` directive. The compiler throws an error: *"The pipe 'date' could not be found"*. You fix this by adding `imports: [CommonModule]`. A Senior Engineer reviews your PR and tells you that importing `CommonModule` is a bad practice in modern Angular. Why? What should you import instead?**
A: **The Bundle Size problem.**
`CommonModule` is a massive legacy module. It contains `NgIf`, `NgFor`, `NgSwitch`, the `DatePipe`, the `CurrencyPipe`, and a dozen other things.
If your Standalone Component only needs the `DatePipe`, importing `CommonModule` forces the bundler to evaluate and potentially include a lot of unnecessary code. The entire purpose of Standalone architecture is granular, explicit dependencies.
*The Fix:* Standalone architecture applies to Directives and Pipes as well! You should import EXACTLY what you need directly from `@angular/common`:
`imports: [NgIf, DatePipe]`
*Note:* In Angular 17+, `NgIf` and `NgFor` are obsolete anyway, replaced by the new native `@if` and `@for` control flow blocks, which require zero imports!
