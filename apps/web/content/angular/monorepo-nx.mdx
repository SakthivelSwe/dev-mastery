## WHY
In an enterprise environment, you rarely build just one Angular application. You might have an `Admin Dashboard`, a `Customer Portal`, and a `Backend API` (Node.js/NestJS). 
If you create three separate Git repositories, sharing code becomes a nightmare. If you write a generic `ButtonComponent` or a `User` interface, you have to publish it as a private NPM package just to share it between your own apps.
A **Monorepo** is a single Git repository that contains *multiple* distinct applications and libraries. **Nx** is a build system engineered by former Google Angular team members that supercharges the Angular CLI, allowing you to manage massive Monorepos with incredible speed and caching.

## THEORY
### Polyrepo vs Monorepo
- **Polyrepo (Multiple Repos):** Every app has its own repo, its own `package.json`, its own CI/CD pipeline, and its own version of Angular. Upgrading Angular requires opening 10 pull requests across 10 repos.
- **Monorepo:** One repo. One `package.json`. One `node_modules` folder. One version of Angular across the entire company. You update Angular once, and all 10 apps are upgraded simultaneously. Code sharing is as easy as `import { Button } from '@myorg/ui-shared'`.

## CODE

### Level 1 — Creating an Nx Workspace
You do not use the standard `ng new` command to create an Nx monorepo. You use the `create-nx-workspace` command.

```bash
npx create-nx-workspace@latest my-org
```
The resulting folder structure looks completely different from a standard Angular app:
```text
my-org/
├── apps/
│   ├── admin-dashboard/     (Angular App)
│   ├── customer-portal/     (Angular App)
│   └── api/                 (NestJS Backend)
├── libs/
│   ├── shared-ui/           (Buttons, Modals)
│   └── data-access/         (HTTP Services, State)
├── package.json             (Only ONE for the whole org!)
└── nx.json                  (Global Nx configuration)
```

### Level 2 — Libraries and Code Sharing
In a Monorepo, your `apps/` folders should be almost empty! They should only contain routing logic. **80% of your code should live in the `libs/` folder.**
To create a shared UI library:
```bash
nx generate @nx/angular:library shared-ui
```
This creates `libs/shared-ui`. You build your `ButtonComponent` inside it.
Nx automatically updates your global `tsconfig.base.json` to map a beautiful import path!

```json
"compilerOptions": {
  "paths": {
    "@my-org/shared-ui": ["libs/shared-ui/src/index.ts"]
  }
}
```

Now, the `admin-dashboard` and the `customer-portal` can BOTH import the button using the exact same path, without publishing to NPM!
```typescript
import { ButtonComponent } from '@my-org/shared-ui';
```

### Level 3 — The Dependency Graph
Because Nx analyzes your code at the AST (Abstract Syntax Tree) level, it knows exactly which apps import which libraries.
If you run:
```bash
nx graph
```
Nx opens a beautiful interactive visualization in your browser showing the architecture of your entire company! It shows `admin-dashboard` pointing to `shared-ui` and `data-access`.

### Level 4 — Computation Caching (The Nx Magic)
Imagine a massive Monorepo with 50 apps and 200 libraries. If you type `nx build admin-dashboard`, it might take 10 minutes.
If you immediately type `nx build admin-dashboard` again, it takes **0.5 seconds**.
How? **Computation Caching.**
When you run a command (build, test, lint), Nx looks at the physical source code of the app and all its imported libraries. It generates a cryptographic hash of that code. It runs the 10-minute build, and saves the output into a local cache folder named after that hash.
When you run the build again, Nx recalculates the hash. Since you didn't change any files, the hash is identical! Nx completely skips the Angular compiler and instantly copies the files from the cache!

## REAL_WORLD
**Nx Cloud (Distributed Caching)**
Local caching is great, but what about your CI/CD pipeline? When a developer pushes code to GitHub, the CI server usually builds everything from scratch, taking 20 minutes.
**Nx Cloud** allows you to share your Computation Cache with your entire team and your CI pipeline over the internet.
1. Developer A builds `admin-dashboard` locally. It takes 10 minutes. Nx uploads the cached result to Nx Cloud.
2. Developer B pulls the code and runs `nx build admin-dashboard`. It takes 0.5 seconds, because it downloads the cache from the cloud!
3. Developer A pushes to GitHub. The CI server runs `nx test admin-dashboard`. It takes 0.5 seconds, because the CI server downloads the cache from the cloud!
This fundamentally changes Enterprise DevOps, reducing CI bills by tens of thousands of dollars.

## INTERVIEW
**Q: You have a Monorepo with 10 Angular Apps. You modify a single file inside `libs/shared-ui/button.component.ts`. You push the code to GitHub. Your CI pipeline runs `nx test`. Instead of testing all 10 apps, Nx magically only runs the tests for 3 specific apps, and skips the other 7. How did Nx know exactly which apps to test?**
A: **Nx Affected.**
Because Nx maintains an internal Dependency Graph of your entire architecture, it mathematically calculates exactly what your code change "affected".
When you run `nx affected:test`, Nx looks at the Git diff (comparing your PR to the `main` branch). It sees you changed `shared-ui`.
It looks at the Dependency Graph and traces the arrows backwards. It sees that `App 1`, `App 2`, and `App 3` import `shared-ui`. It sees that the other 7 apps *do not* import `shared-ui`.
Therefore, it is mathematically impossible for your code change to have broken the other 7 apps. Nx automatically ignores them, saving massive amounts of CI compute time!
