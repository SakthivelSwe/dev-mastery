## WHY
In a traditional website, clicking a link like `<a href="/about">About</a>` causes the browser to completely destroy the current page, make a round-trip HTTP request to the server, download a new HTML file, and re-render the entire browser window from scratch. This is slow and jarring.

Angular is a **Single Page Application (SPA)** framework. When the user navigates, the browser does not refresh. Instead, Angular intercepts the URL change, looks at its internal mapping rules, and seamlessly swaps out the center Component on the screen, leaving the Navbar and Footer completely untouched. This makes the application feel as fast and smooth as a native mobile app.

## THEORY
### The Router Architecture
Angular's router relies on three primary directives/services:
1. **The Route Configuration**: An array of objects defining which URL string matches which Component.
2. **`RouterOutlet`**: A physical placeholder tag in your HTML (`<router-outlet>`). This acts as the "hole" where the Router dynamically injects the chosen Component.
3. **`RouterLink`**: The Angular replacement for the standard `<a href>` tag. It changes the URL in the browser without triggering a full page refresh.

## CODE

### Level 1 — Beginner (Basic Routing Setup)
In a modern Standalone application, you configure the router in `app.routes.ts` and provide it in `main.ts`.

```typescript
// 1. app.routes.ts
import { Routes } from '@angular/router';
import { HomeComponent } from './home/home.component';
import { AboutComponent } from './about/about.component';
import { NotFoundComponent } from './not-found/not-found.component';

export const routes: Routes = [
  // If the user goes to "mysite.com/home", render the HomeComponent
  { path: 'home', component: HomeComponent },
  
  // If they go to "mysite.com/about", render the AboutComponent
  { path: 'about', component: AboutComponent },
  
  // Default Redirect: If they go to exactly "mysite.com/", redirect to "/home"
  { path: '', redirectTo: '/home', pathMatch: 'full' },
  
  // Wildcard Fallback: If they type complete gibberish, render the 404 Component
  { path: '**', component: NotFoundComponent }
];
```

```html
<!-- 2. app.component.html -->

<nav>
  <!-- DO NOT USE href="". It will force a hard page refresh! -->
  <!-- Use routerLink instead! -->
  <a routerLink="/home">Home</a>
  <a routerLink="/about">About</a>
</nav>

<main>
  <!-- MAGIC: When the URL is '/about', Angular physically injects 
       the <app-about> component right here! -->
  <router-outlet></router-outlet>
</main>
```

### Level 2 — Building Blocks (Route Parameters)
How do you handle dynamic URLs, like viewing a specific user profile: `/users/123`? You use Route Parameters (`:id`).

```typescript
// In app.routes.ts
export const routes: Routes = [
  // The colon indicates that "id" is a dynamic variable!
  { path: 'users/:id', component: UserProfileComponent }
];
```

```typescript
// In user-profile.component.ts
import { Component, inject, OnInit } from '@angular/core';
import { ActivatedRoute } from '@angular/router';

@Component({
  template: `<h1>Viewing User ID: {{ userId }}</h1>`
})
export class UserProfileComponent implements OnInit {
  
  // We inject the ActivatedRoute service to read the URL!
  private route = inject(ActivatedRoute);
  userId: string | null = null;

  ngOnInit() {
    // 1. We subscribe to the paramMap Observable to get the 'id' parameter!
    this.route.paramMap.subscribe(params => {
      this.userId = params.get('id');
      console.log('Fetching user data for ID:', this.userId);
    });
  }
}
```

### Level 3 — Intermediate (Programmatic Navigation)
You don't always use `<a routerLink>`. Sometimes you need to trigger a navigation *after* an action completes (e.g., clicking a "Submit" button, saving to the database, and *then* redirecting).

```typescript
import { Component, inject } from '@angular/core';
import { Router } from '@angular/router';

@Component({
  template: `<button (click)="saveAndRedirect()">Save</button>`
})
export class EditorComponent {
  
  // We inject the Router service!
  private router = inject(Router);

  saveAndRedirect() {
    // 1. Save data to database...
    
    // 2. Programmatically redirect the user!
    this.router.navigate(['/users', '123']); 
    // ^ This compiles to "/users/123"
  }
}
```

### Level 4 — Advanced (Route Guards)
What if the user manually types `/admin-dashboard` into the URL bar, but they are not logged in? You must protect the route using a **Route Guard**.

In modern Angular, Guards are simply functions!

```typescript
import { inject } from '@angular/core';
import { Router, CanActivateFn } from '@angular/router';
import { AuthService } from './auth.service';

// 1. Create a function that returns true (allow) or false (block)
export const authGuard: CanActivateFn = (route, state) => {
  const authService = inject(AuthService);
  const router = inject(Router);

  if (authService.isLoggedIn()) {
    return true; // Let them through!
  } else {
    // Kick them out to the login page!
    router.navigate(['/login']);
    return false;
  }
};
```

```typescript
// 2. Attach the Guard to the route configuration
export const routes: Routes = [
  { 
    path: 'admin-dashboard', 
    component: AdminComponent,
    canActivate: [authGuard] // MAGIC: Runs the function before rendering!
  }
];
```

## REAL_WORLD
When building massive Enterprise Dashboards, developers use **Child Routes (Nested Routing)**. 

Imagine you go to `/settings/profile`. You don't want to destroy the main Application Navbar. Furthermore, you don't want to destroy the Settings Sidebar! You only want to change the inner content pane.

You solve this by placing a `<router-outlet>` *inside* another Component that already contains a `<router-outlet>`!

```typescript
const routes: Routes = [
  {
    path: 'settings',
    component: SettingsWrapperComponent, // Contains the Settings Sidebar + a 2nd RouterOutlet
    children: [
      { path: 'profile', component: ProfileEditorComponent },
      { path: 'billing', component: BillingComponent }
    ]
  }
];
```

## INTERVIEW
**Q: Why do we subscribe to `paramMap` instead of just reading `route.snapshot.paramMap.get('id')`?**
A: Angular aggressively reuses Components to save memory. If you are on `/users/1` and you click a link to `/users/2`, Angular *does not destroy* the `UserProfileComponent` and *does not run* `ngOnInit()` again! It keeps the exact same Component instance on the screen and simply updates the URL. If you read the static `snapshot` in `ngOnInit`, your component will be permanently stuck showing User 1. By subscribing to the `paramMap` Observable, your component instantly reacts to the URL change and fetches User 2's data.

**Q: What is `routerLinkActive`?**
A: It is a brilliant built-in directive used for navigation menus. If you write `<a routerLink="/about" routerLinkActive="active-tab">`, Angular automatically attaches the CSS class `active-tab` to the link *only* when the user is currently on the `/about` page. It automatically removes the class when they navigate away.

**Q: What is the `canDeactivate` Guard used for?**
A: `canActivate` blocks users from *entering* a route. `canDeactivate` blocks users from *leaving* a route. It is almost exclusively used to intercept a user trying to navigate away from a form with unsaved changes, allowing you to pop up a warning: "You have unsaved changes! Are you sure you want to leave?"
