## WHY
The basic MongoDB Aggregation Pipeline (`$match`, `$group`, `$sort`) is fantastic for generating simple reports. However, what if you need to execute a complex SQL-style `LEFT OUTER JOIN` to merge data from two different collections? What if you need to pivot arrays into separate documents, or dynamically restructure the exact shape of the output JSON based on conditional logic?

To replace a complex SQL database, you must master the advanced stages of the MongoDB Aggregation Pipeline.

## THEORY
### Data Transformation
A standard `find()` query cannot change the structure of the document. If a document looks like `{ a: 1, b: 2 }`, a `find()` query can only return that exact shape.
The Aggregation Pipeline is a **Data Transformer**. By the time a document reaches the end of the pipeline, it might look absolutely nothing like the original document stored in the database. It can have new calculated fields, arrays unwound into individual objects, and data merged from other collections.

## CODE

### Level 1 — Beginner ($lookup for Joins)
Because MongoDB is a NoSQL database, data is often denormalized. However, there are times when you absolutely must store data in separate collections (e.g., `Users` and `Orders`) and join them together.

The `$lookup` stage performs a Left Outer Join.

```javascript
db.users.aggregate([
    // 1. Find Alice
    { $match: { name: "Alice" } },
    
    // 2. JOIN with the Orders collection
    {
        $lookup: {
            from: "orders",           // The collection to join with
            localField: "_id",        // The primary key on the User document
            foreignField: "userId",   // The foreign key on the Order document
            as: "userOrders"          // The name of the new ARRAY field to create
        }
    }
]);
/* Result:
{
    "_id": 1,
    "name": "Alice",
    // ALL of Alice's orders are embedded directly into a massive array!
    "userOrders": [
        { "_id": 100, "userId": 1, "total": 50 },
        { "_id": 101, "userId": 1, "total": 75 }
    ]
}
*/
```

### Level 2 — Building Blocks ($unwind)
The `$lookup` stage above created an array of orders (`userOrders`). But what if you want to perform a grouping operation on those orders? You can't easily group an array! 
You must "flatten" the array using `$unwind`.

`$unwind` takes a document with an array and clones the document for *every single item* in the array.

```javascript
db.users.aggregate([
    { $match: { name: "Alice" } },
    { $lookup: { from: "orders", localField: "_id", foreignField: "userId", as: "userOrders" } },
    
    // 3. Deconstruct the array!
    { $unwind: "$userOrders" }
]);
/* Result: Notice we now have TWO separate documents for Alice!
{ "_id": 1, "name": "Alice", "userOrders": { "_id": 100, "total": 50 } },
{ "_id": 1, "name": "Alice", "userOrders": { "_id": 101, "total": 75 } }
*/
```
Now that the orders are flat objects, you can pipe them into a `$group` stage to easily calculate the total revenue generated by Alice!

### Level 3 — Intermediate ($project with Conditional Logic)
The `$project` stage allows you to completely reshape the document. You can concatenate strings, perform math, and execute `if/else` logic!

```javascript
db.orders.aggregate([
    {
        $project: {
            _id: 0,
            
            // String Concatenation
            summary: { $concat: ["Order #", { $toString: "$_id" }] },
            
            // Math
            finalPrice: { $subtract: ["$total", "$discount"] },
            
            // Conditional Logic (If / Then / Else)
            statusDescription: {
                $cond: {
                    if: { $eq: ["$status", "DELIVERED"] },
                    then: "Package has arrived!",
                    else: "Still in transit."
                }
            }
        }
    }
]);
```

### Level 4 — Advanced ($facet for Multi-Faceted Search)
Imagine you are building the sidebar filter on an Amazon search results page.
You search for "Laptop". The sidebar needs to show three different summaries:
1. Count of laptops by Brand.
2. Count of laptops by Price Range.
3. Count of laptops by Rating.

Running 3 separate aggregation pipelines is extremely slow. The `$facet` stage allows you to run multiple independent sub-pipelines on the exact same set of documents in parallel!

```javascript
db.products.aggregate([
    { $match: { category: "Laptop" } },
    
    // Branch off into 3 parallel pipelines!
    {
        $facet: {
            "byBrand": [
                { $group: { _id: "$brand", count: { $sum: 1 } } }
            ],
            "byRating": [
                { $match: { rating: { $gte: 4 } } },
                { $count: "highlyRatedCount" }
            ],
            "topExpensive": [
                { $sort: { price: -1 } },
                { $limit: 3 }
            ]
        }
    }
]);
```
This returns a single massive JSON object containing the results of all 3 distinct reports simultaneously, massively reducing database round-trips!

## REAL_WORLD
**Performance and the 100MB RAM Limit**
The Aggregation Pipeline does all of its sorting and grouping in RAM. MongoDB imposes a strict hard limit: an aggregation pipeline stage is only allowed to consume **100MB of RAM**. If a `$group` or `$sort` stage exceeds 100MB, the query immediately crashes with an error.

If you are grouping millions of documents, you must tell MongoDB to spill the excess data to the physical hard drive. You do this by passing the `{ allowDiskUse: true }` option to your aggregation command. It will be much slower, but it will successfully finish instead of crashing!

## INTERVIEW
**Q: I have a massive pipeline: `$match -> $lookup -> $unwind -> $match`. Is this efficient?**
A: **No.** The most critical rule of Aggregation Pipeline tuning is that **`$match` stages must go absolutely first**. 
If your first stage is a `$lookup`, MongoDB joins every single document in the massive collection, builds huge arrays, unwinds them, and *only then* filters them down. This is devastatingly slow. You must aggressively filter the data using `$match` as early as possible so subsequent heavy stages (like joins or sorts) only process a tiny handful of documents.
