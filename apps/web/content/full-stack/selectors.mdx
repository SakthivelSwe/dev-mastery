# CSS Selectors — Combinators, Pseudo-classes, :has() | Level: 1

## 1. Advanced Selector Specifications
Selectors identify the DOM nodes to which style rules apply. Modern CSS engines parse selectors from right to left (the key selector is evaluated first, then the engine climbs up the DOM tree to verify matches).

```
   [Parent] ──(Descendant: " ")──> [Child]
   [Parent] ──(Child: ">")───────> [Direct Child]
   [Node A] ──(Adjacent: "+")────> [Immediate Next Sibling]
   [Node A] ──(General: "~")─────> [Any Next Sibling]
```

### Combinator Mechanics
* **Descendant Selector (`A B`)**: Matches any element `B` that is a descendant of `A`, regardless of depth.
* **Child Selector (`A > B`)**: Matches only elements `B` that are direct children of `A`.
* **Adjacent Sibling Selector (`A + B`)**: Matches element `B` only if it is the immediate sibling following `A`.
* **General Sibling Selector (`A ~ B`)**: Matches all elements `B` that follow `A` as siblings, regardless of whether they are immediately adjacent.

---

## 2. Attribute Matchers & Pseudo-classes
Attribute selectors match elements based on the presence or value of attributes.

* `[attr]`: Presence of attribute.
* `[attr="val"]`: Exact equality.
* `[attr^="start"]`: Starts with prefix.
* `[attr$="end"]`: Ends with suffix.
* `[attr*="contains"]`: Substring match anywhere within the value.

### Pseudo-classes vs Pseudo-elements
* **Pseudo-classes** (single colon `:`): Represent state or structural positions of elements (e.g., `:hover`, `:focus-visible`, `:nth-child(2n)`).
* **Pseudo-elements** (double colon `::`): Generate virtual elements that are not part of the standard DOM tree (e.g., `::before`, `::after`, `::placeholder`, `::selection`). The `content` property is required for `::before` and `::after` to render.

## 3. Modern Functional Selectors: `:is()`, `:where()`, and `:has()`

### Specificity Differences: `:is()` vs `:where()`
Both selectors allow you to write compact selector groups, but they handle specificity differently:
* **`:is()`**: The specificity of the entire `:is()` functional selector is determined by the **most specific** selector in its argument list.
* **`:where()`**: The specificity of `:where()` is **always zero**, regardless of the arguments passed to it.

```css
/* Specificity: (0, 1, 0, 0) because #hero is an ID selector */
:is(.card, #hero) p {
  color: oklch(0.2 0.01 240);
}

/* Specificity: (0, 0, 0, 1) because :where() has zero specificity and p is a type selector */
:where(.card, #hero) p {
  color: oklch(0.2 0.01 240);
}
```

### The Parent Selector: `:has()`
The `:has()` relational pseudo-class allows styling an element based on its descendants or subsequent siblings. This acts as a parent selector.

```css
/* Style a form wrapper only if it contains a focused input */
.form-group:has(input:focus) {
  border-color: oklch(0.6 0.25 140);
  background-color: oklch(0.98 0.01 240);
}

/* Style an H2 element only if it is immediately followed by a paragraph */
h2:has(+ p) {
  margin-bottom: 0.5rem;
}
```

## 5. DevTools Diagnostics & Profiling

1. Open DevTools and inspect a `.card` element.
2. In the **Styles** pane, verify if the `:has()` or `:is()` selector is successfully matching. 
3. If an invalid selector is part of a standard selector list (e.g., `h1, h2, :invalid-pseudo`), the entire rule is discarded. However, if wrapped inside `:is()` or `:where()`, the selector list is **forgiving**—the browser ignores the invalid selector and still applies styles to the valid ones.
4. To debug dynamic pseudo-classes like `:focus` or `:hover`, click the **`:hov`** button in the Styles pane to force element states.

## 6. Performance & Compatibility

### Selector Performance
The `:has()` selector was historically avoided due to performance concerns in rendering engines. Modern browsers utilize invalidation filters to fast-path matching. This ensures that updates to the DOM only trigger style recalculations on elements that could match. 
* Avoid deep, complex combinations inside `:has()`, such as `*:has(div span a)`. This can degrade scrolling performance and increase style recalculation times.

### Browser Compatibility
* **`:is()` and `:where()`**: Supported in all modern browsers (since early 2021). ~97% global support.
* **`:has()`**: Supported in all evergreen browsers (Chrome 105+, Safari 15.4+, Firefox 121+). Global support is ~93%.
* **caniuse reference**: [caniuse.com/#feat=css-has](https://caniuse.com/#feat=css-has)

## WHY
CSS (Cascading Style Sheets) is useless if it doesn't know *which* HTML elements to style. **Selectors** are the targeting system of CSS. They allow you to apply rules to a single, specific button on the page, or to every single paragraph inside a specific article, without affecting anything else. Mastering selectors is the first step to writing clean, maintainable CSS that doesn't accidentally break other parts of your website.

## THEORY
### The Cascade and Specificity
If you write a rule saying "All buttons are blue", and another rule saying "The login button is red", which one wins? 
CSS uses a mathematical scoring system called **Specificity**. 
- Element selectors (e.g., `button`) are worth **1 point**.
- Class selectors (e.g., `.btn`) are worth **10 points**.
- ID selectors (e.g., `#login-btn`) are worth **100 points**.

The rule with the highest score always wins, regardless of which order they were written in the file!

## CODE

### Level 1 — Beginner (Elements, Classes, and IDs)
The three foundational ways to target HTML.

```css
/* 1. Element Selector (Targets ALL <p> tags on the entire page) */
/* Score: 1 */
p {
  color: black;
}

/* 2. Class Selector (Targets elements with class="error") */
/* Prefix with a DOT (.) */
/* Score: 10 */
.error {
  color: red;
  font-weight: bold;
}

/* 3. ID Selector (Targets the ONE element with id="submit-btn") */
/* Prefix with a HASH (#). IDs must be absolutely unique on the page! */
/* Score: 100 */
#submit-btn {
  background-color: blue;
}
```

### Level 2 — Building Blocks (Combinators)
What if you only want to target `<p>` tags that are *inside* a specific `<article>`? You combine selectors!

```css
/* Descendant Combinator (Space) */
/* Targets ANY <p> that is inside the article, no matter how deeply nested it is. */
article p {
  line-height: 1.5;
}

/* Child Combinator (>) */
/* Targets ONLY the <p> tags that are DIRECT children of the article. */
/* If a <p> is inside a <div> inside the article, it is ignored! */
article > p {
  font-size: 18px;
}

/* Chaining (No Space) */
/* Targets a button that ALSO has the 'primary' class. (<button class="primary">) */
button.primary {
  border: 2px solid blue;
}
```

### Level 3 — Intermediate (Pseudo-classes)
Pseudo-classes allow you to style an element based on its **State** (like when the user hovers over it) or its **Position** in the HTML.

```css
/* State: When the user hovers their mouse over the link */
a:hover {
  text-decoration: underline;
}

/* State: When a user clicks and holds the button */
button:active {
  transform: scale(0.95); /* Makes it look pressed down! */
}

/* Position: Targets the VERY FIRST <li> in a list */
li:first-child {
  border-top: none;
}

/* Position: Targets every alternating row in a table! (Zebra striping) */
tr:nth-child(even) {
  background-color: #f2f2f2;
}
```

### Level 4 — Advanced (Attribute Selectors)
You can target elements based on their exact HTML attributes! This is incredibly powerful for styling forms.

```css
/* Target ONLY the input fields where type="text" */
input[type="text"] {
  border: 1px solid gray;
}

/* Target ANY element that has a 'disabled' attribute */
[disabled] {
  opacity: 0.5;
  cursor: not-allowed;
}

/* Target links that go to external websites (href starts with 'https') */
a[href^="https"] {
  color: green;
}
```

## REAL_WORLD
**The `!important` Anti-Pattern**
Imagine you add a CSS class to a button: `.btn-red { color: red; }`, but the button stays blue. You inspect it and realize a 3rd-party library has an ID selector (`#nav-btn { color: blue; }`). The ID has a specificity score of 100, and your class only has a score of 10. The library wins.

Frustrated, you write `.btn-red { color: red !important; }`. It works! 
`!important` overrides the entire mathematical specificity system, forcing the rule to win. 
**Never do this.** 
If you sprinkle `!important` throughout your codebase, you completely destroy the "Cascading" part of CSS. When you later need to override the `!important` rule, you are forced to write an even crazier selector. Instead, you should simply increase the specificity of your selector naturally (e.g., `#nav-btn.btn-red { color: red; }` - Score: 110).

## INTERVIEW
**Q: I have this HTML: `<div class="box"></div>`. I wrote this CSS: `.Box { background: blue; }`. Why isn't the box turning blue?**
A: **CSS Selectors are case-sensitive!** (Or more accurately, the HTML attributes they target, like `class` and `id`, are case-sensitive). The HTML class is `box` (lowercase), but the CSS selector is `.Box` (capital B). They do not match. Always use strictly lowercase letters and hyphens (kebab-case) for CSS class names (e.g., `nav-button`, not `navButton` or `Nav_Button`) to prevent these maddening bugs.

## FEYNMAN CHECK

### Explain CSS Selectors Like I'm 10 Years Old
> Selectors are the QUERIES of CSS — they ask the browser "find every element matching this pattern, and apply these styles to them." `.btn` matches every element with `class="btn"`; `a:hover` matches every link the mouse is currently over; `p > strong` matches `<strong>` that is a DIRECT child of `<p>`. Each selector has a SPECIFICITY score (4-digit number: inline, IDs, classes, elements) that decides which rule wins when two selectors target the same element. This is why `#header .btn` (0,1,1,0) beats `.modal .btn` (0,0,2,0) — and why nobody should ever use IDs in CSS.

### 5 Deep Conceptual Questions

**Q1: What problem do CSS selectors fundamentally solve?**
> **A:** They give CSS its targeting power — one rule applies to many elements without you naming each. They also encode RELATIONSHIPS (descendant, child, sibling, attribute, state) so styling can react to context: a link inside a nav is styled differently from a link inside an article. Without selectors, CSS would degenerate into per-element inline styles, defeating the cascade.

**Q2: What is the ONE mental model that makes selectors click?**
> **A:** Selectors are matched RIGHT-TO-LEFT, not left-to-right. `nav ul li a` doesn't mean "find all navs, then their uls, then..." — it means "find every `<a>`, then check if its ancestor chain includes `li > ul > nav`." This is why ".btn" is fast but "div div div .btn" is slow on large pages: the browser must walk up the ancestor chain for every potential match.

**Q3: What is the most dangerous misconception about selectors? Show it with code.**
> **A:** That nesting selectors deeply "scopes" them safely — instead it locks specificity high and makes overrides require even higher specificity:
> ```css
> /* ❌ Specificity (0,0,4,0). Now every override needs ≥4 classes or !important */
> .page .sidebar .widget .title { color: red; }
>
> /* ✅ Use a single class with BEM-style naming. Specificity (0,0,1,0). Easy to override. */
> .widget__title { color: red; }
> ```

**Q4: How does the browser evaluate selectors at the runtime level?**
> **A:** The browser maintains a hash map of rules indexed by their rightmost simple selector (the "key selector") — class names, IDs, tag names. When matching, it looks up only rules whose key matches the element, then walks UP the ancestor chain to test the rest. This is why `* { color: red }` is slow (every element matches the key) and why `:has(.child)` is computationally expensive (it inverts the direction — checking descendants).

**Q5: Write a one-sentence definition of CSS selectors that a senior engineer at a FAANG company would find technically precise.**
> **A:** "CSS selectors are right-to-left-matched patterns over the DOM tree that compose simple selectors (tag, class, id, attribute, pseudo-class) with combinators (descendant, child, adjacent sibling, general sibling) and resolve specificity as a four-tuple (`a, b, c, d`) → first by !important, then by inline, ID, class, element count, then by source order — which is why deeply-nested selectors and ID-based rules create a specificity arms race that makes design-system overrides painful."

#### Step 3 — Specificity Calculator + Live Match Highlighting
```javascript
// app.js
const input = document.getElementById('sel');
const out   = document.getElementById('out');

// Compute specificity as [inline, ids, classes/attrs/pseudo-classes, elements/pseudo-elements]
function specificity(selector) {
  // Strip pseudo-elements (single colon and double colon)
  const s = selector.replace(/::?[\w-]+(\([^)]*\))?/g, '');
  const ids        = (s.match(/#[\w-]+/g)  || []).length;
  const classesEtc = (s.match(/(\.[\w-]+|\[[^\]]+\]|:[\w-]+(\([^)]*\))?)/g) || []).length;
  const elems      = (s.match(/(^|[\s>+~])[a-z][\w-]*/g) || []).length;
  return [0, ids, classesEtc, elems];
}

input.addEventListener('input', () => {
  const sel = input.value.trim();
  document.querySelectorAll('.winner').forEach(el => el.classList.remove('winner'));
  if (!sel) { out.textContent = ''; return; }

  try {
    const matches = document.querySelectorAll(sel);
    matches.forEach(el => el.classList.add('winner'));
    const spec = specificity(sel);
    out.textContent =
      `Selector:    ${sel}\n` +
      `Specificity: (${spec.join(',')})\n` +
      `Matches:     ${matches.length} element(s)\n` +
      Array.from(matches).map((el, i) =>
        `  ${i + 1}. ${el.tagName.toLowerCase()}` +
        (el.id ? '#' + el.id : '') +
        (el.className ? '.' + [...el.classList].filter(c=>c!=='winner').join('.') : '')
      ).join('\n');
  } catch (err) {
    out.textContent = 'Invalid selector: ' + err.message;
  }
});
```
