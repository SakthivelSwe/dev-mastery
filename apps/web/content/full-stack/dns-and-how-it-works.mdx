## WHY
Humans are terrible at memorizing random numbers. If the Domain Name System (DNS) didn't exist, to check your email you would have to open your browser and type `http://142.250.190.69` instead of `google.com`. 

DNS is the phonebook of the Internet. It is a massive, globally distributed database that translates human-readable domain names into machine-readable IP addresses. If DNS goes down, the internet effectively goes down for 99% of users.

## THEORY
### The Hierarchy of the Internet
DNS is not one single server. It is a hierarchical tree of servers delegating responsibility from top to bottom.

1. **Root Servers:** The absolute top of the tree. There are 13 logical Root Servers globally. They don't know the IP of google.com, but they know who is in charge of `.com`.
2. **TLD (Top-Level Domain) Servers:** Responsible for specific extensions (`.com`, `.org`, `.io`). They don't know google.com's IP, but they know who Google's specific servers are.
3. **Authoritative Name Servers:** The servers actually owned or configured by the domain owner (usually hosted by Route53, Cloudflare, or GoDaddy). These servers hold the actual IP address record.

## CODE

### Level 1 — Beginner (The Recursive Lookup)
When you type `devmastery.com` into your browser, how does it find the IP?

```text
1. Browser Cache: Chrome checks its internal memory. Do I already know this IP?
2. OS Cache: The browser asks Windows/macOS. Do you know it?
3. Recursive Resolver (ISP): Your computer asks your Internet Service Provider (like Comcast) or Google's public resolver (8.8.8.8): "Find this IP for me."

The Resolver does the heavy lifting:
4. Resolver -> Root Server: "Where is devmastery.com?"
   Root: "I don't know, but here is the IP for the .com TLD server."
   
5. Resolver -> TLD Server: "Where is devmastery.com?"
   TLD: "I don't know, but here is the Authoritative Server for devmastery (e.g., Cloudflare)."
   
6. Resolver -> Cloudflare Authoritative Server: "Where is devmastery.com?"
   Cloudflare: "I have the record! It is 104.21.55.100."

7. The Resolver saves this IP in its cache, and sends it back to your browser!
```

### Level 2 — Building Blocks (DNS Record Types)
When you buy a domain name, you manage it by creating different types of DNS Records.

```text
A Record (Address)
Maps a domain directly to an IPv4 address.
[devmastery.com] -> [A] -> [104.21.55.100]

AAAA Record (Quad-A)
Maps a domain directly to an IPv6 address.
[devmastery.com] -> [AAAA] -> [2001:0db8:85a3:0000:0000:8a2e:0370:7334]

CNAME (Canonical Name)
Maps a domain (alias) to ANOTHER domain name, never an IP. 
Used for subdomains to point to the main domain.
[www.devmastery.com] -> [CNAME] -> [devmastery.com]

MX Record (Mail Exchange)
Tells the internet which servers handle email for your domain. 
(Crucial if you want to use GSuite or Office365 for your company email).
[devmastery.com] -> [MX] -> [aspmx.l.google.com]

TXT Record (Text)
Stores arbitrary text. Primarily used today to prove domain ownership 
(e.g., adding a Google site verification string) and for email security (SPF, DKIM).
```

### Level 3 — Intermediate (TTL - Time to Live)
DNS is heavily cached at every level (Browser, OS, ISP). How does a cache know when to expire the record?

Every DNS record has a **TTL (Time to Live)** value, usually set in seconds.
- `TTL = 3600` (1 Hour)
- `TTL = 86400` (24 Hours)

If your website is hosted on AWS, and you migrate it to DigitalOcean, you must update your `A Record` to point to the new DigitalOcean IP. 
If your TTL was set to 24 hours, users around the world will continue to be sent to your old AWS server for up to 24 hours until their local ISP cache finally expires and fetches the new IP!

*Pro-Tip: If you plan to migrate a server on Friday, change your TTL to 5 minutes (`300`) on Wednesday. By Friday, everyone's cache will expire every 5 minutes, allowing your migration to be almost instantaneous.*

### Level 4 — Advanced (Anycast and Geo-DNS)
If you have a global application, you don't want users in Japan connecting to an IP address in New York. You want them to connect to a server in Tokyo.

**Geo-DNS (Latency Based Routing)**
When the Japanese user's ISP resolver asks Route53 for `devmastery.com`, Route53 looks at the geographic origin of the request. Instead of returning the New York IP, Route53 intelligently returns the Tokyo IP. The DNS response literally changes based on who is asking!

## REAL_WORLD
**Why does propagating DNS take so long?**
You buy a new domain, update the name servers, and the hosting provider says "Please wait up to 48 hours for DNS to propagate." 
DNS doesn't actually "propagate" or push data across the internet. The delay is entirely due to **Caching**. Thousands of ISPs around the world have cached the old negative responses ("This domain doesn't exist"). You are simply waiting for those millions of caches to naturally expire according to their TTLs so they will finally query the Root servers again and discover your new domain.

## INTERVIEW
**Q: I want `my-app.com` to point to my Heroku app (`my-app.herokuapp.com`). Should I use an A Record or a CNAME?**
A: You must use a CNAME for subdomains (`www.my-app.com` -> CNAME -> `my-app.herokuapp.com`), because Heroku's IP addresses change dynamically and you don't own them. 
However, **the DNS specification forbids placing a CNAME at the root domain level (the apex)**. You cannot put a CNAME on `my-app.com`! 
To solve this, modern DNS providers (like Cloudflare or Route53) invented proprietary records (called ALIAS or CNAME-flattening). You configure it like a CNAME, but when a client queries it, the DNS provider resolves the target domain internally and returns a standard A Record (IP address) to the client, perfectly satisfying the DNS specification.

## FEYNMAN CHECK

### Explain DNS Like I'm 10 Years Old
> DNS is the internet's phone book. When your phone wants to call "google.com", it asks DNS "what's the number?" and DNS replies with an IP like 142.250.80.46. The book is split into branches (.com, .org, .uk) each maintained by different teams, and your local resolver caches answers so it doesn't ask the master books every time. The non-obvious part: a single DNS query can trigger 4 separate lookups (root → TLD → authoritative → answer) on a cold cache, and each is a UDP round-trip that adds latency to your page load.

### 5 Deep Conceptual Questions

**Q1: What is the difference between recursive and authoritative DNS resolvers?**
> **A:** A recursive resolver (your ISP, 8.8.8.8, 1.1.1.1) does the work of walking the DNS tree for you and caches the result. An authoritative resolver owns a specific zone and answers definitively for it (e.g., `ns-1.awsdns.com` is authoritative for many AWS-hosted domains). Your computer talks to recursive; recursive talks to authoritative.

**Q2: ONE mental model.**
> **A:** "DNS = recursive resolver does the legwork by walking root → TLD → authoritative, caching each answer for its TTL. Cold lookup = 4 RTTs; warm = local cache, 0 RTTs."

**Q3: Misconception with code.**
> **A:** Trusting DNS to be instantly consistent globally.
> ```bash
> # ❌ Just changed DNS A record, expect immediate switch
> # Cached resolvers still serve old IP for up to TTL seconds
> # ✅ Lower TTL BEFORE the change, then change, then raise TTL back
> # 24h before change: TTL 60s
> # At change time: update A record
> # 1h after change confirmed: TTL back to 3600s
> ```

**Q4: How does DNS-based load balancing work, and what are its limits?**
> **A:** Return different IPs to different clients (round-robin, geo-based, or weighted). Limit: client-side caching means you can't pull traffic off a bad server faster than the TTL. Real load balancing happens at L4/L7 (AWS ALB, Cloudflare) where servers can be drained in seconds rather than minutes.

**Q5: Senior one-liner.**
> **A:** "DNS is a hierarchical, eventually-consistent, cache-heavy mapping protocol from names to records — its strengths (cache speed, decentralisation) are also its weaknesses (slow propagation, cache poisoning risk), which is why production systems use short TTLs only during migration windows."

#### Step 2 — Core
```js
// analyse.js — measure cold vs warm lookup time
const dns = require('dns').promises;
const { performance } = require('perf_hooks');
async function timeLookup(domain) {
  const t0 = performance.now();
  await dns.resolve4(domain);
  return performance.now() - t0;
}
async function main(domain) {
  console.log(`${domain}: cold = ${(await timeLookup(domain)).toFixed(1)}ms`);
  console.log(`${domain}: warm = ${(await timeLookup(domain)).toFixed(1)}ms`);
  console.log(`${domain}: warm = ${(await timeLookup(domain)).toFixed(1)}ms`);
}
main(process.argv[2] || 'github.com');
```

#### Step 3 — Test Different Resolvers
```bash
# Switch resolver
dns.setServers(['8.8.8.8']);     // Google
dns.setServers(['1.1.1.1']);     // Cloudflare
dns.setServers(['9.9.9.9']);     // Quad9
# Run timing across all three
```

#### Step 4 — Handle Errors
```js
try { await dns.resolve4(domain); }
catch (e) {
  if (e.code === 'ENOTFOUND') console.error('No A record');
  else if (e.code === 'ETIMEOUT') console.error('Resolver timeout');
}
```

#### Step 5 — Tests
