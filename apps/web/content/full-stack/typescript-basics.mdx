## WHY
JavaScript is a loosely typed, dynamic language. If you create a variable `let age = 25`, you can later write `age = "twenty-five"`. JavaScript allows this.
While this flexibility is great for small scripts, it is a catastrophic nightmare for enterprise applications. If an API returns an Object, but you accidentally misspell a property name (`user.firstName` instead of `user.first_name`), JavaScript will silently return `undefined`, and your application will randomly crash in production 3 weeks later.
**TypeScript** is a superset of JavaScript developed by Microsoft. It introduces strict "Static Typing". It mathematically forces you to declare the exact shape of your data. If you misspell a property or pass a String into a function expecting a Number, TypeScript will physically refuse to compile your code, catching the bug *while you are typing in VS Code*, long before the code ever reaches a user!

## THEORY
### The Compiler (TSC)
Browsers do NOT understand TypeScript. Chrome and Firefox can only execute raw JavaScript.
Therefore, you cannot run `.ts` files directly in a browser.
You must run the **TypeScript Compiler (`tsc`)**. The compiler reads your TypeScript code, enforces all the strict rules, throws red error squiggles if you broke a rule, and if everything passes, it strips away all the Types and translates the code into raw, standard JavaScript (`.js`) that the browser can understand!

## CODE

### 1. Basic Types and Inference
You declare types using a colon `:`.

```typescript
// Explicitly typing variables
let firstName: string = "Alice";
let age: number = 30;
let isAdmin: boolean = true;
let ids: number[] = [1, 2, 3, 4, 5];

// Type Inference (The Magic!)
// TypeScript is incredibly smart. You don't have to type everything!
let city = "New York"; // TS knows this is a String instantly.
// city = 42;          // ERROR! TS will block this reassignment!
```

### 2. Typing Functions
You must specify what type of data goes INTO the function (Parameters), and what type of data comes OUT of the function (Return Type).

```typescript
// The inputs must be numbers. The function MUST return a number!
function calculateTotal(price: number, tax: number): number {
    return price + tax;
}

// ERROR! "15" is a string. TypeScript prevents this bug instantly!
// calculateTotal(100, "15"); 
```

### 3. Interfaces (The Shape of Objects)
This is the most powerful feature of TypeScript. An `Interface` allows you to define the exact required structure of complex objects.

```typescript
// Define the Blueprint!
interface UserProfile {
    id: number;
    username: string;
    email: string;
    // The '?' means this property is OPTIONAL.
    phoneNumber?: string; 
}

// Enforce the Blueprint!
const user: UserProfile = {
    id: 1,
    username: "AliceCodes",
    email: "alice@example.com"
    // Valid! phoneNumber is optional.
};

// ERROR! Type '{ id: number; }' is missing the following properties: username, email
// const badUser: UserProfile = { id: 2 }; 

// Auto-Complete Magic!
// If you type `user.`, VS Code will instantly suggest `id`, `username`, and `email`!
console.log(user.username);
```

### 4. Union Types (Allowing Multiple Types)
Sometimes a variable needs to be flexible, but not *infinitely* flexible.

```typescript
// The status can ONLY be one of these three exact strings!
let status: "loading" | "success" | "error";

status = "success"; // Valid!
// status = "failed"; // ERROR! "failed" is not assignable to the union type!

// A variable that can be a number OR a string
let id: number | string;
id = 101;
id = "ABC-101";
```

## REAL_WORLD
**The "Any" Type (The Escape Hatch)**
When migrating a massive, messy 10-year-old JavaScript codebase to TypeScript, it is impossible to perfectly define Interfaces for 5,000 files simultaneously.
TypeScript provides an escape hatch: the `any` type.
```typescript
let mysteryData: any = "Hello";
mysteryData = 42;
mysteryData = { foo: "bar" };
```
Using `any` tells the TypeScript compiler: *"Turn off all rules. Blindly trust me. Let me do whatever I want."*
In the real world, you should **never** use `any` in new code. It completely defeats the entire purpose of using TypeScript. If you don't know the exact shape of data (e.g., fetching a random API payload), use the `unknown` type instead. `unknown` forces you to write `if/else` checks to verify the data type *before* you are allowed to use it!

## INTERVIEW
**Q: An interviewer looks at your TypeScript interface for an API Response. They see `interface User { name: String, age: Number }`. They point out a subtle mistake that proves you don't fully understand TypeScript. What is the mistake?**
A: **Using uppercase `String` and `Number` wrapper objects.**
In TypeScript, `string` (lowercase) and `String` (uppercase) are two completely different things!
- `string` (lowercase) is the primitive JavaScript data type. (e.g., `"Hello"`). This is what you should use 100% of the time!
- `String` (uppercase) refers to the native JavaScript wrapper Object `new String("Hello")`. 
If you type `name: String`, you are accidentally forcing TypeScript to expect the massive Object wrapper, which will cause massive type-checking failures when you try to assign standard primitive strings to it. Always use lowercase primitives (`string`, `number`, `boolean`) in TypeScript definitions!
