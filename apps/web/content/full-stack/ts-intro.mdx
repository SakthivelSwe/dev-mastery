# TypeScript Introduction — Why Types, tsc, tsconfig

TypeScript is a typed superset of JavaScript that compiles to plain JavaScript. It adds optional static typing to the language, enabling developers to catch errors early in the development lifecycle rather than at runtime.

---

## 1. Structural vs. Nominal Typing

Unlike languages like Java or C# which use **nominal typing** (where type compatibility is determined by explicit declarations or class names), TypeScript uses **structural typing** (often called "duck typing"). 

In a structural type system, two types are compatible if their *shapes* are compatible. If an object has all the properties required by a type, TypeScript allows it, regardless of its explicit class or interface declaration.

### TypeScript Code
```typescript
interface Point2D {
  x: number;
  y: number;
}

function logCoordinates(point: Point2D) {
  console.log(`x: ${point.x}, y: ${point.y}`);
}

// Named object matching the shape
const namedPoint = { x: 10, y: 20, label: "origin" };
logCoordinates(namedPoint); // Allowed! Extra properties are ignored.

// Structural compatibility with classes
class Vector2D {
  constructor(public x: number, public y: number) {}
}
logCoordinates(new Vector2D(5, 10)); // Allowed! Shape matches Point2D.
```

### Compiled JavaScript (tsc output)
```javascript
"use strict";
function logCoordinates(point) {
    console.log(`x: ${point.x}, y: ${point.y}`);
}
const namedPoint = { x: 10, y: 20, label: "origin" };
logCoordinates(namedPoint);
class Vector2D {
    constructor(x, y) {
        this.x = x;
        this.y = y;
    }
}
logCoordinates(new Vector2D(5, 10));
```

## 2. Type Erasure: Compile-Time vs. Runtime

TypeScript's type system is completely **erased** during compilation. Types exist solely to assist the compiler (`tsc`) and your IDE in performing static analysis. Once compiled, the generated JavaScript has no awareness of interfaces, type aliases, or generics. 

There is **zero runtime overhead** added by TypeScript types.

### TypeScript Code (Compile-Time)
```typescript
type UserID = string & { readonly __brand: unique symbol };

interface User {
  id: UserID;
  username: string;
}

function greetUser(user: User): string {
  return `Hello, ${user.username}`;
}
```

### Compiled JavaScript (Runtime)
```javascript
"use strict";
// All type definitions, interfaces, and brands have been erased.
function greetUser(user) {
    return `Hello, ${user.username}`;
}
```

## 3. The `tsc` Compiler: Compilation vs. Type Checking

The TypeScript compiler (`tsc`) performs two main duties:
1. **Transpilation**: Down-leveling newer JavaScript/TypeScript syntax to an older target (e.g., ES5/ES6).
2. **Type Checking**: Validating code against type rules.

Crucially, **type checking is separate from compilation**. By default, even if there are type errors, `tsc` will still emit JavaScript files unless the `--noEmitOnError` flag is set.

```bash
# Run type checking and emit files
tsc

# Run type checking only, without emitting files
tsc --noEmit

# Prevent emitting compiled files if any type-check error is found
tsc --noEmitOnError
```

## 4. Configuring `tsconfig.json` with `strict: true`

The `tsconfig.json` file configures how `tsc` compiles your project. Enabling `"strict": true` is the industry standard and activates a suite of type-checking behaviors that guarantee maximum type safety.

### Complete strict-mode `tsconfig.json`
```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "NodeNext",
    "moduleResolution": "NodeNext",
    "lib": ["ES2022", "DOM"],
    "strict": true,
    "noImplicitAny": true,
    "strictNullChecks": true,
    "strictFunctionTypes": true,
    "strictBindCallApply": true,
    "strictPropertyInitialization": true,
    "noImplicitThis": true,
    "useUnknownInCatchVariables": true,
    "alwaysStrict": true,
    "noUnusedLocals": true,
    "noUnusedParameters": true,
    "noImplicitReturns": true,
    "exactOptionalPropertyTypes": true,
    "outDir": "./dist",
    "rootDir": "./src",
    "esModuleInterop": true,
    "forceConsistentCasingInFileNames": true,
    "skipLibCheck": true
  },
  "include": ["src/**/*"]
}
```

### Strict Mode Flags Explained:

*   **`noImplicitAny`**: Raises an error on expressions and declarations with an implied `any` type.
    ```typescript
    // Error: TS7006: Parameter 'x' implicitly has an 'any' type.
    function double(x) { 
      return x * 2;
    }
    ```
*   **`strictNullChecks`**: Ensures `null` and `undefined` are not assignable to other types except when explicitly declared as unions.
    ```typescript
    let name: string = "Alice";
    // Error: TS2322: Type 'null' is not assignable to type 'string'.
    name = null; 
    ```
*   **`strictPropertyInitialization`**: Ensures class properties are initialized in the constructor or have a default value.
    ```typescript
    class Account {
      // Error: TS2564: Property 'balance' has no initializer and is not definitely assigned in the constructor.
      balance: number; 
      constructor() {}
    }
    ```
*   **`useUnknownInCatchVariables`**: Forces the type of error variables in catch blocks to be `unknown` instead of `any`.
    ```typescript
    try {
      throw new Error("Failed");
    } catch (err) {
      // Error: TS18046: 'err' is of type 'unknown'.
      console.log(err.message); 
    }
    ```

## 5. TypeScript Errors vs. Runtime Errors

TypeScript errors are **compile-time** validations that prevent unsafe code from executing. Runtime errors are failures that occur while the engine executes the compiled JavaScript.

### Compile-Time Error Example
```typescript
interface Configuration {
  port: number;
}

const config: Configuration = {
  port: "8080" 
  // Error: TS2322: Type 'string' is not assignable to type 'number'.
};
```

### Runtime Error Prevention
Without TypeScript, passing an unexpected type can cause silent failures or immediate crashes:
```typescript
function uppercaseFirst(arr: string[]) {
  return arr[0].toUpperCase();
}

// TypeScript prevents this call at compile-time:
// Error: TS2345: Argument of type 'undefined[]' is not assignable to parameter of type 'string[]'.
uppercaseFirst([]); // At runtime, this would throw: TypeError: Cannot read properties of undefined (reading 'toUpperCase')
```

## 6. When TypeScript Helps Most

1. **Large-Scale Refactoring**: Renaming properties or changing function signatures across hundreds of files is trivial because the compiler instantly flags all mismatched call sites.
2. **Explicit API Contracts**: Interfaces define strict boundaries for APIs, keeping frontend and backend schemas in sync.
3. **Team Collaboration**: Code serves as self-documenting executable specifications. Developers do not need to guess the shape of data passed into functions.

*Refer to the official [TypeScript Handbook: The Basics](https://www.typescriptlang.org/docs/handbook/2/basic-types.html) for further information.*
