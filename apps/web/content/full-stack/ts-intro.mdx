# TypeScript Introduction — Why Types, tsc, tsconfig

TypeScript is a typed superset of JavaScript that compiles to plain JavaScript. It adds optional static typing to the language, enabling developers to catch errors early in the development lifecycle rather than at runtime.

---

## 1. Structural vs. Nominal Typing

Unlike languages like Java or C# which use **nominal typing** (where type compatibility is determined by explicit declarations or class names), TypeScript uses **structural typing** (often called "duck typing"). 

In a structural type system, two types are compatible if their *shapes* are compatible. If an object has all the properties required by a type, TypeScript allows it, regardless of its explicit class or interface declaration.

### TypeScript Code
```typescript
interface Point2D {
  x: number;
  y: number;
}

function logCoordinates(point: Point2D) {
  console.log(`x: ${point.x}, y: ${point.y}`);
}

// Named object matching the shape
const namedPoint = { x: 10, y: 20, label: "origin" };
logCoordinates(namedPoint); // Allowed! Extra properties are ignored.

// Structural compatibility with classes
class Vector2D {
  constructor(public x: number, public y: number) {}
}
logCoordinates(new Vector2D(5, 10)); // Allowed! Shape matches Point2D.
```

### Compiled JavaScript (tsc output)
```javascript
"use strict";
function logCoordinates(point) {
    console.log(`x: ${point.x}, y: ${point.y}`);
}
const namedPoint = { x: 10, y: 20, label: "origin" };
logCoordinates(namedPoint);
class Vector2D {
    constructor(x, y) {
        this.x = x;
        this.y = y;
    }
}
logCoordinates(new Vector2D(5, 10));
```

## 2. Type Erasure: Compile-Time vs. Runtime

TypeScript's type system is completely **erased** during compilation. Types exist solely to assist the compiler (`tsc`) and your IDE in performing static analysis. Once compiled, the generated JavaScript has no awareness of interfaces, type aliases, or generics. 

There is **zero runtime overhead** added by TypeScript types.

### TypeScript Code (Compile-Time)
```typescript
type UserID = string & { readonly __brand: unique symbol };

interface User {
  id: UserID;
  username: string;
}

function greetUser(user: User): string {
  return `Hello, ${user.username}`;
}
```

### Compiled JavaScript (Runtime)
```javascript
"use strict";
// All type definitions, interfaces, and brands have been erased.
function greetUser(user) {
    return `Hello, ${user.username}`;
}
```

## 3. The `tsc` Compiler: Compilation vs. Type Checking

The TypeScript compiler (`tsc`) performs two main duties:
1. **Transpilation**: Down-leveling newer JavaScript/TypeScript syntax to an older target (e.g., ES5/ES6).
2. **Type Checking**: Validating code against type rules.

Crucially, **type checking is separate from compilation**. By default, even if there are type errors, `tsc` will still emit JavaScript files unless the `--noEmitOnError` flag is set.

```bash
# Run type checking and emit files
tsc

# Run type checking only, without emitting files
tsc --noEmit

# Prevent emitting compiled files if any type-check error is found
tsc --noEmitOnError
```

## 4. Configuring `tsconfig.json` with `strict: true`

The `tsconfig.json` file configures how `tsc` compiles your project. Enabling `"strict": true` is the industry standard and activates a suite of type-checking behaviors that guarantee maximum type safety.

### Complete strict-mode `tsconfig.json`
```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "NodeNext",
    "moduleResolution": "NodeNext",
    "lib": ["ES2022", "DOM"],
    "strict": true,
    "noImplicitAny": true,
    "strictNullChecks": true,
    "strictFunctionTypes": true,
    "strictBindCallApply": true,
    "strictPropertyInitialization": true,
    "noImplicitThis": true,
    "useUnknownInCatchVariables": true,
    "alwaysStrict": true,
    "noUnusedLocals": true,
    "noUnusedParameters": true,
    "noImplicitReturns": true,
    "exactOptionalPropertyTypes": true,
    "outDir": "./dist",
    "rootDir": "./src",
    "esModuleInterop": true,
    "forceConsistentCasingInFileNames": true,
    "skipLibCheck": true
  },
  "include": ["src/**/*"]
}
```

### Strict Mode Flags Explained:

*   **`noImplicitAny`**: Raises an error on expressions and declarations with an implied `any` type.
    ```typescript
    // Error: TS7006: Parameter 'x' implicitly has an 'any' type.
    function double(x) { 
      return x * 2;
    }
    ```
*   **`strictNullChecks`**: Ensures `null` and `undefined` are not assignable to other types except when explicitly declared as unions.
    ```typescript
    let name: string = "Alice";
    // Error: TS2322: Type 'null' is not assignable to type 'string'.
    name = null; 
    ```
*   **`strictPropertyInitialization`**: Ensures class properties are initialized in the constructor or have a default value.
    ```typescript
    class Account {
      // Error: TS2564: Property 'balance' has no initializer and is not definitely assigned in the constructor.
      balance: number; 
      constructor() {}
    }
    ```
*   **`useUnknownInCatchVariables`**: Forces the type of error variables in catch blocks to be `unknown` instead of `any`.
    ```typescript
    try {
      throw new Error("Failed");
    } catch (err) {
      // Error: TS18046: 'err' is of type 'unknown'.
      console.log(err.message); 
    }
    ```

## 5. TypeScript Errors vs. Runtime Errors

TypeScript errors are **compile-time** validations that prevent unsafe code from executing. Runtime errors are failures that occur while the engine executes the compiled JavaScript.

### Compile-Time Error Example
```typescript
interface Configuration {
  port: number;
}

const config: Configuration = {
  port: "8080" 
  // Error: TS2322: Type 'string' is not assignable to type 'number'.
};
```

### Runtime Error Prevention
Without TypeScript, passing an unexpected type can cause silent failures or immediate crashes:
```typescript
function uppercaseFirst(arr: string[]) {
  return arr[0].toUpperCase();
}

// TypeScript prevents this call at compile-time:
// Error: TS2345: Argument of type 'undefined[]' is not assignable to parameter of type 'string[]'.
uppercaseFirst([]); // At runtime, this would throw: TypeError: Cannot read properties of undefined (reading 'toUpperCase')
```

## 6. When TypeScript Helps Most

1. **Large-Scale Refactoring**: Renaming properties or changing function signatures across hundreds of files is trivial because the compiler instantly flags all mismatched call sites.
2. **Explicit API Contracts**: Interfaces define strict boundaries for APIs, keeping frontend and backend schemas in sync.
3. **Team Collaboration**: Code serves as self-documenting executable specifications. Developers do not need to guess the shape of data passed into functions.

*Refer to the official [TypeScript Handbook: The Basics](https://www.typescriptlang.org/docs/handbook/2/basic-types.html) for further information.*

## WHY
JavaScript is a "dynamically typed" language. You can write `let x = 5` and then later do `x = "hello"`. This flexibility is great for small scripts, but in a large team building a 100,000-line Enterprise application, it becomes catastrophic.

If a function expects a `User` object and you accidentally pass a `Product` object, JavaScript will silently fail at runtime, crashing the application in production for real users.

**TypeScript** is a "statically typed superset of JavaScript" that solves this. It adds a compile-time type system on top of JavaScript. Before your code ever runs, the TypeScript compiler (`tsc`) reads every single line, verifies that every variable and function call is used correctly, and throws a compile error if it finds a bug. This catches 70%+ of bugs before users ever see them.

## THEORY
### How TypeScript Works
TypeScript is NOT a new programming language that runs in the browser. It is a **Transpiler**. You write `.ts` files, run `tsc`, and it produces plain JavaScript `.js` files. The browser or Node.js then runs the JavaScript exactly as normal.

The TypeScript compiler acts as an extremely intelligent pre-flight checks system. Once it's happy with your code, it strips all the type annotations away, leaving clean JavaScript.

## CODE

### Level 1 — Beginner (Adding Types to Variables)
```typescript
// JavaScript: No type safety! This is perfectly valid.
// let count = 0;
// count = "ten"; // No error! This will silently break things at runtime.

// TypeScript: Explicit type annotations
let count: number = 0;
// count = "ten"; // ❌ COMPILE ERROR! TypeScript catches this instantly!

let username: string = "Alice";
let isActive: boolean = true;

// TypeScript also has TYPE INFERENCE — it's smart enough to figure out types!
let score = 100; // TypeScript INFERS this is a 'number'
// score = "high"; // ❌ ERROR! Even without explicit annotation!

// Arrays
let items: string[] = ["apple", "banana"];
// items.push(42); // ❌ ERROR! 42 is not a string.

// Tuples: Fixed-length arrays with specific types at each position
let coordinates: [number, number] = [10.5, 20.7];
```

### Level 2 — Building Blocks (Typing Functions)
```typescript
// JavaScript (no safety):
// function add(a, b) { return a + b; }
// add("10", 5); // Returns "105" (string concatenation!) instead of 15

// TypeScript: Every parameter and the return type are explicitly typed!
function add(a: number, b: number): number {
    return a + b;
}
// add("10", 5); // ❌ COMPILE ERROR! 
add(10, 5);     // ✅ Returns 15.

// Optional Parameters (marked with ?)
function greet(name: string, greeting?: string): string {
    return `${greeting ?? "Hello"}, ${name}!`;
}
greet("Alice");           // ✅ "Hello, Alice!"
greet("Bob", "Welcome");  // ✅ "Welcome, Bob!"

// Arrow Functions
const multiply = (a: number, b: number): number => a * b;
```

### Level 3 — Intermediate (Interfaces and Type Aliases)
`interface` and `type` are how you define the "shape" of a complex object.

```typescript
// INTERFACE (recommended for object shapes and classes)
interface User {
    readonly id: number;          // Cannot be changed after creation!
    username: string;
    email: string;
    age?: number;                 // Optional field
}

function displayUser(user: User): void {
    console.log(`${user.username} (${user.email})`);
}

const alice: User = { id: 1, username: "alice", email: "alice@example.com" };
displayUser(alice); // ✅

// TYPE ALIAS (flexible — can be used for primitives, unions, intersections)
type Status = "active" | "inactive" | "pending";  // Union of literal types!

type AdminUser = User & {       // Intersection: combines User with extra fields
    adminLevel: number;
    permissions: string[];
};
```

### Level 4 — Advanced (Generics)
Generics allow you to write reusable code that works with any type while maintaining full type safety.

```typescript
// A function that works with ANY type!
// <T> is a "Type Parameter" — a placeholder for the actual type.
function getFirstItem<T>(array: T[]): T | undefined {
    return array[0];
}

const firstNum = getFirstItem([1, 2, 3]);   // TypeScript infers T = number
const firstStr = getFirstItem(["a", "b"]); // TypeScript infers T = string

// firstNum.toUpperCase() // ❌ ERROR! It's a number, not a string!
firstStr.toUpperCase();  // ✅ TypeScript knows it's a string!

// Generic Interfaces
interface ApiResponse<T> {
    data: T;
    status: "success" | "error";
    message: string;
}

type UserResponse = ApiResponse<User>;
type ProductListResponse = ApiResponse<Product[]>;
```

## REAL_WORLD
In the Enterprise React + TypeScript ecosystem, TypeScript is non-negotiable. 

**Key benefits in practice:**
1. **Autocomplete:** When you type `user.`, the IDE shows you every available property (`username`, `email`, etc.) automatically.
2. **Refactoring:** Renaming a property in an `interface` automatically updates every file that uses it, across the entire codebase.
3. **Self-documenting code:** The type definitions ARE the documentation. New team members can understand a function's contract by reading the TypeScript types alone.

## INTERVIEW
**Q: What is `any` in TypeScript and why should you avoid it?**
A: `any` is a type escape hatch. Assigning `any` to a variable tells TypeScript "Turn off type checking for this value." It completely defeats the purpose of TypeScript. Avoid it at all costs. Use `unknown` instead (TypeScript still checks `unknown`, but forces you to narrow the type before using it).

**Q: What is the difference between `interface` and `type`?**
A: Both can define object shapes. Key differences: `interface` supports **Declaration Merging** (you can define the same interface in multiple places, and TypeScript merges them — useful for extending third-party library types). `type` is more flexible (it can represent Unions, Mapped Types, and Conditional Types). In general, use `interface` for object shapes and `type` for everything else.

**Q: What does `tsc` do exactly?**
A: The TypeScript Compiler (`tsc`) reads all `.ts` files, resolves module imports, performs full type checking, and if no errors are found, transpiles the TypeScript into pure JavaScript. In production builds (Vite, Next.js), `tsc` is used only for type checking. The actual transpilation is done by much faster tools like **esbuild** or **SWC**.

## FEYNMAN CHECK

### Explain TypeScript Like I'm 10 Years Old
> TypeScript is JavaScript with a SPELL-CHECKER for your data. You annotate `name: string`, `age: number`, and the TypeScript compiler reads your code at BUILD TIME — long before any user runs it — and screams if you try to pass a number where a string was expected, or call `.toUpperCase()` on `null`. Critically, all this type information is ERASED before the code reaches the browser — TS compiles to plain JS. So there is zero runtime cost. Types are a developer tool, not a runtime feature. This is why `typeof user === "User"` doesn't work — at runtime the User interface is gone, replaced by a plain object.

### 5 Deep Conceptual Questions

**Q1: What problem does TypeScript fundamentally solve that ESLint and unit tests don't?**
> **A:** It catches an entirely different class of bug — TYPE MISMATCHES — at compile time across module boundaries, with full intellisense as a side-effect. ESLint catches style and a few simple bugs (unused vars, no-unreachable); unit tests verify behaviour you remembered to test. TypeScript verifies that EVERY call site of EVERY function passes the right shape, every time, with zero runtime cost and no tests required. At Google scale, Microsoft reports TS prevents ~15% of bugs that would otherwise reach production.

**Q2: What is the ONE mental model that makes TypeScript click?**
> **A:** "Types are a parallel universe that describes the shape of values, erased before runtime." Every type — `string`, `User`, `Promise<T>`, `keyof T` — exists ONLY in the .ts source. The compiler reads the type universe, validates that your values are consistent, then THROWS THE TYPES AWAY. Output is plain ES2020+ JavaScript. Once you internalise the type/value split, you stop trying to do impossible things like `if (typeof x === 'User')` or storing types in variables.

**Q3: What is the most dangerous misconception about TypeScript? Show with code.**
> **A:** Believing TS checks types at runtime:
> ```ts
> // ❌ Wrong: assumes TS validates JSON at runtime
> interface User { name: string; age: number }
> const data: User = JSON.parse(apiResponse);   // TS trusts the cast — NO validation
> console.log(data.age.toFixed(2));   // 💥 if api sent age as a string
>
> // ✅ Right: use a runtime validator (zod, valibot, io-ts)
> import { z } from 'zod';
> const UserSchema = z.object({ name: z.string(), age: z.number() });
> const data = UserSchema.parse(JSON.parse(apiResponse));   // throws if invalid
> ```

**Q4: How does TypeScript interact with JavaScript's runtime at the engine level?**
> **A:** It doesn't — and that's the point. The TypeScript compiler (`tsc`) is a transpiler. It parses .ts, builds an AST, runs the type checker (which validates the type universe), emits .js with types stripped, and exits. V8 never sees a type. This is why decorators, enums, and namespaces — which DO emit runtime code — are different from regular type annotations: they have actual JS output. Pure types are zero bytes.

**Q5: One-sentence FAANG-grade definition?**
> **A:** "TypeScript is a structurally-typed, gradually-applied static type checker that transpiles to JavaScript — providing build-time type validation, IDE intellisense, and refactoring safety with zero runtime overhead, while requiring external runtime validators for any data crossing system boundaries (network, disk, user input)."

#### Step 2 — The Module With Bugs (Typed Version)
```ts
// src/userService.ts
export interface User {
  id: number;
  email: string;
  age: number;
  isAdmin: boolean;
}

export function findUser(users: User[], id: number): User | undefined {
  return users.find(u => u.id === id);
}

export function getDisplayName(user: User): string {
  // Bug in original JS: user.name didn't exist; original JS returned undefined silently
  return user.email.split('@')[0];
}

export function promoteToAdmin(user: User): User {
  return { ...user, isAdmin: true };
}
```

#### Step 3 — Caller With a Type Mismatch
```ts
// src/main.ts
import { findUser, getDisplayName, promoteToAdmin, type User } from './userService.js';

const users: User[] = [
  { id: 1, email: 'alice@dev.io', age: 30, isAdmin: false },
  { id: 2, email: 'bob@dev.io',   age: 25, isAdmin: false },
];

// TS will catch: "Argument of type 'string' is not assignable to type 'number'"
const found = findUser(users, '1');           // ❌ compile error
const promoted = promoteToAdmin({ id: 3, email: 'eve@dev.io' });  // ❌ missing age + isAdmin
console.log(getDisplayName(found));            // ❌ found can be undefined
```

#### Step 4 — Error Handling: Fix Each Compile Error
```ts
// src/main.ts (corrected)
const found = findUser(users, 1);
const promoted = promoteToAdmin({ id: 3, email: 'eve@dev.io', age: 28, isAdmin: false });
if (found) {
  console.log(getDisplayName(found));         // narrowed: now User, not User|undefined
} else {
  console.log('No user found');
}
```
