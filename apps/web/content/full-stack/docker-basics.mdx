## WHY
*"But it works on my machine!"*
This is the most infamous excuse in software engineering. You spend 3 weeks building a Node.js API on your Macbook. It works perfectly. You send the code to your coworker, who is using Windows. When they run the code, it crashes instantly because they have the wrong version of Node.js installed, and their environment variables are missing.
When you finally deploy the code to a Linux production server, it crashes again because the server doesn't have the correct database drivers!
**Docker** solves this by physically isolating your application and its entire environment into a **Container**. A container is a standardized, lightweight, portable box that contains your code, the exact version of Node.js you used, and all the required OS libraries. If a Docker container works on your laptop, it is mathematically guaranteed to work on any coworker's laptop, and on any cloud server in the world, identically.

## THEORY
### Virtual Machines vs. Containers
- **Virtual Machine (VM):** To isolate an app, a VM installs an *entire massive Operating System* (like a 20GB copy of Windows or Ubuntu) on top of the host computer. It reserves a fixed amount of RAM (e.g., 4GB) even if the app isn't using it. VMs are extremely slow to boot and massive in size.
- **Docker Container:** Containers do not install a new OS. They aggressively share the host computer's underlying OS kernel! A container only contains the tiny, specific libraries your app needs (often less than 50MB). Containers boot up in milliseconds and consume almost zero overhead RAM.

### The Image and The Container
- **Docker Image:** The rigid, read-only Blueprint. It is the compiled file containing your code and instructions. (Like an `.exe` file).
- **Docker Container:** The actual running instance of the Image. You can launch 100 identical containers from a single Image! (Like double-clicking the `.exe` file 100 times).

## CODE

### 1. The Dockerfile (Creating the Blueprint)
To dockerize a Node.js application, you create a file named exactly `Dockerfile` in the root of your project. This file tells Docker exactly how to build the Image.

```dockerfile
# 1. Base Image: Start with a computer that already has Node.js version 18 installed!
FROM node:18-alpine

# 2. Create a folder inside the container to hold our app
WORKDIR /app

# 3. Copy the package.json file from our laptop into the container
COPY package*.json ./

# 4. Run the install command INSIDE the container
RUN npm install

# 5. Copy the rest of our source code from our laptop into the container
COPY . .

# 6. Open port 3000 on the container so traffic can enter
EXPOSE 3000

# 7. The exact command to start the application when the container boots!
CMD ["npm", "start"]
```

### 2. Building and Running (Terminal Commands)
Now we command Docker to read the blueprint and build the physical Image.

```bash
# BUILD THE IMAGE
# -t names the image "my-node-app". The '.' means "look in this folder for the Dockerfile"
docker build -t my-node-app .

# RUN THE CONTAINER
# -d runs it in the background (Detached).
# -p 8080:3000 maps Port 8080 on your physical laptop to Port 3000 inside the isolated container!
docker run -d -p 8080:3000 my-node-app

# See all currently running containers!
docker ps
```
You can now go to `http://localhost:8080` on your laptop, and you will see your app running safely inside the isolated container!

### 3. Docker Compose (Multi-Container Apps)
What if your Node.js app needs a PostgreSQL database? You don't put the database *inside* the Node container! (Containers should only do ONE thing).
You use `docker-compose.yml` to spin up multiple separate containers and network them together automatically!

```yaml
version: '3.8'
services:
  # Container 1: The Database
  database:
    image: postgres:15
    environment:
      POSTGRES_PASSWORD: mysecretpassword

  # Container 2: Our Node App
  api:
    build: .
    ports:
      - "8080:3000"
    depends_on:
      - database # Wait for the DB to boot up before starting the API!
```
Now, you just run **`docker-compose up -d`**, and Docker will download PostgreSQL, build your Node app, boot them both up, and network them together instantly!

## REAL_WORLD
**Continuous Deployment (CI/CD)**
Before Docker, deploying code to production was terrifying. A developer would SSH into the live server, manually run `git pull`, manually run `npm install`, and hope the server didn't crash.
Today, the entire process is automated using Docker Registries (like Docker Hub or AWS ECR).
When you merge code into `main` on GitHub, an automated script kicks off. It builds the Docker Image and uploads it to the Docker Registry. It then tells the production server: *"Hey! There is a new Image v2.0 ready!"*
The server downloads the `v2.0` Image, boots it up alongside the old `v1.0` container, switches the network traffic to the new container, and quietly deletes the old `v1.0` container. This allows companies to deploy new code 50 times a day with zero seconds of downtime!

## INTERVIEW
**Q: You run `docker ps` and see your container is running perfectly. You open a terminal INSIDE the container using `docker exec`, and you manually edit a configuration file. You restart the container. When it boots back up, your edits are completely gone! Why did this happen?**
A: **Containers are strictly Ephemeral (Temporary).**
The fundamental rule of Docker is that containers are totally disposable. Any file written *inside* a container's filesystem is permanently destroyed the moment the container stops or restarts! The container always reboots from the pristine, immutable Image blueprint.
If you need to persist data (like saving uploaded user images, or running a database), you MUST use a **Docker Volume**. 
A Volume is a special command that punches a hole through the container's isolated wall, mapping a folder inside the container directly to a physical, permanent folder on your host laptop's hard drive!

## FEYNMAN CHECK

### Explain Docker Like I'm 10 Years Old
> Docker is a lunchbox for your code: it packs everything your app needs (OS libs, runtime, dependencies, code) into one sealed box (image) that runs the same on your laptop, a colleague's laptop, or a production server. No more "works on my machine". The non-obvious part: Docker images are layered like a stack of transparencies — each instruction in your Dockerfile adds a layer, and unchanged layers are cached between builds. This is why putting `COPY package.json` before `COPY .` is a 10× speedup for rebuilds.

### 5 Deep Questions
**Q1: Container vs VM — what's the architectural difference?**
> **A:** VM virtualises hardware → each VM has its own kernel → 100s of MB overhead, slow boot. Container shares the host kernel via namespaces (PID, network, mount isolation) and cgroups (resource limits) → MBs not GBs, boots in milliseconds. You get application isolation without OS overhead.

**Q2: ONE model.**
> **A:** "Image = immutable layered filesystem. Container = running instance of an image with its own writable layer on top. Dockerfile = recipe. Registry = library of images."

**Q3: Misconception with code.**
> **A:** Layer ordering wastes cache:
> ```dockerfile
> # ❌ Code changes invalidate node_modules cache — 60s rebuild every time
> COPY . .
> RUN npm install
> # ✅ Dependencies first (cached), code after
> COPY package*.json ./
> RUN npm install
> COPY . .
> ```

**Q4: How do volumes, bind mounts, and tmpfs differ?**
> **A:** Volume (`docker volume create`) — Docker-managed persistent storage, portable. Bind mount (`-v $(pwd):/app`) — direct host path mapped in, used in dev for hot-reload. Tmpfs — in-memory, ephemeral, used for secrets. Volumes are recommended for production state.

**Q5: Senior one-liner.**
> **A:** "A container is a process running with kernel namespaces and cgroups in a unioned filesystem image — which is why containers boot in milliseconds, share the host kernel, and can be packed 10× denser than VMs."

## BUILD

### 🏗️ Mini Project: Dockerise a Node App with Multi-Stage Build
**Time:** 35 minutes

#### Step 1 — Setup
```bash
mkdir docker-app && cd docker-app
npm init -y && npm i express
cat > index.js << 'EOF'
const app = require('express')();
app.get('/', (_, res) => res.send('Hello from Docker'));
app.listen(3000, () => console.log('on :3000'));
EOF
```

#### Step 2 — Multi-Stage Dockerfile
```dockerfile
# Stage 1: build (with dev deps)
FROM node:20-alpine AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .

# Stage 2: runtime (lean, prod-only deps)
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci --omit=dev && npm cache clean --force
COPY --from=build /app .
USER node
EXPOSE 3000
CMD ["node", "index.js"]
```

#### Step 3 — Build & Run
```bash
docker build -t my-app .
docker run -p 3000:3000 --name app my-app
curl http://localhost:3000
```

#### Step 4 — Error Handling
```dockerfile
# Healthcheck so orchestrators know when the app is ready
HEALTHCHECK --interval=30s --timeout=3s   CMD wget --quiet --tries=1 --spider http://localhost:3000/ || exit 1
```

#### Step 5 — Image Size Test
```bash
docker images my-app   # Should be ~150MB (vs 1GB+ without multi-stage)
```

**Expected Output:** `Hello from Docker` and image under 200MB.
