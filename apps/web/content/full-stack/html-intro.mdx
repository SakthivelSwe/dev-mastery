# HTML — Document Structure, DOCTYPE, Semantic Purpose (Level 1)

## 1. Deep-Dive Technical Core
At the foundation of every web document lies the HyperText Markup Language (HTML) specification, governed by the WHATWG Living Standard. When a browser initiates a network request and receives an HTML document, the rendering engine (such as Blink, WebKit, or Gecko) passes the raw bytes to its parsing subsystem.

```
[Raw Bytes] -> [Tokenization] -> [Tree Construction] -> [DOM Tree]
                                        ^
                                        | (CSSOM Integration)
                                        v
                                  [Render Tree]
```

### The Role of `<!DOCTYPE html>`
The very first line of any modern HTML document must be `<!DOCTYPE html>`. This is not an HTML tag; it is a declarative instruction to the browser's parser. 
* **Quirks Mode vs. Standards Mode**: Historically, browsers implemented non-standard layout behaviors to support legacy websites. If the `<!DOCTYPE html>` declaration is omitted or malformed, modern browsers enter **Quirks Mode**, emulating bugs from late-1990s browsers (e.g., incorrect CSS box model calculation where padding is included in width). Including the modern HTML5 DOCTYPE forces the browser to use **Standards Mode** (or No-Quirks Mode), adhering strictly to the W3C and WHATWG specifications.
* **Case Insensitivity**: While written conventionally as `<!DOCTYPE html>`, it is case-insensitive (`<!doctype html>` is equally valid).

### Parser Tokenization and Tree Construction
The parser processes the incoming stream of characters via a state machine:
1. **Tokenization**: Converting characters into tokens (`StartTag`, `EndTag`, `Character`, `Comment`, `DOCTYPE`).
2. **Tree Construction**: Building the Document Object Model (DOM) tree. The parser is highly resilient; it features an error-recovery mechanism defined in the specification that automatically closes unclosed tags (e.g., nesting a `<div>` inside a `<p>` forces the `<p>` to close immediately because paragraphs cannot contain block-level elements).

### Document-Level Elements
* `<html>`: The root element of the document. The `lang` attribute (e.g., `lang="en"`) is mandatory for screen readers to select the correct text-to-speech engine and pronunciation dictionary.
* `<head>`: Contains machine-readable metadata, pointers to stylesheets, scripts, and document titles. It is parsed before the body, and its contents do not render visually.
* `<body>`: Represents the renderable content of the document.

## 3. Rendered Visual Output & CSS Explanation
When parsed and styled by a modern browser:
* **The Header**: Appears as a clean, white horizontal bar fixed to the top. The logo sits on the far left, while the navigation links align to the right, distributed evenly with `gap: 1.5rem`.
* **Skip Link**: Under normal circumstances, this element is completely hidden off-screen (`top: -100px`). However, if a user hits the `Tab` key immediately after the page loads, the skip link shifts into view at `top: 1rem`, styled in high-contrast blue with a thick amber outline when focused.
* **The Main Content**: Centered horizontally on the page with a maximum width of `1200px`. The text utilizes a highly readable line-height (`1.6`) and is constrained to a maximum width of `65ch` (characters) to prevent eye fatigue.
* **Sticky Footer**: Due to the `display: flex; flex-direction: column; min-height: 100vh` on the `body` and `flex: 1` on the `main` tag, the footer is guaranteed to sit at the absolute bottom of the viewport even on extremely short content pages.

## 4. Accessibility, Screen Reader Mechanics & WCAG 2.1 AA
* **Language Declaration (`lang="en"`)**: Essential for **WCAG 2.1 AA Success Criterion 3.1.1 (Language of Page)**. Without this, screen readers like VoiceOver or NVDA will default to the user's system language, causing incomprehensible pronunciation of the content.
* **Skip Link**: Maps to **WCAG 2.1 AA Success Criterion 2.4.1 (Bypass Blocks)**. It allows keyboard-only users to bypass repetitive header navigation and jump straight to the primary content. The target `main` has `tabindex="-1"` so it can programmatically receive focus without being in the natural tab order.
* **ARIA Landmarks**: The `<header>` element is recognized as the `banner` landmark, `<nav>` as `navigation`, `<main>` as `main`, and `<footer>` as `contentinfo`. Screen readers allow users to jump directly between these landmarks using hotkeys (e.g., `D` in JAWS/NVDA).
* **`aria-current="page"`**: Informs assistive technology that the "Home" link represents the active page in the collection, meeting **WCAG 2.1 AA Success Criterion 1.3.1 (Info and Relationships)**.

### Screen Reader Output Scenario (VoiceOver on macOS)
1. User presses `Tab` on page load: *"Link, Skip to main content"*.
2. User presses `Enter`: Focus moves to the main element. Screen reader remains silent but focus is shifted.
3. User presses `Tab` again (bypassing header): Focus lands on the first interactive element inside `main` (if any).
4. Navigating landmarks: *"Banner landmark. Navigation landmark, Primary Site Navigation. List of 3 items. Link, Home, current page."*

## 5. Advanced Patterns & Architectural Integration
In modern Single Page Application (SPA) architectures (React, Vue, Angular) and Server-Side Rendering (SSR) frameworks (Next.js, Nuxt, SvelteKit), managing document structure requires deliberate synchronization.

### Hydration and Root Elements
Most SPAs mount their application inside a container element, e.g., `<div id="app"></div>`.
```javascript
// React 18+ Mounting Pattern
import { createRoot } from 'react-dom/client';
import App from './App';

const container = document.getElementById('app');
const root = createRoot(container);
root.render(<App />);
```
**Architectural Risk**: If the server-rendered HTML does not perfectly match the client-side generated DOM tree during the hydration phase, the framework will trigger a "Hydration Mismatch" error. This often occurs when developer-written JS injects elements (like dates or dynamic IDs) that differ between the server and client environment.

### Dynamic Metadata Management
In SSR, the `<head>` tag must change dynamically based on the current route. Frameworks use packages like `@vueuse/head` or React's built-in `<head>` support to inject title and meta tags before the HTML is streamed to the client. This ensures search engine crawlers receive fully populated metadata without relying on client-side JS execution.

## 6. Edge Cases, Performance & W3C Validation Gotchas
* **Parser-Blocking Scripts**: By default, encountering a `<script>` tag inside `<head>` pauses DOM construction while the script is downloaded and executed. This blocks the main thread. Always mitigate this by using `defer` or `async` attributes.
* **Trailing Slashes and Optional Tags**: Under the HTML5 specification, tags like `</li>`, `</p>`, and `</td>` are technically optional—the parser will auto-infer their closure. However, omitting them is highly discouraged in production because it can lead to subtle rendering differences in legacy browsers and breaks static analysis tools (linters).
* **The DOCTYPE Gotcha**: If any character—even a single whitespace or an HTML comment—appears before `<!DOCTYPE html>`, some versions of Internet Explorer and older rendering engines will fail to parse standard styles and fall back to Quirks Mode. Ensure the DOCTYPE is strictly the first byte of the HTTP response body.

## WHY

Every web page you have ever visited — Google, YouTube, Amazon, Netflix — starts with a single HTML file. HTML (HyperText Markup Language) is the **skeleton of the web**. Before HTML existed, the internet was a disconnected series of plain text files you had to know the exact address of. Tim Berners-Lee invented HTML in 1991 precisely to solve this: a way to embed links inside documents so the web could be navigated like a book.

Without HTML, a browser has no idea what to do with text. It cannot tell the difference between a heading, a paragraph, a navigation menu, or a button. HTML introduces **semantic meaning** — the idea that a `<h1>` tag means "this is the most important heading on this page," a `<p>` means "this is a paragraph of text," and a `<button>` means "this element the user can interact with." This meaning is used by browsers, screen readers, search engine crawlers, and JavaScript code.

Understanding HTML properly is also the foundation of everything else in frontend development. CSS cannot target elements it cannot find. JavaScript cannot manipulate the DOM without understanding its structure. Accessibility features only work when you use the correct semantic HTML. Developers who treat HTML as an afterthought produce broken, inaccessible, poorly-ranked websites.

## THEORY

### What is HTML?
HTML is a **markup language** — not a programming language. It uses **tags** (words surrounded by angle brackets) to describe the structure and meaning of content. Tags come in pairs: an **opening tag** (`<p>`) and a **closing tag** (`</p>`). The content between them is the element's content.

Some elements are **void elements** (self-closing) — they have no content or closing tag: `<br>`, `<img>`, `<input>`, `<hr>`, `<meta>`, `<link>`.

### The Document Object Model (DOM)
When a browser parses your HTML file, it builds a live **tree data structure in memory** called the **DOM (Document Object Model)**. Every HTML tag becomes a **Node** in this tree. The DOM is what JavaScript, CSS selectors, and browser developer tools all interact with — not the raw HTML text.

```
document
  └── <html>
        ├── <head>  → <meta>, <title>, <link>
        └── <body>  → <h1>, <p>, <a>, <img>...
```

### Block vs Inline Elements

| Behaviour | Common Elements | Layout Rule |
|-----------|----------------|-------------|
| **Block** | `<div>`, `<p>`, `<h1>`–`<h6>`, `<ul>`, `<section>` | Full container width, starts new line |
| **Inline** | `<span>`, `<a>`, `<strong>`, `<em>`, `<img>` | Width of content only, flows with text |
| **Inline-block** | `<button>`, `<input>` | Inline flow but respects width/height |

### Attributes
Tags can have **attributes** — key-value pairs that configure the element:
```html
<a href="https://example.com" target="_blank" rel="noopener noreferrer">Link</a>
```
Boolean attributes need no value: `<input disabled>` or `<input disabled="">` are equivalent.

### `<head>` vs `<body>`
- **`<head>`**: Metadata — not displayed. Contains `<title>`, `<meta>`, `<link>` to CSS, `<script>` references.
- **`<body>`**: Everything the user sees — text, images, links, forms.

### DOCTYPE Declaration
`<!DOCTYPE html>` at the top triggers **standards mode** rendering. Without it, browsers enter **quirks mode** — emulating the buggy behaviour of 1990s browsers — causing inconsistent layouts across browsers.

### Global Attributes (available on every element)
| Attribute | Purpose |
|-----------|---------|
| `id` | Unique identifier on the page |
| `class` | CSS/JS grouping selector |
| `style` | Inline CSS |
| `data-*` | Custom data stored in the element |
| `aria-*` | Accessibility metadata |
| `hidden` | Hides element (display:none) |
| `tabindex` | Controls keyboard focus order |
| `lang` | Language of element's content |

## CODE

### Level 1 — Beginner: Complete HTML5 Document Structure
```html
<!DOCTYPE html>
<!-- DOCTYPE: Tells browser to use HTML5 standards mode -->

<html lang="en">
<!-- lang="en": Helps screen readers, search engines, and browser spell-check -->

<head>
  <!-- HEAD = metadata about the page, not shown to users -->

  <meta charset="UTF-8">
  <!-- charset: UTF-8 supports all characters in all languages -->

  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <!-- viewport: CRITICAL for mobile! Without this, mobiles zoom out to show desktop layout -->

  <title>My First Web Page</title>
  <!-- title: Shown in browser tab, bookmarks, and search engine results -->

  <link rel="stylesheet" href="style.css">
  <!-- Links external CSS file to this page -->
</head>

<body>
  <!-- BODY = everything the user actually sees -->

  <h1>Welcome to HTML</h1>
  <!-- h1: The single most important heading — use ONLY ONE per page (SEO rule) -->

  <p>HTML stands for <strong>HyperText Markup Language</strong>.</p>
  <!-- p: Paragraph. strong: Bold AND semantically "important" -->

  <a href="https://developer.mozilla.org" target="_blank" rel="noopener noreferrer">
    MDN Web Docs
  </a>
  <!-- a: Anchor/hyperlink. href = destination. rel="noopener" prevents tab hijacking -->

  <img src="logo.png" alt="DevMastery logo" width="200" height="80">
  <!-- img: Image. alt is MANDATORY — used by screen readers and when image fails to load -->

</body>
</html>

<!-- Expected Output in Browser:
     Tab title: "My First Web Page"
     H1 heading: "Welcome to HTML"
     Paragraph with "HyperText Markup Language" in bold
     Blue clickable link: "MDN Web Docs"
     The logo image (or alt text if image not found)
-->
```

## REAL_WORLD

In production Next.js/React applications, the **HTML output quality** determines both SEO rankings and Core Web Vitals scores. A major mistake in component-heavy apps is generating "div soup" — replacing all semantic HTML with generic `<div>` wrappers. This breaks screen-reader navigation (users can't jump between landmarks), hurts Google's ability to understand the page hierarchy, and fails WCAG 2.1 AA accessibility compliance (which is legally required in many countries).

```html
<!-- ❌ BAD: Div soup — technically works, fails accessibility/SEO audits -->
<div class="header">
  <div class="logo"><img src="logo.png"></div>
  <div class="nav"><div class="nav-item"><a href="/">Home</a></div></div>
</div>
<div class="main"><div class="article"><div class="title">Blog Post</div></div></div>

<!-- ✅ GOOD: Semantic HTML — passes WCAG, helps SEO, screen-reader navigable -->
<header>
  <a href="/"><img src="logo.png" alt="DevMastery home"></a>
  <nav aria-label="Primary">
    <ul><li><a href="/" aria-current="page">Home</a></li></ul>
  </nav>
</header>
<main>
  <article>
    <h1>Blog Post</h1>
  </article>
</main>
```

For large-scale e-commerce sites (Shopify, Amazon), the `<picture>` element with multiple `srcset` formats (AVIF → WebP → JPEG) combined with `loading="lazy"` and `decoding="async"` is critical — it can reduce image payload by 60-80%, directly improving Largest Contentful Paint (LCP) scores. The JSON-LD structured data in `<script type="application/ld+json">` enables Google rich results (star ratings, FAQs, breadcrumbs) in search results, increasing click-through rates by up to 30%.

## INTERVIEW

### Q1: What is the DOM and how does it differ from HTML source code?
**Difficulty:** Easy | **Companies:** All

❌ **Weak answer:** "The DOM is like HTML but as a JavaScript object."

✅ **Strong answer:**
"HTML is a **static text file** — a string of characters stored on disk. The **DOM (Document Object Model)** is a **live, in-memory tree data structure** the browser constructs by parsing the HTML. Every tag becomes a Node object in this tree. The key difference: the DOM is mutable — JavaScript, React, and browser extensions all modify it after page load. The changes exist only in memory; the original HTML file is unchanged. Also, browsers automatically repair invalid HTML — a missing `<tbody>` tag gets added to the DOM even if absent in the HTML. This is why browser DevTools show the DOM (the live state), not always what's in your source HTML file."

### Q2: What is the difference between `<b>/<i>` and `<strong>/<em>`?
**Difficulty:** Easy | **Companies:** All

❌ **Weak answer:** "They look the same — bold and italic."

✅ **Strong answer:**
"`<b>` and `<i>` are **purely presentational** — they apply visual formatting (bold/italic) with no semantic meaning. Screen readers don't alter their reading of these elements. `<strong>` conveys **strong importance** — screen readers may read it with emphasis, and search engines give it more weight in ranking. `<em>` conveys **stress emphasis** that can change the meaning of a sentence. Use `<strong>` for warnings/critical info; `<em>` for stress in prose. Use `<b>` for stylistic bold without importance (product names, keywords); `<i>` for technical terms, foreign phrases, or titles."

### Q3: What does `rel="noopener noreferrer"` do on external links?
**Difficulty:** Medium | **Companies:** Google, Facebook

❌ **Weak answer:** "It's a security thing you add to links."

✅ **Strong answer:**
"`target='_blank'` opens the link in a new tab, but without `rel='noopener'`, the new tab gets a reference to the opening page via `window.opener`. A malicious site could then execute `window.opener.location = 'phishing-site.com'` — redirecting the original tab to a phishing page while the user is on the new tab. `rel='noopener'` severs this `window.opener` reference, preventing the attack. `noreferrer` additionally prevents the browser from sending the `Referer` HTTP header, preserving privacy. Modern browsers automatically apply `noopener` to `target='_blank'` links, but explicitly adding it is best practice for clarity and older browser support."

### Q4: Why is the `alt` attribute on `<img>` mandatory?
**Difficulty:** Easy | **Companies:** All

❌ **Weak answer:** "It shows text if the image doesn't load."

✅ **Strong answer:**
"The `alt` attribute serves three critical purposes: (1) **Accessibility** — screen readers read the `alt` text aloud for visually impaired users; without it, screen readers may read out the full file path or announce 'unlabelled image'. (2) **Fallback** — if the image fails to load (slow connection, broken URL, user has images disabled), the alt text displays in its place. (3) **SEO** — search engine crawlers cannot 'see' images; the alt text tells them what the image shows, helping with image search rankings. For purely decorative images that add no information, use an empty alt (`alt=''`) — this tells screen readers to skip the image entirely, not read the filename."

### Q5: What is the purpose of `<meta name="viewport">`?
**Difficulty:** Medium | **Companies:** Apple, Google, Responsive Design Agencies

❌ **Weak answer:** "It makes the website mobile-friendly."

✅ **Strong answer:**
"Without `<meta name='viewport' content='width=device-width, initial-scale=1.0'>`, mobile browsers assume the page was designed for a 980px desktop viewport and **zoom out** to fit it on the screen, making everything tiny. The `width=device-width` tells the browser to set the viewport width equal to the device's actual screen width (e.g., 375px on an iPhone). The `initial-scale=1.0` sets the initial zoom level to 100%. This declaration is the prerequisite for all CSS media queries to work correctly — without it, your `@media (max-width: 768px)` rules may never trigger on mobile devices because the browser still thinks the width is 980px. It's mandatory in any responsive web design."

## FEYNMAN CHECK

### Explain HTML Like I'm 10 Years Old
> HTML is the skeleton of a web page. It does not say what color anything is or how big — it only says "this is a heading," "this is a paragraph," "this is a list." The browser reads HTML top-to-bottom, builds a tree in memory called the DOM, then asks CSS "how should this look?" and JavaScript "what should this do?" This is why a missing closing tag can shift the entire page layout — the browser's tree-builder has to guess where you meant to close it.

---

### 5 Deep Conceptual Questions

**Q1: What problem does HTML fundamentally solve, and why couldn't you solve it another way?**
> **A:** HTML solves giving structured meaning to text in a way that ANY program — a browser, a screen reader, a search crawler, an LLM scraper — can understand without negotiating a format first. Before HTML there was no shared, hyperlinkable document format that ran in any operating system. You can't replace it with JSON because JSON has no built-in tree-of-content semantics like "this is a heading that belongs to that section."

**Q2: What is the ONE mental model that makes everything about HTML click into place?**
> **A:** The browser turns HTML text into a tree of node objects (the DOM). Every CSS rule and every JavaScript API operates on that tree, not on the original text. Once you internalise "HTML is the build script for an in-memory tree," concepts like `document.querySelector`, CSS specificity, and event bubbling stop feeling magical.

**Q3: What is the most dangerous misconception about HTML? Show it with code.**
> **A:** That HTML is "just markup" with no impact on accessibility, SEO, or performance.
> ```html
> <!-- ❌ Looks the same visually, breaks screen readers and SEO -->
> <div class="header">
>   <div class="title" onclick="goHome()">My Site</div>
>   <div class="nav"><div onclick="go('about')">About</div></div>
> </div>
>
> <!-- ✅ Same look, but keyboard-accessible, screen-reader friendly, indexable -->
> <header>
>   <h1><a href="/">My Site</a></h1>
>   <nav><a href="/about">About</a></nav>
> </header>
> ```

**Q4: How does HTML interact with CSS and JavaScript at the runtime level?**
> **A:** The browser parses HTML into the DOM, parses CSS into the CSSOM, then COMBINES them into the Render Tree (visible nodes only). JavaScript can mutate the DOM, which triggers re-layout. Critical detail: a `<script>` without `async` or `defer` BLOCKS DOM parsing until it has been downloaded and executed — this is why scripts traditionally went at the bottom of `<body>`.

**Q5: Write a one-sentence definition of HTML that a senior engineer at a FAANG company would find technically precise.**
> **A:** "HTML is a declarative, tree-structured markup language whose tokens are parsed by the browser into the DOM — an in-memory object tree that serves as the single source of truth on which CSS layout, JavaScript mutation, and accessibility APIs all operate — which is why semantic tag choice has measurable downstream impact on rendering performance, screen-reader UX, and search ranking."
