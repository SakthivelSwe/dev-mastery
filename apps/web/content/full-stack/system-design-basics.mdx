## WHY
You know how to build a React frontend, a Node.js backend, and a PostgreSQL database. You know how to deploy it using Docker.
But what happens when your application needs to support 10 million concurrent users globally?
If you just build a massive monolithic server, it will melt under the load. 
**System Design** is the architectural planning required to build massive, globally distributed, highly available software systems. It is the transition from "writing code" to "architecting infrastructure." In senior engineering interviews, you will rarely be asked to reverse an array; you will be asked to stand at a whiteboard and design the architecture of Netflix from scratch.

## THEORY
### The Golden Triangle of System Design
Every architectural decision is a tradeoff between three core concepts (The CAP Theorem broadly touches on these):
1. **Scalability:** Can the system handle massive spikes in traffic without degrading performance? (Solution: Horizontal Scaling, Load Balancers).
2. **Availability:** If an Amazon data center is destroyed by a hurricane, does the website stay online? (Solution: Redundancy, Multi-Region Deployment).
3. **Latency:** How long does it take for a user in Australia to download an image from a server in New York? (Solution: CDNs, Edge Caching).

## CODE

You don't write "System Design" in a code editor. You design it on a whiteboard! Let's walk through the architectural evolution of a startup.

### Stage 1: The Monolith (0 to 1,000 Users)
You launch your startup. It is cheap and simple.
- **DNS (Domain Name System):** Routes `myapp.com` to your single server's IP address.
- **Single EC2 Server:** Runs both your Node.js code and your PostgreSQL database on the exact same physical hard drive.
*Flaw:* If the server reboots, the entire company is offline.

### Stage 2: Database Separation (1,000 to 100,000 Users)
The CPU is maxing out.
- **Separate DB:** You move the PostgreSQL database to its own dedicated, highly-optimized server (AWS RDS). The Node.js server now communicates with it over the internal network.
*Flaw:* If the Node.js server crashes, the site goes down.

### Stage 3: Horizontal Scaling (100,000 to 1 Million Users)
Traffic spikes randomly. A single Node server isn't enough.
- **Load Balancer (ALB):** You place an AWS Load Balancer in front of your application. The Load Balancer has a single public IP address.
- **Auto-Scaling Group:** You spin up 5 identical clones of your Node.js server. The Load Balancer mathematically distributes incoming traffic evenly across all 5 servers!
*Flaw:* The 5 Node.js servers are hammering the single PostgreSQL database with read requests, and the database is slowing down.

### Stage 4: Read Replicas and Caching (1 Million to 10 Million Users)
The database is the ultimate bottleneck.
- **Redis Cache:** You introduce a Redis RAM Cache. If data hasn't changed, the Node servers pull it instantly from Redis, bypassing the database entirely.
- **Read Replicas:** You configure PostgreSQL to create 3 Read-Only clones of itself. The Node.js servers send all `SELECT` queries to the clones, and only send `INSERT/UPDATE` queries to the primary Master database!
*Flaw:* Users in Australia are complaining that images load incredibly slowly because the servers are in New York.

### Stage 5: Global Distribution (10 Million+ Users)
You must move data physically closer to the users.
- **CDN (Content Delivery Network):** You move all static assets (Images, CSS, React JS bundles) to a CDN like Cloudflare. The CDN caches these files in hundreds of cities worldwide. The Australian user downloads the images from a server in Sydney in 5 milliseconds!
- **Message Queues:** You introduce Apache Kafka to decouple heavy background tasks (like sending massive email blasts or generating reports) into asynchronous workers, keeping the main HTTP servers lightning fast!

## REAL_WORLD
**Single Point of Failure (SPOF)**
The core philosophy of System Design is eliminating SPOFs.
If your entire architecture relies on a single Load Balancer, and that Load Balancer hardware fails, the fact that you have 10,000 healthy Node.js servers behind it doesn't matter; the site is dead.
Enterprise architecture relies on **Redundancy**.
You run Load Balancer A and Load Balancer B. You deploy your Node servers across three physically isolated AWS Data Centers (Availability Zones) so if one building loses power, the other two take over seamlessly. You replicate your database across different continents! High Availability is expensive, but for companies like Amazon, 1 minute of downtime costs millions of dollars.

## INTERVIEW
**Q: An interviewer asks: "We are designing a distributed database system for a massive global application. We need perfect data accuracy (Consistency) and 100% uptime (Availability) even if the network cables between our data centers are physically cut (Partition Tolerance). How do we architect this?"**
A: **You mathematically cannot. This violates the CAP Theorem.**
The CAP Theorem states that a distributed database can only guarantee TWO of the following three properties simultaneously:
1. **Consistency (C):** Every read receives the most recent write, or an error.
2. **Availability (A):** Every request receives a non-error response, without the guarantee that it contains the most recent write.
3. **Partition Tolerance (P):** The system continues to operate despite an arbitrary number of messages being dropped by the network between nodes.
Because networks *will* always fail eventually (Network Partitions are unavoidable), you are forced to choose between Consistency and Availability.
- **CP Systems (MongoDB, strict SQL):** If the network breaks between the Master and the Clone database, the system will refuse to answer user read requests (sacrificing Availability) to guarantee that nobody accidentally reads stale, incorrect data (maintaining Consistency). (Crucial for banking).
- **AP Systems (Cassandra, DynamoDB):** If the network breaks, the system will continue answering user requests (maintaining Availability) by serving stale, outdated data from the isolated Clones, resolving the conflicts later when the network heals (Eventual Consistency). (Crucial for Twitter likes).

## FEYNMAN CHECK

### Explain System Design Basics Like I'm 10 Years Old
> System design is laying out the major pieces of a software system and the connections between them — clients, load balancers, services, databases, caches, message queues, CDNs. You decide how requests flow, where data lives, where bottlenecks form, and how to scale each piece. The non-obvious part: system design is about TRADE-OFFS, not "best practices" — consistency vs availability, latency vs cost, simple vs scalable. There is no right answer, only choices appropriate to constraints.

---

### 5 Deep Questions
**Q1: Vertical vs horizontal scaling?**
> **A:** Vertical: bigger machine (easy, limits at ~$50k servers). Horizontal: more machines (complex but unbounded). Modern systems start vertical, go horizontal when one machine can't keep up. Stateless services scale horizontally trivially; stateful (DBs) require partitioning/sharding.

**Q2: ONE model.**
> **A:** "Client → CDN → LB → service tier (stateless, horizontal) → cache → DB (stateful, partitioned). Async work → queue → workers. Identify the bottleneck and scale that layer."

**Q3: Misconception with code.**
> **A:** ```yaml
> # ❌ Single point of failure — DB outage = total outage
> service: { db: postgres-primary }
> # ✅ Replica + failover + read-replica routing
> service: { db_writer: postgres-primary, db_reader: [postgres-replica-1, postgres-replica-2] }
> ```

**Q4: What is the CAP theorem?**
> **A:** In a network partition, you choose either Consistency or Availability — not both. Postgres = CP (refuses writes if partitioned). Cassandra = AP (accepts conflicting writes, reconciles later). Real systems mix: financial transactions = CP, social feed = AP. Knowing your domain's tolerance is essential.

**Q5: Senior one-liner.**
> **A:** "System design is the discipline of mapping requirements (RPS, latency SLO, durability, budget) to a topology (services, data stores, async layers, caches) by making explicit trade-offs — and the answer changes every time the constraints change."

## BUILD

### 🏗️ Mini Project: Sketch System Design for a URL Shortener
**Time:** 35 minutes

#### Step 1 — Requirements
```
- 100M URLs created/day, 10B redirects/day
- Read:write = 100:1
- 6-char short codes
- Sub-100ms p99 redirect latency globally
```

#### Step 2 — Capacity Math
```
Reads: 10B/day ÷ 86,400s = 115k req/s avg, peak ~3× = 350k
Writes: 100M/day = 1.2k req/s avg
Storage: 100M × 200B/row = 20GB/day, 7.3TB/year
```

#### Step 3 — Architecture
```
[Client]
  ↓
[Cloudflare CDN cache 5min] ← absorbs 90% of reads
  ↓
[Anycast LB]
  ↓
[Stateless read service (auto-scaling 20-200 pods)]
  ↓
[Redis cache, 95% hit rate, 1ms latency]
  ↓
[Postgres read replica, partitioned by short_code prefix]

Write path: Client → API service → ID generator (Snowflake) → write Postgres + invalidate cache
```

#### Step 4 — Failure Modes
```
- Cache cold start → 5min of slow redirects → solve with warmup
- DB primary fails → automatic failover ~30s → solve with read-only mode
- Region outage → DNS failover to backup region
```

**Expected Output:** Diagram + capacity math + failure plan you could whiteboard in 45 min.
