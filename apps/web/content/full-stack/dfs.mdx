# Depth-First Search (DFS)

Depth-First Search is a graph traversal technique that progresses as deep as possible along each branch before backtracking. It is highly suited for pathfinding, connectivity analysis, and topological sorting.

## 1. Core Mechanics & Theoretical Foundation

### The Stack as a Deep-First Driver
DFS uses a **Last-In, First-Out (LIFO)** strategy. Whether utilizing the system call stack via recursion or managing an explicit `java.util.Deque` implementation, the stack structure ensures that the most recently discovered vertex is prioritized for exploration.

```
[Stack Frame: Node A] -> [Stack Frame: Node B] -> [Stack Frame: Node C] (Deepest)
```

### Tree Traversal Variations
On trees, DFS can be categorized by when the root node is processed relative to its children:
* **Pre-order**: Node $\to$ Left $\to$ Right (Used for copying/cloning trees).
* **In-order**: Left $\to$ Node $\to$ Right (Used for binary search trees to retrieve sorted order).
* **Post-order**: Left $\to$ Right $\to$ Node (Used for bottom-up computations, such as calculating tree height or deleting a tree).

### Graph Traversal Attributes: Discovery and Finish Times
When running DFS on a graph, we track two key lifecycle timestamps for each vertex $u$:
1. **Discovery Time ($d[u]$)**: The step at which the node is first marked visited.
2. **Finish Time ($f[u]$)**: The step at which all of $u$'s descendants have been fully explored.

These intervals satisfy the **Parenthesis Theorem**: for any two vertices $u$ and $v$, either the intervals $[d[u], f[u]]$ and $[d[v], f[v]]$ are entirely disjoint, or one is completely nested within the other.

### Classification of Graph Edges
DFS classifies the edges of a graph into four categories:
1. **Tree Edges**: Edges in the depth-first forest. Edge $(u, v)$ is a tree edge if $v$ was first discovered by exploring $(u, v)$.
2. **Back Edges**: Edges pointing from a descendant to an ancestor in the DFS tree. **The presence of a back edge indicates a cycle in the graph.**
3. **Forward Edges**: Non-tree edges $(u, v)$ connecting a vertex $u$ to a descendant $v$ in a DFS tree.
4. **Cross Edges**: All other edges. They go between vertices in different trees or between vertices in the same tree that are not ancestors/descendants.

```
       [Ancestor]
        /      ^
  Tree /        \ Back Edge (Cycle!)
      v          \
   [Descendant] --+
```

### Recursive Risks: Stack Overflow
Recursive DFS relies on the JVM's call stack. In deep graphs (such as a linked list graph with $> 10,000$ nodes), recursive DFS will throw a `StackOverflowError`. For production-grade applications, an **iterative implementation with an explicit heap-allocated stack** is preferred.

## 2. Mathematical Proofs & Complexity Derivations

### Time Complexity: $O(V + E)$
The proof mirrors BFS:
1. Every vertex is transitioned from `WHITE` (unvisited) to `GRAY` (discovered) to `BLACK` (finished) exactly once. This takes $O(V)$ state transitions.
2. During the lifetime of a vertex, its adjacency list is scanned exactly once. The sum of the lengths of all adjacency lists is $E$ (for directed graphs) or $2E$ (for undirected graphs).
3. Thus, the total execution time is:
$$\text{Total Time} = O(V + E)$$

### Space Complexity: $O(V)$
In the worst-case scenario (a completely linear graph, such as a single line of $V$ vertices), the recursion stack or explicit stack will store $V$ frames. Thus, the worst-case space complexity is $O(V)$.

Let's trace DFS on a directed graph:
* Vertices: `0, 1, 2, 3`
* Edges: `(0->1), (1->2), (2->3), (0->2)`

| Step | Node | Stack (Call Stack) | Visited | Action / State Change |
| :--- | :--- | :--- | :--- | :--- |
| **0** | — | `[]` | `{}` | Start DFS at source `0`. |
| **1** | `0` | `[0]` | `{0}` | Discover `0`. Move to neighbor `1`. |
| **2** | `1` | `[0, 1]` | `{0, 1}` | Discover `1`. Move to neighbor `2`. |
| **3** | `2` | `[0, 1, 2]` | `{0, 1, 2}` | Discover `2`. Move to neighbor `3`. |
| **4** | `3` | `[0, 1, 2, 3]` | `{0, 1, 2, 3}`| Discover `3`. No neighbors. Finish `3`. |

## 5. Implementation (Java & Python)

### Java 21 Implementation
```java
package dsa.dfs;

import java.util.*;

public final class DepthFirstSearch {

    public enum NodeColor { WHITE, GRAY, BLACK }

    public static List<Integer> traverseIterative(Map<Integer, List<Integer>> graph, int source) {
        Objects.requireNonNull(graph, "Graph cannot be null");
        
        List<Integer> traversalOrder = new ArrayList<>();
        Deque<Integer> stack = new ArrayDeque<>();
        Set<Integer> visited = new HashSet<>();

        stack.push(source);

        while (!stack.isEmpty()) {
            int current = stack.pop();

            if (!visited.contains(current)) {
                visited.add(current);
                traversalOrder.add(current);

                List<Integer> neighbors = graph.getOrDefault(current, Collections.emptyList());
                // Push neighbors in reverse order to preserve left-to-right exploration
                for (int i = neighbors.size() - 1; i >= 0; i--) {
                    int neighbor = neighbors.get(i);
                    if (!visited.contains(neighbor)) {
                        stack.push(neighbor);
                    }
                }
            }
        }
        return traversalOrder;
    }
}
```

## 6. Interview Deep Dive

### Probing Questions & Expert Answers

1. **How do you detect cycles in an undirected graph using DFS vs. a directed graph?**
   * *Answer*: 
     * In an **undirected graph**, we pass the `parent` node to the recursive DFS call. If we encounter a visited node that is not the parent of the current node, a cycle exists.
     * In a **directed graph**, a simple visited set is insufficient. We must track nodes in the current recursion stack (the "active path") using a state array (e.g., `WHITE` = unvisited, `GRAY` = visiting, `BLACK` = finished). A cycle is detected if we hit a `GRAY` node.

2. **How can you find the strongly connected components (SCCs) of a directed graph using DFS?**
   * *Answer*: Use **Kosaraju's Algorithm** or **Tarjan's Algorithm**. 
     * Kosaraju's runs DFS on the original graph to get ordering by finish times, transposes the graph, and runs DFS on the transposed graph in reverse order of finish times.
     * Tarjan's uses a single DFS pass tracking `low-link` values to identify SCCs on the fly.

## WHY
Imagine you are trapped in a physical maze. How do you get out? You pick a path, and you walk straight down it until you hit a dead end. Once you hit a dead end, you trace your steps backward (backtrack) to the last intersection, and try a different path.
This is exactly how **Depth-First Search (DFS)** works. It plunges as deep as mathematically possible into a graph or tree before it ever considers looking at alternative neighbors. 
DFS is incredibly powerful. Because it exhaustively explores paths to their conclusion, it is the foundational algorithm for solving Mazes, Sudoku puzzles, finding Connected Components, and detecting Cycles in graphs!

## THEORY
### The Stack Structure
While BFS uses a Queue (FIFO) to search level-by-level, DFS uses a **Stack (LIFO)**.
1. Push the starting node onto the Stack.
2. Pop a node. Mark it visited.
3. Push all of its *unvisited* neighbors onto the Stack.
4. Repeat.

Because Stacks process the *most recently added* items first, the algorithm physically plunges down a single chain of neighbors before it ever processes the neighbors left behind at the starting node!

## CODE

### Level 1 — Recursive DFS
Because the call stack of a computer program *is* a physical Stack, the easiest and most beautiful way to write DFS is using Recursion!

```java
import java.util.*;

public class GraphDFS {
    HashMap<Integer, List<Integer>> adjList = new HashMap<>();

    public void traverse(int startNode) {
        // The visited set prevents infinite loops!
        HashSet<Integer> visited = new HashSet<>();
        // Launch the recursive plunge!
        dfsRecursive(startNode, visited);
    }

    private void dfsRecursive(int current, HashSet<Integer> visited) {
        // 1. Mark the current node as visited immediately
        visited.add(current);
        System.out.println("Visited Node: " + current);

        // 2. Look at all neighbors
        List<Integer> neighbors = adjList.getOrDefault(current, new ArrayList<>());

        for (int neighbor : neighbors) {
            if (!visited.contains(neighbor)) {
                // 3. PLUNGE! We pause the current loop and dive into the neighbor!
                dfsRecursive(neighbor, visited);
            }
        }
    }
}
```

### Level 2 — Iterative DFS
If a graph is massively deep (e.g., a linked list of 100,000 nodes), recursive DFS will crash with a `StackOverflowError`. 
To fix this, we replace the invisible Call Stack with a physical `Stack` object!

```java
public void dfsIterative(int startNode) {
    Stack<Integer> stack = new Stack<>();
    HashSet<Integer> visited = new HashSet<>();

    stack.push(startNode);

    while (!stack.isEmpty()) {
        // Pop the top of the stack (the deepest node!)
        int current = stack.pop();

        // Have we seen it? If not, process it!
        if (!visited.contains(current)) {
            visited.add(current);
            System.out.println("Visited Node: " + current);

            // Push all neighbors onto the stack!
            for (int neighbor : adjList.getOrDefault(current, new ArrayList<>())) {
                if (!visited.contains(neighbor)) {
                    stack.push(neighbor);
                }
            }
        }
    }
}
```

### Level 3 — Cycle Detection
Imagine a package manager like NPM. Package A depends on B. Package B depends on C. What if Package C accidentally depends on A? You have an infinite loop (a Cycle)!
DFS is the best way to detect cycles in a Directed Graph.
We track three states for a node instead of just two:
1. `UNVISITED`
2. `VISITING` (Currently in the recursive call stack)
3. `VISITED` (Fully completely done processing)

```java
public boolean hasCycle(int numNodes) {
    // 0 = Unvisited, 1 = Visiting, 2 = Visited
    int[] state = new int[numNodes]; 

    for (int i = 0; i < numNodes; i++) {
        if (state[i] == 0) {
            if (dfsDetectCycle(i, state)) return true;
        }
    }
    return false;
}

private boolean dfsDetectCycle(int current, int[] state) {
    // 1. Mark as currently in the stack (VISITING)
    state[current] = 1;

    for (int neighbor : adjList.getOrDefault(current, new ArrayList<>())) {
        if (state[neighbor] == 1) {
            // CRITICAL: We hit a node that is CURRENTLY in our call stack!
            // This mathematically proves a Cycle exists!
            return true;
        }
        
        if (state[neighbor] == 0) {
            // Unvisited. Plunge!
            if (dfsDetectCycle(neighbor, state)) return true;
        }
    }

    // 2. We finished exploring all paths from this node. Mark as fully VISITED.
    state[current] = 2;
    return false;
}
```

## REAL_WORLD
**Finding Connected Components**
Imagine a massive social network. Is everyone on Earth connected by 6 degrees of separation? Or are there completely isolated clusters of users who don't connect to the main network?
These isolated clusters are called "Connected Components".
DFS finds them effortlessly. You write a loop that iterates through all 10 Billion users. If User 1 is unvisited, you launch a DFS. The DFS will paint (mark visited) every single user connected to User 1. Once the DFS finishes, you increment a `componentCount` variable. Then the loop continues to User 2. If User 2 is already painted, you skip them! You continue until you find an unpainted user, proving you found a brand new, isolated network!

## INTERVIEW
**Q: You want to find the Shortest Path out of a maze. You write a DFS algorithm. Does DFS guarantee the shortest path? If not, why?**
A: **No, DFS never guarantees the shortest path!**
DFS blindly plunges down the first available path. Imagine a maze where you can turn Left or turn Right to reach the exit. 
If turning Right takes 3 steps to reach the exit, and turning Left takes 50,000 steps looping around the entire maze to reach the exit...
If your DFS loop happens to check `Left` before `Right`, it will plunge down the 50,000 step path, hit the exit, and return that 50,000 step path as the answer! It never even checked the 3-step path!
If you need the Shortest Path in an unweighted graph or maze, you MUST use BFS, which guarantees the shortest path by searching level-by-level!
