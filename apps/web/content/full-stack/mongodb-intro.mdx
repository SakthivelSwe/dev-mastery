# MongoDB — Document Model vs Relational | Level: 2

### Layer 1: Deep Dive / Architectural Core
MongoDB’s storage architecture is driven by the **WiredTiger** storage engine, which employs a highly optimized B-Tree structure for indexes and a row-oriented layout for data files. WiredTiger utilizes a multi-version concurrency control (MVCC) engine to manage concurrent operations without locking entire databases or collections.

```
       +--------------------------------------------------------+
       |                     WiredTiger Engine                  |
       |  +--------------------------------------------------+  |
       |  |                 WiredTiger Cache                 |  |
       |  |  [ Active B-Tree Pages ]  [ Dirty Pages (MVCC) ] |  |
       |  +------------------------+-------------------------+  |
       |                           |                            |
       |               Checkpoints | (Every 60s or 2GB)         |
       |                           v                            |
       |  +------------------------+-------------------------+  |
       |  |                Disk / Storage Block              |  |
       |  |  [ Data Files (.wt) ]     [ Journal Files ]      |  |
       |  +--------------------------------------------------+  |
       +--------------------------------------------------------+
```

#### Memory Management & Eviction
WiredTiger allocates a dedicated cache size (by default, the larger of 50% of (RAM - 1 GB) or 256 MB). It reads data from disk into memory in the form of B-Tree pages. When memory pressure increases, background eviction threads target dirty pages, writing them to disk using a reconciliation process. 

#### The Journal
To ensure durability (ACID compliance), MongoDB writes operations to an in-memory journal buffer, which is flushed to disk (the Journal) sequentially every 100 milliseconds (or upon a write with `j: true`). In the event of a crash, WiredTiger replays the journal from the last consistent checkpoint (taken automatically every 60 seconds or when 2 GB of data has been written).

#### BSON (Binary JSON) Serialization
MongoDB does not store JSON directly. It serializes JSON into BSON, a binary format that introduces type-awareness, length prefixes, and byte-alignment for fast parsing. Key BSON-specific types include:
*   **`ObjectId` (12-byte hex)**: Optimized for distributed unique ID generation without network coordination.
    *   `4-byte timestamp` (seconds since epoch)
    *   `5-byte random value` (unique per process/machine)
    *   `3-byte counter` (starting with a random value)
*   **`Date`**: 64-bit integer representing milliseconds since the Unix epoch.
*   **`NumberDecimal` (Decimal128)**: 128-bit IEEE 754 decimal floating-point format. Crucial for financial calculations to prevent binary floating-point rounding errors.
*   **`Binary`**: Arbitrary binary data (UUIDs, cryptographic keys, compressed payloads).

#### Document Limits & Schema Philosophy
MongoDB enforces a **16MB limit** per BSON document. This hard limit prevents poor application design (such as unbounded arrays) from causing memory exhaustion during serialization/deserialization. 

*Schemaless does not mean "no schema."* It means **schema-flexibility**. The application code defines the implicit schema. Relying purely on application-level flexibility can lead to data corruption; hence, MongoDB 7.x supports server-side `$jsonSchema` validation to enforce document integrity at the engine level.

### Layer 2: Real-World Use Cases & Schema Design
#### Scenario: E-Commerce Product Catalog with Dynamic Attributes
Relational databases struggle with highly polymorphic products (e.g., a laptop has RAM/CPU, while shoes have size/color). This typically forces a slow EAV (Entity-Attribute-Value) anti-pattern in SQL. MongoDB natively handles this by storing attributes as subdocuments or key-value arrays.

```json
{
  "_id": ObjectId("64b1f8e9c9e8a7b6d5c4b3a2"),
  "sku": "TECH-LAP-001",
  "name": "Developer Pro Laptop",
  "category": "Electronics",
  "price": NumberDecimal("1499.99"),
  "attributes": [
    { "name": "CPU", "value": "Apple M3 Max", "type": "string" },
    { "name": "RAM", "value": 64, "type": "number", "unit": "GB" }
  ],
  "metadata": {
    "created_at": ISODate("2023-07-15T08:00:00Z"),
    "version": 1
  }
}
```

### Layer 4: Advanced Patterns & Optimization
#### Schema Evolution and Versioning Pattern
When your document structure changes over time, instead of running slow database migrations that lock tables (as in SQL), append a `schema_version` field to your documents.

```javascript
// Document Version 1
{ "_id": 1, "name": "John Doe", "phone": "123-456-7890", "schema_version": 1 }

// Document Version 2 (Migrated to nested object)
{ "_id": 2, "name": "Jane Smith", "contact": { "phone": "987-654-3210", "email": "jane@test.com" }, "schema_version": 2 }
```

Your application layer handles this gracefully:
```java
public User readUser(Document doc) {
    int version = doc.getInteger("schema_version", 1);
    if (version == 1) {
        String phone = doc.getString("phone");
        return new User(doc.getString("name"), new Contact(phone, null));
    } else {
        Document contactDoc = (Document) doc.get("contact");
        return new User(doc.getString("name"), new Contact(contactDoc.getString("phone"), contactDoc.getString("email")));
    }
}
```

### Layer 5: PostgreSQL Comparison Matrix

| Architectural Vector | MongoDB (7.x) | PostgreSQL (16+) |
| :--- | :--- | :--- |
| **Storage Model** | Document (BSON) stored in WiredTiger B-Trees. | Relational (tuples) in Heap tables with indexes (B-Tree, GIN). |
| **Write Path** | App -> Journal Memory -> Journal Disk (Sequential) -> Data Files (Checkpoint). | App -> Shared Buffers -> WAL (Write-Ahead Log) -> Heap. |
| **Dynamic Attributes** | Native. Subdocuments, indexing nested keys directly. | Supported via `JSONB`. GIN indexes required. |
| **Schema Enforcement** | Flexible. Optional server-side `$jsonSchema` validation. | Strict. DDL migrations required for structural changes. |
| **Horizontal Scale** | Native sharding (range, hash, zone-based routing). | Manual sharding or extensions (e.g., Citus). |
| **Memory Footprint** | Higher. BSON metadata stored per document. | Lower. Schema metadata stored once in catalog table. |

#### When MongoDB Wins:
1.  **Polymorphic Data**: E-commerce, catalogs, content management systems where entities have diverse structures.
2.  **Horizontal Scalability**: High write-throughput workloads requiring seamless sharding across multiple nodes.
3.  **Rapid Prototyping**: Fast-changing application models where schema migrations impede deployment cycles.

#### When PostgreSQL Wins:
1.  **Complex Ad-hoc Reporting**: Deep analytical queries requiring complex multi-table joins.
2.  **Strict Data Normalization**: Systems requiring declarative referential integrity constraints (FKs) enforced by the engine.
3.  **Low Storage Overheads**: Massive record sets where repeating schema keys (as in BSON) would dramatically inflate storage costs.

### Layer 6: Production Gotchas & Best Practices
1.  **The 16MB Limit Violation**: Avoid designing schemas where arrays grow indefinitely (e.g., storing all logs or comments inside a single document). *Mitigation*: Reference the child documents or use the Bucket Pattern.
2.  **WiredTiger Cache Starvation**: By default, containerized deployments (Docker/Kubernetes) may miscalculate host memory, causing WiredTiger to allocate more memory than the container limit, triggering OOM kills. *Mitigation*: Explicitly set `--wiredTigerCacheSizeGB` to match the container's limits.
3.  **BSON Key Bloat**: BSON stores field names inside every single document. Storing `{"long_descriptive_field_name": "value"}` millions of times wastes storage and memory. *Mitigation*: Use short, concise field names (e.g., `{"desc": "value"}`) or rely on compression (WiredTiger block compression defaults to Snappy; Zlib or Zstd can be configured for higher ratios).

## WHY
Traditional SQL databases require you to define a rigid schema before storing any data. If you later need to add a new field to a table, you must run an `ALTER TABLE` migration, which can lock the table and cause downtime on massive production databases.

MongoDB is a **Document Database** that stores data as flexible JSON-like documents (internally BSON — Binary JSON). There is no fixed schema! If one user document has a `phoneNumber` field and another doesn't, that's perfectly fine.

This schema flexibility makes MongoDB ideal for rapid prototyping, content management systems, and scenarios where your data structure evolves frequently.

### The Document Model
MongoDB stores data as **BSON Documents** (Binary JSON). A document is simply a JSON object with a guaranteed unique `_id` field. Documents in the same Collection can have completely different structures — there is no enforced schema at the database level.

### Level 2 — Building Blocks (Querying with Operators)
```javascript
// Comparison operators
db.products.find({ price: { $gt: 100 } });             // price > 100
db.products.find({ price: { $gte: 50, $lte: 200 } });  // 50 <= price <= 200
db.products.find({ category: { $in: ["Electronics", "Gaming"] } });

// Logical operators
db.products.find({
    $and: [
        { price: { $lt: 500 } },
        { stock: { $gt: 0 } }
    ]
});

// Querying arrays
db.users.find({ skills: "JavaScript" }); // Users with "JavaScript" in their skills array
db.users.find({ skills: { $all: ["React", "TypeScript"] } }); // Has BOTH

// Query nested documents using dot notation
db.users.find({ "address.country": "USA" });

// Projection: Return only specific fields (1 = include, 0 = exclude)
db.users.find({ username: "alice" }, { username: 1, email: 1, _id: 0 });
```

### Level 3 — Intermediate (UPDATE and DELETE)
```javascript
// UPDATE ONE document
db.users.updateOne(
    { username: "alice" },                  // filter
    { $set: { email: "alice_new@example.com" } }  // update
);

// ADD a field dynamically (no schema migration needed!)
db.users.updateOne(
    { username: "alice" },
    { $set: { isPremium: true, premiumSince: new Date() } }
);

// UPDATE array elements — push a new skill
db.users.updateOne(
    { username: "alice" },
    { $push: { skills: "Docker" } }
);

// UPDATE MANY: Add a default 'isActive' field to ALL users
db.users.updateMany({}, { $set: { isActive: true } });

// REPLACE ONE document entirely
db.users.replaceOne(
    { username: "alice" },
    { username: "alice", email: "alice@example.com", isActive: true }
);

// DELETE
db.users.deleteOne({ username: "charlie" });
db.users.deleteMany({ isActive: false });
```

### Level 4 — Advanced (Indexes)
```javascript
// Without indexes, MongoDB performs a COLLECTION SCAN — reading every document!
// Create an index on a frequently queried field
db.users.createIndex({ email: 1 });          // 1 = Ascending
db.users.createIndex({ username: -1 });       // -1 = Descending

// Unique Index: Enforces that no two documents have the same email
db.users.createIndex({ email: 1 }, { unique: true });

// Compound Index: Optimizes queries that filter by both fields
db.orders.createIndex({ userId: 1, createdAt: -1 });

// Text Index: Full-text search across string fields
db.products.createIndex({ name: "text", description: "text" });
db.products.find({ $text: { $search: "wireless gaming headset" } });

// Check index usage with explain()
```

## REAL_WORLD
**Embedding vs Referencing** is the fundamental design decision in MongoDB:

- **Embed** (denormalize): Store related data directly inside the parent document. Use when you always access the data together and the sub-document list is small and bounded (e.g., order items inside an order).
- **Reference** (normalize with ObjectId): Store just the `_id` of the related document. Use when the sub-documents are shared across many parents (e.g., a `product` referenced by thousands of `orders`), or when the list is unbounded (e.g., comments that can grow infinitely).

## INTERVIEW
**Q: Does MongoDB support transactions?**
A: Yes, since v4.0, MongoDB supports ACID multi-document transactions. However, they are significantly more expensive than simple single-document operations. MongoDB's document model is specifically designed so that 90% of operations don't need transactions (by embedding related data in one document, you get atomic updates for free).

**Q: How does MongoDB handle the ObjectId `_id` field?**
A: MongoDB auto-generates a 12-byte BSON ObjectId if you don't provide `_id`. It encodes a 4-byte timestamp, 5-byte random value, and 3-byte incrementing counter. This means ObjectIds are globally unique, roughly time-sortable (no need for a separate `createdAt` field for basic ordering!), and generated on the client driver without a database roundtrip.
