## WHY
Imagine building an e-commerce database. You create a single massive table called `Orders`. It contains:
`OrderID, Date, CustomerName, CustomerAddress, ItemName, ItemPrice`.
Bob buys a laptop. He moves to a new house. To update Bob's address, you have to find all 50 of his previous orders and update the address in every single row. If you miss one, the database has conflicting information (Data Anomaly). 
If you delete an order, you accidentally delete Bob's address entirely!
**Database Design (Normalization)** is the mathematical science of organizing tables and relationships to absolutely eliminate data redundancy, prevent update anomalies, and guarantee bulletproof data integrity across massive enterprise systems.

## THEORY
### The Three Normal Forms (Normalizing Data)
Normalization is a step-by-step process of breaking massive tables down into smaller, strictly focused tables.
1. **First Normal Form (1NF):** Eliminate repeating groups. Every cell must contain a single, atomic value. (e.g., You cannot put `"Shirt, Pants, Shoes"` into a single `Items` column. They must be separate rows).
2. **Second Normal Form (2NF):** Eliminate redundant data. (e.g., Bob's Name and Address should not be duplicated across 50 Order rows. They should be moved to a dedicated `Customers` table, and linked via a `CustomerID` Foreign Key).
3. **Third Normal Form (3NF):** Eliminate transitive dependencies. Columns must depend *only* on the Primary Key. (e.g., An `Orders` table shouldn't store `ItemName` and `ItemPrice`. It should only store `ItemID`. The Price belongs exclusively to the `Items` table!).

### Relationships (Entity-Relationship)
When you break tables apart, you must connect them using **Foreign Keys**.
- **One-to-One (1:1):** A User has exactly one UserProfile. (Foreign key on either table).
- **One-to-Many (1:N):** A User has many Orders. (Foreign key `user_id` goes on the Orders table).
- **Many-to-Many (N:M):** A Student has many Classes, and a Class has many Students. SQL cannot physically do this directly! You MUST create a third "Join Table" (e.g., `Enrollments`) containing `student_id` and `class_id` to bridge them!

## CODE

### 1. The Denormalized Nightmare (Bad Design)
This is what an amateur database looks like. It violates all Normal Forms.

```sql
-- DANGER: Highly Redundant!
CREATE TABLE orders_bad (
    order_id INT,
    customer_name VARCHAR(100),
    customer_address VARCHAR(200), -- Duplicated for every order!
    product_name VARCHAR(100),
    product_price DECIMAL,         -- Price might change tomorrow!
    quantity INT
);
```

### 2. The 3NF Normalized Architecture (Good Design)
We break the nightmare into 3 distinct tables: `Customers`, `Products`, and `Orders` (which acts as the bridge connecting them).

```sql
-- 1. The Customers Table (Only holds Customer data)
CREATE TABLE customers (
    customer_id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    address VARCHAR(200) NOT NULL
);

-- 2. The Products Table (Only holds Product data)
CREATE TABLE products (
    product_id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    price DECIMAL NOT NULL
);

-- 3. The Orders Table (The Bridge connecting 1:N relationships)
CREATE TABLE orders (
    order_id SERIAL PRIMARY KEY,
    -- FOREIGN KEYS: We only store the IDs, not the raw data!
    customer_id INT REFERENCES customers(customer_id),
    product_id INT REFERENCES products(product_id),
    quantity INT NOT NULL,
    order_date TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### 3. Querying Normalized Data (The JOIN)
Because the data is perfectly separated, how do we print a receipt showing the Customer's Name and the Product's Name? We use a `JOIN` to mathematically fuse the tables back together during the query!

```sql
SELECT 
    c.name AS Customer, 
    p.name AS Product, 
    o.quantity, 
    (p.price * o.quantity) AS TotalCost
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id
JOIN products p ON o.product_id = p.product_id
WHERE o.order_id = 101;
```

## REAL_WORLD
**Denormalization (Sacrificing Rules for Speed)**
In computer science, every rule has an exception. 
Normalization guarantees perfect data integrity, but it comes at a cost: **Performance**.
If you perfectly normalize a Twitter database, fetching a single Tweet requires running a `JOIN` across the `Tweets`, `Users`, `Likes`, `Retweets`, and `Media` tables. Running a massive 5-way JOIN for millions of tweets per second will melt the database CPU.
At massive scale, engineers intentionally violate Normalization rules (a process called **Denormalization**). They intentionally duplicate the `UserName` directly into the `Tweets` table! Yes, if the user changes their name, the database has to do a massive update operation across 10,000 tweets. But it mathematically eliminates the need to run the expensive `JOIN` on every single read query. In Read-Heavy systems (like Twitter), Denormalization is a necessary evil for performance!

## INTERVIEW
**Q: An interviewer looks at your beautifully normalized e-commerce database. They say: "If a Product costs $10 today, John buys it, and the `Orders` table saves the `product_id`. Tomorrow, the CEO raises the price of the product to $20 in the `Products` table. If John prints his receipt from yesterday, what happens?"**
A: **The receipt will retroactively charge John $20, completely destroying the financial accuracy of the application.**
This is a classic database design trap. 
Normalization rule 3NF states that the `Orders` table should not duplicate the `ItemPrice`. 
However, financial data is **Historical and Immutable**. 
An Order is a snapshot in time. If you only store the `product_id`, the Order will always `JOIN` to fetch the *current* live price of the product, retroactively altering past financial records when prices change.
*The Fix:* To protect financial history, you MUST intentionally violate strict normalization! The `Orders` table (or an `Order_Items` table) MUST contain a hardcoded `purchase_price` column. When the order is placed, you copy the live price into that column. This permanently locks the historical price into the order, ensuring future price changes never affect past receipts!

## FEYNMAN CHECK

### Explain Database Design Like I'm 10 Years Old
> Database design = deciding what data lives in which tables and how they reference each other. Normalisation (1NF/2NF/3NF) eliminates duplication. Foreign keys enforce integrity. Indexes speed up lookups. The non-obvious part: over-normalisation (10 JOINs to read one screen) and under-normalisation (denormalised JSON columns) both cause production pain. The right answer is "normalised for writes, denormalised views for reads" — separating concerns.

---

### 5 Deep Questions
**Q1: Why normalise?**
> **A:** Single source of truth — update once, reflected everywhere. Avoids inconsistency (user changes email, only one row updated). Reduces storage. Cost: more JOINs to read.

**Q2: ONE model.**
> **A:** "Normalise the source-of-truth tables. Index for read patterns. Use materialised views or caches for read-heavy denormalisation. Use foreign keys + constraints for integrity."

**Q3: Misconception with code.**
> **A:** ```sql
> -- ❌ Storing CSV in a column — breaks JOINs, breaks queries
> CREATE TABLE posts (id INT, tags TEXT);  -- 'javascript,react,nodejs'
> -- ✅ Many-to-many junction table
> CREATE TABLE post_tags (post_id INT, tag_id INT, PRIMARY KEY (post_id, tag_id));
> ```

**Q4: When use a many-to-many junction table vs an array column (Postgres)?**
> **A:** Junction table: queryable, indexable, supports cardinality limits, joinable. Array column: faster reads for nested aggregates, harder to constrain. Junction is default; arrays for small fixed-size sets where JOIN cost matters.

**Q5: Senior one-liner.**
> **A:** "Database design balances integrity (normalisation + constraints) against read performance (indexes + denormalised views) — getting it wrong on day one costs 10× the work to migrate at scale."

## BUILD

### 🏗️ Mini Project: Design a Blog Schema
**Time:** 30 minutes

#### Step 2 — Schema
```sql
CREATE TABLE users (id SERIAL PRIMARY KEY, email TEXT UNIQUE NOT NULL, created_at TIMESTAMPTZ DEFAULT now());
CREATE TABLE posts (id SERIAL PRIMARY KEY, user_id INT REFERENCES users(id) ON DELETE CASCADE, title TEXT NOT NULL, body TEXT, published_at TIMESTAMPTZ);
CREATE INDEX idx_posts_user ON posts(user_id);
CREATE INDEX idx_posts_published ON posts(published_at) WHERE published_at IS NOT NULL;

CREATE TABLE tags (id SERIAL PRIMARY KEY, name TEXT UNIQUE NOT NULL);
CREATE TABLE post_tags (post_id INT REFERENCES posts(id) ON DELETE CASCADE, tag_id INT REFERENCES tags(id) ON DELETE CASCADE, PRIMARY KEY (post_id, tag_id));

CREATE TABLE comments (id SERIAL PRIMARY KEY, post_id INT REFERENCES posts(id) ON DELETE CASCADE, user_id INT REFERENCES users(id), body TEXT NOT NULL, created_at TIMESTAMPTZ DEFAULT now());
CREATE INDEX idx_comments_post ON comments(post_id);
```

#### Step 5 — Verify
```sql
EXPLAIN ANALYZE SELECT p.*, u.email FROM posts p JOIN users u ON u.id = p.user_id WHERE p.published_at > now() - INTERVAL '7 days';
```

**Expected Output:** Index Scan on idx_posts_published.
