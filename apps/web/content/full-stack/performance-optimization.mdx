## WHY
If your e-commerce website takes 5 seconds to load, 40% of users will abandon the page before they even see a product. Amazon calculated that a mere 100-millisecond delay in page load time costs them 1% in sales (billions of dollars).
Writing functional code is easy. Writing highly optimized code that runs at 60 Frames Per Second on a cheap 5-year-old Android phone on a 3G network is incredibly difficult. 
**Performance Optimization** is the science of minimizing network payloads, deferring non-critical JavaScript, mathematically indexing databases, and utilizing browser painting mechanics to make a web application feel instantaneously responsive.

## THEORY
### Core Web Vitals (The Google Standard)
Google actively penalizes the SEO ranking of slow websites. They measure speed using 3 strict metrics:
1. **LCP (Largest Contentful Paint):** How long does it take for the largest image or text block to physically appear on the screen? (Goal: < 2.5 seconds).
2. **FID (First Input Delay) / INP (Interaction to Next Paint):** When the user taps a button, how long does it take for the browser to visually respond? (Goal: < 100 milliseconds).
3. **CLS (Cumulative Layout Shift):** Does the page suddenly violently shift downward because an image finally loaded, causing the user to accidentally click the wrong button? (Goal: 0).

## CODE

Let's look at frontend and backend optimization strategies.

### 1. Frontend: Code Splitting (Lazy Loading)
If you build a React app, Webpack bundles all your JS into a massive `bundle.js` file (e.g., 3 Megabytes). The user must download all 3MB before the page can load.
What if the user never visits the "Admin Dashboard" page? Why force them to download the code for it?
**Lazy Loading** allows you to chop the bundle into tiny pieces and only download code on demand!

```jsx
import React, { Suspense, lazy } from 'react';
import { BrowserRouter, Route, Routes } from 'react-router-dom';

// EAGER LOADING: This code is downloaded immediately.
import Home from './Home'; 

// LAZY LOADING: This code is physically chopped out of the main bundle!
// It will ONLY be downloaded from the server if the user actually clicks the Admin link!
const AdminDashboard = lazy(() => import('./AdminDashboard'));

export default function App() {
  return (
    <BrowserRouter>
      <Routes>
        <Route path="/" element={<Home />} />
        
        {/* We must wrap lazy components in Suspense to show a fallback while downloading! */}
        <Route path="/admin" element={
          <Suspense fallback={<div>Loading Dashboard Code...</div>}>
            <AdminDashboard />
          </Suspense>
        } />
      </Routes>
    </BrowserRouter>
  );
}
```

### 2. Frontend: Image Optimization
Images account for 80% of the weight of a typical webpage. Loading raw JPEGs is catastrophic for LCP.
Always use the modern `WebP` or `AVIF` formats (which are 50% smaller than JPEG with the exact same quality). More importantly, always use the `loading="lazy"` attribute!

```html
<!-- DANGEROUS: The browser halts rendering to download this massive image! -->
<img src="hero-banner.jpg" />

<!-- OPTIMIZED: The browser downloads the tiny WebP image ONLY when the user 
     scrolls down close to it! The width/height prevents Layout Shift (CLS)! -->
<img src="hero-banner.webp" width="800" height="600" loading="lazy" alt="Banner" />
```

### 3. Backend: Database Indexing
On the backend, the most common bottleneck is the Database.
If you query `SELECT * FROM users WHERE email = 'alice@example.com'`, the database physically scans every single one of the 10 million rows on the hard drive until it finds Alice (A Full Table Scan). This takes 5 seconds!

You must create an **Index**. An Index is a highly-optimized, alphabetized B-Tree data structure stored in RAM.

```sql
-- Create an Index on the email column!
CREATE INDEX idx_users_email ON users(email);

-- Now, this exact same query will execute in 1 millisecond!
-- The database traverses the B-Tree instantly instead of scanning the hard drive!
SELECT * FROM users WHERE email = 'alice@example.com';
```

## REAL_WORLD
**Debouncing and Throttling**
You build a Google Maps clone. Every time the user drags the map 1 pixel, the frontend fires a `fetch()` request to the backend to get the new pins.
If the user aggressively drags the map, the frontend fires 500 API requests in 2 seconds. The backend crashes, and the user's browser freezes.
You must implement **Debouncing** or **Throttling**.
- **Debouncing:** *"Wait until the user completely stops moving the mouse for 300 milliseconds, and then fire exactly 1 API request."* (Perfect for Search Autocomplete typing).
- **Throttling:** *"The user is moving the mouse continuously. I will enforce a strict speed limit and only allow 1 API request to fire every 500 milliseconds, ignoring all other movements."* (Perfect for Map Dragging or scrolling).
These two microscopic JavaScript techniques prevent accidental DDoS attacks against your own servers!

## INTERVIEW
**Q: An interviewer asks: "We want to improve our React application's performance. We wrapped every single component in `React.memo` and wrapped every function in `useCallback`. Surprisingly, the application is actually running SLOWER than before. Why did this happen?"**
A: **Over-memoization adds CPU overhead.**
`React.memo` tells React to cache a component and skip re-rendering it if its Props haven't changed. `useCallback` caches a function definition in memory.
However, calculating whether Props have changed (shallow equality comparison) requires actual CPU processing time! Allocating memory to store the cached functions takes RAM!
If a component is simple and renders incredibly fast natively (like a basic Button), the CPU cost of mathematically checking the memoization cache is actually *more expensive* than just letting React blindly re-render the button! 
Memoization should NEVER be used globally. You must actively profile the application using React DevTools, identify the specific, massively heavy components (like a Data Grid with 10,000 rows) that are causing measurable lag, and *only* memoize those specific bottlenecks!

## FEYNMAN CHECK

### Explain Performance Optimization Like I'm 10 Years Old
> Performance optimisation is a feedback loop: MEASURE → identify bottleneck → fix → measure again. Never optimise without measuring — you almost certainly guess wrong about where time goes. The non-obvious part: 80% of perceived performance is network + first paint, not CPU. A 50ms backend with 3s of bundle parsing FEELS slow; a 500ms backend with instant render FEELS fast.

---

### 5 Deep Questions
**Q1: What are the Core Web Vitals?**
> **A:** LCP (Largest Contentful Paint) < 2.5s — perceived load speed. FID/INP (Interaction to Next Paint) < 200ms — responsiveness. CLS (Cumulative Layout Shift) < 0.1 — visual stability. Google ranks SEO partly on these — they're not optional for content sites.

**Q2: ONE model.**
> **A:** "Optimisation = profile → identify p95 bottleneck → fix → re-profile. Frontend = network, parse, render. Backend = DB, IO, CPU. Each layer has different tools (Chrome DevTools, APM, flame graphs)."

**Q3: Misconception with code.**
> **A:** ```js
> // ❌ Micro-optimisation that doesn't matter
> for (let i = 0, len = arr.length; i < len; i++) {} // saves 0.001ms
> // ✅ Macro-optimisation that does
> const results = await Promise.all(ids.map(fetchUser)); // parallelise 10 serial fetches
> ```

**Q4: When is asynchronous parallelism the right fix?**
> **A:** When operations are independent (no data dependency). Fetching 10 user records: parallel. Computing then storing: serial. Use `Promise.all` for unbounded parallel, `p-limit` for bounded (avoid hammering downstreams).

**Q5: Senior one-liner.**
> **A:** "Performance optimisation is a measurement discipline — profile, find the p95 bottleneck, fix it, re-profile — because intuition is wrong 80% of the time and the cost of optimising the wrong thing is opportunity cost on real bottlenecks."

## BUILD

### 🏗️ Mini Project: Profile and Fix a Slow Endpoint
**Time:** 35 minutes

#### Step 1 — Slow Endpoint
```js
// server.js
app.get('/api/dashboard', async (req, res) => {
  const user = await db.users.findById(req.user.id);     // 50ms
  const orders = await db.orders.findByUser(req.user.id); // 80ms
  const prefs = await db.prefs.findByUser(req.user.id);   // 30ms
  // Total: 160ms serial
  res.json({ user, orders, prefs });
});
```

#### Step 2 — Profile
```bash
node --prof server.js
# Hit endpoint 1000 times with autocannon
autocannon http://localhost:3000/api/dashboard
# Process the profile
node --prof-process isolate-0x...-v8.log > prof.txt
```

#### Step 3 — Optimise: Parallel
```js
app.get('/api/dashboard', async (req, res) => {
  const [user, orders, prefs] = await Promise.all([
    db.users.findById(req.user.id),
    db.orders.findByUser(req.user.id),
    db.prefs.findByUser(req.user.id),
  ]);
  // Total: 80ms (max of the three)
  res.json({ user, orders, prefs });
});
```

#### Step 4 — Cache Hot Reads
```js
const prefs = await cached(`prefs:${req.user.id}`, 300, () => db.prefs.findByUser(req.user.id));
// Future reads: 1ms instead of 30ms
```

#### Step 5 — Verify
```bash
autocannon http://localhost:3000/api/dashboard
# Before: 160ms p99; After parallel: 80ms; After cache: 50ms
```

**Expected Output:** 3× latency reduction without changing logic.
