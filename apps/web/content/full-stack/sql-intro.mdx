# SQL and Relational Model — Tables, Keys, ACID | Level: 1

The relational model, formulated by Edgar F. Codd in 1970, defines a mathematical approach to managing data using first-order predicate logic. Rather than navigating physical pointers (as in network or hierarchical databases), data is represented logically as relations (tables) composed of tuples (rows) and attributes (columns). 

---

### Core Relational Concepts & Codd's Principles

Codd defined 12 rules (often numbered 0 to 12) that a system must meet to be considered a truly relational database management system (RDBMS). Key principles include:

1. **The Information Rule (Rule 1):** All information in the database is represented explicitly at the logical level and in exactly one way — by values in tables.
2. **Guaranteed Access Rule (Rule 2):** Every individual datum (atomic value) is logically addressable by providing the table name, primary key value, and column name.
3. **Systematic Treatment of Null Values (Rule 3):** The DBMS must support systematic representations of missing or inapplicable information, distinct from default values (e.g., empty strings or zero) and independent of data type.
4. **Physical and Logical Data Independence (Rules 7 & 8):** Changes to the physical storage structures (how data is written to disk) or logical structures (splitting tables) must not break application queries.

#### Primary Keys and Foreign Keys

*   **Primary Key (PK):** A minimal set of attributes that uniquely identifies a tuple in a relation. It enforces the *Entity Integrity* constraint: no part of a primary key can be `NULL`.
*   **Foreign Key (FK):** An attribute or set of attributes in one table that references the primary key of another table. It enforces the *Referential Integrity* constraint: every non-null foreign key value must match an existing primary key value in the referenced table.

---

### ACID Properties: The Foundation of Transactions

Transactions are logical units of work. To maintain consistency across system crashes and concurrent operations, databases must guarantee ACID properties:

*   **Atomicity:** The "all-or-nothing" rule. Either all modifications within a transaction are committed to the database, or the entire transaction is rolled back. Relies on the **Write-Ahead Log (WAL)** to undo changes if a failure occurs.
*   **Consistency:** A transaction must transition the database from one valid state (satisfying all schema constraints, triggers, and foreign keys) to another. If a state transition violates a rule, the engine aborts the transaction.
*   **Isolation:** Concurrent transactions must execute without interfering with one another. The engine uses locking and **MVCC (Multi-Version Concurrency Control)** to give each transaction the illusion that it is running in isolation.
*   **Durability:** Once a transaction commits, its changes survive any subsequent system crash or power loss. This is guaranteed by flushing WAL frames to non-volatile storage (disk sync) before acknowledging the commit.

### SQL vs NoSQL Decision Matrix

| Dimension | Relational (SQL) | NoSQL (Document, Key-Value, Graph, Wide-Column) |
| :--- | :--- | :--- |
| **Data Schema** | Rigid, predefined, enforced at write time (Schema-on-Write). | Flexible, dynamic, schema-less or Schema-on-Read. |
| **Scaling** | Scale-up (vertical). Scale-out (horizontal) requires sharding or distributed SQL engines. | Scale-out (horizontal) via partitioning and replication. |
| **Joins** | Highly optimized, natively supported in memory/disk. | Typically performed in application code or denormalized. |
| **Transactions** | Strict ACID guarantees across multiple tables. | Eventual consistency or limited single-row ACID. |
| **Use Case** | Financial systems, ERPs, complex analytical joins. | High-throughput ingestion, catalog management, real-time caching. |

#### 1. PostgreSQL 16
*   **Architecture:** Multi-process (each connection spawns a backend process).
*   **Standards Compliance:** Highly compliant with ANSI SQL standards. Supports advanced types (JSONB, Arrays, Custom Types) and rich indexing (GiST, GIN, BRIN).
*   **Concurrency:** Robust MVCC implementation; readers never block writers, and writers never block readers.

#### 2. MySQL 8.x
*   **Architecture:** Multi-threaded (single process with worker threads).
*   **Storage Engine:** Pluggable, with InnoDB as the default (supports ACID, clustered indexes).
*   **Quirks:** Traditionally more permissive with data truncation and implicit conversions, though modern versions with strict SQL mode behave closer to standards.

#### 3. SQLite 3
*   **Architecture:** Serverless, self-contained, in-process library.
*   **Storage:** The entire database is stored in a single cross-platform disk file.
*   **Typing:** Dynamic typing (Manifest Typing). A column declared as `INTEGER` can store a string.

    contract_id BIGINT GENERATED ALWAYS AS IDENTITY,

        REFERENCES tenants (tenant_id)

    CONSTRAINT chk_dates CHECK (end_date > start_date)

## WHY
Every application that stores data needs a database. For 50 years, the industry standard for storing structured data has been **Relational Databases** — and the language used to communicate with them is **SQL (Structured Query Language)**.

Whether you are using PostgreSQL, MySQL, SQLite, or Oracle, they all understand SQL (with minor dialect differences). Learning SQL is one of the highest-ROI investments a developer can make, because structured data storage is a universal requirement of virtually every application ever built.

## THEORY
### The Relational Model
A Relational Database stores data in **Tables** (also called Relations). Each Table:
- Has a fixed set of **Columns** (also called Attributes or Fields), each with a specific data type.
- Contains **Rows** (also called Records or Tuples), where each row represents one entity.
- Has a **Primary Key** — one or more columns that uniquely identify each row.

Tables can be linked together using **Foreign Keys** — a column in one table that refers to the Primary Key of another table.

## CODE

### Level 1 — Beginner (Creating a Table)
DDL (Data Definition Language) commands create and modify the database structure.

```sql
-- Create a table for users
CREATE TABLE users (
    id        SERIAL PRIMARY KEY,   -- Auto-incrementing integer, the unique identifier
    username  VARCHAR(50) NOT NULL UNIQUE, -- Max 50 chars, required, must be unique
    email     VARCHAR(100) NOT NULL UNIQUE,
    age       INT CHECK (age >= 18),       -- Must be 18 or older (a constraint)
    created_at TIMESTAMP DEFAULT NOW()     -- Automatically set to current time
);

-- Create a table for posts (references users)
CREATE TABLE posts (
    id         SERIAL PRIMARY KEY,
    user_id    INT NOT NULL REFERENCES users(id) ON DELETE CASCADE, -- Foreign Key!
    title      VARCHAR(200) NOT NULL,
    content    TEXT,
    created_at TIMESTAMP DEFAULT NOW()
);
```

### Level 2 — Building Blocks (Inserting and Querying Data)
DML (Data Manipulation Language) commands manipulate the data itself.

```sql
-- INSERT a row
INSERT INTO users (username, email, age)
VALUES ('alice', 'alice@example.com', 25);

-- INSERT multiple rows at once
INSERT INTO users (username, email, age)
VALUES
    ('bob',     'bob@test.com',     30),
    ('charlie', 'charlie@test.com', 22);

-- SELECT all columns from all rows
SELECT * FROM users;

-- SELECT specific columns
SELECT id, username, email FROM users;

-- SELECT with a WHERE filter
SELECT * FROM users WHERE age > 25;

-- SELECT with sorting and limiting
SELECT username, email
FROM users
ORDER BY created_at DESC
LIMIT 10 OFFSET 0; -- Pagination: Page 1 (first 10 results)
```

### Level 3 — Intermediate (UPDATE, DELETE, and Aggregate Functions)
```sql
-- UPDATE: Modify existing data
UPDATE users
SET email = 'alice_new@example.com'
WHERE id = 1;

-- DELETE: Remove rows
DELETE FROM users WHERE id = 3;
-- WARNING: Never run DELETE without a WHERE clause! It deletes every row in the table.

-- Aggregate Functions
SELECT COUNT(*) AS total_users FROM users;
SELECT AVG(age) AS average_age FROM users;
SELECT MAX(age) AS oldest_user FROM users;
SELECT MIN(age) AS youngest_user FROM users;

-- GROUP BY: Aggregate per category
SELECT age, COUNT(*) AS users_at_this_age
FROM users
GROUP BY age
ORDER BY age;

-- HAVING: Filter on aggregated results (WHERE cannot be used with aggregates)
SELECT age, COUNT(*) as user_count
FROM users
GROUP BY age
HAVING COUNT(*) > 5; -- Only show ages that have more than 5 users
```

### Level 4 — Advanced (NULL Handling)
`NULL` is a special value in SQL representing the absence of data. **`NULL` is NOT zero, and it's NOT an empty string.** NULL does not equal anything, not even itself. Any comparison with NULL returns NULL (not true or false).

```sql
-- NULL in comparisons: ALWAYS use IS NULL or IS NOT NULL
SELECT * FROM users WHERE bio IS NULL;     -- ✅ Correct
SELECT * FROM users WHERE bio = NULL;      -- ❌ WRONG! Always returns 0 rows!
SELECT * FROM users WHERE bio IS NOT NULL;

-- COALESCE: Returns the first non-NULL value in the list
-- Perfect for providing default values!
SELECT username, COALESCE(bio, 'No bio provided') AS display_bio
FROM users;

-- NULLIF: Returns NULL if both values are equal, otherwise returns the first value
-- Useful to avoid "Division by Zero" errors!
SELECT total_sales / NULLIF(total_orders, 0) AS avg_order_value
FROM sales_summary;
```

## REAL_WORLD
As a developer, you should understand at least basic SQL even if you are using an ORM (Object-Relational Mapper) like Hibernate or Prisma. 

ORMs translate your code into SQL automatically, which is very convenient. But they can generate extremely inefficient SQL behind the scenes (the infamous N+1 problem). Being able to look at the generated SQL, understand why it's slow, and optimize it (e.g., by adding an index or rewriting the query) is a crucial senior engineering skill.

Always enable SQL logging in development: `spring.jpa.show-sql=true` (Spring Boot) or `DEBUG=prisma:query` (Prisma).

## INTERVIEW
**Q: What is the difference between `DELETE`, `TRUNCATE`, and `DROP`?**
A: `DELETE FROM users WHERE id = 1` removes specific rows matching the filter. `TRUNCATE TABLE users` instantly removes ALL rows from the table (much faster than DELETE for bulk clearing, but cannot be rolled back in some databases). `DROP TABLE users` destroys the entire table structure, all its data, and all its indexes permanently. These are in increasing order of destructiveness!

**Q: What does `ON DELETE CASCADE` mean?**
A: If `posts.user_id` has `ON DELETE CASCADE` referencing `users.id`, then when a User is deleted from the `users` table, all their Posts in the `posts` table are automatically deleted too. Without this, attempting to delete a User who has Posts would fail with a Foreign Key Constraint violation error.

**Q: What is the difference between `WHERE` and `HAVING`?**
A: `WHERE` filters individual rows BEFORE they are grouped/aggregated. `HAVING` filters groups AFTER the `GROUP BY` aggregation has already happened. You cannot use aggregate functions like `COUNT()` inside a `WHERE` clause.
