## WHY
HTTP is a "Stateless" protocol. When Alice requests her profile page, the server returns it and immediately forgets who Alice is. When she clicks "Settings" one second later, the server looks at the request and says, *"Who are you? You must log in!"*
If users had to type their password for every single mouse click, the internet would be unusable.
**Authentication (Auth)** solves this. When a user logs in successfully once, the server gives them a VIP Wristband (a Token or a Cookie). The browser secretly attaches this wristband to every future request. The server sees the wristband, instantly knows who the user is, and grants access without asking for a password!

## THEORY
### Authentication vs. Authorization
These two words sound similar but mean completely different things.
- **Authentication (AuthN):** "Who are you?" (Checking usernames and passwords).
- **Authorization (AuthZ):** "What are you allowed to do?" (Checking if the user is a standard User or an Admin).

### The Two Architectures
1. **Stateful Sessions (Cookies/Redis):** The server gives the user a random string (e.g., `SessionID=123XYZ`). The server MUST maintain a massive database (often Redis) linking `123XYZ` to Alice. This is highly secure but difficult to scale across multiple servers.
2. **Stateless JWT (JSON Web Tokens):** The server gives the user a cryptographically signed JSON file containing their actual data (e.g., `{"name":"Alice","role":"Admin"}`). The server doesn't need a database! It just mathematically verifies its own signature. It scales infinitely, but is harder to revoke!

## CODE

We will implement a modern Stateless JWT Authentication flow in an Express.js backend.

### 1. Generating the JWT (Login)
When the user proves who they are (by sending the correct password), we generate the VIP wristband!

```javascript
const jwt = require('jsonwebtoken'); // npm install jsonwebtoken

// This must be a highly secure, random string hidden in a .env file!
const SECRET_KEY = "my_super_secret_key"; 

app.post('/api/login', async (req, res) => {
    const { username, password } = req.body;

    // 1. Authenticate (Check the database!)
    const user = await db.users.find({ username, password });
    if (!user) {
        return res.status(401).json({ error: "Invalid credentials" });
    }

    // 2. Generate the Token (The Payload)
    const payload = {
        userId: user.id,
        role: user.role
    };

    // 3. Sign it mathematically! Set it to expire in 1 hour.
    const token = jwt.sign(payload, SECRET_KEY, { expiresIn: '1h' });

    // 4. Send the wristband to the client!
    res.json({ token: token });
});
```

### 2. Protecting a Route (Middleware)
Now, we have a route `/api/dashboard`. Only logged-in users can see this. We create a "Middleware" function (a bouncer at the club door) to check for the wristband!

```javascript
// The Middleware Function (The Bouncer)
function authenticateJWT(req, res, next) {
    // The frontend must send the token in the headers: "Authorization: Bearer <token>"
    const authHeader = req.headers.authorization;

    if (!authHeader || !authHeader.startsWith("Bearer ")) {
        return res.status(401).json({ error: "Access Denied. No token provided." });
    }

    // Extract just the token string
    const token = authHeader.split(" ")[1];

    try {
        // Did WE sign this token? Has it expired? Did a hacker tamper with it?
        const decodedPayload = jwt.verify(token, SECRET_KEY);
        
        // Success! Attach the user data to the request so the next function can use it!
        req.user = decodedPayload;
        
        // Pass the request through to the actual route!
        next(); 
    } catch (err) {
        return res.status(403).json({ error: "Invalid or Expired Token." });
    }
}

// Apply the Bouncer to the route!
app.get('/api/dashboard', authenticateJWT, (req, res) => {
    // Because the bouncer let them in, req.user is guaranteed to exist!
    res.json({ message: `Welcome to your dashboard, User ID: ${req.user.userId}` });
});
```

### 3. The Frontend (Using the Token)
How does a React frontend actually use the token? It saves it to `localStorage`, and intercepts every `fetch()` request to attach it to the headers!

```javascript
// Assume we got this token from the login response
localStorage.setItem("authToken", token);

// When fetching the dashboard later...
async function fetchDashboard() {
    const token = localStorage.getItem("authToken");

    const response = await fetch("https://api.com/dashboard", {
        headers: {
            "Authorization": `Bearer ${token}` // Show the wristband!
        }
    });
    
    // ...handle response
}
```

## REAL_WORLD
**Storing Tokens (LocalStorage vs HttpOnly Cookies)**
In the code above, we stored the JWT in the browser's `localStorage`. This is the easiest method and the most widely taught in tutorials.
**It is also a massive security risk.**
If a hacker manages to inject a single line of malicious JavaScript into your website (Cross-Site Scripting / XSS), that script has full access to `localStorage`. The hacker can instantly steal the JWT and hijack the user's session!
In enterprise production applications, you should **never** store authentication tokens in `localStorage`. You should configure your backend to send the JWT inside an **HttpOnly Cookie**. 
An HttpOnly Cookie is a secure vault built into the browser. The browser automatically attaches the cookie to every HTTP request, but it physically prevents *any* JavaScript code (even your own!) from ever reading the cookie's contents. This makes the token 100% immune to XSS theft!

## INTERVIEW
**Q: You are using stateless JWTs. A user's account is compromised. The user clicks a "Log out on all devices" panic button. How do you implement this feature?**
A: **You can't (easily), which is the fatal flaw of stateless JWTs.**
Because the JWT is stateless, the server doesn't track it in a database. The token is cryptographically valid until its expiration time runs out (e.g., 1 hour). If you hit a "panic button" on the server, the server has no physical way to instantly invalidate the hacker's token, because the token lives entirely on the hacker's laptop!
To solve this, you must compromise on the "stateless" nature of JWTs. 
You must implement a **Token Revocation Blocklist**. When the panic button is pressed, the server writes that specific user's ID to a high-speed Redis database. Now, every time the JWT Bouncer middleware runs, it must *first* verify the mathematical signature, and *then* query the Redis database to see if the user's ID was recently revoked. This effectively turns the stateless JWT into a stateful session!
