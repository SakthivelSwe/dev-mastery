# The Prototype Chain and Object Representation

JavaScript uses a prototype-based inheritance model. Every object maintains an implicit link to another object, forming a chain that is traversed during property lookups.

---

## 1. The `[[Prototype]]` Internal Slot and `__proto__`

Every JavaScript object has an internal slot called `[[Prototype]]`. This slot is a direct pointer to another object or `null`.

```
+------------------+
|    myObject      |
|  - a: 1          |
|  - [[Prototype]] | ----> +------------------+
+------------------+       |  protoObject     |
                           |  - b:

## WHY
In traditional Object-Oriented languages like Java or C++, inheritance works through rigid Classes. A `Dog` class inherits from an `Animal` class. When the code compiles, the computer physically merges the properties of `Animal` into `Dog` to create a fixed blueprint in memory.

JavaScript does **not** have traditional classes. Everything is an Object. 
If JavaScript doesn't have classes, how does an Array "inherit" the `.push()` method? It uses a completely different architectural concept called **Prototypal Inheritance**. Instead of copying blueprints at compile time, JavaScript physically links living objects together in RAM using a hidden pointer. If you don't understand the Prototype Chain, you don't truly understand how JavaScript works under the hood.

## THEORY
### The Hidden Link (`__proto__`)
Every single object in JavaScript contains a hidden, secret property called `__proto__` (or `[[Prototype]]` in the specification). This property is simply a memory pointer that links directly to *another* object. 

### The Delegation Chain
When you try to access a property on an object (e.g., `user.toString()`), the V8 engine follows a strict algorithm:
1. It looks at the `user` object. Does it have a `toString` method? No.
2. It follows the hidden `__proto__` pointer to the parent object. Does the parent have it? Yes! It executes it.
3. If the parent doesn't have it, it follows the parent's `__proto__` pointer to the grandparent.
4. This forms the **Prototype Chain**. It keeps walking up the chain until it hits `null`. If it hits `null`, it returns `undefined`.

## CODE

### Level 1 — Beginner (Discovering the Chain)
```javascript
// We create a completely empty object.
const user = {};

// Wait, if it's empty, how can we call the .toString() method on it without crashing?!
console.log(user.toString()); // "[object Object]"

// WHERE DID THAT METHOD COME FROM?
// The JavaScript engine automatically attached a hidden pointer to 'user' 
// that points to the master 'Object.prototype' in memory!
// The master 'Object.prototype' is where the .toString() method lives!

console.log(user.__proto__ === Object.prototype); // true
```

### Level 2 — Building Blocks (Manual Prototypal Inheritance)
```javascript
// Let's create our own inheritance chain manually!

const animal = {
  eats: true,
  walk() {
    console.log("Animal is walking");
  }
};

const rabbit = {
  jumps: true
};

// We manually point the rabbit's secret pointer to the animal object!
// (Note: __proto__ is considered deprecated for manual assignment. 
// Use Object.setPrototypeOf() in real code).
rabbit.__proto__ = animal;

// The rabbit object DOES NOT have a 'walk' method or an 'eats' property!
// But when we ask for them, JS seamlessly walks up the chain and finds them on 'animal'!
console.log(rabbit.jumps); // true (Found on rabbit)
console.log(rabbit.eats);  // true (Found on animal)
rabbit.walk();             // "Animal is walking" (Found on animal)
```

### Level 3 — Intermediate (Constructor Functions)
```javascript
// Before ES6 introduced the 'class' keyword, this was how developers 
// created hundreds of objects that shared the same methods.

// 1. The Constructor Function
function User(name) {
  this.name = name;
  // We DO NOT put methods inside here! If we did, every time we created 
  // a new User, it would consume more RAM duplicating the function!
}

// 2. The Prototype Object
// We attach the method exactly ONCE to the Master Prototype object!
User.prototype.login = function() {
  console.log(`${this.name} logged in.`);
};

// 3. The 'new' keyword
// The 'new' keyword does the magic: It creates an empty object, 
// sets its __proto__ to 'User.prototype', and returns it!
const alice = new User("Alice");
const bob = new User("Bob");

alice.login(); // Alice logged in.
bob.login();   // Bob logged in.

// Both objects share the EXACT SAME function in memory!
console.log(alice.login === bob.login); // true
```

### Level 4 — Advanced (Array Prototypes and Polyfills)
```javascript
// Let's look at Arrays. 
const arr = [1, 2, 3];

// 'arr' is just an object. How does it get the .map() method?
// arr.__proto__ points to 'Array.prototype'!
console.log(arr.__proto__ === Array.prototype); // true

// And what is above Array? 
// Array.prototype.__proto__ points to the master 'Object.prototype'!
// This is why arrays have access to the .toString() method!

// POLYFILLING (The ultimate hack):
// What if you invent a brand new array method, and you want EVERY array 
// in your entire application to instantly have access to it?
// You just mutate the master Array.prototype!

Array.prototype.getSecondItem = function() {
  return this[1]; // 'this' points to the array calling the method!
};

const fruits = ["Apple", "Banana", "Cherry"];
console.log(fruits.getSecondItem()); // "Banana"

// WARNING: Mutating built-in prototypes in a large application is considered 
// incredibly dangerous and a massive anti-pattern, because it can conflict 
// with future JS updates or third-party libraries!
```

## REAL_WORLD
When ES6 was released in 2015, it introduced the `class` keyword. 
```javascript
class Dog extends Animal {
  bark() { console.log("Woof"); }
}
```
**Do not be fooled.** JavaScript classes are completely fake. They are "Syntactic Sugar" designed to make JavaScript look friendly to Java and C# developers. Under the hood, the JS engine instantly translates the `class` and `extends` keywords directly back into Constructor Functions and `__proto__` chain links.

Understanding the prototype chain is essential for debugging. Have you ever logged an object in the Chrome DevTools and seen a little dropdown arrow labeled `[[Prototype]]` at the bottom of the properties list? That is the Prototype Chain! Opening it allows you to see all the hidden methods the object inherited from its ancestors.

## INTERVIEW
**Q: What is the very top of the Prototype Chain?**
A: `Object.prototype`. All objects, arrays, and functions eventually link up to the master `Object.prototype` (which provides methods like `.toString()` and `.hasOwnProperty()`). If you check the prototype of the master object itself (`Object.prototype.__proto__`), it returns `null`. `null` signifies the absolute end of the chain.

**Q: If `alice` and `bob` both inherit a `status` property from their prototype, what happens if you execute `alice.status = "Active"`? Does `bob`'s status change?**
A: No! This is called "Property Shadowing". When you *read* a property, JS walks up the chain. But when you *write* a property, JS always creates it directly on the target object. `alice` will get a brand new personal `status` property physically attached to her object. When you ask for `alice.status`, JS finds it immediately on her object and never looks at the prototype. The prototype remains untouched, and `bob` will still inherit the original status.

**Q: Why shouldn't you manually reassign `obj.__proto__` in production code?**
A: Because modern JavaScript engines (like V8) aggressively optimize object property access by creating "Hidden Classes" (Shapes). Mutating the `__proto__` pointer of a living object instantly destroys these optimizations, forcing the engine to fall back to a significantly slower dictionary lookup mode. If you need to set a prototype, you should do it at the exact moment of creation using `Object.create(parentPrototype)`.

## FEYNMAN CHECK

### Explain Prototype Chain Like I'm 10 Years Old
> Every JS object has a hidden link `[[Prototype]]` to ANOTHER object — its prototype. When you access `obj.foo`, the engine first checks obj's OWN properties. If missing, it checks `Object.getPrototypeOf(obj).foo`. If still missing, it walks UP the chain until it reaches `null`. This is JavaScript's INHERITANCE mechanism — older than classes, more flexible, still underneath every class. Built-ins: `[1,2,3]` → Array.prototype → Object.prototype → null. `"hello"` → String.prototype → Object.prototype → null. `class Cat extends Animal` just sets `Cat.prototype.__proto__ = Animal.prototype`. The non-obvious payoff: you can add methods to ALL existing arrays via `Array.prototype.myMethod = ...` (but please don't, in production — it pollutes globals).

### 5 Deep Conceptual Questions

**Q1: Why does JavaScript use prototypal inheritance instead of class inheritance?**
> **A:** Historical accident — Brendan Eich wanted Self-style prototypes (1990s research language) and was asked to "make it look like Java." The result: prototype mechanism under the hood, class syntax bolted on top. The advantage: more dynamic — you can add methods at runtime, delegate to ANOTHER object, change prototype mid-life. The disadvantage: harder to reason about than classical OOP, easier to subtly break performance via shape changes.

**Q2: Mental model for the chain?**
> **A:** Three things to remember: (1) Every value has a prototype (except null at the chain's end). (2) Method lookup walks UP the chain; property writes always create OWN properties on the instance. (3) `instanceof` checks if a constructor's .prototype appears anywhere in the chain. Once you internalise "lookup walks up, writes stay local", every weird inheritance bug clicks.

**Q3: Most dangerous misconception?**
> **A:** Mutating Array.prototype:
> ```js
> // ❌ Polluting prototype affects EVERY array in the program
> Array.prototype.first = function() { return this[0]; };
> [1, 2, 3].first();   // 1 — works for everyone
> for (const x in [1, 2, 3]) console.log(x);   // '0', '1', '2', 'first' — leaked!
>
> // ✅ Use a free function or Object.defineProperty with enumerable:false
> Object.defineProperty(Array.prototype, 'first', {
>   value: function() { return this[0]; },
>   enumerable: false,
> });
> // Better: avoid prototype mutation entirely — wrap in a utility function
> ```

**Q4: How do hasOwnProperty, in, and Object.keys differ?**
> **A:** `'x' in obj` checks own OR inherited properties. `obj.hasOwnProperty('x')` checks only own (but breaks if hasOwnProperty was overridden — use `Object.hasOwn(obj, 'x')` ES2022). `Object.keys(obj)` returns own ENUMERABLE string keys. `for...in` walks own + inherited enumerable string keys (gotcha for polluted prototypes). For arrays use `for...of` (iterator) or indexed for-loop, not for-in.

**Q5: FAANG-grade definition?**
> **A:** "JavaScript's prototype chain is the language's inheritance mechanism — every object has an internal [[Prototype]] reference to another object (or null), and property access walks the chain until found — with classes (`extends`) configuring chain links syntactically, and Object.create / Object.setPrototypeOf manipulating them programmatically — providing dynamic, mutable inheritance that classes desugar onto."

#### Step 2 — Polyfills
```js
// proto.js

// Object.create — links the new object's [[Prototype]] to the given proto
export function create(proto) {
  function F() {}
  F.prototype = proto;
  return new F();
}

// instanceof — walks the chain
export function isInstanceOf(obj, ctor) {
  let cur = Object.getPrototypeOf(obj);
  while (cur !== null) {
    if (cur === ctor.prototype) return true;
    cur = Object.getPrototypeOf(cur);
  }
  return false;
}

// Tiny "class" using only prototypes
export function defineClass({ init, methods = {}, parent = null }) {
  function Ctor(...args) {
    if (parent) parent.call(this, ...args);
    if (init)   init.call(this, ...args);
  }
  if (parent) {
    Ctor.prototype = create(parent.prototype);   // chain to parent
    Ctor.prototype.constructor = Ctor;
  }
  for (const [name, fn] of Object.entries(methods)) {
    Object.defineProperty(Ctor.prototype, name, { value: fn, enumerable: false });
  }
  return Ctor;
}
```

#### Step 3 — Use The Factory To Build A Class Hierarchy
```js
// demo.js
import { defineClass, isInstanceOf } from './proto.js';

const Animal = defineClass({
  init(name) { this.name = name; },
  methods: {
    describe() { return `I am ${this.name}, a ${this.constructor.name}`; },
  },
});

const Dog = defineClass({
  parent: Animal,
  init(name, breed) { this.breed = breed; },
  methods: {
    bark() { return `${this.name} the ${this.breed} barks!`; },
  },
});

const rex = new Dog('Rex', 'corgi');
console.log(rex.describe());     // from Animal.prototype
console.log(rex.bark());          // from Dog.prototype

// Walk the chain manually
let cur = rex;
while (cur !== null) {
  console.log('chain:', cur);
  cur = Object.getPrototypeOf(cur);
}

console.log(isInstanceOf(rex, Dog));    // true
console.log(isInstanceOf(rex, Animal)); // true
console.log(rex instanceof Dog);         // true (native check matches)
```

#### Step 4 — Error Handling: hasOwn vs in
```js
// hasOwn.js
const proto = { inherited: true };
const obj   = Object.create(proto);
obj.own = true;

console.log('inherited' in obj);                     // true
console.log(Object.hasOwn(obj, 'inherited'));        // false
console.log(Object.hasOwn(obj, 'own'));              // true

// Buggy old pattern — breaks if obj has its own hasOwnProperty
const bad = { hasOwnProperty: () => 'lol' };
console.log(bad.hasOwnProperty('x'));                // 'lol' !!! shadowed
console.log(Object.prototype.hasOwnProperty.call(bad, 'x'));   // false — safe
console.log(Object.hasOwn(bad, 'x'));                // false — safest modern check
```

#### Step 5 — Tests
```js
// proto.test.js
import { describe, it, expect } from 'vitest';
import { create, isInstanceOf, defineClass } from './proto.js';

describe('create', () => {
  it('links prototype', () => {
    const proto = { greet() { return 'hi'; } };
    const obj = create(proto);
    expect(obj.greet()).toBe('hi');
    expect(Object.getPrototypeOf(obj)).toBe(proto);
  });
});

describe('isInstanceOf', () => {
  it('matches direct constructor', () => {
    const Animal = defineClass({ init(n) { this.n = n; } });
    const a = new Animal('rex');
    expect(isInstanceOf(a, Animal)).toBe(true);
  });
  it('walks the chain to parent', () => {
    const A = defineClass({ init() {} });
    const B = defineClass({ parent: A, init() {} });
    const b = new B();
    expect(isInstanceOf(b, A)).toBe(true);
    expect(isInstanceOf(b, B)).toBe(true);
  });
  it('returns false when not in chain', () => {
    const X = defineClass({ init() {} });
    const Y = defineClass({ init() {} });
    expect(isInstanceOf(new X(), Y)).toBe(false);
  });
});
