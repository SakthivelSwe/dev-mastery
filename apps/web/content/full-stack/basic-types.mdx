# Basic Types — string, number, boolean, any, unknown

TypeScript extends JavaScript's dynamic primitives with a robust static type layer. Understanding how the compiler represents primitives and manages safety-escape hatches is fundamental to writing secure code.

---

## 1. Primitive Types

TypeScript has direct equivalents for all JavaScript primitives:

```typescript
const username: string = "Alice";
const port: number = 8080; // Represents both integers and floating-point values
const isActive: boolean = true;
const maxSafeInt: bigint = 9007199254740991n;
const uniqueKey: symbol = Symbol("key");
```

---

## 2. Special Types: `any` vs. `unknown`

### `any` (The Escape Hatch)
The `any` type represents a dynamic, unchecked type. It opts out of all type checking. When you use `any`, you tell the compiler to trust you completely, which reintroduces runtime fragility.

```typescript
let dynamicValue: any = { greet: () => "Hello" };
dynamicValue.nonExistentMethod(); // Compiles successfully, but crashes at runtime!
```

### `unknown` (The Safe Alternative)
The `unknown` type is the type-safe counterpart of `any`. Anything is assignable to `unknown`, but `unknown` is **not** assignable to anything else (except `any` and itself) without explicit type narrowing or assertion.

```typescript
let safeValue: unknown = "Hello World";

// Compiler Error: TS18046: 'safeValue' is of type 'unknown'.
// const len: number = safeValue.length; 

// You must narrow the type first:
if (typeof safeValue === "string") {
  const len: number = safeValue.length; // Safe and allowed
}
```

## 3. `void` vs. `never`

| Type | Semantic Meaning | Common Use Case |
| :--- | :--- | :--- |
| `void` | The absence of any value. Typically indicates a function returns nothing (`undefined`). | Event handlers, operations with side effects. |
| `never` | A value that can **never** occur. Represents values that are unreachable. | Functions that throw exceptions, infinite loops, or exhaustive checks. |

```typescript
function logMessage(msg: string): void {
  console.log(msg);
  // Implicitly returns undefined
}

function throwError(msg: string): never {
  throw new Error(msg); // Never returns to the caller
}

function infiniteLoop(): never {
  while (true) {} // Execution never ends
}
```

## 4. Type Annotations vs. Type Inference

TypeScript is highly capable of inferring types based on initialization values. This is known as **type inference**. Explicit **type annotations** are only necessary when the compiler cannot infer the correct type, or to enforce a strict contract.

```typescript
// Type Inference (Preferred for local variables)
let x = "hello"; // Inferred as type 'string'
// x = 42; // Compiler Error: TS2322: Type 'number' is not assignable to type 'string'.

// Type Annotation (Necessary for parameter types, or when initializing without a value)
let y: string;
y = "world";
```

## 5. `null` and `undefined` under `strictNullChecks`

When `"strictNullChecks": true` is enabled in `tsconfig.json`, `null` and `undefined` have their own distinct types and are not assignable to primitive types.

```typescript
let name: string;
// name = null; // Compiler Error: TS2322: Type 'null' is not assignable to type 'string'.

let nullableName: string | null = null; // Correct way using a union type
```

### Difference in TypeScript Context
*   `undefined` means a variable has been declared but has not yet been assigned a value.
*   `null` is an assignment value representing no value or an empty object reference.

## 6. Type Erasure: Primitives and Special Types

### Side-by-Side Comparison

```typescript
// --- TypeScript Source (types.ts) ---
function processInput(input: unknown): string {
  if (typeof input === "string") {
    return input.toUpperCase();
  }
  return "Not a string";
}
```

```javascript
// --- Compiled JavaScript (tsc Output - types.js) ---
"use strict";
function processInput(input) {
    if (typeof input === "string") {
        return input.toUpperCase();
    }
    return "Not a string";
}
```
