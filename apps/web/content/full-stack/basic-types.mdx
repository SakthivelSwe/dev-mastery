# Basic Types — string, number, boolean, any, unknown

TypeScript mirrors JavaScript's primitive types and introduces special utility types to handle dynamic or uncertain data structures securely.

---

## 1. Primitive Types

TypeScript has direct type mappings for all JavaScript primitives.

```typescript
const username: string = "Alice";
const score: number = 42.5; // Represents both integers and floating-point values
const isActive: boolean = true;
const bigValue: bigint = 9007199254740991n;
const uniqueKey: symbol = Symbol("key");
```

---

## 2. Special Types: `any` vs. `unknown`

*   **`any`**: Opts out of type checking entirely. It allows you to perform any operation, read any property, or call any method on a variable. Using `any` bypasses the safety guarantees of TypeScript.
*   **`unknown`**: Represents any value but is type-safe. You cannot perform operations on an `unknown` variable without first narrowing its type via type guards or assertions.

### TypeScript Code
```typescript
// Bypassing check with 'any'
let unsafeValue: any = { greet: () => "Hello" };
unsafeValue.someNonExistentMethod(); // Compiles, but crashes at runtime!

// Safe handling with 'unknown'
let safeValue: unknown = "Hello World";

// Error: TS18046: 'safeValue' is of type 'unknown'.
const length1 = safeValue.length; 

// Type narrowing to safely access properties
if (typeof safeValue === "string") {
  const length2 = safeValue.length; // Allowed! Type is narrowed to 'string'.
}
```

### Compiled JavaScript (tsc output)
```javascript
"use strict";
let unsafeValue = { greet: () => "Hello" };
unsafeValue.someNonExistentMethod(); 
let safeValue = "Hello World";
if (typeof safeValue === "string") {
    const length2 = safeValue.length;
}
```

## 3. Void vs. Never

*   **`void`**: Indicates that a function returns no value (implicitly returns `undefined`).
*   **`never`**: Indicates a value that can *never* occur. It is used for function return types when the function throws an exception or enters an infinite loop.

### TypeScript Code
```typescript
function logMessage(msg: string): void {
  console.log(msg);
  // Implicitly returns undefined
}

function throwError(msg: string): never {
  throw new Error(msg); // Never returns to caller
}

function infiniteLoop(): never {
  while (true) {} // Never reaches end of execution
}
```

### Compiled JavaScript (tsc output)
```javascript
"use strict";
function logMessage(msg) {
    console.log(msg);
}
function throwError(msg) {
    throw new Error(msg);
}
function infiniteLoop() {
    while (true) { }
}
```

## 4. Type Annotations vs. Type Inference

*   **Type Annotation**: Manually writing the type (`let x: number = 5;`).
*   **Type Inference**: Allowing the TypeScript compiler to automatically determine the type based on the assigned value. Type inference is preferred in idiomatic TypeScript to avoid redundant code.

```typescript
// Redundant annotation
let name1: string = "Bob";

// Idiomatic inference (inferred as string)
let name2 = "Bob"; 

// Inference is highly effective for complex expressions
const elements = [1, 2, 3].map(x => x.toString()); // Inferred as string[]
```

## 5. Null and Undefined (with `strictNullChecks`)

With `"strictNullChecks": true` enabled, `null` and `undefined` are treated as distinct types and cannot be assigned to variables of other primitive types.

```typescript
let age: number = 30;

// Error: TS2322: Type 'null' is not assignable to type 'number'.
age = null; 

// Correct usage using Union Types
let nullableAge: number | null = 30;
nullableAge = null; // Allowed
```

### Difference in TypeScript Context:
*   `undefined`: Signifies that a variable has been declared but has not yet been assigned a value, or a property is missing from an object.
*   `null`: Represents an intentional absence of any object value.

```typescript
interface UserProfile {
  bio?: string; // Can be string or undefined
  avatar: string | null; // Must be present, but can explicitly be null
}
```
