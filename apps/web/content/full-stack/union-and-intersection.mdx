# Union Types and Intersection Types

Union and intersection types are fundamental to composing complex, flexible types from simpler building blocks without relying on inheritance hierarchies.

---

## 1. Union Types (`|`)

A union type represents a value that can be **one of several types**. You use the pipe (`|`) symbol to separate each type.

```typescript
function printId(id: number | string) {
  // Before using type-specific methods, you must narrow the union
  if (typeof id === "string") {
    console.log(id.toUpperCase());
  } else {
    console.log(id.toFixed(2));
  }
}
```

---

## 2. Intersection Types (`&`)

An intersection type combines multiple types into one. The resulting type will have **all properties** of all intersected types.

```typescript
interface Loggable {
  log: () => void;
}

interface Serializable {
  serialize: () => string;
}

type LoggableAndSerializable = Loggable & Serializable;

const doc: LoggableAndSerializable = {
  log: () => console.log("Logging..."),
  serialize: () => "{}"
};
```

## WHY
In the real world, data is rarely perfectly uniform. A function might return a `User` object on success, or an `Error` object on failure. A configuration setting might accept a `number` (for pixels) or a `string` (for percentages). 

TypeScript provides **Union Types (`|`)** to model data that can be "one of many things", and **Intersection Types (`&`)** to model data that combines "multiple things together". Mastering these two operators allows you to model highly dynamic JavaScript patterns with strict type safety.

## THEORY
### Unions (`|`) = Logical OR
A value can be **this OR that**.
If a value is a union of `A | B`, you can only safely access properties that are common to *both* `A` and `B` until you "narrow" the type down.

### Intersections (`&`) = Logical AND
A value must be **this AND that**.
It creates a new type that combines all properties from `A` and `B`. It is primarily used for combining object shapes.

## CODE

### Level 1 — Beginner (Basic Unions)
Unions are most commonly used for primitives or literal values.

```typescript
// 1. Primitive Unions
function printId(id: number | string) {
    // ❌ Error: Property 'toUpperCase' does not exist on type 'string | number'.
    // id.toUpperCase(); 
    
    // You must narrow the type first!
    if (typeof id === "string") {
        console.log(id.toUpperCase()); // ✅ TS knows it's a string here
    } else {
        console.log(id.toFixed(2));    // ✅ TS knows it's a number here
    }
}

// 2. Literal Unions (Extremely common in React props)
type ButtonVariant = "primary" | "secondary" | "danger";
type ButtonSize = "sm" | "md" | "lg";

function renderButton(variant: ButtonVariant, size: ButtonSize) { ... }

// ✅ Autocomplete will suggest these exact strings!
renderButton("primary", "md");
```

### Level 2 — Building Blocks (Object Unions)
When you union two objects, you can only access properties they both share.

```typescript
interface Bird {
    name: string;
    fly(): void;
}
interface Fish {
    name: string;
    swim(): void;
}

function moveAnimal(animal: Bird | Fish) {
    // ✅ Valid: Both share 'name'
    console.log(animal.name);
    
    // ❌ Error: Fish cannot fly!
    // animal.fly();
    
    // We use the 'in' operator to narrow the type
    if ("fly" in animal) {
        animal.fly(); // ✅ TS knows it's a Bird
    } else {
        animal.swim(); // ✅ TS knows it's a Fish
    }
}
```

### Level 3 — Intermediate (Intersections)
Intersections are used to mix multiple types together into one massive type.

```typescript
interface HasName { name: string; }
interface HasAge { age: number; }
interface HasEmail { email: string; }

// Combine all three interfaces into one!
type UserProfile = HasName & HasAge & HasEmail;

const user: UserProfile = {
    name: "Alice",
    age: 30,
    email: "alice@example.com"
};
```
*Note: This achieves the exact same result as extending interfaces (`interface UserProfile extends HasName, HasAge, HasEmail`), but using `&` is often preferred when combining dynamic `type` aliases.*

### Level 4 — Advanced (Discriminated Unions)
This is the single most important pattern in advanced TypeScript (heavily used in Redux and modern state management).

Instead of guessing what type an object is using the `in` operator, you give every object a **shared literal property** (a discriminator tag) so TypeScript can instantly narrow it!

```typescript
interface NetworkSuccess {
    status: "success"; // The Discriminator Tag
    data: string[];
}

interface NetworkError {
    status: "error";   // The Discriminator Tag
    errorMessage: string;
}

interface NetworkLoading {
    status: "loading"; // The Discriminator Tag
}

// The Discriminated Union
type NetworkState = NetworkSuccess | NetworkError | NetworkLoading;

function handleResponse(response: NetworkState) {
    // We switch on the shared discriminator property
    switch (response.status) {
        case "success":
            // TS knows this is NetworkSuccess
            console.log(response.data); 
            break;
        case "error":
            // TS knows this is NetworkError
            console.error(response.errorMessage); 
            break;
        case "loading":
            console.log("Loading...");
            break;
    }
}
```

## REAL_WORLD
**Never Intersecting Primitives**
What happens if you accidentally Intersect two primitives that cannot possibly overlap?
```typescript
type Confused = string & number;
```
TypeScript will evaluate this as the `never` type. It is impossible for a value to be both a `string` and a `number` at the exact same time. If you ever see a `never` type appearing unexpectedly in your codebase, it means you accidentally used an Intersection (`&`) when you meant to use a Union (`|`).

## INTERVIEW
**Q: What is the difference between `any` and `unknown` in TypeScript?**
A: Both allow a variable to hold any type of value. 
However, `any` turns off the TypeScript compiler completely. You can call `.fly()` on an `any` variable and TS won't warn you. 
`unknown` is a type-safe alternative. TS forces you to narrow the `unknown` type (via `typeof` or `instanceof`) before you are allowed to interact with it. Always prefer `unknown` over `any` when dealing with unpredictable data (like `JSON.parse` responses).

## FEYNMAN CHECK

### Explain Unions & Intersections Like I'm 10 Years Old
> A UNION (`A | B`) says "this value is EITHER an A OR a B." You can only use what's COMMON to both until you narrow it. An INTERSECTION (`A & B`) says "this value is BOTH an A AND a B simultaneously" — must have ALL properties of both. Unions describe ALTERNATIVES (a Response is success OR error); intersections describe COMBINATIONS (a User & Admin has all user props plus admin props). The non-obvious twist: unions are like "or", but on object types you can only access the INTERSECTION of their properties (the common subset) — opposite of intuition. `(Cat | Dog).bark()` fails because not every member has bark; you must narrow with `in 'bark'` or a discriminant first.

### 5 Deep Conceptual Questions

**Q1: What problem do discriminated unions solve?**
> **A:** Modelling "one of N variants" with type-safe access. `type Result = { kind: 'ok'; data: T } | { kind: 'err'; error: E }` lets TS NARROW the type based on the `kind` literal: inside `if (r.kind === 'ok')`, TS knows r is the ok branch with `data` available. This eliminates null-check chains, makes pattern matching exhaustive (with `never` guards), and replaces error-prone `if (err) ... else ...` shapes with provably correct switch statements.

**Q2: Mental model for union narrowing?**
> **A:** TS uses CONTROL FLOW ANALYSIS — it tracks the type at every point based on the runtime checks you've performed. `typeof x === 'string'` narrows to string. `'foo' in x` narrows to objects with foo. `x.kind === 'ok'` narrows discriminated unions. The compiler runs this in your IDE in real-time. The narrowed type is REVERTED when you assign to the variable or call a function that mutates it.

**Q3: Most dangerous misconception? Show with code.**
> **A:** Believing intersection of incompatible types is allowed:
> ```ts
> // ❌ Looks fine but produces `never`
> type Impossible = string & number;   // type is `never`
> const x: Impossible = ???;            // can't construct one
>
> // ❌ Mixing literal unions with intersection sneakily
> type WrongStatus = ('a' | 'b') & ('b' | 'c');   // narrows to 'b' only
>
> // ✅ Use union when alternatives, intersection when combining
> type LoggableUser = User & { log: () => void };
> ```

**Q4: How does narrowing interact with closures at runtime?**
> **A:** TS's control flow analysis is INTRA-FUNCTION. If you narrow `x: string | null` to string and then put `() => x.length` in a callback, TS may RE-WIDEN x to the original union inside the callback (because the callback could run later, after x changed). The fix is to capture the narrowed value in a const: `const s = x; if (s) callback(() => s.length)` — now s is provably `string` for the closure's lifetime.

**Q5: FAANG-grade definition?**
> **A:** "Union types (`A | B`) describe values that inhabit either type, accessing only the intersection of their public surface until narrowed; intersection types (`A & B`) describe values that simultaneously satisfy both type constraints — control-flow analysis narrows unions via typeof, instanceof, 'in', and discriminant property checks, producing the exhaustiveness guarantees that underpin algebraic data types (ADTs) in TypeScript."

#### Step 2 — The Result Type
```ts
// src/result.ts
export type Result<T, E> =
  | { kind: 'ok';  value: T }
  | { kind: 'err'; error: E };

export const ok  = <T>(value: T): Result<T, never> => ({ kind: 'ok',  value });
export const err = <E>(error: E): Result<never, E> => ({ kind: 'err', error });

export function map<T, U, E>(r: Result<T, E>, fn: (v: T) => U): Result<U, E> {
  return r.kind === 'ok' ? ok(fn(r.value)) : r;
}

export function mapErr<T, E, F>(r: Result<T, E>, fn: (e: E) => F): Result<T, F> {
  return r.kind === 'err' ? err(fn(r.error)) : r;
}

export function flatMap<T, U, E>(r: Result<T, E>, fn: (v: T) => Result<U, E>): Result<U, E> {
  return r.kind === 'ok' ? fn(r.value) : r;
}

export function unwrap<T, E>(r: Result<T, E>): T {
  if (r.kind === 'ok') return r.value;
  throw new Error(`unwrap on err: ${JSON.stringify(r.error)}`);
}
```

#### Step 3 — Use It (Replace try/catch)
```ts
// src/main.ts
import { ok, err, map, flatMap, type Result } from './result.js';

type ApiError = { code: 'NOT_FOUND' | 'NETWORK' | 'INVALID'; message: string };

function fetchUser(id: number): Result<{ id: number; name: string }, ApiError> {
  if (id <= 0)         return err({ code: 'INVALID',   message: 'id must be positive' });
  if (id === 999)      return err({ code: 'NOT_FOUND', message: 'no such user' });
  return ok({ id, name: `User ${id}` });
}

const r = flatMap(fetchUser(42), u => ok({ ...u, name: u.name.toUpperCase() }));

// Exhaustive switch — TS forces handling both branches
switch (r.kind) {
  case 'ok':  console.log('Got:', r.value); break;
  case 'err': console.log('Failed:', r.error.code, '-', r.error.message); break;
}
```

#### Step 4 — Error Handling: Exhaustiveness Check
```ts
function describeError(e: ApiError): string {
  switch (e.code) {
    case 'NOT_FOUND': return 'Resource missing';
    case 'NETWORK':   return 'Try again later';
    case 'INVALID':   return e.message;
    default: {
      const _: never = e.code;   // catches missed cases at compile time
      return _;
    }
  }
}
```
