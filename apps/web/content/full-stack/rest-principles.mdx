# REST — Constraints, Richardson Maturity Model

### 1. Deep Architectural Core & Mechanics

Representational State Transfer (REST) is an architectural style defined by Roy Fielding in his 2000 doctoral dissertation. It is not a protocol, but a set of architectural constraints that, when applied collectively, optimize for distributed hypermedia systems.

```
+-------------------------------------------------------------+
|                        Client Layer                         |
+-------------------------------------------------------------+
                              |
                              | HTTP Request (State Transition)
                              v
+-------------------------------------------------------------+
|                 API Gateway / Load Balancer                 |
+-------------------------------------------------------------+
                              |
                              | Routed Stateless Request
                              v
+-------------------------------------------------------------+
|                      Application Server                     |
|  [Stateless Processing Engine]                              |
|  - Validates request                                        |
|  - Maps URL to Resource representation                      |
|  - Executes Domain Logic                                    |
|  - Generates Hypermedia Links (HATEOAS Engine)             |
+-------------------------------------------------------------+
                              |
                              | Database Query / Cache Lookup
                              v
+-------------------------------------------------------------+
|                  Data Store / Cache Layer                   |
+-------------------------------------------------------------+
```

#### The Six Core Constraints
1. **Client-Server Architecture**: Separates user interface concerns from data storage concerns. This decoupling improves portability across multiple platforms and scalability by simplifying server components.
2. **Statelessness**: Each request from a client must contain all the information necessary to understand and process the request. The server must not store any session context about the client. Session state is kept entirely on the client.
3. **Cacheability**: Responses must implicitly or explicitly define themselves as cacheable or non-cacheable to prevent clients from reusing stale or inappropriate data, optimizing network efficiency.
4. **Layered System**: A client cannot ordinarily tell whether it is connected directly to the end server or to an intermediary (e.g., load balancer, cache, or gateway). Intermediaries can improve system scalability by enabling load balancing and shared caches.
5. **Code on Demand (Optional)**: Servers can temporarily extend or customize client functionality by transferring executable code (e.g., compiled components such as Java applets, or client-side scripts like JavaScript).
6. **Uniform Interface**: This is the defining constraint of REST. It decouples the architecture, allowing each part to evolve independently. The four sub-constraints of the Uniform Interface are:
   * **Identification of Resources**: Individual resources are identified in requests (e.g., using URIs). The resources themselves are conceptually separate from the representations returned to the client.
   * **Manipulation of Resources through Representations**: When a client holds a representation of a resource (including metadata), it has enough information to modify or delete the resource on the server, provided it has permission.
   * **Self-Descriptive Messages**: Each message includes enough information to describe how to process it. For example, the media type (e.g., `application/json`, `application/xml`) explicitly tells the parser how to decode the body.
   * **Hypermedia As The Engine Of Application State (HATEOAS)**: Clients deliver state transitions dynamically through hypermedia links included in resource representations. The client needs zero hardcoded knowledge about the workflow structure beyond the entry point URI.

#### Richardson Maturity Model (RMM)
The Richardson Maturity Model breaks down the implementation of RESTful APIs into four distinct levels of design quality and adherence to REST constraints:

* **Level 0 (The Swamp of POX)**: Uses HTTP purely as a transport protocol for Remote Procedure Calls (RPC). There is a single URI (typically `/api` or `/soap`) and a single HTTP method (usually `POST`). All operations, queries, and mutations are defined in the payload.
* **Level 1 (Resources)**: Introduces the concept of individual resources. Instead of making all requests to a single endpoint, the API exposes multiple endpoints representing discrete entities (e.g., `/users`, `/users/42`). However, it still typically uses a single HTTP method (often `POST`) for all actions.
* **Level 2 (HTTP Verbs)**: Leverages the full capabilities of HTTP native protocols. The API uses correct HTTP verbs (e.g., `GET` for safe retrieval, `POST` for creation, `PUT` for complete replacement, `DELETE` for removal) and returns appropriate HTTP status codes (e.g., `201 Created`, `404 Not Found`, `409 Conflict`).
* **Level 3 (Hypermedia Controls - HATEOAS)**: The pinnacle of REST. The server provides not only the resource representation but also dynamic hypermedia links that describe the available actions (state transitions) currently permitted for that resource based on its internal state and client permissions.

### 2. Protocol Specs & Serialization Mechanics

```http

Host: api.enterprise.com
Accept: application/hal+json
Authorization: Bearer eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...
```

#### OpenAPI 3.1.0 Specification
```yaml
openapi: 3.1.0
info:
  title: Order Management Service (HATEOAS compliant)
  version: 1.0.0
paths:
  /api/v1/orders/{id}:
    get:
      summary: Retrieve an order by ID with HAL-compliant hypermedia links
      parameters:
        - name: id
          in: path
          required: true
          schema:
            type: string
      responses:
        '200':
          description: Order details with applicable state transitions
          content:
            application/hal+json:
              schema:
                $ref: '#/components/schemas/OrderRepresentation'
components:
  schemas:
    OrderRepresentation:
      type: object
      properties:
        id:
          type: string
        status:
          type: string
          enum: [AWAITING_PAYMENT, PAID, FULFILLED, CANCELLED]
        totalAmount:
          type: number
          format: double
        currency:
          type: string
        createdAt:
          type: string
          format: date-time
        _links:
          type: object
          properties:
            self:
              $ref: '#/components/schemas/Link'
            customer:
              $ref: '#/components/schemas/Link'
            payment:
              $ref: '#/components/schemas/Link'
            cancel:
              $ref: '#/components/schemas/Link'
    Link:
      type: object
      required:
        - href
      properties:
        href:
          type: string
          format: uri
```
