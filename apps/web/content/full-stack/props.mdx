# Props — Passing Data, TypeScript Types, Children | Level: 2

## Props are Read-Only

Props are the configuration options passed into a component by its parent. Within the component, props are strictly **read-only** (immutable).

In React 18 and 19 (especially in strict mode), the props object is frozen. Attempting to modify props directly will throw a runtime error.

```typescript
interface UserCardProps {
  user: {
    name: string;
  };
}

export const UserCard = (props: UserCardProps) => {
  // CRITICAL VIOLATION: Do not mutate props!
  // props.user.name = "New Name"; // Will throw an error in strict mode
  
  // CORRECT: Treat props as immutable. If you need to modify it, create a copy or use state.
  const uppercaseName = props.user.name.toUpperCase();
  
  return <div>{uppercaseName}</div>;
};
```

## The Prop Drilling Problem

Prop drilling is the process of passing props through multiple levels of nested components to reach a deeply nested child that actually needs the data. The intermediate components do not use the props themselves; they merely act as a conduit.

```
[App] (holds user state)
  └── [Dashboard] (drills user prop)
        └── [Sidebar] (drills user prop)
              └── [UserProfile] (uses user prop)
```

While acceptable for 2–3 levels, deep prop drilling makes components rigid, highly coupled, and difficult to refactor.

### Solutions to Prop Drilling
1. **Component Composition**: Pass the fully constructed child component down instead of raw data.
2. **React Context**: Use the Context API to bypass intermediate components.
3. **State Management Libraries**: Use tools like Redux Toolkit or TanStack Query.

## Spreading Props

The spread operator (`...`) allows you to pass all properties of an object as individual props to a child component.

```typescript
interface ButtonProps extends React.ButtonHTMLAttributes<HTMLButtonElement> {
  variant: 'primary' | 'secondary';
  isLoading?: boolean;
}

export const Button = ({ variant, isLoading, ...restProps }: ButtonProps) => {
  return (
    <button 
      className={`btn btn-${variant}`} 
      disabled={isLoading}
      {...restProps} // Spreads standard attributes like onClick, type, id, tabIndex
    >
      {isLoading ? 'Loading...' : restProps.children}
    </button>
  );
};
```

### Risks of Spreading Props
* **Over-spreading**: You can accidentally pass non-standard HTML attributes to DOM elements, causing React to throw console warnings (e.g., passing custom props like `isLoading` directly to a `<button>` DOM node).
* **Implicit dependencies**: It obscures what properties the child component actually depends on, making the code harder to trace.

## Default Props in Modern React

In early versions of React, default props were defined using a static property on the component class (e.g., `Component.defaultProps`).

In modern functional React, you should use JavaScript’s native **destructuring default parameters** directly in the function signature.

```typescript
interface AvatarProps {
  imageUrl?: string;
  size?: 'sm' | 'md' | 'lg';
  altText?: string;
}

// Modern, native approach to default values
export const Avatar = ({
  imageUrl = '/default-avatar.png',
  size = 'md',
  altText = 'User avatar'
}: AvatarProps) => {
  const dimensions = size === 'sm' ? 32 : size === 'md' ? 64 : 128;

  return (
    <img 
      src={imageUrl} 
      alt={altText} 
      width={dimensions} 
      height={dimensions} 
      className="avatar"
    />
  );
};
```

## Advanced Children Typing Patterns

When typing children, you can use specialized types depending on how restrictive your component API needs to be.

```typescript
import { ReactNode, ReactElement } from 'react';

// 1. React.ReactNode (Most common, highly permissive)
// Allows strings, numbers, elements, arrays, portals, null, undefined
interface LayoutProps {
  children: ReactNode;
}

// 2. React.ReactElement (Restrictive)
// Requires EXACTLY one React element. Throws an error if a string, number, or array is passed.
interface StrictWrapperProps {
  children: ReactElement;
}

// 3. Render Prop / Component-as-a-Prop Pattern
// Passes a function that returns JSX, allowing dynamic arguments
interface TableProps<T> {
  data: T[];
  renderRow: (item: T, index: number) => ReactNode;
}

export function Table<T>({ data, renderRow }: TableProps<T>) {
  return (
    <table>
      <tbody>
        {data.map((item, index) => (
          <tr key={index}>{renderRow(item, index)}</tr>
        ))}
      </tbody>
    </table>
  );
}
```

## Component as a Prop Pattern

Instead of passing raw data and letting the child decide how to render it, you can pass a fully configured React component or a render function as a prop. This is highly effective for building customizable UI libraries.

```typescript
interface HeaderProps {
  title: string;
  renderActionKeys?: () => React.ReactNode; // Component as a prop pattern
}

export const Header = ({ title, renderActionKeys }: HeaderProps) => {
  return (
    <header className="app-header">
      <h1>{title}</h1>
      {renderActionKeys && (
        <div className="header-actions">
          {renderActionKeys()}
        </div>
      )}
    </header>
  );
};
