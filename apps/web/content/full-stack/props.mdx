# Props — Passing Data, TypeScript Types, Children | Level: 2

## Props are Read-Only

Props are the configuration options passed into a component by its parent. Within the component, props are strictly **read-only** (immutable).

In React 18 and 19 (especially in strict mode), the props object is frozen. Attempting to modify props directly will throw a runtime error.

```typescript
interface UserCardProps {
  user: {
    name: string;
  };
}

export const UserCard = (props: UserCardProps) => {
  // CRITICAL VIOLATION: Do not mutate props!
  // props.user.name = "New Name"; // Will throw an error in strict mode
  
  // CORRECT: Treat props as immutable. If you need to modify it, create a copy or use state.
  const uppercaseName = props.user.name.toUpperCase();
  
  return <div>{uppercaseName}</div>;
};
```

## The Prop Drilling Problem

Prop drilling is the process of passing props through multiple levels of nested components to reach a deeply nested child that actually needs the data. The intermediate components do not use the props themselves; they merely act as a conduit.

```
[App] (holds user state)
  └── [Dashboard] (drills user prop)
        └── [Sidebar] (drills user prop)
              └── [UserProfile] (uses user prop)
```

While acceptable for 2–3 levels, deep prop drilling makes components rigid, highly coupled, and difficult to refactor.

### Solutions to Prop Drilling
1. **Component Composition**: Pass the fully constructed child component down instead of raw data.
2. **React Context**: Use the Context API to bypass intermediate components.
3. **State Management Libraries**: Use tools like Redux Toolkit or TanStack Query.

## Spreading Props

The spread operator (`...`) allows you to pass all properties of an object as individual props to a child component.

```typescript
interface ButtonProps extends React.ButtonHTMLAttributes<HTMLButtonElement> {
  variant: 'primary' | 'secondary';
  isLoading?: boolean;
}

export const Button = ({ variant, isLoading, ...restProps }: ButtonProps) => {
  return (
    <button 
      className={`btn btn-${variant}`} 
      disabled={isLoading}
      {...restProps} // Spreads standard attributes like onClick, type, id, tabIndex
    >
      {isLoading ? 'Loading...' : restProps.children}
    </button>
  );
};
```

### Risks of Spreading Props
* **Over-spreading**: You can accidentally pass non-standard HTML attributes to DOM elements, causing React to throw console warnings (e.g., passing custom props like `isLoading` directly to a `<button>` DOM node).
* **Implicit dependencies**: It obscures what properties the child component actually depends on, making the code harder to trace.

## Default Props in Modern React

In early versions of React, default props were defined using a static property on the component class (e.g., `Component.defaultProps`).

In modern functional React, you should use JavaScript’s native **destructuring default parameters** directly in the function signature.

```typescript
interface AvatarProps {
  imageUrl?: string;
  size?: 'sm' | 'md' | 'lg';
  altText?: string;
}

// Modern, native approach to default values
export const Avatar = ({
  imageUrl = '/default-avatar.png',
  size = 'md',
  altText = 'User avatar'
}: AvatarProps) => {
  const dimensions = size === 'sm' ? 32 : size === 'md' ? 64 : 128;

  return (
    <img 
      src={imageUrl} 
      alt={altText} 
      width={dimensions} 
      height={dimensions} 
      className="avatar"
    />
  );
};
```

## Advanced Children Typing Patterns

When typing children, you can use specialized types depending on how restrictive your component API needs to be.

```typescript
import { ReactNode, ReactElement } from 'react';

// 1. React.ReactNode (Most common, highly permissive)
// Allows strings, numbers, elements, arrays, portals, null, undefined
interface LayoutProps {
  children: ReactNode;
}

// 2. React.ReactElement (Restrictive)
// Requires EXACTLY one React element. Throws an error if a string, number, or array is passed.
interface StrictWrapperProps {
  children: ReactElement;
}

// 3. Render Prop / Component-as-a-Prop Pattern
// Passes a function that returns JSX, allowing dynamic arguments
interface TableProps<T> {
  data: T[];
  renderRow: (item: T, index: number) => ReactNode;
}

export function Table<T>({ data, renderRow }: TableProps<T>) {
  return (
    <table>
      <tbody>
        {data.map((item, index) => (
          <tr key={index}>{renderRow(item, index)}</tr>
        ))}
      </tbody>
    </table>
  );
}
```

## WHY
If you build a `<Button>` component that says "Click Me", that's great. But what if you need one button that says "Login", another that says "Sign Up", and another that says "Submit"? 

You don't want to create three separate components (`<LoginButton>`, `<SignUpButton>`). You want to create ONE reusable `<Button>` component and dynamically pass it the text it should display. 

React solves this with **Props** (short for Properties). Props are how Parent components pass data DOWN to their Child components, allowing you to configure and customize the Child at runtime.

## THEORY
### What are Props?
Props are conceptually identical to arguments passed into a standard JavaScript function. 
When a Parent renders `<Avatar name="Alice" />`, React takes the attribute `name="Alice"`, wraps it into a single JavaScript Object `{ name: "Alice" }`, and passes that object as the *first argument* to the `Avatar` component function.

### The Golden Rule of Props
**Props are strictly Read-Only.** A Child component must NEVER attempt to mutate or change the props it was given. If a Parent passes `age={25}`, the Child cannot do `props.age = 26`. Data in React flows strictly in one direction: Top-Down.

### Level 1 — Beginner (Passing and Reading Props)
```javascript
// THE CHILD COMPONENT
// 1. React passes a single 'props' object as the first argument!
export function UserBadge(props) {
  return (
    <div className="badge">
      {/* 2. We read the data out of the object */}
      <h2>{props.name}</h2>
      <p>Role: {props.role}</p>
    </div>
  );
}
```

```javascript
// THE PARENT COMPONENT
import { UserBadge } from './UserBadge';

export default function App() {
  return (
    <div>
      {/* 3. We pass data down using HTML-like attributes! */}
      <UserBadge name="Alice" role="Admin" />
      <UserBadge name="Bob" role="User" />
    </div>
  );
}
```

### Level 2 — Building Blocks (Destructuring Props)
Typing `props.name` and `props.role` over and over gets tedious. The industry standard is to use **JavaScript Object Destructuring** directly in the function signature to instantly unpack the variables!

```javascript
// ✅ Industry Standard: Destructuring
// We instantly unpack 'name' and 'role' from the invisible props object!
export function UserBadge({ name, role }) {
  return (
    <div className="badge">
      <h2>{name}</h2>
      <p>Role: {role}</p>
    </div>
  );
}
```

### Level 3 — Intermediate (Default Props & Passing Data Types)
You aren't limited to passing Strings. You can pass Numbers, Arrays, Objects, or even other Functions! To pass anything other than a literal string, you must use curly braces `{}`.

You can also assign Default values in case the Parent forgets to pass the prop!

```javascript
// 1. We assign a Default Value for 'theme'
export function SettingsCard({ userObject, isPremium, theme = "dark" }) {
  return (
    <div className={`card ${theme}`}>
      <h3>{userObject.username}</h3>
      {isPremium ? <p>💎 Premium Member</p> : <p>Free Tier</p>}
    </div>
  );
}
```

```javascript
// THE PARENT
export default function App() {
  const myUser = { username: "Charlie", age: 30 };

  return (
    <SettingsCard 
      // 2. Passing an Object requires {}
      userObject={myUser} 
      
      // 3. Passing a Boolean requires {}
      isPremium={true} 
      
      // We didn't pass 'theme', so it defaults to "dark"!
    />
  );
}
```

### Level 4 — Advanced (The children Prop)
What if you want to build a generic `<Card>` component, and you want the Parent to decide what goes *inside* the card? 

You can pass JSX into the opening and closing tags of a component. React will automatically grab all of that inner JSX and pass it to the Child via a magical, built-in prop called **`children`**.

```javascript
// THE CHILD (A generic layout wrapper)
export function Card({ title, children }) {
  return (
    <div className="border border-gray-200 shadow-md">
      <h2>{title}</h2>
      
      {/* MAGIC: The Parent's injected JSX is rendered right here! */}
      <div className="card-body">
        {children}
      </div>
      
    </div>
  );
}
```

```javascript
// THE PARENT
export default function Dashboard() {
  return (
    // We pass 'title' as a standard prop...
    <Card title="Sales Metrics">
      
      {/* ...but EVERYTHING inside these tags becomes the 'children' prop! */}
      <img src="chart.png" />
      <button>Download CSV</button>
      
    </Card>
  );
}
```

## REAL_WORLD
If you are using **TypeScript** (which you absolutely should be in Enterprise React), you must strictly define the "Shape" of the Props object using an Interface. 

This gives you massive superpowers: If the Parent forgets to pass a required prop, or passes a String when the Child expects a Number, the IDE will instantly throw a red underline and refuse to compile the code!

```typescript
// ENTERPRISE TYPESCRIPT REACT
interface ButtonProps {
  text: string;
  color?: 'primary' | 'secondary'; // The ? makes it optional!
  isDisabled: boolean;
}

export function Button({ text, color = 'primary', isDisabled }: ButtonProps) {
  return (
    <button disabled={isDisabled} className={`btn-${color}`}>
      {text}
    </button>
  );
}
```

## INTERVIEW
**Q: Can a Child modify its Props?**
A: **Absolutely NOT**. Props are immutable (read-only). If a Child component needs to change the data it was given, it cannot mutate the prop directly. Instead, the Parent must pass down a "Setter Function" as a prop, and the Child must call that function to politely ask the Parent to update the data.

**Q: What happens if you pass a prop but don't define a value? (e.g. `<Input disabled />`)**
A: In JSX, if you pass a prop name with no value, it defaults to the boolean `true`. So `<Input disabled />` is identical to `<Input disabled={true} />`.

**Q: What is Prop Drilling?**
A: It is a notorious React anti-pattern. If the Root component has data, and a Child 5 levels deep needs that data, you have to pass the prop down through 4 intermediate components that don't even care about the data, just to transport it to the bottom. This pollutes the codebase. You solve Prop Drilling by using the Context API or a State Management library like Redux or Zustand.

## FEYNMAN CHECK

Explain to a 10-year-old:
> Props are how parent components send read-only data to children. A child should NEVER modify its props — they are the parent's data. The `children` prop is just a special prop containing whatever JSX is placed between component tags. Prop drilling — passing props through 3+ layers just to reach a deep child — is the pain that led to Context API and state managers. The pattern to avoid it: composition (pass pre-built JSX as props, not raw data).
