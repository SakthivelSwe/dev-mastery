## WHY
You successfully build an API and deploy it to AWS. 
The next morning, you wake up to find your entire database has been deleted, replaced with a single record demanding a Bitcoin ransom. Your AWS bill is $50,000 because someone hijacked your server to mine cryptocurrency. Your company's user data is currently being sold on the dark web.
**API Security** is not a feature; it is the absolute baseline requirement of software engineering. Hackers run automated bots that scan every public IP address on the internet 24/7, actively looking for open ports, missing authentication, and unprotected database queries. If you don't secure your API, it will be compromised within hours of deployment.

## THEORY
### The Big Three Vulnerabilities (OWASP Top 10)
1. **Broken Access Control:** A user logs in perfectly (Authentication works). They then change the URL from `/api/users/12/receipts` to `/api/users/99/receipts`. The server blindly returns User 99's receipts because it forgot to check if User 12 is *authorized* to view User 99's data!
2. **Injection (SQL & NoSQL):** The server blindly trusts user input and concatenates it directly into database queries or shell commands. 
3. **Cross-Site Scripting (XSS):** A hacker leaves a comment on a blog: `<script>stealCookies()</script>`. The server saves it. When innocent users view the blog, their browser executes the hacker's JavaScript, stealing their session tokens!

## CODE

### 1. Fixing Broken Access Control (Authorization)
Authentication proves *who* they are. Authorization proves *what they are allowed to touch*. Every single endpoint MUST verify ownership!

```javascript
// Express.js Route
app.get('/api/users/:targetId/documents', authenticateJWT, async (req, res) => {
    // req.user comes from our JWT Bouncer middleware!
    const loggedInUserId = req.user.id; 
    const targetUserId = req.params.targetId;

    // AUTHORIZATION CHECK: Are they trying to look at someone else's data?
    // And do they lack the 'Admin' role?
    if (loggedInUserId !== targetUserId && req.user.role !== 'Admin') {
        // 403 Forbidden: I know who you are, but you are not allowed to do this!
        return res.status(403).json({ error: "Access Denied." });
    }

    // Safe to fetch the documents!
    const docs = await db.documents.find({ owner: targetUserId });
    res.json(docs);
});
```

### 2. Fixing XSS (Input Sanitization)
Never trust user input. If a user types HTML/JS into an input box, you must **Sanitize** (escape) it before saving it or rendering it!

```javascript
// Install a sanitization library: npm install dompurify jsdom
const DOMPurify = require('dompurify');
const { JSDOM } = require('jsdom');
const window = new JSDOM('').window;
const purify = DOMPurify(window);

app.post('/api/comments', (req, res) => {
    const rawComment = req.body.text; // "<script>alert('Hacked!')</script> Hello!"

    // 1. SANITIZE! This strips out all malicious <script> tags!
    const safeComment = purify.sanitize(rawComment); // " Hello!"

    // 2. Safe to save to the database!
    db.comments.save({ text: safeComment });
    res.status(201).send();
});
```
*Note: Modern Frontend Frameworks (like React) automatically sanitize all data injected via `{}` brackets, making them incredibly resistant to XSS!*

### 3. Rate Limiting (Preventing DDoS & Brute Force)
If a hacker wants to guess a password, they will write a Python script to hit your `/api/login` endpoint 10,000 times a second. You must throttle them!

```javascript
// npm install express-rate-limit
const rateLimit = require('express-rate-limit');

// Create a Limiter rule
const loginLimiter = rateLimit({
    windowMs: 15 * 60 * 1000, // 15 minutes
    max: 5, // Limit each IP to 5 requests per windowMs
    message: "Too many login attempts from this IP, please try again after 15 minutes."
});

// Apply the rule ONLY to the login route!
app.post('/api/login', loginLimiter, async (req, res) => {
    // Handle login logic...
});
```

## REAL_WORLD
**HTTPS (TLS/SSL Encryption)**
You build a perfect API with JWTs, Rate Limiting, and Input Sanitization.
But you deploy it using standard `HTTP` (Port 80).
When Alice logs in from a coffee shop WiFi, her password and her JWT travel through the air as plain, readable text. A hacker sitting in the corner running a "Packet Sniffer" (like Wireshark) intercepts the WiFi radio waves and reads her password in plain English.
**HTTPS (Hypertext Transfer Protocol Secure)** solves this. It encrypts the HTTP data using advanced TLS cryptography *before* it leaves the laptop. The data traveling through the air looks like absolute gibberish. Only the backend server has the mathematical Private Key required to decrypt the gibberish back into a usable password. You must NEVER deploy an API to production without an SSL Certificate enabling HTTPS!

## INTERVIEW
**Q: An interviewer asks: "We want to store user passwords in our database. Should we encrypt them using a two-way encryption algorithm like AES-256, so we can decrypt them if the user forgets their password?"**
A: **No! Never store passwords using two-way encryption!**
If you use two-way encryption, the server must store the decryption key. If a hacker breaches the database, they will also steal the decryption key, allowing them to decrypt every single password instantly!
Passwords MUST be stored using **One-Way Hashing** (specifically slow hashing algorithms like `Bcrypt` or `Argon2`).
A hash is a mathematical meat-grinder. It turns `"password123"` into a chaotic string of characters, but it is mathematically impossible to reverse the process! The server *never* knows the user's actual password. When the user logs in, the server hashes their input, and simply checks if the resulting hashes match! If they forget their password, they cannot recover it; you must send them a secure email link to generate a brand new one.

## FEYNMAN CHECK

### Explain API Security Like I'm 10 Years Old
> API security is the locks, alarms, and bouncer for your back-end. Common threats: unauthorised access (no auth), broken authz (logged-in user sees other users' data), injection (SQL/NoSQL/command), excessive data exposure, rate-limit absence (brute-force, DDoS), and insecure deserialisation. The non-obvious part: most breaches are NOT zero-days — they're missing controls (no rate-limit on login, missing authz check on /admin route, default credentials). This is why OWASP Top 10 has stayed remarkably stable for 10+ years.

### 5 Deep Questions
**Q1: What is BOLA (Broken Object-Level Authorisation) and why is it #1?**
> **A:** API returns object by ID but doesn't verify the caller owns it. `GET /orders/123` returns order 123 even if it belongs to another user — because authn (logged in) was checked but authz (owns this order) was not. Fix: every object endpoint must verify ownership.

**Q2: ONE model.**
> **A:** "Auth = identity. Authz per object = ownership. Validation = input shape. Sanitisation = output. Rate-limit = abuse. Logging = forensics. Each is a separate layer; skipping any creates exploitable gaps."

**Q3: Misconception with code.**
> **A:** ```js
> // ❌ Authenticated but not authorised
> app.get('/orders/:id', authn, async (req, res) => {
>   res.json(await db.orders.findById(req.params.id));  // Any logged-in user reads any order
> });
> // ✅ Verify ownership
> app.get('/orders/:id', authn, async (req, res) => {
>   const order = await db.orders.findById(req.params.id);
>   if (!order || order.userId !== req.user.sub) return res.sendStatus(404);
>   res.json(order);
> });
> ```

**Q4: How do you prevent SQL injection?**
> **A:** ALWAYS use parameterised queries (prepared statements). Never string-concatenate user input into SQL. ORMs do this by default. Even with ORM, raw escape hatches (`pool.query(`SELECT * FROM users WHERE name = '${name}'`)`) reintroduce the vulnerability.

**Q5: Senior one-liner.**
> **A:** "API security is layered defence — authn + per-object authz + input validation + output sanitisation + rate-limit + audit — and 95% of breaches come from a missing layer, not a zero-day, which is why OWASP Top 10 has remained stable for a decade."

#### Step 2 — Core (defence in depth)
```js
const express = require('express');
const helmet = require('helmet');
const rateLimit = require('express-rate-limit');
const jwt = require('jsonwebtoken');
const { z } = require('zod');

const app = express();
app.use(helmet());                          // 1. security headers
app.use(express.json({ limit: '10kb' }));   // 2. body size cap
app.use('/api/', rateLimit({ windowMs: 60_000, max: 60 })); // 3. rate-limit

function authn(req, res, next) { /* JWT verify, set req.user */ next(); }

const QuerySchema = z.object({ page: z.coerce.number().int().min(1).max(1000).default(1) });

app.get('/api/orders/:id', authn, async (req, res) => {
  // 4. input validation
  const id = z.coerce.number().int().positive().parse(req.params.id);
  // 5. per-object authz
  const order = await db.orders.findById(id);
  if (!order || order.userId !== req.user.sub) return res.sendStatus(404);
  // 6. output filtering — never leak internal fields
  const { internalNotes, ...safe } = order;
  res.json(safe);
});

app.use((err, req, res, next) => {
  console.error(err);   // 7. logging
  res.status(500).json({ error: 'internal' });  // 8. no stack trace leak
});

app.listen(3000);
```
