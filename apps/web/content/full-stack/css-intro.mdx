# CSS — Cascade, Specificity, Inheritance | Level: 1

## 1. Core Architecture & Specifications
The CSS Cascade is the engine that governs how conflicts are resolved when multiple declarations target the same element. It operates on a strict hierarchy of precedence, processing style sheets to determine the final, absolute value for every property of every element.

```
       [ Cascade Precedence Cascade ]
       ┌────────────────────────────┐
       │     User Agent Styles      │
       ├────────────────────────────┤
       │     User Normal Styles     │
       ├────────────────────────────┤
       │    Author Normal Styles    │
       ├────────────────────────────┤
       │    Author !important       │
       ├────────────────────────────┤
       │     User !important        │
       ├────────────────────────────┤
       │   User Agent !important    │
       └────────────────────────────┘
```

The cascade processes rules in this order of priority:
1. **Origin & Importance**: The combination of where the style sheet comes from (User Agent, User, Author) and its importance (`!important` status).
2. **Context**: CSS `@layer` rules (Cascade Layers) allow authors to control the precedence of style rules independent of specificity.
3. **Specificity**: A weight calculated from the selectors in a matching rule.
4. **Order of Appearance**: The last declared rule wins if the origins, layers, and specificities are identical.

---

## 2. Specificity Calculation Mechanics
Specificity is represented as a four-part vector: `(inline-styles, ID-selectors, class/attribute/pseudo-class-selectors, type/pseudo-element-selectors)`.

| Selector Type | Specificity Weight | Example |
| :--- | :--- | :--- |
| **Inline Style** | `(1, 0, 0, 0)` | `style="color: oklch(0.6 0.2 140)"` |
| **ID Selector** | `(0, 1, 0, 0)` | `#main-header` |
| **Class, Attribute, Pseudo-class** | `(0, 0, 1, 0)` | `.btn`, `[type="submit"]`, `:hover` |
| **Type, Pseudo-element** | `(0, 0, 0, 1)` | `div`, `h1`, `::before` |
| **Universal, Combinators** | `(0, 0, 0, 0)` | `*`, `>`, `+`, `~` |

### Specificity Calculation Example
```css
/* Specificity: (0, 1, 2, 1) */
#dashboard-nav .menu-item[data-active="true"] a {
  color: oklch(0.6 0.25 140);
}
```
* `1` ID (`#dashboard-nav`) = `0, 1, 0, 0`
* `1` Class (`.menu-item`) = `0, 0, 1, 0`
* `1` Attribute (`[data-active="true"]`) = `0, 0, 1, 0`
* `1` Type (`a`) = `0, 0, 0, 1`
* **Total Specificity**: `(0, 1, 2, 1)`

### The `!important` Directive
The `!important` flag does not increase specificity. Instead, it alters the origin/importance phase of the cascade. An `!important` declaration overrides any normal declaration, regardless of specificity. 
Using `!important` breaks the natural flow of the cascade. It should be avoided except in utility classes (e.g., Tailwind's `!hidden`) or when overriding third-party widget styles that use inline styles.

## 3. Style Lifecycle: Declared to Resolved Values
The rendering engine processes properties through several stages before they are rendered on screen:

```
[ Declared Value ] ──(Cascade)──> [ Specified Value ] ──(Defaults/Inheritance)──> [ Computed Value ] ──(Layout)──> [ Used Value ] ──(Rounding)──> [ Actual Value ]
```

1. **Declared Value**: Any value specified in a stylesheet matching the element.
2. **Specified Value**: The value chosen by the cascade, or inherited/defaulted if no cascade match exists.
3. **Computed Value**: The value resolved during style computation (e.g., resolving relative URIs, converting inherit/initial keywords, converting relative units like `em` or percentages of font size to absolute values). This is the value inherited by child elements.
4. **Used Value**: The value resolved after layout has occurred (e.g., converting percentage widths to absolute pixels based on the parent's actual width).
5. **Actual Value**: The final value used by the rendering engine after rounding is applied (e.g., rounding subpixel offsets to match physical screen pixels).

### `getComputedStyle()` Mechanics
In JavaScript, `window.getComputedStyle(element)` returns the **resolved value**. For most properties, this is the *computed* value, but for legacy reasons, for layout properties (like `width` and `height`), it returns the *used* value (absolute pixel dimensions).

## 4. Inheritance & Explicit Control Keywords

Inherited properties pass their **computed values** down from parent to child. 
* **Inherited by default**: Text properties (`color`, `font-family`, `font-size`, `line-height`, `letter-spacing`, `text-align`, `visibility`).
* **Not inherited by default**: Box-model and layout properties (`margin`, `padding`, `border`, `display`, `position`, `width`, `height`, `z-index`).

### Explicit Control Keywords
* `initial`: Resets the property to its initial value defined in the CSS specification (e.g., `display: inline` for all elements, including `div`).
* `inherit`: Forces inheritance of the parent's computed value, even for non-inherited properties.
* `unset`: Acts as `inherit` if the property is inherited by default, otherwise acts as `initial`.
* `revert`: Rolls back the cascade to the user-agent stylesheet or user stylesheet level, effectively discarding author-defined styles for that property.

## 6. DevTools Inspection Guide

To diagnose cascade issues:
1. Open Chrome DevTools (or Firefox Developer Tools) and inspect the element.
2. In the **Styles** pane, look for declarations that are struck through (e.g., ~~`color: red;`~~). This indicates they were overridden by a higher specificity selector, a later declaration, or a higher cascade origin.
3. Look for the small `i` icon or warning triangle indicating invalid properties.
4. Switch to the **Computed** tab. This pane shows the absolute resolved values currently active on the element (e.g., actual pixel values for `font-size` or `width`). Check the "Show all" checkbox to see inherited and default properties. Expand any property to see the exact selector chain and source file that won the cascade.

## 7. Performance & Compatibility

### Rendering Pipeline Mechanics
During the **Recalculate Style** phase, the browser parses the DOM and matches CSS selectors to DOM elements, calculating the computed values.
* **Complex Selectors Performance**: Deeply nested selectors (e.g., `body div.wrapper main article .content p span`) require the browser to evaluate selectors from right to left. This can cause high CPU utilization during DOM manipulation or animations. Keep selector chains flat (ideally 1–3 levels).

### Browser Compatibility
* **Cascade Layers (`@layer`)**: Supported in all evergreen browsers (Chrome 99+, Firefox 97+, Safari 15.4+). Global support is ~96%.
* **Logical Properties** (`padding-block`, `margin-inline`): Fully supported across all major engines.
* **caniuse reference**: [caniuse.com/#feat=css-cascade-layers](https://caniuse.com/#feat=css-cascade-layers)

## WHY
HTML creates the structure and content of a web page. But without CSS, a webpage looks like a plain text document from 1993 — black text on a white background with blue links. 

**CSS (Cascading Style Sheets)** is the language that controls the visual presentation: colors, fonts, spacing, layout, and animations. It is what transforms a bare HTML skeleton into a beautiful, modern, branded user interface.

Understanding CSS deeply is one of the most underrated skills in web development. It separates developers who "make things work" from developers who create truly stunning user experiences.

## THEORY
### How CSS Works
CSS rules follow a simple structure:
```
selector {
    property: value;
}
```
The **selector** targets specific HTML elements. The **property-value pair** defines what to change about them.

### The Cascade and Specificity
"Cascading" in CSS means that multiple rules can target the same element. When rules conflict, the browser uses a priority system called **Specificity** to decide which rule wins:
1. Inline styles (`style="color: red"`) — Highest priority
2. ID selectors (`#my-id`) 
3. Class selectors (`.my-class`), attribute selectors (`[type="text"]`), pseudo-classes (`:hover`)
4. Element selectors (`p`, `div`, `h1`) — Lowest priority

## CODE

### Level 1 — Beginner (Selectors and Basic Properties)
```css
/* ============================================
   SELECTORS
   ============================================ */

/* Element selector — targets ALL <p> tags */
p {
    color: #333333;
    font-size: 16px;
}

/* Class selector — targets ALL elements with class="btn" */
.btn {
    background-color: #6366f1; /* Indigo */
    color: white;
    padding: 12px 24px;
    border: none;
    cursor: pointer;
}

/* ID selector — targets ONE element with id="hero" */
#hero {
    background-image: url('/images/hero-bg.jpg');
    height: 100vh;
}

/* Descendant selector — targets <a> tags INSIDE a <nav> */
nav a {
    text-decoration: none;
    color: inherit;
}

/* Pseudo-class — styles applied on user interaction */
.btn:hover {
    background-color: #4f46e5;  /* Darken on hover */
    transform: translateY(-2px); /* Subtle lift effect */
}
```

### Level 2 — Building Blocks (The Box Model)
```css
/* EVERY HTML element is a box with four layers:
   Content → Padding → Border → Margin

   CRITICAL: By default, width/height only sizes the CONTENT box.
   If you set width: 200px and add padding: 20px, the element becomes 240px wide!
   
   Fix this with box-sizing: border-box — makes width include padding and border! */

*, *::before, *::after {
    box-sizing: border-box; /* Apply to EVERYTHING — industry standard! */
}

.card {
    /* Content sizing */
    width: 320px;
    height: auto;

    /* Padding: Space INSIDE the element (between content and border) */
    padding: 24px;             /* All four sides */
    padding: 16px 24px;        /* Top/Bottom | Left/Right */
    padding: 10px 20px 15px 20px; /* Top | Right | Bottom | Left */

    /* Border */
    border: 1px solid #e5e7eb;
    border-radius: 12px;       /* Rounded corners! */

    /* Margin: Space OUTSIDE the element (between this and other elements) */
    margin: 0 auto;            /* Center horizontally! */

    /* Box Shadow */
    box-shadow: 0 4px 16px rgba(0, 0, 0, 0.1);
}
```

### Level 4 — Advanced (Pseudo-elements and Modern Selectors)
```css
/* ::before and ::after — inject virtual elements via CSS */
.quote::before { content: '"'; font-size: 4em; color: var(--color-primary); }
.quote::after  { content: '"'; font-size: 4em; color: var(--color-primary); }

/* :has() — The "parent selector" (CSS 2023!) */
/* Style a card differently if it CONTAINS a .featured-badge child */
.card:has(.featured-badge) {
    border-color: gold;
    box-shadow: 0 0 0 3px gold;
}

/* :is() — Reduces repetition in complex selectors */
:is(h1, h2, h3, h4) {
    font-weight: 700;
    line-height: 1.2;
}

/* :not() — Style elements that DON'T match */
.list-item:not(:last-child) {
    border-bottom: 1px solid #e5e7eb; /* Divider between items, but not after the last! */
}

/* CSS Nesting (modern browsers, 2023+) */
.card {
    background: white;
    
    & .card-title { /* Nested! Equivalent to .card .card-title */
        font-size: 1.25rem;
    }
    
    &:hover { /* Nested pseudo-class */
        box-shadow: var(--shadow-md);
    }
}
```

## REAL_WORLD
Modern CSS (2023+) is extraordinarily powerful. Many capabilities that previously required JavaScript (sticky headers, scroll-linked animations, container queries) can now be achieved purely in CSS.

**The Modern CSS Workflow:**
1. Define Design Tokens as CSS Custom Properties in `:root`.
2. Use `box-sizing: border-box` globally.
3. Use `Flexbox` for one-dimensional layouts (rows or columns).
4. Use `CSS Grid` for two-dimensional layouts (rows AND columns simultaneously).
5. Use `clamp()` for fluid typography that automatically scales between screen sizes.

## INTERVIEW
**Q: What is the difference between `display: none` and `visibility: hidden`?**
A: `display: none` completely removes the element from the document flow — it takes up no space, and other elements reflow to fill the gap. `visibility: hidden` makes the element invisible, but it still occupies its original space in the layout (it's just transparent). Critically, `display: none` elements also cannot receive focus for accessibility tools.

**Q: What is specificity and how do you calculate it?**
A: Specificity is a 4-tuple score `(inline, id, class, element)`. Each rule is scored: inline styles = (1,0,0,0), ID selectors add (0,1,0,0), class/attribute/pseudo-class add (0,0,1,0), element/pseudo-element add (0,0,0,1). The rule with the highest score wins. The `!important` declaration bypasses specificity entirely — avoid it, as it creates maintenance nightmares.

**Q: What does `inherit`, `initial`, and `unset` do as CSS values?**
A: `inherit` forces the property to take the same value as the parent element. `initial` resets the property to its CSS specification-defined initial value. `unset` acts as `inherit` for naturally inherited properties and as `initial` for non-inherited ones — useful for truly resetting an element's style.

## FEYNMAN CHECK

### Explain CSS Like I'm 10 Years Old
> CSS is the "painter" that walks through the DOM tree the browser built from HTML and says: "this heading is 2rem and red, this paragraph has 1em of margin, this button has 8px of padding." But CSS doesn't paint immediately — it first computes a SECOND tree called the CSSOM, then the browser MERGES the DOM + CSSOM into the Render Tree (only visible nodes), runs LAYOUT (calculates every box's x/y/width/height), then PAINT (rasterises to pixels), then COMPOSITE (assembles layers onto the screen). This is why one wrong CSS rule can trigger "layout thrashing" — re-running steps 3-5 sixty times per second.

### 5 Deep Conceptual Questions

**Q1: What problem does CSS fundamentally solve, and why couldn't you solve it another way?**
> **A:** CSS solves "separation of concerns" between document structure (HTML) and presentation. Before CSS (pre-1996), styling lived inside HTML as `<font color="red" size="5">` and `<table>`-based layouts — meaning a redesign required editing every page. CSS introduced a CASCADE: one rule can target thousands of elements via selectors, and you can override globally → site-wide → component-level → element-level with predictable precedence. JavaScript could mimic this with inline styles, but you'd lose pseudo-classes, media queries, animations, and the entire render-pipeline optimisation the browser does for CSS.

**Q2: What is the ONE mental model that makes CSS click into place?**
> **A:** CSS is a function: `(element, context) → computed style`. For every element in the DOM, the browser collects every matching rule, sorts them by specificity + source order + `!important`, computes inherited values from the parent, then resolves units (`em` → `px`, `%` → pixels) against the containing block. Once you internalise "every property has a computed value AND a used value, and they can differ," debugging becomes mechanical instead of magical.

**Q3: What is the most dangerous misconception about CSS? Show it with code.**
> **A:** That `!important` "wins" against everything — actually two `!important` rules still battle by specificity:
> ```html
> <!-- ❌ "I'll just use !important to fix this..." -->
> <style>
>   .button       { color: red !important; }
>   .modal .button { color: blue !important; }   /* WINS — higher specificity */
> </style>
> <div class="modal"><button class="button">Text</button></div>
> <!-- Result: blue, not red. !important doesn't bypass specificity. -->
>
> <!-- ✅ Fix the specificity properly without !important -->
> <style>
>   .button       { color: red; }
>   .modal .button { color: red; }
> </style>
> ```

**Q4: How does CSS interact with the browser's rendering pipeline at the runtime level?**
> **A:** The browser parses every `<link rel="stylesheet">` into the CSSOM, which BLOCKS rendering — the browser refuses to paint until all stylesheets are downloaded and parsed (otherwise you'd see a "flash of unstyled content"). It then walks the DOM + CSSOM to build the Render Tree, runs the layout algorithm (which is O(n) in element count but can cascade on a single `width` change), paints layers into bitmaps, and composites them on the GPU. Modifying a layout property (`width`, `top`, `margin`) re-runs layout+paint+composite; modifying `transform` or `opacity` only re-runs composite — which is why `transform: translate` is 60fps and `top: 10px` is jank.

**Q5: Write a one-sentence definition of CSS that a senior engineer at a FAANG company would find technically precise.**
> **A:** "CSS is a declarative, cascading, inherited style-resolution language whose rules the browser parses into the CSSOM, merges with the DOM to produce the Render Tree, and then evaluates through a four-stage pipeline (style → layout → paint → composite) — which is why selector specificity, the containing-block algorithm, and the will-change/compositor-promotion rules have direct measurable impact on Largest Contentful Paint and Cumulative Layout Shift."

#### Step 4 — Error Handling: Honour `prefers-reduced-motion`
```javascript
// app.js
const cheap     = document.getElementById('cheap');
const expensive = document.getElementById('expensive');
const btn       = document.getElementById('go');

// Respect the user's OS-level "reduce motion" setting (a11y requirement)
const reduceMotion = matchMedia('(prefers-reduced-motion: reduce)').matches;

btn.addEventListener('click', () => {
  if (reduceMotion) {
    alert('Reduced-motion mode is on — skipping animation.');
    return;
  }
  animate(0);
});

function animate(frame) {
  if (frame >= 200) return;

  // CHEAP: transform only triggers COMPOSITE
  cheap.style.transform = `translateX(${frame * 2}px)`;

  // EXPENSIVE: left triggers LAYOUT + PAINT + COMPOSITE
  expensive.style.left = (frame * 2) + 'px';

  requestAnimationFrame(() => animate(frame + 1));
}
```
