# Advanced Angular Components: Architecture, Lifecycle, and Internals

Angular components are the fundamental building blocks of modern Angular applications. With the release of Angular 17+, the framework has shifted decisively toward **standalone components** as the default architecture. This transition simplifies the mental model by eliminating `NgModule` declarations, but it also elevates the importance of understanding the underlying compiler mechanics, lifecycle phases, and view query engines.

---

## 1. Component Architecture & Standalone Compilation

In Angular 17+, a component is defined using the `@Component` decorator with `standalone: true`. This tells the Angular compiler (Ivy) to treat the component as a self-contained compilation unit.

### Internal Mechanics: `LView` and `TView`
Under the hood, the Ivy rendering engine represents components using two primary internal data structures:
*   **`TView` (Template View):** A static, shared structure created once per component type. It contains template instructions, element paths, and metadata that do not change between instances.
*   **`LView` (Logical View):** A dynamic, instance-specific array containing the actual state, bound properties, DOM element references, and child component instances.

When Angular compiles a standalone component, it translates the template into a set of sequential Ivy instructions (e.g., `ɵɵelementStart`, `ɵɵtext`, `ɵɵproperty`).

```typescript
import { 
  Component, 
  Input, 
  Output, 
  EventEmitter, 
  ViewEncapsulation, 
  ChangeDetectionStrategy,
  input,
  output,
  model
} from '@angular/core';
import { CommonModule } from '@angular/common';

@Component({
  selector: 'app-advanced-card',
  standalone: true,
  imports: [CommonModule],
  template: `
    <div class="card-container" [class.active]="isActive()">
      <header class="card-header">
        <ng-content select="[card-title]"></ng-content>
      </header>
      <main class="card-body">
        <p>{{ description() }}</p>
      </main>
      <footer class="card-actions">
        <button (click)="emitAction()">Execute</button>
      </footer>
    </div>
  `,
  styles: [`
    :host {
      display: block;
      margin: 1rem 0;
    }
    .card-container {
      border: 1px solid #ccc;
      padding: 1rem;
      transition: border-color 0.3s ease;
    }
    .card-container.active {
      border-color: #007bff;
    }
  `],
  encapsulation: ViewEncapsulation.Emulated,
  changeDetection: ChangeDetectionStrategy.OnPush
})
export class AdvancedCardComponent {
  // Signal-based Inputs (Angular 17.1+)
  description = input.required<string>();
  
  // Model input (writable signal-based two-way binding)
  isActive = model<boolean>(false);

  // Signal-based Output
  actionTriggered = output<void>();

  emitAction(): void {
    this.isActive.set(!this.isActive());
    this.actionTriggered.emit();
  }
}
```

## 2. Advanced Lifecycle Hooks & Execution Order

Angular components orchestrate their state transitions through lifecycle hooks. Understanding their exact execution order and relationship with the change detection loop is crucial for avoiding the notorious `ExpressionChangedAfterItHasBeenCheckedError`.

### Execution Order Lifecycle Flow
1.  **`constructor`**: Instantiate class. DI is initialized, but inputs are not yet available.
2.  **`ngOnChanges`**: Triggered when bound inputs change. Receives a `SimpleChanges` object. Not called if inputs are signal-based (signals track dependencies dynamically).
3.  **`ngOnInit`**: Called once after the first `ngOnChanges`. Component state is initialized.
4.  **`ngDoCheck`**: Runs during every change detection cycle. Used for custom change detection algorithms.
5.  **`ngAfterContentInit`**: Executed after project content (`<ng-content>`) is projected into the component.
6.  **`ngAfterContentChecked`**: Triggered after projected content is checked.
7.  **`ngAfterViewInit`**: Executed after the component's template (and children) has been fully initialized.
8.  **`ngAfterViewChecked`**: Triggered after the component's template has been checked.
9.  **`ngOnDestroy`**: Clean up resources, unsubscribe from observables, disconnect event listeners.

```typescript
import { 
  Component, 
  OnInit, 
  OnChanges, 
  DoCheck, 
  AfterViewInit, 
  OnDestroy, 
  SimpleChanges, 
  Input,
  inject,
  DestroyRef
} from '@angular/core';

@Component({
  selector: 'app-lifecycle-tracker',
  standalone: true,
  template: `<div>Lifecycle Tracker: {{ value }}</div>`
})
export class LifecycleTrackerComponent implements OnInit, OnChanges, DoCheck, AfterViewInit {
  @Input() value = '';

  private destroyRef = inject(DestroyRef);

  constructor() {
    console.log('1. Constructor: View is not ready, inputs are undefined.');
    
    // Modern alternative to ngOnDestroy using DestroyRef
    this.destroyRef.onDestroy(() => {
      console.log('Cleanup executed via DestroyRef.');
    });
  }

  ngOnChanges(changes: SimpleChanges): void {
    console.log('2. ngOnChanges: Input values modified.', changes);
  }

  ngOnInit(): void {
    console.log('3. ngOnInit: Component initialization complete.');
  }

  ngDoCheck(): void {
    console.log('4. ngDoCheck: Manual change detection hook.');
  }

  ngAfterViewInit(): void {
    console.log('5. ngAfterViewInit: DOM and child views are fully loaded.');
  }
}
```

## 3. Component Communication Patterns

Modern Angular applications leverage a mix of Signals and classic decorators to build robust communication bridges.

### Signals-based Communication vs. Decorators
*   **`input()` / `output()`**: Integrated directly into Angular's reactive engine. They do not require change detection ticks to notify child templates of changes.
*   **`model()`**: Simplifies two-way data binding by combining an input and an output into a single writable signal.

### Element, View, and Content Queries
Queries allow components to reference child components, directives, or DOM elements.

```typescript
import { 
  Component, 
  ElementRef, 
  viewChild, 
  viewChildren, 
  contentChild,
  AfterViewInit,
  Directive
} from '@angular/core';

@Directive({
  selector: '[appCustomMarker]',
  standalone: true
})
export class CustomMarkerDirective {}

@Component({
  selector: 'app-query-demo',
  standalone: true,
  imports: [CustomMarkerDirective],
  template: `
    <div #container>
      <ng-content></ng-content>
      <div appCustomMarker>Marked Element</div>
    </div>
  `
})
export class QueryDemoComponent implements AfterViewInit {
  // Signal-based view queries (Angular 17.3+)
  containerEl = viewChild<ElementRef<HTMLDivElement>>('container');
  markerDirective = viewChild(CustomMarkerDirective);
  
  // Content query to read projected content
  projectedHeader = contentChild<ElementRef<HTMLHeadingElement>>('headerProj');

  ngAfterViewInit(): void {
    const container = this.containerEl();
    if (container) {
      container.nativeElement.style.backgroundColor = '#f4f4f4';
    }
    
    console.log('Marker Directive:', this.markerDirective());
    console.log('Projected Header:', this.projectedHeader());
  }
}
```

## 4. View Encapsulation Mechanics

Angular isolates styles using three strategies defined in `ViewEncapsulation`:

1.  **`Emulated` (Default)**: Angular prepends unique attributes (like `_nghost-c12` and `_ngcontent-c12`) to component hosts and elements. CSS selectors are recompiled to target these attributes, preventing leakage.
2.  **`ShadowDom`**: Angular uses the browser's native Shadow DOM API. The component is rendered inside a shadow root, isolating styles completely.
3.  **`None`**: Styles are injected globally. Any selector defined in the component's stylesheet will affect the entire document.

```typescript
// Comparison of compiled CSS output for Emulated Encapsulation:
// Original: 
// .btn { color: red; }
// Compiled: 
// .btn[_ngcontent-c10] { color: red; }
```

## 6. Real-World Production Scenario: Dynamic Component Portal

In enterprise dashboards, we often need to load components dynamically based on dynamic configurations. The following implementation uses `ViewContainerRef` to inject components programmatically.

```typescript
import { 
  Component, 
  ViewChild, 
  ViewContainerRef, 
  Type, 
  OnInit, 
  DestroyRef,
  inject 
} from '@angular/core';

@Component({
  selector: 'app-dynamic-widget-a',
  standalone: true,
  template: `<div class="widget">Widget A State</div>`,
  styles: ['.widget { padding: 10px; background: #e0f7fa; }']
})
export class WidgetAComponent {}

@Component({
  selector: 'app-dynamic-widget-b',
  standalone: true,
  template: `<div class="widget">Widget B State</div>`,
  styles: ['.widget { padding: 10px; background: #fff3e0; }']
})
export class WidgetBComponent {}

@Component({
  selector: 'app-widget-portal',
  standalone: true,
  template: `
    <div class="portal-container">
      <h3>Dynamic Dashboard Portal</h3>
      <button (click)="loadWidget(widgetA)">Load Widget A</button>
      <button (click)="loadWidget(widgetB)">Load Widget B</button>
      <button (click)="clearPortal()">Clear</button>
      
      <div class="anchor-point">
        <ng-container #portalAnchor></ng-container>
      </div>
    </div>
  `
})
export class WidgetPortalComponent implements OnInit {
  @ViewChild('portalAnchor', { read: ViewContainerRef, static: true }) 
  portalAnchor!: ViewContainerRef;

  widgetA = WidgetAComponent;
  widgetB = WidgetBComponent;

  private destroyRef = inject(DestroyRef);

  ngOnInit(): void {
    this.destroyRef.onDestroy(() => this.clearPortal());
  }

  loadWidget(widgetType: Type<any>): void {
    this.portalAnchor.clear();
    // Instantiates component and inserts its view into the container
    const componentRef = this.portalAnchor.createComponent(widgetType);
    
    // We can also pass inputs programmatically
    // componentRef.setInput('inputName', value);
  }

  clearPortal(): void {
    this.portalAnchor.clear();
  }
}
```

This portal architecture provides a flexible way to instantiate components dynamically, bypassing the template compiler for highly dynamic dashboard structures.
