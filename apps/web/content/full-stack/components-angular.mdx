# Architectural Deep Dive: Angular Components

## 1. Architectural Foundations & Core Mechanics

In Angular 17+, components are the fundamental building blocks of the user interface. Under the hood, the Ivy compilation engine transforms a component class decorated with `@Component` into a highly optimized set of instructions. To truly understand Angular components, we must look past the decorator and examine how the compiler transforms our code into the runtime architecture.

### The Ivy Engine: `TView` and `LView`

When Angular compiles a component, it generates two critical internal data structures: the **Logical View (`LView`)** and the **Template View (`TView`)**.

```
+-------------------------------------------------------------+
|                           TView                             |
|  (Shared Template Definition - Static Metadata & Blueprints)|
|  - Directive defs, Binding metadata, Query definitions      |
+-------------------------------------------------------------+
                               |
                               | Instantiates
                               v
+-------------------------------------------------------------+
|                           LView                             |
|  (Logical View Instance - Dynamic State & DOM Elements)     |
|  - [0]: Controller Instance (Component Context)             |
|  - [1..N]: Native DOM Elements, Directive Instances,        |
|            Child LViews, Projected Nodes                    |
+-------------------------------------------------------------+
```

*   **`TView` (Template View)**: This is a static, shared structure created once for each component type. It contains the blueprint of the component's template, including directive definitions, binding metadata, and query definitions. It is read-only at runtime and is shared across all instances of that component.
*   **`LView` (Logical View)**: This represents a specific instance of a component. It is a flat array containing the actual state of the view: native DOM elements, directive instances, child `LView` references, and binding values. 

The first element (`LView[0]`) points to the component controller instance. Every property binding in your template corresponds to an index in this array. During change detection, Angular compares the current expression value with the value stored at the corresponding index in `LView` to determine if the DOM needs to be updated.

### Component Lifecycle Execution Sequence

The lifecycle of an Angular component is governed by the change detection loop. When an `LView` is processed, Angular executes lifecycle hooks in a strict sequence:

```
[Component Creation]
         │
         ▼
 1. Constructor (Dependency Injection resolved)
         │
         ▼
 2. ngOnChanges() (If inputs exist; parses SimpleChanges)
         │
         ▼
 3. ngOnInit() (Component state initialized; inputs available)
         │
         ▼
 4. ngDoCheck() (Custom change detection run)
         │
         ├──────────────────────────────┐
         ▼                              ▼
 5. ngAfterContentInit()        6. ngAfterViewInit()
         │                              │
         ▼                              ▼
 7. ngAfterContentChecked()     8. ngAfterViewChecked()
         │                              │
         └──────────────┬───────────────┘
                        │
                        ▼
               [Change Detection Loop]
                        │ (On Component Destroy)
                        ▼
 9. ngOnDestroy() (Teardown, unsubscribe, clean up)
```

1.  **`constructor`**: The class is instantiated. Dependencies are resolved via the Injector. Inputs are *not* yet available.
2.  **`ngOnChanges`**: Triggered whenever data-bound inputs change. It receives a `SimpleChanges` object containing current and previous values.
3.  **`ngOnInit`**: Executed once after the first `ngOnChanges`. Ideal for initializing component state.
4.  **`ngDoCheck`**: Executed during every change detection cycle, immediately after `ngOnChanges`. Used for custom change detection algorithms.
5.  **`ngAfterContentInit`**: Executed after projected content (`<ng-content>`) has been fully initialized into the component's view.
6.  **`ngAfterContentChecked`**: Executed after every check of projected content.
7.  **`ngAfterViewInit`**: Executed after the component's own view (and child views) has been fully initialized.
8.  **`ngAfterViewChecked`**: Executed after every check of the component's view and child views.
9.  **`ngOnDestroy`**: Executed immediately before the component is destroyed. Critical for preventing memory leaks by unsubscribing from Observables and clearing timers.

## 2. Deep-Dive API & Declarative Synthesis

Angular 17+ introduces modern, declarative APIs that replace legacy decorators and configurations.

### Standalone Component Configuration

Standalone components eliminate the need for `NgModule`. They directly manage their template dependencies via the `imports` array:

```typescript
import { Component, ChangeDetectionStrategy } from '@angular/core';
import { CommonModule } from '@angular/common';
import { CustomDirective } from './custom.directive';

@Component({
  selector: 'app-root-container',
  standalone: true,
  imports: [CommonModule, CustomDirective],
  templateUrl: './root-container.component.html',
  styleUrls: ['./root-container.component.scss'],
  changeDetection: ChangeDetectionStrategy.OnPush
})
export class RootContainerComponent {}
```

### Modern Input Transformations

Input properties can now be declared with transform functions to parse or sanitize values declaratively:

```typescript
import { Component, Input, booleanAttribute, numberAttribute } from '@angular/core';

@Component({
  selector: 'app-data-card',
  standalone: true,
  template: `<div>ID: {{ id }} | Active: {{ active }}</div>`
})
export class DataCardComponent {
  // Coerces string 'true' or empty attribute to boolean true
  @Input({ transform: booleanAttribute }) active: boolean = false;

  // Coerces string to number
  @Input({ transform: numberAttribute }) id: number = 0;
}
```

### Output and Event Emitters

Outputs allow components to communicate events upwards to parent contexts:

```typescript
import { Component, Output, EventEmitter } from '@angular/core';

@Component({
  selector: 'app-action-button',
  standalone: true,
  template: `<button (click)="emitAction()">Click Me</button>`
})
export class ActionButtonComponent {
  @Output() actionOccurred = new EventEmitter<string>();

  emitAction(): void {
    this.actionOccurred.emit('Button clicked successfully!');
  }
}
```

## 3. Advanced Implementation & Production Patterns

Let's build a highly optimized, type-safe, standalone Component that manages a dynamic grid layout. It uses structural directives, content projection, queries, and `OnPush` change detection.

### `dynamic-grid.component.ts`

```typescript
import { 
  Component, 
  Input, 
  Output, 
  EventEmitter, 
  ContentChildren, 
  QueryList, 
  AfterContentInit, 
  ChangeDetectionStrategy,
  Directive,
  TemplateRef
} from '@angular/core';
import { CommonModule } from '@angular/common';

@Directive({
  selector: '[gridItem]',
  standalone: true
})
export class GridItemDirective {
  constructor(public templateRef: TemplateRef<any>) {}
}

export interface GridConfig {
  columns: number;
  gapPx: number;
}

@Component({
  selector: 'app-dynamic-grid',
  standalone: true,
  imports: [CommonModule],
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `
    <div 
      class="grid-container" 
      [style.grid-template-columns]="'repeat(' + config.columns + ', 1fr)'"
      [style.gap.px]="config.gapPx"
    >
      <div 
        *ngFor="let item of items; let idx = index" 
        class="grid-item"
        (click)="handleItemClick(item, idx)"
      >
        <!-- Projecting template dynamic content -->
        <ng-container *ngTemplateOutlet="itemTemplate; context: { $implicit: item, index: idx }"></ng-container>
      </div>
    </div>
  `,
  styles: [`
    .grid-container {
      display: grid;
      width: 100%;
    }
    .grid-item {
      cursor: pointer;
      border: 1px solid #ccc;
      padding: 16px;
      border-radius: 4px;
      transition: background-color 0.2s ease;
    }
    .grid-item:hover {
      background-color: #f5f5f5;
    }
  `]
})
export class DynamicGridComponent<T> implements AfterContentInit {
  @Input({ required: true }) items: T[] = [];
  @Input() config: GridConfig = { columns: 3, gapPx: 16 };
  @Output() itemSelected = new EventEmitter<{ item: T; index: number }>();

  @ContentChildren(GridItemDirective) 
  queryItems!: QueryList<GridItemDirective>;

  protected itemTemplate!: TemplateRef<any>;

  ngAfterContentInit(): void {
    const firstItem = this.queryItems.first;
    if (firstItem) {
      this.itemTemplate = firstItem.templateRef;
    } else {
      throw new Error('DynamicGridComponent requires a projected [gridItem] template directive.');
    }
  }

  protected handleItemClick(item: T, index: number): void {
    this.itemSelected.emit({ item, index });
  }
}
```

## 4. Diagnostics, Performance, & Memory Management

### Avoiding Memory Leaks in Component Subscriptions

A major source of memory leaks in Angular is failing to clean up active subscriptions when a component is destroyed. If a subscription is held in a component instance, the Garbage Collector cannot reclaim the component's `LView` and class instance, causing a leak.

#### The `takeUntilDestroyed` Pattern (Angular 16+)

```typescript
import { Component, OnInit, DestroyRef, inject } from '@angular/core';
import { HttpClient } from '@angular/common/http';
import { takeUntilDestroyed } from '@angular/core/rxjs-interop';
import { interval } from 'rxjs';

@Component({
  selector: 'app-leak-free',
  standalone: true,
  template: `<div>Active Component</div>`
})
export class LeakFreeComponent implements OnInit {
  private destroyRef = inject(DestroyRef);
  private http = inject(HttpClient);

  ngOnInit(): void {
    interval(1000)
      .pipe(takeUntilDestroyed(this.destroyRef))
      .subscribe(val => console.log('Interval Tick:', val));
  }
}
```

### Performance Profiling with `ChangeDetectionStrategy.OnPush`

By default, Angular uses the `Default` change detection strategy, which checks the entire component tree from top to bottom on every asynchronous event (click, timer, HTTP response). 

By switching to `OnPush`, you instruct Angular to skip change detection for this component and its children *unless*:
1.  One of its `@Input` references changes.
2.  An event originates from the component or its children (e.g., a click event handler).
3.  An Observable bound in the template via the `async` pipe emits a new value.
4.  Change detection is explicitly invoked via `ChangeDetectorRef.markForCheck()`.

#### Profiling Change Detection

Using the Angular DevTools chrome extension, you can record change detection cycles. If you notice components re-rendering when their inputs haven't changed, they are likely set to `Default` strategy or are receiving mutated object references instead of immutable updates.

## 6. Anti-Patterns, Refactoring, & Decoupling

### Anti-Pattern: Direct DOM Manipulation via `ElementRef.nativeElement`

```typescript
// BAD: Violates platform abstraction, breaks SSR (Server-Side Rendering)
constructor(private el: ElementRef) {
  this.el.nativeElement.style.backgroundColor = 'red';
}
```

#### Refactoring Solution: Use `Renderer2` or Declarative Directives

```typescript
// GOOD: Safe platform abstraction, compatible with SSR and Web Workers
import { Component, ElementRef, Renderer2, inject, OnInit } from '@angular/core';

@Component({
  selector: 'app-safe-dom',
  standalone: true,
  template: `<div #target>Safe Styled Box</div>`
})
export class SafeDomComponent implements OnInit {
  private el = inject(ElementRef);
  private renderer = inject(Renderer2);

  ngOnInit(): void {
    const targetEl = this.el.nativeElement.querySelector('[target]');
    if (targetEl) {
      this.renderer.setStyle(targetEl, 'background-color', 'red');
    }
  }
}
```

### Anti-Pattern: Over-reliance on Parent-Child Coupling

Using `@Parent` or traversing `fixture.debugElement.parent` in tests tightly couples components together, making reuse impossible.

#### Refactoring Solution: Unify communication via Inputs, Outputs, or scoped State Services.

## WHY
Building a massive web application as one giant HTML file and one massive JavaScript file is unmaintainable. The code becomes spaghetti. 

Angular solves this by using a **Component-Based Architecture**. A Component is a self-contained, highly cohesive block of UI (like a Navbar, a User Card, or a Data Table). By breaking the UI down into small, reusable components, developers can build complex screens by snapping together these building blocks like Legos. If you fix a bug in the `UserAvatarComponent`, it instantly fixes the bug everywhere the avatar is used across the entire application.

## THEORY
### The Triad
An Angular Component is composed of three interconnected parts:
1. **The Class (TypeScript)**: Holds the data state and the business logic (functions).
2. **The Template (HTML)**: Defines what the user actually sees. It binds to the Class's data.
3. **The Styles (CSS/SCSS)**: Defines how the component looks. Crucially, these styles are *encapsulated*.

### View Encapsulation
By default, Angular uses **Emulated View Encapsulation**. If you write `h1 { color: red; }` inside a Component's CSS file, Angular secretly attaches a unique identifier to the HTML element (like `<h1 _ngcontent-c42>`) and rewrites your CSS to target only that identifier (`h1[_ngcontent-c42] { color: red; }`). This guarantees your CSS will *never* bleed out and ruin the styles of other components on the page.

### Level 3 — Intermediate (Passing Data: @Input & @Output)
Components must talk to each other. A parent component passes data DOWN to a child using `@Input()`. A child passes data UP to a parent using `@Output()`.

**The Child Component (`user-card.component.ts`)**:
```typescript
import { Component, Input, Output, EventEmitter } from '@angular/core';

@Component({
  selector: 'app-user-card',
  template: `
    <div class="card">
      <!-- 2. Display the data passed from the parent -->
      <h3>{{ name }}</h3>
      
      <!-- 4. User clicks button, triggers notifyParent() -->
      <button (click)="notifyParent()">Delete User</button>
    </div>
  `
})
export class UserCardComponent {
  // 1. @Input allows the parent to inject data into this property!
  @Input() name: string = '';

  // 3. @Output creates a custom event stream to broadcast to the parent!
  @Output() userDeleted = new EventEmitter<string>();

  notifyParent() {
    // 5. Emit the event upwards, passing the specific user's name!
    this.userDeleted.emit(this.name);
  }
}
```

**The Parent Component (`user-list.component.html`)**:
```html
<!-- 
  [name] is property binding (injects 'Alice' into the @Input)
  (userDeleted) is event binding (listens to the @Output) 
-->
<app-user-card 
    [name]="'Alice'" 
    (userDeleted)="handleDeletion($event)">
</app-user-card>
```

## REAL_WORLD
In modern enterprise Angular, developers are aggressively switching to **Smart vs. Dumb (Presentational) Components**. 

- **Smart Components**: Connect to Services, make HTTP calls, and manage application state. They rarely have complex CSS. They grab data and pass it downwards.
- **Dumb Components**: Have zero external dependencies. They only accept data via `@Input()` and emit events via `@Output()`. They handle all the complex UI rendering.

By strictly separating concerns, you can reuse "Dumb" components across entirely different projects because they don't have hardcoded HTTP calls baked into their logic.

## INTERVIEW
**Q: Can you define the HTML template directly in the `.ts` file?**
A: Yes. You can replace `templateUrl` with `template` and write inline HTML. This is highly recommended for tiny, single-purpose components (often called Single File Components) to avoid cluttering the filesystem with 3-line HTML files.

**Q: What is `ViewEncapsulation.None`?**
A: If you add `encapsulation: ViewEncapsulation.None` to your `@Component` metadata, Angular disables the CSS scoping. Any CSS you write in that component's stylesheet will instantly leak out and become global CSS, affecting the entire application. This is generally considered a terrible practice, used only when creating global theme wrappers.

**Q: If a parent component's property changes, does the child's `@Input()` automatically update?**
A: Yes. Angular's Change Detection mechanism runs continuously. If you pass `[username]="currentName"`, and later do `this.currentName = 'Bob'`, Angular instantly detects the change, pushes the new value down into the child's `@Input()`, and re-renders the child's HTML.

## FEYNMAN CHECK

### Explain Angular Components Like I'm 10 Years Old
> An Angular component is a **TypeScript class + HTML template + CSS styles** packaged together and registered via the @Component decorator. When the Angular runtime finds app-user-list in a parent template, it looks up which component claims that selector, creates an instance, renders the template using the instance's properties, and inserts it into the DOM. The non-obvious detail: Angular creates a **ViewRef** object — a structured internal representation that tracks bindings, queries, and change-detection state separately from the actual DOM, enabling performant incremental updates without touching the entire DOM tree.

### 5 Deep Conceptual Questions

**Q1: What is the selector and why does Angular use CSS selectors?**
> **A:** The selector string ('app-user-list', '[appHighlight]', '.my-class') is a CSS selector. Angular matches components and directives to DOM elements using the same matching algorithm as CSS. Element selectors (app-*) create component-style custom elements; attribute selectors ([appHighlight]) create directive-style behaviour extensions. Using CSS selectors means Angular can determine which components apply to an element purely from the template at compile time — no runtime lookup table needed.

**Q2: What is the mental model for @Input and @Output?**
> **A:** @Input() declares a one-way data binding from PARENT to CHILD — the parent sets it via [propertyName]="value". @Output() declares an event emitter from CHILD to PARENT — the child emits via this.myEvent.emit(data) and the parent listens via (myEvent)="handler($event)". Together they enforce unidirectional data flow: data flows DOWN through inputs, events flow UP through outputs. This is the same philosophy as React props + callbacks, but Angular enforces it via decorator metadata visible to the compiler.

**Q3: Most dangerous misconception about components?**
> **A:** Thinking OnPush components always skip change detection:
> ```typescript
> // WRONG: Mutating an @Input object does not trigger OnPush re-render
> @Component({ changeDetection: ChangeDetectionStrategy.OnPush })
> export class ListComponent {
>   @Input() items: string[] = [];
> }
> // Parent: this.items.push('new') — reference unchanged, OnPush skips!
>
> // CORRECT: Replace the reference
> this.items = [...this.items, 'new item'];
> // Or use signals — they notify automatically
> ```

**Q4: How does Angular's view hierarchy relate to the DOM hierarchy?**
> **A:** Angular maintains a component tree parallel to the DOM tree — each component has an associated ViewRef containing bindings, queries, child views, and a dirty flag. When change detection runs, Angular walks this view tree (not the DOM) — reading component properties and comparing to last values. Only dirty nodes update the DOM. The DetachChangeDetectorRef API lets you remove a subtree from this walk entirely, and markForCheck re-attaches it — the foundation of manual performance tuning.

**Q5: FAANG-grade definition of Angular components.**
> **A:** "An Angular component is a TypeScript class annotated with @Component metadata (selector, template, styles, imports, changeDetection strategy) that Angular instantiates per matching DOM element — creating a ViewRef for incremental DOM diffing, processing @Input/@Output bindings for parent-child communication, querying child elements via @ViewChild/@ContentChild, and participating in change detection either via zone-based dirty marking or signal-based reactive notification."

#### Step 2 — Component
```typescript
// src/app/data-table/data-table.component.ts
import { Component, Input, Output, EventEmitter, ChangeDetectionStrategy, computed, signal } from '@angular/core';
import { NgFor, NgIf } from '@angular/common';

export interface Column<T> { key: keyof T; label: string; sortable?: boolean; }
export interface SortEvent<T> { key: keyof T; direction: 'asc' | 'desc'; }

@Component({
  standalone: true,
  selector: 'app-data-table',
  changeDetection: ChangeDetectionStrategy.OnPush,
  imports: [NgFor, NgIf],
  template: `
    <table>
      <thead>
        <tr>
          <th *ngFor="let col of columns"
              [style.cursor]="col.sortable ? 'pointer' : 'default'"
              (click)="col.sortable && sort(col.key)">
            {{ col.label }}
            <span *ngIf="sortKey() === col.key">{{ sortDirection() === 'asc' ? 'up' : 'down' }}</span>
          </th>
        </tr>
      </thead>
      <tbody>
        <tr *ngFor="let row of pagedRows()">
          <td *ngFor="let col of columns">{{ row[col.key] }}</td>
        </tr>
      </tbody>
    </table>
    <div>
      <button (click)="prevPage()" [disabled]="page() === 0">Prev</button>
      <span>Page {{ page() + 1 }} / {{ totalPages() }}</span>
      <button (click)="nextPage()" [disabled]="page() === totalPages() - 1">Next</button>
    </div>
  `,
})
export class DataTableComponent<T extends Record<string, unknown>> {
  @Input({ required: true }) columns: Column<T>[] = [];
  @Input({ required: true }) set rows(val: T[]) { this._rows.set(val); this._page.set(0); }
  @Input() pageSize = 10;
  @Output() sortChange = new EventEmitter<SortEvent<T>>();

  private _rows = signal<T[]>([]);
  private _page = signal(0);
  sortKey = signal<keyof T | null>(null);
  sortDirection = signal<'asc' | 'desc'>('asc');
  page = this._page.asReadonly();

  pagedRows = computed(() => {
    const start = this._page() * this.pageSize;
    return this._rows().slice(start, start + this.pageSize);
  });
  totalPages = computed(() => Math.max(1, Math.ceil(this._rows().length / this.pageSize)));

  sort(key: keyof T) {
    const dir = this.sortKey() === key && this.sortDirection() === 'asc' ? 'desc' : 'asc';
    this.sortKey.set(key); this.sortDirection.set(dir);
    this.sortChange.emit({ key, direction: dir });
  }
  prevPage() { if (this._page() > 0) this._page.update(p => p - 1); }
  nextPage() { if (this._page() < this.totalPages() - 1) this._page.update(p => p + 1); }
}
```

#### Step 4 — Error Handling: OnPush Reference Safety
```typescript
// Always give OnPush components NEW references
export class ParentComponent {
  rows = signal<any[]>([]);
  addRow(row: any) {
    // WRONG: this.rows().push(row) — OnPush won't detect
    // CORRECT: new reference via signal update
    this.rows.update(current => [...current, row]);
  }
}
```
