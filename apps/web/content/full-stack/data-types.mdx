# JavaScript Data Types — Primitives and References

To write predictable, high-performance JavaScript, you must understand the distinction between **Primitive values** and **Reference values**, how they are stored in memory, and how the engine evaluates them.

---

## 1. Deep-Dive Architectural Concepts

### Memory Architecture: Stack vs. Heap
The JavaScript engine uses two distinct memory structures to manage data:

```
MEMORY MODEL
┌─────────────────────────────────────────┐
│ STACK (Fixed-size, fast access)         │
│ ┌──────────────────┬──────────────────┐ │
│ │ Variable Name    │ Value            │ │
│ ├──────────────────┼──────────────────┤ │
│ │ age              │ 30               │ │
│ │ name             │ "Alice"          │ │
│ │ userPtr          │ 0x7ffd8a42b1     │─┼──┐
│ └──────────────────┴──────────────────┘ │  │
└─────────────────────────────────────────┘  │
                                             │
┌─────────────────────────────────────────┐  │
│ HEAP (Dynamic-size, slow access)        │  │
│ ┌─────────────────────────────────────┐ │  │
│ │ 0x7ffd8a42b1: {                     │ │◄─┘
│ │   street: "Main St",                │ │
│ │   city: "Boston"                    │ │
│ │ }                                   │ │
│ └─────────────────────────────────────┘ │
└─────────────────────────────────────────┘
```

* **The Stack**: A structured, fast-access memory region. It stores all **Primitive values** directly within the variable's stack frame, as their sizes are fixed and known at compile time. It also stores **pointers** (addresses) to objects in the Heap.
* **The Heap**: A large, unstructured memory region. It stores **Reference values** (Objects, Arrays, Functions) whose sizes are dynamic and can change during execution.

### Immutability of Primitives
Primitive values are immutable. You cannot modify a primitive value; you can only reassign a variable to a new primitive value.

```javascript
let str = "Hello";
str[0] = "h";      // Silently fails (or throws in strict mode)
console.log(str);  // Output: "Hello"

str = "world";     // Reassigns the variable to a completely new string in the stack
```

## 2. Core Syntax & Mechanics

### The 7 Primitive Types + Reference Type
JavaScript has 7 primitive types and 1 reference type:

| Type | Category | Description | typeof Result |
|---|---|---|---|
| **Undefined** | Primitive | Assigned to declared variables that have not been initialized. | `"undefined"` |
| **Null** | Primitive | Represents the intentional absence of any object value. | `"object"` (legacy bug) |
| **Boolean** | Primitive | Logical entity: `true` or `false`. | `"boolean"` |
| **Number** | Primitive | Double-precision 64-bit binary format IEEE 754 float. | `"number"` |
| **BigInt** | Primitive | Arbitrary-precision integers (ES2020). | `"bigint"` |
| **String** | Primitive | Sequence of UTF-16 code units. | `"string"` |
| **Symbol** | Primitive | Unique and immutable identifier (ES6). | `"symbol"` |
| **Object** | Reference | Collection of properties (includes Arrays, Functions, Maps). | `"object"` (or `"function"`) |

### The `typeof null` Bug
```javascript
console.log(typeof null); // Output: "object"
```
In the first implementation of JavaScript, values were stored in 32-bit words, consisting of a type tag (1-3 bits) and the actual value. The tag for objects was `000`. The representation of `null` was the machine null pointer (all zeros), which made its type tag read as `000` (Object). This is a known bug that cannot be fixed because it would break millions of legacy websites.

### The Mystery of `NaN`
`NaN` (Not-a-Number) is a special value of the **Number** type, returned when a mathematical operation fails or cannot yield a real number.

```javascript
console.log(typeof NaN); // Output: "number"

// NaN is the only value in JavaScript that is not equal to itself
console.log(NaN === NaN); // Output: false

// Correct detection methods
console.log(isNaN("hello"));       // Output: true (coerces "hello" to NaN first)
console.log(Number.isNaN("hello")); // Output: false (strict, "hello" is not of type Number)
console.log(Number.isNaN(NaN));     // Output: true
```

### Strict vs. Abstract Equality (`===` vs `==`)
* `===` (Strict Equality): Compares both the **type** and the **value**. No type coercion occurs.
* `==` (Abstract Equality): Coerces the operands to a common type before performing the comparison.

```javascript
console.log(1 === "1"); // Output: false
console.log(1 == "1");  // Output: true (coerces string "1" to number 1)
```

## 3. Advanced Patterns & Implementations

### Deep Cloning Objects (Handling Circular References)
Because reference types store pointers, copying an object with a shallow assignment (`const b = a`) only copies the pointer. A true deep clone must recursively copy all nested objects while handling circular references to prevent infinite loops.

```javascript
function deepClone(obj, seen = new WeakMap()) {
  if (obj === null || typeof obj !== "object") {
    return obj; // Return primitives as-is
  }

  // Handle circular references
  if (seen.has(obj)) {
    return seen.get(obj);
  }

  // Handle Dates and RegExps
  if (obj instanceof Date) return new Date(obj);
  if (obj instanceof RegExp) return new RegExp(obj);

  // Initialize clone as array or object
  const clone = Array.isArray(obj) ? [] : {};
  seen.set(obj, clone);

  for (const key of Reflect.ownKeys(obj)) {
    clone[key] = deepClone(obj[key], seen);
  }

  return clone;
}

// Verification
const original = { a: 1, b: { c: 2 } };
original.self = original; // Circular reference

const cloned = deepClone(original);
console.log(cloned !== original);     // Output: true
console.log(cloned.b !== original.b); // Output: true
console.log(cloned.self === cloned);  // Output: true (circular link preserved)
```

## 4. Anti-Patterns & Common Pitfalls

### The Falsy Value Trap
JavaScript has exactly 8 falsy values: `false`, `0`, `-0`, `0n`, `""` (empty string), `null`, `undefined`, and `NaN`. Everything else is truthy, including empty objects `{}` and empty arrays `[]`.

```javascript
// ANTI-PATTERN: Checking for object existence or array elements using truthiness directly
function processData(config) {
  if (!config) {
    console.log("No config provided");
    return;
  }
  console.log("Processing...");
}

processData({}); // Output: "Processing..." (empty object is truthy!)
```

```javascript
// CORRECT PATTERN: Explicitly check for properties or length
function processData(config) {
  if (config === null || typeof config !== "object" || Object.keys(config).length === 0) {
    console.log("Invalid or empty config");
    return;
  }
  console.log("Processing...");
}
```

## 6. Ecosystem & Real-World Application

### Structural Cloning in Modern Runtimes
In modern browsers and Node.js (v17+), you do not need a custom deep-clone utility or `JSON.parse(JSON.stringify(obj))` (which strips functions, Symbols, and breaks on circular references). Use the native **`structuredClone()`** API.

```javascript
const user = {
  name: "Alice",
  joined: new Date(),
  data: new Uint8Array([1, 2, 3])
};

const clone = structuredClone(user);
console.log(clone.joined instanceof Date); // Output: true
console.log(clone.data !== user.data);     // Output: true

## WHY
In Java or C++, when you create a variable, you must explicitly declare its type in memory (e.g., `int x = 5;`). If you try to store a word in that variable (`x = "Hello"`), the compiler violently crashes.

JavaScript is **Dynamically Typed**. You never declare the type of a variable. The V8 engine looks at the data you assign and figures out the type on the fly! `let x = 5;` creates a Number. If you immediately do `x = "Hello"`, V8 seamlessly destroys the number, transitions the variable, and stores a String. While this makes JS incredibly fast to write, it requires developers to deeply understand the 8 fundamental Data Types, otherwise they will accidentally perform math on Strings and crash their applications.

## THEORY
### Primitives vs Objects
JavaScript has exactly 8 native Data Types. They are divided into two distinct categories:
1. **The 7 Primitives**: `String`, `Number`, `Boolean`, `Undefined`, `Null`, `BigInt`, and `Symbol`.
   - Primitives are immutable (they cannot be fundamentally altered).
   - They are stored directly on the fast Call Stack.
   - They are passed by **Value** (copying the raw data).
2. **The 1 Complex Type**: `Object`. 
   - (Note: Arrays and Functions are not distinct types in JS; they are just specialized Objects!)
   - Objects are mutable.
   - They are stored in the massive Memory Heap.
   - They are passed by **Reference** (copying the memory address pointer).

// 6. BigInt (Added in ES2020 to handle massive astronomical numbers)

// 7. Symbol (Added in ES6 to create guaranteed unique identifiers)

### Level 2 — Building Blocks (The typeof Operator)
```javascript
// You can ask the JS Engine what type of data is currently in a bucket!

console.log(typeof "Hello"); // "string"
console.log(typeof 42);      // "number"
console.log(typeof true);    // "boolean"
console.log(typeof undefined); // "undefined"

// THE INFAMOUS JAVASCRIPT BUG:
console.log(typeof null); // Output: "object" 

// Wait, what?! Null is a primitive, not an object! 
// Why does it say "object"? Because of a typo written into the original JS engine 
// in 1995. Fixing it now would break billions of websites, so it remains forever.

// ARRAYS ARE OBJECTS:
console.log(typeof [1, 2, 3]); // Output: "object"
```

### Level 3 — Intermediate (Pass by Value vs Reference)
```javascript
// 1. Primitives are Passed by VALUE
let a = 10;
let b = a; // V8 creates a BRAND NEW copy of the number 10 for bucket 'b'
b = 20;

console.log(a); // 10 (A is completely unaffected)

// 2. Objects are Passed by REFERENCE
const user1 = { name: "Alice" };
const user2 = user1; // V8 does NOT copy the object. It copies the MEMORY POINTER.

// Both variables now point to the exact same object in the Heap!
user2.name = "Bob";

console.log(user1.name); // "Bob" (Disaster! User1 was mutated!)
```

### Level 4 — Advanced (Numbers and Floating Point Math)
```javascript
// Because all Numbers in JS are 64-bit Floating Points (IEEE 754 standard),
// JS is notoriously bad at simple decimal math.

const mathResult = 0.1 + 0.2;
console.log(mathResult); // Output: 0.30000000000000004

// Why?! Because the computer converts 0.1 into base-2 binary. 
// 0.1 results in an infinite repeating fraction in binary! The CPU runs out of memory,
// rounds off the end, and introduces a microscopic error.

// NEVER use JavaScript to calculate financial currency! 
// If you are building a banking app, you must store all money as Cents (Integers)
// and format it into dollars later, or use a specialized library like decimal.js!

const priceInCents = 10;
const taxInCents = 20;
console.log(priceInCents + taxInCents); // 30 (Perfect precision)
```

## REAL_WORLD
When building React applications, the "Pass by Reference" behavior of Objects is the #1 cause of bugs for junior developers. 

React's entire rendering engine relies on checking if data has changed. If you have an array of users `[UserA, UserB]`, and you want to add `UserC`, you might do `users.push(UserC)`. 
However, React checks the memory pointer of the array. Because you mutated the existing array in memory, the pointer hasn't changed! React assumes nothing happened and refuses to update the UI! 

To fix this, you must always treat Objects and Arrays as **Immutable**. You must create a *brand new* array in memory, copy the old users over, and add the new one. This generates a brand new pointer on the Call Stack, forcing React to perfectly update the UI.

## INTERVIEW
**Q: How do you verify if a variable is an Array, since `typeof []` returns "object"?**
A: You must use the built-in `Array.isArray(myVar)` method, which mathematically guarantees the object is an array prototype.

**Q: What is `NaN` and what is its type?**
A: `NaN` stands for "Not a Number". It is the result of an invalid mathematical operation, such as dividing a string by a number (`"hello" / 5`). Ironically, if you check `typeof NaN`, the result is `"number"`. It is a specialized numeric value representing a broken calculation. Furthermore, `NaN` is the only value in JavaScript that is not equal to itself (`NaN === NaN` is false).

**Q: Explain the difference between `null` and `undefined`.**
A: `undefined` means a variable has been declared in RAM, but it currently has no value. It is the engine's default placeholder. `null` is an explicit, empty value that a programmer manually assigns to signify "I deliberately want this variable to be empty". When dealing with optional API fields, you should almost always check for both.

## FEYNMAN CHECK

### Explain JS Data Types Like I'm 10 Years Old
> JavaScript has 8 types — 7 PRIMITIVES (`string`, `number`, `bigint`, `boolean`, `undefined`, `null`, `symbol`) and 1 OBJECT type (which covers everything else: arrays, functions, dates, regexps, Maps, Sets, plain objects). Primitives are IMMUTABLE and PASSED BY VALUE — `let a = 5; let b = a; b = 6` leaves a as 5. Objects are MUTABLE and PASSED BY REFERENCE — two variables can point to the SAME underlying object. The non-obvious detail: `typeof null === 'object'` is a famous bug from JS's first month that can never be fixed without breaking the web. `typeof NaN === 'number'`. `typeof []` is also 'object' — use `Array.isArray()` for arrays. `number` is always a 64-bit IEEE 754 float — there is no separate int type, which is why 0.1 + 0.2 !== 0.3.

---

### 5 Deep Conceptual Questions

**Q1: Why does `0.1 + 0.2` equal `0.30000000000000004`?**
> **A:** JavaScript numbers are IEEE 754 double-precision floats. 0.1 cannot be represented exactly in binary (just like 1/3 cannot be exact in decimal); it's a repeating fraction approximated to ~17 significant digits. When you add two approximations, the rounding errors add up. The fix for money: use integers in cents (`19_99` cents = $19.99), use BigInt for arbitrary integer precision, or use a decimal library (Decimal.js).

**Q2: Mental model for primitives vs objects?**
> **A:** A primitive value is COPIED when assigned or passed; modifying the copy does NOT affect the original. An object is a REFERENCE — both variables point to the SAME memory location. `let a = {x:1}; let b = a; b.x = 5` → a.x is also 5 because a and b reference the same object. This is why "immutable update patterns" (`{...obj, x: 5}`) matter — they create a new object rather than mutate.

**Q3: Most dangerous misconception?**
> **A:** `typeof null === 'object'`:
> ```js
> // ❌ Famous bug — typeof null is 'object', not 'null'
> typeof null;        // 'object'
> typeof undefined;   // 'undefined'
> typeof [];          // 'object' (yes, arrays are objects)
> typeof function(){};// 'function' (special case)
> typeof NaN;         // 'number' (NaN is a number)
>
> // ✅ Robust type checks
> Array.isArray(x);                          // for arrays
> x === null;                                 // for null
> Object.prototype.toString.call(x);         // returns '[object Date]', '[object RegExp]', etc.
> ```

**Q4: How does the JS engine store these types internally?**
> **A:** V8 uses TAGGED POINTERS — every value is a 64-bit pointer where the LOW BITS indicate the type tag (e.g., 1 = small integer "Smi", 0 = pointer to heap object). Small integers (-2^31 to 2^31-1) are stored INLINE in the pointer — no heap allocation, blazing fast. Floats, large numbers, strings, and objects live on the heap with a hidden "Map" pointer at offset 0 describing the shape (used for property-access optimisation). This is why monomorphic code (same shape) is 10× faster than polymorphic.

**Q5: FAANG-grade definition?**
> **A:** "JavaScript values are typed at runtime as one of 7 primitive types (string, number, bigint, boolean, undefined, null, symbol) or as an object — primitives are immutable and copy-by-value, objects are mutable references stored on the heap with V8 hidden classes describing their shape — with type coercion governed by abstract equality and string/number conversion algorithms that produce many of the language's famous footguns."

#### Step 2 — Money Class
```js
// money.js
export class Money {
  /** @param {number} cents - integer cents (NOT dollars) */
  constructor(cents, currency = 'USD') {
    if (!Number.isInteger(cents)) {
      throw new TypeError(`Money requires integer cents, got ${cents}`);
    }
    this._cents = cents;
    this._currency = currency;
    Object.freeze(this);   // immutable
  }

  static fromDollars(dollars, currency = 'USD') {
    // Multiply by 100 then round to avoid 1.10 * 100 = 110.00000000000001
    return new Money(Math.round(dollars * 100), currency);
  }

  add(other) {
    this._sameCurrency(other);
    return new Money(this._cents + other._cents, this._currency);
  }

  subtract(other) {
    this._sameCurrency(other);
    return new Money(this._cents - other._cents, this._currency);
  }

  multiply(factor) {
    return new Money(Math.round(this._cents * factor), this._currency);
  }

  format(locale = 'en-US') {
    return new Intl.NumberFormat(locale, {
      style: 'currency', currency: this._currency,
    }).format(this._cents / 100);
  }

  _sameCurrency(other) {
    if (this._currency !== other._currency) {
      throw new Error(`Currency mismatch: ${this._currency} vs ${other._currency}`);
    }
  }
}
```
