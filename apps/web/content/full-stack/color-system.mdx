### 1. Figma Workflow & Token Sync Pipeline
The color system utilizes a semantic architecture mapping primitives (raw hues) to semantic tokens (roles) and component tokens (scoped targets).

```
[ Primitive Token ] ──> [ Semantic Token ] ──> [ Component Token ]
(e.g., Slate-900)       (e.g., bg-surface)      (e.g., button-primary-bg)
```

In Figma, this is managed across light and dark collections. The JSON output defines theme-specific semantic layers that reference global primitives.

### 3. Accessibility Specifications (WCAG 2.1 AA)
- **Contrast Ratios**: Body text on canvas must meet **4.5:1** contrast. Large headings must meet **3:1**.
- **Interactive States**: Accent borders or active icons must adhere to **3:1** minimum contrast against adjacent backgrounds.
- **Color Blindness**: Critical semantic states (errors, warnings, success) must never rely solely on color. They must include text labels, visual shapes, or patterns.

## WHY
Color is the most immediately recognizable aspect of a brand. However, in UI design, color is not just about aesthetics—it is about communication. If your "Delete Account" button is the same soft blue as your "Save Profile" button, users will make mistakes. 

A systematic approach to color ensures consistency, guarantees accessibility (contrast ratios), and makes it trivial to implement features like Dark Mode. You cannot build a scalable application by picking random hex codes from a color wheel.

## THEORY
### The Core Palette Structure
A professional design system rarely relies on just a "Primary" and "Secondary" color. It requires a structured palette broken down by intent:

1. **Brand/Primary Colors:** The identity of your app (e.g., Twitter Blue, Spotify Green).
2. **Neutral/Grayscale Colors:** Used for text, backgrounds, borders, and subtle UI elements. (You usually need 9-10 shades of gray).
3. **Semantic/Feedback Colors:** Colors with universal meaning.
   - **Success (Green):** Operations completed successfully.
   - **Warning (Yellow/Orange):** Action required, or potential issues.
   - **Danger/Error (Red):** Destructive actions or system failures.
   - **Info (Blue):** Neutral information or highlights.

## CODE

### Level 1 — Beginner (The 50-900 Scale)
Never define just one hex code for a color. Define a **scale** from 50 (lightest) to 900 (darkest), just like CSS font-weights. 

```css
/* Example of a Primary Blue Scale */
:root {
  --blue-50:  #eff6ff; /* Backgrounds for active states */
  --blue-100: #dbeafe; /* Very light subtle backgrounds */
  --blue-200: #bfdbfe;
  --blue-300: #93c5fd;
  --blue-400: #60a5fa;
  --blue-500: #3b82f6; /* The CORE brand color (Buttons) */
  --blue-600: #2563eb; /* Hover state for buttons */
  --blue-700: #1d4ed8; /* Active/Pressed state */
  --blue-800: #1e40af; /* Dark text on light blue backgrounds */
  --blue-900: #1e3a8a; 
}
```

### Level 2 — Building Blocks (Semantic Mapping)
Once you have your global scale (e.g., `blue-500`, `red-500`), you must map them to **Semantic Tokens**. Your UI components should *never* reference the raw color directly.

```css
/* Good: Mapping global scales to semantic meaning */
:root {
  --color-primary-base: var(--blue-500);
  --color-primary-hover: var(--blue-600);
  
  --color-danger-base: var(--red-500);
  --color-danger-hover: var(--red-600);
  
  --text-default: var(--gray-900);
  --text-muted: var(--gray-500);
}

/* The Button component only uses semantic tokens! */
.button-primary {
  background-color: var(--color-primary-base);
  color: white;
}
.button-primary:hover {
  background-color: var(--color-primary-hover);
}
```

### Level 3 — Intermediate (Accessible Contrast - WCAG)
If you put white text on a yellow button, users with visual impairments will not be able to read it. The Web Content Accessibility Guidelines (WCAG) enforce strict mathematics for color contrast.

- **AA Standard:** Text and its background must have a contrast ratio of at least **4.5:1** (or 3:1 for large text).
- **AAA Standard (Strict):** Contrast ratio of at least **7:1**.

```css
/* DANGER: Yellow-400 (#facc15) with White (#ffffff) text has a contrast ratio of 1.2:1! */
/* This is completely illegible and fails all accessibility standards. */
.btn-warning-bad {
  background-color: #facc15; 
  color: #ffffff; /* Fails WCAG */
}

/* FIX: Use Dark Gray or Black text on light/bright backgrounds! */
.btn-warning-good {
  background-color: #facc15;
  color: #111827; /* Passes WCAG (Ratio > 9:1) */
}
```
*Pro Tip: Most design systems dictate that any color from `50` to `400` requires dark text, and `500` to `900` requires white text.*

### Level 4 — Advanced (HSL and Programmatic Generation)
If your user can pick a custom "Theme Color", you can't pre-generate the 50-900 scale. You have to generate it programmatically using **HSL (Hue, Saturation, Lightness)**.

```css
:root {
  /* We define the raw Hue and Saturation of the brand color */
  --brand-h: 220;
  --brand-s: 90%;
  
  /* We generate the scale dynamically by manipulating Lightness! */
  --brand-50:  hsl(var(--brand-h), var(--brand-s), 95%);
  --brand-500: hsl(var(--brand-h), var(--brand-s), 50%); /* Base */
  --brand-900: hsl(var(--brand-h), var(--brand-s), 10%);
}
```
Now, if you change `--brand-h` to `10` (Red) using JavaScript, the *entire* 50-900 scale updates perfectly and instantly!

## REAL_WORLD
**Tinted Grays**
Pure gray (`#808080` or `rgb(128,128,128)`) looks dead and lifeless in a modern UI. Top-tier design systems (like Stripe or Tailwind) use **Tinted Grays**. They inject a tiny amount of the brand's primary color (usually blue or purple) into the gray scale.
- Slate: Gray with a hint of blue.
- Zinc: Gray with a hint of purple.
- Stone: Gray with a hint of yellow/brown (warm).
Using tinted grays makes your application look incredibly premium and cohesive, because the backgrounds harmonize with the primary brand buttons.

## INTERVIEW
**Q: How do you handle Dark Mode in a semantic color system?**
A: You invert the semantic mappings, NOT the global scales! `blue-500` should always look like `blue-500`. However, `--bg-default` (which maps to `gray-50` in Light Mode) will be re-mapped to `gray-900` in Dark Mode. Furthermore, in Dark Mode, you often use *lighter* shades (like `blue-400` instead of `blue-500` or `blue-600`) for primary text to ensure sufficient contrast against the dark backgrounds.

## FEYNMAN CHECK

### Explain Color Systems Like I'm 10 Years Old
> A colour system is like a paint factory with a strict naming rule. Instead of 500 different blues that everyone picks from randomly, you have 10 numbered buckets (blue-100 through blue-900) and a set of meaning names (brand-primary, text-danger, surface-warning). Designers pick from meaning names, not raw buckets, so changing the entire brand colour takes one edit instead of finding every place blue-500 was used. The non-obvious part: raw palette tokens (blue-500) and semantic tokens (textPrimary) are two different layers, and confusing them is why dark mode themes break. This is why Tailwind's arbitrary colours make theme switching impossible.

---

### 5 Deep Conceptual Questions

**Q1: What problem does semantic token naming solve that raw palette naming cannot?**
> **A:** Raw palette names (blue-500) describe the colour itself; semantic names (textPrimary, surfaceDanger) describe the role. When you switch to dark mode, `textPrimary` points to white instead of near-black — one token change, zero component changes. With raw palette names, every component hardcoded to `blue-500` must be manually updated. At 10,000 component usages across a large design system, semantic tokens are the difference between a one-file theme switch and a three-month migration.

**Q2: What is the ONE mental model for a two-layer colour system?**
> **A:** "Layer 1 = the palette (what colours exist as numbered scales); Layer 2 = semantics (what each colour means in context). Components ALWAYS reference Layer 2. Layer 2 references Layer 1. Themes only ever swap Layer 2 mappings." This means a theme is just a mapping file — `{ textPrimary: gray-900 }` in light, `{ textPrimary: gray-100 }` in dark.

**Q3: Most dangerous misconception with code.**
> **A:** Using raw palette tokens in components — dark mode breaks completely.
> ```css
> /* ❌ Hardcoded to light theme — dark mode impossible */
> .button { background: var(--blue-600); color: var(--gray-50); }
>
> /* ✅ Semantic reference — theme swap changes behaviour everywhere */
> .button { background: var(--color-interactive-primary); color: var(--color-text-on-primary); }
> ```

**Q4: How do colour tokens interact with accessibility contrast requirements?**
> **A:** Semantic token pairs (e.g. `textPrimary`/`surfacePrimary`) must be verified to meet WCAG 4.5:1 contrast in EVERY theme at token definition time. This means the design token pipeline should include automated contrast checks when theme mappings are defined, not left to individual components to verify. The correct pattern: define contrast-safe pairs and encode them as the only allowed text-on-surface combinations.

**Q5: Senior one-liner.**
> **A:** "A colour system separates palette (what exists) from semantics (what it means) so that theming is a mapping change, not a component rewrite — which is why using raw hex or palette tokens in components makes theme switching O(n components) instead of O(1 mapping file)."

#### Step 2 — Core Implementation
```css
/* tokens.css */
/* Layer 1: Palette — raw colour scale */
:root {
  --palette-blue-500: #3b82f6;
  --palette-blue-700: #1d4ed8;
  --palette-gray-900: #111827;
  --palette-gray-100: #f3f4f6;
  --palette-red-600:  #dc2626;
  --palette-white:    #ffffff;
}

/* Layer 2: Semantic — light theme */
:root {
  --color-text-primary:       var(--palette-gray-900);
  --color-surface-primary:    var(--palette-white);
  --color-interactive-primary:var(--palette-blue-500);
  --color-interactive-hover:  var(--palette-blue-700);
  --color-text-on-primary:    var(--palette-white);
  --color-feedback-danger:    var(--palette-red-600);
}

/* Layer 2: Semantic — dark theme override */
@media (prefers-color-scheme: dark) {
  :root {
    --color-text-primary:       var(--palette-gray-100);
    --color-surface-primary:    #1f2937; /* palette-gray-800 */
    --color-interactive-primary:var(--palette-blue-500);
    /* blue-500 works on both backgrounds — verify at definition time */
  }
}
```

#### Step 4 — Error Handling
```ts
// Contrast verification at token definition time (Node.js)
import { getContrastRatio } from 'wcag-contrast';
const pairs = [
  { fg: '#111827', bg: '#ffffff', pair: 'textPrimary/surfacePrimary light' },
  { fg: '#f3f4f6', bg: '#1f2937', pair: 'textPrimary/surfacePrimary dark' },
];
for (const { fg, bg, pair } of pairs) {
  const ratio = getContrastRatio(fg, bg);
  if (ratio < 4.5) throw new Error(`FAIL: ${pair} = ${ratio.toFixed(2)}:1`);
  console.log(`PASS: ${pair} = ${ratio.toFixed(2)}:1`);
}
```

#### Step 5 — Tests
```ts
test('all semantic pairs meet WCAG AA', () => {
  const lightPairs = [['#111827','#ffffff'],['#ffffff','#3b82f6']];
  for (const [fg, bg] of lightPairs) {
    expect(getContrastRatio(fg, bg)).toBeGreaterThanOrEqual(4.5);
  }
});
```
