# JavaScript Event Loop — Complete Internal Model

The JavaScript runtime environment is designed around a single-threaded execution model. To achieve high-throughput, non-blocking I/O, engines like V8 coordinate with host environments (the browser’s rendering engine or Node.js’s `libuv` platform) via an asynchronous event loop.

---

## 1. The V8 Call Stack

The V8 Call Stack is a Last-In, First-Out (LIFO) data structure that tracks the execution of synchronous code. 

### Stack Frames (Activation Records)
When a JavaScript function is invoked, V8 allocates a **Stack Frame** (or Activation Record) on the call stack. This frame stores:
*   **Return Address**: The instruction pointer to resume execution at after the function returns.
*   **Local Variables and Parameters**: Primitive values and references to heap-allocated objects.
*   **Execution Context**: A pointer to the current lexical environment and the `this` binding.

```
+------------------------------------------+
|  Frame 3: baz() [Local vars, Return ADDR] |  <-- Stack Pointer (Top)
+------------------------------------------+
|  Frame 2: bar() [Local vars, Return ADDR] |
+------------------------------------------+
|  Frame 1: foo() [Local vars, Return ADDR] |
+------------------------------------------+
|  Global Execution Context                |  <-- Bottom of Stack
+------------------------------------------+
```

### Stack Overflow and `StackGuard`
V8 has a hard limit on the call stack size (typically 10,000 to 50,000 frames depending on the host OS and architecture). This boundary is monitored by V8’s internal `StackGuard` class (`src/execution/stack-guard.h`). 

If a recursive function lacks a base case, it continuously pushes frames:

```javascript
function recurse() {
    recurse();
}
recurse(); // Uncaught RangeError: Maximum call stack size exceeded
```

When the stack pointer crosses the memory address limit set by `StackGuard`, V8 halts execution and throws a `RangeError`.

---

## 2. Web APIs and libuv: Out-of-V8 Execution

V8 is purely an execution engine; it does not have built-in timers (`setTimeout`), network capabilities (`fetch`), or DOM awareness. These are **Host-Defined APIs**.

```
+-----------------------------------------------------------------------+
|                           HOST ENVIRONMENT                            |
|                                                                       |
|   +-------------------+                     +---------------------+   |
|   |     V8 Engine     |                     |      Web APIs       |   |
|   |                   |   Binding Call      |  - setTimeout       |   |
|   |  +-------------+  | ------------------> |  - fetch            |   |
|   |  | Call Stack  |  |                     |  - DOM Events       |   |
|   |  +-------------+  | <------------------ +---------------------+   |
|   +-------------------+   Push Callback                |              |
|             ^                                          |              |
|             |                                          v              |
|             |                               +---------------------+   |
|             +------------------------------ |   Callback Queues   |   |
|                     Event Loop Tick         +---------------------+   |
+-----------------------------------------------------------------------+
```

*   **Browsers**: Web APIs are implemented in C++ by the browser engine (e.g., Blink in Chrome). When you call `fetch()`, V8 makes a C++ binding call to Blink. Blink handles the network socket on a background browser thread.
*   **Node.js**: Asynchronous operations are delegated to **libuv**, a multi-platform C library focusing on asynchronous I/O. libuv manages a system-level thread pool (default size of 4, configurable via `UV_THREADPOOL_SIZE`) and uses OS-specific asynchronous mechanisms like `epoll` (Linux), `kqueue` (macOS), or `IOCP` (Windows).

Once the background operation completes, the host environment (Blink or libuv) wraps the result in a callback and places it into the appropriate queue.

---

## 3. The Queue Architecture

The Event Loop manages two primary categories of queues:

### The Microtask Queue
This queue handles high-priority tasks that must be executed immediately after the current synchronous execution block finishes, before control yields to rendering or macrotasks.
*   **Sources**: `Promise.resolve().then()` (specifically V8's internal `PromiseReactionJob`), `queueMicrotask()`, and `MutationObserver` callbacks.
*   **V8 Implementation**: Managed directly by V8 inside `v8::internal::MicrotaskQueue` (`src/execution/microtask-queue.cc`).

### The Callback Queue (Macrotask Queue)
This queue handles standard asynchronous operations.
*   **Sources**: `setTimeout`, `setInterval`, network callbacks (`fetch`/`xhr`), DOM events, `setImmediate` (Node.js only), and I/O.
*   **Management**: Handled entirely by the host environment (Blink's task scheduler or libuv's event loop).

## 4. The Event Loop Algorithm (Precise Spec Model)

The browser event loop operates according to the HTML5 Specification (Section 8.1.4). The exact execution sequence is as follows:

```
                  +------------------------+
                  |   Execute Sync Code    |
                  |   (Drain Call Stack)   |
                  +------------------------+
                              |
                              v
                +----------------------------+
                | DRAIN Microtask Queue      | <------+
                | (Including newly queued)   |        | (Loop until
                +----------------------------+        |  queue empty)
                              |                       |
                              +-----------------------+
                              |
                              v
                  +------------------------+
                  | Take ONE Macrotask     |
                  |   and execute it       |
                  +------------------------+
                              |
                              v
                +----------------------------+
                | DRAIN Microtask Queue      | <------+
                | (Including newly queued)   |        | (Loop until
                +----------------------------+        |  queue empty)
                              |                       |
                              +-----------------------+
                              |
                              v
                  +------------------------+
                  |  Need Render? (vsync)  |
                  +------------------------+
                             / \
                            /   \
                     Yes   /     \   No
                          v       v
         +--------------------------+     +-------------------+
         | Run requestAnimationFrame|     |  Loop back to     |
         | Run Style, Layout, Paint |     |  next Macrotask   |
         +--------------------------+     +-------------------+
                      |                             ^
                      v                             |
                      +-----------------------------+
```

1.  **Execute Synchronous Code**: The call stack is processed until it is completely empty.
2.  **Drain Microtask Queue**:
    *   The engine processes the Microtask Queue until it is completely empty.
    *   *Critical Detail*: If a microtask queues another microtask, the newly added microtask is executed **during the same tick**. This can lead to infinite microtask loops (starvation of the Macrotask Queue and UI rendering).
3.  **Process One Macrotask**:
    *   The event loop pulls exactly **one** task from the Macrotask Queue and pushes it onto the Call Stack.
    *   The Call Stack executes this task to completion (stack is empty again).
4.  **Drain Microtask Queue**:
    *   The Microtask Queue is drained completely again.
5.  **Render Phase (Browsers only)**:
    *   The browser determines if a frame update is required (typically every 16.6ms for 60Hz displays).
    *   If a render is scheduled, it executes `requestAnimationFrame` (rAF) callbacks, followed by recalculating styles, layout, and painting the pixels.
6.  **Repeat**: Loop back to Step 3.

### Implication: Microtask Precedence
Because the entire microtask queue is drained before the next macrotask is pulled, a resolved Promise callback will *always* execute before a `setTimeout(fn, 0)` callback, even if the timer was registered first.

```javascript
setTimeout(() => console.log("Macrotask (setTimeout)"), 0);

Promise.resolve().then(() => {
    console.log("Microtask 1");
    Promise.resolve().then(() => console.log("Microtask 2 (nested)"));
});

console.log("Synchronous");

// Output:
// 1. Synchronous
// 2. Microtask 1
// 3. Microtask 2 (nested)
// 4. Macrotask (setTimeout)
```

## 5. Node.js Event Loop (libuv)

Node.js uses libuv for its event loop, which differs from the browser specification. The libuv event loop consists of 6 distinct phases executed in a loop:

```
   ┌───────────────────────────┐
┌─>│          TIMERS           │ <--- setTimeout(), setInterval()
│  └─────────────┬─────────────┘
│                v
│  ┌───────────────────────────┐
│  │     PENDING CALLBACKS     │ <--- TCP errors, system-level I/O
│  └─────────────┬─────────────┘
│                v
│  ┌───────────────────────────┐
│  │       IDLE, PREPARE       │ <--- Internal use only
│  └─────────────┬─────────────┘
│                v
│  ┌───────────────────────────┐
│  │           POLL            │ <--- Incoming connections, data, I/O
│  └─────────────┬─────────────┘
│                v
│  ┌───────────────────────────┐
│  │           CHECK           │ <--- setImmediate()
│  └─────────────┬─────────────┘
│                v
│  ┌───────────────────────────┐
│  │      CLOSE CALLBACKS      │ <--- socket.on('close', ...)
└─┬└───────────────────────────┘
  │
  v
[process.nextTick() & Microtasks executed between phases]
```

### The 6 Phases
1.  **Timers**: Executes callbacks scheduled by `setTimeout()` and `setInterval()`.
2.  **Pending Callbacks**: Executes I/O callbacks deferred to the next loop iteration (e.g., system errors like `ECONNREFUSED`).
3.  **Idle, Prepare**: Used internally by libuv for housekeeping.
4.  **Poll**: Retrieves new I/O events. Node.js blocks here if there are no timers or immediate scripts scheduled.
5.  **Check**: Executes `setImmediate()` callbacks.
6.  **Close Callbacks**: Executes close event handlers (e.g., `socket.destroy()`).

### `process.nextTick()` and Microtasks in Node.js
*   `process.nextTick()` is **not** part of libuv's event loop. It belongs to Node.js's native layer.
*   The `nextTick` queue is processed immediately after the current operation completes, regardless of the current phase of the event loop.
*   The `nextTick` queue has strict priority over the Promise microtask queue.

```javascript
setTimeout(() => console.log("Timeout"), 0);
setImmediate(() => console.log("Immediate"));

Promise.resolve().then(() => console.log("Promise"));
process.nextTick(() => console.log("NextTick"));

// Output:
// 1. NextTick
// 2. Promise
// 3. Timeout (depending on entry speed, usually runs before Immediate)
// 4. Immediate
```

## WHY
JavaScript is strictly **Single-Threaded**. It only has one Call Stack, meaning it can only execute one piece of code at a time.
So, if JavaScript is single-threaded, how can it make an HTTP request that takes 5 seconds, whilst simultaneously updating a smooth 60FPS animation on the screen, without freezing the browser?
The answer is the **Event Loop**. It is the invisible conductor of JavaScript. Understanding it is the difference between a Junior developer who guesses why `setTimeout` is behaving weirdly, and a Senior developer who guarantees perfect performance.

## THEORY
### The Big Picture Components
1. **The Call Stack:** Where your JavaScript code actually executes. It runs synchronously, line by line.
2. **Web APIs (or Node.js C++ APIs):** These are NOT JavaScript! When you call `fetch()` or `setTimeout()`, JS offloads the heavy lifting to the browser's C++ threads. The JS thread moves on immediately!
3. **The Callback Queue (Task Queue):** When the Web API finishes its job (the timer hits 0, or the HTTP request returns), it pushes your callback function into this queue.
4. **The Event Loop:** An infinite `while` loop. Its only job is to look at the Call Stack. **IF the Call Stack is completely empty**, it takes the first item from the Callback Queue and pushes it onto the Call Stack to execute.

## CODE

### Level 1 — The Classic Interview Question
Look at this code. What order do the logs print in?

```javascript
console.log("First");

setTimeout(() => {
  console.log("Second");
}, 0); // Wait 0 milliseconds!

console.log("Third");
```
**The Output:** First -> Third -> Second.
*Why?*
1. `console.log("First")` goes on the Call Stack. Prints. Pops off.
2. `setTimeout` goes on the Call Stack. JS passes the callback to the Browser Timer API and instantly pops `setTimeout` off the stack.
3. `console.log("Third")` goes on the Call Stack. Prints. Pops off.
4. Meanwhile, the Browser Timer finishes instantly (0ms) and puts the callback in the Task Queue.
5. The Event Loop sees the Call Stack is finally empty. It moves the callback from the Queue to the Stack.
6. `console.log("Second")` prints!

*Even with a 0ms delay, `setTimeout` guarantees the code will run AFTER all synchronous code has finished executing!*

### Level 2 — Macrotasks vs. Microtasks
The reality is slightly more complex. There isn't just one Queue. There are two!
1. **Macrotask Queue:** `setTimeout`, `setInterval`, DOM Events (clicks).
2. **Microtask Queue:** Promises (`.then()`, `await`), `MutationObserver`.

**The Golden Rule:** The Event Loop prioritizes the Microtask Queue over the Macrotask Queue! It will completely drain the Microtask Queue before it even looks at a Macrotask.

```javascript
console.log("1. Start");

setTimeout(() => console.log("2. Timeout (Macrotask)"), 0);

Promise.resolve().then(() => console.log("3. Promise (Microtask)"));

console.log("4. End");
```
**The Output:** 
`1. Start`
`4. End`
`3. Promise (Microtask)`
`2. Timeout (Macrotask)`

Even though the Timeout and the Promise were scheduled at the exact same time, the Promise executes first because it is placed in the VIP line (the Microtask Queue)!

### Level 3 — Blocking the Event Loop (The Fatal Mistake)
If the Event Loop's rule is *"I can only move items from the Queue to the Stack if the Stack is empty"*, what happens if the Stack is NEVER empty?

```javascript
function freezeBrowser() {
  // A synchronous infinite loop stays on the Call Stack forever!
  while(true) {
    // ... do heavy math
  }
}

document.getElementById('btn').addEventListener('click', () => {
  console.log("Button Clicked!");
});

// If you call this function, the Call Stack locks up.
freezeBrowser(); 
```
If `freezeBrowser` is running, the Call Stack is full.
If the user furiously clicks the button 50 times, the Browser API registers the clicks and puts 50 `console.log` callbacks into the Task Queue.
But the Event Loop can never move them! The Call Stack never empties. The browser UI completely freezes, animated GIFs stop spinning, and eventually, the browser prompts the user to "Kill Page".

### Level 4 — Yielding to the Main Thread
How do you process a massive array of 10 million items without blocking the main thread?
You intentionally chop the work into small chunks and use `setTimeout(..., 0)` to place each chunk at the back of the Macrotask Queue. This gives the browser a chance to render UI updates and process user clicks between chunks!

```javascript
const massiveData = new Array(10000000).fill(1);

function processInChunks(index = 0) {
  // Process 50,000 items at a time
  const chunkSize = 50000;
  for (let i = 0; i < chunkSize && index < massiveData.length; i++) {
    // Math logic...
    massiveData[index]++;
    index++;
  }

  if (index < massiveData.length) {
    // YIELD to the Event Loop! 
    // Put the next chunk at the back of the line!
    setTimeout(() => processInChunks(index), 0);
  } else {
    console.log("Done processing 10M items!");
  }
}
```

## REAL_WORLD
**Node.js vs Browser Event Loop**
The concepts (Call Stack, Micro/Macrotasks) are identical in Node.js, but the underlying C++ libraries are different.
Browsers use the Web APIs (DOM, XMLHTTPRequest). Node.js uses `libuv` to handle file system (`fs`) and network (`http`) operations.
Node.js also introduces a special Microtask: `process.nextTick()`. It is even higher priority than Promises! If you use `process.nextTick()`, it jumps to the absolute very front of the Microtask Queue.

## INTERVIEW
**Q: You write a recursive function that returns a Promise: `function recursive() { return Promise.resolve().then(recursive); }`. You execute it. Will it throw a "Maximum Call Stack Size Exceeded" error (Stack Overflow), or will it freeze the browser tab?**
A: **It freezes the browser tab!**
If you wrote a normal synchronous recursive function (`function rec() { rec(); }`), it would immediately blow up the Call Stack, and JavaScript would throw a Stack Overflow error and terminate the script, saving the browser.
However, with `Promise.resolve().then(recursive)`, you are pushing the recursion into the **Microtask Queue**. 
The Event Loop executes the Promise, pops it off the Stack (Stack size returns to 0), and then immediately queues up another Microtask. 
Because the Event Loop *must* drain the Microtask Queue before doing anything else (like UI rendering), it gets trapped in an infinite loop of executing Microtasks. The Call Stack never overflows (because it empties after every step), but the browser tab is permanently frozen!

## FEYNMAN CHECK

### Explain Event Loop Like I'm 10 Years Old
> The event loop is the SCHEDULER that runs your JavaScript. JS is single-threaded — only ONE thing runs at a time. But asynchronous operations (setTimeout, fetch, click handlers, Promise resolutions) need to run LATER. The runtime keeps queues: MACROTASK QUEUE (setTimeout, setInterval, I/O, UI events) and MICROTASK QUEUE (Promise .then callbacks, queueMicrotask, MutationObserver). The loop is: (1) Run ONE macrotask from the queue. (2) Drain ALL microtasks queued during that macrotask. (3) Browser paints if needed. (4) Go to step 1. The non-obvious power: microtasks always drain BEFORE the next macrotask — that's why `Promise.resolve().then(...)` ALWAYS runs before `setTimeout(..., 0)`.

### 5 Deep Conceptual Questions

**Q1: Why was the event loop designed this way?**
> **A:** JavaScript runs in browsers — it MUST not block UI rendering. Single-threaded execution prevents data races (no locks needed in user code), but requires non-blocking I/O. The event loop is the bridge: native I/O happens in C++ threads underneath, results are posted back to the JS queue, the loop picks them up between synchronous chunks. This let JS scale from "small page scripts" to "Gmail" without redesigning the language.

**Q2: Mental model for queue priority?**
> **A:** "Sync code runs to completion. Then microtasks drain (priority — including chained .then). Then ONE macrotask runs. Then microtasks drain again. Then maybe a paint. Repeat." Microtasks have a footgun: an infinite chain of microtasks STARVES macrotasks AND blocks paint (frozen UI). Browser dev tools warn about long microtask chains.

**Q3: Most dangerous misconception?**
> **A:** `setTimeout(fn, 0)` runs "immediately":
> ```js
> // ❌ Order is NOT (sync, timer, then microtask)
> setTimeout(() => console.log('macro'), 0);
> Promise.resolve().then(() => console.log('micro'));
> console.log('sync');
>
> // Actual output:
> // sync   (sync code first)
> // micro  (microtask queue drained next)
> // macro  (macrotask comes last)
>
> // ✅ Use queueMicrotask for "as soon as possible" without timer overhead
> queueMicrotask(() => console.log('also micro'));
> ```

**Q4: How does requestAnimationFrame fit in?**
> **A:** rAF callbacks fire RIGHT BEFORE the browser paints — typically 60 times per second (every 16.67ms) or matching the display refresh rate. They are scheduled BETWEEN microtask drain and paint. This makes rAF the canonical hook for animations: it runs once per frame, no more (avoiding the over-rendering of setInterval), and is automatically paused for inactive tabs. NEVER do heavy work in rAF — you have ~10ms before the next frame.

**Q5: FAANG-grade definition?**
> **A:** "The JavaScript event loop is the runtime's single-threaded scheduler that processes one macrotask per tick (timers, I/O callbacks, UI events) interleaved with full draining of the microtask queue (Promise reactions, queueMicrotask, MutationObserver) between every macrotask — coordinated with browser paint via requestAnimationFrame and requestIdleCallback — providing non-blocking concurrency without thread-level parallelism."
