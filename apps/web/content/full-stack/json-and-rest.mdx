## WHY
If your frontend (built in JavaScript) wants to ask your backend database (built in Java) for user data, how do they communicate? They don't speak the same language!
They need a universal translator and a standard set of rules.
**JSON (JavaScript Object Notation)** is the universal text format that every programming language on Earth understands. 
**REST (Representational State Transfer)** is the universal architectural rulebook for designing URLs and HTTP methods so that any frontend developer instantly knows exactly how to ask any backend server for data without having to read a 500-page manual.

## THEORY
### What is JSON?
JSON is literally just a giant String (Text) formatted to look exactly like a JavaScript Object. Because it is pure text, it can be sent instantly across the internet via HTTP.
- Keys MUST be wrapped in double quotes `"name"`.
- It cannot contain functions, dates, or logic. Only Strings, Numbers, Booleans, Arrays, and nested Objects.

### What is REST?
REST states that an API should treat data as **Resources** (Nouns, not Verbs).
Instead of having URLs like `/getUserData` or `/deleteUser`, REST forces you to use the standard HTTP Methods (Verbs) applied to a standard Resource URL.
- **GET:** Read data.
- **POST:** Create new data.
- **PUT / PATCH:** Update existing data.
- **DELETE:** Destroy data.

## CODE

### 1. Working with JSON
Every modern language has a built-in JSON parser. In JavaScript, it's the `JSON` object.

```javascript
// A standard JavaScript Object (It exists in RAM)
const userObject = { name: "Alice", age: 25, isAdmin: true };

// 1. Serialization (Stringify)
// Convert the RAM object into a raw Text String so we can send it over the internet!
const jsonText = JSON.stringify(userObject); 
console.log(jsonText); 
// Output: '{"name":"Alice","age":25,"isAdmin":true}'

// 2. Deserialization (Parse)
// We received raw text from a server. Convert it back into a usable JS Object!
const parsedObject = JSON.parse(jsonText);
console.log(parsedObject.name); // Prints: Alice
```

### 2. A Perfect REST API Design
Imagine we are building a blog. Our resource is `articles`. 
A perfect REST API uses the exact same URL (`/articles`) for almost everything, relying entirely on the HTTP Method to determine the action!

| HTTP Method | URL Structure | Action Performed |
| :--- | :--- | :--- |
| **GET** | `https://api.com/articles` | Read a massive array of ALL articles. |
| **GET** | `https://api.com/articles/42` | Read ONLY the specific article with ID 42. |
| **POST** | `https://api.com/articles` | Create a brand new article. (Data sent in the body). |
| **PUT** | `https://api.com/articles/42` | Completely overwrite article 42 with new data. |
| **DELETE** | `https://api.com/articles/42` | Permanently delete article 42. |

### 3. Fetching with POST (Sending JSON)
The default `fetch()` is a GET request. If we want to create data (POST), we must explicitly attach the Method, the Headers, and the JSON Body!

```javascript
async function createNewArticle() {
    const newArticle = {
        title: "Learning REST",
        content: "REST is awesome."
    };

    // We must pass a Configuration Object as the second argument!
    const response = await fetch("https://api.com/articles", {
        method: "POST", // Change the HTTP Verb!
        headers: {
            // CRITICAL: Tell the server we are speaking JSON, not XML or HTML!
            "Content-Type": "application/json" 
        },
        // We cannot send a raw Object! We MUST convert it to a JSON String!
        body: JSON.stringify(newArticle)
    });

    if (response.ok) {
        console.log("Article created successfully!");
    }
}
```

## REAL_WORLD
**REST vs. GraphQL**
REST has been the industry standard for 20 years. However, it has two massive flaws: **Over-fetching** and **Under-fetching**.
If you call `GET /users/1`, a REST API might return the user's name, email, age, address, phone number, and credit card history (a massive 1MB JSON file). What if your frontend only needed their profile picture? You just wasted massive bandwidth (Over-fetching).
Facebook invented **GraphQL** to solve this. Instead of calling specific URLs for specific resources, a GraphQL frontend sends a single complex Query string to one endpoint (e.g., `POST /graphql`). The query literally says: *"Only give me the user's profile picture, nothing else."* The server responds with a tiny 1KB JSON payload matching the exact shape requested!

## INTERVIEW
**Q: You are designing an API. What is the fundamental difference between a `PUT` request and a `PATCH` request when updating a resource?**
A: **Complete Replacement vs. Partial Update.**
- A **PUT** request implies Idempotent Complete Replacement. If an Article has a `title`, `content`, and `author`, and you send a PUT request containing ONLY a new `title`, the server should completely overwrite the existing record. The `content` and `author` fields will be wiped out and set to null! (You must send the *entire* object in a PUT request).
- A **PATCH** request implies a Partial Modification. If you send a PATCH request containing ONLY a new `title`, the server will surgically update the title and leave the existing `content` and `author` completely untouched!

## FEYNMAN CHECK

### Explain JSON & REST Like I'm 10 Years Old
> JSON is the way computers exchange data over the internet — a human-readable text format of nested keys and values (`{"name":"Alice","age":30}`). REST APIs typically send and receive JSON because every language can parse it. The non-obvious part: JSON has only 6 types (string, number, boolean, null, object, array). No dates, no integers vs floats distinction, no bigint, no undefined — which is why `new Date()` and `BigInt` cause silent serialisation bugs in JSON APIs.

---

### 5 Deep Questions
**Q1: What types does JSON support and lack?**
> **A:** Supports: string, number, boolean, null, object, array. Lacks: Date (use ISO strings), BigInt (use strings), undefined (skipped in serialisation), Symbol, function. JSON.stringify silently drops unsupported types — silent data loss.

**Q2: ONE model.**
> **A:** "JSON = subset of JS literals usable as text. Parse converts text→JS value. Stringify converts JS value→text. Both lossy at the type-system boundary."

**Q3: Misconception with code.**
> **A:** ```js
> // ❌ Date silently becomes a string on serialise/parse round-trip
> const obj = { createdAt: new Date() };
> const parsed = JSON.parse(JSON.stringify(obj));
> parsed.createdAt.getTime();  // TypeError: not a Date
> // ✅ Reviver function or schema validator restores types
> JSON.parse(text, (key, value) =>
>   key === 'createdAt' ? new Date(value) : value);
> ```

**Q4: How do you handle BigInt or precise decimals in JSON APIs?**
> **A:** JSON numbers are 64-bit floats — precision degrades above 2^53. For BigInt or currency, serialise as strings (`"amount": "1000000000000.50"`) and parse with BigInt() or Decimal libraries on receipt. Many financial APIs use strings for all monetary values.

**Q5: Senior one-liner.**
> **A:** "JSON is a text-based subset of JS object literals with 6 primitive types — its simplicity drives universal adoption but its lossy type mapping forces production APIs to use string-encoded dates, decimals, and BigInts to survive round-trips."

## BUILD

### 🏗️ Mini Project: Type-Safe JSON Roundtrip with Zod
**Time:** 25 minutes

#### Step 1 — Setup
```bash
mkdir json-zod && cd json-zod
npm init -y && npm i zod
touch index.js
```

#### Step 2 — Core
```js
const { z } = require('zod');
const UserSchema = z.object({
  id: z.number().int(),
  name: z.string().min(1),
  email: z.string().email(),
  createdAt: z.string().datetime().transform(s => new Date(s)),  // ISO → Date
  balance: z.string().regex(/^\d+(\.\d+)?$/),  // BigInt-safe currency
});

const wireData = JSON.parse(JSON.stringify({
  id: 1, name: 'Alice', email: 'a@b.co',
  createdAt: new Date().toISOString(), balance: '1000000.50'
}));

const user = UserSchema.parse(wireData);
console.log(user.createdAt instanceof Date);  // true
console.log(BigInt(user.balance.split('.')[0])); // 1000000n
```

#### Step 4 — Error Handling
```js
try { UserSchema.parse({ id: 'not-a-number' }); }
catch (e) { console.error(e.issues); }  // [{ path: ['id'], message: 'Expected number' }]
```

#### Step 5 — Test
```bash
node index.js
```

**Expected Output:** `true` (Date restored), BigInt parsed correctly.
