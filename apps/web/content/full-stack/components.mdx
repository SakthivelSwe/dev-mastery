# Function Components and Composition | Level: 2

## Functional Components in React 19

In modern React, components are pure JavaScript functions. They accept a single object argument of properties (props) and return a React element tree (JSX).

### Capitalization Convention
React uses capitalization to distinguish between standard HTML tags and custom components during compilation.

* **Lowercase** (`<button>`, `<div>`): Transpiled as strings (e.g., `_jsx("button", ...)`). The browser renders them as standard HTML elements.
* **Uppercase** (`<Button>`, `<Header>`): Transpiled as direct references to the component function (e.g., `_jsx(Button, ...)`). React then invokes this function during reconciliation.

```typescript
import { ReactNode } from 'react';

// Explicitly typing the Props interface
interface ButtonProps {
  onClick: () => void;
  variant: 'primary' | 'secondary';
  children: ReactNode;
}

// Functional Component with standard TypeScript typing
export const Button = ({ onClick, variant, children }: ButtonProps) => {
  return (
    <button 
      className={`btn btn-${variant}`} 
      onClick={onClick}
    >
      {children}
    </button>
  );
};
```

## The Pure Function Rule

React expects your component functions to be **pure** with respect to rendering. A pure function has two characteristics:
1. **Same Inputs, Same Outputs**: Given the same props, the component must always return the exact same JSX structure.
2. **No Side Effects during Render**: The function body must not modify variables outside its scope, make network requests, write to localStorage, or mutate inputs (props).

```typescript
// VIOLATION: Impure Component
let renderCount = 0; // External state mutation

export const ImpureCounter = () => {
  renderCount++; // Side effect during render!
  return <div>Render count: {renderCount}</div>;
};

// CORRECT: Pure Component
interface PureCounterProps {
  count: number;
}

export const PureCounter = ({ count }: PureCounterProps) => {
  return <div>Count: {count}</div>;
};
```

### Why Purity Matters
React's concurrent features, such as time-slicing and server-side rendering, rely on purity. If a render is interrupted or discarded, React must be able to call your component function multiple times without causing bugs or unexpected changes to the system.

## Component Composition and the `children` Prop

Composition is the practice of building complex components by combining simpler, highly reusable components. This is the primary alternative to inheritance in React.

The most common composition pattern uses the special `children` prop, which passes nested JSX elements directly into a container component.

```typescript
import { ReactNode } from 'react';

interface ModalProps {
  isOpen: boolean;
  onClose: () => void;
  title: string;
  children: ReactNode; // Captures nested JSX
}

export const Modal = ({ isOpen, onClose, title, children }: ModalProps) => {
  if (!isOpen) return null;

  return (
    <div className="modal-backdrop" onClick={onClose}>
      <div className="modal-content" onClick={(e) => e.stopPropagation()}>
        <header className="modal-header">
          <h2>{title}</h2>
          <button onClick={onClose}>&times;</button>
        </header>
        <main className="modal-body">
          {children} {/* Renders children here */}
        </main>
      </div>
    </div>
  );
};
```

### Typing the `children` Prop
In TypeScript, you should use `React.ReactNode` to type the `children` prop.

```typescript
// What is React.ReactNode?
// It is a union type representing anything that can be rendered by React:
type ReactNode = 
  | ReactElement 
  | string 
  | number 
  | Iterable<ReactNode> 
  | ReactPortal 
  | boolean 
  | null 
  | undefined;
```

## When to Split Components

Overly large components are difficult to maintain, test, and optimize. Consider splitting a component when it meets any of the following criteria:

1. **Violates Single Responsibility Principle (SRP)**: A component handles its layout, data fetching, form validation, and complex calculations all at once.
2. **Reusability**: You need to use the same visual element or logical behavior in multiple places.
3. **Performance Isolation**: A fast-changing state (like an input value or mouse position) is causing a massive, slow-rendering component tree to rebuild on every keystroke or pixel movement.

### Performance Isolation Example
```tsx
// BEFORE: Every keystroke in <input> re-renders the expensive <HeavyCharts> component
export const Dashboard = () => {
  const [search, setSearch] = useState('');
  return (
    <div className="dashboard">
      <input value={search} onChange={e => setSearch(e.target.value)} />
      <HeavyCharts query={search} />
    </div>
  );
};

// AFTER: We split the search input into its own component or isolate state
export const SearchInput = ({ onSearch }: { onSearch: (val: string) => void }) => {
  const [value, setValue] = useState('');
  
  const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    setValue(e.target.value);
    onSearch(e.target.value);
  };

  return <input value={value} onChange={handleChange} />;
};
```

## Colocating State

When splitting components, you must decide where state should live. The general rule is to **colocate state**: keep state as close to where it is used as possible.

If two or more components need access to the same state, lift that state up to their **nearest common ancestor**. Do not hoist state to the global level unless absolutely necessary.

```
       [Ancestor Component]  <-- Lift state here (nearest common ancestor)
           /          \
          /            \
  [Sibling A]        [Sibling B]
 (Reads State)     (Updates State)
```

### Code Example: Lifting State Up
```typescript
interface SearchBarProps {
  query: string;
  setQuery: (val: string) => void;
}

export const SearchBar = ({ query, setQuery }: SearchBarProps) => (
  <input value={query} onChange={(e) => setQuery(e.target.value)} />
);

interface ResultsListProps {
  query: string;
}

export const ResultsList = ({ query }: ResultsListProps) => {
  // Filters data based on query
  return <div>Results for: {query}</div>;
};

// Nearest Common Ancestor manages the state
export const SearchManager = () => {
  const [query, setQuery] = useState('');

  return (
    <div className="manager">
      <SearchBar query={query} setQuery={setQuery} />
      <ResultsList query={query} />
    </div>
  );
};
```

## WHY
If you write your entire application in a single `index.js` file, it will quickly reach 10,000 lines of code. It becomes impossible to read, impossible to test, and impossible to collaborate on with other developers.

React enforces a **Component-Based Architecture**. A Component is a completely isolated, reusable block of UI. You break your massive application down into tiny pieces: a `<Navbar>`, a `<Sidebar>`, a `<DataGrid>`, and a `<Footer>`. You build each piece independently, and then you compose them together to build complex pages. If the `<Navbar>` breaks, you only have to debug a 50-line file, not a 10,000-line monolith.

## THEORY
### What is a Component?
In modern React, a Component is literally just a standard JavaScript function that returns JSX. 

There are two strict rules you must follow when creating a Component:
1. **The function name MUST be Capitalized** (e.g., `function UserProfile()`). If you name it `function userProfile()`, React will assume it is a standard HTML tag (like `<div>`) and the app will crash.
2. **It must return JSX** (or `null` if you want it to render nothing).

## CODE

### Level 1 — Beginner (Creating a Component)
You create a new file, define a function, and export it.

```javascript
// Button.jsx

// 1. Capitalized function name!
export default function Button() {
  // 2. Returns JSX!
  return (
    <button className="btn-primary">
      Click Me
    </button>
  );
}
```

### Level 2 — Building Blocks (Composing Components)
Once you've created a component, you can import it into other files and use it as if it were a built-in HTML tag!

```javascript
// App.jsx
import Button from './Button';

export default function App() {
  return (
    <div className="container">
      <h1>Welcome to the Dashboard</h1>
      
      {/* We reuse our Button component three times! */}
      <Button />
      <Button />
      <Button />
    </div>
  );
}
```

### Level 3 — Intermediate (Nesting and Hierarchies)
Components naturally form a Tree structure. You have a Root component (`<App />`), which renders Parent components (`<Header />`), which render Child components (`<NavLinks />`).

```javascript
// Layout.jsx
import Navbar from './Navbar';
import Sidebar from './Sidebar';

export default function Layout() {
  return (
    <div className="layout-grid">
      {/* The Parent Layout renders its children */}
      <Navbar />
      <div className="main-content">
        <Sidebar />
        <main>
          <h2>Main Content Area</h2>
        </main>
      </div>
    </div>
  );
}
```
*Note: In React, data only flows in one direction: DOWN the tree. A Parent can pass data to a Child, but a Child cannot easily pass data back up to a Parent.*

### Level 4 — Advanced (Component Purity)
A critical rule of React is that Components must be **Pure Functions** with respect to their rendering logic.

A Pure Function means: *If you give it the exact same inputs, it must return the exact same output, and it must not mutate any variables outside of its own scope.*

```javascript
// ❌ TERRIBLE: Impure Component!
let guestCount = 0; // A global variable outside the component!

export function ImpureGuestList() {
  // Every time this component renders, it mutates the global variable!
  // If React re-renders this component twice (which happens often), 
  // the guestCount will jump by 2 instead of 1!
  guestCount = guestCount + 1;
  return <h1>Guest #{guestCount}</h1>;
}


// ✅ CORRECT: Pure Component!
export function PureGuestList({ count }) {
  // It relies entirely on data passed IN to it (props).
  // It does not mutate anything. It just renders what it is told to render.
  return <h1>Guest #{count}</h1>;
}
```

## REAL_WORLD
Historically (before 2019), React Components were written as Object-Oriented **Classes** that extended `React.Component`. 

You will still see millions of lines of Class Components in older Enterprise codebases.

```javascript
// LEGACY REACT (Do not write new code like this!)
import React, { Component } from 'react';

export default class Welcome extends Component {
  render() {
    return <h1>Hello, {this.props.name}</h1>;
  }
}
```
The React core team officially recommends that **all new code should be written using Functional Components**. Class components are verbose, confusing for JavaScript engines to optimize, and make sharing logic incredibly difficult. Functions are cleaner, faster, and support Hooks.

## INTERVIEW
**Q: Can a Component return multiple elements?**
A: A Component function can only return one single parent object. If you want to return a `<header>` and a `<footer>` side-by-side without wrapping them in a pointless `<div>`, you must wrap them in a React Fragment: `<></>`.

**Q: What is the difference between an Element and a Component?**
A: A **Component** is the actual JavaScript function you write (e.g., `function Button() { ... }`). An **Element** is the lightweight JavaScript object that React creates when you *use* the component in JSX (e.g., `<Button />`). The Element simply describes to React what you want to see on the screen.

**Q: Why must Component names be capitalized?**
A: It's a strict requirement of the JSX compiler. When Babel sees a lowercase tag like `<div>`, it assumes it is a built-in HTML element and passes it directly to the browser. When it sees a capitalized tag like `<Button />`, it realizes it is a custom React Component and attempts to execute the JavaScript function associated with that name.

## FEYNMAN CHECK

Explain to a 10-year-old:
> A React component is a function that takes props and returns JSX. Every time state or props change, React calls your function again — a "render." The render phase is PURE: no side effects, no DOM access, no random numbers. Side effects go in `useEffect`. `React.StrictMode` deliberately invokes your component twice (in development only) to expose impure renders — if you see things happening twice in dev, this is why.
