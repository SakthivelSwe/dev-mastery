## WHY
In the "Version Control Basics" module, you learned how to commit code to the `main` timeline.
But what happens when you work at a company with 50 developers? If everyone commits directly to the `main` timeline simultaneously, the codebase will constantly break. Someone will push a half-finished feature, breaking the build for the other 49 developers trying to work.
**Git Workflows (Branching and Pull Requests)** solve this. They allow you to create an isolated, parallel universe (a Branch) to build your feature safely. When you finish, you formally request permission (a Pull Request) to merge your universe back into the main timeline, ensuring the main code is never broken and always reviewed by senior engineers!

## THEORY
### Branching (Parallel Universes)
A **Branch** is a cheap, lightweight pointer in Git. When you create a branch, you split off from the `main` timeline. You can make 50 commits, delete files, and completely break the application in your branch. The `main` branch remains 100% unaffected. You can instantly switch between the two branches on your laptop!

### The Pull Request (PR)
A **Pull Request** (or Merge Request in GitLab) is not actually a feature of Git; it is a feature of GitHub.
When your branch is finished, you push it to GitHub and open a PR. A PR is a web page that visually highlights exactly which lines of code you added, deleted, or modified. Your team members review the PR, leave comments, and if they approve, they click a green "Merge" button to fuse your branch back into `main`.

## CODE

Here is the exact daily workflow of an enterprise software engineer.

### 1. Starting a New Feature
You are assigned a Jira ticket to build a new Login page. You must NEVER work on `main`!

```bash
# 1. ALWAYS make sure your main timeline is completely up-to-date first!
git checkout main
git pull origin main

# 2. Create a new branch (-b) and immediately switch to it!
# Standard naming convention: feature/<your-name>/<feature-name>
git checkout -b feature/alice/login-page

# (You are now in a parallel universe. Start coding in VS Code!)
```

### 2. Committing to Your Branch
You finish the login page. You commit your work normally.

```bash
git add .
git commit -m "feat: implement user login page UI"

# 3. Push YOUR specific branch up to GitHub!
# (You must use --set-upstream the very first time you push a new branch)
git push --set-upstream origin feature/alice/login-page
```

### 3. The GitHub Pull Request Phase
You go to `github.com`. You open a Pull Request.
Your Senior Engineer reviews it and leaves a comment: *"Please change the button color to blue."*

```bash
# You go back to VS Code (you are still on your branch).
# You change the CSS to blue. You commit again!
git add .
git commit -m "fix: change login button color to blue based on PR review"

# Because the upstream is already set, you just push!
git push
```
GitHub automatically detects the new push and updates the open Pull Request page! The Senior Engineer approves, clicks "Merge", and your code is now officially live on `main`!

### 4. Cleaning Up
Your branch was successfully merged. It is now useless. Delete it to keep your laptop clean!

```bash
# Switch back to the main timeline
git checkout main

# Download your newly merged code from the internet!
git pull origin main

# Delete your local feature branch (-d for delete)
git branch -d feature/alice/login-page
```

## REAL_WORLD
**Stashing (The "Oops" Command)**
You are halfway through building the Login page. Your code is completely broken and unfinished.
Suddenly, your boss runs in: *"There's a critical typo on the live homepage! Fix it immediately!"*
You need to switch to the `main` branch to fix the typo. But Git blocks you! Git says: *"You have uncommitted broken code. If you switch branches now, you will lose your work!"*
You don't want to make a garbage commit like `git commit -m "broken half finished login"`. 
You use **Git Stash**.
`git stash` takes all your modified, uncommitted files, scoops them up, and hides them in a temporary clipboard. Your working directory instantly becomes clean!
You switch to `main`, fix the typo, commit, and push. 
You switch back to your feature branch and run `git stash pop`. Your broken, unfinished code violently explodes back onto your screen exactly how you left it, and you resume working!

## INTERVIEW
**Q: You created a feature branch 3 weeks ago. You finally finish the feature today. Before you open a Pull Request, you run `git push`. A senior engineer stops you and says: "Wait! The `main` branch has moved forward by 100 commits over the last 3 weeks. You need to 'Rebase' your branch before you push." What does `git rebase` do, and why is it preferred over `git merge` here?**
A: **It rewrites history to create a perfectly linear timeline.**
If you use `git merge main` into your feature branch, Git creates a hideous "Merge Commit". It ties the two histories together in a messy diamond shape. If everyone does this, the Git graph looks like a tangled plate of spaghetti, making it impossible to read the history.
`git rebase main` takes your 3-week-old branch, detaches it from its original starting point in history, fast-forwards the time machine to today, and brutally *re-attaches* your branch to the absolute tip of the modern `main` branch!
It mathematically replays your commits one by one on top of the newest code. This ensures your feature integrates perfectly with all the code your coworkers wrote over the last 3 weeks, and it keeps the Git history in a beautiful, readable, perfectly straight line! (Just remember: Never rebase a branch that *other* people are actively working on, as rewriting history destroys their local timelines!)

## FEYNMAN CHECK

### Explain Git Workflow Like I'm 10 Years Old
> A git workflow is the agreed pattern for how a team uses branches, commits, and PRs. Trunk-based: short-lived branches off main, merge daily. GitFlow: long-lived develop + release branches (complex, mostly dead). GitHub Flow: feature branches + PR + CI + merge to main. The non-obvious part: workflow disagreements waste days; pick one, document it, enforce via CI (branch protection, required reviews). Most modern teams use trunk-based or GitHub Flow with feature flags.

---

### 5 Deep Questions
**Q1: Trunk-based vs GitFlow vs GitHub Flow?**
> **A:** Trunk-based: all to main, feature flags for incomplete work, daily merges. GitFlow: develop branch + release branches + hotfix branches — heavy, suits scheduled releases. GitHub Flow: short-lived feature branches → PR → merge — pragmatic default. Most SaaS teams use GitHub Flow or trunk-based.

**Q2: ONE model.**
> **A:** "Branch = isolated work. PR = review checkpoint. CI = quality gate. Merge to main = ship. Short branches reduce conflicts; long branches accumulate debt."

**Q3: Misconception with code.**
> **A:** ```bash
> # ❌ Long-lived feature branch — 100+ commits, painful merge
> git checkout -b epic/redesign  # branches for 3 months
> # ✅ Short-lived branch + feature flag
> git checkout -b feat/new-button  # 1-3 days, behind FLAG_NEW_BUTTON
> ```

**Q4: What is git rebase vs merge?**
> **A:** Merge: creates merge commit, preserves history. Rebase: rewrites your commits on top of target — linear history. Use rebase for feature branches before merging; merge for shared branches. NEVER rebase shared branches — rewrites history others depend on.

**Q5: Senior one-liner.**
> **A:** "Git workflow is the team's contract for branching, reviewing, and merging — short-lived branches with feature flags beat long-lived branches with hopes, and the workflow's enforcement (CI gates + branch protection) matters more than which workflow you pick."

## BUILD

### 🏗️ Mini Project: GitHub Flow with Conventional Commits + Branch Protection
**Time:** 25 minutes

#### Step 1 — Repo Setup
```bash
git init my-project && cd my-project
git checkout -b main
echo "# Project" > README.md
git add . && git commit -m "chore: initial"
git remote add origin git@github.com:you/my-project.git
git push -u origin main
```

#### Step 2 — Branch Protection (on GitHub.com)
```
Settings → Branches → Add rule → main
- Require PR before merging
- Require status checks (CI)
- Require linear history
- Include administrators
```

#### Step 3 — Feature Branch
```bash
git checkout -b feat/login-button
# work, commit
git commit -m "feat(auth): add login button" -m "Closes #42"
git push -u origin feat/login-button
gh pr create --fill
```

#### Step 4 — Conventional Commits
```
feat: new feature
fix: bug fix
chore: tooling/maintenance
docs: docs only
refactor: code change, no behaviour change
test: tests
```

#### Step 5 — Rebase Before Merge
```bash
git fetch origin
git rebase origin/main
git push --force-with-lease
```

**Expected Output:** Linear commit history on main; every change behind a reviewed PR.
