### 1. Figma Workflow & Token Sync Pipeline
In a modern enterprise design system, Figma acts as the single source of truth for design tokens. Using **Figma Variables** or plugins like **Tokens Studio**, designers define primitive and semantic values.

```
[ Figma Variables ] 
       │
       ▼ (Export via API / Tokens Studio)
[ JSON Token Definitions ]
       │
       ▼ (Style Dictionary Build Step)
┌──────────────────────┼──────────────────────┐
│                      │                      │
▼                      ▼                      ▼
[ CSS custom props ]   [ Tailwind Config ]    [ TypeScript Types ]
```

#### GitHub Actions Automation
A GitHub Action automatically triggers on a Webhook from Figma, runs **Style Dictionary** to transform raw JSON files into multiple platform targets (CSS, JS/TS, iOS, Android), and opens a pull request:

```yaml
# .github/workflows/token-sync.yml
name: Sync Design Tokens
on:
  repository_dispatch:
    types: [figma-token-update]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Setup Node.js
        uses: actions/setup-node@v3
        with:
          node-version: 18
      - run: npm ci
      - name: Build Tokens
        run: npm run build-tokens # Executes Style Dictionary
      - name: Create Pull Request
        uses: peter-evans/create-pull-request@v5
        with:
          commit-message: "chore: update design tokens from Figma"
          branch: tokens/figma-update
```

### 3. Accessibility Specifications (WCAG 2.1 AA)
- **Contrast Ratios**: Semantic color mappings must satisfy a minimum contrast ratio of **4.5:1** for normal text (under 18pt) and **3:1** for large text (18pt and above) or interactive UI components (borders, focus rings).
- **Spatial Focus Indicators**: Interactive elements must feature high-contrast outline tokens (`--color-brand-primary`) with at least a **2px** width and **2px** offset to ensure visible focus states.

## WHY
Imagine a massive enterprise with an iOS app, an Android app, and a React web dashboard. The design team decides to change the brand's primary color from Blue to Purple.
Historically, the iOS developer opens Xcode and changes a Swift variable. The Android developer opens Android Studio and changes an XML file. The Web developer opens VSCode and changes a CSS file. 
Inevitably, one of them types `#8A2BE2` instead of `#7A1CD1`, and the apps no longer match. 

**Design Tokens** are the Single Source of Truth for all design decisions. They abstract design properties (colors, typography, spacing) into platform-agnostic data (JSON). This JSON is then automatically compiled into CSS variables, Swift code, and Android XML, guaranteeing 100% synchronization across all platforms and drastically reducing the time required to rebrand or update a system.

## THEORY
### What is a Design Token?
A Design Token is just a key-value pair stored in a JSON file. It represents the smallest possible design decision.
It replaces hardcoded values (like `#3b82f6` or `16px`) with meaningful, semantic names (like `color.primary.base` or `spacing.large`).

### The Token Hierarchy
Tokens are organized into three distinct tiers:
1. **Global Tokens (Core/Primitive):** The raw ingredients. (e.g., `color.blue.500 = #3b82f6`). These are almost never used directly by engineers.
2. **Semantic Tokens (Alias):** The meaning. They point to Global Tokens based on their intended use. (e.g., `color.brand.primary = {color.blue.500}`).
3. **Component Tokens (Specific):** The specific application. They point to Semantic Tokens. (e.g., `button.background.default = {color.brand.primary}`).

## CODE

### Level 1 — Defining Tokens in JSON
Instead of writing CSS, the design team defines the tokens in a universal JSON format (often adhering to the W3C Design Tokens Community Group specification).

```json
// tokens.json
{
  "color": {
    "blue": {
      "500": { "value": "#3b82f6", "type": "color" },
      "600": { "value": "#2563eb", "type": "color" }
    },
    // The Semantic Layer references the Global Layer!
    "brand": {
      "primary": { "value": "{color.blue.500}", "type": "color" },
      "primaryHover": { "value": "{color.blue.600}", "type": "color" }
    }
  },
  "spacing": {
    "small": { "value": "8px", "type": "dimension" },
    "medium": { "value": "16px", "type": "dimension" }
  }
}
```

### Level 2 — Compiling with Style Dictionary
How do we turn that JSON into usable code? We use a build tool like **Style Dictionary** (created by Amazon).
Style Dictionary ingests the JSON and spits out platform-specific files!

**Output 1: For the Web Developer (`variables.css`)**
```css
:root {
  --color-blue-500: #3b82f6;
  --color-brand-primary: var(--color-blue-500);
  --spacing-medium: 16px;
}
```

**Output 2: For the iOS Developer (`StyleDictionaryColor.swift`)**
```swift
public class StyleDictionaryColor {
    public static let brandPrimary = UIColor(red: 0.231, green: 0.510, blue: 0.965, alpha: 1)
}
```

**Output 3: For the Android Developer (`colors.xml`)**
```xml
<resources>
  <color name="color_brand_primary">#3b82f6</color>
</resources>
```
*The Magic:* If the designer changes the JSON file to Purple, the CI/CD pipeline runs Style Dictionary and instantly updates the CSS, Swift, and XML files automatically!

### Level 3 — Connecting Figma to Code
Writing JSON manually is tedious. Modern workflows sync Figma directly to the codebase!
1. The Designer uses the **Figma Tokens Plugin** (or native Figma Variables) to define the colors and spacing visually within Figma.
2. The Designer clicks "Push to GitHub" inside Figma.
3. The plugin generates the `tokens.json` file and automatically opens a Pull Request in the engineering repository!
4. The engineers review the PR. When merged, Style Dictionary compiles the new tokens, and the entire React application instantly reflects the designer's changes without a single engineer writing any CSS!

### Level 4 — Multi-Brand and Theming
Design Tokens make multi-tenant theming trivial.
Imagine building a "White-Label" application that is sold to two different companies: Acme Corp (Red brand) and Globex (Green brand).

You define ONE base semantic token file:
```json
// base-semantic.json
{
  "button": {
    "primary": { "value": "{color.brand.primary}" }
  }
}
```

Then you define TWO different theme files that override the core values:
```json
// theme-acme.json
{ "color": { "brand": { "primary": { "value": "#FF0000" } } } }

// theme-globex.json
{ "color": { "brand": { "primary": { "value": "#00FF00" } } } }
```

When building the application for Acme, Style Dictionary merges `base-semantic.json` and `theme-acme.json`. The React components just use `var(--button-primary)`, and perfectly adapt to the current brand!

## REAL_WORLD
**Dark Mode as a Token Theme**
The most common use case for theming is Dark Mode.
Instead of writing complex CSS `@media (prefers-color-scheme: dark)` overrides in every single component file, you handle it entirely at the Token level.
You create a `light.json` and a `dark.json` token file. 
- In `light.json`: `"bg.surface": { "value": "#ffffff" }`
- In `dark.json`: `"bg.surface": { "value": "#121212" }`
Style Dictionary outputs two separate CSS files. The web app simply toggles a `data-theme="dark"` attribute on the `<body>` tag to swap which CSS variables are active, instantly switching the entire app to Dark Mode!

## INTERVIEW
**Q: We use Tailwind CSS. Tailwind already has a configuration file (`tailwind.config.js`) where we define our colors and spacing. Do we really need to use Design Tokens and Style Dictionary?**
A: **It depends on your cross-platform strategy.**
If your company *only* builds a web application using React/Tailwind, then NO, you do not need Style Dictionary. The `tailwind.config.js` file essentially acts as your Design Token JSON! 
However, if your company also has a native iOS app and a native Android app, Tailwind cannot help you. Tailwind only generates CSS. In a multi-platform environment, you MUST extract the design decisions out of Tailwind into a central `tokens.json` file. You then use Style Dictionary to generate the Swift code, the XML code, *and* a generated `tailwind.config.js` file for the web team, ensuring all three platforms stay perfectly synchronized.

## FEYNMAN CHECK

### Explain Design Tokens Like I'm 10 Years Old
> Design tokens are named variables that store design decisions — colour, spacing, typography, animation timing. Instead of every developer guessing what shade of blue the brand uses, there's one named token (`color.brand.primary = #3b82f6`) that everyone references. Change the token once, and every button, card, and heading that references it updates automatically. The non-obvious part: tokens exist in two layers — primitive (blue-500) and semantic (buttonBackground) — and confusing them makes theming impossible. This is why "just use the hex code" creates a maintenance catastrophe at scale.

---

### 5 Deep Questions
**Q1: Why two layers (primitive + semantic)?**
> **A:** Primitive tokens name raw values (blue-500, 16px). Semantic tokens assign meaning (textPrimary, spacingMd). Themes swap semantic mappings while leaving primitives intact. Without the separation, dark mode requires touching every component; with it, dark mode is one mapping file.

**Q2: ONE model.**
> **A:** "Token = named design decision. Primitive = what. Semantic = what-for. Components use semantic only."

**Q3: Misconception with code.**
> **A:** Using primitive tokens in components:
> ```css
> /* ❌ Locks to light mode palette */
> .card { background: var(--color-gray-100); }
> /* ✅ Theme-agnostic */
> .card { background: var(--color-surface-secondary); }
> ```

**Q4: How do tokens interact with Figma?**
> **A:** Tokens Plugin for Figma (or Tokens Studio) syncs JSON token files with Figma variables/styles bidirectionally. Design changes in Figma export to JSON, pipeline generates CSS/Swift/etc. Development changes to JSON sync back to Figma. The handoff friction disappears — the token IS the source of truth for both.

**Q5: Senior one-liner.**
> **A:** "A design token is a named, platform-agnostic design decision stored as a structured value — the primitive/semantic split makes the difference between O(1) theming and O(n components) repainting."

## BUILD

### 🏗️ Mini Project: Token Validator CLI

**What you will build:** A Node.js script that reads token JSON and validates: all aliases resolve, all colour tokens have valid hex, all semantic tokens reference primitives only.
**Time estimate:** 25 minutes

#### Step 1 — Setup
```bash
mkdir token-validator && cd token-validator
npm init -y && touch validate.js tokens.json
```

#### Step 2 — Core
```js
const tokens = require('./tokens.json');
const hex = /^#([0-9a-f]{3}|[0-9a-f]{6})$/i;
function flatten(obj, prefix='') {
  return Object.entries(obj).flatMap(([k,v]) => typeof v==='object'&&!v.value ? flatten(v, prefix+k+'.') : [[prefix+k, v]]);
}
const flat = Object.fromEntries(flatten(tokens));
let errors = 0;
for(const [name, token] of Object.entries(flat)) {
  if(token.value?.startsWith('{')) {
    const ref = token.value.slice(1,-1);
    if(!flat[ref]) { console.error(`BROKEN ALIAS: ${name} -> ${ref}`); errors++; }
  }
}
if(errors) process.exit(1); else console.log('All tokens valid.');
```

#### Step 5 — Tests
```bash
node validate.js  # Should output: All tokens valid.
# Add a broken alias to tokens.json:
# "test": { "value": "{color.palette.nonexistent}" }
node validate.js  # Should output: BROKEN ALIAS and exit 1
```

**Expected Output:**
```
All tokens valid.
```
