## WHY
In the "Basics" module, you learned how to pass State down from a Parent component to a Child component using Props.
But what happens in a massive enterprise application with 500 components? 
Imagine the user's `theme` preference (Dark Mode vs Light Mode). The `App` component holds the State, but a tiny `Button` component buried 15 layers deep inside the layout needs to know the theme color.
Passing the `theme` prop manually through 14 intermediate components that don't even care about the theme is a nightmare known as **"Prop Drilling"**.
Advanced Framework concepts (like Context, Redux, and Custom Hooks) solve this by teleporting data globally, managing complex asynchronous side-effects, and optimizing render performance for massive applications!

## THEORY
### Global State Management
Instead of passing data vertically down the tree via Props, Global State creates a "Cloud" of data that hovers *above* your entire application. Any component, no matter how deep in the tree, can reach up into the cloud and subscribe to a piece of data. If the data in the cloud changes, only the specific components listening to that data instantly re-render!

### The Component Lifecycle & Side Effects
A component is born (Mounts), it changes over time (Updates), and it eventually dies (Unmounts).
Sometimes, when a component is born, you need it to perform a "Side Effect" that interacts with the outside world (like fetching data from an API, or setting up a WebSocket connection). You must carefully manage these effects, or you will accidentally create infinite network loops that crash your user's browser!

## CODE

We will explore advanced React concepts using Hooks.

### 1. Global State (React Context API)
The Context API solves Prop Drilling. We create a "Provider" that wraps the app, and a "Consumer" (the `useContext` hook) that accesses the data!

```jsx
import { createContext, useContext, useState } from 'react';

// 1. Create the Cloud!
const ThemeContext = createContext();

export default function App() {
    const [theme, setTheme] = useState('dark');

    return (
        // 2. Wrap the app in the Provider, injecting the data into the cloud!
        <ThemeContext.Provider value={{ theme, setTheme }}>
            <Header />
            <MassiveLayoutContainer /> 
        </ThemeContext.Provider>
    );
}

// 15 layers deep...
function DeeplyNestedButton() {
    // 3. Teleport the data directly into this component!
    const { theme, setTheme } = useContext(ThemeContext);

    return (
        <button 
            style={{ background: theme === 'dark' ? 'black' : 'white' }}
            onClick={() => setTheme(theme === 'dark' ? 'light' : 'dark')}
        >
            Toggle Theme
        </button>
    );
}
```

### 2. Side Effects (The `useEffect` Hook)
`useEffect` allows you to run code at specific moments in a component's lifecycle.

```jsx
import { useState, useEffect } from 'react';

export function UserProfile({ userId }) {
    const [userData, setUserData] = useState(null);

    // useEffect takes a function, and a "Dependency Array"
    useEffect(() => {
        // This code runs when the component mounts!
        console.log("Fetching user data...");
        
        fetch(`https://api.com/users/${userId}`)
            .then(res => res.json())
            .then(data => setUserData(data));

        // Optional Cleanup function (Runs when the component unmounts/dies!)
        return () => {
            console.log("Cleaning up! Closing connections...");
        };

    // THE DEPENDENCY ARRAY: [userId]
    // This tells React: "Only re-run this effect if the 'userId' changes!"
    // If the array is empty [], it ONLY runs once when the component is born.
    // If you FORGET the array entirely, it runs on every single render (causing an infinite loop!)
    }, [userId]); 

    if (!userData) return <p>Loading...</p>;
    return <h1>{userData.name}</h1>;
}
```

### 3. Custom Hooks (Reusability)
If you find yourself writing the exact same `useEffect` API fetching logic in 10 different components, you can extract that logic into your own Custom Hook!

```jsx
// A Custom Hook (Must start with the word 'use')
function useWindowWidth() {
    const [width, setWidth] = useState(window.innerWidth);

    useEffect(() => {
        const handleResize = () => setWidth(window.innerWidth);
        window.addEventListener('resize', handleResize);
        
        // Cleanup the event listener!
        return () => window.removeEventListener('resize', handleResize);
    }, []);

    return width; // Return the reactive data!
}

// Now any component can use this logic instantly!
function ResponsiveComponent() {
    const width = useWindowWidth();
    return <h1>The window is {width}px wide!</h1>;
}
```

## REAL_WORLD
**Redux vs. Zustand (The State Wars)**
While the React Context API is great for small things like Themes, it is terrible for high-frequency data changes (like a fast-paced game or a massive trading dashboard), because updating Context forces *every single component* wrapped inside it to aggressively re-render, causing massive UI lag.
Enterprise applications use highly optimized third-party Global State managers.
For a decade, **Redux** was the king. It enforced a strict, boilerplate-heavy architecture (Actions, Reducers, Dispatchers) that made code incredibly verbose but mathematically predictable. 
Today, the industry has largely shifted away from Redux towards **Zustand** (or Redux Toolkit). Zustand provides the exact same high-performance, lag-free global state management, but accomplishes it in 5 lines of code instead of 50, drastically speeding up development time!

## INTERVIEW
**Q: You write a `useEffect` that calls `setInterval` to update a clock on the screen every 1 second. The clock component is conditionally rendered (it appears and disappears when a user clicks a toggle button). You notice that after toggling the clock on and off 10 times, the browser CPU spikes to 100% and the application crashes. Why?**
A: **A Memory Leak (Missing Cleanup Function).**
When the clock component mounts, the `useEffect` runs and creates a `setInterval` timer in the browser's background memory.
When the user toggles the clock *off*, React violently unmounts and destroys the HTML clock component. 
However, React does *not* automatically destroy the `setInterval` timer running in the browser! 
When the user toggles the clock on and off 10 times, there are now 10 separate timers running simultaneously in the background, all trying to update state on destroyed components!
*The Fix:* You MUST return a Cleanup Function from the `useEffect`. `return () => clearInterval(timerId);`. React guarantees it will execute this cleanup function the exact millisecond before the component dies, safely killing the timer and preventing the memory leak!

## FEYNMAN CHECK

### Explain Advanced React/Angular Like I'm 10 Years Old
> Advanced React/Angular means: managing complex state (Redux, NgRx, Zustand), optimising re-renders (memo, useMemo, OnPush), handling side effects safely (useEffect, RxJS), code-splitting for fast initial loads, and server-side rendering for SEO/performance. The non-obvious part: most "advanced" problems are AVOIDABLE — pick smaller components, lift state correctly, and you rarely need Redux. The optimisations that matter most are bundle size and avoiding unnecessary re-renders, not micro-memoisation.

### 5 Deep Questions
**Q1: When do you need Redux/NgRx vs local state?**
> **A:** Local state for data used by one component. Lift to parent if 2-3 components share it. Use Context/Service for tree-wide data (theme, user). Reach for Redux/NgRx only when: complex state machines, time-travel debugging needed, or 10+ components share rapidly-mutating state. Most apps never need it.

**Q2: ONE model.**
> **A:** "Optimise renders, not memory. Memo only when profiler shows a problem. Code-split routes. SSR for SEO + first paint. State at the lowest common ancestor."

**Q3: Misconception with code.**
> **A:** ```tsx
> // ❌ Premature memoisation — overhead > benefit
> const Item = React.memo(({ name }) => <li>{name}</li>);
> // ✅ Profile first — memo only hot components with stable props
> const Row = React.memo(VirtualisedRow);  // 1000+ instances, prop-stable
> ```

**Q4: How does React Suspense / Angular SSR change rendering?**
> **A:** React Server Components + Suspense: server renders shell instantly, streams components as data loads. Angular SSR: full page rendered on server, hydrated on client. Both cut TTI but add complexity (data fetching boundaries, hydration mismatches).

**Q5: Senior one-liner.**
> **A:** "Advanced React/Angular is the discipline of measuring before optimising — bundle-size, render counts, and data-fetching waterfalls — because most performance wins come from architecture (code-splitting, suspense, lifting state) not micro-optimisation (excessive memo)."

## BUILD

### 🏗️ Mini Project: React App with Suspense + Code-Splitting
**Time:** 35 minutes

#### Step 2 — Core
```tsx
import { lazy, Suspense } from 'react';
const Settings = lazy(() => import('./Settings'));
const Profile  = lazy(() => import('./Profile'));

export default function App() {
  const [tab, setTab] = useState<'home'|'settings'|'profile'>('home');
  return (
    <>
      <nav>
        <button onClick={() => setTab('home')}>Home</button>
        <button onClick={() => setTab('settings')}>Settings</button>
        <button onClick={() => setTab('profile')}>Profile</button>
      </nav>
      <Suspense fallback={<div>Loading…</div>}>
        {tab === 'settings' && <Settings />}
        {tab === 'profile'  && <Profile />}
      </Suspense>
    </>
  );
}
```
