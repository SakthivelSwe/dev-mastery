## WHY
In 2005, if a startup wanted to build a web application, they had to physically purchase $50,000 worth of Dell server racks. They had to rent a climate-controlled warehouse, hire technicians to wire the networking cables, and manually install Linux. If their startup went viral and the servers couldn't handle the traffic, they had to order more physical servers, which took 6 weeks to ship!
**Cloud Computing (AWS, GCP, Azure)** changed the world. Amazon realized they had massive warehouses full of millions of servers. They built software that allows you to rent a tiny fraction of a server by the millisecond using a web dashboard. 
Today, a developer in their bedroom can provision an enterprise-grade supercomputer in Tokyo in 3 seconds, scale it to handle 10 million users on a Friday, and instantly delete it on a Monday, paying only $4 for the exact CPU cycles they used.

## THEORY
### The Big Three Cloud Providers
- **AWS (Amazon Web Services):** The undisputed king of the cloud. They invented the industry. They hold the largest market share and have the most services (and the steepest learning curve).
- **Microsoft Azure:** The runner-up. Incredibly popular for massive corporate enterprises that already use Microsoft infrastructure (Active Directory, Windows, .NET).
- **GCP (Google Cloud Platform):** Famous for having the best Data Analytics and Machine Learning services, and for inventing Kubernetes.

### The Service Models
1. **IaaS (Infrastructure as a Service):** The provider gives you a raw, empty virtual computer (e.g., AWS EC2). You have absolute control. You must install the OS, Node.js, and configure the firewalls manually.
2. **PaaS (Platform as a Service):** The provider handles the OS and the software. You just give them your code (e.g., Vercel, Heroku). They run it automatically. Extremely easy, but you lose fine-grained control.
3. **SaaS (Software as a Service):** A fully finished product (e.g., Gmail, Salesforce). You just log in and use it.

## CODE

You don't configure Cloud infrastructure using standard code; you use the Provider's web dashboard. However, modern engineers use **IaC (Infrastructure as Code)** to automate it!

### Infrastructure as Code (Terraform)
Clicking buttons in the AWS Dashboard is dangerous. If you click the wrong button, you might delete the production database. Furthermore, if you need to duplicate your exact server setup in Europe, you have to manually click 500 buttons again.
**Terraform** allows you to write a declarative text file that describes your servers. You run the file, and Terraform automatically commands AWS to build the architecture!

```hcl
# main.tf (Terraform Configuration)

# 1. Tell Terraform we are using AWS, in the US-East region
provider "aws" {
  region = "us-east-1"
}

# 2. Command AWS to spin up a tiny Virtual Machine (EC2 Instance)
resource "aws_instance" "my_web_server" {
  ami           = "ami-0c55b159cbfafe1f0" # The ID for the Ubuntu Linux OS
  instance_type = "t2.micro"              # A tiny, cheap server ($5/month)

  tags = {
    Name = "Production-NodeJS-Server"
  }
}

# 3. Command AWS to create a managed PostgreSQL Database!
resource "aws_db_instance" "my_database" {
  allocated_storage = 20
  engine            = "postgres"
  instance_class    = "db.t3.micro"
  username          = "admin"
  password          = "SuperSecretPassword123"
}
```
If you run `terraform apply` in your terminal, Terraform will connect to AWS, spin up the server and the database in 2 minutes, and wire them together! If you run `terraform destroy`, it instantly deletes everything!

## REAL_WORLD
**S3 (Simple Storage Service)**
Where do you store user profile pictures?
You should *never* store images on your EC2 web server's hard drive! If the Load Balancer auto-scales and spins up 5 clone servers, the other 4 servers won't have the image! When the servers are deleted, the images are permanently lost.
You must use **AWS S3 (Simple Storage Service)**. S3 is not a computer; it is an infinite, highly-available hard drive in the cloud. When a user uploads a photo, your Node.js server sends it directly to an S3 "Bucket". S3 guarantees 99.999999999% durability. Any of your 5 clone servers can safely read the image from the central S3 bucket using an API!

## INTERVIEW
**Q: Your startup's web application goes viral. You are currently running on a single AWS EC2 server. The CPU hits 100%, and the server crashes. You need to handle 10x more traffic. Should you "Scale Vertically" or "Scale Horizontally", and what is the difference?**
A: **You must Scale Horizontally.**
- **Scaling Vertically (Scaling Up):** You turn off your EC2 server, click a button in AWS to upgrade its hardware from 4 Cores to 64 Cores, and turn it back on. While this is incredibly easy and requires zero code changes, it has a fatal flaw: It hits a physical hardware ceiling. You cannot buy an infinite-core CPU. Furthermore, it represents a Single Point of Failure. If that one supercomputer crashes, your entire app is offline.
- **Scaling Horizontally (Scaling Out):** You keep your servers small and cheap, but you use an Auto-Scaling Group and a Load Balancer to spin up 10 identical *clones* of the server. The Load Balancer evenly distributes the traffic. This provides mathematically infinite scalability, and massive high availability (if 2 servers crash, the other 8 keep the app running!). *However*, your application code MUST be perfectly Stateless (e.g., using JWTs and storing images in S3) to function in a horizontally scaled environment!

## FEYNMAN CHECK

### Explain Cloud Basics Like I'm 10 Years Old
> Cloud is renting computing instead of buying it — pay per minute, scale up when busy, scale down when quiet. Big providers (AWS, GCP, Azure) own millions of servers in dozens of regions and rent slices via APIs. The non-obvious part: the cloud is NOT cheaper than owning — it's more flexible. A steady workload at scale is often cheaper on-prem; spiky or experimental workloads are way cheaper on cloud. This is why mature companies repatriate predictable workloads back to colocated hardware.

### 5 Deep Questions
**Q1: IaaS vs PaaS vs SaaS — what each offers?**
> **A:** IaaS (EC2, GCE) = raw VMs, you manage OS up. PaaS (App Engine, Heroku) = runtime managed, you ship code. SaaS (Gmail, Stripe) = ready application via API. Each removes another layer of operations but reduces customisation.

**Q2: ONE model.**
> **A:** "Cloud = APIs over a global fleet. Region = data centre cluster. Availability Zone = isolated facility within region. Service = managed primitive (compute, storage, DB). Cost = compute + storage + egress + ops."

**Q3: Misconception with code.**
> **A:** Hardcoding region in client code:
> ```js
> // ❌ Single point of failure — region outage = total outage
> const s3 = new S3({ region: 'us-east-1' });
> // ✅ Multi-region with failover
> const s3 = new S3({ region: process.env.PRIMARY_REGION });
> // ... retry against backupRegion on failure
> ```

**Q4: How does cloud egress pricing surprise teams?**
> **A:** Data IN is usually free; data OUT to internet is $0.05-0.15/GB. A misconfigured analytics export sending 10TB/day = $30k/month surprise. Cross-AZ traffic within the same region also charges (~$0.01/GB), so chatty microservices spread across AZs can produce 5-figure bills.

**Q5: Senior one-liner.**
> **A:** "Cloud computing is operational abstraction sold by API where the unit economics favour spiky/experimental workloads and punish steady high-throughput ones — which is why FinOps and right-sizing are now first-class disciplines."

## BUILD

### 🏗️ Mini Project: Provision an EC2 Instance via CLI
**Time:** 25 minutes

#### Step 1 — Setup
```bash
# Install AWS CLI, configure with: aws configure
aws --version
```

#### Step 2 — Launch
```bash
aws ec2 run-instances   --image-id ami-0c55b159cbfafe1f0   --instance-type t3.micro   --key-name my-key   --security-groups default   --tag-specifications 'ResourceType=instance,Tags=[{Key=Name,Value=demo}]'
```

#### Step 4 — Costs Visibility
```bash
aws ce get-cost-and-usage --time-period Start=2026-06-01,End=2026-06-29   --granularity DAILY --metrics UnblendedCost
```

#### Step 5 — Tear Down (avoid surprise bill)
```bash
aws ec2 terminate-instances --instance-ids i-xxxxx
```

**Expected Output:** Instance launched and visible in console; cost visible in billing dashboard.
