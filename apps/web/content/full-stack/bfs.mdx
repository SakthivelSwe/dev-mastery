# Breadth-First Search (BFS)

Breadth-First Search is a fundamental graph traversal algorithm that explores vertices level-by-level, starting from a source node. It systematically explores all nodes at distance $d$ from the source before moving to nodes at distance $d+1$.

## 1. Core Mechanics & Theoretical Foundation

### The Queue as a Level-by-Level Coordinator
The fundamental reason BFS is able to find the shortest path in an unweighted graph is its reliance on a **First-In, First-Out (FIFO) queue**. 

When a node $u$ at distance $d$ from the source is dequeued, all of its unvisited neighbors $v$ are enqueued. Because of the FIFO property, these neighbors are guaranteed to be processed after all nodes at distance $d$, but before any nodes at distance $d+2$. This establishes a strict radial frontier of exploration.

```
   [Source] (Level 0)
      |
   +--+--+
   |     |
[Node] [Node] (Level 1)
   |     |
 +-+-----+--+
 |          |
[Node]    [Node] (Level 2)
```

### The Criticality of the Visited Set
In a graph containing cycles, failing to track visited vertices leads to infinite loops and stack/heap exhaustion.
* **With Visited Tracking**: Every node enters the queue at most once.
* **Without Visited Tracking**: In a cycle $A \to B \to C \to A$, node $A$ will continuously re-enqueue $B$, which enqueues $C$, which enqueues $A$, leading to an out-of-memory error.

### BFS Tree vs. BFS Graph
During a BFS traversal, the edges that lead to the discovery of unvisited nodes form a spanning tree (or forest) called the **BFS Tree**. 
* **Tree Edges**: Edges $(u, v)$ where $v$ was first discovered via $u$.
* **Non-Tree Edges**: Edges that point to already visited nodes. In an undirected graph, these can only connect nodes in the same level or adjacent levels.

### Level Tracking Paradigms
1. **The Two-Queue Trick**: Maintain a `currentLevel` queue and a `nextLevel` queue. Swap them when `currentLevel` is empty.
2. **Level Counter (Size Tracking)**: Capture the queue's size at the beginning of each level loop. Run the inner loop exactly `size` times. This is the industry-standard memory-efficient approach.

```java
int level = 0;
while (!queue.isEmpty()) {
    int levelSize = queue.size(); // Freeze size for current frontier
    for (int i = 0; i < levelSize; i++) {
        Vertex u = queue.poll();
        // Process u and enqueue neighbors
    }
    level++;
}
```

## 2. Mathematical Proofs & Complexity Derivations

### Time Complexity: $O(V + E)$
Let $G = (V, E)$ be a graph.
1. **Vertex Initialization**: Every vertex is added to the queue at most once. This contributes $O(V)$ operations.
2. **Edge Relaxation/Exploration**: When vertex $u$ is dequeued, we iterate through its adjacency list.
   * In a directed graph, the sum of the out-degrees of all vertices is exactly equal to the number of edges: $\sum_{v \in V} \text{out-deg}(v) = E$.
   * In an undirected graph, the sum of all degrees is $2E$: $\sum_{v \in V} \text{deg}(v) = 2E$.
3. Combining these two phases:
$$\text{Total Time} = O(V) + O(\text{Edge Operations}) = O(V + E)$$

### Space Complexity: $O(V)$
In the worst-case scenario, the queue must hold an entire level of the graph.
* Consider a **Complete Binary Tree** with $N$ nodes.
* The leaf level contains $\lceil N/2 \rceil$ nodes.
* When processing the second-to-last level, the queue will reach a size of $O(N)$.
* Thus, the auxiliary space complexity is bounded by $O(V)$ to store the queue and the visited set.

### Shortest Path Guarantee: Unweighted vs. Weighted
* **Unweighted Graphs**: BFS guarantees the shortest path (in terms of edge count) because it explores nodes in increasing order of distance. The first time we encounter a target node, we must have arrived via the shortest path.
* **Weighted Graphs**: BFS fails to find the shortest path. A path with more edges but lower cumulative weight might exist. For weighted graphs, **Dijkstra's Algorithm** (which uses a priority queue based on cumulative cost rather than a FIFO queue) is required.

## 5. Implementation (Java & Python)

### Java 21 Implementation
```java
package dsa.bfs;

import java.util.*;

public final class BreadthFirstSearch {

    public static List<Integer> traverse(Map<Integer, List<Integer>> graph, int source) {
        Objects.requireNonNull(graph, "Graph cannot be null");
        
        List<Integer> traversalOrder = new ArrayList<>();
        Queue<Integer> queue = new ArrayDeque<>();
        Set<Integer> visited = new HashSet<>();

        queue.add(source);
        visited.add(source);

        while (!queue.isEmpty()) {
            int current = queue.poll();
            traversalOrder.add(current);

            List<Integer> neighbors = graph.getOrDefault(current, Collections.emptyList());
            for (int neighbor : neighbors) {
                if (!visited.contains(neighbor)) {
                    visited.add(neighbor);
                    queue.add(neighbor);
                }
            }
        }
        return traversalOrder;
    }
}
```

## 6. Interview Deep Dive

### Probing Questions & Expert Answers

1. **How would you implement BFS on a graph so large that it cannot fit into the memory of a single machine?**
   * *Answer*: Use a distributed graph processing framework like Apache Spark (GraphX) or Pregel. Alternatively, perform a **Distributed BFS** using external storage (like Cassandra or DynamoDB) to keep track of the visited set (using Bloom filters to minimize lookups) and a distributed messaging queue (like Kafka) to manage the frontier.

2. **Can you perform BFS on an infinite graph? How do you prevent out-of-memory errors?**
   * *Answer*: Yes, BFS can run on infinite graphs (e.g., state spaces in game trees). To prevent OOM, we must enforce a depth limit (Iterative Deepening BFS/DFS) or use pruning techniques (A* Search) with a heuristic to prioritize paths rather than exploring blindly.

3. **What is Bidirectional BFS, and when is it useful? Derive its complexity.**
   * *Answer*: Bidirectional BFS runs two simultaneous searches: one forward from the source $S$, and one backward from the target $T$. The search terminates when the frontiers intersect. 
   * *Complexity*: If the branching factor is $b$ and the shortest path distance is $d$:
     * Standard BFS explores $O(b^d)$ nodes.
     * Bidirectional BFS explores $O(b^{d/2} + b^{d/2}) = O(b^{d/2})$ nodes, offering a massive reduction in search space.

## WHY
You have a massive graph of friends on a social network. You want to find out if there is a connection between You and Kevin Bacon.
If you use Depth-First Search (DFS), you might follow a single chain of friends down a rabbit hole of 5,000 people across the world, completely missing the fact that your direct coworker knows Kevin Bacon!
**Breadth-First Search (BFS)** is an algorithm that searches outward in concentric circles (like ripples in a pond). It checks all your direct friends first (Level 1), then checks all of their friends (Level 2). Because it searches level by level, BFS mathematically guarantees that the first time it finds the target, it is the **Shortest Path** (in an unweighted graph)!

## THEORY
### The Queue Structure
DFS uses a Stack (LIFO) to plunge deep. BFS uses a **Queue (FIFO)** to maintain order.
1. Put the starting node in the Queue.
2. Dequeue a node, and check if it's the target.
3. If not, add all of its unseen neighbors to the *back* of the Queue.
4. Repeat until the Queue is empty.

Because new neighbors are added to the back, the Queue physically enforces that all nodes at Level 1 are processed completely before Level 2 is ever touched!

## CODE

### Level 1 — Basic BFS Traversal
Let's traverse a graph using an Adjacency List. The most critical part of BFS is the `visited` array/set. In Trees, you can only move downwards. In Graphs, connections go everywhere, including loops! If A points to B, and B points to A, without a `visited` tracker, your BFS will bounce between them in an infinite loop!

```java
import java.util.*;

public class GraphBFS {
    // Using a HashMap Adjacency List
    HashMap<Integer, List<Integer>> adjList = new HashMap<>();

    public void traverse(int startNode) {
        // 1. The Queue controls the Level-by-Level flow!
        Queue<Integer> queue = new LinkedList<>();
        // 2. The Visited Set prevents infinite loops!
        HashSet<Integer> visited = new HashSet<>();

        // Start the algorithm
        queue.add(startNode);
        visited.add(startNode);

        while (!queue.isEmpty()) {
            // Take the front node out of the queue
            int current = queue.poll();
            System.out.println("Visited Node: " + current);

            // Get all neighbors of the current node
            List<Integer> neighbors = adjList.getOrDefault(current, new ArrayList<>());

            for (int neighbor : neighbors) {
                // If we haven't seen this neighbor before...
                if (!visited.contains(neighbor)) {
                    visited.add(neighbor); // Mark it seen IMMEDIATELY!
                    queue.add(neighbor);   // Put it at the back of the line!
                }
            }
        }
    }
}
```

### Level 2 — Finding the Shortest Path (Tracking Levels)
To find the actual distance (number of edges) from the start node to the target, we must count the "Levels" (ripples).
To do this, we measure the size of the queue at the start of the while loop! That size tells us exactly how many nodes exist on the *current* level.

```java
public int getShortestDistance(int startNode, int targetNode) {
    Queue<Integer> queue = new LinkedList<>();
    HashSet<Integer> visited = new HashSet<>();

    queue.add(startNode);
    visited.add(startNode);
    
    int distance = 0; // Track the "ripples"

    while (!queue.isEmpty()) {
        // MAGIC: Take a snapshot of how many nodes are on the current level!
        int nodesInCurrentLevel = queue.size();

        // Process EXACTLY that many nodes!
        for (int i = 0; i < nodesInCurrentLevel; i++) {
            int current = queue.poll();

            // Did we find it?
            if (current == targetNode) return distance;

            for (int neighbor : adjList.getOrDefault(current, new ArrayList<>())) {
                if (!visited.contains(neighbor)) {
                    visited.add(neighbor);
                    queue.add(neighbor);
                }
            }
        }
        // Once the inner loop finishes, we have fully processed the current level!
        // Increment the distance for the next level!
        distance++; 
    }

    return -1; // Target not found
}
```

### Level 3 — Reconstructing the Path
Knowing the distance is 3 is great, but what if you need turn-by-turn directions? We need the actual path `A -> C -> D -> Target`.
We use a `parentMap`. Every time we add a neighbor to the queue, we record *who* added them!

```java
public List<Integer> reconstructPath(int start, int target) {
    Queue<Integer> queue = new LinkedList<>();
    HashSet<Integer> visited = new HashSet<>();
    // Key: Node -> Value: The parent node that discovered it!
    HashMap<Integer, Integer> parentMap = new HashMap<>();

    queue.add(start);
    visited.add(start);

    while (!queue.isEmpty()) {
        int current = queue.poll();

        if (current == target) {
            // TARGET FOUND! Build the path backwards using the Parent Map!
            List<Integer> path = new ArrayList<>();
            int step = target;
            
            while (step != start) {
                path.add(step);
                step = parentMap.get(step); // Trace backwards!
            }
            path.add(start);
            Collections.reverse(path); // Reverse it so it goes Start -> Target
            return path;
        }

        for (int neighbor : adjList.get(current)) {
            if (!visited.contains(neighbor)) {
                visited.add(neighbor);
                queue.add(neighbor);
                // Record history! "I reached 'neighbor' via 'current'"
                parentMap.put(neighbor, current); 
            }
        }
    }
    return new ArrayList<>(); // Empty list if no path
}
```

## REAL_WORLD
**Multi-Source BFS**
Standard BFS starts from a single node, like a single drop of water creating ripples.
What if a city has 5 Fire Stations, and you need to find the distance from EVERY house in the city to its *closest* Fire Station?
You use **Multi-Source BFS**! 
Instead of putting a single node in the Queue at the start, you push ALL 5 Fire Stations into the Queue simultaneously, and mark all 5 as visited.
When the BFS runs, the ripples expand outward from all 5 stations simultaneously. Whichever ripple touches a house first is mathematically guaranteed to be the closest Fire Station to that house!

## INTERVIEW
**Q: You write a BFS. Inside your loop, you pop a node, and THEN you mark it as `visited`. Your code causes the Queue to grow to a massive size and throws an OutOfMemoryError on LeetCode. A Senior Engineer tells you to mark nodes as `visited` the exact moment you *add* them to the queue, not when you *pop* them. Why does this order matter so much?**
A: **Queue Duplication.**
If you wait to mark a node as `visited` until you *pop* it from the Queue, the node is physically sitting inside the Queue completely unprotected.
Imagine Node A and Node B both connect to Node C. 
Node A processes its neighbors and pushes C into the Queue. 
Node B processes its neighbors. It checks if C is in the `visited` set. Because C hasn't been popped yet, it is not marked visited! Node B pushes a SECOND copy of Node C into the Queue!
In a highly interconnected dense graph, this causes the same node to be pushed into the Queue hundreds of times, exponentially exploding the Queue size and crashing the server.
*The Fix:* Always mark nodes as `visited` the exact millisecond you push them into the Queue. This guarantees a node can only ever enter the Queue exactly once.
