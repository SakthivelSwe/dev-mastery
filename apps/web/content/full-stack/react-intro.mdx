# React — Virtual DOM, Reconciliation, JSX | Level: 2

## The Problem: Imperative DOM Manipulation

In traditional web development, developers manipulate the Document Object Model (DOM) imperatively. Using APIs like `document.getElementById()` or `element.appendChild()`, you must explicitly state *how* to transition the UI from one state to another.

```typescript
// Imperative DOM Manipulation: Fragile and hard to scale
const updateNotificationCount = (count: number) => {
  const badge = document.getElementById('notification-badge');
  if (badge) {
    if (count === 0) {
      badge.style.display = 'none';
    } else {
      badge.style.display = 'inline-block';
      badge.textContent = count > 99 ? '99+' : String(count);
    }
  }
};
```

### Why Imperative DOM Manipulation Scales Poorly
1. **State-UI Desynchronization**: As your application grows, tracking every possible state transition and its corresponding DOM modification becomes unsustainable. A single missed edge case leads to UI bugs (e.g., an empty badge remaining visible).
2. **Performance Bottlenecks**: The DOM is a C++ representation of the document. Modifying DOM nodes directly is computationally expensive. It triggers style recalculations, layout reflows, and repainting. Uncoordinated, consecutive DOM writes force the browser to perform synchronous layout thrashing.

React solves this by introducing a **declarative paradigm**. You describe *what* the UI should look like for a given state, and React handles the complex task of efficiently updating the DOM to match that target state.

## The Virtual DOM (VDOM)

The Virtual DOM is not a browser feature; it is a lightweight, in-memory representation of the real DOM. It consists of nested, plain JavaScript objects that describe the node type, its properties (props), and its children.

Here is a simplified representation of a Virtual DOM node:

```typescript
interface VirtualNode {
  type: string | Function;
  props: {
    className?: string;
    children?: VirtualNode[] | string;
    [key: string]: any;
  };
}

// A Virtual DOM representation of: <div className="container">Hello World</div>
const virtualNode: VirtualNode = {
  type: 'div',
  props: {
    className: 'container',
    children: 'Hello World'
  }
};
```

When state changes, React constructs a new Virtual DOM tree representing the updated UI. It then compares this new tree with the previous Virtual DOM tree to calculate the minimal set of changes required to update the real DOM.

## Reconciliation and the Diffing Algorithm

Comparing two trees of size $n$ has a theoretical complexity of $O(n^3)$ using general-purpose tree diffing algorithms. If React used this naive approach, rendering a tree of 1,000 elements would require 1,000,000,000 comparisons, making it completely unusable.

To achieve $O(n)$ complexity, React's **Reconciliation** algorithm relies on two heuristic assumptions:

### Heuristic 1: Different Element Types Produce Different Trees
If two elements have different types (e.g., changing from `<div>` to `<span>`, or from `<ComponentA>` to `<ComponentB>`), React will tear down the entire old subtree and build the new subtree from scratch.

* **Tear Down**: The old DOM nodes are destroyed. Component instances run their cleanup effects and unmount lifecycles.
* **Build Up**: React inserts the new DOM nodes. Component instances are initialized and mounted. Any state associated with the old subtree is completely discarded.

```tsx
// This transition causes React to destroy <OldCounter /> and mount <NewCounter />
// State inside <OldCounter /> is lost.
{isPremium ? (
  <NewCounter value={count} />
) : (
  <OldCounter value={count} />
)}
```

### Heuristic 2: Same Element Type Updates Attributes and Recurses
When comparing two elements of the same type, React retains the underlying DOM node or component instance. It only updates the changed attributes (props) and then recursively diffs the children.

```tsx
// React updates only the 'className' and 'title' attributes, keeping the DOM node intact
<div className="alert alert-info" title="Initial">Message</div>
<div className="alert alert-danger" title="Updated">Message</div>
```

## Fiber Architecture (React 16+)

Prior to React 16, the reconciler (known as the "Stack Reconciler") processed updates recursively in a single, synchronous execution block. Once rendering started, it could not be interrupted. If the component tree was large, this synchronous execution would block the browser's main thread, causing dropped frames (jank) and unresponsive user input.

**React Fiber** is a complete rewrite of the core reconciliation engine. It introduces a virtual stack frame structure where each element in the tree corresponds to a "Fiber" node.

```typescript
interface FiberNode {
  tag: number;             // Identifies the type of component (Functional, Host, etc.)
  key: null | string;      // Unique identifier
  type: any;               // The function or string associated with this fiber
  stateNode: any;          // Reference to the actual DOM node or component instance
  child: FiberNode | null; // Pointer to the first child
  sibling: FiberNode | null; // Pointer to the next sibling
  return: FiberNode | null;  // Pointer to the parent (return address)
  alternate: FiberNode | null; // Pointer to the double-buffered counterpart
}
```

### Key Mechanics of Fiber
1. **Incremental Rendering (Time Slicing)**: Fiber breaks rendering work into small, discrete units of work (individual Fiber nodes). During idle periods, React processes these nodes. Using browser APIs like `requestIdleCallback` (or React's Scheduler polyfill), React can pause rendering to let the browser handle high-priority tasks (like user input or animation) and then resume rendering where it left off.
2. **Concurrency and Priority**: Fiber assigns priorities to different types of updates (e.g., discrete user input has high priority; data fetching responses have lower priority).
3. **Double Buffering**: Fiber maintains two trees simultaneously: the `current` tree (what is currently visible on screen) and the `workInProgress` tree (what is being computed in memory). Once the `workInProgress` tree is fully computed and verified, React swaps the pointers, rendering the updates to the DOM in a single, synchronous batch.

## JSX: Syntactic Sugar

JSX is not valid JavaScript. It is an XML-like syntax extension that must be transpiled into standard JavaScript before execution.

Historically, JSX transpiled directly to `React.createElement()` calls. In modern React (17+), transpilers (like Babel, SWC, or TypeScript) use the new JSX transform, importing JSX runtime functions automatically.

### Transpilation Example

**Source JSX:**
```tsx
import React from 'react';

interface AlertProps {
  type: 'success' | 'error';
  message: string;
}

export const Alert: React.FC<AlertProps> = ({ type, message }) => {
  return (
    <div className={`alert alert-${type}`}>
      <span className="icon" />
      <p>{message}</p>
    </div>
  );
};
```

**Transpiled Output (Modern JSX Transform):**
```javascript
import { jsx as _jsx, jsxs as _jsxs } from "react/jsx-runtime";

export const Alert = ({ type, message }) => {
  return _jsxs("div", {
    className: `alert alert-${type}`,
    children: [
      _jsx("span", { className: "icon" }),
      _jsx("p", { children: message })
    ]
  });
};
```

## Why the `key` Prop Matters

When recursing over the children of a DOM node, React evaluates both lists of children simultaneously. If you insert an element at the beginning of a list, a naive comparison will fail to recognize that the existing items simply shifted down.

```tsx
// Without Keys:
// React compares <li>First</li> with <li>Zero</li> -> updates content
// React compares <li>Second</li> with <li>First</li> -> updates content
// React appends <li>Second</li> -> highly inefficient
<ul>
  <li>First</li>
  <li>Second</li>
</ul>

<ul>
  <li>Zero</li>
  <li>First</li>
  <li>Second</li>
</ul>
```

### The Role of Keys
Keys act as stable identifiers for elements across renders. When keys are provided, React uses them to match children in the old tree with children in the new tree.

```tsx
// With Keys:
// React matches key="1" and key="2", identifying that they merely shifted position.
// It inserts key="0" at the beginning without re-rendering or modifying the existing elements.
<ul>
  <li key="1">First</li>
  <li key="2">Second</li>
</ul>

<ul>
  <li key="0">Zero</li>
  <li key="1">First</li>
  <li key="2">Second</li>
</ul>
```

### Key Rules
* **Must Be Unique Among Siblings**: Keys do not need to be globally unique, only unique within the immediate parent list.
* **Must Be Stable**: Keys should not be generated on the fly (e.g., `key={Math.random()}`). Unstable keys force React to destroy and recreate the entire DOM subtree on every render, causing severe performance degradation and loss of local UI state (such as input focus or scroll position).

## React DevTools Profiler: Visualizing Render Behaviors

When optimizing reconciliation, you can use the **React DevTools Profiler** to analyze why components render.

```
[ Flame Chart ] [ Ranked Chart ]
───────────────────────────────────────────────────────────────────
▼ ParentComponent (0.8ms)
  ├───▼ ChildList (0.5ms)  <-- Rendered due to props change
  │     ├───■ ListItem [key="0"] (0.1ms)
  │     ├───■ ListItem [key="1"] (0.1ms)
  │     └───■ ListItem [key="2"] (0.1ms)
  └───■ StaticHeader (0.0ms) <-- Did not render (Memoized)
```

In the Flame Chart view, colored bars represent individual components.
- **Gray bars** indicate components that did not re-render (their Virtual DOM subtree was reused completely).
- **Colored bars** (ranging from yellow/warm to blue/cool) indicate components that rendered. The width of the bar represents how long the component took to render. Selecting a component reveals the exact reason for its render (e.g., "Props changed: [key]").
