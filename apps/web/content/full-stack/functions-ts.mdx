# TypeScript Functions — Overloads, Generics, this

Functions are the core building blocks of any JavaScript application. TypeScript adds robust static analysis to parameters, return values, context definitions, and polymorphic signatures.

---

## 1. Parameter Types and Return Type Annotations

Parameters must be annotated unless contextually typed. Return types are strongly recommended for self-documentation and to prevent accidental type leakage.

```typescript
function add(a: number, b: number): number {
  return a + b;
}
```

---

## 2. Optional and Default Parameters

*   **Optional Parameters (`?`)** must always follow required parameters.
*   **Default Parameters** automatically infer their type based on their default value.

```typescript
function greet(greeting: string, name?: string, punctuation = "!"): string {
  if (name) {
    return `${greeting}, ${name}${punctuation}`;
  }
  return `${greeting}${punctuation}`;
}
```

---

## 3. Function Overloads

Function overloads allow you to describe a single runtime function that can accept different sets of arguments. You define multiple **overload signatures** followed by a single **implementation signature**. The implementation signature is not exposed to consumers.

```typescript
// Overload Signatures
function format(input: string): string;
function format(input: number, formatStr: string): string;

// Implementation Signature (Must be compatible with all overloads)
function format(input: string | number, formatStr?: string): string {
  if (typeof input === "number" && formatStr) {
    return `${formatStr}: ${input}`;
  }
  return String(input);
}

const s1 = format("hello"); // Valid
const s2 = format(42, "hex"); // Valid
// const s3 = format(42); 
// Compiler Error: TS2575: No overload expects 1 arguments, but overloads do exist that expect either 1 or 2 arguments.
```

---

## 4. `void` vs. `undefined` Returns

*   If a function return type is annotated as `void`, it can return any value, but that value **must be ignored**.
*   If a function return type is annotated as `undefined`, it must explicitly return `undefined` or use an empty `return;` statement.

```typescript
type VoidFn = () => void;
const run: VoidFn = () => {
  return 42; // Allowed, but the return value is ignored by consumer types
};

function returnUndefined(): undefined {
  // Compiler Error: TS2355: A function whose declared type is neither 'void' nor 'any' must return a value.
  return; 
}
```

---

## 5. Explicit `this` Parameters

In JavaScript, `this` is bound dynamically. TypeScript allows you to declare an explicit type for `this` as the **first parameter** of a function. This parameter is compiled away entirely.

```typescript
interface ElementHandler {
  registerClick(callback: (this: HTMLElement, ev: Event) => void): void;
}

const handler: ElementHandler = {
  registerClick(callback) {
    // Inside callback, 'this' is recognized as HTMLElement
  }
};
```

// --- TypeScript Source (functions.ts) ---
function process(val: string): string;
function process(val: number): string;
function process(this: void, val: string | number): string {

// --- Compiled JavaScript (tsc Output - functions.js) ---
"use strict";

## WHY
Functions are the core building blocks of JavaScript. In vanilla JS, you can call a function with the wrong number of arguments, or pass a string into a function that expects a number. The function will execute, fail silently, and return `NaN` or `undefined`.
TypeScript functions solve this by defining a strict "Contract". You explicitly define what types go IN (Parameters), and what type comes OUT (Return Type). If you violate the contract, the code won't even compile.

## THEORY
### The Function Signature
A Function Signature in TypeScript consists of two parts:
1. **Parameter Types:** Annotating the arguments inside the parenthesis `(name: string)`.
2. **Return Type:** Annotating the result after the parenthesis `(): number`.

If a function does not return anything (it just executes a side effect, like `console.log`), its return type is `void`.

## CODE

### Level 1 — Basic Function Typing
You must annotate the parameters. If you don't, TypeScript implicitly types them as `any`, which defeats the purpose of TypeScript!

```typescript
// Parameter 'price' is a number. 
// The ': string' after the parenthesis dictates what the function returns.
function formatPrice(price: number): string {
    return `$${price.toFixed(2)}`;
}

// ERROR: Expected 1 arguments, but got 0.
formatPrice(); 

// ERROR: Argument of type 'string' is not assignable to parameter of type 'number'.
formatPrice("50"); 

let result = formatPrice(50); // Valid. 'result' is strongly typed as a string!
```

### Level 2 — Optional and Default Parameters
In JavaScript, all parameters are technically optional. In TypeScript, parameters are strictly required unless you explicitly mark them as optional using the `?` operator.

```typescript
// 'greeting' is optional. If not provided, it will be undefined.
function sayHello(name: string, greeting?: string): string {
    if (greeting) {
        return `${greeting}, ${name}!`;
    }
    return `Hello, ${name}!`;
}
sayHello("Alice"); // Valid
sayHello("Alice", "Good morning"); // Valid

// Default Parameters (Type Inference happens automatically!)
// Because we assign "Hello", TypeScript infers that 'greeting' is a string.
function sayHelloDefault(name: string, greeting = "Hello"): string {
    return `${greeting}, ${name}!`;
}
```

### Level 3 — The `void` and `never` Types
What if a function doesn't return anything?

```typescript
// 'void' means the function returns nothing (technically it returns undefined in JS).
function logMessage(message: string): void {
    console.log(message);
    // return "Done"; // ERROR: Type 'string' is not assignable to type 'void'.
}
```

What if a function *never finishes executing*? (e.g., it throws an error and crashes the app, or it has an infinite while loop).
```typescript
// 'never' means this function physically cannot reach the end of its execution block.
function crashApp(errorMessage: string): never {
    throw new Error(errorMessage);
}
```

### Level 4 — Typing Function Variables (Callbacks)
When you pass a function *as an argument* to another function (a callback), you must define the signature of the function you are expecting.
You use the Arrow Function syntax to define a function type: `(arg: Type) => ReturnType`.

```typescript
// The second argument 'onComplete' MUST be a function that takes a string and returns nothing.
function fetchData(url: string, onComplete: (data: string) => void) {
    console.log("Fetching " + url);
    // Simulate network request
    onComplete("Here is your data!");
}

fetchData("/api/users", (data) => {
    // TypeScript knows 'data' is a string automatically!
    console.log(data.toUpperCase()); 
});
```

## REAL_WORLD
**Function Overloads**
In JavaScript, it is common to write a single function that does completely different things depending on what arguments you pass it. 
For example, a `getDate` function:
- If you pass a Timestamp (number), it returns a Date object.
- If you pass a Date object, it returns a Timestamp (number).

How do you type a single function with multiple different signatures? You use **Function Overloads**.

```typescript
// Overload Signature 1: Pass a number, get a Date
function transformDate(timestamp: number): Date;
// Overload Signature 2: Pass a Date, get a number
function transformDate(date: Date): number;

// The Actual Implementation (Uses 'any' or Union types internally)
function transformDate(value: any): any {
    if (typeof value === "number") {
        return new Date(value);
    } else {
        return value.getTime();
    }
}

// TypeScript enforces the Overloads perfectly!
let d = transformDate(1700000000); // d is typed as Date
let n = transformDate(new Date()); // n is typed as number
// transformDate("string"); // ERROR: No overload matches this call.
```

## INTERVIEW
**Q: What is the difference between typing a function's return type as `void` vs `undefined`?**
A: **Intent and Callback Compatibility.**
If you type a function as returning `undefined`, you are strictly enforcing that the function *must* physically execute `return undefined;` (or have no return statement at all).
If you type a return as `void`, it means "I don't care what this function returns, the caller should ignore it."
This is crucial for Callbacks. 
If an API expects a callback `(data: string) => void`, you are allowed to pass an arrow function that returns a value (e.g., `(data) => data.toUpperCase()`). TypeScript allows this because `void` tells the API to simply ignore the returned value. If the API expected `(data: string) => undefined`, passing that same arrow function would trigger a compiler error because it returns a string, not undefined!

## FEYNMAN CHECK

### Explain TS Functions Like I'm 10 Years Old
> A function type captures three things: PARAMETER types, RETURN type, and (sometimes) THIS type. `(x: number, y: number) => number` is a function type. TS supports OVERLOADS — multiple signatures for the same function — when one body handles several call patterns (`document.createElement('div')` returns HTMLDivElement, while `document.createElement('a')` returns HTMLAnchorElement). Variance matters: parameter types are CONTRAVARIANT (a function accepting `Animal` can be used where `Dog` is expected, NOT the other way), and return types are COVARIANT (a function returning `Dog` can be used where `Animal` is expected). The non-obvious detail: with `strictFunctionTypes: false` (default in non-strict), parameter variance becomes bivariant — a major source of subtle bugs.

---

### 5 Deep Conceptual Questions

**Q1: What problem do function overloads solve?**
> **A:** A single function signature can't express "if input is A, output is X; if input is B, output is Y." Overloads let you declare multiple input→output relationships, then write ONE implementation that handles all cases. DOM `createElement`, array `reduce`, and Lodash's `_.get` rely heavily on overloads. The 2024 alternative is to use CONDITIONAL types or function-generic constraints, which are often cleaner but harder to read for juniors.

**Q2: Mental model for parameter variance?**
> **A:** A function is "more permissive" (subtype) when it ACCEPTS MORE inputs and RETURNS LESS. `(a: Animal) => Animal` is a subtype of `(a: Dog) => Mammal` because Animal-accepting is broader than Dog-accepting (any caller passing a Dog is satisfied), and Mammal-returning is narrower than Animal-returning (any caller expecting Animal is satisfied). This is contravariance in parameters + covariance in returns = the Liskov Substitution Principle for functions.

**Q3: Most dangerous misconception? Show with code.**
> **A:** Believing optional parameters are the same as defaults:
> ```ts
> // ❌ Optional: type includes undefined
> function greet(name?: string) { return `Hi, ${name.toUpperCase()}`; }   // 💥 if undefined
>
> // ✅ Default: type does NOT include undefined inside body
> function greet(name: string = 'guest') { return `Hi, ${name.toUpperCase()}`; }
>
> // ❌ Optional with default is still optional at call site, but inside the body it's defined
> function greet(name?: string = 'guest') { /* SyntaxError */ }
> ```

**Q4: How do generic functions infer types from arguments?**
> **A:** TS runs an INFERENCE algorithm: for each generic param, it looks at all places that param is used, gathers candidates from actual arguments, and unifies them. `function pick<K extends keyof T, T>(obj: T, key: K)` — when you call `pick({a:1, b:2}, 'a')`, TS infers T from obj's shape, K from the literal 'a' (narrowed to 'a' because the literal type was inferred). This is why type-parameter ORDER matters and why `extends` clauses act as both constraint and inference hint.

**Q5: FAANG-grade definition?**
> **A:** "A TypeScript function type is a signature `(params) => return` with contravariant parameter assignability and covariant return assignability under strictFunctionTypes — supporting overloads via multiple signatures + single implementation, generic inference via call-site argument unification, and rest/spread typed via tuple types — collectively enabling polymorphic APIs without runtime cost."

#### Step 2 — Typed EventEmitter
```ts
// src/emitter.ts
export type EventMap = Record<string, unknown>;

export class TypedEmitter<E extends EventMap> {
  private listeners = new Map<keyof E, Array<(payload: any) => void>>();

  on<K extends keyof E>(event: K, handler: (payload: E[K]) => void): () => void {
    const list = this.listeners.get(event) ?? [];
    list.push(handler);
    this.listeners.set(event, list);
    // Returns a disposer — call to unsubscribe
    return () => {
      const next = (this.listeners.get(event) ?? []).filter(h => h !== handler);
      this.listeners.set(event, next);
    };
  }

  emit<K extends keyof E>(event: K, payload: E[K]): void {
    for (const h of this.listeners.get(event) ?? []) h(payload);
  }
}
```

#### Step 3 — Define App Events
```ts
// src/main.ts
import { TypedEmitter } from './emitter.js';

interface AppEvents {
  'user:login':  { userId: number; ts: number };
  'user:logout': { userId: number };
  'order:paid':  { orderId: string; amount: number; currency: 'USD' | 'EUR' };
}

const bus = new TypedEmitter<AppEvents>();

// Handler payload type is INFERRED from the event name literal
bus.on('user:login', (p) => {
  console.log(`User ${p.userId} logged in at ${new Date(p.ts).toISOString()}`);
});

bus.on('order:paid', (p) => {
  console.log(`Order ${p.orderId}: ${p.amount} ${p.currency}`);
});

bus.emit('user:login', { userId: 42, ts: Date.now() });
bus.emit('order:paid', { orderId: 'ORD-1', amount: 99, currency: 'USD' });
```

import { TypedEmitter } from './emitter.js';

interface E { ping: { n: number } }

describe('TypedEmitter', () => {
  it('delivers payloads to handlers', () => {
    const bus = new TypedEmitter<E>();
    const spy = vi.fn();
    bus.on('ping', spy);
    bus.emit('ping', { n: 7 });

    const bus = new TypedEmitter<E>();

    const bus = new TypedEmitter<E>();
    const calls: number[] = [];
    bus.on('ping', () => calls.push(1));
    bus.on('ping', () => calls.push(2));
    bus.emit('ping', { n: 0 });
