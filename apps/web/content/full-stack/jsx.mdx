# JSX — Expressions, Fragments, Conditional Rendering | Level: 2

## JSX vs HTML: Key Differences

While JSX resembles HTML, it is a strict syntax extension of JavaScript. Because JSX transpiles to JavaScript objects, its attributes map directly to camelCase properties of DOM elements.

### 1. Attribute Naming Conventions
* **`class` vs `className`**: Since `class` is a reserved keyword in JavaScript, JSX uses `className`.
* **`for` vs `htmlFor`**: Similarly, `for` is a reserved keyword, so labels use `htmlFor` to bind to input IDs.
* **Property naming**: DOM attributes like `tabindex` and `onclick` become camelCased as `tabIndex` and `onClick`.

```tsx
// Correct JSX Attribute Mappings
export const FormInput: React.FC = () => {
  return (
    <div className="form-group">
      <label htmlFor="user-email">Email Address</label>
      <input 
        id="user-email" 
        type="email" 
        tabIndex={1} 
        required 
      />
    </div>
  );
};
```

### 2. The `style` Attribute
In HTML, `style` is a string of CSS rules. In JSX, `style` accepts a JavaScript object containing camelCased CSS property names. This prevents injection attacks (XSS) and allows dynamic style calculations.

```tsx
interface CardProps {
  backgroundColor: string;
  isExpanded: boolean;
}

export const Card: React.FC<CardProps> = ({ backgroundColor, isExpanded }) => {
  const dynamicStyle: React.CSSProperties = {
    backgroundColor,
    maxHeight: isExpanded ? '1000px' : '150px',
    transition: 'max-height 0.3s ease-in-out',
    padding: '1.5rem',
    borderRadius: '8px' // Note camelCase instead of border-radius
  };

  return <div style={dynamicStyle}>Card Content</div>;
};
```

## JSX Expressions

You can embed any valid JavaScript expression inside JSX by wrapping it in curly braces `{}`. An expression is code that evaluates to a value.

```tsx
interface UserProfileProps {
  user: {
    firstName: string;
    lastName: string;
    role: 'admin' | 'user';
  };
}

export const UserProfile: React.FC<UserProfileProps> = ({ user }) => {
  // Evaluates to a string inside JSX expression
  const getFullName = () => `${user.firstName} ${user.lastName}`;

  return (
    <div className="profile">
      <h3>{getFullName()}</h3>
      <p>Role: {user.role.toUpperCase()}</p>
      <p>Session ID: {crypto.randomUUID()}</p>
    </div>
  );
};
```

### What Values Can JSX Render?
* **Rendered directly**: Strings, Numbers, and JSX Elements.
* **Ignored (Render nothing)**: Booleans (`true`/`false`), `null`, `undefined`.
* **Errors**: Plain JavaScript objects cannot be rendered directly as children (e.g., `<div>{{ name: 'React' }}</div>` will throw a runtime error).

## Conditional Rendering Pitfalls

The most common pattern for conditional rendering is the logical AND (`&&`) operator. However, this operator has a dangerous edge case in JavaScript.

### The Zero Bug
The expression `expression && <Component />` returns the evaluated value of the left side if it is falsy. If the left side evaluates to the number `0`, React will render `0` to the screen instead of rendering nothing.

```tsx
interface NotificationBadgeProps {
  unreadMessages: string[];
}

export const NotificationBadge: React.FC<NotificationBadgeProps> = ({ unreadMessages }) => {
  const count = unreadMessages.length;

  return (
    <div className="notification-container">
      {/* BUG: If count is 0, this renders "0" to the DOM! */}
      {count && <span className="badge">{count}</span>}

      {/* FIX 1: Explicit boolean conversion */}
      {!!count && <span className="badge">{count}</span>}

      {/* FIX 2: Comparison operator */}
      {count > 0 && <span className="badge">{count}</span>}

      {/* FIX 3: Ternary operator */}
      {count > 0 ? <span className="badge">{count}</span> : null}
    </div>
  );
};
```

## Lists and the `key` Prop

When rendering lists of elements dynamically using `.map()`, you must provide a unique, stable `key` prop to the outer element returned by the callback.

```typescript
interface LogEntry {
  id: string;
  timestamp: number;
  message: string;
}

interface LogViewerProps {
  logs: LogEntry[];
}
```

```tsx
export const LogViewer: React.FC<LogViewerProps> = ({ logs }) => {
  return (
    <ul className="log-list">
      {logs.map((log) => (
        // Correct: unique, stable ID used as the key
        <li key={log.id} className="log-item">
          <span className="time">{new Date(log.timestamp).toLocaleTimeString()}</span>
          <span className="msg">{log.message}</span>
        </li>
      ))}
    </ul>
  );
};
```

### Why Array Index is an Antipattern for Keys
Using the array index (`key={index}`) is safe *only* if the list is static (it will never be sorted, filtered, inserted into, or deleted from). If the list is dynamic, using the index as a key causes severe UI and state bugs.

If you delete the first item from a list of three items:
1. The item at index `0` is removed from the array.
2. The item that was at index `1` is now at index `0`.
3. The item that was at index `2` is now at index `1`.
4. React compares the keys before and after. It sees keys `0` and `1` still exist, and key `2` is gone.
5. React retains the DOM elements for keys `0` and `1`, updating only their props. If these elements contain local state (like an input element with text entered by the user), that state remains bound to the DOM node at index `0` and `1`. The input value stays in the first position, while the props shift, causing the UI to desynchronize.

## React Fragments

A React component must return a single root element. This is because a JavaScript function can only return a single value (or a single transpiled `React.createElement` object).

If you do not want to introduce unnecessary DOM elements (like wrapping everything in a `<div>` which can break CSS Flexbox or Grid layouts), use a **Fragment**.

### Syntax Variations

```tsx
import React from 'react';

interface ColumnProps {
  label: string;
  value: string | number;
}

export const DataRow: React.FC<ColumnProps> = ({ label, value }) => {
  return (
    // Short syntax: renders nothing to the DOM except its children
    <>
      <td>{label}</td>
      <td>{value}</td>
    </>
  );
};

export const DefinitionList: React.FC<{ items: ColumnProps[] }> = ({ items }) => {
  return (
    <dl>
      {items.map((item) => (
        // Explicit syntax: Required when the fragment itself needs a key
        <React.Fragment key={item.label}>
          <dt>{item.label}</dt>
          <dd>{item.value}</dd>
        </React.Fragment>
      ))}
    </dl>
  );
};
```

## The Modern JSX Transform (React 17+)

In older versions of React, every file containing JSX had to import React explicitly:
```javascript
import React from 'react'; // Required for Babel to call React.createElement
```

With the modern JSX transform introduced in React 17, compilers automatically import the necessary entry points from the React package (`react/jsx-runtime`). 

### Advantages
1. **Performance**: The modern transform can optimize memory allocation by using runtime-specific creation functions that are faster than standard `React.createElement`.
2. **Bundle Size**: It allows for minor improvements in bundle size by eliminating unused `React` imports.
3. **Ergonomics**: You no longer need to import `React` in every single file that returns JSX.

## WHY
Historically, web development strictly separated technologies: HTML for structure, CSS for styling, and JavaScript for logic. However, this separation was based on *technologies*, not *features*. If you built a "Like Button", the logic and the HTML were stored in entirely different files, making the codebase difficult to maintain.

React introduced **JSX (JavaScript XML)**. JSX allows you to write HTML tags directly inside your JavaScript files. This fundamentally shifted the paradigm: instead of putting JavaScript into your HTML (like Angular or Vue templates do), React puts HTML into your JavaScript. This allows you to harness the full, unbridled power of the JavaScript engine to dynamically generate user interfaces.

## THEORY
### What is JSX?
JSX is **Syntactic Sugar**. Browsers cannot understand JSX. If you send a `<div />` inside a `.js` file to Chrome, it will throw a Syntax Error. 

Before the code reaches the browser, a compiler (like Babel or SWC) scans your files and translates the beautiful JSX tags into ugly, raw JavaScript function calls (`React.createElement()`).

### The Golden Rules of JSX
Because JSX is secretly JavaScript under the hood, it has strict rules that standard HTML doesn't:
1. **You must return a single Root Element**: A function cannot return two separate elements at the same level.
2. **Close every tag**: Even self-closing tags like `<img>` must end with a slash: `<img />`.
3. **CamelCase properties**: Because `class` is a reserved keyword in JS, you must use `className`. Event listeners like `onclick` become `onClick`.

## CODE

### Level 1 — Beginner (The Syntax)
```javascript
// A standard React component returning JSX
export default function UserProfile() {
  const username = "Alice";
  const avatarUrl = "https://example.com/alice.jpg";

  // We use curly braces { } to evaluate ANY JavaScript expression!
  return (
    // 1. We use className instead of class!
    <div className="profile-card">
      
      {/* 2. We inject variables using {} */}
      <h1>{username}'s Profile</h1>
      
      {/* 3. We use {} to bind dynamic data to HTML attributes! */}
      {/* 4. Notice the self-closing tag /> */}
      <img src={avatarUrl} alt="User Avatar" />
      
      <button>Follow</button>
    </div>
  );
}
```

### Level 2 — Building Blocks (The Root Element Rule)
If you try to return two sibling elements, the compiler will instantly crash.

```javascript
// ❌ FATAL ERROR: Adjacent JSX elements must be wrapped!
export function BrokenComponent() {
  return (
    <h1>Title</h1>
    <p>Description</p>
  );
}

// ✅ FIX 1: Wrap in a div (creates a useless DOM node)
export function FixedWithDiv() {
  return (
    <div className="wrapper">
      <h1>Title</h1>
      <p>Description</p>
    </div>
  );
}

// ✅ FIX 2: Use React Fragments <></> (The Enterprise Standard)
// Fragments allow you to group elements without adding a useless <div> to the real DOM!
export function FixedWithFragment() {
  return (
    <>
      <h1>Title</h1>
      <p>Description</p>
    </>
  );
}
```

### Level 3 — Intermediate (Executing JS inside JSX)
Inside the curly braces `{}`, you can execute any valid JavaScript *expression* (something that evaluates to a value). You cannot write statements (like `if`, `for`, or `switch`).

```javascript
export default function MathComponent() {
  const price = 50;
  const taxRate = 0.1;

  return (
    <div>
      {/* You can do math! */}
      <p>Total: ${price + (price * taxRate)}</p>

      {/* You can call functions! */}
      <p>Uppercase Name: {"alice".toUpperCase()}</p>

      {/* ❌ YOU CANNOT DO THIS: */}
      {/* <p> { if (price > 10) return "Expensive" } </p> */}
      
      {/* ✅ YOU MUST DO THIS (Ternary Operator): */}
      <p>{ price > 10 ? "Expensive!" : "Cheap!" }</p>
    </div>
  );
}
```

### Level 4 — Advanced (Under the Hood Compilation)
To truly master React, you must understand what JSX compiles into.

```javascript
// What you write in your IDE:
const myElement = (
  <h1 className="title">
    Hello, World!
  </h1>
);

// What the Babel Compiler turns it into before sending it to the browser:
const myElement = React.createElement(
  'h1',                    // 1. The HTML tag (or Component)
  { className: 'title' },  // 2. The props/attributes object
  'Hello, World!'          // 3. The children (text or other elements)
);
```
Understanding this compilation reveals *why* you can't return two sibling elements: A JavaScript function can only `return` one object at a time!

## REAL_WORLD
Because JSX evaluates pure JavaScript, it is completely immune to **XSS (Cross-Site Scripting) Injection Attacks** by default.

If a malicious hacker sets their database username to `<script>alert('HACKED')</script>`, and you render it using `<h1>{user.name}</h1>`, React will intercept the string. It automatically sanitizes and escapes the HTML, safely printing the literal text `<script>...` to the screen instead of executing the malicious code in the browser. 

*(If you ever actually need to render raw HTML from a string, you must use the terrifyingly named `dangerouslySetInnerHTML` attribute, which forces you to acknowledge you are creating a security vulnerability).*

## INTERVIEW
**Q: Can you use a JavaScript `for` loop inside JSX?**
A: No. A `for` loop is a statement, not an expression. It does not return a value. If you need to loop over an array in JSX, you must use the `.map()` method, which is an expression that returns a new array of JSX elements.

**Q: Why do we use `className` instead of `class`?**
A: JSX compiles down into JavaScript objects that map directly to the Browser's DOM API. In the DOM API, the property to change a class is `element.className`, because the word `class` is a reserved keyword in JavaScript (used for creating Object-Oriented classes). 

**Q: Do I need to `import React from 'react'` to use JSX?**
A: Historically, yes. Because JSX compiled into `React.createElement()`, the `React` object had to be in scope, or the app would crash. However, starting in React 17 (released in 2020), a new JSX Transform was introduced. The compiler now automatically injects a hidden import from `react/jsx-runtime`. Therefore, in modern React, you no longer need to import React just to use JSX!

## FEYNMAN CHECK

Explain to a 10-year-old:
> JSX is NOT HTML inside JavaScript — it is syntactic sugar for `React.createElement` calls. `<div className="box">` compiles to `React.createElement('div', {className:'box'})`. Every JSX attribute is a JavaScript expression. This is why you use `className` (not `class`), `htmlFor` (not `for`), and `onClick` (not `onclick`). JSX is purely a build-time transformation — no JSX exists at runtime.
