# JSX — Expressions, Fragments, Conditional Rendering | Level: 2

## JSX vs HTML: Key Differences

While JSX resembles HTML, it is a strict syntax extension of JavaScript. Because JSX transpiles to JavaScript objects, its attributes map directly to camelCase properties of DOM elements.

### 1. Attribute Naming Conventions
* **`class` vs `className`**: Since `class` is a reserved keyword in JavaScript, JSX uses `className`.
* **`for` vs `htmlFor`**: Similarly, `for` is a reserved keyword, so labels use `htmlFor` to bind to input IDs.
* **Property naming**: DOM attributes like `tabindex` and `onclick` become camelCased as `tabIndex` and `onClick`.

```tsx
// Correct JSX Attribute Mappings
export const FormInput: React.FC = () => {
  return (
    <div className="form-group">
      <label htmlFor="user-email">Email Address</label>
      <input 
        id="user-email" 
        type="email" 
        tabIndex={1} 
        required 
      />
    </div>
  );
};
```

### 2. The `style` Attribute
In HTML, `style` is a string of CSS rules. In JSX, `style` accepts a JavaScript object containing camelCased CSS property names. This prevents injection attacks (XSS) and allows dynamic style calculations.

```tsx
interface CardProps {
  backgroundColor: string;
  isExpanded: boolean;
}

export const Card: React.FC<CardProps> = ({ backgroundColor, isExpanded }) => {
  const dynamicStyle: React.CSSProperties = {
    backgroundColor,
    maxHeight: isExpanded ? '1000px' : '150px',
    transition: 'max-height 0.3s ease-in-out',
    padding: '1.5rem',
    borderRadius: '8px' // Note camelCase instead of border-radius
  };

  return <div style={dynamicStyle}>Card Content</div>;
};
```

## JSX Expressions

You can embed any valid JavaScript expression inside JSX by wrapping it in curly braces `{}`. An expression is code that evaluates to a value.

```tsx
interface UserProfileProps {
  user: {
    firstName: string;
    lastName: string;
    role: 'admin' | 'user';
  };
}

export const UserProfile: React.FC<UserProfileProps> = ({ user }) => {
  // Evaluates to a string inside JSX expression
  const getFullName = () => `${user.firstName} ${user.lastName}`;

  return (
    <div className="profile">
      <h3>{getFullName()}</h3>
      <p>Role: {user.role.toUpperCase()}</p>
      <p>Session ID: {crypto.randomUUID()}</p>
    </div>
  );
};
```

### What Values Can JSX Render?
* **Rendered directly**: Strings, Numbers, and JSX Elements.
* **Ignored (Render nothing)**: Booleans (`true`/`false`), `null`, `undefined`.
* **Errors**: Plain JavaScript objects cannot be rendered directly as children (e.g., `<div>{{ name: 'React' }}</div>` will throw a runtime error).

## Conditional Rendering Pitfalls

The most common pattern for conditional rendering is the logical AND (`&&`) operator. However, this operator has a dangerous edge case in JavaScript.

### The Zero Bug
The expression `expression && <Component />` returns the evaluated value of the left side if it is falsy. If the left side evaluates to the number `0`, React will render `0` to the screen instead of rendering nothing.

```tsx
interface NotificationBadgeProps {
  unreadMessages: string[];
}

export const NotificationBadge: React.FC<NotificationBadgeProps> = ({ unreadMessages }) => {
  const count = unreadMessages.length;

  return (
    <div className="notification-container">
      {/* BUG: If count is 0, this renders "0" to the DOM! */}
      {count && <span className="badge">{count}</span>}

      {/* FIX 1: Explicit boolean conversion */}
      {!!count && <span className="badge">{count}</span>}

      {/* FIX 2: Comparison operator */}
      {count > 0 && <span className="badge">{count}</span>}

      {/* FIX 3: Ternary operator */}
      {count > 0 ? <span className="badge">{count}</span> : null}
    </div>
  );
};
```

## Lists and the `key` Prop

When rendering lists of elements dynamically using `.map()`, you must provide a unique, stable `key` prop to the outer element returned by the callback.

```typescript
interface LogEntry {
  id: string;
  timestamp: number;
  message: string;
}

interface LogViewerProps {
  logs: LogEntry[];
}
```

```tsx
export const LogViewer: React.FC<LogViewerProps> = ({ logs }) => {
  return (
    <ul className="log-list">
      {logs.map((log) => (
        // Correct: unique, stable ID used as the key
        <li key={log.id} className="log-item">
          <span className="time">{new Date(log.timestamp).toLocaleTimeString()}</span>
          <span className="msg">{log.message}</span>
        </li>
      ))}
    </ul>
  );
};
```

### Why Array Index is an Antipattern for Keys
Using the array index (`key={index}`) is safe *only* if the list is static (it will never be sorted, filtered, inserted into, or deleted from). If the list is dynamic, using the index as a key causes severe UI and state bugs.

If you delete the first item from a list of three items:
1. The item at index `0` is removed from the array.
2. The item that was at index `1` is now at index `0`.
3. The item that was at index `2` is now at index `1`.
4. React compares the keys before and after. It sees keys `0` and `1` still exist, and key `2` is gone.
5. React retains the DOM elements for keys `0` and `1`, updating only their props. If these elements contain local state (like an input element with text entered by the user), that state remains bound to the DOM node at index `0` and `1`. The input value stays in the first position, while the props shift, causing the UI to desynchronize.

## React Fragments

A React component must return a single root element. This is because a JavaScript function can only return a single value (or a single transpiled `React.createElement` object).

If you do not want to introduce unnecessary DOM elements (like wrapping everything in a `<div>` which can break CSS Flexbox or Grid layouts), use a **Fragment**.

### Syntax Variations

```tsx
import React from 'react';

interface ColumnProps {
  label: string;
  value: string | number;
}

export const DataRow: React.FC<ColumnProps> = ({ label, value }) => {
  return (
    // Short syntax: renders nothing to the DOM except its children
    <>
      <td>{label}</td>
      <td>{value}</td>
    </>
  );
};

export const DefinitionList: React.FC<{ items: ColumnProps[] }> = ({ items }) => {
  return (
    <dl>
      {items.map((item) => (
        // Explicit syntax: Required when the fragment itself needs a key
        <React.Fragment key={item.label}>
          <dt>{item.label}</dt>
          <dd>{item.value}</dd>
        </React.Fragment>
      ))}
    </dl>
  );
};
```

## The Modern JSX Transform (React 17+)

In older versions of React, every file containing JSX had to import React explicitly:
```javascript
import React from 'react'; // Required for Babel to call React.createElement
```

With the modern JSX transform introduced in React 17, compilers automatically import the necessary entry points from the React package (`react/jsx-runtime`). 

### Advantages
1. **Performance**: The modern transform can optimize memory allocation by using runtime-specific creation functions that are faster than standard `React.createElement`.
2. **Bundle Size**: It allows for minor improvements in bundle size by eliminating unused `React` imports.
3. **Ergonomics**: You no longer need to import `React` in every single file that returns JSX.
