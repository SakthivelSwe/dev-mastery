## WHY
You know how to build an Express.js server, and you know how to query a database. Now you must connect them. You need to create URLs (Endpoints) that the frontend can call to get data.
You could create endpoints like this:
- `GET /getAllUsers`
- `POST /makeNewUser`
- `POST /deleteUserNow`
This is completely valid code, but it is a **catastrophe** for team collaboration. If every backend developer names URLs randomly based on how they feel that day, frontend developers will be forced to memorize hundreds of arbitrary endpoints.
**REST (Representational State Transfer)** is the industry-standard architectural style. It enforces a strict mathematical pattern for URLs and HTTP methods. If an API is perfectly RESTful, a frontend developer can intuitively guess exactly what the URLs are without ever looking at the documentation!

## THEORY
### Nouns, Not Verbs
The fundamental rule of REST API URLs: **They must only contain Nouns (Resources), NEVER Verbs (Actions).**
The URL represents *what* you are interacting with (e.g., `/users`, `/articles`).
The HTTP Method represents the *Action* (Verb) you are taking upon that resource.

### The 5 Standard Endpoints (CRUD)
Every major resource in your application (Users, Posts, Products) should be mapped to these 5 standard routes. Notice how the URL `/users` is used multiple times, but the server knows exactly what to do based entirely on the HTTP Verb!

| Action | HTTP Verb | URL Structure | Description |
| :--- | :--- | :--- | :--- |
| **C**reate | `POST` | `/users` | Create a new user. (Sends JSON body). |
| **R**ead (All) | `GET` | `/users` | Fetch an array of all users. |
| **R**ead (One) | `GET` | `/users/:id` | Fetch a single user by their ID. |
| **U**pdate | `PUT`/`PATCH` | `/users/:id` | Update a specific user. |
| **D**elete | `DELETE` | `/users/:id` | Delete a specific user. |

## CODE

### 1. Building a RESTful Controller (Express.js)
Here is a perfectly designed RESTful router for an "Articles" resource.

```javascript
const express = require('express');
const router = express.Router();

// Assume we have a database ORM available
const db = require('./fakeDatabase');

// 1. GET ALL
router.get('/articles', async (req, res) => {
    const articles = await db.article.findMany();
    // 200 OK
    res.status(200).json(articles); 
});

// 2. GET ONE (Uses a URL Parameter ':id')
router.get('/articles/:id', async (req, res) => {
    const articleId = req.params.id; // Extract the ID from the URL!
    const article = await db.article.findById(articleId);
    
    if (!article) {
        // 404 Not Found
        return res.status(404).json({ error: "Article not found" }); 
    }
    res.status(200).json(article);
});

// 3. CREATE (POST)
router.post('/articles', async (req, res) => {
    const newArticleData = req.body;
    const createdArticle = await db.article.create(newArticleData);
    
    // 201 Created (The standard status code for successful creation!)
    res.status(201).json(createdArticle);
});

// 4. DELETE
router.delete('/articles/:id', async (req, res) => {
    await db.article.delete(req.params.id);
    
    // 204 No Content (Deletion was successful, nothing to send back!)
    res.status(204).send(); 
});

module.exports = router;
```

### 2. Nested Resources
How do you design a URL when resources belong to other resources? (e.g., A User has many Articles).
You nest the nouns hierarchically!
- `GET /users/42/articles` (Fetch all articles authored by User 42).
- `POST /users/42/articles` (Create a new article authored by User 42).

*Rule of Thumb:* Never nest deeper than two levels! URLs like `/users/42/articles/15/comments/3` are impossible to maintain. If you need a specific comment, just call the comment directly: `/comments/3`!

### 3. Query Parameters (Filtering and Pagination)
If `/users` returns 10 million users, the server will crash. You must use **Query Parameters** (the text after the `?` in a URL) to filter and paginate the collection!
- `GET /users?role=admin&age=25` (Filtering)
- `GET /users?page=2&limit=50` (Pagination)

```javascript
router.get('/users', async (req, res) => {
    // Extract query parameters from the URL
    const { role, limit } = req.query; 
    
    const users = await db.user.findMany({ 
        where: { role: role },
        take: parseInt(limit) || 100 // Default to 100 if no limit provided
    });
    
    res.json(users);
});
```

## REAL_WORLD
**HTTP Status Codes Matter**
Amateur backend developers will wrap their entire controller in a `try/catch` block, and if an error occurs, they still reply with a `res.status(200).json({ error: "Something broke" })`.
This is a fatal violation of REST. The frontend relies heavily on the HTTP Status Code (metadata attached to the response) to know how to react.
- **200s (Success):** 200 (OK), 201 (Created), 204 (No Content).
- **400s (Client Error - The Frontend messed up):** 400 (Bad Request - missing data), 401 (Unauthorized - not logged in), 403 (Forbidden - logged in, but not an Admin), 404 (Not Found).
- **500s (Server Error - The Backend messed up):** 500 (Internal Server Error - database crashed).

If you send a `200 OK` when an error happens, the frontend browser thinks the request succeeded, caches the error message as valid data, and completely breaks the application UI!

## INTERVIEW
**Q: An interviewer looks at your API design. They see you created an endpoint `POST /login` to authenticate users. They say: "Wait, the fundamental rule of REST is 'Nouns, not Verbs'. 'Login' is a verb! How do you make the login endpoint strictly RESTful?"**
A: **You reframe the action as creating a "Session" resource.**
While `POST /login` is incredibly common and widely accepted in the industry, if you want to be mathematically, pedantically RESTful, you must stop thinking about the *action* of logging in.
When a user authenticates, what physical resource is being generated on the server? A **Session** (or an Authentication Token).
Therefore, the perfectly RESTful endpoint is:
`POST /sessions` (or `POST /tokens`).
You are "Creating" a new "Session" resource for that user! When they log out, you destroy the resource: `DELETE /sessions`. This perfectly aligns with REST principles!
