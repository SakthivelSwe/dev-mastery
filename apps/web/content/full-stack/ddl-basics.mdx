# DDL — CREATE TABLE, Data Types, Constraints | Level: 1

Data Definition Language (DDL) is the SQL subset used to define, alter, and destroy relational database structures. Designing a robust schema requires choosing appropriate data types and physical/logical constraints to prevent data corruption at the storage engine level.

---

### Deep Dive: PostgreSQL Data Types vs Other Dialects

#### 1. Numeric Types
*   `INTEGER` (4 bytes, signed range: $-2^{31}$ to $2^{31}-1$).
*   `BIGINT` (8 bytes, signed range: $-2^{63}$ to $2^{63}-1$). Always use `BIGINT` for primary keys and foreign keys in high-throughput enterprise systems to avoid sequence exhaustion.
*   `NUMERIC(precision, scale)` / `DECIMAL`: Arbitrary-precision exact numeric type. Essential for financial and monetary calculations. *Never use floating-point types (`REAL`, `DOUBLE PRECISION`) for currency*, as they introduce binary rounding errors.

#### 2. Character Types
*   `VARCHAR(n)`: Variable-length character type with a maximum length limit $n$.
*   `TEXT`: Variable-length character string of unlimited length (up to 1GB in PostgreSQL). 
*   *Performance Note:* In PostgreSQL, `VARCHAR(n)` and `TEXT` use the same internal storage format (varlena). There is no performance difference between them. The only difference is that `VARCHAR(n)` checks length constraints, which adds a minimal CPU overhead.

#### 3. Date & Time Types
*   `TIMESTAMP`: Stores date and time *without* timezone awareness.
*   `TIMESTAMPTZ`: Stores date and time with timezone information. Internally, PostgreSQL converts the input time to UTC and stores it as an 8-byte integer. When queried, it converts it back to the client's local timezone. Always use `TIMESTAMPTZ` for transaction logs, creation times, and audit tracks.
*   *MySQL Difference:* MySQL uses `DATETIME` (timezone-naive) and `TIMESTAMP` (timezone-aware but limited to a range of 1970 to 2038).

#### 4. Primary Key Generation Strategies
*   `SERIAL` / `BIGSERIAL`: PostgreSQL-specific pseudo-types that auto-create an underlying integer sequence. 
*   `GENERATED ALWAYS AS IDENTITY`: The SQL standard-compliant replacement for `SERIAL` introduced in PostgreSQL 10. It prevents manual inserts into primary key columns unless explicitly forced.
*   `gen_random_uuid()`: Generates a cryptographically strong UUIDv4. Excellent for distributed databases, as keys can be generated safely client-side without sequence collision risk.

-- Enable the pgcrypto module for UUID generation (if using older PG versions, native in PG 13+)
CREATE EXTENSION IF NOT EXISTS "pgcrypto";

DROP TABLE IF EXISTS account_transactions;
DROP TABLE IF EXISTS user_accounts;

-- 1. Create Parent Table using UUID primary keys
CREATE TABLE user_accounts (
    account_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    username VARCHAR(100) NOT NULL UNIQUE,
    email VARCHAR(255) NOT NULL,
    balance NUMERIC(15, 4) NOT NULL DEFAULT 0.0000,
    is_active BOOLEAN NOT NULL DEFAULT TRUE,
    created_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
    
    -- Table Constraints
    CONSTRAINT chk_username_length CHECK (char_length(username) >= 4),
    CONSTRAINT chk_valid_email CHECK (email LIKE '%@%.%')
);

-- 2. Create Transaction Table with Foreign Key and CHECK constraints
CREATE TABLE account_transactions (
    transaction_id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    account_id UUID NOT NULL,
    amount NUMERIC(15, 4) NOT NULL,
    transaction_type VARCHAR(20) NOT NULL,
    executed_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
    
    CONSTRAINT fk_transaction_account FOREIGN KEY (account_id)
        REFERENCES user_accounts (account_id)
        ON DELETE RESTRICT,
        
    CONSTRAINT chk_transaction_type CHECK (transaction_type IN ('DEPOSIT', 'WITHDRAWAL', 'TRANSFER')),
    CONSTRAINT chk_non_zero_amount CHECK (amount <> 0.0000)
);

## WHY
Before you can insert data into a database, you must build the structure to hold it. **Data Definition Language (DDL)** is the subset of SQL used to create, modify, and destroy the actual tables, databases, and schemas. If a database is a filing cabinet, DDL is the act of buying the cabinet and labeling the drawers, while DML (Data Manipulation Language) is the act of putting papers inside.

## THEORY
### The Core DDL Commands
1. **CREATE:** Build a brand new object (Database, Table, Index, View).
2. **ALTER:** Modify the structure of an existing object (e.g., add a new column to a table).
3. **DROP:** Completely destroy an object and all of the data inside it.
4. **TRUNCATE:** Empty a table completely, but keep the structure intact.

## CODE

### Level 1 — Beginner (CREATE TABLE)
When creating a table, you must specify the exact data type for every single column. This is what makes SQL "strongly typed" compared to NoSQL databases.

```sql
-- Create a simple Users table
CREATE TABLE users (
    id INT PRIMARY KEY,
    username VARCHAR(50) NOT NULL,
    email VARCHAR(100) UNIQUE NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### Level 2 — Building Blocks (ALTER TABLE)
Requirements always change. Six months after launching, the Product Manager asks you to add a "Phone Number" field to the user profiles. You do not drop the table and recreate it! You use `ALTER`.

```sql
-- Add a new column to an existing table
ALTER TABLE users 
ADD COLUMN phone_number VARCHAR(20);

-- Change the data type of an existing column (e.g., make it longer)
ALTER TABLE users 
ALTER COLUMN username TYPE VARCHAR(100);

-- Delete a column completely
ALTER TABLE users 
DROP COLUMN phone_number;
```

### Level 3 — Intermediate (DROP vs TRUNCATE)
Both of these commands destroy data, but they operate very differently under the hood.

```sql
-- TRUNCATE deletes all the rows, but the 'logs' table still exists!
-- You can immediately start inserting new rows into it.
TRUNCATE TABLE logs;

-- DROP completely destroys the 'logs' table.
-- If you try to insert into it, the database will throw an error.
DROP TABLE logs;
```
**Why use TRUNCATE instead of `DELETE FROM logs`?** 
`DELETE` scans the table row by row, deleting each one and writing the deletion to the transaction log (so it can be undone). `TRUNCATE` is a DDL command; it simply drops the physical data file on the hard drive and creates a new empty one. `TRUNCATE` can delete 1 billion rows in a fraction of a millisecond, whereas `DELETE` might take 10 minutes!

### Level 4 — Advanced (Constraints)
A database shouldn't just hold data; it should protect the integrity of the data. You enforce rules using DDL Constraints.

```sql
CREATE TABLE products (
    id SERIAL PRIMARY KEY,
    name VARCHAR(200) NOT NULL,
    price DECIMAL(10, 2) NOT NULL,
    
    -- CHECK Constraint: The database will physically reject 
    -- any INSERT or UPDATE where the price is negative!
    CONSTRAINT positive_price CHECK (price >= 0),
    
    -- FOREIGN KEY Constraint: This ensures that 'category_id'
    -- MUST perfectly match an ID that exists in the 'categories' table!
    category_id INT,
    CONSTRAINT fk_category 
      FOREIGN KEY (category_id) 
      REFERENCES categories(id)
      ON DELETE CASCADE
);
```

## REAL_WORLD
**Database Migrations (Flyway / Liquibase)**
In a professional environment, you never open a SQL console and manually type `ALTER TABLE` on the production database. If you make a typo, you could destroy the database.

Instead, developers use **Migration Tools** (like Flyway or Prisma). You write your DDL statements into versioned text files (e.g., `V1__Create_Users.sql`, `V2__Add_Phone_Column.sql`) and commit them to Git. When the application boots up, the migration tool automatically scans the database, determines which scripts haven't been run yet, and safely executes them in order.

## INTERVIEW
**Q: Can a DDL statement (like `ALTER TABLE`) be rolled back inside a Transaction?**
A: **It depends on the database engine!** 
In **PostgreSQL**, DDL statements are fully transactional. If you run `BEGIN; ALTER TABLE users DROP COLUMN email; ROLLBACK;`, the column will be restored instantly.
In **MySQL** and **Oracle**, DDL statements implicitly force an auto-commit. The moment you type `ALTER TABLE`, the transaction is instantly committed to disk and it **cannot** be rolled back. This is a crucial architectural difference to remember when writing migration scripts!
