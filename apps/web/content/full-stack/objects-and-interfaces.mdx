# Objects, Interfaces, and Type Aliases

TypeScript provides two primary ways to declare structural contracts for objects: **Interfaces** and **Type Aliases**. Understanding when and how to use each is critical to designing robust type systems.

---

## 1. Interfaces vs. Type Aliases

### Key Differences
*   **Interfaces** are open-ended; they support **declaration merging**. If you declare two interfaces with the same name in the same scope, they merge into a single definition.
*   **Type Aliases** are closed; they cannot be changed after definition. They can represent any type, including primitives, unions, intersections, and tuples, whereas interfaces are strictly limited to describing object shapes.

```typescript
// Declaration Merging with Interfaces
interface WindowConfig {
  width: number;
}
interface WindowConfig {
  height: number;
}

// WindowConfig now has both width and height
const config: WindowConfig = { width: 1920, height: 1080 };

// Type Aliases (Cannot be redeclared)
type Point = { x: number; y: number };
// type Point = { z: number }; 
// Compiler Error: TS2300: Duplicate identifier 'Point'.
```

### Use Case Recommendations
*   Use **interfaces** for public APIs, libraries, and class implementations, as they support extensibility via merging and inheritance.
*   Use **type aliases** for unions, intersections, utility types, and complex computed shapes.

---

## 2. Optional and Readonly Properties

*   **`?`**: Marks a property as optional, meaning it can be assigned `undefined` or omitted entirely.
*   **`readonly`**: Prevents a property from being reassigned after initialization.

```typescript
interface DatabaseConfig {
  readonly host: string;
  port?: number; // Optional
}

const config: DatabaseConfig = { host: "localhost" };
// config.host = "127.0.0.1"; 
// Compiler Error: TS2540: Cannot assign to 'host' because it is a read-only property.
```

---

## 3. Index Signatures

Index signatures allow you to define types for objects when you do not know all the property names beforehand, but you do know the types of keys and values.

```typescript
interface Dictionary<T> {
  [key: string]: T;
}

const scores: Dictionary<number> = {
  math: 95,
  science: 90,
  // history: "A" // Compiler Error: TS2322: Type 'string' is not assignable to type 'number'.
};
```

---

## 4. Excess Property Checking

When assigning an object literal directly to a typed variable or passing it as an argument, TypeScript performs **excess property checking**. If the object literal contains properties not present in the target type, the compiler raises an error.

```typescript
interface User {
  name: string;
}

// Direct Assignment (Triggers excess property checking)
const user1: User = {
  name: "Alice",
  age: 30 // Compiler Error: TS2353: Object literal may only specify known properties, and 'age' does not exist in type 'User'.
};

// Indirect Assignment via Variable (Bypasses excess property checking)
const intermediate = { name: "Alice", age: 30 };
const user2: User = intermediate; // Compiles successfully because of structural compatibility
```

## 5. Type Erasure: Interfaces and Type Aliases

### Side-by-Side Comparison

```typescript
// --- TypeScript Source (objects.ts) ---
interface Animal {
  name: string;
}

type Food = {
  calories: number;
};

class Dog implements Animal {
  constructor(public name: string) {}
}

const treat: Food = { calories: 150 };
```

```javascript
// --- Compiled JavaScript (tsc Output - objects.js) ---
"use strict";
class Dog {
    constructor(name) {
        this.name = name;
    }
}
const treat = { calories: 150 };
```
