# DML — INSERT, UPDATE, DELETE, SELECT | Level: 1

Data Manipulation Language (DML) manages the retrieval, insertion, modification, and deletion of data within existing schema relations. Writing efficient DML involves minimizing round-trips to the database and understanding the physical implications of mutations.

---

### Core Mechanics & Advanced Mutating Features

#### 1. PostgreSQL `RETURNING` Clause
Unlike standard SQL, which requires a separate query to fetch auto-generated primary keys or default values after an insert or update, PostgreSQL supports the `RETURNING` clause. This returns the modified rows immediately within the same transaction round-trip, saving significant network overhead.

#### 2. UPDATE From Subquery
Using values from one table to update another can be achieved using the `UPDATE ... FROM` syntax. This allows complex joins directly inside the update statement.

#### 3. DELETE with JOIN (The `USING` Clause)
Standard SQL does not support direct joins in a `DELETE` statement. PostgreSQL provides the `USING` clause to join additional tables for conditional deletion.

#### 4. TRUNCATE vs DELETE: Internal Mechanics
*   **`DELETE`:**
    *   Row-by-row operation.
    *   Generates transaction logs (WAL) for every single deleted record.
    *   Triggers are executed for every deleted row (if defined).
    *   Does not release disk space immediately (it marks tuples as dead, waiting for `VACUUM`).
*   **`TRUNCATE`:**
    *   DDL operation under the hood.
    *   Reclaims disk space by deallocating the entire data file (heap relation) and creating a new empty file.
    *   Bypasses row-level triggers.
    *   Extremely fast for large datasets.

### Database Setup and SQL DML

```sql
DROP TABLE IF EXISTS stock_inventory;
DROP TABLE IF EXISTS products;

CREATE TABLE products (
    product_id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    sku VARCHAR(50) UNIQUE NOT NULL,
    title VARCHAR(150) NOT NULL,
    price NUMERIC(12, 2) NOT NULL,
    discontinued BOOLEAN NOT NULL DEFAULT FALSE
);

CREATE TABLE stock_inventory (
    product_id BIGINT PRIMARY KEY REFERENCES products(product_id) ON DELETE CASCADE,
    quantity_available INT NOT NULL CHECK (quantity_available >= 0),
    warehouse_code VARCHAR(10) NOT NULL
);

-- 1. INSERT with RETURNING
INSERT INTO products (sku, title, price) 
VALUES ('PROD-A-101', 'Enterprise Server Rack', 1200.00)
RETURNING product_id, sku, price;

-- Insert supplementary records
INSERT INTO products (sku, title, price) VALUES 
('PROD-B-202', 'Fiber Optic Cable 10m', 45.00),
('PROD-C-303', 'Managed Switch 24-Port', 350.00);

INSERT INTO stock_inventory (product_id, quantity_available, warehouse_code)
VALUES 
((SELECT product_id FROM products WHERE sku = 'PROD-A-101'), 15, 'WH-EAST'),
((SELECT product_id FROM products WHERE sku = 'PROD-B-202'), 100, 'WH-WEST'),
((SELECT product_id FROM products WHERE sku = 'PROD-C-303'), 0, 'WH-EAST');

-- 2. UPDATE from Subquery / Join
-- Update product price based on inventory status (e.g., discount out of stock items)
UPDATE products p
SET price = p.price * 0.90
FROM stock_inventory i
WHERE p.product_id = i.product_id
AND i.quantity_available = 0
RETURNING p.product_id, p.sku, p.price AS discounted_price;

-- 3. DELETE with USING Clause
-- Delete all discontinued products that have no stock in WH-EAST
DELETE FROM products p
USING stock_inventory i
WHERE p.product_id = i.product_id
AND p.discontinued = TRUE
AND i.warehouse_code = 'WH-EAST';
```

### Spring Data JPA & Hibernate Integration

When executing bulk updates or deletes in Hibernate, the persistence context (L1 Cache) can become out-of-sync because the SQL is executed directly in the database, bypassing the entity manager's lifecycle tracking.

import java.util.Optional;

    // Native query mapping RETURNING clause
    @Query(value = "INSERT INTO products (sku, title, price, discontinued) " +

## WHY
Once your tables are created using DDL, you need a way to put data into them, modify that data, and eventually delete it. **Data Manipulation Language (DML)** consists of the three foundational commands that manipulate rows inside the database: `INSERT`, `UPDATE`, and `DELETE`.

*(Note: `SELECT` is technically considered DQL - Data Query Language, though it is often grouped with DML in casual conversation).*

## THEORY
### The ACID Guarantee
Unlike DDL commands (which alter the physical structure of the database), DML commands operate strictly on the data rows. Because they modify data, DML commands are inherently tied to **Transactions**. If you run an `UPDATE` statement that affects 10,000 rows, and the server loses power exactly at row 5,000, the database guarantees that upon reboot, all 5,000 partial updates will be completely rolled back.

## CODE

### 1. INSERT (Creating Data)
The `INSERT INTO` statement adds new rows to a table. You must specify the table name, the columns you are providing, and the corresponding values.

```sql
-- Insert a single row
INSERT INTO users (username, email, age) 
VALUES ('alice123', 'alice@test.com', 28);

-- Insert multiple rows in a single query (Bulk Insert)
-- This is drastically faster than running 3 separate INSERT statements!
INSERT INTO users (username, email, age) 
VALUES 
    ('bob_smith', 'bob@test.com', 35),
    ('charlie_d', 'charlie@test.com', 22),
    ('diana_w', 'diana@test.com', 41);
```
*Tip: If you omit the column names (e.g., `INSERT INTO users VALUES ('alice', ...)`), you must provide a value for exactly every single column in the exact order they were defined in the table. This is dangerous and should be avoided in production code.*

### 2. UPDATE (Modifying Data)
The `UPDATE` statement changes existing data. **You must almost always use a `WHERE` clause!** If you forget the `WHERE` clause, you will update every single row in the entire table.

```sql
-- Safely update a single user's email by targeting their Primary Key
UPDATE users 
SET email = 'alice_new@test.com' 
WHERE id = 1;

-- Update multiple columns at once
UPDATE users 
SET 
    email = 'bob_new@test.com', 
    age = 36 
WHERE username = 'bob_smith';

-- Increment a value atomically
UPDATE products 
SET stock = stock - 1 
WHERE id = 99;
```

### 3. DELETE (Removing Data)
The `DELETE` statement removes rows. Just like `UPDATE`, **if you forget the `WHERE` clause, you will delete the entire table!**

```sql
-- Delete a specific user
DELETE FROM users 
WHERE id = 5;

-- Delete all users who haven't logged in for a year
DELETE FROM users 
WHERE last_login < '2023-01-01';
```

## REAL_WORLD
**Soft Deletes vs Hard Deletes**
In enterprise applications, running `DELETE FROM users WHERE id = 5` (a **Hard Delete**) is often illegal due to auditing and compliance laws. If a user deletes their account, the business still needs a historical record of their transactions.

Instead, companies use **Soft Deletes**. 
You alter the table to include a boolean column: `is_deleted BOOLEAN DEFAULT FALSE`.
When the user clicks "Delete Account", your application does *not* run a `DELETE` statement. It runs an `UPDATE`:
```sql
UPDATE users SET is_deleted = TRUE WHERE id = 5;
```
Then, you must modify all of your `SELECT` queries across your entire application to always include `WHERE is_deleted = FALSE`.

## INTERVIEW
**Q: I want to insert a user. If the user's email already exists in the database, I want to update their age instead of throwing an error. How do I do this?**
A: This is known as an "Upsert" (Update or Insert). Standard SQL does not have an `UPSERT` keyword, but modern dialects provide specialized syntax for this:
- **PostgreSQL / SQLite:** Uses `ON CONFLICT`
  ```sql
  INSERT INTO users (email, age) VALUES ('alice@test.com', 29)
  ON CONFLICT (email) DO UPDATE SET age = EXCLUDED.age;
  ```
- **MySQL:** Uses `ON DUPLICATE KEY UPDATE`
  ```sql
  INSERT INTO users (email, age) VALUES ('alice@test.com', 29)
  ON DUPLICATE KEY UPDATE age = VALUES(age);
  ```
