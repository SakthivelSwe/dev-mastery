## WHY
To become a Full Stack Developer, you must understand the infrastructure your code runs on. The "Internet" and the "World Wide Web" are not magic—they are physical systems of cables, routers, and agreed-upon protocols. If you don't understand how data travels from a browser to your server, you will struggle to debug network errors, configure domains, or understand system architecture.

## THEORY
### Internet vs. World Wide Web
- **The Internet:** The physical infrastructure. A massive global network of connected computers, servers, and fiber-optic cables running across ocean floors.
- **The Web (WWW):** A service that runs *on top* of the Internet. It is a collection of linked documents (HTML) and resources transmitted via the HTTP protocol. 
*(Other services that run on the Internet include Email (SMTP) and File Transfer (FTP)).*

### The Core Protocols (TCP/IP)
For two computers to talk, they must agree on how to communicate.
- **IP (Internet Protocol):** Gives every computer a unique address (like a physical mailing address). Example: `142.250.190.46` (Google's IP).
- **TCP (Transmission Control Protocol):** The delivery system. It breaks data into tiny "packets", sends them across the network, and ensures they are reassembled perfectly on the other side in the correct order.

## CODE

### Level 1 — Beginner (The Journey of a Web Request)
What actually happens when you type `google.com` into your browser and press Enter?

```text
STEP 1: DNS Lookup (The Phonebook)
Your browser knows the name "google.com", but computers only understand IP addresses.
The browser asks a DNS (Domain Name System) server: "What is the IP for google.com?"
DNS replies: "142.250.190.46".

STEP 2: TCP Handshake (Establishing the Connection)
Your browser sends a message to 142.250.190.46 asking to open a connection.
Browser: "SYN" (Can we talk?)
Server:  "SYN-ACK" (Yes, I hear you, can you hear me?)
Browser: "ACK" (Yes, let's talk!)

STEP 3: The HTTP Request
The browser sends an HTTP request over the established TCP connection:
"GET / HTTP/1.1" (Please send me the home page).

STEP 4: The Server Response
The Google server processes the request and sends back the HTML code for the webpage.

STEP 5: Rendering
The browser receives the HTML, parses it, and paints the pixels on your screen.
```

### Level 2 — Building Blocks (Packets and Routers)
Data doesn't travel as one giant file. A 1MB image is broken into roughly 700 small chunks called **Packets**.

```text
Each Packet contains:
- Header: Source IP, Destination IP, and Packet Number (e.g., Packet 4 of 700).
- Payload: The actual piece of data (a chunk of the image).

How do they get there? ROUTERS.
Your home router sends the packet to your ISP's (Internet Service Provider) router. 
That router looks at the Destination IP and forwards it to the next closest router. 
The packets bounce through 10-20 different routers across the country until they 
reach the destination.

Because of network congestion, Packet 5 might take a completely different physical 
path than Packet 6, and arrive out of order! 
TCP on the receiving end waits for all 700 packets and reassembles them perfectly.
```

### Level 3 — Intermediate (IP Addresses - IPv4 vs IPv6)
```text
IPv4 (The Old System)
Format: 192.168.1.1 (Four numbers between 0-255).
Total possible addresses: 4.3 billion.
Problem: We ran out of IPv4 addresses years ago! There are more phones than addresses.

IPv6 (The New System)
Format: 2001:0db8:85a3:0000:0000:8a2e:0370:7334 (Hexadecimal).
Total possible addresses: 340 undecillion (340 followed by 36 zeros).
Enough to assign an IP address to every atom on the surface of the Earth.

NAT (Network Address Translation)
How did we survive running out of IPv4? NAT. 
Your home router has ONE public IP address (facing the internet). 
It creates a private, fake network for your devices (192.168.1.x). 
When your phone makes a request, the router swaps your private IP with the public IP, 
makes the request, gets the response, and forwards it back to your phone. 
The internet never knows your phone's private IP!
```

### Level 4 — Advanced (BGP - Border Gateway Protocol)
If DNS is the phonebook, **BGP is the GPS of the internet.**

```text
The internet is a "Network of Networks". 
Comcast owns a network. AT&T owns a network. AWS owns a network. 
These networks (Autonomous Systems) connect to each other at massive data centers.

BGP is the protocol these networks use to announce to each other: 
"I know the fastest path to Google's servers."

If a fiber-optic cable is cut by an excavator in Chicago, BGP routers instantly 
detect the failure and recalculate a new route through Dallas. 
BGP is what makes the internet decentralized and resilient to nuclear attacks.

(Fun Fact: When Facebook went completely offline globally for 6 hours in 2021, 
it was because an engineer accidentally deleted Facebook's BGP routes. 
The servers were fine, but the internet's GPS forgot how to find them!)
```

## REAL_WORLD
**Fiber Optic Cables vs Satellites**
You might think the internet is beamed through satellites. It's not. 99% of international data is transmitted via physical fiber-optic cables resting on the bottom of the ocean. These cables are about the thickness of a garden hose. Sharks have been known to bite them, so they are wrapped in Kevlar.
Satellite internet (like Starlink) is used primarily for rural areas because it has higher latency (it takes time for light to travel to space and back) compared to a straight line of glass fiber across the ocean.

## INTERVIEW
**Q: What is the difference between TCP and UDP?**
A: **TCP (Transmission Control Protocol)** is reliable. It guarantees delivery and order. If a packet is lost, it asks the sender to resend it. It's used for web browsing, emails, and file transfers where losing a packet corrupts the file.
**UDP (User Datagram Protocol)** is fast but unreliable. It just blasts packets at the destination and doesn't care if they arrive or are out of order. It's used for live video calls (Zoom) and multiplayer gaming. If a frame of video is lost, you don't want the stream to pause to fetch it; you just want the next frame immediately.

## FEYNMAN CHECK

### Explain the Internet Like I'm 10 Years Old
> The internet is a giant network of post offices (routers) that pass tiny envelopes (packets) between buildings (computers). Each envelope has a return address (source IP), a destination address (destination IP), and a tiny piece of the message. Big messages are split across many envelopes and reassembled on arrival. The non-obvious part: no router knows the whole route — each one just picks the next-best post office based on its routing table. This is why a packet can take a different route than the one before it, and why latency varies even between the same two computers.

### 5 Deep Conceptual Questions

**Q1: What problem does packet switching solve that circuit switching cannot?**
> **A:** Circuit switching (old phone networks) reserves an end-to-end path for the duration of a call — wasteful when traffic is bursty. Packet switching breaks data into independent packets that share the network with everyone else, each finding its own route. This means a single fibre link carries millions of conversations simultaneously and survives any individual router failure because packets re-route around it.

**Q2: ONE mental model.**
> **A:** "The internet is layered envelopes: Ethernet wraps IP wraps TCP wraps HTTP. Each layer is opened by a different device — switches read Ethernet, routers read IP, your OS reads TCP, your browser reads HTTP."

**Q3: Most dangerous misconception with code.**
> **A:** Assuming a TCP connection is a wire.
> ```js
> // ❌ Treating TCP as reliable wire — code panics on transient disconnect
> const conn = await connect(host);
> conn.send(payload);  // throws after WiFi drop, app crashes
> // ✅ Retry-with-backoff because TCP can drop any time
> for (let attempt = 0; attempt < 3; attempt++) {
>   try { await sendWithTimeout(payload, 5000); break; }
>   catch (e) { await sleep(2 ** attempt * 1000); }
> }
> ```

**Q4: How do DNS, TCP, TLS, and HTTP interact when you load a webpage?**
> **A:** Browser → DNS lookup (UDP to resolver) → TCP 3-way handshake to the IP → TLS handshake on top of TCP → HTTP request inside TLS → server response → browser renders. Each layer adds round-trips: a cold load to a new domain is typically 4 RTTs before the first byte. HTTP/3 (QUIC) collapses TCP+TLS into one handshake to cut this.

**Q5: One-sentence definition for a senior engineer.**
> **A:** "The internet is a global packet-switched inter-network of autonomous systems connected by BGP, where IP provides best-effort delivery and higher layers (TCP/QUIC/TLS/HTTP) add reliability, encryption, and application semantics — which is why latency and packet loss are properties of the path, not the endpoints."

## BUILD

### 🏗️ Mini Project: Trace the Internet Path to a Website
**What you will build:** A traceroute + DNS lookup + curl chain that visualises every hop your packets take.
**Time estimate:** 20 minutes

---

#### Step 1 — Setup
```bash
mkdir net-trace && cd net-trace
# Windows: install nslookup (built-in), use tracert
# macOS/Linux: dig + traceroute already installed
```

#### Step 2 — DNS Lookup
```bash
# Resolve a domain to IPs (returns A and AAAA records)
nslookup github.com
# Or with dig (Linux/macOS) for full chain
dig +trace github.com
```

#### Step 3 — Trace the Hops
```bash
# Windows
tracert github.com
# Linux/macOS
traceroute github.com
# Each line = one router; latency to each hop in ms
```

#### Step 4 — Error Handling
```bash
# Some routers drop ICMP — you see "* * *" — that's not a failure, just silenced
# Use -T flag (TCP probes) when ICMP is blocked
traceroute -T -p 443 github.com
```

#### Step 5 — Inspect the HTTP Layer
```bash
# Show the full HTTP exchange including DNS time, TCP connect, TLS handshake
curl -w "DNS: %{time_namelookup}s | Connect: %{time_connect}s | TLS: %{time_appconnect}s | TTFB: %{time_starttransfer}s | Total: %{time_total}s\n" -o /dev/null -s https://github.com
```

**Expected Output:**
```
DNS: 0.012s | Connect: 0.045s | TLS: 0.120s | TTFB: 0.250s | Total: 0.310s
```
