## WHY
For over 40 years, Relational (SQL) Databases have been the undisputed bedrock of software engineering. From the banking sector to global airlines to e-commerce, if money is exchanging hands, the data is living in an SQL database.

Why? Because when a server crashes, power is lost, or a hard drive physically breaks midway through a transaction, an SQL database guarantees that your data will not be corrupted, duplicated, or lost. This guarantee is known as the **ACID Properties**, and it is the most important concept in database architecture.

## THEORY
### The ACID Properties
ACID is an acronym describing the 4 theoretical guarantees made by relational databases.

1. **Atomicity (A):** The "All or Nothing" rule. A transaction might consist of 5 separate SQL commands. If the 5th command fails, the database automatically *rolls back* the first 4 commands, acting as if the transaction never happened at all.
2. **Consistency (C):** The database enforces strict rules (Schema, Foreign Keys, Constraints). You cannot insert a row that violates the rules (e.g., trying to set a string "ABC" into an integer column). The database guarantees it will only move from one valid state to another valid state.
3. **Isolation (I):** What happens if User A and User B try to buy the exact same concert ticket at the exact same millisecond? Isolation guarantees that concurrently executing transactions will not interfere with each other. It acts as if they were executed sequentially, one after the other.
4. **Durability (D):** Once the database tells the client "Transaction Successful," the data is permanently saved to non-volatile physical storage (hard drives). If the power cord is immediately unplugged 1 millisecond later, the data is perfectly safe and will be there when the server reboots.

## CODE

### Level 1 — Beginner (Atomicity in Action)
Imagine a bank transfer. Alice sends $100 to Bob. This requires two steps:
1. Deduct $100 from Alice.
2. Add $100 to Bob.

If the power goes out after Step 1, but before Step 2, $100 has vanished from existence! We solve this by wrapping both commands inside a **Transaction Block**.

```sql
-- Start the transaction block
BEGIN;

-- Step 1
UPDATE accounts SET balance = balance - 100 WHERE name = 'Alice';

-- Step 2
UPDATE accounts SET balance = balance + 100 WHERE name = 'Bob';

-- Only if both steps succeed, we commit the changes permanently!
COMMIT;
```
If an error occurs before `COMMIT`, we call `ROLLBACK;`, and the database instantly restores Alice's original balance!

### Level 2 — Building Blocks (Isolation Levels)
Isolation prevents the "Race Condition." However, perfect isolation requires extreme Database Locks (stopping User B until User A finishes), which drastically slows down the database.

Databases allow you to tune the **Isolation Level** depending on how much speed vs accuracy you need.

1. **Read Uncommitted (Lowest):** Extremely fast, but causes "Dirty Reads." You might read a value that another transaction is in the middle of changing, and hasn't committed yet!
2. **Read Committed (Standard):** You will only read data that has been fully committed. This is the default in PostgreSQL.
3. **Repeatable Read:** Guarantees that if you read the same row twice inside your transaction, it will have the exact same value both times, even if someone else changed it in the background.
4. **Serializable (Highest):** Absolute mathematical perfection. The database acts as if every transaction executed in single-file order. Extremely safe, but extremely slow.

### Level 3 — Intermediate (Indexes and B-Trees)
If you have a table with 100 Million users, and you run `SELECT * FROM users WHERE email = 'alice@example.com'`, the database has to scan all 100 million rows one by one. This is called a **Full Table Scan**, and it will take several seconds.

You fix this by creating an **Index**. 
```sql
CREATE INDEX idx_users_email ON users(email);
```

Under the hood, the database creates a separate, highly optimized data structure called a **B-Tree (Balanced Tree)**. The B-Tree stores a sorted list of emails and memory pointers to their physical rows. Because it is a balanced tree, finding an email out of 100 million takes exactly `O(log N)` operations. It finds the email in milliseconds.

### Level 4 — Advanced (The Write Penalty of Indexes)
If Indexes make reading data 10,000x faster, why don't we put an Index on every single column in the table?

Because Indexes incur a massive **Write Penalty**.
When you `INSERT` a new user into the database, you don't just write to the table. You also have to re-sort and update the B-Tree for the `email` index. If you have 5 indexes, one `INSERT` command turns into 6 physical disk writes. If your application is heavily Write-Biased (like IoT sensors), adding too many indexes will cause the database's write speed to grind to a halt.

## REAL_WORLD
One of the most insidious bugs in SQL is the **N+1 Query Problem** often caused by ORMs (Object-Relational Mappers like Prisma or Hibernate). 

If you want to fetch 50 blog posts, and the author of each post:
- An ORM might execute 1 query to fetch the 50 posts.
- It then executes 1 separate query to fetch the author for *each* post!
- Total Queries: 1 (for the posts) + 50 (for the authors) = **51 Queries!**

This destroys database performance. The correct solution is to use a SQL `JOIN` to fetch the posts and the authors in exactly **1 Query**. Understanding how your ORM translates your code into raw SQL is mandatory for senior engineers.

## INTERVIEW
**Q: How does a database guarantee Durability (D) if writing to a hard drive is too slow?**
A: It uses a **WAL (Write-Ahead Log)**. Writing a massive table update to various sectors of a hard drive is slow. Instead, the database first writes a tiny, sequential text entry to an append-only log file (the WAL) saying "I am about to change Alice's balance to 50". Because it is a tiny sequential write, it is incredibly fast. Once the WAL is saved to disk, it tells the user "Success!", and updates the actual table in the background. If the power cuts, upon reboot, the database reads the WAL and reapplies any unfinished changes!

**Q: What is Database Normalization?**
A: It is the process of structuring your tables to reduce data redundancy. Instead of storing the `Company_Name` in every single `User` row (which wastes space and risks typos), you store the `Company_ID`, and put the `Company_Name` in a separate `Companies` table. You then connect them using a Foreign Key. This ensures there is a Single Source of Truth.

**Q: When would you intentionally De-Normalize a database?**
A: When you need extreme Read performance. `JOIN`ing 5 normalized tables together takes heavy CPU power. If a query is too slow, you intentionally "De-Normalize" by duplicating data into a single table to avoid the `JOIN`, trading storage space for read speed.
