## WHY
In a distributed system, you don't run your database on a single machine. You run it on 5 machines (nodes) spread across different data centers to ensure that if one data center burns down, your application stays online.

But having multiple machines introduces a devastating physics problem: **Network Failure**. Networks are notoriously unreliable. Cables get cut, routers crash, and packets drop. What happens when Node A and Node B lose their network connection to each other? 

The **CAP Theorem** (proposed by computer scientist Eric Brewer) states that in the event of a network failure, you are forced to make a terrible choice. You can either keep the system online (but serve potentially incorrect data), or you can turn the system offline to protect the data's accuracy. You cannot have both.

## THEORY
### The Three Pillars (C-A-P)
The CAP theorem states that a distributed system can only provide two of the following three guarantees simultaneously:

1. **Consistency (C):** Every read receives the most recent write, or an error. If Alice changes her password on Node A, and Bob immediately tries to read Alice's password from Node B, Bob MUST see the new password.
2. **Availability (A):** Every request receives a non-error response, without the guarantee that it contains the most recent write. If Node A goes down, Node B will still answer your query, even if Node B's data is slightly outdated.
3. **Partition Tolerance (P):** The system continues to operate despite an arbitrary number of messages being dropped or delayed by the network between nodes (a "network partition").

### The Illusion of Choice
In reality, you don't get to choose two. You only get to choose one. 
**Partition Tolerance (P)** is absolutely mandatory. You do not control the internet. Network partitions *will* happen. Because 'P' is forced upon you, the CAP theorem really states: **When a network partition occurs, you must choose between Consistency (CP) and Availability (AP).**

## CODE

### Level 1 — Beginner (The Scenario)
Imagine you have an ATM network with two database nodes: Node A (New York) and Node B (London). 
Alice has $100 in her account. She deposits $50 at the New York ATM. Node A updates her balance to $150. 
Before Node A can send the new balance over the ocean to Node B, **the transatlantic internet cable is cut by a submarine.** A Network Partition (P) has occurred.

Immediately after, Alice calls her husband Bob in London and tells him to withdraw the $150. Bob goes to the London ATM (which connects to Node B). Node B still thinks Alice only has $100!

The system is now forced to make a decision.

### Level 2 — Building Blocks (CP: Consistency over Availability)
If you configure your database as a **CP System (Consistent and Partition Tolerant)**:

When Bob tries to check his balance in London, Node B will realize that it cannot communicate with Node A to verify the true balance. Because it cannot guarantee **Consistency**, it sacrifices **Availability**. Node B will lock up and return an error: *"System Offline for Maintenance."* 

Bob is angry that the ATM is broken, but the bank is happy because they didn't accidentally give Bob $150 when the London database only thought he had $100.
**Examples:** MongoDB, HBase, Redis (in certain cluster modes).
**Use Cases:** Banking, Billing, Inventory Management.

### Level 3 — Intermediate (AP: Availability over Consistency)
If you configure your database as an **AP System (Available and Partition Tolerant)**:

When Bob tries to check his balance in London, Node B will prioritize **Availability**. It will say, *"I can't talk to New York right now, but here is the data I currently have."* It will return a balance of $100.

Bob is confused because Alice told him there was $150. The data is inconsistent (stale). However, the ATM is still online and functional.
**Examples:** Cassandra, DynamoDB, CouchDB.
**Use Cases:** Social Media Feeds, Like Counters, IoT Sensor Logging. (If Twitter shows you a feed that is missing a tweet from 3 seconds ago, nobody cares. It is better to show an outdated feed than to crash the entire website).

### Level 4 — Advanced (Eventual Consistency and PACELC)
The CAP theorem is slightly outdated because it only discusses what happens *during* a network failure. A more modern framework is the **PACELC Theorem**. 

PACELC states: In the case of a **P**artition, you choose between **A**vailability and **C**onsistency. **E**lse (when the network is running perfectly fine), you choose between **L**atency and **C**onsistency.

If you want absolute Consistency even when the network is fine, Node A must pause and wait for Node B to acknowledge the write before telling the user "Success." This drastically increases **Latency**. 
If you want low Latency, Node A must tell the user "Success" instantly, and sync with Node B in the background. This creates **Eventual Consistency**—meaning for a brief window of milliseconds, the two nodes will hold different data.

## REAL_WORLD
Eventual Consistency is the backbone of the modern internet. 

When you post a video on YouTube, you can immediately see it on your channel. But if your friend searches for it 3 seconds later, they might not find it. The database nodes handling your profile have the video, but the database nodes handling the global Search Index haven't received the background synchronization yet. The system is "Eventually Consistent." Within a few minutes, all nodes worldwide will mathematically converge on the exact same data.

## INTERVIEW
**Q: What is a CA System?**
A: A CA system (Consistent and Available) sacrifices Partition Tolerance. However, because you cannot magically prevent network cables from breaking, a true CA distributed system cannot exist in the real world. The only true CA system is a single monolithic database server sitting on one desk (like a standard PostgreSQL database). If the network goes down, the client can't reach the database at all.

**Q: Can you configure Cassandra to be CP?**
A: Cassandra is famous for being a highly Available (AP) database. However, it offers "Tunable Consistency." You can set a parameter called `QUORUM`. If you have 5 nodes, you can tell Cassandra: *"When a user reads data, you MUST read from at least 3 nodes and make sure they all agree before returning the result."* This forces Cassandra to act more like a CP system, at the cost of significantly higher latency.

**Q: In a system design interview, if the interviewer asks for a "Highly Available" system, which database do you choose?**
A: You immediately lean towards NoSQL AP databases like Cassandra or DynamoDB, but you MUST clarify the use-case. *"Since you asked for High Availability, I recommend Cassandra, which will provide Eventual Consistency. Are we okay with users occasionally seeing a stale "Like" count on a post for a few seconds to achieve this 99.99% uptime?"*
