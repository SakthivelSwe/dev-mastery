## WHY
Data cannot travel faster than the speed of light. 

If your startup's only server is physically located in a data center in New York, a user sitting in New York will experience 20 milliseconds of latency to load your website. However, a user sitting in Sydney, Australia, has to send an HTTP request across the Pacific Ocean via underwater fiber optic cables. The round-trip time for a single packet might be 250 milliseconds. 

If your website requires downloading 50 images, a CSS file, and a React JavaScript bundle, that Australian user will stare at a blank white screen for 5 seconds while those large, static files slowly trickle across the ocean.

We solve the speed of light using a **CDN (Content Delivery Network)**. A CDN pushes your static files out of your main server and distributes them to hundreds of proxy servers physically located in every major city on Earth. 

## THEORY
### How a CDN Works (The Pull Model)
A CDN is basically a massive, geographically distributed Cache.

1. **Edge Servers:** The CDN provider (like Cloudflare, AWS CloudFront, or Fastly) owns physical servers in London, Tokyo, Sydney, Mumbai, and Sao Paulo. These are called Edge Servers or PoPs (Points of Presence).
2. **The Origin Server:** Your actual New York server is called the "Origin."
3. **The First Request (Cache Miss):** An Australian user requests `logo.png`. The request hits the Sydney Edge Server. The Sydney server doesn't have the image yet. It reaches across the ocean to the New York Origin, downloads the image, saves a copy in its local RAM/SSD, and returns it to the user.
4. **Subsequent Requests (Cache Hit):** The next 10,000 Australian users who request `logo.png` never touch the New York server. They download the image directly from the Sydney Edge Server in 10 milliseconds.

## CODE

### Level 1 — Beginner (Static Assets)
What exactly should you put on a CDN? **Static Assets**. 
Static assets are files that do not change based on who the user is. 

**DO PUT ON A CDN:**
- Images (`.jpg`, `.png`, `.svg`)
- Videos (`.mp4`)
- Compiled JavaScript Bundles (`main.js`)
- CSS Stylesheets (`styles.css`)

**DO NOT PUT ON A CDN:**
- Private API Responses (e.g., the JSON response containing the user's bank account balance). If you cache this globally, User B will see User A's bank balance!

### Level 2 — Building Blocks (Push vs Pull CDNs)
There are two ways to get your data onto the CDN:

1. **Pull CDN (The standard):** The CDN starts completely empty. When a user requests a file, the CDN "pulls" it from the Origin server on demand, and caches it for future users. This is perfect for high-traffic sites.
2. **Push CDN:** You manually write a script to "push" all your images up to the CDN servers before any users request them. This is incredibly expensive because you pay to store files on the CDN that users might never actually look at. It is rarely used unless you have strict compliance/security requirements where the Origin server is heavily firewalled and refuses incoming connections from the CDN.

### Level 3 — Intermediate (Cache Invalidation and Versioning)
Because a CDN is just a Cache, it suffers from the exact same Cache Invalidation problem as Redis.

If you update your `styles.css` file on the New York Origin server to make the background black instead of white, the Sydney CDN server won't know! It will continue serving the old white CSS file to Australian users until its Time-To-Live (TTL) expires (which could be 30 days!).

You cannot easily "Invalidate" a CDN because the file is spread across 500 servers globally. It takes time for the purge command to propagate.

**The Fix: File Versioning (Cache Busting)**
You never update a file on a CDN. You create a brand new file with a different name.

```html
<!-- ❌ BAD: If you update this file on the server, the CDN won't notice! -->
<link rel="stylesheet" href="https://cdn.example.com/styles.css" />

<!-- ✅ GOOD: Webpack/Vite appends a unique hash to the filename during the build process! -->
<link rel="stylesheet" href="https://cdn.example.com/styles.f8a91b.css" />
```
When you deploy a new version, Webpack generates `styles.c2x9z.css`. Because the filename is entirely new, the CDN sees it as a brand new request, instantly fetches it from the Origin, and caches it. You completely bypass the invalidation problem!

### Level 4 — Advanced (Dynamic Content Acceleration)
Wait, if you can't put private APIs on a CDN, is the Australian user still doomed to have 250ms latency when they try to fetch their bank balance?

Not necessarily. Modern CDNs offer **Dynamic Content Acceleration**. 
Instead of caching the bank balance, the Australian user connects to the Sydney CDN Edge server. The CDN provider maintains highly optimized, dedicated, private fiber-optic tunnels (backbones) directly between Sydney and New York. The CDN routes the API request through its private tunnel, bypassing the chaotic public internet and cutting the latency down significantly (e.g., from 250ms to 120ms). 

Furthermore, the CDN handles the SSL/TLS encryption handshake locally in Sydney, saving multiple round-trips across the ocean!

## REAL_WORLD
Cloudflare is the most ubiquitous CDN on earth, primarily because it acts as a **Reverse Proxy**. 
Instead of hosting your images on a separate URL like `https://cdn.yourstartup.com/logo.png`, you simply point your main Domain Name (`yourstartup.com`) directly at Cloudflare. 

Cloudflare intercepts every single request to your entire website. If it sees a request for an image, it serves it from the edge cache. If it sees a request for an API, it securely forwards it to your origin server. This also allows Cloudflare to act as a massive shield against **DDoS attacks** (Distributed Denial of Service), absorbing millions of malicious requests at the edge before they ever reach your fragile origin server.

## INTERVIEW
**Q: What is a CDN Edge Server?**
A: An Edge Server is a CDN proxy server located physically close to the user (at the "edge" of the network, far away from your centralized origin database).

**Q: If your Origin server completely crashes, does your website go offline?**
A: If you are using a CDN, your static assets (HTML, CSS, JS, Images) will remain completely online and accessible, served directly from the CDN caches! Users can load the website perfectly fine. However, the moment the JavaScript tries to make an HTTP request to your dynamic API, the request will fail because the Origin is dead. (Cloudflare specifically has an "Always Online" mode that serves a snapshot of the page if the Origin dies).

**Q: Can you execute logic on a CDN?**
A: Yes! This is a cutting-edge paradigm called **Edge Computing** (e.g., Cloudflare Workers, AWS Lambda@Edge). Instead of routing an API request back to New York to execute a simple script, you upload the script to the CDN. The CDN executes the JavaScript locally in Sydney, manipulating headers or returning custom JSON, providing 0ms execution latency.
