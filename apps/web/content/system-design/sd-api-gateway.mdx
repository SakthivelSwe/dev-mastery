## WHY
In a microservices architecture, you might have 50 different services: User Service, Order Service, Payment Service, Notification Service, etc. Without an API Gateway, your mobile app would need to know the internal network addresses of all 50 services, handle authentication for each one separately, and coordinate 10 parallel network calls to assemble a single page.

An **API Gateway** is a single entry point that sits in front of all your microservices. All client traffic goes through the Gateway, which handles cross-cutting concerns like authentication, rate limiting, routing, and load balancing — so your individual services don't have to.

## THEORY
### The API Gateway Pattern
The Gateway acts as a "smart reverse proxy." It:
1. **Routes** requests to the correct upstream microservice.
2. **Authenticates** the request (verifying JWT tokens).
3. **Rate Limits** the client.
4. **Transforms** requests and responses (e.g., protocol translation from REST to gRPC).
5. **Aggregates** responses from multiple services into one response (Backend for Frontend pattern).
6. **Caches** frequent responses.
7. **Logs** and **traces** every request.

### Level 1 — Beginner (The Routing Role)
```
Without API Gateway (Client must know all services!):
Client → POST https://user-service.internal:8001/login
Client → GET  https://product-service.internal:8002/products
Client → POST https://order-service.internal:8003/orders

With API Gateway (Single public endpoint!):
Client → POST https://api.devmastery.com/v1/auth/login
           └── Gateway routes to → user-service:8001/login
           
Client → GET  https://api.devmastery.com/v1/products
           └── Gateway routes to → product-service:8002/products

Client → POST https://api.devmastery.com/v1/orders
           └── Gateway routes to → order-service:8003/orders
```

### Level 3 — Intermediate (The Backend for Frontend Pattern)
Different clients have different data needs:
- Mobile App: Needs a lightweight response (only 5 fields) to save bandwidth.
- Web Browser: Needs a rich response with embedded related data.
- Smart TV: Needs a specialized format.

Instead of one generic API, you create **dedicated gateways per client type**:

```
Mobile App → Mobile BFF Gateway → (aggregates data from User + Order services)
Web Browser → Web BFF Gateway → (aggregates data from User + Product + Order + Analytics)
Smart TV → TV BFF Gateway → (only fetches media metadata)

Each BFF is a thin gateway (Node.js + Express, or Spring Cloud Gateway)
that orchestrates the underlying microservice calls.

Benefits:
- Each client gets exactly the data it needs (no over-fetching).
- Frontend teams own their BFF — they control the aggregation logic.
- Backend microservices remain clean and generic.
```

### Level 4 — Advanced (Service Mesh vs API Gateway)
```
API GATEWAY:
  - Positioned at the EDGE (external clients → internal services).
  - Handles North-South traffic (traffic entering/leaving the cluster).
  - One gateway, many services.
  - Focused on: Auth, Rate Limiting, Routing, SSL Termination.

SERVICE MESH (Istio, Linkerd):
  - Positioned INSIDE the cluster (service-to-service communication).
  - Handles East-West traffic (service A → service B → service C).
  - Each service gets a "sidecar" proxy (e.g., Envoy Proxy).
  - Focused on: mTLS encryption, Circuit Breaking, Observability, Traffic Splitting.

Both work TOGETHER in production Enterprise systems:
Client → API Gateway (Auth, Rate Limiting) → Service Mesh (mTLS, Circuit Breaker) → Services
```

## REAL_WORLD
Major API Gateways used in Enterprise:
- **AWS API Gateway**: Fully managed, native AWS integration, pay-per-request pricing.
- **Kong**: Open-source, extremely extensible with plugins, self-hosted or cloud.
- **Nginx**: Ultra-high performance, commonly used as a basic reverse proxy/gateway.
- **Spring Cloud Gateway**: Java/Spring native, deep Spring Security integration.
- **Traefik**: Docker/Kubernetes native, auto-discovers services, free tier.

## INTERVIEW
**Q: What is SSL/TLS Termination at the API Gateway?**
A: The API Gateway decrypts HTTPS traffic from the client (handles the TLS handshake), then forwards the request to internal services over plain HTTP. This offloads expensive encryption/decryption from backend services to the Gateway. Traffic within the internal cluster is often trusted and unencrypted (or uses mutual TLS via a service mesh), while the Gateway provides the public-facing HTTPS endpoint.

**Q: What is the difference between an API Gateway and a Load Balancer?**
A: A **Load Balancer** is a Layer 4 (TCP) or Layer 7 (HTTP) device that distributes traffic across instances of the SAME service. It doesn't understand the application's routing logic. An **API Gateway** is always Layer 7 and routes different URL paths to completely DIFFERENT upstream services — it understands the application's structure. Modern cloud load balancers (AWS ALB) blur this distinction with path-based routing.

// Robust error handling for production use
function safeProcess(input) {
  try {
    if (input === null || input === undefined) {
      throw new TypeError(`[API Gateway] Input cannot be null or undefined`);
    }
    const instance = createAPIGateway();
    return { success: true, data: instance.process(input) };
  } catch (err) {
    console.error(`[API Gateway] Error:`, err.message);
    return { success: false, error: err.message };
  }
}

console.log(safeProcess(null));            // { success: false, error: '...' }
