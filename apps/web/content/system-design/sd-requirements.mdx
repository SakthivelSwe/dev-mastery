## WHY
In Software Engineering, the #1 cause of project failure is not choosing the wrong database, nor is it writing inefficient algorithms. The #1 cause of project failure is **building the wrong system**. 

If you dive straight into drawing architecture diagrams without understanding the exact constraints, scale, and business goals of the product, you will over-engineer a simple internal tool, or drastically under-engineer a massive global platform. 

In a System Design Interview, if the interviewer says "Design Twitter," and you immediately start talking about Cassandra databases and Redis clusters, **you have already failed.** The very first step of any system design process is **Requirements Gathering**.

## THEORY
### Functional vs. Non-Functional Requirements
To design a system, you must slice the requirements into two distinct categories:

1. **Functional Requirements (The "What")**
   - What exactly does the system do?
   - What are the core features the user will interact with?
   - *Example (Twitter)*: Users can post tweets. Users can follow other users. Users can see a timeline of tweets from people they follow.

2. **Non-Functional Requirements (The "How" / Constraints)**
   - How well does the system perform?
   - What are the scale, latency, availability, and consistency constraints?
   - *Example (Twitter)*: The system must be Highly Available (HA). Latency for viewing the timeline must be under 200ms. The system must handle 100 million Daily Active Users (DAU).

## CODE

### Level 1 — Beginner (The "Happy Path" Functional Requirements)
Start by defining the absolute minimum viable product (MVP) features. Do not overcomplicate the system with edge cases yet.

**Scenario:** "Design a URL Shortener (like Bitly)"

**Functional Requirements:**
1. Given a long URL, the system returns a much shorter, unique URL.
2. When a user clicks the short URL, they are instantly redirected to the original long URL.
3. Users can optionally pick a custom alias for their short URL (e.g., `bit.ly/my-custom-name`).
4. Links expire after a default timespan (e.g., 5 years).

### Level 2 — Building Blocks (Non-Functional Requirements)
Once the features are defined, you must define the constraints that will dictate your database choices and server architecture.

**Non-Functional Requirements for the URL Shortener:**
1. **High Availability:** If the service goes down, millions of links across the internet stop working. Availability is strictly prioritized over Consistency.
2. **Low Latency:** URL redirection should happen in real-time with minimal delay (e.g., < 100ms).
3. **Scale:** The system will generate 100 million new URLs per month.
4. **Read/Write Ratio:** The system is extremely read-heavy. (For every 1 link created, it is clicked 100 times). The Read:Write ratio is 100:1.

### Level 3 — Intermediate (Defining the API Contracts)
Once you have the requirements, you immediately define the API boundaries. This proves you understand how the Client and Server will communicate before you worry about how the Server stores the data.

```typescript
// 1. Create Short URL API
POST /api/v1/data/shorten
Request Body:
{
  "original_url": "https://www.example.com/very/long/path/article",
  "custom_alias": "my-article", // Optional
  "expiration_date": "2030-01-01T00:00:00Z" // Optional
}

Response (201 Created):
{
  "short_url": "https://tiny.url/xyz123"
}
```

```typescript
// 2. Redirect API
GET /api/v1/{short_alias}
Response (301 Moved Permanently):
Headers: {
  "Location": "https://www.example.com/very/long/path/article"
}
```

### Level 4 — Advanced (Extended / Out-of-Scope Requirements)
A Senior Engineer knows exactly where to draw the line. In a 45-minute interview (or a tight Agile sprint), you cannot build the entire universe. You must explicitly state what you are **NOT** building.

**Out of Scope (To be handled later):**
- User Accounts and Authentication.
- Advanced Analytics (tracking exactly which countries clicked the link).
- Billing and Subscription tiers.
- A frontend web UI.

By explicitly stating these are out of scope, you prevent scope creep and focus your architecture entirely on the core scaling problem.

## REAL_WORLD
In the real world, Non-Functional Requirements are often dictated by **Service Level Agreements (SLAs)**. 

If you are building an API for a bank, the SLA might guarantee **99.999% Uptime** (known as "Five Nines"). Five nines means the system is legally only allowed to be down for **5 minutes and 15 seconds per YEAR**. 
If you fail to meet this requirement, your company owes the client millions of dollars in penalty payouts. Designing a system for 99.9% uptime is relatively easy. Designing it for 99.999% uptime requires massive geographic redundancy, automated failovers, and chaos engineering, multiplying the engineering cost by 10x.

## INTERVIEW
**Q: The interviewer asks "Design Netflix." What is the very first thing you say?**
A: You ask clarifying questions to narrow the scope. *"Are we designing the video streaming backend (content delivery), or the client-facing web application (browsing, recommendations, user profiles)?"* You cannot design both in 45 minutes.

**Q: Why do we care about the Read/Write Ratio?**
A: It dictates your caching and database strategy. If a system is Read-Heavy (like Twitter or a URL Shortener), you will heavily invest in caching layers (Redis/Memcached) and Database Read Replicas. If a system is Write-Heavy (like IoT sensor logging or financial transaction ledgers), caching won't help you; you need a database optimized for massive concurrent appends (like Cassandra or a Time-Series DB).

**Q: What is the difference between a 301 and 302 redirect for our URL Shortener?**
A: A `301 (Moved Permanently)` tells the browser to cache the redirect forever. This drastically reduces the load on your servers, but makes it impossible to track analytics (because the browser stops asking your server where the link goes). A `302 (Found / Temporary)` forces the browser to hit your server every single time, increasing server load but allowing you to track exactly how many times the link was clicked.
