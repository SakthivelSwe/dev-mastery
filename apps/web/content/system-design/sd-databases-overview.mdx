## WHY
In 1990, picking a database was easy: you bought Oracle or Microsoft SQL Server. Data was neat, structured, and fit perfectly into tables.

Today, companies deal with wildly different types of data. 
- A Bank needs absolute mathematical certainty that if you transfer $100, the money doesn't magically disappear if the server loses power midway through the transaction.
- An IoT company collects 50 million temperature sensor readings per second. They don't care if they lose 2 readings, but the database *must* be able to ingest 50 million writes per second without crashing.
- A Social Network needs to traverse a graph of 2 billion users to find "Friends of Friends who like Pizza."

No single database can do all of these things efficiently. You must choose the right tool for the job.

## THEORY
### The Great Divide: SQL vs NoSQL
The database ecosystem is heavily divided into two primary categories:

1. **SQL (Relational):** Data is highly structured into strict Tables with Columns and Rows (like an Excel spreadsheet). Relationships between tables are enforced by the database engine.
2. **NoSQL (Non-Relational):** Data is unstructured or semi-structured. It does not use strict tables. It is designed to scale horizontally across hundreds of cheap servers seamlessly.

### The Four Types of NoSQL
"NoSQL" is not a specific database; it is an umbrella term for four completely different architectures:
1. **Key-Value Stores:** (Redis, DynamoDB)
2. **Document Stores:** (MongoDB, Couchbase)
3. **Wide-Column Stores:** (Cassandra, HBase)
4. **Graph Databases:** (Neo4j, Amazon Neptune)

## CODE

### Level 1 — Beginner (Relational / SQL Databases)
**Examples:** PostgreSQL, MySQL, Oracle.
**When to use them:** You should default to PostgreSQL for 90% of all applications you build. Use it when data is highly structured, relationships are clear, and you require **ACID transactions** (guaranteed data integrity for financial/billing systems).

```sql
-- The strict structure of a Relational Database
CREATE TABLE users (
  id INT PRIMARY KEY,
  name VARCHAR(50) NOT NULL,
  balance DECIMAL(10, 2) NOT NULL
);

-- Strict relationships via Foreign Keys
CREATE TABLE orders (
  id INT PRIMARY KEY,
  user_id INT REFERENCES users(id),
  total DECIMAL(10, 2)
);
```

### Level 2 — Building Blocks (Document Stores)
**Examples:** MongoDB, CouchDB.
**When to use them:** When your data structure is constantly changing (e.g., an E-commerce catalog where a TV has a "Screen Size" property but a T-Shirt has a "Fabric Type" property). Relational databases require you to alter the table schema to add columns, which locks the database. Document stores simply save raw JSON objects, allowing ultimate flexibility.

```json
// A MongoDB Document
// There is no schema. You can insert this exact object into the database.
{
  "_id": "5f1b2c3d4e5f",
  "name": "Samsung TV",
  "price": 999.99,
  "specs": {
    "screen_size": 55,
    "resolution": "4K"
  },
  "reviews": [
    { "user": "Alice", "rating": 5 },
    { "user": "Bob", "rating": 4 }
  ]
}
```

### Level 3 — Intermediate (Key-Value & Wide-Column Stores)
**Key-Value Stores (Redis, DynamoDB):** 
Used for ultra-fast, low-latency lookups. You cannot run complex queries like "Find all users older than 25." You can ONLY query by the exact Key. Used for Caching, Session Management, and Shopping Carts.

**Wide-Column Stores (Cassandra):** 
Invented by Facebook. Designed for absolutely massive scale and extreme **Write-Heavy** workloads. Used for logging, Time-Series data, and IoT sensor metrics. It spreads data across hundreds of decentralized nodes. There is no concept of JOINs.

### Level 4 — Advanced (Graph Databases)
**Examples:** Neo4j, Amazon Neptune.
**When to use them:** When the *relationships* between data are more important than the data itself. If you try to find "Friends of Friends of Friends" in an SQL database, you have to run a 4-way `JOIN`, which requires scanning massive tables and will take 10 seconds. Graph databases treat relationships as physical memory pointers, allowing them to traverse billions of connections in milliseconds.

```cypher
// Cypher Query Language (Neo4j)
// Find Alice, find all her friends, and return the restaurants they like!
MATCH (u:User {name: "Alice"})-[:FRIENDS_WITH]->(friend:User)-[:LIKES]->(r:Restaurant)
RETURN r.name
```

## REAL_WORLD
Enterprise architectures use **Polyglot Persistence**. They do not pick just one database.

If you are building Uber:
- The user's billing information and account balance are stored in **PostgreSQL** (SQL) for strict ACID guarantees.
- The real-time driver locations are streamed into **Redis** (Key-Value) for sub-millisecond updates.
- The history of every trip ever taken is dumped into **Cassandra** (Wide-Column) because it generates millions of massive write operations per day that would crush Postgres.
- The driver-rider matching algorithm uses **Neo4j** (Graph) to calculate the most efficient geographic routes and relationships.

## INTERVIEW
**Q: Can you use MongoDB for a banking application?**
A: Historically, no. MongoDB did not support multi-document ACID transactions, meaning money could vanish during a transfer if the server crashed. However, modern MongoDB (v4.0+) *does* support distributed ACID transactions. Despite this, the industry still heavily prefers traditional SQL databases (like Postgres) for financial data because the schema strictly enforces data types, preventing a developer from accidentally saving a string `"100"` into an integer balance column.

**Q: Why is scaling an SQL database hard?**
A: SQL databases are designed to run on a single machine to guarantee data integrity and support complex `JOIN`s. To scale them horizontally across multiple machines (Sharding), you have to break the tables apart, destroying the ability to perform `JOIN`s and Foreign Key constraints, essentially turning your highly structured SQL database into a clunky NoSQL database. 

**Q: What is a Vector Database?**
A: A massive trend in the AI era (e.g., Pinecone, Milvus). They do not store text or numbers; they store high-dimensional Arrays of floats (Vectors). When you type a prompt into an LLM, the LLM converts the text into a Vector, and the Vector Database performs "Similarity Search" to find other Vectors that are mathematically close to it, enabling Semantic Search and Retrieval-Augmented Generation (RAG).
