## WHY
"Design a Notification System" is a deceptively simple interview question. Sending one email is easy. 
But how do you design a system that sends 10 million push notifications to iOS/Android devices simultaneously when a celebrity tweets, without dropping a single message, while respecting user "Do Not Disturb" preferences, and without getting banned by Apple/Google for rate-limiting violations?

This question tests your knowledge of **Asynchronous Task Queues**, **Third-Party Integrations**, and **Retry Mechanisms**.

## THEORY
### The Three Notification Types
A modern notification system must abstract away the delivery method. The core system handles the business logic, and dedicated workers handle the actual delivery.
1. **iOS Push (APNS):** Apple Push Notification Service.
2. **Android Push (FCM):** Firebase Cloud Messaging.
3. **SMS / Email:** Twilio, SendGrid.

## CODE

*(In a System Design interview, you don't write code, you draw the architecture diagram and explain the data flow!)*

### Level 1 — The API and Validation
**The Explanation:**
1. "A microservice (e.g., the Billing Service) needs to notify a user that their payment failed."
2. "It sends an HTTP POST request to our **Notification Service API**: `POST /notify { userId: 123, template: "payment_failed" }`."
3. "The Notification Service first hits the Database to validate the user. Do they exist? Do they have 'Payment Alerts' disabled in their preferences? Are they currently in 'Do Not Disturb' mode?"
4. "If the notification passes validation, we must look up their device tokens. A user might have an iPhone, an iPad, and an Android phone. We fetch all active device tokens for `userId: 123`."

### Level 2 — The Message Queues (The Buffer)
*The Interviewer Trap:* "The Billing Service sent 5 million notification requests in one minute. Can the Notification Service just call the Apple API directly?"
**Your Answer:** "Absolutely not. The Apple API is a network call. It is slow and has strict rate limits. If we call it synchronously, our Notification Service will block, run out of threads, and crash. We MUST use Message Queues."

**The Explanation:**
1. "The Notification Service takes the validated payload and drops it into a **Message Broker** (Kafka or RabbitMQ)."
2. "Crucially, we use **Topic Separation**. We have a dedicated queue for iOS (`queue_ios`), a queue for Android (`queue_android`), and a queue for Email (`queue_email`)."
3. "Why? Because if the Apple API goes down for 5 minutes, the iOS queue will back up. But because they are separated, Android and Email notifications continue flowing perfectly without being blocked by Apple's outage!"

### Level 3 — The Worker Nodes & Third-Party Integration
**The Explanation:**
1. "We have dedicated clusters of **Worker Nodes** assigned to each queue."
2. "The iOS Workers pull messages from the `queue_ios` Kafka topic."
3. "The Workers construct the exact JSON payload required by Apple, open a persistent HTTP/2 connection to Apple APNS, and push the notifications."
4. "The Workers are responsible for handling Rate Limits. If Apple returns an HTTP 429 (Too Many Requests), the Worker catches the error, pauses, and safely puts the message *back* into the queue to be retried later."

### Level 4 — Analytics and State Tracking
*The Interviewer Trap:* "The user complains they never received the notification. How does your support team know what happened to it?"

**The Solution:**
1. "Every notification generated by the API receives a unique `Notification_ID`."
2. "When the Notification Service drops the message into Kafka, it simultaneously inserts a row into the **Notification Log Database** (Cassandra or Elasticsearch) with status `PENDING`."
3. "When the iOS Worker successfully receives a 200 OK from Apple, it drops a 'Success' event into a separate Kafka topic (`notification_events`)."
4. "An Analytics Worker consumes this topic and updates the database row to `DELIVERED`."
*(Advanced: When the user actually clicks the notification on their iPhone, the app sends an API call back to our server, and we update the status to `OPENED`!)*

## REAL_WORLD
**Deduplication**
A massive problem in distributed systems is the "At-Least-Once" delivery guarantee of message queues like Kafka. If a worker node processes an email, sends it via SendGrid, but crashes *before* it can acknowledge the message to Kafka, Kafka will assume it failed and hand the exact same message to another worker! The user gets the email twice.

To prevent this, the Workers must be **Idempotent**. 
You use a fast distributed cache (like Redis). Before the Worker sends the email to SendGrid, it runs: `Redis.setnx(Notification_ID)`. (Set if Not Exists). 
If Redis returns `1` (Success), the Worker sends the email. 
If Redis returns `0` (Failed), it means another worker already processed this exact Notification_ID! The Worker simply drops the message and moves on, guaranteeing Exactly-Once delivery logic.

## INTERVIEW
**Q: A celebrity with 50 million followers tweets. If we loop through 50 million user IDs in the Notification Service, validate their preferences, and push 50 million messages into Kafka, the Database will crash and the process will take hours. How do you design for this 'Fan-Out' scale?**
A: **Pre-computation and Dedicated Fast-Lanes.**
First, we do not query the database 50 million times. The follower list and their device tokens are highly cacheable. We keep massive JSON blobs of active device tokens for the celebrity's followers cached in Redis.
Second, we use the **Fan-Out on Write** pattern. Instead of one server pushing 50 million messages to Kafka, a Master Coordinator splits the 50 million followers into 5,000 chunks of 10,000 users. It pushes these 5,000 "Chunk Tasks" into a queue. Thousands of Worker Nodes pick up the chunks and process them perfectly in parallel. 
Finally, we implement **Priority Queues**. A direct message from your mom should not be delayed by 5 minutes because a celebrity tweeted. We have a "High Priority Queue" for 1-to-1 messages, and a "Bulk Queue" for massive fan-outs, ensuring standard user traffic remains instantly responsive.
